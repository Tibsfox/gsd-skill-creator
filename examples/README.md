# examples/ — The Artifact Library

This directory is the canonical library of every Tibsfox-derived skill, agent, team, and chipset developed with `gsd-skill-creator`. It is a menu, not a package — browse the categories, then use `tools/install.mjs` to pull what you want into your target `.claude/` directory.

## Directory structure

```
examples/
├── CATEGORIES.md            — taxonomy definitions
├── CHANGELOG.md             — narrative archaeology of the library's evolution
├── LICENSE-POLICY.md        — BSL 1.1 + exemption rule
├── .count-badge.md          — auto-generated artifact counts
│
├── skills/
│   ├── gsd/                 research/   media/      dev/        ops/
│   ├── workflow/            patterns/   orchestration/   state/   deprecated/
│
├── agents/
│   ├── gsd/   research/   media/   dev/   ops/   ui/   audit/   deprecated/
│
├── teams/
│   ├── code/   ops/   infra/   migration/   deprecated/
│
├── chipsets/
│   ├── <name>/chipset.yaml + README.md
│   └── deprecated/
│
└── tools/
    ├── install.mjs          — pick what to install
    ├── validate.mjs          — frontmatter + structure checker
    ├── catalog-gen.mjs       — rebuilds .planning/artifact-catalog.csv + .count-badge.md
    └── license-report.mjs    — audits BSL vs exempt
```

See [`CATEGORIES.md`](CATEGORIES.md) for what belongs in each category.

## How to install

`tools/install.mjs` copies artifacts out of `examples/` and into a target `.claude/` directory. It only installs `status: stable` artifacts by default; experimental and deprecated are available with explicit flags.

```bash
# Dry-run: show what would be installed
node examples/tools/install.mjs --dry-run

# Install a single named artifact
node examples/tools/install.mjs --name audio-engineer --type agent

# Install all skills in a category
node examples/tools/install.mjs --type skills --category media

# Install everything (explicit opt-in)
node examples/tools/install.mjs --all --confirm

# Target a specific location
node examples/tools/install.mjs --name research-engine --target ~/.claude
node examples/tools/install.mjs --name research-engine --target ./my-project/.claude

# Include deprecated or experimental
node examples/tools/install.mjs --name old-pattern --include-deprecated
```

The installer writes an audit log to `~/.claude/.gsd-install-log.jsonl` so you can see what was installed and when.

## Authoring principles

These are the patterns every artifact in the library should follow.

### 1. Effective descriptions

Use the "Use when..." pattern with specific trigger keywords:

```yaml
# GOOD — activates reliably
description: Generates conventional commit messages. Use when committing changes,
  writing commit messages, or when user mentions 'commit', 'conventional commits'.

# BAD — unreliable activation
description: Helps with git commits
```

The description field is how Claude Code determines when to apply a skill. Vague descriptions lead to unreliable activation (testing suggests proper descriptions improve activation from ~20% to ~90%).

### 2. Domain separation

Each artifact addresses a distinct concern. Avoid semantic overlap where two skills or agents try to activate for the same user request. The category taxonomy in `CATEGORIES.md` is designed to keep concerns separated — if you can't decide which category something belongs in, it may indicate the artifact is too broad.

### 3. Security-first design

All artifacts follow security-conscious patterns:

- **Read-only by default** — agents use minimal tool access (Read, Glob, Grep)
- **No destructive operations** — no `rm -rf`, force push, DROP TABLE, or similar in default behavior
- **Secrets protection** — never display secret values; recommend `.gitignore` patterns
- **Scoped write access** — agents that write are restricted to specific files/directories
- **Safe team topologies** — read-only review teams; write-scoped generation teams

### 4. Actionable content

Prefer concrete guidance over abstract description:

- Tables for quick reference
- Code examples for patterns
- Checklists for processes
- Anti-patterns to avoid

### 5. Frontmatter convention

Every artifact carries a 9-field frontmatter block. Minimum required:

```yaml
---
name: my-skill
type: skill                   # skill | agent | team | chipset
category: dev                 # see CATEGORIES.md
status: stable                # stable | experimental | deprecated
origin: tibsfox               # tibsfox | gsd | taches-cc-resources | community
modified: false               # true if Tibsfox-changed from upstream
first_seen: 2026-04-10
first_path: examples/skills/dev/my-skill/
description: Short "Use when..." description.
superseded_by: null
---
```

Existing rich frontmatter (`token_budget`, `activation_triggers`, `tools`, etc.) is preserved on artifacts that already have it. Only the 9 fields above are added by Stage 2 back-fill.

See [`LICENSE-POLICY.md`](LICENSE-POLICY.md) for how `origin` and `modified` determine licensing.

## Creating your own

Use these examples as templates:

1. **Copy** an example that's similar to what you want to build
2. **Rename** the directory or filename to your artifact's name
3. **Update** the `name` field in frontmatter to match
4. **Revise** the `description` with your specific "Use when..." triggers
5. **Set** `origin: tibsfox` and `modified: false` (for a new artifact you authored)
6. **Replace** the content with your own guidance
7. **Validate** with `node examples/tools/validate.mjs --name your-artifact`

## Verifying no conflicts

```bash
node examples/tools/validate.mjs --strict
```

The validator checks frontmatter completeness, category consistency, naming rules, deprecation invariants, and catalog drift.

## Deprecation

When an artifact is superseded by a replacement:

1. Change `status: stable` → `status: deprecated`
2. Fill `superseded_by: <name-of-replacement>`
3. Move the artifact into `<type>/deprecated/`
4. Add a dated entry to `CHANGELOG.md`

Deprecated artifacts are kept in the repo forever. They remain browsable and the archaeological intent of the library is that old work is preserved, not hidden.
