# Roadmap: GSD Skill Creator v1.8

## Overview

v1.8 brings capability-aware planning and token efficiency to the GSD workflow. The journey starts with refactoring SkillApplicator into a pipeline abstraction (highest-risk change), then unlocking token budgets, building the capability manifest, wiring capabilities into phases/plans/execution, and layering on cache optimization, research compression, model guidance, post-phase hooks, and parallelization analysis. Ten phases, each delivering a verifiable capability.

- v1.8 Capability-Aware Planning + Token Efficiency - Phases 52-61 (in progress)

## Phase Details

### Phase 52: Pipeline Extraction
**Goal**: SkillApplicator delegates to a composable SkillPipeline so every subsequent feature adds a pipeline stage instead of modifying a monolithic apply() method
**Depends on**: Nothing (first phase of v1.8)
**Requirements**: None directly (infrastructure enabling TOK-01, TOK-04, and all features that modify skill loading)
**Success Criteria** (what must be TRUE):
  1. SkillApplicator.apply() is under 25 lines and delegates to SkillPipeline stages
  2. SkillPipeline processes skills through Score, Resolve, and Load stages with the same observable behavior as before refactoring
  3. A new pipeline stage can be added without modifying SkillApplicator or existing stages
  4. All existing tests pass without modification (pure refactor, no behavior changes)
**Plans:** 2 plans
Plans:
- [x] 52-01-PLAN.md — TDD: SkillPipeline runner infrastructure (PipelineStage, PipelineContext, SkillPipeline class)
- [x] 52-02-PLAN.md — Extract Score/Resolve/Load stages, refactor SkillApplicator facade, wire exports

### Phase 53: Token Budget Tiers
**Goal**: GSD agents operate within configurable token budgets with priority tiers that prevent skill starvation while surfacing what was skipped
**Depends on**: Phase 52
**Requirements**: TOK-04, TOK-07
**Success Criteria** (what must be TRUE):
  1. Each GSD agent has a token budget profile with critical/standard/optional priority tiers
  2. Critical-priority skills load even when standard budget is exhausted (up to a hard ceiling)
  3. When skills are skipped due to budget limits, the skipped list with reasons is surfaced to the user
  4. Running `budget-estimate` CLI reports token cost estimates for skill loading with threshold warnings (50/80/100%)
  5. Default budget increases from 3% to 5-8% for capability-aware mode
**Plans:** 3 plans
Plans:
- [ ] 53-01-PLAN.md — TDD: Budget profile types, PipelineContext extension, and BudgetStage implementation
- [ ] 53-02-PLAN.md — Wire BudgetStage into SkillApplicator, extend ApplyResult, export types
- [ ] 53-03-PLAN.md — budget-estimate CLI command with tier breakdown and threshold warnings

### Phase 54: Capability Manifest
**Goal**: Users see an accurate, auto-generated inventory of all available skills, agents, and teams without manual maintenance
**Depends on**: Phase 52
**Requirements**: CAP-01, CAP-07
**Success Criteria** (what must be TRUE):
  1. CAPABILITIES.md is auto-generated with a machine-readable structured format listing all skills, agents, and teams (global + project-local)
  2. Running new-project or new-milestone triggers capability discovery that populates CAPABILITIES.md
  3. CAPABILITIES.md includes version stamps (git hash or content hash) so staleness is detectable
  4. Regenerating CAPABILITIES.md from the same source produces identical output (deterministic)
**Plans:** 4 plans
Plans:
- [x] 54-01-PLAN.md — TDD: Capability types and CapabilityDiscovery service (dual-scope skill/agent/team scanning)
- [x] 54-02-PLAN.md — TDD: Deterministic manifest renderer (YAML frontmatter + markdown tables)
- [x] 54-03-PLAN.md — TDD: CLI command, barrel exports, public API wiring
- [x] 54-04-PLAN.md — Wire capability discovery into new-project and new-milestone workflows (gap closure)

### Phase 55: Phase and Plan Capability Declarations

  2. Plan frontmatter includes a capabilities field that inherits from the parent phase declarations
  3. Capability references in phases and plans are validated against CAPABILITIES.md (unknown capabilities produce warnings)

### Phase 56: Skill Injection and Dynamic Creation
**Goal**: Executors automatically receive the skills they need based on plan capabilities, and phases can scaffold new capabilities as plan tasks
**Depends on**: Phase 55
**Requirements**: CAP-04, CAP-05
**Success Criteria** (what must be TRUE):
  1. When a plan declares capabilities with `use` verb, the referenced skills are auto-injected into executor context without manual loading
  2. When a phase declares capabilities with `create` verb, plan tasks are generated that scaffold new skill/agent files in project-local `.claude/`
  3. Auto-injected skills respect the token budget system from Phase 53 (critical capabilities load first)
  4. Created capabilities appear in CAPABILITIES.md on next regeneration
**Plans:** 3 plans
Plans:
- [x] 56-01-PLAN.md — TDD: SkillInjector service (resolve use-verb capabilities to disk content with critical budget tier)
- [x] 56-02-PLAN.md — TDD: CapabilityScaffolder service (generate scaffold tasks for create-verb capabilities)
- [x] 56-03-PLAN.md — Wire barrel exports, public API, and GSD workflow documentation (executor, planner, execute-phase)

### Phase 57: Cache-Aware Ordering
**Goal**: Skills are ordered to maximize Anthropic prompt cache hits by grouping stable content before dynamic content
**Depends on**: Phase 52, Phase 56
**Requirements**: TOK-01
**Success Criteria** (what must be TRUE):
  1. Skills declare cacheTier metadata (static/session/dynamic) in their frontmatter
  2. Skills are sorted by cacheTier within relevance bands (cache ordering is a tiebreaker, not a primary sort)
  3. Ordering is deterministic: same inputs always produce same output (tier then alphabetical tiebreaking)
  4. Relevance quality is preserved (cache ordering never changes which skills survive the maxSkills cut)
**Plans:** 2 plans
Plans:
- [x] 57-01-PLAN.md — TDD: CacheTier type and CacheOrderStage (pipeline stage reordering within relevance bands)
- [x] 57-02-PLAN.md — Wire CacheOrderStage into pipeline, add cacheTier to extension type, update exports

### Phase 58: Research Compression
**Goal**: Research artifacts are compressed to distilled skill files at 10-20x reduction so executors get actionable guidance without context bloat
**Depends on**: Phase 54
**Requirements**: TOK-02
**Success Criteria** (what must be TRUE):
  1. Running `compress-research` CLI converts research files to distilled skill files (target: 2-5KB from 20-35KB source)
  2. Compressed skills include `source: auto-generated` and `generatedFrom` metadata linking back to source file with content hash
  3. When the source research file changes, derived skills are detected as stale and excluded from loading
  4. Manual skills always win over auto-generated skills in conflict resolution
**Plans:** 3 plans
Plans:
- [x] 58-01-PLAN.md — TDD: ResearchCompressor service (extract key insights, generate distilled skill content, content hash tracking)
- [x] 58-02-PLAN.md — TDD: StalenessChecker service (staleness detection, manual-wins conflict resolution)
- [x] 58-03-PLAN.md — Wire compress-research CLI command, barrel exports, integration tests

### Phase 59: Model-Aware Activation
**Goal**: Skills declare which model tier they target so the right skills load for the right model profile
**Depends on**: Phase 54
**Requirements**: TOK-03
**Success Criteria** (what must be TRUE):
  1. Skills can declare modelGuidance metadata specifying preferred models (opus/sonnet/haiku) and minimum capability level
  2. When the active model profile does not match a skill's modelGuidance, the skill is gracefully skipped (not an error)
  3. Skipped-due-to-model-mismatch skills appear in the skipped list with the reason
  4. Default behavior (no modelGuidance declared) loads the skill regardless of model profile (backward compatible)
**Plans:** 2 plans
Plans:
- [x] 59-01-PLAN.md — TDD: ModelGuidance type and ModelFilterStage pipeline stage with model-to-tier mapping
- [x] 59-02-PLAN.md — Wire ModelFilterStage into SkillApplicator pipeline, barrel exports, public API

### Phase 60: Post-Phase Invocation and Collector Agents
**Goal**: Phases can trigger capabilities automatically after completion, and read-only collector agents can be generated for context-efficient information gathering
**Depends on**: Phase 56
**Requirements**: CAP-06, TOK-05
**Success Criteria** (what must be TRUE):
  1. When a phase declares capabilities with `after` verb, those capabilities are automatically invoked after phase completion and verification
  2. Read-only collector agents can be auto-generated via CLI that gather information and return compressed summary output
  3. Generated collector agents follow official Claude Code agent format and pass validation
  4. Post-phase invocation integrates with the existing verify-phase workflow (after triggers fire only when verification passes)
**Plans:** 3 plans
Plans:
- [x] 60-01-PLAN.md — TDD: PostPhaseInvoker service (resolve after-verb capabilities to invocation instructions)
- [x] 60-02-PLAN.md — TDD: CollectorAgentGenerator (generate read-only agents with compressed output format)
- [x] 60-03-PLAN.md — Wire barrel exports, generate-collector CLI command, verify-phase workflow integration

### Phase 61: Parallelization Advisor
**Goal**: Plan dependencies are automatically analyzed to recommend wave-based parallel execution instead of manual wave composition
**Depends on**: Phase 55
**Requirements**: TOK-06
**Success Criteria** (what must be TRUE):
  1. Running the parallelization advisor on a set of plans produces wave assignments based on file and dependency analysis
  2. Plans that modify the same files are assigned to the same wave (sequential) and independent plans to different waves (parallel)
  3. Output is advisory only (recommendations with rationale, not automatic execution changes)
  4. Conservative defaults: plans are sequential unless proven independent
**Plans:** 2 plans
Plans:
- [x] 61-01-PLAN.md — TDD: ParallelizationAdvisor service (parse plan frontmatter, build file dependency graph, compute wave assignments, generate advisory report)
- [x] 61-02-PLAN.md — Wire advise-parallelization CLI command, barrel exports, public API
