Create `docs/vv/e2e-user-scenario-verification.md` -- a comprehensive user scenario verification document structured as a NASA SE Phase E operational readiness procedure.

The document must contain these sections in order:

**1. Header and Purpose**
- Title: "End-to-End User Scenario Verification Procedure"
- Requirement traceability: VERIF-07
- NASA SE phase mapping: Phase E (Operations and Sustainment) -- ORR gate (Operational Readiness Review)
- Purpose: Prove the cloud works from a tenant user's perspective by exercising the complete workflow

**2. Pre-Conditions**
- OpenStack deployment complete (all post-deploy blocking gates passed)
- Admin credentials available (openrc file sourced)
- At least one guest image uploaded to Glance (e.g., CirrOS test image)
- External network configured for floating IP allocation
- Reference: Deployment verification (324-01) must have passed

**3. Verification Steps (8 stages)**

Stage 1 -- Authentication:
- Source admin openrc credentials
- Issue authentication token: `openstack token issue`
- Verify token contains: id, expires, project_id
- Create a test project: `openstack project create e2e-test-project`
- Create a test user: `openstack user create e2e-test-user --project e2e-test-project --password <test-password>`
- Assign member role: `openstack role add --project e2e-test-project --user e2e-test-user member`
- Switch to test user credentials and verify authentication
- Services exercised: Keystone

Stage 2 -- Project Setup:
- As test user, verify project access: `openstack project show e2e-test-project`
- List available services: `openstack catalog list`
- Verify service endpoints are accessible from tenant context
- Services exercised: Keystone

Stage 3 -- Network Configuration:
- Create tenant network: `openstack network create e2e-test-net`
- Create subnet with DHCP: `openstack subnet create e2e-test-subnet --network e2e-test-net --subnet-range 10.0.0.0/24 --dns-nameserver 8.8.8.8`
- Create router: `openstack router create e2e-test-router`
- Set router external gateway: `openstack router set e2e-test-router --external-gateway <external-network>`
- Add subnet to router: `openstack router add subnet e2e-test-router e2e-test-subnet`
- Verify network topology: `openstack network list`, `openstack router show e2e-test-router`
- Services exercised: Neutron

Stage 4 -- Security Group Configuration:
- Create security group: `openstack security group create e2e-test-sg`
- Add SSH rule: `openstack security group rule create --protocol tcp --dst-port 22 e2e-test-sg`
- Add ICMP rule: `openstack security group rule create --protocol icmp e2e-test-sg`
- Verify rules: `openstack security group rule list e2e-test-sg`
- Services exercised: Neutron

Stage 5 -- Instance Launch:
- List available images: `openstack image list` (verify CirrOS or test image exists)
- List available flavors: `openstack flavor list` (or create: `openstack flavor create e2e-tiny --ram 512 --disk 1 --vcpus 1`)
- Create keypair: `openstack keypair create e2e-test-key` (save private key)
- Launch instance: `openstack server create e2e-test-instance --image <cirros> --flavor <flavor> --network e2e-test-net --security-group e2e-test-sg --key-name e2e-test-key`
- Wait for ACTIVE: `openstack server show e2e-test-instance -f value -c status` (poll until ACTIVE, timeout 120s)
- Verify console log: `openstack console log show e2e-test-instance` (should show boot messages)
- Services exercised: Nova, Glance, Neutron

Stage 6 -- Storage Attachment:
- Create volume: `openstack volume create e2e-test-volume --size 1`
- Wait for available: `openstack volume show e2e-test-volume -f value -c status` (poll until available)
- Attach to instance: `openstack server add volume e2e-test-instance e2e-test-volume`
- Verify attachment: `openstack volume show e2e-test-volume` (status: in-use, attached to instance)
- Services exercised: Cinder, Nova

Stage 7 -- Floating IP Access:
- Allocate floating IP: `openstack floating ip create <external-network>`
- Associate with instance: `openstack server add floating ip e2e-test-instance <floating-ip>`
- Verify association: `openstack server show e2e-test-instance` (addresses include floating IP)
- Test connectivity: `ping -c 3 <floating-ip>` (should succeed if security group allows ICMP)
- Test SSH (if CirrOS): `ssh -o StrictHostKeyChecking=no -i e2e-test-key cirros@<floating-ip> hostname` (timeout 30s)
- Services exercised: Neutron (floating IP), Nova (networking)

Stage 8 -- Cleanup:
- Detach volume: `openstack server remove volume e2e-test-instance e2e-test-volume`
- Delete instance: `openstack server delete e2e-test-instance --wait`
- Delete volume: `openstack volume delete e2e-test-volume`
- Remove floating IP: `openstack floating ip delete <floating-ip>`
- Delete router interfaces: `openstack router remove subnet e2e-test-router e2e-test-subnet`
- Delete router: `openstack router delete e2e-test-router`
- Delete network: `openstack network delete e2e-test-net` (deletes subnet too)
- Delete security group: `openstack security group delete e2e-test-sg`
- Delete keypair: `openstack keypair delete e2e-test-key`
- Delete user and project: `openstack user delete e2e-test-user`, `openstack project delete e2e-test-project`
- Verify no orphaned resources remain

**4. Expected Results Table**
- Table mapping each stage to: services exercised, expected outcome, pass criteria

**5. Service Coverage Matrix**

  - Keystone: Stages 1, 2

**Part A: Create `scripts/e2e-user-scenario-verification.sh`**

Create an executable bash verification script that automates the 8-stage user scenario from the document. The script must:

1. Use `set -euo pipefail` for safety
2. Accept flags:
   - `--dry-run`: Print what would be done without executing
   - `--skip-cleanup`: Leave test resources in place for debugging
   - `--external-network <name>`: Specify the external network for floating IPs (default: "external")
   - `--image <name>`: Specify the guest image to use (default: "cirros")
   - `--results <path>`: Path to results file (default: configs/evaluation/e2e-user-scenario-results.yaml)
3. Define constants for all test resource names (all prefixed with `e2e-test-`):
   ```bash
   E2E_PROJECT="e2e-test-project"
   E2E_USER="e2e-test-user"
   E2E_NETWORK="e2e-test-net"
   E2E_SUBNET="e2e-test-subnet"
   E2E_ROUTER="e2e-test-router"
   E2E_SG="e2e-test-sg"
   E2E_INSTANCE="e2e-test-instance"
   E2E_VOLUME="e2e-test-volume"
   E2E_KEYPAIR="e2e-test-key"
   E2E_FLAVOR="e2e-tiny"
   ```
4. Implement each of the 8 verification stages as bash functions:

```bash
stage_1_authentication()    # Token issue, project/user create, role assign
stage_2_project_setup()     # Verify project access, list services
stage_3_network_config()    # Create network, subnet, router, connect
stage_4_security_group()    # Create SG, add SSH + ICMP rules
stage_5_instance_launch()   # List images/flavors, create keypair, launch, wait for ACTIVE
stage_6_storage_attach()    # Create volume, wait available, attach to instance
stage_7_floating_ip()       # Allocate floating IP, associate, test connectivity
stage_8_cleanup()           # Delete all e2e-test- resources in reverse order
```

5. Each stage function must:
   - Print a header with stage number and services being exercised
   - Execute the OpenStack CLI commands
   - Check return codes and expected output
   - Record results (pass/fail/skip with details)
   - On failure: print remediation guidance referencing the relevant skill's troubleshooting section
   - Support `--dry-run` mode (print commands without executing)

6. Add a `cleanup_on_exit()` trap function that runs cleanup on script termination (Ctrl+C, error) unless `--skip-cleanup` was specified. This prevents orphaned test resources.

7. Handle the case where OpenStack CLI is not available (not installed or openrc not sourced) -- gracefully exit with instructions.

8. Handle the case where the required image is not available -- skip stages 5-7 with status "skip" and note that a guest image must be uploaded first.

9. Polling functions for async operations:
   - `wait_for_status()`: Poll resource status with configurable timeout (default 120s for instances, 60s for volumes)
   - Check every 5 seconds, report progress

10. Output a summary at the end:
    - Total stages: 8
    - Passed / Failed / Skipped counts
    - Services exercised
    - Overall verdict: PASS or FAIL
    - Path to results file

11. Record the git commit hash at the start for traceability.

12. Make the script executable (chmod +x).
