  <action>
    Create PXE boot menu templates for both BIOS and UEFI boot paths. Both menus must:
    - Display GSD branding (ASCII art splash or title)
    - Offer "GSD Base Image (CentOS Stream 9)" entry -- boots to CentOS installer with base kickstart
    - Offer "GSD Minecraft Server" entry -- boots to CentOS installer with Minecraft kickstart
    - Offer "Boot from local disk" entry -- falls back to normal boot
    - Use template variables for kickstart server URL and other site-specific values
    - Auto-boot to local disk after a timeout (30 seconds) if no selection made

    **Create `infra/templates/pxe/gsd-splash.txt`:**
    ASCII art banner for GSD branding. Keep it compact (fits in 80x25 terminal):

    ```
     ╔═══════════════════════════════════════════════╗
     ║           GSD Network Boot Service            ║
     ║        Get Shit Done — Local Cloud            ║
     ╚═══════════════════════════════════════════════╝
    ```

    **Create `infra/templates/pxe/pxelinux.cfg-default.template`:**
    Standard syslinux/pxelinux menu format:

    ```
    # GSD PXE Boot Menu (BIOS)
    # Template: ${VAR} replaced at deployment time

    DEFAULT vesamenu.c32
    PROMPT 0
    TIMEOUT ${PXE_TIMEOUT_TENTHS}
    ONTIMEOUT localboot

    MENU TITLE GSD Network Boot Service
    MENU BACKGROUND gsd/splash.png

    LABEL base
      MENU LABEL ^1) GSD Base Image (CentOS Stream 9)
      KERNEL centos-stream-9/vmlinuz
      APPEND initrd=centos-stream-9/initrd.img inst.ks=${KICKSTART_SERVER_URL}/base.ks ip=dhcp

    LABEL minecraft
      MENU LABEL ^2) GSD Minecraft Server
      KERNEL centos-stream-9/vmlinuz
      APPEND initrd=centos-stream-9/initrd.img inst.ks=${KICKSTART_SERVER_URL}/minecraft.ks ip=dhcp

    LABEL localboot
      MENU LABEL ^3) Boot from Local Disk
      MENU DEFAULT
      LOCALBOOT 0
    ```

    Notes on pxelinux format:
    - `TIMEOUT` is in tenths of seconds (300 = 30 seconds)
    - `DEFAULT vesamenu.c32` enables graphical menu (falls back to text if unavailable)
    - `MENU DEFAULT` on localboot means timeout boots to local disk (safety)
    - `inst.ks=` is the CentOS/RHEL Anaconda kickstart parameter
    - `ip=dhcp` tells the installer to use DHCP for network during installation

    **Create `infra/templates/pxe/grub.cfg.template`:**
    GRUB2 menu format for UEFI clients:

    ```grub
    # GSD PXE Boot Menu (UEFI)
    # Template: ${VAR} replaced at deployment time

    set default=2
    set timeout=${PXE_TIMEOUT_SECONDS}

    menuentry "GSD Base Image (CentOS Stream 9)" {
      linuxefi centos-stream-9/vmlinuz inst.ks=${KICKSTART_SERVER_URL}/base.ks ip=dhcp
      initrdefi centos-stream-9/initrd.img
    }

    menuentry "GSD Minecraft Server" {
      linuxefi centos-stream-9/vmlinuz inst.ks=${KICKSTART_SERVER_URL}/minecraft.ks ip=dhcp
      initrdefi centos-stream-9/initrd.img
    }

    menuentry "Boot from Local Disk" {
      exit
    }
    ```

    Notes on GRUB format:
    - `set default=2` means "Boot from Local Disk" is default (0-indexed)
    - `linuxefi` and `initrdefi` are the UEFI-specific GRUB commands (not `linux`/`initrd`)
    - `exit` in the local boot entry returns control to the UEFI firmware
    - Timeout in seconds (not tenths like pxelinux)

    **Template variables used across both menus:**
    - `${KICKSTART_SERVER_URL}` -- HTTP URL where kickstart files are served (e.g., `http://192.168.122.10:8080/kickstart`)
    - `${PXE_TIMEOUT_TENTHS}` -- pxelinux timeout in tenths of seconds (BIOS menu only)
    - `${PXE_TIMEOUT_SECONDS}` -- GRUB timeout in seconds (UEFI menu only)
  </action>

<task type="auto">
  <name>Task 2: Create template renderer and master PXE deployment script</name>
  <files>
    infra/scripts/render-pxe-menu.sh

  <action>
    Create two scripts: a template renderer for PXE menu files, and a master deployment script that orchestrates the complete PXE setup.

    **Create `infra/scripts/render-pxe-menu.sh`** (executable, bash):

    A general-purpose template renderer that substitutes `${VAR}` placeholders with values from a local-values YAML file. This is a shared utility that will be reused by kickstart rendering (Phase 171) and other template-driven configs.

    The script should:
    1. Accept arguments:
       - `--template` (path to template file, required)
       - `--values` (path to local-values YAML file, required)
       - `--output` (path to write rendered output, required)
    2. Parse the values file using grep/awk (YAML key: "value" format — no external YAML parser):
       - Read each `key: "value"` or `key: value` line
       - Store in bash associative array
    3. For each `${VAR}` in the template:
       - Look up lowercase `var` in the values (template vars are UPPER, values are lower_snake_case)
       - Replace `${VAR}` with the value
       - If a variable is referenced but not in values file, print ERROR and exit 1
    4. Write rendered output to `--output` path
    5. Print summary: template path, values path, output path, variables substituted count

    Variable name mapping: template `${DHCP_RANGE_START}` maps to values `dhcp_range_start`. Convert: lowercase the template var name.

    **Create `infra/scripts/deploy-pxe.sh`** (executable, bash):

    Master deployment script that orchestrates the full PXE setup. Calls the scripts from Plan 01 and this plan in correct order.

    The script should:
    1. Accept arguments:
       - `--local-values` (path to local-values YAML, required)
       - `--dry-run` (show what would be done without executing)
       - `--skip-download` (skip boot media download, useful for offline setups)
    2. Read configuration from local-values:
       - TFTP root path
       - PXE interface
       - Kickstart server URL
       - All dnsmasq template variables
    3. Execute deployment steps in order:
       a. **Validate prerequisites**: check for dnsmasq, syslinux packages; check if running as root
       b. **Setup TFTP root**: call `setup-tftp-root.sh --tftp-root ${TFTP_ROOT}`
       c. **Download boot media**: call `download-centos-boot-media.sh --tftp-root ${TFTP_ROOT}` (unless `--skip-download`)
       d. **Render dnsmasq config**: call `render-pxe-menu.sh` on `pxe-boot.conf.template` -> `/etc/dnsmasq.d/pxe-boot.conf` (or `--output` from values)
       e. **Render BIOS menu**: call `render-pxe-menu.sh` on `pxelinux.cfg-default.template` -> `${TFTP_ROOT}/pxelinux.cfg/default`
       f. **Render UEFI menu**: call `render-pxe-menu.sh` on `grub.cfg.template` -> `${TFTP_ROOT}/grub.cfg`
       g. **Copy branding**: copy `gsd-splash.txt` to `${TFTP_ROOT}/gsd/`
       h. **Validate dnsmasq config**: `dnsmasq --test --conf-file=/etc/dnsmasq.d/pxe-boot.conf` (syntax check)
       i. **Restart dnsmasq**: `systemctl restart dnsmasq` (or just reload if only config changed)
       j. **Verify service**: `systemctl is-active dnsmasq` and `ss -tulnp | grep :69` (TFTP port)
    4. On `--dry-run`: print each step that would execute but don't run anything. Useful for review.
    5. Error handling:
       - If any step fails, print which step failed and what to check
       - If dnsmasq config validation fails, do NOT restart dnsmasq (preserve working DNS)
       - Backup existing dnsmasq config before overwriting: `cp /etc/dnsmasq.d/pxe-boot.conf /etc/dnsmasq.d/pxe-boot.conf.bak`
    6. Print deployment summary at end:
       - dnsmasq status, TFTP port status, DHCP range, boot media checksums
       - Test instructions: "To test: create a VM with PXE boot enabled on network ${PXE_INTERFACE}"

    **Safety features (critical):**
    - NEVER modify existing dnsmasq.conf — only write to conf.d/ include directory
    - Validate new config BEFORE restarting dnsmasq
    - Backup previous PXE config before overwriting
    - Check that TFTP port 69 is not already in use by another service
    - Verify dnsmasq is still serving DNS after restart: `dig @localhost gsd.local` (or equivalent quick check)

    **Script design principles (both scripts):**
    - `set -euo pipefail` header
    - Determine script directory via `SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` for relative path resolution to other scripts and templates
    - Clear `--help` output with usage examples
    - Colored output for steps (green=success, red=error, yellow=warning) using tput if available
    - All operations logged to stderr; only final summary to stdout
  </action>
