Create `src/brainstorm/pathways/router.ts` implementing the IPathwayRouter interface.

Interface:
```typescript
interface IPathwayRouter {
  recommendPathway(problem_statement: string): PathwayId;
  getPathway(id: PathwayId): PathwayConfig;
  getTechniqueQueue(pathway_id: PathwayId): TechniqueId[];
  adaptTechniqueQueue(
    current_queue: TechniqueId[],
    signal: AdaptationSignal,
    session: SessionState
  ): TechniqueId[];
}

type AdaptationSignal =
  | { type: 'low_energy'; completed_techniques: TechniqueId[] }
  | { type: 'user_request'; requested_technique: TechniqueId }
  | { type: 'saturation'; current_technique: TechniqueId }
  | { type: 'analytical_depth_needed'; current_technique: TechniqueId };
```

**PathwayRouter class implementation:**

Constructor:
- Load all 5 pathway configs from JSON files using `fs.readFileSync` (sync load at startup is fine — configs are static)
- Validate each config with `PathwayConfigSchema.parse()` on load
- Build internal `pathways: Map<PathwayId, PathwayConfig>` and `signal_index: Map<string, PathwayId>` from `recommended_for` words

`recommendPathway(problem_statement: string): PathwayId`:
- Signal word analysis — split `problem_statement` to lowercase tokens
- Score each pathway by counting how many `recommended_for` words appear in the problem tokens
- Return the highest-scoring pathway ID
- Tie-breaking: 'creative-exploration' wins ties (broadest applicability)
- If no signal words match (score = 0 for all), return 'free-form' (safest fallback)

Key signal word test cases from spec:
- Input containing "how might we reduce" → should match 'problem-solving' (`recommended_for` includes "how might we", "reduce", "problem")
- Input containing "imagine new possibilities" → should match 'creative-exploration' (no explicit "imagine" in recommended_for — check that creative-exploration wins when no specific match; OR add "imagine", "new possibilities", "explore" to its recommended_for)

**Update creative-exploration.json recommended_for** to include: `"imagine", "explore", "possibilities", "discover", "create", "new ideas", "starting fresh"`

`adaptTechniqueQueue(current_queue, signal, session): TechniqueId[]`:

Handle each adaptation signal type per spec (05-11-component-specs.md Pathway Router adaptation table):

- `low_energy`: Remove current high-effort techniques from remaining queue. Insert high-energy technique: prefer 'rapid-ideation' or 'rolestorming'. Return new queue.
- `user_request`: Insert `requested_technique` at position 0 of remaining queue (next up). If already in queue, move to front.
- `saturation`: Remove `current_technique` from remaining queue. Skip to next technique.
- `analytical_depth_needed`: Insert 'starbursting' or 'five-whys' after current technique position.

IMPORTANT: `adaptTechniqueQueue` ONLY modifies the REMAINING queue — not completed techniques. The session's `metadata.techniques_used` tracks completed techniques and should never be modified by the router.

Free-Form pathway handling:
- `getTechniqueQueue('free-form')` returns `[]` (empty — user-directed)
- `adaptTechniqueQueue` with free-form can add any requested technique

Import `PathwayConfigSchema` from `'../shared/schemas.js'`. Import `PathwayConfig`, `PathwayId`, `TechniqueId`, `SessionState` from `'../shared/types.js'`. Use `node:fs` for sync JSON loading. Import path using `import.meta.url` for ESM-compatible `__dirname` equivalent.

Do NOT import from den/, vtm/, knowledge/.
