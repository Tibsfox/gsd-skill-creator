Does NOT validate against full chipset schema (that is the staging pipeline's job). Pure function.
- `stageChipsetVariant(options: { basePath: string; name: string; content: string; source: string }): Promise<{ variantPath: string; metadataPath: string }>` — writes variant YAML to `.planning/staging/inbox/chipset-variants/{name}/chipset.yaml` with companion metadata. Creates directory.
- `listStagedVariants(basePath: string): Promise<StagedVariantInfo[]>` — reads `.planning/staging/inbox/chipset-variants/` and returns info for each staged variant directory. Returns empty array if directory does not exist.

Create `src/cloud-ops/observation/types.ts` with:
- `DeploymentAction`: `'bootstrap' | 'prechecks' | 'deploy' | 'reconfigure' | 'upgrade' | 'rollback' | 'verify'` — kolla-ansible action types
- `DeploymentEvent`: `{ action: DeploymentAction; service: string; timestamp: string; durationMs: number; success: boolean; errorMessage?: string }` — single deployment event
- `DeploymentPattern`: `{ sequence: DeploymentAction[]; services: string[]; occurrences: number; avgDurationMs: number; successRate: number; firstSeen: string; lastSeen: string }` — detected pattern
- `PromotionCandidate`: `{ pattern: DeploymentPattern; confidence: number; reason: string; suggestedSkillName?: string }` — candidate for skill promotion
- `DeploymentObserverConfig`: `{ minOccurrences: number; minSuccessRate: number; promotionThreshold: number }` — configurable thresholds
- `DEFAULT_DEPLOYMENT_OBSERVER_CONFIG` constant: `{ minOccurrences: 3, minSuccessRate: 0.9, promotionThreshold: 0.8 }`

Create `src/cloud-ops/observation/deployment-observer.ts`:
- `DeploymentObserver` class following the pattern from `src/observation/promotion-detector.ts`:
  - Constructor takes optional `DeploymentObserverConfig` (defaults to DEFAULT_DEPLOYMENT_OBSERVER_CONFIG)
  - `recordEvent(event: DeploymentEvent): void` — appends event to internal event log. After recording, checks if the last N events form a recognized pattern.
  - `detectPatterns(): DeploymentPattern[]` — scans the event log for recurring action sequences. Groups consecutive events by service, extracts action sequences, counts occurrences of identical sequences, calculates average duration and success rate. Returns patterns meeting minOccurrences threshold.
  - `identifyPromotionCandidates(): PromotionCandidate[]` — runs detectPatterns(), filters to patterns with successRate >= minSuccessRate, scores confidence as `(occurrences / 10) * successRate` capped at 1.0. Returns candidates with confidence >= promotionThreshold. Suggests skill name as `deploy-{services.join('-')}-sequence`.
  - `getEventLog(): readonly DeploymentEvent[]` — returns copy of event log
  - `clearEventLog(): void` — resets event log
  - `getStats(): { totalEvents: number; uniquePatterns: number; promotionCandidates: number }` — summary stats

This is a pure in-memory observer. No file I/O. Events are fed in by the deployment crew's observation loop. The observer detects patterns and identifies promotion candidates which the skill-creator pipeline can then process.

Create barrel `src/cloud-ops/observation/index.ts` exporting DeploymentObserver, all types, and the default config constant.

- Pattern detection: different sequences not conflated
- Promotion candidates: high success rate + enough occurrences = candidate
- Promotion candidates: low success rate filtered out
- Promotion candidates: below minOccurrences filtered out
- Stats return correct counts
- Config override works (custom thresholds)
- Clear event log resets state
