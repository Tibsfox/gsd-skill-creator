Create `src/cloud-ops/git/types.ts` with:
- `DeploymentChangeType`: `'config' | 'deploy' | 'reconfigure' | 'upgrade' | 'rollback' | 'verify' | 'certificate' | 'network'` — type of deployment change
- `DeploymentCommitInfo`: `{ changeType: DeploymentChangeType; services: OpenStackServiceName[]; sePhase: SEPhaseId; summary: string; rationale: string; configFiles?: string[]; riskLevel?: 'low' | 'medium' | 'high' }` — structured commit metadata
- `ParsedDeploymentCommit`: `DeploymentCommitInfo & { rawMessage: string; timestamp?: string }` — parsed from git log

Create `src/cloud-ops/git/commit-rationale.ts`:
- `formatDeploymentCommit(info: DeploymentCommitInfo): string` — formats a structured deployment commit message following this template:
  ```
  deploy({services}): {summary}

  SE Phase: {sePhase} ({phase name from NASA_SE_PHASES})
  Change Type: {changeType}
  Risk Level: {riskLevel || 'low'}
  Services: {services.join(', ')}

  Rationale: {rationale}

  {if configFiles: "Config Files:\n" + configFiles.map(f => "  - " + f).join("\n")}
  ```
  Import NASA_SE_PHASES from src/types/openstack.ts to look up phase name by ID. Use conventional commit format with `deploy` as the type and services as the scope. This ensures git log is searchable by service and SE phase.

- `parseDeploymentCommit(message: string): ParsedDeploymentCommit | null` — parses a deployment commit message back into structured data. Uses regex to extract the deploy({scope}): subject line, then parses the body for SE Phase, Change Type, Risk Level, Services, Rationale, and Config Files sections. Returns null if the message does not match the deployment commit format (not a deployment commit).

- `isDeploymentCommit(message: string): boolean` — quick check if a commit message is a deployment commit (starts with `deploy(`).

Tests:
- Format: produces correct conventional commit format with deploy type
- Format: includes SE phase name lookup (e.g., "Phase C: Final Design & Fabrication")
- Format: includes rationale and config files when provided
- Format: omits config files section when none provided
- Format: defaults risk level to 'low'
- Parse: round-trips with format (format then parse produces original info)
- Parse: returns null for non-deployment commits
- Parse: handles missing optional fields gracefully
- isDeploymentCommit: true for deploy() prefix, false for other types
  </action>

Create `src/cloud-ops/knowledge/types.ts` with:
- `KnowledgeTier`: `'summary' | 'active' | 'reference'` — the 3 tiers
- `TierConfig`: `{ tier: KnowledgeTier; maxTokens: number; timeoutMs: number; paths: string[] }` — tier configuration
- `TierContent`: `{ tier: KnowledgeTier; documents: Array<{ path: string; content: string; sizeBytes: number }>; totalTokens: number; loadTimeMs: number }` — loaded tier content
- `TierLoadResult`: `{ success: boolean; content?: TierContent; error?: string }` — result with error handling
- `TIER_DEFAULTS` constant:
  - summary: `{ maxTokens: 6000, timeoutMs: 2000 }` — always-loaded, <2s
  - active: `{ maxTokens: 20000, timeoutMs: 5000 }` — on-demand, <5s
  - reference: `{ maxTokens: 40000, timeoutMs: 10000 }` — deep dives, 10s budget

Create `src/cloud-ops/knowledge/tier-loader.ts`:
- `KnowledgeTierLoader` class:
  - Constructor takes `basePath: string` (project root) and optional tier config overrides
  - `loadSummaryTier(): Promise<TierLoadResult>` — loads summary-tier documents from `docs/cloud-ops/summary/` directory. Reads all `.md` files in the directory, concatenates content, estimates token count as `Math.ceil(totalChars / 4)` (standard approximation -- the project uses gpt-tokenizer for precise counting elsewhere but this is sufficient for the loader). Enforces the 2-second timeout using `AbortController` with `setTimeout`. If total tokens exceed maxTokens, truncates to fit. Returns TierLoadResult with loadTimeMs measured via `performance.now()`.
  - `loadActiveTier(documentIds?: string[]): Promise<TierLoadResult>` — loads active-tier documents from `docs/cloud-ops/active/`. If documentIds provided, loads only matching files. Otherwise loads all. Enforces 5-second timeout. Same token estimation and truncation.
  - `loadReferenceTier(documentIds: string[]): Promise<TierLoadResult>` — loads specific reference documents from `docs/cloud-ops/reference/`. Always requires document IDs (never loads all reference docs). Enforces 10-second timeout.
  - Private `loadDocuments(dir: string, filter?: string[], timeoutMs: number): Promise<TierContent>` — shared implementation. Uses `readdir` + `readFile` with Promise.race against timeout. Measures load time. Estimates tokens.

- Standalone convenience functions that create a KnowledgeTierLoader instance:
  - `loadSummaryTier(basePath: string): Promise<TierLoadResult>`
  - `loadActiveTier(basePath: string, documentIds?: string[]): Promise<TierLoadResult>`
  - `loadReferenceTier(basePath: string, documentIds: string[]): Promise<TierLoadResult>`

Tests:
- Summary tier: loads .md files from summary directory, returns content with token count
- Summary tier: enforces 2-second timeout (use mock slow reads)
- Summary tier: truncates when exceeding 6000 token budget
- Active tier: loads all documents when no IDs specified
- Active tier: loads only specified documents when IDs provided
- Active tier: enforces 5-second timeout
- Reference tier: requires document IDs (never loads all)
- Reference tier: returns error for non-existent documents
- Token estimation: approximately chars/4
- Empty directory: returns success with empty documents array
- Performance: loadTimeMs is populated and reasonable

Use `vi.useFakeTimers()` or mock `node:fs/promises` for timeout tests. Create temp directories with sample .md files for integration-style tests using `vi.mock` or `mkdtemp`.
