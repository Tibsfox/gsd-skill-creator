Create `src/vtm/cache-optimizer.ts` implementing all three functions to pass the test suite.

**Module structure:**

1. Import types from './types.js' (WaveExecutionPlan, ComponentSpec, WaveTask)
2. Import { encode } from 'gpt-tokenizer' for accurate token counting
3. Export public types: SharedLoadEntry, SchemaReuseEntry, KnowledgeTierEntry
4. Export three pure functions

**detectSharedLoads(plan: WaveExecutionPlan): SharedLoadEntry[]**
- For each wave, collect all tasks
- Group tasks by their produces artifact (exact match)
- Also detect content overlap: extract significant words (>3 chars, not stopwords) from task descriptions, compute Jaccard similarity, flag pairs with >50% overlap
- For each group of 2+ tasks sharing the same artifact or overlapping content, create a SharedLoadEntry
- Use `encode(produces).length` from gpt-tokenizer for estimatedTokens
- cacheable = group.length >= 2

**analyzeSchemaReuse(plan: WaveExecutionPlan, specs: ComponentSpec[]): SchemaReuseEntry[]**
- Build a map of task ID -> wave number from plan.waves
- Identify producers: tasks whose produces string contains 'types', 'schema', 'interface', or 'config' (case-insensitive)
- Identify consumers: for each producer, find tasks in later waves whose dependsOn (from the corresponding ComponentSpec) includes the producer's spec name
- Check if plan.cacheOptimization exists at WaveExecutionPlan top-level (NOT per-task) -- when present, use plan.cacheOptimization.schemaReuse records as primary source of producer/consumer relationships
- For each producer-consumer pair, create a SchemaReuseEntry
- waveBoundariesCrossed = max consumer wave - producer wave

**calculateKnowledgeTiers(plan: WaveExecutionPlan, specs: ComponentSpec[]): KnowledgeTierEntry[]**
- For each task in the plan, determine:
  - currentTier: defaults to 'reference' (all tasks load full content by default)
  - optimalTier: based on task's estimatedTokens threshold: <=2000 -> summary, <=10000 -> active, >10000 -> reference
  - Token sizes: summary = encode of ~2K content representation, active = encode of ~10K, reference = task's estimatedTokens
  - savings = tokensCurrent - tokensOptimal (clamped to 0 minimum)

**Key decisions per CONTEXT.md:**
- Use gpt-tokenizer `encode` for accurate token counting (NOT char/4 heuristic)
- Schema reuse covers ALL wave boundaries, not just Wave 0 -> Wave 1
- Content overlap algorithm for skill matching is Claude's discretion -- use Jaccard similarity on significant words

**Style:** Follow established VTM module patterns -- JSDoc on every export, internal helpers as private functions, constants at module top.
