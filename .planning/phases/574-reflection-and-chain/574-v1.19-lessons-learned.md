## Key Lessons from v1.19

### L1: Focused Releases Score Well
v1.19 (16 commits, +3906 lines) scored 4.35 — matching v1.17's 4.34 (85 commits) and exceeding v1.18's 4.315 (85 commits). The quality-per-commit is higher in focused releases because each commit has clear purpose and clean TDD pairing.

### L2: Refactoring IS Quality Improvement
The status.ts → status-display.ts extraction eliminated P4 copy-paste and improved P7 documentation simultaneously. This is a structural quality improvement that compounds: future status changes only need to touch the rendering module, not the CLI wiring.

### L3: Test Ratio as Leading Indicator
v1.19's 3.09x test ratio is the strongest in chain history. When tests outnumber implementation 3:1, it indicates thorough edge-case coverage (zero-skill, over-budget, no-projection, migration). This correlates with zero fix commits (P11 CLEAN).

### L4: Budget Vocabulary as Generative Seed
The tier grammar (critical/standard/optional × loaded/deferred) and severity spectrum (ok/info/warning/error) form reusable patterns. These could generalize to any resource allocation domain — not just budget management.

### L5: Dual-View Pattern (Installed vs Loadable)
The distinction between what EXISTS (installed) and what RUNS (loadable) is a fundamental architectural pattern. It mirrors game engine LOD: everything is loaded into memory (installed), but only what's visible gets rendered (loadable). This pattern should be tracked as a potential Muse cartridge seed.
