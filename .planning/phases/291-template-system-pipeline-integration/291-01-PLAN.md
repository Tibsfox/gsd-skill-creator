  <action>
Add a new async function `renderMissionDocuments()` to `src/vtm/mission-assembly.ts` that takes a MissionPackage and optional basePath, and returns rendered markdown documents using the template system. This function must:

1. Import and call `createTemplateRegistry(basePath)` to create a registry instance (wires TMPL-04).
2. For each document type in the mission package (readme, milestone-spec, component-spec, wave-plan, test-plan), call `registry.get(name)` which internally calls `loadTemplate()` (wires TMPL-01). If a template is not found on disk (likely in test environments), fall back to generating the document via the existing string builder for README and returning a JSON.stringify representation for typed objects.
3. For each loaded template, build a variables map from the corresponding typed object and call `renderTemplate(template.content, variables)` (wires TMPL-02).
4. For each rendered document, call `validateRenderedTemplate(name, renderedContent, typedObject)` to validate against Zod schemas (wires TMPL-03).
5. Return a `RenderedDocuments` type:
   ```typescript
   export interface RenderedDocument {
     name: string;
     templateName: string;
     content: string;
     validation: ValidationResult;
   }
   export interface RenderedDocuments {
     documents: RenderedDocument[];
     registry: TemplateRegistry;
   }
   ```

For the variables map construction per template type:
- **readme**: { milestoneName, date, status, visionDocument, contents (file manifest as string), howToUse (usage instructions string), executionSummary (formatted string), researchReference, dependencies, notes }
- **milestone-spec**: Spread milestoneSpec fields as variables, convert nested objects to string representations
- **component-spec**: Per component spec, spread fields as variables
- **wave-plan**: Spread waveExecutionPlan fields as variables
- **test-plan**: Spread testPlan fields as variables

For variable maps, convert array/object values to their string representation (JSON.stringify for objects, join for simple arrays) since renderTemplate works with string values. The key integration point is that the template system is CALLED and USED, not that it replaces all string generation.

Then update `src/vtm/pipeline.ts`:

1. Make `runPipeline()` async (returns `Promise<PipelineResult>`).
2. After the mission stage succeeds, call `await renderMissionDocuments(missionPackage)`.
3. Add a `renderedDocuments` field to `MissionStageResult`:
   ```typescript
   renderedDocuments?: RenderedDocument[];
   ```
4. Import `renderMissionDocuments` and `RenderedDocument` from mission-assembly.
5. Wrap the renderMissionDocuments call in a try/catch -- template rendering failures should NOT fail the pipeline (they are additive, not critical). If rendering fails, set renderedDocuments to an empty array and continue.
6. Import `createTemplateRegistry` from template-system.ts and store the registry reference in the mission stage result or use it for the render call.

Then update `src/vtm/vtm-pipeline.ts`:

1. Make `VTMPipeline.run()` async (returns `Promise<PipelineResult>`).
2. Add `await` to the `runPipeline()` call.
3. Update `createVTMPipeline()` -- no changes needed (it returns the class, not a result).

IMPORTANT: Do NOT change the generateReadme() function signature or remove it. It continues to work as a fallback. The template rendering is an additional layer that produces rendered documents alongside the existing typed objects.
