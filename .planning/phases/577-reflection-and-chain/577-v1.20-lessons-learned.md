## Key Lessons from v1.20

### L1: Orchestrator Emergence
Generator.ts evolved from a simple page renderer to a 25-module orchestrator. This wasn't designed top-down — it emerged from iterative collector wiring. The pattern is P8 (Example-to-Infrastructure Pipeline) applied to the generator: what started as a static HTML generator became the central data pipeline. This is the project's strongest P6 instance.

### L2: Graceful Degradation as Architecture
The "never blocks dashboard generation" pattern (7 instances) is no longer ad-hoc error handling — it's a stated architectural principle. Every data source is optional. The dashboard generates with whatever data is available. This is resilient system design: partial data produces partial dashboard, never failure.

### L3: Ad-Hoc Collectors — Honest Trade-Off
The 5 collectors share types.ts but lack a unified DataCollector<T> interface. This is a conscious depth trade-off: each collector was implemented as a standalone function matching its specific data domain. A unified interface would score higher on Depth but might over-abstract at this stage. The honest assessment: it works, it's testable, but refactoring to a shared contract would reduce duplication of the try/catch wrapper pattern.

### L4: Design Tokens as Threshold Governance
CSS custom properties (var(--domain-*, #fallback)) don't reduce the threshold count (FF-02) but they centralize the color dimension. This is partial progress — colors are governed by tokens, but numeric thresholds (budgets, limits, timeouts) remain scattered. The token pattern could generalize to numeric thresholds.

### L5: Scale Independence Confirmed
v1.20 scores 4.35 on 24 commits, matching v1.19's 4.35 on 16 commits and competitive with v1.17's 4.34 on 85 commits. Across the last 5 positions (16-85 commit range), scores stay in 4.25-4.38. Quality is a property of the development process, not the release size.
