  <action>
Create `infra/templates/kickstart/minecraft.ks.template` -- a CentOS Stream 9 kickstart for the Minecraft server VM. This kickstart is a COMPLETE kickstart file (not a %include -- Anaconda's %include over HTTP is fragile). It duplicates the base security directives and adds Minecraft-specific content.

The template MUST include:

**Base sections (same as base.ks.template):**
All base security sections are repeated verbatim: `url`, `text`, locale, network (with `${MINECRAFT_HOSTNAME}` instead of `${HOSTNAME}`), `rootpw --lock`, admin user with SSH key, `selinux --enforcing`, `firewall --enabled --ssh --port=${MINECRAFT_SERVER_PORT}:tcp --port=${MINECRAFT_RCON_PORT}:tcp`, bootloader, disk partitioning.

**Extended packages section** (`%packages`):
- Everything from base: `@^minimal-environment`, `openssh-server`, `sudo`, `vim-minimal`, `curl`, `tar`, `bash-completion`, `policycoreutils-python-utils`
- Java 21: `java-21-openjdk-headless` (headless -- server does not need GUI)
- Additional tools: `screen` (for server console), `jq` (for JSON config parsing), `firewalld` (ensure it is installed)
- `%end`

**Extended %post section** (`%post --log=/root/ks-post.log`):
All base %post content (sudo, SSH hardening, first-boot service install, base image marker) PLUS:

1. **Create Minecraft user**: `useradd --system --create-home --home-dir /opt/minecraft --shell /sbin/nologin minecraft` -- system user, no login shell, dedicated home directory
2. **Create server directories**: `mkdir -p /opt/minecraft/{server,backups,logs}` and `chown -R minecraft:minecraft /opt/minecraft`
3. **Write Minecraft first-boot hook**: Create `/opt/gsd/first-boot.d/50-minecraft-setup.sh` that will run during first boot to:
   - Verify Java 21 is installed: `java -version 2>&1 | grep -q "21\." || exit 1`
   - Log Java version to first-boot log
   - Create a placeholder `server.properties` marker (actual server.properties comes from Phase 175)
   - Set correct SELinux contexts on /opt/minecraft: `restorecon -Rv /opt/minecraft` (if semanage is available)
   - The hook script must be `chmod +x` and owned by root
4. **Write the Minecraft marker**: `echo "GSD Minecraft Server built $(date -u +%Y-%m-%dT%H:%M:%SZ)" > /etc/gsd-minecraft-server`
5. `%end`

**Reboot**: `reboot --eject`

Template variables specific to Minecraft (add to kickstart.local-values.example):
- `minecraft_hostname`: VM hostname (e.g., "mc-server-01")
- `minecraft_server_port`: Game port (default: 25565)
- `minecraft_rcon_port`: RCON port (default: 25575)

Update `infra/templates/kickstart/kickstart.local-values.example` to add these variables in a new "# Minecraft Server" section at the bottom. Keep all existing base variables intact.

Do NOT configure the actual Minecraft server software (that is Phase 173). This kickstart only prepares the OS environment: Java runtime, dedicated user, firewall ports, directory structure.

  <name>Task 2: Kickstart deployment script (render + HTTP serve)</name>
  <files>
    infra/scripts/deploy-kickstart.sh

  <action>
Create `infra/scripts/deploy-kickstart.sh` -- a deployment orchestrator that renders kickstart templates and prepares them for HTTP serving to PXE clients. Follow the established orchestrator pattern from `infra/scripts/deploy-pxe.sh` (step-based execution, dry-run mode, pre-validation).

The script MUST:

1. **Header**: `#!/usr/bin/env bash` with `set -euo pipefail`, same color/logging helpers as deploy-pxe.sh (`info`, `ok`, `warn`, `error`, `die`, `step`, `dry_run_cmd`)
2. **Arguments**: `--local-values PATH` (required), `--output-dir PATH` (required -- where rendered kickstart files go, typically `/var/www/html/kickstart` or a Python HTTP server directory), `--dry-run` (optional), `--help`
3. **Step 1 -- Validate prerequisites**: Check that `render-pxe-menu.sh` exists, local-values file exists, template files exist (base.ks.template and minecraft.ks.template)
4. **Step 2 -- Create output directory**: `mkdir -p ${OUTPUT_DIR}` (create kickstart serve directory)
5. **Step 3 -- Render base kickstart**: Call `render-pxe-menu.sh --template .../base.ks.template --values $LOCAL_VALUES --output $OUTPUT_DIR/base.ks`
6. **Step 4 -- Render Minecraft kickstart**: Call `render-pxe-menu.sh --template .../minecraft.ks.template --values $LOCAL_VALUES --output $OUTPUT_DIR/minecraft.ks`
7. **Step 5 -- Validate rendered kickstarts**: For each rendered file, verify:
   - File is non-empty
   - No unresolved `${VAR}` placeholders remain (grep for `\$\{[A-Z].*\}` should return nothing)
   - Contains `%end` (basic kickstart structure check)
8. **Step 6 -- Set permissions**: `chmod 644 $OUTPUT_DIR/*.ks` (world-readable for HTTP serving)
9. **Step 7 -- Summary**: Print rendered file paths, sizes, and the URL they would be served at (from `kickstart_server_url` in local-values)

The script discovers template and script paths relative to its own location (same `SCRIPT_DIR`/`PROJECT_ROOT` pattern as deploy-pxe.sh). It does NOT start an HTTP server -- that is the operator's responsibility (could be Apache, nginx, Python http.server, or any HTTP server). The script just renders and places files.

Include a usage section that shows how to test locally:
```
# Render kickstarts
./deploy-kickstart.sh --local-values infra/local/kickstart.local-values --output-dir /tmp/kickstart

# Quick HTTP server for testing (serves on port 8080)
# cd /tmp/kickstart && python3 -m http.server 8080
```

Safety features (matching deploy-pxe.sh pattern):
- Backup existing rendered files before overwriting (timestamped .bak)
- Validate rendered output before declaring success
- Never modify templates (only writes to output directory)
