Create `infra/templates/kickstart/base.ks.template` -- a CentOS Stream 9 kickstart file using ${UPPER_SNAKE_CASE} template variables (same pattern as Phase 170 PXE templates, rendered by the existing `infra/scripts/render-pxe-menu.sh`).

The kickstart template MUST include these sections in order:

1. **Installation source**: `url --url=${CENTOS_MIRROR_URL}` for network install from CentOS Stream 9 mirror
2. **Text mode**: `text` (no GUI installer)
3. **Locale/keyboard/timezone**: `lang en_US.UTF-8`, `keyboard us`, `timezone ${TIMEZONE} --utc`
4. **Network**: `network --bootproto=dhcp --device=${NET_DEVICE} --activate --hostname=${HOSTNAME}`
5. **Root password**: `rootpw --lock` (root account locked, no password login)
6. **Admin user**: `user --name=${ADMIN_USER} --groups=wheel --shell=/bin/bash`
7. **SSH key**: `sshkey --username=${ADMIN_USER} "${SSH_PUBLIC_KEY}"` (key-only access)
8. **SELinux**: `selinux --enforcing` (NON-NEGOTIABLE security requirement)
9. **Firewall**: `firewall --enabled --ssh` (only SSH through firewall by default)
10. **Bootloader**: `bootloader --location=mbr --append="crashkernel=auto"`
11. **Disk partitioning**: `autopart --type=lvm` with `clearpart --all --initlabel` and `zerombr`
12. **Packages section** (`%packages`): `@^minimal-environment`, `openssh-server`, `sudo`, `vim-minimal`, `curl`, `tar`, `bash-completion`, `policycoreutils-python-utils` (for SELinux management), then `%end`
13. **Post-install section** (`%post --log=/root/ks-post.log`):
    - Configure sudo for wheel group: `echo '%wheel ALL=(ALL) NOPASSWD: ALL' > /etc/sudoers.d/wheel-nopasswd && chmod 440 /etc/sudoers.d/wheel-nopasswd`
    - Harden sshd_config: `sed -i 's/^#*PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config && sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config && sed -i 's/^#*PubkeyAuthentication.*/PubkeyAuthentication yes/' /etc/ssh/sshd_config`
    - Install the first-boot service (see Task 2 -- the %post section writes the systemd unit and the script from inline heredocs, with ${FIRST_BOOT_SCRIPT_PATH} defaulting to /opt/gsd/first-boot.sh)
    - Enable the first-boot service: `systemctl enable gsd-first-boot.service`
    - Write a marker at `/etc/gsd-base-image` with build timestamp: `echo "GSD Base Image built $(date -u +%Y-%m-%dT%H:%M:%SZ)" > /etc/gsd-base-image`
    - Then `%end`
14. **Reboot**: `reboot --eject`

Template variable naming follows Phase 170 convention: UPPER_SNAKE_CASE in templates, lower_snake_case in values files.

Create `infra/templates/kickstart/kickstart.local-values.example` documenting ALL template variables used by the base kickstart. Follow the format established in `infra/templates/dnsmasq/pxe-boot.local-values.example`:
- Comment header explaining this is an example file
- Grouped by function (installation, network, security, paths)
- Each variable with a descriptive comment
- Sensible defaults where appropriate (timezone: "UTC", net_device: "ens3", admin_user: "gsd")
- SSH_PUBLIC_KEY must have a placeholder value with clear instructions to replace

Variables to document:
- `centos_mirror_url`: CentOS Stream 9 network install URL
- `timezone`: System timezone
- `net_device`: Primary network interface name
- `hostname`: VM hostname
- `admin_user`: Non-root admin username
- `ssh_public_key`: SSH public key for admin user (the ONLY auth method)
- `first_boot_script_path`: Path for first-boot script (default: /opt/gsd/first-boot.sh)

Do NOT include any actual secrets, private keys, or real hostnames. This is an example file.

<task type="auto">
  <name>Task 2: First-boot service framework for deferred configuration</name>
  <files>
    infra/templates/kickstart/first-boot.sh.template
  </files>
  <action>
Create `infra/templates/kickstart/first-boot.sh.template` -- a bash script template that runs on first boot via a systemd oneshot service. This script handles configuration that cannot happen during kickstart (needs network, runtime state, or role-specific setup).

The script template MUST:

1. **Script header**: `#!/usr/bin/env bash` with `set -euo pipefail` and logging to `/var/log/gsd-first-boot.log` via `exec > >(tee -a /var/log/gsd-first-boot.log) 2>&1`
2. **Guard clause**: Check for marker file `/etc/gsd-first-boot-complete`; if it exists, log "First boot already completed" and exit 0. This makes the script idempotent.
3. **Timestamp**: Log the start time with `echo "=== GSD First Boot: $(date -u +%Y-%m-%dT%H:%M:%SZ) ==="`
4. **Wait for network**: Loop up to 30 seconds checking network connectivity (`curl -sf --max-time 5 http://${CENTOS_MIRROR_URL:0:30}... || ping -c1 -W5 8.8.8.8`). Use a simple `until ping -c1 -W5 8.8.8.8 &>/dev/null; do sleep 2; done` with a 30-second timeout.
5. **System updates section** (placeholder): Comment block `# --- System updates (optional) ---` with `# dnf -y update` commented out (operator can enable if desired)
6. **Role-specific hooks directory**: Create `/opt/gsd/first-boot.d/` if it does not exist. Then execute all scripts in that directory in lexicographic order: `for script in /opt/gsd/first-boot.d/*.sh; do ... done`. This is how the Minecraft kickstart (Plan 02) and other roles inject their deferred config.
7. **Completion marker**: Write `/etc/gsd-first-boot-complete` with timestamp. This prevents re-execution.
8. **Disable self**: `systemctl disable gsd-first-boot.service` so it does not run again.
9. **Log completion**: `echo "=== GSD First Boot Complete: $(date -u) ==="`

Also update the base kickstart template's %post section to install this first-boot service. The %post section should write two files:

A) The systemd unit file at `/etc/systemd/system/gsd-first-boot.service`:
```ini
[Unit]
Description=GSD First Boot Configuration
After=network-online.target
Wants=network-online.target
ConditionPathExists=!/etc/gsd-first-boot-complete

[Service]
Type=oneshot
ExecStart=${FIRST_BOOT_SCRIPT_PATH}
RemainAfterExit=false
StandardOutput=journal+console

[Install]
WantedBy=multi-user.target
```

B) The rendered first-boot script at `${FIRST_BOOT_SCRIPT_PATH}` (from the template, with variables substituted). Since the kickstart %post runs during installation, the first-boot script content should be written inline in the %post heredoc with any needed variables already resolved by the kickstart template renderer.

The first-boot.sh.template exists as a standalone reference/documentation file. The actual installation into the VM happens through the base.ks.template %post section which contains the script content inline.

Do NOT use template variables that require runtime resolution inside the first-boot script itself -- all ${VAR} substitution happens at template render time before the kickstart is served to the PXE client.
