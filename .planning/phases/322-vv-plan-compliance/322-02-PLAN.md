**docs/vv/drift-detection-procedures.md** -- Documentation Drift Detection Procedures:

1. **Header section:**
   - Title: "Documentation Drift Detection Test Procedures"
   - Reference: "Requirement VERIF-04, SC-018, CF-SK-022, CF-CH-012"
   - Purpose: Define procedures for testing that doc-verifier detects intentionally introduced drift

2. **Drift categories** (reference doc-verifier SKILL.md):
   - Configuration drift: documented value differs from running value
   - Endpoint drift: documented URL/port differs from actual
   - Procedural drift: documented procedure no longer produces expected result
   - Reference drift: external documentation link no longer valid

3. **Test scenarios** -- for each drift category, define:
   - **Scenario name** (e.g., "DRIFT-CONFIG-001: Keystone token lifetime mismatch")
   - **Setup:** What the documentation says (e.g., "Token lifetime: 3600s")
   - **Intentional drift introduction:** What to change in the doc or system to create mismatch (e.g., "Change documented value to 7200s while system retains 3600s")
   - **Expected detection:** What the doc-verifier should report (e.g., "Configuration drift: keystone.conf [fernet_tokens] token_expiration documented as 7200, actual is 3600")
   - **Pass criteria:** Doc-verifier identifies the specific mismatch with correct values

4. **Minimum 8 test scenarios** covering all 4 drift categories (2 per category):
   - DRIFT-CONFIG-001: Keystone token lifetime mismatch
   - DRIFT-CONFIG-002: Nova compute driver mismatch
   - DRIFT-ENDPOINT-001: Service catalog endpoint URL change
   - DRIFT-ENDPOINT-002: Horizon dashboard port mismatch
   - DRIFT-PROCEDURE-001: Fernet key rotation procedure produces different result
   - DRIFT-PROCEDURE-002: Nova instance creation with wrong flavor name
   - DRIFT-REFERENCE-001: OpenStack docs URL version mismatch
   - DRIFT-REFERENCE-002: SP-6105 section number cross-reference error

5. **Detection verification procedure:**
   - Step 1: Baseline -- run doc-verifier against known-good documentation (expect 0 drift reports)
   - Step 2: Introduce drift -- modify documentation per test scenario
   - Step 3: Detect -- run doc-verifier again (expect drift report identifying the change)
   - Step 4: Validate -- confirm drift report contains correct artifact path, expected value, actual value, and severity
   - Step 5: Restore -- revert documentation to baseline

6. **Integration with doc-sync loop:**
   - How drift detection feeds into the COMM-07 doc-sync communication loop
   - Automatic vs manual drift resolution paths

This document provides detailed, step-by-step test procedures for all 22 safety-critical tests (SC-001 through SC-022) from the test plan. These are the mandatory-pass tests that block deployment if any fails.

**Document structure:**

1. **Header section:**
   - Title: "Safety-Critical Test Procedures -- GSD OpenStack Cloud Platform"
   - Reference: "VERIF-05, SP-6105 SS 5.3, NPR 7123.1 Process 7"
   - Purpose: Provide repeatable test procedures for all 22 safety-critical tests
   - Failure action: BLOCK -- any safety-critical test failure halts the deployment pipeline
   - Total tests: 22

2. **Test procedure format** -- each test follows this structure:
   ```
   ### SC-XXX: [Test Name]
   **Requirement:** [Which requirement this verifies]
   **Component:** [Which component is tested]
   **Verification Method:** [T/A/I/D]
   **Preconditions:** [System state required]
   **Procedure:**
   1. [Step with exact command or action]
      Expected: [What should happen]
   2. [Next step]
      Expected: [Result]
   **Pass Criteria:** [Exact measurable condition]
   **Fail Action:** BLOCK -- [what to do if test fails]
   **Rollback:** [How to recover if test disrupts system]
   ```

3. **All 22 test procedures** -- reference the test plan for descriptions but provide DETAILED step-by-step procedures. Group by domain:

   **Credential & Data Security (SC-001, SC-002, SC-020):**
   - SC-001: Credentials excluded from git -- `git log --all --diff-filter=A -- '*.conf' '*.env' '*.key' '*.pem'`, then grep output for password/token/key patterns
   - SC-002: Local-only directory isolation -- verify `.gitignore` contains `local/`, run `git status --porcelain` and confirm no local/ files appear
   - SC-020: No credential logging -- grep all service log locations for password/token/private key patterns at all verbosity levels

   **Network & Access Security (SC-003, SC-004, SC-009, SC-015):**
   - SC-003: No external binding without HITL -- inspect Kolla-Ansible globals.yml for kolla_internal_vip_address and kolla_external_vip_address, verify internal-only default
   - SC-004: CAPCOM approval for external exposure -- trace crew config for human_interface.sole_agent confirming CAPCOM gate
   - SC-009: Security groups default deny -- verify new security group creation procedure produces group with no ingress rules
   - SC-015: Network isolation between tenants -- verify network architecture enforces project-level isolation

   **Authentication & Authorization (SC-006, SC-007, SC-008, SC-017):**
   - SC-006: Keystone token expiration -- check keystone.conf fernet_tokens section for token_expiration <= 3600
   - SC-007: TLS certificates validated -- verify TLS configuration in globals.yml, certificate paths, and expiry > 30 days
   - SC-008: RBAC policies enforce least privilege -- verify default policy prevents member from performing admin operations
   - SC-017: Audit logging enabled -- verify keystone logging configuration and log rotation setup

   **Operational Safety (SC-005, SC-010, SC-011, SC-021):**
   - SC-005: Destructive ops have rollback -- inspect all runbooks for `destructive: true` tag, verify each has non-empty rollback section
   - SC-010: Backup procedures produce restorable data -- define backup-modify-restore test sequence
   - SC-011: Hardware requirements checked before deploy -- reference pre-deploy-gates.yaml, verify SCOUT blocks on insufficient resources
   - SC-021: Rollback tested for deployment -- define deployment rollback test procedure

   **Agent & System Safety (SC-012, SC-013, SC-014, SC-019):**
