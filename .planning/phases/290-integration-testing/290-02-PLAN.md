  <action>
Create `src/vtm/__tests__/vtm-eval-harness.test.ts` implementing the eval harness for the 5 scenarios defined in `.claude/commands/vision-to-mission/evals.json`.

The eval harness validates that `runPipeline` produces structurally correct output for each scenario type. The evals.json expectations are qualitative (designed for LLM-as-judge evaluation), so the eval harness should test the structural/programmatic properties that can be verified in code.

**Scenario 1: eval-01-vision-from-idea**
Test that given a raw idea (educational drone pack), `parseVisionDocument` on a well-formed vision markdown produces a VisionDocument with all required sections (vision, problemStatement, coreConcept, architecture, modules, chipsetConfig, successCriteria, throughLine). This tests the vision stage in isolation.

Fixture: Create a drone-themed vision document markdown string (similar to the `createVisionMarkdown()` fixture in pipeline.test.ts but with drone content).

Assertions:
- parseVisionDocument returns success=true
- Result has non-empty vision, problemStatement, coreConcept, architecture, modules, successCriteria, throughLine
- chipsetConfig exists with skills and agents
- Modules array has length > 0

**Scenario 2: eval-02-mission-from-vision**
Test that given a pre-parsed VisionDocument (music production pack), `runPipeline` with speed='mission-only' produces a complete mission package.

Fixture: Create a music-themed VisionDocument object.

Assertions:
- result.success === true
- result.stages.mission is defined
- MissionPackage has milestoneSpec, componentSpecs (length > 0), waveExecutionPlan, testPlan
- fileManifest includes milestone-spec, component-spec, wave-plan, test-plan, readme types
- executionSummary has model split with opus/sonnet/haiku
- Component specs have non-empty modelAssignment
- Test plan has tests array with length > 0

**Scenario 3: eval-03-research-compilation**
Test that given an educational VisionDocument (nutrition/cooking), `compileResearch` produces structured output and `checkSourceQuality` runs without errors.

Fixture: Create a nutrition-themed VisionDocument object.

Assertions:
- compileResearch returns a ResearchReference with topics array length > 0
- Each topic has name, content, sources
- chunkKnowledge returns tiers with summary, active, reference
- extractSafety returns a SafetySection with boundaries
- checkSourceQuality returns an array (may be empty — entertainment media check)

**Scenario 4: eval-04-full-pipeline**
Test that given a full vision document markdown (home repair theme), `runPipeline` with speed='full' executes all three stages.

Fixture: Create a home-repair-themed vision markdown.

Assertions:
- result.success === true
- result.speed === 'full'
- result.stages.vision is defined with visionDoc, archetype, diagnostics
- result.stages.research is defined with research, knowledgeTiers, safety
- result.stages.mission is defined with missionPackage
- result.fileManifest.length > 0
- result.executionSummary.totalTasks > 0
- result.durationMs >= 0

**Scenario 5: eval-05-infrastructure-vision**
Test that given an infrastructure/tooling VisionDocument (plugin system), the archetype is 'infrastructure-component' and the pipeline auto-selects 'skip-research' speed.

Fixture: Create a plugin-system-themed VisionDocument object with infrastructure keywords.

Assertions:
- classifyArchetype returns 'infrastructure-component'
- selectPipelineSpeed with empty config returns 'skip-research'
- runPipeline without speed override skips research stage (stages.research is undefined)
- result.stages.mission is defined

Use `describe` blocks grouped by eval ID (eval-01 through eval-05) and import from the VTM module using relative paths (`../pipeline.js`, `../vision-parser.js`, etc.).
