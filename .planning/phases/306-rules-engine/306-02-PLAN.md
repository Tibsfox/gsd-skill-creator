<feature>
  <name>RulesEngine -- Two-Stage Evaluative Detection, Black Hat Constraint, Violation Logging</name>
  <files>src/brainstorm/core/rules-engine.ts, src/brainstorm/core/rules-engine.test.ts</files>
  <behavior>
    Extending the RulesEngine from Plan 01 with three capabilities:

    1. checkMessage(message: BrainstormMessage, phase: SessionPhase): { allowed: boolean; violations: RuleViolation[] }

       During NON-diverge phases: return { allowed: true, violations: [] } -- no content filtering outside Diverge.

       During diverge phase, apply TWO-STAGE detection:
       Stage 1 -- evaluative pattern match (hard_block_patterns from config):
         Blocked patterns (literal or near-literal):
         - "that won't work", "that will not work"
         - "bad idea", "terrible idea", "awful idea"
         - "not feasible", "this is not feasible"
         - "unrealistic"
         - "we already tried", "we've already tried"
         - "everyone knows"
         - Direct verdicts: "that's wrong", "that's incorrect"
       Stage 2 -- constructive-context check (if Stage 1 matches):
         Allow the message through if the content (case-insensitive) also contains:
         - "but combined with", "combined with", "but if we"
         - "what if we", "unless we", "could become"
         - "building on that", "and we could", "could also"
         - "in isolation but", "on its own but"
         These patterns indicate the evaluative phrase is being used constructively.
         If Stage 2 passes (constructive context found) -> { allowed: true, violations: [] }
         If Stage 2 does not pass -> { allowed: false, violations: [RuleViolation] }

       Black Hat constraint (RULES-04, Pitfall 1 prevention):
         If message.payload contains hat_color: 'black' AND phase === 'diverge':
         -> { allowed: false, violations: [RuleViolation with rule: 'no-criticism', violation_type: 'timing'}] }
         This is checked BEFORE the evaluative pattern check.

       Violation records:
         Each blocked message produces a RuleViolation with:
         - rule: 'no-criticism'
         - agent: message.from (if AgentRole, otherwise 'facilitator' as fallback)
         - message_id: message.id
         - violation_type: 'content' (evaluative text) or 'timing' (Black Hat in Diverge) or 'activation' (Critic gate -- from Plan 01)
         - description: human-readable description
         - timestamp: Date.now()
         - phase: the phase parameter
         - action_taken: 'blocked'
         Log violation to _violations array (keyed by session_id from message.session_id).

    2. getViolations(session_id: string): RuleViolation[]
       Returns all logged violations for the given session_id.
       Returns [] if no violations logged for that session.
       Violations accumulate -- calling this multiple times returns the same growing list.

    Test cases (spec table + new):
    - Evaluative message blocked during diverge: "that won't work" -> { allowed: false }
    - Building message allowed during diverge: "and we could also" -> { allowed: true }
    - Black Hat message blocked during diverge: message with payload.hat_color === 'black' -> { allowed: false }
    - Violation logged: blocked message -> RuleViolation in getViolations()
    - [RULES-03] 50-sentence false positive corpus: constructive sentences with evaluative vocabulary -> allowed: true

    50-sentence corpus structure (to build in tests):
    25 sentences that SHOULD be blocked (true positives):
      "that won't work", "bad idea, let's drop it", "this is completely unrealistic",
      "not feasible given our constraints", "we already tried this last year",
      "that's a terrible approach", "everyone knows this doesn't scale",
      "this is clearly wrong", "we should not do this", "too risky, abandon it",
      + 15 more direct evaluative sentences

    25 sentences that SHOULD pass (true negatives / false positive prevention):
      "this won't work in isolation but combined with the previous idea it becomes powerful",
      "what if we took the thing that doesn't work about approach A and inverted it?",
      "building on that -- and we could also expand into neighboring markets",
      "unrealistic on its own but could become feasible if we partner with a vendor",
      "bad for solo use but combined with the team format it gains value",
      "not feasible right now unless we automate the manual steps",
      "we already tried a version but what if we made it intentionally extreme?",
      "that seems unlikely at first but what if we combined it with technique B?",
      + 17 more constructive sentences containing evaluative vocabulary

    False positive rate must be <= 5% (max 1 false positive out of 25 negatives).
    True positive rate must be >= 90% (min 22 true positives out of 25 positives).

    DEFAULT_RULES_ENGINE_CONFIG update:
    Populate evaluative_patterns with the hard_block_patterns list.
    Add encouragement_patterns list (build-on patterns): ["and we could", "what if we", "combined with", "building on"]
  </behavior>
  <implementation>
    Extend src/brainstorm/core/rules-engine.ts:

    1. Add _violations: Map<string, RuleViolation[]> to RulesEngine constructor (private, keyed by session_id)

    2. Implement checkMessage():
       - If phase !== 'diverge': return { allowed: true, violations: [] }
       - Check Black Hat constraint first: if message.payload.hat_color === 'black' -> block with violation_type: 'timing'
       - Check content: access message.payload.content as string (may be undefined -- treat as empty string if absent)
       - Stage 1: test content.toLowerCase() against each hard_block_pattern (substring match)
       - Stage 2: if Stage 1 hit, test same content against constructive context patterns -- if any match, allow
       - If blocked: create RuleViolation, push to _violations map under message.session_id, return violation
       - If allowed: return { allowed: true, violations: [] }

    3. Implement getViolations(session_id):
       - return this._violations.get(session_id) ?? []

       ]

         "and we could", "what if we", "combined with", "building on", "could also",

       ]

    Pattern extraction helpers (private methods or module-level functions):
    - isEvaluativeContent(content: string, patterns: string[]): boolean -- Stage 1
    - hasConstructiveContext(content: string, encouragement: string[]): boolean -- Stage 2

    Type safety note: message.payload is Record<string, unknown> per BrainstormMessageSchema.
    Access content as: const content = typeof message.payload.content === 'string' ? message.payload.content : ''
