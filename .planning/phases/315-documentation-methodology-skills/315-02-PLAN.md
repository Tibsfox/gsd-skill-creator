**Content sections:**

1. **Purpose** -- This skill guides agents performing documentation verification for Phases 322 and 324. Documentation drift (the gap between what docs say and what the system does) is the primary failure mode for operations manuals. The doc-verifier detects drift using NASA's TAID (Test, Analysis, Inspection, Demonstration) verification methods applied to documentation artifacts.

2. **Drift Detection Methods** -- Four categories of documentation drift and how to detect each:
   - **Configuration Drift**: Document says setting X=A but system has X=B. Detection: Parse documented config values, compare against actual config files or API responses.
   - **Endpoint Drift**: Document references URL/port that has changed. Detection: Extract documented endpoints, verify reachability and response.
   - **Procedure Drift**: Documented steps no longer produce documented results. Detection: Execute procedure steps, compare actual output to expected output documented in the procedure.
   - **Version Drift**: Document references software version that has been upgraded. Detection: Extract version references, compare against installed versions.

3. **Verification Methods (TAID Applied to Documentation):**
   - **Test**: Execute documented commands and compare output to documented expected results. Automate with shell scripts that parse procedure steps and run them.
   - **Analysis**: Compare documented configuration values against running configuration. Parse globals.yml, service configs, and compare to documented values.
   - **Inspection**: Review documentation structure for completeness: does every procedure have preconditions, verification, rollback? Does every runbook have the required sections?
   - **Demonstration**: Walk through an end-to-end documented scenario and confirm each step works as described.

4. **Automated Verification Framework** -- How to build verification scripts:
   - Pattern: For each documented command in a procedure, extract the command and its expected output
   - Compare strategy: Exact match for deterministic output, pattern match for dynamic values (timestamps, UUIDs), presence check for structural elements
   - Reporting format: Per-procedure pass/fail with specific drift items identified
   - Drift severity: Critical (procedure will fail), Warning (procedure works but output differs), Info (cosmetic differences)

5. **Verification Workflow:**
   - Step 1: Inventory all documented procedures (scan docs/operations-manual/ and docs/runbooks/)
   - Step 2: For each procedure, extract verifiable assertions (commands + expected outputs)
   - Step 3: Execute assertions against running system (or analyze config files for non-destructive verification)
   - Step 4: Compare actual vs documented
   - Step 5: Report drift with severity and affected document sections
   - Step 6: Update last-verified date and system version in verified documents

6. **Safety Constraints:**
   - NEVER execute destructive operations during verification (no delete, no recreate, no modify)
   - Read-only verification commands only (openstack list, openstack show, config file reads)
   - For procedures involving destructive operations, verify preconditions and rollback sections exist but do not execute the destructive steps
   - All verification runs logged for audit trail

7. **Doc Sync Loop Integration** -- How doc-verifier connects to the communication framework:
   - Operates on the doc-sync communication loop (Priority 4)
   - Reports drift to FLIGHT via the doc-sync loop
   - Triggers EXEC-docs to update documentation when drift is detected
   - Cross-reference: communication loop schema from Phase 312

8. **Cross-References:**
