- name: ops-manual-writer
- description: Guidance for writing OpenStack operations manual procedures following NASA document standards. Use when authoring per-service procedures, maintenance workflows, or operational documentation that must be verified against running infrastructure.
- user-invocable: true
- allowed-tools: Read Grep Glob
- metadata.extensions.gsd-skill-creator with version, createdAt, triggers (intents: "write procedure", "operations manual", "document procedure", "nasa procedure format", "ops manual", "maintenance procedure"; contexts: "writing operations documentation", "creating service procedures", "documenting OpenStack operations")

**Content sections:**

1. **Purpose** -- This skill guides agents producing operations manual content for Phase 320. Every procedure must follow NASA document standards adapted for cloud operations per SP-6105 SS 5.5 and NPR 7123.1 Process 9 (Product Transition).

2. **Procedure Format Template** -- The standard format from the NASA reference document (Section 3.4 of gsd-openstack-nasa-reference.md):
   - PROCEDURE ID: OPS-{SERVICE}-{NUMBER} format
   - TITLE, SE PHASE, NPR REFERENCE, LAST VERIFIED, VERIFIED BY fields
   - PURPOSE, PRECONDITIONS, SAFETY CONSIDERATIONS, PROCEDURE (numbered steps with expected results and if-unexpected branches), VERIFICATION, ROLLBACK, REFERENCES sections
   - Include the exact template with placeholder markers

3. **Writing Standards** -- Rules for procedure authoring:
   - Imperative mood for all procedure steps ("Run", "Verify", not "You should run")
   - Every step must include the exact command or action (no "configure as needed")
   - Expected results stated for every step (what the operator should see)
   - If-unexpected branches for steps that could fail
   - Safety considerations before any destructive operation
   - All external references must include SP-6105 or NPR 7123.1 section numbers

4. **Per-Service Procedure Categories** -- Standard categories each service needs:
   - Service health check (daily)
   - Configuration verification (after changes)
   - Backup and restore
   - Upgrade procedure (minor and major)
   - Troubleshooting common failures
   - Security audit procedure

5. **Verification Requirements** -- How procedures get verified:
   - Every procedure must have a verification section proving success
   - Procedures must be testable against a running system
   - doc-verifier skill handles automated verification (cross-reference)
   - Last-verified date and system version must be recorded

6. **Cross-References** -- Link to:
   - NASA SE Methodology skill for lifecycle phase context
   - Doc-verifier skill for verification methods
   - Filesystem contracts for output location (docs/operations-manual/)
   - NASA-STD-3001 (adapted) for procedure format origin

Keep total content under 8K tokens. The skill should be actionable -- an executor agent reading this skill should produce correctly formatted procedures without needing additional guidance.

**Frontmatter:**
- name: runbook-generator
- description: Guidance for generating OpenStack runbooks with dual task/symptom indexing following NASA procedure standards. Use when creating incident response procedures, operational runbooks, or troubleshooting guides that must be verified against running infrastructure.
- user-invocable: true
- allowed-tools: Read Grep Glob
- metadata.extensions.gsd-skill-creator with version, createdAt, triggers (intents: "runbook", "incident response", "troubleshoot", "on-call", "symptom index", "task index", "generate runbook"; contexts: "creating runbook entries", "building incident response library", "documenting troubleshooting procedures")

**Content sections:**

1. **Purpose** -- This skill guides agents producing runbook library content for Phase 321. Runbooks provide step-by-step procedures indexed by both task (what you want to do) and symptom (what went wrong), following the NASA procedure format from SP-6105 SS 5.3.

2. **Runbook Entry Format** -- The standard format from the vision document (Section 2.3):
   - RUNBOOK: {ID}-{TITLE} with RB-{SERVICE}-{NNN} ID pattern
   - SE Phase Reference, Last Verified Against, Verification Method fields
   - PRECONDITIONS, PROCEDURE (numbered steps with expected results and if-not branches), VERIFICATION, ROLLBACK, RELATED RUNBOOKS sections
   - Include the exact template with placeholder markers

3. **Dual Indexing System** -- How runbooks are indexed two ways:
   - Task Index: Organized by DEPLOY, OPERATE, MONITOR categories (what you want to do)
   - Symptom Index: Organized by failure symptom (INSTANCE WON'T LAUNCH, NETWORK UNREACHABLE, STORAGE UNAVAILABLE, AUTHENTICATION FAILED)
   - Every runbook must appear in at least one task index entry AND be discoverable by symptom
   - Index files: docs/runbooks/task-index.md and docs/runbooks/symptom-index.md

4. **Runbook Categories by Service** -- Standard runbook coverage per OpenStack service:
   - Keystone: token issues, catalog problems, RBAC failures, federation issues
   - Nova: instance launch failures, scheduling problems, hypervisor issues, live migration
   - Neutron: network connectivity, DHCP failures, floating IP issues, security group problems, OVS/OVN issues
   - Cinder: volume creation failures, attachment issues, backend connectivity, snapshot problems
   - Glance: image upload failures, format conversion issues, metadata problems
   - Swift: container access issues, replication problems, quota issues
   - Heat: stack creation failures, template validation, resource dependencies
   - Horizon: dashboard access issues, session problems, panel errors

5. **Naming Convention** -- RB-{SERVICE}-{NNN} format:
   - SERVICE: uppercase service name (KEYSTONE, NOVA, NEUTRON, CINDER, GLANCE, SWIFT, HEAT, HORIZON, KOLLA, GENERAL)
   - NNN: 3-digit sequential number per service, starting at 001
   - Cross-service runbooks use GENERAL prefix

6. **Verification Integration** -- How runbooks connect to verification:
   - Each runbook specifies whether verification is automated or manual
   - Automated verification commands should be executable via doc-verifier
   - Last-verified date tracks when the runbook was confirmed against a running system
   - Unverified runbooks must be marked with visible warning

7. **Cross-References** -- Link to:
   - NASA SE Methodology skill for phase context
   - ops-manual-writer skill for procedure format alignment
   - Doc-verifier skill for verification methods
   - Filesystem contracts for output location (docs/runbooks/)
