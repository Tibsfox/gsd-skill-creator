**Frontmatter:**
- `name: openstack-neutron`
- `description:` Mention: software-defined networking, security groups, floating IPs, DHCP, L3 routing, OVN/OVS, ML2 plugin, network namespaces
- `user-invocable: true`
- `allowed-tools: Read Grep Glob`
- `metadata.extensions.gsd-skill-creator:` with version, createdAt, triggers
- Trigger intents: "neutron", "network", "subnet", "router", "floating ip", "security group", "SDN", "OVN", "OVS", "DHCP", "L3", "port"
- Trigger contexts: "deploying openstack networking", "configuring SDN", "troubleshooting network connectivity", "managing security groups"

**Body sections:**

1. **Introduction** - Neutron's role as the networking abstraction layer, ML2 plugin architecture, OVN vs OVS backends, network types (flat, VLAN, VXLAN, GRE), the agent model

2. **Deploy** - Kolla-Ansible specifics: globals.yml for neutron (`neutron_plugin_agent: ovn` or `openvswitch`, `neutron_external_interface`, `enable_neutron_provider_networks: yes`, `enable_neutron_dvr: no` for single-node), network bridge configuration (`br-ex` for external), container verification (`docker ps | grep neutron`), agent status check (`openstack network agent list`)

3. **Configure** - Key areas: provider networks vs tenant networks, subnet configuration (CIDR, gateway, DHCP range, DNS), router configuration (external gateway, SNAT), security group rules (default deny ingress, allow egress), floating IP pool setup, MTU configuration, DNS integration, allowed address pairs, port security

4. **Operate** - Day-2: network/subnet/router CRUD, floating IP management, security group rule management, network namespace debugging (`ip netns exec`), DHCP agent management, L3 agent management, bandwidth and QoS policies, port diagnostics

5. **Troubleshoot** - Common failure modes (this is where Neutron's complexity shows):
   - Instance has no network connectivity (port binding failure, security group blocking, DHCP failure, wrong network namespace)
   - Floating IP not reachable (router gateway not set, SNAT rules missing, external bridge misconfigured, iptables/nftables rules)
   - DHCP failures (agent down, namespace issues, dnsmasq process, lease file corruption)
   - Security group rules not applying (port security disabled, wrong direction, OVS flow rules stale)
   - Network creation fails (no available VLAN/VXLAN IDs, provider network misconfigured)
   - MTU/fragmentation issues (VXLAN overhead, jumbo frames, path MTU)
   - OVN/OVS specific issues (ovs-vsctl show, OVN northbound/southbound DB sync)

6. **Integration Points** - Keystone auth, Nova port creation during instance boot, metadata service via Neutron, Octavia (LBaaS), VPNaaS

7. **NASA SE Cross-References** - Phase B (network design), Phase C (network configuration), Phase D (connectivity verification), Phase E (network operations)

**Token budget:** Target 7-8K tokens. Neutron is the most complex service -- use the full 8K budget. Networking troubleshooting is where operators spend most of their time, so the troubleshoot section should be the most detailed.

Create `skills/openstack/cinder/SKILL.md`. Cinder provides persistent block storage volumes that attach to Nova instances.

**Frontmatter:**
- `name: openstack-cinder`
- `description:` Mention: block storage, volume management, snapshots, backups, LVM backend, iSCSI, volume types, encryption
- Trigger intents: "cinder", "volume", "block storage", "snapshot", "backup", "LVM", "iSCSI", "volume type", "storage backend"
- Trigger contexts: "deploying openstack storage", "managing volumes", "troubleshooting storage", "configuring storage backends"

**Body sections:**

1. **Introduction** - Cinder's role as the block storage abstraction, backend driver architecture (LVM/iSCSI for single-node, Ceph for multi-node), volume lifecycle

2. **Deploy** - Kolla-Ansible specifics: globals.yml for cinder (`enable_cinder: yes`, `cinder_volume_group: cinder-volumes`, `enable_cinder_backup: yes`), LVM volume group setup (`pvcreate`, `vgcreate cinder-volumes`), container verification, volume service status (`openstack volume service list`)

3. **Configure** - Key areas: volume types and extra specs, QoS policies, volume encryption (LUKS), backend configuration (LVM, NFS, Ceph), replication (for future multi-node), scheduler filters, default volume type, oversubscription ratios

4. **Operate** - Day-2: volume CRUD, attach/detach to instances, snapshot management, volume backup/restore, volume migration, extending volumes, managing volume types, transfer volumes between projects, force-detach stuck volumes

5. **Troubleshoot** - Common failure modes:
   - Volume stuck in "creating"/"deleting" state (LVM/iSCSI target issues, backend full)
   - Volume attach failures (iSCSI initiator not configured, multipath issues, device busy)
   - Snapshot failures (not enough space in VG, COW overflow)
   - Backup failures (backup service not running, backup storage full)
   - Volume group not found (LVM not configured, VG name mismatch)

6. **Integration Points** - Keystone auth, Nova volume attachment (os-brick), Glance volume-from-image, backup to Swift

7. **NASA SE Cross-References** - Phase B (storage design), Phase C (backend configuration), Phase D (volume verification), Phase E (backup/recovery operations)

**Token budget:** Target 5-6K tokens. Cinder is moderately complex.

Create `skills/openstack/swift/SKILL.md`. Swift provides S3-compatible object storage with containers and ACLs.

**Frontmatter:**
- `name: openstack-swift`
- `description:` Mention: object storage, S3 compatible, containers, ACLs, rings, replication, eventual consistency, large objects
- Trigger intents: "swift", "object storage", "container", "s3 compatible", "large object", "ring", "ACL", "tempurl"
- Trigger contexts: "deploying openstack object storage", "managing swift containers", "troubleshooting object storage", "configuring s3 compatibility"

**Body sections:**

1. **Introduction** - Swift's role as distributed object storage, S3 API compatibility, ring architecture (for understanding even in single-node), eventual consistency model, containers vs buckets terminology

2. **Deploy** - Kolla-Ansible specifics: globals.yml for swift (`enable_swift: yes`, disk configuration for swift storage), ring building (single-node simplified), container/object/account service verification, S3 API middleware configuration (`swift3` or `s3api`)

3. **Configure** - Key areas: container/account/object ring configuration (for single-node: 1 replica), storage policies, container sync, cross-domain CORS, tempURL configuration, ACLs (container-level and account-level), large object support (SLO/DLO), rate limiting, object versioning, object expiry

4. **Operate** - Day-2: container/object CRUD via CLI and S3 API, ACL management, large file upload (segmented), container listings and metadata, storage usage reporting, ring rebalancing (relevant for future multi-node), object auditing

5. **Troubleshoot** - Common failure modes:
   - Object upload failures (storage full, ring misconfigured, proxy timeout)
   - S3 API compatibility issues (signature mismatch, missing middleware, unsupported operations)
   - Container listing empty/incomplete (eventual consistency delay, object server unreachable)
   - Permission denied on container operations (ACL misconfiguration, Keystone role mapping)
   - Ring builder errors (no devices added, partition power mismatch)

6. **Integration Points** - Keystone auth (including S3-style credentials), Glance image backend, Cinder backup target, tempURL for unauthenticated access

7. **NASA SE Cross-References** - Phase B (storage architecture), Phase C (ring configuration), Phase D (object upload/download verification), Phase E (storage operations)

**Token budget:** Target 5-6K tokens. Swift is simpler in single-node context but has unique ring concepts.
