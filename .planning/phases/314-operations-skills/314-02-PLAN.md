- Trigger contexts: "troubleshooting network connectivity", "debugging SDN", "analyzing packet flows", "diagnosing floating IP issues"

**Body sections (all required):**

1. **Introduction** - Role of networking debug in cloud operations, the SDN debugging challenge (virtual networks add abstraction layers between user intent and physical packets), the debugging mental model (start at the instance, trace outward through tap device, bridge, OVS/OVN, physical NIC), relationship to CRAFT-network agent

2. **Deploy** - Debug tooling setup: OVS diagnostic commands availability (`ovs-vsctl`, `ovs-ofctl`, `ovs-appctl`), OVN diagnostic commands (`ovn-nbctl`, `ovn-sbctl`, `ovn-trace`), network namespace tools (`ip netns`), tcpdump/tshark availability in containers, Kolla-Ansible network debug container options

3. **Configure** - Key debugging configuration: OVS logging levels (adjusting for debug vs production), OVN tracing enablement, packet capture setup (tcpdump on tap interfaces, bridge ports, physical NICs), flow table inspection setup, network agent debug logging (Neutron L3 agent, DHCP agent, OVS agent, metadata agent)

4. **Operate** - Day-2 debugging procedures:
   - **Connectivity diagnosis workflow:** Instance cannot reach external network -- step-by-step trace from instance through tap, qbr bridge, OVS br-int, br-ex, physical NIC
   - **DHCP troubleshooting:** Instance gets no IP -- check DHCP agent namespace, dnsmasq process, DHCP offer/request packet capture
   - **Floating IP diagnosis:** Cannot reach instance from external -- check NAT rules, router namespace, iptables/nftables, ARP resolution
   - **Security group analysis:** Traffic blocked -- decode OVS flow rules mapped to security group rules, conntrack state
   - **MTU issues:** Large packet drops -- check MTU chain from instance through overlay to physical network
   - **DNS resolution:** Instance cannot resolve names -- check DHCP options, DNS server configuration, metadata agent

5. **Troubleshoot** - Common failure patterns with systematic resolution:
   - Instance has no network connectivity (tap device missing, OVS port not bound, wrong VLAN, namespace routing)
   - Floating IP unreachable from external (router namespace missing, NAT rules absent, ARP not resolving on external bridge)
   - Inter-tenant traffic leaking (VLAN/VXLAN ID collision, security group misconfiguration, OVS flow table corruption)
   - DHCP failures (agent not running, namespace missing, dnsmasq crash, port mismatch)
   - Metadata service unreachable from instance (metadata proxy misconfigured, Neutron metadata agent down, routing through router namespace broken)
   - East-west traffic between instances fails (same subnet: bridge issue, different subnet: router namespace, security groups)

6. **Integration Points** - How networking-debug connects:
   - Neutron skill: core networking knowledge, SDN architecture understanding
   - Security skill: security group rule analysis during network debug
   - Monitoring skill: network metrics (packet drops, latency, errors) guide where to look
   - Capacity skill: network resource exhaustion (port limits, floating IP pool depletion)
   - Nova skill: instance networking (tap devices, port binding, metadata service)
   - CRAFT-network agent: primary consumer for SDN troubleshooting

7. **NASA SE Cross-References** - Phase D (network integration testing), Phase E (operational network troubleshooting), SP-6105 SS 5.2 (Product Integration -- service interface verification)

**Token budget:** Target 6-7K tokens. Networking debug is the most hands-on troubleshooting skill. Prioritize step-by-step diagnostic procedures with exact commands. Every troubleshooting entry should have a "run this, see that, means this" structure.

Create `skills/openstack/kolla-ansible-ops/SKILL.md` following the same pattern. This skill covers day-2 Kolla-Ansible operations -- distinct from the kolla-ansible deployment skill (Phase 313) which covers initial deployment. This skill covers what happens AFTER the cloud is running: upgrades, reconfiguration, container management, rolling updates.

**Frontmatter:**
- `name: openstack-kolla-ansible-ops`
- `description:` Must mention: Kolla-Ansible day-2 operations, upgrades, reconfigure, container management, rolling updates, service restart, maintenance mode, config changes. Explicitly note this is for post-deployment operations, not initial deployment.
- Trigger intents: "kolla upgrade", "kolla reconfigure", "container restart", "rolling update", "service maintenance", "kolla-ansible ops", "config change", "container management"
- Trigger contexts: "upgrading openstack", "reconfiguring services", "managing kolla containers", "performing maintenance"

**Body sections:**

1. **Introduction** - Kolla-Ansible as the infrastructure lifecycle tool (not just deployer), the operational command set (reconfigure, upgrade, stop, deploy, pull), relationship to the kolla-ansible deployment skill (Phase 313 covers initial `kolla-ansible deploy`, this skill covers everything after), container lifecycle in Kolla (image pull, deploy, reconfigure, upgrade), why this separation matters (deployment is one-time, ops is ongoing)

2. **Deploy** - Operational prerequisites: Kolla-Ansible virtual environment maintenance, inventory file management, globals.yml change tracking (git-controlled), password management (`kolla-ansible -i inventory genpasswd` for initial, manual rotation after), image management (registry configuration, local image cache)

3. **Configure** - Key operational configurations: service reconfiguration workflow (`kolla-ansible -i inventory reconfigure` -- what it does: regenerates config files, restarts affected containers), partial reconfiguration (`--tags` for targeting specific services), config validation before apply (`kolla-ansible -i inventory prechecks`), custom config overlays (config_overrides directory structure), globals.yml change management (what changes require reconfigure vs redeploy)

4. **Operate** - Day-2 operations (the core of this skill):
   - **Service reconfigure:** Change globals.yml or custom configs, run prechecks, run reconfigure, verify service health
   - **Minor upgrade (patch):** Pull new images, run upgrade, verify services (within same major release)
   - **Major upgrade:** Pre-upgrade checks, backup databases, pull images, run upgrade, post-upgrade verification, rollback if needed
   - **Container management:** Restart individual service (`docker restart nova_api`), inspect container logs (`docker logs --tail 100 neutron_server`), check container health (`docker inspect --format='{{.State.Health.Status}}' keystone`)
   - **Maintenance mode:** Disable compute service before host maintenance, drain instances, perform maintenance, re-enable, verify
   - **Password rotation:** Generate new passwords, apply via reconfigure, verify all services reconnect
   - **Certificate renewal:** Update certificates, reconfigure affected services (HAProxy, internal TLS), verify TLS connectivity

5. **Troubleshoot** - Common failure modes:
   - Reconfigure fails midway (partial config state -- identify affected services, rerun with `--tags`)
   - Upgrade fails on prechecks (version incompatibility, unsupported configuration, missing migration)
   - Container won't start after upgrade (image pull failure, config incompatibility, database migration needed)
   - Service unhealthy after reconfigure (config syntax error in override, missing dependency)
   - Rollback needed after failed upgrade (database backup restore procedure, image revert, full redeployment)
   - Ansible connection failures (SSH key issues, become password, inventory mismatch)

6. **Integration Points** - How kolla-ansible-ops connects:
   - Kolla-ansible (deployment) skill: shares inventory and globals.yml, this skill picks up where deployment left off
   - Backup skill: always backup before upgrade/reconfigure operations
   - Monitoring skill: verify service health after every operational change
   - Security skill: certificate renewal triggers reconfigure, password rotation
   - All core skills: every service is managed through Kolla-Ansible containers
   - EXEC agent: primary consumer for infrastructure changes

7. **NASA SE Cross-References** - Phase C (configuration management), Phase E (maintenance and upgrade operations), SP-6105 SS 6.5 (Configuration Management -- change control process)

**Token budget:** Target 6-7K tokens. This is the most operationally dense skill -- every procedure should have the exact `kolla-ansible` command with relevant flags. Operators will use this as their primary reference for making changes to the running cloud.
