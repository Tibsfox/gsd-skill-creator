**07-phase-f-closeout.md** -- Phase F: Closeout chapter:
- Title: "Chapter 7: Closeout -- Graceful Shutdown"
- NASA SE cross-ref header: "SE Phase: Phase F | SP-6105 SS 6.1 | NPR 7123.1 SS 6.1 | Review Gate: DR"
- Section 1: When to Decommission -- decision criteria for cloud lifecycle end: hardware end-of-life, migration to different platform, consolidation of infrastructure, end of project. This is not a failure -- it is the planned completion of the lifecycle (per SP-6105 SS 6.1 Technical Planning, lifecycle includes closeout)
- Section 2: Instance Migration Procedures -- before shutting down, ensure all workloads are migrated or archived:
  - Inventory all running instances and their owners
  - Export instance data (volume snapshots, configuration backups)
  - Notify stakeholders with migration timeline
  - Verify migration completeness before proceeding
- Section 3: Data Export and Archive -- systematic data preservation:
  - Export all project data (instances, volumes, networks, security groups)
  - Archive service databases (Keystone, Nova, Neutron, Cinder, Glance)
  - Archive configuration files with version history (complete git repo)
  - Archive monitoring data (Prometheus metrics, Grafana dashboards)
  - Archive all documentation (sysadmin guide, ops manual, runbooks) as versioned snapshot
  - Verify archive integrity (checksums, test restore)
- Section 4: Service Decommissioning Sequence -- shut down services in reverse dependency order:
  1. Heat (orchestration -- no active stacks remaining)
  2. Horizon (dashboard -- no longer needed)
  3. Swift (object storage -- all objects archived)
  4. Cinder (block storage -- all volumes detached and archived)
  5. Neutron (networking -- all ports released)
  6. Nova (compute -- all instances terminated)
  7. Glance (images -- all images archived)
  8. Keystone (identity -- last service, no longer authenticating anyone)
  Explain why reverse order matters (prevents orphaned resources, ensures clean shutdown)
- Section 5: Resource Recovery -- after services are down:
  - Remove Kolla-Ansible containers and images (`kolla-ansible destroy --yes-i-really-really-mean-it`)
  - Clean up storage (LVM volume groups, Swift rings)
  - Restore network configuration to pre-deployment state
  - Verify hardware is clean and available for reuse
  - Update hardware inventory
- Section 6: Lessons Learned Compilation -- capture mission retrospective in NASA LLIS (Lessons Learned Information System) format:
  - What worked well (deployment automation, documentation methodology, monitoring)
  - What could be improved (areas of friction, pain points, process gaps)
  - Actionable recommendations for future deployments
  - Reference to docs/lessons-learned.md for the formal LLIS document
  - Per NASA's LLIS methodology: each lesson should be specific, actionable, and traceable to the experience that generated it
- Section 7: Final Mission Report -- summarize the complete lifecycle:
  - Mission timeline (concept through closeout)
  - Requirements fulfillment summary (which requirements met, which waived, which deferred)
  - Key metrics (uptime, incidents, changes deployed, documentation accuracy)
  - Recommendations for successor systems
- Section 8: Phase Gate Criteria -- checklist for DR (Decommissioning Review): all user data migrated or archived with written confirmation, all services stopped with no OpenStack processes running, hardware resources recovered and inventoried, lessons learned document published in NASA LLIS format with at least 3 actionable recommendations
- Include SP-6105 SS 6.1 and NPR 7123.1 cross-references throughout
