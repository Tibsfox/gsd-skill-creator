  <action>
Create `infra/scripts/vm-lifecycle.sh` -- unified VM lifecycle manager that auto-detects the hypervisor backend and dispatches operations through a single interface.

Structure and behavior:

1. **Shebang + strict mode**: `#!/usr/bin/env bash` + `set -euo pipefail`

2. **Source shared library**: Source `lib/vm-common.sh` for backend detection, validation, logging

3. **Backend loading**: Call `vm_detect_backend()` to determine KVM or VMware, then source the appropriate backend script (`vm-backend-kvm.sh` or `vm-backend-vmware.sh`). Map backend functions to generic names:
   - If KVM: `vm_do_create=kvm_create`, `vm_do_start=kvm_start`, etc.
   - If VMware: `vm_do_create=vmware_create`, etc.

4. **Subcommands** (all idempotent):

   **`create`** -- Create a new VM
   - Args: `--name NAME --ram-mb N --vcpus N --disk-gb N [--iso PATH] [--pxe] [--kickstart-url URL] [--network BRIDGE] [--storage-dir DIR] [--os-variant TYPE]`
   - OR: `--values PATH` (load all params from local-values file via `vm_load_values()`)
   - Validate params via `vm_validate_params()`
   - Call `$vm_do_create` with validated params
   - Idempotent: if VM exists, log "already exists" and exit 0

   **`start`** -- Start a VM: `--name NAME`
   - Call `$vm_do_start`; idempotent (already running = exit 0)

   **`stop`** -- Graceful shutdown: `--name NAME [--timeout SECS]`
   - Call `$vm_do_stop`; idempotent (already stopped = exit 0)

   **`snapshot`** -- Create named snapshot: `--name NAME --snapshot SNAP_NAME [--description TEXT]`
   - Call `$vm_do_snapshot`; idempotent (snapshot exists = exit 0)

   **`clone`** -- Clone VM: `--source NAME --name NEW_NAME [--snapshot SNAP_NAME]`
   - Stop source if running, call `$vm_do_clone`
   - Idempotent: if target exists, exit 0

   **`destroy`** -- Remove VM completely: `--name NAME [--force]`
   - Require `--force` flag as safety (no accidental deletions)
   - Stop if running, call `$vm_do_destroy`
   - Idempotent: if VM does not exist, exit 0

   **`status`** -- Show VM state: `--name NAME`
   - Print: exists/not-found, running/stopped, snapshots list

   **`list`** -- List all managed VMs
   - Backend-specific: `virsh list --all` or `vmrun list`

5. **Global flags**: `--dry-run` (log actions without executing), `--help`, `--backend kvm|vmware` (override auto-detection)

6. **Usage/help**: Clear help text with examples for each subcommand

7. **Exit codes**: 0=success, 1=operation failed, 2=usage error, 3=no backend available

Follow deploy-pxe.sh patterns: step numbering, colored output, clear progress logging. Every operation logs what it's doing and what it found (backend type, VM state before action).

  <action>
Create `infra/scripts/provision-vm.sh` -- master VM provisioning orchestrator following the deploy-pxe.sh pattern (numbered steps, dry-run, validation-before-action).

This script orchestrates the complete VM provisioning workflow:

**Modes:**

1. **`--mode fresh`** -- Full provisioning from scratch via PXE/kickstart:
   - Step 1: Validate prerequisites (vm-lifecycle.sh, resource-budget.yaml, PXE infrastructure)
   - Step 2: Read resource-budget.yaml for VM sizing (minecraft_vm section)
   - Step 3: Read VM local-values for network, storage, and kickstart configuration
   - Step 4: Create VM via `vm-lifecycle.sh create --pxe` with budget-derived sizing
   - Step 5: Start VM (PXE boots into kickstart)
   - Step 6: Wait for installation to complete (poll VM state, configurable timeout, default 15 min)
   - Step 7: Verify VM is running and accessible (check VM state)
   - Step 8: Report provisioning result with timing

2. **`--mode golden`** -- Create golden image from a provisioned VM:
   - Step 1: Validate source VM exists and is running
   - Step 2: Stop source VM gracefully
   - Step 3: Create snapshot named "golden-YYYYMMDD-HHMMSS"
   - Step 4: Start source VM back up
   - Step 5: Report golden image details (snapshot name, source VM)

3. **`--mode clone`** -- Clone from golden image (the <5 minute path):
   - Step 1: Validate source VM and golden snapshot exist
   - Step 2: Clone from golden snapshot to new VM name
   - Step 3: Start the cloned VM
   - Step 4: Wait for boot (poll VM state, timeout 120 seconds)
   - Step 5: Report clone result with timing
   - This is the path that must complete in under 5 minutes per INFRA-11

4. **`--mode destroy`** -- Clean teardown:
   - Step 1: Stop and destroy specified VM via `vm-lifecycle.sh destroy --force`
   - Step 2: Confirm destruction

**Arguments:**
- `--mode fresh|golden|clone|destroy` (required)
- `--name VM_NAME` (required for all modes)
- `--source SOURCE_VM` (required for clone mode)
- `--snapshot SNAP_NAME` (optional for clone, defaults to latest golden-*)
- `--values PATH` (local-values file, defaults to `infra/local/vm-provisioning.local-values`)
- `--dry-run` (preview without executing)
- `--help`

**Integration points:**
- Reads `infra/local/resource-budget.yaml` for VM sizing (from calculate-budget.sh output)
- Reads VM local-values for site-specific configuration
- Calls `vm-lifecycle.sh` for all VM operations (never calls backends directly)
- Uses render-pxe-menu.sh to render any VM-specific templates if needed
- Follows deploy-pxe.sh orchestrator pattern: numbered steps, config backup, validation-first

**Timing measurement:**
- Record start/end time for each mode
- Clone mode: assert and report total time (target: under 5 minutes)
- Log all step durations for debugging
