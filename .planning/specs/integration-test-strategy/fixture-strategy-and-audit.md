## 1. Fixture Strategy

*Satisfies: INTTEST-06 (test fixture directory structure with per-flow snapshot projects)*

### 1.1 Purpose and Design Goals

Contract tests (defined in [contract-testing-approach.md](./contract-testing-approach.md)) verify that data crossing component boundaries conforms to Zod schema contracts. Those tests need **realistic fixture data** that represents actual pipeline inputs and outputs at every stage.

The fixture strategy provides:

1. **One directory per priority flow** -- the 6 flows from Plan 01 each get a dedicated fixture directory with stage-numbered snapshot files.
2. **Expected JSON Schema snapshots** -- each flow directory contains an `expected-schemas/` subdirectory with JSON Schema files generated by `.toJSONSchema()` at fixture creation time.
3. **Realistic data** -- fixture files contain domain-accurate values (real milestone names, valid ISO timestamps, correct enum values), not lorem ipsum or stub data.
4. **Append-only history** -- fixtures are never deleted. Old fixtures become regression tests when schemas evolve.

### 1.3 Fixture File Conventions

**Numbered prefixes (01-, 02-, 03-)** indicate pipeline stage order within each flow. The number reflects the chronological order in which data passes through the pipeline:

- `01-` is the initial input (what enters the pipeline)
- `02-` is an intermediate transformation (what the pipeline produces midway)
- `03-` is the final output (what exits the pipeline)

Some flows have more or fewer stages -- the numbering adapts to the flow's actual pipeline depth.

**File format rules:**

| Rule | Detail |
|------|--------|
| JSON for data payloads | Files containing structured data that would be parsed by `JSON.parse()` in production use `.json` extension |
| YAML for pipeline definitions | Files that are YAML in production (Copper pipeline definitions, skill input files) use `.yaml` extension |
| Markdown for project fixtures | The orchestrator state flow uses actual `.md` files (ROADMAP.md, STATE.md, PROJECT.md) because the orchestrator parses Markdown in production |
| Realistic data values | Every field contains a domain-accurate value. Timestamps are valid ISO 8601 UTC. IDs match their schema patterns (e.g., `msg-YYYYMMDD-NNN`). Enum values are from the actual enum set. No `"foo"`, `"bar"`, `"test"`, or `"lorem ipsum"`. |
| Forward-compatible fields | Fixture files for schemas using `.passthrough()` should include 1-2 unknown fields to verify that passthrough behavior is preserved during parsing |

**`expected-schemas/` directory:**

Each flow directory contains an `expected-schemas/` subdirectory with JSON Schema snapshots:

- Generated by calling `.toJSONSchema()` on the Zod schema at fixture creation time
- Used by structural contract tests for snapshot comparison (`toMatchSnapshot()`)
- File naming: `{schema-name}.schema.json` in kebab-case
- JSON Schema version: Draft 2020-12 (the version Zod 4.3.6 generates)

**`_fixture_meta` top-level field:**

Every JSON fixture file includes a `_fixture_meta` field at the top level:

```json
{
  "_fixture_meta": {
    "source_schema": "EventEnvelopeSchema",
    "created_date": "2026-02-19",
    "created_by_phase": 234,
    "notes": "Realistic telemetry update envelope for ICD-01 contract test"
  },
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "timestamp": "2026-02-19T15:10:05.123Z",
  ...
}
```

For YAML fixture files, the metadata is a top-level `_fixture_meta` key:

```yaml
_fixture_meta:
  source_schema: PipelineSchema
  created_date: "2026-02-19"
  created_by_phase: 234
  notes: "Phase execution pipeline with WAIT, MOVE, SKIP instructions"

metadata:
  name: phase-execution
  ...
```

**Why `_fixture_meta` works with `.passthrough()`:** All production schemas in this project use `.passthrough()`, which preserves unknown fields during parsing. The `_fixture_meta` field passes through parsing without interference because it is an unknown key that Zod's passthrough mode preserves. Contract tests can verify this behavior -- if a schema accidentally drops `.passthrough()`, the `_fixture_meta` field will be stripped, causing the test to fail.

### 1.4 Fixture Lifecycle

| Event | Action | Trigger |
|-------|--------|---------|
| **Created** | Fixture files and expected schemas are generated when contract tests are first implemented | Future implementation milestone creates contract test files |
| **Updated** | Fixture data and expected schemas are re-verified when schema changes trigger freshness policy | Per INTTEST-05 freshness policy defined in Plan 02 |
| **Deleted** | Never | Fixtures are append-only history |
| **Versioned** | Fixture files are committed to git alongside their contract test files | Same commit that creates/updates the contract test |

**Append-only rationale:** When a schema evolves (field added, type changed, constraint modified), the old fixture becomes a regression test. It verifies that the new schema can still parse data in the old shape (backward compatibility) or that it correctly rejects the old shape (intentional breaking change). Deleting old fixtures loses this regression coverage.

**Version strategy:** Fixture files live in `.planning/fixtures/` and are committed to the same git repository as the contract tests that use them. There is no separate fixture versioning system -- git history IS the version history. When a fixture is updated, the git diff shows exactly what changed.

### 1.5 Fixture Loading Convention

Contract tests load fixture data using a standard helper pattern:

```typescript
import { readFileSync } from 'node:fs';
import { resolve, join } from 'node:path';
import * as yaml from 'js-yaml';

/**
 * Root directory for all test fixtures.
 * Resolved relative to the project root (where package.json lives).
 */
const FIXTURES_ROOT = resolve(process.cwd(), '.planning', 'fixtures');

/**
 * Load a JSON or YAML fixture by flow name and stage filename.
 *
 * @param flowName - Directory name under .planning/fixtures/ (e.g., 'console-message-flow')
 * @param stage - Filename within the flow directory (e.g., '01-dashboard-creates.json')
 * @returns Parsed fixture data as unknown (caller validates with Zod)
 *
 * @example
 * const input = loadFixture('console-message-flow', '01-dashboard-creates.json');
 * const result = MessageEnvelopeSchema.safeParse(input);
 */
function loadFixture(flowName: string, stage: string): unknown {
  const filePath = join(FIXTURES_ROOT, flowName, stage);
  const content = readFileSync(filePath, 'utf-8');

  if (stage.endsWith('.yaml') || stage.endsWith('.yml')) {
    return yaml.load(content);
  }
  return JSON.parse(content);
}

/**
 * Load an expected JSON Schema snapshot for schema comparison.
 *
 * @param flowName - Directory name under .planning/fixtures/
 * @param schemaFile - Filename within the expected-schemas/ subdirectory
 * @returns Parsed JSON Schema object
 *
 * @example
 * const expectedSchema = loadExpectedSchema('console-message-flow', 'message-envelope.schema.json');
 * const currentSchema = MessageEnvelopeSchema.toJSONSchema();
 * expect(currentSchema).toEqual(expectedSchema);
 */
function loadExpectedSchema(flowName: string, schemaFile: string): unknown {
  const filePath = join(FIXTURES_ROOT, flowName, 'expected-schemas', schemaFile);
  const content = readFileSync(filePath, 'utf-8');
  return JSON.parse(content);
}

/**
 * Load all JSON fixtures from a subdirectory (e.g., ICD payload examples).
 *
 * @param flowName - Directory name under .planning/fixtures/
 * @param subdir - Subdirectory within the flow (e.g., '03-icd-payload-examples')
 * @returns Map of filename to parsed data
 *
 * @example
 * const payloads = loadFixtureDir('amiga-envelope-flow', '03-icd-payload-examples');
 * // payloads = { 'icd-01-telemetry.json': {...}, 'icd-02-dependency.json': {...}, ... }
 */
function loadFixtureDir(flowName: string, subdir: string): Record<string, unknown> {
  const dirPath = join(FIXTURES_ROOT, flowName, subdir);
  const { readdirSync } = await import('node:fs');
  const files = readdirSync(dirPath).filter(f => f.endsWith('.json') || f.endsWith('.yaml'));
  const result: Record<string, unknown> = {};
  for (const file of files) {
    result[file] = loadFixture(join(flowName, subdir), '../' + subdir + '/' + file);
  }
  return result;
}
```

**Loading rules:**

| Rule | Detail |
|------|--------|
| JSON fixtures | Loaded via `JSON.parse(readFileSync(...))` |
| YAML fixtures | Loaded via `yaml.load(readFileSync(...))` using `js-yaml` (already a project dependency) |
| Return type is `unknown` | Caller is responsible for validating via Zod `.parse()` or `.safeParse()` -- fixtures are untrusted data |
| Path resolution | All paths are relative to `.planning/fixtures/` (the `FIXTURES_ROOT` constant) |
| Expected schemas | Loaded separately via `loadExpectedSchema()` for snapshot comparison |

**Usage in contract tests:**

```typescript
import { describe, it, expect } from 'vitest';
import { EventEnvelopeSchema } from '../../amiga/message-envelope.js';

// Hypothetical loadFixture helper (implemented per convention above)
import { loadFixture, loadExpectedSchema } from '../../../test-utils/fixtures.js';

describe('Console Message Flow - Boundary 2: Adapter to EventEnvelope', () => {
  it('structural: JSON Schema matches expected snapshot', () => {
    const currentSchema = EventEnvelopeSchema.toJSONSchema();
    const expectedSchema = loadExpectedSchema(
      'console-message-flow',
      'event-envelope.schema.json'
    );
    expect(currentSchema).toEqual(expectedSchema);
  });

  it('semantic: adapter output validates against EventEnvelopeSchema', () => {
    const adapterOutput = loadFixture('console-message-flow', '03-adapter-output.json');
    const result = EventEnvelopeSchema.safeParse(adapterOutput);
    expect(result.success).toBe(true);
  });

  it('semantic: _fixture_meta survives passthrough', () => {
    const adapterOutput = loadFixture('console-message-flow', '03-adapter-output.json');
    const result = EventEnvelopeSchema.safeParse(adapterOutput);
    expect(result.success).toBe(true);
    if (result.success) {
      expect(result.data._fixture_meta).toBeDefined();
    }
  });
});
```

Every JSON fixture includes a `_fixture_meta` top-level field:

```json

    "source_schema": "SchemaName",
    "created_date": "YYYY-MM-DD",
    "created_by_phase": NNN,
    "notes": "Description of what this fixture represents"
  }
}
```

This field passes through Zod `.passthrough()` schemas without interference and serves as

1. Create the fixture file in the appropriate flow directory
2. Use realistic, domain-accurate values (no lorem ipsum or stub data)
3. Include the `_fixture_meta` field

6. Commit the fixture alongside its contract test file

## Lifecycle

- **Created:** when contract tests are first implemented
- **Updated:** when schema changes trigger the freshness policy
- **Deleted:** never (old fixtures become regression tests)
- **Versioned:** committed to git alongside contract tests
```

## 2. EventDispatcher Spec Compliance Audit

*Satisfies: INTTEST-07 (spec compliance audit step with bypass detection patterns)*

### 2.1 Audit Purpose

The [EventDispatcher Specification](../shared-eventdispatcher-specification/eventdispatcher-spec.md) establishes a single `notify 8.2` inotify instance with `tokio::broadcast` multi-subscriber fan-out as the canonical file-watching architecture. All 6 consumers (ChipsetRouter, SessionObserver, DashboardNotifier, SiliconMonitor, ConsoleWatcher, StagingWatcher) must use this shared EventDispatcher rather than independent file-watching mechanisms.

The compliance audit detects components that **bypass** the EventDispatcher by using direct filesystem watching APIs. Bypass patterns create the exact problems the EventDispatcher was designed to solve:

- Duplicated inotify watch allocations (exhausting the system limit)
- Inconsistent debouncing strategies
- No shared event classification or envelope format
- No centralized backpressure or failure handling

### 2.2 What the Audit Detects

Four categories of bypass patterns, each representing a different mechanism for observing filesystem changes outside the EventDispatcher:

| Category | Pattern | Why It's a Bypass |
|----------|---------|-------------------|
| **Direct `fs.watch`** | TypeScript code calling `fs.watch()` or `fs.watchFile()` | Creates an independent inotify watch outside the EventDispatcher's single-instance architecture |
| **Polling patterns** | `setInterval` combined with `fs.stat` or `fs.statSync` to detect file changes | Duplicates the EventDispatcher's event detection with a less efficient polling approach (the pattern [Phase 232 migration plan](../shared-eventdispatcher-specification/migration-plan.md) addresses for `src/dashboard/refresh.ts`) |
| **Direct `notify` crate** | Rust code using `use notify::` outside the EventDispatcher module | Creates a separate notify watcher instance, consuming additional inotify watches and bypassing the shared broadcast channel |
| **Unauthorized watching libraries** | Third-party file-watching libraries (chokidar, watchpack, etc.) | Introduces a parallel file-watching stack that duplicates the EventDispatcher's responsibilities |

### 2.3 Audit Grep Patterns

Executable grep/ripgrep commands for each detection category. These commands are designed to run from the project root directory without any runtime infrastructure -- pure static analysis.

#### Category 1: Direct `fs.watch` / `fs.watchFile` Usage

```bash
# TypeScript: Direct fs.watch or fs.watchFile usage
# Excludes: EventDispatcher implementation directories, test files
rg "fs\.watch\(|fs\.watchFile\(" \
  --type ts \
  --glob '!**/eventdispatcher/**' \
  --glob '!**/event_dispatcher/**' \
  --glob '!**/*.test.ts' \
  --glob '!**/*.spec.ts'
```

**Expected findings:** 0 (no production TypeScript code should use direct `fs.watch`)

#### Category 2: Polling Patterns

```bash
# TypeScript: setInterval combined with fs.stat or fs.statSync
# Detects the polling anti-pattern the EventDispatcher replaces
rg "setInterval.*fs\.(stat|statSync)|fs\.(stat|statSync).*setInterval" \
  --type ts \
  --glob '!**/eventdispatcher/**' \
  --glob '!**/event_dispatcher/**' \
  --glob '!**/*.test.ts' \
  --glob '!**/*.spec.ts'
```

```bash
# Additional: setInterval with file-change detection intent
# Catches patterns where setInterval and fs.stat are on nearby lines
rg "setInterval" --type ts -l | while read f; do
  rg "fs\.(stat|statSync|readFileSync)" "$f" --quiet && echo "REVIEW: $f"
done
```

**Expected findings:** 1 known exception (`src/dashboard/refresh.ts` -- see Section 2.4)

#### Category 3: Direct `notify` Crate Usage in Rust

```bash
# Rust: Direct notify crate imports outside EventDispatcher module
# The EventDispatcher implementation itself (watcher.rs) is expected to use notify
rg "use notify::" \
  --type rust \
  --glob '!**/watcher.rs' \
  --glob '!**/event_dispatcher.rs' \
  --glob '!**/eventdispatcher/**'
```

**Expected findings:** 0 (only `src-tauri/src/watcher.rs` should import `notify` directly, and it is excluded)

#### Category 4: Unauthorized File-Watching Libraries

```bash
# TypeScript: chokidar (common Node.js file watcher)
rg "require\(['\"]chokidar|from ['\"]chokidar" --type ts

# TypeScript: watchpack (webpack's file watcher)
rg "require\(['\"]watchpack|from ['\"]watchpack" --type ts

# TypeScript: nsfw (native filesystem watcher)
rg "require\(['\"]nsfw|from ['\"]nsfw" --type ts

# TypeScript: @parcel/watcher
rg "require\(['\"]@parcel/watcher|from ['\"]@parcel/watcher" --type ts

# Rust: hotwatch crate (alternative to notify)
rg "use hotwatch::" --type rust

# Rust: inotify crate directly (bypassing notify abstraction)
rg "use inotify::" --type rust \
  --glob '!**/watcher.rs' \
  --glob '!**/event_dispatcher.rs'
```

**Expected findings:** 0 (no third-party file-watching libraries should be in use)

### 2.4 Known Exceptions

These are components that the audit will find but which are expected findings -- not compliance violations. Each has a documented rationale and a planned resolution path.

#### Exception 1: `src-tauri/src/watcher.rs`

| Field | Value |
|-------|-------|
| **File** | `src-tauri/src/watcher.rs` |
| **Pattern found** | `use notify::` (direct notify crate usage) |
| **Why expected** | This file IS the EventDispatcher implementation (Rust side). It is the single notify instance that all subscribers share. Using `notify` directly here is correct -- it is not a bypass. |
| **Resolution** | Permanent exception. This file will always use `notify` directly. |
| **Audit handling** | Excluded from Category 3 grep via `--glob '!**/watcher.rs'` |

#### Exception 2: `src/dashboard/refresh.ts`

| Field | Value |
|-------|-------|
| **File** | `src/dashboard/refresh.ts` |
| **Pattern found** | `setInterval` + file-stat-based polling for auto-refresh |
| **Why expected** | Currently uses timer-based polling for dashboard auto-refresh. The [Phase 232 migration plan](../shared-eventdispatcher-specification/migration-plan.md) Section 3 documents its transition to a DashboardNotifier EventDispatcher subscriber. |
| **Resolution** | Will be resolved when Migration 2 (Dashboard Auto-Refresh) is implemented in a future implementation milestone. After migration, the `setInterval` pattern is replaced by event-driven refresh via the EventDispatcher. |
| **Audit handling** | Listed as a known expected finding in Category 2. The audit reports it but classifies it as "expected." |

#### Exception 3: Test Files

| Field | Value |
|-------|-------|
| **Files** | `*.test.ts`, `*.spec.ts` |
| **Pattern found** | May use `fs.watch`, `setInterval + fs.stat`, or other file-watching patterns in test utilities |
| **Why expected** | Test files may need to observe filesystem changes as part of test infrastructure (e.g., waiting for a file to be written before asserting). This is test tooling, not production code. |
| **Resolution** | Permanent exception. Test infrastructure is not subject to the EventDispatcher architectural constraint. |
| **Audit handling** | Excluded from all grep patterns via `--glob '!**/*.test.ts'` and `--glob '!**/*.spec.ts'` |

### 2.6 When to Run the Audit

| Trigger | Rationale |
|---------|-----------|
| **Milestone verification** (`/gsd:verify-work`) | Ensures no bypass patterns were introduced during the milestone's implementation |
| **New EventDispatcher subscriber added** | Verify the new subscriber uses the EventDispatcher and does not reintroduce direct watching |
| **Refactoring touches filesystem-watching code** | Any change to watcher.rs, refresh.ts, or files importing `fs.watch` should trigger a re-audit |
| **Pre-merge check for PRs touching `src-tauri/src/`** | Rust backend changes could introduce direct `notify` usage outside the EventDispatcher |
| **Pre-merge check for PRs adding `fs.watch` imports** | TypeScript changes adding filesystem watching should be caught before merge |
| **After completing EventDispatcher Migration 2** | Verify `src/dashboard/refresh.ts` no longer uses polling, allowing its removal from the exceptions list |

**Automation path:** The grep patterns in Section 2.3 can be composed into a shell script or CI step. A non-zero exit code from any unexpected finding fails the check. Example:

```bash
#!/usr/bin/env bash
# eventdispatcher-audit.sh -- Run from project root
set -euo pipefail

UNEXPECTED=0

echo "## EventDispatcher Compliance Audit"
echo "Date: $(date -u +%Y-%m-%d)"
echo ""

# Category 1: Direct fs.watch
echo "### Category 1: Direct fs.watch/watchFile usage"
HITS=$(rg "fs\.watch\(|fs\.watchFile\(" \
  --type ts \
  --glob '!**/eventdispatcher/**' \
  --glob '!**/event_dispatcher/**' \
  --glob '!**/*.test.ts' \
  --glob '!**/*.spec.ts' \
  --count-matches 2>/dev/null || echo "0")
if [ "$HITS" = "0" ]; then
  echo "- PASS: 0 findings"
else
  echo "- FINDING: $HITS matches"
  UNEXPECTED=$((UNEXPECTED + 1))
fi
echo ""

# Category 2: Polling patterns
echo "### Category 2: Polling patterns"
HITS=$(rg "setInterval" --type ts -l \
  --glob '!**/*.test.ts' \
  --glob '!**/*.spec.ts' \
  --glob '!**/eventdispatcher/**' 2>/dev/null || echo "")
for f in $HITS; do
  if echo "$f" | grep -q "dashboard/refresh"; then
    echo "- EXPECTED: $f (pending Migration 2)"
  else
    if rg "fs\.(stat|statSync|readFileSync)" "$f" --quiet 2>/dev/null; then
      echo "- FINDING: $f -- setInterval + fs access"
      UNEXPECTED=$((UNEXPECTED + 1))
    fi
  fi
done
[ -z "$HITS" ] && echo "- PASS: 0 findings"
echo ""

# Category 3: Direct notify crate
echo "### Category 3: Direct notify crate usage"
HITS=$(rg "use notify::" --type rust \
  --glob '!**/watcher.rs' \
  --glob '!**/event_dispatcher.rs' \
  --glob '!**/eventdispatcher/**' \
  --count-matches 2>/dev/null || echo "0")
if [ "$HITS" = "0" ]; then
  echo "- PASS: 0 findings"
else
  echo "- FINDING: $HITS matches"
  UNEXPECTED=$((UNEXPECTED + 1))
fi
echo ""

# Category 4: Unauthorized libraries
echo "### Category 4: Unauthorized file-watching libraries"
LIBS=("chokidar" "watchpack" "nsfw" "@parcel/watcher")
LIB_HITS=0
for lib in "${LIBS[@]}"; do
  COUNT=$(rg "require\(['\"]${lib}|from ['\"]${lib}" --type ts --count-matches 2>/dev/null || echo "0")
  if [ "$COUNT" != "0" ]; then
    echo "- FINDING: $lib ($COUNT matches)"
    LIB_HITS=$((LIB_HITS + 1))
  fi
done
RUST_HITS=$(rg "use hotwatch::|use inotify::" --type rust \
  --glob '!**/watcher.rs' \
  --count-matches 2>/dev/null || echo "0")
if [ "$RUST_HITS" != "0" ]; then
  echo "- FINDING: hotwatch/inotify crate ($RUST_HITS matches)"
  LIB_HITS=$((LIB_HITS + 1))
fi
if [ "$LIB_HITS" = "0" ]; then
  echo "- PASS: 0 findings"
fi
UNEXPECTED=$((UNEXPECTED + LIB_HITS))
echo ""

# Summary
echo "### Summary"
echo "Unexpected findings: $UNEXPECTED"
if [ "$UNEXPECTED" -eq 0 ]; then
  echo "Compliance: PASS"
  exit 0
else
  echo "Compliance: FAIL"
  exit 1
fi
```

### 2.7 Audit Evolution

The audit's known exceptions list is a living document that shrinks as the EventDispatcher adoption progresses:

| Migration Event | Exception Change | Audit Impact |
|-----------------|-----------------|--------------|
| Migration 2 complete (Dashboard Auto-Refresh) | Remove `src/dashboard/refresh.ts` from Category 2 exceptions | Category 2 expected findings drops from 1 to 0 |
| All 6 subscribers implemented | No new exceptions expected (all use EventDispatcher by design) | Categories 1-4 should all show PASS with 0 findings |
| New subsystem adds file watching | New exception if legitimately outside EventDispatcher scope, or FAIL if bypass | Audit catches the addition immediately |

**Goal state:** After full EventDispatcher adoption across all 6 subscribers, the audit should report:

- Category 1: PASS (0 findings)
- Category 2: PASS (0 findings) -- dashboard polling migration complete
- Category 3: PASS (0 findings) -- only watcher.rs uses notify (excluded)
- Category 4: PASS (0 findings)
- Permanent exceptions: watcher.rs (implementation), test files (test infrastructure)
- Compliance: PASS

The audit serves as a ratchet -- once a bypass pattern is removed, the audit prevents its reintroduction. Each resolved exception permanently tightens the compliance baseline.
