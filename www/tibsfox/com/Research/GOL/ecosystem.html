Go’s greatest ecosystem strength is that most of the tools you need are built in. go build, go test, go fmt, go vet, go doc, go mod, pprof — all ship with the compiler. No npm, no pip, no Maven needed for the basics.

Cross-compilation

# Build for Linux on any OS
GOOS=linux GOARCH=amd64 go build -o myapp-linux

# Build for ARM64
GOOS=linux GOARCH=arm64 go build -o myapp-arm64

# Build for Windows
GOOS=windows GOARCH=amd64 go build -o myapp.exe

# Static binary (no C dependencies)
CGO_ENABLED=0 go build -o myapp

net/http server

package main

import (
    "encoding/json"
    "log"
    "net/http"
)

type User struct {
    ID   int    `json:"id"`
    Name string `json:"name"`
}

func main() {
    http.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
        id := r.PathValue("id") // Go 1.22 pattern routing
        user := User{ID: 1, Name: "Foxy"}
        w.Header().Set("Content-Type", "application/json")
        json.NewEncoder(w).Encode(user)
    })
    log.Fatal(http.ListenAndServe(":8080", nil))
}

Why Go won cloud-native: single static binary (trivial Docker images), fast startup, low memory per goroutine, excellent networking stdlib, trivial cross-compilation, and a culture of simplicity that makes large-team codebases maintainable.

Containerization

# Dockerfile for a Go app
FROM golang:1.24-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /myapp

FROM scratch
COPY --from=builder /myapp /myapp
ENTRYPOINT ["/myapp"]
# Result: ~10 MB image with no OS, no runtime, just the binary
