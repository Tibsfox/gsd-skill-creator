“Don’t communicate by sharing memory; share memory by communicating.” — Rob Pike, Go Proverbs. Go’s concurrency model descends from Hoare’s CSP (1978) through Pike’s own Newsqueak, Limbo, and Alef at Bell Labs.

Goroutines

A goroutine is a lightweight thread managed by the Go runtime. Initial stack: ~4 KB (grows/shrinks dynamically). Cost: trivial — millions of goroutines in a single process. Start one with the go keyword:

go func() {
    fmt.Println("hello from goroutine")
}()

// With sync.WaitGroup for coordination
var wg sync.WaitGroup
for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(n int) {
        defer wg.Done()
        fmt.Println(n)
    }(i)
}
wg.Wait()

The GMP Scheduler

Go uses M:N scheduling: M goroutines multiplexed onto N OS threads.

  • G = goroutine (the work unit)
  • M = machine (OS thread)
  • P = processor (logical CPU, sized by GOMAXPROCS)

Each P has a local run queue of Gs. Work-stealing: idle Ps steal Gs from busy Ps. Preemption is asynchronous since Go 1.14 (signal-based). Syscall handling: the G’s M is handed off so another M can run Gs on that P.

Channels

// Unbuffered: sender blocks until receiver is ready (synchronous)
ch := make(chan string)
go func() { ch <- "hello" }()
msg := <-ch

// Buffered: sender blocks only when buffer is full
ch := make(chan int, 10)
ch <- 42

// Close: signals no more values
close(ch)

// Range over channel: receives until closed
for v := range ch { fmt.Println(v) }

// Directional channels in function signatures
func producer(out chan<- int) { out <- 42 }
func consumer(in <-chan int) { fmt.Println(<-in) }

    if errors.Is(err, context.DeadlineExceeded) {
        log.Println("request timed out")
    }

Concurrency Patterns

Pipeline

func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums { out <- n }
        close(out)
    }()
    return out
}

func sq(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in { out <- n * n }
        close(out)
    }()
    return out
}

// Usage: gen |> sq |> print
for v := range sq(gen(1, 2, 3)) {
    fmt.Println(v) // 1, 4, 9
}

Worker pool

func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        results <- j * 2
    }
}

jobs := make(chan int, 100)
results := make(chan int, 100)
for w := 0; w < 5; w++ { go worker(w, jobs, results) }
for j := 0; j < 50; j++ { jobs <- j }
close(jobs)
for r := 0; r < 50; r++ { fmt.Println(<-results) }

errgroup

import "golang.org/x/sync/errgroup"

g, ctx := errgroup.WithContext(context.Background())
g.Go(func() error { return fetchUsers(ctx) })
g.Go(func() error { return fetchOrders(ctx) })
if err := g.Wait(); err != nil {
    log.Fatal(err) // first error cancels the group via ctx
}

Race Detector

go test -race and go run -race enable the race detector (based on ThreadSanitizer). It catches data races at runtime with ~10× slowdown. Always run tests with -race. Go does not prevent data races at compile time (unlike Rust) — the race detector is the safety net.

Common Pitfalls

  • Goroutine leaks — forgotten goroutines blocking on channels that nobody reads from
  • Loop variable capture — goroutines capturing a loop variable by reference (fixed in Go 1.22)
  • Sending to a closed channel — panics
  • Nil channel — send/receive blocks forever (useful in select for disabling a case)
  • Deadlock — all goroutines asleep; Go runtime detects and crashes
