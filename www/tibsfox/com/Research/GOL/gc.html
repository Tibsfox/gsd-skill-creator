GC & Runtime

Concurrent Tri-Color Mark-and-Sweep, Sub-Millisecond Pauses, GOGC, GOMEMLIMIT, pprof, The “No Knobs” Philosophy

Go’s GC is radically simpler than the JVM’s: one collector, one tuning knob (GOGC), and sub-millisecond pauses. The trade-off: lower throughput than Java’s Parallel GC for batch workloads. The bet: most Go programs are network services where latency matters more than throughput.

The Collector

Go uses a concurrent, non-generational, non-compacting, tri-color mark-and-sweep collector. It runs concurrently with the application, with two very short STW pauses (typically <1 ms each):

  1. Mark Setup (STW) — enable write barrier, scan stacks for roots
  2. Mark (concurrent) — trace from roots, mark reachable objects
  3. Mark Termination (STW) — drain remaining work, disable write barrier
  4. Sweep (concurrent) — reclaim unmarked memory, lazy per-span

GC Evolution

Go 1.0 Stop-the-world collector. Pauses could be hundreds of milliseconds.
Go 1.3 Precise GC (can identify all pointers). Previously conservative.
Go 1.5 Concurrent GC — tri-color mark-and-sweep, write barrier, <10 ms pauses. Rick Hudson’s breakthrough.
Go 1.8 Sub-millisecond pauses typical. Eliminated most remaining STW work.
Go 1.19 GOMEMLIMIT — soft memory limit, reducing OOM risk in containers.

Tuning: GOGC and GOMEMLIMIT

# GOGC: target heap growth ratio (default 100 = 100% growth before GC)
GOGC=200 ./myapp     # less frequent GC, higher throughput, more memory
GOGC=50  ./myapp     # more frequent GC, lower memory, more CPU on GC
GOGC=off ./myapp     # disable GC entirely (for short-lived processes)

# GOMEMLIMIT (Go 1.19+): soft memory ceiling
GOMEMLIMIT=1GiB ./myapp  # GC tries to stay under 1 GiB
# Combine both:
GOGC=100 GOMEMLIMIT=512MiB ./myapp

The “no knobs” philosophy: Go deliberately avoids the JVM’s hundreds of GC flags. Rick Hudson: “If the GC needs a lot of tuning, the GC is broken.” GOGC controls the CPU/memory trade-off. GOMEMLIMIT prevents OOM in bounded environments. That’s it.

Memory Allocator

  • Size classes — objects are allocated into spans of fixed size classes (8 bytes to 32 KB)
  • mcache — per-P (per-processor) allocation cache, lock-free
  • mcentral — shared cache of spans per size class
  • mheap — the global heap, backed by OS memory
  • Large objects (>32 KB) allocated directly from mheap
  • No compaction — fragmentation is managed by size classes

Profiling with pprof

Go has the best built-in profiling of any language. runtime/pprof for CLI apps, net/http/pprof for live servers:

import _ "net/http/pprof"
// then: go tool pprof http://localhost:6060/debug/pprof/heap
// or:   go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

The Execution Tracer

runtime/trace and go tool trace give a microsecond-resolution view of goroutine scheduling, GC events, syscalls, and network I/O. Far more detailed than profiles — shows exactly what happened, when, on which processor.

Escape Analysis

The Go compiler decides at compile time whether a value can stay on the stack (cheap) or must escape to the heap (requires GC). go build -gcflags="-m" shows escape analysis decisions. Reducing heap escapes is the #1 performance optimization in Go.

Memory Model

Go’s memory model (go.dev/ref/mem) defines happens-before relationships for concurrent access. Goroutine creation, channel operations, mutex lock/unlock, and sync.Once all establish happens-before edges. Reads/writes of shared variables without synchronization are data races (undefined behavior).
