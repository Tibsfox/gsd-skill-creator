OpenMP — Shared-Memory Parallelism

OpenMP was created in 1997 by a committee dominated by Fortran users (KAI, SGI, Cray, IBM, DOE labs). The original spec was essentially a standardization of the !$OMP PARALLEL DO directives that SGI and KAI had been shipping for years. C and C++ bindings came second.

program omp_example
  use omp_lib
  implicit none
  integer, parameter :: n = 1000000
  real(8) :: a(n), b(n), c(n)
  real(8) :: total
  integer :: i

  ! Initialize in parallel
  !$omp parallel do
  do i = 1, n
    a(i) = real(i, 8)
    b(i) = real(i, 8) * 2.0_8
  end do
  !$omp end parallel do

  ! Compute and reduce in parallel
  total = 0.0_8
  !$omp parallel do reduction(+:total)
  do i = 1, n
    c(i) = a(i) + b(i)
    total = total + c(i)
  end do
  !$omp end parallel do

  print *, 'Total =', total
end program omp_example
# Compile with OpenMP
gfortran -O3 -fopenmp omp_example.f90 -o omp_example
ifx     -O3 -qopenmp omp_example.f90 -o omp_example

# Run with 8 threads
OMP_NUM_THREADS=8 ./omp_example

MPI — Distributed-Memory Parallelism

MPI (Message Passing Interface) is the de facto standard for distributed-memory parallelism in HPC. Fortran is one of the two reference languages (the other being C). Every supercomputer ships an MPI implementation.

program mpi_hello
  use mpi_f08          ! modern Fortran 2008+ MPI bindings
  implicit none
  integer :: ierr, rank, size

  call MPI_Init(ierr)
  call MPI_Comm_rank(MPI_COMM_WORLD, rank, ierr)
  call MPI_Comm_size(MPI_COMM_WORLD, size, ierr)

  print *, 'Hello from rank', rank, 'of', size

  call MPI_Finalize(ierr)
end program mpi_hello
# Compile and run
mpifort -O3 mpi_hello.f90 -o mpi_hello
mpirun -np 4 ./mpi_hello

Coarrays — PGAS in the Language Standard

Coarrays are part of the Fortran language. They were designed by Robert Numrich and John Reid as a descendant of Cray F-- and the CAF research project at Rice University. The idea predated MPI’s one-sided communication and is arguably cleaner than MPI-3 RMA.

A coarray is an array with an extra codimension distinguishing images (processes). Writing A(:)[2] = B(:) means “copy B on this image to A on image 2” — a one-sided put expressed in standard syntax.

program coarray_example
  implicit none
  integer :: me, np
  real :: data(100)[*]    ! coarray: each image has 100 reals
  integer :: i

  me = this_image()
  np = num_images()

  ! Each image initializes its own data
  do i = 1, 100
    data(i) = real(me * 100 + i)
  end do

  sync all   ! barrier across all images

  ! Image 1 reads from image 2
  if (me == 1 .and. np >= 2) then
    print *, 'Image 1 reading from image 2:'
    print *, data(1:5)[2]   ! one-sided get from image 2
  end if

  sync all
end program coarray_example
# Compile and run with Intel
ifx -O3 -coarray=shared coarray_example.f90 -o coarray_example
./coarray_example                              # auto-detects images

# Compile and run with GFortran via OpenCoarrays
caf -O3 coarray_example.f90 -o coarray_example
cafrun -np 4 ./coarray_example

DO CONCURRENT — The Universal Parallel Loop

DO CONCURRENT, introduced in Fortran 2008 and extended in 2018, is the forward-looking way to express parallelism in pure standard Fortran. It tells the compiler “this loop has no iteration dependencies and no ordering constraints — run it however you like.”

program concurrent_example
  implicit none
  integer, parameter :: n = 1000, m = 1000
  real(8) :: a(n,m), b(n,m), c(n,m)
  real(8) :: total
  integer :: i, j

  call random_number(a)
  call random_number(b)

  ! The genius: same source compiles to serial, SIMD,
  ! OpenMP, or GPU depending on compiler flags
  total = 0.0_8
  do concurrent (i = 1:n, j = 1:m) reduce(+:total)
    c(i,j) = a(i,j) * b(i,j)
    total = total + c(i,j)
  end do

  print *, 'Total =', total
end program concurrent_example
Compiler FlagTarget

The vision realized: The same source code — without directives, without annotations, without preprocessor macros — compiles to serial, vectorized, OpenMP, or GPU execution depending on compiler flags. DO CONCURRENT is as close as any language has come to the holy grail of “write once, run on any parallel hardware,” and Fortran got there first.

OpenACC — Directive-Based GPU Offload

OpenACC was developed by PGI, Cray, and NVIDIA in 2011-2012 as a directive-based GPU programming model. !$ACC PARALLEL LOOP marks a loop for GPU execution; the compiler handles data movement through !$ACC DATA regions. OpenACC was the default GPU path for Fortran climate codes through the 2010s.

program openacc_example
  implicit none
  integer, parameter :: n = 1000000
  real(8) :: a(n), b(n), c(n)
  integer :: i

  call random_number(a)
  call random_number(b)

  !$acc data copyin(a, b) copyout(c)
  !$acc parallel loop
  do i = 1, n
    c(i) = sqrt(a(i)*a(i) + b(i)*b(i))
  end do
  !$acc end parallel loop
  !$acc end data

  print *, 'c(1) =', c(1)
end program openacc_example
# Compile with nvfortran
nvfortran -O3 -acc -gpu=cc80 openacc_example.f90 -o openacc_example

CUDA Fortran — Maximum GPU Control

CUDA Fortran is NVIDIA’s proprietary extension exposing CUDA kernels, device memory, and streams as Fortran language features. The only non-C way to write CUDA code. Popular in weather and chemistry communities deeply invested in NVIDIA hardware.

module my_kernels
  contains
  attributes(global) subroutine vec_add(a, b, c, n)
    real, device :: a(*), b(*), c(*)
    integer, value :: n
    integer :: i
    i = (blockIdx%x - 1) * blockDim%x + threadIdx%x
    if (i <= n) c(i) = a(i) + b(i)
  end subroutine vec_add
end module my_kernels

program cuda_example
  use cudafor
  use my_kernels
  implicit none
  integer, parameter :: n = 1000000
  real, allocatable, device :: d_a(:), d_b(:), d_c(:)
  real :: h_a(n), h_b(n), h_c(n)

  call random_number(h_a)
  call random_number(h_b)

  allocate(d_a(n), d_b(n), d_c(n))
  d_a = h_a; d_b = h_b           ! host to device

  ! Launch kernel: 1024-thread blocks
  call vec_add<<<(n+1023)/1024, 1024>>>(d_a, d_b, d_c, n)

  h_c = d_c                       ! device to host
  print *, 'c(1) =', h_c(1)
end program cuda_example
# Build with nvfortran
nvfortran -O3 -cuda cuda_example.cuf -o cuda_example

The Vectorization Story

Auto-vectorization — the compiler turning a scalar loop into SIMD — is where Fortran’s array semantics pay off most visibly. A loop like:

do i = 1, n
  c(i) = a(i) + b(i) * s
end do

compiles, on any modern x86-64 target, into AVX-512 FMAs processing 8 doubles at a time. The compiler can prove the loop is vectorizable because Fortran’s aliasing rules guarantee a, b, and c don’t overlap. The equivalent C loop requires restrict on all three pointers, and even then the compiler has to do more work.

On AVX-512 targets, well-written Fortran kernels routinely achieve 80-90% of peak single-core throughput on BLAS-like workloads. On ARM SVE (Fugaku, Grace), Fortran compilers target scalable vectors natively.
