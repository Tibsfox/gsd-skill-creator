Arrays — The Killer Feature

! c = a + b ! column-major, whole-array, first-class

! Arrays are why Fortran exists
In 1957 it was about writing math formulas. In 2026 it’s about auto-vectorization, cache behavior, aliasing rules, and BLAS integration. The language-level array semantics are why Fortran code outperforms hand-written C loops.

Array Declarations

! Basic declarations — two styles
real :: a(100)                       ! classic
real, dimension(100) :: b            ! dimension attribute

! Multi-dimensional
real :: matrix(10, 20)               ! 10 rows, 20 columns
real :: cube(3, 4, 5)                ! three dimensions

! Lower bound other than 1
real :: offset(0:99)                 ! indices 0 through 99
real :: physical(-10:10, -10:10)    ! centered grid

! With kind
real(kind=8) :: big(1000)            ! double precision
integer(kind=8) :: counters(100)    ! 64-bit integers

! Parameter arrays (constants)
integer, parameter :: N = 1000
real, parameter :: primes(4) = [2.0, 3.0, 5.0, 7.0]

Whole-Array Operations

The killer feature: Fortran treats arrays as first-class values. Arithmetic and functions operate element-wise on entire arrays. You write c = a + b instead of a loop. The compiler produces vectorized SIMD code automatically.

real :: a(1000), b(1000), c(1000)
real :: s

! Initialize
a = 0.0                              ! scalar broadcast
b = 1.0                              ! scalar broadcast

! Element-wise arithmetic on whole arrays
c = a + b                            ! c(i) = a(i) + b(i) for all i
c = a * b + 2.0 * sin(a)             ! elementwise compound expression
c = sqrt(a**2 + b**2)                ! intrinsics apply elementwise

! Reduction
s = sum(c)
s = sum(c * c)                       ! dot product via reduction

Array Slicing

real :: a(100), m(10, 20)

! 1-D slices — syntax: start:end:stride
b = a(1:10)                          ! first 10 elements
b = a(91:100)                        ! last 10 elements
b = a(::2)                           ! every other element (1, 3, 5, ...)
b = a(1:50:3)                        ! stride of 3
b = a(100:1:-1)                      ! reversed

! 2-D slices
row = m(5, :)                        ! row 5 (all columns)
col = m(:, 3)                        ! column 3 (all rows)
sub = m(1:5, 1:5)                    ! 5x5 upper-left corner
every_other_row = m(::2, :)          ! alternating rows

! Assignment to slices
m(1:5, :) = 0.0                      ! zero out first 5 rows
m(:, 1) = m(:, 2) + m(:, 3)          ! column arithmetic

Array Constructors

integer :: v(5)
v = [1, 2, 3, 4, 5]                  ! modern (Fortran 2003+)
v = (/ 1, 2, 3, 4, 5 /)              ! older syntax

! Implied DO
real :: x(10)
x = [(real(i)**2, i = 1, 10)]        ! 1, 4, 9, 16, ..., 100

! Type-specified
y = [real :: 1, 2, 3]

! Range using step
integer :: evens(5)
evens = [(i, i = 2, 10, 2)]          ! 2, 4, 6, 8, 10

! Concatenation
real :: c(20)
c = [a(1:10), b(1:10)]

Allocatable Arrays

Dynamic arrays allocated at runtime. Automatically deallocated when out of scope.

real, allocatable :: data(:)
integer :: n, ierr

read *, n
allocate(data(n), stat=ierr)
if (ierr /= 0) stop 'allocation failed'

data = 0.0
! ... use data ...
! deallocate happens automatically at scope exit (Fortran 2003+)

! Automatic reallocation on assignment (Fortran 2003+)
real, allocatable :: x(:)
x = [1.0, 2.0, 3.0]                  ! allocates size 3
x = [x, 4.0]                          ! reallocates to size 4
x = sin([(real(i)*0.1, i=1,100)])    ! reallocates to size 100

! Multi-dimensional allocatable
real, allocatable :: matrix(:,:)
allocate(matrix(rows, cols))
matrix = 0.0

Assumed-Shape Arguments

The modern idiom for array arguments: the callee writes a(:) and the compiler passes a descriptor with size and bounds information.

module stats
  implicit none
contains

  pure function mean(a) result(m)
    real, intent(in) :: a(:)           ! assumed shape
    real :: m
    m = sum(a) / size(a)
  end function mean

  pure function variance(a) result(v)
    real, intent(in) :: a(:)
    real :: v, avg
    avg = mean(a)
    v = sum((a - avg)**2) / size(a)
  end function variance

  subroutine normalize_rows(m)
    real, intent(inout) :: m(:,:)
    integer :: i
    do i = 1, size(m, 1)
      m(i, :) = m(i, :) / sum(m(i, :))
    end do
  end subroutine normalize_rows

end module stats

Important: Assumed-shape arguments require an explicit interface — the caller must know the signature. Module procedures provide this automatically; external procedures need an interface block. This is the #1 reason to organize code into modules.

Column-Major Storage

Fortran stores arrays column-major: the first index varies fastest in memory. This is opposite to C’s row-major storage.

! A 3x4 matrix in Fortran
real :: m(3, 4)

! Memory layout:
! m(1,1) m(2,1) m(3,1)  m(1,2) m(2,2) m(3,2)  m(1,3) m(2,3) m(3,3)  m(1,4) m(2,4) m(3,4)
!    ---column 1---        ---column 2---        ---column 3---        ---column 4---

Performance implication: Loops should vary the first index fastest for best cache behavior:

! GOOD: column-major friendly (fast)
do j = 1, n
  do i = 1, n
    m(i, j) = m(i, j) + 1.0
  end do
end do

! BAD: row-major order (cache unfriendly, slow)
do i = 1, n
  do j = 1, n
    m(i, j) = m(i, j) + 1.0
  end do
end do

WHERE — Array Conditional Assignment

real :: a(100), b(100)

! Conditional element-wise operation
where (a > 0.0)
  b = sqrt(a)
elsewhere
  b = 0.0
end where

! Single-line WHERE
where (a < 0.0) a = 0.0

Why Fortran Arrays Beat NumPy

PropertyFortranNumPy
Compile-time optimizationYes — whole pipeline fusedPer-operation kernel dispatch
Bounds analysisStatic where possibleRuntime checks
Aliasing rulesNon-aliasing by defaultMust assume aliasing
VectorizationCompiler-level (AVX-512, SVE)NumPy ops call BLAS (which is Fortran)
Memory overheadRaw arrays, no Python objectsPyObject wrappers
Parallel executionDO CONCURRENT → GPU, OpenMPLimited (GIL, threading issues)

The performance story: NumPy users benefit from Fortran indirectly. When you write A @ B in NumPy, you’re calling dgemm from BLAS, which is Fortran (or hand-tuned assembly mimicking Fortran semantics). Writing native Fortran skips the Python overhead entirely.
