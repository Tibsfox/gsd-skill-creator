! The polyglot HPC stack
Every numerical Python package eventually calls Fortran. NumPy, SciPy, scikit-learn — all wrap LAPACK and BLAS, both written in Fortran. Python is the front-end. Fortran is the engine.

Notice the pattern: every language eventually calls BLAS/LAPACK, which are written in Fortran. This is not coincidence — it’s the result of 40+ years of optimization, vendor tuning (Intel MKL, AMD AOCL, OpenBLAS), and platform support that no replacement has matched.

Calling Fortran from Python (f2py)

f2py ships with NumPy. It generates Python wrappers from Fortran source code. This is how SciPy wraps LAPACK; it’s how nearly every numerical Python package gets its speed.

! sumsq.f90
subroutine sumsq(n, total)
  integer, intent(in) :: n
  integer, intent(out) :: total
  integer :: i
  total = 0
  do i = 1, n
    total = total + i*i
  end do
end subroutine sumsq
# Build a Python module from Fortran
$ python -m numpy.f2py -c sumsq.f90 -m sumsq

# Use it from Python
>>> import sumsq
>>> sumsq.sumsq(100)
338350

Calling C from Fortran (ISO_C_BINDING)

! Wrap a C function in Fortran
module my_c_wrapper
  use, intrinsic :: iso_c_binding
  implicit none

  interface
    function c_strlen(s) bind(C, name="strlen")
      import :: c_char, c_size_t
      character(kind=c_char), intent(in) :: s(*)
      integer(c_size_t) :: c_strlen
    end function c_strlen
  end interface
end module my_c_wrapper

program use_c
  use my_c_wrapper
  implicit none
  character(kind=c_char), parameter :: hello(*) = [character(kind=c_char) :: 'h','i','!',c_null_char]
  print *, "C strlen returned:", c_strlen(hello)
end program use_c

Calling Fortran from C

! sumsq.f90 — exported C-callable Fortran
function sumsq(n) bind(C, name="sumsq") result(total)
  use, intrinsic :: iso_c_binding
  integer(c_int), value :: n
  integer(c_int) :: total
  integer :: i
  total = 0
  do i = 1, n
    total = total + i*i
  end do
end function sumsq
/* main.c */
#include <stdio.h>
extern int sumsq(int n);
int main() {
    printf("%d\n", sumsq(100));
}
/* gfortran -c sumsq.f90 && gcc main.c sumsq.o -lgfortran -o main */

Calling Fortran from Julia

# Julia ccall: direct calls to Fortran functions
result = ccall((:sumsq, "libsumsq"), Cint, (Cint,), 100)
println(result)  # 338350

Julia’s ccall works with both C and Fortran libraries. This is why Julia’s LinearAlgebra stdlib calls LAPACK directly — no wrapper layer.

The Escalation Ladder — HPC Edition

RungLanguageWhen to Reach
1Python + NumPy/SciPyPrototype, exploration, glue. The default for any new project.
2NumPy with vectorized operationsWhen pure Python is too slow but the algorithms fit array semantics.
3Cython / NumbaWhen you need to optimize a hot loop without leaving Python.
4Modern FortranWhen the numerical algorithm is the bottleneck and you need maximum performance with minimum cognitive overhead.
5C / C++When you need libraries that don’t exist in Fortran (graphics, networking, OS APIs).
6OpenMP / coarraysWhen one core isn’t enough and you want to stay in the same language.
7MPIWhen you need to scale to a cluster.
8CUDA / HIP / OpenACCWhen you need GPU acceleration. nvfortran is the easiest path from Fortran.
9SIMD intrinsics / assemblyWhen the compiler can’t vectorize your hot loop and you must.

The pragmatic answer: Most scientific code lives at rungs 1-2 (Python + NumPy). The 5-10% that’s performance-critical lives at rung 4 (Fortran). The 0.1% that needs distributed computing lives at rung 7 (MPI on top of Fortran or C++). This stratification is stable, well-understood, and unlikely to change — because Fortran is already the right answer for what it does.
