1. The $250K Engineer Problem

At GTC 2026, Jensen Huang articulated a vision that reframed the economics of AI-assisted engineering. A $500,000 engineer, he argued, should consume $250,000 per year in AI tokens — compute power, inference costs, the raw fuel of machine intelligence. Not as a luxury, but as a baseline expectation. The reasoning: if AI provides a 10x amplification of engineering output, then $250K in tokens per engineer is the cheapest force multiplier in the history of industrial labor.

The Nvidia Math

Nvidia employs roughly 8,000 engineers. At $250K per engineer in annual token spend, that implies approximately $2 billion per year in AI inference costs — just for internal engineering. Huang's framing was not hypothetical. He said "we're trying to" — present tense, active investment. Nvidia is building the future where every knowledge worker has an AI copilot, and the copilot's bill is a line item on the operating budget, not a rounding error.

This is the inflection point. When the world's largest GPU company treats token consumption as a core engineering expense — not an experiment — the market follows.

The 10x Logic: If an engineer produces $1M in annual value, and AI amplifies that to $10M, then $250K in token costs yields a 40:1 return. The question is not whether to spend it — it's whether you can afford not to. But the corollary is equally important: waste in token consumption directly erodes that ratio.

Token Economy Fundamentals

All major LLM providers price by tokens — fragments of text roughly equivalent to 0.75 words or 4 characters. Two prices apply: input tokens (what you send to the model) and output tokens (what the model generates). Output is always more expensive because it requires sequential, autoregressive generation — each token depends on every token before it.

Full window cost is computed as: (context window size) × (input price per token). This is the cost of a single turn that fills the entire context. In a 20-turn conversation where context accumulates, the total input cost is the sum of all turns' context sizes — a geometric growth pattern.

2. The Six Waste Vectors

Token waste falls into six categories. Each has a distinct mechanism, a measurable cost, and — critically — a known solution. The first step in optimization is diagnosis: knowing where tokens go.

Waste Vector 1

Raw Document Ingestion

Typical waste: 60-80% of input tokens

PDFs are the worst offenders. A 10-page technical PDF enters the context as roughly 100,000 tokens — including headers, footers, page numbers, formatting artifacts, embedded font declarations, table-of-contents repetition, and legal boilerplate. The actual informational content is typically 25,000-35,000 tokens. The rest is structural noise that the model must process but gains nothing from.

Even worse: some tools Base64-encode PDFs before passing them to the model, inflating token counts by an additional 33%. A 10-page PDF that should cost $1.50 in input tokens (at Opus pricing) can cost $5.00 or more through inefficient ingestion.

The fix: Convert to Markdown before ingestion. A Rust-based PDF-to-Markdown converter produces clean, structured text at zero token cost. The conversion runs locally, preserves semantic structure, and reduces input tokens by 60-80%.

Waste Vector 2

Conversation Sprawl

Typical waste: geometric growth, 2-5x by turn 15

Every turn in a conversation re-sends the entire history. Turn 1 sends your prompt (1K tokens). Turn 2 sends your prompt + the model's response + your follow-up (4K tokens). Turn 3 sends everything so far + the new exchange (9K tokens). By turn 20, you are sending 200K+ tokens of context per turn — most of which is rehashing earlier exchanges the model has already processed.

The cost is not linear — it is quadratic. Total input tokens for an N-turn conversation grow as O(N²). A 10-turn conversation costs roughly 5x what 10 independent single-turn queries would cost, even if the total information exchanged is identical.

The fix: Summarize and compress conversation history at regular intervals. Use structured context injection instead of raw chat history. Start new conversations when the topic shifts. Use tools like /compact in Claude Code to compress context without losing critical state.

Waste Vector 3

Plugin/Connector Overhead

Typical waste: 5,000-30,000 tokens per session (fixed tax)

Before you type a single character, your AI assistant has already consumed tokens. System prompts, tool definitions, skill descriptions, MCP server schemas, and safety instructions are injected into every turn. In Claude Code with a full skill set and multiple MCP servers, this "plugin tax" can exceed 20,000 tokens — sent on every single turn.

With 57 commands and 34 skills loaded, the tool definition block alone is substantial. Each tool includes a name, description, parameter schema, and usage examples. Multiply by 50+ tools and you have a fixed cost that dwarfs many actual queries.

In a 20-turn session, 20,000 tokens of plugin overhead × 20 turns = 400,000 tokens of overhead — at Opus pricing, that is $6.00 spent before a single useful token is generated.

The fix: Audit your loaded plugins. Disable tools you are not using. Use deferred tool loading (tools loaded on-demand rather than at session start). Design skill descriptions to fit within the 250-character cap. Consider session-specific tool profiles that load only what is needed for the current task.

Waste Vector 4

No Model Mixing

Typical waste: 5-60x cost on routine tasks

Using Opus for everything is like hiring a PhD to sort mail. Commit message generation, file triage, code formatting, data validation — these tasks do not require Opus-level reasoning. They require fast pattern matching that Haiku handles at 60x lower cost with comparable quality.

A commit message takes ~500 tokens of context (the diff) and produces ~50 tokens of output. On Opus: $0.01 input + $0.004 output = $0.014. On Haiku: $0.0001 input + $0.00006 output = $0.00016. That is an 87x cost difference for a task where both models produce acceptable results.

Across 50 commits per day for an active developer: Opus costs $0.70/day, Haiku costs $0.008/day. Annualized: Opus = $175, Haiku = $2. Same quality. Same speed. Ninety-nine percent cheaper.

The fix: Implement a model router. Classify tasks by complexity and route to the cheapest model that meets the quality bar. Use Haiku for triage, classification, and formatting. Use Sonnet for code generation and analysis. Reserve Opus for research synthesis, complex reasoning, and novel problem-solving.

Waste Vector 5

Redundant Context

Typical waste: 30-60% of repeated sessions

When you start a new conversation about the same project, you re-send the same README, the same architecture overview, the same coding conventions. Each session starts from zero, rebuilding context that the previous session already established. Without prompt caching, this is pure waste — paying full price for tokens the model has already seen.

Anthropic's prompt caching (when available) reduces repeated input costs by 90%. But caching has limits: cache entries expire, cache keys are sensitive to exact byte-matching, and not all providers support it. The structural solution is to eliminate the redundancy at the source.

The fix: Use structured context documents (CLAUDE.md, MEMORY.md) that are compact and cached. Build persistent state files that carry forward between sessions. Use embeddings and semantic search to retrieve only the relevant context for each query, rather than sending everything every time.

Waste Vector 6

Over-Retrieval

Typical waste: 50-90% of retrieved content

RAG systems often retrieve full documents when only a paragraph is relevant. A query about a specific function retrieves the entire source file (500 lines) when the function definition (20 lines) would suffice. A question about a configuration option retrieves the full config documentation (10,000 tokens) when the relevant section is 200 tokens.

The retrieval granularity problem compounds with the number of retrieved chunks. If your RAG pipeline retrieves 5 documents averaging 2,000 tokens each, you are injecting 10,000 tokens of context. If the actual relevant content is 500 tokens, you are paying 20x for the privilege of imprecise retrieval.

The fix: Use fine-grained chunking in your vector store. Retrieve at the paragraph or function level, not the document level. Implement re-ranking to filter retrieved chunks before injection. Use metadata filters to narrow retrieval scope before semantic search even begins.

3. Architecture for Efficiency

Token efficiency is not a setting you toggle — it is an architecture you build. Our approach uses three layers, each handling a different class of computation. The principle: the LLM should think, not compute. Everything deterministic, parseable, or transformable happens locally, at zero token cost.

Layer 1: Rust Local Computation

Rust handles everything that is deterministic, CPU-bound, or I/O-intensive. These operations have well-defined inputs and outputs — no ambiguity, no reasoning required. Sending them to an LLM is pure waste.

  • PDF-to-Markdown conversion: A Rust CLI tool parses PDF structure, extracts text, preserves headings and lists, and outputs clean Markdown. Zero tokens consumed. A 10-page PDF that would cost $1.50 in Opus input tokens costs $0.00 when converted locally. Processing time: ~200ms.
  • METAR parsing: Raw METAR strings (KPAE 011856Z 19012KT 10SM FEW045 SCT060 16/08 A3002) are structured data masquerading as free text. A Rust parser converts them to structured JSON in microseconds. Sending METAR to an LLM for interpretation wastes ~500 tokens per station.
  • Git diff compression: A 500-line diff contains mostly unchanged context lines. A Rust compressor extracts only the semantically meaningful changes — additions, deletions, and their immediate context — reducing 500 lines to ~50 tokens of structured change summary.
  • JSON/TOML/YAML validation: Schema validation is a compile-time problem, not a reasoning problem. serde validates structure in nanoseconds. Asking an LLM to validate JSON is like hiring an architect to check if a door is locked.
  • Embedding generation: ONNX Runtime with a local GPU (RTX 4060 Ti) generates embeddings at ~1,000 vectors/second. Each embedding that would cost ~100 API tokens costs zero when generated locally. For our 1,087-page pgvector database, local generation saved approximately 108,700 tokens.
The rtk Pattern: The rtk (Rust Token Kit) pattern wraps common data transformations in a Rust CLI that sits between the data source and the LLM. The pattern: data → rtk transform → compressed representation → LLM. Measured reduction: 60-90% of input tokens for structured data tasks.

TOON: Token-Oriented Object Notation

TOON is a custom serialization format designed specifically for LLM consumption. It eliminates the syntactic overhead of JSON (braces, quotes, colons, commas) while preserving semantic structure. The format uses whitespace-significant indentation and type inference, similar to YAML but with tighter encoding rules.

TOON vs JSON Token Comparison (Same Data) JSON: {"name":"Token Economics","version":"1.0","sections":10,"published":true} → 28 tokens
TOON: name Token Economics | version 1.0 | sections 10 | published true → 19 tokens
Reduction: 32%

Across a large plan document (5,000 tokens in JSON), TOON encoding reduces to ~3,400 tokens — an 18-40% reduction depending on nesting depth and string content. The savings compound: every time the document is re-sent in conversation context, the reduction applies again.

Layer 2: Go MCP Server Infrastructure

Go powers persistent MCP (Model Context Protocol) servers that replace per-query web fetches with structured, cached, pre-filtered data services. Each MCP server is a single binary with fast startup (~50ms), low memory footprint (100-320MB), and built-in caching.

  • mcp-weather: Fetches METAR observations and NWS forecasts, caches responses (5-minute TTL for METAR, 1-hour for forecasts), and returns structured JSON. Replaces WebFetch calls that would include HTML boilerplate, navigation, ads, and cookie banners. Token reduction per query: ~80%.
  • mcp-usgs: Bridges the USGS earthquake API, returning structured event data with magnitude, location, depth, and timestamp. A WebFetch of the USGS earthquake page returns ~15,000 tokens of HTML. The MCP server returns ~1,500 tokens of clean data.
  • mcp-pgvector: Semantic search server backed by PostgreSQL with pgvector. Takes a natural language query, generates embeddings locally, performs cosine similarity search, and returns the top-K most relevant chunks. Eliminates the round-trip of sending query text to an embedding API.
  • mcp-foxfield: Phone sensor bridge that collects GPS, accelerometer, barometer, and compass data from mobile devices. Streams structured telemetry instead of requiring the LLM to parse sensor dumps.
Why Go, not Python? A Python MCP server with FastAPI + uvicorn + numpy requires ~500MB of memory at startup and takes 2-3 seconds to initialize. A Go binary requires 100-320MB and starts in ~50ms. For MCP servers that may be spun up on-demand for each session, startup time and memory footprint directly impact user experience and system resource availability. Go's goroutine model also handles concurrent requests more efficiently than Python's GIL-constrained threading.

Layer 3: Structured Representations Plan Compression

The third layer addresses how we represent plans, instructions, and structured knowledge. Prose is expensive. Structure is cheap. The same information expressed as a Markdown narrative versus a compressed YAML plan can differ by 50-70% in token count.

Format Example (3-task plan) Tokens vs Prose
Prose Markdown "First, we need to create a new Rust module that handles PDF parsing. This module should accept a file path..." ~5,200 baseline
JSON plan {"tasks":[{"id":1,"action":"create","target":"src/pdf.rs","details":"..."}]} ~3,800 -27%
YAML plan tasks:\n- id: 1\n action: create\n target: src/pdf.rs ~3,200 -38%
Compressed YAML T1: create src/pdf.rs [PDF parser, lopdf, extract text+headings] ~2,200 -58%
TOON T1 create src/pdf.rs | lopdf | extract text headings ~1,900 -63%

UML and PlantUML serve as compressed system notation. A class diagram that would take 500 tokens to describe in prose takes ~120 tokens in PlantUML syntax. Sequence diagrams compress even more aggressively because they eliminate the need to describe temporal ordering in natural language — the diagram format encodes it structurally.

Template abstraction further compounds savings. When 80% of a plan follows a known pattern (TDD cycle, migration, API endpoint), the plan need only specify the 20% that varies. A template reference like @template:tdd-cycle(target=src/pdf.rs, tests=12) encodes in 15 tokens what would otherwise take 200+ tokens of boilerplate.

6. The Five Commandments

These are not guidelines. They are laws. Every token dollar wasted on violations of these commandments is a dollar that could have funded actual intelligence.

I

Convert to Markdown, Always

No PDF, no DOCX, no HTML, no RTF enters the model without conversion. Every document passes through a local converter (Rust, Pandoc, or equivalent) before it touches the context window. The converter strips formatting artifacts, removes headers and footers, collapses whitespace, and outputs clean semantic Markdown. This single rule eliminates 60-80% of document ingestion waste. There are zero exceptions. If the converter fails, the document is not ready for the model.

II

Compress Conversation History, Never Accumulate

Conversations are not archives — they are working memory. When a conversation exceeds 10 turns or 100K context tokens, it must be compressed. Compression means: summarize decisions made, extract action items, discard exploratory dead-ends, and carry forward only the state needed for the next turn. Use /compact, use structured handoff documents, use session state files. Never let context grow unbounded. The O(N²) cost of accumulation is the single largest source of token waste in multi-turn interactions.

III

Audit Your Plugin Tax

Every loaded skill, tool, and MCP server adds tokens to every turn. Know the cost. Measure it. If you have 57 commands and 34 skills loaded but are only using 5 of them, you are paying a 90% overhead tax on every single turn. Use deferred loading. Use session-specific profiles. Disable what you do not need. The plugin tax is insidious because it is invisible — it does not appear in your prompts, but it consumes your context budget silently.

IV

Use the Cheapest Model That Works

Opus is for thinking. Sonnet is for building. Haiku is for sorting. Local is for computing. Match the model to the task, not the task to the model. If you catch yourself using Opus to generate a commit message, to classify an error, or to format a string, you are burning $15/M tokens on a $0.25/M task. Model routing should be automatic, not manual. Build it into your infrastructure so the default is always the cheapest sufficient option.

V

Ask: Does This Need the LLM at All?

The most efficient token is the one never sent. Before every query, ask: is this a reasoning task or a computation task? Can grep solve this? Can a regex handle it? Can a Rust parser do it faster, cheaper, and more reliably? If the answer to any of these is yes, the LLM should not be involved. Every file search done by Glob instead of the LLM, every JSON validation done by serde instead of the LLM, every METAR observation parsed by a Rust decoder instead of the LLM — each is a token cost eliminated permanently.

7. The Self-Audit Checklist

We call this "The Stupid Button" — the set of questions that, if answered honestly, immediately reveal whether you are wasting tokens. Ask these six questions at the start of every session and at every 10-turn boundary. If you answer "yes" to any of them, stop and fix it before continuing.

1
Am I sending raw documents or converted Markdown?

If you pasted a PDF, a DOCX, or raw HTML into the conversation, you are paying 2-4x more than necessary. Stop. Convert it. Then send the Markdown. This applies to every document, every time, without exception.

2
How many turns deep am I? Should I start fresh?

Check your turn count. If you are past turn 15, your context is dominated by conversation history, not by your current question. Consider: is the historical context still relevant, or am I paying to carry forward dead weight? If the topic has shifted, start a new conversation. If the topic is continuous, compress the history first.

3
What plugins/skills are loaded that I'm not using?

List your active tools. Count them. For each one, ask: did I use this in the last 5 turns? Will I use it in the next 5? If the answer is no to both, it is dead weight. Every unused tool definition is consuming tokens on every turn. In a 20-turn session with 30 unused tools, you might be spending 200,000+ tokens on tool definitions you never invoke.

4
Could a smaller model handle this task?

Be honest about the complexity of your current request. Are you asking Opus to write a for-loop? To rename a variable? To generate boilerplate? These tasks do not require the world's most capable reasoning model. They require fast, cheap pattern matching. Route them to Haiku or Sonnet and save 5-60x on that turn.

5
Am I asking the model to compute something my machine could?

If you are asking the LLM to count lines in a file, calculate a hash, parse a date, validate a URL, or check if a port is open — stop. These are deterministic operations with exact answers. Your local machine can do them in milliseconds at zero token cost. The LLM adds nothing except cost and the possibility of hallucinated results.

6
Am I repeating context the model already has?

If you are pasting the same README, the same architecture overview, or the same coding conventions into multiple conversations, you need persistent context. Use CLAUDE.md for project-level context. Use MEMORY.md for cross-session state. Use prompt caching where available. Structure your workflows so context is injected once and persisted, not re-sent manually on every session start.

The Rule of Thumb: If answering these six questions takes more than 60 seconds, your setup is too complicated. Simplify your tool configuration, reduce your loaded plugins, and establish default behaviors so that efficient usage is the path of least resistance, not an active effort.

8. Research Connections

Token efficiency is not an isolated optimization problem. It connects to fundamental research in biological computation, neural architecture, hardware design, and business sustainability. These connections are not metaphors — they are structural analogs that inform our approach.

Slime Mold Optimization

Physarum polycephalum — a single-celled organism with no nervous system — solves shortest-path problems, builds efficient transport networks, and allocates resources across competing objectives. It does this with zero computational overhead in the LLM sense: no model weights, no matrix multiplications, no gradient descent. It is biological proof that intelligence does not require expensive computation.

The lesson for token economics: many problems that we reflexively send to an LLM have efficient algorithmic solutions. Graph traversal, resource allocation, scheduling, optimization — these have well-studied algorithms that run in polynomial time on local hardware. The LLM should be reserved for problems that genuinely require flexible reasoning over ambiguous inputs.

Research link: Ecology Research — Slime Mold Intelligence

Attention Residuals (AttnRes)

Recent research on selective attention mechanisms shows that not all transformer layers contribute equally to output quality. Attention Residuals (AttnRes) techniques identify and skip redundant attention computations within the model itself, reducing inference cost by 15-30% without measurable quality degradation on most benchmarks.

This is token efficiency at the hardware level: the model internally optimizing which computations matter. The same principle applies at the application level — not all context tokens contribute equally to output quality. Identifying and eliminating low-value context tokens is the application-layer analog of AttnRes.

BitNet: 1-Bit Weights

BitNet and related quantization research demonstrate that transformer models can operate with dramatically reduced precision — 1-bit weights instead of 16-bit — with surprisingly small quality losses. A 1-bit model requires 16x less memory and computes 16x faster per operation. This is the hardware-level complement to application-level token efficiency.

The combined effect is multiplicative: if you send 4x fewer tokens (application optimization) to a model that runs 4x faster per token (hardware optimization), you achieve a 16x improvement in cost-performance. This is the trajectory we are tracking.

TurboQuant: KV Cache Compression

The key-value (KV) cache is the memory structure that stores attention state across tokens. As context grows, the KV cache grows linearly, consuming GPU memory that could otherwise serve more concurrent requests. TurboQuant and similar techniques compress the KV cache by 2-4x, effectively doubling or quadrupling the context that fits in a given GPU's memory.

For our purposes, KV cache compression means longer contexts become economically viable. But this does not eliminate the input token cost — you still pay per-token at the API level. The optimization stack must work at both levels: reduce tokens sent (application), and reduce cost-per-token (infrastructure).

Fox Companies: Sustainability Arithmetic

For a bootstrapped AI company, token economics is not an optimization — it is an existential constraint. A startup with $100K in annual AI budget cannot afford $38/day/engineer in unoptimized Opus usage. At 5 engineers, that is $47,500/year — nearly half the budget consumed by inefficiency.

The same startup with optimized token usage spends $12.64/day/engineer, or $15,800/year for 5 engineers. The $31,700 saved is the difference between hiring another engineer and running out of runway. At Mythos-class pricing, the gap widens to $312,600 — the difference between a viable company and bankruptcy.

Every optimization in this document is an investment in sustainability. The compound effect of disciplined token management across an organization is the difference between building something that lasts and burning through capital on inference costs.
