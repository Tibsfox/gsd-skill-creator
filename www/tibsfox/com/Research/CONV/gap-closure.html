Gap-Closure Opportunities

← Convergent Substrate hub · Full PDF §7

The convergence findings explain why gsd-skill-creator already looks like the April 2026 literature. The mirror question — where the literature has patterns gsd-skill-creator does not yet ship — is the gap-closure opportunity. Five candidates are documented below, ranked by architectural leverage against the 14-row component map.

Half B of this milestone selects 3 to 5 of these for implementation. The final selection is made after this reference lands, based on test-budget fit and coverage across safety / theoretical-backing / architectural-gap / MCP-security / orchestration categories.

1 — Four-tier skill trust framework (T1-T4)

Source: Jiang et al., 2602.12430 (S). Supporting: 2604.07551, 2604.17125, 2603.22489.

The paper's 26.1% community-skill vulnerability baseline is the number the Staging Layer must beat, and the T1–T4 trust framework is the reference architecture for Wasteland / GASTOWN / DoltHub skill intake. Implementation: src/trust-tiers/ with tier-tagged cartridge metadata, CAPCOM gates that gate promotion based on tier transitions, and a governance table in docs. Estimated test cost: 20-25. High leverage because it directly closes a published threat-model gap.

2 — Two-Gate guardrail formalization (τ + K[m])

Source: Wang & Dorchen, 2510.04399 (A). Supporting: 2604.16968.

Formalizes the validation margin τ and capacity cap K[m] that prove self-modification safety is conditional on bounded capacity. Implementation: src/bounded-learning/two-gate/ defining the two thresholds explicitly, an oracle-inequality log at each CAPCOM boundary, and a documentation cross-reference proving that the 20% change cap is the gsd-skill-creator realization of K[m]. Estimated test cost: 15. High leverage because it converts an intuited governance constraint into a theorem-backed one.

3 — Compression Spectrum cross-layer

Source: Shen et al., 2604.15877 (S). Supporting: 2604.08224, 2604.01348.

Implements the missing diagonal explicitly: a single subsystem that compresses between episodic memory, procedural skills, and declarative rules. Implementation: src/compression-spectrum/ with adapters into the existing session memory, SKILL.md loader, and hooks. Estimated test cost: 25. High leverage because it fills a published gap the paper documents no system currently addresses. See the dedicated compression-spectrum page for detail.

4 — CASCADE three-tier MCP defense

Source: Abasikelesh Turgut et al., 2604.17125 (A). Supporting: 2603.22489, 2604.07551.

Three tiers of increasingly expensive input validation: regex+phrase-weighting+entropy, then BGE embedding + Ollama Llama3 fallback, then pattern-based output filtering. Reports 95.85% precision at 6.06% FPR with fully local operation. Implementation: src/mcp-defense/cascade/ with a progressive-fallback design and a test fixture of known tool-poisoning attempts. Estimated test cost: 15. High leverage for planning-bridge MCP server hardening — tool poisoning is the number-one published client-side vulnerability.

5 — Reasoning-graph evidence-centric feedback

Source: Penaroza, 2604.07595 (A). Supporting: 2604.14228.

Persists per-evidence chain-of-thought as structured edges on a traversal graph. Self-improving without retraining: all gains come from context engineering. Implementation: src/reasoning-graphs/ layered over the existing knowledge graph, with a retrospective-review pass that ingests past CAPCOM decisions as evidence edges. Estimated test cost: 20. Medium-to-high leverage — the pattern is empirically validated and would feed directly into the Drift defenses shipped in v1.49.569.

Below the cut line

Three further candidates are available if Half B capacity allows a sixth slot:

  • StageMem lifecycle memory — source 2604.16774. Would replace the current usage-count heuristic with confidence + strength metrics.
  • TalkLoRA inter-expert communication module — source 2604.06291. Makes the Amiga inter-chip communication pattern mathematical rather than metaphorical.
  • ECM capability/identity split documentation — source 2604.07799. Formal documentation of the Amiga Principle using ECM terminology. See the capability-evolution page.

Selection criteria

The Half B selection decision is gated on three criteria:

  1. Combined estimated test cost fits the milestone over-deliver budget (~80-120 tests target; over-deliver acceptable per user directive).
  2. Coverage of at least one candidate from each of {safety-layer, theoretical-backing, architectural-gap, MCP-security, orchestration}.
  3. Highest-leverage candidate per category.

Candidates 1–5 as ranked above satisfy all three criteria and are the recommended selection. Half B planning appends CONV-13 through CONV-22 to REQUIREMENTS.md with definitive module list, per-module sub-IDs, and estimated phase ranges.
