# RFC Document Template & Style Guide

> **Domain:** Internet Standards Authoring
> **Module:** 6 -- Required Sections, RFC 2119 Language, Boilerplate, and the Complete xml2rfc v3 Template
> **Through-line:** *RFC 7322 is a style guide, not a suggestion. Every RFC published since 2014 must conform to its requirements: specific sections in a specific order, with specific boilerplate text that varies by stream and IPR status. The requirements language keywords -- MUST, SHOULD, MAY -- carry precise interoperability meaning defined in RFC 2119 and clarified by RFC 8174. Getting these right is not pedantry; it is the difference between a specification that machines can implement consistently and prose that humans interpret differently. The template below includes every required section, every boilerplate block, and every structural constraint. Pass idnits. Ship the draft.*

## 1. Required Document Structure

RFC 7322 (the RFC Style Guide) mandates a specific document structure for all RFCs [1]. The order is not a convention -- it is a requirement enforced by the RFC Editor.

| # | Element | Required | Notes |
|---|---------|----------|-------|
| 1 | First-page header | Yes | Author(s), organization, stream, category, date, ISSN 2070-1721 |
| 2 | Title | Yes | Full document title |
| 3 | Abstract | Yes | Concise summary; no normative references |
| 4 | Status of This Memo | Yes* | Stream-specific boilerplate; RFC Editor supplies final text |
| 5 | Copyright Notice | Yes* | IETF Trust boilerplate; RFC Editor supplies |
| 6 | Table of Contents | Yes* | Auto-generated by xml2rfc |
| 7 | Introduction | Yes | First numbered section |
| 8 | Requirements Language | Conditional | Required if RFC 2119 keywords are used |
| 9 | Main body sections | Yes | Numbered; application-specific content |
| 10 | IANA Considerations | Yes | Required in I-Ds; may state "no actions" |
| 11 | Security Considerations | Yes | BCP 72; must address security implications |
| 12 | References | Yes | Split into Normative and Informative |
| 13 | Appendices | Optional | Labeled A, B, C (not numbered) |
| 14 | Acknowledgements | Optional | After appendices; not numbered |
| 15 | Contributors | Optional | Significant contributors not listed as authors |
| 16 | Author's Address | Yes | Postal, phone, and/or email |

*Supplied by RFC Editor during editorial process; must be present as placeholder in I-D.

## 2. Section-by-Section Guide

### Abstract

The abstract is a concise summary (typically 100-300 words) that describes what the document specifies or proposes. It MUST NOT contain:
- Normative references (no citations)
- RFC 2119 keywords (MUST/SHOULD/MAY)
- Forward references to document sections

The abstract should be comprehensible without reading the rest of the document [1].

### Introduction

Section 1 MUST be the Introduction. It provides background, describes the problem, summarizes related work, and states what the document contributes. If RFC 2119 requirements language is used, the Requirements Language boilerplate paragraph MUST appear in the Introduction or in a dedicated Requirements Language subsection [2].

### Main Body

Numbered sections containing the technical content. Use subsections freely. Every claim that affects interoperability should use RFC 2119 keywords.

### IANA Considerations

Required in all Internet-Drafts. If the document requires no IANA actions, state: "This document has no IANA actions." If the document creates new registries, modifies existing ones, or allocates code points, specify the exact actions required [3].

### Security Considerations

Required in ALL RFCs, per BCP 72 (RFC 3552). Cannot be omitted even if the document has no security implications -- in that case, state explicitly that there are no security implications and explain why. The section must address threats, vulnerabilities, and mitigations relevant to the specification [4].

### References

MUST be split into two subsections:
- **Normative References:** Documents that MUST be read to implement or understand this RFC
- **Informative References:** Background, related work, not required for implementation

### Author's Address

Every RFC MUST include contact information for each author. At minimum: name, organization, and one of postal address, phone, or email. Maximum 5 authors on the first page; additional contributors go in the Contributors section [1].

## 3. RFC 2119 Requirements Language

RFC 2119 (BCP 14) defines seven keywords with precise normative meaning. RFC 8174 clarifies that only UPPERCASE instances carry normative weight [2][5].

### Keyword Definitions

| Keyword | Meaning |
|---------|---------|
| **MUST** / **REQUIRED** / **SHALL** | Absolute requirement; no exceptions |
| **MUST NOT** / **SHALL NOT** | Absolute prohibition; no exceptions |
| **SHOULD** / **RECOMMENDED** | Strong recommendation; valid reasons to deviate may exist, but full implications must be understood |
| **SHOULD NOT** / **NOT RECOMMENDED** | Strong recommendation against; valid exceptions exist |
| **MAY** / **OPTIONAL** | Truly optional; implementations may or may not include |

### Required Boilerplate (BCP 14)

When using RFC 2119 keywords, include this exact paragraph in the Introduction or a Requirements Language subsection:

```
REQUIREMENTS LANGUAGE BOILERPLATE
================================================================

  The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
  "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
  and "OPTIONAL" in this document are to be interpreted as described
  in BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in
  all capitals, as shown here.
```

### Usage Rules

1. Keywords MUST be in ALL CAPITALS when used in their normative sense
2. Lowercase "must," "should," "may" do NOT carry normative meaning (RFC 8174)
3. Do not use normative keywords in the Abstract
4. Use keywords consistently -- the same requirement should always use the same keyword
5. When quoting text from another RFC that uses these keywords, clarify whether the normative meaning applies in your context [5].

## 4. Boilerplate Text

The boilerplate text in an RFC varies by stream and IPR status. The RFC Editor supplies the final boilerplate, but I-Ds must include correct placeholder text [6].

### Status of This Memo (by Stream)

| Stream | Boilerplate Summary |
|--------|-------------------|
| IETF (Standards Track) | "This is an Internet Standards Track document..." |
| IETF (BCP) | "This memo documents an Internet Best Current Practice..." |
| IETF (Informational) | "This document is not an Internet Standards Track specification..." |
| IETF (Experimental) | "This document is not an Internet Standards Track specification; it is published for examination..." |
| Independent | "This document is not an Internet Standards Track specification; it is published for informational purposes..." with note that it does not carry IETF consensus |
| IAB | "This document is a product of the Internet Architecture Board..." |
| IRTF | "This document is a product of the Internet Research Task Force..." |

### IPR Trust Statement

The standard IPR attribute for most submissions is `ipr="trust200902"`, which grants the IETF Trust standard rights per BCP 78. Variants exist for restrictive licenses, but `trust200902` is correct for nearly all new submissions [7].

### Copyright Notice

All RFCs carry the IETF Trust copyright notice. The text is standardized and must not be modified. xml2rfc generates it automatically from the `ipr` attribute [6].

## 5. Reference Management

References are split into two mandatory subsections within the References section.

### Normative vs. Informative

| Type | Criterion | Example |
|------|-----------|---------|
| Normative | MUST read to implement this RFC | RFC 2119 (if using MUST/SHOULD/MAY) |
| Informative | Background; helps understanding but not required | RFC 7322 (Style Guide reference) |

### Classification Rules

- Any document cited with a normative keyword (MUST implement, SHOULD conform to) is a Normative reference
- A Standards Track document SHOULD NOT have a normative reference to a non-standards-track document (downref)
- If a downref is necessary, it must be explicitly called out and approved by the IESG
- Self-references (citing the document itself) are not permitted [1]

### Reference Format in xml2rfc v3

```
REFERENCE EXAMPLES
================================================================

  <!-- Normative reference using entity -->
  &RFC2119;

  <!-- Informative reference (inline) -->
  <reference anchor="RFC7322">
    <front>
      <title>RFC Style Guide</title>
      <author initials="H." surname="Flanagan" fullname="H. Flanagan"/>
      <author initials="S." surname="Ginoza" fullname="S. Ginoza"/>
      <date year="2014" month="September"/>
    </front>
    <seriesInfo name="RFC" value="7322"/>
    <seriesInfo name="DOI" value="10.17487/RFC7322"/>
  </reference>

  <!-- Non-RFC reference -->
  <reference anchor="POSIX" target="https://pubs.opengroup.org/onlinepubs/9699919799/">
    <front>
      <title>The Open Group Base Specifications Issue 7</title>
      <author><organization>IEEE/Open Group</organization></author>
      <date year="2018"/>
    </front>
  </reference>
```

## 6. IANA Considerations

Every I-D MUST include an IANA Considerations section. This section tells IANA what, if anything, needs to happen when the document is published [3].

### When No IANA Action Is Needed

```
<section anchor="iana">
  <name>IANA Considerations</name>
  <t>This document has no IANA actions.</t>
</section>
```

### When IANA Action Is Required

Specify precisely:
- **New registry creation:** Registry name, initial values, registration policy (First Come First Served, Expert Review, Standards Action, etc.)
- **New code point allocation:** Which registry, which value, which meaning
- **Registry modification:** What changes, why, impact on existing entries

## 7. Security Considerations

BCP 72 (RFC 3552) provides guidance on writing Security Considerations sections [4].

### Minimum Requirements

1. Identify the threat model
2. Describe potential vulnerabilities
3. Specify mitigations
4. Note any residual risks

If the document genuinely has no security implications, state this explicitly and explain why. "This document has no security considerations" without explanation will be rejected by the IESG [4].

### Common Security Topics

| Topic | When to Address |
|-------|----------------|
| Authentication | If the protocol identifies parties |
| Authorization | If access control is relevant |
| Confidentiality | If sensitive data is transmitted |
| Integrity | If data modification is a threat |
| Availability | If denial-of-service is possible |
| Privacy | If personal data is involved |
| Cryptographic agility | If crypto algorithms are specified |

## 8. Common Style Rules

RFC 7322 defines detailed style requirements [1]:

### Text Conventions

- Line length: 72 characters for text output (xml2rfc handles this)
- No tabs in plain text output
- Section numbering: Arabic numerals (1, 1.1, 1.1.1)
- Appendices: Lettered (A, B, C), not numbered
- Figures and tables: Labeled and numbered sequentially

### Writing Style

| Rule | Guidance |
|------|----------|
| Voice | Active preferred over passive |
| Tense | Present tense for specifications; past for background |
| Person | Third person preferred; "this document" not "we" |
| Abbreviations | Define on first use; use consistently thereafter |
| Inclusive language | Per NIST SP 800-33; avoid exclusionary terms |

### Citation Style

- RFC references: `[RFC2119]` (symbolic, not `[1]`)
- Non-RFC references: `[AuthorYear]` or `[AcronymYear]`
- URLs in references: Use `target` attribute; avoid inline URLs in body text

## 9. Complete xml2rfc v3 Template

The following is a complete, annotated xml2rfc v3 template that includes all required sections for an Internet-Draft. This template is designed to pass idnits validation with zero errors.

```
COMPLETE XML2RFC v3 TEMPLATE
================================================================

<?xml version="1.0" encoding="utf-8"?>
<!DOCTYPE rfc [
  <!ENTITY RFC2119 SYSTEM
    "https://xml2rfc.ietf.org/public/rfc/bibxml/reference.RFC.2119.xml">
  <!ENTITY RFC8174 SYSTEM
    "https://xml2rfc.ietf.org/public/rfc/bibxml/reference.RFC.8174.xml">
]>

<?rfc toc="yes"?>
<?rfc sortrefs="yes" symrefs="yes"?>
<?rfc strict="yes" compact="yes"?>

<rfc xmlns:xi="http://www.w3.org/2001/XInclude"
     category="info"
     docName="draft-yourname-topic-00"
     ipr="trust200902"
     submissionType="IETF"
     consensus="true"
     obsoletes=""
     updates="">

<front>
  <title abbrev="Short Title">Full Document Title</title>
  <seriesInfo name="Internet-Draft"
              value="draft-yourname-topic-00"/>
  <author fullname="Author Name" initials="A." surname="Name">
    <organization>Organization</organization>
    <address>
      <postal>
        <city>City</city>
        <region>State</region>
        <country>US</country>
      </postal>
      <email>author@example.com</email>
    </address>
  </author>
  <date year="2026"/>
  <area>General</area>
  <keyword>keyword1</keyword>
  <abstract>
    <t>Document abstract. No citations, no normative keywords.</t>
  </abstract>
</front>

<middle>
  <section anchor="intro" numbered="true" toc="default">
    <name>Introduction</name>
    <t>Introduction text.</t>
    <t>
      The key words "MUST", "MUST NOT", "REQUIRED", "SHALL",
      "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED",
      "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document
      are to be interpreted as described in BCP 14
      <xref target="RFC2119"/> <xref target="RFC8174"/>
      when, and only when, they appear in all capitals,
      as shown here.
    </t>
  </section>

  <section anchor="overview" numbered="true" toc="default">
    <name>Overview</name>
    <t>Main technical content.</t>
  </section>

  <section anchor="iana" numbered="true" toc="default">
    <name>IANA Considerations</name>
    <t>This document has no IANA actions.</t>
  </section>

  <section anchor="security" numbered="true" toc="default">
    <name>Security Considerations</name>
    <t>Security analysis of the specification.</t>
  </section>
</middle>

<back>
  <references>
    <name>References</name>
    <references>
      <name>Normative References</name>
      &RFC2119;
      &RFC8174;
    </references>
    <references>
      <name>Informative References</name>
    </references>
  </references>
  <section anchor="ack" numbered="false" toc="default">
    <name>Acknowledgements</name>
    <t>Thanks to reviewers.</t>
  </section>
</back>
</rfc>
```

## 10. RFC as Literature

The RFC series is not just a technical archive -- it is a form of literature. The best RFCs combine rigorous specification with clear, readable prose. Jon Postel's specifications read as cleanly as scientific papers. RFC 2119's seven keywords encode an entire philosophy of interoperability in a single page. RFC 1149 (IP over Avian Carriers) is a complete protocol specification that also functions as satire.

### Writing Quality Matters

The RFC Editor's editorial review exists because clarity is a technical requirement. Ambiguous prose produces ambiguous implementations. Every sentence in a normative section that uses MUST or SHOULD is a contract between the author and every implementer who will ever read it. RFC 2119's genius is making that contract explicit: when you capitalize MUST, you are not expressing an opinion -- you are issuing a binding requirement that implementations must satisfy for interoperability [2].

The RFC series also serves as a historical record of how the Internet's architects thought about problems. Reading the early RFCs -- RFC 1, RFC 791, RFC 793 -- reveals not just protocols but design philosophies. The Robustness Principle ("be conservative in what you send, be liberal in what you accept") is not a protocol specification; it is a statement about how systems should treat each other. That it appears in a technical document rather than a philosophical treatise is itself significant: on the Internet, engineering IS philosophy [8].
