## 1. Archive Structure

The RFC Editor maintains the canonical archive at `https://www.rfc-editor.org`. Every RFC published since 1969 is available, in perpetuity, at a predictable URL pattern [1]:

```
URL PATTERNS
================================================================

  Text:  https://www.rfc-editor.org/rfc/rfc{number}.txt
  HTML:  https://www.rfc-editor.org/rfc/rfc{number}.html
  PDF:   https://www.rfc-editor.org/rfc/rfc{number}.pdf
  XML:   https://www.rfc-editor.org/rfc/rfc{number}.xml

  Examples:
    https://www.rfc-editor.org/rfc/rfc791.txt    (IPv4)
    https://www.rfc-editor.org/rfc/rfc9000.html   (QUIC)
    https://www.rfc-editor.org/rfc/rfc2119.txt    (BCP 14)
```

The archive is organized flat -- all RFCs in a single directory, numbered sequentially. There are no subdirectories by year, topic, or status. The number IS the identifier. Numbers are never reused, never reassigned, and never skipped (though some numbers were assigned but never published, creating small gaps in the early series) [2].

### Immutability Principle

Once published, an RFC is never changed. Not a single character. If there is an error, the RFC Editor publishes a new RFC that obsoletes or updates the original. The original remains in the archive with its original content, forever. This immutability is a fundamental design principle -- it means that any citation to an RFC number always refers to the same document [3].

Errata (minor corrections that do not warrant a new RFC) are tracked separately at `https://www.rfc-editor.org/errata/` and linked from the RFC's metadata page.

## 2. Available Formats

The RFC series has undergone a format evolution. Early RFCs were plain ASCII text with strict 72-character line widths. Modern RFCs are produced in multiple formats from an XML source.

| Format | Extension | Availability | Notes |
|--------|-----------|--------------|-------|
| Plain text | .txt | All RFCs | Historical primary format; 72-char lines; still produced for all new RFCs |
| HTML | .html | Most RFCs | Modern rendering; generated by rfc2html for pre-v3 RFCs; canonical for post-v3 |
| PDF | .pdf | Post-v3 RFCs | Paginated; generated by xml2rfc v3 |
| XML (RFCXML v3) | .xml | Post-v3 source | Canonical source format since RFC format transition (~2019) |
| XML (RFCXML v2) | .xml | Older submissions | Legacy format; convertible to v3 via `xml2rfc --v2v3` |
| EPUB | .epub | Limited | Available for some I-Ds via tools.ietf.org |

### The Format Transition

RFC 7990 (2016) and RFC 7994 (2016) initiated the transition from plain-text-only to XML-based production. Under the new model, the XML source is canonical, and all other formats (HTML, TXT, PDF) are derived outputs. This transition was completed gradually; RFCs published before ~2019 may only have .txt available natively, with .html generated retroactively [4].

## 3. rsync Modules

The RFC Editor provides rsync access for bulk mirroring. rsync is the recommended method for maintaining a complete local archive because it supports incremental updates -- only changed or new files are transferred on subsequent syncs [5].

### Available rsync modules at rsync.rfc-editor.org

| Module Name | Contents | Typical Size |
|-------------|----------|-------------|
| `rfcs-text-only` | All RFCs as .txt; includes bcp/, fyi/, ien/, std/ subdirectories | ~400 MB |
| `rfcs` | All RFCs in all formats (txt + html + pdf + xml) | ~2.5 GB |
| `rfcs-pdfs` | PDF versions only | ~1.2 GB |
| `rfcs-html` | HTML versions only | ~800 MB |

### rsync Command Syntax

```
RSYNC MIRROR COMMANDS
================================================================

  # Text-only mirror (smallest; all RFCs as .txt)
  rsync -avz --delete rsync.rfc-editor.org::rfcs-text-only ./rfc-mirror/

  # Full mirror (all formats: txt, html, pdf, xml)
  rsync -avz --delete rsync.rfc-editor.org::rfcs ./rfc-mirror-full/

  # PDF only
  rsync -avz --delete rsync.rfc-editor.org::rfcs-pdfs ./rfc-pdfs/

  # HTML only
  rsync -avz --delete rsync.rfc-editor.org::rfcs-html ./rfc-html/

  Flags:
    -a    archive mode (recursive, preserve permissions/timestamps)
    -v    verbose (show files being transferred)
    -z    compress during transfer
    --delete   remove local files no longer on the remote
```

### IETF rsync Endpoint

The IETF also provides rsync access at `rsync.ietf.org` for Internet-Drafts and other IETF materials. The tools endpoint at `rsync.tools.ietf.org` provides EPUB versions [5].

```
# Internet-Drafts (active)
rsync -avz rsync.ietf.org::internet-drafts ./internet-drafts/

# EPUB/eBook versions
rsync -avz --include="*.epub" --exclude="*" \
  rsync.tools.ietf.org::tools/ebook/ ./rfc-ebooks/
```

## 4. Building a Local Mirror

A complete local mirror requires three components: the RFC documents, the index file, and a maintenance schedule.

### Step-by-Step Mirror Setup

```
LOCAL MIRROR SETUP
================================================================

  1. Create the directory structure:
     mkdir -p /srv/rfc-archive/{text,full,index}

  2. Initial sync (text-only, fastest):
     rsync -avz --delete \
       rsync.rfc-editor.org::rfcs-text-only \
       /srv/rfc-archive/text/

  3. Download the index:
     rsync -avz \
       rsync.rfc-editor.org::rfcs-text-only/rfc-index.xml \
       /srv/rfc-archive/index/

  4. Optional: Full mirror (all formats):
     rsync -avz --delete \
       rsync.rfc-editor.org::rfcs \
       /srv/rfc-archive/full/

  5. Verify:
     ls /srv/rfc-archive/text/rfc*.txt | wc -l
     # Should show 9700+ files
```

### Initial Transfer Time

The initial sync depends on bandwidth. On a typical broadband connection:

| Mirror Type | Size | Estimated Time (50 Mbps) |
|-------------|------|--------------------------|
| Text-only | ~400 MB | ~1 minute |
| Full (all formats) | ~2.5 GB | ~7 minutes |
| PDF-only | ~1.2 GB | ~3 minutes |
| HTML-only | ~800 MB | ~2 minutes |

Subsequent incremental syncs transfer only new or changed files and typically complete in seconds [5].

## 5. rfc-index.xml

The `rfc-index.xml` file is the master metadata index for the entire RFC series. It contains structured information about every RFC: number, title, authors, date, status, stream, format availability, Obsoletes/Updates links, DOI, and more [6].

### Index Structure

```
RFC-INDEX.XML STRUCTURE (SIMPLIFIED)
================================================================

  <rfc-index>
    <rfc-entry>
      <doc-id>RFC0791</doc-id>
      <title>Internet Protocol</title>
      <author>
        <name>J. Postel</name>
      </author>
      <date>
        <month>September</month>
        <year>1981</year>
      </date>
      <format>
        <file-format>ASCII</file-format>
        <char-count>97779</char-count>
        <page-count>45</page-count>
      </format>
      <keywords>
        <kw>IP</kw>
        <kw>datagram</kw>
      </keywords>
      <abstract>...</abstract>
      <current-status>INTERNET STANDARD</current-status>
      <publication-status>INTERNET STANDARD</publication-status>
      <stream>IETF</stream>
      <doi>10.17487/RFC0791</doi>
      <updated-by>
        <doc-id>RFC1349</doc-id>
        <doc-id>RFC2474</doc-id>
        <doc-id>RFC6864</doc-id>
        <doc-id>RFC8200</doc-id>
      </updated-by>
    </rfc-entry>
    <!-- ... 9700+ entries ... -->
  </rfc-index>
```

### Key Fields

| Field | Description |
|-------|-------------|
| `doc-id` | RFC number (RFC0001 through RFC9999+) |
| `title` | Document title |
| `author` | One or more author entries |
| `date` | Publication month and year |
| `current-status` | Current maturity level (PROPOSED STANDARD, INTERNET STANDARD, etc.) |
| `stream` | Publication stream (IETF, IAB, IRTF, INDEPENDENT, EDITORIAL) |
| `obsoleted-by` | List of RFCs that obsolete this one |
| `obsoletes` | List of RFCs this one obsoletes |
| `updated-by` | List of RFCs that update this one |
| `updates` | List of RFCs this one updates |
| `doi` | Digital Object Identifier |

The `rfc-index.txt` plain-text version is also available for simpler parsing [6].

## 6. Local Search Indexing

With a local mirror and rfc-index.xml, you can build offline search capability. Several approaches work:

### grep-based Search (Simplest)

```
# Search by keyword in text files
grep -rl "QUIC" /srv/rfc-archive/text/rfc*.txt

# Search by title in index
grep -i "transport" /srv/rfc-archive/index/rfc-index.txt

# Find all RFCs by a specific author
grep "Postel" /srv/rfc-archive/index/rfc-index.txt
```

### SQLite Index (Structured)

Parse `rfc-index.xml` into a SQLite database for structured queries:

```
SQLITE SCHEMA FOR RFC INDEX
================================================================

  CREATE TABLE rfcs (
    number      INTEGER PRIMARY KEY,
    title       TEXT NOT NULL,
    date        TEXT,
    status      TEXT,
    stream      TEXT,
    abstract    TEXT,
    page_count  INTEGER,
    doi         TEXT
  );

  CREATE TABLE authors (
    rfc_number  INTEGER REFERENCES rfcs(number),
    name        TEXT NOT NULL
  );

  CREATE TABLE relations (
    rfc_number   INTEGER REFERENCES rfcs(number),
    relation     TEXT CHECK(relation IN
      ('obsoletes','obsoleted_by','updates','updated_by')),
    related_rfc  INTEGER
  );

  CREATE INDEX idx_title ON rfcs(title);
  CREATE INDEX idx_status ON rfcs(status);
  CREATE INDEX idx_author ON authors(name);
```

### Full-Text Search Engines

For larger-scale search, tools like `recoll`, `ripgrep`, or `tantivy` can index the full text of all 9,700+ RFCs. A full-text index of the text-only archive typically requires 200-400 MB of index storage [7].

## 7. Bulk Download Alternatives

For environments where rsync is not available, the RFC Editor provides alternative download methods [5]:

| Method | URL | Notes |
|--------|-----|-------|
| Web browsing | rfc-editor.org/rfc/ | Individual document download |
| Bulk zip/tar | rfc-editor.org/retrieve/bulk/ | Compressed archives of all RFCs |
| IETF tools | tools.ietf.org | Alternative document mirror |
| DOI resolution | doi.org/10.17487/RFC{number} | Redirects to rfc-editor.org |

The bulk download page provides complete archives in both .zip and .tar.gz formats, updated periodically. This is suitable for one-time downloads but not for maintaining a continuously current mirror [5].

## 8. Keeping the Mirror Current

A cron job provides automated daily synchronization.

```
CRON JOB FOR DAILY RFC SYNC
================================================================

  #!/bin/bash
  # /etc/cron.daily/rfc-sync
  # Runs daily; only transfers new/changed files

  LOGFILE="/var/log/rfc-sync.log"
  MIRROR_DIR="/srv/rfc-archive/text"
  INDEX_DIR="/srv/rfc-archive/index"

  echo "=== RFC Sync: $(date -u +%Y-%m-%dT%H:%M:%SZ) ===" >> "$LOGFILE"

  # Sync text-only RFCs
  rsync -avz --delete \
    rsync.rfc-editor.org::rfcs-text-only \
    "$MIRROR_DIR/" >> "$LOGFILE" 2>&1

  # Sync index separately (ensures always current)
  rsync -avz \
    rsync.rfc-editor.org::rfcs-text-only/rfc-index.xml \
    "$INDEX_DIR/" >> "$LOGFILE" 2>&1

  echo "=== Sync complete ===" >> "$LOGFILE"
```

### Sync Frequency Recommendations

| Use Case | Frequency | Rationale |
|----------|-----------|-----------|
| Standards-compliant development | Daily | New RFCs publish 2-4 per week |
| Research reference | Weekly | Sufficient for most citation needs |
| Air-gapped environment | On-demand | Manual sync via sneakernet |
| Bandwidth-constrained | Monthly | Acceptable lag for archival use |

New RFCs are published at a rate of approximately 200-250 per year (2020-2024 average), or roughly 4-5 per week. Daily sync ensures a maximum lag of ~24 hours from publication [2].

## 9. Obsoletes and Updates Chains

The RFC archive's most powerful metadata feature is the Obsoletes/Updates chain. This web of relationships tracks how specifications evolve over time.

```
EXAMPLE: HTTP SPECIFICATION CHAIN
================================================================

  RFC 1945 (1996) HTTP/1.0 [Informational]
       │
       └── obsoleted by ──► RFC 2068 (1997) HTTP/1.1
                                 │
                                 └── obsoleted by ──► RFC 2616 (1999) HTTP/1.1
                                                           │
                                                           ├── obsoleted by ──► RFC 7230 (2014)
                                                           ├── obsoleted by ──► RFC 7231 (2014)
                                                           ├── obsoleted by ──► RFC 7232 (2014)
                                                           ├── obsoleted by ──► RFC 7233 (2014)
                                                           ├── obsoleted by ──► RFC 7234 (2014)
                                                           └── obsoleted by ──► RFC 7235 (2014)
                                                                                    │
                                                                                    └── all obsoleted by ──►
                                                                                         RFC 9110 (2022)
                                                                                         RFC 9111 (2022)
                                                                                         RFC 9112 (2022)
```

### Relationship Types

| Relation | Meaning |
|----------|---------|
| Obsoletes | Completely replaces the referenced RFC; old RFC is reclassified as Historic |
| Updates | Partially modifies the referenced RFC; both remain active |
| Obsoleted-by | This RFC has been completely replaced |
| Updated-by | This RFC has been partially modified |

When citing an RFC, always check whether it has been obsoleted. The `rfc-index.xml` file contains this metadata for every RFC. The RFC Editor website also displays prominent banners on obsoleted RFCs [6].

## 10. Archive Size and Storage Planning

| Component | Size (2026 estimate) | Growth Rate |
|-----------|---------------------|-------------|
| Text-only archive | ~400 MB | ~15 MB/year |
| Full archive (all formats) | ~2.5 GB | ~100 MB/year |
| rfc-index.xml | ~25 MB | ~500 KB/year |
| Internet-Drafts (active) | ~3.0 GB | ~500 MB/year |
| SQLite search index | ~200 MB | ~8 MB/year |
| **Total (text + index + search)** | **~625 MB** | **~25 MB/year** |
| **Total (full + I-Ds + search)** | **~5.7 GB** | **~600 MB/year** |

The entire text-only RFC archive comfortably fits on a USB flash drive. A full archive including all formats, Internet-Drafts, and search indexes requires less than 6 GB -- trivial by modern storage standards [5].
