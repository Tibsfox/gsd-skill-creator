C’s type system is deliberately thin: it describes how many bits a value occupies and how the CPU should interpret them. The standard leaves many sizes implementation-defined so C can target everything from 8-bit microcontrollers to 64-bit supercomputers. The power of C is that the model matches the machine. The danger of C is the same thing.

The Type System

The standard only guarantees minimum ranges. sizeof(char) == 1 always, but a char could be more than 8 bits on exotic platforms. Signedness of plain char is implementation-defined.

Fixed-Width Integers — <stdint.h> (C99)

#include <stdint.h>

int8_t   a;   // exactly 8 bits, signed
uint16_t b;   // exactly 16 bits, unsigned
int32_t  c;   // exactly 32 bits, signed
uint64_t d;   // exactly 64 bits, unsigned

int_least16_t e;  // at least 16 bits, smallest such type
int_fast32_t  f;  // at least 32 bits, "fastest" such type
intmax_t      g;  // largest available integer
intptr_t      h;  // integer wide enough to hold a pointer

Pointers — The Fundamental Concept

int x = 42;
int *p = &x;         // p holds the address of x
int value = *p;       // dereference: get what p points to
*p = 100;             // write through the pointer: x is now 100

// Pointer arithmetic (in units of the pointed-to type)
int arr[5] = {10, 20, 30, 40, 50};
int *q = arr;         // decays to &arr[0]
q++;                  // now points to arr[1]
printf("%d\n", *q);   // 20

// Array-pointer duality: a[i] == *(a+i)
arr[2] == *(arr + 2);  // true

// Pointer to pointer
int **pp = &p;
**pp = 200;           // x is now 200

// Function pointers
int (*fn)(int, int) = &add;
int result = fn(3, 4);

// void* — the generic pointer
void *any = &x;       // can point to anything
int *back = any;      // implicit cast back

The restrict Qualifier (C99)

// Promise: no other pointer will access these ranges during the call
void add_arrays(int *restrict dest,
                const int *restrict a,
                const int *restrict b,
                size_t n) {
    for (size_t i = 0; i < n; i++)
        dest[i] = a[i] + b[i];
}

restrict is a promise to the compiler: “no other pointer in this function body will read or write the memory these pointers access.” It enables vectorization and other optimizations that would be unsafe under C’s default aliasing rules. Violating the promise is undefined behavior.

struct Point p = { .x = 3.0, .y = 4.0 };  // designated initializer (C99)

Memory Management

#include <stdlib.h>
#include <string.h>

// Allocate
int *arr = malloc(100 * sizeof(int));      // uninitialized
int *zeroed = calloc(100, sizeof(int));    // zero-initialized

if (arr == NULL) { /* handle OOM */ }

// Resize
int *bigger = realloc(arr, 200 * sizeof(int));
if (bigger == NULL) {
    free(arr);   // original still valid on failure
    return -1;
}
arr = bigger;

// Free
free(arr);
arr = NULL;      // common idiom: prevent use-after-free

The canonical C memory bugs:

  • Use-after-free: Accessing memory after free()
  • Double-free: Calling free() twice on the same pointer
  • Memory leak: Losing the last pointer to allocated memory
  • Buffer overflow: Writing past the end of an allocation
  • Null dereference: Forgetting to check malloc’s return
  • Uninitialized read: Reading malloc’d memory before writing

These are the entire memory-safety debate in six bullets. Every one of them is caught by Rust’s borrow checker. Every one of them is runtime-detected by AddressSanitizer. Every one of them has caused a CVE.

Undefined Behavior — Power and Peril

Undefined behavior (UB) is C’s great sin and its great power. When the standard says an operation is “undefined,” the compiler is free to do anything — produce any result, crash, format your hard drive, summon nasal demons. This latitude is what enables aggressive optimization.

Canonical Cases of UB

CaseExample
Signed integer overflowint x = INT_MAX; x + 1;
Strict aliasing violationfloat f; int i = *(int*)&f;
Reading uninitialized memoryint x; printf("%d", x);
Out-of-bounds accessint a[5]; a[10] = 0;
Null dereferenceint *p = NULL; *p = 0;
Use-after-freefree(p); *p = 0;
Data raceTwo threads writing the same non-atomic variable
Division by zeroint x = 5 / 0;

Why UB exists: If signed overflow were defined, the compiler couldn’t assume (x + 1) > x, which blocks a huge class of loop optimizations. If strict aliasing were relaxed, every pointer access would need alias analysis. UB is a performance bargain.

Why UB is dangerous: Modern optimizers aggressively assume UB doesn’t happen. If the compiler can prove that a branch leads to UB, it may delete the branch — or the check that would have prevented it. This has produced famous security bugs where security checks were optimized away because the compiler assumed the undefined case could never occur.

The culture: Expert C programmers develop a sixth sense for UB. They run -fsanitize=undefined in testing. They read compiler warning output like scripture. They know the dragons.

C99 Improvements (1999)

  • // comments (finally, after 27 years)
  • inline functions — compiler hint for performance-critical small functions
  • VLAs (variable-length arrays) — int arr[n];. Controversial; optional in C11+
  • <stdint.h> — fixed-width integer types
  • <stdbool.h> — bool, true, false
  • restrict — aliasing hint for optimization
  • Compound literals: (struct Point){ .x = 1, .y = 2 }
  • Designated initializers: int arr[5] = { [2] = 10 };
  • long long (at least 64 bits)
  • Variadic macros: #define FOO(...) bar(__VA_ARGS__)
  • For loop variable scoping: for (int i = 0; ...)

C11 Improvements (2011)

  • _Generic — type-based macro dispatch (Poor man’s overloading)
  • Atomics (<stdatomic.h>) — lock-free programming primitives
  • Threads (<threads.h>) — optional; rarely implemented
  • alignof / alignas — explicit alignment control
  • _Noreturn — functions that don’t return (exit, abort)
  • Anonymous structs/unions — cleaner tagged-union idioms
  • static_assert — compile-time assertions
  • char16_t, char32_t — Unicode character types

C23 Improvements (2024) — The Big Modernization

// nullptr — finally!
int *p = nullptr;  // no more (void*)0 ambiguity

// typeof — borrowed from GCC extension
#define SWAP(a, b) do { \
    typeof(a) _tmp = (a); \
    (a) = (b); \
    (b) = _tmp; \
} while (0)

// Bit-precise integers
_BitInt(7) tiny = 42;       // exactly 7 bits
_BitInt(128) huge = 0;       // exactly 128 bits

// Typed enums
enum Priority : uint8_t { LOW = 0, MEDIUM = 1, HIGH = 2 };

// constexpr for objects
constexpr int BUFFER_SIZE = 4096;

// auto type inference (limited form)
auto x = 42;           // int
auto y = 3.14;         // double

// [[attribute]] syntax (borrowed from C++)
[[noreturn]] void fatal(const char *msg);
[[nodiscard]] int compute_important_value(void);
[[deprecated("use new_api instead")]] void old_api(void);

// #embed directive — include binary data
const unsigned char logo[] = {
    #embed "logo.png"
};

// Binary literals and digit separators
int mask  = 0b1010'1100;
int limit = 1'000'000;

// u8 string literals guaranteed UTF-8
const char8_t *greeting = u8"\u4F60\u597D";

Canonical C Idioms

Error Handling with goto

int process_file(const char *path) {
    FILE *fp = NULL;
    char *buffer = NULL;
    int result = -1;

    fp = fopen(path, "r");
    if (fp == NULL) goto cleanup;

    buffer = malloc(1024);
    if (buffer == NULL) goto cleanup;

    if (fread(buffer, 1, 1024, fp) == 0) goto cleanup;

    result = 0;   // success

cleanup:
    free(buffer);
    if (fp) fclose(fp);
    return result;
}

This is one of the legitimate uses of goto — unwinding resources on error. It’s used throughout the Linux kernel, where it’s considered idiomatic rather than harmful.

The do { ... } while (0) Macro Wrap

#define CHECK(x) do { \
    if (!(x)) { \
        fprintf(stderr, "Check failed: %s\n", #x); \
        abort(); \
    } \
} while (0)

// Usage (works correctly as a statement, even with if/else)
if (condition)
    CHECK(ptr != NULL);
else
    printf("skipped\n");
