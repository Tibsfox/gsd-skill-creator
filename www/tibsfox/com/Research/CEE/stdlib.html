Standard Library & libc

stdio, stdlib, POSIX, glibc, musl, Bionic, Cosmopolitan — The Universal Runtime
Overview Core libc Compilers Systems ABI Rosetta Learn History
The C standard library (libc) is the oldest piece of running code most people on Earth depend on every day. It began as a small set of headers in K&R C (1978), was formalized in ANSI C (1989), and has been continuously extended through C99, C11, C17, and C23. Every Unix system has a libc. Every Windows system has the MSVCRT or UCRT. Every Android phone has Bionic. Every embedded device either ships with newlib or dietlibc or rolls its own. libc is the universal runtime of modern computing.

printf — The Most Famous Function

#include <stdio.h>

int main(void) {
    int    n = 42;
    double f = 3.14159;
    char  *s = "hello";

    printf("integer:  %d\n", n);          // 42
    printf("hex:      %x\n", n);          // 2a
    printf("float:    %.2f\n", f);        // 3.14
    printf("string:   %s\n", s);          // hello
    printf("pointer:  %p\n", (void*)&n);  // 0x7ff...
    printf("width:    %10d\n", n);        //         42
    printf("zero pad: %05d\n", n);        // 00042
    printf("type:     %lld\n", 123LL);    // 123 (long long)
    printf("percent:  %%\n");             // %

    return 0;
}

printf’s format string is a runtime-interpreted mini-language. The compiler has no idea what arguments will actually be passed — it’s all variadic. This is why modern C compilers ship -Wformat to catch mismatches at compile time. It’s also why printf has been the source of countless security bugs (format string attacks where user input is passed as the format argument).

libc Implementations

The C standard library is a specification, not a single implementation. Each OS (and sometimes each distro) ships its own libc. They all provide the same interface, but their sizes, performance characteristics, and licenses differ wildly.

musl — Minimal, Static-Friendly

AttributeValue
CreatorRich Felker, 2011
LicenseMIT
Default onAlpine Linux, OpenWrt, Void Linux, many Docker images
Size~700 KB for the shared library
DesignStrict standards compliance, small, simple, static-linking friendly

Alpine Linux’s rise is musl’s rise. Alpine became popular for Docker images because a musl-based base image is ~5 MB compared to glibc’s ~80 MB. Every time a Docker image starts with FROM alpine, you’re using musl. Sometimes this causes subtle bugs when programs assume glibc-specific behavior.

POSIX — Beyond ANSI C

Portable Operating System Interface. IEEE 1003.1 / The Open Group. POSIX standardizes what Unix-like systems share beyond the C standard library. If your program uses only <stdio.h> and <stdlib.h>, it’s portable to anything. If it uses POSIX APIs, it’s portable to any Unix-like system.

The String Function Minefield

The C string functions are simultaneously the most-used and most-dangerous parts of the standard library. Every buffer overflow CVE traces back to one of these functions being used incorrectly:

FunctionRiskSafer Alternative
strcpy(dst, src)No bounds check. Writes until null terminatorsnprintf, strlcpy (BSD), or manual length checks
strcat(dst, src)No bounds check. Worse than strcpy because it starts at dst’s endsnprintf, strlcat (BSD)
sprintf(buf, ...)No bounds check on outputAlways use snprintf
gets(buf)Catastrophically unsafe. Reads until newlineRemoved in C11. Use fgets
strncpy(dst, src, n)Does NOT guarantee null termination if src is too long. Also pads with nullsStill subtly unsafe. Check length first
scanf("%s", buf)No bounds on %sfgets + sscanf

The BSD functions strlcpy and strlcat (Todd Miller, OpenBSD, 1998) are safer alternatives but are not in the C standard. Microsoft’s Annex K (strcpy_s, etc.) is in C11 but optional and rarely implemented on Linux. The de facto safe pattern is snprintf, which is universally available.
