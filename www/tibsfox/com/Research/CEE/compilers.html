C has two reigning compilers (GCC and Clang), half a dozen viable alternatives, and a build-system ecosystem that is notorious for being simultaneously the most and least mature of any language. The compilation toolchain for C is ancient, layered, and everywhere — because C is everywhere.

GCC — GNU Compiler Collection

Richard Stallman, 1987. Originally the GNU C Compiler. Renamed to Compiler Collection in 1999 when it had grown well beyond C. In 1997 the EGCS fork appeared out of frustration with FSF’s slow release cadence; EGCS became the official GCC in April 1999. License: GPLv3+ with the GCC Runtime Library Exception (so programs compiled by GCC aren’t forced into GPL).

GCC Supported Languages (2026)

C, C++, Objective-C, Objective-C++, Fortran, Ada, Go, D, Modula-2, COBOL, Rust (gccrs), Algol 68 — plus associated runtime libraries.

GCC Pipeline

Source → Frontend (language-specific parsing) → GENERIC (language-independent tree) → GIMPLE (3-address form, SSA) → Tree-SSA optimizations → RTL (Register Transfer Language, low-level) → Target machine code

This multi-stage IR is why GCC can share optimization passes across languages. A Fortran program and a C program both go through GIMPLE.

Clang / LLVM

Chris Lattner, University of Illinois at Urbana-Champaign, 2000. LLVM (Low Level Virtual Machine) started as Lattner’s master’s thesis. Clang (the C language family frontend) was added in 2007, sponsored by Apple. LLVM 1.0 released 2003; Clang became usable for C around 2009.

License: Originally UIUC license, now Apache 2.0 with LLVM Exception. Permissive — companies can build proprietary products on top.

Why Clang Won Apple

Apple had been contributing to GCC for years but grew frustrated with GCC’s GPL licensing (which conflicted with Apple’s need to ship compiler libraries in their SDK) and monolithic architecture (hard to use GCC as a library for IDEs and refactoring tools). Clang solved both: permissive license + modular, library-based design. Apple became the primary sponsor. By macOS 10.7 (2011), Clang was the default compiler. By macOS 10.9 (2013), GCC was gone from Xcode entirely.

Build Systems

make — The Original

Stuart Feldman, Bell Labs, 1976. Written to solve Feldman’s frustration with rebuilding C programs manually. The entire concept of dependency-based build systems came from this project. “The dollar sign is the most expensive mistake I ever made,” Feldman later said of $ for variable expansion — but the fundamental model of targets, dependencies, and recipes has never been seriously challenged.

# Makefile
CFLAGS = -Wall -Wextra -O2
LDFLAGS = -lm

program: main.o util.o
	$(CC) $(LDFLAGS) -o $@ $^

%.o: %.c
	$(CC) $(CFLAGS) -c -o $@ $<

clean:
	rm -f *.o program

GNU Autotools — The Terrible But Essential Toolchain

configure + automake + autoconf + libtool. The classic ./configure && make && make install that everyone has typed thousands of times. Generates portable shell scripts that detect platform capabilities. Notoriously slow and cryptic, but extraordinarily portable — this is how tarballs have worked since the 1990s.

CMake — Now Dominant

Kitware, 2000. Originally built to support ITK (Insight Toolkit) for medical image analysis. Generates native build files (Makefiles, Ninja files, Visual Studio projects, Xcode projects) from a high-level CMakeLists.txt description. Now the de facto standard for new C/C++ projects. Complex and verbose, but wins on portability.

# CMakeLists.txt
cmake_minimum_required(VERSION 3.20)
project(myapp VERSION 1.0 LANGUAGES C)

add_executable(myapp main.c util.c)
target_compile_features(myapp PUBLIC c_std_23)
target_compile_options(myapp PRIVATE -Wall -Wextra)
target_link_libraries(myapp PRIVATE m)

Meson — The Python-Based Challenger

Jussi Pakkanen, 2013. Written in Python, emits Ninja build files. Syntax is deliberately not Turing-complete — “you shouldn’t need to debug your build system.” Adopted by GNOME, systemd, and many newer projects.

Ninja — The Ultra-Fast Executor

Evan Martin, Google, 2010. Deliberately not a build system — it’s a build executor. Designed to be the backend target of other tools (CMake, Meson, Bazel). Handles tens of thousands of targets in milliseconds. Originally built to replace Make in the Chrome build system.

Bazel — Google’s Monorepo Build

Google-developed build system for monorepos. Hermetic, reproducible, caches everything. Used internally at Google for two decades before being open-sourced. Adopted by Envoy, Istio, and other Google-adjacent projects.

Sanitizers — Runtime Bug Detection

Modern compilers include sanitizers — compile-time instrumentation that adds runtime checks for specific bug classes. Both GCC and Clang support them. Enable with -fsanitize=....

SanitizerFlagDetects
AddressSanitizer (ASan)-fsanitize=addressUse-after-free, buffer overflows, double-free, memory leaks (on exit)
ThreadSanitizer (TSan)-fsanitize=threadData races, deadlocks in multithreaded code
MemorySanitizer (MSan)-fsanitize=memoryUse of uninitialized memory (Clang only)
UndefinedBehaviorSanitizer (UBSan)-fsanitize=undefinedSigned overflow, alignment violations, null deref, array OOB
LeakSanitizer (LSan)-fsanitize=leakStandalone leak detector (subset of ASan)

Valgrind (Julian Seward, 2002) is the historical alternative — a dynamic binary instrumentation framework with Memcheck for memory errors. Slower than sanitizers (20× vs 2×) but works on unmodified binaries.

IDE & Editor Tooling

clangd is Clang’s language server (LSP). It provides autocomplete, go-to-definition, diagnostics, and refactoring to any LSP-compatible editor. Combined with VS Code, Neovim, Emacs, or any other LSP-aware editor, clangd gives you IDE-quality C/C++ support anywhere.

The modern minimal C development stack: VS Code + clangd + CMake Tools + CodeLLDB. Cross-platform. Free. Good enough for the Linux kernel.
