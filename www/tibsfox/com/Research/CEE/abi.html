The C ABI — Universal Glue

The Esperanto of Computing, Calling Conventions, FFI, Name Mangling, Struct Layout
Overview Core libc Compilers Systems ABI Rosetta Learn History
The C ABI is not a specification. There is no ISO document titled “The C Application Binary Interface.” What there is, instead, is a set of platform conventions that every C compiler on a given OS and architecture has agreed to follow — and that every other language on that platform is forced to mimic if it wants to call C code. This ad-hoc convergence, born of Unix’s portability and 50 years of inertia, is why C became the Esperanto of programming languages.

What Is an ABI?

An Application Binary Interface is the contract between two pieces of compiled code about how to communicate. It specifies:

  • Calling convention — how arguments are passed (which registers, which stack slots) and how return values come back
  • Stack layout — who saves which registers, how the stack frame is organized, where the return address lives
  • Name mangling — how function and variable names appear in the binary (C says: exactly as written, no mangling)
  • Struct layout — field ordering, padding, alignment rules
  • Integer sizes — how big int and long actually are on this platform
  • Object file format — ELF (Linux), Mach-O (macOS), COFF/PE (Windows)
  • Exception handling (in C++, not C) — how unwinding propagates across function boundaries
  • Symbol visibility — what’s exported, what’s hidden

Name Mangling — Or the Lack Thereof

C’s single most important contribution to interop: names are not mangled. A C function void foo(int x) appears in the binary as the symbol foo (or _foo on macOS for historical reasons). That’s it.

Compare to C++

// C
void greet(int n) { ... }              // symbol: greet

// C++ (Itanium ABI, used by GCC/Clang on Linux/macOS)
void greet(int n) { ... }              // symbol: _Z5greeti
namespace foo { void greet(int) { } } // symbol: _ZN3foo5greetEi
class Bar { void greet(int); };        // symbol: _ZN3Bar5greetEi

// C++ (Microsoft ABI)
void greet(int n) { ... }              // symbol: ?greet@@YAXH@Z

C++ must mangle names because it supports overloading (two functions named greet with different argument types need different symbols), namespaces, classes, and templates. C doesn’t have any of that, so it doesn’t need to mangle.

This is why extern "C" exists in C++. It tells the C++ compiler: “for this block, use C’s no-mangling rule so C code can call me.” Every cross-language FFI depends on this. When Python calls into a C library, it looks for a symbol named exactly my_function. If the library were C++ without extern "C", Python would need to know the mangling convention of the compiler that built it — a nightmare.

Struct Layout — Padding and Alignment

C struct layout is mostly standardized but has enough wiggle room to cause real FFI pain:

struct Example {
    char  a;   // offset 0, size 1
    // 3 bytes padding on 32-bit systems (align int to 4)
    int   b;   // offset 4, size 4
    char  c;   // offset 8, size 1
    // 7 bytes padding (align pointer to 8 on 64-bit)
    void *p;   // offset 16, size 8
    // struct size: 24 bytes (multiple of largest alignment)
};

Rules:

  • Fields are laid out in declaration order (C guarantees this)
  • Each field is aligned to its natural alignment (usually its size)
  • Padding is inserted between fields to satisfy alignment
  • Struct size is rounded up to a multiple of the largest field’s alignment
  • Reordering fields can reduce size: {void*, int, char, char} is 16 bytes, {char, int, void*, char} can be 24

Tools like pahole (part of dwarves) show struct layout from DWARF debug info — essential for optimizing cache behavior in the Linux kernel.

#pragma pack — Forcing a Layout

#pragma pack(push, 1)
struct NetworkPacket {
    uint8_t  type;
    uint32_t length;    // at offset 1 (!), unaligned read
    uint8_t  flags;
    uint16_t checksum;
};  // total size: 8 bytes, no padding
#pragma pack(pop)

Used for network protocols, file formats, hardware registers — anywhere the layout is externally dictated. On many architectures, unaligned access is slower or generates faults, so pack is a double-edged sword.

Integer Sizes — The Great ABI Ambiguity

C89 specifies only minimum sizes for its integer types. Actual sizes depend on the platform:

TypeC89 MinimumLinux/macOS x86-64Windows x86-6432-bit ARM
char1 byte111
short2 bytes222
int2 bytes444
long4 bytes844
long long8 bytes (C99)888
size_t—884
void*—884

The long problem. On 64-bit Linux, long is 8 bytes (LP64 model). On 64-bit Windows, long is still 4 bytes (LLP64 model). A struct with a long field has a different size on these two platforms. This is why C99 introduced <stdint.h> with int32_t, int64_t, intptr_t, etc. — guaranteed-size types that work across ABIs.

Rule of thumb for modern C: never use bare int/long in a struct that crosses language or platform boundaries. Use <stdint.h> types.

Calling C from Every Language

Python (ctypes)

import ctypes
libm = ctypes.CDLL("libm.so.6")
libm.sqrt.argtypes = [ctypes.c_double]
libm.sqrt.restype = ctypes.c_double
print(libm.sqrt(2.0))  # 1.4142135623730951

Rust (extern "C")

extern "C" {
    fn sqrt(x: f64) -> f64;
}

fn main() {
    unsafe { println!("{}", sqrt(2.0)); }
}

Go (cgo)

package main

// #include <math.h>
import "C"
import "fmt"

func main() {
    fmt.Println(C.sqrt(2.0))
}

Node.js (N-API via node-ffi-napi)

const ffi = require('ffi-napi');

const libm = ffi.Library('libm', {
  'sqrt': ['double', ['double']]
});

console.log(libm.sqrt(2.0));

Ruby (Fiddle)

require 'fiddle'
libm = Fiddle.dlopen('libm.so.6')
sqrt = Fiddle::Function.new(
  libm['sqrt'],
  [Fiddle::TYPE_DOUBLE],
  Fiddle::TYPE_DOUBLE
)
puts sqrt.call(2.0)

Every example above calls the exact same symbol: sqrt. Same binary. Same calling convention. Same ABI. This is the miracle of the C ABI — five different languages, one contract, 50 years of stability.

Why the C ABI Is So Durable

C has no generics. No overloading. No exceptions that unwind the stack. No virtual dispatch tables. No namespaces. No templates. No RAII.

Every one of those features, if included, would complicate the ABI. C++ has all of them, and its ABI is correspondingly complex — so complex that “the C++ ABI” is really a set of platform-specific implementations (Itanium, MSVC, ARM64EC) that don’t interoperate with each other, let alone with other languages.

C’s poverty of features is its ABI’s wealth. Because the language does so little, the binary interface does only the minimum: pack arguments into registers, jump to the function, pack the return into a register, jump back. That minimum has remained unchanged since 1972 in spirit and since the late 1980s in practice. A C library compiled in 1990 can still be called by Rust written in 2026 as long as the architecture matches.

This is not an accident. It is the single most important reason C survives — and why every new systems language, no matter how ambitious, must speak C at its boundary.
