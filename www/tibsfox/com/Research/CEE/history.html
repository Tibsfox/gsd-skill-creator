History & Culture

Dennis Ritchie, Ken Thompson, Bell Labs, K&R, ANSI, and the Language That Rewrote Unix
Overview Core libc Compilers Systems ABI Rosetta Learn History
More than half a century after its creation in a back room at Bell Labs, C remains the bedrock on which nearly all modern computing rests. Every operating system kernel in widespread use is written in C or a close descendant. Every mainstream language runtime — CPython, Ruby’s MRI, PHP, Perl, Lua, V8’s base, the JVM’s HotSpot core — is itself a C program. When a Python program calls NumPy, which calls BLAS, which calls a vendor kernel, which calls the Linux system call interface, it is C calling conventions all the way down. C is not merely old. C is structural.

Dennis Ritchie — The Quiet Architect

Born September 9, 1941, Bronxville, New York. Grew up in New Jersey after his father Alistair E. Ritchie joined Bell Labs as a switching systems engineer. Alistair co-authored The Design of Switching Circuits (1951). Dennis grew up steeped in the Bell Labs tradition before he ever walked through its doors.

Harvard: Physics undergraduate 1963, PhD in applied mathematics 1968. Thesis: Program Structure and Computational Complexity — subrecursive hierarchies, pure theoretical CS in the Turing/Kleene lineage. Famously never formally received his PhD because he declined to pay the fee to have the thesis bound and deposited in the Harvard library.

Bell Labs, 1967: Computing Sciences Research Center in Murray Hill, NJ. Joined the Multics group. When Bell pulled out of Multics in 1969, he and Ken Thompson turned to a smaller system on a cast-off PDP-7 — Unix. Ritchie was the systems thinker: Thompson was the virtuoso hacker. Ritchie saw that Unix needed a language neither assembly nor academic, but something in between.

The comparison was made, and keeps being made: Jobs sold devices that ran software; Ritchie wrote the software that runs the devices. Every iPhone Steve Jobs ever sold was running a kernel descended from Unix and written in C.

Ken Thompson — The Hacker Who Built the Universe

Born February 4, 1943, New Orleans. BS and MS in electrical engineering and CS from UC Berkeley. Joined Bell Labs in 1966.

Thompson’s great insight was that Multics was too big but the idea was too good to abandon. In 1969, with his wife and infant son temporarily visiting family in California — the famous “three weeks of uninterrupted hacking” story — Thompson wrote the first version of Unix on a cast-off DEC PDP-7: kernel, shell, editor, assembler. All in PDP-7 assembly.

To bootstrap himself into a higher-level language, Thompson created B in 1969 — a stripped-down BCPL, typeless, tiny footprint. When Bell acquired a PDP-11 in 1970, B’s word-oriented assumptions were a poor fit for the new byte-addressable machine. Ritchie attacked the problem; that attack became C. Thompson’s attention moved on.

Thompson and Ritchie were complementary. Ritchie was the thinker who loved a precise specification; Thompson was the builder who loved a working system. Elegance pulled taut against pragmatism — that tension is the tension you can still feel in C itself.

The Lineage — CPL → BCPL → B → NB → C

C is the last link in a chain of languages each of which was an attempt to make the previous one smaller, simpler, and more implementable.

LanguageYearCreatorWhat
CPL~1963Christopher Strachey et al. (Cambridge + University of London)Combined Programming Language. ALGOL 60 block structure + systems programming. Too large, never fully implemented
BCPL1967Martin Richards (Cambridge Mathematical Laboratory)Basic CPL. Typeless, curly braces (actually $(/$)), portable via O-code intermediate
B1969Ken ThompsonBCPL stripped for the PDP-7. Typeless. main(), printf, i++, while, if/else all already looked like C
NB (New B)~1971Dennis RitchieAdded types (int, char, pointers) and struct-like composites
C1972Dennis RitchieNB crystallized: proper pointers with type info, preprocessor, richer structs, self-bootstrapping compiler

The name “C” is, in the direct and unpretentious Bell Labs style, simply the letter after B. There was never a “language A” at Bell Labs.

The Unix Symbiosis

You cannot tell C’s story without Unix’s, because they grew up together and each made the other possible.

  • Mid-1972: First attempt to rewrite Unix in C. Failed — C didn’t yet have structs
  • 1972: Ritchie adds structs. Rewrite succeeds
  • Late 1973: Unix V4 is the first kernel written mostly in C (with some assembly)
  • June 1974: Unix V5 shipped outside Bell Labs, with C kernel. Ritchie & Thompson’s CACM paper “The UNIX Time-Sharing System” describes it to the world
  • 1977: Steve Johnson and Ritchie port Unix to the Interdata 8/32. The first major port. This established the pattern: write the OS in C, port the C compiler, everything else follows

The decision to write Unix in C was, at the time, considered mildly eccentric. The conventional wisdom was that OS kernels had to be in assembly. Ritchie’s bet was that a language close enough to the metal — with pointers, bitwise operators, no hidden runtime — could be almost as fast as assembly while being dramatically easier to port. He was right, and the consequences were enormous.

The Bidirectional Symbiosis

DirectionEffect
Unix → CEvery interesting machine wanted Unix. Running Unix meant having a C compiler. C became available on every interesting machine
C → UnixA Unix port reduced to a C-compiler port — a tractable problem for a small team. Dozens of ports followed through the 1980s

K&R — The White Book

1978, Prentice-Hall: The C Programming Language by Brian W. Kernighan and Dennis M. Ritchie. White cover, blue Helvetica. 228 pages. One of the most influential computer books ever written.

Brian Kernighan was not a co-designer of C. He was a Bell Labs colleague — Princeton-trained CS, unusually gifted writer, co-creator of AWK (the “k”), author of Software Tools (with P.J. Plauger). Kernighan wrote most of the prose. Ritchie wrote the reference manual appendix. The partnership produced a book that is simultaneously a tutorial, a reference, and an essay on programming taste.

WG14 — The C Committee

ISO/IEC JTC1/SC22/WG14 is the international working group that maintains C. It works alongside ANSI’s INCITS J11 (the U.S. technical advisory group). The committee meets twice a year, hosts mailing list discussions year-round, and publishes papers via its public document repository (WG14 N-documents). Recent chairs include David Keaton. The process is slower than almost any other language committee — and deliberately so. C changes don’t get rolled back. A feature added in C99 must still compile in C23 and beyond.

The Memory Safety Debate

February 2024: The White House ONCD report. The Office of the National Cyber Director published “Back to the Building Blocks: A Path Toward Secure and Measurable Software,” explicitly calling for adoption of memory-safe languages (Rust, Go, C#, Swift, Java, Python) in new code. The report cited Microsoft and Google research showing that ~70% of security vulnerabilities in their C/C++ codebases were memory-safety bugs.

The report did not say “stop writing C.” It said “prefer memory-safe where possible.” But it was the loudest official voice ever raised against C’s memory model.

The C/C++ Community’s Responses

ResponseApproach
Rust in the Linux kernelKernel 6.1 (Dec 2022) accepted the first Rust code. “Rust for Linux” led by Miguel Ojeda. Several production drivers in Rust by 2026

Fil-CFilip Pizlo. Fully memory-safe C compiler (2024+). Runtime checks everything

The Bet

C and Rust coexist for decades. The Linux kernel will be mostly C for the foreseeable future. Embedded systems will stay C. The C ABI will remain the universal FFI target. But new code for networked services, browsers, and security-critical components is increasingly Rust. This is not a replacement — it’s a layering.

Dennis Ritchie would have understood. He added restrict in C99 because aliasing was costing optimization. He would have watched Rust’s borrow checker with curiosity and probably some small fondness, noting that the problem Rust solves is precisely the problem he left unsolved in 1972 because solving it would have meant not shipping.
