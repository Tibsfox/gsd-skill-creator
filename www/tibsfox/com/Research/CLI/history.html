$ The shell as we know it did not arrive fully formed. It is the product of a lineage that begins with mechanical teletypes clattering at 110 baud, passes through the phosphor glow of DEC's video terminals, and arrives at today's GPU-rendered terminal emulators running shells that manipulate structured data as fluently as plain text.

1. Teletypes: Where “TTY” Comes From

Before computers had screens, they had teletypes. The Teletype Corporation manufactured electromechanical teleprinters from the 1920s onward, originally for telegraph and Telex networks. When the earliest interactive computers needed a human interface, the teletype was the obvious choice: a mature, mass-produced device for converting between text and electrical signals.

The Teletype Model 33 (ASR-33), introduced in 1963, became the canonical computer terminal:

SpecificationValue
Speed10 characters/second (110 baud)
Character setASCII (7-bit, 128 characters)
Paper width8.5 inches (80 columns at 10 cpi)
Weight~35 pounds
Cost (1963)~$750 ($7,500 in 2025 dollars)

The ASR-33 was the device Ken Thompson and Dennis Ritchie used when they created Unix on the PDP-7 in 1969. Its physical characteristics shaped Unix in ways that persist today:

Short, lowercase commands (ls, cp, rm, cat) reflect a world where every character was physically struck by a mechanical typehead.

CR+LF. The carriage physically returned (CR, ASCII 13) and paper advanced (LF, ASCII 10). This is why Windows uses \r\n and Unix uses \n — Unix assumed a driver that handled the translation.

No backspace. Paper couldn’t unprint. The # character was Unix’s original erase character, @ the kill character — conventions predating video terminals.

The abbreviation tty entered Unix’s DNA from the start. Even in 2026, on a machine that has never been near a teletype, tty returns the pseudo-terminal device name: /dev/pts/3. The kernel’s terminal subsystem still implements line discipline, echo, and canonical-mode editing because the abstraction was designed for teletypes, and the abstraction survived.

2. Video Terminals: DEC’s Empire

The fundamental limitation of the teletype: it could only append. The video display terminal replaced paper with a CRT screen, enabling cursor movement, screen clearing, and full-screen applications.

VT52 (1975)

DEC’s first widely adopted video terminal. 24 lines, 80 columns. Supported cursor addressing via escape sequences (ESC A = cursor up, ESC H = cursor home). Established the pattern of ESC as a control introducer.

VT100 (1978) — The Standard

SpecificationValue
Display24 lines × 80 columns (132-column mode available)
Screen12″ monochrome CRT (green or amber phosphor)
ProcessorIntel 8080
Escape sequencesANSI X3.64 compliant
CommunicationRS-232, up to 19,200 baud
Price (1978)~$1,800 ($8,500 in 2025 dollars)

The VT100 was the most influential terminal ever manufactured. It was the first widely available terminal to implement ANSI X3.64 escape codes. Because the VT100 was so popular, every other manufacturer had to be “VT100 compatible,” and every terminal emulator written since implements VT100 as its baseline. The 80×24 geometry became the de facto standard — stty size on a fresh terminal: 24 80.

Other notable terminals: Wyse 50/60 (corporate standard), Televideo 925/950 (business), IBM 3270 (block-mode mainframe terminal), Heathkit H19 (VT52-compatible kit terminal).

3. The 80-Column Standard

1928: IBM’s 80-column punch card (Herman Hollerith’s revised format) — 12 rows × 80 columns, one character per column. Billions manufactured. Buildings designed around them.

1978: The VT100 defaulted to 80 columns. Programs expecting 80-column card input got 80-column terminal output. FORTRAN assumed 80 columns (1-5 labels, 6 continuation, 7-72 code, 73-80 sequence). COBOL used all 80.

2026: PEP 8 recommends 79 characters. The Linux kernel style guide recommends 80. Go’s gofmt naturally produces short lines. The standard relaxes to 100-120 in many guides, but 80 remains the gravitational center.

4. The Shell Lineage

Pre-Unix: CTSS and Multics (1963-1965)

CTSS RUNCOM (1963) at MIT — the concept of executing stored command sequences. The abbreviation rc (“run commands”) is the direct ancestor of .bashrc.

Louis Pouzin coined “shell” for the Multics command interpreter: a program that surrounds the kernel like a shell surrounds a nut. The word, and the concept, followed Ken Thompson from Multics to Unix.

Thompson Shell (1971)

Ken Thompson’s /bin/sh for V1 Unix. ~900 lines of C. Supported I/O redirection (>, <), background execution (&), and — after Doug McIlroy’s insistence — pipes (|, added in V3 Unix, 1973). No variables, no control flow, no scripting. Just commands and pipes.

Bourne Shell (1979)

Stephen Bourne, British computer scientist (PhD from Cambridge on ALGOL 68), at Bell Labs. V7 Unix. The first real scripting language: variables ($HOME), control flow (if/fi, case/esac, for/do/done), here-documents, trap, command substitution. The reversed-keyword delimiters reflected his ALGOL background. Every #!/bin/sh script written in 45 years traces back to Bourne’s design.

C Shell (1978)

Bill Joy at UC Berkeley (2BSD). Designed for interactive use: history substitution (!!), aliases, job control (Ctrl-Z, bg, fg). C-like scripting syntax was widely criticized — Tom Christiansen’s “Csh Programming Considered Harmful” (1995) is gospel: use csh interactively if you like; write scripts in sh.

KornShell (1983)

David Korn at Bell Labs. Combined Bourne’s scripting with csh’s interactive features. First Bourne-compatible shell with command-line editing (emacs and vi modes), coprocesses (|&), associative arrays (ksh93), arithmetic ($(( ))). Default on commercial Unix (AIX, HP-UX, Solaris). POSIX shell standard is essentially “ksh88 minus the ksh-specific parts.”

Bash (1989)

Brian Fox for the GNU Project, at Richard Stallman’s request. Written from scratch (no AT&T code). Chet Ramey became sole maintainer in 1994 — one of the longest single-maintainer tenures in free software. POSIX.2 compliant with GNU extensions. Readline, programmable completion, arrays, process substitution. Default on most Linux distributions. The shell you can count on being present.

Zsh (1990)

Paul Falstad at Princeton. Named after Zhong Shao’s login name. The maximalist shell: extended globbing with qualifiers, spell correction, loadable modules, themeable prompts. oh-my-zsh (Robby Russell, 2009) made it mainstream. Powerlevel10k (Roman Perepelitsa, 2019) pushed prompt customization to its limit. macOS default since Catalina (2019).

Fish (2005)

Axel Liljencrantz. “Finally, a command line shell for the 90s.” Syntax highlighting and autosuggestions out of the box. Web-based configuration. Deliberately not POSIX — the philosophy is that POSIX syntax is a historical accident, not a design ideal. Rewritten in Rust for v4.0 (2024). User-friendly heresy.

PowerShell (2006)

Jeffrey Snover at Microsoft. The Monad Manifesto (2002) proposed an object pipeline instead of text. .NET integration. Cross-platform since 2016. A fundamentally different philosophy: structured data instead of text streams.

Modern Renaissance (2020s)

Nushell (structured data tables), Oil/Oils (fixing bash’s syntax), Elvish (functional shell), Xonsh (Python + shell hybrid), Murex (typed pipelines). The shell is not done evolving.

5. ANSI Escape Codes

The ANSI X3.64 standard (1979, later ECMA-48) formalized escape sequences for terminal control. Every sequence begins with ESC [ (the Control Sequence Introducer, CSI):

ESC[1m     -- bold
ESC[31m    -- red foreground
ESC[42m    -- green background
ESC[0m     -- reset all attributes
ESC[2J     -- clear entire screen
ESC[H      -- cursor to home position
ESC[10;20H -- cursor to row 10, column 20

SGR (Select Graphic Rendition) controls bold, italic, underline, strikethrough, blink, reverse video, and colors. The VT100 implemented the standard; every terminal emulator since has followed.

6. termcap → terminfo → ncurses

Not all terminals spoke the same escape dialect. The solution was a database of terminal capabilities:

SystemAuthorYearFormat
termcapBill Joy (UC Berkeley)1978Text file, capability names
terminfoMark Horton (AT&T)1981Compiled binary, faster lookup
ncursesThomas Dickey1993Library wrapping terminfo

The TERM environment variable tells the curses layer which capabilities to assume. TERM=xterm-256color means: xterm-compatible sequences, 256 colors. Setting it wrong causes visual glitches. This database-of-capabilities pattern — where programs ask “can this terminal do X?” instead of assuming — is why terminal applications work across vastly different environments.
