"Data is the new oil, but storage is the refinery." Storage decisions are among the hardest to reverse — the filesystem you choose on day one may be the filesystem you maintain for a decade. Understanding the trade-offs at a systems level is essential.

1. Storage Types

1.1 Block Storage

Block storage presents raw volumes divided into fixed-size blocks (typically 512 bytes or 4 KiB). The storage system knows nothing about the data — the host is responsible for writing a filesystem on top. Block storage provides the lowest latency and highest IOPS, making it the standard for databases, virtual machines, and boot volumes.

Protocols:

  • iSCSI (Internet SCSI): Encapsulates SCSI commands over TCP/IP. Runs on standard Ethernet, no special hardware required. Widely used for VMware datastores and shared storage in SMB environments. Use jumbo frames (MTU 9000) and dedicated VLAN for performance. Multipath I/O (MPIO) provides redundancy across multiple network paths.
  • Fibre Channel (FC): Dedicated storage network using 16/32/64 Gbps links. Low latency (~1ms), lossless fabric, zoning for access control. Expensive but proven for enterprise databases. FC-NVMe extends NVMe over FC fabrics.
  • NVMe-oF (NVMe over Fabrics): Extends the NVMe command set over RDMA (RoCE, InfiniBand) or TCP. Delivers near-local NVMe latency (~20-50 microseconds) over the network. The future of high-performance networked storage. Linux supports NVMe/TCP natively since kernel 5.0.

1.2 File Storage

File storage presents a filesystem hierarchy over the network. The server handles filesystem operations; clients mount and use it like a local directory.

  • NFS (Network File System): The Unix/Linux standard. NFSv3 is stateless (relies on NLM for locking, uses multiple UDP/TCP ports requiring complex firewall rules). NFSv4 (RFC 7530) modernizes everything: a single TCP port (2049), Kerberos authentication (sec=krb5p for encryption), stateful operation with lock delegation, and compound operations (multiple ops per RPC for latency reduction). NFSv4.1 adds sessions and pNFS (parallel NFS, striping data across multiple storage servers for bandwidth scaling). NFSv4.2 adds server-side copy (copy_file_range()), hole punching, application I/O advise, and labeled NFS (SELinux contexts). Performance tuning: adjust rsize/wsize to 1 MiB for modern networks, enable async on the export cautiously (better write performance but data loss risk on server crash), use nconnect=8 (kernel 5.3+) for parallel TCP connections per mount, and consider actimeo=0 for cache-coherent workloads at the cost of increased metadata traffic.
  • SMB/CIFS: Windows-native file sharing. SMB3 (used in Windows 10/11 and Samba 4.x) adds encryption, multichannel (multiple NICs aggregated transparently), transparent failover (clients reconnect seamlessly after server failover), directory leasing (client-side caching of directory metadata), and RDMA support (SMB Direct for lossless, low-latency transfers). Samba on Linux provides full interoperability with Active Directory including domain membership, Group Policy, and DFS (Distributed File System) namespace support.

1.3 Object Storage

Object storage uses a flat namespace of buckets containing objects (data + metadata + unique ID). No directory hierarchy — objects are retrieved by key. Designed for massive scale, not low latency.

  • S3 API: Amazon's Simple Storage Service defined the de facto standard API (GET, PUT, DELETE, LIST). Now supported by dozens of providers and self-hosted solutions.
  • MinIO: S3-compatible object storage you run yourself. Single binary, supports erasure coding, encryption, versioning, and replication. Can serve as the backend for application storage, backup targets (restic, borgmatic), and data lake infrastructure.
  • Ceph RGW: Ceph's RADOS Gateway provides S3 and Swift-compatible object storage on top of a Ceph cluster. Suitable for multi-petabyte deployments with self-healing and replication.

2. Filesystem Deep Dive

2.1 ext4

ext4 is the default Linux filesystem since 2008 (kernel 2.6.28) and remains the most widely deployed. It is the safe, boring choice — which is exactly what you want for boot partitions, small-to-medium data volumes, and general-purpose use.

Key features: Journaling (metadata-only by default, data=ordered), extents (contiguous block allocation replacing indirect block maps), delayed allocation (batches writes for better layout), online resize (grow only), directory indexing (htree for O(log n) lookups).

Journaling modes explained: ext4 supports three journaling modes via the data= mount option. data=journal writes both metadata and data to the journal before committing to disk — safest but slowest (roughly 50% write penalty). data=ordered (the default) writes data to disk before committing metadata to the journal — ensures no stale data exposure after a crash. data=writeback only journals metadata; data may be written in any order — fastest but after a crash, recently written files may contain garbage data from previous files. For database volumes, data=ordered is the standard; only switch to data=writeback if the application (like PostgreSQL or MySQL) manages its own write ordering via fsync.

Limits: Maximum filesystem size: 1 EiB. Maximum file size: 16 TiB. Maximum filename: 255 bytes. No built-in checksumming of data blocks (silent bit-rot goes undetected — this is the primary reason for preferring ZFS or Btrfs on long-lived data), no snapshots, no transparent compression, no deduplication. The metadata_csum feature (enabled by default since e2fsprogs 1.44) adds CRC32c checksums to metadata structures, catching some corruption but not data-level bit-rot.

# Create ext4 filesystem with labels and features
mkfs.ext4 -L data -O metadata_csum,64bit /dev/sdb1

# Tune existing filesystem
tune2fs -m 1 /dev/sdb1          # Reduce reserved blocks from 5% to 1%
tune2fs -c 0 -i 0 /dev/sdb1    # Disable fsck count/interval (use fstrim instead)

# Check filesystem integrity
e2fsck -f /dev/sdb1             # Force check (requires unmount)

2.2 XFS

XFS is the default on RHEL 9, Rocky Linux, and AlmaLinux. Designed by SGI for IRIX, it excels at parallel I/O and large files. XFS uses allocation groups — independent sub-volumes that allow multiple threads to allocate space simultaneously without lock contention.

Key features: Online growth (but not shrink), delayed allocation, extent-based allocation, reflink support (since kernel 4.9 — enables instant file copies and efficient deduplication), guaranteed rate I/O (for media streaming), online defragmentation (xfs_fss).

Performance: In benchmarks with 1 billion files (George Mason University, 2024), XFS handled the load without any setup changes. XFS leads in SQLite writes and random reads. It is the go-to filesystem for large file workloads, parallel I/O (think multiple database instances or media servers), and enterprise Linux deployments.

# Create XFS with reflink enabled (default on RHEL 9)
mkfs.xfs -m reflink=1 -L data /dev/sdb1

# Grow online (while mounted)
xfs_growfs /mount/point

# Filesystem repair
xfs_repair /dev/sdb1            # Requires unmount

# Defragment a mounted filesystem
xfs_fsr /mount/point

2.3 Btrfs

Btrfs (B-tree filesystem) is a copy-on-write (CoW) filesystem designed to address the limitations of ext4. It is the default on openSUSE, Fedora Workstation, and used by Synology NAS. Facebook/Meta runs Btrfs at scale on its production servers.

Key features: Snapshots (instant, space-efficient CoW snapshots), built-in checksumming (CRC32c or xxhash for data and metadata), subvolumes (independent filesystem trees sharing the same pool), transparent compression (zstd, lzo, zlib), online balance and resize (grow and shrink), send/receive (incremental backup via snapshot diff).

Limitations: RAID 5/6 support remains experimental and data-losing in failure scenarios — never use it in production. Btrfs ranked last or second-to-last in four benchmark workloads on Linux 6.11 (2024). Write performance suffers under heavy random workloads due to CoW amplification. Best suited for snapshot-heavy workloads, home servers, backup targets, and environments where data integrity matters more than raw write speed.

# Create Btrfs filesystem
mkfs.btrfs -L data /dev/sdb1

# Create subvolumes (common layout for snapshots)
btrfs subvolume create /mnt/@
btrfs subvolume create /mnt/@home
btrfs subvolume create /mnt/@snapshots

# Mount with compression
mount -o compress=zstd:3,subvol=@ /dev/sdb1 /

# Take a read-only snapshot
btrfs subvolume snapshot -r /mnt/@ /mnt/@snapshots/2026-04-08

# Send/receive for incremental backup
btrfs send /mnt/@snapshots/2026-04-08 | btrfs receive /backup/

# Scrub (verify checksums, detect corruption)
btrfs scrub start /mnt
btrfs scrub status /mnt

2.4 ZFS

ZFS, originally developed by Sun Microsystems, is the most feature-complete filesystem/volume manager available. It combines filesystem, volume management, RAID, caching, compression, checksumming, and replication into a single integrated system. On Linux, it runs via OpenZFS (kernel module, not in mainline due to CDDL licensing).

Architecture:

  • vdevs (virtual devices): The building blocks. A vdev can be a single disk, mirror, raidz (1-3 parity), or special allocation class. You cannot add disks to an existing vdev (unlike Btrfs), but you can add new vdevs to a pool.
  • Pools: Aggregate one or more vdevs into a storage pool. All datasets in the pool share the total capacity.
  • Datasets: Filesystems or zvols (block devices) within a pool. Each dataset inherits pool properties but can override them (compression, quota, recordsize, mountpoint).

Caching layers:

  • ARC (Adaptive Replacement Cache): In-memory read cache, typically uses 50% of system RAM. Combines recency and frequency to outperform simple LRU.
  • L2ARC: Second-level read cache on a fast SSD. Extends ARC for workloads that exceed RAM. Use NVMe for L2ARC; the benefit with SATA SSDs is marginal.
  • SLOG (Separate Intent Log): Accelerates synchronous writes by placing the ZFS Intent Log (ZIL) on a fast, durable device (Optane, high-endurance NVMe). Critical for NFS servers (NFSv3 sync writes) and database workloads.

Key features: End-to-end checksumming (every block, detected on read via scrub), transparent compression (LZ4 default — faster than Btrfs's zlib, also supports zstd), native encryption (AES-256-GCM), send/receive (incremental replication), bookmarks (reference point without snapshot overhead), special allocation class (metadata and small blocks on fast SSDs).

# Create a mirrored pool
zpool create tank mirror /dev/sda /dev/sdb

# Create a raidz2 pool (double parity, equivalent to RAID 6)
zpool create datapool raidz2 /dev/sd{a,b,c,d,e,f}

# Add an SLOG and L2ARC
zpool add tank log mirror /dev/nvme0n1p1 /dev/nvme1n1p1
zpool add tank cache /dev/nvme2n1

# Create a dataset with compression and quota
zfs create -o compression=lz4 -o quota=500G -o recordsize=1M tank/backups

# Create a dataset optimized for PostgreSQL
zfs create -o recordsize=8K -o compression=lz4 -o atime=off \
    -o primarycache=metadata -o logbias=latency tank/pgdata

# Snapshot and send/receive
zfs snapshot tank/data@2026-04-08
zfs send tank/data@2026-04-08 | ssh backup-server zfs receive pool/data

# Incremental send (only changes since last snapshot)
zfs send -i tank/data@2026-04-07 tank/data@2026-04-08 | \
    ssh backup-server zfs receive pool/data

# Scrub (verify all checksums — run monthly)
zpool scrub tank
zpool status tank   # Check scrub progress and error counts

3.2 The Write Hole Problem

In RAID 5/6, a power failure during a write can leave data and parity blocks inconsistent. On restart, the array does not know which stripe was being written — this is the write hole. Mitigations: battery-backed write cache (hardware RAID), write-intent bitmap (mdadm), or use a CoW filesystem (ZFS/Btrfs) that avoids in-place updates entirely.

3.3 RAID Is Not Backup

RAID protects against disk failure. It does not protect against: accidental deletion, ransomware encryption, filesystem corruption, software bugs writing bad data, fire/flood destroying the server, or controller failure corrupting the entire array. You need backup in addition to RAID, always.

3.4 Rebuild Times and URE Risk

Modern disks have an Unrecoverable Read Error (URE) rate of approximately 1 in 1014 bits (1 per 12.5 TB read) for enterprise drives, and 1 in 1015 for better-grade drives. During a RAID 5 rebuild, the entire remaining array is read. With six 20 TB drives in RAID 5, rebuilding reads ~100 TB — statistically likely to encounter a URE, causing the rebuild to fail and the array to be lost. This is why RAID 5 with large disks is considered dangerous. Use RAID 6, RAID 10, or ZFS raidz2/raidz3 for arrays with large capacity drives.

Rebuild times: a 20 TB disk at 200 MB/s sustained reads takes approximately 28 hours to rebuild. During this window, performance is degraded and another disk failure means data loss (for RAID 5).

3.5 Software RAID with mdadm

# Create a RAID 10 array
mdadm --create /dev/md0 --level=10 --raid-devices=4 /dev/sd{a,b,c,d}

# Monitor the array
cat /proc/mdstat
mdadm --detail /dev/md0

# Replace a failed disk
mdadm /dev/md0 --fail /dev/sdb --remove /dev/sdb
# (physically replace the disk)
mdadm /dev/md0 --add /dev/sde

# Save configuration
mdadm --detail --scan >> /etc/mdadm/mdadm.conf

# Enable write-intent bitmap (reduces resync time after unclean shutdown)
mdadm --grow /dev/md0 --bitmap=internal

3.6 ZFS and Btrfs RAID

ZFS raidz (single parity), raidz2 (double), and raidz3 (triple) integrate parity into the filesystem layer, eliminating the write hole problem through CoW semantics. ZFS also supports dRAID (distributed RAID) for faster rebuilds on large pools. Btrfs RAID 1 and RAID 10 are production-ready; RAID 5/6 remain experimental and should never be used for data you care about.

4. LVM

4.1 Architecture: PV, VG, LV

The Logical Volume Manager (LVM) provides a layer of abstraction between physical disks and filesystems, enabling flexible volume management without repartitioning.

  • Physical Volumes (PV): A partition or whole disk initialized for LVM. pvcreate /dev/sdb
  • Volume Groups (VG): One or more PVs pooled together. vgcreate data_vg /dev/sdb /dev/sdc
  • Logical Volumes (LV): Virtual partitions carved from a VG. lvcreate -L 100G -n db_lv data_vg

The key advantage: logical volumes can be resized, moved, and snapshotted without unmounting (for supported filesystems). Multiple physical disks appear as a single pool of storage.

4.2 Snapshots (CoW)

LVM snapshots use copy-on-write: when a block in the origin volume is overwritten, the original block is first copied to the snapshot volume. This means the snapshot only consumes space for changed blocks. Use snapshots for: pre-upgrade backups, consistent database dumps (snapshot, mount snapshot, dump, remove snapshot), and testing configuration changes.

# Create a snapshot (allocate space for changed blocks)
lvcreate --size 20G --snapshot --name db_snap /dev/data_vg/db_lv

# Mount the snapshot read-only
mount -o ro /dev/data_vg/db_snap /mnt/snap

# After use, remove the snapshot
lvremove /dev/data_vg/db_snap

Warning: If a snapshot fills up (more data changed than allocated snapshot space), it becomes invalid and must be removed. Monitor with lvs (check the Snap% column). Size snapshots generously for write-heavy workloads.

4.3 Thin Provisioning

LVM thin provisioning allows creating logical volumes larger than the physically available space. Blocks are allocated on first write, not at creation time. A thin pool consists of a data LV and a metadata LV:

# Create a thin pool (100G data, auto-sized metadata)
lvcreate --type thin-pool --size 100G --name thin_pool data_vg

# Create a thin volume (virtual size 500G, backed by the 100G pool)
lvcreate --type thin --virtualsize 500G --name vm_disk --thinpool thin_pool data_vg

# Monitor thin pool usage
lvs -o+lv_size,pool_lv,data_percent,metadata_percent data_vg

Auto-extension: Configure the LVM daemon to automatically extend thin pools before they fill. In /etc/lvm/lvm.conf, set thin_pool_autoextend_threshold = 70 and thin_pool_autoextend_percent = 20. When the pool reaches 70% usage, it expands by 20%. Ensure the VG has free space for this to work, and always monitor — a full thin pool with no room to extend corrupts all thin volumes.

4.4 LVM + LUKS Encryption

The standard Linux encryption stack layers LUKS (Linux Unified Key Setup) on top of a partition or LV, then optionally puts LVM on top of the encrypted layer for flexible volume management:

# Encrypt a partition
cryptsetup luksFormat /dev/sdb1
cryptsetup luksOpen /dev/sdb1 crypt_data

# Create LVM on top of encrypted volume
pvcreate /dev/mapper/crypt_data
vgcreate secure_vg /dev/mapper/crypt_data
lvcreate -L 50G -n secrets_lv secure_vg

4.5 LVM Cache (dm-cache)

dm-cache uses a fast SSD as a transparent cache for a slow HDD logical volume. Write-back mode caches writes on the SSD for maximum performance; write-through writes to both SSD and HDD for safety.

# Assume slow_lv exists on spinning disks, fast NVMe is /dev/nvme0n1
pvcreate /dev/nvme0n1
vgextend data_vg /dev/nvme0n1

# Convert to cached LV (writeback mode, 50G cache)
lvcreate --type cache --cachemode writeback -L 50G \
    -n cache_nvme data_vg/slow_lv /dev/nvme0n1

4.6 Practical Workflows

Adding a disk to an existing VG:

pvcreate /dev/sdd
vgextend data_vg /dev/sdd
lvextend -l +100%FREE /dev/data_vg/data_lv
# For ext4:
resize2fs /dev/data_vg/data_lv
# For XFS:
xfs_growfs /mount/point

Migrating data between disks (online):

# Add new disk to VG
pvcreate /dev/sde
vgextend data_vg /dev/sde

# Move all extents from old disk to new disk (online, no downtime)
pvmove /dev/sdb /dev/sde

# Remove old disk from VG
vgreduce data_vg /dev/sdb
pvremove /dev/sdb

5. Backup Strategies

5.1 The 3-2-1 Rule (and Extensions)

The foundational backup rule: 3 copies of data, on 2 different media types, with 1 offsite copy. The modern extension adds: 3-2-1-1-0 — the extra 1 is an air-gapped or immutable copy (protection against ransomware), and 0 means zero errors in backup verification.

Example implementation for a production database server:

  1. Production data on ZFS raidz2 (copy 1)
  2. Hourly ZFS snapshots with zfs send to a local backup server (copy 2, different media)
  3. Daily restic backup to S3 with object lock / immutability (copy 3, offsite + immutable)

5.2 Full, Incremental, and Differential

TypeBacks UpRestore SpeedStorageBackup Speed

DifferentialChanges since last fullModerate (full + latest differential)ModerateModerate

Grandfather-Father-Son (GFS) rotation: Daily backups retained for a week (son), weekly backups for a month (father), monthly backups for a year (grandfather). This balances granularity for recent recovery with long-term retention without unbounded storage growth.

5.3 Backup Tool Comparison

rsync: Not a backup tool per se, but the foundation of many backup scripts. rsync -avz --delete creates a mirror. Combine with --link-dest for space-efficient incremental snapshots (hardlinks to unchanged files). Simple, transparent, but no encryption, deduplication, or verification.

BorgBackup: Deduplicating, encrypting backup. Uses content-defined chunking for efficient deduplication across backups. Produces 10-20% smaller repositories than alternatives thanks to its mature chunking algorithm and wider compression options. Requires Borg installed on both client and repository server. Key management requires explicit export/backup of the repository key — losing both the key and passphrase means permanent data loss.

# Initialize an encrypted repository
borg init --encryption=repokey-blake2 /backup/borg-repo

# Create a backup with compression
borg create --compression zstd,6 --stats --progress \
    /backup/borg-repo::'{hostname}-{now:%Y-%m-%d_%H:%M}' \
    /etc /home /var/lib/postgresql

# Prune old backups (keep 7 daily, 4 weekly, 6 monthly)
borg prune --keep-daily=7 --keep-weekly=4 --keep-monthly=6 /backup/borg-repo

# Verify backup integrity
borg check --verify-data /backup/borg-repo

# CRITICAL: Export and safely store the key
borg key export /backup/borg-repo /safe/location/borg-key.txt

Restic: Single static binary, native S3/SFTP/REST backend support, simpler key management (password-derived keys, no separate key file). Faster backup creation (~112 seconds vs Borg's ~211 seconds in benchmarks) but produces slightly larger repositories. The better choice for cloud-first and mixed-environment deployments.

# Initialize a repository on S3
export AWS_ACCESS_KEY_ID=...
export AWS_SECRET_ACCESS_KEY=...
restic -r s3:s3.amazonaws.com/my-backup-bucket init

# Create a backup
restic -r s3:s3.amazonaws.com/my-backup-bucket backup /etc /home /var/lib/postgresql

# Apply retention policy
restic -r s3:s3.amazonaws.com/my-backup-bucket forget \
    --keep-daily 7 --keep-weekly 4 --keep-monthly 6 --prune

# Verify backup integrity
restic -r s3:s3.amazonaws.com/my-backup-bucket check --read-data

Bacula/Bareos: Enterprise backup systems with a director (scheduler), file daemons (agents on backed-up machines), and storage daemons. Support tape, disk, and cloud targets. Bareos is the actively maintained open-source fork. Use for environments with hundreds of servers requiring centralized management, scheduling, and tape support.

5.4 Database-Consistent Backups

Backing up database files while the database is running risks inconsistency. Each database engine provides its own mechanism:

  • PostgreSQL: pg_dump (logical, single database), pg_dumpall (all databases + roles), pg_basebackup (physical, for PITR). For point-in-time recovery: continuous WAL archiving + pg_basebackup.
  • MySQL/MariaDB: mysqldump --single-transaction (InnoDB, consistent snapshot), Percona XtraBackup (hot physical backup, no locking for InnoDB), mariabackup (MariaDB's fork of XtraBackup).
  • MongoDB: mongodump (logical), filesystem snapshots during db.fsyncLock() (block writes briefly), or oplog-based PITR with Percona Backup for MongoDB.
  • Redis: BGSAVE creates an RDB snapshot in the background. Copy the RDB file while LASTSAVE confirms completion. For AOF: copy the AOF file (append-only, crash-safe).

Backup verification: Untested backups are not backups. Schedule automated restore tests: spin up a temporary instance, restore the latest backup, run validation queries, and alert on failure. A backup that cannot be restored is worthless.

6. Disaster Recovery

6.1 RTO and RPO

Recovery Time Objective (RTO): The maximum acceptable time from disaster to full service restoration. An RTO of 1 hour means the business can tolerate 1 hour of downtime. Drives the choice of DR infrastructure — shorter RTO requires more expensive hot-standby systems.

Recovery Point Objective (RPO): The maximum acceptable data loss measured in time. An RPO of 15 minutes means you can afford to lose at most 15 minutes of transactions. Drives the backup/replication frequency — shorter RPO requires more frequent or continuous replication.

Real-world examples:

SystemRTORPODR Strategy
E-commerce checkout5 minutes0 (zero data loss)Multi-region active-active, synchronous replication
Corporate email4 hours1 hourWarm standby, async replication
Development servers24 hours24 hoursDaily backups, cold restore
Regulatory archives72 hoursN/A (immutable data)Offsite tape or glacier storage

6.2 DR Site Types

Hot site: Fully operational duplicate environment with continuous data synchronization. Failover in seconds to minutes. Cost: 2x or more of production infrastructure. Suitable for Tier 1 / revenue-critical systems. Achieved via multi-region cloud deployments, database synchronous replication, and global load balancing with health checks.

Warm site: Pre-provisioned infrastructure (VM templates, reserved instances, network configuration) with frequent asynchronous replication. Failover requires spinning up services and applying recent data, taking minutes to hours. The most common enterprise DR strategy — balances cost against acceptable downtime.

Cold site: Physical space with power, cooling, and network connectivity, but no running systems. Recovery requires shipping or downloading data, installing OS and applications, and configuring everything. RTO: 24-72+ hours. Least expensive but highest recovery time.

6.3 Replication Strategies

Synchronous replication: Every write is confirmed on both primary and replica before acknowledging to the application. Zero data loss (RPO = 0) but introduces write latency proportional to the round-trip time between sites. Practical only within the same metro area (~1-5ms RTT). PostgreSQL synchronous replication, ZFS synchronous send, and DRBD Protocol C implement this.

Asynchronous replication: The primary acknowledges writes immediately; the replica catches up in the background. Data loss equals the replication lag (typically seconds to minutes). Works over any distance. PostgreSQL streaming replication (default async), MySQL binlog replication, and ZFS periodic send/receive implement this.

6.4 Cloud DR Patterns

  • Backup and restore: Cheapest. Back up to another region's S3/GCS. Recovery means provisioning infrastructure and restoring data. Hours-scale RTO.
  • Pilot light: Core infrastructure (database replica, DNS) running in the DR region. Application servers are stopped/minimal. On failover, scale up compute. Minutes-to-hours RTO.
  • Warm standby: A scaled-down but fully functional copy in the DR region. Scale up on failover. Minutes-scale RTO.
  • Multi-site active-active: Both regions serve production traffic. On failure of one region, the other absorbs all traffic. Near-zero RTO/RPO. Highest cost and operational complexity (data consistency, conflict resolution, split-brain prevention).

6.5 DR Testing

A DR plan that has not been tested will fail when needed. Testing approaches:

  • Tabletop exercise: Walk through the DR plan in a meeting. Low cost, identifies procedural gaps and missing documentation.
  • Parallel test: Bring up the DR site alongside production, validate data consistency, but do not redirect production traffic.
  • Full cutover test: Actually redirect production to the DR site for a scheduled window. The gold standard but highest risk — plan carefully.

Runbooks: Every DR plan must have a step-by-step runbook that a team member can follow under stress at 3 AM. Include: contact lists with phone numbers (not just Slack handles), escalation paths with time thresholds ("if step 3 takes more than 15 minutes, escalate to on-call manager"), credential access procedures (where passwords are stored, who has vault access, how to unseal), exact commands to run (copy-pasteable, not pseudocode), validation checks at each step ("after DNS failover, verify with dig +short @8.8.8.8 app.example.com"), expected output at each step, and rollback instructions for every step. Store runbooks in a system accessible during an outage — not exclusively in the system that is down.

DR metrics to track: Recovery Time Actual (RTA) from your last test — compare against your RTO target. Replication lag (current and p99 over the last 30 days). Backup success rate and last successful restore date. Time since last DR test. These metrics should be on a dashboard reviewed monthly.

7. Database Administration Basics

7.1 PostgreSQL

PostgreSQL is the sysadmin's database: open-source, reliable, well-documented, and ubiquitous. Key internals every sysadmin should understand:

WAL (Write-Ahead Log): PostgreSQL writes all changes to the WAL before modifying data files. This guarantees crash recovery: on startup, uncommitted WAL entries are replayed. WAL files are 16 MB segments in pg_wal/. Continuous archiving of WAL files enables Point-in-Time Recovery (PITR). Key settings:

# postgresql.conf — WAL tuning
wal_level = replica              # Required for replication and PITR
max_wal_size = 2GB               # Checkpoint target — larger = fewer checkpoints
min_wal_size = 256MB             # Minimum WAL retained
checkpoint_timeout = 15min       # Max time between checkpoints
checkpoint_completion_target = 0.9  # Spread checkpoint I/O over 90% of interval
archive_mode = on                # Enable WAL archiving
archive_command = 'cp %p /archive/%f'  # Archive WAL files for PITR

VACUUM: PostgreSQL's MVCC (Multi-Version Concurrency Control) creates new row versions on UPDATE/DELETE. Dead rows accumulate and must be cleaned by VACUUM. Autovacuum runs automatically but may fall behind on busy tables. Manual intervention:

# Check autovacuum activity and dead rows
SELECT schemaname, relname, n_dead_tup, last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC LIMIT 10;

# Aggressive manual vacuum for a bloated table
VACUUM (VERBOSE, ANALYZE) large_table;

# Full vacuum (reclaims space to OS, but locks table exclusively)
VACUUM FULL large_table;  -- Use sparingly, pg_repack preferred

Connection Pooling with PgBouncer: PostgreSQL forks a new backend process for every client connection, consuming 5-10 MB of RAM each. At 500 connections, that is 2.5-5 GB just for connection overhead. PgBouncer (a lightweight C proxy, ~2 KB overhead per connection) pools connections in three modes:

  • Session mode: One-to-one mapping, pool returns connection when client disconnects. Safe for all workloads but limited pooling benefit.
  • Transaction mode: Connection returns to pool after each transaction completes. The right default for web applications. Breaks prepared statements and session-level settings (SET, advisory locks).
  • Statement mode: Connection returns after each statement. Rarely used; breaks multi-statement transactions.
# /etc/pgbouncer/pgbouncer.ini
[databases]
myapp = host=127.0.0.1 port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432
listen_addr = *
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt
pool_mode = transaction
default_pool_size = 25
max_client_conn = 1000
server_idle_timeout = 300
log_connections = 0
log_disconnections = 0

Replication: Streaming replication sends WAL records to standby servers in real-time (synchronous or asynchronous). Logical replication (PostgreSQL 10+) replicates individual tables or subsets, enabling cross-version replication, selective sync, and multi-master patterns with conflict resolution (via BDR or Citus).

7.2 MySQL / MariaDB

InnoDB internals: InnoDB (the default storage engine) uses a buffer pool as its primary cache — set innodb_buffer_pool_size to 70-80% of available RAM on dedicated database servers. It implements MVCC through undo logs and supports row-level locking. The redo log (innodb_redo_log_capacity, replacing the old innodb_log_file_size in MySQL 8.0.30+) serves the same purpose as PostgreSQL's WAL — crash recovery by replaying committed transactions. The undo log stores old row versions for MVCC reads and rollback. Monitor buffer pool hit ratio with SHOW ENGINE INNODB STATUS — if the hit rate drops below 99%, the buffer pool is undersized for the working set.

Replication: Binary log (binlog) replication streams row-level (binlog_format=ROW, the safe default) or statement-level changes to replicas. Group Replication (MySQL 8.0+) provides virtually synchronous multi-primary with automatic conflict detection (certification-based), distributed recovery (a joining node catches up automatically), and built-in failure detection. ProxySQL sits in front of MySQL topologies to provide connection pooling, read/write split (routing SELECT to replicas, writes to the primary), query caching, query rewriting, and graceful failover handling. For MariaDB, MaxScale provides similar proxy functionality with built-in Galera Cluster and replication awareness.

7.3 Redis

Redis is an in-memory data store used for caching, sessions, queues, and real-time analytics. Persistence options:

  • RDB (Redis Database): Point-in-time snapshots at configured intervals (save 900 1 = snapshot if 1+ write in 900 seconds). Compact, fast to load, but data between snapshots is lost on crash.
  • AOF (Append Only File): Logs every write operation. Three fsync policies: always (durable but slow), everysec (recommended — at most 1 second of data loss), no (OS decides when to flush).
  • RDB + AOF: The recommended hybrid approach. AOF for durability, RDB for faster restarts and backup.

Redis Sentinel: Provides high availability through automatic failover. Sentinel processes monitor the master, detect failure (configurable quorum), elect a new master from replicas, and reconfigure clients. Minimum three Sentinel instances for quorum. Redis Cluster: Shards data across multiple masters using hash slots (16,384 slots). Each master has one or more replicas. Provides both HA and horizontal scaling, but with limitations (multi-key operations must be in the same hash slot, use hash tags {} for co-location).

8. Storage Performance

8.1 IOPS vs. Throughput vs. Latency

These three metrics are distinct and often traded against each other:

  • IOPS (Input/Output Operations Per Second): How many discrete read/write operations the storage can handle. Matters for databases (small random I/O). A 4K random read workload is IOPS-bound.
  • Throughput (MB/s or GB/s): How much data can be transferred per second. Matters for sequential workloads (video streaming, backups, large file transfers). A 1 MB sequential read workload is throughput-bound.
  • Latency (microseconds or milliseconds): Time from I/O request to completion. Determines responsiveness. p99 latency (99th percentile) matters more than average — one slow I/O in a transaction-heavy database cascades into application-visible delays.

8.2 NVMe vs. SAS vs. SATA

InterfaceMax BandwidthTypical IOPS (4K random read)Typical LatencyQueue Depth
SATA III SSD600 MB/s~100,00080-200 microseconds32

The NVMe advantage is not just speed — it is architectural. NVMe bypasses the SCSI/AHCI command layer, talks directly to the PCIe bus, supports 65,535 queues with 65,536 entries each (vs. SATA's single queue of 32), and enables the CPU to submit I/O without register-level synchronization. For latency-sensitive workloads (databases, search indices, message brokers), NVMe is transformative.

8.3 I/O Schedulers

The Linux I/O scheduler sits between the filesystem and the block device, reordering and merging I/O requests:

  • mq-deadline: Ensures no request waits beyond a deadline. Good for databases on SSDs (prevents starvation of reads by heavy writes). Default for SATA/SAS SSDs.
  • BFQ (Budget Fair Queueing): Provides fair bandwidth allocation and low latency for interactive workloads. Good for desktops and mixed workloads on HDDs/SSDs.
  • none (noop): No reordering — passes requests directly to the device. The right choice for NVMe, where the device's internal controller handles scheduling far more efficiently than the kernel can.
# Check current scheduler
cat /sys/block/nvme0n1/queue/scheduler
# Output: [none] mq-deadline

# Change scheduler (non-persistent)
echo none > /sys/block/nvme0n1/queue/scheduler

# Persistent: add to udev rules
# /etc/udev/rules.d/60-io-scheduler.rules
ACTION=="add|change", KERNEL=="nvme*", ATTR{queue/scheduler}="none"
ACTION=="add|change", KERNEL=="sd*", ATTR{queue/scheduler}="mq-deadline"

8.4 Benchmarking Tools

fio (Flexible I/O Tester) is the gold standard for storage benchmarking. It can simulate virtually any I/O workload:

# Random 4K read IOPS test (database-like workload)
fio --name=rand-read --ioengine=libaio --direct=1 --bs=4k \
    --rw=randread --numjobs=8 --iodepth=32 --size=4G \
    --runtime=60 --time_based --group_reporting --filename=/dev/nvme0n1

# Sequential 1M write throughput test (backup-like workload)
fio --name=seq-write --ioengine=libaio --direct=1 --bs=1M \
    --rw=write --numjobs=1 --iodepth=8 --size=10G \
    --runtime=60 --time_based --group_reporting --filename=/dev/nvme0n1

# Mixed 70/30 read/write (OLTP-like workload)
fio --name=mixed --ioengine=libaio --direct=1 --bs=8k \
    --rw=randrw --rwmixread=70 --numjobs=4 --iodepth=16 --size=4G \
    --runtime=120 --time_based --group_reporting --filename=/dev/nvme0n1

Important: Always test with --direct=1 to bypass the page cache (otherwise you are benchmarking RAM, not storage). Use --numjobs and --iodepth to match your expected workload concurrency. Run tests long enough (60+ seconds) for results to stabilize.

Other tools: dd for quick sequential throughput estimates (not rigorous), bonnie++ for filesystem-level benchmarks, sysbench for database-oriented I/O tests (MySQL/PostgreSQL workloads), ioping for latency measurement (the ping of storage).

8.5 Caching Layers

Multiple caching layers exist between the application and the disk platter/flash cell:

  1. Application cache: Connection pools, query caches, in-process caches (Redis, Memcached from the app's perspective).
  2. Page cache: The kernel's filesystem cache in RAM. Transparently caches reads and buffers writes. Inspect with free -h (the "buff/cache" column) and vmstat 1 (si/so for swap, bi/bo for block I/O).
  3. bcache / dm-cache / LVM cache: SSD-backed caching of HDD volumes (covered in section 4.5). bcache is a kernel module; dm-cache is managed through LVM.
  4. ZFS ARC / L2ARC: ZFS's integrated read cache (covered in section 2.4).
  5. Device write cache: The drive's own volatile write buffer. Improves write performance but risks data loss on power failure unless battery-backed. Check with hdparm -W /dev/sda; disable for databases on drives without power loss protection.

8.6 Storage Tiering

Modern infrastructure often uses multiple storage tiers:

Automated tiering (via ZFS special allocation class, Ceph's device classes, or LVM dm-cache) moves data between tiers based on access frequency. Place metadata and small blocks on NVMe, active data on SSD, and bulk/cold data on HDD. Monitor access patterns with iostat, iotop, and filesystem-specific statistics to validate tier placement.

# Monitor I/O per device with 1-second intervals
iostat -xz 1

# Key columns:
# r/s, w/s    — reads/writes per second (IOPS)
# rMB/s, wMB/s — throughput
# r_await, w_await — average latency (ms) per I/O
# %util       — device utilization (100% = saturated for single-queue devices)

# Per-process I/O monitoring
iotop -oPa
