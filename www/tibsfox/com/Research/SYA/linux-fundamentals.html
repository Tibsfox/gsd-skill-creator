Linux runs on everything from Raspberry Pis to 100% of the world’s top 500 supercomputers. For systems administrators, deep Linux knowledge is not optional — it is the foundation on which every other skill in this digest rests.

Kernel Architecture

The Linux kernel is monolithic but modular. All core services (process scheduling, memory management, filesystem drivers, network stack) run in kernel space, but functionality can be extended at runtime through loadable kernel modules (LKMs) without recompilation.

Key kernel subsystems:

  • Process scheduler: CFS (Completely Fair Scheduler) since 2.6.23, replaced by EEVDF in 6.6+
  • Memory management: Virtual memory, page tables, OOM killer, cgroups v2 memory controller
  • VFS (Virtual Filesystem Switch): Abstraction layer enabling ext4, XFS, Btrfs, NFS to coexist
  • Network stack: Netfilter (iptables/nftables), TC (traffic control), XDP/eBPF for programmable networking
  • Device model: sysfs, udev, device trees for hardware abstraction

Boot Process

Understanding the boot sequence is essential for troubleshooting startup failures:

StageComponentRole
1. FirmwareBIOS / UEFIHardware initialization, POST, locate bootloader
2. BootloaderGRUB2 / systemd-bootLoad kernel + initramfs into memory
3. Kernel initLinux kernelMount initramfs, detect hardware, mount real root
4. Init systemsystemd (PID 1)Start services in dependency order, reach target
5. Logingetty / display managerPresent login prompt or graphical login

systemd has replaced SysVinit on all major distributions. Its unit files (.service, .timer, .socket, .mount) provide declarative service management with dependency resolution, socket activation, and cgroup-based resource control.

Universal formats (Flatpak, Snap, AppImage) supplement distro packages for desktop applications but are generally avoided on servers due to sandboxing overhead and update control concerns.

Filesystem Hierarchy Standard (FHS)

The FHS defines the directory structure of Linux systems. Key directories every sysadmin must know:

/etc — System configuration files (editable text)

/var — Variable data: logs (/var/log), mail, spool, databases

/usr — User programs and libraries (read-only after install)

/home — User home directories

/tmp — Temporary files (cleared on reboot)

/proc — Virtual filesystem exposing kernel and process info

/sys — Virtual filesystem for device and driver information

/opt — Third-party application packages

/srv — Data served by the system (web, FTP)

User Management

User authentication in Linux is a layered stack:

PAM (Pluggable Authentication Modules) provides the framework — modular, stackable authentication that can enforce password policies, two-factor authentication, and account lockouts. NSS (Name Service Switch) resolves user/group lookups, abstracting whether the source is /etc/passwd, LDAP, or SSSD. For centralized environments, LDAP (via OpenLDAP or FreeIPA) provides directory services, while SSSD caches and connects to Active Directory or IPA.

Process Management

Core tools: ps (snapshot), top/htop (real-time), kill/pkill (signals), nice/renice (priority). Understanding process states (Running, Sleeping, Stopped, Zombie, Dead) and signal semantics (SIGTERM vs SIGKILL vs SIGHUP) is fundamental.

cgroups v2 and namespaces provide the kernel primitives that underpin containers. A sysadmin who understands cgroups understands the resource isolation model of Docker and Kubernetes at the kernel level.

Shell Scripting

Bash remains the default shell on nearly all Linux distributions. POSIX shell compatibility matters for portability across systems. Key scripting patterns for sysadmins: log rotation, backup scripts, health checks, user provisioning, and automated deployments. Modern alternatives include Python for complex logic and shellcheck for static analysis of shell scripts.

Permissions & Access Control

Linux permissions operate at multiple layers:

LayerMechanismScope
DAC (Discretionary)chmod, chown, chgrpOwner/group/other rwx bits, setuid/setgid/sticky
ACLs (Extended)setfacl, getfaclFine-grained per-user/per-group permissions beyond rwx
MAC (Mandatory)SELinux / AppArmorKernel-enforced policy, prevents privilege escalation even by root
Capabilitiessetcap, getcapGranular root privileges (e.g., bind to port 80 without full root)

SELinux (Red Hat family) uses labels and policies to confine every process. AppArmor (Debian/Ubuntu family) uses path-based profiles. Both implement mandatory access control but with different configuration models.

Job Scheduling

cron remains ubiquitous for time-based job scheduling. The crontab format (minute hour day month weekday command) is universal knowledge. systemd timers provide an alternative with better logging integration, dependency management, and calendar event expressions. For one-time tasks, at and batch handle deferred execution.

# systemd timer equivalent of a daily cron job
[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
