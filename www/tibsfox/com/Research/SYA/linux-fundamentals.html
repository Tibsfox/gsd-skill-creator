Linux Fundamentals for Systems Administrators

PNW Research Series — Systems Administration (SYA) • Deep Research Module
Last updated: April 2026 • Estimated reading time: 35 minutes

Linux is the operating system that runs the modern internet. As of 2025, it powers over 96% of the world's top one million web servers, all 500 of the TOP500 supercomputers, the majority of cloud instances, and the vast majority of container workloads. For a systems administrator, deep fluency in Linux is not optional — it is the foundation upon which every other skill is built. This module covers the eight fundamental domains that every Linux sysadmin must master, from kernel internals to shell scripting to mandatory access control.

1. Kernel Architecture

1.1 The Monolithic vs. Microkernel Debate

On January 29, 1992, Andrew S. Tanenbaum — professor of computer science at Vrije Universiteit Amsterdam and creator of Minix — posted a message to the Usenet group comp.os.minix titled "LINUX is obsolete." Tanenbaum argued that monolithic kernels (where all OS services run in a single address space) were an architectural dead end, and that microkernels (where services run as separate user-space processes communicating via message passing) represented the future of operating system design. He predicted that Linux's tight coupling to the x86 architecture would render it irrelevant as newer processor architectures emerged.

Linus Torvalds responded the next day, acknowledging that microkernel design was theoretically superior but defending his pragmatic choice: a monolithic kernel was simpler to implement, performed better (no context-switching overhead for system calls), and was "good enough" for the real world. He also noted that Minix had its own design flaws, including the lack of multithreading support.

History decisively vindicated Torvalds's pragmatism. Linux runs on more architectures than any other operating system (x86, ARM, RISC-V, s390x, MIPS, PowerPC, and many others), and the monolithic-with-modules approach proved flexible enough to accommodate three decades of hardware evolution. The debate itself became one of the most famous in computing history, and its lessons about pragmatism versus theoretical purity remain relevant.

1.2 Loadable Kernel Modules

Linux's answer to the rigidity of a monolithic kernel is the Loadable Kernel Module (LKM) system. Modules are compiled code that can be dynamically loaded into and unloaded from the running kernel without a reboot. Most device drivers, filesystem implementations, and network protocols are distributed as modules.

# List loaded modules
lsmod

# Load a module
modprobe <module_name>

# Remove a module
modprobe -r <module_name>

# Show module information
modinfo <module_name>

# List module dependencies
modprobe --show-depends <module_name>

Module parameters can be set at load time or persistently via /etc/modprobe.d/*.conf files. For example, to blacklist a problematic driver:

# /etc/modprobe.d/blacklist-nouveau.conf
blacklist nouveau
options nouveau modeset=0

1.3 The /proc and /sys Filesystems

Linux exposes kernel internals through two virtual filesystems:

/proc (process filesystem) provides a window into running processes and kernel state. Each process gets a directory (/proc/<pid>/) containing its memory maps, file descriptors, status, and environment. System-wide information lives in files like /proc/meminfo, /proc/cpuinfo, /proc/loadavg, and /proc/net/.

/sys (sysfs) was introduced in Linux 2.6 to expose the kernel's device model in a structured hierarchy. Unlike /proc's somewhat ad hoc layout, /sys is organized by subsystem (/sys/class/, /sys/block/, /sys/devices/, /sys/bus/). It is the preferred interface for device and driver information.

1.4 Kernel Parameters and sysctl

The sysctl interface allows administrators to read and modify kernel parameters at runtime. Parameters are namespaced (e.g., net.ipv4.ip_forward, vm.swappiness, kernel.pid_max) and can be set temporarily or persisted in /etc/sysctl.d/*.conf.

# View current value
sysctl net.ipv4.ip_forward

# Set temporarily (lost on reboot)
sysctl -w net.ipv4.ip_forward=1

# Persist across reboots — add to /etc/sysctl.d/99-custom.conf:
net.ipv4.ip_forward = 1
vm.swappiness = 10
net.core.somaxconn = 65535

# Apply all sysctl config files
sysctl --system

Common tuning parameters for production servers:

1.5 Kernel Compilation and Custom Builds

While most administrators use distribution-supplied kernels, understanding the compilation process remains valuable for embedded systems, real-time workloads, security hardening, and hardware bring-up. The general workflow:

# Download and extract kernel source
wget https://cdn.kernel.org/pub/linux/kernel/v6.x/linux-6.12.tar.xz
tar xf linux-6.12.tar.xz && cd linux-6.12

# Start from current running config
cp /boot/config-$(uname -r) .config

# Interactive configuration (ncurses-based)
make menuconfig

# Alternatively, update config for new kernel version
make olddefconfig

# Compile (parallel build using all cores)
make -j$(nproc)

# Install modules and kernel
make modules_install
make install

# Update bootloader
grub-mkconfig -o /boot/grub/grub.cfg

Key reasons to build a custom kernel include: disabling unnecessary subsystems to reduce attack surface, enabling kernel hardening options (CONFIG_HARDENED_USERCOPY, CONFIG_STACKPROTECTOR_STRONG), adding patches not yet in mainline (such as real-time PREEMPT_RT), and tuning the HZ timer frequency for latency-sensitive workloads. Distribution kernels are compiled with CONFIG_HZ=1000 (RHEL) or CONFIG_HZ=250 (Ubuntu), which represents a trade-off between throughput and latency that may not suit every workload.

1.6 The Kernel Release Cycle

The Linux kernel follows a time-based release model managed by Linus Torvalds. Each release cycle lasts approximately 9-10 weeks:

  1. Merge window (2 weeks) — Subsystem maintainers send pull requests to Torvalds. Up to 1,000 changes per day are merged. Only code that has been reviewed and tested in subsystem trees is accepted.
  2. RC phase (7-8 weeks) — After the merge window closes, Torvalds releases -rc1, signaling that no new features will be accepted. Weekly release candidates (-rc2, -rc3, etc.) follow, with only bug fixes permitted. Typically -rc7 is the last, though additional RCs are released if needed.
  3. Release — The stable kernel is tagged and released.

There is exactly one person who can merge patches into the mainline kernel: Linus Torvalds. But the project uses a lieutenant system — subsystem maintainers act as gatekeepers for their areas (networking, memory management, filesystems, etc.), reviewing and merging patches into their trees before sending pull requests to Torvalds.

1.6 Major Kernel Subsystems

The kernel's major subsystems that sysadmins interact with most frequently include:

  • Scheduler (CFS) — The Completely Fair Scheduler allocates CPU time using a red-black tree of runnable tasks, weighted by nice values. EEVDF (Earliest Eligible Virtual Deadline First) replaced CFS as the default in kernel 6.6 (October 2023).
  • Virtual Filesystem (VFS) — An abstraction layer that allows different filesystem implementations (ext4, XFS, Btrfs, tmpfs, NFS) to coexist behind a uniform interface.
  • Netfilter — The kernel's packet filtering framework, upon which iptables and its successor nftables are built. Provides firewall, NAT, and packet mangling capabilities.
  • Cgroups (Control Groups) — Resource limitation and accounting for groups of processes. Cgroups v2, the current standard, provides a unified hierarchy for CPU, memory, I/O, and PID limits.
  • Namespaces — Process isolation primitives (PID, mount, network, user, UTS, IPC, cgroup, time). Namespaces are the foundation of container technology.

2. The Boot Process

2.1 Firmware: BIOS and UEFI

When a machine powers on, the first code to execute is firmware. Traditional BIOS (Basic Input/Output System), dating from the IBM PC era (1981), performs a Power-On Self-Test (POST) to validate hardware integrity, then searches for a bootloader in the Master Boot Record (MBR) of the first boot device — a 512-byte sector containing up to 446 bytes of bootloader code and a partition table.

UEFI (Unified Extensible Firmware Interface), which has largely replaced BIOS on modern hardware, is a far more capable firmware environment. UEFI can boot from GPT (GUID Partition Table) disks larger than 2 TB, supports an EFI System Partition (ESP) formatted as FAT32 where bootloaders are stored as regular files, and provides Secure Boot — a chain of trust that verifies the cryptographic signature of every component from firmware through bootloader to kernel, preventing rootkit installation at the boot level.

2.2 GRUB2: The Grand Unified Bootloader

GRUB2 (GRand Unified Bootloader version 2) is the default bootloader for most Linux distributions. It reads its configuration from /boot/grub/grub.cfg (BIOS) or /boot/efi/EFI/<distro>/grub.cfg (UEFI), which defines available kernels, initramfs images, and boot parameters. This file is auto-generated — administrators edit /etc/default/grub and drop scripts into /etc/grub.d/, then run grub-mkconfig (or grub2-mkconfig on RHEL-family systems) to regenerate it.

# /etc/default/grub — common settings
GRUB_TIMEOUT=5
GRUB_DEFAULT=0
GRUB_CMDLINE_LINUX="rhgb quiet crashkernel=auto"
GRUB_DISABLE_RECOVERY="true"

# Regenerate GRUB config after editing
grub-mkconfig -o /boot/grub/grub.cfg

Key boot parameters that sysadmins use for troubleshooting:

  • single or systemd.unit=rescue.target — boot to single-user/rescue mode
  • rd.break — break into an emergency shell before root filesystem mount (useful for password recovery)
  • init=/bin/bash — bypass init entirely, boot directly to a shell
  • nomodeset — disable kernel mode-setting (fix for broken display drivers)

2.3 Kernel Loading and initramfs

GRUB loads two files from /boot: the compressed kernel image (vmlinuz-<version>) and the initial RAM filesystem (initramfs-<version>.img). The initramfs is a compressed cpio archive containing a minimal filesystem with the drivers and tools needed to mount the real root filesystem — particularly important when root lives on LVM, RAID, encrypted volumes, or network storage that requires drivers not compiled into the kernel.

# Rebuild initramfs (Debian/Ubuntu)
update-initramfs -u

# Rebuild initramfs (RHEL/Fedora)
dracut --force

# Rebuild initramfs (Arch)
mkinitcpio -P

# Inspect initramfs contents
lsinitrd /boot/initramfs-$(uname -r).img        # RHEL/Fedora
lsinitramfs /boot/initrd.img-$(uname -r)         # Debian/Ubuntu

2.4 The Init System Wars: SysVinit to Upstart to systemd

Once the kernel mounts the root filesystem, it executes the first userspace process: PID 1, the init system. The evolution of init systems is one of the most contentious chapters in Linux history.

SysVinit (System V init) was the traditional init system, inherited from AT&T Unix. It used shell scripts in /etc/init.d/ and a concept of runlevels (0=halt, 1=single-user, 3=multi-user, 5=graphical, 6=reboot) to manage service startup. Services started sequentially, one at a time, making boot slow on machines with many services.

Upstart, developed by Canonical and introduced in Ubuntu 6.10 (2006), replaced sequential startup with an event-driven model. Services could declare dependencies on events rather than runlevels, allowing parallel startup. Red Hat adopted Upstart for RHEL 6 (2010). However, Upstart's event model proved difficult to reason about at scale.

systemd, created by Lennart Poettering and Kay Sievers in 2010, took a more comprehensive approach. It replaced not just init but also the logging daemon (with journald), the cron daemon (with timers), the inetd superserver (with socket activation), device management (integrating udev), and login session management (with logind). systemd became the default in Fedora 15 (2011), followed by Arch Linux (2012), RHEL 7 (2014), SUSE (2014), Debian 8 "Jessie" (2015, after a contentious Technical Committee vote), and Ubuntu 15.04 (2015, when Mark Shuttleworth announced the switch on February 14, 2014).

The systemd controversy was fierce. Critics argued that systemd violated the Unix philosophy by absorbing too many functions into a single project, created tight coupling between components, introduced binary log formats (breaking traditional text-based log analysis), and was developed with insufficient regard for portability to non-Linux systems. The controversy included personal attacks and alleged death threats against Poettering. Supporters countered that systemd's integrated design solved real problems — parallel service startup, socket activation, cgroup-based resource management, and reliable service supervision — that the patchwork of SysVinit scripts could not.

By 2016, the debate was effectively settled by adoption: every major Linux distribution had switched to systemd. It remains the standard init system today.

3. Package Management

3.1 The Debian Family: dpkg and APT

dpkg is the low-level package manager for Debian and its derivatives (Ubuntu, Mint, Pop!_OS). It handles installation, removal, and querying of individual .deb packages but does not resolve dependencies or interact with remote repositories.

APT (Advanced Package Tool), released in 1998 and first included in Debian 2.1 (1999), adds dependency resolution, repository management, and automatic downloads on top of dpkg. APT is considered one of the defining features of Debian-based systems.

# Update package index
apt update

# Upgrade all installed packages
apt upgrade

# Install a package (with dependency resolution)
apt install nginx

# Remove a package (keep config files)
apt remove nginx

# Remove a package and its config files
apt purge nginx

# Search for packages
apt search "web server"

# Show package details
apt show nginx

# List installed packages
dpkg -l | grep nginx

# Pin a package to a specific version
# /etc/apt/preferences.d/nginx-pin
Package: nginx
Pin: version 1.24.*
Pin-Priority: 1001

3.2 The Red Hat Family: RPM, YUM, and DNF

RPM (Red Hat Package Manager) is the low-level format for RHEL, Fedora, CentOS, Rocky Linux, AlmaLinux, and SUSE. Like dpkg, RPM handles individual packages without dependency resolution.

YUM (Yellowdog Updater Modified) added repository and dependency support, becoming the standard for RHEL 5-7 and CentOS. DNF (Dandified YUM), introduced in Fedora 18 and becoming the default in Fedora 22 (2015), was a complete rewrite addressing YUM's performance issues, undocumented APIs, broken dependency resolution, and high memory usage. DNF became the default in RHEL 8.

# Update package index and upgrade
dnf upgrade

# Install a package
dnf install httpd

# Remove a package
dnf remove httpd

# Search for packages
dnf search "web server"

# List installed packages
rpm -qa | grep httpd

# Show which package owns a file
rpm -qf /usr/sbin/httpd

# Create a local repository
createrepo /path/to/rpms/
# Add to /etc/yum.repos.d/local.repo:
[local]
name=Local Repository
baseurl=file:///path/to/rpms/
enabled=1
gpgcheck=0

3.3 Arch Linux: Pacman

Pacman manages packages in Arch Linux's .pkg.tar.zst format. It is notably faster than APT and DNF, tightly integrated with Arch's rolling-release model, and uses a terser syntax:

# Synchronize and upgrade
pacman -Syu

# Install a package
pacman -S nginx

# Remove a package and unused dependencies
pacman -Rns nginx

# Search for a package
pacman -Ss "web server"

# Query installed package info
pacman -Qi nginx

3.4 Universal Packaging: Flatpak, Snap, and AppImage

Traditional package managers are distribution-specific. Three formats attempt to solve cross-distribution compatibility:

FormatCreatorMechanismBest For
FlatpakFreeDesktop / Red HatOSTree (content-addressed, Git-like), shared runtimes, sandboxedDesktop applications
SnapCanonicalSquashFS image mounted read-only, centralized Snap Store, daemon-basedDesktop, server, IoT

As of 2026, Flatpak has emerged as the community preference for desktop applications, while Snap maintains its position in Ubuntu's ecosystem and server/IoT deployments. AppImage remains popular for its simplicity but lacks automatic updates and sandboxing.

3.5 GPG Signing and Repository Security

Package repositories use GPG signatures to ensure integrity. When you add a third-party repository, you must import its GPG key:

# Debian/Ubuntu: import a GPG key
curl -fsSL https://example.com/repo.gpg | gpg --dearmor -o /etc/apt/keyrings/example.gpg

# RHEL/Fedora: import an RPM GPG key
rpm --import https://example.com/RPM-GPG-KEY-example

# Verify a package signature
rpm -K package.rpm
dpkg-sig --verify package.deb

4. Filesystem Hierarchy Standard (FHS)

4.1 Overview and History

The Filesystem Hierarchy Standard (FHS) defines the directory structure for Unix-like systems. Originally developed by the Linux community in the early 1990s, version 3.0 was published on June 3, 2015 by the Linux Foundation and has since been maintained by FreeDesktop (republished November 2025). Understanding FHS is essential for every sysadmin — it determines where configuration lives, where logs accumulate, where binaries execute, and how backup strategies must be designed.

4.2 Key Directories and Their Purposes

4.3 The /usr Merge

Historically, /bin, /sbin, and /lib were separate from /usr/bin, /usr/sbin, and /usr/lib. The distinction dated from the earliest Unix systems, where /usr might be on a separate disk that was not available during early boot. On modern systems with initramfs, this distinction is unnecessary.

The /usr merge (also called the "merged-/usr" or "UsrMerge") makes /bin, /sbin, and /lib symlinks to their counterparts under /usr. Fedora completed the merge in Fedora 17 (2012), Arch Linux followed suit, Debian completed it in Debian 12 "Bookworm" (2023), and Ubuntu in 23.04. This simplifies the filesystem, eliminates the need to decide which directory a binary belongs in, and avoids subtle bugs caused by duplicate paths.

# On a merged-/usr system:
$ ls -la /bin
lrwxrwxrwx 1 root root 7 Jan  1  2024 /bin -> usr/bin

$ ls -la /sbin
lrwxrwxrwx 1 root root 8 Jan  1  2024 /sbin -> usr/sbin

5. User and Group Management

5.1 The Core Files

Linux user management revolves around three files:

/etc/passwd — one line per user, colon-delimited: username:x:UID:GID:GECOS:home:shell. The x in the password field indicates that the actual password hash is stored in /etc/shadow.

/etc/shadow — contains password hashes and aging information. Readable only by root. Fields include the hash algorithm (e.g., $6$ for SHA-512, $y$ for yescrypt), salt, hash, and password expiry dates.

/etc/group — defines groups and their members: groupname:x:GID:member1,member2.

# Create a user with specific UID, home directory, and shell
useradd -u 1500 -m -d /home/jdoe -s /bin/bash -G wheel,docker jdoe

# Set password
passwd jdoe

# Modify user (add to supplementary group)
usermod -aG docker jdoe

# Lock/unlock account
usermod -L jdoe    # lock
usermod -U jdoe    # unlock

# Set password aging (expire in 90 days, warn 14 days before)
chage -M 90 -W 14 jdoe

# View password aging info
chage -l jdoe

5.2 PAM: Pluggable Authentication Modules

PAM is the framework that allows Linux to support multiple authentication methods without modifying individual applications. When a user runs sudo, ssh, login, or su, the application calls PAM, which consults a stack of modules defined in /etc/pam.d/.

A PAM configuration file contains four management groups:

  • auth — verifies the user's identity (password, token, biometric)
  • account — checks account validity (expired? locked? time restrictions?)
  • password — handles password changes
  • session — sets up/tears down the user session (mount home directory, set ulimits, log access)
# Example: /etc/pam.d/sshd
auth       required     pam_sepermit.so
auth       substack     password-auth
auth       include      postlogin
account    required     pam_nologin.so
account    include      password-auth
password   include      password-auth
session    required     pam_selinux.so close
session    required     pam_loginuid.so
session    required     pam_selinux.so open
session    required     pam_namespace.so
session    optional     pam_keyinit.so force revoke
session    include      password-auth
session    include      postlogin

5.3 NSS and SSSD

NSS (Name Service Switch) determines where the system looks up user and group information. Configured in /etc/nsswitch.conf, it can chain multiple sources:

# /etc/nsswitch.conf
passwd:     files sss
group:      files sss
shadow:     files sss
hosts:      files dns myhostname

SSSD (System Security Services Daemon) is the modern integration layer for centralized identity. It replaces the older nss_ldap, pam_ldap, and pam_krb5 modules, providing a unified client for LDAP directories, Kerberos authentication, and Active Directory domains. SSSD caches credentials locally, enabling login even when the network directory is temporarily unreachable.

5.4 sudo Configuration

The sudoers file (/etc/sudoers, edited exclusively with visudo) controls privilege escalation. Drop-in files in /etc/sudoers.d/ are preferred for maintainability.

# Allow user jdoe to run any command as root
jdoe    ALL=(ALL)       ALL

# Allow group wheel with password
%wheel  ALL=(ALL)       ALL

# Allow specific commands without password (use sparingly!)
deploy  ALL=(ALL)       NOPASSWD: /usr/bin/systemctl restart myapp

# DANGER: NOPASSWD for all commands is a security risk
# jdoe  ALL=(ALL)       NOPASSWD: ALL   # DON'T do this in production

NOPASSWD dangers: Granting NOPASSWD: ALL effectively gives the user unrestricted root access with no authentication barrier. If the user's account is compromised, the attacker has immediate root access. Best practice is to grant NOPASSWD only for specific, well-defined commands needed for automation.

6. Process Management

6.1 Process States

Every process in Linux occupies one of several states, visible in the STAT column of ps output:

StateCodeMeaning
RunningRCurrently executing or in the run queue
Sleeping (Interruptible)SWaiting for an event (I/O, signal); can be woken
Sleeping (Uninterruptible)DWaiting for I/O (typically disk); cannot be killed — a common cause of "unkillable" processes
StoppedTStopped by a signal (SIGSTOP or Ctrl+Z)
ZombieZTerminated but parent has not called wait(); process entry remains in the process table

Processes in D state cannot be killed, even with SIGKILL. This is by design: the process is in the middle of a kernel operation (usually disk I/O) that cannot be safely interrupted. If you see many processes in D state, the likely cause is a storage subsystem problem — a hung NFS mount, a failing disk, or a saturated I/O bus.

6.2 Signals

Signals are the primary mechanism for communicating with processes. The two most important to understand:

  • SIGTERM (15) — the polite request. The process receives the signal and can handle it: flush buffers, close connections, clean up temporary files, then exit. This is what kill <pid> sends by default and what systemd sends first when stopping a service.
  • SIGKILL (9) — the nuclear option. The kernel terminates the process immediately. The process gets no chance to clean up. Use only when SIGTERM fails. kill -9 <pid>.

Other commonly used signals:

# Hang up (often used to reload configuration)
kill -HUP <pid>    # SIGHUP (1) — nginx, Apache reload config on HUP

# Interrupt (same as Ctrl+C)
kill -INT <pid>    # SIGINT (2)

# Continue a stopped process
kill -CONT <pid>   # SIGCONT (18)

# User-defined (often used for log rotation)
kill -USR1 <pid>   # SIGUSR1 (10)

6.3 Zombie Processes: Diagnosis and Resolution

A zombie process (Z state) is a process that has terminated but whose parent has not yet read its exit status via the wait() system call. The process table entry remains, consuming a PID slot but no memory or CPU. A few zombies are harmless and transient, but a large accumulation indicates a buggy parent process that fails to reap its children.

# Find zombie processes
ps aux | awk '$8 == "Z"'

# Identify the parent of a zombie
ps -o ppid= -p <zombie_pid>

# Resolution options:
# 1. Send SIGCHLD to the parent (prompts it to call wait())
kill -CHLD <parent_pid>

# 2. If that fails, kill the parent (init/systemd adopts and reaps orphans)
kill <parent_pid>

# 3. Fix the application code to properly handle SIGCHLD

6.4 Priority and Resource Control

Nice values range from -20 (highest priority) to 19 (lowest). Only root can set negative nice values. The default is 0.

# Start a process with low priority
nice -n 19 ./backup.sh

# Change priority of a running process
renice -n 10 -p <pid>

6.4 Cgroups v2 for Resource Limits

Cgroups v2 (control groups version 2), the current standard since Linux 4.5 and required by recent systemd versions, provides a unified hierarchy for resource management. Unlike cgroups v1, which had separate hierarchies for each controller, v2 uses a single hierarchy where a process belongs to exactly one cgroup.

Key controllers:

  • cpu — CPU time allocation (weight-based or bandwidth throttling)
  • memory — memory limits (memory.max hard limit, memory.high soft reclaim target)
  • io — block I/O throttling and proportional weight
  • pids — limit the number of processes (prevent fork bombs)

systemd integrates directly with cgroups. Every systemd service runs in its own cgroup, and resource limits can be set in the unit file:

# /etc/systemd/system/myapp.service
[Service]
ExecStart=/usr/bin/myapp
MemoryMax=1G
MemoryHigh=768M
CPUQuota=200%
TasksMax=512
Restart=on-failure
RestartSec=5s

6.5 The OOM Killer

When the system runs out of memory and swap, the OOM (Out of Memory) killer activates. It selects a process to terminate based on a scoring algorithm that considers memory usage, process age, and the oom_score_adj value. Linux 4.19 (October 2018) introduced cgroup-aware OOM killing, which can terminate an entire cgroup as a unit (controlled by memory.oom.group), guaranteeing workload integrity rather than killing random individual processes.

# View OOM score of a process (higher = more likely to be killed)
cat /proc/<pid>/oom_score

# Adjust OOM priority (-1000 = never kill, 1000 = always kill first)
echo -500 > /proc/<pid>/oom_score_adj

# In a systemd unit file:
[Service]
OOMScoreAdjust=-500

7. Shell Scripting

7.1 Bash Fundamentals

Bash (Bourne Again Shell) is the default interactive shell on most Linux distributions and the most common scripting language for system administration tasks. Every bash script should begin with a shebang line and safety options:

#!/usr/bin/env bash
set -euo pipefail

# -e (errexit): exit immediately if any command fails
# -u (nounset): treat references to unset variables as errors
# -o pipefail: pipeline fails if ANY command in the pipe fails,
#              not just the last one

7.2 The POSIX sh vs. Bash Distinction

This distinction is critical and the source of many subtle bugs. /bin/sh on most Linux systems is not bash — on Debian and Ubuntu, it is dash, a minimal POSIX-compliant shell optimized for speed. Many bash features are not available in POSIX sh:

FeatureBashPOSIX sh (dash)
Arraysarr=(a b c)Not available
[[ ]] conditionalsSupportedNot available (use [ ])
set -o pipefailSupportedAdded in POSIX:2024, not in older dash
Process substitution<(command)Not available
{1..10} brace expansionSupportedNot available
$'...' ANSI-C quotingSupportedNot available
Here strings<<< "string"Not available
Functions with localSupportedPartially supported

Rule of thumb: If your script uses any bash-specific features, the shebang must be #!/bin/bash (or #!/usr/bin/env bash), never #!/bin/sh. This matters particularly in CI/CD environments and Docker containers, where the base image may use Alpine Linux (which ships busybox ash, not bash) or minimal Debian images where bash is not installed by default.

7.2.1 Portability Testing

To verify that a script works under POSIX sh, test it explicitly with dash:

# Install dash if not present
apt install dash

# Run your script under dash to catch bashisms
dash ./myscript.sh

# Or use checkbashisms (from devscripts package)
checkbashisms ./myscript.sh

The checkbashisms utility, part of Debian's devscripts package, statically analyzes scripts for bash-specific constructs and is invaluable when writing scripts intended for broad portability.

7.3 Common Sysadmin Scripting Patterns

Lock files prevent concurrent execution:

#!/usr/bin/env bash
set -euo pipefail

LOCKFILE="/var/run/mybackup.lock"

cleanup() {
    rm -f "$LOCKFILE"
}
trap cleanup EXIT

if ! mkdir "$LOCKFILE" 2>/dev/null; then
    echo "Another instance is running (lockdir exists: $LOCKFILE)" >&2
    exit 1
fi

# ... do work ...

Note: mkdir is atomic on all filesystems and is preferred over creating a regular file for locking.

Signal handlers for graceful shutdown:

#!/usr/bin/env bash
set -euo pipefail

RUNNING=true

handle_signal() {
    echo "Caught signal, shutting down gracefully..."
    RUNNING=false
}
trap handle_signal SIGTERM SIGINT

while $RUNNING; do
    # ... do work ...
    sleep 5
done

echo "Exited cleanly."

Process substitution for comparing command outputs:

# Compare files on two remote hosts without temp files
diff <(ssh host1 cat /etc/hosts) <(ssh host2 cat /etc/hosts)

Heredocs for multi-line strings and remote commands:

# Execute commands on a remote host
ssh user@host << 'REMOTE'
    systemctl status nginx
    df -h /var/log
    tail -5 /var/log/nginx/error.log
REMOTE

7.4 ShellCheck and Quality

ShellCheck is an essential static analysis tool for shell scripts. It catches common bugs, portability issues, and style problems. Every sysadmin script should pass ShellCheck before deployment:

# Install
apt install shellcheck    # Debian/Ubuntu
dnf install ShellCheck    # Fedora/RHEL

# Run on a script
shellcheck myscript.sh

# Integrate with CI/CD
shellcheck --format=json myscript.sh

7.5 When to Move to Python

Shell scripts are appropriate for:

  • Gluing together existing commands
  • Simple file operations and text processing
  • Boot scripts and system initialization
  • Quick automation of repetitive tasks

Move to Python (or another general-purpose language) when:

  • You need data structures beyond arrays (dictionaries, classes, nested structures)
  • Error handling becomes complex (bash's error handling is fragile)
  • You need to parse structured data (JSON, YAML, XML)
  • The script exceeds ~200 lines
  • You need unit testing
  • You are interacting with APIs (HTTP requests, database queries)

8. Permissions and Access Control

8.1 Traditional UGO Permissions

The classic Unix permission model assigns three permission types (read, write, execute) to three categories (user/owner, group, others):

# View permissions
ls -la /etc/shadow
-rw-r----- 1 root shadow 1234 Apr  1 12:00 /etc/shadow

# Set permissions (numeric)
chmod 750 /opt/myapp           # rwxr-x---
chmod 644 /etc/myapp.conf      # rw-r--r--

# Set permissions (symbolic)
chmod u+x script.sh            # Add execute for owner
chmod g-w,o-r file.txt         # Remove write for group, read for others
chmod a+r public.html          # Add read for all

8.2 Special Permission Bits

  • setuid (4000) — When set on an executable, it runs with the privileges of the file owner, not the invoking user. Example: /usr/bin/passwd is setuid root so that any user can change their own password (which requires writing to /etc/shadow). Setuid is a significant security surface and should be audited regularly: find / -perm -4000 -type f.
  • setgid (2000) — On an executable, runs with the file's group. On a directory, new files inherit the directory's group (essential for shared project directories).
  • sticky bit (1000) — On a directory, prevents users from deleting files they don't own. The classic example is /tmp, where all users can create files but only file owners (and root) can delete them.
# Set setuid
chmod u+s /usr/local/bin/myutil    # or chmod 4755

# Set setgid on a shared directory
chmod g+s /opt/shared/             # or chmod 2775

# Set sticky bit
chmod +t /tmp/                     # or chmod 1777

# Audit setuid binaries system-wide
find / -perm -4000 -type f -exec ls -la {} \; 2>/dev/null

8.3 POSIX Access Control Lists (ACLs)

Traditional UGO permissions are limited to one owner, one group, and everyone else. POSIX ACLs provide fine-grained access control for multiple users and groups:

# View ACLs
getfacl /opt/project/

# Grant read/write to a specific user
setfacl -m u:jdoe:rw /opt/project/report.txt

# Grant read to a specific group
setfacl -m g:auditors:r /var/log/app/

# Set default ACL on a directory (inherited by new files)
setfacl -d -m g:developers:rwx /opt/project/

# Remove a specific ACL entry
setfacl -x u:jdoe /opt/project/report.txt

# Remove all ACLs
setfacl -b /opt/project/report.txt

A + sign in ls -l output indicates that ACLs are present on a file. ACLs are supported on ext4, XFS, Btrfs, and most modern filesystems. They are essential in environments where the traditional three-category model is insufficient — for example, a shared project directory where multiple teams need different access levels, or an audit directory where auditors need read access without belonging to the operations group.

8.3.1 Practical ACL Case Study: Shared Development Directory

Consider a project directory where developers need read/write access, QA needs read-only access, and a CI service account needs read/write. Traditional permissions can only accommodate one group:

# Create the directory with setgid for group inheritance
mkdir -p /opt/project
chgrp developers /opt/project
chmod 2770 /opt/project

# Grant read-only to the QA group
setfacl -m g:qa:rx /opt/project
setfacl -d -m g:qa:rx /opt/project        # default ACL for new files

# Grant read/write to CI service account
setfacl -m u:ci-runner:rwx /opt/project
setfacl -d -m u:ci-runner:rwx /opt/project

# Verify the complete ACL
getfacl /opt/project
# file: opt/project
# owner: root
# group: developers
# flags: -s-
# user::rwx
# user:ci-runner:rwx
# group::rwx
# group:qa:r-x
# mask::rwx
# other::---
# default:user::rwx
# default:user:ci-runner:rwx
# default:group::rwx
# default:group:qa:r-x
# default:mask::rwx
# default:other::---

8.4 SELinux

Security-Enhanced Linux (SELinux), developed by the NSA and integrated into the mainline kernel, implements mandatory access control (MAC). Unlike discretionary access control (DAC, i.e., traditional permissions), where file owners control access, MAC policies are set by the administrator and enforced by the kernel regardless of file ownership.

SELinux assigns a security context (label) to every process, file, port, and other system object. The format is user:role:type:level. The type field is the most commonly used in targeted policy (RHEL's default): a process of type httpd_t can only access files of type httpd_sys_content_t.

# Check SELinux status
getenforce                          # Enforcing, Permissive, or Disabled
sestatus                            # Detailed status

# View file contexts
ls -Z /var/www/html/
# -rw-r--r--. root root unconfined_u:object_r:httpd_sys_content_t:s0 index.html

# View process contexts
ps -eZ | grep httpd
# system_u:system_r:httpd_t:s0    1234 ?  00:00:01 httpd

# Restore default file contexts
restorecon -Rv /var/www/html/

# Toggle a boolean (e.g., allow httpd to connect to the network)
setsebool -P httpd_can_network_connect on

# Investigate a denial
ausearch -m avc -ts recent
audit2why < /var/log/audit/audit.log

# Generate a policy module from denials (use with caution)
audit2allow -a -M mypolicy
semodule -i mypolicy.pp

SELinux modes:

  • Enforcing — policies are enforced, violations are denied and logged
  • Permissive — policies are not enforced but violations are logged (useful for debugging)
  • Disabled — SELinux is completely off (re-enabling requires a full filesystem relabel)

Used by default on RHEL, Rocky Linux, AlmaLinux, CentOS Stream, and Fedora.

8.5 AppArmor

AppArmor takes a path-based approach to mandatory access control, in contrast to SELinux's label-based model. AppArmor profiles define what files, capabilities, and network access a program is allowed. Profiles can operate in enforce mode (violations blocked) or complain mode (violations logged but allowed).

# Check AppArmor status
aa-status

# Put a profile in complain mode
aa-complain /etc/apparmor.d/usr.sbin.nginx

# Put a profile in enforce mode
aa-enforce /etc/apparmor.d/usr.sbin.nginx

# Generate a profile interactively
aa-genprof /usr/sbin/myapp

AppArmor is generally considered easier to learn and configure than SELinux, though it provides less granular control. It is the default MAC system on Ubuntu, Debian, SUSE Linux Enterprise Server, and openSUSE.

8.6 Linux Capabilities

Traditionally, a process either ran as root (with all privileges) or as an unprivileged user (with none). Linux capabilities break root's monolithic privilege into discrete units that can be granted individually:

CapabilityPermission GrantedCommon Use Case
CAP_NET_BIND_SERVICEBind to ports below 1024Run a web server as non-root on port 80
CAP_NET_RAWUse raw socketsAllow ping without setuid
CAP_SYS_ADMINBroad system admin operations"The new root" — avoid granting this
CAP_DAC_OVERRIDEBypass file permission checksBackup tools that need to read all files
CAP_CHOWNChange file ownershipDeployment tools
CAP_SETUIDChange process UIDPrivilege dropping daemons
 
# Grant a capability to a binary
setcap cap_net_bind_service=+ep /usr/bin/myserver

# View capabilities on a binary
getcap /usr/bin/myserver

# View capabilities of a running process
getpcaps <pid>

# In a systemd unit file (preferred for services)
[Service]
AmbientCapabilities=CAP_NET_BIND_SERVICE
CapabilityBoundingSet=CAP_NET_BIND_SERVICE

8.7 umask

The umask defines the default permissions for newly created files and directories by specifying which permission bits to remove. The default umask of 0022 means new files get 644 (rw-r--r--) and new directories get 755 (rwxr-xr-x).

# View current umask
umask

# Set a more restrictive umask (files: 640, directories: 750)
umask 0027

# Set system-wide default in /etc/login.defs
UMASK 027

# Or via PAM in /etc/pam.d/common-session:
session optional pam_umask.so umask=0027

For shared project directories, combine setgid, ACLs, and an appropriate umask to ensure that all files created by team members are readable and writable by the project group.
