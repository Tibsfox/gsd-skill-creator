The cloud did not eliminate systems administration; it transformed it. Instead of racking servers, sysadmins now architect VPCs, configure IAM policies, and manage container orchestration at a scale that would have been unimaginable a decade ago. The complexity did not decrease — it moved up the stack.

Kubernetes Architecture

Kubernetes (from Google’s internal Borg system) is the standard container orchestration platform. Understanding its architecture is non-negotiable for modern sysadmins.

Control Plane:

  • kube-apiserver: RESTful API, all communication passes through it
  • etcd: Distributed key-value store, source of truth for cluster state
  • kube-scheduler: Assigns pods to nodes based on resource requirements, affinity, taints
  • kube-controller-manager: Runs control loops (ReplicaSet, Deployment, Node, Job controllers)
  • cloud-controller-manager: Cloud-provider-specific controllers (load balancers, volumes)

Worker Nodes:

  • kubelet: Node agent, ensures containers run in pods according to PodSpec
  • kube-proxy: Network proxy implementing Service abstraction (iptables/IPVS)
  • Container runtime: containerd (default), CRI-O (Red Hat)

Pod Lifecycle & Workload Types

Pods move through states: Pending → Running → Succeeded/Failed. Workload controllers manage pod replicas:

Helm

Helm is the package manager for Kubernetes. Charts are templated Kubernetes manifests with configurable values. Helm manages releases (install, upgrade, rollback) and provides a chart repository ecosystem. Helmfile manages multiple releases declaratively. Kustomize (built into kubectl) provides an alternative templating approach using overlay patches.

Container Runtimes & Docker Security

containerd (CNCF graduated) is the default runtime for Kubernetes since dockershim removal in K8s 1.24. CRI-O is the minimal runtime for OpenShift/RHEL. Docker remains the primary tool for building images.

Docker security best practices:

  • Multi-stage builds: Minimize final image size, exclude build dependencies
  • Non-root user: USER nonroot in Dockerfile, avoid running as PID 1 root
  • Image scanning: Trivy, Grype, or Snyk for CVE detection before deployment
  • Minimal base images: Alpine, distroless, or scratch where possible
  • Read-only filesystem: --read-only flag prevents container filesystem writes
  • Seccomp & AppArmor profiles: Restrict syscalls available to containers

Registry Management

Container registries store and distribute images. Harbor (CNCF graduated) provides vulnerability scanning, RBAC, replication, and audit logging for private registries. Cloud-managed options: ECR (AWS), ACR (Azure), Artifact Registry (GCP). Sigstore/cosign signs images cryptographically for supply chain security.

RBAC (Role-Based Access Control)

Kubernetes RBAC controls who can do what within the cluster. Four objects: Role (namespace-scoped permissions), ClusterRole (cluster-wide), RoleBinding (assigns Role to subjects), ClusterRoleBinding (assigns ClusterRole). Follow the principle of least privilege: grant only the permissions each workload or user needs.

# ClusterRole: read-only access to pods
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
