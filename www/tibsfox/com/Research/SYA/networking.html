"The network is the computer." — John Gage, Sun Microsystems. For the sysadmin, the network is the infrastructure that makes everything else possible. Understanding it at a protocol level is the difference between guessing and diagnosing.

1. The OSI and TCP/IP Models

1.1 The Seven-Layer OSI Model

The Open Systems Interconnection (OSI) model, formalized by ISO in 1984, provides a conceptual framework for understanding network communication. Its seven layers — Physical, Data Link, Network, Transport, Session, Presentation, and Application — describe how data moves from a user application on one machine to an application on another.

1.2 The TCP/IP Model — What Actually Runs

In practice, the four-layer TCP/IP model (Link, Internet, Transport, Application) is what the kernel implements. The OSI layers 5-7 collapse into a single "Application" layer. Most sysadmins spend their time in layers 2-4: configuring VLANs and bonds (L2), managing IP addressing and routes (L3), and troubleshooting connection states and firewall rules (L4).

1.3 Encapsulation, Headers, and MTU

Data moves down the stack via encapsulation: application data becomes a TCP segment (with a 20-byte header), then an IP packet (with another 20-byte header), then an Ethernet frame (with a 14-byte header and 4-byte FCS). At each hop, routers strip and re-add L2 headers while preserving L3+.

The Maximum Transmission Unit (MTU) defines the largest L3 payload an L2 frame can carry. Standard Ethernet MTU is 1500 bytes. Jumbo frames (MTU 9000) improve throughput for bulk transfers within datacenters by reducing per-packet overhead, but every device on the path must support them — a single 1500-MTU hop will force fragmentation or drop packets if the Don't Fragment (DF) bit is set.

Path MTU Discovery (PMTUD) relies on ICMP "Fragmentation Needed" (type 3, code 4) messages. Blocking ICMP at firewalls breaks PMTUD, causing mysterious connection stalls where the TCP handshake succeeds but data transfer hangs — a classic "black hole" problem.

1.4 When the Models Break Down

Real protocols do not always fit neatly into layers. MPLS operates between L2 and L3 (sometimes called "Layer 2.5"). TLS sits between transport and application. QUIC implements its own transport semantics over UDP, effectively merging transport and encryption. The models remain useful as mental frameworks for troubleshooting ("start at the bottom, work up"), but rigid adherence to the OSI model is a classroom exercise, not an operational philosophy.

2. TCP/IP Deep Dive

2.1 The TCP Three-Way Handshake

Every TCP connection begins with a three-way handshake that synchronizes sequence numbers between client and server:

Client --> SYN (seq=x)              --> Server
Client <-- SYN-ACK (seq=y, ack=x+1) <-- Server
Client --> ACK (ack=y+1)            --> Server

The handshake takes one round-trip time (RTT). On a connection with 100ms RTT, this means 100ms before any data can flow. Connection reuse (HTTP keep-alive, connection pooling) exists precisely to amortize this cost.

2.2 Congestion Control: Reno, CUBIC, and BBR

TCP congestion control determines how fast a sender ramps up and backs off. The evolution matters because it directly affects throughput on your servers:

TCP Reno (1990) uses Additive Increase / Multiplicative Decrease (AIMD). It increases the congestion window by one segment per RTT and halves it on packet loss. Reno works but recovers slowly from losses and severely underutilizes high-bandwidth, high-latency links (the "long fat network" problem).

CUBIC (2006, Linux default since 2.6.19) replaces linear growth with a cubic function that quickly probes back toward the pre-loss window. It is loss-based: it increases sending rate until packets are dropped, then backs off. CUBIC is the default on Linux, macOS, and Windows.

BBR (Bottleneck Bandwidth and Round-trip propagation time), developed by Google and now at version 3, takes a fundamentally different approach. Instead of reacting to loss, BBR builds a model of the network path by continuously probing for available bandwidth and minimum RTT. Under 200ms RTT and 500 Mbps bandwidth, BBR improves goodput by approximately 115% compared to CUBIC. Google reported 4% throughput improvement globally and up to 14% in developing regions when deploying BBR on its CDN. However, BBR can be unfair to competing CUBIC flows — APNIC testing found BBR grabbing more than its fair share on shared links.

To enable BBR on Linux:

# Check current algorithm
sysctl net.ipv4.tcp_congestion_control

# Enable BBR (requires kernel 4.9+)
echo "net.core.default_qdisc=fq" >> /etc/sysctl.d/99-bbr.conf
echo "net.ipv4.tcp_congestion_control=bbr" >> /etc/sysctl.d/99-bbr.conf
sysctl -p /etc/sysctl.d/99-bbr.conf

# Verify
sysctl net.ipv4.tcp_congestion_control
# net.ipv4.tcp_congestion_control = bbr

2.3 TIME_WAIT and Socket Exhaustion

After a TCP connection closes, the side that initiates the close enters TIME_WAIT for 2 × MSL (Maximum Segment Lifetime, typically 60 seconds on Linux). This prevents delayed packets from a previous connection from being misinterpreted in a new connection on the same port tuple. On busy servers handling thousands of short-lived connections (load balancers, API gateways), TIME_WAIT sockets can accumulate and exhaust the available port space (roughly 28,000 ephemeral ports by default).

Mitigation strategies:

# Allow reuse of TIME_WAIT sockets for new connections (safe since kernel 4.12)
net.ipv4.tcp_tw_reuse = 1

# Expand ephemeral port range
net.ipv4.ip_local_port_range = 1024 65535

# SO_REUSEADDR: allows binding to an address already in TIME_WAIT
# (set in application code: setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)))

# SO_REUSEPORT: allows multiple sockets to bind the same port (load distribution)
# Used by Nginx, Envoy for multi-worker accept() balancing

Warning: tcp_tw_recycle was removed in kernel 4.12 because it broke connections behind NAT. Never use it.

2.4 UDP, ICMP, and Why Blocking ICMP Breaks Things

UDP provides connectionless, best-effort delivery. No handshake, no acknowledgment, no congestion control. It is essential for DNS (port 53), NTP (port 123), DHCP (ports 67/68), SNMP (ports 161/162), syslog (port 514), and increasingly for QUIC (port 443). The lack of head-of-line blocking makes UDP ideal for real-time applications (VoIP, gaming, video).

ICMP is the control plane of IP. Types you must allow:

  • Type 3 (Destination Unreachable) — especially Code 4 (Fragmentation Needed): required for Path MTU Discovery
  • Type 11 (Time Exceeded) — required for traceroute to function
  • Type 0/8 (Echo Reply/Request) — ping, the most basic reachability test

Blocking all ICMP is a common but harmful security misconception. RFC 4890 provides guidance on filtering ICMPv6, and the principle extends to v4: allow error messages, rate-limit echo if desired, but never drop unreachables.

2.5 IPv4 Exhaustion and IPv6 Adoption

IANA exhausted its free pool of IPv4 /8 blocks in 2011. Regional Internet Registries (RIRs) followed: APNIC (2011), RIPE (2012), LACNIC (2014), ARIN (2015), AFRINIC (2017). The IPv4 transfer market now prices a /24 (256 addresses) at approximately $40-60 per address.

As of early 2026, global IPv6 adoption stands at approximately 45-49% of traffic to Google (varying by day of week — higher on weekends due to mobile use). France leads with 86% IPv6 penetration. Germany and India run majority IPv6 traffic. The United States, Brazil, and Japan hover around 50%. Asia-Pacific economies account for 64% of global IPv6 users, with an aggregate capability of 50%.

Transition mechanisms for sysadmins:

  • Dual-stack: Run IPv4 and IPv6 simultaneously. The simplest approach, but requires double the firewall rules, monitoring, and DNS records (A + AAAA).
  • NAT64/DNS64: Allows IPv6-only clients to reach IPv4 servers. The DNS64 server synthesizes AAAA records, and NAT64 translates at the gateway. Reduces IPv4 dependency.
  • 464XLAT: Combines customer-side NAT46 (CLAT) with NAT64 (PLAT) so legacy IPv4 apps work on IPv6-only networks. Used extensively by mobile carriers.

3. DNS

3.1 Recursive vs. Authoritative

DNS has two fundamentally different server roles. Authoritative servers hold the zone data and answer queries for their domains definitively. Recursive (resolver) servers accept queries from clients, traverse the DNS hierarchy starting from root servers, cache answers, and return them to clients. A resolver asks the root for .com, .com for example.com, then example.com's authoritative server for the final answer.

Never run an open recursive resolver facing the internet — it will be exploited for DNS amplification attacks. If you run a resolver, restrict it to your network (Unbound's access-control, BIND's allow-recursion).

3.2 Zone Files and Record Types

A zone file defines the DNS records for a domain. The essential record types:

CNAME restrictions: A CNAME cannot coexist with any other record type at the same name. This means you cannot use CNAME at the zone apex (where SOA and NS records exist). Solutions include ALIAS/ANAME records (provider-specific) or flattening (Cloudflare's approach).

3.3 DNSSEC Chain of Trust

DNSSEC adds cryptographic signatures to DNS, preventing cache poisoning. The chain of trust works through three record types:

  • DNSKEY — Contains the zone's public keys. Two types exist: the Key Signing Key (KSK, flag 257) signs other DNSKEY records; the Zone Signing Key (ZSK, flag 256) signs all other records.
  • RRSIG — A digital signature over a resource record set. Every answer includes an RRSIG that a resolver validates using the matching DNSKEY.
  • DS (Delegation Signer) — A hash of the child zone's KSK, published in the parent zone. This links the chain: the root signs .com's DS, .com signs example.com's DS, and so on up to the root trust anchor, which is validated through the IANA Root Signing Ceremony.

Verify DNSSEC with dig:

# Check DNSKEY records
dig @8.8.8.8 example.com DNSKEY +dnssec +short

# Trace the full chain
dig @8.8.8.8 example.com A +dnssec +trace

# Check DS records at parent
dig @8.8.8.8 example.com DS +short

3.4 DNS-over-HTTPS (DoH) and DNS-over-TLS (DoT)

Traditional DNS is unencrypted UDP on port 53, visible to any on-path observer. DoT (RFC 7858) wraps DNS in TLS on port 853. DoH (RFC 8484) tunnels DNS inside HTTPS on port 443, making it indistinguishable from normal web traffic. For sysadmins operating internal resolvers, DoT is generally preferred: it uses a dedicated port (easy to firewall and monitor), while DoH is harder to control since it blends with HTTPS traffic.

3.5 Split-Horizon DNS

Split-horizon (or split-brain) DNS returns different answers for the same query depending on the source. Internal clients resolve app.example.com to a private IP (10.0.1.50), while external clients get the public IP (203.0.113.50). Implemented via views in BIND, response policies in Unbound, or separate zones in CoreDNS. Essential for environments with private services behind NAT.

4. DHCP

4.1 The DORA Process

DHCP assigns IP addresses through a four-step broadcast handshake known as DORA:

  1. Discover — Client broadcasts (255.255.255.255) from 0.0.0.0, seeking any DHCP server. The Ethernet frame uses broadcast MAC (ff:ff:ff:ff:ff:ff).
  2. Offer — One or more servers respond with an available IP address, subnet mask, gateway, DNS servers, and lease duration.
  3. Request — Client broadcasts its acceptance of a specific offer (it broadcasts so other servers know to release their offered addresses).
  4. Acknowledge — The chosen server confirms the lease. The client configures its interface.

Lease renewal: At 50% of lease time (T1), the client unicasts a renewal request to the original server. At 87.5% (T2), if no response, it broadcasts a rebind request to any server. If the lease expires, the client must restart the DORA process.

4.2 Relay Agents

DHCP broadcasts do not cross router boundaries. In multi-subnet environments, a DHCP relay agent (configured on the router or a server in each subnet) converts broadcasts to unicast packets directed at the DHCP server. In Cisco IOS, this is ip helper-address. On Linux, dhcrelay (ISC) or the relay functionality in Kea.

4.3 PXE Boot with Options 66/67

For network booting (PXE), DHCP provides two critical options: Option 66 (next-server / TFTP server address) and Option 67 (boot filename). Modern UEFI systems often require different boot files than legacy BIOS, so DHCP servers must detect the client architecture (Option 93) and serve the appropriate filename — pxelinux.0 for BIOS, shimx64.efi for UEFI Secure Boot.

4.4 Server Comparison: ISC DHCP vs. Kea vs. dnsmasq

ISC DHCP (dhcpd) is the legacy standard but reached end-of-life in 2022. Its successor, ISC Kea, features a REST API for dynamic configuration, database backends (MySQL, PostgreSQL, Cassandra) for lease storage, and high-availability with hot-standby failover. dnsmasq combines DNS forwarding and DHCP in a single lightweight binary — ideal for small networks, VMs, and containers, but lacks the scalability for enterprise use.

4.5 IPv6: DHCPv6 vs. SLAAC

IPv6 hosts can obtain addresses through two mechanisms. SLAAC (Stateless Address Autoconfiguration, RFC 4862) uses Router Advertisements (RA) to learn the prefix, then generates the interface identifier from the MAC address (EUI-64) or randomly (privacy extensions, RFC 8981). DHCPv6 provides stateful assignment with additional options (DNS servers, NTP, domain search list). Many deployments use SLAAC for address assignment plus DHCPv6 for options only (the "stateless DHCPv6" model, indicated by the O-flag in the RA).

5. HTTP/TLS

5.1 HTTP/1.1, HTTP/2, and HTTP/3

HTTP/1.1 (1997, RFC 2616/7230-7235): Text-based, one request per TCP connection at a time (head-of-line blocking). Browsers work around this by opening 6-8 parallel TCP connections per domain. Connection: keep-alive reuses connections but does not solve HOL blocking.

HTTP/2 (2015, RFC 7540/9113): Binary framing, multiplexed streams over a single TCP connection, header compression (HPACK), server push. Eliminates application-layer HOL blocking but is still vulnerable to TCP-level HOL blocking: a single lost TCP segment stalls all multiplexed streams.

HTTP/3 (2022, RFC 9114): Replaces TCP with QUIC (RFC 9000), a UDP-based transport with built-in TLS 1.3. Each QUIC stream is independently flow-controlled, so a lost packet only affects its own stream. Connection migration allows seamless handoff when a client's IP changes (WiFi to cellular). As of early 2026, approximately 35% of websites advertise HTTP/3 support (W3Techs). Over 75% of Meta's internet traffic uses QUIC/HTTP/3.

5.2 TLS 1.3 Handshake

TLS 1.3 (RFC 8446) reduces the handshake from two round trips (TLS 1.2) to one. The client sends supported cipher suites and key shares in the ClientHello; the server responds with its key share in the ServerHello. Encrypted application data can flow immediately after one RTT. Removed: RSA key exchange, static DH, CBC mode ciphers, MD5/SHA-1, compression, renegotiation. Only five cipher suites survive:

TLS_AES_256_GCM_SHA384
TLS_AES_128_GCM_SHA256
TLS_CHACHA20_POLY1305_SHA256
TLS_AES_128_CCM_SHA256
TLS_AES_128_CCM_8_SHA256

0-RTT Resumption: When reconnecting to a previously visited server, TLS 1.3 can send encrypted data with zero round trips using a pre-shared key (PSK) from the previous session. The server provides a Session Ticket that encapsulates the PSK; on the next connection, the client includes the ticket and immediately sends encrypted data without waiting for a response. This cuts connection setup time by 30-50% for resumptions. However, 0-RTT data lacks forward secrecy and is vulnerable to replay attacks — servers must ensure 0-RTT requests are idempotent (GET, not POST with side effects).

5.3 Certificate Management

Let's Encrypt and ACME: The ACME protocol (RFC 8555) automates certificate issuance and renewal. Validation methods include HTTP-01 (place a file at /.well-known/acme-challenge/), DNS-01 (create a TXT record — required for wildcards), and TLS-ALPN-01 (prove TLS control). Use certbot or acme.sh for standalone servers; cert-manager for Kubernetes.

OCSP Stapling: Instead of clients checking certificate revocation with the CA (slow, privacy-leaking), the server periodically fetches its own OCSP response and staples it to the TLS handshake. Configure in Nginx:

ssl_stapling on;
ssl_stapling_verify on;
ssl_trusted_certificate /etc/ssl/certs/chain.pem;
resolver 8.8.8.8 8.8.4.4 valid=300s;

HSTS (HTTP Strict Transport Security): The Strict-Transport-Security header tells browsers to only connect via HTTPS for a specified duration. Include includeSubDomains and consider HSTS preloading for production domains.

6. SSH

6.1 Key-Based Authentication

SSH key pairs consist of a private key (stays on the client, never transmitted) and a public key (placed in ~/.ssh/authorized_keys on the server). The server sends a challenge encrypted with the public key; only the holder of the private key can decrypt it. Always use a passphrase and ssh-agent to cache it for the session.

ed25519 vs RSA: Ed25519 keys are 256 bits but offer security equivalent to RSA-3072. They are faster to generate and verify, produce shorter signatures, and are resistant to several side-channel attacks. RSA-4096 remains acceptable but Ed25519 is the modern default:

ssh-keygen -t ed25519 -C "admin@example.com" -f ~/.ssh/id_ed25519

6.2 SSH Agent Forwarding (and Why It Is Dangerous)

Agent forwarding (-A or ForwardAgent yes) makes your local SSH keys available on remote servers, allowing you to hop between machines without copying keys. The danger: anyone with root on the intermediate server can use your forwarded agent socket to authenticate as you to any server your key has access to, for as long as you are connected. Prefer ProxyJump instead.

6.3 ProxyJump and Tunneling

ProxyJump (-J) routes the SSH connection through a bastion host without agent forwarding. Your keys never leave your local machine; the bastion sees only encrypted traffic:

# Command line
ssh -J bastion.example.com target.internal

# In ~/.ssh/config
Host target
    HostName target.internal
    ProxyJump bastion.example.com
    User admin

SSH Tunneling:

  • Local forward (-L): Forward a local port to a remote destination. ssh -L 5432:db.internal:5432 bastion makes the remote database accessible at localhost:5432.
  • Remote forward (-R): Expose a local service through the remote server. ssh -R 8080:localhost:3000 server makes your local dev server accessible at server:8080.
  • Dynamic SOCKS proxy (-D): ssh -D 1080 server creates a SOCKS5 proxy. Configure your browser to use localhost:1080 as a SOCKS proxy to tunnel all traffic through the SSH server.

6.4 SSH Certificates vs. Keys

For fleets beyond 10 servers, SSH certificates solve the scalability problem of authorized_keys. A Certificate Authority (CA) signs short-lived certificates with embedded principals (usernames) and expiration times. Servers trust the CA key, not individual user keys — no need to distribute authorized_keys files. Revocation is handled by a single RevokedKeys file or KRL (Key Revocation List) on each server.

7. Firewalls

7.1 iptables Chains and Tables

iptables organizes rules into tables (filter, nat, mangle, raw) and chains (INPUT, OUTPUT, FORWARD for the filter table). Packets traverse chains based on direction: INPUT for packets destined to the host, OUTPUT for locally-generated packets, FORWARD for routed packets. The nat table has PREROUTING (before routing decision) and POSTROUTING (after routing). Rules are evaluated top-to-bottom; first match wins.

# Classic iptables ruleset
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow established connections
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow SSH, HTTP, HTTPS
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow ICMP (don't break PMTUD!)
iptables -A INPUT -p icmp --icmp-type destination-unreachable -j ACCEPT
iptables -A INPUT -p icmp --icmp-type time-exceeded -j ACCEPT
iptables -A INPUT -p icmp --icmp-type echo-request -m limit --limit 5/sec -j ACCEPT

# Log and drop everything else
iptables -A INPUT -j LOG --log-prefix "IPT-DROP: " --log-level 4
iptables -A INPUT -j DROP

7.2 nftables Migration

nftables replaced iptables in the Linux kernel (3.13+) and is now the default in RHEL 8+, Debian 10+, Ubuntu 20.04+, and Fedora 18+. Key advantages: atomic ruleset loading (no race conditions), sets and maps (one rule matches multiple ports/IPs), unified IPv4/IPv6 handling, and a cleaner syntax.

#!/usr/sbin/nft -f
# /etc/nftables.conf

flush ruleset

table inet filter {
    set allowed_tcp_ports {
        type inet_service
        elements = { 22, 80, 443 }
    }

    chain input {
        type filter hook input priority 0; policy drop;

        # Established/related connections
        ct state established,related accept

        # Loopback
        iif "lo" accept

        # ICMP / ICMPv6
        ip protocol icmp accept
        ip6 nexthdr icmpv6 accept

        # Allowed services (single rule, uses set)
        tcp dport @allowed_tcp_ports accept

        # Log drops
        log prefix "nft-drop: " counter drop
    }

    chain forward {
        type filter hook forward priority 0; policy drop;
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}

Migration path: use iptables-translate to convert individual rules, or iptables-save | iptables-restore-translate for entire rulesets. The iptables-nft compatibility layer uses iptables syntax but nftables kernel backend — a stepping stone, not a destination.

7.3 Connection Tracking (conntrack)

Stateful firewalls track connections in the conntrack table. Each entry stores source/destination IP, ports, protocol, state (NEW, ESTABLISHED, RELATED, INVALID), and timeout. The table size defaults to 65,536 on many systems — insufficient for busy load balancers. Monitor and tune:

# Current entries / maximum
cat /proc/sys/net/netfilter/nf_conntrack_count
cat /proc/sys/net/netfilter/nf_conntrack_max

# Increase maximum (busy LB or NAT gateway)
echo 262144 > /proc/sys/net/netfilter/nf_conntrack_max

# Reduce timeouts for TIME_WAIT
echo 120 > /proc/sys/net/netfilter/nf_conntrack_tcp_timeout_time_wait

# Inspect the table
conntrack -L | head -20

7.4 Cloud Security Groups

Cloud providers implement stateful firewalls at the hypervisor level. AWS Security Groups are stateful (return traffic auto-allowed), default-deny inbound, default-allow outbound, and support up to 60 inbound + 60 outbound rules per SG with 5 SGs per ENI. Azure NSGs support priority-based rule evaluation (100-4096), subnet or NIC association, and augmented security rules with service tags and application security groups. Unlike host firewalls, cloud SGs operate below the OS — misconfigured host iptables and correctly configured SGs (or vice versa) cause confusing failures. Always configure both layers.

8. Load Balancing

8.1 Layer 4 vs. Layer 7

L4 load balancing operates on TCP/UDP: it sees IP addresses and ports but not application content. Decisions are made at connection establishment. Lower latency, higher throughput, protocol-agnostic. Use for databases, SMTP, any non-HTTP protocol.

L7 load balancing operates on HTTP: it can inspect headers, cookies, URL paths, and even request bodies. Enables path-based routing (/api to backend A, /static to backend B), header-based routing (A/B testing via cookie), and content-aware health checks (verify HTTP 200 from /healthz).

    bind *:443 ssl crt /etc/ssl/certs/example.pem alpn h2,http/1.1
    http-request redirect scheme https unless { ssl_fc }

    # Path-based routing
    acl is_api path_beg /api
    use_backend api_servers if is_api
    default_backend web_servers

backend web_servers
    balance roundrobin
    option httpchk GET /healthz
    http-check expect status 200
    server web1 10.0.1.10:8080 check inter 5s rise 2 fall 3 weight 100
    server web2 10.0.1.11:8080 check inter 5s rise 2 fall 3 weight 100
    server web3 10.0.1.12:8080 check inter 5s rise 2 fall 3 weight 50  # half capacity

backend api_servers
    balance leastconn
    option httpchk GET /api/health
    cookie SERVERID insert indirect nocache
    server api1 10.0.2.10:9090 check cookie api1
    server api2 10.0.2.11:9090 check cookie api2

8.4 High Availability with keepalived/VRRP

A single load balancer is a single point of failure. VRRP (Virtual Router Redundancy Protocol) allows two or more nodes to share a virtual IP (VIP). The master holds the VIP; if it fails, the backup takes over within seconds. keepalived implements VRRP on Linux and integrates with LVS for L4 load balancing and health-check scripts for application awareness.

9. Routing

9.1 Static Routes

Static routes are the simplest form: manually defined paths for specific destinations. Sysadmins use them for default gateways, VPN routes, and inter-VLAN routing on servers with multiple interfaces:

# Add a route: traffic to 10.10.0.0/16 goes via 192.168.1.1
ip route add 10.10.0.0/16 via 192.168.1.1

# Default gateway
ip route add default via 192.168.1.1

# Route for a VPN subnet via a specific interface
ip route add 172.16.0.0/12 dev wg0

# Persistent routes (Debian/Ubuntu: /etc/network/interfaces or netplan)
# Persistent routes (RHEL: /etc/sysconfig/network-scripts/route-eth0)

9.2 BGP Fundamentals

Border Gateway Protocol is the routing protocol of the internet. Each Autonomous System (AS) — a network under single administrative control — is identified by an ASN (32-bit, 1-4294967295). BGP peers exchange prefix announcements over TCP port 179, advertising which IP ranges they can reach and the AS path to get there.

Key concepts for sysadmins at multi-homed organizations:

  • eBGP (external BGP): between different ASes. iBGP (internal BGP): within the same AS.
  • Prefix announcements: Only announce your own prefixes and customer prefixes — never re-announce peer or transit prefixes (route leaks cause outages).
  • Max-prefix limits: Safety valve preventing a misconfigured peer from flooding your routing table (the global table has 1M+ prefixes).
  • Looking glass servers: Public BGP route viewers (e.g., RIPE RIS, RouteViews) let you see how your prefixes propagate globally.
  • RPKI/ROA: Resource Public Key Infrastructure cryptographically validates that an AS is authorized to announce a prefix. Adoption is growing — configure your routers to drop RPKI-invalid prefixes.

9.3 OSPF

Open Shortest Path First is an interior gateway protocol (IGP) used within a single organization. It divides the network into areas (Area 0 is the backbone; all other areas must connect to it). OSPF elects a Designated Router (DR) and Backup DR (BDR) on each multi-access segment to reduce flooding overhead. Link-State Advertisements (LSAs) describe the network topology; each router runs Dijkstra's SPF algorithm to compute shortest paths.

Sysadmins encounter OSPF when managing server routing in multi-subnet data centers, especially with container networking (Calico uses BGP, but OSPF-integrated environments exist) and when configuring VPN servers that participate in dynamic routing.

9.4 When Sysadmins Need Routing Knowledge

You will need routing skills for: multi-homed servers (selecting which interface for which traffic), container networking (Docker/Kubernetes overlay networks add routes), VPN configuration (WireGuard AllowedIPs creates routes), policy-based routing (route traffic by source IP or port, not just destination), and VRF (Virtual Routing and Forwarding — multiple independent routing tables on one host).

10. VLANs and Network Segmentation

10.1 802.1Q VLAN Tagging

VLANs create separate broadcast domains on a single physical switch. 802.1Q inserts a 4-byte tag into the Ethernet frame containing a 12-bit VLAN ID (1-4094). Access ports carry untagged traffic for a single VLAN. Trunk ports carry tagged traffic for multiple VLANs between switches or to servers running virtual machines.

On Linux, create a VLAN interface:

# Create VLAN 100 on eth0
ip link add link eth0 name eth0.100 type vlan id 100
ip addr add 10.100.0.5/24 dev eth0.100
ip link set eth0.100 up

10.2 VLAN Design Patterns

VLAN-per-subnet is the most common design: each VLAN maps to one IP subnet. Inter-VLAN routing is handled by a Layer 3 switch or router (router-on-a-stick for small deployments, dedicated L3 switch for production).

Security-driven segmentation:

  • DMZ: Public-facing servers in a separate VLAN with restrictive firewall rules — only allow inbound HTTP/HTTPS, restrict egress to specific internal services.
  • PCI-DSS zones: Cardholder data environment (CDE) isolated in its own VLAN with strict access controls and monitoring. Segmentation reduces scope of PCI audits.
  • Management VLAN: IPMI/iLO/iDRAC interfaces on a dedicated, non-routable VLAN. These BMC interfaces have a history of critical vulnerabilities — never expose them to the production network.
  • Private VLANs (PVLAN): Restrict communication between ports on the same VLAN. Useful for hosting environments where tenants share a subnet but should not see each other's traffic. Ports are classified as promiscuous (can talk to all), isolated (can only reach promiscuous ports), or community (can talk to same community and promiscuous).

11. VPN Technologies

11.1 WireGuard

WireGuard is a modern VPN protocol in the Linux kernel (since 5.6) with approximately 4,000 lines of code (compared to 600,000+ for OpenVPN). It uses Curve25519 for key exchange, ChaCha20 for encryption, Poly1305 for authentication, and BLAKE2s for hashing — a fixed, opinionated cryptographic selection with no cipher negotiation.

Performance benchmarks (2025): WireGuard achieves 920-960 Mbps throughput with 1-3ms latency overhead and 8-15% CPU usage, compared to OpenVPN's 650-780 Mbps with 8-12ms latency overhead and 45-60% CPU usage. WireGuard consistently achieves 95%+ of bare metal speed. On mobile networks, WireGuard reconnects in 1-3 seconds versus OpenVPN's 5-15 seconds.

# /etc/wireguard/wg0.conf — Server
[Interface]
PrivateKey = <server-private-key>
Address = 10.0.0.1/24
ListenPort = 51820
PostUp = iptables -A FORWARD -i wg0 -j ACCEPT; iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE

[Peer]
PublicKey = <client-public-key>
AllowedIPs = 10.0.0.2/32

# /etc/wireguard/wg0.conf — Client
[Interface]
PrivateKey = <client-private-key>
Address = 10.0.0.2/24
DNS = 10.0.0.1

[Peer]
PublicKey = <server-public-key>
Endpoint = vpn.example.com:51820
AllowedIPs = 0.0.0.0/0, ::/0    # Full tunnel
PersistentKeepalive = 25

11.2 OpenVPN

OpenVPN operates in user space, supporting both TUN (L3, routed) and TAP (L2, bridged) modes. It uses OpenSSL for crypto and supports certificate-based authentication (PKI with easy-rsa or step-ca). The flexibility is both its strength and weakness: more configuration surface means more places to misconfigure. Still widely deployed because of its maturity, extensive audit history, and ability to traverse restrictive firewalls by running on TCP port 443.

11.3 IPsec (IKEv2)

IPsec is a kernel-level framework using IKEv2 (RFC 7296) for key negotiation and ESP (Encapsulating Security Payload) for encryption. Native support on every major OS (Windows, macOS, iOS, Android, Linux via strongSwan). Primarily used for site-to-site VPNs between routers/firewalls. More complex to configure than WireGuard but deeply integrated into enterprise network equipment.

11.4 Modern Mesh VPNs

Tailscale builds on WireGuard with a coordination server that handles key distribution, NAT traversal (DERP relays), and ACLs. Zero-configuration for users — install, authenticate, connected. Ideal for distributed teams accessing internal services.

Nebula (developed by Slack) creates a peer-to-peer mesh network with certificate-based identity. Nodes communicate directly when possible, with lighthouse nodes for discovery. Self-hosted, no external coordination dependency.

Split tunneling: Route only specific subnets through the VPN (WireGuard's AllowedIPs = 10.0.0.0/8) versus routing all traffic (AllowedIPs = 0.0.0.0/0). Split tunneling reduces VPN server load and preserves local network access but means non-tunneled traffic bypasses corporate security controls.

12. Network Troubleshooting

12.1 Systematic Layer-by-Layer Approach

When a network problem is reported, work from the bottom up:

  1. L1 (Physical): Is the link up? ip link show, check for "state UP", verify cable/SFP, check link speed (ethtool eth0).
  2. L2 (Data Link): Is ARP resolving? ip neigh show. Are VLANs correct? Is the switch port in the right VLAN?
  3. L3 (Network): Is the IP configured? ip addr show. Can you ping the gateway? Is the route correct? ip route get <destination>.
  4. L4 (Transport): Is the port open? ss -tlnp. Is the firewall allowing traffic? nft list ruleset or iptables -L -n -v. Is conntrack full?
  5. L7 (Application): Is the service responding correctly? curl -v, check application logs.

12.3 Common Problems and Diagnosis

MTU issues: TCP handshake works (small packets) but data transfer stalls. Test with ping -M do -s 1472 target (1472 + 28 bytes header = 1500). If it fails, reduce payload size until it succeeds — that reveals the path MTU. Fix by adjusting interface MTU or enabling TCP MSS clamping on the firewall:

iptables -t mangle -A FORWARD -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu

Asymmetric routing: Packets take different paths in each direction. Causes problems with stateful firewalls (return packets arrive on a different interface, conntrack has no entry, packets are dropped). Diagnose with tcpdump on both interfaces. Fix with policy routing or by adjusting firewall rules.

DNS failures: Intermittent resolution failures often indicate: a misconfigured resolver (/etc/resolv.conf overwritten by DHCP or systemd-resolved), DNSSEC validation failures (broken chain on the authoritative side), or DNS caching serving stale records after a zone change. Always check from multiple resolvers: dig @8.8.8.8, dig @1.1.1.1, dig @your-local-resolver.

TLS certificate chain errors: The server must send its certificate and all intermediate certificates. Missing intermediates cause failures in clients that do not have the intermediate cached. Test with openssl s_client -connect host:443 -showcerts and verify the chain with openssl verify -CAfile ca-bundle.crt server.crt.
