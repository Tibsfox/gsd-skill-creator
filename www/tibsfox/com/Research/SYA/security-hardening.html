"The only truly secure system is one that is powered off, cast in a block of concrete, and sealed in a lead-lined room with armed guards." — Gene Spafford, Purdue University

Security hardening is the practice of reducing a system's attack surface by eliminating unnecessary software, tightening configurations, applying patches, and enforcing access controls. It is not a one-time task—it is a continuous discipline that spans the full lifecycle of every system you operate. This module covers the major domains of systems security: from industry benchmarks and patch management through intrusion detection, mandatory access controls, cryptographic infrastructure, secrets management, and compliance frameworks.

1. CIS Benchmarks

1.1 What CIS Benchmarks Are

The Center for Internet Security (CIS) publishes consensus-based configuration guidelines for over 100 technologies, including operating systems, cloud platforms, databases, and container runtimes. Each benchmark is a document containing hundreds of specific configuration recommendations, developed through collaboration between security practitioners, vendors, and government agencies.

CIS benchmarks are the most widely adopted hardening standards in the industry. They are referenced by NIST, PCI DSS, HIPAA, FedRAMP, and SOC 2 auditors as evidence of due diligence.

1.2 Level 1 vs. Level 2

ProfileScopeImpactUse Case
Level 1Essential security hardening that can be implemented without significant performance impact or reduced functionality.LowAll production systems. This is your baseline.
Level 2Defense-in-depth measures that may reduce functionality or performance. More restrictive access controls, additional auditing, and stricter network policies.ModerateHigh-security environments: financial systems, healthcare, government.

Each level further subdivides into Server and Workstation profiles. A production database server should target Level 1 Server at minimum; a workstation used for software development has different requirements.

1.3 CIS-CAT Scanning

CIS-CAT Pro is the official assessment tool. It scans a system against a selected benchmark and produces a compliance report showing pass/fail/not-applicable status for each recommendation. The free CIS-CAT Lite covers a subset of recommendations. For automated scanning at scale, most organizations integrate CIS-CAT into their CI/CD pipeline or use configuration management tools (Ansible, Puppet, Chef) to both enforce and validate compliance.

1.4 Automated Hardening with Ansible

The Ansible Lockdown project provides pre-built Ansible roles aligned with CIS benchmarks for major platforms:

  • UBUNTU24-CIS — Ubuntu 24.04 LTS
  • UBUNTU22-CIS — Ubuntu 22.04 LTS
  • RHEL9-CIS — Red Hat Enterprise Linux 9
  • RHEL8-CIS — Red Hat Enterprise Linux 8

Each role contains variables for every CIS recommendation, allowing you to enable or disable individual controls based on your environment:

# group_vars/all.yml for UBUNTU24-CIS
# Level 1 Server (default)
ubtu24cis_level: 1

# Disable controls that conflict with our environment
ubtu24cis_rule_1_1_1_1: true    # Ensure cramfs is disabled
ubtu24cis_rule_1_1_1_2: true    # Ensure freevxfs is disabled
ubtu24cis_rule_5_2_4: true      # Ensure SSH MaxAuthTries is set to 4
ubtu24cis_rule_5_2_6: false     # SKIP: SSH X11 forwarding (we need it)
ubtu24cis_rule_6_1_10: true     # Ensure no world-writable files exist

# Run with check mode first
# ansible-playbook -i inventory site.yml --check --diff
# Then apply
# ansible-playbook -i inventory site.yml

Critical workflow: Always run in check mode (--check --diff) first to see what would change. Apply in a staging environment, validate that services still function, then roll to production. CIS hardening can break applications that depend on insecure defaults.

1.5 Common CIS Findings

Across Ubuntu and RHEL systems, the most frequent findings are:

  • Unnecessary services enabled: avahi-daemon, cups, rpcbind running on servers that do not need them.
  • Weak file permissions: World-readable cron files, world-writable directories outside /tmp, overly permissive /etc/shadow permissions.
  • Missing audit rules: auditd not configured to log privilege escalation, file system changes, or login events.
  • Password policy: No password complexity requirements, no login attempt lockout, no password aging.
  • Kernel parameters: IP forwarding enabled, ICMP redirects accepted, source routing not disabled.

2. Patch Management

2.1 The Vulnerability Lifecycle

Understanding the timeline from vulnerability discovery to patch deployment is essential for prioritizing work:

  1. Discovery: A researcher or attacker finds a vulnerability.
  2. Disclosure: Responsible disclosure to the vendor (or public disclosure if zero-day).
  3. CVE Assignment: MITRE assigns a Common Vulnerabilities and Exposures identifier (e.g., CVE-2024-3094 for the xz backdoor).
  4. CVSS Scoring: The National Vulnerability Database (NVD) assigns a Common Vulnerability Scoring System score (0.0-10.0).
  5. Vendor Patch: The software vendor releases a fix.
  6. Distribution Patch: Linux distributions package the fix into their repositories.
  7. Deployment: You apply the patch to your systems.

The window between steps 3 and 7 is your exposure window. Every day that window is open, your systems are vulnerable to known exploits.

2.2 CVSS Scoring

Score RangeSeverityPatch Timeline
9.0 – 10.0CriticalEmergency: patch within 24-72 hours. Consider emergency maintenance window.
7.0 – 8.9HighPatch within 7 days. Prioritize internet-facing systems.
4.0 – 6.9MediumPatch within 30 days during regular maintenance windows.
0.1 – 3.9LowPatch within 90 days or next scheduled maintenance cycle.

CVSS is not the whole story. A CVSS 7.0 vulnerability in an internet-facing web server is far more urgent than a CVSS 9.0 in an air-gapped system. Prioritize based on: CVSS score, exploitability (is there a public exploit?), exposure (internet-facing vs. internal), and asset criticality.

2.4 Kernel Live Patching

Rebooting production systems for kernel patches introduces downtime. Live patching technologies apply kernel fixes without restarting:

  • Canonical Livepatch: Service for Ubuntu systems. Patches critical and high-severity kernel CVEs in-memory. Free for up to 5 machines (Ubuntu Pro required for more).
  • kpatch (Red Hat): Builds and applies binary patches to the running kernel. Used by RHEL's kernel live patching service.
  • kgraft (SUSE): Similar technology for SUSE Linux Enterprise.

Live patching is not a replacement for regular reboots—some patches require a full restart—but it dramatically reduces the urgency of scheduling maintenance windows for routine kernel CVEs.

3. Intrusion Detection

3.1 Host-Based IDS (HIDS)

Host-based intrusion detection monitors activity on individual hosts: file integrity, log analysis, rootkit detection, and policy compliance.

OSSEC / Wazuh: Wazuh is the actively-maintained fork of OSSEC. It provides real-time log analysis, file integrity monitoring (FIM), rootkit detection, active response (automatic blocking of malicious IPs), and vulnerability detection. Wazuh agents run on monitored hosts and report to a central Wazuh server.

# /var/ossec/etc/ossec.conf (Wazuh agent)
<ossec_config>
  <syscheck>
    <!-- File integrity monitoring -->
    <directories check_all="yes" realtime="yes">/etc,/usr/bin,/usr/sbin</directories>
    <directories check_all="yes" realtime="yes">/boot</directories>
    <ignore>/etc/mtab</ignore>
    <ignore>/etc/hosts.deny</ignore>
    <frequency>7200</frequency>
  </syscheck>

  <rootcheck>
    <rootkit_files>/var/ossec/etc/rootcheck/rootkit_files.txt</rootkit_files>
    <rootkit_trojans>/var/ossec/etc/rootcheck/rootkit_trojans.txt</rootkit_trojans>
  </rootcheck>
</ossec_config>

AIDE (Advanced Intrusion Detection Environment): Lightweight file integrity checker. Creates a database of file hashes and attributes, then compares the current state against the database on subsequent runs. Simpler than Wazuh but lacks real-time monitoring and active response.

# Initialize AIDE database
sudo aide --init
sudo mv /var/lib/aide/aide.db.new /var/lib/aide/aide.db

# Check for changes (run via cron daily)
sudo aide --check

# After legitimate changes, update the database
sudo aide --update

3.2 Network-Based IDS/IPS (NIDS)

Network-based intrusion detection inspects network traffic for malicious patterns.

Suricata: The dominant open-source NIDS/IPS. Multi-threaded, high-performance, supports deep packet inspection, TLS/JA3 fingerprinting, and can operate in IDS (passive) or IPS (inline blocking) mode. Suricata uses the Emerging Threats (ET) ruleset, which contains thousands of signatures for known attack patterns.

# Example Suricata rule: detect SSH brute force
alert ssh $EXTERNAL_NET any -> $HOME_NET 22 (
    msg:"ET SCAN Potential SSH Brute Force";
    flow:to_server,established;
    threshold: type both, track by_src, count 5, seconds 60;
    classtype:attempted-admin;
    sid:2019876; rev:3;
)

# Example: detect SQL injection in HTTP traffic
alert http $EXTERNAL_NET any -> $HOME_NET any (
    msg:"ET WEB_SERVER SQL Injection Attempt - UNION SELECT";
    flow:to_server,established;
    http.uri;
    content:"UNION"; nocase;
    content:"SELECT"; nocase; distance:0;
    classtype:web-application-attack;
    sid:2019877; rev:1;
)

Snort 3: The original open-source IDS (now maintained by Cisco). Version 3 is a complete rewrite with multi-threading, Lua scripting, and improved performance. Shares rule format compatibility with Suricata.

3.3 Signature vs. Anomaly Detection

Signature-based: Matches traffic or behavior against known patterns (rules). High accuracy for known threats, zero detection of novel attacks. Requires constant rule updates.

Anomaly-based: Builds a baseline of "normal" behavior and alerts on deviations. Can detect unknown threats but produces more false positives. Machine learning models are increasingly used for anomaly detection, but they require tuning and retraining as normal behavior evolves.

Production deployments use both: signatures catch the known threats, anomaly detection provides a safety net for the unknown.

3.4 The Detection Engineering Discipline

Detection engineering treats intrusion detection rules as code: versioned, tested, reviewed, and continuously improved. This is a shift from the traditional approach of enabling a vendor ruleset and hoping for the best.

The detection engineering lifecycle:

  1. Hypothesis: Based on threat intelligence or ATT&CK techniques, identify a behavior you want to detect (e.g., "detect Kerberoasting attempts via anomalous TGS requests").
  2. Data source validation: Confirm that the required log sources are collected and parsed correctly. You cannot detect what you cannot see.
  3. Rule development: Write the detection logic. Test against known-good and known-bad samples.
  4. Tuning: Deploy in detection-only mode. Measure false positive rate. Adjust thresholds, add exclusions for known-good behavior.
  5. Production deployment: Enable alerting. Document the expected false positive rate, investigation steps, and response procedures.
  6. Continuous improvement: Review alert quality monthly. Track metrics: true positive rate, false positive rate, mean time to detect, mean time to respond.

3.5 False Positive Management

The curse of intrusion detection is false positives. A rule that fires 100 times a day on legitimate traffic trains operators to ignore it, which means a real attack hidden in the noise goes unnoticed. Systematic management:

  • Track false positive rates per rule. A rule with greater than 90% false positives needs immediate tuning or removal.
  • Tune threshold values rather than disabling rules entirely. Raising a brute force threshold from 3 to 10 attempts may eliminate noise while still catching real attacks.
  • Use suppression lists for known-good sources. Your vulnerability scanner's IP should not trigger IDS alerts during scheduled scans.
  • Review and update rules monthly. As infrastructure changes, detection rules must evolve.
  • Tag rules with MITRE ATT&CK technique IDs for context. This enables coverage gap analysis—you can see which techniques have no detection rules.
  • Maintain a detection coverage matrix: map your detection rules against ATT&CK techniques and identify blind spots. A SIEM with 500 rules that all detect Initial Access but none detect Lateral Movement has a critical gap.

4. SIEM

4.1 Architecture

A Security Information and Event Management (SIEM) system aggregates logs from across your infrastructure (servers, firewalls, IDS, applications, cloud services), normalizes them into a common format, correlates events across sources, and applies detection rules to identify security incidents.

Core components:

  • Log ingestion: Collect from agents (Wazuh, Beats, Fluentd), syslog, APIs, cloud trails.
  • Normalization: Parse diverse log formats into structured fields (source IP, destination port, user, action).
  • Correlation engine: Match patterns across multiple events and sources.
  • Detection rules: Signatures, thresholds, and statistical anomalies that trigger alerts.
  • Case management: Track investigation and response for triggered alerts.

4.3 MITRE ATT&CK Framework

MITRE ATT&CK is a knowledge base of adversary tactics, techniques, and procedures (TTPs) based on real-world observations. It organizes attack behavior into a matrix of 14 tactics (the "why"—Reconnaissance, Initial Access, Execution, Persistence, etc.) and hundreds of techniques (the "how"—Phishing, PowerShell, Scheduled Tasks, etc.).

For SIEM operators, ATT&CK provides:

  • Detection rule mapping: Tag every detection rule with the ATT&CK technique it detects. This reveals coverage gaps—if you have no rules for Lateral Movement techniques, that is a blind spot.
  • Threat intelligence integration: Threat reports describe adversary TTPs using ATT&CK IDs, allowing you to map intelligence directly to your detection rules.
  • Purple teaming: Red teams emulate specific ATT&CK techniques while blue teams validate detection coverage.

4.4 Common Detection Use Cases

# Wazuh rule: detect brute force (5+ failed logins in 60 seconds)
<group name="authentication_failures">
  <rule id="100100" level="10" frequency="5" timeframe="60">
    <if_matched_group>authentication_failed</if_matched_group>
    <description>Multiple authentication failures from same source</description>
    <mitre>
      <id>T1110</id>  <!-- Brute Force -->
    </mitre>
  </rule>
</group>

Key detection scenarios:

  • Brute force: Multiple failed authentication attempts from a single source within a time window (T1110).
  • Lateral movement: Authentication from one internal host to many others in rapid succession, especially using service accounts (T1021).
  • Data exfiltration: Unusual outbound data volumes, especially to uncommon destinations or over DNS/ICMP tunnels (T1048).
  • Privilege escalation: sudo usage by unexpected users, modification of /etc/sudoers, setuid bit changes (T1548).
  • Persistence: New cron jobs, systemd services, or SSH authorized_keys entries created by non-administrative users (T1053, T1098).

4.5 SOAR Integration

Security Orchestration, Automation, and Response (SOAR) platforms automate the response to SIEM alerts. When a brute force alert fires, the SOAR playbook can automatically: enrich the alert with threat intelligence (is the source IP known-malicious?), block the IP at the firewall, create an investigation ticket, and notify the security team. This reduces mean time to response from minutes to seconds for common attack patterns.

A practical SOAR workflow for SSH brute force detection:

  1. Trigger: Wazuh generates alert (rule 100100, multiple authentication failures).
  2. Enrichment: Query AbuseIPDB and VirusTotal for the source IP. Check if it appears in internal asset inventory (is this a legitimate internal system?).
  3. Decision: If the IP is external and has a high abuse confidence score, proceed to block. If internal, escalate to the security team for investigation (possible compromised host or credential stuffing from inside the network).
  4. Response: Add the IP to the firewall blocklist. Create a Jira ticket with the enrichment data. Send a Slack notification to the security channel.
  5. Documentation: Log all actions taken in the case management system for audit trail and post-incident review.

Open-source SOAR options include Wazuh's built-in active response module, Shuffle (a visual SOAR platform), and TheHive + Cortex (case management with automated analysis). For smaller teams, even a set of well-written scripts triggered by webhook integrations can provide significant response automation.

5.2 Key Types: Ed25519 Is the Standard

Ed25519 is the recommended key type for SSH in 2026. It produces compact 256-bit keys, is faster to generate and verify than RSA, has a simpler implementation with fewer side-channel attack opportunities, and is supported by OpenSSH 6.5+ (released 2014, so effectively universal).

# Generate an Ed25519 key
ssh-keygen -t ed25519 -C "admin@example.com" -f ~/.ssh/id_ed25519

# For systems that still require RSA (legacy), use 4096-bit minimum
ssh-keygen -t rsa -b 4096 -C "admin@example.com"

5.3 SSH Certificates

SSH certificates solve the "known_hosts management at scale" problem. Instead of distributing individual public keys to every server, you create a Certificate Authority (CA), sign user keys with it, and configure servers to trust the CA. Benefits:

  • No more known_hosts warnings when servers are rebuilt.
  • Certificates can have expiration dates (e.g., 8 hours for temporary access).
  • Principals define which usernames a certificate is valid for.
  • Centralized revocation via a Key Revocation List (KRL).
# Create a CA key pair
ssh-keygen -t ed25519 -f /etc/ssh/ca_user_key -C "SSH User CA"

# Sign a user's public key (valid for 8 hours, for user "deploy")
ssh-keygen -s /etc/ssh/ca_user_key \
  -I "deploy-20260408" \
  -n deploy \
  -V +8h \
  ~/.ssh/id_ed25519.pub

# Server configuration: trust the CA
# /etc/ssh/sshd_config
TrustedUserCAKeys /etc/ssh/ca_user_key.pub

5.4 FIDO2 / U2F Hardware Keys

OpenSSH 8.2+ supports FIDO2 security keys (YubiKey 5, SoloKeys, Google Titan) as SSH authentication factors. The private key material never leaves the hardware device, making key theft via malware impossible.

# Generate an SSH key backed by a FIDO2 security key
ssh-keygen -t ed25519-sk -C "admin@example.com"
# -sk suffix indicates the key requires a security key for signing

# For resident keys (stored on the security key itself)
ssh-keygen -t ed25519-sk -O resident -C "admin@example.com"

# The key works like any other SSH key but requires
# physical touch of the security key for each authentication

5.5 Fail2ban and sshguard

Fail2ban monitors log files for failed authentication attempts and temporarily bans offending IP addresses using firewall rules (iptables, nftables, or firewalld).

# /etc/fail2ban/jail.local
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
findtime = 600    # 10-minute window
bantime = 3600    # 1-hour ban
action = %(action_mwl)s  # Ban + mail with whois + log lines

[sshd-aggressive]
enabled = true
port = ssh
filter = sshd[mode=aggressive]
logpath = /var/log/auth.log
maxretry = 1
findtime = 86400    # 24-hour window
bantime = 604800    # 1-week ban for repeat offenders

5.6 Bastion Hosts / Jump Servers

Never expose SSH on critical servers directly to the internet. Use a bastion host—a minimal, hardened server that serves as the sole entry point to your network.

# ~/.ssh/config on the operator's workstation
Host bastion
    HostName bastion.example.com
    User admin
    IdentityFile ~/.ssh/id_ed25519
    Port 22

Host production-*
    ProxyJump bastion
    User deploy
    IdentityFile ~/.ssh/id_ed25519

Host production-web01
    HostName 10.0.1.10

Host production-db01
    HostName 10.0.2.10

# Usage: ssh production-web01
# Transparently connects through bastion

Security through obscurity debate: Changing the SSH port from 22 to a non-standard port reduces automated scanning noise but provides no real security against targeted attacks. It is a noise-reduction measure, not a security control. Do it if you want cleaner logs, but never rely on it as your only defense.

5.7 SSH Audit Tools

Validating your SSH configuration is as important as writing it. Two essential tools:

ssh-audit is an open-source tool that analyzes SSH server and client configurations, identifying weak algorithms, outdated key exchange methods, and known vulnerabilities:

# Install and run ssh-audit against your server
pip install ssh-audit
ssh-audit example.com

# Output highlights:
# (rec) remove diffie-hellman-group14-sha1 -- using weak SHA-1 hash
# (rec) remove hmac-sha1 -- using weak SHA-1 hash
# (rec) add chacha20-poly1305@openssh.com -- modern AEAD cipher
# (info) OpenSSH 9.6 running on Ubuntu 24.04

Lynis is a comprehensive security auditing tool that checks SSH configuration as part of a broader system hardening assessment. It produces a hardening index and specific recommendations aligned with CIS benchmarks:

# Run Lynis SSH audit
sudo lynis audit system --tests-from-group "ssh"

# Or a full system audit that includes SSH
sudo lynis audit system

Run these audit tools after any SSH configuration change and as part of your regular security review cycle (quarterly at minimum). Integrate them into your CI/CD pipeline if you manage SSH configurations via Ansible or similar tools—audit the configuration before deploying it to production.

6. Mandatory Access Control

6.1 Why MAC Matters

Discretionary Access Control (DAC)—the standard Unix permission model (user/group/other, read/write/execute)—has a fundamental weakness: any process running as root can access anything. If an attacker exploits a vulnerability in a root-running service, DAC provides no containment.

Mandatory Access Control (MAC) enforces security policies that even root cannot override. The kernel enforces access decisions based on security labels, not just file ownership. A compromised Apache process running as root but confined by SELinux to the httpd_t type still cannot read /etc/shadow—the policy does not permit it.

6.2 SELinux

SELinux (Security-Enhanced Linux) is the most comprehensive MAC implementation, developed by the NSA and integrated into the Linux kernel. It is the default on RHEL, CentOS, Fedora, and Amazon Linux.

Security contexts follow the format user:role:type:level. For most administration, the type is the critical field:

# View the SELinux context of a file
ls -Z /var/www/html/index.html
# Output: system_u:object_r:httpd_sys_content_t:s0 /var/www/html/index.html

# View the SELinux context of a running process
ps -eZ | grep httpd
# Output: system_u:system_r:httpd_t:s0  1234 ?  00:00:01 httpd

# The httpd_t process type can read httpd_sys_content_t file type
# But cannot read user_home_t or shadow_t

Booleans are toggles that enable or disable specific policy features without writing custom rules:

# List all booleans related to httpd
getsebool -a | grep httpd

# Allow Apache to connect to the network (for reverse proxy)
setsebool -P httpd_can_network_connect on

# Allow Apache to send email
setsebool -P httpd_can_sendmail on

Troubleshooting: When SELinux blocks an operation, it logs an AVC (Access Vector Cache) denial to /var/log/audit/audit.log. The diagnostic tools:

# Find recent denials
ausearch -m avc -ts recent

# Analyze a denial and get a human-readable explanation
audit2why < /var/log/audit/audit.log

# Generate a policy module to allow the denied action
# WARNING: Review the output before applying!
audit2allow -a -M my_custom_policy
semodule -i my_custom_policy.pp

Never blindly run audit2allow and install the result. Review each allowed access. The tool generates the minimum policy to permit what was denied, but that denial might be exactly the security control you want.

6.3 AppArmor

AppArmor is the default MAC system on Ubuntu, Debian, and SUSE. It uses path-based access control (rather than SELinux's label-based approach), which makes it simpler to understand but less granular.

AppArmor profiles define what a program can access:

# /etc/apparmor.d/usr.sbin.nginx
#include <tunables/global>

/usr/sbin/nginx {
  #include <abstractions/base>
  #include <abstractions/nameservice>

  # Binary
  /usr/sbin/nginx mr,

  # Configuration
  /etc/nginx/** r,

  # Log files
  /var/log/nginx/** w,

  # Web content (read only)
  /var/www/** r,

  # PID and socket files
  /run/nginx.pid rw,
  /run/nginx/ rw,

  # Network
  network inet stream,
  network inet6 stream,

  # Deny everything else implicitly
}

Modes:

  • Enforce: Violations are blocked and logged.
  • Complain: Violations are logged but allowed. Use this for developing and testing new profiles.
# Put a profile in complain mode
sudo aa-complain /etc/apparmor.d/usr.sbin.nginx

# Switch to enforce mode after testing
sudo aa-enforce /etc/apparmor.d/usr.sbin.nginx

# Generate a profile from observed behavior
sudo aa-genprof /usr/sbin/nginx
# Run the application through its normal operations, then finalize the profile

6.4 Seccomp-BPF for Container Hardening

Seccomp (Secure Computing Mode) restricts the system calls a process can make. Docker and Kubernetes apply default seccomp profiles that block dangerous syscalls (e.g., mount, reboot, ptrace, kexec_load).

// Custom seccomp profile (JSON) — allow only essential syscalls
{
  "defaultAction": "SCMP_ACT_ERRNO",

Defense in depth: Use seccomp to limit the kernel syscall surface, and AppArmor/SELinux to limit file and network access. Together, they create multiple containment layers that must all be bypassed for a container escape.

6.5 When MAC Saves You from Root Compromise

Consider a real-world scenario: an attacker exploits a remote code execution vulnerability in your Apache web server and gains a shell running as root. Without MAC, it is game over—root can read /etc/shadow, install backdoors, pivot to other systems, and exfiltrate data. With SELinux in enforcing mode:

  • The Apache process runs in the httpd_t domain, regardless of the Unix UID.
  • SELinux policy only allows httpd_t to read files labeled httpd_sys_content_t, write to httpd_log_t, and connect to allowed network ports.
  • The attacker's shell inherits the httpd_t context. It cannot read /etc/shadow (shadow_t type), cannot write to /usr/bin (bin_t type), cannot load kernel modules, and cannot connect to arbitrary network ports for a reverse shell.
  • The attacker has root privileges but is confined to a sandbox. The blast radius is limited to what the web server was supposed to do.

This is why MAC is a critical layer in defense-in-depth. DAC (traditional Unix permissions) assumes root is trusted. MAC does not. In a world where software vulnerabilities are inevitable, MAC limits the damage when prevention fails.

Kubernetes Security Profiles Operator (SPO): For container environments at scale, manually creating seccomp and AppArmor profiles for every workload is impractical. SPO automates this by recording the syscalls and file accesses a workload actually uses during a profiling period, then generating a minimal security profile. This brings the MAC methodology into the container orchestration layer.

7. PKI and Certificate Management

7.1 X.509 Certificate Structure

An X.509 certificate binds a public key to an identity. Key fields:

FieldPurpose
SubjectIdentity of the certificate holder (CN, O, OU, etc.)
IssuerIdentity of the CA that signed the certificate
Subject Alternative Names (SANs)DNS names and IPs the certificate is valid for. Modern browsers require SANs; CN alone is deprecated.
Validity PeriodNot Before and Not After timestamps
Public KeyThe subject's public key (RSA 2048+, ECDSA P-256, or Ed25519)
SignatureThe CA's digital signature over the certificate contents
ExtensionsKey Usage, Extended Key Usage, Basic Constraints, OCSP, CRL

Certificate chains: A server certificate is signed by an intermediate CA, which is signed by a root CA. Clients trust the root CA (stored in their trust store) and can verify the chain. Always configure your server with the full chain (server cert + intermediate), not just the leaf certificate.

7.2 Let's Encrypt and ACME

Let's Encrypt provides free, automated, publicly-trusted TLS certificates via the ACME (Automated Certificate Management Environment) protocol. ACME automates the entire certificate lifecycle: domain validation, certificate issuance, and renewal.

# Certbot: the reference ACME client
# Standalone mode (certbot runs its own web server)
sudo certbot certonly --standalone -d example.com -d www.example.com

# Nginx plugin (certbot modifies nginx config automatically)
sudo certbot --nginx -d example.com

# Renewal (runs automatically via systemd timer or cron)
sudo certbot renew --quiet

# Test renewal without actually renewing
sudo certbot renew --dry-run

Certificate lifetime trends: The industry is pushing toward shorter certificate lifetimes. Let's Encrypt issues 90-day certificates. Google has proposed 90-day maximum certificate validity for all public CAs. Apple proposed reducing validity to 47 days starting in 2027. This makes automation non-optional—manual certificate management is incompatible with 47-day lifecycles.

7.3 Internal CA: step-ca

For internal services (service-to-service mTLS, internal APIs, database connections), you need a private CA. step-ca (by Smallstep) is a lightweight, open-source CA with built-in ACME server support:

# Initialize a CA
step ca init --name "Internal CA" \
  --provisioner admin@example.com \
  --dns ca.internal.example.com \
  --address :443

# Issue a certificate using ACME (just like Let's Encrypt)
step ca certificate api.internal.example.com api.crt api.key \
  --provisioner acme

# Configure applications to use step-ca as their ACME server
# STEP_CA_URL=https://ca.internal.example.com
# Same certbot/cert-manager workflow, different CA

cert-manager (Kubernetes): Automates certificate issuance and renewal in Kubernetes clusters. Supports Let's Encrypt, step-ca, Vault PKI, and other ACME-compatible CAs as issuers:

apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account
    solvers:
      - http01:
          ingress:
            class: nginx
---
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: api-tls
  namespace: production
spec:
  secretName: api-tls-secret
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
  dnsNames:
    - api.example.com
    - api-v2.example.com

7.4 mTLS for Service-to-Service Authentication

Mutual TLS (mTLS) requires both the client and server to present certificates. This provides cryptographic identity verification for every service-to-service connection, eliminating the need for API keys or tokens for internal authentication.

Service meshes (Istio, Linkerd) automate mTLS by injecting sidecar proxies that handle certificate issuance, rotation, and mTLS negotiation transparently to the application.

7.5 Certificate Monitoring

Certificate expiration is a leading cause of outages. Monitor proactively:

# Prometheus blackbox exporter: check certificate expiry
- alert: TLSCertificateExpiringSoon
  expr: probe_ssl_earliest_cert_expiry - time() < 30 * 24 * 3600
  for: 1h
  labels:
    severity: high
  annotations:
    summary: "TLS certificate for {{ $labels.instance }} expires in less than 30 days"

Certificate Transparency (CT) logs: Public, append-only logs of all publicly-trusted certificates. Monitor CT logs for unauthorized certificates issued for your domains using tools like certspotter or crt.sh.

8. Secrets Management

8.1 The Problem

Secrets—database passwords, API keys, TLS private keys, encryption keys, service account tokens—are the most sensitive data in any infrastructure. The default approach (environment variables, .env files, hardcoded strings in configuration) is catastrophically insecure:

  • .env files get committed to Git repositories, leaked in container images, and exposed in process listings.
  • Hardcoded secrets appear in version control history forever, even if removed in a later commit.
  • Shared credentials cannot be rotated without coordinating every system that uses them.
  • Static secrets have no expiration, no audit trail, and no way to revoke access for a single consumer.

8.2 HashiCorp Vault

Vault is the industry standard for centralized secrets management. Its architecture separates storage (where encrypted secrets live) from access (how applications authenticate and retrieve secrets).

Core concepts:

  • Auth methods: How clients prove their identity. Token, AppRole (for machines), Kubernetes (pod service accounts), LDAP, OIDC, AWS IAM, and more.
  • Secret engines: Backends that generate or store secrets. KV (key-value), database (dynamic credentials), PKI (certificates), transit (encryption as a service), SSH (signed keys), AWS/GCP/Azure (cloud IAM credentials).
  • Policies: HCL (HashiCorp Configuration Language) rules that define which paths a token can access and with what capabilities (create, read, update, delete, list).
  • Leases: Every secret has a time-to-live (TTL). When the lease expires, the secret is revoked. This ensures secrets do not live forever.
# Enable the database secret engine for dynamic Postgres credentials
vault secrets enable database

vault write database/config/mydb \
  plugin_name=postgresql-database-plugin \
  connection_url="postgresql://{{username}}:{{password}}@db.example.com:5432/mydb?sslmode=require" \
  allowed_roles="readonly" \
  username="vault_admin" \
  password="initial_password"

vault write database/roles/readonly \
  db_name=mydb \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl="1h" \
  max_ttl="24h"

# Application requests credentials (returns unique username/password, valid for 1 hour)
vault read database/creds/readonly
# Key             Value
# lease_id        database/creds/readonly/abc123
# lease_duration  1h
# username        v-app-readonly-abc123-1234567890
# password        A1b2C3d4E5-random-generated

Transit engine (encryption as a service):

# Create an encryption key
vault write -f transit/keys/payment-data

# Encrypt data (Vault never stores the plaintext)
vault write transit/encrypt/payment-data \
  plaintext=$(echo -n "4111-1111-1111-1111" | base64)
# Returns: vault:v1:encrypted-ciphertext-here

# Decrypt data
vault write transit/decrypt/payment-data \
  ciphertext="vault:v1:encrypted-ciphertext-here"
# Returns: base64-encoded plaintext

8.4 Secret Rotation

Static secrets are a ticking time bomb. Every credential should have a rotation strategy:

  • Dynamic secrets (Vault): Generated on demand with short TTLs. Rotation is automatic—every consumer gets unique, short-lived credentials.
  • Automated rotation: AWS Secrets Manager can rotate RDS passwords on a schedule. Vault's database engine creates fresh credentials per request.
  • Rotation procedure: For static secrets that cannot be made dynamic: issue new credential, update all consumers, verify functionality, revoke old credential. Never delete the old credential before all consumers are updated.

8.5 Zero-Trust Secret Access

The principle: no entity is trusted by default, even inside the network perimeter. Every secret access must be authenticated, authorized, audited, and time-limited. Vault's audit log records every secret read, write, and lease renewal, providing a complete chain of custody for sensitive data.

9.2 SOC 2: Type I vs. Type II

Type I evaluates the design of controls at a point in time. It answers: "Do you have the right controls in place?" An auditor examines your policies, procedures, and technical controls on a specific date.

Type II evaluates the operating effectiveness of controls over a period (typically 6-12 months). It answers: "Do your controls actually work consistently?" An auditor reviews evidence (logs, change records, access reviews) across the entire audit period.

Type II is what customers and partners care about. Type I is a stepping stone. Most organizations start with Type I, then transition to Type II once their controls have been operating for 6+ months.

9.3 PCI DSS: The 12 Requirements

PCI DSS applies to any organization that stores, processes, or transmits payment card data. The 12 requirements (v4.0):

  1. Install and maintain network security controls (firewalls, segmentation).
  2. Apply secure configurations to all system components.
  3. Protect stored account data (encryption at rest, tokenization, masking).
  4. Protect cardholder data with strong cryptography during transmission.
  5. Protect all systems and networks from malicious software (antivirus, EDR).
  6. Develop and maintain secure systems and software (SDLC, patching).
  7. Restrict access to system components and cardholder data by business need-to-know.
  8. Identify users and authenticate access to system components (MFA, strong passwords).
  9. Restrict physical access to cardholder data.
  10. Log and monitor all access to system components and cardholder data.
  11. Test security of systems and networks regularly (vulnerability scans, penetration tests).
  12. Support information security with organizational policies and programs.

9.4 NIST 800-53: Control Families

NIST 800-53 Rev 5 organizes 1,000+ security and privacy controls into 20 families. The families most relevant to systems administrators:

  • AC (Access Control): Account management, least privilege, separation of duties, session controls.
  • AU (Audit and Accountability): Audit logging, log review, log retention, timestamps.
  • CM (Configuration Management): Baseline configurations, change control, software restrictions.
  • IA (Identification and Authentication): Multi-factor authentication, credential management, PKI.
  • IR (Incident Response): Incident handling procedures, reporting, testing.
  • SC (System and Communications Protection): Encryption, network segmentation, boundary protection.
  • SI (System and Information Integrity): Patch management, malware protection, monitoring.

NIST 800-53 controls map to nearly every other framework. Implementing NIST 800-53 gives you a foundation that supports SOC 2, PCI DSS, HIPAA, FedRAMP, CMMC, and ISO 27001 compliance simultaneously.

9.5 Evidence Collection and Audit Preparation

Compliance is not just about having controls—it is about proving they work. Auditors require evidence:

  • Configuration evidence: Screenshots or exports of security settings (firewall rules, IAM policies, encryption configurations).
  • Operational evidence: Logs showing controls in action (access reviews conducted, patches applied, incidents responded to).
  • Policy evidence: Documented policies and procedures, signed by management, reviewed annually.
  • Testing evidence: Vulnerability scan reports, penetration test results, disaster recovery test records.

Automate evidence collection wherever possible. Pull firewall rules from infrastructure-as-code repositories, export access review completions from your IdP, and generate vulnerability scan reports on schedule. Manual evidence collection does not scale and introduces human error.

9.6 Compliance as Code

Compliance as code transforms static control frameworks into executable, testable policies:

Chef InSpec: Write compliance checks as Ruby DSL profiles that run against target systems:

# CIS Ubuntu 24.04 Level 1 - Check SSH root login disabled
control 'cis-5.2.10' do
  impact 1.0
  title 'Ensure SSH root login is disabled'
  desc 'The PermitRootLogin parameter should be set to no'

  describe sshd_config do
    its('PermitRootLogin') { should eq 'no' }
  end
end

# CIS - Check that audit log files are not world-readable
control 'cis-4.1.4.1' do
  impact 1.0
  title 'Ensure audit log files are mode 0640 or less permissive'

  describe file('/var/log/audit/audit.log') do
    it { should_not be_more_permissive_than('0640') }
  end
end

Open Policy Agent (OPA) with Rego: Define policies for infrastructure-as-code, Kubernetes admission control, API authorization, and more:

# Rego policy: deny Kubernetes pods running as root
package kubernetes.admission

deny[msg] {
  input.request.kind.kind == "Pod"
  container := input.request.object.spec.containers[_]
  container.securityContext.runAsUser == 0
  msg := sprintf("Container '%v' must not run as root (UID 0)", [container.name])
}

deny[msg] {
  input.request.kind.kind == "Pod"
  container := input.request.object.spec.containers[_]
  not container.securityContext.runAsNonRoot
  msg := sprintf("Container '%v' must set securityContext.runAsNonRoot to true", [container.name])
}

Regula (Fugue): OPA-based tool for validating Terraform, CloudFormation, and Kubernetes YAML against compliance rules (CIS, NIST, PCI DSS, HIPAA) before deployment.

The goal is to shift compliance left: catch violations in CI/CD before they reach production, rather than discovering them in an annual audit.

Putting It All Together: Defense in Depth

Security hardening is not a checklist you complete once. It is a set of overlapping layers, each compensating for the potential failure of others:

LayerControls
PerimeterFirewalls, bastion hosts, VPN, WAF
NetworkSegmentation, mTLS, IDS/IPS (Suricata)
HostCIS hardening, MAC (SELinux/AppArmor), patch management, HIDS (Wazuh)
ApplicationSecure coding, dependency scanning (Trivy), secrets management (Vault), input validation
DataEncryption at rest and in transit, access controls, backup verification
IdentityMFA, SSH certificates, FIDO2 keys, least privilege, just-in-time access
DetectionSIEM (Wazuh/Elastic), log correlation, MITRE ATT&CK mapping, threat hunting
ResponseIncident response procedures, SOAR automation, forensic readiness
GovernanceCompliance frameworks, audit evidence, policy as code, continuous assessment

No single layer is sufficient. An attacker who bypasses your firewall still faces network segmentation, host-level MAC, application-level input validation, encrypted data at rest, and a SIEM watching for lateral movement. The attacker must bypass every layer; the defender only needs one layer to hold.

"Security is a process, not a product." — Bruce Schneier

Start with the highest-impact, lowest-effort controls: enable automatic security updates, enforce SSH key authentication, apply CIS Level 1, enable a host-based IDS. Then build outward: add network detection, implement secrets management, formalize your compliance posture. Each layer you add raises the cost of a successful attack.
