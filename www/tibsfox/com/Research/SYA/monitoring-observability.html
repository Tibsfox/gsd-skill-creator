Monitoring & Observability for Systems Administrators

PNW Research Series — Systems Administration Deep Dive
Last updated: April 2026 • Covers Prometheus 3.x, OpenTelemetry 1.x, Grafana 11.x, Loki 3.x

"Monitoring tells you whether a system is working. Observability lets you ask why it isn't." — Cindy Sridharan, Distributed Systems Observability

Modern infrastructure generates telemetry at a scale that would have been inconceivable a decade ago. A single Kubernetes cluster running fifty microservices can easily produce hundreds of thousands of metric time series, gigabytes of log data per day, and millions of distributed trace spans. The challenge for systems administrators is not collecting data—it is building systems that turn that data into understanding. This module covers the full observability lifecycle: from the three foundational signal types, through the tools that collect, store, and visualize them, to the practices that make on-call sustainable and reliability measurable.

1. The Three Pillars of Observability

Observability rests on three complementary signal types. Each answers different questions about system behavior, and no single pillar is sufficient on its own.

1.1 Logs

A log is an immutable, timestamped record of a discrete event. Logs are the oldest form of telemetry—every Unix daemon has been writing to syslog since the 1980s—and they remain the most detailed signal available.

Structured vs. Unstructured Logs. Unstructured logs are free-form text strings: Apr 08 14:23:01 web01 nginx: 192.168.1.50 - - [08/Apr/2026:14:23:01 +0000] "GET /api/users HTTP/1.1" 200 1543. They are human-readable but expensive to parse at scale. Structured logs emit events as key-value pairs, typically JSON:

{
  "timestamp": "2026-04-08T14:23:01Z",
  "level": "info",
  "service": "api-gateway",
  "method": "GET",
  "path": "/api/users",
  "status": 200,
  "duration_ms": 42,
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7"
}

Structured logging unlocks machine parsing, filtering, and correlation. The trace_id field above is a correlation ID—it ties this log line to a specific distributed trace, allowing you to jump from a log entry directly to the full request flow.

Log Levels. The standard severity hierarchy (from most to least urgent): FATAL, ERROR, WARN, INFO, DEBUG, TRACE. In production, most teams run at INFO level and enable DEBUG dynamically when investigating issues. A common mistake is logging too much at INFO—every log line costs storage, ingestion bandwidth, and query time.

Correlation IDs. In a microservice architecture, a single user request may traverse ten or more services. Without a correlation ID propagated through every hop, matching log lines across services is nearly impossible. The W3C Trace Context standard (traceparent header) provides a standardized way to propagate trace and span IDs, and these same IDs should appear in your log lines.

1.2 Metrics

A metric is a numeric measurement collected over time. Unlike logs (which record individual events), metrics aggregate behavior into time series, making them far more efficient for dashboards, alerting, and trend analysis.

Prometheus defines four core metric types:

TypeDescriptionExample
CounterMonotonically increasing value. Resets only on process restart.http_requests_total
GaugeValue that can go up and down.node_memory_available_bytes
HistogramSamples observations into configurable buckets. Tracks count, sum, and bucket boundaries.http_request_duration_seconds
SummarySimilar to histogram but calculates quantiles client-side. Cannot be aggregated across instances.rpc_duration_seconds

When to use histograms vs. summaries: Prefer histograms. They can be aggregated across instances using histogram_quantile(), and Prometheus 3.0's native histograms eliminate the need to pre-define bucket boundaries entirely. Summaries are useful only when you need exact quantiles from a single instance and cannot tolerate the approximation inherent in bucket-based histograms.

1.3 Traces

A distributed trace follows a single request as it propagates through multiple services. Each unit of work within the trace is called a span. Spans form a tree: the root span represents the initial request, and child spans represent downstream calls.

Key concepts:

  • Trace ID: A globally unique identifier for the entire request flow (128-bit, hex-encoded).
  • Span ID: Identifies a single unit of work within the trace (64-bit).
  • Parent Span ID: Links a span to its parent, forming the tree structure.
  • W3C Trace Context: The standard propagation format. The traceparent HTTP header carries version-trace_id-span_id-trace_flags. Example: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01.

Traces excel at answering "where did this request spend its time?" and "which downstream dependency caused this failure?" They are essential for debugging latency in microservice architectures but are the most expensive signal to collect and store at full fidelity. Most production systems use sampling (head-based or tail-based) to control costs.

1.4 Monitoring vs. Observability

Monitoring answers known questions: "Is the CPU above 90%?" "Is the error rate above 1%?" You define the questions in advance and set up dashboards and alerts.

Observability lets you ask new questions without deploying new instrumentation. If your system emits rich, structured telemetry with high cardinality (many unique label combinations), you can explore failures you did not anticipate. Observability is a property of the system, not a tool you buy.

In practice, you need both. Monitoring catches the known failure modes through alerting. Observability enables investigation of novel failures through ad-hoc querying across logs, metrics, and traces.

2. Metrics with Prometheus

Prometheus is the de facto standard for infrastructure and application metrics in cloud-native environments. It graduated from the CNCF in 2018 and released version 3.0 in November 2024—the first major release since 2.0 in 2017.

2.1 Architecture: The Pull Model

Prometheus uses a pull-based (scrape) model: the Prometheus server periodically sends HTTP GET requests to /metrics endpoints exposed by your applications and infrastructure. This is a deliberate design choice:

  • Failure detection: If a scrape fails, Prometheus knows the target is down. Push-based systems cannot distinguish "the target stopped sending" from "the target is healthy but has nothing to report."
  • Central control: Scrape intervals, timeouts, and target lists are configured in one place.
  • No client-side buffering: Targets do not need to queue and retry metric shipments.

A minimal scrape configuration:

# prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: "node-exporter"
    static_configs:
      - targets: ["node01:9100", "node02:9100", "node03:9100"]

  - job_name: "api-service"
    metrics_path: "/metrics"
    scheme: "https"
    tls_config:
      insecure_skip_verify: false
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        target_label: __address__
        regex: (.+)
        replacement: ${1}

Service discovery is critical at scale. Instead of listing every target manually, Prometheus can discover targets from Kubernetes, Consul, EC2, DNS, file-based discovery, and more. The relabel_configs block filters and transforms discovered targets before scraping.

2.2 PromQL Fundamentals

PromQL is a functional query language for selecting and aggregating time series data. Key patterns:

# Rate of HTTP requests per second over the last 5 minutes
rate(http_requests_total[5m])

# 99th percentile request latency from histogram
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

# Total error rate across all instances, grouped by service
sum by (service) (rate(http_requests_total{status=~"5.."}[5m]))
  /
sum by (service) (rate(http_requests_total[5m]))

# CPU saturation: average usage above 80% sustained over 10 minutes
avg_over_time(node_cpu_seconds_total{mode="idle"}[10m]) < 0.2

Critical rule: When calculating rates from counters, use a range selector at least 4x your scrape interval. With a 15-second scrape interval, use [1m] at minimum—[5m] is a safer default for dashboards.

Recording rules pre-compute expensive queries and store them as new time series, reducing dashboard load times:

# rules/recording.yml
groups:
  - name: api_latency
    interval: 30s
    rules:
      - record: job:http_request_duration_seconds:p99_5m
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

2.3 Alerting Rules and Alertmanager

Prometheus evaluates alerting rules on its configured evaluation_interval and sends firing alerts to Alertmanager, which handles deduplication, grouping, silencing, inhibition, and routing to receivers (email, Slack, PagerDuty, webhooks).

# rules/alerting.yml
groups:
  - name: api_alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
            /
          sum(rate(http_requests_total[5m])) by (service)
            > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High 5xx error rate on {{ $labels.service }}"
          description: "Error rate is {{ $value | humanizePercentage }} over the last 5 minutes."
          runbook_url: "https://wiki.example.com/runbooks/high-error-rate"

The for clause requires the condition to be true for 5 consecutive minutes before firing, preventing transient spikes from triggering alerts.

2.4 Long-Term Storage: Thanos, Cortex, and Mimir

Prometheus stores data locally on disk with a default retention of 15 days. For long-term storage, three major options exist:

SolutionArchitectureKey Feature
ThanosSidecar uploads TSDB blocks to object storage (S3, GCS). Query layer deduplicates and federates across clusters.No single point of failure; global query view
CortexHorizontally scalable, multi-tenant Prometheus-compatible storage using consistent hashing.Multi-tenancy out of the box
Grafana MimirFork of Cortex by Grafana Labs. Fully open-source (AGPLv3).Simplified operations, native support for Prometheus native histograms

2.5 Prometheus 3.0 Highlights

Released November 2024, Prometheus 3.0 introduced several major changes:

  • Native OTLP ingestion: Prometheus can receive OpenTelemetry metrics via the /api/v1/otlp/v1/metrics endpoint, eliminating the need for a separate collector for OTLP-to-Prometheus conversion.
  • UTF-8 metric and label names: Dots, slashes, and other characters are now valid. OpenTelemetry metrics (which use dots as namespace separators, e.g., http.server.duration) can be stored without mangling.
  • Native histograms: Exponentially-bucketed histograms with automatic bucket boundaries. Dramatically reduces cardinality compared to classic histograms while preserving accuracy.
  • Remote Write 2.0: String interning, metadata, exemplars, and native histogram support in the wire protocol. Reduces payload size by 30-50%.
  • New UI: Completely rewritten web interface.

3. Visualization with Grafana

3.1 Dashboard Design Principles

Good dashboards tell a story. Two methodologies dominate infrastructure dashboard design:

The USE Method (Brendan Gregg) for infrastructure resources (CPU, memory, disk, network):

  • Utilization: Percentage of the resource busy (e.g., CPU usage percentage).
  • Saturation: Amount of work queued because the resource is full (e.g., CPU run queue length, disk I/O wait).
  • Errors: Count of error events (e.g., disk read errors, network packet drops).

The RED Method (Tom Wilkie, Grafana Labs) for request-driven services:

  • Rate: Requests per second.
  • Errors: Failed requests per second.
  • Duration: Latency distribution (histograms/percentiles).

Use USE dashboards for your infrastructure layer (nodes, disks, network). Use RED dashboards for your application layer (APIs, microservices). Together, they cover the full stack.

3.2 Panel Types and Variables

Grafana provides time series, stat, gauge, bar chart, table, heatmap, logs, traces, and node graph panels. For latency visualization, the heatmap panel with histogram data is particularly powerful—it shows how the latency distribution shifts over time, not just the average.

Template variables make dashboards reusable. A $service variable populated from label values lets you switch between services without duplicating dashboards:

# Variable query (Prometheus data source)
label_values(http_requests_total, service)

3.3 Dashboards as Code

Hand-crafted dashboards in the Grafana UI are fragile. Production teams provision dashboards through code:

  • Grafana provisioning: YAML files in /etc/grafana/provisioning/dashboards/ pointing to JSON dashboard definitions stored in Git.
  • Grafonnet: A Jsonnet library for generating Grafana dashboard JSON programmatically.
  • Terraform: The grafana_dashboard resource in the Grafana Terraform provider manages dashboards declaratively.

3.4 Grafana Loki for Logs

Grafana Loki is a log aggregation system that indexes only labels (metadata), not the full text of log lines. This makes it dramatically cheaper to run than Elasticsearch for high-volume logging. Loki uses the same label model as Prometheus, so queries like {namespace="production", app="api-gateway"} |= "error" | json | status >= 500 feel natural to Prometheus users.

3.5 Grafana Tempo for Traces

Grafana Tempo is a distributed tracing backend that requires only object storage (S3, GCS, Azure Blob). It accepts traces via OTLP, Jaeger, and Zipkin protocols. Tempo automatically generates RED metrics from ingested traces and pushes them to Mimir/Prometheus, giving you service-level metrics without any application instrumentation.

3.6 Annotations for Deployments

Annotations overlay event markers on time series graphs, providing crucial context for correlation. When you see a latency spike, was there a deployment 5 minutes before? Annotations answer this instantly.

# Push a deployment annotation to Grafana via API
curl -X POST http://grafana:3000/api/annotations \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $GRAFANA_API_KEY" \
  -d '{
    "dashboardUID": "abc123",
    "time": 1712592000000,
    "tags": ["deploy", "api-service", "v2.4.1"],

  }'

3.7 Alert Rules in Grafana

Grafana Alerting (unified alerting, introduced in Grafana 9) provides a built-in alerting engine that can evaluate rules against any data source—not just Prometheus. Alert rules can query Loki (log-based alerts), Elasticsearch, CloudWatch, or any supported data source. This means you can set up a single alerting pipeline for metrics, logs, and even business data from SQL databases, all routed through the same notification policies and contact points.

# Grafana provisioned alert rule (YAML)
# /etc/grafana/provisioning/alerting/rules.yml
apiVersion: 1
groups:
  - orgId: 1
    name: api-alerts
    folder: Production
    interval: 1m
    rules:
      - uid: high-error-rate
        title: High Error Rate
        condition: C
        data:
          - refId: A
            relativeTimeRange:
              from: 300
              to: 0
            datasourceUid: prometheus
            model:
              expr: sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
          - refId: B
            datasourceUid: prometheus
            model:
              expr: sum(rate(http_requests_total[5m])) by (service)
          - refId: C
            datasourceUid: "-100"
            model:
              type: math
              expression: "$A / $B > 0.05"
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Error rate above 5% on {{ $labels.service }}"

3.8 Cross-Signal Correlation

The power of the Grafana stack emerges when you link signals together. A metric alert fires, you click through to the Loki logs filtered by the same time range and labels, find a log line with a trace ID, and click through to the full trace in Tempo. This "metrics to logs to traces" workflow is the core value proposition of unified observability platforms.

To enable this workflow, your data sources must share common labels. If Prometheus metrics have service="api-gateway", your Loki logs must have the same service label, and your traces must include service.name="api-gateway" as a resource attribute. OpenTelemetry's semantic conventions make this natural—all three signals use the same resource attributes. Without consistent labeling, cross-signal correlation breaks down, and you are back to searching across disconnected tools.

4. Log Management

4.1 The ELK Stack

The ELK stack (Elasticsearch, Logstash, Kibana) has been the dominant log management platform for over a decade.

  • Elasticsearch: Distributed search and analytics engine. Indexes the full text of every log line, enabling arbitrary full-text queries. Extremely powerful but resource-intensive—a production cluster typically needs dedicated master, data, and ingest nodes.
  • Logstash: Log processing pipeline. Accepts input from many sources (files, syslog, Beats, Kafka), applies filters (grok, mutate, date), and ships to Elasticsearch. Resource-heavy; many teams replace it with Beats (lightweight shippers) or Fluent Bit.
  • Kibana: Visualization layer with dashboards, Lens (drag-and-drop visualization), and KQL (Kibana Query Language) for searching logs.

Scaling challenges: Elasticsearch clusters require careful capacity planning. Each indexed field consumes memory for its inverted index. High-cardinality fields (like request IDs) can cause memory pressure. Shard sizing, index lifecycle management (ILM), and hot-warm-cold architectures are essential for cost control at scale. Elastic's licensing changes (from Apache 2.0 to SSPL in 2021, then partial relicensing under AGPL in 2024) have also driven some organizations to alternatives.

4.2 Fluentd and Fluent Bit

Fluentd is a CNCF graduated project for unified log collection. It uses a plugin architecture (700+ plugins) to collect from diverse sources and ship to diverse destinations. Written in Ruby with performance-critical paths in C.

Fluent Bit is the lightweight counterpart, written entirely in C, with a footprint under 1 MB. It is the standard log collector in Kubernetes environments (deployed as a DaemonSet) and ships logs to Elasticsearch, Loki, S3, or any OTLP-compatible backend.

4.3 Loki: Labels, Not Indexes

Loki's design philosophy is radical simplicity: index only the metadata labels, store the log content as compressed chunks in object storage. This means Loki uses a fraction of the CPU and RAM per ingested log compared to Elasticsearch, but full-text search requires scanning chunks—so filtering by labels first is essential for query performance.

# LogQL: Loki's query language
# Find error logs from the api-gateway in the last hour
{namespace="production", app="api-gateway"} |= "error"

# Parse JSON logs and filter by status code
{app="api-gateway"} | json | status >= 500

# Compute error rate from logs (metric query)
sum(rate({app="api-gateway"} |= "error" [5m])) by (service)

4.4 System-Level Logging

journald: systemd's binary log format. Queryable with journalctl. Structured by default (unit name, PID, priority, etc.). Forward to Loki or Elasticsearch via Fluent Bit or Promtail.

rsyslog / syslog-ng: Traditional syslog daemons. Still relevant for network devices, legacy systems, and compliance requirements. rsyslog handles millions of messages per second on modern hardware.

logrotate: Prevents disk exhaustion by rotating, compressing, and removing old log files. A typical configuration:

# /etc/logrotate.d/app
/var/log/app/*.log {
    daily
    missingok
    rotate 14
    compress
    delaycompress
    notifempty
    create 0640 app app
    sharedscripts
    postrotate
        systemctl reload app.service > /dev/null 2>&1 || true
    endscript
}

4.5 Log Shipping Patterns

How logs get from their source to your aggregation system matters for reliability and cost:

  • Direct shipping: Applications send logs directly to the aggregation backend (e.g., Loki, Elasticsearch). Simplest setup but tightly couples your application to the logging infrastructure. If the backend is down, logs may be lost or application performance may degrade from backpressure.
  • Agent-based shipping: A local agent (Fluent Bit, Promtail, Beats) reads log files or receives logs via a local socket and ships them to the backend. Decouples the application from the backend. The agent can buffer, retry, and apply transformations. This is the recommended pattern for most deployments.
  • Kafka-buffered shipping: For very high-volume environments, logs are shipped to Apache Kafka first. Kafka provides durability and backpressure handling. Consumers (Logstash, Fluent Bit, or OTel Collector) read from Kafka and write to the final backend. This pattern handles ingestion spikes gracefully and allows multiple consumers to process the same log stream for different purposes (security analysis, operational monitoring, compliance archival).

4.7 Structured Logging Best Practices

Emit JSON from your applications. Include: timestamp (ISO 8601), severity level, service name, trace ID, span ID, and the message. Avoid logging sensitive data (passwords, tokens, PII). Use log levels consistently—ERROR means something is broken and needs attention, WARN means something unexpected happened but the system recovered, INFO means normal operation milestones.

A practical example of structured logging configuration in a Go application using slog:

// Go structured logging with slog (standard library, Go 1.21+)
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelInfo,
}))

logger.Info("request completed",
    slog.String("method", r.Method),
    slog.String("path", r.URL.Path),
    slog.Int("status", statusCode),
    slog.Duration("duration", elapsed),
    slog.String("trace_id", span.SpanContext().TraceID().String()),
    slog.String("span_id", span.SpanContext().SpanID().String()),
)

The key principle: log lines should be machine-parseable without sacrificing human readability. JSON logging with consistent field names across all services enables powerful queries in your log aggregation system without writing complex regex parsers.

5. OpenTelemetry

OpenTelemetry (OTel) is a CNCF project that provides a single, vendor-neutral standard for generating, collecting, and exporting telemetry data (traces, metrics, logs). It is rapidly becoming the universal telemetry pipeline.

5.1 OTLP: The Protocol

The OpenTelemetry Protocol (OTLP) defines how telemetry data is encoded and transported. It supports gRPC and HTTP/protobuf transports. OTLP carries all three signal types (traces, metrics, logs) in a unified format, which means a single exporter endpoint can receive all your telemetry. OTLP Specification 1.10.0 is the latest stable release.

5.2 The Collector

The OpenTelemetry Collector is the central component of most OTel deployments. It receives telemetry (via OTLP, Jaeger, Zipkin, Prometheus, and other formats), processes it (batching, filtering, sampling, attribute manipulation), and exports it to one or more backends.

Two deployment modes:

  • Agent mode: Deployed as a sidecar or DaemonSet alongside applications. Receives telemetry locally, applies lightweight processing, and forwards to a gateway.
  • Gateway mode: Centralized collector that receives from multiple agents. Applies heavier processing (tail-based sampling, enrichment) and exports to storage backends.
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: "0.0.0.0:4317"
      http:
        endpoint: "0.0.0.0:4318"

processors:
  batch:
    send_batch_size: 8192
    timeout: 200ms
  memory_limiter:
    check_interval: 1s
    limit_mib: 2048
    spike_limit_mib: 512

exporters:
  otlp/tempo:
    endpoint: "tempo.monitoring:4317"
    tls:
      insecure: true
  prometheusremotewrite:
    endpoint: "http://mimir.monitoring:9009/api/v1/push"
  loki:
    endpoint: "http://loki.monitoring:3100/loki/api/v1/push"

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [otlp/tempo]
    metrics:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [prometheusremotewrite]
    logs:
      receivers: [otlp]
      processors: [memory_limiter, batch]
      exporters: [loki]

5.3 Instrumentation: SDK vs. Zero-Code

SDK instrumentation gives full control. You add the OTel SDK to your application and manually create spans, record metrics, and emit logs. This produces the richest telemetry but requires code changes.

Zero-code (auto) instrumentation injects telemetry without modifying application code. Java, Python, .NET, and Node.js all have auto-instrumentation agents. Grafana Beyla uses eBPF to instrument any HTTP/gRPC application at the kernel level—no agent injection, no code changes, no language dependency.

Recommendation: Start with auto-instrumentation for broad coverage, then add manual instrumentation for business-critical paths where you need custom spans and attributes.

5.4 Semantic Conventions

Semantic conventions are standardized attribute names and values that ensure telemetry from different libraries and services is consistent and queryable. For example, http.request.method, http.response.status_code, and url.path are standardized attribute names for HTTP telemetry. Database, messaging, RPC, and system conventions are actively being stabilized. In 2024, conventions for AI/LLM observability were added to cover generative AI workloads.

5.5 Resource Attributes

Resource attributes describe the entity producing telemetry: service.name, service.version, deployment.environment, host.name, k8s.pod.name. These are attached to all telemetry from that entity and are essential for filtering and grouping in your backend.

5.6 Why OTel Is Becoming the Standard

Before OpenTelemetry, every observability vendor had its own proprietary agent and protocol. Switching from Datadog to Grafana Cloud meant re-instrumenting every application. OTel breaks this lock-in by providing a vendor-neutral collection layer. You instrument once with OTel, and your Collector exports to any backend. If you switch vendors, you change exporter configuration—not application code.

The convergence is accelerating. Prometheus 3.0 added native OTLP ingestion. Grafana, Datadog, New Relic, Honeycomb, Splunk, and AWS all accept OTLP natively. The OTel Collector is expected to reach its stable v1 release in 2025. For new deployments, there is no reason to use vendor-specific agents when OTel covers the same ground with full portability.

5.7 Vendor-Neutral Pipeline Design

A well-designed telemetry pipeline separates collection from storage. The recommended architecture:

  1. Application layer: OTel SDKs or auto-instrumentation emit telemetry via OTLP to a local collector agent.
  2. Agent layer: OTel Collector in agent mode (DaemonSet or sidecar) receives telemetry, applies lightweight processing (resource detection, basic filtering), and forwards to a gateway.
  3. Gateway layer: OTel Collector in gateway mode receives from all agents, applies heavy processing (tail-based sampling, attribute enrichment, redaction of sensitive data), and exports to storage backends.
  4. Storage layer: Specialized backends for each signal type (Prometheus/Mimir for metrics, Loki for logs, Tempo for traces). Each backend is optimized for its signal type's query patterns.

This layered approach provides resilience (agents buffer if the gateway is temporarily unavailable), control (sampling and redaction happen in one place), and flexibility (adding a new export destination is a gateway configuration change, not an application change).

6. Alerting and On-Call

6.1 Alert Fatigue: The Silent Killer

Alert fatigue is the single biggest threat to operational effectiveness. When on-call engineers receive hundreds of alerts per day, they stop paying attention. Critical alerts get lost in the noise, and real incidents go unnoticed. Studies show that teams receiving more than 5-10 actionable alerts per on-call shift experience significant burnout and degraded incident response.

Causes of alert fatigue:

  • Alerting on causes instead of symptoms (e.g., "CPU is high" instead of "users are seeing errors").
  • Thresholds set too aggressively (tight thresholds on noisy metrics).
  • Duplicate alerts from multiple monitoring systems for the same issue.
  • Alerts that fire but require no action ("informational" alerts that should be dashboards).
  • Missing silence/inhibition rules that allow cascading failures to generate dozens of alerts.

Mitigations:

  • Apply the Google SRE principle: alert on symptoms, not causes. Users experience errors and latency, not CPU usage.
  • Every alert must be actionable. If the response is "look at it and do nothing," it should not be an alert.
  • Every alert must link to a runbook that explains what triggered it, why it matters, and what steps to take.
  • Schedule regular alert reviews (monthly or quarterly) to prune noisy alerts and adjust thresholds.
  • Use inhibition rules in Alertmanager to suppress downstream alerts when a root cause alert is firing.

Low (P4)Next business dayTicket createdDisk filling slowly, certificate expiring in 30 days

Not every alert needs to wake someone at 3 AM. Only Critical and High severity should page. Medium and Low are for business-hours review.

6.4 On-Call Tools

PagerDuty: Industry standard for incident management. Features ML-based alert grouping (Event Intelligence), escalation policies, on-call schedules, and integrations with every monitoring tool.

Grafana OnCall: Open-source on-call management integrated into the Grafana stack. Routes alerts from Grafana Alerting to on-call schedules with escalation chains, Slack/Telegram notifications, and phone calls via Twilio.

OpsGenie (Atlassian): Similar to PagerDuty with strong Jira/Confluence integration.

6.5 On-Call Schedules and Sustainability

On-call is a human system, not just a technical one. Sustainable on-call requires:

  • Rotation size: A minimum of 5-6 people in a rotation ensures no one is on-call more than one week in five. Smaller rotations lead to burnout.
  • Follow-the-sun: For global teams, hand off on-call responsibility to the team in the next time zone rather than having a single person cover overnight hours.
  • Compensation: On-call should be compensated—either through additional pay, time off in lieu, or other recognition. Uncompensated on-call breeds resentment and turnover.
  • Post-incident review: After every significant incident, conduct a blameless postmortem. Focus on systemic improvements (better monitoring, automated remediation, architecture changes), not individual fault.
  • Operational load tracking: Measure the number of pages per shift, time spent on incidents, and toil hours. If operational load exceeds 50% of an engineer's time, the system needs reliability investment, not more on-call staffing.

6.6 Runbooks

A runbook is a documented procedure for responding to a specific alert. Every alert annotation should include a runbook_url. A good runbook contains:

  • Alert description: What this alert means in plain language. What user impact is expected.
  • Likely causes: The 3-5 most common root causes, ordered by frequency.
  • Diagnostic steps: Specific commands to run, dashboards to check, and log queries to execute. Do not make the on-call engineer think at 3 AM—give them the exact steps.
  • Remediation: For each likely cause, the specific fix. Include rollback procedures for deployment-related issues.
  • Escalation criteria: When to wake up the next person. If the issue is not resolved within N minutes, escalate to the team lead or the team that owns the failing dependency.

"Runbook as Code" approaches store runbooks in Git alongside alerting rules, ensuring they stay in sync. When you add an alert, the pull request should include the runbook. When you change the alert threshold, update the runbook in the same commit.

7. SLOs, SLIs, and Error Budgets

7.1 The Concept

A Service Level Indicator (SLI) is a quantitative measure of a service attribute: availability, latency, throughput, or correctness. An SLI is expressed as a ratio: good events divided by total events. Example: "the proportion of HTTP requests that returned a 2xx or 3xx status code."

A Service Level Objective (SLO) is a target for an SLI over a time window. Example: "99.9% of requests succeed over a rolling 30-day window." SLOs are internal reliability contracts—they define how reliable a service needs to be, not how reliable it can be.

An error budget is the inverse of the SLO. A 99.9% SLO means a 0.1% error budget: out of 1,000,000 requests in 30 days, 1,000 can fail without violating the objective. The error budget is what makes SLOs actionable—it quantifies how much unreliability is acceptable.

7.2 Defining SLIs

SLI TypeDefinitionGood Event
AvailabilitySuccessful requests / total requestsHTTP 2xx, 3xx, 4xx (client errors are not your fault)
LatencyRequests faster than threshold / total requestsResponse time < 300ms
ThroughputRequests processed / requests receivedNo queue overflow or rejection
CorrectnessCorrect responses / total responsesResponse body matches expected schema

7.3 Error Budget Policy

The error budget policy defines what happens when the budget is exhausted. Google's SRE approach:

  • Budget remaining: Feature development proceeds. Teams accept calculated risk for velocity.
  • Budget at risk (below 25%): Feature launches require additional review. Focus shifts toward reliability work.
  • Budget exhausted: Feature freeze. All engineering effort redirects to reliability improvements until the budget recovers. No new deployments except reliability fixes.

This creates a self-correcting feedback loop: teams that ship unreliable code burn their error budget and lose deployment velocity. Teams that invest in reliability preserve their budget and maintain freedom to ship features.

7.4 SLO-Based Alerting: Burn Rate

Traditional threshold-based alerting (e.g., "error rate > 5%") is either too slow or too noisy. SLO-based alerting uses burn rate—how fast the error budget is being consumed—to trigger alerts at the right urgency.

Burn rate = 1 means you will exactly exhaust your budget at the end of the window. Burn rate = 14.4 means you will exhaust your 30-day budget in 2% of the window (about 14.4 hours).

The recommended approach is multi-window, multi-burn-rate alerting:

SeverityBurn RateLong WindowShort WindowBudget Consumed
Page (critical)14.4x1 hour5 minutes2% in 1 hour
Page (high)6x6 hours30 minutes5% in 6 hours
Ticket (medium)3x1 day2 hours10% in 1 day
Ticket (low)1x3 days6 hours10% in 3 days

Both windows must be active simultaneously to fire. The long window catches sustained problems; the short window prevents alerting on issues that have already resolved.

# Burn rate alerting rule (Prometheus)
# 99.9% SLO over 30 days = 0.001 error budget
# Fast burn: 14.4x over 1h AND 5m
- alert: HighBurnRate_Critical
  expr: |
    (
      sum(rate(http_requests_total{status=~"5.."}[1h])) by (service)
      /
      sum(rate(http_requests_total[1h])) by (service)
    ) > (14.4 * 0.001)
    and
    (
      sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
      /
      sum(rate(http_requests_total[5m])) by (service)
    ) > (14.4 * 0.001)
  labels:
    severity: critical

7.5 Setting Your First SLO

Practical steps:

  1. Measure current performance for 2-4 weeks to establish a baseline.
  2. Set the SLO slightly below your observed performance (if you are at 99.95%, set the SLO at 99.9%).
  3. Define the error budget policy with your team and stakeholders.
  4. Implement burn rate alerts.
  5. Review monthly: are you burning budget too fast? Tighten reliability. Never burning budget? Your SLO may be too loose—consider tightening it or investing the headroom in shipping faster.

8. Capacity Planning

8.1 Resource Utilization Monitoring

Capacity planning begins with understanding current resource consumption. The four golden resources:

  • CPU: Monitor node_cpu_seconds_total (idle, user, system, iowait modes). High iowait indicates disk bottlenecks, not CPU bottlenecks.
  • Memory: Monitor node_memory_MemAvailable_bytes (not MemFree—Linux uses free memory for page cache, which is reclaimable). OOM kills (node_vmstat_oom_kill) are the critical signal.
  • Disk: Monitor usage (node_filesystem_avail_bytes), I/O throughput (node_disk_read_bytes_total), I/O latency (node_disk_io_time_seconds_total), and IOPS (node_disk_reads_completed_total).
  • Network: Monitor bandwidth (node_network_transmit_bytes_total), errors (node_network_transmit_errs_total), and drops (node_network_transmit_drop_total).

8.2 Saturation Signals

Utilization alone is insufficient. A CPU at 80% might be fine—or it might be at the knee of a latency curve where small increases cause exponential latency growth. Saturation signals tell you when a resource is overloaded:

  • CPU saturation: node_load1 (1-minute load average) compared to CPU count. Load average above CPU count means processes are queuing.
  • Memory saturation: Active swap usage (node_memory_SwapFree_bytes) and page fault rate. Any swap activity on a server is a red flag.
  • Disk saturation: I/O queue depth and I/O wait percentage.
  • Network saturation: Dropped packets, TCP retransmissions (node_netstat_Tcp_RetransSegs).

8.3 Forecasting

Prometheus's predict_linear() function extrapolates a time series into the future using linear regression:

# Predict disk full in 24 hours
predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[7d], 24*3600) < 0

For more sophisticated forecasting (seasonal decomposition, growth modeling), export metrics to time series databases and use dedicated tools. But for capacity alerting, predict_linear() handles the 80% case: "will this disk fill up tomorrow?"

8.4 Load Testing

Capacity planning is incomplete without load testing. You need to know your system's breaking point before production traffic finds it.

  • k6 (Grafana): Modern load testing tool with JavaScript scripting. Designed for developer experience and CI/CD integration. Supports ramping VUs (virtual users), constant arrival rate, and externally controlled test patterns.
  • Locust: Python-based load testing with a web UI for real-time monitoring of test runs.
  • wrk: C-based HTTP benchmarking tool for maximum raw throughput testing.
// k6 test script: ramp from 0 to 200 users over 5 minutes
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 50 },
    { duration: '3m', target: 200 },
    { duration: '2m', target: 200 },  // sustained load
    { duration: '1m', target: 0 },    // ramp down
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],  // 99th percentile under 500ms
    http_req_failed: ['rate<0.01'],     // error rate under 1%
  },
};

export default function () {
  const res = http.get('https://api.example.com/users');
  check(res, { 'status is 200': (r) => r.status === 200 });
  sleep(1);
}

8.5 Auto-Scaling Triggers

Auto-scaling translates monitoring data into infrastructure actions. The key is choosing the right metric to scale on:

  • CPU-based scaling: The simplest and most common. HPA targets a CPU utilization percentage (e.g., scale up when average CPU across pods exceeds 70%). Works well for CPU-bound workloads but misleading for I/O-bound services that may be saturated at 30% CPU.
  • Memory-based scaling: Useful for JVM and cache-heavy workloads. Less responsive than CPU because memory usage changes slowly.
  • Custom metrics scaling: Scale on application-specific metrics like queue depth, active connections, or request latency. Requires KEDA (Kubernetes Event-Driven Autoscaling) or the Prometheus adapter. Most accurate but requires more setup.
  • Request-rate scaling: Scale based on requests per second per pod. Ensures each pod handles a bounded number of concurrent requests.
# Kubernetes HPA with custom Prometheus metric
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"   # Scale up when pods average over 100 req/s
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300   # Slow scale-down prevents flapping

8.6 Rightsizing and Cost

Over-provisioning wastes money. Under-provisioning causes outages. The goal is to maintain enough headroom (typically 30-40% for CPU, one node's worth of capacity in a cluster) to absorb traffic spikes and single-node failures, without paying for resources that sit idle. Auto-scaling helps but introduces its own complexity—scaling too slowly causes degradation during spikes, scaling too aggressively wastes money. The stabilization window in the HPA configuration above is one mechanism for balancing responsiveness against cost: a 300-second scale-down window prevents removing pods during brief traffic lulls, avoiding the "flapping" pattern where pods are constantly created and destroyed.

9. Synthetic Monitoring and Health Checks

9.1 Synthetic Monitoring

Real user monitoring (RUM) tells you how actual users experience your service. Synthetic monitoring probes your service continuously from external vantage points, catching outages even when no real users are active (late-night maintenance windows, low-traffic services).

Types of synthetic checks:

  • HTTP checks: Send an HTTP request and verify status code, response body, and response time.
  • TCP checks: Verify that a port is open and accepting connections.
  • DNS checks: Verify that DNS resolution returns expected records.
  • Script/browser checks: Execute multi-step synthetic transactions (login, search, checkout) using headless browsers.

Prometheus Blackbox Exporter probes endpoints and exposes the results as Prometheus metrics:

# blackbox.yml
modules:
  http_2xx:
    prober: http
    timeout: 5s
    http:
      valid_http_versions: ["HTTP/1.1", "HTTP/2.0"]
      valid_status_codes: [200]
      method: GET
      fail_if_ssl: false
      fail_if_not_ssl: true
      tls_config:
        insecure_skip_verify: false
  tcp_connect:
    prober: tcp
    timeout: 5s
  icmp:
    prober: icmp
    timeout: 5s
# prometheus.yml scrape config for blackbox
- job_name: "blackbox-http"
  metrics_path: /probe
  params:
    module: [http_2xx]
  static_configs:
    - targets:
        - "https://api.example.com/healthz"
        - "https://www.example.com"
  relabel_configs:
    - source_labels: [__address__]
      target_label: __param_target
    - source_labels: [__param_target]
      target_label: instance
    - target_label: __address__
      replacement: "blackbox-exporter:9115"

9.2 Health Check Endpoints

Kubernetes defines three probe types that map to health check endpoints:

EndpointProbe TypePurpose
/healthzLivenessIs the process alive? Failure triggers a pod restart.
/readyzReadinessIs the process ready to serve traffic? Failure removes the pod from service endpoints.
/livezStartupHas the process finished starting? Prevents liveness checks from killing slow-starting containers.

Design principles:

  • Liveness checks should be simple (process is alive, not deadlocked). Do not check downstream dependencies in liveness—if the database is down, restarting your app will not fix it and will cause cascading failures.
  • Readiness checks should verify downstream dependencies. If the database connection pool is exhausted, remove the pod from load balancing until it recovers.
  • Health checks should be fast (<200ms). They are called frequently (every 10-30 seconds).

9.3 Canary Deployments and Progressive Delivery

Canary deployments route a small percentage of traffic (1-5%) to the new version and monitor error rates, latency, and business metrics before rolling out to 100%. Tools like Argo Rollouts and Flagger automate canary analysis using Prometheus metrics, automatically promoting or rolling back based on SLI measurements. This is synthetic monitoring applied to deployment safety.
