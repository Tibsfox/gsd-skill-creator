Monitoring tells you when something is broken. Observability tells you why. The distinction matters: monitoring is necessary but insufficient for complex distributed systems where failure modes cannot be predicted in advance.

The Three Pillars of Observability

PillarWhat It CapturesToolsQuery Pattern
LogsDiscrete events with timestamps and contextElasticsearch, Loki, Fluentd, journald“What happened at 3:42 AM?”
MetricsNumeric measurements over time (counters, gauges, histograms)Prometheus, InfluxDB, StatsD, Graphite“What is the request rate trend?”
TracesRequest path across distributed services with timingJaeger, Zipkin, Tempo, OpenTelemetry“Why is this request slow?”

The industry is converging on OpenTelemetry (OTel) as the unified standard for collecting all three pillars through a single SDK and collector pipeline. OTel is vendor-neutral and backed by the CNCF.

Prometheus & Grafana Stack

Prometheus is a pull-based time-series database that scrapes metrics endpoints at configured intervals. Its query language (PromQL) is the de facto standard for metric querying. Key components:

  • Exporters: node_exporter (system metrics), blackbox_exporter (probing), mysqld_exporter, etc.
  • Alertmanager: Routes, groups, silences, and deduplicates alerts
  • Grafana: Visualization layer with dashboards, supports multiple data sources
  • Thanos / Cortex / Mimir: Long-term storage and multi-cluster federation

histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))

ELK Stack

Elasticsearch (search and analytics engine) + Logstash (log pipeline and transformation) + Kibana (visualization). The ELK stack handles centralized log aggregation and analysis. Filebeat replaces Logstash for lightweight log shipping. Elastic SIEM adds security analytics on top of the same infrastructure.

Alternative: Grafana Loki takes a log-label approach (like Prometheus for logs), avoiding full-text indexing for dramatically lower resource consumption.

OpenTelemetry

OTel provides a unified observability framework: auto-instrumentation SDKs for major languages, the OTLP (OpenTelemetry Protocol) for data transmission, and the OTel Collector for receiving, processing, and exporting telemetry to any backend. This eliminates vendor lock-in and reduces instrumentation burden.

The collector supports pipelines: receivers (OTLP, Prometheus, Jaeger) → processors (batching, filtering, tail sampling) → exporters (to any backend). This architecture allows switching observability backends without re-instrumenting applications.

Alerting & On-Call

Effective alerting follows key principles:

  • Alert on symptoms, not causes — “error rate > 5%” not “CPU > 80%”
  • Every alert must be actionable — if it fires and nobody needs to act, delete it
  • Use severity levels — P1 (page immediately), P2 (respond within hours), P3 (next business day)
  • Runbooks for every alert — link in the alert itself, reduce mean time to resolve
  • Alert fatigue kills — noisy alerts train people to ignore them; prune aggressively

On-call tools: PagerDuty, OpsGenie (Atlassian), Grafana OnCall (open source). Rotation patterns: follow-the-sun (global teams), weekly rotation, primary/secondary escalation.

SLOs, SLIs & Error Budgets

The Google SRE framework formalized service-level management:

ConceptDefinitionExample
SLI (Indicator)The metric that measures service qualityLatency p99, availability %, error rate
SLO (Objective)The target value for the SLI99.9% availability over 30 days
SLA (Agreement)The contractual commitment with consequences99.9% availability or credit refund
Error Budget100% minus SLO = acceptable failure allowance99.9% SLO = 43.2 min/month error budget
Error Budget = (1 - SLO) × time period
99.9% SLO = 0.1% budget = 43.2 minutes/month = 8.76 hours/year

When error budget is exhausted, teams freeze feature releases and focus on reliability. This creates an objective framework for balancing velocity against stability.

Capacity Planning

Capacity planning prevents outages before they happen. Key practices: trending analysis (extrapolate current growth curves), load testing (simulate peak traffic), headroom maintenance (keep utilization below 70% for burst absorption), and cost modeling (right-size instances before and after scaling events).

Health Checks & Synthetic Monitoring

Health checks verify that services are functioning correctly. Types: liveness (is the process alive?), readiness (can it accept traffic?), startup (has it finished initializing?). Kubernetes uses all three for pod lifecycle management.

Synthetic monitoring runs scripted user journeys against production at regular intervals, detecting issues before real users encounter them. Tools: Grafana Synthetic Monitoring, Checkly, Uptime Robot.
