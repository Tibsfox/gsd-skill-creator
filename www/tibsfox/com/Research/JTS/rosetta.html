“Java is to JavaScript as ham is to hamster.” — the oldest running joke in programming, and one of the most expensive naming decisions in the history of computing.

JavaScript is NOT Java

In December 1995, Netscape licensed the “Java” trademark from Sun to ride the Java hype wave. The result: thirty years of confusion. JavaScript is a dynamically typed, prototype-based, single-threaded language with first-class functions, closures, and an event-driven execution model. It shares almost nothing with Java except C-like syntax, curly braces, and the string “Java” in the name.

Same Program, Different Languages

JavaScript (modern)

const shapes = [
  { type: "circle", r: 2.0 },
  { type: "square", side: 3.0 },
];

function area(shape) {
  switch (shape.type) {
    case "circle": return Math.PI * shape.r ** 2;
    case "square": return shape.side ** 2;
  }
}

for (const s of shapes) console.log(area(s));

TypeScript (discriminated union)

type Circle = { type: "circle"; r: number };
type Square = { type: "square"; side: number };
type Shape = Circle | Square;

function area(s: Shape): number {
  switch (s.type) {
    case "circle": return Math.PI * s.r ** 2;
    case "square": return s.side ** 2;
  }
}

const shapes: Shape[] = [
  { type: "circle", r: 2.0 },
  { type: "square", side: 3.0 },
];
shapes.forEach(s => console.log(area(s)));

Python

import math
from dataclasses import dataclass

@dataclass
class Circle:
    r: float
    def area(self): return math.pi * self.r ** 2

@dataclass
class Square:
    side: float
    def area(self): return self.side ** 2

shapes = [Circle(2.0), Square(3.0)]
for s in shapes:
    print(s.area())

Java (sealed + records, Java 21)

sealed interface Shape permits Circle, Square {}
record Circle(double r) implements Shape {}
record Square(double side) implements Shape {}

double area(Shape s) {
    return switch (s) {
        case Circle c -> Math.PI * c.r() * c.r();
        case Square sq -> sq.side() * sq.side();
    };
}

TypeScript vs Java Generics — Erasure Both Ways

A curious parallel: both Java and TypeScript erase generic type information at runtime. Java generics (2004) use type erasure to maintain bytecode backwards compatibility. TypeScript (2012) uses type erasure because it must — it compiles to JavaScript, which has no type system at all.

The result: in both languages, you cannot ask “is this a List<String>?” at runtime. In Java, you can at least check instanceof List. In TypeScript, at runtime you have only typeof (primitives) and instanceof (constructor chain). For anything more, you need runtime validation (Zod, io-ts, Valibot in TS; Jackson/Gson in Java).

TypeScript vs C# — The Hejlsberg Connection

Anders Hejlsberg designed both C# and TypeScript. C# (2000) is nominally typed with reified generics — type info survives at runtime. TypeScript (2012) is structurally typed with erased generics. Why the difference? C# controls its own runtime (.NET CLR); TypeScript must produce plain JavaScript. Structural typing was the pragmatic choice: it lets you pass a {name: string} wherever a Named interface is expected, without declaring implements Named. This duck-typing-at-compile-time is what makes TS feel “light” compared to Java or C# while still catching type errors.

On the Escalation Ladder

From the PRL and CPP research projects:

  • Bash — shell glue
  • JavaScript / TypeScript / Python / Perl — dynamic/high-level scripting. JS owns the browser; Python owns data science; Perl owns text processing.
  • Java / C# — managed enterprise runtimes
  • C++ / Rust — systems programming
  • C — the substrate

JavaScript is the only language on this ladder that runs natively in every web browser on earth. That is its superpower and the reason Atwood’s Law holds: if you need to ship to a browser, you will eventually write JavaScript (or something that compiles to it).

When to Reach for JavaScript / TypeScript

Yes, use JS/TS

Web frontends (React, Vue, Svelte). Full-stack web apps (Next.js, SvelteKit). API servers (Express, Fastify, NestJS). Mobile (React Native). Desktop (Electron, Tauri). Edge functions (Workers, Deno Deploy). Rapid prototyping.

Consider JS/TS

CLI tools (Commander, Yargs, Oclif). Build tools (if you’re in the JS ecosystem). Serverless functions. Real-time services (Socket.IO, WebSocket).

Probably use something else

Data science/ML (Python). Systems programming (Rust, C++). Embedded (C). Enterprise with existing Java/C# stacks. HFT (C++, Java with Azul). Batch processing (Java/Spark). Numerical computing (Python, Julia, C++).

Do NOT use JS

Kernel development. Hard real-time systems. Bare-metal firmware. Anything where a GC pause is unacceptable. Anything where the npm dependency tree is a security risk you can’t tolerate.
