Engines & Runtimes

V8, SpiderMonkey, JavaScriptCore, Node.js, Deno, Bun, Workers — The Event Loop and What Runs Your Code

JavaScript has no standard runtime. Instead, it has a competitive ecosystem of engines and runtimes, each with its own JIT pipeline, GC strategy, and design philosophy. The engine determines your performance ceiling; the runtime determines your I/O model.

V8 Internals

V8 was built by a team led by Lars Bak at Google — the same engineer who previously built the HotSpot JVM and the Animorphic Strongtalk VM. The lineage from Self → Strongtalk → HotSpot → V8 is one of the most important in VM engineering.

Tiered compilation pipeline:

  1. Ignition (interpreter) — bytecode execution, profiling counters
  2. Sparkplug (baseline compiler, 2021) — fast non-optimizing compile from bytecode to machine code
  3. Maglev (mid-tier optimizer, 2023) — SSA-based optimizer, faster than TurboFan but less aggressive
  4. TurboFan (top-tier optimizer) — sea-of-nodes IR, speculative optimization, inlining, escape analysis
  • Hidden classes (shapes) — V8 assigns each object a shape describing its layout; same-shaped objects share optimized property access code
  • Inline caches (ICs) — cache property lookup results for monomorphic/polymorphic call sites
  • Orinoco GC — parallel, concurrent, incremental. Young gen: Scavenger (semi-space copying). Old gen: concurrent Mark-Compact.
  • Pointer compression (2019) — 32-bit pointers on 64-bit, ~40% heap reduction

SpiderMonkey

Mozilla’s engine, descended from Eich’s 1995 original. Current pipeline:

  • Interpreter (bytecode)
  • Baseline Compiler — fast non-optimizing JIT
  • Warp (2020) — the current optimizing JIT, replaced IonMonkey

GC: generational, incremental marking, concurrent sweeping. Strong ECMAScript compliance focus — SpiderMonkey often implements spec proposals first.

JavaScriptCore (JSC / Nitro)

Apple/WebKit’s engine. Four-tier compilation:

  1. LLInt (Low-Level Interpreter)
  2. Baseline JIT
  3. DFG (Data Flow Graph) optimizer
  4. FTL (Faster Than Light) with B3 backend

Used by Bun, which chose JSC over V8 because of its strong numeric optimization and startup speed.

Node.js

Ryan Dahl presented Node.js at JSConf EU in November 2009. His pitch: non-blocking I/O with JavaScript’s callback model. The stack: V8 + libuv + OpenSSL + core modules.

  • libuv — cross-platform async I/O (Ben Noordhuis). epoll on Linux, kqueue on macOS, IOCP on Windows. 4 worker threads for blocking ops (fs, crypto, dns.lookup).
  • npm (Isaac Schlueter, 2010) — the package manager that became the world’s largest package registry.
  • io.js fork (2014) over governance; reconciled into the Node.js Foundation (2015).
  • ESM support: experimental in Node 12, stable in 14, "type": "module" default in 16+.
  • LTS cadence: even-numbered versions (18, 20, 22) get long-term support.

The Event Loop

       +-------------------------------------------+
       |              The Event Loop                 |
       |                                             |
       |  1. Timers (setTimeout, setInterval)        |
       |  2. Pending callbacks (deferred I/O)        |
       |  3. Idle, prepare (internal)                |
       |  4. Poll (I/O polling - epoll/kqueue/IOCP)  |
       |  5. Check (setImmediate)                    |
       |  6. Close callbacks                         |
       |                                             |
       |  --- between each phase ---                 |
       |  Microtask queue (promises, queueMicrotask) |
       |  process.nextTick queue (Node.js)           |
       +-------------------------------------------+

JavaScript is single-threaded. The event loop is what gives it concurrency: I/O operations are delegated to the kernel (or the libuv thread pool), and callbacks are queued for execution when the operation completes. CPU-bound work blocks the loop — use worker_threads for heavy computation.

Deno

Ryan Dahl’s “10 Things I Regret About Node.js” (November 2018, JSConf EU) launched Deno: a rewrite in Rust on top of V8.

  • TypeScript-first — built-in TypeScript compiler, no tsconfig.json required by default
  • Secure by default — --allow-net, --allow-read, --allow-write permission flags
  • URL-based imports — no node_modules; import { serve } from "https://deno.land/std/http/server.ts"
  • Built-in tooling: fmt, lint, test, bench, repl, doc, bundle, compile
  • Deno KV (2023) — built-in key-value store
  • Deno Deploy — edge runtime platform (V8 Isolates)
  • Node.js compat: npm: and node: specifiers for interop

Bun

Jarred Sumner (2022). Written in Zig, uses JavaScriptCore (not V8). Extreme speed — fastest npm install, fastest test runner, fastest transpiler. Drop-in Node.js compatibility.

  • Built-in: bundler, transpiler, test runner, package manager
  • Bun.serve(), Bun.file(), Bun.write(), Bun.sql() — fast I/O primitives
  • Corporate: Oven (startup)
  • Goal: replace Node.js + npm + Babel/SWC + webpack/Vite + Jest in a single binary

Worker Threads (Node.js)

worker_threads (Node 10.5+) provides real OS threads for CPU-bound work:

import { Worker, isMainThread, parentPort } from "worker_threads";

if (isMainThread) {
  const worker = new Worker(new URL(import.meta.url));
  worker.on("message", msg => console.log("result:", msg));
  worker.postMessage({ data: [1, 2, 3] });
} else {
  parentPort.on("message", msg => {
    const sum = msg.data.reduce((a, b) => a + b, 0);
    parentPort.postMessage(sum);
  });
}

SharedArrayBuffer enables zero-copy data sharing between threads. Not the same as cluster (which forks processes).

Memory and GC

  • V8 heap: new space + old space + large object space + code space
  • Default Node.js heap: auto-scales to available memory (~1.7 GB pre-Node 12)
  • --max-old-space-size=4096 to set max heap (MB)
  • --expose-gc to allow explicit global.gc() (debugging only)
  • Common leak patterns: closures over large state, event emitter listeners never removed, timers that hold references, growing Maps/Sets used as caches
