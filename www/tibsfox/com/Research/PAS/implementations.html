The CDC 6000 Original (1970)

The story of Pascal as a running program begins at ETH Zürich, where Niklaus Wirth held a professorship in informatics starting in 1968. The target machine was the Control Data Corporation CDC 6600, one of the most powerful scientific computing platforms of its era, designed by Seymour Cray: 60-bit word length, 10-peripheral-processor architecture, roughly one million floating-point operations per second.

The first Pascal compiler was developed during 1969–1971 by Urs Ammann, E. Marmier, and R. Schild under Wirth’s direction. The bootstrap process was itself a demonstration of structured programming:

  1. The compiler was written in Pascal itself — on paper, as a design document.
  2. R. Schild translated this Pascal source by hand into a low-level language available on the CDC 6000.
  3. This hand-translated compiler compiled the Pascal source of the compiler.
  4. The resulting compiler could compile its own source code — Pascal was self-hosting.

The CDC 6000 Pascal compiler was a single-pass, top-down, recursive-descent parser generating native machine code. Single-pass compilation was not just an implementation convenience; it was a design constraint Wirth imposed on the language itself. The advantages on 1970 hardware were substantial: no intermediate representation written to disk, minimal memory required, and compilation speed competitive with FORTRAN. By 1972, Pascal was being used in introductory programming courses at ETH.

P-Code and the P-Machine (1973–1975)

By 1973, Pascal had proven itself at ETH but faced a portability challenge: the CDC 6000 compiler generated native code for a specific, expensive mainframe. If Pascal was to become a universal teaching language, it needed to run on the rapidly proliferating variety of computing platforms.

The solution emerged from Kesav V. Nori, Urs Ammann, Kathleen Jensen, and Hans-Heinrich Nägeli at ETH: the Pascal-P compiler, which introduced compilation to a portable bytecode for an abstract virtual machine — one of the most influential ideas in the history of programming language implementation.

Instead of generating native code, Pascal-P generated “P-code” — instructions for an imaginary stack-based machine called the P-machine. To run Pascal on any real hardware, you needed only a P-machine interpreter for your platform. Writing an interpreter took a few hundred lines of assembly language, compared to thousands for a full compiler.

Predecessor to the JVM

The P-code strategy was one of the earliest practical demonstrations of the “virtual machine” approach. Its influence is direct and traceable:

  • Java (1995): James Gosling and the Java team at Sun explicitly cited UCSD Pascal’s p-System as an influence on the JVM. “Write once, run anywhere” was a marketing restatement of what Pascal-P had demonstrated two decades earlier.
  • .NET CLR (2002): Multiple source languages compiling to Common Intermediate Language. Anders Hejlsberg, architect of C# and the CLR’s primary client, had spent his formative years building Pascal compilers.
  • Python, Ruby, Lua: Modern interpreted languages use bytecode compilation internally, a concept tracing back through UCSD Pascal to the original P-code work at ETH.

UCSD Pascal and the p-System (1978)

Kenneth Bowles, a professor at the University of California, San Diego, recognized around 1974 that the proliferation of incompatible microprocessor architectures made it impractical to write software for each platform separately. His solution went far beyond what ETH had envisioned: where Pascal-P was a compiler targeting a virtual machine, UCSD Pascal was an entire operating system built around that virtual machine.

The p-System provided a screen editor, Pascal compiler, file manager, linker, assembler, and debugger — all running on the P-machine, all booting from floppy disk on the modest hardware of the era. The system was self-hosting: the compiler was written in Pascal and compiled to P-code.

Key Innovations

Separately compilable units: UCSD Pascal introduced the concept of “units” — independently compiled modules. This directly influenced Ada’s packages and was adopted by Turbo Pascal from version 4.0 onward.

String type: A counted string (first byte = length, followed by character data), solving Standard Pascal’s fixed-length string problem. Adopted unchanged by Turbo Pascal.

IBM offered the UCSD p-System as one of three OS options for the original IBM PC in 1981. The p-System ultimately lost to DOS, but its architectural innovations — virtual machine abstraction, integrated development environment, separately compilable modules — became standard features of the computing platforms that followed.

Wirth himself credited the p-System, and UCSD Pascal in particular, with popularizing Pascal beyond the academic community. Kenneth Bowles’s vision of a portable computing environment was vindicated by Java, .NET, and every modern platform that uses bytecode compilation.

Pascal/MT+, created by Michael Lehman at MT MicroSYSTEMS of Solana Beach, California, was notable for compiling to native machine code rather than interpreting P-code, giving it a performance advantage over UCSD Pascal. MT MicroSYSTEMS was acquired by Digital Research in 1981.

Microsoft’s inability to compete in the Pascal market is ironic: Anders Hejlsberg, the architect of Turbo Pascal, would later join Microsoft and become the lead architect of C#. Microsoft lost the Pascal compiler war to Borland but eventually acquired the human capital that made Borland’s products great.

This entire competitive landscape was swept away by Turbo Pascal’s arrival in late 1983, which rendered most CP/M Pascal compilers commercially irrelevant within a year.

Turbo Pascal (1983–1992)

Anders Hejlsberg

Anders Hejlsberg was born on December 2, 1960, in Copenhagen, Denmark. While studying electrical engineering at the Technical University of Denmark, he began writing a Pascal compiler for the Nascom microcomputer, a British Z80-based single-board machine. The compiler compiled to native Z80 code, was remarkably fast and compact, and Hejlsberg had studied Wirth’s Algorithms + Data Structures = Programs to implement it. He marketed it as “Compas Pascal” and later “PolyPascal” through his company PolyData.

Philippe Kahn, founder of Borland International in Scotts Valley, California, discovered PolyPascal and licensed the compiler core. Borland added a screen editor and an integrated development environment. The result was Turbo Pascal.

The Revolution

Turbo Pascal was released on November 20, 1983, for CP/M and MS-DOS at $49.95 — roughly one-tenth the cost of competing Pascal compilers. The compiler itself was approximately 33KB, shipped on a single floppy disk. On an IBM PC with a 4.77 MHz 8088 processor, it could compile roughly 10,000 lines per second. Competing compilers took minutes for the same task.

What made it special:

  • Speed: Single-pass compilation with all data in RAM. No disk access during compilation. Assembly-language implementation.
  • Integration: One of the first true IDEs for personal computers — editor, compiler, and runtime in a single program. Error messages displayed in the editor with the cursor at the error location.
  • Price: $49.95 made it accessible to students and hobbyists, not just universities and corporations.
  • Extensions: Variable-length strings, break/continue/exit, short-circuit boolean evaluation, inline assembly, direct memory access — addressing virtually all of Kernighan’s criticisms of Standard Pascal.

Turbo Pascal sold over 15,000 copies in its first four months. The “Turbo” name was inspired by the Porsche Turbo — an association with speed and engineering excellence.

What Made Turbo Pascal Special

Speed in depth. The compilation speed was achieved through several deliberate design choices: single-pass compilation (parsing and code generation fused into one pass), RAM-based operation (no intermediate file written to disk), assembly-language implementation of the compiler itself, and a minimal custom object format that required far less processing than standard OMF. From version 4.0, compiled units (.TPU files) stored all exported information in pre-parsed binary format — no character-by-character parsing of header files, in stark contrast to C’s #include mechanism.

Language extensions that addressed every Kernighan critique. Turbo Pascal added variable-length strings (up to 255 characters, length-prefixed), units for separate compilation with encapsulation, break/continue/exit for structured loop control, guaranteed short-circuit boolean evaluation, else clauses in case statements, inline assembly for low-level access, direct memory and port access (Mem, Port arrays), DOS/BIOS interrupt calls, typed constants (initialized variables), and procedural types for callbacks.

Turbo Vision. Version 6.0 introduced Turbo Vision, a text-mode application framework providing windows, menus, dialog boxes, scrollbars, and event-driven programming — a miniature windowing system rendered entirely from text characters. Turbo Vision programs looked and felt like graphical applications despite running in a DOS text console:

program MyApp;
uses App, Objects, Menus, Drivers, Views;

type
  TMyApp = object(TApplication)
    procedure InitMenuBar; virtual;
    procedure InitStatusLine; virtual;
  end;

var
  MyApp: TMyApp;
begin
  MyApp.Init;
  MyApp.Run;
  MyApp.Done;
end.

OOP in 5.5. Object-oriented programming was introduced with a simpler model than C++’s — single inheritance, virtual methods, constructors, and destructors using the object keyword. The simplicity was deliberate: learnable by Pascal programmers who had never encountered OOP concepts.

The Blue IDE

Turbo Pascal’s text-mode IDE — white or yellow text on a blue background with pull-down menus — became iconic. For an entire generation, “programming” meant this color scheme. The integration of editor, compiler, and debugger into a single keystroke (F9 to compile, Ctrl-F9 to compile and run) defined what development environments would look like for the next decade. Before Turbo Pascal 4.0, most programmers used separate tools with separate command-line interfaces. After it, the expectation was that everything would be in one application. This integration seems obvious in retrospect — every modern IDE provides it — but in 1983, it was revolutionary. Turbo Pascal did not merely provide an IDE; it invented the modern concept of what an IDE should be.

Community and Cultural Impact

Turbo Pascal generated an enormous user community, particularly in the pre-Internet era of bulletin board systems. The SWAG (SourceWare Archive Group) collection grew to thousands of Pascal code snippets. Many BBS systems themselves were written in Turbo Pascal, as was a significant portion of the shareware industry — its ability to produce small, self-contained .EXE files with no runtime dependencies made it ideal for software distributed on floppy disks.

The demoscene — the subculture of programmers creating audiovisual demonstrations pushing hardware to its limits — included a significant Turbo Pascal contingent, particularly in Europe. Eastern European and Latin American educational systems adopted Turbo Pascal and maintained it longer than Western countries, producing programmers with strong algorithm design and systems programming skills.

Freeware Releases

Borland released Turbo Pascal versions 1.0 (2000), 3.02 (2002), and 5.5 (2002) as freeware, acknowledging their historical significance. These can still be downloaded and run in DOS emulators, allowing new generations to experience the tools that shaped their predecessors’ education.

Object Pascal and Delphi (1985–Present)

Apple Object Pascal (1985)

The first Object Pascal was created at Apple Computer by Larry Tesler, in consultation with Niklaus Wirth himself. It grew out of Clascal, an object-oriented extension used for the Apple Lisa. Object Pascal added classes with single inheritance, virtual methods, and dynamic binding — deliberately conservative, with Wirth’s involvement ensuring the extensions were minimal and clean. Its purpose was to support MacApp, Apple’s Macintosh application framework.

Delphi (1995)

Delphi, released on February 14, 1995, was Borland’s answer to Visual Basic — a rapid application development environment combining Visual Basic’s ease of use with native code compilation. Created by Hejlsberg and his team, Delphi’s Object Pascal was a major evolution:

  • Classes (using class instead of object)
  • Properties (getters/setters wrapped in a single declaration)
  • Exception handling (try/except/finally)
  • Interfaces (COM-compatible multiple inheritance of specification)
  • Visual Component Library (VCL) with a form designer
  • Runtime type information (RTTI)

Delphi’s VCL was superior to Microsoft’s MFC (Microsoft Foundation Classes) in several ways: properties replaced getter/setter pairs with a single declaration settable at design time; events were type-safe method pointers eliminating MFC’s message-map boilerplate; streaming serialized forms as human-readable text (DFM files) that could be diffed and version-controlled; and RTTI enabled the IDE to inspect component properties at design time.

Delphi produced single .EXE files with no runtime dependencies — no VBRUNxxx.DLL, no MSVCRT.DLL. Distribution was trivial compared to Visual Basic or C++. Many successful commercial applications were built in Delphi, including early versions of Skype.

Anders Hejlsberg’s Departure

In 1996, Microsoft recruited Hejlsberg from Borland with a $500,000 signing bonus (Borland counter-offered $1,000,000, which Hejlsberg declined). At Microsoft, he became the lead architect of C# (2000) and later announced TypeScript (2012). C# is a direct descendant of Delphi’s Object Pascal: properties, events and delegates, component-oriented design, strongly typed references, and structured exception handling all trace directly to Delphi.

Borland’s Arc

Borland’s trajectory from a two-room office above a Jaguar repair garage to a billion-dollar company and back to irrelevance took barely two decades. The company rebranded as Inprise, then back to Borland, spun off its developer tools as CodeGear (2006), which was acquired by Embarcadero Technologies (2008). Embarcadero continues to develop and sell Delphi today. The Borland name was acquired by Micro Focus in 2009. But the products that Borland created during its ascent — particularly Turbo Pascal and Delphi — reshaped the software development industry in ways that outlasted the company itself.

Free Pascal and Lazarus

Free Pascal (FPC) is a mature, open-source Pascal compiler targeting an extraordinary range of architectures: x86 (16/32/64-bit), ARM, AArch64, PowerPC, SPARC, MIPS, Motorola 68000, AVR, and the JVM. It supports Windows, Linux, macOS, FreeBSD, DOS, Android, iOS, Haiku, Nintendo GBA/DS/Wii, AmigaOS, and various embedded platforms. Language compatibility spans Turbo Pascal 7.0, most Delphi versions, and parts of the ISO standards.

Lazarus is a cross-platform IDE built on Free Pascal, designed as a free, open-source equivalent of Delphi. It provides a visual form designer, the Lazarus Component Library (LCL) mirroring Delphi’s VCL, and the ability to build GUI applications for Windows, Linux, macOS, and FreeBSD from a single codebase.

Together, Free Pascal and Lazarus ensure that Pascal remains a living language — not a historical curiosity. Active development continues with regular releases, adding support for new platforms, new language features (generics, anonymous functions, advanced RTTI), and new libraries.

The Free Pascal community maintains an active ecosystem: the Online Package Manager (OPM) provides hundreds of third-party packages, and the compiler’s multi-target architecture means that code written in Free Pascal can target everything from Arduino-class microcontrollers to 64-bit server applications. For educators who still value Pascal’s pedagogical properties, Free Pascal and Lazarus provide a modern, free, cross-platform development environment that honors Pascal’s heritage while embracing contemporary requirements.

Wirth’s Language Trajectory

Wirth’s own languages form a complementary lineage, each smaller and more focused than the last:

Where most language designers add features, Wirth removed them. Oberon-07 has fewer features than Pascal. This is deliberate: “Make it as simple as possible, but no simpler.”
