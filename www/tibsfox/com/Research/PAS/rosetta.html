The Algol family tree, language comparisons, and the Hejlsberg lineage from Turbo Pascal to TypeScript

Every programming language is a reaction against something. Algol 60 was a reaction against machine code. Algol 68 was a reaction against Algol 60’s limitations. Pascal was a reaction against Algol 68’s complexity. C was a reaction against Pascal’s restrictions. The entire history of programming languages is a conversation, and Pascal’s contribution to that conversation was the radical idea that simplicity is a feature.

The Algol Family Tree

  1. Algol 58 (1958) — The ancestor. IAL (International Algorithmic Language). Backus, Bauer, Naur. Defined the idea of a formally specified programming language.
  2. Algol 60 (1960) — The breakthrough. Block structure, lexical scoping, recursion, BNF grammar. Wirth called it “the most influential programming language ever designed.” Never widely used in industry but shaped everything that followed.
  3. Algol W (1966) — Wirth’s first departure. Designed at Stanford as a simpler successor to Algol 60 with record types and while loops. Lost the Algol successor competition to Algol 68. The seed of Pascal.
  4. Pascal (1970) — Wirth’s reaction against Algol 68’s complexity. Designed at ETH Zürich for teaching structured programming. Small, clean, strict typing, no separate compilation (initially). Became the world’s most popular teaching language.
  5. Modula-2 (1978) — Wirth’s improvement on Pascal. Added modules (separate compilation), coroutines, low-level system access. Fixed most of Kernighan’s complaints. Less successful commercially because Turbo Pascal had already captured the market.
  6. Oberon (1987) — Wirth’s distillation. Removed features from Modula-2 to get something even simpler. Type extension (single inheritance), garbage collection, the Oberon operating system. Beautiful but too minimal for industry.
  7. Oberon-2 (1991) — Hanspeter Mössenböck’s extension. Added type-bound procedures (methods). The final step in Wirth’s language trajectory: each language simpler than the last.

The branch that Pascal rejected: Algol 68 (1968) was the “official” successor to Algol 60, designed by van Wijngaarden’s committee. It was extraordinarily powerful and extraordinarily complex — orthogonal design, two-level grammars, coercions, modes, united modes. Wirth was on the committee and left in disagreement. Pascal was his explicit reaction: the same structured programming goals, achieved with a fraction of the mechanism. The Algol 68 branch led to its own tradition (Bourne shell, some influence on C), but it was Pascal that won the teaching language war.

The parallel branch: Simula 67 (Dahl and Nygaard, Norway, 1967) grew from Algol 60 in a different direction — classes, objects, inheritance, virtual procedures. Simula influenced Smalltalk (Kay, Xerox PARC, 1972), which influenced the entire OOP tradition. Pascal and Simula shared a common ancestor in Algol 60 but diverged completely: Pascal toward simplicity and teaching, Simula toward modeling and objects. They would not converge until Object Pascal and Delphi borrowed OOP concepts decades later.

One Problem, Seven Languages — Hello World

Pascal

program HelloWorld;
begin
  writeln('Hello, world!');
end.

C

#include <stdio.h>

int main(void) {
    printf("Hello, world!\n");
    return 0;
}

Ada

with Ada.Text_IO;

procedure Hello is
begin
   Ada.Text_IO.Put_Line ("Hello, world!");
end Hello;

Modula-2

MODULE HelloWorld;
FROM InOut IMPORT WriteString, WriteLn;
BEGIN
  WriteString("Hello, world!");
  WriteLn;
END HelloWorld.

Java

public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, world!");
    }
}

Python

print("Hello, world!")

Rust

fn main() {
    println!("Hello, world!");
}

Pascal’s program declaration and end. with a period are inherited from Algol convention. The writeln procedure is built-in, not imported — a design choice that Kernighan criticized (no user-defined I/O) but that made teaching simpler. Compare with Ada’s explicit with import, C’s #include, and Java’s class ceremony. Python is the only language that is more concise.

Linked List Node — Type Definitions Compared

Pascal

type
  PNode = ^Node;
  Node = record
    data: integer;
    next: PNode;
  end;

C

typedef struct node {
    int data;
    struct node *next;
} Node;

Java

class Node {
    int data;
    Node next;
}

Rust

struct Node {
    data: i32,
    next: Option<Box<Node>>,
}

ML (Standard ML)

datatype 'a list = Nil | Cons of 'a * 'a list

Pascal introduced forward pointer declarations (PNode = ^Node before Node is defined) which was revolutionary — it allowed recursive data structures in a single-pass compiler. C solved the same problem with struct node *next using the struct tag. Rust requires Box for heap indirection and Option for nullable pointers, making the cost explicit. ML’s algebraic data type is the most abstract — and Pascal’s variant records are its direct ancestor.

Variant Records — The Ancestor of Algebraic Types

Pascal (variant record)

type
  ShapeKind = (Circle, Rectangle, Triangle);
  Shape = record
    x, y: real;
    case kind: ShapeKind of
      Circle:    (radius: real);
      Rectangle: (width, height: real);
      Triangle:  (base, altitude: real);
  end;

C (tagged union)

enum ShapeKind { CIRCLE, RECTANGLE, TRIANGLE };

struct Shape {
    double x, y;
    enum ShapeKind kind;
    union {
        struct { double radius; } circle;
        struct { double width, height; } rectangle;
        struct { double base, altitude; } triangle;
    };
};

Rust (enum)

enum Shape {
    Circle { x: f64, y: f64, radius: f64 },
    Rectangle { x: f64, y: f64, width: f64, height: f64 },
    Triangle { x: f64, y: f64, base: f64, altitude: f64 },
}

Haskell (algebraic data type)

data Shape
  = Circle    { x :: Double, y :: Double, radius :: Double }
  | Rectangle { x :: Double, y :: Double, width :: Double, height :: Double }
  | Triangle  { x :: Double, y :: Double, base :: Double, altitude :: Double }

Pascal’s variant record is the bridge between C’s untyped union and ML/Haskell’s algebraic data types. The case kind: ShapeKind of discriminant tags the variant, providing a degree of type safety that raw C unions lack. But Pascal did not enforce exhaustive matching — you could access the radius field of a Rectangle and the compiler would not stop you. Rust and Haskell close that hole with exhaustive pattern matching enforced at compile time. The lineage is clear: Pascal variants → ML datatypes (1973) → Haskell (1990) → Rust enums (2015).

Design Philosophy Comparison

AxisPascalCounterpartTrade-off
Type safety Pascal — strict, no implicit conversions, subrange types, set types C — permissive, implicit integer promotion, void pointers, casts Pascal catches more bugs at compile time; C gives the programmer more power and more rope
Complexity Pascal — small language, one-pass compiler, minimal features Ada — large language, packages, generics, tasking, exceptions, annexes Pascal is easier to learn and implement; Ada is more capable for large systems
Teaching Pascal — designed for students, structured programming, clear syntax Java — adopted for teaching (1998+), OOP-first, managed memory Pascal teaches procedural thinking; Java teaches object-oriented thinking; neither teaches both well
Typing Pascal — static, strong, explicit, verbose Python — dynamic, strong, implicit, concise Pascal catches type errors at compile time; Python catches them at runtime but writes faster

The Hejlsberg Lineage

Anders Hejlsberg is the most consequential language implementor of the personal computer era. His trajectory is the story of how Pascal’s DNA ended up inside the most popular programming language of the 2020s:

  1. PolyPascal (1981) — Hejlsberg writes a Pascal compiler for the Nascom microcomputer in Denmark while still a student. Borland notices.
  2. Turbo Pascal (1983) — Borland releases Hejlsberg’s integrated compiler+editor for CP/M and DOS at $49.95. Compiles 30× faster than alternatives. Sells 250,000 copies in two years. Destroys UCSD Pascal’s market.
  3. Turbo Pascal 5.5 (1989) — Adds object-oriented extensions. The beginning of Object Pascal.
  4. Delphi (1995) — Borland’s visual RAD environment for Windows, built on Object Pascal. Hejlsberg is chief architect. Wildly popular for database applications, client-server tools, and Windows desktop software through the late 1990s.
  5. C# (2000) — Microsoft hires Hejlsberg away from Borland. He leads the design of C#, which is essentially “what Delphi would look like if you started over on the CLR.” Properties, delegates, value types, LINQ — all carry Delphi DNA.
  6. TypeScript (2012) — Hejlsberg designs TypeScript at Microsoft. A typed superset of JavaScript that compiles to plain JS. By 2026 it is the dominant language for web development. The chain is complete: Pascal → Turbo Pascal → Delphi → C# → TypeScript.

One person. Four decades. A direct line from Blaise Pascal’s name to the language most web developers write every day.

Wirth’s Language Trajectory

  1. Euler (1965) — Wirth’s doctoral project. An Algol-like language exploring expression-oriented design. Experimental, never widely used. Proved Wirth could design a language and build a compiler.
  2. Algol W (1966) — Stanford. A cleaned-up Algol 60 with record types, while loops, string handling. Wirth’s proposal for the Algol successor. Rejected by the IFIP committee in favor of Algol 68.
  3. Pascal (1970) — ETH Zürich. Wirth’s reaction against Algol 68’s complexity. Designed for teaching. Small, clean, one-pass compilable. Became the world’s most popular teaching language.
  4. Modula (1975) — Experiment with modules and concurrency for real-time systems (the Lilith workstation). Never widely distributed.
  5. Modula-2 (1978) — The production version. Modules with separate compilation, coroutines, system-level access. Fixed Pascal’s major deficiencies. Used in European education and embedded systems.
  6. Oberon (1987) — Radical simplification. Removed enumeration types, variant records, for loops, subrange types. Added type extension (single inheritance) and garbage collection. The Oberon System was written entirely in Oberon.
  7. Oberon-2 (1991) — Mössenböck’s extension adding type-bound procedures (methods) and FOR loops back. The final language in the sequence. Each step from Algol W to Oberon-2 was a simplification.
Wirth’s trajectory is the opposite of most language designers’. Where others add features over time, Wirth removed them. Each language was simpler than its predecessor. This is the architectural philosophy that made Pascal great as a teaching language — and that made Modula-2 and Oberon too austere for commercial success.
