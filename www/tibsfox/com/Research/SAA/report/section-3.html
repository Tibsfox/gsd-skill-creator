3.1 What Are Subagents?

Subagents are specialized AI assistants that run in their own independent context window within Claude Code. When Claude encounters a task that matches a subagent's description, it delegates the work automatically, spawning the subagent with its own system prompt, specific tool access, and independent permission settings. The subagent completes its task and returns results to the parent conversation, keeping its exploration, intermediate reasoning, and tool calls contained in a separate context.

Definition: What Is a Subagent?

A subagent is a self-contained AI worker with its own identity (system prompt from a markdown file), capabilities (tool allowlist/denylist), autonomy level (permission mode), and resource constraints (model selection, max turns). It runs in a forked context window, meaning its reasoning and tool use do not consume tokens in the parent conversation. When it finishes, only its final output is returned to the caller.

This architecture solves a fundamental problem in agentic programming: context pollution. Without subagents, a single conversation accumulates tool outputs, intermediate reasoning, and exploratory dead ends until the context window fills up or the signal-to-noise ratio degrades. Subagents act as disposable workspaces. The parent conversation sends a task, the subagent explores freely, and only the distilled result comes back.

Claude Code's subagent system is not a framework you import or configure through an API. It is a native capability of the runtime. You define agents as markdown files with YAML frontmatter, place them in the right directory, and Claude automatically discovers them and delegates matching tasks. There is no orchestration code to write, no message passing to implement, no lifecycle to manage. The runtime handles spawning, context isolation, tool permissioning, and result collection.

How Delegation Works

When Claude Code receives a user prompt, it evaluates the task against all available subagent descriptions. If a subagent's description matches the task, Claude invokes the Agent tool with the agent name and a prompt. The subagent receives:

  1. Its own system prompt (the full markdown body of its definition file)
  2. The delegated prompt from the parent
  3. Access to only the tools specified in its frontmatter
  4. The permission mode specified in its frontmatter
  5. Any preloaded skills specified in its frontmatter

The subagent then operates autonomously within these constraints, using tools, reasoning through problems, and producing output. When it reaches its conclusion (or hits its maxTurns limit), its final response is returned to the parent conversation as a single message.

Subagents vs. Agent Teams

Dimension Subagents Agent Teams (Leader-Worker)
Context Independent context window per agent. Parent sees only the final result. Shared task list and mailbox. Leader coordinates via structured messages.
Communication One-shot: parent sends prompt, receives result. No mid-task interaction. Ongoing: leader assigns tasks, workers report back, leader redistributes.
Coordination None between subagents. Each operates in isolation. Parent orchestrates sequentially or in parallel. Built-in. Leader manages dependencies, redistributes failed tasks, aggregates results.
Parallelism Supported via background: true or run_in_background parameter. Parent polls for completion. Native. Workers execute concurrently by default. Leader monitors progress.
State Ephemeral. Subagent state is discarded after completion (unless using persistent memory). Persistent within session. Task list, completion status, and mailbox survive across turns.
Use Case Focused delegation: research, verification, implementation of a single bounded task. Complex workflows: multi-step projects requiring coordination between specialists.
Definition Markdown files with YAML frontmatter in .claude/agents/. Defined in managed settings or via Team tool at runtime.
Research Insight

The subagent model maps directly to the Cognitive Architecture for Language Agents (CoALA) framework described by Sumers et al. Each subagent is a self-contained cognitive unit with its own perception (tools), reasoning (system prompt + model), and action (tool execution) layers. The parent conversation acts as a meta-cognitive layer, deciding which cognitive unit to activate based on task requirements. This mirrors CoALA's "module selection" mechanism where a central controller routes tasks to specialized processing modules.

3.2 Built-in Subagents

Claude Code ships with several built-in subagents that handle common tasks. These are always available regardless of project configuration and demonstrate the core patterns that custom subagents build upon.

Explore Subagent

Explore

Model: Haiku (cost-optimized for high-volume search)

Tools: Read, Grep, Glob (read-only)

Purpose: Codebase exploration and search. Used automatically when Claude needs to find files, understand project structure, or locate specific code patterns.

Thoroughness Levels:

  • Quick: Surface-level search. Checks obvious locations, common naming patterns. Fast, low-cost. Good for "where is the config file?" type queries.
  • Medium: Standard exploration. Follows imports, checks related files, reads function signatures. Balances speed and coverage.
  • Very thorough: Deep search. Examines every potential match, follows dependency chains, reads full file contents. Used for complex architectural questions or when initial searches come up empty.

Why Haiku? Explore is called frequently during normal conversation — often dozens of times per session. Using Haiku keeps the cost per exploration low (roughly 10x cheaper than Opus) while providing sufficient capability for file search and pattern matching. The read-only tool restriction means there is no risk from using a smaller model; it cannot modify anything.

Plan Subagent

Plan

Model: Inherits from parent conversation

Tools: Read, Grep, Glob (read-only)

Purpose: Research and planning in plan mode. When Claude Code is in plan mode (/plan), this subagent handles research tasks, reading files and searching for information to build plans without making any changes.

Key Property: Inherits the parent's model because planning quality scales with model capability. A plan built by Opus will be more architecturally sound than one built by Haiku. Since planning happens less frequently than exploration, the higher cost is justified by better plan quality.

General-Purpose Subagent

General-Purpose (Task Tool)

Model: Inherits from parent conversation

Tools: All tools available to the parent

Purpose: Complex multi-step tasks that benefit from context isolation. When Claude decides a task is complex enough to warrant a separate context window (to avoid polluting the main conversation), it spawns a general-purpose subagent that has full capabilities.

Use Cases: Large refactoring operations, multi-file changes, research-then-implement workflows where the research phase would consume too much context.

Other Built-in Subagents

statusline-setup

Model: Sonnet

Purpose: Terminal status line configuration. Handles the setup and customization of Claude Code's terminal status display.

Claude Code Guide

Model: Haiku

Purpose: Self-help and documentation lookup. Answers questions about Claude Code features, commands, and configuration. Uses Haiku because documentation lookup is a pattern-matching task that does not require deep reasoning.

Model Selection Pattern in Built-in Agents

The built-in subagents demonstrate a deliberate model-routing pattern: Haiku for high-frequency, low-stakes tasks (exploration, documentation lookup), inherited model for quality-sensitive tasks (planning, complex implementation). This pattern — matching model capability to task requirements — is the single most impactful optimization for cost control in production subagent systems. A project that routes 80% of its subagent calls through Haiku and reserves Opus for the remaining 20% can reduce agent costs by 60-70% with minimal quality impact.

3.3 Creating Subagents

There are four ways to create subagents in Claude Code, each suited to different workflows and levels of permanence.

Method 1: The /agents Interactive Command (Recommended)

The /agents slash command opens an interactive interface for creating, editing, and managing subagents. This is the recommended approach for most users because it guides you through the available options and validates your configuration.

Terminal 
# Launch interactive agent creation
/agents

# The interface presents options:
# 1. Create a new agent
# 2. Edit an existing agent
# 3. Delete an agent
# 4. View agent details

When you select "Create a new agent," you are prompted for the name, description, tools, model, and permission mode. The command validates your inputs and writes the resulting markdown file to .claude/agents/ in your project directory.

Method 2: Manual Markdown Files

For full control, you can create agent definition files manually. Each agent is a markdown file with YAML frontmatter that specifies its configuration, followed by a markdown body that becomes the agent's system prompt.

Markdown (.claude/agents/security-reviewer.md) 
---
name: security-reviewer
description: Reviews code for security vulnerabilities including OWASP Top 10,
  injection flaws, authentication issues, and sensitive data exposure.
tools: Read, Grep, Glob
model: sonnet
permissionMode: plan
maxTurns: 30
color: red
---

You are a security code reviewer specializing in the OWASP Top 10.

## Review Process

1. **Reconnaissance:** Identify entry points (API routes, form handlers, auth flows)
2. **Static Analysis:** Search for known vulnerable patterns
3. **Data Flow:** Trace user input from entry to storage/output
4. **Authentication:** Verify session management, token handling, password storage
5. **Authorization:** Check access control on every endpoint and resource
6. **Output Encoding:** Verify XSS prevention on all rendered content

## Report Format

For each finding:
- **Severity:** Critical / High / Medium / Low / Informational
- **Location:** File path and line number
- **Description:** What the vulnerability is
- **Impact:** What an attacker could do
- **Remediation:** How to fix it with code example

The markdown body after the frontmatter separator (---) becomes the agent's full system prompt. This is where you define the agent's identity, instructions, processes, and output format. The more specific and structured this prompt is, the more reliably the agent performs its task.

Method 3: CLI --agents JSON Flag

For session-only agents that do not need to persist, you can define them inline when launching Claude Code. This is useful for one-off tasks, CI/CD pipelines, or experiments.

Bash 
# Define a session-only agent via CLI flag
claude --agents '[{
  "name": "log-analyzer",
  "description": "Analyzes log files for errors, patterns, and anomalies",
  "tools": ["Read", "Grep", "Glob", "Bash"],
  "model": "haiku",
  "permissionMode": "default",
  "systemPrompt": "You analyze log files. Focus on error patterns, frequency, and root causes."
}]'

# The agent is available for this session only
# It will not persist after the session ends

Session-only agents defined via --agents take priority 2 in the resolution order (after managed settings but before project-level agents), meaning they can temporarily override project agents with the same name.

Method 4: Generate with Claude

Within the /agents interface, you can select "Generate with Claude" to have Claude create an agent definition based on a natural language description of what you need. This is the fastest path from idea to working agent.

Terminal 
# In the /agents interface, select "Generate with Claude"
# Then describe what you need:

"I need an agent that reviews database migration files for
backward compatibility issues, checks that rollback scripts
exist, and verifies that indexes are not dropped without
replacement. It should be read-only and use Sonnet."

# Claude generates the full agent definition including:
# - Appropriate frontmatter (tools, model, permissions)
# - Detailed system prompt with review checklist
# - Output format specification
Production Experience

In our production system (26 agents, 190+ completed missions), every agent started as a manual markdown file. The interactive tools are excellent for getting started, but production agents evolve through iteration: you run a task, observe the agent's behavior, refine the system prompt, adjust tool access, and repeat. The markdown format makes this iteration loop fast — edit the file, re-run the task, observe the difference. Version control (git) tracks how each agent's prompt has evolved over time, providing a clear history of what changes improved or degraded performance.

3.4 Subagent Scope and Priority

Subagent definitions can come from multiple sources. When the same agent name appears in more than one location, Claude Code resolves the conflict using a strict priority order. Higher-priority definitions completely override lower-priority ones — there is no merging.

Priority Location Scope Persistence Use Case
1 (Highest) Managed settings Organization-wide Permanent (admin-controlled) Enterprise security policies, compliance agents, org-wide standards
2 --agents CLI flag Current session Session only Experiments, CI/CD pipelines, one-off tasks, temporary overrides
3 .claude/agents/ Current project Committed to repo Project-specific workflows, team agents, domain specialists
4 ~/.claude/agents/ All projects User's home directory Personal utilities, cross-project tools, individual preferences
5 (Lowest) Plugin agents/ directory Where plugin is enabled Plugin lifecycle Third-party agent providers, shared community agents

Priority Resolution in Practice

Understanding the priority system is critical for managing agent configurations across teams and environments. Consider this scenario:

Example: Layered Agent Configuration

An organization has a security-reviewer agent defined at three levels:

  • Managed settings (Priority 1): Defines the security reviewer with SOC2 compliance requirements and mandatory OWASP checks. This is the baseline that all projects must follow.
  • Project .claude/agents/ (Priority 3): Defines a security-reviewer with project-specific API patterns and custom authentication flows added to the review checklist.
  • User ~/.claude/agents/ (Priority 4): Defines a security-reviewer with the user's preferred output format and verbosity level.

Result: Only the managed settings version runs. The project and user versions are completely ignored — not merged, not layered. If you need project-specific additions, they must be incorporated into the managed settings definition or given a different agent name (e.g., project-security-reviewer).

This "highest wins" resolution means that organizations can enforce agent standards without worrying about project-level or user-level definitions overriding them. It also means that developers experimenting with session-only agents (--agents) can temporarily override project agents for testing without modifying committed files.

Naming Convention

Agent names must be lowercase with hyphens (e.g., security-reviewer, gsd-executor). Names are unique within each priority level. To avoid conflicts, use descriptive prefixes that signal the agent's domain: gsd- for workflow agents, research- for research agents, infra- for infrastructure agents. Our production system uses gsd- as the prefix for all 13 workflow agents and descriptive standalone names for the remaining 13.

3.5 Complete Frontmatter Reference

The YAML frontmatter in an agent markdown file controls every aspect of the subagent's behavior. Below is the complete reference for all available fields as of April 2026.

Field Required Type Description
name Yes string Unique identifier. Lowercase letters and hyphens only. Must be unique within its priority level. This is how the agent is referenced when invoking via the Agent tool.
description Yes string Natural language description of when Claude should delegate to this agent. This is the primary signal Claude uses for automatic delegation. Be specific: "Reviews Python code for security vulnerabilities" is better than "Security agent." Maximum 250 characters.
tools No comma-separated string Allowlist of tools the agent can use. If omitted, the agent inherits all tools available to the parent. Restricting tools is the primary mechanism for enforcing agent safety boundaries. Example: Read, Grep, Glob
disallowedTools No comma-separated string Denylist of tools removed from the inherited set. Use when you want "everything except X." Applied before the tools allowlist if both are present.
model No string Model to use. Accepts shorthand (sonnet, opus, haiku) or full model IDs (claude-sonnet-4-20250514). If omitted, inherits the parent conversation's model. See Section 3.6 for full resolution order.
permissionMode No string Controls how the agent handles permissions. One of: default, acceptEdits, auto, dontAsk, bypassPermissions, plan. See Section 3.13 for details on each mode.
maxTurns No integer Maximum number of agentic turns (tool-use cycles) before the agent is forced to return. Prevents runaway agents from consuming excessive resources. No default limit if omitted.
skills No list of strings Skills whose full content is preloaded into the agent's context at startup. This is not just "making skills available" — the entire skill content is injected into the system prompt. See Section 3.11.
mcpServers No object or list MCP servers scoped to this agent. Can be inline definitions (server starts/stops with agent) or string references to parent session servers. See Section 3.8.
hooks No object Lifecycle hooks (PreToolUse, PostToolUse, etc.) scoped to the agent's lifetime. Automatically cleaned up when the agent finishes. See Section 3.12.
memory No object Persistent memory configuration with three scopes: user, project, local. Memory survives across conversations. See Section 3.9.
background No boolean When true, the agent always runs as a background task. The parent is not blocked and receives a notification when the agent completes. Useful for long-running research or analysis tasks.
effort No string Reasoning effort level: low, medium, high, or max. Only effective with Opus models. Controls how much compute the model allocates to reasoning. Higher effort = better quality but more tokens and latency.
isolation No string Set to "worktree" for git worktree-based isolation. The agent operates on a separate copy of the repository, preventing file conflicts with the parent or other agents. See Section 3.10.
color No string Display color in the terminal UI. One of: red, blue, green, yellow, purple, orange, pink, cyan. Helps visually distinguish agent output in logs and the status line.
initialPrompt No string A prompt that is automatically submitted as the first turn when the agent runs as a main agent (not as a subagent). Used for agents that should start working immediately without user input.

Complete Example: Production Agent Definition

This is a real production agent from our system, demonstrating how multiple frontmatter fields work together:

Markdown (.claude/agents/research-fleet-commander.md) 
---
name: research-fleet-commander
description: Launches parallel research agent fleets, aggregates results,
  and produces structured research documents. Proven at 28-doc HEL series
  and 12-doc OPEN problems scale.
tools: Read, Write, Edit, Bash, Grep, Glob, WebSearch, WebFetch
color: cyan
effort: high
maxTurns: 60
---

You are a research fleet commander. You take a research topic,
break it into parallel investigation tracks, launch specialist
agents for each track, aggregate their findings, and produce
structured research documents with cross-references.

## Fleet Dispatch Pattern

1. **Decompose** the research topic into 3-6 parallel tracks
2. **Launch** one specialist agent per track (background: true)
3. **Monitor** for agent completion notifications
4. **Aggregate** findings into structured documents
5. **Cross-reference** across documents for consistency
6. **Produce** final output in the project's research format

Key design decisions in this definition:

  • effort: high — Research synthesis requires deep reasoning, so we explicitly request high-effort computation from the model.
  • maxTurns: 60 — Research tasks are long-running but not unbounded. 60 turns provides enough runway for complex research while preventing infinite loops.
  • tools includes WebSearch and WebFetch — Research agents need internet access, but most of our other agents do not. Tool allowlisting keeps web access confined to agents that need it.
  • color: cyan — When multiple agents are running in parallel, color coding makes it immediately visible which agent is producing which output in the terminal.

3.6 Model Resolution Order

When a subagent is spawned, its model is determined by a four-step resolution chain. The first match wins:

Model Resolution Chain

  1. CLAUDE_CODE_SUBAGENT_MODEL environment variable — Global override. When set, all subagents use this model regardless of any other configuration. Useful for cost control in CI/CD environments where you want to force Haiku for all subagent calls.
  2. Per-invocation model parameter — The calling code can specify a model when invoking the Agent tool. This overrides the agent's frontmatter for this specific invocation. Used by orchestration skills that need dynamic model routing.
  3. Frontmatter model field — The model specified in the agent's YAML frontmatter. This is the agent author's intended model for this agent's workload.
  4. Main conversation's model — If no model is specified at any level, the subagent inherits whatever model the parent conversation is using. This is the most common case for agents that do not specify a model.

Model Shorthand

The model field accepts shorthand names that resolve to the latest version of each model family:

Shorthand Resolves To Typical Use
haiku Latest Claude Haiku High-frequency search, triage, simple categorization
sonnet Latest Claude Sonnet Implementation, code generation, standard analysis
opus Latest Claude Opus Architecture, complex reasoning, research synthesis

You can also use full model IDs (e.g., claude-sonnet-4-20250514) to pin a specific version. This is useful when you need reproducible behavior and do not want model updates to change agent performance.

Cost Implications of Model Selection

Model selection is the single largest lever for controlling subagent costs. The price difference between models is significant:

~10x
Cost difference: Haiku vs Opus per token
~3x
Cost difference: Sonnet vs Opus per token
60-70%
Cost reduction with intelligent model routing

Our production system routes approximately 40% of subagent invocations through the inherited Opus model (for planning, verification, and complex implementation), 35% through Sonnet (for standard code generation and analysis), and 25% through Haiku (for exploration and triage). This distribution evolved through observation: we started with everything inheriting Opus and gradually pushed workloads to smaller models until we found the quality threshold for each task type.

3.7 Tool Control

Tool control is the primary mechanism for enforcing safety boundaries on subagents. By restricting which tools an agent can use, you define what it is physically capable of doing — regardless of what its system prompt says. A read-only agent with tools: Read, Grep, Glob cannot modify files even if instructed to, because the Write and Edit tools are not available.

Allowlist: The tools Field

The tools field defines an explicit allowlist. Only the named tools are available to the agent. All other tools are unavailable.

YAML Frontmatter 
# Read-only agent: can search and read, cannot modify anything
tools: Read, Grep, Glob

# Implementation agent: can read and write, no shell access
tools: Read, Write, Edit, Grep, Glob

# Full-access agent: can do everything including run commands
tools: Read, Write, Edit, Bash, Grep, Glob

# Research agent: read-only + web access
tools: Read, Grep, Glob, WebSearch, WebFetch

Denylist: The disallowedTools Field

The disallowedTools field removes specific tools from the inherited set. This is the inverse of tools: instead of listing what the agent can use, you list what it cannot use.

YAML Frontmatter 
# Everything except shell access and web access
disallowedTools: Bash, WebSearch, WebFetch

# Everything except file modification
disallowedTools: Write, Edit

Combined: Both Fields Together

When both tools and disallowedTools are present, resolution follows this order:

  1. disallowedTools is applied first, removing tools from the inherited set
  2. tools is then resolved against the remaining set

In practice, using both fields simultaneously is rare. Most agents use one or the other.

Controlling Subagent Spawning: The Agent() Syntax

A powerful but less obvious capability is controlling which other subagents a subagent can spawn. By default, a subagent can invoke any agent available in the system. The Agent() syntax in the tools field restricts this:

YAML Frontmatter 
# This agent can only spawn 'worker' and 'researcher' subagents
tools: Read, Write, Edit, Bash, Grep, Glob, Agent(worker, researcher)

# This agent can use tools but cannot spawn any subagents
tools: Read, Write, Edit, Bash, Grep, Glob
# (Agent tool is not listed, so subagent spawning is disabled)

This is essential for managing agent hierarchies. An orchestrator agent might be allowed to spawn executors and verifiers but not other orchestrators, preventing recursive orchestration loops. Our gsd-executor does not include the Agent tool in its toolset, meaning it cannot spawn subagents — it focuses entirely on executing its assigned plan.

Tool Control as the Safety Foundation

Tool control is not just a convenience feature — it is the foundation of subagent safety. A well-designed tool configuration makes it impossible for an agent to perform dangerous actions, regardless of prompt injection attacks, hallucination, or instruction following failures. A security reviewer with tools: Read, Grep, Glob physically cannot execute shell commands or modify files. This is a hard boundary enforced by the runtime, not a soft boundary enforced by prompting. Design your tool sets with the principle of least privilege: give each agent exactly the tools it needs and nothing more.

3.8 MCP Server Scoping

MCP (Model Context Protocol) servers provide tools, resources, and prompts to agents via a standardized protocol. In the context of subagents, MCP servers can be scoped to individual agents, meaning the server starts when the agent spawns and stops when the agent finishes. This keeps specialized tools out of the main conversation's context, reducing noise and improving focus.

Inline Server Definitions

Define MCP servers directly in the agent's frontmatter. The server lifecycle is tied to the agent — it starts when the agent is invoked and is terminated when the agent completes.

YAML Frontmatter 
---
name: browser-tester
description: Tests web applications using Playwright browser automation.
tools: Read, Grep, Glob, Bash
mcpServers:
  playwright:
    command: npx
    args: ["@anthropic-ai/mcp-playwright"]
    env:
      BROWSER: chromium
      HEADLESS: "true"
---

You are a browser testing agent. Use the Playwright MCP tools
to navigate web pages, fill forms, click buttons, and verify
that the application behaves correctly.

With this configuration, the Playwright MCP server (and its tools for navigating pages, clicking elements, and taking screenshots) is only available when the browser-tester agent is running. The main conversation never sees Playwright tools in its tool list, which means Claude never tries to use browser automation unless it has specifically delegated to this agent.

String References: Sharing Parent Servers

If the parent session already has an MCP server running, subagents can reference it by name instead of defining it inline. The subagent shares the parent's server connection rather than starting a new instance.

YAML Frontmatter 
---
name: db-analyst
description: Analyzes database schemas, queries, and performance.
tools: Read, Grep, Glob
mcpServers:
  - postgres  # References the parent's 'postgres' MCP server
---

You analyze database schemas and query performance.
Use the postgres MCP tools to inspect tables, indexes,
and execution plans.

When to Use Inline vs. Reference

Approach Server Lifecycle Best For
Inline definition Starts with agent, stops with agent Tools that should not be available to the main conversation (browser automation, email sending, deployment tools). Keeps specialized capabilities isolated to the agent that needs them.
String reference Shares parent's running server Tools that the parent already uses and the subagent also needs (database connections, file watchers). Avoids spinning up duplicate server instances.
Production Experience

MCP server scoping is one of the most underutilized subagent features. In our production system, we use it to keep the Unison language tools scoped to research agents (so the main conversation does not see 30+ Unison-specific tools cluttering its tool list) and to scope math co-processor tools to computation agents. The noise reduction in the main conversation is significant: instead of Claude sorting through 80+ tools on every decision, it sees only the 15-20 tools relevant to general work. Specialized tools appear only when a specialized agent is running.

3.9 Persistent Memory

By default, a subagent's state is ephemeral — everything it learns and reasons about is discarded when it finishes. The memory field changes this by giving the agent access to persistent markdown files that survive across conversations. This enables agents to learn from past invocations, maintain running logs, and build up domain knowledge over time.

Three Memory Scopes

Scope Location Visibility Use Case
user ~/.claude/agent-memory/ Same user, all projects Personal preferences, cross-project patterns, learned workflows. Example: a code reviewer that remembers your preferred review style across all repositories.
project .claude/agent-memory/ All users, same project Project-specific knowledge shared across the team. Example: a debugger that remembers past incidents and their root causes, available to all team members. Committed to version control.
local .claude/agent-memory-local/ Same user, same project Personal project notes that should not be shared. Example: your own investigation logs or work-in-progress notes. Typically gitignored.

How Memory Works

When a subagent with memory is spawned:

  1. The system prompt includes memory management instructions and the first 200 lines of the agent's MEMORY.md file
  2. The Read, Write, and Edit tools are automatically enabled for the memory file path, even if the agent's tools field does not include them
  3. The agent can read its full memory, append new findings, update outdated entries, and reorganize the file
  4. When the agent completes, the memory file persists on disk for the next invocation
YAML Frontmatter 
---
name: incident-analyzer
description: Analyzes production incidents, identifies root causes,
  and tracks patterns across incidents.
tools: Read, Grep, Glob, Bash
memory:
  project: true
color: red
---

You are an incident analyzer. Before starting any analysis,
read your full memory file to check for related past incidents.

After completing analysis, update your memory with:
- Incident date and severity
- Root cause (brief)
- Resolution steps taken
- Patterns noticed (connections to past incidents)
- Recommendations for prevention

Memory Best Practices

Effective Memory Patterns

  • Consult before work: The agent should always read its memory before starting a new task. Past context prevents repeating investigations and surfaces relevant prior findings.
  • Update after completing tasks: Every invocation should leave the memory file richer than it found it. Even "nothing notable" is worth recording to avoid re-checking the same area.
  • Structure with headings: Use markdown headings and bullet points for fast scanning. The first 200 lines are injected into context, so put the most important information at the top.
  • Prune periodically: Memory files grow. Include a "review and prune" instruction in the system prompt so the agent occasionally consolidates entries and removes outdated information.
  • Separate hot from cold: Keep frequently referenced information (current patterns, active issues) at the top. Move historical entries to the bottom or a separate section.

3.10 Worktree Isolation

When multiple subagents work on the same repository simultaneously, file conflicts are inevitable. Agent A writes to src/utils.ts while Agent B is reading it, or both agents try to modify the same configuration file. The isolation: "worktree" field solves this by giving each agent its own copy of the repository via git worktrees.

How Worktree Isolation Works

  1. Spawn: When the agent is invoked, Claude Code creates a new git worktree in a temporary directory, checked out to the current branch.
  2. Execute: The agent's working directory is set to the worktree. All file operations (Read, Write, Edit, Bash) operate on the worktree copy, not the main checkout.
  3. Complete: When the agent finishes, its changes exist in the worktree. The parent can merge them back into the main checkout.
  4. Cleanup: If the agent made no changes, the worktree is automatically cleaned up. If changes were made, the worktree persists until explicitly removed.
YAML Frontmatter 
---
name: parallel-refactorer
description: Refactors a module in isolation using a git worktree.
tools: Read, Write, Edit, Bash, Grep, Glob
isolation: worktree
permissionMode: acceptEdits
color: green
---

You refactor code modules. You operate in a git worktree,
so your changes are isolated from the main checkout.

After completing your refactoring:
1. Run tests to verify nothing is broken
2. Commit your changes with a descriptive message
3. Report what you changed and the test results

Real Pattern: /batch and Worktrees

The /batch command in Claude Code uses worktree isolation internally. When you submit multiple tasks to /batch, each task gets its own worktree. This allows true parallel execution: four agents can each refactor a different module simultaneously without interfering with each other. When all tasks complete, the results can be merged back in sequence.

Worktree Isolation Architecture

Main checkout: /project/
  |
  +-- Worktree A: /tmp/project-wt-a/  (Agent 1: refactoring auth module)
  |
  +-- Worktree B: /tmp/project-wt-b/  (Agent 2: refactoring database layer)
  |
  +-- Worktree C: /tmp/project-wt-c/  (Agent 3: updating API endpoints)

Each worktree shares the same .git directory but has independent
working files. Agents can read, write, and commit independently.

When to Use Worktree Isolation

  • Parallel execution: Multiple agents modifying files simultaneously. Without worktrees, the second agent to write to a shared file wins and the first agent's changes are lost.
  • Experimental changes: An agent that might produce changes you want to review before merging. The worktree acts as a staging area.
  • Safe rollback: If the agent's changes turn out to be wrong, you can simply delete the worktree. No need to revert commits in the main checkout.
  • Long-running tasks: Agents that take many turns and produce incremental changes. The worktree accumulates changes while the main checkout remains clean for other work.

Production Note: Worktree Overhead

Worktree creation adds a few seconds of overhead per agent invocation (git needs to create the worktree and checkout files). For small, fast tasks, this overhead is not worth it — just use standard isolation. Reserve worktree isolation for tasks where file conflicts are likely or where you need true parallel execution. In our production system, we use worktrees primarily for wave-based parallel execution (where 3-6 agents execute simultaneously) and for the /batch command.

3.11 Skills Preloading

The skills field in agent frontmatter controls which skills are preloaded into the agent's context at startup. This is a commonly misunderstood feature, so the distinction is important:

Skills Preloading vs. Skills Availability

Preloading means the full content of the skill file is injected into the agent's system prompt at startup. The agent sees the entire skill definition as part of its instructions. This consumes context window tokens but ensures the agent has the skill's knowledge immediately, without needing to discover or load it.

Availability is separate from preloading. Subagents do not automatically inherit skills from the parent conversation. If you want a subagent to use a specific skill, you must explicitly list it in the skills field.

YAML Frontmatter 
---
name: api-developer
description: Implements API endpoints following project conventions
  and error handling patterns.
tools: Read, Write, Edit, Bash, Grep, Glob
skills:
  - api-conventions
  - error-handling-patterns
  - typescript-patterns
permissionMode: acceptEdits
color: blue
---

You implement API endpoints. The preloaded skills contain
the project's API conventions and error handling patterns.
Follow them exactly.

When this agent spawns, its system prompt includes the full content of three skill files. This means the agent starts with detailed knowledge of the project's API naming conventions, request/response formats, error codes, and TypeScript patterns. Without preloading, the agent would need to discover and read these conventions during its execution, consuming turns and potentially missing important guidelines.

Context Budget Considerations

Each preloaded skill consumes context window tokens. A skill with 2,000 tokens of content reduces the agent's available context by 2,000 tokens. For Opus with a 1M token context, this is negligible. For Haiku with a smaller context, it can be significant.

Strategy When to Use Trade-off
Preload all relevant skills Agent needs skill knowledge immediately and consistently. Context budget is not a concern (Opus model, few skills). Higher context consumption but faster, more reliable agent behavior.
Preload critical skills only Agent has many potential skills but only a few are critical for every invocation. Balances context consumption with immediate access to essential knowledge.
No preloading (rely on tool use) Agent can discover needed information via Read/Grep. Skills are large or numerous. Saves context but adds turns and introduces risk of the agent missing guidelines.

In our production system, we preload 1-3 skills per agent. The gsd-executor preloads no skills (it reads its plan directly). The gsd-planner preloads the gsd-workflow skill. Research agents preload domain-specific skills for the topic they are investigating.

3.12 Hooks in Subagents

Hooks are deterministic programs that run at specific points in the agent lifecycle. When defined in a subagent's frontmatter, hooks are scoped to that agent's lifetime — they activate when the agent spawns and are automatically cleaned up when the agent finishes. This scoping prevents hook interference between agents and keeps the parent conversation's hook environment clean.

Hook Lifecycle Points

Hook When It Fires Common Use in Subagents
PreToolUse Before the agent invokes a tool Validation (prevent writes to protected paths, enforce naming conventions, block dangerous commands)
PostToolUse After a tool returns results Logging (record file modifications), formatting (auto-lint after writes), metrics (track tool usage)
Stop When the agent is about to return Converted to SubagentStop in subagent context. Used for cleanup, summary generation, state persistence.

Example: Read-Only Database Agent with Query Validation

YAML Frontmatter 
---
name: db-reader
description: Reads database schemas and executes read-only queries
  for analysis. All queries are validated before execution.
tools: Read, Grep, Glob, Bash
hooks:
  PreToolUse:
    - matcher: Bash
      hooks:
        - type: command
          command: |
            if echo "$TOOL_INPUT" | grep -iqE '(INSERT|UPDATE|DELETE|DROP|ALTER|TRUNCATE|CREATE)'; then
              echo '{"decision": "block", "reason": "Write operations are not permitted"}'
            fi
permissionMode: acceptEdits
color: purple
---

You are a read-only database analyst. You can query the database
to understand schemas, data distributions, and query performance.

CRITICAL: You must NEVER execute write operations. Your PreToolUse
hook will block any query containing INSERT, UPDATE, DELETE, DROP,
ALTER, TRUNCATE, or CREATE statements. This is a safety measure.
If you need to suggest schema changes, describe them in your
report without executing them.

In this example, the PreToolUse hook intercepts every Bash command the agent attempts to run. If the command contains SQL write keywords, the hook returns a block decision, preventing the command from executing. The agent sees the block message and can adjust its approach. This is a hard safety boundary enforced by deterministic code, not by the LLM following instructions.

Stop Hook Conversion

When a Stop hook is defined in a subagent's frontmatter, it is converted to a SubagentStop hook at runtime. This conversion is important because it changes the hook's semantics: a regular Stop hook can prevent the conversation from ending, but a SubagentStop hook can only modify the subagent's final output — it cannot prevent the subagent from returning to its parent. This ensures that subagents always complete and return results, even if the stop hook wants to add information or formatting.

Hook Scoping is Automatic

You do not need to manage hook lifecycle manually. When a subagent spawns, its hooks are registered. When the subagent completes (regardless of whether it succeeded, failed, or hit its maxTurns limit), its hooks are deregistered. This means hooks from one subagent invocation never interfere with the next invocation or with other subagents running in parallel.

3.13 Permission Modes

The permissionMode field controls how the agent handles tool permission requests. This determines the level of human oversight during agent execution.

Mode Behavior Use Case
default Prompts the user for permission on each potentially destructive action (file writes, shell commands). Same behavior as the main conversation. High-stakes agents where you want to review every action. Rarely used for subagents because the interruptions defeat the purpose of delegation.
acceptEdits Automatically approves file edits (Write, Edit) but still prompts for shell commands (Bash). A middle ground between full oversight and full autonomy. Implementation agents that write code but whose shell commands should be reviewed (in case they run tests, install packages, or modify system state).
auto Automatically approves all tool use. No permission prompts. The agent runs to completion without human intervention. Trusted agents with well-tested system prompts and appropriate tool restrictions. Most production subagents use this mode.
dontAsk Similar to auto but skips the permission system entirely rather than auto-approving. Slight performance benefit from not evaluating permission rules at all. High-frequency agents where permission evaluation overhead matters. Read-only agents where there is no risk.
bypassPermissions Ignores all permission checks, including those from hooks and managed settings. The most permissive mode available. Internal system agents that need to operate without any restrictions. Use with extreme caution.
plan Read-only mode. The agent can use read tools (Read, Grep, Glob) but all write operations are blocked. Equivalent to plan mode in the main conversation. Research and analysis agents that should never modify files. Provides a hard guarantee of read-only behavior at the permission level, complementing tool restrictions.

Permission Inheritance Rules

Two inheritance rules override the subagent's frontmatter:

  1. bypassPermissions from parent takes precedence: If the parent conversation is running with bypassPermissions, all subagents inherit this mode regardless of their frontmatter setting. This ensures that when a user grants full permissions, subagents do not revert to prompting.
  2. auto from parent overrides frontmatter: If the parent conversation is running with auto mode, the subagent's frontmatter permissionMode is ignored and auto is used. This prevents a subagent from being more restrictive than its parent (which would cause unexpected permission prompts during an otherwise autonomous workflow).

Recommended Permission Strategies

  • Read-only agents: Use permissionMode: plan with tools: Read, Grep, Glob. Double defense: both tool restriction and permission mode prevent writes.
  • Implementation agents: Use permissionMode: acceptEdits with explicit tool allowlists. Allows file changes but keeps shell oversight.
  • Autonomous agents: Use permissionMode: auto with narrow tool allowlists and hooks for critical validation. Trust the agent to act but constrain what it can act on.
  • Never use bypassPermissions unless you have a specific reason. The difference between auto and bypassPermissions is that auto still respects hook-based blocks, while bypassPermissions ignores them. Hooks are your safety net.

3.15 Subagent Design Patterns

After building and operating 26 production agents across 190+ missions, five design patterns have emerged as consistently effective. Each pattern solves a specific architectural problem and can be applied independently or in combination.

Pattern 1: Read-Only Explorer

Problem

You need an agent to investigate a question about your codebase, but you do not want it to modify anything. The investigation might involve reading dozens of files, following import chains, and searching for patterns — all of which would pollute the main conversation's context.

Solution

Create an agent with tools: Read, Grep, Glob and permissionMode: plan. The tool restriction prevents file modification at the capability level. The permission mode provides a secondary guard at the policy level. The agent explores freely, and only its final analysis is returned to the parent.

Markdown 
---
name: architecture-reviewer
description: Analyzes codebase architecture and reports on module
  boundaries, dependency violations, and structural concerns.
tools: Read, Grep, Glob
permissionMode: plan
model: sonnet
maxTurns: 25
color: blue
---

Analyze the codebase architecture. Focus on:
1. Module boundaries (are they clean or leaky?)
2. Dependency direction (do low-level modules import high-level ones?)
3. Circular dependencies
4. Dead code and unused exports

Report findings as a structured document with severity levels.

When to Use

Code reviews, architecture audits, security scanning, compliance checks, documentation generation — any task where the output is a report, not code changes.

Key Properties

  • Safety: Cannot modify files under any circumstances
  • Cost: Use Sonnet or Haiku since read-only analysis does not require Opus-level reasoning
  • maxTurns: Set a reasonable limit (20-30) to prevent runaway exploration

Pattern 2: Focused Executor

Problem

You need an agent to implement a specific, well-defined change. The change should be made autonomously (without permission prompts for every file edit) but within clear boundaries.

Solution

Create an agent with specific write tools and permissionMode: acceptEdits. The agent can modify files freely but shell commands still require approval. Combine with a detailed system prompt that defines exactly what the agent should change and what it should not touch.

Markdown 
---
name: test-writer
description: Writes unit tests for specified modules following
  the project's testing conventions.
tools: Read, Write, Edit, Bash, Grep, Glob
skills:
  - test-generator
permissionMode: acceptEdits
color: green
---

Write unit tests for the specified module.

## Rules
- Follow existing test patterns (check __tests__/ directories)
- Use the project's test framework (detect from package.json)
- Cover happy path, edge cases, and error conditions
- Run tests after writing to verify they pass
- Do NOT modify the source code being tested

When to Use

Test generation, code scaffolding, documentation writing, configuration updates, migration scripts — tasks with a clear scope and bounded impact.

Pattern 3: Model Routing

Problem

Different phases of a workflow require different levels of model capability. Using Opus for everything is expensive. Using Haiku for everything sacrifices quality on complex tasks.

Solution

Define multiple agents for different phases of the workflow, each configured with the appropriate model:

Phase Model Rationale
Triage / Classification Haiku Pattern matching, categorization, and routing do not require deep reasoning. Haiku handles these at 10x lower cost.
Implementation Sonnet Code generation, refactoring, and standard analysis are Sonnet's sweet spot. Good quality at moderate cost.
Architecture / Planning Opus Complex reasoning, multi-factor trade-off analysis, and system-level design require the strongest model.
Verification Opus (or inherit) Verification is as important as planning. A weak verifier misses issues, undermining the entire workflow.

Cost Impact

In our production system, model routing reduced average per-mission cost by approximately 60% compared to using Opus for all subagent calls. The quality impact was negligible: Haiku correctly triaged 97% of tasks, and Sonnet's code generation matched Opus quality on straightforward implementation tasks. The savings come from the volume distribution: triage and exploration account for 40-50% of all subagent invocations, and these run perfectly on Haiku.

Pattern 4: Background Worker

Problem

A long-running task (comprehensive research, full codebase analysis, security audit) would block the main conversation for minutes. You want to start the task and continue working on other things while it runs.

Solution

Set background: true in the agent's frontmatter, or pass run_in_background: true when invoking the Agent tool. The agent runs asynchronously, and the parent receives a notification when it completes.

Markdown 
---
name: full-audit
description: Comprehensive codebase audit covering security,
  performance, and code quality. Runs in the background.
tools: Read, Grep, Glob, Bash
background: true
model: sonnet
maxTurns: 50
color: orange
---

Perform a comprehensive audit of the codebase.

## Audit Areas
1. **Security:** OWASP Top 10, secret exposure, dependency vulns
2. **Performance:** N+1 queries, memory leaks, hot paths
3. **Quality:** Dead code, duplicated logic, naming consistency

Write your findings to audit-report.md when complete.

When to Use

Any task that takes more than 30 seconds and whose results are not needed immediately. Research tasks, full codebase scans, report generation, data processing.

Combining with Worktrees

Background workers pair naturally with worktree isolation. A background agent writing to files in the main checkout would conflict with the user's ongoing work. With isolation: worktree, the background agent works on its own copy.

Pattern 5: Worktree Parallel Execution

Problem

You have multiple independent tasks that could run simultaneously, but they all modify files in the same repository. Sequential execution is safe but slow.

Solution

Combine isolation: worktree with multiple parallel agent invocations. Each agent gets its own copy of the repository and can work independently. After all agents complete, merge their changes sequentially.

Architecture Diagram 
Orchestrator (parent conversation)
    |
    +-- Agent 1 [worktree A] --> Refactor auth module
    |                            Commit: "refactor(auth): extract token validation"
    |
    +-- Agent 2 [worktree B] --> Refactor database layer
    |                            Commit: "refactor(db): add connection pooling"
    |
    +-- Agent 3 [worktree C] --> Update API routes
    |                            Commit: "feat(api): add pagination endpoints"
    |
    (all complete)
    |
    +-- Merge worktree A into main checkout
    +-- Merge worktree B into main checkout
    +-- Merge worktree C into main checkout

Real-World Application: Wave-Based Execution

Our GSD workflow uses this pattern for wave-based parallel execution. Plans are grouped into waves by dependency order. All plans in Wave 1 can execute in parallel (they have no dependencies on each other). When Wave 1 completes, Wave 2 plans (which may depend on Wave 1 outputs) execute in parallel. This maximizes parallelism while respecting dependencies.

Wave Execution Timeline

Time -->

Wave 1: [Agent A: auth] [Agent B: db] [Agent C: config]
         ============== ============= ================
                    |         |              |
                    v         v              v
                (merge)   (merge)        (merge)
                              |
Wave 2:            [Agent D: api routes]  [Agent E: tests]
                   ===================== ===============
                              |                  |
                              v                  v
                           (merge)            (merge)
                                                 |

Key Considerations

  • Merge conflicts: If two agents modify the same file, the sequential merge will surface the conflict. This is by design — it is better to catch conflicts at merge time than to have agents silently overwrite each other's changes.
  • Overhead: Each worktree adds a few seconds of setup time. For 3-6 agents, the total overhead (10-20 seconds) is negligible compared to the time saved by parallel execution (often minutes).
  • Resource usage: Each worktree consumes disk space (a full checkout of the working tree, though the .git directory is shared). For large repositories, monitor disk usage if spawning many agents.

Combining Patterns

These five patterns are composable. A production workflow often combines several:

Example: Full Development Workflow

  1. Read-Only Explorer (Haiku) — Investigates the codebase to understand the current architecture and identify relevant files.
  2. Model Routing (Opus) — Planner agent creates a detailed implementation plan based on the exploration results.
  3. Worktree Parallel (Sonnet) — Three executor agents implement different parts of the plan simultaneously, each in its own worktree.
  4. Read-Only Explorer (Sonnet) — Verifier agent checks that the implementation matches the plan requirements.
  5. Background Worker (Sonnet) — Full test suite runs in the background while the user reviews the changes.

This workflow uses three different models across five agent invocations, with worktree isolation for parallel execution and background processing for non-blocking operations. Total cost is roughly 40% of what it would be using Opus for everything, with minimal quality impact.

Conclusion

Claude Code's custom subagent system represents a fundamentally different approach to multi-agent AI than the framework-based systems (LangChain, CrewAI, AutoGen) covered in other research. Instead of writing orchestration code, defining message schemas, or managing agent lifecycles programmatically, you write markdown files. The runtime handles everything else: discovery, delegation, context isolation, tool permissioning, model routing, and result collection.

Key takeaways from this section:

  • Subagents are markdown files. YAML frontmatter for configuration, markdown body for the system prompt. No code required. Version-controlled alongside your project.
  • Tool control is the safety foundation. Restricting tools creates hard boundaries that no prompt injection or hallucination can bypass. Design tool sets with the principle of least privilege.
  • Model routing is the cost lever. Use Haiku for exploration and triage (40-50% of calls), Sonnet for implementation (30-40%), and Opus for planning and verification (10-20%). This distribution reduces costs by 60-70% with negligible quality impact.
  • Read-only by default. 15 of our 26 production agents have read-only tool sets. Most agent tasks are analysis, not modification.
  • Worktree isolation enables parallelism. Without worktrees, parallel agents cause file conflicts. With worktrees, you get true parallel execution with clean merge semantics.
  • Persistent memory enables learning. Agents with memory improve over time, building up project-specific knowledge that survives across conversations.
  • Hooks enforce invariants. Deterministic hook programs provide safety guarantees that do not depend on LLM instruction following. Use them for critical validations.

As we move into Section 4, we will examine the token economics that underpin all of these design decisions — how context window sizes, model pricing, and token consumption patterns shape the way you architect subagent systems for production use.

## Process
1. Decompose the topic into 3-6 independent research tracks
2. For each track, spawn a specialist agent:
   - Use Agent tool with run_in_background: true
   - Provide a focused research prompt for each track
3. Wait for all agents to complete
4. Use document-builder to synthesize findings
5. Use fact-checker to verify claims in the final document
6. Produce the final research report

## Agent Spawning Rules
- Never spawn more than 6 agents simultaneously
- Each agent gets a focused, specific prompt
- Include expected output format in the prompt
- Set reasonable maxTurns for each agent

CI/CD Agent Definition

A session-only agent defined via CLI flag for use in CI/CD pipelines.

Bash (CI/CD Pipeline) 
# In a GitHub Actions workflow or similar CI environment
CLAUDE_CODE_SUBAGENT_MODEL=haiku \
claude --agents '[{
  "name": "pr-reviewer",
  "description": "Reviews pull request changes for issues",
  "tools": ["Read", "Grep", "Glob"],
  "permissionMode": "plan",
  "maxTurns": 20,
  "systemPrompt": "Review the PR diff. Check for: bugs, security issues, style violations, missing tests. Output as GitHub PR review comments."
}]' \
--prompt "Review the changes in this PR: $(git diff main...HEAD)"

# CLAUDE_CODE_SUBAGENT_MODEL=haiku forces all subagents to use Haiku
# This keeps CI costs predictable regardless of agent definitions
