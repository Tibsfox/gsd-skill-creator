13.1 The Orchestration Challenge

Every AI agent system starts the same way: one agent, one task, one context window. The agent reads a prompt, thinks through the problem, calls some tools, and produces output. It works beautifully for small, self-contained tasks. And then you try to scale it.

Where Single-Agent Execution Breaks

The fundamental constraint of large language model agents is the context window. Even with modern models offering 200K or 1M token windows, there is a hard ceiling on how much state a single agent can hold simultaneously. When your project involves generating 28 research documents across a coordinated series, no single context window can hold the outline, style guide, cross-references, source material, previously generated documents, and the working draft of the current document all at once. The math simply does not work.

But context size is only the most visible constraint. The deeper problem is coordination overhead. A single agent writing 28 documents sequentially will spend an enormous proportion of its token budget on re-establishing context. Each time the context resets—whether from hitting the window limit, from a crash, or from session boundaries—the agent must reload project state, recall what has been completed, figure out what comes next, and re-derive any decisions it made in previous sessions. In our early attempts at sequential execution, we measured coordination overhead consuming 30–40% of total token spend. On a 28-document series, that is the equivalent of throwing away 8–11 full documents worth of compute just on bookkeeping.

Parallel execution solves the throughput problem but introduces an entirely new category of failures: merge conflicts, inconsistent cross-references, duplicated effort, and partial completions that leave the system in an unrecoverable state. You need coordination, but coordination requires shared state, and shared state requires communication channels, and communication channels require protocols, and protocols require enforcement. This is the orchestration problem.

The Core Tension

Orchestration exists to manage a fundamental tension: agents need autonomy to be effective, but autonomous agents need coordination to be coherent. Every pattern in this section is a different approach to resolving that tension for a specific class of problems. None of them replace the others. They compose.

State Management Across Context Resets

Context resets are not exceptional events in agentic systems—they are the normal operating condition. Every agent session has a finite lifetime. Context windows fill. Sessions time out. Crashes happen. Users close terminals. The question is not whether state will be lost, but what the recovery cost will be when it happens.

Traditional software engineering solved this problem decades ago with databases, transaction logs, and checkpoint/restart protocols. But AI agent state is fundamentally different from application state. An agent's most valuable state is not the data it has produced but the reasoning context it has accumulated: the decisions it made and why, the alternatives it rejected, the constraints it discovered. This reasoning context is implicit in the conversation history and cannot be serialized into a JSON file without significant loss.

Our approach accepts this loss as unavoidable and designs around it. Instead of trying to perfectly preserve reasoning context, we externalize decision outcomes into durable state files and design our systems so that any agent can reconstruct enough context from those files to continue the work. The key insight is that decisions need to survive context resets, but the reasoning behind those decisions does not. An agent resuming from a checkpoint does not need to know why a particular document structure was chosen; it needs to know what structure was chosen and what remains to be done.

The Pattern Language

Over the course of 190+ projects, a coherent pattern language has emerged. These patterns did not arrive as a designed system. They grew organically from repeated failures and their solutions, each pattern addressing a specific class of problem that appeared often enough to warrant a named, reusable solution.

13.2 Fleet Mission — Parallel Agent Dispatch

Fleet Mission is the simplest and most frequently used orchestration pattern. A commander agent dispatches N worker agents, each with an independent task, monitors their completion, and merges their results into a unified output. It is the parallel equivalent of a for loop: when tasks are independent and can run simultaneously, Fleet Mission eliminates the serial bottleneck.

Architecture

Fleet Mission Architecture
                    +------------------+
                    |    COMMANDER     |
                    |  (orchestrator)  |
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
        +-----v----+  +-----v----+  +------v---+
        | AGENT #1 |  | AGENT #2 |  | AGENT #N |
        |  task-a   |  |  task-b   |  |  task-n   |
        +-----+----+  +-----+----+  +------+---+
              |              |              |
              +--------------+--------------+
                             |
                    +--------v---------+
                    |    AGGREGATOR    |
                    | merge results    |
                    +------------------+

The commander is responsible for three things: decomposing the work into independent units, dispatching agents with clear instructions, and aggregating the results once all agents have completed. The workers are ephemeral—they spin up, do their work, write their output to a known location, and terminate. The commander monitors completion by checking for the presence and validity of output files.

When to Use Fleet Mission

Independent Research Tasks

When you need analysis of multiple topics where each analysis is self-contained. The 28-document HEL (Higher Education Landscape) series used Fleet Mission with 4 agents running in parallel, each producing complete documents that were later cross-referenced.

Parallel File Generation

When producing multiple output files from a shared specification. The 12-document OPEN (Open Problems in Every Neighborhood) series dispatched agents to generate LaTeX-formatted research documents simultaneously, cutting production time by 75%.

Multi-Aspect Analysis

When a single subject requires analysis from multiple angles. Rather than one agent switching contexts repeatedly, dispatch separate agents for financial analysis, technical review, competitive landscape, and regulatory assessment.

Implementation

Fleet Mission relies on Claude Code's run_in_background capability to launch agents without blocking the commander's context. Each agent receives its task specification as a complete, self-contained prompt that includes all necessary context—no references to shared state, no assumptions about what other agents are doing.

# Fleet Mission dispatch — conceptual interface
fleet-mission dispatch \
  --agents 4 \
  --task-file tasks.json \
  --output-dir ./results/ \
  --aggregate SUMMARY.md \
  --timeout 30m

# tasks.json structure
{
  "fleet_id": "hel-series-2026-04",
  "agents": [
    {
      "id": "agent-1",
      "task": "Research document: Community Colleges",
      "output": "results/hel-community-colleges.html",
      "context_files": ["outline.md", "style-guide.md"]
    },
    {
      "id": "agent-2",
      "task": "Research document: Trade Schools",
      "output": "results/hel-trade-schools.html",
      "context_files": ["outline.md", "style-guide.md"]
    }
    // ... additional agents
  ]
}

Task Sizing

Through repeated iteration across dozens of fleet dispatches, we converged on a reliable rule: 5–6 tasks per teammate is the optimal load. Fewer than 5 and the overhead of agent setup is not amortized. More than 6 and context exhaustion becomes a risk, especially on tasks requiring detailed research or code generation.

This number is not arbitrary. It reflects the practical limits of Claude Code agent sessions. Each agent needs context for its task specification, any reference materials, its working state, and its accumulated output. At 5–6 tasks, the total context stays comfortably within the usable range of the context window, leaving headroom for the agent to reason and iterate without hitting compaction boundaries prematurely.

Fleet Mission Sizing Guidelines

4 agents for synthesis tasks (research, analysis, document generation). Synthesis requires more context per task and benefits from deeper reasoning. Four agents balance throughput with quality.

3 agents for pure documentation runs (straightforward content generation from existing sources). Documentation tasks are more mechanical and benefit from having fewer agents to coordinate.

5–6 tasks per agent regardless of agent count. This is the load ceiling. Beyond it, quality degrades measurably.

Error Handling and Partial Failure Recovery

In a fleet of 4 agents running 5 tasks each, individual task failures are inevitable. The question is not whether failures will occur but how the system responds when they do.

Fleet Mission implements a partial completion model. When a task fails, its output slot is marked as incomplete but the fleet continues. The commander tracks which tasks succeeded and which failed. After all agents complete (or time out), the commander produces the aggregated output from successful tasks and a failure report for unsuccessful ones.

This approach is critical for production use. In the HEL series, agent #3 failed on document 14 (a particularly complex topic requiring data from multiple paywalled sources). Rather than failing the entire 28-document fleet, the system completed the other 27 documents, flagged document 14 for manual retry, and produced the aggregated series index with a placeholder for the missing document. The manual retry succeeded on the second attempt with revised source instructions.

Failure Mode Detection Recovery Strategy
Agent timeout No output file after deadline Mark incomplete, retry with simpler prompt
Partial output Output file exists but fails validation Preserve partial, dispatch repair agent
Complete failure Agent crashes, no output Log context, retry with fresh agent
Quality failure Output passes validation but fails review Queue for second-pass revision agent

13.3 Gastown Convoy Model — Multi-Agent Orchestration

If Fleet Mission is a parallel for loop, the Gastown Convoy Model is a full operating system for multi-agent work. Named after Gastown, Vancouver's oldest neighborhood—a place where steam clocks, cobblestone streets, and modern restaurants coexist in carefully coordinated density—the Gastown model manages the full lifecycle of large-scale agent operations: creation, dispatch, monitoring, recovery, and retirement.

The name is deliberate. Gastown the neighborhood works because decades of careful coordination keep its disparate elements functioning together. Remove the coordination and you get gridlock, noise, and decay. The same is true of multi-agent systems. The agents themselves are simple. The orchestration is everything.

Architecture

Gastown is composed of four named components, each with a specific role in the orchestration lifecycle. These components are implemented as separate skills that compose into the full system.

Gastown Convoy Model — Component Architecture
  +=====================================================+
  |                  MAYOR-COORDINATOR                  |
  |  Convoy creation | Agent allocation | Progress      |
  +========+====================+===========+==========+
           |                    |           |
           v                    v           v
  +-----------------+  +----------------+  +------------------+
  | SLING-DISPATCH  |  | WITNESS-       |  | BEADS-STATE      |
  | 7-stage routing |  | OBSERVER       |  | Crash-recoverable|
  | pipeline        |  | Health monitor |  | persistence      |
  +---------+-------+  | Stall detect   |  +------------------+
            |          | Nudge sending  |
            v          +--------+-------+
  +-----------------+           |
  | POLECAT-WORKER  |           |
  | Ephemeral       |<----------+  (health checks)
  | autonomous work |
  +-----------------+

Mayor-Coordinator

The mayor-coordinator is the central nervous system of Gastown. It creates convoys (logical groupings of related work items), allocates agents to work items within each convoy, tracks progress across all active convoys, and handles escalation when things go wrong.

A convoy is not just a batch of tasks. It is a coherent unit of work with dependencies, ordering constraints, and shared context. When the PNW Research Series needed 50+ projects generated, the mayor-coordinator organized them into convoys by Rosetta cluster: Ecology projects in one convoy, Electronics in another, Music in a third. This clustering meant that agents within a convoy shared domain context, reducing the per-task context loading cost.

# Mayor-coordinator — convoy creation
mayor-coordinator create-convoy \
  --name "ecology-cluster" \
  --work-items ecology-tasks.json \
  --max-agents 4 \
  --token-budget 2000000 \
  --timeout 2h

# Convoy state file (.gastown/convoys/ecology-cluster.json)
{
  "convoy_id": "ecology-cluster",
  "status": "active",
  "created": "2026-03-31T14:00:00Z",
  "agents": {
    "agent-eco-1": { "status": "working", "current_task": "forest-ecology" },
    "agent-eco-2": { "status": "working", "current_task": "marine-biology" },
    "agent-eco-3": { "status": "idle",    "completed": 3 },
    "agent-eco-4": { "status": "working", "current_task": "watershed-analysis" }
  },
  "progress": { "total": 12, "completed": 7, "failed": 0, "in_progress": 3 },
  "token_usage": { "budget": 2000000, "consumed": 1340000, "remaining": 660000 }
}

Sling Dispatch

Sling dispatch is the work routing pipeline that connects the mayor-coordinator to the polecat workers. It implements a 7-stage routing process that ensures every work item is properly prepared, validated, and dispatched with a complete audit trail. (Sling dispatch is detailed fully in Section 13.5.)

Witness-Observer

The witness-observer runs a continuous patrol loop, checking on the health of all active agents in the convoy. It detects stalled agents (agents that have not produced output within a configurable threshold), sends nudge signals to stalled agents, and escalates to the mayor-coordinator when nudging fails to restore progress.

Stall detection is surprisingly important in practice. Agents do not crash cleanly—they often enter degenerate states where they are still technically running but making no progress. Common causes include: infinite tool call loops, context windows filling with error messages, and agents getting stuck in revision cycles where each revision introduces new problems. The witness-observer catches these cases by monitoring output file modification times and agent heartbeat signals.

Polecat-Worker

The polecat-worker is the ephemeral execution unit. Each polecat spins up, receives its work item from sling dispatch, enters autonomous execution mode via GUPP (see Section 13.4), completes its task, writes its output, and terminates. Polecats are stateless by design—all persistent state lives in beads-state files managed by the mayor-coordinator.

The name "polecat" reflects the pattern's characteristics: small, fast, independent, and not particularly concerned with what anyone else is doing. A polecat worker does not coordinate with other polecats. It does not know about the convoy it belongs to. It receives a task, does the task, and exits. All coordination happens above it, in the mayor-coordinator and sling dispatch layers.

Convoy Lifecycle

CREATE → POPULATE → DISPATCH → MONITOR → COMPLETE → ARCHIVE

Create: The mayor-coordinator creates a convoy definition with a name, token budget, agent limit, and timeout. The convoy state file is written to .gastown/convoys/.

Populate: Work items are loaded into the convoy. Each work item includes a task specification, required context files, expected output location, and validation criteria. The mayor-coordinator checks that the total estimated cost fits within the token budget before accepting the work items.

Dispatch: The mayor-coordinator feeds work items into sling dispatch, which routes them to available polecat workers. Dispatch is not all-at-once; the sling manages a single-active-work-item model per agent to prevent overloading.

Monitor: The witness-observer runs patrol loops at configurable intervals. Each patrol checks agent health, updates the convoy progress tracker, and sends nudge signals to stalled agents. The mayor-coordinator checks for convoy-level conditions: budget exhaustion, timeout, or all work items completed.

Complete: When all work items are done (or the convoy times out / exhausts budget), the mayor-coordinator transitions the convoy to the complete state. A completion report is generated listing all successful tasks, failed tasks, and resource usage.

Archive: The convoy state is moved from the active directory to the archive. Output files are left in place for consumption by downstream processes. The archive serves as an audit trail for debugging and post-mortem analysis.

Proven Scale: The PNW Research Series

The Gastown model was proven at its largest scale during the PNW Research Series, a 190+ project research program spanning 13 Rosetta clusters. The series produced over 2,439 files totaling more than 600,000 lines of content across topics ranging from ecology and electronics to music theory and space science.

At peak execution, the system ran 4 parallel agents per convoy across multiple convoys, with the mayor-coordinator managing 50+ active work items simultaneously. The witness-observer detected and recovered from 12 agent stalls during the full run. Total completion rate across all convoys was 97.3%, with the remaining 2.7% handled by manual retry after automated attempts failed.

Production Observation

The Gastown model's greatest strength is not its throughput but its recoverability. When Agent #2 stalled on the Electronics convoy at 2:00 AM during an overnight run, the witness-observer detected the stall within 3 minutes, sent a nudge, and when the nudge failed, escalated to the mayor-coordinator which redistributed the stalled work to Agent #4 (which had just completed its queue). The entire recovery happened autonomously. No human intervention was required. The overnight run completed all 14 Electronics projects by morning.

13.4 GUPP Propulsion — Polled-to-Proactive Execution

GUPP stands for Get Up and Push Protocol. It addresses a fundamental limitation of AI assistant runtimes: they are inherently reactive. An AI agent waits for input, processes it, produces output, and then waits again. Left to its default behavior, an agent will complete one task and then sit idle until the next instruction arrives. In a multi-agent system, this means your expensive, capable agents spend most of their time doing nothing.

The Reactive Problem

Consider a typical agent session. The user dispatches a task. The agent completes it in 3 minutes. The agent then waits. If the user is managing 4 agents and checking them in round-robin, Agent #1 might wait 10–15 minutes before receiving its next task. In a 2-hour session, each agent might complete only 8 tasks despite being capable of 24+. Two-thirds of the compute capacity is wasted on waiting.

The root cause is that AI runtimes are pull-based. The agent pulls work from the user. In production orchestration, we need push-based execution: the system pushes work to agents as soon as they are ready, with zero idle time between tasks.

The GUPP Solution: Interrupt Controller

GUPP converts polled execution to proactive execution by implementing an interrupt controller pattern. Instead of returning to idle after completing a task, the GUPP-enabled agent checks a hook-persistence file for the next work item. If one exists, the agent picks it up and begins execution immediately, without waiting for user input.

GUPP Interrupt Controller Flow
  +------------------+
  |  Agent completes  |
  |  current task     |
  +--------+---------+
           |
           v
  +------------------+     YES    +------------------+
  | Check hook file  +---------->| Load next work   |
  | for pending work |            | item from hook   |
  +--------+---------+            +--------+---------+
           |                               |
           | NO                            v
           v                      +------------------+
  +------------------+            | Enter autonomous |
  | Dead Man's Switch|            | execution mode   |
  | - report idle    |            +--------+---------+
  | - await dispatch |                     |
  +------------------+                     v
                                  +------------------+
                                  | Execute task     |
                                  | Write output     |
                                  +--------+---------+
                                           |
                                           v
                                  (loop back to top)

Per-Runtime Strategies

Different AI runtimes have different capabilities, constraints, and hook mechanisms. GUPP implements per-runtime strategies that achieve the same "always be working" behavior through runtime-appropriate mechanisms.

Runtime Hook Mechanism GUPP Strategy
Claude Code PostToolUse hooks, settings.json Hook-persistence file checked after each tool call. CLAUDE.md instructions for autonomous behavior. Subagent dispatch via Agent tool.
Codex Task queue, environment persistence Task queue integration. Environment variables for work item location. Sequential task processing via queue drain.
Gemini Function calling, conversation context Function call chain that includes next-task retrieval. System instruction modifications for proactive behavior.
Cursor .cursorrules, composer context Rules file injection with work queue. Composer instructions for continuous execution. File-based task signaling.

Configurable Thresholds

GUPP includes configurable thresholds that control the transition between proactive and idle states:

# GUPP configuration
{
  "gupp": {
    "enabled": true,
    "check_interval_ms": 5000,
    "idle_timeout_s": 300,
    "max_consecutive_tasks": 20,
    "dead_mans_switch": {
      "enabled": true,
      "heartbeat_interval_s": 60,
      "stale_threshold_s": 180
    },
    "hook_file": ".gupp/pending-work.json",
    "runtime": "claude-code"
  }
}

The Dead Man's Switch is a safety mechanism. If an agent's heartbeat file has not been updated within the stale threshold, the system assumes the agent has crashed or stalled and makes its queued work available for redistribution. This prevents the silent failure mode where a crashed agent holds work items hostage indefinitely.

The "Always Be Working" Philosophy

GUPP embodies a philosophical position about how AI agents should behave in production systems: idle agents are wasted capacity. When you are paying per-token for compute and your agents are sitting idle between tasks, you are paying for nothing. GUPP eliminates the idle gap by making the agent itself responsible for seeking its next unit of work.

This is a departure from most agent frameworks, which treat agents as passive tools that respond to explicit invocations. GUPP treats agents as active workers in a production line. They do not wait to be told what to do. They check the work queue. If there is work, they do it. If there is no work, they signal that they are available. The orchestrator (mayor-coordinator) manages the queue. The agent manages itself.

GUPP in Practice

In the PNW Research Series, GUPP reduced agent idle time from an estimated 40% (under manual dispatch) to under 5% (under automated GUPP-driven dispatch). For a 4-agent fleet running a 6-hour mission, this recovered approximately 5.6 agent-hours of productive work that would otherwise have been lost to waiting.

13.5 Sling Dispatch — Work Routing Pipeline

Sling Dispatch is the routing pipeline that connects work items to available agents. It is the delivery mechanism for the entire Gastown system. Every piece of work that flows from the mayor-coordinator to a polecat worker passes through sling dispatch's 7-stage pipeline.

The 7-Stage Pipeline

FETCH → ALLOCATE → PREPARE → HOOK → DISPATCH → CONFIRM → ARCHIVE

1. Fetch

Retrieve the next work item from the convoy's work queue. The fetch stage respects ordering constraints: if work item B depends on work item A, B will not be fetched until A is complete. Fetch also checks that the convoy's token budget has remaining capacity for the estimated cost of the work item.

2. Allocate

Select an available agent from the convoy's agent pool. Allocation considers agent load (current active tasks), agent history (agents that previously failed on similar tasks are deprioritized), and agent specialization (if configured). The single-active-work-item model means each agent can only be allocated one task at a time.

3. Prepare

Assemble the complete context package for the work item. This includes the task specification, any context files referenced by the work item, the output format requirements, and the validation criteria. The prepare stage resolves file paths, verifies that referenced files exist, and packages everything into a self-contained instruction set.

4. Hook

Write the prepared instruction to the target agent's hook-persistence file (the file monitored by GUPP). This is an atomic file operation: the hook file is written to a temporary location and then renamed to the target path, ensuring that partial writes never leave the hook file in a corrupt state.

5. Dispatch

Signal the target agent that work is available. For Claude Code agents, this is implicit (the GUPP check will find the hook file). For other runtimes, dispatch may involve an explicit API call or message. The dispatch stage records the dispatch timestamp and the agent assignment in the convoy state.

6. Confirm

Verify that the target agent has acknowledged the work item. Confirmation is detected by monitoring the agent's heartbeat or checking for the presence of a work-in-progress marker file. If confirmation does not arrive within a configurable timeout, the sling retries dispatch or escalates to the mayor-coordinator.

7. Archive

Once the work item is confirmed as dispatched, the dispatch record is archived for audit purposes. The archive includes the full instruction package, the agent assignment, the timestamps for each stage, and the outcome. This archive is essential for post-mortem debugging and for training the system's allocation heuristics.

Single-Active-Work-Item Model

Sling dispatch enforces a strict single-active-work-item model: each agent has at most one active task at any time. This constraint is deliberate. AI agents do not context-switch well. Unlike human developers who can hold multiple projects in their heads and switch between them, an LLM agent's "memory" is its context window. Giving an agent two tasks simultaneously means both tasks share the context window, reducing the effective context available for each.

The single-active-work-item model means agents complete their current task before receiving the next one. This maximizes the quality of each individual task at the cost of slightly higher dispatch overhead. In practice, the dispatch overhead is negligible compared to the quality loss from context sharing.

File-Based Dispatch with Atomic Operations

All sling dispatch operations use the filesystem as the coordination layer. This is a deliberate architectural choice. File-based dispatch is:

  • Observable: Any process can read the dispatch state by listing files in known directories. No special client libraries or API connections required.
  • Durable: Files survive process crashes. A half-dispatched work item can be detected and recovered by inspecting file states.
  • Atomic: File rename operations on POSIX systems are atomic. The write-to-temp-then-rename pattern ensures that no process ever reads a partially-written dispatch file.
  • Debuggable: When something goes wrong, the entire dispatch history is a directory of JSON files that can be inspected with standard tools.

Failure Modes and Recovery

Stage Failure Mode Recovery
Fetch Empty queue or budget exhausted Wait for queue replenishment or convoy completion
Allocate No available agents Queue the work item; dispatch when an agent frees up
Prepare Missing context files Fail the work item with diagnostic; do not dispatch incomplete instructions
Hook Filesystem write failure Retry with exponential backoff; escalate if persistent
Dispatch Agent unreachable Mark agent as unhealthy; reallocate work item
Confirm No acknowledgment within timeout Retry dispatch; if retry fails, reallocate to different agent
Archive Archive write failure Non-critical; log warning but do not block pipeline

13.6 Communication Patterns

Multi-agent systems need inter-agent communication. Agents need to share status, coordinate decisions, signal completion, and request assistance. The challenge is that AI agents do not have sockets, message queues, or shared memory. They have filesystems and tool calls. Three communication patterns have emerged from production use, each optimized for a different temporal and durability requirement.

Mail-Async — Durable Asynchronous Messaging

Mail-async is a write-once, read-many filesystem mail system. It is designed for communication that must survive across context resets, session boundaries, and even multi-day mission timelines. When Agent A needs to tell Agent B something that Agent B might not read for hours or days, mail-async is the correct channel.

Architecture

Mail-async uses a directory-based mailbox structure. Each agent has an inbox directory. Sending a message writes a timestamped envelope file to the recipient's inbox. The envelope contains the message payload, sender identity, timestamp, and optional priority and expiry fields.

# Mail-async directory structure
.mail/
  agent-eco-1/
    inbox/
      2026-04-01T14-30-00Z_from-mayor.json
      2026-04-01T15-45-00Z_from-agent-eco-3.json
    sent/
      2026-04-01T14-00-00Z_to-mayor.json

# Envelope format
{
  "envelope_id": "msg-2026-04-01-143000-mayor",
  "from": "mayor-coordinator",
  "to": "agent-eco-1",
  "timestamp": "2026-04-01T14:30:00Z",
  "priority": "normal",
  "expires": null,
  "subject": "Cross-reference update",
  "body": "Document ECO-07 has been completed. Update your
           cross-references to include the watershed data
           from ECO-07 section 3.2.",
  "attachments": ["results/eco-07-watershed-data.json"]
}

Properties

Write-Once, Read-Many

Messages are immutable after writing. No process ever modifies a sent message. This eliminates an entire class of race conditions and makes the mail history a reliable audit trail.

Persistence Across Resets

Because messages are files on disk, they survive context resets, session timeouts, and process crashes. An agent resuming from a crash can check its inbox and find all messages that arrived while it was down.

When to Use

Cross-session communication where the recipient may not be active. Audit trails that must survive the entire mission lifecycle. Complex instructions that need to be referenced multiple times. Asynchronous coordination between convoys operating on different timelines.

Nudge-Sync — Synchronous Immediate Signaling

Nudge-sync is the opposite of mail-async: it is designed for immediate, ephemeral state signaling. Where mail-async prioritizes durability, nudge-sync prioritizes latency. The pattern uses a latest-wins single-file approach: each agent has a single nudge file, and writing to it overwrites whatever was there before.

Architecture

Each agent monitors a single file at a known path. Any process that needs to signal the agent writes to that file. The agent checks the file at its GUPP polling interval. Because the file is overwritten on each write, only the most recent signal matters. This makes nudge-sync ideal for state changes ("your convoy status changed"), health checks ("are you alive?"), and coordination signals ("pause your current work").

# Nudge file location
.nudge/agent-eco-1.json

# Nudge payload (latest-wins — overwrites previous)
{
  "nudge_type": "health_check",
  "from": "witness-observer",
  "timestamp": "2026-04-01T14:30:00Z",
  "payload": {
    "action": "report_status",
    "respond_to": ".nudge/witness-response.json"
  }
}

# Agent response (written to respond_to path)
{
  "agent": "agent-eco-1",
  "status": "working",
  "current_task": "forest-ecology",
  "progress": "section 4 of 7",
  "timestamp": "2026-04-01T14:30:02Z"
}

Near-Zero Latency

Nudge-sync achieves near-zero signaling latency because it requires no queue processing, no message routing, and no delivery confirmation. The sender writes a file; the receiver reads the file at its next poll. With GUPP's 5-second check interval, the maximum latency is 5 seconds. In practice, because GUPP checks are interleaved with tool calls, the effective latency is often under 2 seconds.

When to Use

  • State changes: Notifying an agent that its convoy status has changed (e.g., budget reduced, priority increased).
  • Health checks: The witness-observer checking whether an agent is responsive.
  • Immediate coordination: Telling an agent to pause work because a dependency has changed.
  • NOT for durable messages: If the message needs to survive a context reset, use mail-async instead.

SendMessage — Built-in Agent Teams Messaging

The third communication channel is not one we built but one we adopted: Claude Code's native sendMessage function for agent teams. When multiple agents are running within a single Claude Code session as an official team (with defined roles and the Agent tool), sendMessage provides direct teammate-to-teammate communication within the session context.

Properties

sendMessage is session-scoped: messages exist only within the active session and do not persist. It is synchronous: the sending agent blocks until the receiving agent processes the message and responds. It is typed: the Claude Code SDK defines the message format, ensuring consistency. These properties make it the best choice for real-time coordination within a team session, but unsuitable for cross-session or cross-process communication.

When to Use

  • Within official Claude Code agent team sessions where all agents share a session context.
  • When immediate, synchronous response is needed and both agents are active in the same session.
  • NOT for cross-session communication (messages do not persist).
  • NOT for cross-process communication (sendMessage is scoped to the Claude Code session).

13.7 Refinery Merge — Deterministic Merge Queue

When multiple agents produce code or content simultaneously, their outputs must eventually converge into a single coherent artifact. Git merge is the standard tool for this, but parallel agent development introduces merge conflicts at a rate that makes manual resolution impractical. Refinery Merge solves this by implementing a sequential, deterministic merge queue that processes merge requests one at a time, ensuring that each merge is clean before the next begins.

Architecture

Refinery Merge operates as a DMA (Direct Memory Access) pattern: merge requests are queued and processed in order, with each request having exclusive access to the target branch during its processing window. No two merges ever execute simultaneously. This eliminates merge conflicts caused by concurrent modifications.

Refinery Merge Queue
  Agent #1 ──┐
  Agent #2 ──┤     +------------------+     +------------------+
  Agent #3 ──┼────>| MERGE QUEUE      |────>| MERGE PROCESSOR  |
  Agent #4 ──┤     | (FIFO ordering)  |     | 1. checkout      |
  Agent #N ──┘     +------------------+     | 2. rebase        |
                                            | 3. verify        |
                                            | 4. merge         |
                                            | 5. push          |
                                            +------------------+
                                                     |
                                                     v
                                            +------------------+
                                            |   main branch    |
                                            | (always clean)   |
                                            +------------------+

The Merge Process

Each merge request passes through a 5-step process:

  1. Checkout: The merge processor checks out the agent's feature branch and the current state of the target branch (typically main).
  2. Rebase: The feature branch is rebased onto the latest target branch. This applies the agent's changes on top of whatever other agents have already merged. If the rebase produces conflicts, the merge is rejected and sent back to the originating agent with conflict details.
  3. Verify: After rebasing, the merge processor runs validation checks: build succeeds, tests pass, lint is clean. This ensures that the rebase did not introduce subtle breakage even if it completed without explicit conflicts.
  4. Merge: The rebased and verified branch is merged into the target branch using a fast-forward merge. The fast-forward strategy is intentional: it produces a linear history that is easier to bisect and debug.
  5. Push: The updated target branch is pushed to the remote. The merge processor holds a lock on the target branch during this step to prevent race conditions with external pushes.

Why Sequential Processing Matters

The natural instinct is to parallelize merges. After all, if we are parallelizing the work, why serialize the integration? The answer is that merge conflicts are inherently order-dependent. Two branches that merge cleanly against the current main may conflict with each other. By processing merges sequentially, each merge sees the most current state of main, including all previously merged changes. This means that conflicts are detected at rebase time rather than at merge time, and they are detected by the agent that caused them rather than by a downstream agent that cannot resolve them.

In practice, sequential merge processing is fast enough that it does not become a bottleneck. Individual merges typically complete in 30–90 seconds. Even with a queue of 10 pending merges, the total queue drain time is under 15 minutes. The time saved by avoiding manual conflict resolution far exceeds the time spent waiting in the queue.

Integration with Done-Retirement

Refinery Merge is tightly coupled with the Done-Retirement pipeline (Section 13.8). When an agent completes its work, it submits a merge request to the refinery queue as the second stage of the retirement pipeline. The merge must succeed before the agent's work is considered complete. If the merge fails, the retirement pipeline halts and the agent is given the conflict details to resolve before retrying.

13.8 Done Retirement — Completion Pipeline

Completing work is harder than doing work. In agent systems, the "last mile" of a task—validating output, committing changes, pushing to the remote, notifying the coordinator, cleaning up temporary state, and shutting down cleanly—is where most silent failures occur. Done Retirement is a 7-stage pipeline that makes completion deterministic and observable.

The 7-Stage Pipeline

VALIDATE → COMMIT → PUSH → SUBMIT → NOTIFY → CLEANUP → TERMINATE
Stage Action Entry Criteria Exit Criteria
Validate Run output validation against task spec Agent reports task complete All validation checks pass
Commit Stage and commit output files Validation passed Clean git commit with conventional message
Push Push branch to remote Commit succeeded Remote branch updated
Submit Submit merge request to refinery queue Push succeeded Merge request accepted into queue
Notify Send completion notification to mayor Submit succeeded Mayor acknowledges completion
Cleanup Remove temporary files, clear hook persistence Notify succeeded Working directory clean
Terminate Agent exits cleanly Cleanup succeeded Agent process ended

Partial Completion Recovery

Each stage writes its completion state to a progress file before advancing to the next stage. If the agent crashes mid-pipeline, the recovery process reads the progress file and resumes from the last completed stage. This means a crash during the PUSH stage does not re-run VALIDATE and COMMIT—it picks up at PUSH and continues forward.

# Done-retirement progress file
# .done-retirement/agent-eco-1-task-forest-ecology.json
{
  "task_id": "forest-ecology",
  "agent_id": "agent-eco-1",
  "pipeline_state": "push",
  "stages": {
    "validate": { "status": "complete", "timestamp": "2026-04-01T16:00:00Z" },
    "commit":   { "status": "complete", "timestamp": "2026-04-01T16:00:15Z",
                  "commit_sha": "a1b2c3d" },
    "push":     { "status": "in_progress", "started": "2026-04-01T16:00:20Z" },
    "submit":   { "status": "pending" },
    "notify":   { "status": "pending" },
    "cleanup":  { "status": "pending" },
    "terminate":{ "status": "pending" }
  }
}

Atomic Operations

Each stage is designed to be atomic: it either completes fully or fails without leaving partial state. The COMMIT stage uses git commit which is inherently atomic. The PUSH stage uses git push which either succeeds completely or fails completely. The CLEANUP stage deletes files one at a time but tracks each deletion, so a crash during cleanup can resume by checking which files still exist.

This atomicity is critical because the Done-Retirement pipeline often runs in degraded conditions. By the time an agent reaches the retirement pipeline, its context window is typically near capacity (the agent has been doing work and accumulating context throughout the task). Failures are more likely in this state, making recovery capability especially important.

Real Example: Research Project Completion

When Agent #2 completed the Marine Biology research project (ECO-09) in the Ecology convoy:

  1. Validate: Checked that the output file contained all 7 required sections, that section word counts met minimums, that all cross-references resolved, and that the HTML validated against the template.
  2. Commit: Staged the output file and committed with message feat(research): ECO-09 marine biology.
  3. Push: Pushed the agent's branch to the remote.
  4. Submit: Submitted a merge request to the refinery queue.
  5. Notify: Sent a mail-async message to the mayor-coordinator reporting completion with the commit SHA and a summary.
  6. Cleanup: Removed the hook-persistence file, cleared the GUPP pending-work entry, deleted draft files from the working directory.
  7. Terminate: Agent exited cleanly. The witness-observer detected the exit and updated the convoy state to show Agent #2 as available for reallocation.

13.9 Beads State — Crash-Recoverable Persistence

Beads State is the persistence layer for the entire Gastown system. It provides git-friendly, crash-recoverable state management for agent identities, work item tracking, and convoy progress. The name reflects the design: each state update is a bead on a string, forming an ordered sequence that can be replayed from any point.

Agent Identity Persistence

In a system where agents are ephemeral (polecat workers spin up and terminate for each task), there is still a need for persistent identity. The same logical agent—"agent-eco-1"—may be instantiated across multiple sessions over the lifetime of a convoy. Beads State maintains the identity mapping: which logical agent is assigned to which work items, what its completion history looks like, and where its output should be written.

# Beads state — agent identity file
# .gastown/beads/agents/agent-eco-1.json
{
  "agent_id": "agent-eco-1",
  "convoy_id": "ecology-cluster",
  "created": "2026-04-01T14:00:00Z",
  "session_count": 4,
  "current_session": null,
  "completed_tasks": [
    "forest-ecology",
    "alpine-meadows",
    "riparian-zones"
  ],
  "failed_tasks": [],
  "total_tokens_consumed": 485000,
  "heartbeat": "2026-04-01T18:30:00Z",
  "state": "idle"
}

Work Item Tracking

Every work item in the system has a beads state file that records its lifecycle: creation, assignment, progress checkpoints, completion, and archival. The state file is updated at each lifecycle transition using atomic write operations (write-to-temp, rename-to-target).

Checkpoint and Restore

For long-running tasks, agents can write checkpoint data to their beads state. A checkpoint captures the agent's progress at a specific point: which sections of a document have been written, which data sources have been processed, what intermediate results have been computed. If the agent crashes and is restarted, it reads the checkpoint and resumes from that point rather than starting the entire task over.

Checkpoints are especially valuable for research tasks that involve web fetches, database queries, or other operations whose results cannot be reproduced exactly. By checkpointing after expensive operations, the system avoids repeating them on restart.

# .gastown/beads/work/forest-ecology.json
{
  "work_id": "forest-ecology",
  "convoy_id": "ecology-cluster",
  "assigned_to": "agent-eco-1",
  "status": "in_progress",
  "checkpoint": {
    "timestamp": "2026-04-01T15:30:00Z",
    "sections_completed": [1, 2, 3, 4],
    "sections_remaining": [5, 6, 7],
    "intermediate_data": {
      "species_count": 342,
      "data_sources_processed": ["usda-plants", "gbif", "inat"],
      "draft_word_count": 4200
    }
  },
  "history": [
    { "event": "created",    "timestamp": "2026-04-01T14:10:00Z" },
    { "event": "assigned",   "timestamp": "2026-04-01T14:10:05Z", "agent": "agent-eco-1" },
    { "event": "started",    "timestamp": "2026-04-01T14:15:00Z" },
    { "event": "checkpoint", "timestamp": "2026-04-01T15:30:00Z", "sections": 4 }
  ]
}

Git Friendliness

All beads state files are JSON, which diffs cleanly in git. State files are organized in a directory hierarchy that mirrors the convoy structure (.gastown/beads/agents/, .gastown/beads/work/, .gastown/beads/convoys/). This means the entire orchestration state can be committed to git, providing version-controlled history of the orchestration itself.

In practice, we commit beads state at convoy lifecycle boundaries (creation, completion, archival) but not at every state update. The intermediate state updates happen at high frequency (every few minutes) and committing each one would create excessive git history. The atomic write operations ensure that the on-disk state is always consistent even without git commits at every update.

Recovery from Mid-Operation Crashes

The recovery protocol when the system restarts after a crash:

  1. Scan convoy directories for active convoys (status != "archived").
  2. For each active convoy, scan agent state files. Mark any agent whose heartbeat is older than the Dead Man's Switch threshold as "presumed_crashed".
  3. For each presumed-crashed agent, scan its assigned work items. Items with checkpoints are resumed from the checkpoint. Items without checkpoints are returned to the convoy work queue for reassignment.
  4. Restart the witness-observer with the updated convoy state. The observer begins its patrol loop and detects any agents that are actually still running (their heartbeat will update, changing them from "presumed_crashed" to "healthy").
  5. Resume sling dispatch for any queued work items. The dispatch pipeline checks the allocate stage to see if the queue has capacity for new assignments.

This recovery protocol has been tested extensively across the PNW Research Series. The most extreme recovery was a system crash during an overnight run that left 3 convoys with 8 active agents in mid-task. On restart, the recovery protocol correctly identified 6 tasks that had checkpoints, resumed them without rework, and reassigned the 2 tasks without checkpoints to fresh agents. Total recovery time was under 4 minutes. No work was lost.

13.10 Runtime HAL — Multi-Runtime Abstraction

Runtime HAL (Hardware Abstraction Layer) addresses a practical problem: the orchestration patterns described in this section were developed on Claude Code, but they should work on any AI runtime capable of tool use and file operations. Different runtimes have different capabilities, different tool interfaces, different context window sizes, and different hook mechanisms. Runtime HAL provides an abstraction layer that detects the active runtime and adapts skill behavior accordingly.

The Problem

Consider a skill like GUPP that needs to check a hook file for pending work. On Claude Code, this is straightforward: the skill reads a file using the Read tool and checks its contents. On Codex, the same operation requires a different tool call. On Gemini, it might require a function call with different parameter names. On Cursor, it requires reading from a different location with different conventions.

Without Runtime HAL, every skill would need per-runtime conditional logic: "if Claude Code, do X; if Codex, do Y; if Gemini, do Z." This creates a maintenance nightmare as the number of skills multiplied by the number of runtimes produces a combinatorial explosion of code paths.

Capability Detection

Runtime HAL detects the active runtime through a cascade of checks:

  1. Environment variables: Claude Code sets CLAUDE_CODE; Codex sets CODEX_SANDBOX; each runtime has telltale environment markers.
  2. File markers: The presence of .claude/ indicates Claude Code. .cursor/ indicates Cursor. Configuration files in known locations indicate specific runtimes.
  3. Process introspection: Checking the parent process name or the available tool set can disambiguate when environment variables are ambiguous.

Once the runtime is detected, HAL loads a capability map that describes what the runtime can do: which tools are available, what their parameter signatures look like, what file paths the runtime uses for configuration, and what the context window limits are.

Feature Negotiation

Not all runtimes support all features. Claude Code has subagent dispatch via the Agent tool; Cursor does not. Codex has persistent environment; Gemini does not. Runtime HAL handles this through feature negotiation: skills query HAL for capability support before attempting to use runtime-specific features.

Portable Skill Execution

With Runtime HAL in place, skills are written against the HAL API rather than against runtime-specific tool names. When a skill needs to read a file, it calls hal.readFile(path) rather than using the Claude Code Read tool directly. HAL translates the abstract operation into the runtime-specific implementation. This means a skill written and tested on Claude Code works on Codex without modification (assuming the runtime supports the required capabilities).

The four runtimes currently supported represent the major agent development platforms as of April 2026. As new runtimes emerge, they can be added to HAL with a new capability map and detector, without modifying any existing skills.

13.11 Token Budget Management

Multi-agent orchestration without cost control is a fast path to unexpected bills. When you dispatch 4 agents to run 5 tasks each, the total token consumption is not 4x a single agent—it is potentially much higher, because each agent loads its own context, generates its own reasoning tokens, and may retry failed operations. Token Budget Management provides hard limits with pre-execution checking, per-convoy token allocation, and real-time budget tracking.

Hard Limits with Pre-Execution Checking

Every convoy has a token budget set at creation time. Before any work item is dispatched, the sling dispatch pipeline (at the FETCH stage) checks that the convoy's remaining budget can accommodate the estimated cost of the work item. If the budget is insufficient, the work item stays in the queue and the convoy enters a "budget_exhausted" state that triggers escalation to the mayor-coordinator.

Cost estimation uses a model based on historical consumption data. For research document generation, the average token cost is approximately 50,000 tokens per document (including input context, reasoning, tool calls, and output). For code generation tasks, the average is higher: approximately 80,000 tokens per file due to the iterative test-fix cycles. These averages include a 20% safety margin to account for variance.

Per-Convoy Token Allocation

The mayor-coordinator allocates token budgets at the convoy level, not the individual agent level. This allows the system to redistribute budget dynamically. If Agent #1 completes its tasks under budget, the surplus becomes available for Agent #3 which might be running over budget on a complex task. Per-convoy allocation prevents the failure mode where one agent exhausts its individual budget on a hard task while other agents have unused budget sitting idle.

Budget Tracking Across Multiple Agents

Each agent reports its token consumption at task completion as part of the Done-Retirement pipeline (at the NOTIFY stage). The mayor-coordinator aggregates these reports to maintain a real-time view of convoy-level budget consumption. The witness-observer includes budget health in its patrol reports, flagging convoys that are consuming tokens faster than estimated.

The Economics of Orchestration at Scale

Orchestration itself has a cost. The mayor-coordinator, witness-observer, and sling dispatch all consume tokens. In our measurements, orchestration overhead typically accounts for 8–12% of total convoy token consumption. This is the "tax" you pay for coordination. The question is whether the productivity gains from orchestration exceed this tax.

For the PNW Research Series, the math was clear. Without orchestration, a single agent producing 190+ projects sequentially would have required an estimated 12.5 million tokens (including the 30–40% coordination overhead of re-establishing context after each reset). With Gastown orchestration, the total consumption was approximately 9.8 million tokens: 8.6 million for productive work and 1.2 million for orchestration overhead. The orchestrated approach consumed 22% fewer tokens while completing the work in one-quarter of the wall-clock time.

  7. Witness-Observer patrols every 60 seconds, checking agent-eco-1's heartbeat via Nudge-Sync. Agent-eco-1 responds to each health check, confirming it is working on section 6.
  8. Agent-eco-1 completes the document. Done Retirement kicks in: validate (all 7 sections present), commit, push, submit merge request.
  9. Refinery Merge picks up the merge request, checks out agent-eco-1's branch, rebases onto main, verifies the output, and merges.
  10. Done Retirement continues: notify mayor (via Mail-Async), cleanup temporary files, terminate agent.
  11. Mayor-coordinator receives the completion notification, updates convoy progress (1/12 complete), and signals Sling Dispatch to send the next work item to the now-available agent-eco-1.
  12. The cycle repeats until all 12 work items are complete and the convoy transitions to COMPLETE and then ARCHIVE.

This entire flow—from convoy creation to final archive—runs autonomously. No human intervention is required unless an escalation is triggered (budget exhaustion, repeated agent failures, or validation failures that exceed the retry limit).

13.13 Lessons Learned at Scale

Theory is cheap. Production is expensive. The patterns in this section were not designed on a whiteboard and then implemented. They were extracted from production failures, refined through iteration, and validated through repeated use at increasing scale. The lessons documented here represent hundreds of hours of agent execution time and dozens of post-mortem analyses.

Agent Supervision: Trust but Verify

Standing Rule

Always check agent reads and writes against instructions. Agents will occasionally misinterpret their instructions, especially when the context window is under pressure. An agent told to "write the marine biology section" might write the freshwater ecology section instead, because a similar document was in its context from a previous task. The witness-observer's job is not just to check if agents are alive but to verify that their outputs match their assignments.

This rule emerged from a production incident during the Music cluster generation. Agent #2 was assigned to write the "Jazz Harmony" document but instead wrote a second copy of "Blues Origins" (which Agent #1 had already completed). The agent's context had retained too much of Agent #1's output from a shared reference file, and the similarity between the two topics caused the agent to drift. After this incident, we added output-title validation to the Done Retirement VALIDATE stage: the output document's title must match the assigned task specification. A simple check that would have caught the error in under 1 second.

Task Sizing: The 5-6 Rule

We arrived at the "5–6 tasks per teammate" guideline through empirical measurement across dozens of fleet dispatches. Below are the actual observations:

Tasks per Agent Completion Rate Quality Score (avg) Context Exhaustion
1–2 100% 4.5/5.0 Never
3–4 98% 4.3/5.0 Rare (<5%)
5–6 95% 4.1/5.0 Occasional (10%)
7–8 85% 3.6/5.0 Frequent (30%)
9+ 70% 3.1/5.0 Common (50%+)

The 5–6 range represents the sweet spot where completion rate remains acceptably high (95%), quality stays above 4.0, and context exhaustion is manageable. Below 5, you are underutilizing agents. Above 6, the quality drop and failure rate increase the total cost (including rework) beyond the savings from reduced agent count.

Simple First, Complex Later

When loading a convoy with work items, sort them by complexity: put the simplest tasks first. This is counterintuitive—you might think it is better to tackle the hardest tasks first while the agents are fresh. But there are two practical reasons for simple-first ordering:

  1. Early success builds momentum. An agent that successfully completes its first three tasks has demonstrated that its context, tools, and instructions are working correctly. Any failures on subsequent harder tasks can be attributed to the task difficulty rather than to systemic setup problems.
  2. Debugging is easier. If an agent fails on its first task, you do not know whether the failure is due to the task, the instructions, the context, or the tooling. If it succeeds on three simple tasks and then fails on a complex one, the diagnosis space is much narrower.

Agent Failures Are Normal

This is perhaps the most important lesson: agent failures happen, and they are not exceptional. In any fleet of 4 agents running 5 tasks each, you should expect 1–2 task failures. The question is not how to prevent all failures (you cannot) but how to design for graceful recovery.

The fallback plan is always the same: write it directly. If an agent fails on a task after two retries, do not keep retrying. Switch to direct execution: the commander agent (or a human) takes the failed task and completes it manually. This is faster than diagnosing why the agent failed and cheaper than a third retry that will probably also fail.

4 Agents for Synthesis

Research, analysis, and document generation tasks benefit from 4 parallel agents. These tasks require more context per task, more reasoning depth, and more iteration. Four agents provide good throughput without overwhelming the orchestrator's monitoring capacity.

3 Agents for Doc Runs

Straightforward documentation generation (converting existing source material into formatted output) works best with 3 agents. The tasks are more predictable, the context requirements are lower, and the reduced agent count simplifies coordination.

Context Is RAM

Treat the agent's context window as volatile memory. Anything in the context will be lost on reset. Anything that must survive must be written to disk (state on disk). Design every interaction assuming the context could reset at any moment. This is not pessimism; it is production thinking.

State Is on Disk

The corollary of "context is RAM" is "state is on disk." Every decision, every checkpoint, every progress marker must be persisted to the filesystem. The beads-state system exists for exactly this purpose. An agent that does not persist its state is an agent that loses its state.

Never Self-Interrupt

During autonomous execution, an agent should never stop to ask for input. If the agent encounters ambiguity, it should make a reasonable default choice, document the choice, and continue. Stopping for input breaks the GUPP flow, wastes context on re-establishing state after the human responds, and defeats the purpose of autonomous execution.

This rule has a corollary: every task specification must be complete enough for the agent to execute without clarification. If an agent regularly stops to ask questions, the problem is not the agent; it is the task specification. Invest more in the PREPARE stage of sling dispatch to ensure that every work item is self-contained.

Post-Mortem Discipline

Every convoy that experiences a failure gets a post-mortem analysis. The analysis examines:

  • Root cause: Was the failure due to task specification, agent behavior, tooling, or external factors?
  • Detection latency: How long between the failure occurring and the system detecting it? If the witness-observer took more than 5 minutes to detect a stall, the patrol interval needs adjustment.
  • Recovery cost: How many tokens and how much wall-clock time did recovery consume? Is there a pattern improvement that would reduce this cost for future convoys?
  • Prevention: Can the failure mode be prevented in future runs? If so, does the prevention belong in the task specification, the sling dispatch pipeline, the validation stage, or the agent instructions?

Post-mortem findings are fed back into the pattern implementations. The current versions of every pattern in this section reflect dozens of such improvements. The sling dispatch pipeline started with 4 stages and grew to 7 after post-mortems revealed gaps in the prepare and confirm stages. The Done-Retirement pipeline added the VALIDATE stage after a post-mortem found that agents were committing output that did not match specifications. The witness-observer's stall detection thresholds were tuned based on 47 separate stall incidents across 3 months of production use.

The Meta-Lesson

The meta-lesson from 190+ projects is this: orchestration patterns are not invented, they are discovered. Every pattern in this section started as a manual workaround for a specific failure. When the same workaround was applied three times, it became a named pattern. When it was applied ten times, it became a reusable skill. When it was applied fifty times, it became part of the system architecture. The system grew from the bottom up, shaped by the specific pressures of production use. Patterns that were theoretically elegant but did not survive production contact were discarded. Patterns that were ugly but reliable were kept and refined. The result is a system that is neither clean nor beautiful but is deeply practical. It works because every piece of it has been tested against the harsh reality of what agents actually do when you give them real work at scale.
