Executive Summary

Performance optimization in AI agent systems has evolved from a purely infrastructure concern to a multi-dimensional design discipline. In 2026, the frontier has shifted: the most impactful performance levers are no longer just caching and batching, but fine-grained reasoning controls, intelligent model routing, parallel execution architectures, and hardware-accelerated computation. This section covers the full performance stack, from extended thinking modes and effort levels through GPU-accelerated math co-processors and production-scale metrics from systems with 21,298 passing tests and 50+ version chains.

Key Insight: The most powerful performance improvement in agentic AI is not making individual operations faster, but choosing the right level of reasoning for each task. Extended thinking with effort controls, model routing between Haiku/Sonnet/Opus, and parallel dispatch with worktree isolation together deliver order-of-magnitude improvements in throughput-per-dollar while maintaining or improving quality.

1. Extended Thinking

1.1 What Extended Thinking Is

Extended thinking is a mode in which the model explicitly shows its reasoning process before delivering a final answer. Rather than producing a response in a single pass, the model generates an internal chain of thought that explores the problem space, considers alternatives, checks its own logic, and then synthesizes a conclusion. This reasoning trace is visible to the user or calling system, providing transparency into how the model arrived at its answer.

In Claude Code, extended thinking is enabled through specific keywords in the system prompt or skill frontmatter. The most common trigger is the keyword ultrathink, which signals the model to engage its deepest reasoning mode before responding.

How Extended Thinking Works

Standard Mode:
  Input → [Single-Pass Generation] → Output

Extended Thinking Mode:
  Input → [Thinking Block] → [Verification] → [Refined Output]
              ↓
         Visible reasoning:
         - Problem decomposition
         - Hypothesis generation
         - Self-checking
         - Alternative exploration
         - Synthesis

1.2 When Extended Thinking Helps

Extended thinking is not universally beneficial. It adds latency and token cost. The performance improvement comes from accuracy gains on problems where single-pass reasoning is insufficient:

Task Category Extended Thinking Benefit Example
Complex Architecture Decisions High -- explores trade-offs systematically Designing a multi-service data pipeline with consistency requirements
Debugging Multi-File Issues High -- traces causality across components Race condition spanning three async modules
Mathematical Proofs Very High -- step-by-step verification prevents errors Eigenvalue spacing analysis, convergence proofs
Security Audits High -- systematic threat modeling Analyzing authentication flow for OWASP Top 10
Simple File Edits Low -- adds latency without accuracy gain Renaming a variable, fixing a typo
Boilerplate Generation Low -- pattern matching is sufficient Creating a standard REST endpoint

1.3 Enabling Extended Thinking in Skills

In the Claude Code skill system, extended thinking can be activated through skill frontmatter or by including the ultrathink keyword in the skill instructions. This allows skill authors to declare that a particular skill's work requires deep reasoning:

---

1. Map all component dependencies before suggesting changes
2. Identify circular dependencies and hidden coupling
3. Evaluate at least 3 alternative approaches
4. Verify backward compatibility of proposed changes
5. Check for emergent failure modes in the interaction graph

Design Principle: Extended thinking is a performance investment, not a universal improvement. The key insight is that spending 5x more tokens on reasoning for a hard architectural decision that prevents a week of refactoring is enormously cost-effective. Spending 5x more tokens to rename a variable is waste. The effort level system (Section 2) provides the mechanism to make this trade-off systematically.
Research Insight

Chain-of-Thought Scaling Laws: Research in 2025-2026 demonstrates that reasoning quality scales with the length and depth of the chain-of-thought, but with diminishing returns. The optimal thinking budget depends on task complexity. Simple tasks see no improvement beyond minimal thinking. Complex multi-step problems continue to improve with extended reasoning budgets up to approximately 10,000 tokens of thinking, after which gains plateau for most engineering tasks.

2. Effort Levels -- Fine-Grained Performance Control

2.1 The Four Effort Levels

Claude Code supports four discrete effort levels that control how much reasoning the model applies to each request. This is the single most impactful performance control available to skill and agent authors, because it determines the fundamental trade-off between speed and accuracy for every operation.

Low

Speed: Fastest response times

Use cases: Fast triage, simple lookups, status checks, yes/no decisions, straightforward file reads

Reasoning depth: Minimal -- pattern matching and direct retrieval

Token budget: Minimal thinking tokens

Example: "Is this file a test file?" / "What branch am I on?"

Medium

Speed: Balanced response times

Use cases: Standard implementation work, routine code edits, test writing, documentation updates

Reasoning depth: Standard -- considers context and basic alternatives

Token budget: Moderate thinking tokens

Example: "Add input validation to this function" / "Write tests for this module"

High

Speed: Slower, thorough responses

Use cases: Complex analysis, detailed implementation, multi-file refactoring, security review

Reasoning depth: Deep -- explores alternatives, checks edge cases

Token budget: Extended thinking enabled

Example: "Refactor auth to support OAuth2 + SAML" / "Analyze this race condition"

Max (Opus Only)

Speed: Slowest -- deep deliberation

Use cases: Architectural decisions, proof verification, critical security audits, complex debugging

Reasoning depth: Maximum -- full extended thinking with self-verification

Token budget: Unlimited thinking tokens

Example: "Design the trust system architecture" / "Verify this convergence proof"

2.2 Setting Effort Levels

Effort levels can be configured at three scopes, with inner scopes overriding outer ones:

Scope How to Set Applies To Override Behavior
Session CLI flag or settings.json All operations in the current session Default baseline; overridden by skill or subagent settings
Skill Skill frontmatter: effort: high All operations while the skill is active Overrides session default
Subagent Subagent frontmatter or dispatch config All operations by that subagent instance Overrides both session and skill defaults

2.3 Effort Inheritance

When no explicit effort level is set, agents inherit from their parent scope. This creates a natural cascade:

Effort Level Inheritance Chain

Session (effort: medium)  ← default baseline
  ↓
  Skill A (effort: high)  ← overrides for this skill's context
    ↓
    Subagent X (no override)  ← inherits "high" from Skill A
    Subagent Y (effort: low)  ← explicit override to "low"
  ↓
  Skill B (no override)  ← inherits "medium" from session
    ↓
    Subagent Z (effort: max)  ← explicit override to "max"

2.4 Performance Impact by Effort Level

Metric Low Medium High Max
Response Latency ~1-3s ~3-10s ~10-30s ~30-120s
Token Usage (Input+Output) 500-2K 2K-10K 10K-50K 50K-200K
Accuracy (Complex Tasks) 60-70% 75-85% 85-95% 92-99%
Accuracy (Simple Tasks) 95%+ 97%+ 98%+ 98%+
Cost per Operation $0.001-0.01 $0.01-0.10 $0.10-1.00 $1.00-5.00
Critical Observation: For simple tasks, accuracy is roughly constant across all effort levels -- the extra reasoning provides no benefit. The differentiation happens on complex tasks, where the gap between low (60-70%) and max (92-99%) is enormous. This is why effort routing matters: you want to spend max effort only where it makes a measurable accuracy difference.
GSD Integration

GSD Model Profiles: The GSD workflow system supports model profiles (quality/balanced/budget/inherit) that map to effort levels. The /gsd-set-profile command switches between profiles, and individual skills can override with frontmatter. In production, our balanced profile uses medium effort for execution phases and high effort for planning phases, reflecting the principle that planning quality has outsized impact on execution outcomes.

3. Parallel Execution Patterns

3.1 Git Worktree Isolation

The foundation of safe parallel execution in code-centric agent systems is filesystem isolation. Without it, parallel agents writing to the same files create merge conflicts, race conditions, and corrupted state. Git worktrees provide the solution: each parallel agent gets its own complete working directory, all sharing the same Git object store.

Worktree Isolation Architecture

Main Repository: /project
  ↓
  git worktree add /tmp/agent-1-work feature-branch
  git worktree add /tmp/agent-2-work feature-branch
  git worktree add /tmp/agent-3-work feature-branch
  git worktree add /tmp/agent-4-work feature-branch

┌─────────────────────────────────────────────────┐
│              Shared Git Object Store             │
│         (commits, blobs, trees -- read-only)     │
└──────┬──────────┬──────────┬──────────┬──────────┘
       ↓          ↓          ↓          ↓
 ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
 │ Agent 1  │ │ Agent 2  │ │ Agent 3  │ │ Agent 4  │
 │ Worktree │ │ Worktree │ │ Worktree │ │ Worktree │
 │ /tmp/a1  │ │ /tmp/a2  │ │ /tmp/a3  │ │ /tmp/a4  │
 │ Files: A │ │ Files: B │ │ Files: C │ │ Files: D │
 └──────────┘ └──────────┘ └──────────┘ └──────────┘
       ↓          ↓          ↓          ↓
       └──────────┴──────────┴──────────┘
                      ↓
              Merge / Rebase
                      ↓
              Main Repository

3.2 Fleet-Mission Pattern

The fleet-mission pattern dispatches N agents simultaneously, each operating in an isolated worktree with a focused task. A coordinator agent monitors progress, collects results, and merges outputs. This pattern is proven at scale in our production system, where we routinely dispatch 3-5 agents per wave.

Skill Configuration 
# fleet-mission dispatch example
# Each agent gets: isolated worktree + focused instructions + effort level

Fleet: 4 agents
  Agent 1: "Implement user authentication module"
    worktree: /tmp/fleet-auth
    effort: high
    files: src/auth/*

  Agent 2: "Write API endpoint tests"
    worktree: /tmp/fleet-tests
    effort: medium
    files: tests/api/*

  Agent 3: "Update documentation for new endpoints"
    worktree: /tmp/fleet-docs
    effort: low
    files: docs/api/*

  Agent 4: "Security audit of input validation"
    worktree: /tmp/fleet-security
    effort: max
    files: src/validation/*

Coordinator: merge results, resolve conflicts, run integration tests

3.3 Wave-Based Execution

Wave execution groups independent tasks into parallel batches, with synchronization points between waves. This is the primary execution model for GSD phase execution and has been proven across 50+ version chains:

Wave Execution Flow

Wave 1: [Agent A] [Agent B] [Agent C]  ← independent, run in parallel
             ↓         ↓         ↓
         ════════════════════════════  ← synchronization barrier
             ↓
Wave 2: [Agent D] [Agent E]            ← depend on Wave 1 results
             ↓         ↓
         ════════════════════════════  ← synchronization barrier
             ↓
Wave 3: [Agent F]                      ← synthesis agent, depends on all
             ↓
         Final Result

3.4 The /batch Skill Pattern

For large-scale parallel work, the /batch skill dispatches 5-30 independent units across parallel worktrees. Each unit gets its own agent, its own filesystem, and its own context window. The batch skill handles coordination, progress tracking, and result aggregation:

Batch Parameter Range Impact
Unit count 5-30 More units = more parallelism, but more coordination overhead
Worktree isolation Required for file-modifying tasks Prevents filesystem conflicts between agents
Effort per unit Configurable per-unit Complex units get high effort; simple units get low
Merge strategy Sequential rebase or parallel merge Rebase produces cleaner history; merge is faster

4. Model Routing for Performance

4.1 The Right Model for the Right Job

Not every task requires the most powerful (and expensive) model. Model routing is the practice of dynamically selecting which model handles each sub-task based on the task's complexity, latency requirements, and accuracy needs. This is one of the highest-impact cost optimizations available.

Haiku -- Fast Exploration

Strengths: Extremely fast, very low cost, good at pattern matching

Best for: Triage, classification, simple lookups, status checks, formatting

Latency: Sub-second for most queries

Cost: ~10-50x cheaper than Opus per token

When to use: The task has a clear, unambiguous answer that does not require multi-step reasoning

Sonnet -- Balanced Implementation

Strengths: Strong coding, good reasoning, fast enough for interactive use

Best for: Standard implementation, test writing, code review, documentation

Latency: 2-10 seconds for typical tasks

Cost: ~3-5x cheaper than Opus per token

When to use: The task requires reasoning but not deep architectural thinking

Opus -- Complex Reasoning

Strengths: Deepest reasoning, best at architecture, handles ambiguity

Best for: Architecture decisions, complex debugging, proof verification, novel problems

Latency: 10-120 seconds for complex tasks

Cost: Highest per-token cost

When to use: The task requires exploring multiple approaches, handling edge cases, or the cost of an error is high

4.2 Dynamic Routing Patterns

Model Selection Decision Tree

                     Incoming Task
                          ↓
                   Complexity Assessment
                          ↓
              ┌───────────┼───────────┐
              ↓           ↓           ↓
           Simple      Standard     Complex
              ↓           ↓           ↓
           Haiku       Sonnet       Opus
              ↓           ↓           ↓
         <1 second    2-10 sec    10-120 sec
              ↓           ↓           ↓
         $0.001       $0.01-0.10   $0.10-5.00

Routing Signals:
  - Task type (lookup vs analysis vs architecture)
  - File count (1 file = simpler, 10+ files = complex)
  - Change scope (rename vs refactor vs redesign)
  - Error cost (low stakes vs production-critical)

4.3 Heterogeneous Agent Fleets

In multi-agent systems, different agents can use different models based on their role. This is a natural extension of model routing applied at the agent architecture level:

Agent Role Recommended Model Rationale
Planner / Architect Opus Planning quality has outsized impact on execution outcomes
Executor / Implementer Sonnet Follows clear plans; needs good coding, not deep reasoning
Verifier / Reviewer Opus or Sonnet Opus for adversarial review; Sonnet for standard checks
Triage / Router Haiku Fast classification; route to the right agent quickly
Documentation Writer Sonnet Good prose quality at reasonable cost
Security Auditor Opus (max effort) Cost of missing a vulnerability is very high
Cost Impact: A well-designed routing strategy can reduce total API costs by 60-80% compared to using the most capable model for every task, while maintaining or improving overall quality. The key insight is that most tasks in a typical workflow are simple enough for Haiku or Sonnet -- only 10-20% of operations genuinely benefit from Opus-level reasoning.

5. Context Window Management

5.1 The Context Budget Problem

Context is the scarcest resource in agentic AI. Models degrade in quality as context grows: retrieval accuracy drops, instructions get diluted, and irrelevant information creates noise. Managing context is therefore a core performance discipline.

Claude Code models support context windows up to 200K tokens (Claude 3.5 Sonnet) and 1M tokens (Claude Opus 4.6). However, effective context -- the amount of context that the model can actually use well -- is significantly smaller than the maximum window. Research consistently shows that models perform best when context is focused and relevant.

5.2 Auto-Compaction

Claude Code automatically compacts the conversation when context approaches the window limit. This process summarizes earlier conversation turns, preserving key facts while discarding verbatim tool outputs and intermediate reasoning. Understanding compaction behavior is essential for session management:

Aspect Behavior Performance Impact
Trigger Context usage exceeds ~80% of window Automatic -- no user action needed
What's Preserved Key decisions, current task state, file modifications Core context survives compaction
What's Lost Verbatim file contents, intermediate tool outputs, reasoning traces May need to re-read files after compaction
Quality After Slight degradation in recall of early-session details Design sessions to be resilient to compaction

5.3 Hooks for State Preservation

Claude Code provides PreCompact and PostCompact hooks that fire before and after auto-compaction. These are the mechanism for preserving critical state across compaction boundaries:

JSON (settings.json) 
{
  "hooks": {
    "PreCompact": [
      {
        "type": "command",
        "command": "cat .planning/STATE.md",
        "description": "Inject current project state before compaction"
      }
    ],
    "PostCompact": [
      {
        "type": "command",
        "command": "echo '--- Session resumed after compaction ---' && cat .planning/STATE.md",
        "description": "Re-inject state after compaction completes"
      }
    ]
  }
}

5.4 Manual Compaction

The /compact command triggers manual compaction. This is useful when context has accumulated significant noise (large file reads, extensive tool output) and you want to reclaim context space without waiting for the automatic trigger. Strategic use of /compact can extend effective session duration by 2-3x.

5.5 Strategies to Delay Compaction

Compaction causes information loss. The best strategy is to delay it as long as possible:

Strategy Mechanism Context Savings
Subagent Delegation Offload large tasks to subagents with fresh context windows High -- subagent context does not count against parent
Targeted File Reads Read only the lines you need (offset + limit) instead of full files Medium -- avoids loading thousands of unnecessary lines
Grep Before Read Search for relevant content before reading files Medium -- loads only matching lines
Filesystem State Write intermediate results to disk instead of holding in context High -- context only holds current working set
Structured Handoffs When context is exhausted, write a handoff document and start fresh Resets context entirely with no loss
Research Insight

Context as RAM: The most productive mental model for context windows is to treat them like RAM in a traditional computer. Context is fast to access but limited in size. Disk (filesystem) is large but slower. The art of performance is keeping the right data in context at the right time, just as the art of systems programming is managing the memory hierarchy. Compaction is the equivalent of garbage collection -- necessary, but best minimized through good memory discipline.

6. Background Agents

6.1 Non-Blocking Parallel Work

Background agents run in separate processes, freeing the primary session for interactive work. This is the key enabler for "fire and forget" patterns where long-running tasks (documentation generation, test suite execution, security scanning) proceed without blocking the developer's primary workflow.

Background Agent Architecture

Primary Session (Interactive)
  ↓
  Dispatch background agent
  ↓                              ┌──────────────────────┐
  Continue working                  │   Background Agent   │
  ↓                              │   (separate process)  │
  Receive notification  ←─────── │   Result: complete    │
  ↓                              └──────────────────────┘
  Integrate results

6.2 Implementation Patterns

Bash 
# Launch a background agent for long-running work
claude --background "Generate comprehensive API documentation for all 47 endpoints"

# Background agent with specific permissions
claude --background --yes "Run full test suite and report failures to /tmp/test-results.md"

# Multiple background agents for parallel work
claude --background "Security audit of src/auth/" &
claude --background "Performance profile of src/api/" &
claude --background "Dependency audit of package.json" &

6.3 Combining Background Agents with Worktree Isolation

Maximum parallelism is achieved by combining background agents with worktree isolation. Each background agent operates in its own worktree, preventing filesystem conflicts:

Bash 
# Create isolated worktrees for parallel background agents
git worktree add /tmp/bg-auth feature-auth
git worktree add /tmp/bg-tests feature-tests
git worktree add /tmp/bg-docs feature-docs

# Launch agents in their isolated worktrees
(cd /tmp/bg-auth && claude --background "Implement OAuth2 flow") &
(cd /tmp/bg-tests && claude --background "Write integration tests") &
(cd /tmp/bg-docs && claude --background "Update API docs") &

# Continue interactive work in the main repo
# Results are available when agents complete

6.4 When NOT to Use Background Agents

Dependency Chains: Background agents should only be used for tasks that are independent of the current session's work. If a background agent's output is needed before the next step in the primary session, the primary session will block waiting for it -- defeating the purpose. Use background agents for:
  • Documentation generation (does not affect code)
  • Test execution (read-only analysis)
  • Security scanning (read-only analysis)
  • Research tasks (output to separate files)

Do NOT use background agents for:

  • Tasks whose output is the input to the next step
  • File modifications that the primary session needs immediately
  • Sequential pipelines with ordering constraints

7. Agent Memory Architecture

7.1 Three Types of Long-Term Memory

Modern AI agents require sophisticated memory systems that mirror human cognitive architecture. Research from 2025-2026 has established three distinct types of long-term memory as essential for autonomous agent operation:

Memory Type Purpose Implementation Key Question
Episodic Memory Stores chat interactions and specific experiences Vector databases, conversation logs, temporal sequences "What happened when?"
Semantic Memory Stores structured factual knowledge Knowledge graphs, RAG systems, fact repositories "What is true?"
Procedural Memory Captures expertise and successful action patterns Skills, tool definitions, workflow templates "How do I do this?"

9. GPU-Accelerated Computation

9.1 The Math Co-Processor

AI reasoning and numerical computation are complementary but distinct capabilities. LLMs excel at reasoning, planning, and natural language understanding but are unreliable at precise arithmetic, matrix operations, and statistical calculations. A GPU-accelerated math co-processor bridges this gap by providing deterministic, hardware-accelerated computation accessible through MCP tools.

Architecture

Hardware: NVIDIA RTX 4060 Ti (8GB VRAM)

Protocol: MCP (Model Context Protocol)

Tools: 18 MCP tools across 5 domains

Integration: Native tool calls from Claude Code

Performance

Matrix multiply (1024x1024): <10ms (GPU) vs ~5s (Python/NumPy CPU)

9.3 When Local GPU Beats API Calls

The decision between local GPU computation and API-based computation depends on latency, data size, and privacy requirements:

Factor Local GPU Wins API Wins
Latency Sub-millisecond for small operations Not applicable -- network latency dominates
Data volume Large matrices/datasets (transfer cost > compute cost) Small computations (overhead of local GPU not justified)
Privacy Sensitive data never leaves the machine No privacy constraints
Determinism CUDA operations are deterministic (with fixed seed) LLM arithmetic is probabilistic
Cost Fixed hardware cost, no per-operation charge Pay-per-use, scales with volume
Scale Limited by VRAM (8GB on RTX 4060 Ti) Scales to arbitrary size

10. Agentic RAG

10.1 Evolution to Agentic RAG in 2026

Retrieval-Augmented Generation has evolved from simple "retrieve then generate" pipelines into sophisticated agentic systems where AI agents dynamically plan retrieval strategies, evaluate source quality, and self-correct their outputs. The key shift is from static retrieval to adaptive, multi-step retrieval guided by agent reasoning.

Aspect Traditional RAG Agentic RAG (2026)
Retrieval Strategy Fixed, single-hop Adaptive, multi-step with backtracking
Context Awareness Current query only Full conversation history, user context, task state
Error Handling No validation Self-correction, quality assessment, re-retrieval
Tool Integration Vector DB only Dynamic tool selection: web search, APIs, databases, code search
Source Evaluation Rank by similarity score Quality assessment, conflict resolution, provenance tracking

10.2 Self-Correcting Retrieval Patterns

Corrective RAG (CRAG)

Introduces a lightweight retrieval evaluator that assesses document quality before generation:

Query → [Retrieve Docs] → [Quality Evaluator]
                               ↓
                    ┌──────────┼──────────┐
                    ↓          ↓          ↓
                 High       Medium      Low
                Quality    Quality    Quality
                    ↓          ↓          ↓
                  Use    [Refine +    [Web Search +
                 Docs    Re-rank]     Re-retrieve]
                    ↓          ↓          ↓
                    └──────────┼──────────┘
                               ↓
                          [Generate]

Self-RAG

Models trained to decide when retrieval is necessary and to critique their own outputs:

  • Agent determines if retrieval is needed (some queries can be answered from context)
  • Generates multiple candidate responses
  • Self-evaluates for factuality and citation accuracy
  • Selects the best response based on internal quality scores

Adaptive RAG

Query complexity drives retrieval strategy selection:

  • Simple queries: Direct retrieval (single-hop)
  • Medium complexity: Multi-hop retrieval with intermediate reasoning
  • High complexity: Agentic planning with iterative refinement and backtracking

10.4 Agentic RAG with Tool Use

The most powerful RAG pattern combines retrieval with tool use. The agent does not just retrieve documents -- it can execute code, query APIs, search the web, and run computations as part of the retrieval-and-reasoning loop:

Python (Conceptual) 
# Agentic RAG with tool use
class AgenticRAG:
    def __init__(self):
        self.vector_db = PgVectorStore("maple@tibsfox", schema="artemis")
        self.web_search = WebSearchTool()
        self.code_search = GrepTool()
        self.evaluator = QualityEvaluator()

    async def query(self, question, context):
        # Step 1: Classify query complexity
        complexity = self.assess_complexity(question)

        # Step 2: Multi-source retrieval based on complexity
        if complexity == "simple":
            docs = await self.vector_db.search(question, k=3)
        elif complexity == "medium":
            # Combine vector search with code search
            vec_docs = await self.vector_db.search(question, k=5)
            code_docs = await self.code_search.search(question)
            docs = self.merge_and_rerank(vec_docs, code_docs)
        else:
            # Full agentic loop: retrieve, evaluate, re-retrieve
            docs = await self.iterative_retrieve(question, max_rounds=3)

        # Step 3: Quality evaluation (CRAG pattern)
        quality = self.evaluator.assess(docs, question)
        if quality < 0.5:
            # Low quality: augment with web search
            web_docs = await self.web_search.search(question)
            docs = self.merge_and_rerank(docs, web_docs)

        # Step 4: Generate with self-critique
        response = await self.generate_with_verification(question, docs)
        return response

11.1 Quantitative Scaling Principles

Research from "Towards a Science of Scaling Agent Systems" (December 2025) establishes that scaling is governed by quantifiable trade-offs rather than simple "more agents is better" heuristics. A predictive model using empirical coordination metrics achieves cross-validated R squared of 0.513.

11.2 Architecture Patterns for Scale

Hub-and-Spoke Architecture

A central orchestrator manages all agent interactions, creating predictable workflows with strong consistency. Ideal for compliance-heavy workflows (finance, healthcare). Trade-off: potential bottleneck at orchestrator.

Mesh Architecture

Agents communicate directly, creating resilient systems that handle failure gracefully. High fault tolerance but complex coordination logic and debugging difficulty.

Hybrid Approaches (Recommended)

The winning pattern uses high-level orchestrators for strategic coordination while allowing local mesh networks for tactical execution. This maps directly to the GSD mayor-coordinator pattern: the mayor dispatches work via sling, monitors progress, and handles failures, while individual agents (polecats) execute autonomously within their assigned scope.

11.3 Task-Dependent Scaling Dynamics

Task Type Best Architecture Performance Impact Real Example
Distributed Financial Reasoning Centralized +80.9% improvement Multi-agent financial analysis
Parallel Research Fleet-Mission 3-4x throughput 190+ research projects in parallel waves
Sequential State-Dependent Single Agent Multi-agent degrades Git rebase requiring ordered commits
Large-Scale Content Convoy (Hybrid) 3x with 25% overhead 50+ project mission packs
Critical Insight: Not all tasks benefit from multi-agent approaches. Tasks requiring strictly sequential state-dependent reasoning experience performance degradation when split across multiple agents. The decision to use multi-agent must be based on task structure analysis, not a default assumption that more agents means better results.

11.4 Protocol Standards

Model Context Protocol (MCP)

Standardizes how agents connect to external tools, databases, and APIs. MCP is the USB-C of AI agents -- a universal interface that enables any agent to use any tool. Our system uses MCP for both the math co-processor (18 tools) and the Unison programming environment (30+ tools).

Agent-to-Agent Protocol (A2A)

Google's protocol enabling cross-platform agent collaboration. Complements MCP by defining how agents from different vendors communicate. Together, MCP and A2A are establishing the HTTP-equivalent standards for agentic AI.

13.2 Performance Optimization Checklist

Effort and Reasoning

  • Set default session effort to medium
  • Override to high/max for planning and architecture skills
  • Override to low for triage, status, and lookup skills
  • Use extended thinking (ultrathink) only for tasks where reasoning depth matters
  • Track effort level vs accuracy to calibrate over time

Model Routing

  • Use Haiku for classification, triage, and simple lookups
  • Use Sonnet for standard implementation and documentation
  • Reserve Opus for architecture, complex debugging, and security audits
  • Monitor cost-per-task to identify routing opportunities

Parallelism

  • Use worktree isolation for all parallel file-modifying tasks
  • Target 3-5 agents per wave (diminishing returns beyond 6)
  • Size tasks at 5-6 per agent to fit within a single context window
  • Use background agents for independent long-running work
  • Never parallelize tasks with ordering dependencies

Context Management

  • Use targeted file reads (offset + limit) instead of full file reads
  • Grep before Read to minimize context consumption
  • Delegate large tasks to subagents to preserve parent context
  • Configure PreCompact/PostCompact hooks for state preservation
  • Write structured handoff documents when context approaches limits

GPU Acceleration

  • Use MCP math co-processor for deterministic numerical computation
  • Route matrix operations, FFT, and Monte Carlo to GPU
  • Keep AI reasoning and numerical computation separate
  • Monitor VRAM usage to avoid GPU memory exhaustion

RAG Performance

  • Implement CRAG pattern for quality-aware retrieval
  • Use adaptive retrieval based on query complexity
  • Maintain pgvector indices for fast approximate nearest neighbor search
  • Combine vector search with code search for technical queries

13.3 Common Pitfalls

Pitfall Impact Solution
Using max effort for all tasks 10-50x cost increase with no quality gain on simple tasks Match effort to task complexity; default to medium
Parallelizing sequential work Race conditions, merge conflicts, corrupted state Analyze task dependencies before parallelizing
Using Opus for everything 5-50x cost vs Haiku for equivalent results on simple tasks Route by complexity; 80% of tasks work fine with Sonnet/Haiku
Ignoring compaction behavior Loss of critical context mid-session Configure hooks; write state to filesystem; use handoffs
No observability until production issues Silent quality degradation, debugging nightmares Deploy monitoring from development phase
Skipping quality metrics No feedback loop for improvement Track chain scores, error rates, and cost-per-task

13.4 Future-Proofing

  1. Adopt Open Standards: Use MCP, A2A, and OpenTelemetry to avoid vendor lock-in
  2. Modular Effort Control: Design skills and agents with explicit effort declarations so routing can be optimized independently
  3. Continuous Evaluation: Maintain test suites that evolve with your agents (our 21,298 tests provide the regression safety net)
  4. Cost Monitoring: Track cost-per-task metrics to identify optimization opportunities before they become budget problems
  5. Hardware Acceleration: Build MCP bridges to local compute as GPU capabilities grow; today's RTX 4060 Ti is tomorrow's baseline
