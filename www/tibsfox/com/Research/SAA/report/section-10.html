Section 10

The GSD Orchestrator

From Conversational Interface to Distributed Multi-Agent DevOps System

Research current as of: February 2026

The GSD (Get Stuff Done) Agent represents the convergence of agentic AI and DevOps automation into a single orchestrating intelligence. Rather than requiring developers to understand which agent to invoke or what context to provide, the GSD Agent acts as the "front door" to the entire agentic DevOps system. Users describe what they want to accomplish; the system figures out the rest. This section defines the GSD Agent architecture, its evolution from a simple conversational interface to an enterprise-grade distributed system, and the implementation roadmap.

10.1 The GSD Agent Vision

The GSD Agent is a project-aware, context-holding orchestrator that guides users through the entire DevOps project lifecycle. It maintains the project's requirements, design decisions, and constraints as persistent context, ensuring both users and sub-agents stay aligned on goals.

Core Design Principles

Principle Description Implementation
User-first simplicity The user never needs to understand the underlying agent architecture Natural language interface, GSD handles routing and orchestration
Vision persistence Project goals, requirements, and constraints maintained as living context Project Vision Store with version-controlled documents
Adaptive verbosity System calibrates output level from silent to fully transparent 5-level verbosity controller, adjustable at any time
Human-in-the-loop Autonomous execution with defined escalation points 7 gate types with configurable approval thresholds
Progressive complexity Start simple, scale to distributed systems as demand grows 3-tier architecture: Direct → Queue → Stream
Composable architecture Every sub-agent is a modular skill, independently testable and replaceable MCP-compatible agent definitions with AgentBus abstraction

Why This Matters

The current pain point in agentic AI adoption is that users must understand the system to use it. They need to know which agent to invoke, what context to provide, and how to chain outputs. The GSD Agent inverts this model: it acts as the conversational layer on top of the Internal Developer Platform, making the entire DevOps lifecycle accessible through natural language.

10.3 GSD Orchestrator Core Capabilities

Project Vision Store

The most critical differentiator of the GSD Agent is its persistent project context. This is the "brain" that keeps everyone aligned:

Project Vision Store Structure
project-vision/ ├── requirements.md # What we're building and why ├── design.md # Architecture decisions and constraints ├── standards.md # Code, security, compliance standards ├── progress.md # Current state, milestones, blockers ├── decisions.md # Decision log with rationale ├── team.md # Roles, permissions, escalation paths └── config.yaml # Verbosity, notification preferences, # agent permissions, approval gates

These documents serve as the source of truth that every sub-agent references. When the Code Review Agent evaluates a PR, it checks standards.md. When the Release Agent prepares a deployment, it checks requirements.md for feature completeness. When the Security Agent scans, it checks standards.md for compliance requirements.

Verbosity Controller

Level Name Behavior
1 Silent Only surfaces at human-in-the-loop gates (approvals, blockers, failures). Everything else runs autonomously.
2 Summary Sends periodic digests (daily or per-milestone) plus gates. No play-by-play.
3 Standard Reports major phase transitions, decisions made, and gates. The recommended default.
4 Detailed Shows each agent's reasoning and output. Good for learning or auditing.
5 Transparent Full visibility into every agent action, tool call, and decision. Debug mode.

10.4 Human-in-the-Loop Gates

These are the defined moments where GSD pauses autonomous execution and requests human input:

Gate Type Trigger Example
Architecture Decision Design choice with significant implications "I recommend microservices over monolith. Here's why. Proceed?"
Scope Change Requirement that conflicts with or extends the vision "This feature conflicts with the original scope. Adjust or defer?"
Deployment Approval Release to production or sensitive environment "All tests pass. Security scan clean. Deploy to production?"
Cost Threshold Action exceeding budget guardrails "This infrastructure change adds ~$X/month. Approve?"
Security Exception Vulnerability or policy violation detected "Critical CVE in dependency X. Block release or accept risk?"
Failure Recovery System failure requiring human judgment "Deploy failed. Rollback succeeded. Root cause: [X]. How to proceed?"
Ambiguity Resolution Agent cannot determine intent "You said 'fast.' Optimize for latency or development speed?"

10.5 Messaging Architecture Evolution

The GSD system scales from a single-user chat interaction to an enterprise-grade distributed system through three architectural tiers.

Tier 1: Direct Messaging (1-3 Users)

Synchronous request-response. GSD Agent is a single process that calls sub-agents as functions. Messages flow through the chat platform's API (Slack Bolt SDK, Teams Bot Framework).

  • Communication: Synchronous function calls
  • State: In-memory or SQLite/Redis
  • Limitations: Single point of failure, no parallelism
  • Stack: Slack Bolt, LangGraph, Redis, Git repo for Vision Store

Tier 2: Message Queue (5-50 Users)

Asynchronous message passing. GSD publishes task messages to agent-specific queues. Agents consume, process, and publish results back. Agents are decoupled and can run in parallel.

  • Communication: AMQP (RabbitMQ) or NATS
  • State: PostgreSQL + Redis, Git-backed Vision Store
  • Capabilities: Parallel agents, fault tolerance, priority routing, audit trail
  • Stack: RabbitMQ/NATS, Kubernetes, OpenTelemetry, MCP + A2A

Tier 3: Event Streaming (50-1000+ Users)

Full event-driven architecture with Apache Kafka. Every agent decision, tool call, and user interaction is an immutable event. Apache Flink provides real-time event correlation.

  • Communication: Kafka topics with consumer groups
  • State: Event-sourced via Kafka log + PostgreSQL + Redis
  • Capabilities: Multi-project, multi-tenant, event replay, CEP with Flink
  • Stack: Kafka, Flink, Kubernetes + Helm, OPA + Vault + Falco

Tier Comparison

Capability Tier 1: Direct Tier 2: Queue Tier 3: Stream
Users 1-3 5-50 50-1000+
Projects 1 5-20 Unlimited
Parallelism None Per-agent Full event-driven
Fault tolerance None Message retry Durable log + replay
Audit trail Logs only Message logs Immutable event stream
Real-time analytics No Basic Full CEP with Flink
Cost (monthly) Free-$50 $200-$2K $2K-$20K+

AgentBus Abstraction

The key migration enabler: agents are written against an abstract bus interface. Switching tiers means swapping the bus implementation, not rewriting agents.

AgentBus Interface (Python)
# Abstract interface — same for all tiers class AgentBus: async def publish(self, topic: str, message: AgentMessage): ... async def subscribe(self, topic: str, handler: Callable): ... async def request(self, topic: str, message: AgentMessage) -> AgentMessage: ... # Tier 1: In-process function calls class DirectBus(AgentBus): """No network. Sub-agents called as functions.""" # Tier 2: Message queue class RabbitMQBus(AgentBus): """AMQP protocol. Work queues with acknowledgment.""" # Tier 3: Event stream class KafkaBus(AgentBus): """Immutable event log. Consumer groups for scaling."""

10.6 Phase-by-Phase GSD Behavior

PLAN Phase

GSD asks about goals, constraints, and success criteria. Routes to Requirements Agent, Risk Agent, Compliance Agent. Generates and maintains requirements.md and design.md. Presents architecture decisions at HITL gates.

CREATE Phase

User describes features; GSD decomposes into tasks. Routes to Code Generation Agent and IaC Agent. Code Review Agent evaluates all generated code. Security Agent scans for vulnerabilities (shift-left).

GSD: PR #7 ready. 3 new endpoints, all tests passing, no security issues.
GSD: Code coverage: 87%. Security scan: clean. Ready for review.

VERIFY Phase

GSD automatically triggers test suites on code changes. Routes to Test Generation Agent for new coverage. Performance Agent runs load tests before release candidates. Security Agent runs DAST against staging.

GSD: All 147 tests pass. Code coverage 87%. No critical vulnerabilities.
GSD: Performance baseline: p99 latency 45ms (within SLO of 100ms).

RELEASE Phase

GSD: Deploy v1.3.2 to production via canary (10% → 50% → 100%)?
User: Yes, go ahead.
[GSD routes to Release Agent via ArgoCD, monitors via Operate/Monitor agents]
GSD: v1.3.2 live. Canary complete. All SLOs green.

OPERATE Phase

Incident Agent monitors for issues 24/7. Self-healing Agent handles auto-recovery. GSD only alerts user for severity 1-2 incidents or when auto-remediation fails.

GSD: Incident #42: Memory spike on pod-3. Auto-scaled. Resolved in 90 seconds.
[Verbosity level 3: reported because it was auto-resolved]

MONITOR Phase

AIOps Agent analyzes telemetry continuously. Cost Agent tracks spending against budget. SLO Agent monitors error budgets. GSD sends digests per verbosity setting.

GSD: Weekly summary:
  ✓ 99.97% uptime (SLO target: 99.9%)
  ✓ 4 successful deployments, 0 rollbacks
  ⚠ Test suite time increased 23% (now 8 min)
  💰 $127 cloud spend (-8% vs last week)

The Feedback Loop

Closing the Loop

GSD feeds operational insights back into planning. Incident patterns become backlog items. Performance regressions become sprint tasks. Cost trends inform architecture decisions. This creates a continuous improvement cycle where the system learns from its own operational data.

10.7 Implementation Roadmap

Phase 0: GSD Agent Core (Months 1-2)

Working conversational agent for project setup. Slack/Teams bot (Tier 1). Project Vision Store. Verbosity controller. MCP for basic tool access.

Phase 1: Specialized Sub-Agents (Months 2-4)

Code Review, Test Generation, Build, and Plan agents. HITL gate framework. Agent Router with MCP + direct calls. AgentBus abstraction in place.

Phase 2: Message Queue (Months 4-6)

Tier 2 migration (RabbitMQ/NATS). Containerized agents on Kubernetes. Release, Security, FinOps agents. Multi-user support. OpenTelemetry tracing.

Phase 3: Operations & Self-Healing (Months 6-9)

Incident Triage, Self-Healing, AIOps, SLO, Runbook Execution agents. Knowledge Agent for learning from incidents. Retrospective Agent.

Phase 4: Event Streaming (Months 9-12+)

Tier 3 migration (Apache Kafka + Flink). Multi-project, multi-team orchestration. Event sourcing. DORA metrics automation. A2A cross-org collaboration.

Phase 5: Platform Maturity (12+ Months)

Self-service golden paths via conversation. Backstage integration. Custom agent creation by users. Agent skill marketplace. Fine-tuned models.

10.9 Risk Considerations

Risk Impact Mitigation
Over-automation GSD makes decisions the user should make Configurable HITL gates. Default to "ask" for high-impact actions.
Context drift Project vision becomes stale Regular vision review prompts. Decisions logged with rationale.
Agent loops Sub-agents stuck in retry loops Circuit breakers, max iteration limits, dead letter queues, alerting.
Token cost explosion Complex orchestration consumes excessive tokens Token budgets per agent per task. Model routing. Caching.
Security exposure Agents with tool access are an attack surface Least-privilege per agent. Tool call auditing. Approval gates for destructive actions.
Vendor lock-in Tight coupling to specific LLM provider Agent abstraction layer. MCP for standardization. Swap LLM per agent.
Pilot purgatory Building forever, never shipping Phase 0 is a usable product. Each phase adds value incrementally.
