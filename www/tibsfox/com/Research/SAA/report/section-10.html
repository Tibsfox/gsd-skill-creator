The GSD Orchestrator

A Mature Workflow Orchestration System at 190+ Project Scale

Research current as of: April 2026

GSD (Get Shit Done) is an operational workflow orchestration system built as Claude Code skills, agents, and commands. What began as a conversational DevOps concept has matured into a battle-tested system with 57 commands, 34 skills, 26 specialized agents, and a proven track record across 190+ completed research and engineering projects. This section documents GSD as it actually operates: the command inventory, agent fleet, skill system, wave-based execution model, lifecycle phases, state management, multi-branch workflows, and quality systems that have been refined through thousands of hours of real use.

10.1 What GSD Is

GSD is a workflow orchestration system built entirely as Claude Code skills, agents, and slash commands. It does not run as a separate server, daemon, or SaaS product. It lives inside the developer's working environment — a collection of markdown files, JSON configuration, and Claude Code extension points that together create a structured project management backbone.

The name is deliberately blunt. GSD exists to get work done: to turn vague intentions into shipped artifacts through a disciplined lifecycle of discussion, planning, execution, verification, and shipping. It enforces structure without bureaucracy, provides guardrails without friction, and scales from a single afternoon task to a multi-month, multi-branch research program.

The Core Architecture

GSD operates through three interlocking layers:

The .planning/ Directory

Every GSD-managed project has a .planning/ directory at its root, containing the project's management backbone. This directory is always gitignored — it never enters version control. It contains living documents that track requirements, roadmaps, state, and phase-specific artifacts. The .planning/ directory is to GSD what the .git/ directory is to Git: the local state that makes everything work.

Skills and Agents

34 skills provide contextual intelligence that activates automatically based on what the developer is doing. 26 agents provide specialized execution capabilities that can be dispatched in parallel. Skills are persistent context; agents are ephemeral workers. Together they form the execution engine.

57 Commands

The slash command interface provides the developer's control surface. Commands range from high-level orchestration (/gsd:autonomous runs an entire milestone) to fine-grained operations (/gsd:note captures a single idea). The command set has grown organically from real usage patterns, not from a theoretical design.

The Lifecycle

Every unit of work in GSD follows a five-phase lifecycle:

Discuss
→
Plan
→
Execute
→
Verify
→
Ship

This lifecycle applies at every scale: a single feature, a milestone with 20 phases, or a 190-project research series. The phases are not waterfall stages — they are a loop. Verification often sends work back to execution; planning can trigger additional discussion; shipping can reveal issues that restart the cycle.

The Four Backbone Files

GSD's state lives in four files within .planning/:

File Purpose Updated By
ROADMAP.md Defines all phases with names, descriptions, dependencies, and status Planning commands, milestone creation
STATE.md Tracks current phase, active work, completion status, and blockers Automatically by phase transitions
REQUIREMENTS.md Captures project requirements gathered during the discuss phase Discussion commands, manual updates
config.json Workflow configuration: model profiles, toggles, workspace settings Settings commands, manual edits

Branch-Based Workflow

GSD follows a branch-based development model where the dev branch is the primary working branch and main receives merges only after human review. This separation ensures that autonomous execution never pushes directly to production. Additional branches serve specialized purposes: long-running review branches, mission-specific branches, and experimental branches can all coexist, each with its own .planning/ state.

Design Philosophy

GSD was not designed top-down from a specification. It was grown bottom-up from daily use. Every command exists because someone needed it. Every skill was created to solve a recurring problem. Every agent was built to handle a task that was too complex or too tedious for inline execution. The system reflects the accumulated experience of 190+ projects — not a theoretical vision of what project management should look like.

10.2 The Command Inventory (57 Commands)

GSD's 57 commands are organized into functional groups. Each command is implemented as a Claude Code slash command — a markdown file in .claude/commands/gsd/ that defines the command's behavior, parameters, and interaction pattern. Commands range from single-action operations (capture a note) to multi-step orchestrations (run an entire milestone autonomously).

Project Lifecycle Commands

These commands manage the top-level project structure: creating projects, defining milestones, and completing work cycles.

/gsd:new-project

Initializes a new GSD-managed project. Creates the .planning/ directory structure, gathers project context through adaptive questioning, and produces the initial PROJECT.md with scope, goals, and constraints. This is the entry point for any new body of work.

/gsd:new-milestone

Starts a new milestone cycle. Updates PROJECT.md with the milestone definition, creates a fresh ROADMAP.md with phase definitions, and routes to the requirements-gathering flow. Each milestone represents a coherent body of work with a defined outcome.

/gsd:complete-milestone

Archives a completed milestone. Validates that all phases have been verified, generates a milestone summary, archives phase directories, and prepares the project for the next version. Includes an audit step that checks for outstanding verification items.

Phase Workflow Commands

The core of GSD's lifecycle. Each phase moves through discuss, plan, execute, verify, and ship stages. These commands drive that progression.

/gsd:discuss-phase

Gathers context through adaptive questioning before planning begins. Supports an --auto flag that lets Claude answer its own questions using available context, enabling fully autonomous operation. Produces a structured context document that feeds into planning.

/gsd:plan-phase

Creates a detailed phase plan (PLAN.md) using 3-level NASA SE planning methodology: research first (RESEARCH.md), then plan with verification loop. The plan includes task breakdown, wave assignments, success criteria, and rollback procedures.

/gsd:execute-phase

Executes all plans in a phase using wave-based parallelization. Groups independent tasks into waves, dispatches parallel agents per wave, and synthesizes results between waves. Each task produces an atomic commit for bisectability.

/gsd:verify-work

Validates built features through conversational UAT (User Acceptance Testing). Uses goal-backward analysis: starts from what was promised in the plan and works backward to confirm each goal was met. Flags discrepancies and generates verification reports.

/gsd:ship

Creates a PR, runs adversarial review, and prepares for merge. Handles the full shipping pipeline: branch preparation, PR creation with structured body, automated review, and merge readiness verification.

Command Growth Pattern

The command inventory grew from approximately 12 commands at GSD v1.0 to 57 at current state. Growth was not planned — commands were added when a workflow gap was identified through real use. The /gsd:forensics command, for example, was created after a particularly difficult debugging session where understanding what went wrong required manually traversing git history and planning artifacts. The /gsd:plant-seed command emerged from repeatedly losing forward-looking ideas that only became relevant milestones later. Every command has a story.

10.3 The Agent Fleet (26 Agents)

GSD's agents are specialized Claude Code subagents defined in .claude/agents/. Each agent has a focused mandate, specific tools it can access, and a defined interaction pattern. Agents are ephemeral — they are spawned for a task, execute, produce output, and terminate. They do not maintain state between invocations; all persistent state lives in the .planning/ directory or in git.

Core Workflow Agents

These three agents form the backbone of GSD's lifecycle:

gsd-planner

Creates executable phase plans with detailed task breakdowns. Reads the roadmap, requirements, and any research artifacts to produce a PLAN.md with tasks grouped into waves, success criteria per task, and verification checkpoints. Uses the 3-level NASA SE planning methodology: Level 1 (goals), Level 2 (approach), Level 3 (task-level detail with wave assignments).

gsd-executor

Runs wave-based parallel execution with atomic commits. Takes a PLAN.md as input, dispatches tasks per wave, monitors progress, handles deviations, and ensures each completed task produces an atomic commit. The executor is the workhorse — it spends more time active than any other agent in the fleet.

gsd-verifier

Performs goal-backward verification. Starts from the promises made in the plan and works backward through the implementation to confirm each goal was achieved. Produces a verification report that identifies gaps, partial completions, and unexpected additions. This is not test execution — it is semantic verification of intent.

Planning Support Agents

gsd-phase-researcher

Researches before planning begins. Analyzes the codebase, existing patterns, dependencies, and potential approaches to produce a RESEARCH.md that the planner uses as input. Prevents planning in the dark.

gsd-plan-checker

Validates plans before execution begins. Checks for missing dependencies, unrealistic task sizing, wave ordering problems, and conflicts with existing code. Acts as a pre-flight check that catches planning errors before they become execution failures.

Research Agents

fleet-commander

Coordinates parallel agent fleets for large-scale research and execution. Dispatches N agents, monitors completion percentages, handles stragglers, and merges results. Used when a task can be decomposed into independent parallel work items — for example, processing 50 research projects simultaneously.

synthesizer

Merges outputs from parallel agents into coherent results. When a fleet of 4 agents each produces research on different topics, the synthesizer combines their findings, resolves contradictions, identifies gaps, and produces a unified output.

project-researcher

Deep-dives into a single research topic. Produces structured research documents with citations, analysis, and recommendations. Proven across 190+ research projects in the PNW Research Series.

Documentation Agents

doc-writer

Generates documentation from code, plans, and artifacts. Understands the project's documentation style and produces content that matches existing tone, structure, and depth. Used for both internal planning documents and published research.

doc-verifier

Cross-references documentation against the codebase to ensure accuracy. Catches stale references, outdated code examples, missing API documentation, and claims that don't match implementation.

Quality and Security Agents

security-auditor

Scans for security issues in code, configuration, and dependencies. Particularly important in GSD because the system is self-modifying — skills and agents can change their own behavior. The security auditor ensures these modifications stay within safety boundaries.

integration-checker

Verifies cross-phase integration: confirms that work done in phase N doesn't break work done in phase N-1. Checks for interface violations, dependency conflicts, and regression risks across phase boundaries.

nyquist-auditor

Coverage auditing based on the Nyquist principle: if you sample at twice the frequency of the signal, you capture everything. Applied to verification, this means checking enough of each phase's output to have confidence in the whole. Identifies coverage gaps and recommends additional verification.

Orchestration and Communication Agents

Agent Design Principles

Three principles govern agent design in GSD. First, agents are ephemeral: they spawn, execute, and die. No agent maintains state between invocations. Second, agents are specialized: each has a single focused mandate. The planner never executes; the executor never plans. Third, agents are composable: the fleet-commander can dispatch any combination of agents, and the mayor-coordinator can create convoys of any size. These principles emerged from experience. Early versions tried to have agents that both planned and executed, and the failure modes were severe — context pollution, role confusion, and unpredictable behavior.

10.4 The Skill System (34+ Skills)

Skills are the contextual intelligence layer of GSD. Unlike agents (which are ephemeral workers), skills are persistent context that activates automatically based on what the developer is doing. A skill provides domain knowledge, workflow guidance, and behavioral modification without explicit invocation. Skills live in .claude/skills/ and are loaded by Claude Code based on trigger conditions.

Core Workflow Skills

gsd-workflow

The main routing skill. Handles GSD workflow routing, phase execution, and lifecycle management. When a developer types a GSD command, this skill provides the context for how the command should behave given the current project state. It is the connective tissue between all other GSD components.

session-awareness

Project state recovery on session start. When a new Claude Code session begins, this skill reads .planning/STATE.md, recent git history, and any handoff documents to reconstruct the project's current state. Eliminates the "where was I?" problem that plagues context-window-limited AI assistants.

skill-integration

Adaptive learning with bounded safety. Manages the skill-creator integration, observes session patterns, and can suggest new skills based on recurring behaviors. The "bounded" part is critical — this is a self-modifying system, so skill-integration operates within strict safety constraints enforced by the security-hygiene skill.

security-hygiene

The safety guardrail for GSD's self-modifying nature. Monitors skill creation, agent definition, and command modifications for security risks. Ensures that adaptive learning never introduces unauthorized capabilities or expands tool access beyond defined boundaries.

Orchestration Skills

fleet-mission

Parallel agent fleet dispatch with progress tracking and result aggregation. Provides the intelligence for launching N agents, monitoring completion, handling failures, and merging results. Used by the fleet-commander agent but also available as standalone context for ad-hoc parallel work.

mayor-coordinator

The Northbridge coordination pattern. Creates convoys of agents, dispatches work via the sling system, monitors progress across all active agents. Named after the role of a mayor in coordinating city services — doesn't do the work itself, but ensures everyone is working and nothing falls through the cracks.

sling-dispatch

Instruction dispatch pipeline for routing work items to available agents. Implements a 7-stage process: fetch the work item, allocate an agent, prepare the context, hook into the monitoring system, dispatch, monitor execution, and collect results.

Communication Skills

mail-async

Durable asynchronous messaging for inter-agent communication. Implements write-once, read-many filesystem mail. Agents write messages to a known filesystem location; other agents poll for messages addressed to them. Durable because it uses the filesystem rather than in-memory channels.

nudge-sync

Synchronous immediate signaling. Implements a latest-wins single-file nudge pattern. When an agent needs immediate attention from another agent (or the developer), it writes a nudge file. The recipient picks it up on their next polling cycle. Unlike mail-async, nudges are ephemeral — only the latest one matters.

Skills vs. Agents: The Distinction

The distinction between skills and agents is sometimes confusing because several names appear in both lists (e.g., done-retirement, data-fidelity). The difference is in activation: a skill loads its context into the current session passively, providing knowledge and behavioral guidance. An agent is spawned as a separate execution context to perform work. The done-retirement skill tells the current session how retirement works. The done-retirement agent actually performs the retirement steps. Skills inform; agents execute.

10.5 Wave-Based Execution

Wave-based execution is GSD's core execution model. It solves the fundamental tension between parallelism (which speeds up work) and ordering (which ensures correctness). The model is simple in concept but has significant implications for how work is structured, committed, and verified.

The Model

Given a set of tasks in a phase plan, GSD groups them into waves. Within a wave, all tasks are independent — they can execute in parallel without conflicts. Between waves, there is a synchronization barrier: wave N+1 does not start until wave N is complete. This creates a partially ordered execution that maximizes parallelism while preventing conflicts.

Wave Execution Example (from PLAN.md)
## Wave 1 (parallel, no dependencies) - Task 1.1: Create data model for weather stations - Task 1.2: Set up API client for METAR sources - Task 1.3: Build HTML template for weather display ## Wave 2 (depends on Wave 1) - Task 2.1: Implement data fetching pipeline (needs 1.2) - Task 2.2: Connect data model to display template (needs 1.1, 1.3) ## Wave 3 (depends on Wave 2) - Task 3.1: Integration testing across full pipeline - Task 3.2: Performance optimization and caching

Parallel Agent Dispatch

Within a wave, GSD dispatches 3-5 parallel agents (the exact count depends on task complexity and available context budget). Each agent receives its task description, the relevant codebase context, and the plan's success criteria. Agents work in isolation — they cannot see each other's in-progress work.

This isolation is by design. Shared mutable state between parallel agents creates race conditions, merge conflicts, and unpredictable outcomes. By ensuring each agent works on an independent piece of the codebase, GSD eliminates an entire class of coordination failures.

Atomic Commits

Each completed task produces exactly one atomic commit. This is a hard rule with significant benefits:

Benefit How It Works
Bisectability If a regression is introduced, git bisect can identify the exact task that caused it because each commit maps to one logical change
Revertability A single task can be reverted without affecting other work from the same wave
Traceability Every commit message references its phase and task number, creating a direct link from code to plan
Review clarity Each commit in a PR represents one coherent change, making review tractable even for large PRs

Synthesis Between Waves

After each wave completes, GSD runs a synthesis step before starting the next wave. The synthesis step:

  1. Verifies all tasks in the wave completed successfully
  2. Checks for unexpected file conflicts (two agents modifying the same file)
  3. Updates STATE.md with wave completion status
  4. Adjusts the next wave's task descriptions if the current wave's output changes the context
  5. Handles failures — a failed task can be retried, deferred to a later wave, or escalated to the developer

Wave Commit Markers

When session boundaries force combining waves into a single commit (because the context window is exhausted and a new session starts), GSD uses wave commit markers to preserve the logical structure:

Wave Commit Marker Format
feat(weather): implement full data pipeline Wave 1: Created data model and API client Wave 2: Connected pipeline components and display Wave 3: Integration testing and performance tuning

This format preserves bisect intent in the commit message even when commit boundaries don't align with wave boundaries. A developer reading the history can still understand the logical progression of work.

Scaling Observations

Wave-based execution has been tested at various scales across 190+ projects:

Scale Typical Wave Count Agents per Wave Observations
Single feature 1-2 waves 1-2 Often faster than sequential because parallel tasks eliminate waiting
Phase (5-10 tasks) 2-4 waves 3-5 The sweet spot. Good parallelism, manageable synthesis
Milestone (20+ tasks) 5-8 waves 3-5 Requires careful dependency analysis. Wave ordering errors are costly
Research convoy (50+ items) 10+ waves 4 The Gastown convoy model. Agent sizing of 5-6 tasks per teammate is optimal

Lessons Learned

The most important lesson from wave-based execution: planning is the hard part; once the plans are done the code is easy. A well-structured plan with correct wave assignments executes smoothly. A plan with dependency errors or ambiguous task descriptions fails in execution. This is why GSD invests so heavily in the plan-checker agent and the research-before-planning pattern — catching errors before execution is orders of magnitude cheaper than debugging execution failures.

10.6 The discuss → plan → execute → verify → ship Lifecycle

Each phase of a milestone moves through five stages. The stages are not bureaucratic overhead — each one exists because skipping it has, historically, produced measurably worse outcomes. This section describes each stage from operational experience.

Discuss

The discuss phase gathers context through adaptive questioning. GSD asks about goals, constraints, known risks, and success criteria. It reads existing project context (previous phases, codebase state, related work) and synthesizes questions that fill gaps in understanding.

Discuss Phase: --auto Flag
# Interactive mode: GSD asks, developer answers /gsd:discuss-phase # Autonomous mode: GSD answers its own questions from context /gsd:discuss-phase --auto

The --auto flag is critical for autonomous execution. When running /gsd:autonomous, GSD uses the --auto flag on every discuss phase, allowing it to answer its own questions using available context from the codebase, previous phases, and project documentation. This enables fully hands-off milestone execution.

The discuss phase produces a structured context document that feeds into planning. It identifies unknowns, flags risks, and establishes the scope boundary. Discuss without plan is useless; plan without discuss is dangerous.

Plan

Planning follows the 3-level NASA SE methodology, adapted from NASA's systems engineering approach to mission planning:

Level 1: Goals

What are we trying to achieve? What does success look like? What are the non-goals (things we explicitly won't do)? Level 1 is a paragraph, not a document.

Level 2: Approach

How will we achieve the goals? What technologies, patterns, and strategies will we use? What are the key decisions and their rationale? Level 2 is the architecture.

Level 3: Task Breakdown

What are the specific tasks? How are they grouped into waves? What are the success criteria per task? What are the rollback procedures? Level 3 is the PLAN.md.

Before planning, GSD optionally runs the research phase, which produces a RESEARCH.md. This document analyzes the codebase, identifies relevant patterns, surveys approaches, and provides the planner with informed context. The research-before-planning pattern prevents the most common planning failure: making assumptions about the codebase that turn out to be wrong.

After the plan is created, the plan-checker agent validates it: checks for missing dependencies, unrealistic sizing, wave ordering problems, and conflicts with existing code. Plans that fail validation are revised before execution begins.

PLAN.md Structure
# Phase 7: Weather Data Pipeline ## Goals Build a real-time weather data pipeline that fetches from 8 METAR sources, 7 NWS/CWOP/NDBC sources, and 4 satellite feeds, with client-side rendering and hourly refresh. ## Approach Static HTML with client-side JavaScript fetch. No server-side rendering. Data cached in localStorage with TTL. Progressive enhancement: METAR first, then additional sources. ## Tasks ### Wave 1 (parallel) - [ ] Task 1.1: METAR API client (success: fetches from all 8 sources) - [ ] Task 1.2: HTML template with responsive grid (success: renders on mobile) - [ ] Task 1.3: Data model for normalized weather observations ### Wave 2 (depends on Wave 1) - [ ] Task 2.1: NWS/CWOP/NDBC integration (needs 1.1, 1.3) - [ ] Task 2.2: Satellite feed parsing (needs 1.3) - [ ] Task 2.3: Display rendering with live data (needs 1.1, 1.2, 1.3) ### Wave 3 (depends on Wave 2) - [ ] Task 3.1: localStorage caching with TTL - [ ] Task 3.2: Error handling and fallback display - [ ] Task 3.3: Integration test across all sources ## Verification - All 19 data sources return valid data - Page loads in under 2 seconds on 3G - Graceful degradation when sources are unavailable - Hourly refresh works without page reload

Execute

Execution follows the wave-based model described in Section 10.5. The executor takes the PLAN.md, dispatches agents per wave, monitors progress, and produces atomic commits. Key execution behaviors:

  • Deviation handling: When an agent encounters something unexpected (a dependency that doesn't exist, a pattern that conflicts with the plan), it reports the deviation. The executor decides whether to adapt the plan, retry with modified instructions, or escalate to the developer.
  • Checkpoint commits: Long-running tasks produce intermediate commits so that progress is preserved even if the session is interrupted.
  • State tracking: STATE.md is updated after each wave with completion status, enabling session recovery.
  • Failure modes: Agent failures happen. The executor always has a fallback plan: retry with different context, decompose the task further, or write the code directly without a subagent.

Verify

Verification is not testing (though tests may be part of it). Verification is goal-backward analysis: start from what was promised in the plan and work backward through the implementation to confirm each promise was kept.

The verifier agent reads the plan's success criteria, examines the produced artifacts (code, documentation, test results), and produces a verification report. The report identifies:

  • Met: Goals that were fully achieved
  • Partial: Goals that were partially achieved with documented gaps
  • Missed: Goals that were not achieved
  • Bonus: Work that was delivered beyond the plan's scope

Conversational UAT (User Acceptance Testing) is the human-facing side of verification. GSD walks the developer through the results, demonstrates the behavior, and asks targeted questions about whether the implementation matches their intent. This catches the subtle misalignments that automated verification misses.

Ship

Shipping is the final stage. In GSD's operational context, shipping means:

  1. PR creation: A clean PR branch is created, filtering out .planning/ commits that would expose internal state. The PR body includes a structured summary, key changes, and verification status.
  2. Adversarial review: The PR is reviewed adversarially: the reviewer's job is to find problems, not to approve. Cross-references diffs against approved specs, verifies runtime claims against source code, and flags anything that doesn't match the plan.
  3. Merge preparation: Once the review passes, the branch is prepared for merge into the target branch (typically dev, with main receiving merges only after human verification).

The Full Autonomous Loop

When running /gsd:autonomous, all five stages execute without human input for each remaining phase in the milestone. Discuss uses --auto. Planning uses available context and the plan-checker for validation. Execution uses wave-based dispatch. Verification uses goal-backward analysis. Shipping creates PRs ready for human review. The developer can walk away and return to find a complete milestone with PRs waiting for approval. This mode has been used successfully for 58 releases in a single milestone.

10.7 State Management

State management is GSD's most critical infrastructure. Because AI assistants lose context between sessions, and because sessions can end unexpectedly (context window exhaustion, network interruption, user closing the terminal), every piece of project state must be persisted to disk in a recoverable format.

STATE.md

STATE.md is the single source of truth for the project's current position. It tracks:

STATE.md Structure
# Project State ## Current Position - **Milestone:** v1.49 - **Phase:** 7 of 12 - **Stage:** execute - **Wave:** 2 of 3 ## Active Work - Task 2.1: NWS integration (in progress) - Task 2.2: Satellite parsing (complete) - Task 2.3: Display rendering (blocked on 2.1) ## Completed Phases - Phase 1: Project setup (verified) - Phase 2: Data model (verified) - Phase 3: METAR client (verified) - Phase 4: HTML templates (verified) - Phase 5: API integration (verified) - Phase 6: Testing framework (verified) ## Blockers - None ## Notes - NWS API returning 503 intermittently; added retry logic - Switched from fetch() to XMLHttpRequest for IE11 compat

ROADMAP.md

ROADMAP.md defines the milestone's phase structure. Each phase has a name, description, dependencies, and status:

ROADMAP.md Structure
# Roadmap: v1.49 — PNW Research Series ## Phase 1: Project Setup Status: complete Description: Initialize project structure, CI, and tooling ## Phase 2: Data Model Status: complete Depends on: Phase 1 Description: Define weather observation data model ## Phase 3: METAR Client Status: complete Depends on: Phase 2 Description: Build METAR API client for 8 sources ## Phase 7: Weather Pipeline Status: in-progress Depends on: Phase 3, Phase 4, Phase 5 Description: Full data pipeline with all 19 sources ## Phase 8: Performance Optimization Status: planned Depends on: Phase 7 Description: Caching, lazy loading, bundle optimization ## Phase 12: Final Release Status: planned Depends on: Phase 8, Phase 9, Phase 10, Phase 11 Description: Release preparation, changelog, deployment

config.json

config.json holds workflow configuration that persists across sessions:

config.json Structure
{ "project": "pnw-research", "version": "1.49", "model_profile": "quality", "workflow": { "auto_discuss": true, "plan_checker": true, "wave_parallelism": 4, "atomic_commits": true, "conventional_commits": true }, "agents": { "max_parallel": 5, "timeout_minutes": 10, "retry_on_failure": true } }

Phase Directories

Each phase gets its own directory under .planning/phases/N/ containing all artifacts for that phase:

Phase Directory Structure
.planning/phases/7/ ├── RESEARCH.md # Pre-planning research ├── PLAN.md # The execution plan ├── DISCUSS.md # Context gathered during discussion ├── VERIFY.md # Verification report ├── RETROSPECTIVE.md # Lessons learned (carry-forward) └── artifacts/ # Any supporting artifacts

Recovery from Invalid States

GSD can reach invalid states through several paths: a session ends mid-execution, a manual edit corrupts STATE.md, or a failed agent leaves partial state. GSD handles these through:

  • /gsd:health: Diagnoses the .planning/ directory and identifies inconsistencies (phase marked complete in ROADMAP but no VERIFY.md exists, STATE.md references a phase that doesn't exist in ROADMAP, etc.)
  • /gsd:resume-work: Reads the last known state, compares it to the current git history, and reconstructs where work was interrupted. Handles the "context window died mid-execution" case.
  • Handoff documents: /gsd:pause-work creates a structured handoff document that captures everything needed to resume. These documents are the primary recovery mechanism for planned interruptions.
  • Session awareness skill: Automatically activates on session start, reads state files, and reconstructs project context without explicit invocation.

The .planning/ gitignore Rule

.planning/ is always gitignored. This is a hard rule with no exceptions. The reasoning: planning artifacts contain working notes, intermediate reasoning, model profile configurations, and other content that should never enter version control. The published output (code, documentation, research) tells the story. The planning artifacts are the rough drafts, and rough drafts don't ship. This also prevents accidental exposure of internal project state in public repositories.

The .planning/ Isolation Model

Because .planning/ is gitignored, each branch's planning state is inherently local. When a developer switches branches, the .planning/ directory does not change. This means:

  • Each branch can have its own ROADMAP, STATE, and phase directories
  • Branch switching does not corrupt planning state
  • Multiple branches can be active simultaneously (in worktrees) with independent planning
  • The session-awareness skill detects branch mismatches and warns the developer

Workspaces

For truly isolated parallel work, GSD supports workspaces. A workspace is a git worktree with its own .planning/ directory. This enables running multiple milestones in parallel on different branches without any state interference.

Workspace Commands
# Create an isolated workspace for the NASA branch /gsd:new-workspace --branch nasa --name nasa-missions # List all active workspaces /gsd:list-workspaces # Remove a workspace when done /gsd:remove-workspace nasa-missions

Workstreams

Within a single branch, workstreams enable parallel phase execution. Multiple phases can be in different stages simultaneously: one phase in execution, another in verification, a third in planning. Workstreams maintain independent state within the same .planning/ directory.

Multi-Project Scale

The 190+ project count represents the PNW Research Series, where each project is a self-contained research document in the www/ directory. These projects span 13 Rosetta clusters (Ecology, Electronics, Infrastructure, Energy, Creative, Business, Vision, Broadcasting, Science, Music, Space, AI & Computation, Graphics) and total over 2,439 files and 600,000+ lines of content.

At this scale, GSD uses the Gastown convoy model: groups of projects are processed together in convoys, with the mayor-coordinator dispatching work via sling-dispatch to parallel agents. A typical convoy processes 50+ projects in a single session, with 4 agents executing in parallel and 5-6 tasks assigned per agent.

The Branch Rule

Work on dev, NOT main. Main receives merges from dev after human review. This rule exists because autonomous execution can produce large volumes of commits very quickly, and having a human gate between autonomous output and the stable branch prevents cascade failures. The developer reviews the diff between dev and main before merging, ensuring that autonomous execution produced the intended result.

10.9 The Quality System

Quality in GSD is not a single check at the end. It is a system of interlocking practices that operate at every stage of the lifecycle. These practices evolved from real failures — each one exists because its absence caused a measurable problem.

Carry-Forward Retrospectives

Every completed phase produces a RETROSPECTIVE.md. The retrospective captures what went well, what went poorly, and what should change for the next phase. The critical innovation: retrospectives are carry-forward. Each new phase's planning process reads the previous phase's retrospective and incorporates its lessons.

This creates a continuous improvement cycle where the system literally gets better with each phase. A bug pattern identified in phase 3's retrospective becomes a check in phase 4's planning. A performance issue identified in phase 7 becomes a non-functional requirement in phase 8. The knowledge accumulates within the milestone.

Carry-Forward Retrospective Example
# Phase 7 Retrospective ## What Went Well - Wave-based execution processed all 19 data sources in 3 waves - Atomic commits made debugging the NWS 503 issue trivial ## What Went Poorly - Task 2.1 was undersized — NWS API required more error handling than planned - Wave 2 took 3x longer than estimated due to API rate limiting ## Carry-Forward for Phase 8 - MUST: Add rate limiting estimates to all API-dependent task sizing - SHOULD: Include retry/backoff strategy in task success criteria - CONSIDER: Pre-flight API availability check before execution starts

Nyquist Validation

Borrowed from signal processing: the Nyquist theorem states that to reconstruct a signal, you must sample at twice its frequency. Applied to verification: to have confidence in a phase's completeness, you must verify at least twice as many aspects as you think are critical.

The nyquist-auditor agent applies this principle: for each phase, it identifies the critical outputs, doubles the verification points, and checks coverage. If a phase claims to implement 5 features, the auditor verifies 10+ aspects (the 5 features plus edge cases, error handling, integration points, documentation accuracy, and performance). Coverage gaps are reported and can be filled retroactively using /gsd:validate-phase.

Integration Checking

The integration-checker agent verifies cross-phase integrity: work done in phase N does not break work done in phase N-1. This catches a class of bugs that phase-level verification misses: interface changes, dependency conflicts, naming collisions, and behavioral regressions that only manifest when components are combined.

UAT Audit

The /gsd:audit-uat command performs a cross-phase audit of all outstanding verification items. It scans every phase's verification report and identifies items that were flagged as "partial" or "missed" but never resolved. This prevents verification debt from accumulating silently across a milestone.

Quality is tracked quantitatively across milestones through chain scores. Each milestone version receives a score from 1.0 to 5.0 based on execution quality, plan adherence, retrospective learning, and verification completeness. Release Pipeline Discipline

GSD enforces a strict release pipeline discipline: one release at a time, sequential, no batching. Each release carries forward lessons from the previous one. The temptation to batch multiple releases for efficiency is explicitly resisted because batching obscures which changes caused which outcomes, making retrospective learning impossible.

This discipline has been tested at scale: 58 releases were shipped in a single milestone (v1.49, the Seattle 360 Engine), each one sequential, each one incorporating carry-forward learning from its predecessor. The result was a monotonically improving quality curve across all 58 releases.

Execution Patterns That Work

After 190+ projects, certain patterns have proven reliably effective:

Research Before Planning

Always run a research phase before planning. Plans made without understanding the existing codebase fail at execution. The 10 minutes spent on RESEARCH.md saves hours of execution rework.

5-6 Tasks Per Agent

When dispatching work to parallel agents, assign 5-6 tasks per agent. Fewer tasks underutilize the context window. More tasks cause quality degradation as the agent loses focus.

Simple Tasks First

Sort tasks by complexity: execute simple tasks first. This builds momentum, creates known-good foundations for complex tasks, and catches environmental issues early when they're cheap to fix.

3-Level NASA SE Planning

The 3-level planning structure (goals, approach, task breakdown) works reliably at every scale. Skip any level and the plan degrades: goals without approach are vague, approach without tasks is unexecutable, tasks without goals are purposeless.

Atomic Commits Always

One task, one commit. No exceptions. The debugging cost of multi-task commits is higher than the small overhead of atomic commits. This has been validated across thousands of commits.

Context is RAM, State is Disk

Treat the AI's context window like volatile memory: it will be lost. Treat .planning/ files like persistent storage: they survive session boundaries. Write everything important to disk immediately. Never rely on context for recovery.

Execution Patterns That Failed

Equally important are the patterns that were tried and abandoned:

Agents That Both Plan and Execute

Early versions tried to have agents that created plans and then executed them. The failure mode: agents would silently modify their own plans during execution to match what they actually produced, making verification meaningless. Separation of planning and execution is now a hard rule.

Shared Mutable State Between Parallel Agents

Tried sharing a common state file between parallel agents. Race conditions, corrupted state, and unpredictable behavior. Agents now work in complete isolation within a wave.

Batching Releases for Efficiency

Tried batching 5-10 releases at once for speed. Impossible to determine which changes caused which outcomes. Carry-forward learning broke down. Sequential releases are slower but produce measurably better quality.

Planning Without Research

Skipping the research phase to save time. Plans made incorrect assumptions about the codebase, leading to execution failures and rework that took longer than the research would have. Research-before-planning is now the default.

The System's Trajectory

GSD is not a finished product. It is a living system that continues to evolve with each project. The command count grows when new workflow gaps are identified. The skill count grows when new domains are encountered. The agent count grows when new specialized capabilities are needed. The quality system improves through carry-forward retrospectives.

The next major milestone (v1.50, targeting April 21, 2026) will test GSD at its largest scale yet, integrating work from multiple branches (dev, nasa, artemis-ii) into a unified release. The system's ability to manage this integration will either validate the multi-branch architecture or reveal its limits. Either outcome will improve GSD.

The sophistication is in the process, not the technology.
