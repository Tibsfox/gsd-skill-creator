Executive Summary

The token optimization landscape has fundamentally changed in early 2026. Claude Opus 4.6 introduced 1 million token context windows, shifting the conversation from "how do we fit everything in?" to "how do we use this space wisely?" Meanwhile, the maturation of agent architectures—with subagent isolation, model routing, and skill-level budget management—has created entirely new categories of optimization that did not exist even six months ago.

This section covers both the foundational techniques that remain essential (compression, caching, RAG optimization) and the new strategies demanded by modern agentic systems: context-as-RAM thinking, fleet dispatch economics, effort-level tuning, and skill description budgets. The result is a comprehensive framework for organizations spending anywhere from $100 to $100,000 per month on LLM inference.

Key Finding: In production agentic systems, the largest cost driver is not individual prompts—it is context accumulation across agent turns. 2. 1M Context Window Strategies

The Context-as-RAM Principle

Claude Opus 4.6's 1 million token context window changes the fundamental nature of LLM interaction. In a traditional 4K-128K context, every token is precious real estate that must be carefully managed. At 1M tokens, context becomes more like working memory—everything in context is active knowledge that the model can reason over, connect, and synthesize.

This principle, which we call "context is RAM," has profound implications for how agent systems are architected:

  • No need to summarize prematurely: Raw data, full file contents, and complete conversation histories can remain in context without compression
  • Cross-reference becomes natural: With an entire codebase loaded, the model can identify patterns across files without explicit retrieval
  • Session continuity improves: A 1M context window can hold an entire day's worth of agentic work without hitting limits
  • RAG becomes optional for small-to-medium corpora: If your entire knowledge base fits in 500K tokens, you may not need RAG at all
Practical Scale: 1 million tokens is approximately 750,000 words, or roughly 3,000 pages of text. That is an entire codebase (most projects are 100K-500K tokens), plus documentation, plus conversation history, plus tool outputs—all simultaneously in context.

When to Use the Full Window vs. Keeping It Lean

More context is not always better. There is a cost dimension (every input token costs money), a latency dimension (larger contexts increase time-to-first-token), and a quality dimension (extremely large contexts can dilute attention to the most relevant information). The right strategy depends on the task:

Strategy Context Size When to Use Cost Implication
Full Window 500K-1M tokens Cross-codebase refactoring, architectural analysis, research synthesis $7.50-$15.00 input per call (Opus)
Focused Window 50K-200K tokens Implementation tasks, targeted debugging, document analysis $0.75-$3.00 input per call (Opus)
Lean Window 5K-50K tokens Classification, triage, simple generation, high-volume tasks $0.004-$0.04 input per call (Haiku)

Diminishing Returns at Scale

Research and practice show that context quality follows a characteristic curve. Performance improves steadily as relevant context increases, but returns diminish beyond a threshold that depends on the task complexity:

The Quality Curve

  • 0-50K tokens: Each additional piece of relevant context meaningfully improves output quality
  • 50K-200K tokens: Gains continue but flatten; focus on relevance over volume
  • 200K-500K tokens: Marginal gains for most tasks; useful primarily for cross-reference tasks
  • 500K-1M tokens: Primarily beneficial for whole-codebase operations and research synthesis

The Cost Curve

  • Linear cost: Token costs scale linearly regardless of quality gains
  • Break-even analysis: Each additional 100K tokens costs $1.50 on Opus; is that context worth $1.50 of quality improvement?
  • Latency penalty: Larger contexts increase TTFT by 1-3 seconds per 100K tokens
  • The right question: "Would a targeted search find this information faster and cheaper?"

Effective 1M Context Strategies

1. Layered Context Loading

Load context in concentric rings of relevance: core task context first (always loaded), supporting context second (loaded on demand), and background context last (loaded only for synthesis tasks).

# Layer 1: Core (always, ~5K tokens)
# - System prompt, tool definitions, task description

# Layer 2: Working Set (~50-200K tokens)
# - Files being modified, relevant test files
# - Recent conversation turns

# Layer 3: Reference (~200K-500K tokens)
# - Related modules, documentation, specs
# - Loaded when cross-referencing needed

# Layer 4: Full Codebase (~500K-1M tokens)
# - Entire repository for architectural decisions
# - Loaded only for refactoring/analysis tasks

2. Smart Pre-loading

For sessions expected to span many turns, pre-load relevant context once rather than fetching it repeatedly across turns. The cache hit benefit compounds over a long session.

# Turn 2-20: Zero additional reads needed

GSD Integration

The 50% Context Budget Rule: Even with 1M tokens available, GSD enforces a 50% context budget for each execution plan. At 0-30% usage, output quality is at peak. At 30-50%, quality remains good. Beyond 50%, quality begins to degrade as the model shifts to efficiency mode. Beyond 70%, output becomes rushed and unreliable. GSD addresses this by sizing plans to 2-3 tasks that complete within the 50% mark, splitting remaining work into new plans if needed. This rule applies regardless of total context window size—it is the percentage that matters, not the absolute number.

3. Subagent Context Savings

The Isolated Context Window

Subagents are the single most powerful token optimization technique in modern agentic systems. When a main conversation dispatches a subagent, that subagent runs in its own isolated context window. It can consume 200K tokens exploring files, running searches, reading documentation—and the main conversation's context grows by only the summary returned, typically 500-2,000 tokens.

Core Insight: A research task that would consume 200K tokens if performed inline in the main context takes zero tokens of main context when delegated to a subagent. The results are summarized back, consuming perhaps 1K tokens. That is a 200x context efficiency gain in the orchestrator's window.

4. Skill Description Budget Management

Why Skill Descriptions Matter for Token Budget

In Claude Code, skills are loaded into context so the model knows what capabilities are available. Each skill includes a description that tells Claude when to activate it. With 34+ skills in a production system, these descriptions consume meaningful context space on every single interaction—whether the skills are used or not.

The Hidden Cost: If each of 34 skills had a 500-character description, that is 17,000 characters (~4,250 tokens) consumed before the user even types their first message. Over a 50-turn conversation, that same skill context is sent 50 times via the API—totaling ~212,500 tokens just for skill descriptions. At Opus pricing, that is $3.19 per conversation, or $31.90 for a 10-conversation day, purely on skill metadata.

The 250-Character Cap

Claude Code enforces a 250-character maximum per skill description. This is not arbitrary—it is a budget constraint designed to keep skill overhead manageable at scale. Writing effective 250-character descriptions requires the same discipline as writing good commit messages: front-load the key use case, be specific about activation triggers, and eliminate filler words.

Budget Calculation and Overrides

The total budget for all skill descriptions is calculated as:

Skill Description Budget Formula

  • Default: 1% of context window OR 8,000 characters, whichever is larger
  • For 200K context (Sonnet/Haiku): max(2,000, 8,000) = 8,000 characters
  • For 1M context (Opus 4.6): max(10,000, 8,000) = 10,000 characters
  • Override: Set SLASH_COMMAND_TOOL_CHAR_BUDGET environment variable to specify a custom limit

With 34 skills at 250 characters each, total consumption is 8,500 characters—just over the 8,000-character fallback. In practice, most descriptions are shorter, keeping total usage around 5,000-6,000 characters.

Description Writing Best Practices

  1. Front-load the trigger: Start with WHEN this skill activates, not WHAT it does. "Use when deploying Docker containers" beats "Provides Docker best practices for..."
  2. Name specific keywords: Include the exact words users type: "'deploy', 'container', 'Dockerfile'"
  3. Be exclusionary: Say what the skill does NOT handle to prevent false activations
  4. Eliminate adjectives: "comprehensive," "advanced," and "best" consume characters without adding signal
  5. Test with real queries: Does the description cause the skill to activate for the right queries and stay silent for the wrong ones?

5. Extended Thinking and Effort Levels

Extended Thinking (Ultrathink)

Extended thinking allows Claude to reason through complex problems before generating a response. When activated, the model produces internal "thinking tokens" that are visible in the response but represent additional computational work. This is triggered in Claude Code skills via the ultrathink keyword in skill frontmatter or system prompts.

Cost Implication: Extended thinking tokens are billed as output tokens. On Opus 4.6 at $75/MTok output, a 10K-token thinking block costs $0.75 before the actual response even begins. Use extended thinking only when the quality improvement justifies this premium.

Effort Levels

Claude Code supports four effort levels that control reasoning depth. These can be set globally, per-skill, or per-agent via frontmatter configuration:

Effort Level Reasoning Depth Token Overhead Best For Model Availability
low Minimal reasoning, fast responses ~0x (no thinking) Triage, classification, simple routing All models
medium Standard reasoning, balanced ~1-2x thinking tokens Implementation, standard tasks All models
high Deep reasoning, thorough analysis ~3-5x thinking tokens Complex debugging, design decisions Sonnet, Opus
max Maximum reasoning, exhaustive ~5-10x thinking tokens Architecture, proofs, critical decisions Opus only

Model Override Per Skill/Agent

6. Token-Efficient Agent Patterns

Fleet Dispatch with Summary Aggregation

The fleet dispatch pattern sends multiple agents to work in parallel, each in an isolated context window. The orchestrator receives only structured summaries, not raw context. This is the most powerful token optimization pattern for multi-agent systems.

The Gastown Convoy Model

Named after the orchestration pattern used in production at scale, the Gastown model has three principles:

  1. The orchestrator stays lean: It dispatches work and aggregates results. It never reads files directly, never runs searches, never accumulates raw context. Its context contains only task descriptions and structured summaries.
  2. Workers consume tokens: Workers (subagents) do the heavy lifting—reading files, searching code, analyzing data. They consume tokens in their own isolated windows. When done, they produce a structured summary.
  3. Results are compressed at the boundary: The boundary between worker and orchestrator is a compression point. A worker might consume 200K tokens to produce a 2K summary. The orchestrator never sees the 200K.

Background Agents (run_in_background)

Background agents run asynchronously, with their results returned only when explicitly requested. This provides two optimization benefits: the main conversation can continue working while the background agent explores, and the background agent's full context never enters the main conversation—only the final result does.

# Background agent pattern in Claude Code:
# 1. Dispatch with run_in_background: true
# 2. Continue working on other tasks
# 3. Results arrive as notification
# 4. Only the summary enters main context

# Production example: 4 research agents
# Each consumes ~200K tokens exploring
# Main context receives 4 x 2K = 8K tokens
# vs. inline: 800K tokens in main context
# Context savings: 99%

Worktree Isolation

Agents with isolation: worktree operate in a separate git worktree, allowing them to make changes without affecting the main working directory. Beyond the git benefit, worktree isolation ensures the agent's file reads and modifications happen in a separate execution context, preventing context pollution in the orchestrator.

Context Compaction: PreCompact and PostCompact Hooks

Claude Code automatically compresses prior conversation turns as context approaches limits. The PreCompact and PostCompact hooks provide extension points for custom state preservation:

PreCompact Hook

Fires before compaction begins. Use it to:

  • Save critical state to disk (decisions made, files modified)
  • Write a session summary that will survive compaction
  • Checkpoint progress for crash recovery

PostCompact Hook

Fires after compaction completes. Use it to:

  • Re-inject critical context that may have been compressed away
  • Load state files written by PreCompact
  • Update progress tracking with post-compaction context size
State Preservation Pattern: Write a .session-state.json file in the PreCompact hook that captures: current task, files modified, decisions made, and next steps. The PostCompact hook injects this file back into context. This ensures critical information survives compaction events, even in very long sessions.

Token Budget Tracking

7. Prompt Compression Techniques

When Compression Still Matters

With 1M context windows, compression is less about fitting content into limited space and more about cost reduction and attention quality. Even when you have the room, sending 500K tokens of marginally relevant context dilutes model attention and costs $7.50 on Opus input alone. Compression remains valuable for high-volume workloads, cost-sensitive applications, and scenarios where every dollar matters.

Hard Prompt Methods (Token Removal)

These methods remove redundant tokens from natural language prompts while retaining semantic meaning:

  • LLMLingua / LLMLingua-2: Microsoft Research's open-source toolkit. Uses smaller language models to rank token importance, achieving up to 20x compression with minimal quality loss. Integrated with LlamaIndex and widely adopted in production RAG systems.
  • LongLLMLingua: Optimized variant for long-context scenarios with question-aware coarse-to-fine compression. Particularly effective for RAG applications where retrieved context is 10-50x larger than needed.
  • SelectiveContext: Uses lexical analysis to identify and preserve key tokens. Simpler than LLMLingua but effective for straightforward compression tasks.

Soft Prompt Methods (Embedding-Based)

  • GIST: Compresses prompts into "gist" tokens, achieving up to 26x compression. Requires model-specific training but produces extremely compact representations.
  • ICAE (In-Context Autoencoder): Learns compressed representations through training, compressing long contexts into "memory slots" with 4x compression at 90%+ performance retention.
  • 500xCompressor: Achieves extreme compression ratios up to 480x for specialized, repetitive tasks. Requires significant training investment.

8. Context Caching Strategies

How Prompt Caching Works

Prompt caching stores and reuses frequently accessed context at the provider level, offering 90% cost reduction and up to 85% latency improvement on cache hits. Claude processes prompts in order—tools, then system messages, then conversation messages—and caches content marked with cache_control breakpoints.

Caching Architecture for Agent Systems

In multi-agent architectures, caching strategy must account for what is stable (cacheable) vs. what changes each turn (not cacheable):

Context Layer Stability Cache Strategy Typical Size
Tool definitions Stable across session Cache with 1-hour TTL 5-15K tokens
System prompt + skills Stable across session Cache with 1-hour TTL 3-8K tokens
CLAUDE.md / project config Stable across session Cache with 1-hour TTL 2-5K tokens
Loaded file contents Semi-stable (changes on edit) Cache with 5-min TTL 10-200K tokens
Conversation history Grows each turn Previous turns cacheable Varies
Current user message Changes every turn Never cached 0.1-5K tokens

Cache Hit Rate Economics

The break-even point for caching depends on the cache write multiplier and expected number of reads:

Break-Even Analysis

  • 5-min cache (1.25x write): Breaks even after 2 cache reads within the 5-minute window. A 20-turn conversation reading the same system prompt 20 times saves 90% x 19 reads = 17.1x the base cost.
  • 1-hour cache (2.0x write): Breaks even after 3 cache reads. Useful for sustained agent sessions that span 30-60 minutes with many tool calls.
  • Target hit rate: >70% for cost-effective caching. Below 40%, caching can actually increase costs due to write overhead.

Prompt Structure for Maximum Cache Hits

Poor Cache Structure

// Variable content first (breaks cache)
User: "What about widget X?"

// Static content after (never cached)
System: Long instructions...
Tools: [tool definitions...]
Knowledge: [product catalog...]

Cache hit rate: ~0%

Optimal Cache Structure

// Static content first (cached)
Tools: [tool definitions...]      // cache_control
System: Long instructions...      // cache_control
Knowledge: [product catalog...]   // cache_control

// Variable content last
User: "What about widget X?"

Cache hit rate: 90%+

Key Constraints

  • Minimum cache size: 1,024 tokens per cache checkpoint
  • Maximum breakpoints: Up to 4 cache breakpoints per prompt
  • Cache lifetime: 5 minutes default (refreshed on each hit) or 1 hour optional
  • Workspace isolation: Since February 2026, caches are isolated per workspace, not per organization
  • Prefix matching: Cache requires exact prefix match—any change to cached content invalidates downstream cache

9. RAG Token Optimization

RAG in the 1M Context Era

The availability of 1M token context windows does not eliminate the need for RAG—it changes where the boundary sits. For corpora under ~500K tokens, direct context loading may outperform RAG. For larger corpora, RAG remains essential but can be optimized significantly.

When to Skip RAG

  • Entire corpus fits in <500K tokens
  • Queries require cross-document reasoning
  • High accuracy is critical (no retrieval errors)
  • Usage volume is low (cost of full context is acceptable)

When RAG Wins

  • Corpus exceeds 1M tokens
  • Queries are focused and specific
  • High volume (cost of full context per query is prohibitive)
  • Corpus changes frequently (embedding index is cheaper to update than re-loading)

The RAG Optimization Stack

Five layers of optimization, each building on the previous:

2. Retrieval Hybrid (dense + sparse, BM25 + embeddings) 20-30% fewer irrelevant chunks Medium
3. Re-ranking Cross-encoder filtering (retrieve 20, keep 5) 50-75% context reduction Medium
4. Compression LLMLingua or xRAG on retrieved chunks 60-80% further reduction High
5. Caching Cache stable knowledge base as context prefix 90% on cache hits Low

10. Intelligent Model Routing

The Routing Opportunity

For 70-80% of production workloads, mid-tier models perform identically to premium models. With the April 2026 pricing spread—Haiku at $0.80/MTok vs. Opus at $15.00/MTok, an 18.75x difference—the routing opportunity is enormous. A system that routes 60% of requests to Haiku, 30% to Sonnet, and 10% to Opus achieves 81% cost reduction compared to all-Opus routing.

Simple Short, factual, FAQ-like, classification Haiku 4.5 $2.40
Medium Multi-step, some reasoning, code generation Sonnet 4.6 $9.00
Complex Deep reasoning, cross-system analysis, architecture Opus 4.6 $45.00

2. Cascade Routing (Try Smaller First)

Attempt with the smallest model first, escalating only when confidence is low:

async def cascade_route(query, context):
    # Try Haiku first (cheap, fast)
    haiku_response = await call_haiku(query, context)
    if haiku_response.confidence > 0.85:
        return haiku_response  # ~70% of queries stop here

    # Escalate to Sonnet
    sonnet_response = await call_sonnet(query, context)
    if sonnet_response.confidence > 0.90:
        return sonnet_response  # ~25% need this tier

    # Final escalation to Opus
    return await call_opus(query, context)  # Only ~5% reach here

3. Agent-Level Routing

In agentic systems, routing happens at the agent dispatch level rather than per-query. Each agent is assigned a model based on its role in the workflow:

# GSD workflow with per-agent model routing:
agents:
  explorer:
    model: haiku        # Cheap file exploration
    effort: low
    tasks: ["find relevant files", "grep for patterns"]

  implementer:
    model: sonnet       # Good code generation
    effort: medium
    tasks: ["write code", "write tests"]

  architect:
    model: opus         # Deep reasoning
    effort: max
    tasks: ["design API", "review architecture"]

  reviewer:
    model: sonnet       # Solid analysis
    effort: high
    tasks: ["code review", "verify requirements"]

Routing Best Practices

  1. Start conservative: Route 50% to Haiku, measure quality, then increase the percentage
  2. Monitor per-route quality: Track accuracy and user satisfaction for each routing tier
  3. Use confidence scores: Let models self-report confidence for cascade routing decisions
  4. Combine with caching: Cache routing decisions for repeated query patterns
  5. Account for routing overhead: Routing adds 5-20ms latency; ensure this is acceptable
  6. Set up fallbacks: If a cheaper model fails, automatically escalate rather than returning poor results

12. Implementation Roadmap

Phased Optimization Approach

Phase 1: Quick Wins (Week 1-2)

Expected Savings: 30-50%
  • Deploy cost monitoring: Instrument all API calls with token tracking and cost attribution (1 day)
  • Enable prompt caching: Add cache_control to static context—system prompts, tool definitions, knowledge bases (2 days)
  • Optimize output verbosity: Instruct models to be concise, use structured output formats (1 day)
  • Use batch API: Migrate non-interactive workloads to batch processing for 50% discount (2 days)
  • Set effort levels: Configure effort: low for triage tasks, effort: medium for standard work (1 day)

Phase 2: Architecture (Week 3-6)

Expected Savings: 50-70% (cumulative)
  • Implement model routing: Deploy complexity-based routing across Haiku/Sonnet/Opus (1 week)
  • Add subagent isolation: Move exploration and research tasks to isolated subagent contexts (1 week)
  • Optimize RAG pipeline: Semantic chunking, hybrid retrieval, re-ranking (2 weeks)
  • Trim skill descriptions: Enforce 250-character cap, front-load triggers (2 days)

Expected Savings: 70-85% (cumulative)
  • Fleet dispatch with summary aggregation: Implement the Gastown convoy model for parallel work (2 weeks)
  • Context compaction hooks: Deploy PreCompact/PostCompact state preservation (1 week)
  • Token budget enforcement: Per-agent and per-convoy budget limits with hard stops (1 week)
  • Prompt compression: Integrate LLMLingua for high-volume RAG contexts (1 week)

The Token Optimization Checklist

  1. Measure first: Deploy cost monitoring before optimizing. You cannot improve what you do not measure.
  2. Route by complexity: Send simple tasks to Haiku ($0.80/MTok), implementation to Sonnet ($3/MTok), architecture to Opus ($15/MTok).
  3. Isolate agent contexts: Never let exploration tokens pollute the orchestrator's context window.
  4. Cache stable content: System prompts, tool definitions, and knowledge bases should be cached. Target >70% hit rate.
  5. Batch non-interactive work: 50% discount for free on any workload that can tolerate 24-hour latency.
  6. Match effort to task: Use effort: low for triage, effort: max only for architecture decisions.
  7. Trim skill descriptions: 250 characters, front-load triggers, eliminate filler.
  8. Compress RAG context: Retrieve broadly, re-rank aggressively, send only the top 5 chunks.
  9. Monitor output length: At $75/MTok output on Opus, every unnecessary paragraph costs money.
  10. Use the 50% context budget rule: Complete work within 50% of available context to maintain quality.
  11. Preserve state across compaction: PreCompact/PostCompact hooks prevent information loss in long sessions.
  12. Track per-project costs: Attribute costs to specific projects, teams, and work types for informed budgeting.

Common Pitfalls

  1. All-Opus everything: Using Opus for tasks that Haiku handles perfectly. This is the most expensive mistake.
  2. Ignoring output costs: Focusing on input optimization while generating verbose, unrestricted output.
  3. Inline exploration: Having the main agent read 50 files when a subagent could do it and return a summary.
  4. Premature compression: Compressing 5K-token contexts where the overhead exceeds the savings.
  5. Over-loading context: Loading an entire codebase for a task that needs 3 files.
  6. Ignoring cache TTL: Low-frequency workloads where cache write costs exceed read savings.
  7. Max effort everywhere: Running all agents at effort: max when most tasks need medium.
  8. Not monitoring: Flying blind on costs until the monthly bill arrives.
The Core Principle: Token optimization in the 1M context era is not about fitting things in—it is about spending tokens where they create the most value. A dollar spent on Opus reasoning about architecture creates more value than a dollar spent on Opus reading log files. Route the dollars to where they matter.
