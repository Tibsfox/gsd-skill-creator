14.1 The Memory Problem in Agentic Systems

Every agentic system eventually runs into the same wall: the agent forgets. Not because it lacks intelligence, but because its memory is fundamentally ephemeral. A conversation ends, a context window fills, a process restarts, and everything the agent learned — every decision rationale, every discovered constraint, every hard-won understanding of the codebase — vanishes. This is not a theoretical limitation. It is the single most practical obstacle to using AI agents for sustained, multi-session work.

Having operated an agentic system across 190+ research projects and months of continuous development, we can report that memory management is not an afterthought or a nice-to-have feature. It is the foundation on which everything else rests. Without reliable memory, skills cannot adapt, orchestration cannot coordinate, and agents cannot build on prior work. Every session starts from zero.

Context Windows Are Finite

Even with 1M-token context windows, there is always more project state than fits in context. A codebase with 21,000+ tests, 34 skills, 57 commands, and hundreds of source files simply cannot be loaded in its entirety. The agent must know what to load and when, which requires memory about the project that exists outside the context window itself.

Sessions End, Context Resets

A conversation that runs for 8 hours builds an enormous amount of implicit state: which approaches were tried and rejected, which files were modified, which constraints were discovered. When that session ends — whether by choice, by crash, or by reaching context limits — all of that state disappears unless it has been explicitly persisted. The next session starts with a blank slate and a system prompt.

Multiple Agents Need Shared State

When four parallel agents are executing different waves of a mission pack, they each operate in independent context windows. Agent A discovers that a CSS variable name changed. Agent B is about to use the old name. Without shared state outside any individual context window, coordination is impossible. The filesystem becomes the shared memory bus.

Conversation Memory vs. Persistent Knowledge

There is a fundamental gap between what an agent remembers within a conversation (everything said so far) and what it knows across conversations (nothing, by default). Bridging this gap is the entire memory problem. The solution is not to make context windows infinitely large — it is to build structured persistence systems that load the right knowledge at the right time.

Why File-Based Persistence Wins

After extensive experimentation, a clear pattern emerges: file-based persistence outperforms every alternative for agentic memory. This is counterintuitive if you come from a database-centric background, but the reasons are compelling:

Property File-Based Database-Based In-Context Only
Git integration Native — version controlled, diffable, branchable Requires export/import Not applicable
Agent readability Direct — agents read files natively Requires query tools Already in context
Human readability Open in any editor Requires client Visible in transcript
Cross-session durability Survives everything Survives everything Lost on session end
Multi-agent access Any agent can read any file Connection required Only the current agent
Crash recovery Files survive crashes Transaction-dependent Lost completely
Portability Copy the directory Migration required Not portable

The filesystem is the universal interface. Every agent can read files. Every tool can write files. Git tracks changes. Humans can inspect and edit directly. There is no impedance mismatch, no connection pooling, no schema migration. A markdown file in .planning/ is simultaneously a memory store, a documentation artifact, and a version-controlled record of decisions.

Practitioner Insight

We maintain 50+ memory files, dozens of planning documents, structured handoffs, and state machines — all as plain files on disk. This is not a prototype. It is a production system that has supported months of continuous operation across hundreds of sessions. The filesystem is the most reliable, most debuggable, and most portable persistence layer available to agentic systems today.

14.2 CLAUDE.md — The Instructions Hierarchy

Before an agent processes its first message, before any skill loads or any memory file is consulted, there is an instructions layer that shapes everything the agent does. In Claude Code, this layer is the CLAUDE.md hierarchy — a cascade of configuration files that define project conventions, user preferences, local overrides, and enterprise policies. Understanding this hierarchy is essential because instructions are the highest-priority form of memory: they load automatically, they are always in context, and they override default behavior.

The Four-Level Hierarchy

Instructions cascade from most specific to most general, with more specific instructions taking precedence when conflicts arise:

  • Enterprise / Managed Instructions — Highest priority. Set by organization administrators. Cannot be overridden by users or projects. Used to enforce security policies, compliance requirements, and organizational standards. Developers cannot modify or see the full content of these instructions.
  • Project Instructions (.claude/CLAUDE.md) — Checked into the repository. Shared by all contributors. Defines the project's tech stack, commit conventions, build commands, file layout, and architectural constraints. This is where team-wide agreements live.
  • User Instructions (~/.claude/CLAUDE.md) — Global to the user, across all projects. Defines personal preferences like verbosity level, commit message style, or preferred testing frameworks. Applies everywhere unless overridden by project-level instructions.
  • Local Instructions (.claude/CLAUDE.local.md) — Gitignored. Per-developer, per-machine configuration. Used for local paths, environment-specific settings, or experimental overrides that should not be committed to the repository.

What Belongs Where

Project Instructions

Committed to the repo, shared by the team

  • Tech stack and language versions
  • Build, test, and lint commands
  • Commit message conventions
  • Key file locations and architecture
  • Boundary rules (e.g., "src/ never imports desktop/")
  • Important caveats about the codebase
# Example: .claude/CLAUDE.md

## Tech Stack
- TypeScript (src/), Rust (src-tauri/), GLSL (shaders)
- Build: `npm run build`
- Test: `npm test` (Vitest, 21,298 tests)

## Commit Convention
- Conventional Commits: <type>(<scope>): <subject>
- Imperative mood, lowercase, no period

## Strict Boundary
- src/ never imports desktop/@tauri-apps/api
- desktop/ never imports Node.js modules

User Instructions

Global preferences, all projects

  • Preferred verbosity level
  • Default editor and tool preferences
  • Personal coding style preferences
  • How to handle commit authorship
  • Session startup behavior
# Example: ~/.claude/CLAUDE.md

## Preferences
- Lowest verbosity in conversation
- Full verbosity in documentation
- Never suggest pruning docs
- Ask what I'm working on first
- Autonomous execution preferred

Path-Specific Rules with Glob Patterns

Instructions can be scoped to specific file paths using glob patterns. This enables rules that only apply when working in certain parts of the codebase:

# In CLAUDE.md:
# When working in the test directory:
# *.test.ts files should use describe/it blocks, not test()
# Mock external services, never hit real APIs

# When working in src-tauri/:
# All Rust code must handle errors with Result, never unwrap()
# Use serde for serialization

This is especially powerful in monorepos or projects with distinct subsystems. The instructions for the Rust backend can differ entirely from the instructions for the TypeScript frontend, and both coexist in the same CLAUDE.md file.

Modular Instructions with @include

As instruction files grow, the @include directive allows splitting them into focused modules:

# CLAUDE.md
@include .claude/instructions/commit-rules.md
@include .claude/instructions/testing-conventions.md
@include .claude/instructions/architecture-boundaries.md

Each included file is loaded as if its contents were inline at the include point. This keeps the root CLAUDE.md concise while allowing detailed conventions to be maintained in dedicated files. It also enables conditional inclusion — different team members can include different instruction sets in their local configuration.

Instructions Are Not Memory

A common mistake is treating CLAUDE.md as a memory dump — recording everything the agent should know. This is wrong. Instructions define behavior: how the agent should act, what conventions to follow, what rules to enforce. Factual knowledge about the project (current state, recent decisions, open questions) belongs in the memory system, not in instructions. Overloading CLAUDE.md with project state makes it bloated, unstable, and hard to maintain.

14.3 Auto Memory System

Claude Code provides a built-in persistent memory system that bridges the gap between ephemeral conversations and long-term knowledge. Unlike the instructions hierarchy which defines behavior, the memory system stores knowledge — facts about the user, the project, decisions made, and lessons learned. This memory persists across conversations automatically, loading relevant memories based on semantic matching against the current context.

The system supports five distinct memory types, each serving a specific purpose. Understanding these types and using them correctly is the difference between a memory system that helps and one that becomes noise.

User User Memories

User memories capture who the developer is — their expertise, background, and working style. These memories shape how Claude approaches problems and communicates solutions. They persist across all projects because the user is the same person regardless of which codebase they are working in.

Expertise Profiling

Recording a developer's skill set allows the agent to calibrate its explanations and suggestions. A "senior Go developer, new to React" will receive different guidance than a "junior developer learning both." The agent can skip basics in areas of expertise and provide more scaffolding in unfamiliar territory.

---
name: developer-expertise
description: Developer skill profile for response calibration
type: user
---
Senior TypeScript/Rust developer. 20+ years systems programming.
Familiar with: GPU compute, CUDA, shader languages, embedded.
New to: Unison language (actively exploring).

Why: Calibrate explanations - skip basics in TS/Rust,
provide more context for Unison patterns.

How to apply: Default to expert-level code suggestions.
Explain Unison concepts with comparisons to familiar langs.

Working Style

Capturing preferences about communication, workflow, and interaction style prevents the agent from repeatedly violating the developer's preferences. This is especially important because these preferences are often non-obvious and specific to the individual.

---
name: working-style
description: Communication and workflow preferences
type: user
---
Lowest verbosity in conversation.
Full verbosity in documentation and research output.
Autonomous execution preferred - don't stop for input.
Never suggest pruning documentation.

Why: Developer values efficiency in dialogue but
thoroughness in artifacts.

How to apply: Keep conversational responses concise.
Write exhaustively in docs, research, and release notes.

Feedback Feedback Memories

Feedback memories are corrections and confirmations — what to do and what not to do, learned from actual interaction. These are the most operationally important memories because they prevent the agent from repeating mistakes. Every time the developer says "don't do that" or "yes, that was the right call," it is a feedback memory waiting to be saved.

Corrections

When the agent does something wrong, the correction should be captured immediately so it never happens again. The key is recording not just the rule but why it matters.

---
name: no-planning-commits
description: Never commit .planning/ files to git
type: feedback
---
HARD RULE: NEVER git add/commit/push .planning files.

Why: .planning/ is gitignored by design. It contains
local-only state, personal notes, and IP that must
never reach the remote repository.

How to apply: Before any git add, check that no paths
start with .planning/. If the user asks to commit
planning files, warn them and refuse.

Confirmations

Positive feedback is equally valuable. When the agent makes a good decision, recording it reinforces the pattern for future sessions.

---
name: release-pipeline-quality
description: Release quality over quantity approach
type: feedback
---
ONE release at a time, sequential, with
carry-forward retrospectives. No batching.

Why: User confirmed this approach produces higher
quality output. Each release improves on the last.

How to apply: Never batch multiple releases.
Complete one fully before starting the next.
Include retrospective in every release.

Project Project Memories

Project memories capture ongoing work context — active initiatives, architectural decisions, constraints, and current state. These are the most dynamic memories, changing as the project evolves. They answer the question: "What is happening in this project right now?"

Active Initiative Tracking

---
name: artemis-ii-launch
description: Artemis II mission tracking on artemis-ii branch
type: project
---
Artemis II LAUNCHED on artemis-ii branch (2026-04-01).
Subtitle: "The Space Between the Moon and the Earth"
pgvector DB: maple@tibsfox, schema artemis, 6 tables.
1,087 pages loaded. sweep.py for hourly updates.
35+ live pages. Day 8 operations.

Why: Primary active project. All work happens here.
Branch context critical for every operation.

How to apply: Default to artemis-ii branch.
Check sweep.py status before page updates.
Never merge to main without explicit request.

A critical rule for project memories: convert relative dates to absolute dates. "Last week we decided..." becomes meaningless after the next session. "On 2026-03-29 we decided..." remains useful forever. This seems obvious but is routinely violated.

Reference Reference Memories

Reference memories are pointers to external systems — where to find information that lives outside the codebase. They do not contain the information itself; they contain the directions for finding it.

External System Pointers

---
name: gsd2-setup
description: GSD-2 CLI location and configuration
type: reference
---
gsd-pi v2.59.0, standalone CLI.
Commands: /gsd auto + doctor + KNOWLEDGE.md
Coexists with GSD v1.

Why: Developer uses both GSD versions. Need to know
which CLI to invoke and where it's installed.

How to apply: When GSD commands are needed, check
which version is appropriate for the task.

The MEMORY.md Index

Every conversation begins by loading MEMORY.md — a single index file that provides one-line pointers to all memory files. This is the most constrained artifact in the entire memory system: it has a hard limit of 200 lines. Everything that exceeds 200 lines is truncated and lost to the session.

This constraint is fundamental, not incidental. Loading all memory into every session would be wasteful and noisy. The index provides just enough context for the agent to know what memories exist and when to load them in full. The design forces discipline: each memory gets a single line, organized by semantic topic.

MEMORY.md Structure

# Memory: project-name

## HOT -- Current State (always in context)
**Active initiative** -- summary
- [Detail link](detail-file.md) -- one-line description
- Key fact or constraint

## WARM -- Reference (session start)
### Completed Milestones (Condensed)
- Milestone summary with key numbers

### Standing Rules
- **HARD RULE**: critical behavioral constraint
- Session startup protocol description

## COLD -- Archive
- Archived project context (rarely needed)

The HOT/WARM/COLD organization mirrors CPU cache hierarchy. HOT memories are always relevant. WARM memories are useful at session start. COLD memories are archived context that is rarely needed but preserved for reference. The agent reads HOT first, consults WARM as needed, and only reaches into COLD when specifically asked.

Operating at 50+ memory files, we have found that the 200-line limit on MEMORY.md is simultaneously the system's greatest constraint and its greatest discipline mechanism. It forces ruthless prioritization. If a memory is not important enough to earn a line in the index, it is either not important enough to save, or it needs to be consolidated with another memory. The limit prevents the index from becoming a dumping ground.

14.4 What NOT to Save

The memory system's effectiveness depends as much on what you exclude as what you include. Every unnecessary memory adds noise, consumes index lines, and increases the chance that the agent acts on stale information. The single most common mistake in agentic memory systems is saving too much.

Code Patterns

Don't save: "The authentication module uses JWT with RS256 signing."

Why not: The code itself is the authoritative source. The agent can read auth.ts directly. A memory about code patterns becomes stale the moment the code changes, and the agent has no way to know the memory is outdated without reading the code anyway.

Instead: Save a reference memory pointing to where authentication is implemented, if the location is non-obvious.

Git History

Don't save: "In commit abc123 we refactored the database layer."

Why not: Git is a better database for git history than memory files. The agent can run git log, git diff, and git blame. Duplicating this information in memory creates a shadow history that diverges from the real one.

Instead: Save the decision behind the refactor if it is non-obvious from the commit message.

Debugging Solutions

Don't save: "Fixed the race condition by adding a mutex around the counter."

Why not: The fix is in the code. If someone asks about it, they can read the code and the commit message. Saving debugging narratives creates archaeological layers of "we tried X, then Y, then Z" that are useless after the fix ships.

Instead: If the debugging revealed a systemic issue (e.g., "this library has a known concurrency bug"), save that as a project constraint.

CLAUDE.md Content

Don't save: "The project uses Conventional Commits with imperative mood."

Why not: This is already in CLAUDE.md, which loads automatically into every session. Duplicating it in memory creates two sources of truth that can diverge. When the convention changes, you have to update both places.

Instead: Keep conventions in CLAUDE.md exclusively. Use memory only for things that don't belong in instructions.

Ephemeral Task Details

Don't save: "Currently debugging the WebSocket timeout issue on line 342."

Why not: This information is useful for exactly one session. By the next session, the issue is either fixed (making the memory stale) or the line number has changed (making the memory wrong). Ephemeral task state belongs in the conversation, not in persistent memory.

Instead: If the task spans sessions, use a context handoff document (Section 14.5) rather than a memory file.

Obvious Tooling

Don't save: "Use npm install to install dependencies."

Why not: The agent already knows standard tooling. Saving obvious facts wastes index lines and implies that the agent needs to be taught things it was trained on. Save only non-obvious tooling quirks.

Instead: Save unusual build requirements, like "must run the install script before building" or "requires CUDA 12.1 specifically."

Decision Rule

Before saving a memory, apply this test: "If this memory were deleted, would the agent be unable to recover the information from the codebase, git history, or CLAUDE.md?" If the answer is no — if the information can be recovered from existing sources — don't save it. Memory is for knowledge that exists nowhere else: decisions, preferences, constraints, and cross-project context.

14.5 Context Handoffs — Session Continuity

Context handoffs are the bridge between sessions. When a session ends — whether planned or forced by context limits — the handoff document captures everything the next session needs to continue seamlessly. This is not a memory file; it is a structured transfer document designed for a specific moment in time. Memories store long-term knowledge. Handoffs store short-term session state.

The distinction matters operationally. A memory file about "branch strategy" is relevant for months. A handoff document about "what I was doing 30 minutes ago when the context window filled" is relevant for exactly one resumption. Mixing the two degrades both.

The Handoff Structure

Effective handoffs follow a rigid structure. We developed this format through trial and error across hundreds of sessions, and every field exists because its absence caused a resumption failure at some point:

# HANDOFF — [Project/Session Identifier]
## Date and Duration
Session started: 2026-04-06 14:30 PST
Session ended: 2026-04-06 23:00 PST (8.5 hours)

## What Was Accomplished
- [Concrete deliverable 1 with metrics]
- [Concrete deliverable 2 with metrics]
- [Concrete deliverable 3 with metrics]

## Current State
- Branch: artemis-ii
- Last commit: [hash] [message]
- Build status: clean / broken (details)
- Test status: 21,298 passing, 0 failing

## Decisions Made
1. **[Decision]**: [What was decided]
   Rationale: [Why this choice]
   Alternatives considered: [What was rejected and why]
   Revisit if: [Conditions that would change this decision]

## Open Questions
- [ ] [Question 1] — blocking / non-blocking
- [ ] [Question 2] — needs user input / can decide independently

## What To Do Next
1. [Immediate next step]
2. [Following step]
3. [Step after that]

## Gotchas and Warnings
- [Thing that will bite you if you don't know about it]
- [Constraint that isn't obvious from the code]

Real-World Handoff Example

Our HANDOFF-ARTEMIS-II-SESSION-003.md captures 8.5 hours of mission operations work. It records that 35+ live pages were updated via sweep.py, that the pgvector database holds 1,087 loaded pages across 6 tables, that math sections 8-10 were developed (live telemetry, O2O Shannon entropy, circadian drift analysis), and that specific research threads were started but not completed. Without this handoff, the next session would have spent its first hour rediscovering what was already done.

What Makes a Good Handoff

  • Concrete, not vague: "Wrote 3 sections, 4,200 words" not "made progress on the report"
  • Verifiable state: "Last commit abc123, build clean, 21,298 tests passing" not "everything works"
  • Decision rationale: Record why, not just what. Future you will ask "why did we..."
  • Revisit conditions: "Revisit if the API changes" prevents decisions from becoming dogma
  • Gotchas first: Put warnings at the top of mind, not buried at the bottom

The Resume Protocol

  1. Read the handoff — Load the full document, not just the summary
  2. Verify state — Check git status, branch, build, tests. The handoff might be stale if someone else worked between sessions
  3. Check for drift — Compare handoff's "current state" against actual state. If they diverge, the handoff is partially outdated
  4. Ask the user — "The handoff says X. Are we continuing from there, or has the plan changed?"
  5. Continue — Pick up from "What To Do Next" in the handoff

When to Create Handoffs

Handoffs should be created in three situations:

Planned Session Ends

The user says "I'm done for the day" or "let's wrap up." Create a thorough handoff while the full context is still available. This is the easiest case because there is time to be comprehensive.

Context Window Approaching Limits

When the conversation is getting long and the agent starts compacting context, it is time to write a handoff before information is lost. The worst handoffs are written after the context has already degraded. Proactive handoff creation is a skill that develops with practice.

Task Switching

When the user needs to switch to a different project or branch mid-stream, create a handoff for the current work before switching. This is essentially a bookmark that preserves the full context of the interrupted work.

14.6 Beads State — Crash-Recoverable Persistence

Handoffs are written by agents for agents. But what about state that needs to be machine-readable, atomically updated, and recoverable after crashes? This is where structured state persistence comes in. The Beads State pattern — named for its use in the Gastown orchestration chipset — provides git-friendly, crash-recoverable JSON state files that track agent identities, work items, and checkpoint/restore state.

Design Principles

Git-Friendly JSON

State files are JSON with deterministic key ordering and consistent formatting. This means git diff shows meaningful changes, not noise from reordered keys or reformatted whitespace. Every state change produces a clean, reviewable diff. The files can be committed alongside code, creating a version-controlled history of state transitions.

{
  "agent_id": "cedar-executor-003",
  "state": "executing",
  "phase": "72",
  "wave": 2,
  "checkpoint": {
    "completed_items": ["72.1", "72.2", "72.3"],
    "current_item": "72.4",
    "started_at": "2026-04-06T14:30:00Z",
    "last_update": "2026-04-06T16:45:00Z"
  }
}

Agent Identity Persistence

When orchestrating multiple agents, each needs a stable identity that persists across process restarts. Beads State assigns each agent a unique identifier tied to its role, and records its current state so that if the process crashes and restarts, the agent can resume from its last checkpoint rather than starting over.

In our system, agent identities follow a naming convention: [role]-[function]-[sequence]. This makes state files self-documenting — you can tell what cedar-executor-003 is doing without reading anything else.

Atomic Writes Prevent Corruption

The core safety mechanism is atomic writes: state is written to a temporary file first, then renamed to the target path. On POSIX systems, file rename is an atomic operation — it either succeeds completely or fails completely. There is no state where the file is half-written.

// Atomic state write pattern
async function writeState(path: string, state: object): Promise<void> {
  const tmp = path + '.tmp.' + process.pid;
  const content = JSON.stringify(state, null, 2) + '\n';
  await fs.writeFile(tmp, content, 'utf-8');
  await fs.rename(tmp, path);  // Atomic on POSIX
}

This pattern means that a crash during a write leaves either the old state (rename didn't happen yet) or the new state (rename completed). It never leaves a corrupted intermediate state. For agentic systems where crashes are not exceptional but expected — context windows fill, processes are killed, machines restart — this guarantee is essential.

Work Item Tracking with Checkpoint/Restore

Beads State tracks work items through their lifecycle, providing checkpoint and restore capabilities that enable crash recovery:

queued → assigned → executing → checkpoint → completed

Each transition is recorded with a timestamp and the agent that performed it. If an agent crashes while executing, the recovery protocol reads the state file, finds the last checkpoint, and resumes from there. Items that were assigned but never checkpointed are returned to the queue. Items that were checkpointed can be restored to their checkpoint state.

Recovery Protocol

Crash Recovery Steps

  1. Read state file — If it exists and parses correctly, the last known state is recovered
  2. Check agent identity — Verify the agent claiming this state matches the recorded identity
  3. Scan for orphaned work — Items in "executing" state with no active agent are orphaned and returned to the queue
  4. Restore from checkpoint — If the agent's last action was a checkpoint, restore that state and continue
  5. Verify filesystem — Check that files referenced in the state actually exist on disk (they might have been partially written)
  6. Resume or restart — If the state is consistent, resume. If not, log the inconsistency and restart the work item from scratch

Integration with Orchestration

Beads State integrates with the orchestration patterns described in Section 13. The Mayor-Coordinator reads state files to understand which agents are alive and what they are working on. The Sling-Dispatch pattern writes state files when assigning work. The Witness-Observer reads state files to detect stalled agents. The state files are the shared memory that makes distributed agent coordination possible without a central database.

14.7 Thread Management — Cross-Session Work

Threads are persistent context containers for work that spans multiple sessions but does not warrant a full project. Where handoffs capture the state at a single session boundary, threads maintain a continuous narrative across many sessions. Where memory files store atomic facts, threads store evolving context for a specific initiative.

When to Use Each Persistence Mechanism

Mechanism Scope Lifetime Best For
Memory files Single fact or rule Indefinite Preferences, constraints, standing rules
Handoffs Session boundary Until consumed Session continuity, task switching
Threads Multi-session initiative Weeks to months Research tracks, feature branches, ongoing investigations
Beads State Machine state Duration of operation Agent coordination, work item tracking, crash recovery
CLAUDE.md Behavioral rules Project lifetime Conventions, build commands, architectural boundaries

Thread Lifecycle

Threads follow a creation-resumption-archival lifecycle:

Creation

A thread is created when a piece of work is identified as spanning multiple sessions. The creation captures the initial context: what the work is, why it matters, what the current state of knowledge is, and what the expected outcomes are. Thread names should be descriptive and stable — they become the identifier used to resume work across sessions.

# Thread: RH-Montgomery-Dyson Research
## Created: 2026-04-03
## Status: active

### Context
Investigating eigenvalue spacing patterns in
random matrices (Montgomery-Dyson) and their
connection to Riemann Hypothesis zeta zeros.

### Current State
- Literature review: 12 papers read
- Numerical experiments: GUE ensemble generated
- Open question: pair correlation convergence rate

### Next Actions
- Reproduce Montgomery's 1973 pair correlation
- Compare with actual zeta zero spacing data

Resumption

Resuming a thread loads the thread document and picks up from the last recorded state. Unlike handoffs, which are consumed and discarded, threads are updated — each session appends its contributions to the thread's history. This creates a running log of progress that becomes valuable documentation in its own right.

Archival

When a thread's work is complete, it is archived rather than deleted. The archive preserves the decision history, research findings, and lessons learned. Archived threads can be referenced later if the topic resurfaces. In our system, archived threads are moved to a completed/ subdirectory with a final status summary.

Thread vs. Handoff: The Decision Rule

The distinction is simple: a handoff is a bookmark; a thread is a notebook. If you are stopping work temporarily and will resume exactly where you left off, create a handoff. If you are tracking an ongoing initiative that will evolve over many sessions with different subtasks and changing priorities, create a thread. In practice, many handoffs get promoted to threads when work that was expected to be brief turns out to be extended.

14.8 State Machines in Agentic Workflows

Agentic workflows are, at their core, state machines. The agent is always in some state, and its available actions depend on that state. Making this state machine explicit — rather than implicit in the conversation flow — is one of the most impactful architectural decisions in agentic system design. Explicit state machines make workflows debuggable, recoverable, and auditable.

GSD Workflow States

The GSD (Get Shit Done) orchestrator implements a five-state workflow that governs all project work:

discuss → plan → execute → verify → ship

Discuss

Gather context, clarify requirements, identify constraints. The agent asks questions, surfaces assumptions, and ensures alignment before planning begins. No code is written in this state. No plans are created. The only output is shared understanding.

Transition guard: User confirms that requirements are understood and scope is agreed upon.

Plan

Break work into phases, create detailed plans with verification criteria, analyze dependencies, and estimate effort. Each phase gets a PLAN.md with specific deliverables, test criteria, and implementation steps.

Transition guard: Plan document exists and has been reviewed. Dependencies are identified.

Execute

Implement the plan. Write code, run tests, commit changes. Execution follows the plan unless a blocking issue is discovered, in which case the state machine transitions back to plan (replanning) rather than improvising.

Transition guard: All plan items are implemented. Build passes. Tests pass.

Verify

Validate that the implementation meets the requirements defined in the discuss phase and the criteria specified in the plan phase. Run UAT, check edge cases, verify performance. This is adversarial — the goal is to find problems, not confirm success.

Transition guard: All verification criteria pass. No blocking issues found.

Ship

Create PR, run review, merge to target branch, create release notes. This is the final gate. Code that reaches this state has been discussed, planned, executed, and verified. Shipping is a formality, not a decision point.

Transition guard: PR approved, CI passes, release notes written.

STATE.md as Single Source of Truth

The current workflow state is recorded in .planning/STATE.md. This file is the single source of truth for "where are we in the workflow?" Every session reads this file at startup to determine the current state and available transitions.

# Project State

## Current Phase: 72
## State: execute
## Branch: dev

### Phase 72: Trust Relationship Provider
- Milestone: v1.50
- Started: 2026-04-05
- Plan: .planning/phases/72/PLAN.md
- Status: Wave 2 of 3 complete
- Blocking: None
- Next: Execute wave 3 (integration tests)

The state file serves multiple purposes simultaneously: it tells the agent what to do, it tells the human what is happening, it provides crash recovery (if the agent restarts, it reads STATE.md and knows where to resume), and it enables audit (the git history of STATE.md is a complete log of workflow transitions).

Recovering from Invalid States

State machines can enter invalid states through crashes, manual edits, or bugs. The recovery strategy is: always fall back to the last valid state. The GSD health check (/gsd-health) diagnoses state inconsistencies and suggests repairs:

Common Invalid States

  • State says "execute" but no PLAN.md exists: State was advanced without creating a plan. Fix: revert to "plan" state.
  • State says "verify" but build is broken: Execution did not complete successfully. Fix: revert to "execute" state and fix the build.
  • State says "ship" but there are uncommitted changes: Execution artifacts were not committed. Fix: commit first, then continue shipping.
  • Phase number in state doesn't match any roadmap entry: The roadmap was renumbered without updating the state. Fix: reconcile phase numbers.

How State Drives Agent Behavior

The state machine is not just a tracking mechanism — it actively shapes what the agent does. An agent in "discuss" state will ask questions and refuse to write code. An agent in "execute" state will follow the plan and refuse to change scope. An agent in "verify" state will look for problems rather than writing new features. The state machine provides behavioral guardrails that prevent the agent from jumping ahead, going backward, or doing the wrong kind of work for the current phase.

This is more disciplined than how most humans work, and that is precisely the point. Humans naturally blend planning and execution, mixing design decisions into implementation. Agents amplify this tendency because they can generate code very quickly. The state machine forces separation of concerns: think first, plan second, build third, check fourth, ship fifth. The discipline is externalized into the state machine rather than relying on the agent's self-control.

14.9 The Memory Lifecycle

Memories are not static artifacts. They are born, they age, they become stale, they are updated, and eventually they are archived or deleted. Treating memories as immutable records — write once, read forever — leads to a memory system that becomes progressively less trustworthy over time. Active lifecycle management is essential.

Creation: Save Immediately

When you learn something useful — a user preference, a project constraint, a decision rationale — save it immediately. Do not wait until the end of the session. Context degrades as conversations get longer, and memories written late in a session are often incomplete or imprecise. The best memory is one written at the moment of learning, when the context is fresh and the reasoning is clear.

Rule: If the information would be valuable in the next session, save it in this session. Do not defer.

Validation: Verify Before Acting

Memories describe the world as it was when the memory was created. The world changes. A memory that says "the config module is in src/config/index.ts" might be wrong if someone moved it last week. Before acting on a memory that references specific files, functions, or system states, verify it against the current codebase.

Rule: If a memory names a file or function, verify it exists before using it. Trust but verify.

Staleness: Memories Decay

Every memory has a half-life. Project memories decay fastest (current state changes constantly). User memories decay slowest (people's expertise and preferences are relatively stable). Feedback memories are somewhere in between (conventions evolve). Actively monitor for stale memories and update or remove them.

Rule: If a memory conflicts with observable reality, update the memory, not your understanding of reality.

Cleanup: Prune the Dead

Memories that are wrong are worse than no memory at all. A wrong memory causes the agent to take confident action in the wrong direction, which is harder to detect and correct than starting from ignorance. Regularly review memories and remove ones that are outdated, superseded, or no longer relevant.

Rule: A smaller, accurate memory system outperforms a larger, partially-stale one every time.

The "Never Guess from Memory" Rule

This is the most important operational rule in the memory system: if a memory names a specific file, function, API endpoint, or system state, verify that it still exists before acting on it. Memories are hints, not facts. They point you in the right direction, but the codebase is the source of truth. An agent that blindly trusts its memories will eventually try to import a module that was renamed, call a function that was deleted, or write to a file that was moved. Always verify.

This rule applies even to memories that were saved minutes ago. In a multi-agent system where several agents are modifying the codebase simultaneously, a file that existed when the memory was created might have been moved by another agent since then.

14.10 Scaling Memory Across Projects

A single project with a handful of memory files is manageable by intuition. Fifty memory files across multiple projects, with cross-project dependencies and shared user preferences, requires deliberate organization. We operate at this scale, and the patterns described here were discovered by necessity.

Per-Project Memory Directories

Claude Code stores project-specific memories in ~/.claude/projects/[project-path]/memory/. Each project gets its own directory, its own MEMORY.md index, and its own namespace for memory files. This isolation is critical: memories about the Rust backend of one project should not pollute the context of a TypeScript-only project.

~/.claude/
  CLAUDE.md                          # Global user instructions
  projects/
    -media-foxy-ai-project-alpha/
      memory/
        MEMORY.md                    # 200-line index for project alpha
        trust-system.md
        branch-strategy.md
        ...
    -media-foxy-ai-project-beta/
      memory/
        MEMORY.md                    # 200-line index for project beta
        api-architecture.md
        deployment-constraints.md
        ...

Cross-Project Patterns

User memories are global — they apply across all projects because the user is the same person everywhere. This is where preferences, expertise profiles, and behavioral rules live. They are stored in ~/.claude/CLAUDE.md (instructions) and the user-level memory directory.

However, some knowledge genuinely spans projects without being user-specific. A team's coding conventions, a shared library's quirks, or an infrastructure constraint that affects multiple services — these need to be duplicated or referenced across project memory directories. The practical approach is to keep one canonical memory file and reference it from other projects' MEMORY.md indices:

## Cross-Project References
- **Shared infra constraints:** See project-alpha/memory/infra-constraints.md
- **Team conventions:** Defined in ~/.claude/CLAUDE.md, apply everywhere

Memory Organization at Scale

At 50+ memory files, flat organization breaks down. MEMORY.md becomes the organizational backbone, using semantic sections to group related memories:

Organizational Pattern for Large Memory Systems

# Memory: gsd-skill-creator

## HOT -- Current State (always in context)
**Active project 1** -- one-line summary
- [detail](file.md) -- pointer
**Active project 2** -- one-line summary
- [detail](file.md) -- pointer

### Core Skills & Tools
- [Skill reference](skill.md) -- what it does
- [Tool reference](tool.md) -- when to use it

### Key Subsystems
- Subsystem A: files, lines, test count
- Subsystem B: files, lines, test count

## WARM -- Reference (session start)
### Completed Milestones (Condensed)
- v1.49: score 4.75, DACP capstone
- v1.48: score 4.50, physical infrastructure

### Standing Rules
- HARD RULE: never commit .planning/
- Branch strategy: dev/main/v1.50

### Agent Lessons
- Task sizing: 5-6 per agent
- 4 agents for synthesis, 3 for docs

Keeping MEMORY.md Under 200 Lines

The 200-line limit is a hard constraint, not a target. Strategies for staying under it:

Consolidation

When two memories cover related topics, merge them into one. Five memory files about different aspects of the branch strategy become one comprehensive branch-strategy.md with a single MEMORY.md line.

Compression

Replace verbose summaries with terse pointers. "The trust system build plan was approved on March 9 with 12 design decisions reviewed by the muse council" becomes "Trust system: 12 decisions approved, [detail](trust-system.md)."

Archival

Completed project memories move from HOT to WARM to archived. A milestone that shipped three months ago does not need the same index space as the currently active work. Condense it to a single line with key metrics.

Pruning

Some memories outlive their usefulness. A debugging constraint that was discovered and fixed does not need to persist. A reference to an external system that is no longer used can be deleted. Regular pruning keeps the index relevant.

Archival Patterns for Completed Work

When a project or milestone is completed, its memories should not be deleted — they contain decision rationale and lessons learned that may be valuable later. Instead, they are compressed and moved to the WARM or COLD sections of MEMORY.md:

# Before (HOT, 8 lines):
**Trust System Build** -- 12 design decisions, muse council reviewed
- [Trust system details](trust-system.md) -- full design
- Stage 1 DONE: trust-relationship.ts (63 tests)
- Stage 1 DONE: trust-relationship-provider.ts (32 tests)
- Total: 95 tests passing
- Build order: 3a -> 4 -> 2 -> 3b -> 5 -> 6
- Next: Stage 3a (trust evaluation)
- Blocking: None

# After (WARM, 2 lines):
### Completed: Trust System
- Stages 1-2 complete: 95 tests, [details](trust-system.md)

This compression ratio — 8 lines to 2 — is typical. The detail file still exists and can be read on demand. The index just needs to acknowledge its existence and provide the pointer.

14.11 Practical Patterns

The theory of agentic memory is useful. The practice is what matters. These patterns are not derived from first principles — they are extracted from months of daily operation, each one motivated by a specific failure or inefficiency that the pattern resolved.

Session Startup Protocol

Every session should begin the same way, regardless of what work is planned. The consistency ensures that no critical context is missed and that the agent and the user are aligned before any work begins.

The Five-Step Startup

  1. Load memory — MEMORY.md is loaded automatically. Scan HOT section for current state. Note any items flagged as blocking or urgent.
  2. Check state — Read .planning/STATE.md to determine workflow state. Verify branch, phase, and wave. Check for pending handoffs.
  3. Verify environment — Run build, check test status, verify git state. Ensure the environment matches what the memory and state files describe. If it does not, investigate before proceeding.
  4. Ask the user — "What are we working on today?" Do not assume. The user may have changed priorities, received new requirements, or want to work on something completely different from what the state files suggest.
  5. Confirm and begin — Confirm the plan, verify the branch, and start work. This takes 60 seconds and prevents 60 minutes of wasted effort on the wrong task.

Decision Documentation

Every non-trivial decision should be documented with its rationale. This is not bureaucracy — it is a direct investment in future efficiency. When a future session asks "why did we use RSA instead of Ed25519?" the decision memory provides the answer instantly instead of requiring a 30-minute archaeological dig through git history and old conversations.

Decision Memory Format

---
name: auth-signing-algorithm
description: Choice of RSA-256 for JWT signing
type: project
---
Decision: Use RS256 (RSA) for JWT signing, not Ed25519.

Why: Our deployment target requires FIPS 140-2 compliance.
Ed25519 is not FIPS-approved. RS256 with 2048-bit keys is.

Alternatives considered:
- Ed25519: faster, smaller keys, but not FIPS-compliant
- ES256: FIPS-compliant but less widely supported by
  our client libraries
- HS256: symmetric, not suitable for multi-service auth

Revisit if: FIPS requirement is removed, or Ed25519
gains FIPS approval (NIST is evaluating as of 2026).

The "revisit if" clause is critical. Without it, decisions become permanent by default. With it, they are explicitly tied to the conditions that justified them. When those conditions change, the decision is automatically flagged for reconsideration.

Standing Rules

Some behavioral instructions are permanent and non-negotiable. These are standing rules — they apply to every session, every task, every agent. They belong in both CLAUDE.md (for automatic loading) and in a feedback memory (for reinforcement).

Examples of Standing Rules

  • "Never commit .planning/ files" — These are gitignored and contain local-only state and potentially sensitive IP. Committing them would leak internal planning artifacts into the public repository.
  • "Ask the user what they're working on before starting" — Prevents the agent from assuming context and doing the wrong work. A 10-second question saves minutes or hours of misdirected effort.
  • "Autonomous execution, no stopping for input during batch work" — Once the user approves a plan, execute it fully. Do not pause every 30 seconds to ask "should I continue?" The approval was the checkpoint.
  • "Docs are the story. Never suggest pruning documentation." — Documentation is a first-class artifact, not overhead. It is part of the deliverable.

Standing Rule Characteristics

  • Unconditional: They apply always, with no exceptions based on context
  • Brief: One or two sentences, not paragraphs of explanation
  • Actionable: They describe a specific behavior to follow or avoid
  • Motivated: Each rule exists because violating it caused a problem
  • Durable: They change infrequently, if ever

Unanswered Questions Tracker

In long-running projects, questions get asked and never answered. Sometimes the user is thinking about it. Sometimes the question was lost in the flow of conversation. Sometimes nobody noticed it was asked. An unanswered questions tracker prevents this information loss.

The Pattern

When a question is asked and not answered in the current session, record it in the handoff or thread document with its status:

## Unanswered Questions
- [ ] Should we support WebSocket fallback for HTTP/2 clients?
      Asked: 2026-04-03. Context: API design discussion.
      Blocking: No, but affects architecture decision.

- [ ] What is the latency budget for the trust evaluation pipeline?
      Asked: 2026-04-05. Context: Trust system Stage 3a planning.
      Blocking: Yes - determines caching strategy.

- [x] Do we need FIPS compliance for the signing algorithm?
      Asked: 2026-04-02. Answered: 2026-04-03. Answer: Yes.
      Resolution: Use RS256, documented in auth-signing-algorithm.md

The key discipline is resurfacing: at the start of each session, check for unanswered blocking questions and raise them before doing any other work. Non-blocking questions can wait, but blocking questions should be resolved before the work they block is attempted.

Branch Strategy Memory

Multi-branch workflows create a persistent need for branch context. Which branch is for what purpose? What can be merged where? What is the current branch? These answers need to be immediately available, not discovered by reading git config.

Branch Strategy Example

---
name: branch-strategy
description: Branch purposes and merge rules
type: project
---
## Branches
- **dev** -- active working branch heading to main
- **main** -- human-verified merges from dev
- **v1.50** -- Unit Circle review branch (long-running)
- **artemis-ii** -- Mission tracking and research
- **nasa** -- NASA mission catalog (720 missions)

## Rules
- BRANCH RULE: Work on dev, NOT main
- Main receives merges from dev after human review
- Never force-push to main
- artemis-ii is independent, does not merge to dev

Why: Prevents accidental pushes to main and keeps
long-running branches isolated from each other.

How to apply: Check current branch before any commit.
If on main, switch to dev first.

Cross-Session Research Continuity

Research work is especially vulnerable to memory loss because insights build on each other over many sessions. A research thread on eigenvalue spacing in random matrices (Montgomery-Dyson conjecture) spans weeks of sessions, each building on the last. Without persistent memory, each session would re-derive insights that were discovered three sessions ago.

The pattern that works: maintain a research thread document that accumulates findings, and link it from MEMORY.md. Each session reads the thread, adds its contributions, and updates the thread. The thread becomes both memory and documentation — it captures the research journey as it unfolds, and at the end, it is a nearly publication-ready artifact.

Multi-Agent Memory Coordination

When multiple agents operate concurrently, their memory needs are different from single-agent systems. Each agent has its own context window but shares the filesystem. The coordination pattern is:

Shared State Files

Agents coordinate through files, not through shared memory. Agent A writes its state to .state/agent-a.json. Agent B reads it. This is the only reliable cross-agent communication channel because it survives crashes, works across process boundaries, and is debuggable with standard tools.

Memory File Conventions

Agent-specific discoveries go in agent-specific state files (ephemeral). Cross-cutting discoveries — "this CSS variable was renamed" — go in shared memory files that all agents read. The distinction prevents memory conflicts where two agents try to write the same memory file simultaneously.

Operational Insight

After operating this memory system across 190+ projects, hundreds of sessions, and months of continuous development, the most important lesson is this: memory management is not a feature you add to an agentic system. It is the foundation on which everything else is built. Skills, orchestration, multi-agent coordination, state machines — all of these depend on reliable, well-organized, actively-maintained memory. Invest in it early, maintain it continuously, and treat it with the same discipline you give to your codebase. Because in an agentic system, memory is the codebase for the agent's understanding of the world.
