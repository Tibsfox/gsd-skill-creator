Claude Agent SDK — Building Custom Agent Applications

The Claude Agent SDK provides the programmatic foundation for building custom agent applications outside of Claude Code itself. Available in both Python (anthropic package with claude_agent_sdk) and TypeScript (@anthropic-ai/sdk), the SDK exposes the same capabilities that power Claude Code's own agent infrastructure.

SDK Capabilities

Tool Use and Function Calling

Define custom tools with JSON Schema parameters. The model selects and invokes tools based on the task. Tool results flow back into the conversation for reasoning. Supports both synchronous and streaming execution patterns.

Key pattern: Define tools declaratively, let the model decide when to invoke them. This is the foundation of agent behavior—the model acts, observes results, and adapts.

Streaming and Batches

Streaming: Server-sent events for real-time output. Essential for interactive agent applications where users need to see work in progress. Both text and tool-use deltas stream incrementally.

Batches: Submit up to 10,000 requests for asynchronous processing at 50% cost reduction. Ideal for bulk analysis, testing, and evaluation pipelines where latency is acceptable.

Structured Outputs

Constrain model output to match a JSON Schema exactly. Eliminates parsing failures and validation overhead. The model generates syntactically valid JSON that conforms to the specified schema on every response.

Application: Agent coordination messages, structured reports, API payloads, configuration generation.

Agent Orchestration

Build agents that spawn and coordinate other agents. The query() function accepts agent definitions with isolated tool sets, custom prompts, and resource limits. This enables the supervisor-worker pattern at the SDK level.

Key fields: effort, maxTurns, isolation, memory, skills, mcpServers.

SDK Architecture Patterns

Python 
from claude_agent_sdk import query, ClaudeAgentOptions, AgentDefinition

# Pattern 1: Supervisor with MCP-connected specialists
async def research_pipeline(topic: str):
    """Multi-agent research pipeline with tool specialization."""
    specialists = {
        "searcher": AgentDefinition(
            description="Web research specialist",
            prompt="Find primary sources. Verify claims. Return structured findings.",
            tools=["WebSearch", "WebFetch", "Read"],
            effort=0.8,
            max_turns=20
        ),
        "analyst": AgentDefinition(
            description="Data analysis specialist",
            prompt="Analyze data for patterns. Produce structured insights.",
            tools=["Read", "Bash"],
            mcp_servers={
                "math": {"command": "math-coprocessor", "args": ["--gpu"]}
            }
        ),
        "writer": AgentDefinition(
            description="Technical writing specialist",
            prompt="Synthesize research into publication-quality documents.",
            tools=["Read", "Write", "Edit"]
        )
    }

    async for message in query(
        prompt=f"Research '{topic}': search → analyze → write report.",
        options=ClaudeAgentOptions(
            allowed_tools=["Task"],
            agents=specialists,
            model="claude-opus-4-6"
        )
    ):
        if hasattr(message, "result"):
            return message.result

Plugins Ecosystem

The plugins system represents the distribution layer for the skills, agents, and configuration patterns described throughout this report. A plugin bundles related capabilities into a single installable package with defined lifecycle, namespacing, and security boundaries.

Plugin Architecture

What a Plugin Contains
ComponentDescriptionExample
Skills SKILL.md files with instructions, triggers, and constraints code-review, test-generator, security-hygiene
Agents Agent definitions with prompts, tool restrictions, and resource limits PR reviewer, deploy coordinator, incident responder
Configuration Settings, model preferences, context injection Team coding standards, deployment targets
Metadata Version, dependencies, compatibility, namespace @org/devops-team v2.1.0

Plugin Lifecycle

Installation

Plugins install into .claude/plugins/<namespace>/. Skills are symlinked or copied to .claude/skills/. Agent definitions merge into the available agent pool. Configuration applies as overrides to project settings.

Namespacing prevents collisions: @acme/devops/code-review and @internal/code-review coexist without conflict.

Updates and Versioning

Semantic versioning with lockfile support. Breaking changes in major versions. Plugins declare minimum Claude Code version compatibility. Organizations can pin to specific versions for stability or float to latest for features.

Plugin Security Model

Security Boundary: What Plugins Cannot Do

The plugin security model enforces a critical constraint: plugins cannot include hooks, mcpServers, or permissionMode configurations for their agents. This means:

  • No hooks: A plugin cannot install PreToolUse or PostToolUse hooks that execute arbitrary code. Hooks remain under the control of the project owner or organization administrator.
  • No MCP servers: A plugin cannot connect to external services on installation. MCP server connections require explicit configuration by the user or organization.
  • No permission escalation: A plugin agent cannot set bypassPermissions or change the permission mode of the host environment.

This design prevents the most dangerous class of supply-chain attacks: a malicious plugin that silently installs a hook to exfiltrate code, connects to an attacker-controlled MCP server, or escalates permissions to modify protected files.

Enterprise Distribution

Organizations distribute plugins through managed settings—the enterprise configuration layer that applies policies across all team members. An organization administrator can:

  • Require specific plugins for all projects (e.g., security-hygiene, compliance-audit)
  • Block plugins from untrusted sources
  • Pin plugin versions across the organization
  • Configure shared MCP server connections that plugins can reference
  • Set permission policies that override plugin defaults

GSD as Plugin Prototype: The GSD workflow system already functions as an unofficial plugin: 57 commands, 34 skills, structured agents, and configuration files distributed via a skill-creator installer (node project-claude/install.cjs). The transition from this ad-hoc distribution to a formal plugin package is a natural evolution and demonstrates the design patterns the plugins ecosystem needs to support.

Content-Addressed Code

Content-addressed code identifies programs by the hash of their content rather than by name or file path. The Unison programming language is the production implementation of this idea. Our research co-processor prototype explores what content-addressing means for agent-generated code specifically.

The Unison Model

Code Identified by Hash, Not Name

In Unison, every definition—function, type, ability—is stored by the hash of its abstract syntax tree. Names are metadata, not identity. This produces several properties that matter deeply for agentic development:

  • No merge conflicts from renaming. Two agents can independently rename the same function without conflict, because the underlying hash identity does not change.
  • Structural typing. Types with the same structure are the same type regardless of what they are called. This makes cross-domain concepts interoperable—a Temperature from a weather module and a Temperature from a physics module can unify if their structures match.
  • Abilities map to pipeline stages. Unison abilities (similar to algebraic effects) cleanly model agent capabilities—an agent that can Read and Write has different abilities than one that can only Read.
  • Dependency resolution is exact. Because every definition is content-addressed, dependency graphs are precise and reproducible. No version ranges, no diamond dependency problems.

Research Co-Processor Prototype

Our research co-processor implements a content-addressed pipeline in Unison with 19/19 tests passing. The architecture demonstrates how agent-generated code could work in a content-addressed world:

Pipeline StageUnison ConceptAgent Mapping
Type definitions Structural types with hash identity Agent output schemas that compose without coordination
Abilities Algebraic effects declaring required capabilities Agent tool permissions (Read, Write, Bash, MCP)
Handlers Implementations that satisfy ability requirements Runtime environments providing tool implementations
Pipeline composition Functions composed by type compatibility Agent chains where output types match input types

Why This Matters for Agent-Generated Code

Today, when multiple agents modify a codebase simultaneously, they face all the coordination problems of traditional software: merge conflicts, naming collisions, import path breakage, and dependency version mismatches. Content-addressed code eliminates an entire class of these problems:

Today: File-Based Code

Agent A renames processData to transformData. Agent B adds a call to processData. Merge conflict. Both agents did valid work, but coordination failure means manual resolution.

Future: Content-Addressed Code

Agent A updates the metadata name to transformData. Agent B references the hash #a8f3c2. No conflict—the hash identity is stable regardless of name. Both changes merge cleanly.

Open question: Could skills and agents themselves be content-addressed? A skill identified by the hash of its instructions rather than its file path would be version-immune, composable, and universally referenceable. This remains speculative but aligns with the direction of agent infrastructure.

Self-Modifying Systems at Scale

A self-modifying system is one where the AI agent can create, edit, and evolve its own tooling—skills, agents, hooks, and configuration—based on observed usage patterns. This is among the most powerful and most dangerous capabilities in agent infrastructure.

Where Self-Modification Adds Value

Bounded Learning: The Skill-Integration Pattern

The skill-integration system demonstrates bounded self-modification. It observes agent sessions, identifies recurring tool sequences and correction patterns, and suggests new skills that encode these patterns. The critical word is suggests—a human reviews and approves before any skill is activated.

Production experience: 34+ skills have evolved through this observation cycle in our system. The learning pipeline follows a strict pattern:

  1. Observe: Record tool calls, corrections, file access patterns during normal operation
  2. Detect: Identify repeated sequences (e.g., "always run tests after editing src/")
  3. Draft: Generate a candidate SKILL.md with clear trigger conditions and instructions
  4. Review: Present the suggestion with evidence. Human accepts, modifies, or rejects.
  5. Activate: Accepted skills enter the active pool and are monitored for effectiveness

The Observer Pattern

Session observation is the input mechanism for self-modification. The observer records structured data about each session without interfering with execution:

  • Tool sequences: Which tools are called in what order. Repeated sequences suggest automation potential.
  • Correction patterns: When a user corrects agent output, the correction encodes a preference that could become a skill constraint.
  • File access patterns: Which files are always read together. This suggests context dependencies that a skill could pre-load.
  • Error recovery: How the agent recovers from failures. Successful recovery patterns become procedural memory.

When Self-Modification Adds Risk

Risk Categories for Self-Modifying Systems

RiskDescriptionMitigation
Drift Skills accumulate subtle biases over many iterations, moving system behavior away from original intent Version control all skills. Regular audit against original objectives. Effectiveness monitoring with auto-deactivation.
Complexity explosion Too many skills create interaction effects. Skill A triggers Skill B which conflicts with Skill C. Token budget per skill. Total skill count limits. Dependency analysis before activation.
Security bypass A self-generated skill inadvertently creates a path around security guardrails Security-hygiene skill reviews all skill modifications. Hooks cannot be created by skills.
Feedback loops Observer detects its own observation patterns as "useful" and generates meta-skills recursively Exclude observer metadata from observation. Hard recursion depth limits.

Design principle: Self-modification should be bounded, observable, and reversible. Bounded means clear limits on what can be modified (skills yes, hooks no). Observable means every modification is logged and auditable. Reversible means any skill can be deactivated or rolled back to a previous version without side effects.

The Runtime HAL Pattern

A Runtime Hardware Abstraction Layer (HAL) detects which AI runtime is active and adapts skill behavior accordingly. This pattern enables portable skills that work across runtimes:

TypeScript 
// Runtime detection: which AI assistant is running?
function detectRuntime(): Runtime {
  if (process.env.CLAUDE_CODE) return "claude-code";
  if (process.env.CODEX_CLI)   return "codex";
  if (process.env.GEMINI_CLI)  return "gemini";
  if (process.env.CURSOR_AGENT) return "cursor";
  return "unknown";
}

// Adapt skill output format for the active runtime
function emitSkillInstruction(runtime: Runtime, instruction: string) {
  switch (runtime) {
    case "claude-code":
      return `<skill-instruction>${instruction}</skill-instruction>`;
    case "codex":
      return `[SYSTEM] ${instruction}`;
    case "gemini":
      return `Instructions: ${instruction}`;
    default:
      return instruction;
  }
}

Agent Skills Open Standard

The Agent Skills initiative (agentskills.io) proposes a runtime-agnostic standard for skill definition. The core idea: define skills in a format that any compliant runtime can load, with runtime-specific adapters handling the translation. Key design constraints:

  • Declarative triggers: When should the skill activate? File patterns, command patterns, content patterns.
  • Capability requirements: What tools does the skill need? Read, Write, Bash, MCP servers.
  • Output contracts: What does the skill produce? Structured output schemas.
  • Security annotations: What trust level is required? Read-only, edit-approved, full autonomy.

Assessment: Convergence is more likely than fragmentation. MCP has already been adopted by four of the five major runtimes. Skill format convergence will follow, driven by the same economic pressure that standardized container images and package formats. The timeline is 12-18 months for meaningful interoperability.

Edge Deployment and Local Compute

Not all agent work should traverse the network. Latency-sensitive operations, privacy-critical data, and computationally intensive tasks benefit from local execution. The edge deployment pattern combines cloud-hosted LLM reasoning with local GPU-accelerated tools.

GPU-Accelerated MCP Tools

Case Study: Math Co-Processor

Our math co-processor implements 18 MCP tools running on a local RTX 4060 Ti (8GB VRAM). The tools expose GPU-accelerated operations—matrix decomposition, FFT, Monte Carlo simulation, symbolic evaluation—through the standard MCP protocol. The LLM reasons about what computation to perform; the local GPU executes it.

Cloud API roundtrip (matrix multiply) ~800ms
Local GPU via MCP (same operation) ~12ms

The 67x speedup is not an optimization trick—it is the difference between sending data to a cloud endpoint and executing on hardware three feet away. For iterative computations where an agent might invoke the same tool hundreds of times during a research session, local execution transforms feasibility.

Hybrid Cloud + Local Architecture

Cloud (LLM Reasoning)
  • Natural language understanding and generation
  • Complex reasoning and planning
  • Code generation and review
  • Cross-document synthesis

Why cloud: LLM inference at Opus-class quality requires infrastructure beyond typical developer hardware. The reasoning layer stays in the cloud.

Local (Tool Execution)
  • File system operations (Read, Write, Edit)
  • GPU computation (math, rendering, ML inference)
  • Database queries (pgvector, SQLite)
  • Build and test execution

Why local: Data never leaves the machine. Latency is minimal. No per-invocation API costs. Hardware is already present.

Privacy-Preserving Agent Architectures

Local compute enables agent architectures where sensitive data never leaves the developer's machine:

  • Local embedding: Run all-MiniLM-L6-v2 or similar models locally to embed documents into pgvector without sending content to cloud APIs.
  • Local RAG: Retrieval-augmented generation where the retrieval step queries a local vector database. Only the retrieved snippets (not the full corpus) are sent to the cloud LLM.
  • Filtered context: Hooks strip sensitive data from context before it reaches the cloud. A PreToolUse hook on Read can redact credentials, PII, or proprietary algorithms from file contents.
  • Air-gapped tools: MCP servers that operate entirely within the local network. The LLM instructs; the tool executes locally and returns only results.

Audit Trails Through Hooks

Hooks provide the deterministic enforcement layer that compliance frameworks demand. Unlike skills (which are advisory), hooks execute unconditionally:

JSON (settings.json) 
{
  "hooks": {
    "PreToolUse": [
      {
        // Log every tool invocation for SOC2 audit trail
        "matcher": "*",
        "command": "echo \"$(date -u +%Y-%m-%dT%H:%M:%SZ) TOOL=$TOOL_NAME\" >> .audit/tool-log.jsonl"
      },
      {
        // Block access to regulated data directories
        "matcher": "Read",
        "if": "input.file_path matches 'phi/|pci/|gdpr/'",
        "command": "echo 'BLOCK: Regulated data access requires explicit approval' && exit 2"
      }
    ],
    "PostToolUse": [
      {
        // Verify no sensitive data in output
        "matcher": "Bash",
        "command": "echo \"$TOOL_OUTPUT\" | grep -qiE 'ssn|credit.card|password' && exit 2 || exit 0"
      }
    ]
  }
}

Permission Management at Scale

The managed settings model provides organizational control over agent permissions:

Organization Level

Set by administrators. Cannot be overridden by projects or users. Enforces minimum security posture across all agent operations.

Controls: Allowed/blocked tools, required hooks, MCP server whitelist, permission mode ceiling, model restrictions.

Project Level

Set per repository via .claude/settings.json. Can restrict but not relax organization-level settings. Team leads configure project-specific policies.

Controls: Project-specific hooks, skill activation rules, agent definitions, MCP server configuration.

The "Trust No One" Security Model

Zero-Trust Applied to Agent Systems

The zero-trust principle from network security applies directly to agent architectures. In a zero-trust agent system:

  • No agent is trusted by default. Every agent starts with minimal permissions and earns expanded access through demonstrated reliability.
  • Every tool invocation is validated. Hooks verify that the tool call is authorized, the parameters are within bounds, and the output is safe before it enters the agent context.
  • Every agent output is verified. Supervisor agents validate subagent results. Cross-checking between independent agents detects compromised outputs.
  • Every external data source is untrusted. Content retrieved from MCP servers, web searches, or file reads is treated as potentially adversarial.
  • Trust is contextual and temporary. An agent trusted for code review is not automatically trusted for deployment. Trust does not persist across sessions without explicit re-establishment.

This model increases overhead but dramatically reduces the blast radius of any single compromise. The hook system makes enforcement practical: deterministic code guards run on every operation regardless of what the LLM decides.

Strategic Recommendations

These recommendations are ordered by risk-adjusted value: the ratio of benefit gained to complexity introduced. Start at the top and work down.

For Individuals

Individual

1. Start with Skills Before Agents

Skills are the lowest-risk, highest-value entry point to agent infrastructure. A single SKILL.md file can encode team conventions, code review standards, or deployment checklists. Skills require no infrastructure, no API keys, and no runtime dependencies. They activate automatically based on context and degrade gracefully—if a skill is wrong, the human simply ignores it.

Action: Create one skill this week. Encode your most-repeated instruction to Claude Code. Observe whether it reduces correction frequency over the next 10 sessions.

Risk: Near zero. A bad skill wastes a few context tokens. Remove it and move on.

Individual

2. Use Hooks for Deterministic Guardrails

Hooks are the complement to skills. Where skills are advisory ("consider doing X"), hooks are mandatory ("X must happen before Y"). A PreToolUse hook on Bash that blocks rm -rf / runs deterministically regardless of what the model thinks is a good idea. Hooks are the right tool when the constraint is unconditional.

Action: Install one safety hook (e.g., commit message format validation, blocked command patterns). Graduate to audit logging hooks once comfortable.

Risk: A misconfigured hook can block legitimate operations. Test hooks in a non-critical project first.

Individual

3. Model Routing for Cost Control

Not every task requires Opus. Classification, extraction, and simple code generation run well on Haiku at 1/60th the cost. Complex architecture decisions, multi-file refactoring, and research synthesis justify Opus. Sonnet covers the middle ground. Route by task complexity, not by habit.

Action: Audit your last week of Claude usage. Identify tasks that could have used a smaller model. Set up model routing in your SDK applications or use GSD's model profile system.

Savings potential: 40-70% cost reduction for mixed workloads with no quality degradation on routed tasks.

For Teams

Team

4. Invest in Memory and State Management Early

Agent effectiveness degrades catastrophically across context resets without persistent memory. Teams that invest in structured state management (CLAUDE.md, MEMORY.md, handoff documents, state files) see compounding returns: each session starts closer to productive work instead of re-establishing context.

Action: Establish a CLAUDE.md for every active project. Document conventions, file locations, and standing instructions. Create a memory directory for cross-session state that does not belong in code.

Pattern: GSD's STATE.md + ROADMAP.md + handoff documents demonstrate this at scale: 190+ projects maintained across hundreds of sessions with minimal context loss.

Team

5. Build for Observability First

You cannot improve what you cannot measure. Hook-based logging captures every tool invocation, every file modification, and every agent spawn. This data drives optimization: which agents are effective, which skills reduce errors, which workflows are bottlenecked.

Action: Configure PostToolUse hooks that log tool name, duration, and outcome. Aggregate weekly. Look for patterns: repeated failures, slow tools, unnecessary operations.

Advanced: The witness-observer pattern runs a monitoring agent that patrols other agents for stalls, errors, and anomalies. Deploy for long-running fleet operations.

Team

6. Plan for Multi-Runtime Portability

Teams that lock into a single runtime's proprietary skill format will face migration costs when better alternatives emerge or organizational requirements change. Investing in portable skill definitions now—even if the open standard is not finalized—reduces future lock-in.

Action: Structure skills with clear separation between intent (what the skill does) and mechanism (how it activates in a specific runtime). Keep the intent layer runtime-agnostic. Add runtime-specific activation as a thin adapter.

Watch: Agent Skills (agentskills.io) open standard. MCP server compatibility across runtimes. Runtime HAL patterns for cross-runtime skill loading.

For Organizations

Organization

7. Governance as Infrastructure, Not Bureaucracy

Effective agent governance is not a policy document—it is enforcement code. Managed settings, organizational hooks, and plugin whitelists provide governance that executes automatically. Every agent session in the organization inherits the same security baseline without developer action or compliance training.

Action: Define organizational managed settings: required safety hooks, blocked operations, approved MCP servers, permission mode ceilings. Deploy to all team members. Audit quarterly.

Framework alignment: Map managed settings to your compliance requirements (SOC2, HIPAA, GDPR). Each setting should trace to a specific control objective.

Organization

8. Graduated Autonomy Adoption

Start agents at Level 0 (read-only, human approves everything). Promote to Level 1 (edit-approved) after demonstrated reliability. Reserve Level 2+ for CI/CD pipelines and verified workflows. Never grant Level 3 (full autonomy) without comprehensive logging, monitoring, and circuit breakers.

Action: Define autonomy levels for your organization. Map each level to specific permission modes, allowed tools, and required hooks. Create promotion criteria: N sessions without errors, N tasks completed successfully, human review of agent decision quality.

Conclusion: The Ground Truth of April 2026

Six months ago, this section was written as a set of predictions. Today it can be written as a status report. The major developments that analysts forecast for 2026 have either arrived or are clearly in progress:

Arrived
  • Multi-agent orchestration in production (convoy model, fleet dispatch, worktree isolation)
  • 1M context windows as standard operating environment
  • Claude Agent SDK for custom agent applications (Python + TypeScript)
  • Hook-based governance with new event types (PostCompact, FileChanged, PermissionDenied)
  • MCP as the dominant tool integration protocol (4/5 major runtimes)
  • 34+ self-evolved skills through bounded learning
In Progress
  • Plugin ecosystem for skill/agent distribution (security model defined, marketplace emerging)
  • A2A protocol maturation (50+ partners, early deployments)
  • Multi-runtime skill portability (Runtime HAL demonstrated, open standard proposed)
  • Content-addressed code for agent workflows (Unison prototype, 19/19 tests)
  • Enterprise governance automation via managed settings
  • Edge/local compute integration for privacy and latency

The Guiding Principle

The single most important insight from the first half of 2026 is that deterministic guardrails around intelligent agents is the architecture that works. Skills provide intelligence—adaptive, context-aware guidance that improves agent output quality. Hooks provide guardrails—unconditional enforcement that runs regardless of what the model decides. Memory provides continuity—state that survives context resets. MCP provides capability—standardized access to tools and data. The agents that succeed in production are not the most autonomous; they are the ones with the clearest boundaries.

The second half of 2026 will be defined by distribution (plugins), interoperability (multi-runtime portability), and governance (enterprise compliance automation). Organizations that built strong foundations in H1—skills, hooks, memory, MCP—will be positioned to adopt these capabilities as they mature. Those still evaluating will find the gap widening.

The agent future is not a future. It is the present operating environment.
