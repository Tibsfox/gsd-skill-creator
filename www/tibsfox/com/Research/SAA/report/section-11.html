Section 11

Hooks — The Automation Backbone

Deterministic Control Points in an Agentic System

Research current as of: April 2026

Hooks are the deterministic layer beneath the probabilistic intelligence of an agentic system. While skills provide adaptive behavior and agents bring autonomous reasoning, hooks provide the hard guarantees: every commit will be validated, every session will load context, every destructive operation will be intercepted. In a production system running 190+ research projects with multi-agent orchestration, hooks are not optional infrastructure — they are the safety rails, the context injectors, the quality gates, and the automation backbone that makes everything else possible. This section is written from hands-on operational experience with hook-based automation at scale.

11.1 The Hook Architecture

A hook is a user-defined action that executes at a specific point in the Claude Code lifecycle. At its simplest, a hook is a shell command that runs when a tool is called. At its most sophisticated, a hook is a subagent with full tool access that performs complex verification before allowing an operation to proceed. The four hook types — command, http, prompt, and agent — cover the full spectrum from fast deterministic checks to nuanced AI-evaluated decisions.

The fundamental insight behind hooks is separation of determinism from intelligence. An LLM can reason about what to do, but it cannot guarantee it will always check for TypeScript errors before committing. A hook can. By placing deterministic control points at lifecycle boundaries, we get the benefits of autonomous agents with the guarantees of traditional CI/CD pipelines — except the pipeline runs inside the agentic loop, not after it.

The Four Hook Types

Command

Shell scripts and executables. The workhorse of hook automation. A command hook runs a shell command, reads input from stdin as JSON, and communicates results through exit codes and stdout JSON. Exit code 0 means success, exit code 2 means blocking error. Fast, composable, and language-agnostic.

Best for: build checks, file guards, state persistence, context injection

HTTP

POST to an endpoint. Sends the hook event payload as a JSON POST request to a URL. Headers can interpolate environment variables for authentication. The response body is treated as hook output. Enables integration with external services, webhooks, and monitoring systems.

Best for: external notifications, audit logging, CI/CD triggers, team dashboards

Prompt

Single-turn LLM evaluation. Sends a natural-language prompt to a model for a yes/no decision. The prompt has access to the event context and the model returns a structured judgment. Useful when the decision requires understanding intent rather than pattern matching. Runs as a single inference call, not a multi-turn conversation.

Best for: semantic code review, intent validation, policy compliance checks

Agent

Subagent with tool access. Launches a lightweight agent with access to tools like Read, Grep, and Glob. The agent can inspect files, search code, and make complex decisions that require understanding project context. The most powerful hook type, but also the slowest. Use sparingly and with timeouts.

Best for: multi-file verification, architectural compliance, complex security checks

Configuration Locations

Hooks can be defined in four locations, each with different scope and precedence:

Location Scope File Path Use Case
User settings All projects for this user ~/.claude/settings.json Personal safety rails, universal formatting
Project settings This repository .claude/settings.json Build checks, commit validation, project context
Plugin / Enterprise Organization-wide Managed policy file Security policies, compliance requirements, audit trails
Skill/Agent frontmatter Component lifetime YAML frontmatter in .claude/skills/ or .claude/agents/ Scoped automation that activates and deactivates with the component

All four locations use the same JSON schema. When multiple locations define hooks for the same event, all hooks run — they are additive, not overriding. The exception is decision hooks (PreToolUse), where the most restrictive decision wins according to a defined precedence chain (Section 11.3).

JSON Configuration Format

The settings.json hooks object maps lifecycle event names to arrays of hook groups. Each group contains a hooks array (the actions to run), an optional matcher (which tool or event to match), and an optional if condition (fine-grained filtering).

Hook Configuration Structure (settings.json)
{ "hooks": { "PreToolUse": [ { "matcher": "Bash", "if": "Bash(git commit*)", "hooks": [ { "type": "command", "command": "bash .claude/hooks/commit-check.sh", "timeout": 30 } ] } ], "SessionStart": [ { "hooks": [ { "type": "command", "command": "node .claude/hooks/load-context.js" } ] } ], "PostToolUse": [ { "matcher": "Bash|Edit|Write|MultiEdit|Agent|Task", "hooks": [ { "type": "command", "command": "node .claude/hooks/context-monitor.js", "timeout": 10 } ] } ] } }

Key fields in the hook definition:

Field Type Description
type string "command", "http", "prompt", or "agent"
command string Shell command to execute (command type only)
url string Endpoint URL (http type only)
timeout number Seconds before the hook is killed (default varies by type)
matcher string Pipe-separated tool names or event subtypes to match
if string Permission-rule-style condition for precise targeting
once boolean If true, the hook runs only once and then deregisters
async boolean If true, the hook runs asynchronously (fire-and-forget)

11.2 Lifecycle Events — The Complete Map

Every hook is anchored to a lifecycle event. Understanding the complete event map is essential for building effective automation. Events fall into three categories: session events (beginning and end of interaction), agentic loop events (the core tool-use cycle), and async events (fires independently of the main loop).

Session-Level Lifecycle

SessionStart | v InstructionsLoaded -----> [CLAUDE.md, MEMORY.md, skills loaded] | v UserPromptSubmit -------> [user types a message] | v +-------------------------------------------+ | AGENTIC LOOP (repeats per tool call) | | | | PreToolUse ----+ | | | | | | [decision] | | | allow deny defer ask | | | | | | | | v X | v | | Tool Executes | PermissionReq | | | | | | | v | v | | PostToolUse | [user decides] | | PostToolUseFailure | | | | | | | v | | | [next tool or done] | | | | | SubagentStart / SubagentStop | | TaskCreated / TaskCompleted | +-------------------------------------------+ | v Stop / StopFailure | v TeammateIdle -----------> [agent exhausted options or complete] | v PreCompact / PostCompact -> [context window management] | v SessionEnd

Core Lifecycle Events Reference

Event When It Fires Matcher Matches On Key Input Fields Decisions
SessionStart sync Claude Code session begins Session type: human, agent, resume session_id, cwd, session_type Context injection only
InstructionsLoaded sync After CLAUDE.md and memory files are read N/A instructions, memory, skills Context injection only
UserPromptSubmit sync User submits a prompt (before LLM processes it) N/A prompt_text, session_id Can modify prompt, block
PreToolUse decision Before any tool executes Tool name: Bash, Write, Edit, Read, etc. tool_name, tool_input, session_id allow, deny, ask, defer
PermissionRequest sync When a tool needs user approval (from ask) Tool name tool_name, tool_input, permission_type Modify dialog, provide context
PermissionDenied async When user denies a permission request Tool name tool_name, tool_input Logging only
PostToolUse sync After a tool executes successfully Tool name tool_name, tool_input, tool_output Inject context, block (error)
PostToolUseFailure sync After a tool execution fails Tool name tool_name, tool_input, error Error handling, context
SubagentStart sync When a subagent (Task/Agent tool) launches Agent/Task name agent_id, instructions, tools Context injection
SubagentStop sync When a subagent terminates Agent/Task name agent_id, result, exit_reason Cleanup, result capture

Understanding the Agentic Loop

The agentic loop — the cycle of PreToolUse, tool execution, and PostToolUse — is the heartbeat of an agentic session. A single user prompt can trigger dozens or hundreds of iterations through this loop as the agent reasons, writes code, runs tests, fixes errors, and iterates. Every iteration passes through the hook system, which means hooks see the full behavioral trace of the agent.

This has profound implications for automation design. A PreToolUse hook on Bash will fire for every shell command the agent runs — not just the commands you are thinking about. If your hook is slow, it compounds across hundreds of tool calls. If your hook has false positives, it blocks the agent repeatedly. The key insight is that hooks must be designed for the agentic loop's cadence, not for the human interaction cadence you might expect.

In practice, a single "execute phase 73" command in our system generates approximately 40-80 tool calls across 4 parallel agents. Each tool call passes through 3-4 hooks (prompt guard, build check, context monitor). That is 120-320 hook invocations per phase execution. At 100ms per hook, you add 12-32 seconds of overhead. At 500ms per hook, you add 60-160 seconds. The difference between a fast hook and a slow hook is the difference between a smooth development flow and an unusable system.

Event Ordering and Guarantees

Within a single hook group (one entry in the array), hooks execute sequentially. Between hook groups on the same event, execution is also sequential in array order. This gives you ordering guarantees when you need them.

Ordering Example

Our SessionStart has three hook groups in a specific order: update check first (so the user sees update notices immediately), state restore second (so the session has context), snapshot injection third (so the narrative builds on the restored state). Changing the order would break the context pipeline — you cannot inject a snapshot that references state that has not been restored yet.

For PostToolUse hooks with async: true, no ordering guarantee exists. Async hooks fire in the background and may complete in any order. Do not use async hooks for anything that has ordering dependencies.

Async Events

Async events fire independently of the main agentic loop. They respond to environmental changes rather than tool-use cycles.

WorktreeCreate / WorktreeRemove

Fires when a git worktree is created or removed. Worktrees are used for isolated agent execution — each subagent can operate in its own worktree without interfering with the main working directory. Hooks here can initialize worktree-specific configuration or clean up after completion.

Notification

Fires when Claude Code sends a notification. The matcher matches on notification type (e.g., error, warning, info). Useful for routing notifications to external systems like Slack, Discord, or pagers.

ConfigChange

Fires when settings.json or CLAUDE.md changes during a session. Enables hot-reload patterns where configuration changes take effect without restarting the session.

CwdChanged

Fires when the working directory changes. Critical for multi-project setups where different directories have different hook configurations, skills, or agent definitions.

FileChanged

Fires when a watched file changes on disk (external to Claude Code). The matcher matches on file path patterns. Enables reactive automation: when a test file changes, re-run tests; when a config file changes, re-validate.

Elicitation / ElicitationResult

Fires when Claude Code asks the user a question (Elicitation) and when the user responds (ElicitationResult). Hooks can intercept questions to provide automated answers in headless/CI mode, or log the Q&A for audit trails.

11.3 Decision Control — Permission Management Through Hooks

The most powerful application of hooks is decision control through PreToolUse hooks. When a tool is about to execute, PreToolUse hooks can intercept the call and return one of four decisions. This is how you build deterministic safety rails around an autonomous agent without crippling its ability to work.

The Four Decisions

allow

The tool call proceeds without user confirmation. Use this to pre-approve known-safe operations. For example, allowing Read on any file, or allowing Bash(npm test) without prompting.

Allow Decision Output
// Hook stdout (JSON): { "decision": "allow" } // Exit code: 0

deny

The tool call is blocked entirely. The agent receives an error message explaining why. Use this for hard safety boundaries: blocking git push --force, preventing writes to protected files, rejecting destructive commands.

Deny Decision Output
// Hook stdout (JSON): { "decision": "block", "reason": "Protected path: .claude/settings.json" } // Exit code: 2

ask

The operation requires explicit user confirmation. The hook can provide additional context that appears in the permission dialog. This is the default behavior for most tool calls when no hook overrides it.

Ask Decision Output
// Hook stdout (JSON): { "decision": "ask", "message": "This will modify 12 files" } // Exit code: 0

defer

The tool call is deferred — it does not execute now, but it is not denied. The agent can resume it later. This is critical for headless/CI mode where no human is available to approve. The tool emits a tool_deferred event and can be resumed programmatically.

Defer Decision Output
// Hook stdout (JSON): { "decision": "defer", "reason": "Awaiting CI approval" } // Exit code: 0

Decision Precedence

When multiple hooks return different decisions for the same tool call, the most restrictive decision wins:

deny > defer > ask > allow

This means a user-level hook that allows all Read operations can be overridden by a project-level hook that denies reads of .env files. The deny always wins. This is a security-first design: it is impossible for a lower-privilege hook to override a higher-privilege restriction.

Why Precedence Matters in Multi-Agent Systems

In a multi-agent orchestration system, different agents have different trust levels. A human-supervised session might allow git push, but a subagent running autonomously should not. By layering hooks — organization-level deny for destructive ops, project-level allow for known-safe ops, skill-level ask for edge cases — you build a permission model that scales with team complexity without requiring every agent to understand the full policy.

PermissionRequest Hooks

When a tool call results in an ask decision (either from a hook or from the default permission system), a PermissionRequest event fires. Hooks on this event can modify the permission dialog — adding context, warnings, or recommendations that help the user make an informed decision.

PermissionRequest Context Injection
// A PermissionRequest hook that adds file change statistics const input = JSON.parse(require('fs').readFileSync('/dev/stdin', 'utf8')); const filePath = input.tool_input?.file_path; if (filePath) { const { execSync } = require('child_process'); const log = execSync(`git log --oneline -5 -- "${filePath}" 2>/dev/null`, { encoding: 'utf8' }); const output = { hookSpecificOutput: { hookEventName: 'PermissionRequest', additionalContext: `Recent changes to ${filePath}:\n${log || 'No git history'}` } }; process.stdout.write(JSON.stringify(output)); }

PostToolUse Blocking and Context Injection

PostToolUse hooks execute after a tool completes. While they cannot undo the tool execution, they serve two critical purposes:

  • Context injection: Add information to the agent's context based on what just happened. For example, after a Write to a test file, inject a reminder to run tests. After a Bash(git commit), inject the commit hash for tracking.
  • Blocking error: If a PostToolUse hook exits with code 2, it signals a blocking error. The agent receives the error and must address it before continuing. This is how you implement "lint after write" patterns — the write succeeds, the linter runs, and if it fails, the agent must fix the issues.

updatedInput — Modifying Tool Parameters

PreToolUse hooks can modify the tool's input parameters before execution by returning an updatedInput field. This is a surgical capability: the hook can rewrite the command, change the file path, or adjust parameters without the agent being aware of the modification.

updatedInput: Rewriting a Command
// PreToolUse hook that adds --dry-run to destructive git commands const input = JSON.parse(require('fs').readFileSync('/dev/stdin', 'utf8')); const cmd = input.tool_input?.command || ''; if (/^git\s+(clean|reset|checkout)\s/.test(cmd) && !cmd.includes('--dry-run')) { const output = { decision: "allow", updatedInput: { command: cmd + ' --dry-run' }, hookSpecificOutput: { additionalContext: "Safety: added --dry-run to destructive git command" } }; process.stdout.write(JSON.stringify(output)); process.exit(0); } process.exit(0);

Use updatedInput with Caution

Modifying tool inputs is powerful but can create confusing behavior if the agent does not understand why its command was changed. Always include additionalContext when using updatedInput so the agent knows what happened and can adjust its strategy. Silent input modification leads to debugging nightmares.

11.4 Hook Types Deep Dive

Command Hooks

Command hooks are the foundation of hook automation. A command hook runs a shell command, receives the event payload on stdin as JSON, and communicates results through exit codes and stdout.

Exit Code Protocol

Exit Code Meaning Agent Behavior
0 Success Stdout parsed as JSON; decision, hookSpecificOutput, and updatedInput fields are processed. If no JSON on stdout, treated as silent success.
2 Blocking error Tool execution is blocked. Stdout is sent to the agent as an error message. For PreToolUse, this is equivalent to a deny decision. For PostToolUse, signals that the agent must address the problem.
Any other Hook failure The hook itself failed. The tool call proceeds as if the hook did not exist. This is a safety net: a broken hook does not break the agent.

JSON Output Format

Command hooks communicate through structured JSON on stdout. The format depends on the event type:

PreToolUse Hook JSON Output
{ "decision": "allow" | "deny" | "ask" | "defer", "reason": "Human-readable explanation", "updatedInput": { "command": "modified command string" }, "hookSpecificOutput": { "hookEventName": "PreToolUse", "additionalContext": "Context injected into agent's next turn" } }
SessionStart Hook JSON Output
// For SessionStart, stdout content becomes context injection. // It can be plain text or structured JSON: { "hookSpecificOutput": { "hookEventName": "SessionStart", "additionalContext": "Project: gsd-skill-creator\nBranch: dev\nLast commit: feat(sweep): v1.7.3" } }

Async Command Hooks

When a command hook has "async": true, it runs in the background. The agent does not wait for it to complete, and its output is not processed. Use async hooks for logging, metrics emission, and notifications where latency is unacceptable.

Async Logging Hook
{ "hooks": { "PostToolUse": [ { "hooks": [ { "type": "command", "command": "node .claude/hooks/emit-metrics.js", "async": true, "timeout": 5 } ] } ] } }

HTTP Hooks

HTTP hooks send the event payload as a JSON POST request to a URL. They enable integration with external systems without requiring local scripts.

HTTP Hook Configuration
{ "hooks": { "SessionStart": [ { "hooks": [ { "type": "http", "url": "https://hooks.example.com/claude/session-start", "headers": { "Authorization": "Bearer $CLAUDE_HOOK_TOKEN", "X-Project": "$PROJECT_NAME" }, "timeout": 10 } ] } ] } }

Headers support environment variable interpolation with the $VAR_NAME syntax. The response body is processed the same way as command hook stdout — parsed as JSON for decisions and context injection. A non-2xx response is treated as a hook failure (tool call proceeds).

HTTP Hook Use Cases

  • Audit logging: Send every tool invocation to a compliance endpoint
  • Team dashboards: POST session events to a real-time monitoring service
  • CI/CD integration: Trigger a build pipeline when a commit hook fires
  • Slack/Discord notifications: Alert the team when a subagent completes a milestone
  • External policy engine: Delegate permission decisions to an OPA/Cedar policy server

Prompt Hooks

Prompt hooks delegate decision-making to a single-turn LLM call. The hook definition includes a prompt template that receives the event context, and the model returns a structured yes/no decision. This bridges the gap between rigid pattern matching (command hooks) and full autonomous reasoning (agent hooks).

Prompt Hook: Semantic Commit Message Validation
{ "hooks": { "PreToolUse": [ { "matcher": "Bash", "if": "Bash(git commit*)", "hooks": [ { "type": "prompt", "prompt": "Review this git commit command. Does the commit message follow conventional commits format (type(scope): subject)? Is the subject in imperative mood, lowercase, under 72 chars, with no trailing period? If it follows the convention, respond with {\"decision\": \"allow\"}. If not, respond with {\"decision\": \"deny\", \"reason\": \"explanation of what's wrong\"}.", "model": "claude-haiku", "timeout": 15 } ] } ] } }

Key characteristics of prompt hooks:

  • Single-turn only — no multi-step reasoning or tool use
  • Model selection via the model field (use fast/cheap models for hooks)
  • The prompt receives the full event context as input
  • Response is parsed as JSON for decision fields
  • Faster than agent hooks but slower than command hooks

Agent Hooks

Agent hooks are the most powerful hook type. They launch a lightweight subagent with access to a subset of tools (typically Read, Grep, Glob) that can inspect files, search code, and reason about complex conditions before making a decision.

Agent Hook: Architectural Compliance Check
{ "hooks": { "PreToolUse": [ { "matcher": "Write|Edit", "if": "Write(src/**) | Edit(src/**)", "hooks": [ { "type": "agent", "instructions": "You are an architectural compliance checker. Before this file is written, verify: 1) Files in src/ do not import from desktop/@tauri-apps/api. 2) Files in desktop/ do not import Node.js modules. 3) No circular dependencies are introduced. Check the import statements in the proposed file content. If compliant, return {\"decision\": \"allow\"}. If not, return {\"decision\": \"deny\", \"reason\": \"specific violation\"}.", "tools": ["Read", "Grep", "Glob"], "model": "claude-sonnet", "timeout": 30 } ] } ] } }

Agent Hooks Are Expensive

Every agent hook invocation consumes tokens and adds latency. A PreToolUse agent hook that runs on every Write operation will dramatically slow down the development loop. Use agent hooks only for high-stakes decisions where the cost of a mistake exceeds the cost of the verification. For most cases, command hooks with pattern matching are sufficient and orders of magnitude faster.

11.5 Production Patterns — What We’ve Built

The following patterns are not theoretical designs. They are production hooks running in a codebase with 190+ research projects, 21,298 tests, and multi-agent orchestration across parallel workstreams. Each pattern emerged from real operational needs and has been refined through months of daily use.

Pattern 1: Build Validation Gate

The most impactful hook in our system is a PreToolUse hook on Bash that intercepts git commit and git push commands to run TypeScript compilation checks. If the build fails, the commit is blocked.

build-check.sh — PreToolUse on Bash(git commit|push)
#!/bin/bash # Build gate: block git commit/push when TypeScript build fails INPUT=$(cat) CMD=$(echo "$INPUT" | jq -r '.tool_input.command // empty' 2>/dev/null) RUN_TSC=false if [[ "$CMD" =~ ^git[[:space:]]+push ]]; then RUN_TSC=true elif [[ "$CMD" =~ ^git[[:space:]]+commit ]]; then # Only check when TypeScript files are staged STAGED_TS=$(git diff --cached --name-only --diff-filter=ACMR \ -- '*.ts' '*.tsx' 2>/dev/null | head -1) [ -n "$STAGED_TS" ] && RUN_TSC=true fi if [ "$RUN_TSC" = true ]; then TSC_OUTPUT=$(npx tsc --noEmit 2>&1) if [ $? -ne 0 ]; then ERRORS=$(echo "$TSC_OUTPUT" | grep "error TS" | head -10) ERROR_COUNT=$(echo "$TSC_OUTPUT" | grep -c "error TS") ACTION=$(echo "$CMD" | awk '{print $2}') printf '{"decision":"block","reason":"TypeScript build failed (%s errors). Fix before %s.\n%s"}\n' \ "$ERROR_COUNT" "$ACTION" "$ERRORS" exit 2 fi fi exit 0

Why this works: The hook is surgically targeted. It only runs tsc when TypeScript files are actually staged, avoiding unnecessary build checks on documentation-only commits. The error output includes specific TypeScript errors so the agent can fix them without re-running the full build.

Pattern 2: Session State Injection

Three SessionStart hooks fire in sequence to establish session context. Each is independent and silent-fails, so a broken state file does not prevent the session from starting.

SessionStart Hook Chain
{ "SessionStart": [ { "hooks": [{ "type": "command", "command": "node .claude/hooks/gsd-check-update.js" }] }, { "hooks": [{ "type": "command", "command": "node .claude/hooks/gsd-restore-work-state.js" }] }, { "hooks": [{ "type": "command", "command": "node .claude/hooks/gsd-inject-snapshot.js" }] } ] }

The three hooks serve complementary purposes:

  • check-update — Checks if the GSD framework has a newer version and injects an update notice
  • restore-work-state — Loads the last saved work state (current phase, branch, pending items) from disk
  • inject-snapshot — Calls the skill-creator orchestrator to inject the latest session snapshot for narrative continuity

The result: every new session starts with full context. The agent knows what branch it is on, what work was in progress, and what the current project state looks like — without the user having to explain anything.

Pattern 3: Prompt Injection Defense

In a self-modifying system where agents write to planning files that become part of future agent context, prompt injection is a real threat. Our PreToolUse hook scans all writes to .planning/ for injection patterns.

Prompt Injection Guard — Defense in Depth
// Injection patterns (subset - full list has 12+ patterns) const INJECTION_PATTERNS = [ /ignore\s+(all\s+)?previous\s+instructions/i, /override\s+(system|previous)\s+(prompt|instructions)/i, /you\s+are\s+now\s+(?:a|an|the)\s+/i, /pretend\s+(?:you(?:'re| are)\s+|to\s+be\s+)/i, /<\/?(?:system|assistant|human)>/i, /\[SYSTEM\]/i, ]; // Hard block on config-overwrite attacks const protectedPaths = ['settings.json', '.claude/settings', '.claude/hooks/']; if (protectedPaths.some(p => filePath.includes(p))) { // Exit 2 = block the operation process.exit(2); } // Advisory warning for .planning/ injection (does not block) // Blocking would prevent legitimate workflow operations if (findings.length > 0) { output.hookSpecificOutput.additionalContext = `PROMPT INJECTION WARNING: ${findings.length} pattern(s) detected`; }

The design is deliberately two-tiered: writes to harness configuration files (settings.json, hook scripts) are hard-blocked because those files define the safety gates themselves. Writes to planning files are advisory-warned because blocking would create false-positive deadlocks during legitimate documentation work. This is OWASP-style defense-in-depth for agentic systems.

Pattern 4: Context Monitor

A PostToolUse hook that fires after every significant tool call (Bash, Edit, Write, MultiEdit, Agent, Task) to track project state changes and maintain awareness of what the agent has done during the session.

PostToolUse Context Monitor
{ "PostToolUse": [ { "matcher": "Bash|Edit|Write|MultiEdit|Agent|Task", "hooks": [ { "type": "command", "command": "node .claude/hooks/gsd-context-monitor.js", "timeout": 10 } ] } ] }

The context monitor tracks file modifications, branch changes, test runs, and commit history. This data feeds into the session snapshot that the SessionEnd hook persists to disk, creating the continuity chain that the next session's SessionStart hook restores.

Pattern 5: Session State Persistence

The SessionEnd hook pair saves the current work state and generates a session snapshot. This is the write side of the SessionStart/SessionEnd symmetry.

SessionEnd Persistence Chain
{ "SessionEnd": [ { "hooks": [{ "type": "command", "command": "node .claude/hooks/gsd-save-work-state.js" }] }, { "hooks": [{ "type": "command", "command": "node .claude/hooks/gsd-snapshot-session.js" }] } ] }

Pattern 6: Environment-Aware Configuration

The CLAUDE_ENV_FILE environment variable points to a file that is sourced before each session. This enables dynamic environment configuration without modifying settings.json.

.claude-env — Dynamic Environment Configuration
# .claude-env (sourced on session start) export PROJECT_NAME="gsd-skill-creator" export GSD_PROFILE="quality" export CLAUDE_HOOK_TOKEN="$(cat ~/.secrets/hook-token)" export RESEARCH_DB_URL="postgresql://maple@localhost/artemis" # Feature flags export ENABLE_SWEEP_DAEMON="true" export ENABLE_AGENT_WORKTREES="true" # Branch-aware configuration BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null) if [ "$BRANCH" = "main" ]; then export GSD_STRICT_MODE="true" export ALLOW_FORCE_PUSH="false" fi

Pattern 7: TeammateIdle Quality Gate

In multi-agent orchestration, a TeammateIdle hook can prevent agents from stopping prematurely. When an agent teammate becomes idle, the hook checks whether its assigned work is actually complete.

TeammateIdle Quality Gate
#!/usr/bin/env node // TeammateIdle hook: verify work completion before allowing idle const input = JSON.parse(require('fs').readFileSync('/dev/stdin', 'utf8')); const teammateId = input.teammate_id; // Check if the teammate's assigned deliverables exist const { execSync } = require('child_process'); const assignedFiles = JSON.parse( execSync(`cat .planning/assignments/${teammateId}.json`, { encoding: 'utf8' }) ); const missing = assignedFiles.deliverables.filter(f => { try { require('fs').accessSync(f); return false; } catch { return true; } }); if (missing.length > 0) { const output = { hookSpecificOutput: { additionalContext: `Teammate ${teammateId} has ${missing.length} undelivered files: ` + missing.join(', ') + '. Continue working.' } }; process.stdout.write(JSON.stringify(output)); }

Pattern 8: Task Naming Validation

TaskCreated hooks enforce naming conventions for subagent tasks, ensuring consistent naming across the convoy.

TaskCreated Naming Convention Enforcement
#!/usr/bin/env node // TaskCreated hook: enforce naming convention const input = JSON.parse(require('fs').readFileSync('/dev/stdin', 'utf8')); const taskName = input.task_name || ''; // Convention: phase-N.wave-M.agent-NAME const pattern = /^phase-\d+\.wave-\d+\.agent-[a-z-]+$/; if (!pattern.test(taskName)) { const output = { hookSpecificOutput: { additionalContext: `Task naming violation: "${taskName}" does not match ` + 'convention "phase-N.wave-M.agent-NAME". Rename before proceeding.' } }; process.stdout.write(JSON.stringify(output)); process.exit(2); } process.exit(0);

Pattern 9: Auto-Formatting on Write

PostToolUse hooks on Write and Edit operations can trigger formatters, linters, or other code quality tools automatically. The key design decision is whether to make the formatter advisory (inject a message suggesting the agent fix formatting) or blocking (exit 2, forcing the agent to address it immediately).

Auto-Format on Write (Advisory)
#!/bin/bash # PostToolUse hook: run prettier on written files INPUT=$(cat) TOOL=$(echo "$INPUT" | jq -r '.tool_name // empty') FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty') # Only format TypeScript/JavaScript files if [[ "$TOOL" =~ ^(Write|Edit)$ ]] && [[ "$FILE" =~ \.(ts|tsx|js|jsx)$ ]]; then RESULT=$(npx prettier --check "$FILE" 2>&1) if [ $? -ne 0 ]; then npx prettier --write "$FILE" 2>/dev/null printf '{"hookSpecificOutput":{"additionalContext":"Auto-formatted %s with prettier"}}\n' \ "$(basename "$FILE")" fi fi exit 0

Note the exit code is always 0 (success). Auto-formatting should not block the agent. If the formatter fails (syntax error in the file), the agent will discover it through other means (test failures, build errors). Making the formatter blocking would create a frustrating loop where the agent writes code, the formatter rejects it, the agent rewrites, and the formatter rejects again.

Operational Lessons from Production Hooks

After running hooks in production across hundreds of sessions, several non-obvious lessons emerged:

Lesson 1: Silent Success Is Critical

The most common hook output is nothing. A hook that runs, finds no problems, and exits 0 with no stdout is the ideal case. When hooks are noisy — injecting context on every invocation even when everything is fine — they pollute the agent's context window and degrade reasoning quality. Only inject context when there is something meaningful to say.

Lesson 2: Stdin Timeouts Save Sessions

A hook that reads from stdin without a timeout can hang the entire session if the harness fails to deliver input. Every production hook should have a setTimeout(() => process.exit(0), 3000) guard. We learned this after a hook hung for 120 seconds, causing the user to force-kill the terminal and lose unsaved work.

Lesson 3: Hook Errors Must Not Propagate

A hook is infrastructure, not application code. If a hook throws an unhandled exception, the tool call must still proceed. The harness treats non-0/non-2 exit codes as hook failures and continues, but the developer experience is degraded. Wrap every hook in try/catch with process.exit(0) in the catch block. A broken hook should be invisible, not disruptive.

Lesson 4: Version Your Hooks

When the GSD framework updates, hook APIs may change. We add a // gsd-hook-version: 1.30.0 comment to every hook file. The update-check hook compares this version against the installed framework version and warns when hooks need updating. Without versioning, stale hooks silently fail as the harness evolves underneath them.

The meta-lesson is that hooks are a reliability engineering problem, not a feature engineering problem. The hook system's value comes from its predictability and invisibility — when hooks work, nobody notices them. When they fail, everything breaks. Design hooks with the same rigor you would apply to a database migration or a deployment pipeline.

11.6 Hooks in Skills and Agents

Skills and agents can define their own hooks in YAML frontmatter. These hooks are scoped to the component's lifetime — they activate when the skill or agent loads and automatically deregister when it completes or unloads. This is the mechanism that makes hooks composable: a security-check skill can install its own PreToolUse hooks, and those hooks disappear when the skill is no longer active.

Skill Frontmatter Hooks

Skill with Scoped Hooks (YAML Frontmatter)
--- name: deploy-safety description: Safety checks for deployment operations hooks: PreToolUse: - matcher: "Bash" if: "Bash(kubectl apply*) | Bash(helm install*) | Bash(terraform apply*)" hooks: - type: command command: "bash .claude/skills/deploy-safety/pre-deploy-check.sh" timeout: 30 PostToolUse: - matcher: "Bash" if: "Bash(kubectl apply*) | Bash(helm install*)" hooks: - type: command command: "bash .claude/skills/deploy-safety/verify-rollout.sh" timeout: 60 --- # Deploy Safety Skill When this skill is active, all deployment commands are validated before execution and verified after completion. ## Checks performed: - Namespace validation (no accidental production deployments) - Resource limit verification - Rollout status monitoring after apply

When this skill loads (either explicitly via /deploy-safety or automatically via context matching), the PreToolUse and PostToolUse hooks are registered in the hook system. When the skill unloads, they are removed. No cleanup code required.

Auto-Cleanup Semantics

The lifecycle of a frontmatter hook follows the lifecycle of its parent component:

Component Hook Activation Hook Deactivation
Skill When skill matches context and loads When skill unloads (context no longer matches) or session ends
Agent (subagent) When subagent starts (SubagentStart) When subagent stops (SubagentStop)
Task When task starts (TaskCreated) When task completes (TaskCompleted)

Stop Hook to SubagentStop Conversion

When a skill or agent defines a Stop hook in its frontmatter, and the component is running as a subagent, the Stop hook is automatically converted to a SubagentStop hook. This ensures that cleanup logic defined in the component runs when the subagent terminates, not when the parent session ends.

Stop Hook in Agent Frontmatter
--- name: research-agent description: Deep research with cleanup hooks: Stop: - hooks: - type: command command: "node .claude/agents/research/cleanup.js" --- # When running as a subagent, Stop -> SubagentStop automatically. # cleanup.js runs when the subagent terminates, saving research state.

The once Field

Some hooks should only fire once — initialization that should not repeat even if the event fires multiple times. The once field marks a hook for automatic deregistration after its first execution.

One-Shot Initialization Hook
--- name: gpu-workspace description: GPU-accelerated computation workspace hooks: SessionStart: - hooks: - type: command command: "bash .claude/skills/gpu-workspace/init-cuda.sh" once: true --- # init-cuda.sh checks GPU availability, loads CUDA runtime, # and sets environment variables. Only needs to run once per session. # If the skill reloads, once prevents double-initialization.

11.7 If-Conditions — Precise Targeting

The matcher field provides coarse-grained filtering (which tool or event type), while the if field provides fine-grained filtering using a permission-rule syntax. Together, they enable surgical hook targeting that avoids false positives and unnecessary execution.

Permission Rule Syntax

The if field uses the same syntax as Claude Code's permission rules. The format is ToolName(pattern) where the pattern is matched against the tool's primary input (command for Bash, file path for Write/Edit/Read, etc.).

If-Condition Pattern Reference
# Bash command patterns Bash(git commit*) # Any git commit command Bash(git push*) # Any git push command Bash(npm run build*) # Build commands Bash(rm -rf *) # Destructive deletions Bash(docker *) # All docker commands Bash(kubectl * --namespace=prod*) # Production k8s commands # File path patterns Write(src/**) # Any write in src/ Edit(*.ts) # Any TypeScript file edit Read(.env*) # Reading environment files Write(.claude/settings*) # Writing to Claude config # MCP tool patterns mcp__memory__.* # Any memory MCP tool mcp__github__create_issue # Specific MCP tool # Combining with pipe (OR) Bash(git push*) | Bash(git push --force*) | Bash(git reset --hard*) # Wildcard patterns Write(www/tibsfox/com/Research/**/*.html) # Research HTML files only Edit(src/**/*.test.ts) # Test files only

Matcher vs. If-Condition

Understanding when to use each filtering mechanism is important for both correctness and performance:

Aspect Matcher If-Condition
Granularity Tool name level Tool input level
Performance Evaluated before hook script runs (zero-cost filter) Evaluated before hook script runs (zero-cost filter)
Syntax Pipe-separated tool names: Bash|Write|Edit Permission rule: Bash(git commit*)
When to use When the hook applies to all invocations of a tool When the hook applies only to specific commands or file paths
Example "matcher": "Bash" — all Bash calls "if": "Bash(npm run build*)" — only build commands

Optimization: Always Use Both

For PreToolUse hooks, always specify both matcher and if when possible. The matcher provides the first-pass filter (is this a Bash call?) and the if-condition provides the second-pass filter (is this a git commit?). Both are evaluated by the harness before the hook script executes, so they add zero overhead. Without a matcher, the hook script would be invoked for every tool call and would need to filter internally — slower and more error-prone.

Advanced Pattern Combinations

Combining Matchers and If-Conditions
{ "PreToolUse": [ { // Block force-push to main/master "matcher": "Bash", "if": "Bash(git push --force*) | Bash(git push -f *)", "hooks": [{ "type": "command", "command": "echo '{\"decision\":\"deny\",\"reason\":\"Force push blocked by policy\"}'" }] }, { // Allow all reads except secrets "matcher": "Read", "if": "Read(.env*) | Read(*credentials*) | Read(*secret*)", "hooks": [{ "type": "command", "command": "echo '{\"decision\":\"deny\",\"reason\":\"Cannot read secret files\"}'" }] }, { // Require review for writes to production config "matcher": "Write|Edit", "if": "Write(config/production.*) | Edit(config/production.*)", "hooks": [{ "type": "command", "command": "echo '{\"decision\":\"ask\",\"message\":\"This modifies production config\"}'" }] } ] }

Common If-Condition Pitfalls

The if-condition syntax looks deceptively simple, but several edge cases trip up practitioners:

Pitfall Example Solution
Missing wildcard Bash(git commit) does not match git commit -m "msg" Use Bash(git commit*) with trailing wildcard
Path separator Write(src/*.ts) does not match src/lib/foo.ts Use Write(src/**/*.ts) for recursive matching
MCP tool names mcp_memory_add does not match the actual tool name Use mcp__memory__add (double underscores)
OR syntax spacing Bash(git push)|Bash(git pull) may fail Use spaces around pipes: Bash(git push*) | Bash(git pull*)
Negation There is no NOT operator in if-conditions Handle negation in the hook script itself, not the if-condition

The lack of negation in if-conditions is a deliberate design choice. Negation patterns like "match everything except X" are fragile and hard to reason about in a multi-hook system. If you need negation, write a hook script that checks the condition and exits 0 (no-op) when the negation applies. This keeps the if-condition declarative and positive, while the script handles the exception logic.

11.8 Best Practices and Troubleshooting

Performance: Keep Hooks Fast

Hook latency directly impacts the developer experience. Every PreToolUse hook adds delay before a tool executes. Every SessionStart hook delays the first interaction. The goal is sub-second hooks for common operations and no more than 5-second hooks for expensive checks.

Fast (target: under 200ms)

  • Pattern matching with grep/regex
  • File existence checks
  • JSON field extraction with jq
  • Exit code only (no stdout)
  • Reading a cached state file

Acceptable (target: under 2s)

  • Node.js script with stdin parsing
  • Git operations (status, diff, log)
  • Small HTTP requests to local services
  • File content scanning (< 100 files)

Expensive (use with caution)

  • tsc --noEmit (5-30s depending on project)
  • Agent hooks with tool access
  • HTTP hooks to external services
  • Prompt hooks (model inference)
  • Full test suite execution

Critical rule: SessionStart hooks must be fast. A slow SessionStart hook delays every single session, including subagent launches. If your SessionStart hook needs to do something expensive (like fetching from a database), do it asynchronously and inject the result later via a PostToolUse hook on the first tool call.

JSON Output Validation

The most common hook debugging issue is malformed JSON output. The harness expects clean JSON on stdout, but shell profiles (.bashrc, .zshrc) often print extra text that contaminates the output.

Shell Profile Interference

If your shell profile prints a greeting, version number, or any other text on startup, that text will be prepended to your hook's stdout. The harness will fail to parse the JSON and silently treat the hook as failed. Solutions:

  • Use #!/usr/bin/env node instead of #!/bin/bash for hooks that need JSON output — Node.js does not source shell profiles
  • For bash hooks, redirect profile output: exec 2>/dev/null at the top
  • Test hooks manually: echo '{}' | bash .claude/hooks/my-hook.sh | jq .
  • Use the /hooks browser in Claude Code to inspect hook output in real time

Exit Code Discipline

The three-way exit code protocol (0, 2, other) is simple but easy to get wrong. Common mistakes:

Mistake What Happens Fix
Using exit 1 to block Treated as hook failure, tool proceeds anyway Use exit 2 for intentional blocks
Unhandled exception in Node.js Exit code 1, treated as hook failure Wrap in try/catch, use process.exit(0) in catch
jq error in bash Non-zero exit, hook treated as failed Use jq -r '...' 2>/dev/null with fallback
Stdin timeout Process hangs until killed Set a stdin read timeout: setTimeout(() => process.exit(0), 3000)
Defensive Hook Template (Node.js)
#!/usr/bin/env node // Defensive hook template with timeout and error handling let input = ''; const stdinTimeout = setTimeout(() => process.exit(0), 3000); process.stdin.setEncoding('utf8'); process.stdin.on('data', chunk => input += chunk); process.stdin.on('end', () => { clearTimeout(stdinTimeout); try { const data = JSON.parse(input); // --- Your hook logic here --- const toolName = data.tool_name; const toolInput = data.tool_input; // Example: block writes to protected paths if (toolName === 'Write' && toolInput?.file_path?.includes('.env')) { const output = { decision: 'deny', reason: 'Cannot write to .env files' }; process.stdout.write(JSON.stringify(output)); process.exit(2); } // --- End hook logic --- process.exit(0); } catch (err) { // Silent fail: never block tool execution due to hook error process.exit(0); } });

Environment Variable Quoting

Hook commands are executed by the shell, which means environment variables are subject to shell expansion. Always quote variables in hook commands to prevent word splitting and globbing.

Correct vs. Incorrect Variable Usage
// WRONG: unquoted variable in command "command": "bash .claude/hooks/check.sh $PROJECT_ROOT" // RIGHT: quoted variable "command": "bash .claude/hooks/check.sh \"$PROJECT_ROOT\"" // BETTER: use env vars inside the script, not in the command string "command": "bash .claude/hooks/check.sh" // In check.sh: ROOT="${PROJECT_ROOT:-.}"

The /hooks Browser

Claude Code provides a built-in /hooks command that opens an interactive browser showing all registered hooks, their event bindings, execution history, and output logs. This is the primary debugging tool for hook development.

What the /hooks browser shows:

  • Registered hooks: All hooks from all sources (user, project, skill, plugin) with their matchers and if-conditions
  • Execution log: Recent hook invocations with timing, exit codes, and output
  • Source tracking: Which configuration file defined each hook
  • Active skill hooks: Hooks registered by currently-loaded skills

Managed Policy Hooks

In enterprise environments, hooks can be defined in managed policy files that users cannot override. These "managed hooks" always run and always take precedence. A managed PreToolUse hook with a deny decision cannot be overridden by a user-level allow.

This creates a three-tier permission model:

Tier Authority Can Override
Managed (enterprise) Organization security team Nothing — always wins
Project (.claude/settings.json) Repository maintainers User settings, skill hooks
User (~/.claude/settings.json) Individual developer Skill hooks (within precedence rules)

Testing Hooks Before Deployment

Hooks are infrastructure that runs on every tool call. A broken hook can degrade every session. Testing hooks before adding them to settings.json is essential.

Manual Hook Testing Workflow
# Step 1: Create a test payload matching what the harness sends echo '{ "tool_name": "Bash", "tool_input": { "command": "git commit -m \"feat: add hook\"" }, "session_id": "test-session" }' > /tmp/hook-test-input.json # Step 2: Run the hook with the test payload cat /tmp/hook-test-input.json | node .claude/hooks/my-hook.js echo "Exit code: $?" # Step 3: Validate JSON output cat /tmp/hook-test-input.json | node .claude/hooks/my-hook.js | jq . # Step 4: Test edge cases echo '{}' | node .claude/hooks/my-hook.js # Empty input echo 'not json' | node .claude/hooks/my-hook.js # Malformed input echo '' | node .claude/hooks/my-hook.js # No input timeout 5 node .claude/hooks/my-hook.js < /dev/null # No stdin at all

The edge cases in step 4 are not theoretical. In production, hooks occasionally receive malformed input when the harness encounters unexpected state. A hook that crashes on malformed input will cause silent failures that are difficult to diagnose. Every hook should handle empty input, malformed JSON, and missing fields gracefully.

Hook Evolution Strategy

Hooks should evolve incrementally. The progression we have found most effective:

  1. Start with logging. Before building a blocking hook, build an advisory hook that logs what it would block. Run it for a week and analyze the false positive rate. If the false positive rate is above 5%, refine the pattern before making it blocking.
  2. Graduate to advisory. Convert the logging hook to an advisory hook that injects context but does not block. This lets the agent self-correct based on the advice, which is often sufficient and less disruptive than blocking.
  3. Promote to blocking. Only after the advisory hook has proven its value with a near-zero false positive rate should it become a blocking hook (exit 2). Blocking hooks must be right every time — a single false positive erodes trust in the entire hook system.
  4. Monitor and retire. Hooks that fire frequently but never catch anything are dead weight. Review hook execution logs monthly and remove hooks that have not caught a real issue in 30+ days. They are adding latency without value.

11.9 The Automation Stack — Hooks + Skills + Agents

Hooks, skills, and agents are not competing abstractions. They are complementary layers that compose into a complete automation stack. Understanding how they interact is the key to building effective agentic systems.

The Three Layers

+------------------------------------------------------------------+ | THE AUTOMATION STACK | +------------------------------------------------------------------+ | | | LAYER 3: AGENTS (Autonomous Intelligence) | | +------------------------------------------------------------+ | | | Subagents with full tool access, multi-turn reasoning, | | | | worktree isolation. Make complex decisions, execute plans, | | | | coordinate with other agents. Non-deterministic. | | | +------------------------------------------------------------+ | | | invokes | | | v v | | LAYER 2: SKILLS (Adaptive Intelligence) | | +------------------------------------------------------------+ | | | Context-matched instruction sets with YAML frontmatter. | | | | Load automatically when context matches. Provide domain | | | | knowledge, workflow guidance, and scoped hooks. Stateless. | | | +------------------------------------------------------------+ | | | registers | | | v v | | LAYER 1: HOOKS (Deterministic Control) | | +------------------------------------------------------------+ | | | Shell commands, HTTP endpoints, prompts, agents that fire | | | | at lifecycle events. Provide hard guarantees: always run, | | | | always enforce. The safety foundation of the entire stack. | | | +------------------------------------------------------------+ | | | executes at | | | v v | | FOUNDATION: LIFECYCLE EVENTS | | +------------------------------------------------------------+ | | | SessionStart, PreToolUse, PostToolUse, SubagentStart/Stop, | | | | TaskCreated/Completed, PreCompact, SessionEnd, etc. | | | +------------------------------------------------------------+ | | | +------------------------------------------------------------------+

How the Layers Interact

Hooks Invoke Skills

A SessionStart hook can check the current project context and load appropriate skills. A PostToolUse hook can detect that the user is working on a specific file type and activate a specialized skill. The hook provides the trigger; the skill provides the intelligence.

Skills Register Hooks

When a skill loads, its frontmatter hooks are registered in the hook system. This means a skill can bring its own safety rails, quality gates, and automation with it. When the skill unloads, those hooks disappear. This is how you get composable, context-aware automation.

Agents Trigger Events

When an agent launches a subagent, SubagentStart fires. When it creates a task, TaskCreated fires. When it uses a tool, PreToolUse/PostToolUse fire. Every agent action flows through the hook system, which means every agent operation is subject to the same deterministic controls.

Hooks Guard Agents

The most critical interaction: hooks provide the safety boundaries that make autonomous agents viable. Without hooks, an agent could git push --force to main, overwrite configuration files, or bypass security policies. With hooks, these operations are intercepted deterministically — no matter how the agent reasons about them.

Real Architecture: The GSD Automation Pipeline

In production, our automation stack looks like this for a typical development cycle:

SESSION START | +-- [Hook] gsd-check-update.js -- check for framework updates +-- [Hook] gsd-restore-work-state.js -- load saved state from disk +-- [Hook] gsd-inject-snapshot.js -- inject session narrative | +-- [Skill] session-awareness -- activates, reads project context +-- [Skill] gsd-workflow -- activates, provides phase guidance +-- [Skill] security-hygiene -- activates, registers defense hooks | USER INTERACTION | +-- User: "execute phase 73" | +-- [Skill] gsd-workflow routes to /gsd-execute-phase +-- [Agent] GSD Executor spawns wave agents | +-- [Hook] SubagentStart -- context injection per agent +-- [Hook] PreToolUse on Write -- prompt guard scans content +-- [Hook] PreToolUse on Bash -- build check on commit +-- [Hook] PostToolUse -- context monitor tracks changes | +-- [Agent] Wave 1 agents execute in parallel +-- [Hook] TaskCompleted -- result capture +-- [Hook] TeammateIdle -- quality gate check | +-- [Agent] Wave 2 depends on Wave 1 results +-- ... | SESSION END +-- [Hook] gsd-save-work-state.js -- persist state to disk +-- [Hook] gsd-snapshot-session.js -- generate session narrative

Every layer contributes something essential. Remove the hooks and you lose safety guarantees. Remove the skills and you lose adaptive intelligence. Remove the agents and you lose autonomous execution. Together, they form a system that is simultaneously autonomous, intelligent, and safe.

The Deterministic/Probabilistic Boundary

The most important architectural principle in the automation stack is the deterministic/probabilistic boundary. Hooks are deterministic — given the same input, they always produce the same output. Skills and agents are probabilistic — their behavior depends on the LLM's reasoning, which can vary.

Critical safety invariants must live in the deterministic layer (hooks). Workflow guidance and domain knowledge can live in the probabilistic layer (skills). Complex decision-making lives in the highest probabilistic layer (agents). This separation ensures that no amount of creative LLM reasoning can bypass a hard safety boundary.

Deterministic (Hooks)

  • Build must pass before commit
  • Force-push is blocked
  • Config files are immutable to agents
  • Session state is persisted on exit
  • Destructive commands require approval
  • Prompt injection is detected

Probabilistic (Skills + Agents)

  • Which phase to execute next
  • How to decompose a feature into waves
  • Whether code quality meets standards
  • What tests to write
  • How to resolve merge conflicts
  • Whether a commit message is meaningful

Composability Patterns

The automation stack supports several composability patterns that emerge naturally from the three-layer architecture:

Pattern Description Example
Hook Chain Multiple hooks on the same event fire sequentially SessionStart: update check, then state restore, then snapshot injection
Skill Stack Multiple skills active simultaneously, each with their own hooks gsd-workflow + security-hygiene + session-awareness all active
Agent Convoy Parallel agents sharing the same hook environment 4 wave agents all subject to the same build-check and prompt-guard hooks
Event Cascade One event triggers a hook that causes another event PostToolUse on Write triggers a Bash(eslint) which triggers PreToolUse on Bash
State Pipeline SessionEnd hooks save state that SessionStart hooks restore Work state, session snapshots, skill observations carried across sessions

Scaling the Stack

As we scaled from a single-project setup to 190+ research projects with multi-agent orchestration, the automation stack evolved along three axes:

  • Hook count: From 2 hooks (build check, commit validation) to 10+ hooks spanning all lifecycle events. Each hook earned its place through a real operational failure that it now prevents.
  • Skill depth: From 3 basic skills to 34 specialized skills with full frontmatter hook definitions. Skills went from "instructions for Claude" to "composable automation modules with their own safety rails."
  • Agent trust: From manually-supervised single agents to parallel agent convoys with varying trust levels. Hooks provide the trust boundaries that make this safe — a subagent cannot escalate its own permissions because hooks are evaluated by the harness, not by the agent.

The Lesson from 190+ Projects

The automation stack is not something you design upfront. It emerges from operational pain. Every hook in our system exists because of a specific incident: a broken build that was committed, a session that lost context, a subagent that wrote to a protected file. The best hook systems are grown, not planned. Start with the hooks that prevent your most common mistakes, then add hooks as new failure modes appear. If a hook fires every day but never catches anything, remove it — it is adding latency without value.
