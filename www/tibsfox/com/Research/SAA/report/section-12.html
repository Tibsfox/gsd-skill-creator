12.1 From Single Sessions to Agent Teams

The evolution of Claude Code's execution model traces a clear arc from constrained to expansive. In the beginning, a single Claude session handled everything: reading code, reasoning about architecture, making edits, running tests. This was already transformative compared to traditional IDE tooling, but it hit fundamental scaling limits. A single session is one thread of consciousness working on one thing at a time. Complex projects demand parallel thought.

The first major expansion was subagents, introduced through the Task tool. A lead session could spawn a child process, hand it a prompt, wait for the result, and continue. This was powerful for delegation — send a subagent to run tests while the lead works on refactoring, or spawn a researcher to explore a codebase while the lead plans the implementation. But subagents are ephemeral. They execute, return a result, and vanish. There is no ongoing conversation between them, no shared workspace, no persistent coordination.

Agent Teams represent the next evolutionary step. Instead of spawning disposable workers, you create persistent teammates that share a task list, communicate through a mailbox system, and coordinate their work in real time. Each teammate is a full Claude Code session with its own context window, running in parallel. The lead orchestrates but does not micromanage — teammates claim tasks, report progress, send messages, and operate with genuine autonomy.

The Three Generations of Claude Code Execution

Generation 1 — Single Session: One agent, sequential execution, entire conversation in one context window. Sufficient for small tasks, but every context switch costs tokens and cognition.

Generation 2 — Subagents: Lead + ephemeral workers via Task tool. Fire-and-forget delegation. No inter-worker communication. Good for parallelizable subtasks with clear boundaries.

Generation 3 — Agent Teams: Lead + persistent teammates with shared task list, mailbox communication, and coordinated file access. Full multi-session orchestration. The model for complex, long-running work.

When to Use Teams vs Subagents

The decision between subagents and teams is not about one being "better" — it is about matching the coordination model to the work structure. Subagents are the right choice when the work is parallelizable but independent: run five test suites simultaneously, search multiple directories, or validate different aspects of a change. Each worker needs no knowledge of what the others are doing.

Teams become necessary when the work requires ongoing coordination. When teammates need to communicate about what they have found, when one person's output feeds into another's input, when shared state must be managed carefully to prevent conflicts. A multi-layer feature implementation, for example, where the database schema informs the API design which informs the frontend — that demands a team, not a collection of disconnected subagents.

Dimension Subagents (Task tool) Agent Teams
Lifetime Ephemeral — execute and return Persistent — run alongside lead
Communication Result only — one-way return Bidirectional mailbox + broadcast
Shared state None — isolated context Shared task list, file system
Token cost Moderate — spawn cost only High — each teammate is a full session
Coordination overhead Minimal — no inter-worker sync Significant — task management, messaging
Ideal for Independent, parallelizable subtasks Interdependent, coordinated work
Max practical count 3–5 concurrent subagents 3–5 teammates (diminishing returns beyond)
Context inheritance Explicit via prompt only CLAUDE.md loaded, lead history NOT carried

Architecture: The Team Model

An agent team consists of four core components:

Team Lead

Your primary Claude Code session. The lead creates teammates, assigns initial tasks, coordinates work, and synthesizes results. The lead is always you — there is no promotion mechanism. If the lead session ends, the team dissolves.

Teammates

Independent Claude Code sessions running in parallel. Each teammate has its own context window, tool access, and permission scope. Teammates can read and write files, run commands, and use all tools available to the lead (minus team management tools specific to the lead role).

Shared Task List

A centralized list of work items with states (pending, in_progress, completed), assignments, and dependencies. Both the lead and teammates can create, update, and complete tasks. This is the coordination backbone.

Mailbox System

Asynchronous message passing between team members. The lead can send targeted messages to specific teammates or broadcast to all. Teammates can message the lead or other teammates. Messages are delivered automatically when the recipient checks for them.

Token Cost Implications

Agent Teams consume substantially more tokens than single-session or subagent workflows. Each teammate is a full Claude Code session with its own context window, system prompt, CLAUDE.md loading, and conversation history. A team of four teammates effectively runs five concurrent sessions (four teammates plus the lead). At typical session lengths, a 30-minute team operation can consume 3–5x the tokens of a sequential single-session approach to the same work.

This cost is justified when the parallelism delivers genuine value — when the wall-clock time savings matter, when the coordination overhead is lower than the serialization cost, or when the problem genuinely requires multiple simultaneous perspectives (such as adversarial code review). It is not justified for tasks that could be handled sequentially by a single session with subagent delegation.

Practitioner Guidance: Token Economics

Budget 3–5x your normal session token cost when using teams. The multiplier depends on team size and task duration. Short, focused team sessions (15–20 minutes of coordinated work) offer the best cost-to-value ratio. Extended team sessions where teammates idle waiting for dependencies burn tokens on context maintenance without producing output.

12.2 Starting and Configuring Teams

Agent Teams are an experimental feature that requires explicit opt-in. This is deliberate — teams are powerful but resource-intensive, and Anthropic gates them behind a feature flag to ensure practitioners make a conscious choice to use them.

Enabling Agent Teams

Set the environment variable before launching Claude Code:

Bash 
# Enable agent teams for the current session
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# Launch Claude Code normally
claude

# Or combine in a single command
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 claude

Alternatively, you can persist the setting in your shell profile (~/.bashrc, ~/.zshrc) so teams are always available:

Shell Profile (~/.bashrc or ~/.zshrc) 
# Always enable agent teams
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

Natural Language Team Creation

Once enabled, you create teams through natural language. There is no special command syntax — you simply describe the team you want and Claude creates it. This is one of the most elegant aspects of the design: the team creation interface is the same conversational interface you use for everything else.

Team Creation Prompts 
# Simple team creation
"Create a team with 3 teammates to review this PR"

# Specialized team with roles
"Create a team for implementing the authentication module:
 - A backend teammate for the API routes and middleware
 - A frontend teammate for the login/signup components
 - A test teammate to write integration tests as we go"

# Research fleet
"Create a team of 4 researchers. Each should investigate one of:
 1. Current authentication libraries for Node.js
 2. OAuth 2.0 best practices for SPAs
 3. JWT vs session-based auth tradeoffs
 4. Rate limiting and brute force protection"

When you issue a team creation prompt, the lead spawns each teammate as a separate Claude Code session. Each teammate receives an initial prompt that includes their role description, their assigned tasks, and instructions for how to coordinate with the rest of the team.

Display Modes

Agent Teams support two display modes that determine how you interact with the team visually:

In-Process Mode (Default)

All teammates run within the same terminal window as the lead. You see the lead's output by default and can cycle through teammates using Shift+Down arrow. This is the simpler setup — no external tools required — but it can feel cramped when managing more than 2–3 teammates since you can only see one session at a time.

Best for: Quick teams, 2–3 teammates, when you primarily interact with the lead.

Split-Pane Mode (tmux / iTerm2)

Each teammate gets its own terminal pane, allowing you to see all teammates simultaneously. Requires tmux on Linux/macOS or iTerm2's native split-pane support on macOS. This is the preferred mode for serious team work — you can watch each teammate's progress in real time, spot issues early, and intervene precisely.

Best for: Complex teams, 3–5 teammates, when you need simultaneous visibility.

Configuration via ~/.claude.json

The teammateMode setting in your Claude configuration controls the default display mode:

~/.claude.json 
{
  "teammateMode": "in_process",   // or "tmux" or "iterm2"
  "permissions": {
    "allow": [
      "Read",
      "Write",
      "Edit",
      "Bash(npm test:*)",
      "Bash(git status)",
      "Bash(git diff)"
    ]
  }
}

The permissions you set apply to all teammates. This is important — if you pre-approve common operations like reads, writes, and test execution, your teammates can work autonomously without stopping for permission prompts. Without pre-approval, every file write from every teammate triggers a permission request, which rapidly makes team work impractical.

Team Storage

Claude Code stores team configuration and task state on disk, enabling state persistence within a session:

Team Storage Structure 
~/.claude/
  teams/
    {team-name}/
      config.json           # Team configuration, member definitions
  tasks/
    {team-name}/
      task-001.json          # Individual task state
      task-002.json          # Assignment, status, dependencies
      task-003.json          # Completion data, results

This file-based storage model means that within a session, tasks survive teammate restarts and the lead can inspect task state directly if needed. However, once the lead session ends, the team configuration is not automatically resumable — a limitation we discuss in Section 12.9.

12.3 Team Coordination Mechanics

The coordination layer is what separates Agent Teams from simply running multiple Claude Code sessions in separate terminals. Without coordination, parallel sessions create merge conflicts, duplicate work, and contradictory changes. The team coordination mechanics solve these problems through structured task management, dependency tracking, and communication channels.

Task Lifecycle

Every piece of work flows through a defined state machine:

Task State Machine

pending → in_progress → completed

Tasks begin as pending when created by the lead or a teammate. A teammate (or the lead) claims a pending task, moving it to in_progress. When the work is done, the task moves to completed with an optional result or summary. Tasks can also be marked as blocked if they depend on other incomplete tasks.

The lead creates the initial task list during team setup. This is where task design matters enormously — tasks that are too granular create coordination overhead, while tasks that are too large prevent parallelism. The sweet spot, validated across dozens of team sessions, is 5–6 tasks per teammate. This gives each teammate enough work to stay productive without requiring constant check-ins, while keeping individual tasks small enough to complete in a reasonable time frame.

Task Dependencies and Automatic Unblocking

Tasks can declare dependencies on other tasks. A task with unmet dependencies remains in a blocked state and will not be claimed by any teammate. When the dependency completes, the blocked task automatically becomes available. This creates natural ordering without requiring the lead to manually sequence work.

Task Dependencies Example 
// Task list with dependency chain
Task 1: "Define the database schema for user authentication"
  Status: completed
  Assignee: backend-teammate

Task 2: "Implement the User model and migrations"
  Status: in_progress
  Assignee: backend-teammate
  Depends on: Task 1 // auto-unblocked when Task 1 completed

Task 3: "Create API routes for /auth/login and /auth/register"
  Status: pending (blocked)
  Depends on: Task 2 // will unblock when Task 2 completes

Task 4: "Write frontend login form component"
  Status: in_progress
  Assignee: frontend-teammate
  Depends on: none — can start immediately using API contract

Task 5: "Integration tests for auth flow"
  Status: pending (blocked)
  Assignee: test-teammate
  Depends on: Task 3, Task 4 // needs both backend and frontend

This dependency model eliminates a common failure mode: teammates racing ahead on work that depends on incomplete upstream changes. Without dependencies, you get the test writer trying to test endpoints that do not exist yet, or the frontend teammate integrating against an API that is still being designed.

Self-Claiming vs Lead-Assigned Tasks

Tasks can be either pre-assigned by the lead during creation or left unassigned for teammates to claim. The two approaches serve different coordination styles:

Lead-Assigned Tasks

The lead explicitly assigns each task to a specific teammate when creating the task list. This ensures each teammate works within a defined scope and prevents overlap. Preferred when teammates have distinct specializations (e.g., a security reviewer should not claim frontend tasks).

Self-Claimed Tasks

Tasks are created without assignment and teammates claim them based on availability. The first teammate to claim a task owns it. This maximizes throughput when teammates are fungible — any can do any task — because fast finishers can pick up more work without waiting for the lead to reassign.

In practice, a hybrid approach works well: assign the first round of tasks explicitly to ensure each teammate starts in the right area, then leave subsequent tasks unassigned for self-claiming as teammates finish their initial work.

File Locking for Race Condition Prevention

When multiple teammates are editing the same codebase, file conflicts are a real risk. Two teammates editing the same file simultaneously will produce unpredictable results — one set of changes will overwrite the other, or worse, the file will end up in an inconsistent state that neither teammate intended.

The team coordination layer implements file-level locking. When a teammate begins editing a file, other teammates are blocked from modifying that same file until the lock is released. This is not a hard filesystem lock — it operates at the coordination protocol level — but it prevents the most common class of concurrent modification errors.

File Ownership Best Practice

Rather than relying on file locking as a safety net, design your task decomposition so that each teammate owns different files. If you are implementing a feature that touches the database, API, and frontend, assign each layer to a different teammate and make clear which directories they own. This eliminates contention entirely rather than merely managing it.

Teammate Communication

The mailbox system supports two communication primitives:

Direct Message (SendMessage)

Send a message to a specific teammate by name. Used for targeted coordination: "Hey backend-teammate, what is the response shape for the /auth/login endpoint? I need it for the frontend form." The recipient sees the message next time it checks for incoming messages, which happens automatically during idle periods.

Broadcast

Send a message to all teammates simultaneously. Used for announcements that affect everyone: "I have changed the database schema — the users table now has a verified_at column. Update your code accordingly." Broadcast is the team equivalent of a Slack channel message.

Idle Notifications and Automatic Message Delivery

When a teammate has no active tasks and is not currently executing a tool, it enters an idle state. During idle, teammates automatically check for new messages and newly available tasks. This creates a natural cadence: work, check for messages, claim next task, work, repeat.

The lead receives notifications when teammates go idle, which serves as a signal that the teammate either needs more work or has completed everything assigned. This prevents the silent-teammate problem where a teammate finishes but no one notices for minutes.

Task Sizing Guidance

Task sizing is the single most important design decision when setting up a team. Based on extensive production use, the guidance is clear:

5–6
Tasks per teammate (optimal)
10–20 min
Per-task completion target
45–90 min
Total team session duration

Tasks smaller than 5 minutes create excessive coordination overhead — the teammate spends more time claiming and reporting than working. Tasks larger than 30 minutes risk context drift and make it hard for the lead to detect when something has gone wrong. The 10–20 minute range gives teammates enough runway to make meaningful progress while keeping the feedback loop tight enough for the lead to steer.

12.4 Teammate Control

The lead maintains control over the team through several mechanisms. This is not about micromanagement — it is about having the tools to steer, inspect, and intervene when the team's work diverges from the intended direction.

Navigating Teammates (In-Process Mode)

When running in in-process mode, you view one teammate at a time. Navigation uses keyboard shortcuts:

In-Process Navigation 
# Cycle through teammates
Shift+Down Arrow     # Move to next teammate's view
Shift+Up Arrow       # Move to previous teammate's view

# The status bar shows:
# - Current teammate name
# - Active task
# - Message indicator (if unread messages)

In split-pane mode (tmux/iTerm2), each teammate gets its own pane and you can see all of them simultaneously. Navigate between panes using your terminal multiplexer's standard key bindings (e.g., Ctrl+B then arrow keys in tmux).

Direct Messaging to Teammates

From the lead's session, you can send a direct message to any teammate by name. This is how you provide course corrections, ask for status updates, or share information that a specific teammate needs:

Directing Teammates 
# Send a directive to a specific teammate
"Tell the backend-teammate to use bcrypt instead of argon2
 for password hashing — our production servers don't have
 the argon2 native module installed."

# Ask for a status update
"Ask the test-teammate how many tests are passing now."

# Share context
"Send a message to all teammates: the API base URL has
 changed from /api/v1 to /api/v2. Update all references."

Plan Approval Workflow

For tasks that involve significant changes, the lead can require plan approval before implementation proceeds. This creates a review gate that prevents teammates from making large, potentially wrong changes without oversight.

The approval workflow follows four stages:

Stage Actor Action
1. Read-Only Analysis Teammate Explore the codebase, understand the existing architecture, identify files to modify
2. Plan Proposal Teammate Write a plan describing what files will be changed, what the changes will do, and why
3. Approve or Reject Lead Review the plan. Approve to proceed, reject with feedback to revise
4. Implement Teammate Execute the approved plan. Changes should match the approved scope

Plan approval adds overhead but prevents a common failure mode: a teammate implementing an approach that conflicts with the lead's mental model or another teammate's work. For teams of 4–5 where the stakes are high (production code, database migrations, security-critical changes), plan approval is worth the cost.

Shutdown Requests

Teammates can be shut down gracefully by the lead. The shutdown process gives the teammate a chance to finish its current tool execution, save any work in progress, and report a final status. This is preferable to an abrupt termination, which can leave files in an inconsistent state.

Graceful Shutdown 
# Graceful shutdown of a specific teammate
"Shut down the frontend-teammate — we've decided to
 handle the UI in a separate session."

# Shutdown with final task
"Tell the test-teammate to finish its current test file
 and then shut down. We have enough coverage."

# Full team shutdown
"Wind down the team. Have everyone commit their changes
 and report what they accomplished."

Team Cleanup

All team lifecycle operations — creation, modification, shutdown — flow through the lead. Teammates cannot create other teammates, shut down peers, or modify the team configuration. This single-point-of-control model prevents chaotic self-modification and ensures the human operator (through the lead) always maintains authority over team composition.

When the lead session ends, all teammates are terminated. There is no "headless team" mode where teammates continue without a lead. This is a deliberate safety constraint: an unsupervised team of AI agents modifying code with no human in the loop is a risk that the current design explicitly prevents.

12.5 Using Subagent Definitions for Teammates

The .claude/agents/ directory, used to define specialized subagents with custom prompts and tool restrictions (as covered in Section 3), can also serve as the foundation for teammate configuration. When creating a team, you can reference an agent definition by name, and the teammate inherits the definition's prompt and tool restrictions.

How Agent Definitions Apply to Teammates

When you reference a .claude/agents/ definition for a teammate:

  • Tools allowlist is honored: If the agent definition restricts tools to ["Read", "Glob", "Grep"], the teammate can only use those tools, regardless of the lead's broader permissions. This enforces the principle of least privilege at the teammate level.
  • Body is appended as instructions: The markdown body of the agent definition file becomes the teammate's system instructions, guiding its behavior and focus areas.
  • Team coordination tools always available: Regardless of the tools allowlist, teammates always retain access to SendMessage, task management, and other team coordination tools. These are infrastructure, not application-level tools, and restricting them would break coordination.

Important Limitation

skills and mcpServers specified in agent definitions are not applied to teammates. Only the tools allowlist and the body instructions carry over. If your agent definition relies on specific MCP servers or skills, those capabilities will not be available to the teammate. Plan your agent definitions accordingly when they are intended for team use.

Agent Definition Structure

.claude/agents/security-reviewer.md 
---
tools:
  - Read
  - Glob
  - Grep
  - Bash(grep)
  - Bash(git log)
  - Bash(git diff)
---

# Security Reviewer

You are a security-focused code reviewer. Your job is to identify
vulnerabilities, insecure patterns, and compliance issues.

## Review Checklist

- [ ] Input validation and sanitization
- [ ] Authentication and authorization checks
- [ ] SQL injection, XSS, CSRF vulnerabilities
- [ ] Secrets/credentials in code or config
- [ ] Dependency vulnerabilities (known CVEs)
- [ ] Error handling that leaks internal details
- [ ] Rate limiting and resource exhaustion
- [ ] Cryptographic misuse (weak algorithms, hardcoded keys)

## Output Format

For each finding, report:
1. **Severity**: Critical / High / Medium / Low / Info
2. **Location**: File path and line number
3. **Issue**: What the vulnerability is
4. **Impact**: What an attacker could do
5. **Fix**: Specific remediation steps
.claude/agents/test-runner.md 
---
tools:
  - Read
  - Write
  - Edit
  - Glob
  - Grep
  - Bash(npm test:*)
  - Bash(npx vitest:*)
  - Bash(npx jest:*)
---

# Test Runner

You write and execute tests. You have full read/write access to
test files and can run test commands, but cannot modify source code.

## Guidelines

- Write tests BEFORE reporting a task as complete
- Use the project's existing test framework and patterns
- Cover happy path, edge cases, and error conditions
- Each test file should be self-contained
- Run the full test suite after writing to catch regressions
- Report: tests written, tests passing, tests failing, coverage delta

Referencing Definitions When Creating Teams

Team Creation with Agent Definitions 
# Reference agent definitions by name
"Create a team for reviewing the authentication PR:
 - Use the security-reviewer agent as a security teammate
 - Use the test-runner agent as a testing teammate
 - Create a general code-quality teammate for style and patterns"

# The security-reviewer teammate will be restricted to
# Read/Glob/Grep/Bash(grep)/Bash(git log)/Bash(git diff)
# It cannot write files — true read-only security review.
#
# The test-runner can write test files and run tests
# but cannot modify source code.

This pattern — defining specialized agents in .claude/agents/ and reusing them as teammates — creates a library of roles that can be composed into different team configurations for different tasks. A project might maintain a dozen agent definitions and combine 3–5 of them per team session based on the work at hand.

Custom Teammate Definitions vs Generic Teammates

Aspect Agent Definition Teammate Generic Teammate
Tool access Restricted to definition's allowlist Inherits lead's full tool access
Instructions Definition body + lead's spawn prompt Lead's spawn prompt only
Behavioral consistency High — follows defined checklist/format Variable — depends on prompt quality
Reusability Reusable across teams and sessions One-off, defined per team creation
Setup effort Upfront investment in definition files Zero — just describe in natural language

For teams you create frequently (code review, release preparation, research), invest in agent definitions. For one-off team tasks, generic teammates with detailed spawn prompts are faster to set up and sufficient for the purpose.

12.6 Permissions, Context, and Communication

Understanding how context, permissions, and communication flow through a team is essential for effective team design. Several aspects of team behavior are non-obvious and, if misunderstood, lead to teammates that are confused, blocked, or working on the wrong thing.

Permission Inheritance

Teammates inherit the lead's permission settings. If the lead has pre-approved Read, Write, Edit, and Bash(npm test) in ~/.claude.json, every teammate gets those same approvals. This is a flat inheritance model — there is no per-teammate permission override at the team level (though agent definitions can restrict tools to a subset, as covered in Section 12.5).

The practical implication: configure your permissions before creating the team. If teammates will need to write files, edit code, and run tests, pre-approve those operations. Otherwise, you will spend your entire team session approving permission requests from four teammates simultaneously, which is both tedious and defeats the purpose of parallel work.

Recommended Permission Pre-Approval for Teams 
// ~/.claude.json — permissions for a typical team session
{
  "permissions": {
    "allow": [
      "Read",
      "Write",
      "Edit",
      "Glob",
      "Grep",
      "Bash(npm test:*)",
      "Bash(npx vitest:*)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(ls:*)"
    ],
    "deny": [
      "Bash(git push:*)",
      "Bash(git reset:*)",
      "Bash(rm -rf:*)"
    ]
  }
}

CLAUDE.md Loading

Each teammate loads CLAUDE.md files the same way a regular Claude Code session does. The project-level CLAUDE.md, any directory-level CLAUDE.md files, and the user-level instructions all apply. This means teammates automatically receive project conventions, coding standards, and behavioral instructions without the lead needing to repeat them in spawn prompts.

This is a significant advantage over subagents, which receive only their spawn prompt as context. A teammate working on a TypeScript project inherits the "use single quotes, no semicolons, functional style" instruction from CLAUDE.md automatically. A subagent would need that instruction explicitly included in its prompt.

Lead Conversation History Does NOT Carry Over

This is the single most important thing to understand about team context, and the most common source of confusion for new team users: teammates do not see the lead's conversation history.

When you have spent 20 minutes discussing architecture with the lead, exploring the codebase, and forming a plan, and then you create a team — the teammates start with a blank conversation. They have CLAUDE.md context and their spawn prompt, but they have no idea what you and the lead discussed. Every relevant decision, every piece of context, every constraint that matters must be explicitly included in the teammate's spawn prompt or communicated via messages.

The Context Transfer Problem

This is the hardest part of team management. You must write spawn prompts that include enough context for the teammate to be effective without including so much that the prompt itself consumes significant context window. The skill is in identifying what the teammate needs to know vs what it can discover on its own by reading code and CLAUDE.md.

Rule of thumb: Include architectural decisions, constraints, and the "why" in spawn prompts. Let teammates discover the "what" (existing code structure, file locations) through their own exploration.

Effective Spawn Prompts

A spawn prompt for a teammate should answer four questions:

  1. What is your role? — "You are the backend teammate responsible for API routes."
  2. What is the context? — "We are implementing OAuth 2.0 authentication. The existing auth uses session cookies."
  3. What are your constraints? — "Do not modify the User model. Use the existing database connection in src/db/."
  4. What are your initial tasks? — "Start with the /auth/callback route, then implement token refresh."
Well-Structured Spawn Prompt 
# Example: spawning a backend teammate
"Create a backend teammate with these instructions:

 You are the backend engineer for this authentication migration.

 CONTEXT:
 - We are migrating from session-based auth to OAuth 2.0 + JWT
 - The existing User model (src/models/user.ts) must not change
 - Database is PostgreSQL via Prisma (schema at prisma/schema.prisma)
 - All new routes go in src/routes/auth/

 CONSTRAINTS:
 - Use the passport.js library already in package.json
 - JWT secret comes from process.env.JWT_SECRET (already configured)
 - All routes must return standard ApiResponse shape from src/types/
 - Do NOT touch src/middleware/session.ts — it stays for legacy support

 TASKS:
 1. Implement POST /auth/oauth/callback (Google provider)
 2. Implement POST /auth/token/refresh
 3. Implement DELETE /auth/token (logout / token revocation)
 4. Add rate limiting middleware for auth routes (10 req/min)
 5. Write JSDoc comments for all new route handlers"

Message Delivery Mechanics

Messages between team members are asynchronous. When you send a message, it is queued for the recipient and delivered when the recipient next checks for messages. This happens automatically during idle periods and between tool executions, but it is not instantaneous. A teammate in the middle of a long file write or test execution will not see the message until the current operation completes.

This asynchronous model means you cannot rely on messages for time-critical coordination. If you need a teammate to stop what it is doing immediately, the shutdown mechanism is more appropriate than a message. Messages are for coordination, context sharing, and course correction — not for real-time control.

12.7 Agent Team Patterns — Battle-Tested

The following patterns emerged from production use across dozens of team sessions. They are not theoretical — they are the configurations that have proven reliable, efficient, and genuinely valuable when applied to real projects. We present them in order of increasing complexity.

Pattern 1: Parallel Code Review (3–5 Reviewers)

The most immediately useful team pattern. Instead of running a single comprehensive code review, spawn multiple reviewers, each applying a different analytical lens to the same code. The lead synthesizes findings into a unified review.

Configuration

Team size: 3–5 reviewers + lead as synthesizer

Duration: 15–25 minutes

Token cost: 3–5x single review (but completed in 1/3 the wall-clock time)

Reviewer Focus Agent Definition
Security Vulnerabilities, auth issues, injection, secrets security-reviewer.md
Performance N+1 queries, memory leaks, algorithmic complexity perf-reviewer.md
Test Coverage Missing tests, edge cases, assertion quality test-reviewer.md
API Compatibility Breaking changes, versioning, contract violations Generic teammate
Accessibility ARIA, keyboard nav, contrast, screen reader support Generic teammate

Why it works: Each reviewer operates in read-only mode (no file conflicts possible), each applies a fundamentally different analysis (no duplicate work), and the lead gets five expert perspectives in the time it takes to get one. The synthesis step is where the lead adds value: resolving conflicts between reviewers (performance says "cache this," security says "don't cache credentials"), prioritizing findings, and producing a coherent review document.

Pattern 2: Competing Hypothesis Debugging

When a bug defies simple diagnosis, spawn multiple agents to test different root cause theories simultaneously. This is the adversarial debate structure applied to debugging.

Configuration

Team size: 3–5 investigators + lead as judge

Duration: 20–40 minutes

Structure: Each investigator is assigned a different hypothesis and must either prove or disprove it

Competing Hypothesis Setup 
# Bug: users intermittently get 500 errors on checkout

"Create a debugging team with 4 investigators:

 Investigator A: Test the hypothesis that this is a race condition
   in the payment processing. Look at concurrent request handling
   in src/services/payment.ts and check for shared mutable state.

 Investigator B: Test the hypothesis that this is a database
   connection pool exhaustion issue. Check connection pool config,
   look for connection leaks, analyze query duration patterns.

 Investigator C: Test the hypothesis that this is a third-party
   API timeout. Check the Stripe integration timeouts, retry logic,
   and error handling in src/integrations/stripe/.

 Investigator D: Test the hypothesis that this is a data validation
   issue with specific product configurations. Look for edge cases
   in cart calculation, especially with discounts and tax."

Why it works: Parallel hypothesis testing is inherently faster than serial debugging. Each investigator builds a case for or against their theory, gathering evidence from the codebase. The lead does not need domain expertise in every subsystem — the investigators bring that. When an investigator reports "I found shared mutable state in the payment service that would cause exactly this symptom under concurrent requests," the lead has a high-confidence diagnosis in 20 minutes instead of 2 hours.

The adversarial element: After initial investigation, the lead can share each investigator's findings with the others and ask them to challenge the conclusions. "Investigator B, Investigator A claims the race condition in payment.ts explains the intermittent failures. Given what you found about connection pools, could the pool exhaustion be the actual trigger?" This cross-examination surfaces the strongest theory.

Pattern 3: Multi-Layer Feature Implementation

For features that span multiple architectural layers (database, API, frontend, tests), assign each layer to a different teammate with explicit dependency chains.

Configuration

Team size: 3–4 implementers + lead as architect

Duration: 45–90 minutes

Critical success factor: Clear file ownership boundaries

Teammate Owns Depends On
Database prisma/, src/models/ Nothing (starts first)
Backend src/routes/, src/services/ Database (needs schema)
Frontend desktop/src/, desktop/components/ Backend (needs API contract)
Tests tests/, __tests__/ Backend + Frontend

Why it works: The dependency chain prevents integration failures. The database teammate defines the schema first. The backend teammate builds against that schema (dependency met). The frontend teammate can start on UI components immediately (no dependency on backend for visual layout) but waits for the API contract before wiring up data fetching. The test teammate writes test skeletons early and fills in assertions as the implementation solidifies.

Key insight: Give the database teammate explicit instructions to broadcast the schema to all teammates once the migration is written. Give the backend teammate instructions to broadcast the API response shapes once routes are defined. These broadcasts propagate the shared contracts that prevent integration surprises at the end.

Pattern 4: Research Fleet

Deploy multiple agents to research different aspects of a topic simultaneously, then aggregate findings into a comprehensive analysis. This pattern has been proven at scale in our production research pipeline, delivering 28-document research packages (91,000 words) and 12-document targeted investigations.

Configuration

Team size: 4–6 researchers + lead as editor-in-chief

Duration: 30–60 minutes

Output: Structured research documents per researcher, synthesized by lead

Research Fleet Deployment 
# Research topic: evaluating WebSocket alternatives for real-time features

"Create a research team:

 Researcher 1: WebSocket protocol deep-dive
   - Current spec status, browser support matrix
   - Connection lifecycle, reconnection strategies
   - Scalability characteristics at 10K, 100K, 1M connections

 Researcher 2: Server-Sent Events (SSE)
   - When SSE is sufficient vs when WebSocket is needed
   - SSE with HTTP/2 multiplexing
   - Polyfill landscape and browser support

 Researcher 3: WebTransport and QUIC
   - Current spec status, browser support
   - Advantages over WebSocket for unreliable delivery
   - When to bet on emerging standards vs proven ones

 Researcher 4: Real-time infrastructure
   - Managed services (Pusher, Ably, PubNub) vs self-hosted
   - Cost analysis at different scale points
   - Operational complexity comparison

 Each researcher: produce a structured document with
 Summary, Pros, Cons, When To Use, When To Avoid, and
 sources for every factual claim."

Why it works: Research is embarrassingly parallel — each researcher investigates an independent topic with no coordination needed during the research phase. The aggregation phase, where the lead synthesizes findings into a decision recommendation, is where the real value emerges. The lead can compare claims across researchers, identify contradictions (Researcher 1 says WebSocket scales to 1M connections, Researcher 4's infrastructure analysis suggests 100K is the practical limit without specialized infrastructure), and produce a decision document that accounts for multiple perspectives.

The fleet-commander variant: For larger research projects, designate one researcher as the "fleet commander" who does no primary research but instead reads all other researchers' output and produces the synthesis. This frees the lead to focus on steering and allows the synthesis to happen within the team rather than requiring the lead to do it manually.

Pattern 5: Adversarial Validation

After a feature is implemented (by a team or a single session), spawn a validation team that tries to break it. One teammate attempts exploits, another stress-tests edge cases, a third checks specification compliance.

Configuration

Team size: 3 validators + lead

Duration: 15–30 minutes

Mindset: Each validator is trying to find failures, not confirm success

Red Team (Attacker)

Attempts to exploit the implementation through malicious input, authentication bypass, privilege escalation, and resource exhaustion. Uses read-only access to analyze the code and documents potential attack vectors.

Edge Case Explorer

Tests boundary conditions: empty inputs, maximum-length strings, Unicode edge cases, concurrent operations, null/undefined propagation. Writes and runs tests targeting the boundaries that implementation tends to miss.

Spec Auditor

Compares the implementation against the original requirements or specification. Identifies features that were specified but not implemented, behaviors that deviate from spec, and assumptions that were made without explicit specification support.

Why it works: The implementation team has a bias toward confirming their work is correct. A separate validation team, given the explicit mandate to find problems, applies a fundamentally different cognitive lens. This is the same principle behind independent QA teams in traditional software engineering, but executed in minutes instead of days.

12.8 Best Practices

These recommendations distill operational experience from dozens of team sessions across projects ranging from small libraries to large multi-module applications. They are ordered from most impactful to least.

1. Start with 3–5 Teammates

The optimal team size is 3–5 teammates. Below 3, you get insufficient parallelism to justify the coordination overhead — just use subagents. Above 5, you hit diminishing returns as the lead spends more time managing teammates than doing productive work, file conflicts become more likely, and token costs scale linearly while output scales sublinearly.

1–2
Use subagents instead
3–5
Optimal team size
6+
Diminishing returns zone

2. Size Tasks Appropriately

Each task should take a teammate 10–20 minutes to complete. Tasks shorter than 5 minutes have too much coordination overhead relative to productive work. Tasks longer than 30 minutes risk the teammate going off-track without the lead noticing. The 5–6 tasks per teammate target gives a full session of work while keeping each unit small enough to verify.

Anti-pattern: Creating one enormous task per teammate ("implement the entire backend") defeats the purpose of coordination. The shared task list exists to make progress visible and manageable. Single-task teammates are just expensive subagents.

3. Design for File Ownership, Not File Locking

File locking is a safety net, not a coordination strategy. Design your task decomposition so that each teammate operates in a different set of files. This eliminates contention entirely rather than merely managing it. When two teammates need to modify the same file, consider whether the file should be split, or whether the work should be serialized through task dependencies.

File Ownership Design 
# GOOD: Clear file boundaries
Teammate A: owns src/auth/       (authentication logic)
Teammate B: owns src/api/        (API routes)
Teammate C: owns tests/          (test files)
Teammate D: owns docs/           (documentation)

# BAD: Overlapping ownership
Teammate A: "implement login feature" (touches auth + api + tests)
Teammate B: "implement signup feature" (touches auth + api + tests)
# Both teammates will fight over src/auth/middleware.ts

4. Monitor and Steer

Do not create a team and walk away. Agent teams require the same active management that human engineering teams need. Check teammate progress every 5–10 minutes. Read their messages. Watch for teammates that are stuck, confused, or heading in the wrong direction. A 2-minute course correction at minute 10 prevents a 20-minute rewrite at minute 30.

Specific signals to watch for:

  • A teammate that has been on the same task for more than 20 minutes without progress
  • A teammate that is modifying files outside its assigned ownership area
  • Multiple teammates sending messages about the same confusion
  • A teammate that has gone idle but has remaining tasks

5. Pre-Approve Common Operations

Nothing kills team productivity faster than permission prompts. Four teammates writing files simultaneously means four permission prompts hitting your terminal in rapid succession. Pre-approve all operations that teammates will need (read, write, edit, test execution, git status) and deny only the dangerous ones (git push, rm -rf, destructive operations).

6. Start with Review Before Implementation

For complex features, run a review/research team first to understand the codebase, then create an implementation team based on the findings. This two-phase approach prevents the most common team failure mode: teammates implementing changes based on incorrect assumptions about the existing code.

The Two-Phase Pattern

Phase 1 (15 min): 3 read-only researchers analyze the codebase area, document the existing architecture, identify integration points, and flag potential pitfalls.

Phase 2 (30–45 min): 3–4 implementers execute the feature with full context from Phase 1. Each implementer's spawn prompt includes the Phase 1 findings.

Total time: 45–60 minutes. Without Phase 1, the implementation team would spend their first 15 minutes rediscovering the codebase individually (3–4x the token cost of one shared analysis).

7. Use Explicit Task Dependencies

When tasks have natural ordering (schema before models, models before routes, routes before tests), express these as explicit dependencies in the task list. This prevents the "test teammate writes tests against an API that doesn't exist yet" failure mode and ensures work flows naturally through the dependency graph without manual sequencing by the lead.

8. Broadcast Shared Contracts

When one teammate produces output that other teammates depend on (API response shapes, database schema, interface definitions), instruct that teammate to broadcast the output to all team members. This is the team equivalent of updating a shared design document. Without broadcasts, each downstream teammate has to discover the contract by reading the upstream teammate's files, which introduces latency and potential for reading an intermediate state rather than the final output.

12.9 Limitations and Workarounds

Agent Teams are an experimental feature with real constraints. Understanding these limitations before committing to a team workflow prevents wasted time and tokens. Each limitation is paired with the practical workaround that mitigates it.

Limitation 1: No Session Resumption

When the lead session ends (whether intentionally, due to a crash, or due to context exhaustion), the team is gone. You cannot resume a team session. All teammates are terminated, and while the files they modified persist on disk, the team coordination state (task list, messages, in-progress work) is lost.

Workaround

Before ending a team session, ask the lead to produce a summary document: which tasks are complete, which are in-progress, what decisions were made, what files were modified. Save this to a file (e.g., .planning/team-handoff.md). When you start a new session, feed this document as context to bootstrap a new team from where the previous one left off. This is manual but reliable.

Additionally, design your team sessions to be completable within a single session. If the work is too large, break it into multiple team sessions with clear handoff points between them, rather than relying on a single long-running team.

Limitation 2: Task Status Lag

Task status updates are not real-time. When a teammate completes a task, the status update propagates to the shared task list asynchronously. The lead may see a task as "in_progress" for a short period after the teammate has actually completed it. This is usually only a few seconds, but under heavy load (many teammates, many tasks), the lag can be noticeable.

Workaround

Do not make time-critical decisions based on task status alone. If you need to know whether a teammate has truly finished, send them a direct message asking for confirmation. Use task status as a general progress indicator, not as a real-time signal.

Limitation 3: One Team Per Session

You can only run one team at a time in a Claude Code session. You cannot create Team A for the backend and Team B for the frontend simultaneously. If you need multiple independent teams, you need multiple Claude Code sessions (which defeats some of the coordination benefits).

Workaround

Design your team to encompass all the parallel work you need. Rather than separate backend and frontend teams, create a single team with backend and frontend teammates. If you genuinely need two independent teams (e.g., working on two different features simultaneously), run two separate Claude Code sessions, each with its own team, and coordinate between them manually through shared files.

Limitation 4: Fixed Lead, No Promotion

The lead is always the initial Claude Code session. A teammate cannot be promoted to lead, and the lead cannot be demoted to teammate. If the lead gets confused or goes off-track, you cannot replace it — you can only steer it back or restart the session.

Workaround

Keep the lead's role light. The lead should orchestrate and synthesize, not do heavy implementation work. If the lead's context window fills with implementation detail, it loses the ability to coordinate effectively. Delegate all heavy work to teammates and keep the lead focused on task management, message routing, and result aggregation.

Limitation 5: Split Panes Require External Tools

The split-pane display mode requires tmux (Linux/macOS) or iTerm2 (macOS). Standard terminal emulators without pane support fall back to in-process mode, where you can only see one teammate at a time. This limits situational awareness.

Workaround

Install tmux. It is available in every Linux distribution's package manager (apt install tmux, brew install tmux) and adds no meaningful overhead. For users who dislike tmux's key bindings, configure a minimal ~/.tmux.conf with intuitive navigation and ignore the rest of tmux's features. The split-pane visibility alone is worth the 2-minute setup.

Limitation 6: No Nested Teams

Teammates cannot create their own teams. The team model is flat: one lead, N teammates, no hierarchy. This means you cannot implement a "team of team leads" pattern where each lead manages a sub-team.

Workaround

If you need hierarchical delegation, combine teams with subagents. A teammate can still use the Task tool to spawn subagents for its own subtasks. This gives you a two-level hierarchy: lead → teammates → subagents. The teammate manages its subagents independently, and the lead coordinates teammates. This is sufficient for most complex workflows without needing full nested teams.

Limitation 7: No Cross-Team Communication

If you are running two separate team sessions (to work around the one-team-per-session limit), there is no built-in mechanism for teams to communicate with each other. Each team operates in complete isolation.

Workaround

Use the filesystem as a communication channel. Have each team write its output to a shared directory, and have the other team read from that directory. This is exactly the approach used by our Gastown convoy model (Section 12.10) and it works reliably at scale. A simple JSON file or markdown document serves as the shared state between teams.

12.10 Teams vs The Gastown Model

Claude Code's built-in Agent Teams and our Gastown convoy model represent two fundamentally different approaches to multi-agent coordination. Understanding when to use each — and how to combine them — is critical for practitioners operating at scale.

Agent Teams: Built-In, Structured, Session-Scoped

Agent Teams are a first-party feature integrated into Claude Code's runtime. They provide a shared task list, mailbox communication, file locking, and teammate lifecycle management. The coordination is structured and opinionated: tasks have defined states, communication flows through channels, and the lead maintains explicit control.

Strengths:

  • Zero setup cost — enable the feature flag and start prompting
  • Built-in coordination primitives (task list, mailbox, file locking)
  • Teammate visibility through in-process cycling or split panes
  • Permission inheritance simplifies security configuration
  • CLAUDE.md loading gives teammates project context automatically

Constraints:

  • Session-scoped — team dissolves when lead ends
  • One team per session, no nesting
  • Fixed lead, no recovery from lead failure
  • Higher token cost per unit of work due to full session per teammate
  • Limited to a single Claude Code installation's context

Gastown Convoy: Custom, Flexible, File-Coordinated

The Gastown convoy model is our custom orchestration pattern, developed through dozens of multi-agent production sessions running 50+ projects in parallel. It uses the filesystem as the coordination backbone: JSON state files for agent identity and work assignment, markdown documents for context handoff, and directory-based ownership boundaries for conflict prevention.

Strengths:

  • Survives session boundaries — state is on disk, not in memory
  • Scales beyond 5 agents (proven at 50+ concurrent projects)
  • Cross-process coordination — agents can be different runtimes entirely
  • Full hierarchical delegation (mayor → convoy leads → workers)
  • Observable — all state is inspectable as files on disk
  • Recoverable — crashed agents can resume from file state

Constraints:

  • Requires custom infrastructure (skills, state management, dispatch)
  • Higher setup cost per session
  • No built-in communication channel (uses file-based messaging)
  • Coordination overhead increases with custom protocol complexity
  • Requires operational experience to tune effectively

When to Use Each Approach

Use Agent Teams When

  • The work fits in a single session (under 90 minutes)
  • You need 3–5 parallel workers with built-in coordination
  • The team pattern is ad hoc (one-off review, specific investigation)
  • You want zero setup cost — just describe the team and go
  • All work happens in a single repository

Use Gastown Convoy When

  • The campaign spans multiple sessions or days
  • You need more than 5 concurrent agents
  • Session persistence and crash recovery are required
  • The work involves multiple repositories or runtimes
  • You need hierarchical delegation with sub-convoys
  • Observability and auditability are critical

Hybrid Patterns: The Best of Both

The most sophisticated orchestration combines both approaches. Use Gastown for the campaign-level coordination (tracking which missions are complete, managing handoffs between sessions, persisting state) and Agent Teams for the intra-session execution (parallelizing the actual work within each session).

Hybrid Example: Research Series Production

Consider a research series that needs to produce 12 documents across 3 sessions:

Gastown layer (cross-session):

  • Campaign state file tracks which documents are complete, in-progress, or pending
  • Each session reads the state file, picks up where the last session left off
  • Beads-state provides crash recovery within each session
  • Witness-observer monitors agent health and detects stalls

Agent Teams layer (intra-session):

  • Each session creates a team of 4 researchers
  • Researchers work in parallel on their assigned documents
  • Built-in task list tracks per-document progress
  • Mailbox enables researchers to share cross-references
  • Session ends with team output written to campaign state

This hybrid approach delivers the best of both worlds: the low-setup-cost, built-in coordination of Agent Teams for parallel work within each session, and the persistence, recoverability, and scalability of Gastown for managing the overall campaign.

The Future Convergence

As Agent Teams mature from experimental to stable, the gap between built-in and custom orchestration will narrow. Features like session persistence, cross-session teams, and hierarchical delegation are natural extensions of the current team model. When those arrive, the Gastown patterns that currently fill those gaps will either be absorbed into the built-in model or will evolve to address the next level of coordination complexity.

For practitioners building multi-agent systems today, the recommendation is clear: learn both approaches. Use Agent Teams for their simplicity and built-in coordination where they fit. Use custom orchestration patterns where they do not. The skill of knowing which tool to reach for, and how to combine them, is what separates effective multi-agent practitioners from those who struggle with coordination overhead.
