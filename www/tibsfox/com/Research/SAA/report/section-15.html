15.1 Scale and Context

Everything described in sections 1 through 14 of this report — the skill architectures, the orchestration patterns, the token management strategies, the security models — is drawn from a production system that has been running daily since late 2025. This section presents the evidence. These are not hypothetical architectures or conference-paper prototypes. They are case studies from a system called gsd-skill-creator that has generated over 190 research projects, passed more than 21,000 tests, and shipped 58 releases in a single milestone cycle.

The System Under Study

gsd-skill-creator — a self-modifying adaptive learning layer for Claude Code, open source at github.com/Tibsfox/gsd-skill-creator

The technology stack spans three languages and two runtimes. The core library and CLI are TypeScript. The desktop application backend is Rust, built on Tauri v2 for native performance and system-level access. Shader programs are written in GLSL for GPU-accelerated computation. The build system uses Vite v6, and the test suite runs on Vitest. The frontend is a Vite webview inside the Tauri shell. This is not a toy project or a demo — it is a production desktop application with a full build pipeline, a mature test suite, and months of daily use.

The hardware is deliberately modest. A single workstation with an RTX 4060 Ti (8 GB VRAM) and 60 GB of system RAM serves as the development and execution environment. There is no cloud compute, no cluster, no elastic scaling. Every result in this section was produced on that one machine, which is important because it demonstrates that agentic programming at scale does not require enterprise infrastructure. The constraint is architectural, not computational.

Branch Architecture

The system operates across six concurrent branches, each serving a distinct purpose in the development lifecycle:

Branch Purpose Activity Level

Each branch represents a different operational pattern: dev follows a conventional feature-branch workflow, nasa uses milestone-driven batch execution, artemis-ii runs hourly automated sweeps, and wasteland is a fully autonomous agent sandbox. The fact that all six coexist on the same codebase, with the same skill and agent infrastructure, is itself a demonstration of the architecture's flexibility.

The 13 Rosetta clusters organize the research output into coherent knowledge domains: Ecology, Electronics, Infrastructure, Energy, Creative, Business, Vision, Broadcasting, Science, Music, Space, AI & Computation, and Graphics. Music is the largest cluster at 26 projects. These clusters are not arbitrary categories — they emerged organically from the research content and now serve as the organizational backbone for cross-referencing and navigation.

15.2 Case Study: PNW Research Series — 190+ Projects

The Challenge

Scale: 190+ projects Status: Complete 13 knowledge clusters ~600K+ lines output

Generate more than 190 research projects across 13 knowledge clusters, each with multiple HTML modules, consistent styling, proper citations, and cross-references. Every project needed to meet the same quality bar — comprehensive content, correct formatting, working navigation, and factual accuracy. The content ranged from ecology field guides to electronics reference manuals to music theory analyses, requiring deep domain knowledge across vastly different fields.

No human could write 600,000+ lines of research content across 13 domains and maintain consistency. The question was whether an agentic system could.

The Approach: Mission-Pack Pipeline

The solution was a three-stage pipeline that transformed a topic description into a complete, published research project:

VISION WAVE PLAN PARALLEL EXECUTION +------------------+ +------------------+ +------------------+ | Topic + Scope | ---> | Wave 1: Research | | Agent A: Module 1| | Quality criteria | | Wave 2: Write | ---> | Agent B: Module 2| | Template refs | | Wave 3: Review | | Agent C: Module 3| | Cross-refs | | Wave 4: Publish | | Agent D: Module 4| +------------------+ +------------------+ +------------------+ | ATOMIC COMMITS | +------------------+ | FTP Sync | | Index Update | | Cross-ref Link | +------------------+

The research-engine skill handled the vision-to-document pipeline. Given a topic, it produced structured HTML documents with proper citations, consistent formatting, and navigation elements. The vision-to-mission skill converted high-level creative descriptions into executable GSD mission packages. Together, these two skills formed the backbone of the content generation system.

At the execution layer, the Gastown convoy model managed batch dispatch at the 50+ project scale. This model treats a group of related projects as a convoy that moves through the pipeline together, with a coordinator (the "mayor") tracking progress, dispatching work via the sling-dispatch skill, and aggregating results. The convoy metaphor proved accurate: projects move at the speed of the slowest member, but the coordinator can reroute and rebalance work as faster agents complete their tasks.

Agent Configuration

Each wave ran with a fixed team size and clear task boundaries:

Task Type Agents Tasks Per Agent Rationale
Synthesis 4 Opus agents 5-6 Cross-referencing requires full reasoning capability
Document generation 3 agents 5-6 Template-driven work needs fewer agents with more focus
Index & navigation 1 agent All Global consistency requires single-threaded execution
Quality review 1 agent Spot-check Adversarial review catches what generators miss

The 5-6 tasks per agent number was not arbitrary. It emerged from empirical observation: agents given fewer than 4 tasks underutilize their context window; agents given more than 7 begin to lose track of earlier work and make more errors. The sweet spot sits around 5-6, where the agent has enough work to build momentum but not so much that quality degrades.

What We Learned

Agent supervision is critical

Every agent's reads and writes must be checked against its instructions. An unsupervised agent will drift from its task within 3-4 operations. The witness-observer skill was built specifically to patrol agent activity and send corrective nudges when an agent strays from its assigned work. This is not a theoretical concern — it happened repeatedly during the first dozen projects until the monitoring infrastructure was built.

Simple tasks first, complex later

Sorting tasks by complexity and assigning simple ones first lets agents warm up with straightforward work before tackling harder problems. An agent that starts with a complex cross-referencing task is more likely to produce errors than one that has already completed two or three simpler modules and has built up context about the project's structure.

Agent failures happen — always have a fallback

At the 190-project scale, agent failures are not edge cases; they are statistical certainties. An agent might produce malformed HTML, lose track of its instructions, or simply run out of context. The system was designed with a fallback path: if an agent's output fails validation, the coordinator marks the task for manual writing. This pragmatic approach prevented cascading failures and kept the pipeline moving.

Context is RAM, state is on disk

This principle shaped the entire architecture. An agent's context window is volatile — it empties when the session ends. But the work must persist. Every meaningful decision, every intermediate result, and every coordination signal is written to disk immediately. Context handoffs serve as "save games" that allow new agents to pick up where previous ones left off.

Results

2,439+ Files Generated
~600K+ Lines of Content

The consistency of the output is what makes this case study significant. All 190+ projects use the same CSS framework, the same navigation patterns, the same citation format, and the same heading hierarchy. A reader moving from an ecology project to an electronics project encounters the same visual language, the same interaction patterns, and the same level of detail. This kind of consistency across hundreds of documents is practically impossible with manual authoring. It is a direct consequence of template-driven generation with strong validation gates.

15.3 Case Study: NASA Mission Catalog — 720 Missions

The Challenge

Scale: 720 missions 6 releases complete 6 parallel tracks Branch-isolated

Build a comprehensive catalog of 720 NASA missions, from Mercury through Artemis, with structured metadata, mission timelines, technical specifications, crew information, and cross-references between related missions. The catalog needed to be more than a database — each mission entry required narrative context, historical significance, and technical depth that a simple data dump could not provide.

Architecture: 6-Track Parallel Execution

The NASA catalog introduced a parallel track architecture that ran five content tracks simultaneously, gated by a sixth quality assurance track:

TRACK 1: Early Space Program TRACK 2: Shuttle Era Mercury, Gemini, Apollo STS missions, ISS assembly | | TRACK 3: Robotic Missions TRACK 4: Deep Space Mars rovers, Voyager, probes Outer planets, heliosphere | | TRACK 5: Modern Era TRACK 6: QA Gate Commercial crew, Artemis Cross-track validation | | +-----------------------------------+ | RELEASE PIPELINE One at a time, sequential Carry-forward retrospectives

Each content track operated on its own slice of the mission catalog, allowing four Opus agents to work in parallel without stepping on each other's data. Track 6 — the QA gate — ran after each track completed a batch, cross-checking for consistency in date formats, mission numbering, cross-references, and factual accuracy.

The release pipeline enforced a strict discipline: one release at a time, sequential, with carry-forward retrospectives. This meant that v1.1 could not begin until v1.0 was fully validated, published, and its retrospective documented. Each retrospective identified specific improvements for the next release, creating a feedback loop that measurably improved quality over time.

Patterns Used

Fleet-Mission Dispatch

The fleet-mission skill launched parallel agents for each track, monitored completion status via beads-state, and merged results when all tracks in a wave completed. Progress tracking used a simple file-based state machine — each agent wrote its status to a known path, and the coordinator polled for completion.

Done-Retirement Pipeline

Completed releases went through a 7-stage retirement protocol: validate, commit, push, submit (PR), notify, cleanup, and archive. This ensured that finished work was properly packaged and that working directories were cleaned up for the next release.

Context Handoffs

Each session ended with a structured handoff document containing: what was done, what decisions were made, what gotchas were discovered, and what the next session should start with. These handoffs eliminated the "cold start" problem that plagues multi-session workflows.

CSV-Driven Research

Mission data was maintained in CSV files that served as the source of truth. Agents read from the CSV to generate content, and any corrections flowed back to the CSV first, then propagated to the generated documents. This single-source-of-truth pattern prevented the data drift that occurs when content is edited in place.

Quality Through Retrospectives

The carry-forward retrospective pattern produced measurable improvement across releases. Each retrospective was structured:

v1.0 Retrospective: Date format inconsistencies found across 12% of entries. Fix: Added date validation to QA gate. Result: v1.1 had zero date format errors.
v1.1 Retrospective: Cross-references between related missions were missing for 23 entries. Fix: Added a cross-reference validation pass. Result: v1.2 achieved 100% cross-reference coverage.
v1.2 Retrospective: Mission timelines had inconsistent granularity (some day-level, some hour-level). Fix: Standardized timeline granularity rules per mission type. Result: v1.3 had uniform timeline detail.
v1.3-v1.5: Progressively refined crew information, technical specifications, and historical context based on gap analysis from previous releases.

Results

720 Missions Cataloged
6 Releases Complete
5+1 Parallel Tracks
2 TB Data Budget

The NASA catalog demonstrated that the carry-forward retrospective pattern can substitute for automated testing when the domain does not lend itself to unit tests. Research content quality is inherently subjective, but structured retrospectives with specific metrics (date format errors, cross-reference coverage, timeline granularity) transform subjective quality into measurable criteria. Each release was objectively better than the last, and the improvement trajectory was documented.

Branch Isolation as Architecture

The NASA catalog operates on its own dedicated branch (nasa), completely isolated from the main development branch and all other active branches. This isolation is deliberate and structural, not merely organizational. The NASA branch carries its own set of data files (CSV catalogs, image assets, mission metadata) that would pollute the main branch if merged. More importantly, the branch-based isolation allows the NASA catalog to run its own release pipeline — with its own version numbering (v1.0 through v1.6), its own retrospective cadence, and its own quality gates — without interfering with the v1.49.x release series running on the dev branch.

This pattern of branch-per-mission proved so effective that it was subsequently adopted for the Artemis II coverage (its own branch) and the wasteland muse sandbox (its own branch). Each branch is a self-contained project with its own orchestration, its own handoff documents, and its own session history. The shared infrastructure — skills, agents, commands, hooks — lives on the main/dev branches and is consumed by all mission branches. This creates a clean separation between the platform (shared) and the missions (isolated).

The local-first data strategy (2 TB budget, no cloud dependencies) meant that the entire catalog could be built and served from the workstation. Network failures, API rate limits, and third-party service outages could not disrupt the research pipeline. Every data source was downloaded, cached locally, and referenced from the local filesystem. This decision added upfront work (downloading and organizing 2 TB of data) but eliminated an entire category of runtime failures.

15.4 Case Study: Artemis II Live Mission Coverage

The Challenge

Real-time coverage 35+ live data sources Hourly automated updates 340 KB new content / session

Provide comprehensive, continuously updated coverage of NASA's Artemis II crewed lunar mission. This was not a retrospective analysis — it was live coverage during an active spaceflight, requiring real-time data ingestion, automated updates, and rapid content generation as events unfolded. The coverage needed to be accurate, timely, and comprehensive, drawing from weather stations, satellite imagery, telemetry data, and research databases.

What We Built

The Artemis II coverage system comprised four integrated components:

sweep.py

The hourly update script that monitored 35+ live data sources, including 8 METAR weather stations, 7 NWS/CWOP/NDBC sources, and 4 satellite imagery feeds. Each sweep cycle pulled fresh data, compared it against the previous state, and generated update diffs for any changed sources.

sweep_daemon.py

A continuous daemon that wrapped sweep.py with FTP sync capabilities. The daemon maintained a persistent connection to the publication server, detected new sweep results, and pushed updates automatically. It handled connection failures, retry logic, and deduplication.

weather-map.html

A live weather visualization page integrating 19 data sources: 8 METAR stations for surface observations, 7 NWS/CWOP/NDBC feeds for marine and atmospheric data, and 4 satellite imagery sources for cloud cover and storm tracking. All rendered client-side with JavaScript.

Discovery Engine

A 6-hour cadence arXiv scanner that identified new papers relevant to the mission's science objectives. Papers were filtered by relevance, summarized, and queued for incorporation into the research pages.

Orchestration in Action: A Single Session

The most intensive demonstration of the orchestration architecture occurred during a single research update session that produced 340 KB of new content across 6 waves using 13 parallel agents. This is the anatomy of that session:

Wave Execution Log

Wave 1 — Data Corrections (4 agents): Audited 9 existing research pages and applied 33 edits to correct data that had been superseded by newer sources. Corrections ranged from updated launch window calculations to revised radiation dose estimates. Each agent owned 2-3 pages and committed atomically.
Wave 2 — New Pages Batch A (3 agents): Generated 3 new research pages in parallel: CubeSats (small satellite experiments on the mission), Laser Communications (optical comm technology being tested), and Radiation Environment (Van Allen belt transit analysis). Each page was 8,000-12,000 words with citations.
Wave 3 — New Pages Batch B (3 agents): Generated 3 more research pages: Study Guide (educational resource for students following the mission), DIY Projects (hands-on activities inspired by mission science), and AI Guide (how AI systems support mission operations).
Wave 4 — New Pages Batch C (3 agents): Generated the final 3 research pages: Apollo Comparison (detailed comparison with Apollo 8/10/13), Earth Ecology (environmental monitoring from lunar orbit), and Canada CSA (Canadian Space Agency contributions including Canadarm3).
Wave 5 — Synthesis (1 agent): Updated the master index page to include all 9 new pages, generated knowledge node cross-references linking related topics across the new and existing pages, and validated all internal links.
Wave 6 — Publication (1 agent): FTP synced 38 files to the live site: 9 new HTML pages, 9 updated existing pages, 1 updated index, and 19 supporting assets. The sync was atomic — a staging directory was built, validated, and then swapped into place.

The total elapsed time for all 6 waves was a single session. The key architectural insight was the wave dependency structure: Waves 2-4 were independent and could run in parallel, but Wave 5 (synthesis) required all content to be complete, and Wave 6 (publication) required synthesis to be complete. This dependency graph is the wave-based parallel execution pattern in its purest form.

The Sweep Architecture

EVERY HOUR: +----------------+ +------------------+ +----------------+ | METAR Stations | | NWS/CWOP/NDBC | | Satellite Feeds| | 8 sources | | 7 sources | | 4 sources | +-------+--------+ +--------+---------+ +-------+--------+ | | | +----------+------------+------------------------+ | +--------v--------+ | sweep.py | | Compare vs last | | Generate diffs | +--------+--------+ | +--------v--------+ | sweep_daemon.py | | FTP sync | | Retry logic | | Deduplication | +--------+--------+ | +--------v--------+ | tibsfox.com | | Live pages | +--------+--------+

The sweep architecture proved that agentic systems can handle continuous, time-sensitive operations. The daemon ran 24 hourly sweeps on its first day, producing 24 versioned releases (v1.0.0 through v1.0.23) of weather data. Each release was a complete snapshot that could be independently verified. The system has continued operating daily since its deployment, accumulating thousands of sweep cycles.

Lessons from Real-Time Coverage

Real-time coverage of an active space mission introduced constraints that do not exist in retrospective research. Data sources update on their own schedule, not the system's schedule. Weather observations arrive every hour. Satellite imagery updates every 15 minutes during passes. Telemetry data streams continuously during active mission phases. The sweep architecture had to accommodate all of these cadences without missing data and without overwhelming the publication pipeline with redundant updates.

The deduplication logic in sweep_daemon.py solved the redundancy problem. Each sweep result was hashed, and the daemon only triggered an FTP sync when the hash changed. During quiet periods (no weather changes, no new satellite passes), the daemon detected zero-diff sweeps and skipped the sync entirely. During active periods (storm systems moving through the launch area, mission phase transitions), the daemon detected changes on nearly every sweep and synced accordingly. This adaptive behavior emerged naturally from the hash-comparison design — it was not explicitly programmed as a "quiet mode" or "active mode."

The discovery engine's 6-hour arXiv scan cadence was deliberately slower than the hourly weather sweeps. Research papers do not arrive in real time; the arXiv posts new papers in daily batches with updates throughout the day. A 6-hour scan captured all new papers with minimal redundant queries. The engine filtered papers by keyword relevance (Artemis, lunar, Orion, SLS, radiation, cislunar) and scored them against the existing research page topics to identify gaps. Papers that scored above the relevance threshold were queued for human review before incorporation, maintaining the accuracy standard while automating the discovery process.

15.5 Case Study: GPE Global Power Efficiency — Deep Research Update

The Challenge

Deep research ~86K words total 10 gap modules Source verification

The Global Power Efficiency project required a comprehensive energy research update with rigorous source verification, systematic gap analysis, and actionable policy recommendations. Unlike the batch-generation pattern of the PNW series, this case study demanded deep, sustained research into a single domain, with each module building on the findings of previous modules and all modules cross-referencing a common evidence base.

Execution Architecture

The GPE update proceeded through three distinct phases, each requiring a different orchestration pattern:

Phase 1: Source Verification

Six existing sources were audited for accuracy, currency, and completeness. The data-fidelity skill launched parallel fact-checkers that cross-referenced claims against primary sources, identified outdated statistics, and flagged unverifiable assertions. This phase produced a verification report that served as the foundation for all subsequent writing.

Phase 2: Gap Module Generation

Ten research gap modules were identified and assigned to parallel agents, each specializing in a distinct energy domain:

Industrial Efficiency

Manufacturing process optimization, waste heat recovery, industrial symbiosis networks

Transport Electrification

EV adoption curves, charging infrastructure, grid impact modeling, heavy transport alternatives

Geothermal, Nuclear & Storage

Enhanced geothermal systems, SMR deployment, grid-scale battery economics

Behavioral Economics

Energy consumption behavior, nudge theory applications, demand response psychology

Just Transition

Workforce retraining, community impact, economic diversification for fossil fuel regions

Thermal Networks

District heating, thermal storage, waste heat distribution, seasonal thermal management

Building Equity

Energy poverty, retrofit access, affordable efficiency, housing stock analysis

Climate Cooling

Passive cooling strategies, urban heat island mitigation, natural ventilation design

Data Literacy

Energy data interpretation, smart meter comprehension, consumption visualization

Verification Methods

Measurement and verification protocols, energy savings validation, M&V best practices

Each agent owned one domain exclusively. This specialization pattern produced higher-quality output than the alternative of having each agent write across multiple domains, because the agent could build deep context about its assigned topic and maintain consistent terminology and framing throughout its module.

Phase 3: Synthesis and Publication

The synthesis phase was deliberately single-threaded. One agent restructured the chart visualizations with geo-grouped color coding, updated the master index, and generated the lawmakers action page — an 11,233-word document covering 15 policy strategies with model legislation templates. This phase could not be parallelized because it required holistic awareness of all 10 gap modules and their relationships.

Key Pattern: Parallel Research, Sequential Synthesis

The GPE case study proved that research modules can be generated in parallel when they address distinct domains, but synthesis — the act of weaving those modules into a coherent whole — must be sequential. An agent performing synthesis needs access to all the parts simultaneously, and it needs to make global decisions about structure, emphasis, and cross-referencing that cannot be decomposed. This parallel-research, sequential-synthesis pattern appears in every subsequent case study.

15.6 Case Study: Math Co-Processor — GPU-Accelerated MCP

The Challenge

GPU compute 18 MCP tools 5 computation chips CUDA acceleration

Build a GPU-accelerated mathematical computation engine and expose it to any Claude session via the Model Context Protocol. The system needed to handle linear algebra, spectral analysis, statistics, symbolic computation, and calculus — all with the performance characteristics of native code, not the latency of API calls to external services.

Chip Architecture

The co-processor was designed around a chip-based modular architecture, where each chip encapsulates one mathematical domain and exposes a set of MCP tools:

MATH CO-PROCESSOR +------------------------------------------------------------------+ | MCP Interface Layer | | 18 tools exposed via Model Context Protocol | +------------------------------------------------------------------+ | | | +------------+ +----------+ +----------+ +--------+ +------+ | | | ALGEBRUS | | FOURIER | | STATOS | | SYMBEX | |VECTRA| | | | Linear | | Spectral | | Stats | | Symbol | |Calc | | | | Algebra | | Analysis | | | | Exec | |ulus | | | +------------+ +----------+ +----------+ +--------+ +------+ | | | gemm | | fft | | describe | | eval | |grad | | | | solve | | ifft | | regress | | verify | |batch | | | | eigen | | spectrum | | monte | | | |xform | | | | svd | | | | carlo | | | | | | | | det | | | | | | | | | | | +------------+ +----------+ +----------+ +--------+ +------+ | | | +------------------------------------------------------------------+ | Rust Backend | CUDA (RTX 4060 Ti) | +------------------------------------------------------------------+

The five chips — Algebrus (linear algebra), Fourier (spectral analysis), Statos (statistics), Symbex (symbolic execution), and Vectora (calculus) — each own a bounded domain of mathematical operations. This boundary enforcement prevents chips from duplicating functionality and makes the system easier to test, extend, and reason about.

Development Methodology: TDD with Autonomous Execution

The math co-processor was built using strict test-driven development with autonomous execution. The pattern was consistent across all five chips:

  1. RED: Write failing tests that define the expected behavior of each MCP tool. For example, algebrus_gemm needed to multiply two 3x3 matrices and return the correct result to 6 decimal places.
  2. GREEN: Implement the minimal code to make the tests pass. The Rust implementation called CUDA kernels for GPU-accelerated computation.
  3. COMMIT: Each RED-GREEN cycle produced 2-3 atomic commits: one for the test, one for the implementation, and optionally one for refactoring.

All plans were autonomous — no human checkpoints. The agent received a plan document specifying the tests and implementations for one chip, and it executed the entire plan without stopping for input. This autonomous execution pattern worked because the plans were specific enough (exact test cases, exact function signatures) that the agent did not need to make judgment calls during execution.

MCP as Interface Layer

The decision to use MCP as the interface layer proved pivotal. Any Claude session — whether a Claude Code CLI, a desktop application, or another agent — can call the math co-processor's tools by connecting to the MCP server. The tools appear as native capabilities in the agent's tool list, with typed parameters and structured responses.

This means an agent writing a research paper can, mid-generation, call statos_regression to run a statistical regression on a dataset, embed the results in its document, and continue writing — all without leaving the generation context. The math becomes a tool call, not a context switch.

Rust Backend for Performance-Critical Operations

The decision to implement the computation backend in Rust rather than Python or JavaScript was driven by two requirements: deterministic memory management for large matrix operations, and direct CUDA interop without the overhead of a Python runtime. Rust's ownership model guarantees that GPU memory buffers are allocated and freed predictably, which matters when the RTX 4060 Ti's 8 GB of VRAM is shared between the co-processor and whatever else the workstation is running.

The Rust-CUDA integration uses raw bindings to the CUDA driver API. Each chip manages its own CUDA context and stream, so Algebrus can run a matrix multiplication concurrently with Fourier running an FFT. The streams are synchronized only when a higher-level operation requires results from multiple chips — for example, a spectral decomposition that uses Fourier for the transform and Algebrus for the eigenvalue computation.

Performance benchmarks on the RTX 4060 Ti showed that GPU-accelerated operations outperformed CPU equivalents by 15-40x for matrices larger than 512x512, and by 3-8x for smaller matrices. Below 64x64, the overhead of GPU memory transfer made CPU execution faster, so the co-processor automatically routes small operations to CPU and large operations to GPU. This routing logic is transparent to the calling agent — it simply calls algebrus_gemm and gets the fastest result.

15.7 Case Study: Self-Modifying System Safety

The Challenge

Self-modification Runtime skill updates Security-critical

gsd-skill-creator modifies its own skills, agents, and hooks at runtime. The system that generates skills is itself governed by skills. The system that validates agent behavior is itself an agent. This recursive self-reference creates a unique safety challenge: how do you ensure that a self-modifying system doesn't modify itself into an unsafe state?

The Security Architecture

The answer, developed over months of iteration, is a layered defense system where each layer constrains the layer below it:

LAYER 4: IMMUTABLE SECURITY SKILL security-hygiene: cannot be modified by the system it protects +------------------------------------------------------------------+ | Monitors all writes to .claude/skills/ and .claude/agents/ | | Validates skill structure before activation | | Enforces file path restrictions | | Audit trail via git history | +------------------------------------------------------------------+ LAYER 3: BOUNDED LEARNING GUARDS skill-integration: constrains what the adaptive layer can learn +------------------------------------------------------------------+ | Token budget per skill observation | | Maximum skill size limits | | Pattern validation against known-good templates | | Reject skills that modify security-hygiene | +------------------------------------------------------------------+ LAYER 2: SKILL VALIDATION PIPELINE Pre-activation checks before any skill becomes active +------------------------------------------------------------------+ | Syntax validation (valid markdown/YAML) | | Path validation (must be in .claude/skills/) | | Content validation (no shell exec, no network access) | | Dependency validation (all referenced tools exist) | +------------------------------------------------------------------+ LAYER 1: FILE SYSTEM BOUNDARIES Hard limits enforced by the operating system +------------------------------------------------------------------+ | Skills only in .claude/skills/ | | Agents only in .claude/agents/ | | Hooks only in .claude/hooks/ | | .planning/ is gitignored (never published) | +------------------------------------------------------------------+

The Trust Principle

The system operates on a principle called "trust no one" — trust is earned, not given. This manifests in several concrete ways:

  • New skills start disabled. A skill generated by the adaptive learning layer is not activated until it passes validation and is explicitly approved. The system never assumes that generated output is safe.
  • Modifications are logged. Every change to a skill file is tracked in git history, creating an immutable audit trail. If a skill is modified and the system subsequently misbehaves, the exact change can be identified and reverted.
  • The security skill is immutable. The security-hygiene skill that monitors all other skills is itself protected from modification. It cannot be updated by the system it protects. This breaks the recursive loop: the watchdog is not itself watched, but it is also not modifiable.
  • Memory system provides institutional knowledge. The MEMORY.md file and its linked detail documents provide cross-session awareness that prevents an agent in a new session from unknowingly undoing a safety decision made in a previous session.

Lessons from Production

Self-modification requires explicit safety boundaries

The system cannot be trusted to set its own boundaries. Safety boundaries must be defined externally (by the human operator) and enforced by components that the system cannot modify. This is the same principle that operating systems use for kernel vs. userspace: the kernel enforces rules that userspace code cannot change.

Automated validation catches what humans miss

During the first month of operation, the validation pipeline caught 14 skill modifications that would have introduced subtle behavioral changes. A human reviewing the same changes in a code review would likely have approved most of them — they looked reasonable in isolation. But the automated validator caught that they violated structural constraints (like exceeding the 250-character description cap) that a human would not think to check.

Audit trails are essential

Three times during development, a skill modification caused unexpected behavior in an unrelated part of the system. In each case, the git history provided the forensic evidence needed to identify the root cause. Without the audit trail, these regressions would have been extremely difficult to diagnose.

The security skill must itself be immutable

Early in development, the security-hygiene skill was itself subject to the adaptive learning layer. This created a vulnerability: the system could learn to relax its own security constraints. The fix was simple and absolute — mark the security skill as immutable and reject any attempt to modify it, regardless of the source.

15.8 Case Study: Unit Circle Chain — Mathematical Rigor

The Challenge

50 versions Chain scoring Proof validation Temporal analysis

Track and validate mathematical quality across 50 versions of the system, using chain scoring to measure improvement trajectories and proof validation to verify mathematical claims. The v1.50 milestone — the fiftieth major version — required a retrospective analysis that could demonstrate sustained quality across the entire version history.

Chain Scoring Methodology

Each version received a chain score between 1.0 and 5.0, assessed across four dimensions: correctness of mathematical claims, completeness of proofs, consistency of notation, and rigor of reasoning. The scoring was performed by the uc-proof-engineer agent, which applied formal verification techniques to mathematical content.

Temporal Analysis with PyDMD

The uc-observatory system used PyDMD (Dynamic Mode Decomposition) to perform temporal analysis of the chain scores, identifying periodic patterns in quality variation. This analysis revealed:

  • Quality oscillation: Scores follow a 3-4 version oscillation pattern. After a high-scoring release, the next 1-2 releases typically score lower as the system absorbs new capabilities, followed by a recovery as those capabilities mature.
  • Convergence trend: The oscillation amplitude decreased over time. Early versions showed swings of 0.8-1.0 points; later versions oscillated within a 0.31-point range (4.44-4.75). The system was converging toward stability.
  • Feature complexity correlation: Releases with higher architectural complexity (holomorphic dynamics, upstream intelligence) consistently scored lower than releases focused on tool-building (site generator, DACP capstone). This suggests that reasoning-heavy work is harder for agents to validate than construction-heavy work.

Proof Validation Agent

The uc-proof-engineer agent applied three levels of verification to mathematical claims:

Level 1: Structural Verification

Check that proofs follow valid logical structure: premises are stated, inference rules are named, and conclusions follow from premises. This catches the most common error — proofs that skip steps or assert conclusions without justification.

Level 2: Computational Verification

Where proofs involve computation (eigenvalue calculations, matrix operations, statistical tests), verify the computations using the math co-processor. This catches numerical errors and precision issues.

Level 3: Cross-Reference Verification

Check that mathematical claims are consistent across documents. If a theorem is stated in one section and applied in another, verify that the statement and application match. This catches the subtle inconsistencies that arise when different agents write different sections.

The v1.50 Milestone Significance

The v1.50 milestone carries special significance: it represents the fiftieth major version of a system that tracks its own quality mathematically. The Unit Circle metaphor is literal — each version maps to a position on the unit circle (2π/50 radians per version), and the chain score at each position determines the radius at that angle. A perfect score of 5.0 at every version would trace a perfect circle. The actual trace, with its oscillations between 4.44 and 4.75, produces a slightly irregular circle that visually represents the quality variation across the version history.

The v1.50 review branch exists specifically for this retrospective analysis. It is a long-running branch that accumulates the chain scoring data, the temporal decomposition results, and the proof validation reports. Unlike the dev or nasa branches, which are active workspaces, v1.50 is an analytical branch — it observes the work done on other branches and produces quality assessments. This separation between the work and the assessment of the work prevents the self-evaluation bias that would occur if agents scored their own output.

The PyDMD temporal decomposition was applied to the full 50-version score history, not just the final 8. The decomposition revealed three dominant dynamic modes: a slow convergence mode (the overall upward trend), a 3-4 version oscillation mode (the quality recovery cycle described above), and a high-frequency mode correlated with major architectural changes. These modes explain 94% of the variance in chain scores, suggesting that quality in a self-modifying system follows predictable dynamics rather than random variation.

15.9 Cross-Cutting Patterns That Emerged

Across all seven case studies, certain patterns appeared repeatedly. These are not patterns that were designed in advance and then applied — they are patterns that emerged from the work and were subsequently codified. Their reliability is empirical: they have been tested across 190+ projects, 720 missions, real-time data pipelines, deep research, GPU computation, self-modifying safety systems, and mathematical proof validation. The fact that these patterns emerged independently in different domains — and that they kept working as the system scaled — is stronger evidence of their validity than any single controlled experiment could provide.

Planning is the hard part, code is easy

This was the single most consequential insight from the entire production history. A well-written plan — one that specifies exactly what to build, exactly what tests to write, and exactly what the output should look like — transforms execution from a creative act into a mechanical one. Agents excel at mechanical execution. They struggle with ambiguity. The 3-level NASA SE planning methodology (discuss, plan, execute) works reliably because it front-loads all the ambiguity into the discussion phase, where a human can participate, and leaves only unambiguous work for autonomous execution.

Wave-based parallel execution

Group independent tasks into waves. Execute each wave with parallel agents. Synthesize between waves. The optimal configuration, established empirically: 3-6 waves per pipeline, 3-5 agents per wave, 5-6 tasks per agent. Smaller waves (2 agents) underutilize parallelism; larger waves (8+ agents) create coordination overhead that negates the throughput gain. The synthesis steps between waves are critical — they ensure that the outputs of one wave are consistent before the next wave builds on them.

Atomic commits preserve bisectability

Each agent commits its own work as an atomic unit. When wave boundaries do not align with commit boundaries (because a session was interrupted), wave markers in the commit message preserve the intent: "Wave 2: cubesats, laser-comms, radiation pages" tells a future reader which logical wave produced this commit, even if the commit contains work from a single agent's portion of that wave. Conventional Commits format (feat(scope): subject) is enforced by a PreToolUse hook, ensuring uniform commit messages across all agents.

Context handoffs prevent information loss

Write handoffs before context exhaustion. Include decisions (what was decided and why), gotchas (what will trip up the next agent), and next steps (what to do first when resuming). The handoff is a "save game" — it captures enough state that a completely new agent, with no memory of the previous session, can pick up work at full speed. This pattern has been used hundreds of times across all six branches and has proven robust even when the resuming agent is a different model version than the one that wrote the handoff.

Parallel research, sequential synthesis

Research modules that address distinct domains can be generated in parallel. Synthesis — the act of weaving modules into a coherent whole — must be sequential. This pattern appeared in every case study: PNW Research (parallel modules, sequential index), NASA (parallel tracks, sequential QA gate), GPE (parallel gap modules, sequential lawmakers page), Artemis II (parallel pages, sequential knowledge nodes). The reason is fundamental: synthesis requires global awareness that cannot be partitioned.

State on disk, context in RAM

An agent's context window is volatile. When the session ends, everything in context is lost. But work must survive session boundaries. The solution is aggressive state externalization: every decision is written to a file, every coordination signal is persisted, every intermediate result is committed. The memory system (MEMORY.md and linked detail documents) provides cross-session continuity that spans weeks and months. The beads-state skill handles crash-recoverable state for in-flight orchestration.

Scale Progression

The system did not arrive at this scale overnight. The growth followed a predictable progression:

Month 1-2: Single-agent workflows. One skill, one command, manual everything. The focus was on getting the basic build pipeline working: TypeScript compilation, Vitest test runner, and a single "hello world" skill.
Month 3-4: Multi-agent dispatch. Fleet-mission skill introduced for parallel work. The first convoy (5 projects) proved the model worked but exposed coordination gaps that led to the sling-dispatch and nudge-sync skills.
Month 5-6: Full orchestration. Mayor-coordinator, beads-state, and the complete Gastown chipset enabled convoys of 50+ projects. The PNW Research Series scaled from 20 to 100+ projects during this period.
Month 7+: Specialized systems. The math co-processor, sweep daemon, proof validation, and self-modifying safety architecture were all built on the foundation established in months 1-6. Each new system reused the core orchestration patterns.

The Compounding Effect

Every skill built makes the next skill easier to build. Every pattern proven makes the next project faster to execute. The 190th project was dramatically faster to produce than the 10th, not because the system got faster, but because the orchestration infrastructure had matured to the point where execution was mechanical.

15.11 What Worked, What Didn't

Honest assessment is more valuable than success theater. The system that produced the results described in this section also produced failures, dead ends, and forced adaptations. Both sides are documented here.

What Worked Well

  • Parallel agent dispatch for independent tasks. When tasks genuinely do not depend on each other, parallel execution delivers near-linear speedup. Four agents processing four independent research domains complete roughly 4x faster than a single agent processing them sequentially.
  • Template-driven content generation. Providing agents with a concrete HTML template, complete with CSS classes, navigation patterns, and example content, eliminated the most common source of inconsistency: each agent inventing its own formatting.
  • Carry-forward retrospectives. The practice of writing a structured retrospective after each release and carrying its findings into the next release's plan produced measurable quality improvement over time. It is the simplest form of organizational learning.
  • Context handoffs between sessions. The handoff document format (decisions, gotchas, next steps) has been used hundreds of times and reliably enables session continuity. Even handoffs written months ago can be consumed by a new agent.
  • Memory system for cross-session continuity. MEMORY.md and its linked detail documents provide a form of institutional memory that survives context window exhaustion, session boundaries, and even model version changes.
  • TDD with autonomous execution. When plans specify exact test cases, agents execute them flawlessly. The RED-GREEN cycle is ideally suited to autonomous execution because each step has unambiguous success criteria.

What Didn't Work / Had to Adapt

  • Agents sometimes drift from instructions. Even with clear instructions, an agent given a 10-step task list will occasionally skip step 7 or reinterpret step 4. The witness-observer skill was built specifically to catch this, but the underlying problem (instruction following degradation in long contexts) remains a limitation of current LLMs.
  • Very large agents perform worse than focused ones. An agent given 15 tasks in a single session produces lower-quality output than three agents given 5 tasks each. The inflection point is around 7 tasks — beyond that, error rates increase noticeably.
  • Batch committing loses bisect granularity. Early in the project, agents were allowed to batch their work into a single commit at the end of a session. This made git bisect useless for regression analysis. The switch to atomic commits (one commit per logical change) restored bisectability but required more disciplined agent behavior.
  • Pure autonomous without quality gates risks drift. A fully autonomous pipeline that runs without any human checkpoints will eventually drift from the intended outcome. The solution was strategic quality gates: human review at wave boundaries, automated validation at commit time, and adversarial review at release time.
  • Token budgets need explicit management at scale. A convoy of 13 agents consuming tokens simultaneously can exhaust rate limits within minutes. The token-budget skill was built to enforce per-agent and per-convoy token limits, but token management remains one of the most operationally complex aspects of the system.
  • Cross-agent coordination adds latency. The coordination overhead of the mayor-coordinator pattern is not free. For small tasks (under 5 minutes of work), the overhead of dispatching to an agent, waiting for it to start, and collecting its results exceeds the cost of doing the work inline. The gsd-fast command exists specifically for these small tasks.

The Adaptation Curve

The most important meta-observation is that the system adapted to its own failures. Each failure mode described above was addressed not by abandoning the approach but by building a specific countermeasure:

Failure Mode Countermeasure Built Effectiveness
Agent instruction drift witness-observer skill (patrol loops, nudges) Reduced drift incidents by ~80%
Oversized agent tasks 5-6 task limit per agent (empirical) Eliminated quality degradation
Batch commit granularity Atomic commit convention + hook enforcement Restored full bisectability
Autonomous drift Strategic quality gates (wave, commit, release) Bounded maximum drift distance
Token exhaustion token-budget skill (per-agent limits) Zero rate-limit failures since deployment
Coordination overhead gsd-fast for inline small tasks 90% latency reduction for sub-5-min work

This adaptation loop — observe failure, build countermeasure, verify effectiveness — is itself an agentic pattern. The system did not merely execute work; it learned from its execution failures and modified its own orchestration infrastructure to prevent recurrence. This is the bounded self-modification described in Section 15.7, applied to the orchestration layer rather than the safety layer.

Remaining Limitations

Several limitations remain unresolved as of April 2026:

  • Single-machine constraint. All work runs on one workstation. Distributing agents across multiple machines would require a network coordination layer that does not yet exist in the architecture.
  • LLM reasoning ceiling. Chain scores have plateaued at 4.75. Improving mathematical rigor beyond this level likely requires more capable base models or dedicated theorem-proving tools.
  • Context window as hard limit. Despite aggressive state externalization, some tasks genuinely require more context than any current model provides. These tasks must be decomposed, which introduces information loss at decomposition boundaries.
  • Reproducibility. LLM outputs are non-deterministic. Running the same pipeline twice may produce different (though equally valid) results. This makes exact reproduction impossible, though the quality range is consistent.

The Honest Assessment

This system is not perfect. It fails, it drifts, it requires supervision. But it has produced 190+ research projects, cataloged 720 missions, maintained live mission coverage, generated 86,000 words of energy research, built a GPU-accelerated math engine, validated itself mathematically across 50 versions, and done it all on a single workstation. The patterns work. The evidence is in the output.
