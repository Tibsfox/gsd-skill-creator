The Language Itself

Terms, unification, SLD resolution, cut, negation as failure, DCGs, meta-programming — the mental-model shift from everything else
Overview Language History Implementations Expert Systems CLP Datalog & ASP Rosetta Learn
“Algorithm = Logic + Control.” — Robert Kowalski (1979). Every algorithm has two orthogonal components: the logic that specifies what the problem is, and the control that specifies how the search should proceed. Prolog takes the radical position that the logic alone should be sufficient to define the program, and that a general-purpose theorem prover can supply the control. You write down what you know; Prolog figures out how to derive what you ask.

1. The Declarative Vision

Conventional programming languages fuse logic and control inseparably. When you write a for loop in C, you are simultaneously specifying the logical structure of the computation and dictating its execution order. Prolog takes the declarative dream seriously: a Prolog program is a set of logical axioms, and running it is asking the system to prove a theorem.

% Facts: Things we know
parent(tom, bob).
parent(tom, liz).
parent(bob, ann).
parent(bob, pat).

% Rule: What grandparenthood means
grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

?- grandparent(tom, X).
X = ann ;
X = pat ;
false.

You never told Prolog how to find grandchildren. You told it what a grandparent is. The system figured out the rest.

Programming by stating facts and rules

In imperative languages you write recipes. In functional languages you write transformations. In Prolog you write definitions. Consider the length of a list:

ParadigmCodeWhat it says
Pythondef length(lst): c=0; for _ in lst: c+=1; return cRecipe: walk, count
Haskelllength [] = 0; length (_:xs) = 1 + length xsTransformation: input → output
Prologlength([], 0). length([_|T], N) :- length(T, N0), N is N0+1.Relation: when does length hold?

The Prolog version is not a function from lists to numbers. It is a relation that holds between a list and a number whenever the number is the length of the list. In principle you could run it backwards — ask “what list has length 3?” — although Prolog’s answer to that particular query is an infinite generator of lists, which is a separate issue.

Horn clauses

Prolog’s logical foundation is a restricted fragment of first-order logic called Horn clauses. A Horn clause is a disjunction of literals with at most one positive literal, equivalent to an implication:

(p1 ∧ p2 ∧ … ∧ pn) → q

Which in Prolog syntax is q :- p1, p2, ..., pn. — “q is true if all of the pi are true.” Why this specific fragment? Because SLD resolution is a sound and complete proof procedure for Horn clauses, and it can be implemented efficiently. Full first-order logic is undecidable; Horn clause logic is tractable. The trade-off is that you cannot express disjunction in the head — sometimes a real pain, but in exchange you get a language that actually runs.

2. Terms — The Universal Data Structure

Everything in Prolog is a term. Data is a term. Code is a term. A query is a term. Prolog is profoundly homoiconic — the language never draws a line between syntax and semantics, between program and data. The grammar of terms is tiny: exactly four kinds.

KindDescriptionExamples
AtomSymbolic constant; equal only to itselffoo, bar, +, 'Hello, World!'
NumberInteger (arbitrary precision in modern Prologs) or float42, -3, 3.14, 2^1000
VariableStarts with uppercase or _; can be bound once per branchX, Result, _foo, _
CompoundFunctor + arguments; identified by name/aritypoint(1,2), date(2026,4,9)

Lists, strings, operators, tuples, records — everything you might expect to be a distinct type in another language — is built from these four primitives.

Lists are sugar for .(H,T)

[]                      % empty list (atom)
[a]                     % .(a, [])
[a, b, c]               % .(a, .(b, .(c, [])))
[H|T]                   % head/tail pattern
[a, b, c | Rest]        % three known elements followed by Rest

The lexical rule “uppercase starts a variable” is load-bearing. Writing likes(Alice, Bob) when you meant likes(alice, bob) produces a clause that is vacuously true for any two people — because Alice and Bob are universally quantified variables, not proper names.

Operators are just syntactic sugar

X + Y is not special syntax. It is the compound term +(X, Y) written with infix convention because + has been declared as an operator. You can see the underlying structure with =.. (“univ”):

?- X = 1 + 2, X =.. L.
X = 1+2,
L = [+, 1, 2].

You declare your own operators with op/3. The DCG arrow --> is just a user-defined operator — syntax to make grammar rules look like grammar rules.

3. Unification — The Heart of the Language

Unification is the single most important mechanism in Prolog. Everything rests on it. Given two terms, unification asks: is there a way to bind the variables so that the terms become syntactically identical? If yes, unification succeeds and produces the bindings; if no, it fails.

?- foo(1, 2) = foo(1, 2).              % Identical
true.

?- foo(X, 2) = foo(1, Y).              % Bind X=1, Y=2
X = 1, Y = 2.

?- foo(X, X) = foo(1, 2).              % Need X=1 AND X=2. Impossible.
false.

?- [H|T] = [a, b, c].                  % Standard list destructuring
H = a, T = [b, c].

The =/2 operator is exactly unification. It is not assignment. It is not equality testing. It is the process of finding a substitution that makes two terms syntactically equal.

Robinson's algorithm (1965)

J. Alan Robinson published unification in 1965 as the computational heart of resolution theorem proving. The algorithm is astonishingly simple:

  1. If both terms are identical atoms or numbers, succeed.
  2. If either term is an unbound variable, bind it to the other. Succeed.
  3. If both are compound with the same functor and arity, recursively unify corresponding arguments.
  4. Otherwise, fail.

This single operation generalizes pattern matching, parameter passing, and equation solving. Robinson's algorithm always produces a most general unifier (MGU) — the “smallest” or “least committed” substitution that works. Further constraints may pin things down later.

The occurs check

Step 2 has a subtle problem. What if the “other term” contains the variable? Consider X = foo(X). Naively: bind X to foo(X), which contains X, which is bound to foo(X)… an infinite term. Not a well-formed first-order term at all.

The occurs check would catch this by testing whether the variable already occurs in the term before binding. It is mathematically correct. But it turns unification from amortized constant time into linear time, and most Prolog programs never actually need it — so Prolog’s default unification is technically unsound. ISO provides unify_with_occurs_check/2 for the cases where you need it.

SWI-Prolog embraces the cyclic case as a feature, supporting rational trees as first-class citizens. Other systems treat it as a latent bug. This is a genuine disagreement in the Prolog world.

Unification is bidirectional — the time-travel effect

If you come from a functional language, unification looks like pattern matching. It is not. Pattern matching is a restricted, one-directional form of unification. In Haskell, length [1,2,3] matches (x:xs) by flowing information one way: input to pattern. Prolog’s unification is bidirectional — both sides can contain variables, and binding happens on both sides.

This sounds like a small difference. It is not. Because unification is bidirectional, the same predicate can often be called in multiple modes:

append([], L, L).
append([H|T], L, [H|R]) :- append(T, L, R).

% Mode 1: Concatenate two lists
?- append([1,2], [3,4], X).
X = [1, 2, 3, 4].

% Mode 2: Split a list (what two lists concatenate to [1,2,3]?)
?- append(X, Y, [1,2,3]).
X = [], Y = [1,2,3] ;
X = [1], Y = [2,3] ;
X = [1,2], Y = [3] ;
X = [1,2,3], Y = [] ;
false.

% Mode 3: Find the suffix starting at a prefix
?- append([1,2], Y, [1,2,3,4]).
Y = [3,4].

You wrote the definition of concatenation. You got decomposition, prefix matching, and suffix extraction for free. This is the time-travel effect — predicates running “backwards” — that makes Prolog feel unique.

4. SLD Resolution and Proof Search

When you ask a Prolog query, the system tries to prove it using SLD resolution. The acronym is unhelpful but each letter matters:

  • Selection — pick a subgoal to work on next (Prolog picks leftmost)
  • Linear — the proof is a linear sequence of resolution steps
  • Definite — restricted to definite clauses (Horn clauses with exactly one positive literal)

Prolog’s default strategy: leftmost subgoal first, clauses tried top-to-bottom, depth-first with backtracking. This is simple to implement and gives reasonable performance, but it is incomplete — some provable goals will cause an infinite loop.

Choice points and backtracking

Whenever Prolog encounters a subgoal that could be satisfied by more than one clause, it creates a choice point — a record of the state so it can come back and try the next clause if the current one fails. This is how search happens for free.

food(apple).
food(bread).
food(pizza).

likes(alice, X) :- food(X).

?- likes(alice, X).
X = apple ;     % Prolog returns the first solution
X = bread ;     % User types ; to backtrack
X = pizza ;
false.          % No more choices

Order of clauses and order of subgoals both matter in practice, even though they “shouldn’t” in the pure logical reading. Left recursion (p(X) :- p(Y), edge(Y, X).) causes infinite loops. Generate-and-test is the canonical search pattern.

5. The Cut (!)

The cut (!) is Prolog’s escape hatch from the declarative ideal. It prunes choice points — tells the search engine “don’t backtrack past this point.” It is the single most controversial feature of the language, and the one that makes the operational reading essential.

O’Keefe’s classic distinction: green cuts don’t change the declarative meaning (they just prune redundant work), while red cuts do change the meaning. Red cuts are powerful and dangerous; green cuts are merely powerful.

% Green cut: after finding a member, don't look for it again.
first_member(X, [X|_]) :- !.
first_member(X, [_|T]) :- first_member(X, T).

% Red cut: if X is positive, return abs(X) = X, else -X.
% The cut changes the meaning if you remove it!
abs_val(X, X)  :- X >= 0, !.
abs_val(X, NX) :- NX is -X.

The cleaner modern alternative for the red-cut case is if-then-else (-> ;), which is syntactic sugar over cut but makes intent clearer:

abs_val(X, Y) :-
    ( X >= 0 -> Y = X
    ; Y is -X
    ).

Markus Triska’s modern school of strict-ISO Prolog advocates avoiding cut whenever possible, preferring CLP(FD) constraints and careful pattern matching. The old school still uses it. Both work.

6. Negation as Failure

Prolog’s \+/1 (historically not/1) is negation as failure, not classical logical negation. \+ Goal succeeds iff Goal fails. This is subtly but importantly different from “Goal is false.”

bird(sparrow).
bird(eagle).

flies(X) :- bird(X), \+ penguin(X).
penguin(X) :- fail.  % no penguins in our database

?- flies(sparrow).
true.

?- flies(penguin).   % No fact. Prolog assumes it's not a bird.
false.

This is the closed world assumption: what cannot be proved is assumed to be false. It is a pragmatic stance that works well for closed domains (a database of facts) but breaks down for open domains (knowledge about the real world).

Floundering

Negation-as-failure misbehaves on non-ground goals. If you write \+ p(X) while X is unbound, Prolog tries to prove p(X), which may succeed with some binding, and then returns failure for \+ p(X) — discarding the binding. You get the right answer for the wrong reasons, or the wrong answer.

The fix is stable model semantics, introduced by Gelfond and Lifschitz in 1988 and elaborated into modern Answer Set Programming (see Datalog & ASP). ASP solvers properly handle negation in logic programs with loops, at the cost of abandoning Prolog’s procedural reading entirely.

7. Lists and Recursion

Every repetition in Prolog is a recursive call. There are no loops. The classics:

% Membership
member(X, [X|_]).
member(X, [_|T]) :- member(X, T).

% Append (three-mode magic)
append([], L, L).
append([H|T], L, [H|R]) :- append(T, L, R).

% Reverse (naive, O(n^2))
reverse([], []).
reverse([H|T], R) :- reverse(T, RT), append(RT, [H], R).

% Reverse with accumulator (linear)
reverse(L, R) :- reverse(L, [], R).
reverse([], Acc, Acc).
reverse([H|T], Acc, R) :- reverse(T, [H|Acc], R).

Difference lists

The accumulator pattern generalizes to difference lists: represent a list as a pair L-Hole, where L is the list with Hole as an unbound tail. Appending is then O(1):

% Difference-list append: (A-B) ++ (B-C) = (A-C)
append_dl(A-B, B-C, A-C).

This is the technique behind fast DCG parsing and many industrial-strength Prolog programs.

findall, bagof, setof

To collect all solutions of a query into a list, use the “meta-call” predicates:

?- findall(X, member(X, [1,2,3]), L).
L = [1, 2, 3].

?- findall(X, (member(X, [1,2,3,4,5]), X > 2), L).
L = [3, 4, 5].

bagof/3 and setof/3 are similar but fail when there are no solutions (instead of returning []) and respect free variables in subtle ways.

8. Arithmetic — The Famous is/2 Gotcha

Every Prolog learner trips over this. Arithmetic in standard Prolog is not relational; it is evaluated by the is/2 operator, which requires the right-hand side to be ground (fully instantiated) at evaluation time.

?- X = 1 + 2.       % Unification. X is bound to the TERM 1+2.
X = 1+2.            % Not evaluated!

?- X is 1 + 2.      % Evaluation. X is bound to 3.
X = 3.

?- X is Y + 1.      % Error: Y is not instantiated.
ERROR: is/2: Arguments are not sufficiently instantiated.

This is perhaps the most famous Prolog gotcha. Newcomers write X = Y + 1 expecting addition and get term construction. The comparison operators are equally confusing:

OperatorMeaningExample
=UnificationX = 1+2 binds X = 1+2
==Strict term equality (no unification)1+2 == 1+2 → true
=:=Arithmetic equality (evaluate both sides)1+2 =:= 3 → true
=..Univ (term ↔ list)foo(1,2) =.. L → L=[foo,1,2]
isEvaluate arithmetic and unifyX is 1+2 binds X = 3

The fix for relational arithmetic is CLP(FD) (Constraint Logic Programming over Finite Domains). With library(clpfd), X #= Y + 1 is a constraint that works bidirectionally, and Prolog can reason about it symbolically. See the CLP page.

9. Definite Clause Grammars (DCGs)

DCGs are Prolog’s killer feature for parsing. They are syntactic sugar over difference lists that make grammar rules look like grammar rules. The --> arrow is an operator; at load time each DCG rule is rewritten into a regular Prolog clause with two extra arguments.

% DCG grammar
sentence --> noun_phrase, verb_phrase.
noun_phrase --> [the], noun.
noun_phrase --> [a], noun.
verb_phrase --> verb, noun_phrase.
noun --> [cat].
noun --> [dog].
noun --> [ball].
verb --> [sees].
verb --> [chases].

% Run it with phrase/2
?- phrase(sentence, [the, cat, chases, a, ball]).
true.

?- phrase(sentence, S).         % Generate a sentence!
S = [the, cat, sees, the, cat] ;
S = [the, cat, sees, the, dog] ;
...

DCGs can carry parse-tree arguments, semantic values, or anything else. They compose cleanly, and for context-free or mildly context-sensitive grammars they are among the most concise parsing formalisms ever devised. Colmerauer’s original 1972 French Q&A system was a DCG before the notation even existed.

A DCG for arithmetic expressions:

expr(E)   --> term(T), expr_rest(T, E).
expr_rest(Acc, E) --> [+], term(T), { Acc2 is Acc + T }, expr_rest(Acc2, E).
expr_rest(Acc, Acc) --> [].
term(N)   --> [N], { number(N) }.

?- phrase(expr(V), [1, +, 2, +, 3]).
V = 6.

10. Meta-Programming — Code Is Data

Prolog is homoiconic in a way even Lisp envies. Code is a term. You can inspect, generate, and transform programs using the same tools you use for data. call/N invokes a goal dynamically. clause/2 retrieves the body of any defined predicate. assert/1 and retract/1 modify the database at runtime.

A Prolog meta-interpreter in ten lines

solve(true) :- !.
solve((A, B)) :- !, solve(A), solve(B).
solve((A ; _)) :- solve(A).
solve((_ ; B)) :- solve(B).
solve(\+ A) :- !, \+ solve(A).
solve(Goal) :-
    clause(Goal, Body),
    solve(Body).

That is a working Prolog interpreter written in Prolog. You can extend it with tracing, step-counting, proof recording, or constraint propagation — each addition is a few more lines. This is why meta-circular evaluation is a standard Prolog teaching exercise.

11. The Gotchas Hall of Fame

  • is/2 vs =/2. X = 1+2 is unification (term construction); X is 1+2 is evaluation.
  • Uppercase atoms. likes(Alice, Bob) uses variables, not proper names.
  • Red cuts. A cut without surrounding guards can silently change meaning.
  • Left recursion. p(X) :- p(Y), edge(Y, X). loops forever under SLD.
  • Missing occurs check. X = f(X) produces a cyclic term instead of failing.
  • Floundering negation. \+ p(X) with unbound X gives meaningless answers.
  • Singleton variables. foo(X, Y) :- bar(X). — is Y a typo or a placeholder? Prolog warns.
  • String types. What "hello" means depends on your system (char-code list? atom list? SWI string?).

12. A Gallery of Classic Programs

Quicksort (the famous 4-liner, plus partition)

quicksort([], []).
quicksort([Pivot|T], Sorted) :-
    partition(Pivot, T, Less, Greater),
    quicksort(Less, LSorted),
    quicksort(Greater, GSorted),
    append(LSorted, [Pivot|GSorted], Sorted).

partition(_, [], [], []).
partition(P, [H|T], [H|L], G) :- H =< P, partition(P, T, L, G).
partition(P, [H|T], L, [H|G]) :- H >  P, partition(P, T, L, G).

N-Queens (CLP(FD) version)

:- use_module(library(clpfd)).

queens(N, Qs) :-
    length(Qs, N),
    Qs ins 1..N,
    all_different(Qs),
    safe(Qs),
    label(Qs).

safe([]).
safe([Q|Qs]) :- no_attack(Q, Qs, 1), safe(Qs).

no_attack(_, [], _).
no_attack(Q, [Q1|Rest], D) :-
    Q #\= Q1 + D,
    Q #\= Q1 - D,
    D1 #= D + 1,
    no_attack(Q, Rest, D1).

Tower of Hanoi (six lines)

hanoi(1, From, To, _) :-
    format("Move disk from ~w to ~w~n", [From, To]).
hanoi(N, From, To, Via) :-
    N > 1, M is N - 1,
    hanoi(M, From, Via, To),
    hanoi(1, From, To, _),
    hanoi(M, Via, To, From).

Zebra puzzle setup

The famous “Einstein’s riddle” — 5 houses, 5 nationalities, 5 pets, 5 drinks, 5 cigarettes, 14 clues — translates into Prolog almost verbatim as member/2 and next_to/3 constraints. About 25 lines of Prolog. The equivalent Python solution is 100+.

13. The Mental Model

After all of this, what does it actually feel like to program in Prolog? The core insight: a Prolog program is a set of relations, and running it is asking “is this true? and if so, how?”

  1. Predicates are bidirectional by default. Your code describes relationships, not computations. The same predicate can usually be used in multiple modes.
  2. Recursion replaces iteration entirely. No for, no while. Every repetition is a recursive call. The accumulator pattern is the canonical shape.
  3. Choice is first-class. Disjunction, multiple clauses, and backtracking make non-determinism native. Search algorithms fall out of just writing the rules.
  4. Data and code are the same thing. Every program is a term. Meta-interpreters, DSL compilers, and program analyzers fall out of basic language features.
  5. Failure is information. fail and unbinding are productive operations. Failure drives backtracking; backtracking drives search; search solves problems.
  6. The declarative dream is slightly out of reach. Prolog’s fixed search strategy means you must still think operationally. This is the recurring tension of the language.
  7. Small programs do enormous work. When Prolog fits, it fits by an order of magnitude in code size. When it doesn’t, it fights you.

Prolog is a language for describing what you know, not what you do. Its power lies in what happens automatically once you have described enough: unification parameterizes your predicates, backtracking searches your solution space, SLD resolution proves your theorems. You write less, and the system does more — as long as you can stomach the occasional occurs check, the floundering negation, the inscrutable cut, and the famous is/2 gotcha.

It is a language from 1972 that still feels, in 2026, like a glimpse of how programming could have been. Not a replacement for everything, but a way of thinking that every serious programmer should encounter at least once.
