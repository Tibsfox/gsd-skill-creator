The point of the Rosetta project isn’t that every language is the same. It’s that translating a problem between paradigms teaches you what the problem really is. Write it in Prolog and you learn what is declarative about it. Write it in SQL and you learn what is relational. Write it in Haskell and you learn what is compositional. Write it in Z3 and you learn what is a constraint.

One Problem, Seven Languages — Family Relationships

The classic introductory Prolog example: given parent facts, compute grandparents, siblings, and ancestors. Watch how each paradigm reshapes the same problem.

Prolog (the reference)

parent(tom, bob).
parent(tom, liz).
parent(bob, ann).
parent(bob, pat).
parent(pat, jim).

grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

sibling(X, Y) :- parent(P, X), parent(P, Y), X \= Y.

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).

?- ancestor(tom, jim).
true.

?- grandparent(tom, X).
X = ann ; X = pat.

Datalog (no function symbols, guaranteed terminating)

.decl parent(p: symbol, c: symbol)
.decl grandparent(gp: symbol, gc: symbol)
.decl ancestor(a: symbol, d: symbol)

parent("tom", "bob").
parent("tom", "liz").
parent("bob", "ann").
parent("bob", "pat").
parent("pat", "jim").

grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).

.output ancestor

This is Souffle syntax. The Datalog program is nearly identical but always terminates because there are no function symbols and the fact base is finite.

SQL (recursive CTEs)

CREATE TABLE parent (p TEXT, c TEXT);
INSERT INTO parent VALUES
  ('tom','bob'), ('tom','liz'), ('bob','ann'),
  ('bob','pat'), ('pat','jim');

-- grandparents
SELECT p1.p AS gp, p2.c AS gc
FROM parent p1 JOIN parent p2 ON p1.c = p2.p;

-- ancestors (recursive CTE)
WITH RECURSIVE ancestor(a, d) AS (
  SELECT p, c FROM parent
  UNION
  SELECT a.a, p.c FROM ancestor a JOIN parent p ON a.d = p.p
)
SELECT * FROM ancestor;

SQL with recursive CTEs (SQL:1999) is Turing-complete and in practice is the “Datalog that everyone already has installed.” The shape of the query is the same.

miniKanren (in Scheme/Racket/Clojure)

(define (parento p c)
  (conde
    ((== p 'tom) (== c 'bob))
    ((== p 'tom) (== c 'liz))
    ((== p 'bob) (== c 'ann))
    ((== p 'bob) (== c 'pat))
    ((== p 'pat) (== c 'jim))))

(define (ancestoro a d)
  (conde
    ((parento a d))
    ((fresh (m) (parento a m) (ancestoro m d)))))

(run* (q) (ancestoro 'tom q))
; => (bob liz ann pat jim)

miniKanren is a tiny, pure, embedded relational DSL. It lives in ~200 lines of host language. Popular in Scheme, Clojure (core.logic), Racket, and Python (pykanren).

Haskell (list comprehensions as logic programming)

parent :: [(String, String)]
parent = [("tom","bob"),("tom","liz"),("bob","ann"),
          ("bob","pat"),("pat","jim")]

grandparent :: String -> [String]
grandparent gp = [gc | (p1, c1) <- parent, p1 == gp,
                       (p2, gc) <- parent, p2 == c1]

-- Fixed-point ancestor relation
ancestor :: [(String, String)]
ancestor = go parent
  where
    go rel = let rel' = rel ++ [(a, c) | (a, b) <- rel,
                                         (b', c) <- parent, b == b']
             in if rel == rel' then rel else go rel'

Haskell’s list comprehensions are syntactic Prolog — the connection is explicit in the comprehension calculus literature (Wadler, 1987).

Python (declarative via pyDatalog or networkx)

from pyDatalog import pyDatalog as pdl

pdl.create_terms('parent, ancestor, grandparent, X, Y, Z')

+parent('tom', 'bob')
+parent('tom', 'liz')
+parent('bob', 'ann')
+parent('bob', 'pat')
+parent('pat', 'jim')

grandparent(X, Z) <= parent(X, Y) & parent(Y, Z)
ancestor(X, Y)    <= parent(X, Y)
ancestor(X, Y)    <= parent(X, Z) & ancestor(Z, Y)

print(ancestor('tom', Y))
# [('bob',), ('liz',), ('ann',), ('pat',), ('jim',)]

pyDatalog is one of several Python libraries that embed logic programming — others include PyKE, problog.py, and PySwip (full SWI-Prolog bridge).

Z3 (constraint solving)

from z3 import *

P = Function('parent', StringSort(), StringSort(), BoolSort())
A = Function('ancestor', StringSort(), StringSort(), BoolSort())

facts = [P(StringVal('tom'), StringVal('bob')),
         P(StringVal('tom'), StringVal('liz')),
         P(StringVal('bob'), StringVal('ann')),
         P(StringVal('bob'), StringVal('pat')),
         P(StringVal('pat'), StringVal('jim'))]

# ancestor as fixed point: encode recursively via ForAll
# (Z3 handles fixed points via Datalog fixedpoint engine, muZ)

Z3 has a built-in Datalog/muZ fixed-point engine. You can feed it rules directly. In production, Z3’s fixed-point engine is often how modern projects do Datalog-style queries without installing a separate tool.

When Logic Programming Beats Everything Else

Recursive fact-base queries. “Given these rules and these facts, what follows?” Datalog and Prolog express this in five lines; an imperative version is 50+.

Combinatorial search and constraints. N-queens, Sudoku, scheduling, graph coloring. CLP(FD) and ASP have first-class propagation and search that you’d reimplement by hand in Python.

Symbolic reasoning. Theorem proving, type inference, program analysis. The Rust borrow checker (Polonius) is literally Datalog. Java pointer analysis (Doop) is Datalog. Infer is not, but the moral equivalent.

Natural-language grammars. DCGs remain one of the most concise ways to write a parser for a non-trivial grammar, especially for linguistic formalisms (HPSG, CCG).

Hypothesis testing via unification. Configuration, planning, diagnosis, legal reasoning, rule engines. The expert-system lineage.

When Logic Programming Loses

Number-crunching and linear algebra. Use NumPy, Julia, Fortran, CUDA. Prolog’s arithmetic is procedural and slow.

Imperative state machines. Use Go, Rust, C. Prolog can model them but it’s awkward.

Machine learning. Use PyTorch, JAX. Logic programming is a collaborator to ML (neuro-symbolic), not a replacement.

UI and I/O-heavy code. Use anything else. Prolog has HTTP servers (SWI’s is very good), but nobody reaches for it first.

Translation Heuristics

  1. From Prolog to SQL: Ground facts become tables; unification becomes JOIN; recursion becomes WITH RECURSIVE; negation-as-failure becomes NOT EXISTS.
  2. From Prolog to Haskell: Facts become list elements; rules become list comprehensions; unification becomes pattern matching; backtracking becomes the list monad.
  3. From Prolog to Python: Use a library (pyDatalog, PySwip, problog, or embed miniKanren). Do not try to reimplement unification by hand unless you’re writing a toy.
  4. From Prolog to Datalog: Remove function symbols; stratify negation; trust bottom-up evaluation.
  5. From imperative to Prolog: Invert control — describe the solution, not the procedure. Let unification and search do the work. If you find yourself writing explicit loops, you’re not speaking Prolog yet.
