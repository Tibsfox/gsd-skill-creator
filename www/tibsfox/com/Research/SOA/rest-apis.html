REST & Modern APIs

Fielding, the Richardson Maturity Model, OpenAPI, GraphQL, AsyncAPI — the simpler approach that beat SOAP
Overview Principles History Enterprise SOA REST & APIs Microservices Implementations Rosetta Learn
REST won because it was approximately right and simple, not because it was exactly right. Fielding’s 2000 dissertation described how the web actually worked; most of the APIs that call themselves “RESTful” are not, by his definition. It didn’t matter. By 2010, REST + JSON was the default way to build a web API, and every front-end developer with an iPhone app agreed.

1. Fielding’s REST (2000)

Roy Thomas Fielding, PhD dissertation at UC Irvine, 2000: Architectural Styles and the Design of Network-based Software Architectures. 180 pages. Chapter 5 defines REST. It is probably the most-read PhD dissertation in software architecture history.

The six constraints of REST

  1. Client-server separation.
  2. Statelessness — each request contains all the information needed; no server-side session state.
  3. Cacheability — responses must label themselves as cacheable or not.
  4. Uniform interface — the defining constraint, itself four sub-constraints:
    • Identification of resources (URIs)
    • Manipulation of resources through representations
    • Self-descriptive messages
    • HATEOAS — Hypermedia As The Engine Of Application State
  5. Layered system — clients cannot tell if they’re talking to the server or an intermediary.
  6. Code-on-demand (optional) — servers can extend client functionality by transferring executable code (e.g., JavaScript).

The argument: the Web is an implementation of REST. HTTP + URIs + HTML (or JSON) forms a REST architecture, and the Web’s success — its scalability, resilience, ability to evolve over decades — is attributable to these constraints. Fielding was not inventing something new; he was giving a name and systematic description to what the Web already was.

REST as a critique of RPC

The dissertation contains an implicit and often-explicit critique of the RPC tradition. RPC tries to make a distributed system look like a local program by hiding the network; REST embraces the network by exposing it.

When Fielding was asked years later whether SOAP/WSDL architectures were “RESTful,” his response: no, they were RPC systems mislabeled as Web architectures. SOAP used HTTP only as a transport for tunneling method calls. It ignored HTTP verbs (almost all SOAP calls were POSTs), ignored URIs (all calls went to the same endpoint with an operation name in the body), ignored caching, and ignored content negotiation.

2. The Richardson Maturity Model (2008)

In 2008, software architect Leonard Richardson gave a QCon San Francisco talk proposing a four-level maturity model. Martin Fowler popularized it in a March 2010 blog post: “Richardson Maturity Model: steps toward the glory of REST.”

  1. Level 0: The Swamp of POX — single endpoint, single HTTP verb (usually POST). The service uses HTTP as a tunnel for XML messages. SOAP lives here. XML-RPC lives here.
  2. Level 1: Resources — distinct URIs for distinct resources, but still typically one verb.
  3. Level 2: HTTP Verbs — GET, POST, PUT, DELETE in their standard semantics; meaningful HTTP status codes. Most “REST APIs” in production live here.
  4. Level 3: Hypermedia Controls (HATEOAS) — responses include links to related resources; clients navigate by following links rather than constructing URIs. Almost nobody.

Fielding himself noted that only Level 3 is actually REST. Levels 0, 1, 2 are HTTP-based APIs but not REST architectures. The vast majority of “RESTful APIs” are Level 2, and Fielding-purists continue to point this out, to the continued annoyance of everyone else.

3. Why “RESTful” Won

For all the pedantry about what REST really was, the name won. By 2010, “RESTful API” was the default thing you built when someone asked for a web API:

  • JSON over XML. Around 2006, a grassroots shift from XML to JSON happened in the Rails and Django communities. JSON was simpler, more compact, and native to JavaScript.
  • The iPhone effect. iPhone launched June 2007, App Store July 2008. Every company suddenly needed a mobile backend. REST+JSON was perfect; SOAP+XML was a disaster.
  • Tooling disappeared. You did not need a WSDL compiler. curl | jq replaced an entire category of code generation.
  • Documentation by example. Instead of a 2000-line WSDL, REST APIs were documented by human-written pages showing example requests and responses.
  • The Rails movement. Ruby on Rails 1.0 (December 2005), Rails 2.0 (December 2007) made REST-by-convention the standard way to build Rails applications.
  • Twitter and the public API era. Twitter’s 2006 API proved that public APIs could be simple and popular.

The lesson: approximately-right-but-simple almost always beats exactly-right-but-complicated in infrastructure technology.

4. HTTP as the REST Protocol

Verbs and semantics

VerbSemanticsIdempotent?Safe?
GETRetrieve a representation of a resourceYesYes (no side effects)
HEADLike GET but only headersYesYes
POSTCreate a new resource, or non-idempotent actionNoNo
PUTCreate or replace a resource at a specific URIYesNo
PATCHPartial updateDebatableNo
DELETERemove a resourceYesNo
OPTIONSDescribe communication options (CORS preflight)YesYes

Status codes (the important ones)

  • 2xx Success: 200 OK, 201 Created, 202 Accepted, 204 No Content
  • 3xx Redirect: 301 Moved Permanently, 302 Found, 304 Not Modified (cache still valid)
  • 4xx Client error: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests
  • 5xx Server error: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout

Headers that matter

  • Authentication: Authorization: Bearer ..., Authorization: Basic ...
  • Content negotiation: Accept: application/json, Content-Type: application/json
  • Caching: Cache-Control, ETag, If-None-Match, Last-Modified, If-Modified-Since
  • Rate limiting: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
  • Idempotency: Idempotency-Key (Stripe pattern)
  • Versioning: API-Version, or vendor media types
  • Tracing: traceparent, tracestate (W3C Trace Context)

5. OpenAPI / Swagger

Swagger was started by Tony Tam at Wordnik around 2011. Swagger 1.0, 1.1, 1.2, 2.0 became the de facto REST API description language. In 2015, Swagger 2.0 was donated to the Linux Foundation as the OpenAPI Initiative, and Swagger 2.0 was renamed OpenAPI 2.0. OpenAPI 3.0 (2017), 3.1 (2021).

openapi: 3.1.0
info:
  title: Users API
  version: 1.0.0
paths:
  /users/{id}:
    get:
      summary: Get a user by ID
      parameters:
        - in: path
          name: id
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: User found
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          description: User not found
components:
  schemas:
    User:
      type: object
      required: [id, name, email, created_at]
      properties:
        id:         { type: integer }
        name:       { type: string }
        email:      { type: string, format: email }
        created_at: { type: string, format: date-time }

OpenAPI tooling

  • Swagger UI — interactive API documentation
  • Swagger Editor — browser-based YAML editor with live preview
  • Swagger Codegen / OpenAPI Generator — client and server stubs in every language
  • Stoplight — API design platform
  • Redocly — OpenAPI toolchain with better docs rendering
  • SwaggerHub — SmartBear’s hosted collaboration platform
  • Spectral — linting for OpenAPI specs
  • Prism — mock server from an OpenAPI spec

Contract-first vs code-first

Two workflows:

  • Contract-first: write the OpenAPI YAML, generate stubs, fill in implementations. Better for API-first companies.
  • Code-first: write annotated Java/Go/Python/C# code, generate OpenAPI from it. Better for code-centric teams.

The “API-first” movement — Stripe, Twilio, Plaid, Adyen, SendGrid — treats the API as the product. Contract-first is the natural workflow for this culture.

6. AsyncAPI — OpenAPI for Async

Founded 2016 by Fran Méndez. AsyncAPI 1.x, 2.x, 3.x. Describes Kafka topics, MQTT channels, AMQP exchanges, WebSockets. The async equivalent to OpenAPI, filling the gap that OpenAPI explicitly ignores.

asyncapi: 3.0.0
info:
  title: User Service Events
  version: 1.0.0
channels:
  user.created:
    address: user.created
    messages:
      userCreated:
        $ref: '#/components/messages/UserCreated'
components:
  messages:
    UserCreated:
      payload:
        type: object
        properties:
          userId: { type: integer }
          email:  { type: string }
          ts:     { type: string, format: date-time }

Tooling: AsyncAPI Generator, Microcks (mocking), Studio (browser editor).

7. GraphQL

Facebook internal 2012, public release 2015. Original authors: Lee Byron and Dan Schafer.

# Schema (SDL)
type Query {
  user(id: ID!): User
  users(limit: Int = 10): [User!]!
}

type User {
  id: ID!
  name: String!
  email: String!
  posts(limit: Int = 5): [Post!]!
}

type Post {
  id: ID!
  title: String!
  body: String!
}

# Query
query {
  user(id: "123") {
    name
    posts(limit: 3) {
      title
    }
  }
}

The client specifies exactly which fields it wants. The server returns exactly those fields. This solves the “over-fetching” and “under-fetching” problems of REST for clients with diverse needs — typically mobile apps.

GraphQL servers

  • Apollo — the commercial leader; Apollo Server, Apollo Client, Apollo Federation
  • Relay — Facebook’s reference client
  • Hasura — auto-generates GraphQL from a Postgres database
  • PostGraphile — similar, PostgreSQL-focused
  • DGS — Netflix’s GraphQL framework
  • graphql-js, graphql-go, graphql-java, Juniper (Rust), Strawberry/Graphene (Python)

GraphQL federation

Apollo Federation lets multiple backend services contribute to a single GraphQL schema. A gateway stitches together partial schemas from each service. This is GraphQL’s answer to microservices: each service exposes its own slice of the schema, and the federation gateway presents a unified query surface.

The operational complexity of GraphQL

GraphQL is a specific-case win, not a REST replacement:

  • The N+1 problem. A naive implementation fetching posts for users issues one query per user. The solution is DataLoader — batch and cache field resolutions within a single request.
  • HTTP caching breaks. All GraphQL queries go to a single endpoint (usually POST), so CDNs and proxy caches don’t help.
  • Rate limiting is harder. A single query can be infinitely expensive. Tools like query complexity analysis are required.
  • Authorization granularity is harder — you have to check every field resolver.

When GraphQL fits (mobile clients with over-fetching problems, API aggregation across services, rapid front-end iteration), it fits very well. When it doesn’t, it adds operational complexity without benefit.

8. The API-First Movement

A generation of companies treated their API as the product. Some well-known examples:

  • Stripe — payments. Pioneered idempotency keys, versioning via date headers, obsessive API documentation. Every engineer interacts with the API before writing code.
  • Twilio — communications (SMS, voice). REST APIs for phone calls.
  • Plaid — financial data access. Bank account integration via a single API.
  • SendGrid — transactional email (acquired by Twilio).
  • Algolia — search as a service.
  • Auth0 — authentication as a service (acquired by Okta).
  • Cloudinary — image/video API.
  • Mapbox — maps API.

The common traits: excellent documentation, first-class SDKs in every major language, version stability (Stripe is famous for never breaking old API versions), uniform error handling, webhooks for async notifications, idempotency for safe retries. These companies showed that a well-designed REST API could be a billion-dollar product.

9. REST API Versioning

URL versioning

/api/v1/users/123. Simplest, most debuggable. Technically violates REST’s uniform-interface principle. Wins in practice.

Header versioning

Accept: application/vnd.github.v3+json. Cleaner URLs but harder to test with a browser or curl.

Date-based versioning (Stripe)

Stripe-Version: 2024-06-20. Each API change gets a date. Clients pin a version and get that version’s behavior forever. Stripe maintains every version that has ever existed, back-compatibly. The cost is huge but the benefit — never breaking customers — is enormous.

No versioning (Protobuf)

Additive schema evolution only. Add optional fields, never remove, never reuse field numbers. gRPC follows this discipline.

10. The Pragmatic Mix in 2026

A typical modern API surface uses several paradigms in different places:

  • REST + JSON for public APIs and partner integrations
  • gRPC + Protobuf for internal service-to-service calls (see microservices page)
  • GraphQL for mobile client aggregation
  • WebSockets for real-time browser updates
  • Server-Sent Events for server push to browsers
  • Webhooks for async notifications to partners
  • Kafka / AsyncAPI for internal event streaming

Each is the right tool for a specific boundary. The mistake is picking one paradigm and forcing it everywhere.

11. What REST Got Right

  • HTTP verbs as semantics. GET means read, PUT means replace, DELETE means remove. The browser, the CDN, and the load balancer all understand.
  • URIs as identity. Every resource has a name you can link to, bookmark, share, cache.
  • Status codes as a vocabulary. 404 means not found in every language.
  • JSON as the lingua franca. Every programming language in 2026 can read and write JSON out of the box.
  • Plain HTTP caching. ETags, If-None-Match, CDN invalidation — all free if you use REST correctly.
  • Debuggability. curl + a browser + your dev tools. No special client needed.
