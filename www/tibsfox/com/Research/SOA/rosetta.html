The best way to understand a service-oriented paradigm is to translate the same operation into each of them. “Get a user by ID” is the “hello world” of service APIs. Watch how the same three-parameter operation — in, out, one-shot — changes shape as you move across four decades of distributed systems.

One Operation, Six Paradigms — GetUser(id) -> User

The operation: given a user ID, return the user record with name, email, and created-at. That’s it. Watch the same idea in each major service-oriented stack from 1991 to 2026.

1. CORBA IDL (1991)

// user.idl
module UserService {
   struct User {
      long    id;
      string  name;
      string  email;
      long    created_at;  // seconds since epoch
   };

   exception NotFound { long id; };

   interface UserRepository {
      User get_user (in long id) raises (NotFound);
   };
};

Compile with idlj user.idl (Java), idl user.idl (C++), etc. Each target language gets its own stub and skeleton. The ORB (Object Request Broker) handles the wire protocol (IIOP). Your client calls user_repo.get_user(123) and it feels like a local method call — until the network fails and you have to handle the distributed exceptions.

2. SOAP / WSDL (2000)

<?xml version="1.0" encoding="UTF-8"?>
<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header>
    <wsse:Security xmlns:wsse="http://docs.oasis-open.org/wss/2004/01/oasis-200401-wss-wssecurity-secext-1.0.xsd">
      <wsse:UsernameToken>
        <wsse:Username>alice</wsse:Username>
        <wsse:Password>***</wsse:Password>
      </wsse:UsernameToken>
    </wsse:Security>
  </soap:Header>
  <soap:Body>
    <GetUserRequest xmlns="http://example.com/users">
      <UserId>123</UserId>
    </GetUserRequest>
  </soap:Body>
</soap:Envelope>

Accompanied by a WSDL document that describes the operation, the message types, the port types, the bindings (SOAP over HTTP, usually), and the service endpoint. The WSDL is typically 100+ lines of XML for a single operation. Machine-readable; not human-readable.

3. REST + JSON (2008, approximately)

GET /api/v1/users/123 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "Alice Chen",
  "email": "alice@example.com",
  "created_at": "2024-06-15T14:30:00Z"
}

Six lines of request, five lines of response. No XML. No WSDL. No SOAP envelope. curl-debuggable. Every browser can make one. Every mobile app can consume one. This is why REST won.

4. OpenAPI description (2015+)

paths:
  /users/{id}:
    get:
      summary: Get a user by ID
      parameters:
        - in: path
          name: id
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: User found
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          description: User not found

components:
  schemas:
    User:
      type: object
      required: [id, name, email, created_at]
      properties:
        id:         { type: integer }
        name:       { type: string }
        email:      { type: string, format: email }
        created_at: { type: string, format: date-time }

Same operation, now with a machine-readable schema. OpenAPI gives you Swagger UI docs, client/server code generation in any language, mock servers, contract testing, and API gateway configuration — all from this one file. This is “WSDL done right” — the documentation/contract benefits without the XML tax.

5. gRPC / Protocol Buffers (2015)

// user.proto
syntax = "proto3";
package example.users.v1;

message User {
  int64  id         = 1;
  string name       = 2;
  string email      = 3;
  int64  created_at = 4;  // unix seconds
}

message GetUserRequest {
  int64 id = 1;
}

service UserService {
  rpc GetUser (GetUserRequest) returns (User);
}

Compile with protoc --go_out=. --go-grpc_out=. user.proto. You get:

  • A typed stub for the client: client.GetUser(ctx, &GetUserRequest{Id: 123})
  • A typed server interface you implement
  • Binary wire format (much smaller than JSON)
  • HTTP/2 multiplexing
  • Built-in deadlines, cancellation, streaming
  • Code generation in every major language

gRPC is “CORBA done right.” The IDL-driven code generation is the same idea. Protobuf is the wire format CORBA should have had. HTTP/2 is the transport IIOP should have been.

6. GraphQL (2015)

# Schema
type Query {
  user(id: ID!): User
}

type User {
  id: ID!
  name: String!
  email: String!
  createdAt: DateTime!
}

# Query
query {
  user(id: "123") {
    name
    email
  }
}

# Response
{
  "data": {
    "user": {
      "name": "Alice Chen",
      "email": "alice@example.com"
    }
  }
}

The client specifies exactly which fields it wants. The server returns exactly those fields. This solves the “over-fetching” and “under-fetching” problems of REST for clients with diverse needs (typically mobile apps). The server pays with complexity (N+1 queries, caching, rate limiting).

7. Event-driven / Kafka (2011+)

# Topic: users.created (or users.v1.events)
# Partition key: user_id
# Value schema: AsyncAPI or Avro in a schema registry

{
  "eventId": "01JA3F2G5K8H7Y6M4N2P1Q0R9S",
  "eventType": "user.created",
  "eventTime": "2024-06-15T14:30:00Z",
  "userId": 123,
  "data": {
    "name": "Alice Chen",
    "email": "alice@example.com"
  }
}

# Consumer code (pseudocode)
for event in consumer.subscribe("users.created"):
    handle_user_created(event.data)

Instead of asking “give me user 123,” the event-driven model says “tell me whenever a user is created/updated/deleted.” Consumers maintain their own materialized views of user data. The trade-off: eventual consistency instead of synchronous correctness. The benefit: services are decoupled in time as well as space.

When Each Paradigm Wins

REST + JSON wins when you need

  • Public APIs for third-party developers. Discoverability, curl-debuggability, tooling ubiquity.
  • Browser clients. JavaScript and JSON are native friends.
  • HTTP caching. GET requests get ETags, CDNs, and proxy caches for free.
  • Simple CRUD over databases. The operation maps naturally to HTTP verbs on resources.

gRPC wins when you need

  • Internal service-to-service communication. Wire efficiency, strong typing, streaming, deadlines.
  • Polyglot teams. Every language gets the same generated client from one .proto file.
  • Backward-compatible evolution. Protobuf’s field-number discipline makes schema evolution safe.
  • Bidirectional streaming. Chat, real-time updates, watch APIs, agent-coordinator patterns.

GraphQL wins when you need

  • Mobile clients with over-fetching problems. Specify exactly what you want.
  • API aggregation. Federate multiple backend services into one query.
  • Rapid UI iteration. Front-end teams can add new queries without waiting for a new REST endpoint.

Event-driven / Kafka wins when you need

  • High-throughput event streams. Millions of events per second per topic.
  • Decoupled consumers. New consumers join the stream without asking the producer.
  • Audit trails. The log is the source of truth.
  • CQRS and event sourcing. Different read models from the same event log.

SOAP wins when you need

  • B2B integration with a legacy partner. Banks, insurance companies, telecoms, government agencies.
  • WS-Security message-level signing. Non-HTTPS security requirements.
  • Existing WSDL tooling investment. You have it, don’t rewrite it.

Do not start a new SOAP project in 2026 unless you have a specific B2B integration reason. For greenfield work, use REST + OpenAPI or gRPC.

Translation Heuristics

  1. From SOAP to REST: each SOAP operation becomes an HTTP endpoint. Complex types become JSON objects. WS-Security becomes OAuth 2.0 + JWT. UDDI registration becomes a service registry or DNS. Expect to reduce contract size by 5–10×.
  2. From REST to gRPC: the HTTP verbs become RPC methods. Path parameters become request message fields. Query parameters become optional fields. Response JSON becomes the response message. Expect wire-size reduction of 3–5× and latency improvement of 20–50%.
  3. From request-response to event-driven: flip the direction. Instead of “client asks for user,” the user service emits user.updated events, and consumers maintain their own views. This is the biggest mental shift — you are trading synchronous consistency for temporal decoupling.
  4. From synchronous to streaming: use gRPC streaming or Server-Sent Events for continuous updates. Use WebSockets for bidirectional browser communication.
  5. From monolith to service-oriented: start with the seams — the places where one module talks to another. Extract one service at a time, starting with the most stable boundary. Do not try to decompose the whole monolith at once.

When to Stay Monolithic

The 2023–2024 pendulum swing from microservices back to monoliths is not a rejection of service-oriented principles. It is a recognition that the operational cost of microservices is substantial and only pays off at a certain scale.

Stay monolithic when:

  • Your team is < 20 engineers. Conway’s Law hasn’t kicked in yet.
  • Your traffic is < 10,000 requests/second. One well-tuned server can handle it.
  • Your deployment cadence is daily or weekly. You don’t need per-service independence yet.
  • You don’t have dedicated platform engineering. Running Kubernetes + mesh + observability full-time is a job, not a side task.
  • The domain is not yet understood. Premature service boundaries are expensive to move.

DHH’s “majestic monolith” essay, Shopify’s explicit modular monolith architecture, and Amazon Prime Video’s 2023 migration back from microservices all make the same point: start monolithic, extract services when you hit a pain point. That pain point will come — but don’t pay for it before you need to.
