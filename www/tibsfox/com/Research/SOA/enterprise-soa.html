Enterprise SOA (SOAP Era)

SOAP envelopes, WSDL contracts, UDDI registries, WS-Security, BPEL, Enterprise Service Buses — the classic stack
Overview Principles History Enterprise SOA REST & APIs Microservices Implementations Rosetta Learn
The enterprise SOA era had the right ideas and the wrong implementations. Contract-first design, loose coupling, service boundaries, messaging patterns — the principles are still the foundation of modern distributed systems. The machinery — SOAP envelopes, WSDL documents, UDDI registries, WS-* specifications, Enterprise Service Buses — did not survive contact with the simpler JSON/REST alternatives that followed. But a lot of production code still runs on this stack in 2026, and understanding it is understanding how we got here.

1. XML-RPC — The Simpler Ancestor

In spring 1998, Dave Winer at UserLand Software corresponded with Bob Atkinson and Mohsen Al-Ghosein at Microsoft about designing a simple, text-based, HTTP-transported RPC. The result was XML-RPC, published publicly on June 15, 1998.

<?xml version="1.0"?>
<methodCall>
  <methodName>examples.getStateName</methodName>
  <params>
    <param><value><i4>41</i4></value></param>
  </params>
</methodCall>

Six data types (int, double, boolean, string, datetime, base64), two containers (array, struct). The whole specification fit on a single web page. Winer insisted on keeping it minimal and resisted Microsoft’s desire to add type systems and namespaces. XML-RPC survives in niches — WordPress XML-RPC API, Trackbacks, some early APIs — but was eclipsed by SOAP within a few years. The philosophical principle lives on: the simpler ancestor of every protocol family is usually the one that actually gets used.

2. SOAP — Simple Object Access Protocol

SOAP 1.1 was published as a W3C Note on May 8, 2000, co-authored by Don Box (DevelopMentor), David Ehnebuske (IBM), Gopal Kakivaya (Microsoft), Andrew Layman (Microsoft), Noah Mendelsohn (Lotus), Henrik Frystyk Nielsen (Microsoft), Satish Thatte (Microsoft), and Dave Winer. The coalition was Microsoft + IBM + Lotus + UserLand + DevelopMentor.

The SOAP envelope structure

<?xml version="1.0"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>
    <!-- optional headers for routing, security, reliability, transaction -->
    <wsa:Action>http://example.com/GetStockPrice</wsa:Action>
    <wsse:Security>
      <wsse:UsernameToken>
        <wsse:Username>alice</wsse:Username>
      </wsse:UsernameToken>
    </wsse:Security>
  </soap:Header>
  <soap:Body>
    <m:GetStockPrice xmlns:m="http://example.com/stock">
      <m:Symbol>IBM</m:Symbol>
    </m:GetStockPrice>
  </soap:Body>
</soap:Envelope>

The envelope/header/body pattern was deliberately borrowed from email (which has envelope metadata, headers, and a body). SOAP messages could be routed through intermediaries, each processing headers relevant to them and passing the body through unchanged.

SOAP encoding styles

Four styles, with the last being the default by 2005:

  • RPC/encoded — the original, used SOAP Section 5 encoding rules. Not interoperable.
  • RPC/literal — RPC model with XML Schema types.
  • Document/literal — a document carrying typed XML.
  • Document/literal wrapped — the dominant style by 2005; wraps each operation in an element named for the operation.

SOAP ran over HTTP, SMTP, JMS, and MQ. SOAP 1.2 (W3C Rec, June 2003) officially dropped the acronym expansion — “SOAP” became just a name.

Don Box and the Young Person’s Guide

In March 2000, Don Box wrote A Young Person’s Guide to the Simple Object Access Protocol for MSDN Magazine. Box had the rare gift of writing about enterprise middleware without sounding like a committee press release. The essay introduced thousands of developers to SOAP. Box later joined Microsoft as an architect in May 2002. His book Essential XML (2000) and his columns cemented his reputation as SOAP’s public face.

3. WSDL — Web Services Description Language

Submitted to W3C in September 2000 as a joint IBM/Microsoft/Ariba proposal. Lead authors: Erik Christensen (Microsoft), Francisco Curbera (IBM), Greg Meredith (Microsoft), Sanjiva Weerawarana (IBM, later WSO2). WSDL 1.1 became a W3C Note in March 2001 — that’s the version that actually shipped into production. WSDL 2.0 (W3C Rec, 2007) is technically superior but almost nobody adopted it.

The six elements

  1. types — XML Schema type definitions
  2. message — named groupings of typed parts
  3. portType — abstract interface (operations with input/output messages)
  4. binding — concrete protocol bindings (usually SOAP over HTTP)
  5. port — a specific endpoint for a binding
  6. service — a collection of ports
<wsdl:definitions name="StockQuote"
    targetNamespace="http://example.com/stock.wsdl"
    xmlns:xsd="http://www.w3.org/2001/XMLSchema">

  <wsdl:types>
    <xsd:schema targetNamespace="http://example.com/stock.xsd">
      <xsd:element name="TradePriceRequest">
        <xsd:complexType>
          <xsd:sequence>
            <xsd:element name="tickerSymbol" type="xsd:string"/>
          </xsd:sequence>
        </xsd:complexType>
      </xsd:element>
    </xsd:schema>
  </wsdl:types>

  <wsdl:message name="GetLastTradePriceInput">
    <wsdl:part name="body" element="TradePriceRequest"/>
  </wsdl:message>

  <wsdl:portType name="StockQuotePortType">
    <wsdl:operation name="GetLastTradePrice">
      <wsdl:input message="GetLastTradePriceInput"/>
      <wsdl:output message="GetLastTradePriceOutput"/>
    </wsdl:operation>
  </wsdl:portType>

  <wsdl:binding name="StockQuoteSoapBinding" type="StockQuotePortType">
    <soap:binding style="document"
      transport="http://schemas.xmlsoap.org/soap/http"/>
    <!-- operation bindings omitted -->
  </wsdl:binding>

  <wsdl:service name="StockQuoteService">
    <wsdl:port name="StockQuotePort" binding="StockQuoteSoapBinding">
      <soap:address location="http://example.com/stockquote"/>
    </wsdl:port>
  </wsdl:service>

</wsdl:definitions>

A nontrivial WSDL file was 500–2000 lines of dense XML. Reading one by hand was possible but unpleasant. Developers used tools: wsdl2java (Apache Axis), wsimport (JAX-WS), svcutil.exe (.NET). The workflow was WSDL-first → generate stubs → write client code that called the stubs. Conceptually identical to the CORBA IDL workflow.

4. UDDI — The Yellow Pages That Nobody Used

UDDI (Universal Description, Discovery, and Integration) was announced September 2000 by IBM, Microsoft, and Ariba. The vision: every company in the world would publish its Web Services into a federated UDDI directory. A developer writing a new application could query UDDI to find, say, “shipping cost calculator services from companies with headquarters in Ohio with an A+ rating,” retrieve the WSDL, and generate a client in minutes.

UDDI 1.0 (2000), 2.0 (2002), 3.0 (OASIS ratified February 2005). IBM, Microsoft, and SAP ran the public UDDI Business Registry.

It did not work. Not at a technical level (UDDI the protocol worked fine) but at a social level. Companies did not want to publish their service catalogs to a public directory. The trust model was wrong. The business model was wrong. Most enterprises found UDDI over-engineered compared to a wiki or a shared spreadsheet.

In January 2006, IBM, Microsoft, and SAP simultaneously announced they were shutting down their public UDDI Business Registry nodes. The Yellow Pages had no listings, and nobody had ever really used them. The lesson: a registry that nobody writes to is not a discovery mechanism.

5. The WS-* Swamp

Once SOAP, WSDL, and UDDI were in place, the industry embarked on the WS-* explosion — an ever-growing set of specifications, all with “WS-” prefixes, each adding some capability on top of base SOAP.

By 2007 there were over sixty WS-* specifications. The set became known informally as the WS-* swamp. The problem was not any single spec — most addressed real problems competently. The problem was aggregate complexity. A production deployment using WS-Security + WS-Addressing + WS-ReliableMessaging + WS-AtomicTransaction + WS-Policy required engineers to understand each spec, the specific vendor interpretations, and the interactions between them. The learning curve was measured in person-years.

6. SOAP Toolkits and Vendors

Every language ecosystem had its own SOAP toolkit:

  • Apache Axis (1.x, 2003; Axis2, 2006) — the Java reference implementation
  • Apache CXF (2006, from XFire + Celtix) — the successor to Axis for Java
  • Microsoft .NET ASMX (2002) — later replaced by WCF (Windows Communication Foundation, 2006)
  • IBM WebSphere — the enterprise Java SOAP stack
  • BEA WebLogic — ditto
  • Sun/Oracle Metro (JAX-WS + WSIT)
  • Spring-WS — the lighter Spring alternative
  • gSOAP for C/C++ — still widely used in embedded/industrial
  • Perl SOAP::Lite — the infamous Perl binding
  • Python ZSI, suds, Zeep — successive attempts at Python SOAP

Why every language ecosystem had its own SOAP toolkit and they were all painful: the WS-* specs left interpretation gaps, each vendor filled them differently, and cross-stack interop bugs were endemic.

7. Enterprise Service Buses

The term “ESB” was coined (or popularized) by Gartner analyst Roy Schulte in a 2002 research note. The concept: a middleware backbone handling routing, protocol translation, transformation, orchestration, and security for all Web Services in an enterprise. Instead of each service talking directly to every other service (creating an N-squared problem), all services would talk through the ESB, which would mediate everything.

The ESB promise vs the ESB reality

Promise: location transparency, format transformation, protocol bridging, routing, orchestration, monitoring, governance — all in one product, graphically modeled, enterprise-wide.

Reality: vendor lock-in, single point of failure, ops complexity, performance bottleneck, 18–36 month projects costing millions. The ESB team became a political choke point for every cross-team integration. Schema changes required coordination meetings. New services required approval. The promised reuse rarely materialized because the ESB team was optimizing for consistency at the cost of developer velocity.

This was the genesis of what would later become a major cultural force in microservices: the rejection of the central integration layer in favor of smart endpoints and dumb pipes. That phrase, coined in the Fowler/Lewis 2014 microservices article, was a direct repudiation of the ESB architecture.

8. Apache Camel and Enterprise Integration Patterns

Gregor Hohpe and Bobby Woolf, Enterprise Integration Patterns (Addison-Wesley, 2003). 65 patterns that named and systematized the ways systems exchange messages. Still the canonical reference twenty-three years later.

The patterns: Message Channel, Point-to-Point Channel, Publish-Subscribe Channel, Message Router, Content-Based Router, Message Filter, Dynamic Router, Recipient List, Splitter, Aggregator, Resequencer, Composed Message Processor, Scatter-Gather, Routing Slip, Process Manager, Message Broker, Message Translator, Envelope Wrapper, Content Enricher, Content Filter, Claim Check, Normalizer, Canonical Data Model, and many more.

Apache Camel (2007) was the library that implemented EIP as a Java fluent DSL:

from("file:/data/inbox")
    .choice()
        .when(xpath("/order/total > 1000"))
            .to("jms:queue:high-value")
        .otherwise()
            .to("jms:queue:normal")
    .end();

Camel had 300+ components (JMS, HTTP, FTP, SFTP, Kafka, S3, database, mail, etc.) and used endpoint URIs as the routing unit. Why Camel outlived most ESBs: it was a library, not a product. You could embed it in your own application rather than deploying it as a central bottleneck. Camel K (2019) brought it to Kubernetes as a cloud-native integration platform. The EIP patterns themselves remain the canonical vocabulary — every Kafka topic, every Kubernetes service, every Istio VirtualService is an EIP pattern with a new name.

9. BPEL — Business Process Execution Language

WS-BPEL 2.0 (OASIS, April 2007, succeeding BPEL4WS 1.1 from 2003). Authors: IBM, Microsoft, BEA, SAP, Siebel. A BPEL document describes a long-running business process as a composition of Web Service calls.

Construct catalog

  • receive — wait for an incoming message
  • reply — respond to a receive
  • invoke — call another Web Service
  • assign — copy/manipulate variables
  • pick — wait for one of several messages (like a select)
  • while, flow, sequence, scope
  • fault handlers and compensation handlers for error recovery

BPEL engines

Every ESB vendor built one: IBM WebSphere Process Server, Oracle BPEL Process Manager, Microsoft BizTalk, SAP NetWeaver, Apache ODE (open source), ActiveBPEL, Sun Java CAPS.

BPEL was the most hyped language in enterprise computing around 2007–2008. Its problems: verbose XML, unreadable graphical tools, source-control merge hell, abstraction-level mismatch (business analysts could not actually write BPEL). By 2015 it was effectively dead as a new-deployment technology. Its spiritual successors are Camunda (BPMN 2.0 engine from Jakob Freund and Bernd Rücker, founded 2008), Temporal, Cadence, Airflow, and Argo Workflows — workflow engines that use code or YAML rather than XML and focus on specific problem domains.

10. SOA Governance — The Industrial Complex

Somewhere around 2005, the SOA conversation shifted from “how do we build services” to “how do we govern services.” An entire product category emerged to sell governance tooling:

  • Systinet (acquired by Mercury 2006, then HP 2007) — UDDI-based service registry
  • SOA Software / Akana (acquired by Rogue Wave 2016, Perforce 2019) — Policy Manager
  • AmberPoint (acquired by Oracle 2010) — runtime monitoring
  • Progress Actional — runtime SOA management
  • Software AG CentraSite — OSGi-based registry
  • IBM WSRR (WebSphere Service Registry and Repository)
  • Oracle Enterprise Repository

Governance boards met weekly. New services required design reviews. Schema changes required backward-compatibility checks. The cultural effect: SOA became associated, for developers, with red tape. “Governance” became a trigger word for developer resistance. When microservices arrived in 2014 with a you build it, you run it ethos and no central governance function, part of the appeal was the explicit rejection of the SOA governance era.

11. Why Enterprise SOA Failed

The definitive obituary was Anne Thomas Manes’s January 5, 2009 Burton Group blog post SOA is Dead; Long Live Services. See the history page for the full quote and reaction. The failure modes, catalogued in many post-mortems:

  1. Complexity overwhelm — WS-* stack too large
  2. Vendor lock-in via the ESB
  3. The reuse gap — most services were used by exactly one consumer
  4. WS-* specs that nobody implemented completely
  5. Integration projects that took years
  6. The political dimension — governance boards as power tools
  7. Canonical data models nobody agreed on
  8. Tool/vendor lifecycle mismatches — Systinet acquired and deprecated, Sonic merged and renamed, ActiveBPEL abandoned, all mid-project

12. What Enterprise SOA Got Right

The principles were never wrong. The implementations were wrong.

  • Contract-first design is still the right way. Modern equivalents: OpenAPI, Protobuf, AsyncAPI, GraphQL SDL. Same idea, less ceremony.
  • Loose coupling is still the fundamental virtue. Modern equivalents: tolerant readers, schema evolution rules, async messaging.
  • Messaging patterns (EIP) are still relevant. Every Kafka topic is a Pub-Sub Channel. Every consumer group is a Competing Consumers pattern. Every dead-letter queue is a Dead Letter Channel.
  • Service boundaries are still how you decompose systems. Modern equivalents: bounded contexts, domain services, team topology.

Every microservices book of the 2010s is, in a sense, a rediscovery of what the SOA community had already figured out by 2005. What changed was not the ideas but the tooling: Docker and Kubernetes replaced the ESB as the deployment substrate; REST + JSON + OpenAPI replaced SOAP + XML + WSDL as the contract language; distributed tracing and metrics replaced the governance dashboard; DevOps culture replaced the governance board.

13. Legacy Enterprise SOA Today (2026)

  • SAP, Oracle, major banks, insurance, healthcare still run SOAP stacks in production.
  • WS-Security still used in B2B integrations (EDI-style partner connections).
  • ESBs still run at large enterprises — not new deployments, but the existing installations are stable and cheaper to maintain than replace.
  • The “SOA facade” pattern: modern microservices in front of legacy SOAP backends, providing a REST/JSON interface while the internal service-to-service calls remain SOAP.
  • Hiring for SOAP/WSDL/WS-Security expertise is still happening in 2026. The work pays well because nobody new wants to learn it.
  • New SOAP projects essentially don’t exist except where B2B integration with a legacy partner forces it.

If you find yourself doing enterprise SOA work in 2026, it is because you are maintaining or extending something that was built during the 2005–2012 hype era. The principles live on; the specific machinery is legacy.
