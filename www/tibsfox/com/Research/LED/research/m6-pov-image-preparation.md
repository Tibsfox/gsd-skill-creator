# POV Image Preparation & Mechanical Assembly

A persistence of vision display is only as good as the image data it renders and the mechanical platform that spins it. This page covers the complete pipeline from rectangular source image to polar-coordinate bitmap stored in flash memory, plus the mechanical assembly -- motor selection, balancing, enclosure, and safety -- that makes the display physically viable.

> **SAFETY WARNING:** A spinning POV arm at 1800+ RPM is physically dangerous. Always fully enclose the assembly in a polycarbonate shield before powering the motor. Verify all components are mechanically secured and balanced. Never reach toward a spinning assembly.

---

## 1. Image Pipeline Overview

Converting a standard rectangular image into POV display data requires a multi-stage pipeline. Each stage reduces or transforms the data to fit the constraints of a spinning LED array.

```
  Source Image (PNG/BMP)
       |
       v
  [1] Resize to target resolution
       |
       v
  [2] Rectangular-to-Polar coordinate transform
       |
       v
  [3] Color depth reduction (24-bit -> APA102 format)
       |
       v
  [4] Gamma correction for perceived linearity
       |
       v
  [5] Serialize to byte array (C header or binary blob)
       |
       v
  [6] Store in RP2040 flash (2MB+)
       |
       v
  [7] Read by PIO state machine during rotation
```

## 2. Rectangular to Polar Coordinate Transform

### 2.1 The Core Problem

A POV display does not scan pixels left-to-right, top-to-bottom like an LCD. It scans radially outward from the hub at each angular position. The display is inherently a polar coordinate system:

```
  Rectangular image (x, y):        Polar display (r, theta):

  +------------------+              .  .  .  . . . .  .  .  .
  |                  |            .    .   . . . . .   .    .
  |   (x, y)        |          .   .   .  . . . .  .   .   .
  |     *            |         .  .  LED[0]        .  .  .
  |                  |        . .     LED[1]         . .
  |                  |        .       LED[2]  ...       .
  +------------------+        .       LED[n]           .
                               .  .  . . . . . . .  .  .
                                  at angle theta
```

### 2.2 The Transform Algorithm

For each angular position (theta) and each radial position (r = LED index), sample the source image at the corresponding (x, y) coordinate:

```
Given:
  image_width, image_height  = source image dimensions
  num_leds                   = radial resolution (LED count on arm)
  num_angles                 = angular resolution (columns per revolution)
  center_x = image_width / 2
  center_y = image_height / 2
  max_radius = min(center_x, center_y)

For each angle a in [0, num_angles):
  theta = a * 2 * pi / num_angles

  For each led r in [0, num_leds):
    radius = (r / num_leds) * max_radius

    x = center_x + radius * cos(theta)
    y = center_y + radius * sin(theta)

    polar_image[a][r] = bilinear_sample(source_image, x, y)
```

### 2.3 Python Implementation

```python
import numpy as np
from PIL import Image

def rect_to_polar(src_path, num_leds=150, num_angles=180):
    """Convert rectangular image to polar coordinate array for POV display."""
    img = np.array(Image.open(src_path).convert('RGB'))
    h, w, _ = img.shape
    cx, cy = w / 2.0, h / 2.0
    max_r = min(cx, cy)

    # Output: angles x leds x 3 (RGB)
    polar = np.zeros((num_angles, num_leds, 3), dtype=np.uint8)

    for a in range(num_angles):
        theta = a * 2.0 * np.pi / num_angles
        cos_t = np.cos(theta)
        sin_t = np.sin(theta)

        for r in range(num_leds):
            radius = (r / num_leds) * max_r
            x = cx + radius * cos_t
            y = cy + radius * sin_t

            # Nearest-neighbor sampling (fast)
            ix = int(np.clip(x, 0, w - 1))
            iy = int(np.clip(y, 0, h - 1))
            polar[a, r] = img[iy, ix]

    return polar


def polar_to_c_header(polar, output_path):
    """Write polar image data as C header file for RP2040."""
    num_angles, num_leds, _ = polar.shape

    with open(output_path, 'w') as f:
        f.write(f"// POV image: {num_angles} angles x {num_leds} LEDs\n")
        f.write(f"// Generated by pov_image_prep.py\n\n")
        f.write(f"#define POV_NUM_ANGLES {num_angles}\n")
        f.write(f"#define POV_NUM_LEDS {num_leds}\n\n")
        f.write("const uint8_t pov_image[POV_NUM_ANGLES][POV_NUM_LEDS][3] = {\n")

        for a in range(num_angles):
            f.write("  { // angle %d\n" % a)
            for r in range(num_leds):
                rgb = polar[a, r]
                f.write("    {%3d, %3d, %3d}" % (rgb[0], rgb[1], rgb[2]))
                if r < num_leds - 1:
                    f.write(",")
                f.write("\n")
            f.write("  }")
            if a < num_angles - 1:
                f.write(",")
            f.write("\n")

        f.write("};\n")
```

### 2.4 Bilinear Interpolation for Quality

Nearest-neighbor sampling (shown above) is fast but produces aliasing artifacts -- jagged edges where the polar grid does not align with source pixels. Bilinear interpolation smooths these artifacts by blending the four nearest source pixels:

```
Bilinear sample at (x, y):

  Given fractional coordinates (x=3.7, y=2.3):

    (3,2)-----(4,2)
      |   *     |        * = sample point (3.7, 2.3)
      |         |
    (3,3)-----(4,3)

  fx = x - floor(x) = 0.7
  fy = y - floor(y) = 0.3

  result = (1-fx)(1-fy) * pixel(3,2)    0.3 * 0.7 = 0.21
         + (fx)(1-fy)   * pixel(4,2)    0.7 * 0.7 = 0.49
         + (1-fx)(fy)   * pixel(3,3)    0.3 * 0.3 = 0.09
         + (fx)(fy)     * pixel(4,3)    0.7 * 0.3 = 0.21
                                                     ----
                                                     1.00
```

## 3. Image Resolution vs LED Count

### 3.1 Resolution Matching

The source image should be at least as large as the POV display's effective resolution. An image smaller than the display will be upscaled, causing blurriness with no benefit.

```
POV display effective resolution:

  Radial:   num_leds = 150
  Angular:  num_angles = 180
  Effective pixels: 150 x 180 = 27,000

  Minimum source image: 300 x 300 pixels (inscribed circle = 150px radius)
  Recommended source:   512 x 512 or larger (allows downsampling quality)
```

### 3.2 Center Pixel Density Problem

Near the rotation hub (small radius), many angular positions map to the same source pixel. At the hub center (r=0), all angles sample the exact same pixel. This creates a visual "bullseye" artifact.

Mitigation strategies:
- Start LED indexing at a physical offset from the hub (e.g., first LED is 20mm from center)
- Reduce angular resolution near the hub (fewer angles for inner LEDs)
- Use the first 5-10 LEDs as a static color (hub accent light)

### 3.3 Outer Edge Resolution

At the tip of the arm (maximum radius), the arc distance between adjacent angular positions is largest:

```
Arc distance at tip = 2 * pi * r_max / num_angles

At r_max = 150mm, 180 angles:
  Arc = 2 * pi * 150mm / 180 = 5.24mm per angular pixel

At r_max = 150mm, 360 angles:
  Arc = 2 * pi * 150mm / 360 = 2.62mm per angular pixel

For comparison, a typical APA102 LED pitch on a strip is 16.7mm (60/m).
The angular resolution at the tip exceeds the radial LED pitch,
so angular resolution is NOT the bottleneck — radial LED count is.
```

## 4. Color Depth and APA102 Format

### 4.1 APA102 Color Frame Structure

Each APA102 LED accepts a 32-bit data frame with a 5-bit global brightness field and three 8-bit color channels:

```
  APA102 LED frame (32 bits):

  [111][GGGGG] [BBBBBBBB] [GGGGGGGG] [RRRRRRRR]
   ^     ^         ^           ^           ^
   |     |         |           |           |
   |     |         Blue        Green       Red
   |     |
   |     Global brightness (0-31)
   |
   Start bits (always 111)

  Effective color depth:
    5-bit global x 8-bit per channel = 13-bit effective per channel
    Total: 5 + 8 + 8 + 8 = 29 bits of color information
    (compared to 24-bit for WS2812B: 8+8+8)
```

### 4.2 Converting 24-bit RGB to APA102 Format

```python
def rgb_to_apa102(r, g, b, global_brightness=31):
    """Convert 8-bit RGB to APA102 32-bit frame.

    Args:
        r, g, b: 0-255 color values
        global_brightness: 0-31 (default max)

    Returns:
        4 bytes: [0xE0|brightness, blue, green, red]
    """
    return bytes([
        0xE0 | (global_brightness & 0x1F),  # 111 + 5-bit brightness
        b & 0xFF,                             # Blue
        g & 0xFF,                             # Green
        r & 0xFF                              # Red
    ])
```

### 4.3 Gamma Correction

LED brightness is linear with current, but human brightness perception is logarithmic. A value of 128 (50% of 255) does not look "half as bright" as 255 -- it looks about 75% as bright. Gamma correction compensates for this perceptual nonlinearity:

```
Gamma-corrected value = 255 * (input / 255) ^ gamma

Standard gamma = 2.2 (matches sRGB displays)

Lookup table (8-bit input -> 8-bit output, gamma 2.2):
  Input:    0    1    2   16   32   64  128  192  255
  Output:   0    0    0    1    3   13   54  127  255

Python generation:
  gamma_table = [int(255 * (i / 255.0) ** 2.2 + 0.5) for i in range(256)]
```

For POV displays, gamma correction is essential because the already-reduced perceived brightness (due to duty cycle) makes nonlinearity more visible. See [APA102 POV Design](m6-apa102-pov-design.md) for how the global brightness register interacts with gamma correction.

## 5. Bitmap Storage in Flash Memory

### 5.1 RP2040 Flash Capacity

The RP2040 (Raspberry Pi Pico) has 2MB of flash memory by default, with some boards offering 4MB or 16MB. The flash stores both the program code and image data.

```
Memory budget for a typical POV program:

  RP2040 flash:           2,097,152 bytes (2MB)
  Program code + libs:    ~200,000 bytes (200KB)
  Available for images:   ~1,897,152 bytes (~1.8MB)

Image size calculation:
  180 angles x 150 LEDs x 3 bytes (RGB) = 81,000 bytes (79KB)
  360 angles x 150 LEDs x 3 bytes (RGB) = 162,000 bytes (158KB)

  At 79KB per image, you can store ~24 images in 1.8MB
  At 158KB per image, you can store ~12 images in 1.8MB
```

### 5.2 Storage Formats

**C header array (compile-time):** The image is compiled directly into the firmware binary. Simplest approach, no filesystem needed. Change images by recompiling.

**Binary blob (runtime):** The image is stored as a raw binary file in a reserved flash region. A simple offset table at a known address maps image names to flash addresses. Allows updating images without recompiling code.

**LittleFS filesystem:** The RP2040 can mount a LittleFS filesystem on a portion of flash. Images are stored as files and read at runtime. Supports USB mass storage mode for drag-and-drop image upload.

```
Flash layout with LittleFS:

  0x10000000  +------------------+
              |  Program code    |  (~200KB)
              +------------------+
  0x10032000  |  LittleFS        |  (~1.8MB)
              |  /image1.bin     |
              |  /image2.bin     |
              |  /image3.bin     |
              |  ...             |
              +------------------+
  0x10200000  (end of 2MB flash)
```

### 5.3 Compression

For storing many images, simple run-length encoding (RLE) can reduce size significantly for images with large areas of solid color:

```
RLE encoding (per-angle column):
  [count] [R] [G] [B]   -- repeat this color 'count' times

  Uncompressed column (150 LEDs): 450 bytes
  RLE column (typical logo image): 60-150 bytes (60-70% compression)

  Not useful for photographic images (poor RLE compression).
  For photos, store fewer images at full resolution instead.
```

## 6. Motor Selection

### 6.1 Requirements

The motor must spin the POV arm at a stable, controllable speed. Key requirements:

| Parameter | Requirement | Why |
|-----------|-------------|-----|
| Speed range | 1200-3600 RPM | 1200 RPM minimum for POV fusion, 3600 for bright environments |
| Speed stability | <2% variation | Image jitter is visible above 2% speed variation |
| Torque | >0.05 Nm | Must overcome air drag and bearing friction with loaded arm |
| Vibration | Minimal | Imbalance amplifies at high RPM |
| Durability | >1000 hours | Continuous operation without maintenance |

### 6.2 Brushless DC Motors (BLDC)

Brushless DC motors are the standard choice for POV displays. They have no brushes to wear out, generate less electrical noise, and offer precise speed control via electronic speed controllers (ESCs).

```
BLDC motor specifications for POV:

  KV rating:  Motor speed constant (RPM per volt)
              A 1000KV motor at 12V: 12,000 RPM (no load)
              Under load: ~70% of no-load speed = 8,400 RPM
              This is too fast — either use lower voltage or
              select a lower KV motor.

  Recommended: 200-500 KV motor at 12V
    200KV x 12V = 2,400 RPM no-load, ~1,700 RPM loaded
    500KV x 12V = 6,000 RPM no-load, ~4,200 RPM loaded

  Suitable motors:
    - RC gimbal motors (low KV, designed for smooth rotation)
    - Small outrunner motors (2204, 2206 size class)
    - Pancake/flat BLDC motors for low-profile designs
```

### 6.3 ESC Speed Control

An ESC (Electronic Speed Controller) drives the motor's three phases in sequence. Most RC-grade ESCs accept a standard PWM servo signal (1000-2000us pulse at 50Hz) for speed control:

```
  RP2040 GPIO ---[PWM 50Hz]---> ESC ---[3-phase AC]---> BLDC Motor

  PWM signal:
    1000us pulse = minimum speed (or motor off, ESC-dependent)
    1500us pulse = ~50% speed
    2000us pulse = maximum speed

  The RP2040 can generate this PWM signal on any GPIO pin.
  See RP2040 PIO programming in the cross-references below.
```

### 6.4 Closed-Loop Speed Control

For stable image display, the motor speed must be regulated. The Hall effect sensor that provides the rotation reference (one pulse per revolution) also provides RPM measurement:

```
Measured RPM = 60 / (time between Hall pulses in seconds)

PID control loop (runs once per revolution):
  error = target_RPM - measured_RPM
  integral += error * dt
  derivative = (error - prev_error) / dt

  pwm_adjust = Kp * error + Ki * integral + Kd * derivative
  esc_pwm = base_pwm + pwm_adjust

  Typical PID gains (tuned experimentally):
    Kp = 0.1, Ki = 0.05, Kd = 0.01
```

## 7. Mechanical Assembly

### 7.1 Hub Design

The hub is the central mount point where the motor shaft connects to the POV arm. It must be rigid, concentric, and allow passage of power/data if using slip rings.

```
  Side view:
                    Arm (PCB or LED strip on carrier)
  ==================|===========================
                    |
              +-----+-----+
              |   Hub      |   <- Aluminum or 3D-printed
              |   Clamp    |      Bore matches motor shaft
              +-----+-----+
                    |
              +-----+-----+
              |   Motor    |   <- BLDC motor
              |   Shaft    |
              +-----+-----+
                    |
              +-----+-----+
              |   Base     |   <- Weighted base plate
              +-----------+
```

### 7.2 Counterweight and Balancing

An unbalanced arm generates vibration force proportional to imbalance mass times the square of angular velocity. At 1800 RPM, even 1 gram of imbalance at 150mm radius produces:

```
F = m * omega^2 * r
  = 0.001 kg * (188.5 rad/s)^2 * 0.15m
  = 5.33 N (about 0.54 kg-force)

This force oscillates at 30 Hz (rotation frequency), causing
the entire assembly to vibrate at 30 Hz — clearly audible
and mechanically destructive over time.
```

**Balancing procedure:**

1. Mount a counterweight arm opposite the LED arm (180 degrees)
2. Start the motor at low speed (300-500 RPM)
3. Feel for vibration direction and amplitude
4. Add/remove small weights (washers, nuts, putty) to the counterweight arm
5. Increase speed gradually, re-checking balance at each step
6. At target RPM, vibration should be barely perceptible

**Symmetric two-arm design:** Use two identical LED arms at 180 degrees. Both display the same image data. This naturally balances the assembly and doubles the refresh rate (each arm paints the full image, so effective refresh = 2 x rotation rate). The cost is doubling the LED count and data bandwidth.

### 7.3 Bearing Selection

The motor's built-in bearings handle the rotational loads. For added stability, an outboard bearing at the base of the arm carrier can reduce wobble:

| Bearing Type | RPM Rating | Notes |
|-------------|-----------|-------|
| Motor stock (ball bearing) | 10,000+ | Usually adequate for POV speeds |
| 608ZZ skate bearing | 30,000+ | Cheap, widely available, fits 8mm shafts |
| Flanged ball bearing | 20,000+ | Provides axial constraint |

### 7.4 Power Transfer to Spinning Arm

The spinning arm needs power (and sometimes data) from the stationary base. Options:

| Method | Complexity | Reliability | Noise |
|--------|-----------|------------|-------|
| Battery on arm | Low | High | Zero |
| Slip rings | Medium | Medium (wear) | Some |
| Wireless power (Qi) | High | High | Low |
| Capacitive coupling | Medium | High | Low |

**Battery on arm** is simplest: a small LiPo cell (3.7V, 500mAh) mounted at the hub powers the LEDs and controller. Runtime at 150 LEDs x 20mA average = 3A from 5V: about 10 minutes. Sufficient for displays, recharge between sessions.

**Slip rings** use spring-loaded brushes on concentric rings. A 4-conductor slip ring (V+, GND, Data, Clock) passes SPI signals to the spinning arm. Slip ring contacts introduce noise -- use twisted pair and differential signaling for data integrity at high SPI clock rates.

## 8. Enclosure Design

### 8.1 Safety Enclosure Requirements

A POV display enclosure must:
- Contain any component that detaches from the spinning arm
- Be transparent enough to see the display clearly
- Allow airflow for motor and LED cooling
- Provide vibration isolation from the mounting surface

### 8.2 Polycarbonate Shield

Polycarbonate (Lexan) is the standard material for POV enclosures. It is nearly shatter-proof and optically clear.

```
  Top view (looking down at POV):

  +-----------------------------------+
  |                                   |
  |     Polycarbonate cylinder        |
  |                                   |
  |       +-------------------+       |
  |       |                   |       |
  |       |   LED arm sweep   |       |
  |       |    (360 degree)   |       |
  |       |                   |       |
  |       +-------------------+       |
  |                                   |
  +-----------------------------------+

  Minimum clearance: 30mm beyond arm tip
  (catches detached components without direct impact)

  Cylinder dimensions for 150mm arm:
    Inner diameter: 360mm (150mm arm + 30mm clearance each side)
    Height: 80-100mm (covers arm profile + motor)
    Wall thickness: 3mm minimum (withstands component impact)
```

### 8.3 Base Weight

The base must be heavy enough that motor torque and vibration do not cause the assembly to "walk" across the surface. A weighted steel base plate (2-5 kg) with rubber feet provides stability.

```
  Required base mass (rule of thumb):
    Base mass > 10 x arm mass

    Arm mass (LED strip + PCB + wiring): ~50g
    Minimum base mass: 500g
    Recommended: 2-3 kg (includes motor and enclosure weight)

  Rubber isolation feet reduce vibration transfer to the surface
  and prevent the assembly from sliding.
```

## 9. Assembly Safety Checklist

Before powering the motor for the first time, verify each item:

| # | Check | Method |
|---|-------|--------|
| 1 | All solder joints on arm inspected | Visual inspection under magnification |
| 2 | LED strip mechanically secured full length | Adhesive + cable ties or clamp |
| 3 | No loose wires or components on arm | Tug test on each connection |
| 4 | Counterweight installed and preliminary balanced | Manual rotation, check for heavy side |
| 5 | Polycarbonate enclosure installed | Enclosure fully seated, no gaps |
| 6 | Motor shaft hub clamp tightened | Hex wrench to specification |
| 7 | Base weighted and stable | Push test on all sides |
| 8 | ESC current limit set | Programmed via ESC programming card |
| 9 | Kill switch accessible | Motor power disconnect within arm's reach |
| 10 | First test at LOW speed | Start at 300 RPM, observe for 30 seconds |
