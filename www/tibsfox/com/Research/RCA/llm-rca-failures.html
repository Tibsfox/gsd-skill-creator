LLM Reasoning Failures in RCA

12 Pitfall Types, 1,675 Agent Runs, Architecture-Level vs Model-Level Failures
Overview Classical Systems SRE AI Agents Cloud LLM Failures Healthcare Manufacturing Our Arch Math
The most important finding in AI-driven RCA is not how to make it work — it is understanding why it systematically fails. Kim et al.’s study of 1,675 agent runs across 5 models reveals that LLM reasoning failures in root cause analysis are architectural, not model-level. Swapping models does not fix the fundamental pitfalls.

The Study: 1,675 Agent Runs, 5 Models, 12 Pitfalls

Kim et al. conducted the largest systematic study of LLM-based root cause analysis, running 1,675 agent executions across 5 frontier models (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro, Llama 3.1 70B, Mistral Large) on real cloud incident data. The study identified 12 distinct pitfall types that cause AI agents to fail at RCA tasks.

The central finding: these pitfalls persist across all models. They are not quirks of a specific model but structural limitations of how LLMs approach investigation tasks. This means the solution lies in architecture and scaffolding, not in model upgrades.

Three Failure Categories

The 12 pitfalls cluster into three high-level categories of reasoning failure:

Stalled Reasoning

The agent gets stuck and cannot make progress toward identifying the root cause:

  • Exploration paralysis: Agent collects data but never converges on a hypothesis
  • Tool call loops: Repeating the same tool call expecting different results
  • Premature termination: Declaring a root cause without sufficient evidence
  • Scope fixation: Investigating one subsystem while ignoring relevant others

Biased Reasoning

The agent reaches a conclusion, but the conclusion is systematically skewed:

  • Confirmation bias: Seeking evidence that supports the first hypothesis while ignoring contradictory data
  • Recency bias: Attributing the root cause to the most recent change, even when evidence points elsewhere
  • Familiarity bias: Defaulting to common root causes from training data rather than investigating the specific incident
  • Authority anchoring: Over-weighting information from “authoritative” sources (documentation, runbooks) even when the data contradicts them

Confused Reasoning

The agent’s reasoning process becomes internally inconsistent:

  • Hallucinated data interpretation: Describing patterns in data that do not exist, or misreading metric values
  • Causal inversion: Confusing cause and effect (e.g., treating the symptom as the root cause)
  • Context window artifacts: Losing track of earlier findings as the context fills with tool outputs
  • Contradictory conclusions: Stating incompatible root causes within the same analysis

Architecture-Level, Not Model-Level

The study’s most consequential finding: when the same RCA task is given to different models with identical prompting and tooling, the same pitfalls appear at similar rates. This means:

  • Upgrading from GPT-4o to Claude 3.5 does not fix confirmation bias in RCA
  • Larger context windows reduce but do not eliminate context window artifacts
  • Better reasoning models (o1-style) reduce but do not eliminate hallucinated data interpretation
  • The fix must be architectural: structured investigation protocols, external verification, guardrails that prevent specific pitfall patterns

This finding aligns with AgentRx’s approach (see AI Agent Debugging): the solution is a pipeline with multiple stages, constraints, and evaluation — not a better single-call prompt.

FVDebug — Formal Verification for Debugging

FVDebug applies formal verification techniques to AI agent debugging. Rather than relying on LLM judgment (which suffers from the same pitfalls), FVDebug uses symbolic methods to verify whether an agent’s reasoning chain satisfies logical constraints:

  • Does the conclusion follow from the evidence?
  • Is the evidence consistent (no contradictions)?
  • Were all relevant data sources consulted?
  • Does the causal chain satisfy temporal ordering?

FVDebug represents a promising direction: using formal methods to verify LLM reasoning rather than replacing it.

Key Findings

  • Most prevalent pitfalls (hallucinated data interpretation + incomplete exploration) persist across ALL models regardless of capability tier
  • Prompt engineering alone cannot resolve the dominant pitfalls
  • Inter-agent communication protocol enrichment reduced communication-related failures by up to 15 percentage points
  • Architecture-level issues dominate over model-level issues — failures stem from system design, not individual model weaknesses

Counter-Intuitive Findings

  • Traces can hurt: Removing trace data improved accuracy by 6–7% — models were distracted by voluminous, noisy trace data rather than aided by it
  • Anchoring bias causes 15–45% accuracy reduction — the strongest negative association with correctness across all failure types
  • Simpler workflows often match complex ones: Straight-Shot (non-agentic) often equaled or surpassed ReAct and Plan-and-Execute for core localization tasks
  • Metrics are the most valuable signal: Withholding metrics reduced localization by 7–15% (most essential modality); log exclusion degraded fault-type accuracy by 13%
  • “No single model dominates across all RCA subtasks” — model selection should reflect target RCA objective rather than model size alone

Mitigation Recommendations from the Literature

  1. Early hypothesis diversification (tree-of-thought approaches)
  2. Self-consistency and critique mechanisms for belief revision
  3. Evidence sufficiency validation before ranking hypotheses
  4. Explicit domain guidance for diagnostic triage heuristics
  5. Better trace alert filtering and integration strategies
  6. Workflow selection matching model capacity
