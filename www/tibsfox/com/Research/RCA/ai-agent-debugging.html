When the system under investigation is itself an AI agent, RCA enters uncharted territory. Non-deterministic reasoning chains, hallucinated tool calls, and emergent multi-step failures demand fundamentally new approaches. Three research threads — AgentRx, OpenRCA, and AgentDebug — are defining this frontier.

AgentRx — Microsoft Research

AgentRx (Microsoft Research, 2026) introduces a 4-stage pipeline for automated root cause analysis of AI agent failures. It achieves +23.6% improvement in fault localization accuracy over baseline approaches.

The 4-stage pipeline:

  1. Trajectory Normalization: Convert diverse agent execution traces into a standardized format, handling variations in tool calling conventions, retry patterns, and output formatting across different agent frameworks
  2. Constraint Synthesis: Automatically derive correctness constraints from successful execution trajectories — what should have happened based on the task specification and historical successes
  3. Guarded Evaluation: Compare failed trajectories against synthesized constraints, identifying the specific step where divergence occurred while accounting for acceptable variation in non-deterministic execution
  4. LLM Judge: Use a separate LLM as a calibrated evaluator to classify the identified failure step, categorize the failure type, and generate an actionable diagnosis

AgentRx’s key insight: agent failures are best understood not by examining the final output, but by identifying the critical failure step — the specific point in the trajectory where the agent’s reasoning first went off-track.

AgentDebug — Corrective Feedback

AgentDebug takes a different approach: rather than post-hoc analysis, it provides corrective feedback during execution. By monitoring the agent’s trajectory in real-time and comparing it against expected behavior patterns, AgentDebug can intervene at the critical failure step before the error cascades.

Results: +24% accuracy improvement in task completion when AgentDebug’s corrective feedback is active. The improvement is largest for multi-step tasks (>10 tool calls) where early errors have the most opportunity to cascade.

Critical Failure Step Identification

All three approaches converge on a fundamental concept: the critical failure step (CFS). In any failed agent trajectory, there exists a specific step where:

  • The agent’s internal state (beliefs, plans) first diverged from correctness
  • All subsequent failures can be traced back to this divergence
  • Intervention at this step (and not later) would have prevented the failure

CFS identification is complicated by non-deterministic reasoning chains: the same prompt can produce different action sequences, making it impossible to define a single “correct” trajectory. AgentRx handles this through constraint synthesis from the distribution of successful trajectories, not from any single golden path.

The CFS concept maps directly to STAMP’s notion of unsafe control actions (see Systems Thinking) — the agent is a controller whose process model became inconsistent with reality at the CFS.

AgentRx — Full Pipeline Details

Deep extraction from the AgentRx paper (Barke et al., Microsoft Research, Feb 2026) and the MSR blog announcement (March 12, 2026) provides the complete operational picture of the 4-stage pipeline:

Stage 1: Trajectory Normalization

Heterogeneous logs from different agent systems (LangChain, AutoGen, CrewAI, custom frameworks) are converted into a common intermediate representation. This normalization step is what makes AgentRx domain-agnostic — the downstream stages operate on the unified format regardless of the originating agent system.

Stage 2: Constraint Synthesis

Constraints are automatically generated from two sources:

  • Tool schemas: API response validity requirements derived directly from tool definitions (e.g., “if the tool returns a list, the agent must iterate it rather than treating it as a scalar”)
  • Domain policies: User confirmation mandates, access restrictions, and business rules (e.g., “before executing a financial transaction, the agent must confirm with the user”)

The key distinction: constraint synthesis is automatic from schemas, not manual annotation. This makes it scalable across agent deployments without per-domain engineering.

Stage 3: Guarded Evaluation

Constraints are evaluated step-by-step against the trajectory. Each constraint is only checked when its guard condition applies (e.g., a “payment confirmation” constraint only fires when a payment tool is invoked). The output is an auditable validation log documenting every violation with evidence — creating an accountability chain that humans can review.

Stage 4: LLM-Based Judging

A language model judge analyzes the validation log together with the failure taxonomy to identify the Critical Failure Step (CFS) — the first unrecoverable error in the trajectory. This is the point where intervention would have prevented the cascade of downstream failures.

Grounded-Theory Taxonomy Derivation

The 9-category failure taxonomy was derived using grounded-theory methodology — built bottom-up from real failures rather than imposed top-down from theoretical categories. This is the same methodology that our skill-creator observation loop uses: observe real agent behavior, classify patterns, then formalize into reusable knowledge.
