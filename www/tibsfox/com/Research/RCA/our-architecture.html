This makes it both a practitioner and a subject of root cause analysis — we build RCA tools and we need them for our own failures. gsd-forensics Skill

The gsd-forensics skill is our primary RCA tool. Triggered after GSD workflow failures, it performs:

  • Git archaeology: Analyze commit history, diff chains, and branch topology to reconstruct the failure timeline
  • Artifact inspection: Read PLAN.md, STATE.md, ROADMAP.md to identify discrepancies between plan and execution
  • State reconstruction: Rebuild the system state at the point of failure from .planning/ artifacts
  • Causal chain construction: Five Whys-style iterative analysis through the evidence
  • Recommendation generation: Specific, actionable corrective measures

The forensics skill implements a blameless approach by design — it investigates systemic factors (missing verification steps, incomplete plans, ambiguous requirements) rather than attributing failure to specific agents.

incident-analyzer Agent

The incident-analyzer agent is the three-pillar observability component of our architecture:

  • Logs: Git log, agent execution logs, hook output — reconstructs the sequence of events
  • Metrics: Test pass/fail counts, execution time, file change volume — quantifies the impact
  • Traces: Agent call chains, skill activation sequences, subagent delegation paths — identifies where in the execution chain the failure originated

Correlation across these three sources enables identification of the critical failure step — the specific agent action that first introduced the error (mapping directly to AgentRx’s CFS concept from AI Agent Debugging).

security-hygiene Skill

The security-hygiene skill implements proactive FMEA for our self-modifying system:

  • Skill modification review: When a skill is created or modified, assess the risk (Severity × Likelihood × Detectability)
  • Boundary enforcement: Verify that src/ never imports desktop/ and vice versa — a structural barrier (Swiss Cheese defense layer)
  • Credential scanning: Prevent accidental commit of .env, API keys, tokens — barrier analysis for data leakage
  • Self-modification audit: Track which skills modified which other skills, creating an audit trail for forensic investigation

Context Rot as Root Cause

In long-running agent sessions, context rot emerges as a uniquely AI-native root cause. As the context window fills with intermediate results, earlier instructions and constraints degrade or are lost during compaction. This manifests as:

  • Agents forgetting project conventions established early in the session
  • Contradictory actions in later phases that violate decisions from earlier phases
  • Gradual drift from the original plan as compacted context loses nuance
  • Knowledge degradation: the agent “knows less” about the project than it did 30 minutes ago

Our mitigation: externalized state in .planning/ files, CLAUDE.md, and MEMORY.md. The agent’s critical knowledge is stored on disk, not in context. This is a barrier analysis approach — adding a defense layer between the hazard (context compaction) and the target (correct execution).

Error-as-Feedback Loops

In our architecture, errors are not terminal — they are feedback signals. The meta-agentic bootstrapping pattern treats every failure as information:

Loop: Agent executes → Failure detected → Error analyzed (forensics) → Correction synthesized → Agent re-executes with correction → Success or next iteration

This maps directly to the STAMP control loop: the agent is the controller, the codebase is the controlled process, test results are feedback, and corrective actions are control actions. When the loop fails, it is because the agent’s process model (its understanding of the codebase) became inconsistent with reality.

Consolidated Implications for Our Architecture

Each of the 8 deeply extracted papers included specific implications for gsd-skill-creator. Consolidated by architectural area:

Agent Communication & Handoffs

  • Our multi-agent architecture (GSD executor, verifier, planner) must address inter-agent communication explicitly — richer protocol design is more impactful than model upgrades (Kim et al.)
  • State tracking across agent handoffs (context-handoff skill, beads-state) is architecturally critical, not optional (Kim et al.)
  • Formalize agent handoff protocols: the 15 percentage point improvement from communication enrichment is the single largest actionable finding

Constraint Synthesis & Guards

  • The constraint synthesis pattern (tool schemas + domain policies → executable guards) maps directly to our hook system and security-hygiene skill (AgentRx)
  • The constraint-from-schema approach can be applied to our MCP tool definitions to auto-generate guards (AgentRx blog)
  • Open-source AgentRx availability means direct integration into our diagnostic pipeline is feasible (CC BY 4.0)

Failure Classification & Forensics

  • The 9-category taxonomy provides a concrete classification for our gsd-forensics skill to adopt (AgentRx)
  • “Critical Failure Step” identification is exactly what our witness-observer and polecat-worker need for stalled-agent detection (AgentRx)
  • Our gsd-forensics skill should follow blameless postmortem principles — focus on systemic causes, not “which model is bad” (Google SRE)
  • The postmortem template structure should be the output format of gsd-forensics (Google SRE)
  • Trigger criteria map to our phase/milestone completion checks — define explicit thresholds for when forensic investigation is warranted (Google SRE)

Research Co-Processor & Diagnostics

  • Hallucination and incomplete exploration are the top risks for our research co-processor pipeline; constraint-based guard rails at each agent step are essential (Kim et al.)
  • The trace distraction finding is critical: more data is not always better — our research co-processor must filter and summarize rather than dump raw traces (Riddell et al.)
  • Anchoring bias (15–45% accuracy reduction) maps directly to our agent workflows — first-hypothesis stickiness is a real threat; our planner/verifier split helps but may need explicit de-anchoring steps (Riddell et al.)
  • Belief update failure (RF-09) explains why our agents sometimes persist with wrong approaches — we need explicit “evidence review” checkpoints in phase execution (Riddell et al.)
  • Metric data is the most valuable signal type; prioritize structured metrics (test counts, build times, error rates) over raw log dumps in diagnostic pipelines (Riddell et al.)

Dynamic Dependency & Causal Reasoning

  • The dynamic causal graph approach is directly applicable to our multi-agent dependency chains — phase dependencies, skill activation sequences, and agent handoffs form a dynamic graph (DynaCausal)
  • The “deviation intensity” trap (where the most visibly broken thing is not the root cause) maps to our debugging experience — the loudest test failure is often a downstream symptom (DynaCausal)
  • SCO’s causal ordering constraint is a principle we should adopt: when diagnosing GSD pipeline failures, always rank upstream causes above downstream symptoms (DynaCausal)
  • Contrastive learning (normal vs anomalous embeddings) could improve our pgvector-based pattern matching for mission failure detection (DynaCausal)

Pattern Learning & Knowledge Architecture

  • The 74.38% recurrence rate validates our approach of persistent learning via skills — most failures repeat, so learned patterns have high reuse value (Wang & Qi survey)
  • Multi-modal fusion is the trend; our architecture already combines multiple signal types (git history, test results, phase plans, execution traces) — formalizing this as RCA input is natural (Wang & Qi survey)
  • Knowledge graph approaches align with our pgvector + Rosetta cluster architecture (Wang & Qi survey)
  • The survey’s evaluation framework (MTTD, MTTR, interpretability, generalizability) gives us concrete metrics for measuring our research co-processor (Wang & Qi survey)
  • The 7-goal framework maps remarkably well to our architecture: data correlation (Rosetta clusters), robustness (beads-state crash recovery), adaptive learning (skill-creator observation loop), real-time (GSD fast/quick paths), interpretability (session reports), multi-granularity (phase → plan → code), actionability (gsd-do routing) (Fang et al.)
  • The point-finding vs graph-building distinction is critical — our gsd-forensics should build propagation graphs, not just identify single failure points (Fang et al.)
  • The effectiveness-data-cost triangle is our token-budget constraint in different clothes (Fang et al.)
  • Cultural reinforcement through “reading clubs” and “wheel of misfortune” suggests we should build a failure pattern library that skills can learn from (Google SRE)
