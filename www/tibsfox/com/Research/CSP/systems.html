1. Linux & the Command Line

The command line is the programmer’s workshop. Every cloud server, every container, every Raspberry Pi runs Linux. The shell is the universal interface.

Everything Is a File

In Unix, devices are files (/dev/sda), processes have file descriptors (/proc/1234/fd/), and even network sockets are file handles. This abstraction means one set of tools (read, write, redirect) works on everything.

# Files and directories
ls -la                    # list with permissions, sizes, dates
chmod 755 script.sh       # rwxr-xr-x: owner can do anything, others read+execute
chown foxy:foxy file.txt  # change ownership

# Processes
ps aux | grep python      # find running Python processes
kill -9 1234              # force-kill process 1234
top                       # live process monitor (htop is better)

# Pipes and redirection — the Unix philosophy
cat access.log | grep 404 | wc -l         # count 404 errors
find . -name "*.html" | xargs wc -l       # count lines in all HTML files
curl -s api.weather.gov/stations/KPAE/observations/latest | python3 -m json.tool

The pipe (|) is the fundamental Unix innovation: small programs that do one thing well, connected together. cat reads, grep filters, wc counts, sort orders. Each is simple. Together they solve complex problems.

Package Management

# Debian/Ubuntu
sudo apt update && sudo apt install postgresql python3 nodejs
dpkg -l | grep julia     # check if installed

# Python
pip install numpy pandas psycopg2-binary
python3 -m venv .venv && source .venv/bin/activate  # virtual environment

# Node.js
npm install              # install from package.json
npx vitest               # run without installing globally

Try This

  1. Open a terminal. Run uname -a to see your kernel version. Run df -h to see disk space. Run free -h to see memory.
  2. Create a shell script: echo '#!/bin/bash' > hello.sh && echo 'echo "Hello from $(hostname)"' >> hello.sh && chmod +x hello.sh && ./hello.sh
  3. Pipe chain: history | awk '{print $2}' | sort | uniq -c | sort -rn | head -10 — your 10 most-used commands.

2. Containers & Orchestration

A container packages your application with its dependencies into an isolated, reproducible unit. Docker is the tool. Kubernetes is the orchestrator that runs containers across clusters.

Docker: Package Once, Run Anywhere

# Dockerfile — recipe for building a container image
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["python3", "server.py"]

# Build and run
docker build -t my-app .
docker run -p 8000:8000 my-app

# Docker Compose — multi-container applications
# docker-compose.yml
services:
  web:
    build: .
    ports: ["8000:8000"]
    depends_on: [db]
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: tibsfox
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:

The key insight: a container is not a VM. It shares the host kernel. Isolation comes from Linux namespaces (separate process trees, network stacks, mount points) and cgroups (CPU/memory limits). Overhead: near zero.

Kubernetes: Containers at Scale

ConceptWhat It Is
PodSmallest deployable unit. One or more containers that share networking and storage.
ServiceStable network endpoint for a set of pods. Load balances across replicas.
DeploymentDeclares desired state (3 replicas of my-app:v2). K8s reconciles actual to desired.

The WLCG runs DIRAC pilot jobs across 170+ sites — same pattern as Kubernetes: declare work, scheduler assigns it to available compute, monitor completion, retry failures. Our Gastown convoy model follows identical principles.

3. Networking & Security

Defense in Depth

Security is layers. No single measure is sufficient. Every layer assumes the layer above it has been compromised.

LayerToolWhat It Protects
PerimeterFirewall (iptables, ufw)Block unauthorized inbound connections
TransportTLS 1.3Encrypt data in transit. Certificate-based authentication.
AccessSSH keys, mTLSProve identity. Never use passwords for server access.
NetworkVPN, WireGuardEncrypted tunnels between networks. Zero-trust assumes no trusted network.
ApplicationOWASP, CSP headersInput validation, content security policy, rate limiting.
DataEncryption at restLUKS, dm-crypt. If the disk is stolen, data is unreadable.
# SSH key authentication (never passwords)
ssh-keygen -t ed25519 -C "foxy@artemis"
ssh-copy-id user@server

# Firewall basics
sudo ufw default deny incoming
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable

# TLS certificate (Let's Encrypt)
sudo certbot --nginx -d tibsfox.com
Our DRP zero-trust communications page applies these principles to disaster recovery: TCP/IP over ham radio, mesh networking with no trusted infrastructure. The Artemis II TDRS link uses the same layered security model — encryption, authentication, redundancy.

4. Distributed Systems

The CAP Theorem

A distributed system can provide at most two of three guarantees:

PropertyMeaningExample
ConsistencyEvery read returns the most recent writeBank balance is always accurate
AvailabilityEvery request gets a response (no timeouts)Website always loads
Partition ToleranceSystem works despite network failures between nodesTwo data centers can’t reach each other

In practice, network partitions happen. You must choose: CP (consistent but sometimes unavailable — PostgreSQL, ZooKeeper) or AP (always available but sometimes stale — Cassandra, DynamoDB). There is no CA system in a real network.

Consensus: Getting Distributed Nodes to Agree

Raft (Diego Ongaro, 2013) is the most widely used consensus protocol. A cluster of nodes elects a leader. The leader accepts writes and replicates them to followers. If the leader fails, followers elect a new one. etcd (Kubernetes’s brain), CockroachDB, and HashiCorp Consul all use Raft.

The key insight: a majority quorum is sufficient. In a 5-node cluster, 3 nodes agreeing is consensus. This tolerates 2 failures. Our muse council operates on the same principle — 5 muses deliberate, majority carries. The math is identical: f failures tolerated requires 2f+1 nodes.

5. Monitoring & Observability

You can’t fix what you can’t see. Observability has three pillars:

PillarWhat It CapturesTool
LogsDiscrete events with timestamps. “What happened.”journalctl, Loki, ELK
MetricsNumerical time-series. “How much, how fast, how often.”Prometheus, Grafana
TracesRequest flow across services. “Where did the time go.”OpenTelemetry, Jaeger

Prometheus & Grafana

Prometheus scrapes metrics from endpoints in a text format. Our prometheus/artemis_sweep.prom file is exactly this format:

# HELP artemis_met_hours Mission elapsed time in hours
# TYPE artemis_met_hours gauge
artemis_met_hours{mission="artemis-ii"} 46.92

# HELP muk_wx_temp_celsius Weather temperature by source
# TYPE muk_wx_temp_celsius gauge
muk_wx_temp_celsius{source="kpae",station="Paine Field"} 12.0
muk_wx_temp_celsius{source="ksea",station="Sea-Tac"} 13.0

Each metric has a name, labels (key-value pairs in braces), and a value. Prometheus stores time-series data and provides PromQL for querying. Grafana visualizes it. Together they answer questions like: “What was the temperature trend at KPAE over the last 4 days?” or “How many sweep samples have we collected per day?”

SLOs: Service Level Objectives

An SLO defines “good enough.” Example: “99.9% of API requests complete in under 500ms.” The error budget is the allowed failure: 0.1% of requests can be slow. When you’ve burned your error budget, you stop deploying new features and fix reliability.

Our sweep cadence has an implicit SLO: 100% hourly coverage. Every missed hour is a gap in the coverage table. The budget is zero — we don’t accept gaps. This is why backfilling exists: if a session boundary causes a gap, we fill it retroactively.

Our Prometheus metrics file tracks 28+ weather samples across 4 days, plus space weather and river data. The space weather feed is a monitoring dashboard for the mission. The sweep coverage table is our SLO tracking sheet — 100% coverage, no error budget.

Try This

  1. Run journalctl -f to watch system logs in real time. Open another terminal and start a service — watch the logs appear.
  2. Read our prometheus/artemis_sweep.prom file. Each line is a metric. Imagine these plotted as time series — temperature over 4 days, Kp index through the CME event, sample count growing.
  3. Install Prometheus and Grafana locally: docker run -d -p 9090:9090 prom/prometheus and docker run -d -p 3000:3000 grafana/grafana. Connect them. Import a dashboard.

The Infrastructure Mindset

Infrastructure is invisible when it works. The best systems administrator is the one whose users never know they exist. The shell, the container, the network, the monitor — these are the spaces between the code and the user. They don’t make the application work. They make it reliable.

The WLCG has been processing LHC data since 2009. It has survived hardware failures, network outages, entire computing centers going offline, and a global pandemic. It works because the infrastructure is layered, redundant, and monitored. The same principles apply to a single-server deployment: automate, monitor, plan for failure.
