GSD — Get Shit Done

An AI-Native Project Management Substrate for Hybridization, Measurement, and Evolution
Overview History Methodologies Metrics AI & Tools GSD Rosetta Learn
There is no “right way” to manage projects. We explore, experiment, and learn to find what works best for us. Our metrics and memory functions give us the data to work with, so we can try innovative new hybrid approaches to these pathways.

This is not a marketing slogan. It is the operating principle of GSD. Read the rest of this page through that lens.

1. What GSD Is

GSD (Get Shit Done) is an AI-native project management substrate built as a skill and agent layer for Claude Code. It is not a methodology in the traditional sense. It is a flexible framework that can absorb elements of waterfall, agile, kanban, lean, stage-gate, or any other approach — and it uses metrics and memory to learn which combinations work for the team and project at hand.

The word substrate is deliberate. A substrate is a base layer on which other things grow. GSD does not tell you how to manage your project. It gives you the ingredients — commands, artifacts, metrics, memory layers — from which you compose your way of managing this project. And then it gives you the feedback signals you need to evolve that approach over time.

The closest historical analogue is not Scrum or PMBOK but Unix: a small set of composable primitives, each doing one thing well, intended to be combined in ways the original authors never anticipated. GSD’s commands are pipes; the artifacts are streams; the human is the shell.

2. The GSD Philosophy — Stated Plainly

There is no “right way” to manage projects. We explore, experiment, and learn to find what works best for us. Our metrics and memory functions give us the data to work with, so we can try innovative new hybrid approaches to these pathways.

GSD was designed to be hybridizable:

  • You can run GSD in pure-waterfall mode — one phase, all at once, full specification upfront, single execution pass, single verification pass.
  • You can run GSD in agile-sprint mode — short phases, iterative, retrospective-driven, planning happens in the small, not in the large.
  • You can run GSD in kanban mode — continuous flow, no phase boundaries, work items pulled from a queue as capacity allows.
  • You can run GSD in stage-gate mode — explicit CAPCOM gates between phases, with formal go/no-go decisions and retrospective audits.
  • You can run GSD in exploratory mode — /gsd:explore for socratic ideation, /gsd:note for capture, /gsd:plant-seed for forward-looking ideas with trigger conditions, then promote to active work later.
  • Most teams end up running GSD in hybrid modes specific to their workflow, project type, and stage of development. This is normal. This is the goal. This is what the philosophy points at.

The phrase “there is no right way” is not a permission slip for sloppiness. It is an acknowledgment that the right way for you on this project at this moment is something only you can discover, and only by experimenting. GSD gives you the experimental apparatus — the substrate, the signals, the memory — and steps out of the way.

3. The GSD Lifecycle — The Substrate

What follows is a list of GSD commands. Read it as a menu, not a recipe. You use the items that fit your situation. Many teams only use a subset. Some teams use commands in orders the author did not anticipate. That is the point.

CommandWhat it doesWhen to use it
/gsd:new-project Initialize with deep context gathering. Produces PROJECT.md, ROADMAP.md, REQUIREMENTS.md. At project birth. Once per project.
/gsd:discuss-phase Adaptive questioning. Surfaces assumptions before planning. Asks the questions you would have wished you had asked. Before planning, when the phase is non-trivial.
/gsd:plan-phase Generate a PLAN.md with tasks, dependencies, waves, verification criteria. When you know what you want and need to break it down.
/gsd:execute-phase Wave-based parallel execution. Atomic commits. STATE.md tracking throughout. When the plan is approved and you want to build.
/gsd:verify-work Conversational UAT with measurable criteria. Checks deliverables against the plan. After execution, before declaring done.
/gsd:ship PR creation, review, merge preparation. When work is verified and ready to land.
/gsd:session-report Retrospective with token metrics, outcomes, lessons learned. End of session. Captures the data you will need next time.
/gsd:pause-work / /gsd:resume-work Context persistence across sessions. Compresses state into a handoff document. When stopping mid-work. When restarting after a break.
/gsd:note Zero-friction idea capture. Append, list, promote. When an idea hits and you don’t want to lose it.
/gsd:plant-seed Forward-looking idea capture with trigger conditions. Surfaces automatically at the right milestone. When an idea is for later, not now.
/gsd:explore Socratic ideation. Think through ideas before committing to plans. Before you know what to plan. Pre-planning thinking.
/gsd:fast / /gsd:quick Bypass overhead for trivial tasks. Atomic commits, state tracking, no agents. When the task is small and the substrate would be overkill.

Read this list again. None of these are required. You can run an entire GSD project using only three of them. You can run an entire GSD project using ten of them. The only commitment GSD asks is that you use git for execution and that you let the artifacts speak for themselves.

4. Metrics as Feedback Signals (Not Targets)

GSD instruments every phase with metrics. They are signals, not targets. The point is not to optimize them. The point is to have enough data to make informed experiments with your process.

MetricWhat it measuresWhat to do when it spikes
Token consumption per phase Planning-to-execution ratio. How much context-building vs. doing. If high relative to outcomes, the plan may be too abstract or the context window too cluttered. Try smaller phases or chunked context.
Commit frequency Actual work rate. Atomic commits per hour or per phase. If low, the plan may be too vague or the team is blocked. Investigate before adding pressure.
Research-to-code ratio Context-building vs. producing. High ratio may mean over-researching. Low ratio may mean under-researching. Both are signals to recalibrate, not failures.
Verification pass rates Quality of planning. Does what was built match what was planned? Low pass rate means the plan step needs more rigor. High pass rate may mean criteria are too easy.
Phase completion time variance Estimation accuracy. How much actual time differed from planned time. Use to calibrate future estimates. Don’t use to evaluate the team.
Session-to-session handoff size Knowledge preservation efficiency. How much context survived the session boundary. Large handoffs mean state is bleeding across sessions. Try smaller phases or better artifact discipline.

None of these metrics are targets. None of them should appear on a dashboard with a green/red traffic light. They are signals that tell you whether your current workflow is working. Low verification pass rate? The plan step needs more rigor. High research-to-code ratio? You may be over-researching. The point is to have enough data to make informed experiments with your process.

Read the Metrics page for the broader history of why this framing matters — the CHAOS reports, Campbell’s Law, EVM, DORA, and the metrics trap. GSD’s contribution is not new metrics. It is the discipline of treating metrics as feedback for hybridization, not as targets to hit.

5. Memory as a Persistence Layer for Learning

GSD treats memory as a first-class concern. Every artifact is designed to outlive the session that produced it.

  • STATE.md — current state of the work. Updated continuously during execution. The single source of truth for “where are we right now?”
  • Handoff documents — compressed context snapshots for session transitions. Generated by /gsd:pause-work. Read by /gsd:resume-work. The mechanism by which work survives context resets.
  • Session reports — retrospective data with token estimates, work summary, lessons learned. Generated by /gsd:session-report. The retrospective that actually happens because it is built into the workflow.
  • Git history — the immutable execution log. Every commit is a status update. The PR is the work item. The merge is the delivery. Git is the most reliable PM tool ever built; GSD just uses it.
  • REQUIREMENTS.md / ROADMAP.md / PROJECT.md — the stable artifacts. Updated when intent changes, not when status changes.
  • Memory files in ~/.claude/memory — cross-project patterns, user preferences, lessons learned that transcend any single project. The mechanism by which GSD accumulates wisdom across projects.

Memory is how GSD accumulates wisdom across projects. Not just “what did we build” but “what process worked when we built it, and what didn’t.” The lessons from project A inform project B, not because someone wrote a wiki page that no one reads, but because the artifacts themselves carry the learning forward.

6. Hybridization Examples

The following are real patterns observed in GSD projects. None of them are “the right way.” All of them are right for the team and project they emerged in.

Agile-Waterfall Hybrid

Waterfall for phases that have regulatory requirements (the team must produce a complete spec before any code is written and the spec must be reviewed by external parties). Agile for phases that are experimental (rapid iteration with small commits and frequent verification). The same project may use both modes in different phases. GSD does not care.

Kanban-Wave Hybrid

Use wave-based parallelism within a phase (like agile sprints — parallel work items completing roughly together) but pull work items from a backlog (like kanban) instead of committing to a sprint scope upfront. The waves provide concurrency. The kanban pull provides flow. The combination provides both.

Stage-Gate + Iteration

CAPCOM gates between phases (stage-gate semantics — an explicit go/no-go decision with documentation). Iterative execution within phases (agile semantics — rapid feedback loops, atomic commits, sub-phase retrospectives). The gates protect the macro structure; the iteration delivers the micro work.

Exploratory + Production

Use /gsd:explore and /gsd:note for the research/ideation phase, when you don’t yet know what to build. Then switch to formal /gsd:plan-phase + /gsd:execute-phase for the production phase, when you do. The same project flows through both modes at different times.

Single-Phase Mode (the “Just Ship It” Pattern)

Some projects do not need a multi-phase structure at all. /gsd:fast and /gsd:quick exist for trivial tasks. A single phase can hold everything if that is what fits. The substrate scales down as well as up.

7. GSD’s Unique Solves

The Documentation Debt Problem

Traditional PM produces documentation as a separate activity. The PM writes the plan, the team writes the code, someone else writes the docs — if anyone remembers. GSD eliminates this by making documentation be execution. PLAN.md is the plan. The commit history is the execution log. STATE.md is the live status. session-report is the retrospective. There is no separate documentation phase because the documentation is the execution. You cannot skip it.

The Context Loss Problem

Knowledge work is context-dependent, and context is fragile. When a developer leaves a team, when a session ends, when a context window fills up, the knowledge that was in someone’s head evaporates. GSD’s handoff documents encode state for future sessions. /gsd:pause-work compresses the live context into a document; /gsd:resume-work reconstitutes it. The result is that work survives context resets.

The Estimation Problem

Estimation is where most PM methodologies break. Estimates are wrong, often by 2x, sometimes by 10x. GSD’s response is to not estimate. Plan, execute, measure, adjust. Use the metrics to calibrate. Treat each phase as a probe, not a commitment. If a phase took longer than expected, the lesson is in the variance, not in “who do we blame.”

The Over-Planning Problem

Some PM systems require so much upfront planning that the planning consumes the project. /gsd:fast and /gsd:quick exist for tasks where the substrate would be overkill. The full lifecycle is for complex work; the lightweight commands are for everything else.

The Methodology Lock-in Problem

Most PM tools embody a methodology and resist others. GSD is a substrate, not a prescription. Switch approaches when the data says so. The tools do not fight you when you change your mind.

8. The Human Role in GSD

In a GSD project, the human is not the scheduler, the estimator, the status tracker, or the documentation writer. The AI handles those. The human is:

  • Vision holder. Sets direction. Decides what success looks like. Articulates why this project exists.
  • Quality gate. Approves plans before execution. Verifies results before shipping. Says “yes” or “no, that’s not what I meant.”
  • Decision maker. Resolves ambiguity when the AI cannot. Makes the calls that require values, judgment, or context the AI does not have.
  • Experimenter. Tries new approaches to the workflow itself. Reads the metrics. Adjusts the process. Notices when something isn’t working and changes it.

Notice what is not on this list. Status updates. Estimation. Documentation. Schedule maintenance. Burndown chart updating. Standup-running. These are activities that AI handles in GSD — not because the AI is “better” at them but because the human time is more valuably spent on the four roles above.

9. The AI Role in GSD

  • Planner. Generates plans from context. Decomposes vague intent into concrete tasks with dependencies, waves, and verification criteria.
  • Executor. Produces deliverables. Writes the code, the tests, the documentation. Commits atomically. Pushes when ready.
  • Verifier. Checks work against criteria. Runs the tests. Flags discrepancies. Asks the human when something is ambiguous.
  • Documenter. Produces artifacts as a byproduct of execution. STATE.md, PLAN.md, session reports, handoffs — all generated automatically, all reflecting actual execution.
  • Context manager. Maintains state across sessions. Knows where the work was last left. Picks up where it left off.

Notice what is not on this list. Vision setting. Strategy. Final “is this what we want” calls. The AI does not decide what the project should accomplish. The AI does not decide whether a deliverable meets the human’s actual intent. Those remain human responsibilities and likely will for a long time.

10. GSD Compared to Traditional Methodologies

The table below compares GSD to common methodologies. Important caveat: GSD can emulate any of these when the metrics say so. The comparison is not “which is better” but “how does GSD relate to each.”

MethodologyRelationship to GSD
Scrum No fixed sprint length, but the wave model captures the iteration concept. GSD can run with sprint-like timeboxes if that fits. The retrospective is built in (/gsd:session-report).
Kanban Supports continuous flow mode (pull from backlog), but defaults to phased work. The /gsd:explore and /gsd:note commands act as a low-friction backlog when you want kanban semantics.
Waterfall Can run in single-phase mode with full upfront specification, but defaults to iterative. GSD does not prevent waterfall — it just makes iteration cheap enough that few teams choose pure waterfall.
SAFe No organizational overhead. GSD scales via parallel workstreams (/gsd:workstreams) rather than via additional roles, ceremonies, and planning levels. The assumption is that AI agents can handle the coordination overhead that SAFe addresses with human roles.
Stage-Gate The CAPCOM gate concept maps directly. /gsd:audit-milestone and /gsd:complete-milestone serve as gate decisions with documented criteria.
PMBOK 7 (principles-based) Closest philosophical alignment. PMBOK 7’s shift from processes to principles is the same shift GSD embodies. The 12 PMBOK principles map cleanly onto GSD’s commands.
“GSD as a prescription” This is a straw man. GSD is not prescriptive. If anyone tells you there is a “right way” to do GSD, they have misunderstood the substrate.

11. The Experimentation Imperative

Don’t use GSD “correctly.” Use it experimentally.

If you are reading this page hoping to learn the right way to run a GSD project, you are reading the wrong page. There is no right way. There is only the way that works for you and your team and your project, this week, on this codebase, with this stakeholder, in this part of the lifecycle. That way will not be the way that works next week, on a different codebase, for a different stakeholder, in a different part of the lifecycle. The substrate is constant; the configuration is not.

What you should do instead:

  • Try different combinations of commands and see what fits. You can use /gsd:plan-phase without /gsd:discuss-phase. You can run /gsd:execute-phase on a hand-written plan. You can skip the verification step on a throwaway prototype. Try the unusual combinations.
  • Measure what works. Run /gsd:session-report at the end of every session. Look at the token counts, the commit counts, the verification pass rates. Notice patterns. The metrics are signals; treat them that way.
  • Remember it in memory files. When you find a hybrid that works, write it down in a memory file. When you find a hybrid that fails, write that down too. The point of memory is to make this the last time you have to learn that particular lesson.
  • Share experimental findings. Session reports, handoff documents, retrospectives, blog posts. The PM profession has been short on data for decades because reporting failure was career-ending. In GSD, the failures are part of the substrate — they are how you learn what works.
  • Evolve your personal GSD dialect over time. By the end of your tenth project, you will have a way of using GSD that no one else uses, that fits you and your work perfectly, that would be incomprehensible to a beginner. This is not a sign of fragmentation. It is a sign of mastery.

The PM history covered on the History page is, in large part, a story of people trying to find the “one true way” to manage projects — and a parallel story of those efforts repeatedly failing because no one true way exists. GSD takes the opposite bet. It assumes there is no one true way, designs the substrate accordingly, and gives you the apparatus to discover what works for you. The discovery is the work.

Start experimenting.
