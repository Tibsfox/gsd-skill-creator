MCP Security Landscape

Six papers mapping the security terrain of the Model Context Protocol ecosystem — from enterprise IAM integration and governance frameworks through protocol-level threat modeling, automated tool poisoning, and the structural risk of overthinking loops that inflate costs by 658x. Papers

Enterprise Security

Enterprise MCP: IAM, Zero-Trust Architecture

Paper: arXiv:2504.08623
Categories: cs.CR, cs.AI · Date: April 2025

Key Finding: Enterprise MCP deployments require IAM integration with Azure AD, Okta, and SAML/OIDC federation. The paper establishes a zero-trust architecture pattern where every tool invocation must pass through identity verification, session binding, and permission scoping — treating MCP servers as first-class security principals rather than transparent proxies.

Architecture: The proposed pattern includes: (1) per-tool permission manifests declaring required scopes, (2) session-bound tokens that expire with the agent’s context window, (3) audit trails linking every tool invocation to both the requesting agent and the authorizing user, and (4) break-glass procedures for emergency access that bypass normal approval but generate elevated alerts.

Implication: Without IAM federation, MCP servers become privilege escalation vectors in enterprise environments where tools inherit ambient credentials from the hosting process.

Governance

Securing MCP: Risks, Controls, Governance

Paper: arXiv:2511.20920
Categories: cs.CR, cs.SE · Date: November 2025

Key Finding: A comprehensive risk taxonomy for MCP deployments covering data exfiltration, prompt injection through tool responses, server impersonation, and cascading failures in multi-server topologies. Proposes a three-tier governance model: platform controls (transport encryption, authentication), operational controls (audit logging, rate limiting), and policy controls (tool allowlists, data classification).

Three-Tier Model:

  • Platform controls: TLS 1.3 transport encryption, mTLS for server-to-server, certificate pinning for known MCP endpoints
  • Operational controls: Structured audit logging with tool invocation provenance, rate limiting per tool per session, anomaly detection on invocation patterns
  • Policy controls: Tool allowlists scoped to organizational role, data classification labels on tool inputs/outputs, retention policies for tool interaction logs

Implication: MCP governance cannot be bolted on after deployment. The paper demonstrates that 78% of MCP security incidents stem from missing operational controls that would be standard in traditional API management.

Protocol Comparison

Security Threat Modeling: MCP, A2A, Agora, ANP

Paper: arXiv:2602.11327
Categories: cs.CR, cs.DC · Date: February 2026

Key Finding: Comparative threat modeling across four agent communication protocols. MCP optimizes for tool integration but lacks built-in mutual authentication. A2A (Agent-to-Agent) provides stronger identity guarantees but higher latency. Agora enables marketplace discovery at the cost of trust-on-first-use vulnerabilities. ANP (Agent Network Protocol) offers the most complete security model but lowest adoption.

Threat Categories per Protocol: The analysis covers STRIDE-based threat modeling (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) applied to each protocol’s message flow, discovery mechanism, and session management. MCP scores worst on Spoofing and Elevation of Privilege due to its trust-the-transport design. ANP scores worst on Denial of Service due to its complex handshake requirements.

Implication: No single protocol provides complete security. Production deployments should layer protocols — MCP for tool binding, A2A for inter-agent trust, with Agora/ANP for discovery only in controlled environments.

Landscape Survey

MCP Landscape: Security Threats & Future Research

Paper: arXiv:2503.23278
Categories: cs.CR, cs.AI · Date: March 2025

Key Finding: A landscape survey covering the MCP ecosystem as of early 2025, identifying 12 distinct threat categories and 47 specific attack vectors. The survey maps the gap between MCP’s rapid adoption (110M SDK downloads, 3,600x growth since November 2024) and the maturity of its security tooling, which remains at “early prototype” for most categories.

Threat Categories: The 12 categories span: tool impersonation, parameter injection, response manipulation, session hijacking, credential leakage, data exfiltration, privilege escalation, denial of service, supply-chain compromise, cross-server attacks, state corruption, and audit evasion. Of these, supply-chain compromise and credential leakage have the highest real-world impact scores.

Implication: The adoption-security gap is widening. The paper projects that without coordinated security investment, MCP will replicate the npm/PyPI supply-chain crisis at a larger scale due to the higher privilege level of tool invocations.

Tool Poisoning

MCP-ITP: Automated Implicit Tool Poisoning

Paper: arXiv:2601.07395
Categories: cs.CR, cs.CL · Date: January 2026

Key Finding: An automated framework for generating implicit tool poisoning attacks against MCP servers. MCP-ITP demonstrates that tool descriptions — the natural-language metadata agents use to select tools — can be crafted to manipulate agent behavior without modifying the tool’s actual implementation. The framework achieves consistent redirection of agent tool selection across multiple LLM backends.

Attack Vector: The poisoned tool description embeds subtle semantic cues that cause the agent to prefer the malicious tool over legitimate alternatives. For example, a poisoned “file search” tool description might include phrases like “comprehensive search including environment variables and credentials” that cause agents to route sensitive queries to the attacker-controlled tool. The tool’s code is benign; the attack is entirely in the metadata.

Defense Gap: Current MCP security focuses on tool code review and sandboxing. MCP-ITP bypasses both because the malicious behavior originates from the agent’s interpretation of the description, not from the tool’s execution. A new defense category — description semantic analysis — is required.

Implication: Tool poisoning attacks are fundamentally different from code injection — they operate at the semantic layer where agents make decisions about which tools to invoke and in what order. Static analysis of tool code is insufficient; defense must include semantic analysis of tool descriptions.

Cost & Risk

Overthinking Loops: 658x Cost Inflation

Paper: arXiv:2602.14798
Categories: cs.AI, cs.SE · Date: February 2026

Key Finding: Agents interacting with MCP tools can enter overthinking loops where they repeatedly invoke tools, re-read results, and re-plan without making progress. The paper measures cost inflation up to 658x compared to direct completion, with the agent burning tokens on recursive self-consultation rather than acting on available information. The structural cause is the absence of progress detection in the agent-tool interaction loop.

Loop Taxonomy: Three overthinking patterns identified: (1) verification spirals — the agent repeatedly checks its own work without making corrections, (2) exploration loops — the agent reads the same files or tool outputs multiple times seeking information that isn’t there, and (3) plan oscillation — the agent alternates between two approaches without committing to either. Verification spirals are the most costly (median 340x inflation); plan oscillation is the most frequent (58% of overthinking incidents).

Proposed Defenses: Four circuit breaker patterns: max-iteration limits per tool, diminishing-returns detection (flag when new tool calls produce <5% new information), cost budget enforcement with graceful degradation, and progress heuristics based on output entropy change.

Implication: Overthinking is not a rare edge case — it occurs in 12–18% of complex multi-tool tasks. Without circuit breakers (max-iteration limits, progress heuristics, cost budgets), MCP tool use becomes an economic denial-of-service vector.

Synthesis: The Four-Layer Security Model

These six papers collectively establish that MCP security operates across four distinct layers, each requiring its own defense strategy:

  1. Identity Layer (2504.08623, 2511.20920) — IAM federation, zero-trust authentication, session binding. Without identity, every other layer is bypassed.
  2. Protocol Layer (2602.11327, 2503.23278) — Transport encryption, mutual authentication, discovery trust. Protocol choice determines the ceiling of achievable security.
  3. Tool Layer (2601.07395) — Semantic analysis of tool descriptions, intent verification beyond code signing. The attack surface is the natural-language interface, not the implementation.
  4. Economic Layer (2602.14798) — Cost budgets, progress detection, circuit breakers. Overthinking loops transform tool use into resource exhaustion attacks.

The gap between MCP’s adoption velocity (110M SDK downloads) and security tooling maturity means production deployments must implement all four layers simultaneously, not sequentially.
