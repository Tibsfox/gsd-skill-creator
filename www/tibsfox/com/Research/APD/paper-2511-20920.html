Securing MCP: Risks, Controls, Governance

arXiv: 2511.20920

Date: November 2025 · Categories: cs.CR, cs.SE

Abstract

A comprehensive risk taxonomy for MCP deployments covering data exfiltration, prompt injection through tool responses, server impersonation, and cascading failures in multi-server topologies. The paper contrasts dynamic user-driven agent systems with static API integrations, demonstrating that traditional API security models are insufficient for agentic contexts where tool selection is non-deterministic. Proposes a three-tier governance model spanning platform, operational, and policy controls.

Key Findings

  1. 78% of MCP security incidents stem from missing operational controls
  2. Three-tier governance model: platform controls (TLS 1.3, mTLS), operational controls (audit logging, rate limiting), policy controls (tool allowlists, data classification)
  3. Dynamic agent systems introduce risks absent in static API integrations — non-deterministic tool selection creates attack surfaces
  4. MCP governance cannot be bolted on after deployment; it must be designed in from the start
  5. Certificate pinning for known MCP endpoints prevents server impersonation attacks

Methodology

Risk taxonomy development based on analysis of MCP deployment patterns in enterprise environments. Comparative analysis between dynamic agent-driven tool selection and static API integration security models. Framework validated against known incident patterns.

Implications for Our Architecture

The three-tier model maps directly to our GSD skill security layers: platform controls align with hook-based enforcement, operational controls with the security-hygiene skill's audit patterns, and policy controls with skill permission manifests. The 78% operational gap finding validates our emphasis on deterministic hooks over advisory guardrails. The policy-based access control model should inform the trust-relationship provider's permission scoping.
