MCP Security

Enterprise-Grade Security for the Model Context Protocol

arXiv: 2504.08623

Authors: Narajala, Habler

Date: April 2025 · Categories: cs.CR, cs.AI

Abstract

Enterprise MCP deployments require IAM integration with Azure AD, Okta, and SAML/OIDC federation. This paper establishes a zero-trust architecture pattern where every tool invocation must pass through identity verification, session binding, and permission scoping — treating MCP servers as first-class security principals rather than transparent proxies. The paper proposes structured approval workflows incorporating security, data governance, privacy, and legal considerations, with periodic recertification of deployed tools.

Key Findings

  1. MCP servers must be treated as first-class security principals with per-tool permission manifests declaring required scopes
  2. Session-bound tokens that expire with the agent's context window prevent credential leakage across sessions
  3. Audit trails must link every tool invocation to both the requesting agent and the authorizing user
  4. Break-glass procedures for emergency access generate elevated alerts while bypassing normal approval
  5. Without IAM federation, MCP servers become privilege escalation vectors inheriting ambient credentials

Methodology

Architecture analysis of enterprise MCP deployment patterns, examining IAM integration requirements across Azure AD and Okta identity providers. The authors propose a four-component security architecture: per-tool permission manifests, session-bound tokens, audit trail linking, and break-glass procedures. Evaluated against existing enterprise API management patterns to identify gaps.

The evaluation methodology compares MCP security requirements against established patterns from OAuth 2.0, SAML federation, and OIDC flows. Each proposed component is assessed for compatibility with existing enterprise identity infrastructure, deployment complexity, and operational overhead. The paper also analyzes failure modes when individual components are missing.
