The LHC generates roughly one petabyte of collision data per day. No single computing center on Earth can store and process this alone. CERN's answer: don't build a bigger center — federate every center. The Worldwide LHC Computing Grid connects 170+ sites across 42 countries into a single coherent computing fabric. It is the proof that distributed autonomous systems can operate at planetary scale with shared governance, heterogeneous infrastructure, and no central authority over execution. Sound familiar? It should. This is what we're building with Gastown.

Tier Architecture

WLCG organizes computing resources into tiers based on capability, responsibility, and connectivity. The model is hierarchical in data flow but federated in governance — each site operates autonomously.

Tier 0 — CERN Data Centre

Tier 0

The source. Tier 0 is CERN itself. Raw collision data from the four LHC detectors flows here first. Performs first-pass reconstruction (raw → calibrated physics objects) and distributes copies to all Tier 1 sites within 48 hours. Stores one full copy of all raw and reconstructed data permanently. ~15% of total WLCG compute capacity.

Gastown parallel: This is the mayor-coordinator. It doesn't do all the work — it receives the signal, does first-pass triage, and distributes work items to the tier below.

CERN Data Centre, Meyrin + Wigner (Budapest) · ~100K cores · ~340 PB disk

Tier 1 — National Computing Centers (14 sites)

Tier 1

The backbone. 14 large national computing centers in 12 countries. Connected to CERN by dedicated 100 Gbps fiber (LHCONE network). Each stores a fraction of the total dataset and provides significant CPU for scheduled reprocessing. Custodial storage: tape archives holding a second copy of all data. Available 24/7 with professional operations teams.

Gastown parallel: These are polecat-workers with persistent state and guaranteed availability. They don't choose their work — they receive dispatched beads and execute them reliably. The dedicated fiber is like our mail-async durable messaging.

BNL, CNAF, FNAL, GridKA, IN2P3, JINR, KISTI, NDGF, NL-T1, PIC, RAL, RRC-KI, ASGC, TRIUMF

Tier 2 — University Sites (160+ sites)

Tier 2

The workforce. 160+ sites at universities and research labs. Provide the bulk of CPU for Monte Carlo simulation and user analysis. Heterogeneous: different hardware, different middleware, different local policies. Connect via national research networks (1–10 Gbps). Fat Tier-2s (T2D) provide more services; thin Tier-2s (T2C) provide raw compute only.

Gastown parallel: This is the convoy model. Heterogeneous agents with different capabilities, dispatched work via sling, monitored by witness. The fat/thin distinction mirrors our quality/balanced/budget model profiles — same interface, different capacity.

160+ sites · ~1M cores combined · majority of Monte Carlo production

Tier 3 — Local Clusters

Tier 3

The edge. Individual physics groups' local clusters. No formal SLA with WLCG. Used for final-stage analysis, development, and testing. Pull data on demand from Tier 1/2 via XRootD federation. Ephemeral — can come and go without affecting the grid.

Gastown parallel: Local development environments. An agent that spins up, does work, disappears. No state contract with the orchestrator. This is how our worktree isolation works — ephemeral, pull what you need, produce artifacts, vanish.

Protocols & Middleware Stack

XRootD — Data Federation Protocol

Protocol Chipset Lesson

Developed at SLAC/CERN. Provides transparent data access across all tiers. A client requests a file; the redirector service locates it across the entire grid and streams it. No need to know where data physically lives. Supports caching, parallel streams, and third-party copy. Handles >1M file opens/day across the federation.

Chipset lesson: Our context-packet DAG resolver does the same thing — agent requests context, system locates and delivers it without the agent knowing the source. XRootD's redirector = our context resolver. The pattern is universal: request what you need, not where it lives.

HTCondor — Workload Management

Protocol Chipset Lesson

High-throughput computing framework from UW-Madison. WLCG uses HTCondor-CE (Computing Element) as the interface between the grid and local batch systems. Jobs are submitted to a central negotiator that matches job requirements to available resources. ClassAds (key-value descriptions) enable capability-based matching rather than hard-coded routing.

Chipset lesson: ClassAds = our hook-persistence work descriptors. Capability-based matching = our model profile routing. HTCondor doesn't care what hardware runs the job, only that the requirements match. Describe the work, not the worker.

DIRAC — Interware

Protocol Chipset Lesson

LHCb's workload management system, now used by 70+ virtual organizations. Sits between experiments and the grid. Provides job management, data management, accounting, and monitoring through a unified API regardless of the underlying middleware (Grid, Cloud, HPC). "Pilot job" model: DIRAC sends lightweight pilots that claim resources, then pull real jobs from a central task queue.

Chipset lesson: The pilot job model IS our GUPP (Get Up and Push Protocol). The pilot lands on a resource, checks for work, executes it, reports back. DIRAC's agent framework with specialized agents for different subsystems mirrors our polecat/witness/mayor separation. This is convergent evolution.

FTS — File Transfer Service

Protocol

CERN-developed reliable bulk data transfer service. Manages millions of file transfers per day between sites. Handles retry logic, checksum verification, bandwidth scheduling, and priority queues. Supports GridFTP, HTTP-TPC, and XRootD protocols. Activity-based scheduling prevents any single transfer campaign from monopolizing links.

Chipset lesson: Activity-based scheduling = our convoy batching. Priority queues = our wave-based execution. FTS proves that reliable bulk transfer at scale requires: checksums, retries, bandwidth fairness, and activity classification. All patterns we implement.

CVMFS — Software Distribution

Protocol

CernVM File System. Distributes experiment software to all 170+ sites via HTTP-based content-addressable storage with aggressive caching. A single publication at CERN propagates to every site within minutes. Read-only, deduplicated, cryptographically verified. ~500 TB of software served to ~160K unique clients daily.

Chipset lesson: Content-addressable + read-only + cached = exactly how our skills distribute. A skill published once propagates to all agents. CVMFS is skills distribution at CERN scale.

Rucio — Data Management

Protocol

ATLAS-developed, now multi-experiment data management system. Tracks the location, state, and lifecycle of >1 billion files across the grid. Enforces replication policies (e.g., "keep 2 copies on Tier 1 tape, 1 on Tier 2 disk"). Manages data placement, deletion, and access patterns. The metadata catalog for the entire ATLAS dataset.

Chipset lesson: Rucio = our beads-state persistence layer. It tracks what exists, where it lives, and what the lifecycle policy is. The replication rules mirror our "always keep state in git + local JSON" dual-write pattern.

Architectural Lessons for GSD & Skill-Creator

1. Federate, Don't Centralize

Architecture

WLCG works because no single site owns all the data or all the compute. Sites join voluntarily, operate autonomously, and contribute to the whole. Our multi-agent system should work the same way: agents are autonomous, contribute to shared state, and the orchestrator coordinates rather than controls.

2. The Pilot Job Pattern Is Universal

Pattern

DIRAC's pilot jobs land on a resource, discover what's available, pull work from a queue, execute, report. This is exactly GUPP. WLCG validated this pattern at planetary scale with 1.4M cores. Our implementation at 4-8 agents is the same architecture, different magnitude.

3. Content-Addressed Distribution Scales

Pattern

CVMFS distributes 500 TB of software to 160K clients daily using content-addressed, cached, read-only filesystems. Our skills are the same: published once, immutable, cached locally, verified by hash. If we ever need to scale skills distribution, CVMFS is the blueprint.

4. Heterogeneity Is a Feature, Not a Bug

Architecture

WLCG sites run different hardware, different OS versions, different local batch systems. The middleware layer (HTCondor-CE, DIRAC) abstracts this away. Our runtime-hal (Hardware Abstraction Layer) does the same: detect Claude Code, Codex, Gemini, Cursor — present a uniform interface. WLCG proves this works at 170+ sites.

5. Monitoring Must Be Passive and Universal

Operations

WLCG's MonIT system collects metrics from all sites without requiring sites to push. Dashboards aggregate. Alerts trigger on anomalies. No site is required to explain itself — the monitoring system observes and reports. This is our witness-observer pattern: watch, don't interfere, escalate when stuck.

6. Data Lifecycle Policies Prevent Rot

Operations

Rucio enforces rules: "2 copies on tape, 1 on disk, delete after 2 years." Without automated lifecycle management, data accumulates until storage is full. Our .planning/ lifecycle (gitignored, session-scoped, archived per milestone) is the same pattern. Rucio just does it for a billion files.
