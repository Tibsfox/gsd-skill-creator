1. Course Overview

Course: 17-214/17-614 — ML in Production / Machine Learning in Production
University: Carnegie Mellon University, Spring 2025
Instructors: Christian Kastner (primary), Claire Le Goues (co-instructor)
Scope: Full semester, 16 lectures from requirements engineering through deployment, monitoring, fairness, and operations

This is not a machine learning course. It is a systems engineering course that happens to include ML components. The central argument, developed across all 16 lectures: building a model is the easy part. Making that model safe, reliable, and maintainable inside a production system is where the real engineering happens.

The course progresses through four arcs:

ArcLecturesCore Question
Foundations1–5What should the system do, and how do we know the model is any good?
Human & Engineering6–8How do teams build and test ML pipelines?
Deployment9–11Where does the model run, and what data feeds it?
Operations & Safety12–16How do we monitor, update, and keep production systems safe?

Kastner and Le Goues bring complementary perspectives: Kastner from software engineering and systems architecture, Le Goues from software testing and formal methods. The result is a course that treats ML systems with the same rigor applied to safety-critical software — because, increasingly, ML systems are safety-critical software.

2. Safety ≠ Reliability — The Central Thesis

The single most important idea in the course, developed across Lectures 2, 4, and the Safety lecture:

A more reliable model is not a safer system.

RLHF, Constitutional AI, DPO — all are training-time interventions that improve model reliability (reduce incidence of unsafe outputs). But they do not constitute system-level safety. You can reduce how often an NC-17 movie is recommended to a minor, but that does not make the recommendation system safe.

The Gas Tank Analogy

Making gas tanks more rigid (more reliable) made them more dangerous (bigger explosions). Instead, adding pressure valves (system-level mitigation) made them safe. The lesson: safety comes from system design, not component quality.

Four Distinct Concerns

ConcernDefinitionRelationship to Safety
SafetyPrevention of system-level harmsThe goal — requires all four

The Correctness Gap

Lecture 2 establishes the philosophical foundation: models cannot provide guarantees — period. SQL injection has a known structural fix (parameterized queries); prompt injection does not. You can increase reliability (distance between mistakes) but cannot guarantee correctness. This means every ML system needs engineering around the model to handle the inevitable failures.

Risk = Probability of Harm × Severity of Harm. The error budget concept from SRE applies directly: if you never make mistakes, you have probably invested too much in quality assurance. The question is not “will the model fail?” but “what happens when it does?”

3. Guardrails as System Primitives

If models will fail, the engineering response is system-level guardrails — deterministic interventions that constrain the damage a model failure can cause.

Input Guards

  • Prompt injection detection — pattern matching on inputs before they reach the model
  • Input validation — schema enforcement, type checking, range constraints
  • Rate limiting — throttle request volume to prevent abuse

Output Guards

  • Toxic content filtering — deterministic post-processing on model output
  • Output clipping — constrain numerical outputs to valid ranges
  • Vocabulary filters — block specific terms or patterns in generated text

Tool Access Controls

  • Human approval requirements for high-risk actions
  • Tool-level permissions enforced in code, not in prompts
  • Sandboxing — isolate model execution from sensitive system resources

The SQL Injection Analogy

The 20-year-old web security lesson applies directly: always validate on the server side. Always validate using some mechanism other than the model itself before executing dangerous actions. This principle — borrowed from security engineering and applied to ML systems — is the operational foundation of the guardrails approach.

Deterministic over model-based guardrails. Deterministic guards (pattern matching, schema validation, code-level permission checks) cannot be bypassed by prompt injection. Model-based guards (asking a second LLM to evaluate the first) inherit the same vulnerability. The CMU and OWASP positions converge here: prefer deterministic enforcement wherever possible.

4. Hazard Analysis for ML Systems

Before building guardrails, you need to know what can go wrong. Lectures 2 and 4 introduce three hazard analysis techniques adapted from safety engineering for ML systems.

FMEA — Failure Mode and Effects Analysis

StepQuestionML Application
1. Identify componentsWhat can fail?Model, data pipeline, feature store, serving infrastructure, monitoring
2. Failure modesHow can it fail?Wrong prediction, stale data, schema drift, latency spike, silent degradation
3. EffectsWhat happens when it fails?Wrong recommendation, denied credit, missed diagnosis, safety incident
4. Severity × ProbabilityHow bad and how likely?Risk priority number guides mitigation investment

FTA — Fault Tree Analysis

Work backward from a catastrophic outcome to identify all paths that could produce it. For an autonomous vehicle fatality: sensor failure AND perception model error AND planning model override AND no human intervention AND no fail-safe braking. Each gate in the tree is an engineering opportunity.

STPA — Systems-Theoretic Process Analysis

The most powerful technique for ML systems because it treats the entire system — human operators, software, hardware, organizational processes — as a control structure. Unsafe control actions arise not just from component failures but from incorrect interactions between functioning components.

The Alignment Gap

Misalignment produces negative externalities: transport robots blocking wheelchair ramps, social media optimizing for engagement while producing teenage depression. The objective function optimizes correctly, but the system fails at the safety level. Hazard analysis is the methodology for identifying these gaps before deployment.

5. Planning for Mistakes — Five Mitigation Strategies

Lecture 4 presents five strategies for handling model mistakes, ordered by decreasing human involvement:

StrategyMechanismStrengthWeakness
1. Human-in-the-Loop Human confirms or overrides every decision Maximum safety for critical decisions Fails at scale; automation bias after 15 minutes of rubber-stamping
2. Guardrails Constrain the system’s action space Prevents catastrophic outcomes without human bottleneck Cannot catch novel failure modes outside defined constraints
3. Doer-Checker Independent component validates decisions Catches errors the model cannot self-detect Requires true independence — two models trained on same data are correlated, not independent
4. Fail-Safe Defaults When uncertain, default to the safe option Handles unknown unknowns gracefully Over-conservative defaults reduce system utility
5. Undo / Appeal Allow affected parties to reverse decisions Legally required (GDPR, EU AI Act); catches long-tail errors Post-hoc — damage may already be done

Case Studies

Apple Card Discrimination. Goldman Sachs’ algorithm gave women lower credit limits than men with identical credit profiles. Apple support response: “It’s just the algorithm. We’re not trying to discriminate.” No appeal mechanism, no human override path, no fail-safe for bias detection. This is what happens when none of the five strategies are implemented.

Post Office Horizon Scandal. Fujitsu’s accounting software incorrectly flagged 600+ postal workers for stealing. Workers were prosecuted based on “the computer said so.” A British judicial scandal — now a case study in software accountability and the absence of undo mechanisms.

Target Canada — $5.4B Data Quality Failure. 75,000 products entered by inexperienced staff with no validation rules. Only ~30% of entries were correct. Wrong product dimensions meant items wouldn’t fit in shipping containers, shelves went bare, and reorders never triggered. All 133 Canadian stores closed. Stock price rose on the announcement.

The Doer-Checker in Practice

The autonomous train example from Lecture 4: the ML model sets speed (doer); a gyro sensor detects excessive cornering force (checker); the braking system activates (recovery). Independence is critical — the checker must use a different modality, different training data, or a fundamentally different algorithm. LLM-as-judge is a doer-checker instance: the judge model should differ architecturally from the generator.

Christian Kastner teaches the systems-thinking lectures: correctness, risk, deployment, data quality, safety, and operations. His recurring theme: “I don’t like it when an answer to ‘how do we make the system better’ is ‘make the model better.’ Because it’s good to make the model better, but it doesn’t actually fundamentally remove most risks.”

The CMU course provides the theoretical foundation for how we think about verification and validation in agentic systems. Here is how the 16-lecture arc maps to concrete engineering practice.

The Open Problem: Nondeterministic Testing

The oracle problem from Lecture 7 and the silent-bug problem from Lecture 8 combine into the central challenge of testing agentic systems: outputs vary between runs, so assertEqual(agent_output, expected) is not viable. The CMU answer, synthesized across lectures:

  • Metamorphic testing (L7) — define input-output relationships, not exact outputs
  • Property-based testing (L7) — define constraints any valid output must satisfy
  • Pipeline modularization (L8) — isolate deterministic components and test conventionally; use invariants only at the nondeterministic model boundary
  • Stubs (L8) — replace the nondeterministic model with a deterministic fake during integration testing
