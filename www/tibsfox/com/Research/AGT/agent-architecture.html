Agent Architecture: Seven Convergent Signals

Building agents is 80% non-glamorous plumbing work and 20% AI. The most common failure mode in agentic systems is not underengineering. It’s actually overengineering. Premature complexity is where most projects go to die. — Nate B Jones, on the Claude Code architecture leak

Seven independent sources — an MIT inference paper, a $2.5B production system leak, a recursive composition framework, a stem-cell metaphor from Amazon/Berkeley, a Stanford/MIT meta-harness study, a practitioner deep-dive on Claude Code, and a Warp CEO’s industry survey — converge on the same architectural principles. This page synthesizes the key claims, maps them to each other, and shows the coherent picture that emerges.

1. Recursive Language Models (MIT)

Source: MIT RLM paper — inference paradigm for arbitrarily long contexts
Key idea: Give the LLM a REPL, not a bigger context window

Recursive Language Models are not a new architecture but an inference scaffold. Instead of dumping an entire massive context into the LLM’s window, the RLM gives the model a Python REPL and lets it write code to explore, decompose, and transform prompt data programmatically — like a human in a Jupyter notebook.

Three Technical Breakthroughs

MechanismWhat It DoesWhy It Matters
Symbolic variable return Sub-agent results return as Python variables in the parent’s REPL, not raw text dumped into context Eliminates context pollution — the inverse of how most agent frameworks work
Recursive self-calling The LLM calls llm_query() to spawn child RLM agents, which can themselves recurse with configurable depth limits Natural task decomposition without a fixed orchestration graph
Non-autoregressive output Answers constructed inside Python variables rather than token-by-token generation Theoretically unlimited output length; avoids counting errors

RLMs outperform RAG, ReAct, CodeAct, and sub-agent architectures on long-context benchmarks. The cost structure is input-heavy (cached) with minimal output tokens — inverting the typical agent cost profile. The key insight: the LLM controls what enters its own context window through code.

2. The 12 Primitives

Source: Nate B Jones — “I Broke Down Anthropic’s $2.5 Billion Leak”
Key idea: The harness is the product, not the model

Claude Code’s leaked architecture reveals 12 primitives organized in two tiers: Day One Basics (non-negotiables) and Operational Maturity (production lessons). The $2.5B run-rate product is sustained primarily by back-end engineering, not novel AI techniques.

The 12 Primitives at a Glance

#PrimitiveCore Function
1Tool RegistryMetadata-first capability catalog — 207 commands + 184 tools, loaded on demand
2Permission SystemTiered trust (built-in > plug-in > skills) with an 18-module security architecture for Bash alone
3Session PersistenceFull engine reconstruction from stored JSON after crashes
4Workflow StateTask state separate from conversation state — what step we’re in, not what we’ve said
5Token BudgetHard limits with pre-execution budget checking; stops before overspend
6Structured StreamingTyped events (tool match, crash reason) — a black box for every run
7System Event LoggingProvable reconstruction of any agent run: context loaded, permissions granted, actions taken
8Dual-Level VerificationL1: did the agent complete? L2: do harness changes preserve safety invariants?
9Tool Pool AssembliesSession-specific tool selection from 184 available, not all-or-nothing
10Transcript CompactionAuto-compress conversation history while preserving instructions
11Permission Audit TrailThree handlers (interactive, coordinator, swarm) with full provenance
12Agent Type SystemSix typed agents (Explore, Plan, Verify, Guide, General, Status) with constrained capabilities

The Critical Separation

Almost every agentic framework conflates conversation state with task state. Claude Code treats them as fundamentally different problems. Resuming a conversation is not the same as resuming a workflow. This single architectural decision prevents duplicate writes, double-sent messages, and rerun expensive operations after crashes.

3. Meta-Agentic Bootstrapping (p3nGu1nZz)

Source: Dr. Dimitrios Geromichalos — three videos on recursive composition, self-organizing intelligence, and SDD
Key idea: Embed an already-capable agent inside a rule-based world that governs success

The meta-agentic bootstrapping pattern has two parts: a domain-specific scaffold (the governance layer) that defines what success looks like, and an inner coding agent (the execution layer) whose job is to work within those constraints. The scaffold is “the boss” — it sets the rules of the game.

The Self-Correcting Engine

  1. Agent builds something
  2. External validator (not the agent itself) sweeps against scaffold rules
  3. Failures produce targeted error signals sent back to the agent
  4. Agent repairs and retries until all quality gates pass

Four Mechanisms

MechanismFunction
Artifact grammarStrict structural definitions that constrain the solution space
Externalized correctnessThe agent does not grade its own homework; outside validators do
Error-as-feedback loopErrors become targeted learning signals, not terminal failures
Division of laborScaffold governs, agent executes; separation is non-negotiable

Recursive Composition

The scaffold + agent + validation pattern can itself become a single component. You nest meta-agents inside each other, creating layers of governance and execution. The question shifts from “what tools should we give this model?” to “what kind of world, with what rules, for what kind of agent?”

The companion SDD (Specification Driven Development) paper formalizes this: the spec is a first-class artifact that dictates the tests, the tests guide the code, and “spec debt” is as dangerous as technical debt. The MDA priority framework — Mechanics first, then Dynamics, then Aesthetics — orders what to specify and when.

4. Pluripotent AI

Source: “STEM Agent” — Amazon + UC Berkeley, March 2026
Key idea: Don’t build specialized agents — build one that self-differentiates

Borrowed from developmental biology: a stem cell is undifferentiated yet capable of specializing into any cell type. The STEM agent starts with no preconceived domain knowledge — pure reasoning capability — and self-differentiates based on environment and problems encountered. It discovers capabilities at runtime through MCP, then crystallizes specialized handlers as needed.

The Differentiation Lifecycle

StageBiological AnalogAgent Behavior
CrystallizedCell fate commitmentPattern detected from episodic memory
CommittedProgenitor cell3 successful activations — can shortcut the pipeline
MatureFully differentiated10+ successes — gets priority routing
ApoptosisProgrammed cell deathSuccess drops below threshold — skill is pruned

The architectural theorem: by forcing all domain capabilities through an MCP layer, reasoning logic becomes strictly orthogonal to domain knowledge. You can update the LLM’s meta-reasoning without touching tools, or vice versa. The decoupling is mechanical, not aspirational.

The 8-phase cognitive pipeline (perceive, adapt, skill match, reason, plan, execute, format, learn) runs against 10 behavioral parameters including reasoning depth, exploration vs. exploitation balance, and self-reflection frequency. A caller profiler learns a 20+ dimensional representation of each human user via exponential moving average, silently in the background.

5. The Meta-Harness Pattern

Source: Stanford RISC Lab + MIT — Omar (DSPy creator) et al., March 30, 2026
Key idea: Improving the code around the LLM can matter as much as improving the LLM itself

A harness is the stateful program wrapping a language model that determines what context the model sees at each step — RAG pipelines, graph RAG, formal reasoning engines, vector databases, agentic loops, all orchestration logic. The meta-harness is one level above: a second LLM that reads the entire file system of prior candidate source code, execution traces, and evaluation scores, then proposes optimized harness configurations.

How It Works

  1. Read the full file system — all prior candidates, execution traces, scores. A single evaluation can produce up to 10 million tokens of diagnostic information. No summarization allowed; any lossy compression degrades results.
  2. Causal reasoning over logs — which sequences to analyze, which dependencies matter, which steps caused failures
  3. Propose new harness code — local edits or substantial rewrites of orchestration logic
  4. Evaluate and loop — write everything back to the file system (exceeds any context window), repeat

The meta-harness treats harness optimization as a policy search problem in code space: given a frozen core LLM M and a task distribution X, find the harness H that maximizes expected reward. It achieved 4.7–7.7 percentage point improvements across text classification, mathematical reasoning, and coding benchmarks.

Safety Constraints

  • Skills constrain outputs and safety-relevant behavior, not the diagnostic procedure
  • Verifiable reward systems are essential — the outer LLM only works well where correctness is deterministic
  • Full observability without summarization — any information reduction causes the harness engineering to fail
  • The harness controls what the core LLM is allowed to see — absolute gatekeeper function

6. Agent Orchestration 2026

Source: Zach Lloyd, Warp CEO (former Google Principal Engineer) + Claude Code deep dive
Key idea: The industry is converging on convoy/mayor patterns

Zach Lloyd argues that 2026 is the year agent orchestration moves from individual laptops to cloud-based, team-visible, programmable infrastructure. Developers routinely run 3–5 concurrent agents against large codebases (~1M lines), and laptops are the bottleneck. Five orchestration patterns are crystallizing across the industry:

PatternDescriptionIndustry Examples
Parallel single-threaded Multiple independent agents in cloud sandboxes, each on a separate task Warp Oz, Stripe Minions, Ramp
Named agents / scheduled Version-controlled skills running on cron — auto-updating docs, dead-code cleanup Warp named agents, GitHub Agents
Agent-as-app-backend Agent capabilities embedded in TUI/web apps for issue triage, de-duplication Power Fixer (open source)
Human-in-the-loop handoff Agents do 80% of work, engineer steers from phone/web Warp cloud handoff
Multi-agent coordination Agents orchestrating each other via APIs/SDKs for complex collaborative tasks Next frontier (emerging)

The Claude Code deep dive confirms the primitives needed: the agentic loop (context gathering, action execution, result verification), the Bash tool as universal capability bridge (replaces dozens of bespoke integrations), skills as loadable expertise (markdown with YAML front matter, injected just-in-time), and plan mode separating thinking from doing. The three-tier memory hierarchy (user-level, project-level, auto-memory) enables session resumption with full context restoration.

The required infrastructure primitives converge across both sources: environments, hosting, tracking, handoff, and programmability. Long-running tasks require agents that survive beyond context windows. Multi-agent teams (not just parallel singles) will emerge for hard tasks.

7. Implications for Our Architecture

Six cross-source patterns — unanimous or near-unanimous agreement across independent researchers

Pattern 1: Scaffold > Model (Unanimous — 7/7 sources)

Every source converges: the harness/scaffold/environment matters more than the raw model. The 12 Primitives analysis says it directly: “80% plumbing, 20% AI.” Meta-agentic bootstrapping frames it as “the difference between giving someone a hammer versus putting them in a fully equipped workshop with a supervisor and a blueprint.” The self-organizing intelligence paper argues complex intelligence emerges from simple rules, not from a more powerful central controller. The meta-harness paper proves it empirically: improving the code around the LLM yields measurable gains with a frozen model.

Pattern 2: Externalized Verification (5/7 sources)

The agent should not grade its own homework. Meta-agentic bootstrapping uses external validators scanning against scaffold rules. The 12 Primitives reveal dual-level verification (agent checks work + harness checks harness). SDD ties tests one-to-one to spec criteria. The meta-harness requires verifiable reward systems where correctness is deterministic. RLMs use the REPL itself as a verification environment where code either runs or it doesn’t.

Pattern 3: Recursive Composition (4/7 sources)

Systems that compose themselves from smaller validated units. Meta-agentic bootstrapping nests scaffold+agent+validator as single components. RLMs spawn child agents that return symbolic variables to parents. Pluripotent AI differentiates and re-differentiates through a lifecycle. The meta-harness uses inner/outer sphere nesting. The key question shifts from “what tools?” to “what kind of world?”

Pattern 4: State = Files, Not Memory (5/7 sources)

Agent identity and state should be durable, model-independent, and filesystem-based. The 12 Primitives persist sessions as JSON files with full engine reconstruction. The meta-harness writes everything back to the filesystem because it exceeds any context window. The Claude Code deep dive describes the three-tier memory hierarchy as filesystem artifacts. Agent orchestration 2026 requires state that survives beyond context windows.

Pattern 5: The Simplicity Imperative (4/7 sources)

The most powerful insight that cuts against over-engineering. The 12 Primitives analysis warns: “premature complexity is where most projects go to die.” Self-organizing intelligence shows complex behavior from “a swarm of simple things, each just following simple local rules.” Claude Code’s Bash tool — a single tool that bridges thousands of CLI utilities — outperforms bespoke integrations. RLMs use a standard Python REPL rather than a custom orchestration protocol.

Pattern 6: The Governance/Emergence Tension

The deepest architectural insight from synthesizing all sources: governance (scaffold, spec, constraints) and emergence (self-organization, local rules) are not competing philosophies but complementary forces in tension. The self-organizing intelligence paper argues for structure emerging from learning. Meta-agentic bootstrapping argues for strict governance layers. Pluripotent AI resolves the tension: the core is undifferentiated (emergence), but the differentiation signals come from the environment (governance). The system works when the forces balance.

The Unified Architecture Frame

GOVERNANCE LAYER (scaffold, spec, constraints)
  |
  |-- SCAFFOLD: rules, artifact grammars, success criteria
  |-- COMPOSE: nests governance+execution as recursive components
  |
EXECUTION LAYER (agents using atomic primitives)
  |
  |-- QUERY, COMPUTE, OBSERVE, COMPARE, CONNECT, PUBLISH, COMMIT, VERIFY
  |
INFRASTRUCTURE LAYER (harness providing 12 primitives)
  |
  |-- Registry, Permissions, Session, Workflow, Budget, Streaming,
  |-- Logging, Verification, Tool Pools, Compaction, Audit, Types
  |
EMERGENCE LAYER (self-organizing properties)
  |
  |-- SYNCHRONIZE: phase alignment between components
  |-- EMERGE: structure arising from local rules
  |-- DIFFERENTIATE: pluripotent specialization lifecycle

These four layers are not a hierarchy to implement top-down. They are forces in tension. Governance constrains execution. Execution demands infrastructure. Infrastructure enables emergence. Emergence challenges governance. The system works when the tensions balance — like particles finding their energy minimum.
