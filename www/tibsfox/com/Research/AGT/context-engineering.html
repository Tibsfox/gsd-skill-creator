1. The Context Crisis

The agentic era introduced an unexpected resource constraint: context windows are consumed before the first user message arrives. Tool definitions, system prompts, agent instructions, and memory documents compete for the same fixed budget that reasoning needs to occupy.

Tool Definition Bloat

Adam Jones at Anthropic (AgenticAI Foundation, 2026) demonstrated that MCP tool definitions alone can consume 100,000 tokens — half of a 200K context window — before any work begins. With tool selection accuracy collapsing from 43% to 14% under bloated toolsets, the problem is not just resource waste but active quality degradation.

Context Rot Starts Early

Chroma Research (via the AI Agent Handbook, R9) found that context quality degradation begins at approximately 25% fill — not at 100%. The assumption that a 200K window gives you 200K of useful space is wrong. Effective capacity may be closer to 50K before accuracy begins to erode.

The Compaction Death Spiral

When the window fills, compaction events discard 60–70% of conversation per event. After 2–3 compaction cycles, cumulative loss reaches 88–95%. Ivett Ordog (Queue #38) described the resulting sawtooth pattern: the window fills, compacts, drops, fills again — each cycle losing fidelity. Agent MD files and memory documents that survived the first compaction consume space that the second compaction needs for actual work.

The Compaction Cascade

Cycle 1: System prompt + agent MD + tool defs fill to ~60%. Compaction removes 60–70% of conversation. Surviving context still includes all static documents.

Cycle 2: Compacted context + new work fills remaining space. Second compaction has less useful material to preserve. Static documents crowd out working memory.

Cycle 3: By this point, 88–95% of the original session intent is lost. The agent is operating on fragments.

2. Context as Formula

Chris (Queue #34, Clip Cannon / OCR Provenance) argues there is exactly one correct framework for building AI systems. It reduces to a formula:

Output = AI_Model(Context_Window(Start, End, Prompt, Memory))

Where:
  Start  = current system state (loaded into context at session start)
  End    = desired output state (defined as precisely as possible)
  Prompt = the bridge description between start and end
  Memory = external corpus accessible via RAG with provenance

Define the Output First

The key inversion: start from what you want, not from what you have. Chris maintains 16 documents describing his codebase, loaded into Claude’s 1M context window at session start. “Read these documents and then stop. So you can understand the system.” The AI needs both anchors — start and end — to build a high-probability connection.

Epistemic Symmetry

The theoretical foundation: given an answer or output, LLMs can generate the questions that would have produced it with claimed >90% semantic similarity. Question and answer become bidirectional. This is why defining the end state clearly is more important than crafting the perfect question — the model can derive the reasoning path from the destination.

Full State Verification

After the AI builds an initial connection, you must stop and verify every transformation. Chris calls this “full state verification” and insists: never proceed to the next step until the current step is verified. This is the critical counterweight to the formula — the model finds the path, the human validates each step.

“AI will get me most of the way there, but then I will need to do full state verification — manually testing everything and fine-tuning at each step till I get everything perfectly just the way I want it before I move on to the next step.” — Chris, Queue #34

Anti-Pattern: Parallel Generation

Chris argues explicitly against generating multiple competing solutions. Five parallel agents create five crude connections; you then evaluate all five. Better: make one connection and iteratively optimize it. The exception — and Chris does not address this — is parallel agents working on independent start-end pairs, which avoids the evaluation bottleneck.

3. LightRAG & Graph RAG

Standard vector RAG (Pinecone, Supabase, 2024–2025 era) chunks documents, embeds them, and retrieves by cosine similarity. LightRAG (Queue #9) adds a second layer: during ingestion, it extracts entities and relationships to build a knowledge graph.

Why Graph Matters

Vector search is a sophisticated ctrl+F — it finds semantically similar chunks but cannot answer cross-document questions. A knowledge graph connects “Anthropic” to “Claude Code” via a “created” edge. Traversing these edges surfaces connections that pure embedding similarity would never discover.

Provenance Is Non-Negotiable

The OCR Provenance system (Chris, Queue #34) demonstrates a critical principle: every chunk must link back to its source document. Standard RAG returns a chunk with no context. Provenance-based RAG returns a chunk plus the original document location, allowing the model to navigate to full context when needed. Chris built 13 embedders including custom temporal and cause-effect embedders where causes and effects are adjacent in semantic space.

Skills as RAG Endpoints

The integration pattern wraps LightRAG’s REST API (Docker container on port 9621) as Claude Code skills. Query, upload, explore, and status endpoints become auto-activating capabilities. The system can also run fully local via Ollama for both embeddings and LLM work.

4. Token Optimization

Three distinct approaches to reclaiming context space, each targeting a different layer of the problem.

Ghost Tokens (token-optimizer, R8)

Ghost tokens are placeholders that represent compacted information without consuming full token budgets. Combined with progressive checkpoints at 20% / 35% / 50% fill levels (lightweight snapshots rather than full compaction), and a read-deduplication layer saving 8–30%, the approach aims to flatten the compaction sawtooth into a gentler curve.

The critical finding from R8: context quality drops from 93% to 76% MRCR accuracy at 256K fill. This is the empirical basis for moving compaction triggers earlier.

98% Compression via Sandbox (context-mode, R10)

The context-mode project achieves 98% compression (315KB down to 5.4KB) by virtualizing the context into a sandbox with FTS5/BM25 retrieval. Instead of loading full session history into the window, only the segments relevant to the current query are retrieved. This transforms the context window from a flat buffer into an indexed store.

Progressive 3-Tier Skill Loading (R9)

The AI Agent Handbook documents a pattern achieving 94% token reduction for skill-heavy sessions:

TierContentWhen Loaded
Tier 1Description stub only (~50 tokens)Always — session start
Tier 2Full skill definition (~500–2K tokens)On first use
Tier 3Reference docs, examples, edge casesOn explicit invocation

This is lazy loading for agent instructions. Sub-agent context isolation provided an additional 90.2% improvement by preventing skill bleed between agents in multi-agent sessions.

5. Cognitive Load

Ivett Ordog (Queue #38) frames the context problem through cognitive science: “If it doesn’t fit your head, you can’t reason about it.” The principle applies equally to humans reviewing AI output and to AI agents processing their context.

The Developer Burden

Chris Lattner (Queue #31) observes that AI tools shifted the engineering role from 20% thinking / 80% coding to 80% thinking / 20% coding — “quite taxing” in his words. Kilo (Queue #3, 25 trillion tokens across 1.5 million developers) reports that half of all developers do not use AI every day. The trust ladder — autocomplete, chat, single agent, orchestration — breaks at any rung when latency exceeds 200ms or when the agent edits the wrong file.

Agent MD Files Hit Diminishing Returns

Ordog cites research asking “Are Repository Level Context Files Helpful for Coding Agents?” — the answer: not really. The mechanism: context window fills with system prompt + agent MD + user prompt + tool calls + file reads. Larger agent MD files compress the working space available for reasoning. The agent becomes “a day worker who reads all the books, does the job, and leaves” — but the books crowd out the work.

Habit Hooks as Lazy Context Injection

Ordog’s solution: move behavioral rules out of static context files and into deterministic triggers. A habit hook is a post-commit script that detects a specific code smell (duplication, feature envy, functions >15 lines) and injects a short action plan into the agent’s context only when relevant. This is demand-driven context injection — the difference between eager and lazy evaluation applied to agent instructions.

The 80% Demo Problem

Lattner identifies a failure mode where someone builds an 80% demo with AI tools and declares success. “Having some crazy thing that’s really exciting and very cool is not actually success. What is success is when code gets in the product and ships.” The cognitive load of closing the remaining 20% — tests, integration, edge cases — is where AI amplifies existing discipline or the lack of it.

6. The Evaluation Problem

Jess from Braintrust (Queue #2) makes the case that most teams ship AI features based on “vibes” — a PM trying a few prompts and saying it looks good. This does not work.

10–15% Non-Determinism

Identical eval configurations can swing 10–15% across runs due to LLM non-determinism. With small datasets (10 rows), a single failure shifts the score by 10%. Multiple trials per task are essential. Jess explicitly states she would not publish results from a single run.

The Four-Part Eval Structure

ComponentPurposeAnti-Pattern
DatasetGolden cases, edge cases, failure modesTiny samples that swing on one failure
TaskPrompt + model configurationTuning to benchmarks instead of real workloads
ScoringDeterministic, LLM-as-judge, or humanexpect(result).toBeTruthy()
ExperimentDataset + task + scorer = one runSingle-run conclusions

Agentic Search vs. Vector Search

In Braintrust’s experiment: agentic search (Claude Code using grep, find, cat to follow references) matched or beat vector search on accuracy (68% vs 60% on SWE-Bench/Django) while being cheaper in tokens. The insight: vector search provides proximity but lacks connective tissue — import chains, call graphs, type definitions — that agentic search discovers through chain-of-thought exploration.

Drift as Organizational Risk

Al Shalloway (Queue #36, AmpGPT) identifies drift as the primary deployment risk for axiom-based AI systems. When organizational context layers on top of foundational reasoning without preserving the connection to first principles, the system diverges. His three-layer architecture (objectives / capabilities / practices) with drift detection built in addresses this. The parallel to context engineering: agent instructions that drift from their original intent after compaction cycles produce the same category of failure.
