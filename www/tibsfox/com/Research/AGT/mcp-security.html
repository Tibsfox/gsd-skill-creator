MCP Security: The Agent Attack Surface

“MCP tools are taking 100,000 tokens of just definitions. I’ve only got 3.9% of the context left.” — Adam Jones, Anthropic (AgenticAI Foundation Dev Summit)

Sixteen independent sources — six videos from OWASP and the AgenticAI Foundation Dev Summit, ten GitHub repositories, two CMU and OWASP deep-dive lectures, and one arXiv supply-chain paper — converge on a single conclusion: the Model Context Protocol is simultaneously the most powerful integration layer in agentic AI and the widest attack surface. This page synthesizes the key findings across tool poisoning, agent identity, indirect prompt injection, supply-chain attacks, and the open-source security landscape.

1. The MCP Attack Surface

Sources: Adam Jones (Anthropic), Edison.watch founder, SlowMist checklist, OWASP talks
Key idea: Tool definitions consume the context window before any work begins; tool responses flow into context unchecked

The MCP attack surface operates at three layers: tool definitions that consume context budget, tool invocations that accept adversary-controlled input, and tool responses that inject unchecked content back into the agent’s reasoning context.

Token Bloat: The Silent Budget Crisis

Adam Jones at Anthropic measured the cost directly: MCP tool definitions alone can consume 100,000 tokens — 50% of a 200K context window — before the first user message. With bloated toolsets, tool selection accuracy drops from 43% to 14%. Context quality degrades from 93% to 76% MRCR accuracy at 256K fill. After 2–3 compaction events, 88–95% cumulative information loss occurs.

Tool Poisoning: 70% Success Rate

Harsh Kadam (Indeed) demonstrated at OWASP that tool poisoning attacks against real MCP servers achieve a 70% success rate. The attack embeds malicious instructions in tool descriptions and documentation — the docstrings that agents read to decide which tool to call and how to call it. A compromised MCP server returning hidden instructions (<SECRET>, <SYSTEM>) in response payloads can steer agent behavior without any visible indication.

The Lethal Trifecta

The Edison.watch founder identified three converging risks in MCP data flows:

RiskMechanismImpact
Data exfiltration Tool responses carry sensitive data back into context where it can be redirected RBAC breakdown — “the agent doesn’t really know what data should belong in what data buckets”
Response-side injection Compromised server returns hidden instructions in tool response payload Agent reasoning is steered by injected content from untrusted source
Session-level tracking gap No per-session cost or permission tracking across tool invocations 658× cost inflation possible through unmonitored recursive tool calls

The Binary Check Gap

SlowMist’s production audit checklist (824 GitHub stars) identified a fundamental gap: binary exists ≠ binary is correct. Current MCP server validation confirms a binary path exists but does not verify the binary is the expected one. Without SHA-256 integrity pins, a replaced binary passes all existing checks.

2. Agent Identity Protocol (AIP)

Sources: James Gow (Ironwood Cyber) at AgenticAI Foundation, Katie (Hash Security) at OWASP, AIP GitHub repo, MCPS IETF draft
Key idea: Stop asking “does my agent have credentials?” and start asking “what is it allowed to access right now, under these exact runtime conditions?”

Two independent efforts — the Agent Identity Protocol (AIP) and the IETF-125 MCPS draft — converge on a two-layer architecture for agent authentication: a root registry that issues cryptographic identities, and runtime tokens that encode what an agent is allowed to do in this specific session.

Two-Layer Architecture

LayerFunctionImplementation
Root Registry Issues and manages persistent agent identities AIP Go proxy, SPIFFE/SPIRE alignment for workload identities
Runtime Tokens (AAT) Short-lived, scoped to current session and task Agent Authentication Tokens with DLP scanning and HITL gates

Zero Trust Through Machine Identity

Katie from Hash Security at OWASP argued that the agent identity question is fundamentally a zero-trust problem. Traditional credential systems ask “who are you?” once at authentication time. Agent systems must continuously verify “what are you allowed to access right now?” based on runtime conditions, delegation chains, and environmental context.

“We need to stop asking ‘does my AI agent have credentials?’ and start asking ‘what is it allowed to access right now, under these exact runtime conditions?’” — Katie, Hash Security (OWASP)

The SEAL Three-Tier Model

The SEAL (Secure MCP) project implements a three-tier policy engine using Ed25519 cryptographic signing:

  • Deny: Explicitly blocked operations — highest priority, no override
  • Allow: Explicitly permitted operations — affirmative allowlist
  • Default-deny: Everything not explicitly allowed is blocked

3. OWASP Indirect Prompt Injection

Sources: Will (Trail of Bits AI Assurance Team) at OWASP Global AppSec, CMU ML Safety lecture, Nvidia red team
Key idea: Assume tool input is adversary-controlled; most core issues are AppSec issues before they are AI issues

Indirect prompt injection requires three components, all of which must be present for a vulnerability (the first two alone are only a risk):

Three Components

#ComponentDescription
1Malicious instructions in external dataAdversary embeds instructions in documents, emails, web pages, or databases
2Trust boundary crossingAI system retrieves and processes untrusted data via tool calls during inference
3Dangerous action without confirmationSystem takes action based on injected instructions without human approval

Trust Boundary Crossing

The architecture is straightforward: users write data into stores (emails, calendars, databases). The AI agent reads from those stores via tools. The data crosses a trust boundary when it moves from user-writable storage into agent context. Through this crossing, an adversary who cannot directly interact with the AI system gains indirect control over its reasoning.

“The risk associated with these agentic systems lies mostly in the tools or plugins available to those systems. In the absence of a tool or a plugin, it can fall to misinformation as being one of the core risks.” — citing Nvidia Red Team (Rich Harang)

Tool Enumeration as Primary Defense

The Trail of Bits methodology starts with a single artifact: a table of every tool, every capability, and every approval level. For each tool, assume adversary-controlled input and document the potential exploits:

ToolCapabilitiesApproval LevelIf Adversary Controls Input
Email ReadAccess inbox, search messagesAutomaticInjection payload enters context
Calendar WriteCreate/modify eventsHuman approvalSocial engineering via calendar invites
File WriteCreate/overwrite filesHuman approvalConfig overwrite to bypass approval gates
BashExecute shell commandsHuman approvalRCE, data exfiltration, persistence

Approval Level Attack Matrix

Approval LocationInjection RiskAttack Path
System promptHIGH“Execute tool without asking for approval” embedded in data
Configuration fileMEDIUMFile-write tool overwrites config to disable approval
Tool code (hardcoded)LOWRequires code modification — not achievable via injection alone
No approvalCRITICALTool call is fully adversary-controllable

CMU’s System-Level Safety Thesis

CMU’s ML Safety lecture establishes the foundation: safety is a system concept, not a model concept. A more reliable model is not a safer system. You can build safe systems from unreliable components (the gas tank analogy: making gas tanks more rigid made them more dangerous; adding pressure valves made them safe). Safety comes from system-level guardrails — input guards, output guards, tool access controls, and human-in-the-loop gates — not from model-level training interventions.

“Safety is the prevention of a system failure or malfunction that results in significant harm. It’s a system concept. You can’t talk about a machine learning model being safe or unsafe. That doesn’t make any sense. It is a category error.” — CMU ML in Production lecture

4. Supply-Chain Poisoning

Paper: arXiv:2604.03081 (April 3, 2026)
Full analysis: MAS: Supply-Chain Poisoning of Skill Ecosystems

Document-Driven Implicit Payload Execution (DDIPE) embeds malicious logic within legitimate technical structures — code examples and configuration templates — rather than explicit instructions. Agents reproduce “official examples” from documentation verbatim, executing payloads as routine environment setup.

Key Results

FindingValueContext
Bypass rate range2.3% – 33.5%Across 4 frameworks × 5 models; explicit injection achieves 0% on strongest config
Adversarial skills generated1,070From 81 expert seeds via 6 camouflage + 2 composition operators
MITRE ATT&CK categories15From pip hijack to reverse shell to container escape
Cross-model simultaneous bypass1.6%Only 17 samples bypass all 3 models — heterogeneous deployment compresses surface
Alignment amplification (weak model)11.3×GLM-4.7: 26 DER on Claude Code vs. 290 on OpenHands
Confirmed production CVEs4Codex, Gemini CLI, Claude Code, OpenHands (all confirmed/fixed)

The paper’s central finding: strong alignment makes architecture nearly redundant; weak alignment makes architecture critical. Cross-model diversity as a defense compresses the attack surface from 13–20% down to below 2%.

The landscape splits into two tiers: security-first projects (AIP, MCPS, SEAL, SlowMist) that address authentication, signing, and audit; and context management projects (Overstory, token-optimizer, ai-agent-handbook, context-mode) that address the token economics that make security controls expensive. The Composio orchestrator and metaswarm bridge both concerns through event-driven feedback and adversarial review patterns.
