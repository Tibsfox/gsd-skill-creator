Regina REXX

Mark Hessling’s Regina REXX is the most portable REXX implementation. It runs on Linux, Windows, macOS, OS/2, and numerous other platforms. It is ANSI X3.274-1996 compliant with extensions. Available via apt install regina-rexx on Debian/Ubuntu.

Architecture Lessons for Our Port

  • Tokenizer: Regina’s lexer handles the “no reserved words” challenge with a two-pass approach: first pass identifies clause structure, second pass disambiguates instructions vs. commands.
  • PARSE engine: Template compilation produces an internal representation that’s evaluated against the source string. Our TypeScript port should follow this pattern.
  • Numeric engine: Regina implements decimal arithmetic with configurable precision. Study rexxmath.c for the algorithm.
  • Variable pool: Hash table with separate handling for simple variables and stems. Stem resolution chains through dot-separated components.

BRexx — The Minimal Implementation

Vasilis Vlachoudis’s BRexx is critical for our GPU port. At ~15,000 lines of C, it proves that a functional REXX interpreter can be tiny. BRexx strips away everything non-essential while maintaining ANSI compliance for core features. Its memory footprint (~60KB binary) is small enough to fit in GPU shared memory.

What BRexx Teaches About GPU Porting

  • Fixed-size string buffers instead of dynamic allocation — maps directly to GPU shared memory
  • Simple hash tables for variables — implementable in GPU shared memory
  • Compact bytecode representation — fits in GPU constant memory
  • No external dependencies — essential for GPU kernel code

Open Object Rexx (ooRexx)

IBM donated Object REXX to open source in 2004. RexxLA (Rexx Language Association) maintains it. ooRexx adds classes, methods, message passing (OO-style), and collection classes to standard REXX. The OO extensions are backward-compatible — every classic REXX program runs unchanged.

Relevant for Our Port

  • Message passing in OO context parallels ARexx port messages. ooRexx’s ~send method is conceptually similar to ADDRESS.
  • Collection classes (List, Queue, Set, Bag, Table, Directory) show how to extend stems to proper data structures.
  • Concurrency support (ooRexx has threads with message-based synchronization) informs our multi-script GPU execution model.

NetRexx — REXX on the JVM

Mike Cowlishaw created NetRexx in 1996 as a JVM language that brings REXX readability to the Java ecosystem. NetRexx compiles to Java bytecode and can use any Java class directly.

Cross-Language Bridging Lessons

  • How Cowlishaw himself mapped REXX semantics to a typed host language
  • String-to-type conversion strategies for a REXX-on-JVM bridge
  • PARSE on JVM: NetRexx implements PARSE as a compiled method — study for our TypeScript approach

OS/2 REXX — Deep System Integration

IBM shipped REXX with OS/2, where it served as the Workplace Shell scripting language. OS/2 REXX had the deepest system integration of any REXX implementation before ARexx:

  • Workplace Shell scripting: Desktop objects, folders, and settings all scriptable
  • REXXUTIL: System utility function library (file operations, registry access, process management)
  • SOM integration: System Object Model bridging from REXX to compiled objects

OS/2 REXX’s relationship to the Workplace Shell is the closest parallel to ARexx’s relationship to AmigaOS — both were the scripting glue for their entire operating system.
