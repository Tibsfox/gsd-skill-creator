The ARexx port system turned the Amiga from a collection of programs into a living, scriptable environment. Every application that exposed a port became programmable. Every script could orchestrate a symphony of applications.

exec.library Message Passing

ARexx ports are built on AmigaOS’s exec.library message-passing IPC — the lowest level of the operating system.

/* Amiga exec.library structures (C) */
struct MsgPort {
    struct Node mp_Node;       /* ln_Name = public port name */
    UBYTE       mp_Flags;      /* PA_SIGNAL: signal task on message */
    UBYTE       mp_SigBit;     /* signal bit (0-31) */
    void       *mp_SigTask;    /* task to signal */
    struct List mp_MsgList;    /* pending messages */
};

struct Message {
    struct Node    mn_Node;
    struct MsgPort *mn_ReplyPort;  /* where to send reply */
    UWORD          mn_Length;
};

Any task creates a MsgPort, optionally adds it to the public port list (discoverable by name), and waits. Another task finds the port (FindPort()), sends a message (PutMsg()), and the receiver is signaled. Fast (no kernel transitions — just pointer manipulation), reliable (ordered delivery), universal (every part of AmigaOS used it).

The RexxMsg Structure

struct RexxMsg {
    struct Message rm_Node;     /* standard Amiga message header */
    APTR    rm_TaskBlock;       /* ARexx internal */
    APTR    rm_LibBase;         /* ARexx internal */
    LONG    rm_Action;          /* command type + modifier flags */
    LONG    rm_Result1;         /* return code (0 = success) */
    LONG    rm_Result2;         /* result string pointer */
    STRPTR  rm_Args[16];       /* up to 16 argument strings */
    APTR    rm_PassPort;        /* forwarding port */
    STRPTR  rm_CommAddr;        /* current host address name */
    STRPTR  rm_FileExt;         /* default file extension */
    LONG    rm_Stdin;           /* I/O redirection */
    LONG    rm_Stdout;
};
FieldPurpose
rm_ActionBitfield: RXCOMM (command), RXFUNC (function call), RXCLOSE (shutdown). Flags: RXFF_RESULT (caller wants result string), RXFF_STRING (arg is string).
rm_Args[0]Command string (e.g., "LOAD FILE picture.iff") or function name.
rm_Result1Return code. 0 = success. Maps to REXX’s RC variable.
rm_Result2Result string (Argstring with length prefix + hash). Maps to RESULT variable.
rm_CommAddrName of the target port (e.g., "IMAGEFX").

The Command Dispatch Cycle

/* Step-by-step: how an ARexx command reaches an application */

1. Script: ADDRESS 'IMAGEFX'
2. Script: "LOAD FILE picture.iff"
3. ARexx creates RexxMsg, rm_Args[0] = "LOAD FILE picture.iff"
4. ARexx sends to port "IMAGEFX" via PutMsg()
5. ARexx suspends script, waits for reply
6. ImageFX GetMsg() receives RexxMsg
7. ImageFX parses command, executes load
8. ImageFX sets rm_Result1=0, rm_Result2="loaded 640x480"
9. ImageFX ReplyMsg()
10. ARexx receives reply, sets RC=0 RESULT="loaded 640x480"
11. Script resumes

The ADDRESS Instruction

/* Set default host */
ADDRESS 'IMAGEFX'

/* Send single command to a specific host, then revert */
ADDRESS 'DOPUS.1' 'lister new'

/* Toggle between two hosts */
ADDRESS 'IMAGEFX'        /* current = IMAGEFX */
ADDRESS 'GOLDED.1'       /* current = GOLDED, previous = IMAGEFX */
ADDRESS                   /* toggle: current = IMAGEFX again */

/* Dynamic port selection */
portname = 'MYAPP'
ADDRESS VALUE portname

/* Check current host */
SAY ADDRESS()             /* returns current host name */

/* Port discovery */
IF SHOW('P', 'IMAGEFX') THEN SAY "ImageFX is running"
ports = SHOW('P')         /* list all public ports */

How Applications Implemented ARexx Support

  1. Create a named public port (e.g., "IMAGEFX")
  2. Add the port to the signal mask in the main event loop
  3. When RexxMsg arrives: parse command string, dispatch to handler, set result, reply
  4. Define a command vocabulary (LOAD, SAVE, CROP, SCALE, FILTER, etc.)
  5. Optionally: support function call mode (RXFUNC) for return values
/* Typical application command handler (pseudocode C) */
void HandleARexxMsg(struct RexxMsg *msg) {
    char *cmd = msg->rm_Args[0];

    if (strncmp(cmd, "LOAD", 4) == 0) {
        char *filename = cmd + 5;
        int result = LoadImage(filename);
        msg->rm_Result1 = result ? 0 : 10;
        if (msg->rm_Action & RXFF_RESULT)
            msg->rm_Result2 = CreateArgstring(result_str, strlen(result_str));
    }
    /* ... more commands ... */

    ReplyMsg((struct Message *)msg);
}

Real-World Multi-Application Workflow

/* ARexx: Batch process images, catalog results, update log */
OPTIONS RESULTS

/* Get file list from Directory Opus */
ADDRESS 'DOPUS.1'
'lister query' handle 'files'
filelist = RESULT

/* Process each image in ImageFX */
ADDRESS 'IMAGEFX'
DO i = 1 TO WORDS(filelist)
  file = WORD(filelist, i)
  'LoadBuffer "Work:Images/' || file || '" FORCE'
  'Scale 640 480'
  'Filter Sharpen 50'
  'SaveBufferAs JPEG "Work:Output/' || file || '"'
END

/* Update the log in GoldEd */
ADDRESS 'GOLDED.1'
'OPEN FILE "Work:Output/processing.log"'
'GOTO END'
'INSERT "Processed' WORDS(filelist) 'files at' TIME() DATE()'"'
'SAVE'

SAY "Done." WORDS(filelist) "files processed across 3 applications."

The MCP bridge: In our implementation, ADDRESS 'TOOL_NAME' resolves to an MCP tool call. The command string is parsed into tool parameters. RC maps to success/error. RESULT maps to the JSON result stringified. Port discovery (SHOW('P')) maps to list_tools. ARexx scripts become first-class MCP agents.
