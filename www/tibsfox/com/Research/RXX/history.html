1. The Problem: IBM Mainframe Scripting Before REXX

By the late 1970s, IBM’s VM/CMS had EXEC and EXEC 2 — both inadequate. EXEC was primitive: no proper subroutines, no structured control flow, &GOTO for everything. EXEC 2 added variables and &LOOP, but had no expression evaluator — adding two numbers required calling an external command. On the TSO side, CLIST was universally disliked: cryptic errors, fragile parsing.

Two major IBM environments. Both with scripting languages that couldn’t keep up with growing demands for automation. This was the landscape when Mike Cowlishaw decided to fix it.

2. Michael Frank Cowlishaw

Mike Cowlishaw (born c. 1949-1950, UK) was a researcher at IBM’s United Kingdom Laboratories in Hursley Park, near Winchester, Hampshire — a Baroque country house that IBM acquired in 1958. Home of CICS and a center for programming language research.

Cowlishaw studied mathematics at the University of Birmingham. His math background profoundly influenced REXX: arbitrary precision, decimal floating point, the insistence that 0.1 + 0.2 should equal 0.3. He was deeply familiar with PL/I — powerful but enormously complex — and understood both its strengths and burdens.

His insight: a scripting language should have the readability of English, the power of a real programming language, but the simplicity that PL/I had sacrificed. A novice should write scripts immediately; an expert should never feel constrained.

Cowlishaw began designing REXX in March 1979. Initially called REX (Reformed Executor), renamed to REXX (REstructured eXtended eXecutor). The first interpreter was written in PL/I on VM/CMS. By 1982, REXX was the official scripting language for VM/CMS. By 1983, IBM shipped it as standard in VM/SP Release 3.

His book — “The REXX Language: A Practical Approach to Programming” (Prentice-Hall, 1985, 2nd ed. 1990) — became one of the most respected programming language books of the 1980s. Cowlishaw was elected Fellow of the Royal Academy of Engineering and received an OBE. He later created NetRexx and led IEEE 754-2008 decimal floating-point arithmetic.

3. The Ten Design Principles

#PrincipleHow It Manifests
1ReadabilityPrograms readable as English prose. No sigils, no braces.
2Natural data typingNo declarations. Context determines interpretation.
3Minimal notationFew special characters. No operator overloading.
4Unlimited precisionNUMERIC DIGITS — no artificial limits on numeric accuracy.
5Content-addressable arraysStems: associative lookup by any string key.
6Extensive built-in functions70+ functions — no external library for common operations.
7Powerful parsingPARSE — first-class template-based string decomposition.
8Debugging supportTRACE — built-in execution tracing. Unique among scripting languages.
9Consistent and reliableNo surprises, no undefined behavior.
10Small implementationPractical on resource-constrained systems (512KB Amiga).

The Principle of Least Astonishment

  • No reserved words. IF, THEN, DO, SAY can all be variable names. The parser disambiguates by context.
  • Decimal arithmetic. 0.1 + 0.2 = 0.3, not 0.30000000000000004.
  • Strings are the universal type. No type system to learn, no casting. x = 42 stores the string "42".

4. The Language at a Glance

/* A REXX program readable by someone who has never seen REXX */
SAY "Enter numbers, one per line. Enter 'done' when finished."
total = 0
count = 0
DO FOREVER
  PULL value
  IF value = 'DONE' THEN LEAVE
  total = total + value
  count = count + 1
END
IF count > 0 THEN
  SAY "The average is" total / count
ELSE
  SAY "No numbers were entered."

23 instructions. 70+ built-in functions. One data type. No declarations. No compilation step. From these minimal building blocks: arbitrary-precision arithmetic, complex string parsing, dynamic code generation, structured error handling, and inter-process communication.

5. ANSI X3.274-1996: Standardization

REXX was formally standardized by ANSI in 1996 as X3.274-1996 — “Information Technology — Programming Language REXX.” The standard codified the language as implemented by IBM, with minor adjustments. It covers: clause syntax, expressions, variables, control structures, PARSE, built-in functions, conditions, and the trace facility. It does NOT cover: the ARexx port system, object-oriented extensions, or platform-specific I/O — those are extensions.

6. William S. Hawes and ARexx

William S. Hawes, based in Maynard, Massachusetts, operating as Wishful Thinking Development. Already known for WShell (advanced Amiga command shell). He recognized that the Amiga’s preemptive multitasking and exec.library message passing created a unique opportunity: if applications could communicate through a standard language and protocol, the Amiga would become a scriptable environment unlike anything on personal computers.

He chose REXX because: (1) established standard, (2) designed for scripting, (3) accessible to non-programmers, (4) powerful enough for real work.

ARexx was released commercially in 1987 (~$49.95). The critical turning point: Commodore licensed ARexx and included it as standard in AmigaOS 2.0 (1990, shipped with the Amiga 3000). Unprecedented — no other personal computer OS shipped a standard scripting language with inter-application IPC. Apple wouldn’t ship AppleScript until 1993.

7. Timeline

  1. 1979 — Cowlishaw begins REXX design at IBM Hursley. First interpreter in PL/I on VM/CMS.
  2. 1982 — REXX becomes official VM/CMS scripting language.
  3. 1985 — The REXX Language published. REXX spreads beyond IBM.
  4. 1987 — William S. Hawes releases ARexx commercially for the Amiga ($49.95).
  5. 1988 — IBM ships REXX with OS/2 1.1. Deep Workplace Shell integration follows.
  6. 1990 — Commodore includes ARexx in AmigaOS 2.0. Every Amiga application adds port support.
  7. 1996 — ANSI X3.274-1996 standardizes the REXX language.
  8. 1997 — Cowlishaw creates NetRexx (REXX on JVM).
  9. 2004 — IBM donates Object REXX to open source. RexxLA stewardship begins (ooRexx).
  10. 2026 — This project: ARexx reborn for GSD-OS. TypeScript → x86-64 → CUDA.

What REXX Invented

  • PARSE — template-based string decomposition (no prior art)
  • NUMERIC DIGITS — user-configurable arbitrary precision arithmetic
  • INTERPRET — dynamic code execution as a first-class instruction
  • TRACE — built-in execution tracing (unique among scripting languages)
  • No reserved words — keywords disambiguated by context
