What SC:Learn Does

SC:Learn is the knowledge ingestion pipeline that transforms external knowledge — documentation, code, research papers, API references — into skills, agents, and teams that Claude Code can use automatically. It is how the skill library grew from zero to 69 installed skills. When you point SC:Learn at a source of knowledge, it acquires, sanitizes, analyzes, and extracts reusable behavioral patterns, then generates skill files that auto-activate in the right contexts. The entire pipeline is gated by human approval because the skill creator is a self-modifying system.

External knowledge flows left to right through seven stages. The HITL (Human-In-The-Loop) gate at stage 3 is the critical security boundary — nothing enters the skill library without human review. The output is a set of YAML-frontmatter markdown files that Claude Code loads automatically on session start.

The Seven Stages

1
Acquire
Source detection and content retrieval

The pipeline begins by detecting and retrieving knowledge from external sources. SC:Learn accepts multiple input types and examines each for extractable content.

  • Local files and archives — individual files, directories, .zip, .tar.gz
  • GitHub repositories — clones or fetches specific paths from public repos
  • URLs and web content — documentation sites, wikis, rendered pages

What it looks for

  • Documentation (READMEs, guides, reference manuals)
  • Code patterns (idioms, helper functions, architectural patterns)
  • API references (function signatures, parameter types, return values)
  • Tutorials (step-by-step workflows, getting-started guides)
2
Sanitize
Safety first

Every piece of acquired content passes through tiered sanitization before it can proceed further. The tier depends on source trust level.

  • Credential removal — strips API keys, tokens, passwords, connection strings
  • Secret detection — pattern matching for .env values, private keys, OAuth secrets
  • PII filtering — removes email addresses, phone numbers, names from examples
  • Structure validation — ensures content is well-formed and parseable
  • Provenance tracking — every extracted item is traced back to its original source URL, file path, and retrieval timestamp
Tiered sanitization

TRUSTED sources (your own repos) get lightweight sanitization. KNOWN sources (popular open-source projects) get standard sanitization. STRANGER sources (unknown URLs, untested repos) get aggressive sanitization with additional pattern scanning.

3
HITL Gate
Human approval — non-negotiable

The Human-In-The-Loop gate is the security boundary of the entire pipeline. The security-hygiene skill enforces this gate. Nothing passes without explicit human approval.

  • STRANGER sources require explicit approval — unknown content is quarantined until reviewed
  • Content preview — the human sees exactly what will be extracted before approving
  • Rejection tracking — rejected content is logged for audit, never silently discarded
Why this gate exists

The skill creator is a self-modifying system. Skills auto-activate and influence Claude's behavior in every session. A malicious or poorly-written skill could cause real damage — from incorrect code generation to data exfiltration. The HITL gate ensures a human always stands between external knowledge and the system's behavior.

4
Analyze
Structure detection and concept mapping

Approved content is analyzed for structure, concepts, and relationships. This stage determines what kind of knowledge the content represents and how it connects to existing skills.

  • Content type identification — tutorial, API reference, architecture doc, code sample, configuration guide
  • Concept mapping — identifies key concepts and maps their relationships (e.g., "bpy.ops" relates to "Blender operators")
  • Dependency detection — finds prerequisite knowledge areas (e.g., "Kubernetes patterns" depends on "Docker patterns")
  • Confidence scoring — each extraction candidate receives a confidence score based on source quality and content clarity
5
Extract + Wire
Primitive extraction and dependency wiring

The extraction stage pulls out concrete behavioral primitives from the analyzed content and wires dependencies between them.

  • Skills — auto-activating behavioral guidance (the most common output)
  • Agents — composed expertise for complex, multi-step tasks
  • Teams — multi-agent coordination patterns for parallel work

Each extraction includes

  • Triggers — when to activate (intents, file patterns, context clues)
  • Applies-to — what files and contexts this knowledge covers
  • Safety constraints — boundaries the skill must not cross
  • Dependencies — other skills this one builds on or coordinates with
6
Dedup + Merge
Collision detection against the existing library

Before generating output, every extraction is checked against the existing 69 installed skills for overlap. This prevents library bloat and conflicting guidance.

  • Jaccard similarity matching — extractions with >85% overlap against an existing skill are flagged as duplicates
  • Complementary merge — if a new extraction adds useful content to an existing skill, it is merged rather than duplicated
  • Conflict resolution — contradictory guidance between new and existing content is flagged for human review
  • Library health — prevents the skill count from growing without bound; quality over quantity
7
Generate + Report
Output and availability

The final stage produces the actual skill files and a summary report of everything that happened during the pipeline run.

  • YAML-frontmatter markdown — the standard skill file format (details below)
  • Agent definitions — generated if composed multi-step behavior was detected
  • Team definitions — generated if multi-agent coordination patterns were detected
  • Pipeline report — what was extracted, merged, rejected, and why
Immediate availability

Generated skills land in .claude/skills/ and are available on the next session start. No manual registration or configuration is required — Claude Code discovers and loads them automatically based on their trigger conditions.

Safety Architecture

The skill creator is a self-modifying system. Skills auto-activate and directly influence Claude's behavior, tool usage, and decision-making in every session. This makes safety not just important but foundational. Three independent layers provide defense in depth.

Layer 1
Sanitization
Automated. Removes credentials, secrets, PII. Tiered by source trust level.
Layer 2
HITL Gate
Human. Nothing enters the library without explicit approval. Non-negotiable.
Layer 3
Security Hygiene
Ongoing audit. Watches for permission escalation, data exfiltration, unbounded operations.

What the security-hygiene skill monitors

  • Permission escalation — skills attempting to grant themselves broader access
  • Data exfiltration patterns — unexpected network calls, file reads outside scope
  • Unbounded operations — infinite loops, recursive expansions, runaway generation
  • Skill modification — any edit to an existing skill triggers audit review
  • Agent composition — new agent definitions are checked for safe coordination patterns

Practical Example: Learning from the Blender Docs

Here is a complete walkthrough of SC:Learn processing the Blender 4.4 Python API documentation into a usable skill.

Acquire
SC:Learn receives the Blender 4.4 Python API docs. It detects the bpy module structure, identifies 14 sub-modules (bpy.ops, bpy.types, bpy.props, etc.), and retrieves 2,400+ pages of reference material.
Sanitize
Source is classified as KNOWN (official Blender Foundation docs). Standard sanitization runs. No credentials found. 3 example snippets contained hardcoded file paths — replaced with placeholders. Provenance: docs.blender.org/api/4.4/, retrieved 2026-03-15.
HITL
Human reviews the acquisition summary: 14 modules, 2,400 pages, KNOWN trust level. Preview shows extracted operator patterns and property type definitions. Human approves.
Analyze
Content type: API reference + code samples. Key concepts identified: operators (bpy.ops.*), properties (IntProperty, FloatProperty), mesh data access, scene management. Dependency detected: requires Python basics (already covered by existing skills). Confidence: 0.92.
Extract
Extracts one primary skill: blender-scripting. Triggers: import bpy, *.blend files, intent keywords "blender", "bpy", "mesh". Includes: operator invocation patterns, property registration boilerplate, common mesh manipulation workflows.
Dedup
Checked against 69 existing skills. No overlap found (no existing Blender skills). Complementary connection noted with gpu-ecosystem skill (shared GPU compute context). No merge needed — proceeds to generation.
Generate
Generates .claude/skills/blender-scripting.md with YAML frontmatter. Report: 1 skill created, 0 merged, 0 rejected. Skill activates on next session start whenever someone works with bpy imports or .blend files.

Configuration

SC:Learn behavior is controlled through a small set of configuration options that balance thoroughness against token budget and safety.

Option Values Description
trust_level TRUSTED, KNOWN, STRANGER Source trust classification. Determines sanitization depth and HITL gate strictness. STRANGER sources require explicit approval for every extraction.
extraction_depth shallow, deep Shallow extracts skills only. Deep extracts skills, agents, and teams. Use deep for rich documentation with multi-step workflows.
dedup_threshold 0.0 – 1.0 Jaccard similarity threshold for duplicate detection. Default is 0.85. Lower values are more aggressive about merging.
output_dir path Where generated skills land. Default: .claude/skills/. Agents go to .claude/agents/.
max_skills integer Maximum number of skills to generate from a single source. Prevents runaway extraction from very large documentation sets.
provenance true, false Whether to embed source URLs and timestamps in generated skill files. Default: true.

The Skill Anatomy

Every skill generated by SC:Learn follows the same format: a YAML frontmatter block defining metadata and triggers, followed by the skill content in markdown. 

---
name: blender-scripting
description: "Blender Python API patterns for bpy operators, properties, mesh access, and scene management"
triggers:
  intents:
    - "blender"
    - "bpy"
    - "mesh scripting"
  contexts:
    - "working with Blender Python API"
    - "writing Blender add-ons"
applies_to:
  - "**/*.py"
  - "**/*.blend"
provenance:
  source: "docs.blender.org/api/4.4/"
  retrieved: "2026-03-15"
  trust: "KNOWN"
---

When invoking Blender operators, always use the poll() check pattern:

if bpy.ops.mesh.subdivide.poll():
