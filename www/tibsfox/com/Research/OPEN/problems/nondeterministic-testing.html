Non-Deterministic System Testing

What does "correct" mean for a system that gives different answers each time?

The Problem

Traditional software testing assumes determinism: given the same input, the system produces the same output. Traditional ML evaluation assumes fixed benchmarks: given a test set, measure accuracy. Agentic AI systems break both assumptions. They take multi-step journeys where each step's output depends on prior steps, tool results, and stochastic sampling. Run the same task twice and you get different trajectories, different tool calls, and potentially different final answers.

This creates a fundamental testing challenge. You cannot write an assertion that checks for a specific output, because the output varies. You cannot run a test suite and expect consistent pass/fail results, because the same test may pass on one run and fail on the next. The concept of "regression" becomes fuzzy -- did the system get worse, or did this run happen to sample an unlucky trajectory?

The problem compounds in multi-agent systems where multiple non-deterministic agents interact. The combinatorial explosion of possible trajectories makes exhaustive testing impossible. Statistical approaches (run N times, check that the success rate exceeds a threshold) are expensive and provide weak guarantees. The field needs new testing paradigms designed for non-deterministic, multi-step, multi-agent systems.

Latest Research

Agent Contracts: Testing Under Non-Determinism

arXiv:2601.08815

Proposes "agent contracts" as a testing framework for non-deterministic AI systems. Instead of asserting specific outputs, contracts specify invariants that must hold across all possible trajectories: resource bounds (never exceed N tokens), safety constraints (never call tool X without authorization), and outcome distributions (success rate above threshold over K runs). The contract framework enables property-based testing for agents, analogous to how QuickCheck enables property-based testing for functions. Achieves 90% token reduction through contract-guided early termination.

Our Connection

We face this problem every day. Our 21,298 tests are deterministic unit tests that verify specific code behavior. But the agent orchestration layer -- the Gastown chipset running multiple agents in parallel -- is fundamentally non-deterministic. We test the deterministic substrate but cannot systematically test the non-deterministic layer above it.

  • Nyquist Verification: Our approach to verification uses sampling theory -- verify enough points to reconstruct the signal. This is one approach to non-deterministic testing: statistical sampling of trajectories rather than exhaustive checking.
  • Witness-Observer: The observer pattern provides runtime monitoring that compensates for the impossibility of pre-deployment exhaustive testing. If you cannot test all paths, you observe in production.
  • Beads-State Crash Recovery: Our crash recovery mechanism provides a form of testing through resilience -- the system is tested by actual failures and must recover correctly. This is chaos engineering applied to agent orchestration.
  • ACE M5: The verification and validation milestone for the Agent Compute Engine. Non-deterministic testing is the central technical challenge.

Session 6 Research Connections

New connections discovered during the Artemis II Session 6 research sweep:

  • Deterministic Atomic Buffers from Game Engines: Game engines solve the non-determinism problem through order-independent accumulation. Physics engines compute forces from multiple sources (gravity, collision, player input) and accumulate them into a single deterministic buffer before applying the update. The order of force computation does not matter — only the accumulated result. Applied to agent testing: instead of asserting specific trajectories, accumulate the agent's effects (files created, tests passed, invariants maintained) into a buffer and assert on the buffer's final state. This transforms trajectory-dependent testing into state-dependent testing, which is deterministic.
  • Harness Integrity Tests — 16 Invariants Built: The harness integrity test pattern (discovered in Session 6 meta-harness research) defines invariants that must hold regardless of the agent's non-deterministic trajectory. Our system now has 16 such invariants: resource budget not exceeded, no unauthorized tool calls, output schema valid, state file integrity, etc. These invariants are deterministic assertions on a non-deterministic system — they never check what path the agent took, only that the guardrails held. This is property-based testing adapted for agents, and it works: 16 invariants, zero false positives across 50+ convoy runs.

What Would Constitute a Breakthrough

A testing framework that provides coverage guarantees for non-deterministic multi-agent systems with bounded compute. Specifically: given an agent system with known stochasticity parameters, determine the minimum number of test runs required to achieve a target confidence level that a safety property holds across all possible trajectories.

A practical breakthrough would be a "non-deterministic CI" system that integrates with existing test runners, automatically determines the required sample size for each test, and reports both point estimates and confidence intervals for pass rates. This would make non-deterministic testing as ergonomic as deterministic testing while providing honest uncertainty quantification.
