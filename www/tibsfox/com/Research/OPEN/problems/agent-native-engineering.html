The Problem

Software engineering conventions — variable naming, code organization, documentation formats, API design — have been optimized for human readability for six decades. Readable code, clean architecture, meaningful names, comprehensive comments: all assume a human developer will read, understand, and modify the code. But AI agents are increasingly the primary consumers of codebases. They read code to understand it, modify it to implement features, and review it for quality. Their needs differ fundamentally from human developers.

The key tension: conventions optimized for humans may be suboptimal for agents, and vice versa. Verbose comments help humans but waste context window tokens for agents. Short variable names save tokens but confuse humans during review. The research question is whether we need dual-mode conventions, agent-specific code views, or entirely new paradigms that serve both audiences.

This is not a hypothetical problem. Every AI coding assistant today navigates codebases designed for humans. The inefficiency is measurable: wasted tokens on boilerplate, confusion from naming conventions that carry implicit human context, and navigation overhead from directory structures optimized for IDE file trees rather than semantic search.

Latest Research

Beyond Human-Readable: Rethinking SE for the Agent Era

arXiv:2604.07502 · Ustynov, April 2026

Proposes "semantic density optimization" — removing informationally empty tokens while retaining high-value content. Controlled experiments on logging formats revealed a counterintuitive finding: aggressive compression increased total session cost by 67% despite reducing input tokens by 17%, because the interpretive burden shifted to more expensive reasoning. Introduces the "program skeleton" concept — a specialized code navigation paradigm for agents that exposes structure without surface syntax. Also proposes rehabilitating classical anti-patterns (global state, deep coupling) when they serve agent consumption. The core message: cost-effectiveness depends on where the interpretive burden falls, not on raw token count.

Triage: Routing SE Tasks to Cost-Effective LLM Tiers

arXiv:2604.07494 · Madeyski, April 2026

Uses code health metrics as routing signals to assign SE tasks to appropriate LLM tiers (light, standard, heavy). Key finding: medium-tier models benefit significantly from clean, well-maintained code, while frontier models show less sensitivity to code quality. This enables cost savings by routing tasks on healthy code to cheaper models. Evaluated on SWE-bench Lite with 300 tasks, deriving analytical thresholds for when routing is cost-effective. Directly validates our quality/balanced/budget profile system — their formal methodology provides the theoretical grounding for what we do operationally with gsd:set-profile.

We are a living experiment in agent-native engineering. The gsd-skill-creator codebase is simultaneously written by humans and consumed by AI agents. Our CLAUDE.md files, skill metadata, and SKILL.md/AGENT.md formats are agent-native documentation — designed to be machine-parsed. Our human documentation lives in docs/ and .planning/. The dual audience is already here.

Our Connection

GSD AI Engineering This problem is our daily operating environment.

  • CLAUDE.md as agent-native docs: Our project instructions file is pure agent-native engineering — structured for machine parsing, not human reading. It contains the exact information an agent needs to operate in the codebase, nothing more.
  • SKILL.md / AGENT.md formats: Our metadata formats use YAML frontmatter + structured markdown, optimized for machine extraction with human readability as a secondary concern. This is the dual-mode format the research calls for.
  • GSD model profiles: The quality/balanced/budget system is a manual implementation of the Triage routing framework. Clean phases with good specifications route to budget models; ambiguous exploratory work routes to quality. The research formalizes our heuristic.
  • Program skeleton alignment: Our codebase intelligence (gsd:map-codebase, gsd:scan) already produces structural summaries that function as program skeletons — agent-consumable views that expose architecture without requiring full file reads.
  • Semantic density in practice: The 67% cost increase from over-compression validates our CLAUDE.md approach: we keep key context verbose rather than compressing it, because the reasoning cost of decompression exceeds the token savings.

What Would Constitute a Breakthrough

A code representation format that is simultaneously optimal for human comprehension and agent consumption — not a compromise, but a format where both audiences extract maximum value. This likely requires a layered representation: a semantic core that agents parse directly, with human-readable views generated on demand (similar to how databases store normalized data and generate views).

For our system, a breakthrough would be automatic dual-mode rendering: the same codebase presented to humans as clean, documented TypeScript and to agents as semantic skeletons with dependency graphs and type signatures — both generated from a single source of truth. Unison's content-addressed code is the closest existing system to this vision.
