In mid-2026, Dexter Horthy gave a talk at a developer conference documenting what went wrong when thousands of engineers adopted the Research-Plan-Implement (RPI) methodology for AI coding agents. RPI is a three-phase workflow: research the codebase, create a plan, implement the code. It was widely adopted after going viral on Hacker News, with prompts used by teams from startups to Fortune 500 companies.

The talk is valuable not because RPI failed — it didn't — but because the failure modes Horthy documents are the exact problems any agent orchestration system must solve. GSD independently converged on solutions to most of them. Where we diverge is instructive.

What They Got Wrong

Wrong #1

The 85-Instruction Mega-Prompt

The original RPI planning prompt contained 85+ instructions in a single monolithic prompt. Research from late 2025 showed frontier LLMs can only follow ~150-200 instructions with consistency. Combined with CLAUDE.md, system prompts, tool definitions, and MCP server descriptions, the effective instruction budget was exhausted before the workflow even started. Critical steps — like asking the user design questions before writing the plan — were randomly skipped.

The fix: Split the single prompt into focused phases: questions → research → design → structure → plan → implement → PR. Each phase has <40 instructions.

GSD equivalent: Our skill/command system was built this way from the start. discuss-phase, plan-phase, execute-phase are separate prompts with separate context windows. The Gastown chipset separates mayor/polecat/witness into distinct roles with focused instructions. We never had a mega-prompt because the architecture prevented it.
Wrong #2

Telling the Research Agent What You're Building

If you tell the model "I'm building feature X, research the codebase," the research output contains opinions about how to build X. Good research should be pure facts — objective compression of what's true about the code. Opinions contaminate the planning phase downstream.

The fix: Separate question generation (knows the ticket) from research execution (doesn't know the ticket). One context window generates the queries, a fresh context window executes them blind.

GSD equivalent: Our gsd-phase-researcher agent gets a research brief, not the full implementation plan. The discuss-phase gathers context through questions before the researcher runs. The separation is architectural — different agents with different context.
Wrong #3

"Don't Read the Code"

"In August I said don't read the code. I was wrong. I am humble enough to admit when I was wrong." They tried not reading code for six months. Had to rip out and replace large parts of the system. A 1,000-line plan produces ~1,000 lines of code, and the code often diverges from the plan.

The fix: Read the code, not the plan. Plans are a means to an end, not a deliverable. The code is the deliverable. Review that.

GSD learning: Our verify-work and Nyquist validation steps check the actual output, not just the plan. But this is a reminder that autonomous execution still needs human code review. "Planning is the hard part" remains true, but the output still needs eyes on it.
Wrong #4

"Magic Words" Required for Basic Functionality

The planning prompt had built-in steps for asking the user design questions. But ~50% of the time, the model skipped them. Users discovered you had to add "work back and forth with me starting with your open questions and outline before writing the plan" to trigger the interactive flow. Horthy: "If you built a tool that requires hours and hours of training and reps to get good results from, go fix the tool."

The fix: Use code control flow, not prompt control flow. Classify the input, then route to the right focused prompt deterministically. The if-statement is more reliable than an instruction.

GSD equivalent: Our gsd:do command routes freeform text to the right workflow automatically. gsd:next determines the next step deterministically from project state. We use code for control flow and prompts for content — which is exactly what Horthy converged on.

What They Got Right (And We Also Got Right)

Validated

There Is No Magic Prompt

"There is no magic prompt." The search for the one perfect system prompt is a dead end. Good results come from good architecture: focused phases, appropriate context, human judgment at decision points.

GSD: Our architecture is 34 skills, 57 commands, dozens of agents — each focused on one thing. No single prompt runs the show. The orchestration IS the prompt.
Validated

Do Not Outsource the Thinking

"You the engineer are an important part of this process." The model generates options, the human chooses. The model researches, the human interprets. The model writes code, the human reviews. Agency without abdication.

GSD: Our discuss-phase exists specifically for this. Adaptive questioning surfaces decisions the human needs to make BEFORE the agent goes to work. The human steers, the agent executes.
Validated

Vertical Plans Beat Horizontal Plans

Horizontal: all database → all services → all API → all frontend. Nothing testable until the end. Vertical: mock endpoint → wire frontend → service layer → database → integration. Testable at every step. Models love horizontal plans. You have to force vertical.

GSD: Our wave-based execution IS vertical planning. Each wave delivers a testable slice. The plan-checker verifies plans are vertical before execution begins. This is one of our strongest architectural decisions.
Validated

Seek Leverage, Not Speed

"Going 10x faster doesn't matter if you throw it all away in 6 months. Shoot for 2-3x." The goal is sustainable quality at higher throughput, not raw speed. Every speedup that reduces quality is a net negative.

GSD: Our model profiles (quality/balanced/budget) exist for this. Quality mode uses Opus for critical work. Budget mode uses Haiku for routine tasks. The system self-selects the appropriate speed/quality tradeoff.

Where We Diverge — The Gastown Question

Divergence

"I'm Mid on Agent Swarms and the Whole Gastown Thing"

Horthy directly called out the Gastown convoy model as a concern. His argument: multi-agent swarms make it harder to ensure quality. You lose visibility into what each agent decided. The blast radius of a bad decision scales with the number of agents.

Our response: He's right that quality assurance is the hard problem. But Gastown has explicit answers:

• witness-observer — passive monitoring agent that watches all others without interfering
• Nyquist verification — coverage-based validation that every requirement was met
• verify-work — goal-backward analysis checking the output delivers what was promised
• atomic commits — each wave is a single commit, reviewable in isolation
• plan-checker — verifies the plan will achieve the goal before execution starts

The question isn't "should you use multiple agents?" — it's "do you have quality gates between them?" CERN's WLCG runs 1.4M cores across 170 sites with federated governance. The answer to scale isn't centralization — it's monitoring, verification, and accountability at every tier.

Action item: Document our quality gates more explicitly. Horthy's concern is valid for systems without them. We have them — we need to prove they work by measuring defect rates across supervised vs. unsupervised agent runs.

Key Numbers from the Talk

Data Points

Quantitative Findings

• ~150-200 instructions — max that frontier LLMs follow with consistency (cited arXiv paper, late 2025)

• 85 instructions — the original RPI mega-prompt size (too many for one context)

• <40 instructions — the new per-phase prompt size (well within budget)

• ~40% context usage — the "dumb zone" threshold where quality degrades (matches our token economics research)

• 1,000 line plan ≈ 1,000 lines of code — within 10%. Plans don't compress.

• 200 lines — the design discussion document. High-leverage review artifact.

• 50%+ of users — got skipped interactive steps without "magic words"

• 2-3x — the sustainable productivity multiplier target (not 10x)

• 10,000+ users — grabbed the open-source RPI prompts

Verdict: Convergent Evolution Validates the Architecture

RPI and GSD were developed independently. RPI started as prompts for Claude Code; GSD started as a full project management system with skills, agents, and chipsets. Both converged on the same core principles: separate research from planning, use focused prompts with small instruction budgets, force vertical plans, don't outsource the thinking, and verify the output.

Where GSD goes further: multi-agent orchestration with quality gates (witness, Nyquist, plan-checker), wave-based parallel execution, model profile routing, and the Gastown chipset for convoy-scale work. Where RPI v2 goes further: explicit instruction budget accounting, the 200-line design discussion artifact, and the "C header file" structure outline as a lightweight review target.

Action items for GSD: (1) Add explicit instruction counting to our skill prompts. (2) Consider a "design concept" artifact between discuss and plan. (3) Measure and publish defect rates for supervised vs. unsupervised agent runs to address Horthy's quality concern. (4) Track the ~150-200 instruction limit as a hard constraint in skill design.
