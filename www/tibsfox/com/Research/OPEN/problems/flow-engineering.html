The Problem

Early multi-agent systems used open-ended chat loops: agents talk to each other in a conversation until the task is done. This approach is flexible but unreliable. Agents lose track of the task, repeat themselves, get stuck in loops, or drift into irrelevant discussion. The conversation has no structure to enforce progress toward completion.

Flow engineering replaces chat loops with explicit workflow graphs. The workflow is modeled as a state machine: nodes are tool calls or LLM invocations, edges are permissible transitions, and the system enforces that execution follows the graph. This constrains the agent's behavior to a known set of states and transitions, making the system predictable, debuggable, and recoverable.

The open question is where to draw the line between structure and flexibility. Too rigid a workflow graph cannot handle novel situations. Too flexible and you are back to unreliable chat loops. The frontier is adaptive workflow graphs that adjust their structure based on runtime observations while maintaining enough constraints to guarantee progress.

Latest Research

AFlow: MCTS Workflow Search (ICLR 2025 Oral)

arXiv:2410.10762

Uses Monte Carlo Tree Search to automatically discover optimal workflow graphs for a given task type. Instead of hand-designing the state machine, AFlow searches the space of possible workflows and finds one that maximizes task completion while minimizing cost. ICLR 2025 Oral paper. Demonstrates that automatically discovered workflows often outperform hand-designed ones because they find non-obvious orderings and parallelizations that human designers miss.

LangGraph 1.0: State Machines for Agents

LangGraph 1.0 (2026)

Production framework for building agent workflows as state machines. Nodes are computational steps (LLM calls, tool invocations, conditionals), edges are state transitions, and the graph is the entire execution plan. Provides built-in persistence, human-in-the-loop checkpoints, and recovery from failures. The practical observation is that converting chat loops to state machines reduces failure rates by 60-80% in production deployments.

Plan-Execute-Verify-Replan

arXiv:2603.11445

Proposes a four-stage workflow pattern: Plan (decompose the task), Execute (run each subtask), Verify (check results against requirements), Replan (adjust if verification fails). This is a minimal state machine that handles the most common failure mode (incorrect execution) through verification and replanning. Shows that adding explicit verification reduces error rates by 40% compared to plan-execute-only workflows.

What Would Constitute a Breakthrough

An adaptive workflow graph that modifies its own structure at runtime while maintaining formal guarantees about progress and termination. The graph starts with a base structure and adds/removes nodes and edges based on observed execution, but a proof system ensures that the modified graph still converges to a terminal state and satisfies all safety invariants.

A practical breakthrough would be automatic workflow discovery: given a task description and a set of available tools, automatically generate an optimal workflow graph without human design. AFlow's MCTS approach is a step in this direction, but current methods require extensive simulation and do not generalize well to novel task types.
