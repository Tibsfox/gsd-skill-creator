File-Native Agent Architecture

When the filesystem IS the message bus

The Problem

Most multi-agent frameworks assume agents communicate through in-memory message passing, shared databases, or API calls. But the most reliable, debuggable, and crash-recoverable agent systems we’ve built use the filesystem as the primary coordination mechanism. Files are the messages. Directories are the queues. Git is the audit log. The question: is this a pragmatic hack, or a fundamentally better architecture for LLM agents?

The Observation

Our Gastown orchestration chipset (14 skills, 8 patterns) uses files for everything:

.gastown/
  beads/           # Work items (JSON files, one per task)
  hooks/           # Agent assignments (one file per agent, contains current work ID)
  mail/            # Async messages (directory per agent, one file per message)
  nudge/           # Sync signals (latest-wins single file per agent)
  convoys/         # Grouped dispatches (JSON manifests)
  merge-requests/  # Completed work awaiting integration

Every state transition is a file write. Every message is a file. Every agent’s identity, current task, and communication history is a readable directory listing. When an agent crashes, its state is on disk. When you want to debug, you ls and cat. When you want an audit trail, you git log.

The Deeper Claims

Claim 1: Files are the natural persistence layer for LLM agents

LLM agents already work with files: they read code, write code, read docs, write docs. The filesystem is their native environment. Using files for coordination means the same tools (Read, Write, Grep, Glob) that agents use for their primary work also handle their coordination. No separate messaging SDK. No serialization layer. The coordination protocol is the same as the work protocol.

Claim 2: Git provides free exactly-once delivery and total ordering

A git commit is an atomic, totally-ordered, content-addressed snapshot of state. Git merge is a conflict-detection-and-resolution protocol. Git push is a distributed consensus operation (fast-forward or reject). These are the exact primitives that distributed systems papers spend decades designing. Git already solved them.

Claim 3: Directory structure is a natural topic/queue abstraction

A directory is a queue. Files within it are messages. Subdirectories are topics. The filesystem hierarchy maps directly onto the pub/sub patterns that message brokers provide. mail/agent-cedar/ is Cedar’s inbox. mail/agent-cedar/msg-001.json is a message. ls mail/agent-cedar/ | wc -l is the queue depth. No Kafka. No RabbitMQ. No Redis pub/sub.

Claim 4: The GUPP model (Get Up and Push Protocol) solves the polling problem

Traditional agent loops poll for work: “Is there a task for me?” GUPP inverts this: work is hooked to an agent by writing a file to its hook directory. The agent doesn’t poll — it’s interrupted. File watchers (inotify on Linux, FSEvents on macOS) provide the interrupt mechanism. This is event-driven architecture using filesystem events as the event bus.

Open Questions

Q1: Does file-native scale beyond single-machine?

The current architecture assumes a shared filesystem (or git repo). For multi-machine coordination, you need network-accessible storage (NFS, S3, distributed filesystem) or git push/pull as the sync mechanism. At what scale does the filesystem abstraction break? Is git push the right sync primitive, or does it become a bottleneck?

Q2: What’s the latency floor?

File I/O on SSD is ~100μs for a 4KB write. inotify delivers events in ~1ms. This is fast enough for agent coordination where each agent turn takes seconds to minutes. But is it fast enough for sub-second agent interactions? Where is the latency floor compared to in-memory message passing (~1μs)?

Q3: Can file-native replace MCP for local tool servers?

MCP (Model Context Protocol) uses stdio or HTTP for tool communication. Could a file-native MCP transport work? Agent writes a request file, tool watches the directory, writes a response file. The protocol is the same; the transport changes. Would this simplify MCP server implementation?

Q4: Is there a formal model?

Can file-native agent coordination be formalized as a CSP (Communicating Sequential Processes) system? Each agent is a process. Each directory is a channel. File writes are channel sends. File reads are channel receives. If the mapping is clean, existing CSP verification tools could prove properties about agent systems.

Evidence From Production

The Gastown chipset has coordinated 50+ research projects across 4 mission days (Artemis II), handling convoy dispatch, wave-based parallel execution, merge request queuing, and crash recovery — all via file-native patterns. Key metrics:

  • Zero data loss from agent crashes (state always on disk)
  • Sub-second coordination latency (inotify + SSD)
  • Debug time: seconds (ls + cat vs log-diving)
  • Cross-runtime: Claude Code, Codex, Gemini CLI all use same file protocol
  • Audit: every state change in git history, bisectable

The WLCG has used file-based job coordination since 2009, processing ~1 PB/day across 170+ sites. Their DIRAC system submits pilot jobs as file descriptions, tracks status via filesystem state, and collects output as files. This is file-native agent architecture at planetary scale.
