Ownership & Borrowing

THE Defining Feature — Move Semantics, Borrowing Rules, Lifetimes, and the Borrow Checker
Overview Ownership Traits Async Ecosystem Unsafe Rosetta Learn History
Every value in Rust has exactly one owner. When the owner goes out of scope, the value is dropped. There is no garbage collector, no reference counting by default, no manual free(). Ownership is the mechanism.

The Three Rules of Ownership

  1. Each value in Rust has exactly one owner.
  2. There can only be one owner at a time.
  3. When the owner goes out of scope, the value is dropped.

Move Semantics

By default, assigning a value to another variable moves it. The original binding becomes invalid. This is not a shallow copy followed by a potential double-free; it is a transfer of ownership that the compiler enforces statically.

let s1 = String::from("hello");
let s2 = s1; // s1 is MOVED into s2

// println!("{}", s1); // ERROR: value used after move
println!("{}", s2);     // OK: s2 owns the string

This applies to function calls too. Passing a value to a function moves it unless the type implements Copy:

fn take_ownership(s: String) {
    println!("I own: {}", s);
} // s is dropped here

fn main() {
    let greeting = String::from("hello");
    take_ownership(greeting);
    // greeting is no longer valid here
}

Copy and Clone

TraitWhenHow
CopySmall, stack-only types (all numbers, bool, char, tuples of Copy types)Implicit bitwise copy on assignment; both bindings remain valid
CloneTypes that can be deep-copied explicitlyExplicit .clone() call; can be arbitrarily expensive (heap allocation)
 
let x: i32 = 42;
let y = x;           // Copy: x is still valid
println!("{} {}", x, y);

let s1 = String::from("deep");
let s2 = s1.clone(); // Clone: explicit heap copy
println!("{} {}", s1, s2);

A type can implement Copy only if all its fields are Copy and it does not implement Drop.

Borrowing

Instead of transferring ownership, you can borrow a value by taking a reference. References are non-owning pointers that the compiler tracks.

The Two Borrowing Rules:

  • You can have any number of immutable references (&T) — OR —
  • Exactly one mutable reference (&mut T)
  • Never both at the same time.

This is the “aliasing XOR mutation” rule. It prevents data races at compile time.

fn print_length(s: &String) {       // immutable borrow
    println!("length: {}", s.len());
} // s goes out of scope, but doesn't drop the String (not the owner)

fn add_exclamation(s: &mut String) { // mutable borrow
    s.push('!');
}

fn main() {
    let mut greeting = String::from("hello");

    // Multiple immutable borrows: OK
    let r1 = &greeting;
    let r2 = &greeting;
    println!("{} {}", r1, r2);

    // Mutable borrow after immutable borrows are done: OK
    add_exclamation(&mut greeting);
    println!("{}", greeting); // "hello!"

    // This would NOT compile:
    // let r3 = &greeting;
    // add_exclamation(&mut greeting);
    // println!("{}", r3); // ERROR: cannot borrow as mutable while immutable borrow exists
}

Lifetimes

Lifetimes are Rust’s way of proving that references don’t outlive the data they point to. Most of the time, lifetimes are inferred by the compiler (lifetime elision). When the compiler can’t figure it out, you annotate explicitly with 'a:

// The returned reference lives as long as the shortest of the two inputs
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("short");
        result = longest(s1.as_str(), s2.as_str());
        println!("{}", result); // OK: both s1 and s2 are alive
    }
    // println!("{}", result); // ERROR if result could be s2 (s2 is dropped)
}

Lifetime elision rules

The compiler applies three rules automatically so you don’t annotate most functions:

  1. Each reference parameter gets its own lifetime parameter
  2. If there is exactly one input lifetime, it is assigned to all output lifetimes
  3. If there is a &self or &mut self parameter, its lifetime is assigned to all output lifetimes

'static

The 'static lifetime means “lives for the entire program.” String literals (&'static str) have this lifetime. T: 'static as a trait bound means the type contains no non-static references (it can be an owned type like String).

The Borrow Checker

Niko Matsakis’s key contribution to the language. The borrow checker is a compiler pass that verifies ownership, borrowing, and lifetime rules at compile time. It prevents:

Use-After-Free

A reference cannot outlive the data it borrows. The compiler rejects code where a reference escapes its owner’s scope.

Double-Free

Each value has exactly one owner. When the owner is dropped, the value is freed exactly once. No second free is possible.

Dangling References

You cannot return a reference to a local variable. The compiler sees that the reference would outlive the value and rejects it.

Data Races

Aliasing XOR mutation: you cannot have a mutable reference and any other reference at the same time. This eliminates data races at compile time.

Non-Lexical Lifetimes (NLL)

Rust 2018 introduced NLL: the borrow checker now tracks when a reference is last used, not when it goes out of scope. This makes many previously-rejected programs compile correctly:

let mut v = vec![1, 2, 3];
let first = &v[0]; // immutable borrow
println!("{}", first); // last use of `first`
v.push(4);             // OK with NLL: first is no longer in use

Polonius

The next-generation borrow checker, still in development. Polonius uses a Datalog-based analysis to accept even more valid programs while maintaining the same safety guarantees.

Structs with Lifetimes

struct Excerpt<'a> {
    text: &'a str,  // the struct borrows from some String
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().unwrap();
    let excerpt = Excerpt { text: first_sentence };
    println!("Excerpt: {}", excerpt.text);
}

The lifetime annotation 'a on the struct says: “an Excerpt cannot outlive the string it borrows from.” The compiler enforces this everywhere the struct is used.

Ownership and Collections

Collections own their elements. A Vec<String> owns the strings. When the vector is dropped, all the strings are dropped. You can iterate by reference (&v), by mutable reference (&mut v), or by value (v.into_iter() which consumes the vector):

let names = vec!["Alice".to_string(), "Bob".to_string()];

// Borrow: names is still valid after the loop
for name in &names {
    println!("{}", name);
}

// Consume: names is moved into the loop, no longer valid after
for name in names {
    println!("owns: {}", name);
}
// names is no longer valid here
