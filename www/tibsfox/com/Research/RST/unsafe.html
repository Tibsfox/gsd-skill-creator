“Unsafe doesn’t turn off the borrow checker.” It gives you five additional superpowers that safe Rust denies. Everything else — ownership, lifetimes, type checking — still applies inside unsafe blocks.

What unsafe Allows

An unsafe block or function lets you:

  1. Dereference raw pointers (*const T, *mut T)
  2. Call unsafe functions (including extern "C" FFI)
  3. Access or modify mutable statics
  4. Implement unsafe traits (Send, Sync, etc.)
  5. Access union fields

Everything else — ownership, borrowing, lifetimes, type checking — is still enforced. unsafe is the minimal escape hatch, not an off switch.

let x = 42;
let raw = &x as *const i32;  // create a raw pointer (safe)

unsafe {
    println!("raw value: {}", *raw); // dereference (requires unsafe)
}

FFI — Calling C from Rust

extern "C" {
    fn strlen(s: *const std::ffi::c_char) -> usize;
}

fn main() {
    let c_str = std::ffi::CString::new("hello").unwrap();
    let len = unsafe { strlen(c_str.as_ptr()) };
    println!("length: {}", len); // 5
}

Calling Rust from C

#[no_mangle]
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}
// Compile as cdylib, call from C as: int add(int, int);

Tooling

  • bindgen — auto-generates Rust bindings from C headers
  • cbindgen — auto-generates C headers from Rust code
  • #[repr(C)] — ensures struct layout matches C

Interior Mutability

Sometimes you need to mutate data behind a shared reference (&T). Rust provides controlled ways to do this:

TypeCheckThread-safe?Use for
Cell<T>Compile-timeNoReplace values in-place (Copy types only)
RefCell<T>RuntimeNoRuntime-checked mutable borrows (panics on violation)
OnceCell<T> / OnceLock<T>Single initOnceLock isLazy initialization

use std::cell::RefCell;

let data = RefCell::new(vec![1, 2, 3]);

// Runtime borrow checking
{
    let mut v = data.borrow_mut(); // panics if already borrowed
    v.push(4);
}
println!("{:?}", data.borrow()); // [1, 2, 3, 4]

Memory Layout

AttributeMeaning
#[repr(Rust)]Default; layout is unspecified and may be reordered for optimization
#[repr(C)]C-compatible layout; fields in declaration order with C padding rules
#[repr(transparent)]Single-field struct has the same layout as the field (newtype pattern)
#[repr(packed)]No padding between fields (may cause unaligned access)
#[repr(align(N))]Minimum alignment of N bytes
 
use std::mem;

#[repr(C)]
struct Header {
    magic: u32,
    version: u16,
    flags: u16,
}

assert_eq!(mem::size_of::<Header>(), 8);
assert_eq!(mem::align_of::<Header>(), 4);

Zero-Sized Types (ZSTs)

Types with size_of == 0: unit type (), empty structs, PhantomData<T>. They occupy no memory and are optimized away. PhantomData lets you express ownership or lifetime relationships without storing data.

Drop and RAII

The Drop trait is Rust’s destructor. When a value goes out of scope, drop() runs automatically. Drop order is reverse of declaration order.

struct FileHandle {
    name: String,
}

impl Drop for FileHandle {
    fn drop(&mut self) {
        println!("closing file: {}", self.name);
    }
}

fn main() {
    let _f = FileHandle { name: "data.txt".into() };
    println!("using file...");
} // "closing file: data.txt" printed here
  • std::mem::drop(value) — drop a value early (just takes ownership and lets it go out of scope)
  • ManuallyDrop<T> — prevents automatic drop; you must call ManuallyDrop::drop() yourself
  • std::mem::forget(value) — leak the value (no drop). NOT unsafe but usually a bug.

The Miri Interpreter

Miri is an interpreter for Rust’s intermediate representation (MIR) that detects undefined behavior in unsafe code: out-of-bounds access, use-after-free, uninitialized reads, data races, and invalid values. Run with cargo +nightly miri test. Essential for anyone writing unsafe Rust.

When to Use Unsafe

  • FFI — calling C libraries (extern "C")
  • Performance-critical code — when the borrow checker prevents an optimization you can prove is safe
  • Implementing low-level abstractions — Vec, HashMap, Arc, Mutex are all built on unsafe internally
  • Hardware access — MMIO registers, DMA buffers, interrupt handlers

The rule: Write safe APIs backed by minimal unsafe internals. The unsafe surface should be small, well-documented, and tested with Miri. Most Rust code should never need unsafe.
