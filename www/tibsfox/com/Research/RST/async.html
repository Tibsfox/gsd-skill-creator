“Fearless concurrency” is not marketing. Rust’s type system makes data races a compile error. If your program compiles, shared mutable state is properly synchronized. No other mainstream language offers this guarantee.

Send and Sync — The Foundation

Two marker traits make fearless concurrency work:

  • Send: a type can be moved to another thread. Most types are Send. Rc<T> is NOT (non-atomic ref counting is not thread-safe).
  • Sync: a type can be shared (via &T) across threads. T is Sync if &T is Send. Cell<T> and RefCell<T> are NOT Sync.

The compiler auto-derives Send and Sync for types composed of Send/Sync fields. You cannot accidentally share non-thread-safe data across threads — the compiler refuses.

Threads

use std::thread;

// Spawn a thread with a move closure
let handle = thread::spawn(move || {
    println!("hello from a thread");
    42
});

let result = handle.join().unwrap(); // wait for thread, get result
println!("thread returned: {}", result);

// Scoped threads (Rust 1.63) — borrow from the parent stack
let mut data = vec![1, 2, 3];
thread::scope(|s| {
    s.spawn(|| {
        println!("scoped thread sees: {:?}", &data);
    });
    s.spawn(|| {
        data.push(4); // mutable borrow — safe because scoped
    });
}); // all scoped threads join here

Shared State: Arc + Mutex

The canonical pattern for shared mutable state across threads:

use std::sync::{Arc, Mutex};
use std::thread;

let counter = Arc::new(Mutex::new(0));
let handles: Vec<_> = (0..10).map(|_| {
    let counter = Arc::clone(&counter);
    thread::spawn(move || {
        let mut num = counter.lock().unwrap();
        *num += 1;
    })
}).collect();

for h in handles { h.join().unwrap(); }
println!("Result: {}", *counter.lock().unwrap()); // 10

Rust’s Mutex is safer than C++/Java: The data is inside the lock. You cannot access the data without holding the lock guard. The guard is a smart pointer that releases the lock when dropped (RAII). There is no way to forget to unlock.

Message Passing: Channels

use std::sync::mpsc; // multi-producer, single-consumer
use std::thread;

let (tx, rx) = mpsc::channel();

thread::spawn(move || {
    tx.send("hello from thread".to_string()).unwrap();
});

let msg = rx.recv().unwrap();
println!("received: {}", msg);

For multi-producer, multi-consumer: use crossbeam-channel or flume.

Async/Await

Stabilized in Rust 1.39 (November 2019). Rust’s async model is lazy (nothing happens until a future is polled) and zero-cost (compiles to state machines, not heap-allocated closures).

async fn fetch_data(url: &str) -> Result<String, reqwest::Error> {
    let body = reqwest::get(url).await?.text().await?;
    Ok(body)
}

#[tokio::main]
async fn main() {
    match fetch_data("https://httpbin.org/get").await {
        Ok(body) => println!("{}", body),
        Err(e) => eprintln!("error: {}", e),
    }
}

The Future trait

pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

Poll::Ready(value) means done. Poll::Pending means not ready yet — the runtime will call poll again when a waker fires.

The Async Runtime Problem

Rust has NO built-in async runtime. The language provides async/await syntax and the Future trait. You must choose a runtime to execute futures:

  • Tokio — the dominant runtime. Multi-threaded work-stealing scheduler. Used by axum, reqwest, tonic, tower. #[tokio::main].
  • async-std — simpler, follows std API patterns. Less ecosystem support.
  • smol — minimal, lightweight. Good for learning or small apps.

use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let (result_a, result_b) = tokio::join!(
        async { sleep(Duration::from_secs(1)).await; "a" },
        async { sleep(Duration::from_secs(2)).await; "b" },
    );
    println!("{} {}", result_a, result_b); // after 2 seconds
}

Rayon — Data Parallelism

Josh Stone and Niko Matsakis. Drop-in parallelism for iterator chains using a work-stealing thread pool:

use rayon::prelude::*;

let sum: i64 = (0..1_000_000)
    .into_par_iter()           // parallel iterator
    .filter(|n| n % 2 == 0)
    .map(|n| n * n)
    .sum();

let mut data = vec![5, 3, 1, 4, 2];
data.par_sort();               // parallel sort

Atomics

std::sync::atomic provides lock-free atomic types with explicit memory ordering:

OrderingMeaning
RelaxedNo ordering guarantees; only atomicity
AcquireReads after this see writes before the matching Release
ReleaseWrites before this are visible to the matching Acquire
AcqRelBoth Acquire and Release
SeqCstSequentially consistent; total order agreed by all threads

See Rust Atomics and Locks by Mara Bos for the definitive treatment.

Pin and Unpin

Pin is Rust’s solution to self-referential structs. When the compiler generates a state machine for an async fn, local variables that are referenced across .await points create self-referential data. Moving the struct would invalidate those internal pointers.

Pin<&mut T> guarantees the value will not be moved in memory. Unpin is a marker trait for types that are safe to move even when pinned (most types are Unpin). This is widely considered one of Rust’s most confusing concepts.

Lock-Free Data Structures

The crossbeam crate (Aaron Turon) provides epoch-based memory reclamation and lock-free data structures: SegQueue (unbounded lock-free queue), ArrayQueue, deque (work-stealing).
