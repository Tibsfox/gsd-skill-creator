Rust has the most welcoming community and the best free documentation of any systems programming language. “The Book” is genuinely good, the compiler errors are genuinely helpful, and the community genuinely wants you to succeed. The hard part is the borrow checker, not the people.

A Reading Order That Works

  1. “The Book” (Klabnik & Nichols, free online) — read it cover to cover, do the exercises
  2. Rustlings — fix broken code to reinforce ownership, borrowing, traits
  3. Build something real: a CLI tool with clap, an HTTP server with axum, or a game with bracket-lib
  4. Programming Rust (Blandy/Orendorff/Tindall) — the comprehensive reference when you hit walls
  5. Jon Gjengset’s Crust of Rust videos — for intermediate topics (iterators, smart pointers, async)
  6. Rust for Rustaceans (Gjengset) — when you’re comfortable with ownership and want to go deep
  7. Rust Atomics and Locks (Bos) — when you need low-level concurrency
  8. Read ripgrep source — Andrew Gallant’s code is the gold standard for idiomatic Rust
  9. The Rustonomicon — when you need unsafe

This progression takes several months of serious study and produces a programmer who can write production Rust confidently.

The Borrow Checker Experience

Every Rust learner hits a wall with the borrow checker. This is normal. The typical arc:

  1. Week 1-2: “This is nice, it’s like C++ but cleaner.”
  2. Week 3-4: “WHY WON’T THIS COMPILE.” The borrow checker rejects your design. You fight it.
  3. Week 5-8: “Oh. The borrow checker was right. My design was wrong.” You redesign.
  4. Month 3+: “I think in ownership now. The borrow checker catches things I didn’t even know were bugs.”
  5. Month 6+: “I miss the borrow checker when I write other languages.”

The compiler error messages are genuinely excellent — they explain what went wrong, why, and often suggest the fix. Trust them.
