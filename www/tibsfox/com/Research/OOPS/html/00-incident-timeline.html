The Claude Code Source Code Incident – March 31, 2026

What Happened

On March 31, 2026, security researcher Chaofan Shou discovered that Anthropic’s Claude Code – their flagship agentic CLI tool – had its entire source code exposed through a source map file inadvertently published to the npm registry. The exposure was not a hack, not an exploit, not a zero-day. It was a packaging oversight: a .map file that should have been excluded from the npm publish was included because of a missing line in .npmignore.

A 59.8 MB JavaScript source map generated by Bun’s bundler (which Claude Code uses to compile TypeScript into a single executable) was included in version 2.1.88 of the @anthropic-ai/claude-code package. Source maps are debugging artifacts that map compiled JavaScript back to original TypeScript source. Bun generates them by default unless explicitly excluded. The .npmignore file – which tells npm what NOT to publish – lacked a *.map exclusion. So the source map shipped to every mirror of the npm registry alongside the compiled binary.

The exposure revealed approximately 512,000 lines of TypeScript across approximately 1,900 files – the complete client-side Claude Code codebase. This represented months of engineering work by one of the most well-funded AI companies in the world, covering everything from tool dispatch to context management to unreleased features gated behind 44 feature flags.

What Was NOT Exposed

Precision matters when discussing security incidents. This was the client-side CLI tool source code – the application that runs on the user’s machine. It was NOT:

  • Anthropic’s AI model weights or architecture (these live on Anthropic’s servers, not in the CLI)
  • Training data or fine-tuning datasets
  • Customer data, API keys, or authentication secrets
  • Server-side API infrastructure or model serving code
  • Conversation history or telemetry databases
  • Internal corporate communications or HR data

The exposure was analogous to reverse-engineering a desktop application – which companies have done legally for decades – except that Anthropic accidentally provided the source instead of forcing researchers to decompile the binary. The information was always theoretically accessible through binary analysis (and many researchers had already extracted architectural insights from the string table of the compiled Bun SEA binary). The source map simply made the analysis dramatically easier and more complete.

The pattern suggests a systemic gap in Anthropic’s release engineering: not a failure of security controls on sensitive systems, but a failure of mundane build pipeline hygiene. The fix for the source map issue was literally one line in .npmignore: *.map. That this line was missing – and that the 2025 incident did not trigger a systematic audit of build artifact exclusions – points to an organizational process gap rather than a technical one.

What the Source Code Revealed

Scale

  • 512,000 lines of TypeScript across approximately 1,900 files
  • 59.8 MB source map (the compiled JavaScript bundle that the map references is approximately 15 MB)
  • 44 feature flags covering unreleased features, A/B tests, and graduated rollouts
  • 150+ references to KAIROS, the most discussed discovery from the leak

KAIROS – Autonomous Daemon Mode

The most significant technical discovery was KAIROS, named after the Ancient Greek concept of “the opportune moment” (as opposed to chronos, sequential time). KAIROS appeared in the source code more than 150 times, indicating it is not a prototype but a substantial, well-developed subsystem.

What KAIROS is: A persistent background agent that operates as an always-on daemon, not waiting for user input to take action. Where today’s Claude Code is reactive – the user types a command, the agent responds – KAIROS represents a proactive mode where the agent continuously observes, reasons, and acts.

What the code reveals about KAIROS:

  • Daemon architecture: KAIROS operates as a persistent background process, not as a request-response cycle. It has its own event loop, its own state, and its own decision-making authority within defined boundaries.
  • 15-second blocking budget: KAIROS operations have a hard ceiling of 15 seconds for any blocking action. This prevents the daemon from interfering with the user’s interactive work while still allowing substantive background operations.
  • autoDream memory consolidation: While the user is idle, KAIROS performs “memory consolidation” through an autoDream system. This process merges observations from the session, removes contradictions between stored facts, and converts raw observations into structured insights. The metaphor is deliberate: the system literally “dreams” during downtime, reorganizing knowledge.
  • Append-only daily log files: KAIROS maintains daily logs of observations, decisions, and actions. The append-only design ensures forensic auditability – you can always trace what the daemon did and why.
  • Feature-gated, compiled, production-adjacent: KAIROS is not research code living in a separate branch. It is compiled into the production binary, sitting behind feature flags, ready to be activated for specific users or rollout percentages. The infrastructure to turn it on is already deployed.

What KAIROS means for the industry:

KAIROS represents the first concrete evidence from a major AI company that autonomous background agents are not speculative – they are engineered, tested, and waiting for deployment. The implications extend well beyond Claude Code:

  1. The agent loop becomes persistent. Every current AI coding tool operates in a request-response pattern: the human acts, the AI responds. KAIROS breaks this pattern fundamentally. The agent is always running, always observing, always ready to act without being asked. This is the difference between a tool and a collaborator.

  2. Memory consolidation becomes a first-class concern. The autoDream system indicates that Anthropic considers memory management not just a technical necessity but a cognitive architecture feature. Agents that dream – that process, consolidate, and restructure their knowledge during downtime – will develop session-over-session coherence that reactive agents cannot match.

  3. Daemon-mode agents require new trust models. An agent that acts autonomously needs guardrails that interactive agents do not. The 15-second blocking budget is one such guardrail. The append-only logs are another. But the broader question – how do you trust an agent that acts without being asked? – is one the industry has not yet answered at scale. Our trust system work (trust-relationship.ts, 95 tests passing) is directly relevant here.

  4. The competitive landscape shifts. Every AI coding tool company – Cursor, Windsurf, Codex, Gemini Code Assist – now knows that Anthropic has a production-ready daemon mode. The race to ship autonomous background agents has effectively started, with Anthropic holding the architectural lead.

The Mythos Model Revelation

Five days before the source map leak, Fortune reported that Anthropic had accidentally exposed approximately 3,000 files that referenced “Mythos” – an internal codename (also called “Capybara”) for a new model tier beyond Opus. The source map leak confirmed and expanded on this revelation.

What the source code revealed about Mythos:

  • Internal references to both “fast” and “slow” variants of Mythos, suggesting a model family rather than a single model
  • Integration points in the codebase where Mythos could be selected as the backing model for specific operations
  • Feature flags controlling Mythos access, indicating staged rollout plans

What Mythos means:

Anthropic’s current public model hierarchy is Haiku (fast/cheap) -> Sonnet (balanced) -> Opus (highest capability). Mythos as a tier “beyond Opus” suggests either a larger model, a more capable model, or a model with qualitatively different capabilities (such as extended reasoning, multimodal generation, or persistent memory). The “fast” and “slow” variants parallel the existing pattern where each tier offers speed/capability tradeoffs.

For tool builders, Mythos matters because it implies the model substrate is about to change significantly. Tools optimized for Opus’s capabilities and limitations – context windows, reasoning depth, instruction following – may need adaptation when Mythos becomes available. The existence of integration points in Claude Code’s source suggests this is not a distant eventuality but an imminent one.

Context Entropy Solution

The source code revealed Anthropic’s internal approach to context entropy – the gradual degradation of agent performance as conversations grow long and context windows fill. This is the same problem our PostCompact hook work addresses and the same challenge we experience in extended multi-agent sessions.

The solution involves: - Sophisticated context compression algorithms that preserve high-value information while discarding redundant or low-relevance content - The PreCompact/PostCompact hook pair, which allows the system to save and restore critical state around compaction events - Memory relevance scoring (the memory_survey pattern) that evaluates which stored knowledge is relevant to the current task before loading it into context

Feature Flags and Unreleased Capabilities

The 44 feature flags represent a substantial pipeline of capabilities in various stages of development and rollout:

  • Flags controlling access to new model tiers and variants
  • Flags for experimental UI features and interaction modes
  • Flags for infrastructure changes (new API endpoints, protocol updates)
  • Flags for graduated rollouts of KAIROS subcomponents

The feature flag system itself is architecturally significant: it indicates Anthropic operates Claude Code as a continuously deployed service with fine-grained rollout control, not as a versioned release with periodic updates.

Security Advisory

The Supply Chain Attack

Important: Between 00:21 and 03:29 UTC on March 31, 2026, users who installed or updated Claude Code via npm may have been exposed to a supply chain attack that piggybacked on the exposure.

Within hours of the leak, malicious actors published fake axios packages (versions 1.14.1 and 0.30.4) to npm containing a Remote Access Trojan (RAT). These packages were dependency-squatted – they had version numbers designed to be pulled in by semver ranges in projects that depended on axios. The timing was not coincidental: the attackers exploited the flood of npm traffic around Claude Code to increase the probability that developers would install malicious dependencies.

This attack was NOT from Anthropic. The malicious axios packages were published by third parties exploiting the confusion around the leak.

Specific Mitigation Steps

If you installed or updated any npm packages during the March 31 window, take these steps:

  1. Check for malicious axios versions:

    # Check your project's node_modules
find node_modules -name "package.json" -path "*/axios/*" | xargs grep '"version"'

# Specifically check for the known malicious versions
npm ls axios | grep -E '1\.14\.1|0\.30\.4'

  2. If found, remove immediately:

    rm -rf node_modules/axios
npm install axios@1.7.9  # known safe version as of March 2026

  3. Check for persistence mechanisms: The RAT established persistence through cron jobs and modified shell profiles. Check:

    crontab -l  # look for unfamiliar entries
cat ~/.bashrc ~/.zshrc  # look for unfamiliar lines at the end

  4. Rotate credentials: If your system was compromised, rotate any credentials that were accessible on the affected machine: npm tokens, git credentials, SSH keys, cloud provider access keys, database passwords.

  5. Use lockfiles: If you were not already using package-lock.json (npm) or pnpm-lock.yaml (pnpm) with exact version pinning, this incident is a concrete reason to start. Lockfiles prevent semver ranges from pulling in unexpected versions.

  6. Enable npm audit:

    npm audit

    The malicious axios versions were flagged in the npm advisory database within hours of discovery.

Broader Supply Chain Implications

This incident illustrates a structural vulnerability in the npm ecosystem: high-profile events that drive installation spikes create windows of opportunity for supply chain attacks. The pattern is predictable and will recur. Developers should treat any high-traffic npm event (major package release, security incident, viral package) as a period of elevated supply chain risk and avoid updating dependencies during the immediate aftermath unless necessary.

GitHub Archival: The Velocity of Open Knowledge

The archival GitHub repository (Kuberwastaken/claude-code) reached 41,500+ forks within 24 hours. To put this velocity in perspective:

Metric Value Context
Forks in 24 hours 41,500+ Most popular open-source projects take months or years to reach this fork count
Stars in 24 hours 1,100+ Stars accumulate slower than forks because starring is a passive action while forking indicates active use
Fork-to-star ratio ~37:1 Normal ratio is ~0.1-0.5:1 (more stars than forks). The inverted ratio here indicates most users wanted a local copy, not just to bookmark

The 37:1 fork-to-star ratio is the critical signal. In normal open-source projects, the vast majority of GitHub users star a repository (bookmark it for later) while a small percentage fork it (create a copy they intend to use or modify). Here, the ratio was inverted by two orders of magnitude: users were forking 37 times for every star. This means the primary action was creating a personal copy of the source code before it could be taken down – a digital preservation reflex.

This velocity tells us several things about the market:

  1. Intense demand for transparency. Tens of thousands of developers wanted to understand how Claude Code works at a source level. This is not a niche interest – it represents a significant fraction of Claude Code’s active user base.

  2. The code is already distributed. With 41,500+ forks, the source code is now stored in 41,500+ GitHub accounts, plus their local clones, plus forks-of-forks, plus mirrors on other platforms. It cannot be un-distributed. Anthropic’s practical option is to accept this reality and consider whether controlled open-sourcing would be strategically preferable to uncontrolled leaking.

  3. Competitive analysis is underway. Every major AI coding tool company – and many startups – will have engineers studying this code. The architectural decisions Anthropic made are now public knowledge. The competitive advantage shifts from “knowing how it works” to “executing better on the shared knowledge.”

  4. The Claw Code project is a leading indicator. Within hours of the leak, the “Claw Code” project began rewriting Claude Code’s architecture in Rust. If Claude Code’s architecture is good (and the evidence suggests it is), multiple open-source implementations will emerge, creating a competitive ecosystem that Anthropic does not control.

For those of us building tooling on top of Claude Code – and we are among the most deeply integrated – the leak has immediate practical implications:

1. Architectural Validation

Our independent development of skills, agents, hooks, memory, and worktree-based isolation converged with Anthropic’s internal architecture. The source code confirms this convergence was not coincidental but rather the result of solving the same engineering problems. This is enormously validating: it means our architectural decisions are sound, our abstractions are aligned with the platform’s direction, and our extensions (GUPP, DACP, Gastown convoy) fill genuine gaps.

2. Feature Pipeline Visibility

The 44 feature flags provide a window into what Anthropic is building next. For tool builders, this is actionable intelligence: - Features that align with our existing work (KAIROS aligns with GUPP) confirm our direction - Features that we have not anticipated (new notification types, memory survey) reveal improvement opportunities - Features that compete with our extensions (teams, dispatch) signal where we should align rather than diverge

3. Hook System Completeness

We now know the complete hook taxonomy: 13+ event types, many of which we were not using. This is directly actionable – see OOPS doc 04 for our hook-by-hook analysis and implementation plan.

4. Trust Model Requirements

KAIROS makes trust a first-class engineering concern, not just an academic one. An always-on daemon agent that acts autonomously needs earned trust relationships, bounded autonomy, and audit trails. Our trust system (trust-relationship.ts with 95 tests) is directly applicable to this emerging requirement.

5. The Build Pipeline Lesson

The most actionable lesson for every tool builder: audit your own build pipelines. If Anthropic – a company valued at tens of billions of dollars with world-class engineering talent – can ship a source map to npm, so can you. The fix is mechanical:

  • Use files in package.json (allowlist) instead of only .npmignore (blocklist)
  • Add a CI step that checks published package contents before npm publish
  • Never rely solely on .npmignore – it fails open (includes everything not explicitly excluded)
  • Test your packaging: npm pack and inspect the tarball before publishing

What We Learn

1. Our Architecture Converged with Anthropic’s

Skills, agents, hooks, memory, worktrees – we independently built the same patterns. The source confirms this at the directory structure level: .claude/agents/, .claude/skills/, SKILL.md frontmatter, MEMORY.md. Sixteen of our patterns have direct parallels in Claude Code’s internal architecture (see OOPS doc 01 for the full map). This convergence is not flattery – it is engineering validation.

2. KAIROS Validates Our GUPP Concept

We built GUPP (Get Up and Push Protocol) to convert passive agent behavior into proactive execution. GUPP addresses RLHF-trained passivity bias: agents that summarize plans instead of executing them, that ask “shall I proceed?” when the work is already on their hook. KAIROS is Anthropic’s answer to the same problem – an always-on daemon mode that does not wait for permission to act. Different implementation (daemon vs. interrupt-driven), same fundamental insight: agents should push, not wait.

The difference in approach is instructive. GUPP operates at the orchestration layer, using hooks and nudges to enforce execution on agents that are otherwise reactive. KAIROS operates at the agent layer, making the agent itself proactive. Both are valid; the ideal system likely combines both – a proactive agent (KAIROS) operating within an execution enforcement framework (GUPP).

3. The autoDream Memory Consolidation Parallels Our Memory System

We built MEMORY.md with tiered relevance: HOT (always loaded), WARM (loaded on session start), COLD (loaded on demand). KAIROS’s autoDream does something functionally similar but automated – merging observations during idle periods, removing contradictions, converting raw session data into structured facts. Our memory system is manually curated; KAIROS proposes an automated curation loop. The convergence suggests both approaches are necessary: automated consolidation for volume, human curation for accuracy and priority.

4. Context Entropy Is the Real Problem

Our PostCompact hook work (see OOPS doc 04) addresses the same challenge Anthropic was solving internally. Long sessions degrade because context fills, compacts, and loses critical information. The PreCompact/PostCompact hook pair is the save/restore mechanism for this problem. The memory_survey pattern (scoring relevance before loading) is the prevention mechanism. Both approaches are needed: prevent overload (survey), and recover from compaction (save/restore).

5. The Future Is Agent Autonomy

Both our work and KAIROS point the same direction: AI agents that do not just respond to prompts but actively manage their own context, memory, and initiative. The 360 engine – 57 autonomous releases without human intervention – is a working example of this future. KAIROS is Anthropic’s engineering commitment to the same vision. The question is no longer “will agents be autonomous?” but “how do we make autonomous agents trustworthy?”

What Happens Next

For Anthropic

The practical damage is limited but real. The source code exposure does not compromise their AI models, their competitive moat in model quality, or their customer data. It does compromise their competitive advantage in client-side architecture, reveals their feature pipeline, and creates a reference architecture for competitors and open-source alternatives.

Anthropic’s most strategically sound response would be controlled open-sourcing of Claude Code’s client – similar to how Meta open-sources PyTorch while keeping their training infrastructure proprietary. The client code is already public in practice (41,500+ copies distributed globally). Open-sourcing it would convert an embarrassment into a strategic asset: community contributions, third-party integrations, and ecosystem lock-in through adoption.

For the Industry

The KAIROS revelation accelerates the industry’s timeline for autonomous agent development. Every competitor now has a concrete architectural reference for how to build a daemon-mode coding agent. The competitive question shifts from “should we build this?” to “how fast can we ship our version?”

The architectural patterns revealed – skill systems, hook taxonomies, context compaction, memory survey – will likely become industry standards. When everyone has the same architectural blueprint, differentiation shifts to execution quality, model capabilities, and ecosystem depth.

For Tool Builders Like Us

Our position strengthens. We have the deepest third-party integration on the platform. Our architectural convergence means we are aligned with both Anthropic’s direction and the emerging industry standard. Our extensions (GUPP, DACP, Gastown, trust system) address genuine gaps that the source code confirms exist. The question is not whether to continue – it is how to leverage this moment of clarity to build faster and share more openly.
