A. Simulations — What to Build Locally

A1. Python: Connector State Machine Simulator

A Python simulation that models the MR-1 tail plug connector as a finite state machine with three binary variables (A, B, C). The student can trigger disconnections in any order and observe which states are valid, which are invalid, and which contain the sneak circuit. Visualizes the state machine as a directed graph, highlights the intended path versus all possible paths, and shows the Boolean expression for engine cutoff in each state.

Key learning moments:

  1. Click B first (simulating MR-1) and watch the state machine transition to state 4 (0,1,0). The sneak circuit term (NOT A) AND B evaluates to TRUE. Engine cutoff.
  2. Try all 6 permutations: 5 red rows, 1 green row.
  3. Probability plot shows factorial scaling: for 10 operations, P(correct) is effectively zero.

Libraries: numpy, matplotlib, networkx | Difficulty: Beginner-Intermediate | Duration: 2–3 hours

A2. Python: Thrust Curve and Rise Height Calculator

Computes the Mercury-Redstone vehicle’s altitude, velocity, and acceleration as a function of time during the first 10 seconds of flight. Accounts for varying mass (propellant burn), atmospheric drag, and thrust buildup. The student can adjust vehicle mass, thrust, and drag coefficient and observe how parameters affect the rise rate — and specifically, the velocity at the 4-inch connector separation point.

Key learning moments:

  1. The altitude curve barely rises above zero before flattening — the engine cut off. The dashed “would-have-been” curve soars upward.
  2. Parametric study: at T/W = 1.14 (MR-1), separation velocity is 0.53 m/s. At T/W = 1.25 (military Redstone), it is 0.70 m/s — 32% faster.

Libraries: numpy, matplotlib, scipy | Difficulty: Beginner-Intermediate | Duration: 2–3 hours

A3. Web: Interactive Launch Sequence Game

A browser-based game where the player must disconnect three virtual cables (A, B, C) in the correct order to launch a Mercury-Redstone rocket. Wrong order replicates MR-1’s actual failure: the rocket rises 4 inches, the engine stops, the escape tower fires, parachutes deploy on the pad. Correct order: rocket launches with animation and sound.

Difficulty modes: Easy (3 cables, 6 permutations) → Medium (4 cables, 24) → Hard (5 cables, 120) → Nightmare (6 cables, 720).

Technology: HTML5 Canvas + JavaScript | Difficulty: Intermediate | Duration: 4–6 hours

A4. Web: MR-1 Event Timeline

An interactive second-by-second timeline of the MR-1 flight, from T-5 seconds through parachute deployment on the pad. Key events plotted on a horizontal timeline with precise timestamps. Zoomable to the critical 80-millisecond window where the sneak circuit formed. State machine panel synchronized with timeline position.

Key learning moment: Zooming into T+0.420s to T+0.500s shows the sneak circuit forming in 20 ms: B disconnects at T+0.430, sneak circuit activates at T+0.432, A disconnects at T+0.460 — but the damage is done. The cause (20 ms) and the effect (engine shutdown at T+1.530) are separated by a factor of 50 in time.

Technology: HTML5 Canvas + JavaScript | Difficulty: Intermediate | Duration: 3–4 hours

B. Machine Learning — What to Train

B1. Sequence Anomaly Detector

Train an LSTM sequence classifier to detect out-of-order connector disconnection sequences. Input: sequences of 3–10 timestamped events with timing noise. Output: binary classification (nominal / anomalous) plus identification of which event is out of order.

Training data: 10,000 synthetic sequences per class. Timing noise: Gaussian, sigma = 5–50 ms. Expected accuracy: >95% for well-separated events, degrades to ~80% when noise makes ordering ambiguous. Attention maps highlight the first out-of-order event.

Libraries: numpy, torch, matplotlib | GPU: Not needed | Difficulty: Intermediate

C. Computer Science — The Race Condition

C1. Race Conditions in Concurrent Programming

MR-1’s sneak circuit is a hardware race condition: two signals racing to change state, with system behavior depending on which arrives first.

# MR-1 race condition in Python pseudocode:
ground_power_connected = True
flight_control_active = False

def disconnect_A():     # Pin A -- ground power
    time.sleep(random.uniform(0.01, 0.05))
    ground_power_connected = False

def disconnect_B():     # Pin B -- control signals
    time.sleep(random.uniform(0.01, 0.05))
    flight_control_active = True

# If B completes before A: SNEAK CIRCUIT! Engine cutoff.

# Fix 1 (MR-1A): Simultaneous release -- atomic operation
# Fix 2 (Software): Mutex lock
# Fix 3 (Dictyostelium): Redundancy + majority voting

The key insight: the race condition exists because the intermediate state (A connected, B disconnected) is reachable. Eliminating it requires making that state unreachable, safe, deterministic, or tolerable.

D. Game Theory — The Test Program Dilemma

D1. How Much to Test Before You Fly

MR-1’s failure could have been caught by testing at the actual separation speed. The test program tested at design-nominal speed. The failure was in the UNTESTED region of the test space. The game-theoretic optimum: distribute test points to maximize detection probability across the full envelope, with extra coverage near boundaries. MR-1’s test program was not Bayesian — it tested where failure was expected rather than where failure was unknown.

F. Problem-Solving Methodology — The Interface Audit

F1. Interface Hazard Analysis

The five-step methodology born from MR-1:

  1. Identify all interfaces — every physical connection between systems
  2. Enumerate signals/energy — what crosses the boundary, in what direction, when
  3. Identify sequencing requirements — must this signal change before/after another?
  4. Verify enforcement — how is the correct sequence enforced? What happens if enforcement fails? Has it been tested at actual operating conditions?
  5. Document and review — create an ICD with both sides of the interface present

Now standard in aerospace (NASA NPR 7120.5) and adopted by automotive (ISO 26262), nuclear (10 CFR 50), and medical device (IEC 62304) industries.

G. GSD Improvements — Sequence Validation Chipset

G1. Sequence Validator Chipset

Four chips for gsd-skill-creator: sequence-model (define expected ordering as a DAG), sequence-monitor (track actual execution order, compare to DAG), sneak-path-detector (enumerate all paths, identify bypasses), sequence-enforcer (block or warn on out-of-order operations). Hooks into the phase execution pipeline. MR-1’s connector had an implicit sequence that was not explicitly validated. This chipset makes all sequence dependencies explicit and validated in real time.

Voltaire died on May 30, 1778, in Paris, reportedly saying “Now is not the time for making new enemies” when asked to renounce Satan on his deathbed. He had spent 84 years making enemies of every institution that valued certainty over evidence. He would have understood MR-1 instantly. Not the engineering — he was a writer, not an engineer. But the epistemology. The tail plug connector was designed with certainty: the pins WILL disconnect in this order. Voltaire’s entire career was spent dismantling exactly this kind of certainty. Pins have tolerances. Separation speeds vary. The world is not a mechanism that executes the designer’s intent. It is a mechanism that executes physics, and physics does not read schematics. Pangloss would have explained MR-1 as necessary in the best of all possible space programs. This is retrospectively true. But it is not a reason to celebrate the failure. It is a reason to analyze it, document it, fix it, and fly again. Il faut cultiver notre jardin. The garden, in November 1960, was Cape Canaveral. The weeds were sneak circuits. The gardeners were engineers with 28 days and a redesigned connector. They cultivated. They flew. They succeeded. Voltaire would have approved.
