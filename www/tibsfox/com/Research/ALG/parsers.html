Parser Technology

From Knuth’s 1965 LR paper to yacc (1975), bison (1985), ANTLR (1990s), Ford’s PEG (2004), and Tree-sitter (2018)

“Yacc (1975) gave compiler writers a way to generate parsers from grammars. ANTLR (1990s) made top-down parsing mechanical. PEG (2004) removed ambiguity. Tree-sitter (2018) made parsers fast enough to run on every keystroke. Each step kept the BNF inheritance visible.”

Hand-Written Recursive Descent

The first parsers were hand-written. The FORTRAN 0 parser, ALGOL 58 compilers, the early Lisp readers — all were hand-coded. Recursive descent was the natural pattern: you wrote a function per syntactic category, each function read a bit of input and called other functions to parse sub-structures. The parser mirrored the grammar, with each non-terminal becoming a function.

Recursive descent works beautifully for LL(1) grammars and works well with hand-written backtracking for LL(k) and some non-LL grammars. It produces parsers that are straightforward to read, easy to debug, and easy to add error recovery to. It is still the technique used by Clang’s C++ parser, Roslyn’s C# parser, and the Rust compiler’s parser — all production compilers for production languages, all hand-written recursive descent.

Operator Precedence and Pratt Parsing

Bauer and Samelson, at the Technische Hochschule München in the late 1950s, introduced operator precedence parsing as a specialized technique for expressions. The idea: assign each operator a precedence number, use a stack to track operators being deferred, and reduce whenever the stack top has higher precedence than the incoming operator. This insight came from their work on ALGOL 58’s expression problem.

A related technique, Pratt parsing (Vaughan Pratt, 1973, “Top Down Operator Precedence”), generalizes operator precedence by associating “binding powers” with tokens and using a simple recursive procedure to parse expressions with arbitrary infix, prefix, and postfix operators. Pratt parsers are still the technique of choice for hand-written expression parsers. The Rust compiler’s expression parser is Pratt-based. Pratt parsing is easy to extend — adding a new operator is just adding a binding power table entry.

Knuth’s LR Paper (1965)

Donald Knuth’s On the Translation of Languages from Left to Right, published in Information and Control, 1965, is the paper that founded modern parser theory. Knuth asked: what is the largest class of context-free languages that can be parsed in linear time by a deterministic algorithm using a bounded amount of lookahead?

His answer: LR(k), for any fixed k. He gave a constructive algorithm for building the parsing table from the grammar, proved correctness, and established that LR(k) for k ≥ 1 captures exactly the deterministic context-free languages. He also introduced the concepts of item, closure, goto, and LR state that every subsequent parsing treatment has used.

The paper was largely theoretical at the time of publication: LR(1) tables were too big to build by hand and too big to store on the computers of 1965. The theory had to wait for better computers and for LALR(1) compression before it became useful.

DeRemer’s LALR(1) — 1969

Frank DeRemer’s MIT PhD thesis, Practical Translators for LR(k) Languages (1969), and his subsequent CACM paper Simple LR(k) Grammars (1971) introduced two important compressions of Knuth’s LR(1): SLR (Simple LR) and LALR (Lookahead LR). SLR uses FOLLOW sets as lookaheads directly; LALR uses item-set-specific lookaheads. LALR is strictly more powerful than SLR but strictly less than canonical LR(1). For real programming languages, LALR(1) is enough. DeRemer’s thesis made LR parsing practical by cutting table sizes by an order of magnitude.

Yacc — Yet Another Compiler-Compiler (1975)

§ Yacc at Bell Labs

Stephen C. Johnson at Bell Labs released Yacc in 1975 as part of the Unix distribution. Yacc was a LALR(1) parser generator: you wrote a grammar file with semantic actions in C, and Yacc generated a C parser. Yacc was paired with Lex (Mike Lesk and Eric Schmidt, also 1975), the lexer generator.

Together, Yacc and Lex became the dominant parser-writing toolchain for two decades. Every early C compiler, every database query language, every configuration file format, every domain-specific language built in the 1970s and 1980s used Yacc and Lex.

A Yacc grammar file looks like BNF with semantic actions attached:

expr:   expr '+' term   { $$ = $1 + $3; }
    |   expr '-' term   { $$ = $1 - $3; }
    |   term            { $$ = $1; }
    ;

term:   term '*' factor { $$ = $1 * $3; }
    |   term '/' factor { $$ = $1 / $3; }
    |   factor          { $$ = $1; }
    ;

factor: NUMBER          { $$ = $1; }
    |   '(' expr ')'    { $$ = $2; }
    ;

The $$ is the value of the left-hand side; $1, $3 are the values of symbols on the right. Yacc generates a C file that declares a yyparse() function driving an LALR(1) state machine. Lex generates a yylex() that the parser calls to get tokens. You link them together and you have a parser.

Bison — GNU’s Yacc (1985)

Bison was written by Robert Corbett originally as a Berkeley project and later brought under GNU by Richard Stallman. It is a yacc-compatible parser generator under the GPL. Bison has been the default parser generator for GNU projects and has slowly accumulated features beyond standard yacc: GLR parsing, push parsers, reentrant parsers, C++ output, better error messages, locations, named references. For decades of GNU history, “writing a parser” meant “writing a Bison file.”

ANTLR — Another Tool for Language Recognition

Terence Parr built PCCTS (Purdue Compiler Construction Tool Set) at Purdue University in the early 1990s, then renamed it to ANTLR when he moved. ANTLR is a LL(*) parser generator — it uses top-down parsing with arbitrary-lookahead decisions, which is strictly more powerful than LL(k) for any fixed k and handles most real grammars that programmers write naturally.

ANTLR’s advantages over yacc:

  • Readable error messages. ANTLR parsers produce errors like “missing ‘;’ at ‘if’” rather than “syntax error at line 37.”
  • Top-down parsing. The generated parser code is recursive descent, which is easier to debug and step through than an LR state machine.
  • Multiple target languages. ANTLR can generate parsers in Java, C#, Python, JavaScript, Go, C++, Swift, PHP, and Dart.
  • Tree grammars and visitor patterns. ANTLR generates AST node classes and visitor interfaces automatically.
  • Semantic predicates. You can attach runtime conditions to productions, giving you context-sensitive parsing without leaving the grammar.

ANTLR4, released in 2013, is the current version. It introduced Adaptive LL(*), an even more powerful top-down technique, and significantly improved error recovery. ANTLR is widely used in commercial compilers, DSLs, IDE tooling, and static analyzers. Twitter’s search query language, Apache Hive’s HQL, Groovy’s parser, and many others are built on ANTLR.

GLR Parsing — Tomita, 1986

Masaru Tomita’s Generalized LR (GLR), published in his 1986 book Efficient Parsing for Natural Language, extends LR parsing to handle arbitrary context-free grammars, including ambiguous ones. Instead of maintaining a single parser state stack, GLR maintains a graph of possible states, forking when shift/reduce conflicts occur and rejoining when forks produce the same state. The result is a parse forest rather than a single tree when the grammar is ambiguous.

GLR was invented for natural language processing (Japanese, in Tomita’s case), where grammars are inherently ambiguous. It later found use in programming language tools where the grammar needs to handle legacy ambiguities: the Elkhound parser generator (UC Berkeley, 2003) was used in the CCured C analyzer and handled full C++. Bison has GLR mode since version 1.50 (2002).

PEG — Parsing Expression Grammars (2004)

Bryan Ford’s 2004 POPL paper Parsing Expression Grammars: A Recognition-Based Syntactic Foundation introduced PEGs as an alternative to BNF-style context-free grammars. The crucial difference: PEG’s alternation operator is ordered. A / B means “try A first, and only try B if A fails.”

PEGs have features BNF lacks:

  • Ordered alternation A / B instead of |
  • And-predicate &A — lookahead: succeed if A would match, but consume nothing
  • Not-predicate !A — negative lookahead
  • Greedy repetition A*, A+ — always consumes maximally, no backtracking
  • No ambiguity possible — the ordered-choice operator picks a single parse tree deterministically

Packrat parsing, Ford’s companion paper (Packrat Parsing, ICFP 2002), showed that PEGs can be parsed in linear time by memoizing intermediate results. The memoization table can be large, but for practical grammars it is tractable.

Python switched to a PEG-based parser in 2020 (Python 3.9, PEP 617). Guido van Rossum himself wrote the PEG implementation and blogged about the transition. The old pgen LL(1) generator was retired. The move gave Python access to features that were awkward in LL(1), like the walrus operator (:=) in arbitrary contexts.

Parser Combinators

Parsec (Daan Leijen, 2001) is the famous Haskell parser combinator library. Its influence is immense — most modern combinator libraries descend from it or from a library inspired by it.

A parser combinator library provides small parser primitives and higher-order functions to combine them. A parser is a first-class value in the host language; you build complex parsers by composing simple ones.

Advantages: full integration with the host language (types, module system, tooling), flexible and easy to extend, and the parser can be modified at runtime. Disadvantages: slower than generated parsers (usually), no global grammar optimization, and no separate grammar specification to read.

Modern combinator libraries:

  • Parsec / attoparsec / megaparsec — Haskell. The canonical combinator libraries.
  • nom (Rust) — binary and text parser combinators with streaming support. Used by Bevy, WebAssembly parsers, and others.
  • chumsky (Rust) — modern combinators with excellent error recovery and error messages.
  • combine (Rust) — older Rust combinator library.
  • pyparsing (Python) — Python combinators, long-standing.
  • Parboiled (Java/Scala) — PEG-based combinators on the JVM.

Earley Parsing — The General CFG Algorithm

Jay Earley’s 1968 PhD thesis at Carnegie Mellon, and his 1970 CACM paper An Efficient Context-Free Parsing Algorithm, introduced an algorithm that parses any context-free grammar, including ambiguous and left-recursive ones, in O(n3) worst case, O(n2) for unambiguous grammars, and O(n) for LR grammars.

Earley parsing is less famous than LR and LL because for most programming languages, the more restrictive LL(1) or LALR(1) is fast enough. But Earley has staged a comeback in two places: natural language processing (where grammars are often ambiguous by necessity), and modern parsing libraries that want to support arbitrary user grammars. Marpa (Jeffrey Kegler), the Python library Lark, and Nearley (JavaScript) all use variants of Earley parsing.

LALRPOP, Menhir, and Modern Generators

  • LALRPOP (Rust) — LALR(1) parser generator that integrates with Rust’s type system. Grammar lives in a .lalrpop file; the generator produces Rust code. Used by RustPython and several research compilers.
  • Menhir (OCaml) — LR(1) parser generator, the successor to ocamlyacc. Can handle canonical LR(1), not just LALR(1). Its error message generator is particularly good.
  • Happy (Haskell) — LALR(1) generator, used by GHC (the Haskell compiler) for parsing Haskell itself.
  • JavaCC (Java) — LL(k) parser generator, older, still used in some Java-based projects.
  • Jison (JavaScript) — Bison port for JavaScript. Used by CoffeeScript’s compiler.

Tree-sitter — Incremental Parsing (2018)

§ Parsing Every Keystroke

Max Brunsfeld at GitHub built Tree-sitter around 2018, initially for the Atom editor. Tree-sitter is a parser generator designed for one specific use case that the previous generation of generators did poorly: running inside a text editor on every keystroke, fast enough to not lag the user’s typing.

The requirements Tree-sitter met:

  • Fast. Parses in milliseconds for typical source files, microseconds for incremental updates.
  • Incremental. Given a small edit to a source file, Tree-sitter reparses only the affected subtree. The rest of the tree is reused.
  • Robust. Broken input (the half-typed state of a file the user is editing) must still produce a useful tree. Tree-sitter uses error recovery to continue past syntactically invalid regions, emitting error nodes.
  • Language-agnostic. One library, dozens of language grammars. Grammar files are in a JavaScript-based DSL that gets compiled to a C parser.

The parsing algorithm is a GLR variant that handles ambiguity by maintaining multiple stacks. Because Tree-sitter parses on every keystroke, it cannot refuse to parse syntactically broken input — the user has just typed a character, and the tree has to be ready for highlighting before the next character arrives.

Tree-sitter is now used by:

  • Atom (the original home)
  • Neovim (since 0.5, 2021) — built-in syntax highlighting uses Tree-sitter
  • Helix — Rust-based modal editor, Tree-sitter for everything syntactic
  • Emacs (since version 29, 2023) — treesit.el module
  • VS Code — in recent versions for some languages
  • Zed — the collaborative editor from the Atom team
  • GitHub — for syntax highlighting on GitHub.com

Over 200 Tree-sitter grammars exist, covering every mainstream programming language and most niche ones. A Tree-sitter grammar is written in JavaScript and looks like EBNF with JavaScript hooks. The generated parser is C code that is then compiled and loaded dynamically. See tree-sitter.github.io for the ecosystem.

The Long Arc

The arc is visible in hindsight:

1959 Backus’s ICIP paper

BNF as notation.

1960 ALGOL 60 Report

First language fully specified in BNF.

1965 Knuth LR(k) paper

Theoretical foundation of bottom-up parsing.

1968 Earley’s thesis

General CFG parsing in O(n3).

1969 DeRemer LALR(1)

Practical LR parsing with compressed tables.

1975 Yacc + Lex at Bell Labs

Johnson’s LALR(1) generator + Lesk & Schmidt’s lexer generator. The Unix parser toolchain.

1977 Wirth EBNF paper

“What Can We Do About the Unnecessary Diversity of Notation for Syntactic Definitions” — CACM.

1985 Bison

GNU yacc replacement. GPL.

1986 Tomita GLR

Generalized LR for ambiguous grammars.

1986 Dragon Book

Aho, Sethi, Ullman Compilers. The canonical LR-centric compiler textbook.

1990s ANTLR (PCCTS)

Parr’s LL(*) generator. Multiple target languages, readable errors.

2001 Parsec (Haskell)

Leijen’s monadic parser combinators. The canonical combinator library.

2002 Ford packrat parsing

Linear-time parsing with memoization.

2004 Ford PEG paper

Parsing Expression Grammars as BNF alternative.

2013 ANTLR4

Adaptive LL(*). Much better error recovery.

2018 Tree-sitter

Incremental GLR for editors. Brunsfeld at GitHub.

2020 Python switches to PEG

PEP 617. Python 3.9. Retires pgen.

2026 BNF still in use

67 years on, every modern language manual still has a BNF-descended grammar section.

BNF is the connective tissue. The grammar file in Yacc, in ANTLR, in Tree-sitter, in every parser generator, is a close descendant of the notation in the ALGOL 60 Report. An ALGOL 60 programmer reading a Tree-sitter grammar in 2025 would recognize the shape of it immediately. Sixty-five years of parser technology have not displaced the notation.
