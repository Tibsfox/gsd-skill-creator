The ALGOL 60 Report

Block structure, call-by-name, proper recursion, Jensen’s device, and the 20-page document that taught the world how to specify a language

“The Report is a classic in the literature of programming language definition. Its authors resolved many issues that were left unresolved in previous languages, and at the same time left a specification that was sufficiently precise for the language to be implemented.”

— Peter Naur, 20 years later

The Three Languages of ALGOL

The ALGOL 60 Report opens with a distinction that no previous language specification had made clearly. ALGOL, it says, is not one language but three:

  • The reference language is the one specified in the Report. It uses a notation with Greek letters, boldface keywords, and genuine mathematical symbols (×, ÷, ≤, ≥, →, ≠). This is the canonical form.
  • The publication language is what you would print in a journal. It could use different typographical conventions as long as the mapping back to the reference language was clear. Boldface keywords. True mathematical notation.
  • The hardware language is whatever a particular implementation actually accepts. Since most machines of 1960 could not display Greek letters or boldface, implementations had to ASCII-ify. This led to stropping: marking keywords with apostrophes, quotes, or dots to distinguish them from identifiers — 'BEGIN' or .BEGIN or "BEGIN".

The three-language ideal was a response to a real problem. In 1960, you might write an algorithm at Copenhagen, send it to a colleague at Munich, who would read it on a machine in Zurich, who would then implement it at Berkeley. Each machine had different character sets, different keyboards, different terminal capabilities. The reference/publication/hardware distinction let all three talk to each other.

The publication language ideal: ALGOL programs were supposed to be typeset for journals using real mathematical notation. Keywords in boldface. Relational operators as proper mathematical symbols. Subscripts as actual subscripts. When you read an ALGOL 60 paper in Communications of the ACM from 1961, you are reading the language as it was meant to be seen. When you type ALGOL into a teletype, you are reading the hardware language, an approximation. The reference language sits between them, connecting the two.

Block Structure and Lexical Scope

If ALGOL 60 had given us nothing but block structure, it would still be one of the most important languages in history. Every block-structured language — C, Pascal, Ada, Modula-2, Oberon, Java, C#, JavaScript, Rust, Go, Swift — is a direct descendant.

The Block as a Unit

A block in ALGOL 60 is a sequence of declarations followed by a sequence of statements, enclosed in begin and end:

begin
  integer i, j;
  real x;
  i := 0;
  x := 3.14;
  j := i + 1
end

The declarations introduce names; the statements use them. The names are local to the block: they come into existence when control enters the begin, and they cease to exist when control leaves the end.

This sounds utterly obvious today. It was not obvious in 1960. Fortran had no blocks. COBOL had paragraphs and sections but no lexical nesting. Assembly had labels and nothing else. The idea that a program could be a tree of nested scopes, each with its own declarations, was a genuine invention.

Nested Blocks and Shadowing

begin
  integer x;
  x := 1;
  begin
    integer y;
    y := x + 1;
    begin
      integer x;
      x := y * 2;
      comment inner x shadows outer x
    end;
    comment here x is 1 again, y is 2
  end;
  comment here only x (=1) is visible
end

When the inner block declares its own x, the outer x is shadowed — not destroyed, just hidden. This is lexical scoping: the meaning of a name at a given point in the program text is determined by the innermost enclosing declaration of that name.

The Stack Discipline and the Display

Blocks nest. When control enters a block, we push that block’s locals onto an allocation area; when control exits, we pop them off. The stack discipline is what makes ALGOL 60 efficiently compilable. A block’s locals live at a known offset from the top of the stack at the moment the block is active.

For nested procedures with lexical scope, Dijkstra introduced the display: an array indexed by static nesting level, where display[k] always points to the activation record of the enclosing block at level k. Accessing a variable at level k becomes display[k] + offset, two loads instead of walking a chain. Dijkstra described this in his 1960 paper on recursive procedures; the Burroughs B5000 implemented it in hardware.

Proper Recursion

Fortran I (1957) explicitly forbade recursion. The compiler allocated local variables statically — each subroutine had one activation record permanently assigned to it. A recursive call would silently corrupt the program.

ALGOL 60 took the opposite position. Section 5.4.4 of the Report makes recursion explicit. McCarthy (fresh from Lisp) insisted on it. Dijkstra argued that recursion was the natural consequence of block structure, and that a language which could not express recursive definitions was fighting against mathematics itself.

integer procedure factorial(n); value n; integer n;
begin
  if n = 0 then
    factorial := 1
  else
    factorial := n * factorial(n - 1)
end

Note the assignment-to-function-name syntax: the result of an ALGOL 60 typed procedure is whatever value has been assigned to the procedure’s name inside the body. There is no return statement — that would come in Pascal, C, and their descendants.

Knuth’s “Man or Boy?” Test

In 1964, Donald Knuth published a short test program designed to separate real ALGOL 60 implementations from fake ones. Any implementation that truly supported all of: proper lexical scoping, proper recursion, non-local access via a display, and call-by-name with thunks, would produce a specific sequence of outputs. Any implementation that cut corners would crash, loop, or produce wrong answers.

begin
  real procedure A(k, x1, x2, x3, x4, x5);
    value k; integer k;
  begin
    real procedure B;
    begin
      k := k - 1;
      B := A := A(k, B, x1, x2, x3, x4)
    end;
    if k <= 0 then A := x4 + x5 else B
  end;
  outreal(A(10, 1, -1, -1, 1, 0))
end

Knuth called it the “Man or Boy?” test because an ALGOL 60 compiler that handled it correctly was a “man”; one that did not was a “boy.” The expected output for A(10, 1, -1, -1, 1, 0) is -67. Getting this right requires juggling non-local access through multiple levels of the display, call-by-name thunks that reference outer variables by aliasing, and recursive procedures whose formal parameters are themselves procedures. It was the hardest ALGOL 60 test case ever devised, and it separated compilers that were faithful to the Report from those that cheated.

Call-By-Value vs Call-By-Name

Call-by-name is the single weirdest thing in ALGOL 60. It is also the feature that most influenced theoretical computer science, inspired the development of denotational semantics, and — in a direct line — led to the lazy evaluation of Haskell and the by-name parameters of Scala.

Call-by-Value

The easy case. When you call a procedure and pass an expression as a parameter, the expression is evaluated once at the call site, and the resulting value is bound to the formal parameter. Subsequent uses of the formal parameter look up that stored value.

integer procedure square(n); value n; integer n;
begin square := n * n end

ALGOL 60 required explicit marking for value parameters via value n; otherwise, the default was call-by-name.

Call-by-Name: The Copy Rule

Call-by-name is specified by the copy rule. The compiler does not evaluate the actual parameter. Instead, it substitutes the text of the actual parameter expression in place of every reference to the formal parameter inside the body, as if by textual replacement, and then evaluates the resulting body.

Put another way: every reference to the formal parameter inside the body re-evaluates the actual parameter expression, in the caller’s scope, every time.

integer procedure double(x); integer x;
begin double := x + x end

No value x — so x is call-by-name. If you call double(f()) where f is a procedure with side effects, then f is called twice, once for each reference to x in the body.

Thunks

How do you implement call-by-name? You cannot literally substitute text, because the procedure body is already compiled. Instead, the compiler generates a small anonymous procedure — a thunk — for each by-name actual parameter. The thunk, when called, evaluates the actual parameter expression in the caller’s environment (it has access to the caller’s variables via the display) and returns the current value.

The term thunk was coined for exactly this purpose, reportedly by Peter Ingerman in 1961. Folklore has it that it was chosen because the value of the parameter had “already been thought about” by the time the procedure used it — the past tense of think being, jokingly, thunk. The word stuck, and is now the standard term in compiler literature for a zero-argument closure that lazily computes a value.

Jensen’s Device

The most famous use of call-by-name is Jensen’s device, named after Jørn Jensen, Peter Naur’s colleague at Regnecentralen in Copenhagen. Jensen’s device exploits call-by-name to build a crude form of higher-order function — a decade before ML and twenty years before anyone was using the phrase first-class functions in mainstream programming.

§ The Idea

Pass an expression as a by-name parameter, along with a by-name parameter that is a variable used inside that expression. When the procedure body mutates the variable and then references the expression, the expression re-evaluates with the new value of the variable. You have effectively passed a function of one argument using only call-by-name.

The classic summation procedure:

real procedure sum(i, lo, hi, term);
  value lo, hi;
  integer i, lo, hi;
  real term;
begin
  real s;
  s := 0;
  for i := lo step 1 until hi do
    s := s + term;
  sum := s
end

The parameters lo and hi are passed by value — they are just numbers. But i and term are by-name. You call it like this:

real total;
integer k;
total := sum(k, 1, 100, k * k)

Inside sum, the for loop increments k from 1 to 100 (because i is by-name and aliased to k). For each value of k, the reference to term inside the loop re-evaluates k * k in the caller’s scope, using the current value of k. The result is the sum of squares from 1 to 100.

You have, in effect, just written sum(f) = Σ f(k) for k = 1..100, passing f as the expression k * k and k as the summation variable. This is a lambda in disguise. Without higher-order functions, without lambda expressions, without function types, Jensen figured out how to pass functions around.

You can use the same trick for integration, sorting by a key function, searching with a predicate — anywhere you would pass a function today.

The Aliasing Problem: Why swap Fails

Jensen’s device is elegant. Call-by-name also has a notorious failure mode. Consider the obvious swap procedure:

procedure swap(a, b); integer a, b;
begin
  integer t;
  t := a;
  a := b;
  b := t
end

With call-by-name, calling swap(i, j) works fine. Now call swap(i, A[i]). This should swap i and A[i]. But here is what happens:

  1. t := a — look up a, which is the thunk for i. Current value: say, 3. So t := 3.
  2. a := b — look up b, which is the thunk for A[i]. Current i is 3, so look up A[3]. Say it is 99. Assign 99 to a, which means assigning 99 to i. Now i = 99.
  3. b := t — assign t (which is 3) to b. b is the thunk for A[i]. But i is now 99, so this assigns 3 to A[99], not to A[3].

The array element we were trying to swap with is still untouched. We have corrupted A[99] instead. The by-name semantics re-evaluates the subscript expression every time we reference b, and because step 2 mutated i, step 3 references a different array element.

This is the aliasing problem, and it cannot be fixed by clever programming. It is a fundamental consequence of the copy-rule semantics. It made generic utility procedures like swap dangerous to write in ALGOL 60. This bug — or rather the recognition of it — was part of what killed call-by-name in mainstream language design.

The Descendants of Call-By-Name

Call-by-name did not die. It was reborn in cleaner form in several places:

  • Lazy evaluation (call-by-need) is call-by-name with memoization: the expression is evaluated the first time the formal parameter is referenced, and the result is cached so subsequent references reuse it. Haskell is built entirely on call-by-need. Clean and Miranda before it, the same.
  • Scala’s by-name parameters: Scala has explicit => T parameter types that behave exactly like ALGOL 60 call-by-name. Used heavily for control abstractions, assertions, and logging.
  • Lisp macros are essentially call-by-name at the syntactic level. The macro arguments are re-inserted into the body textually.
  • Denotational semantics: Strachey and Wadsworth’s 1974 paper “Continuations: A Mathematical Semantics for Handling Full Jumps” developed the mathematical framework for describing languages like ALGOL 60 precisely. Strachey’s earlier work on the semantics of call-by-name directly motivated the development of lambda-calculus-based language semantics, which eventually became the foundation of all modern programming language theory.

Haskell exists because ALGOL had call-by-name.

Control Flow

If-Then-Else

ALGOL 60 had if as both a statement and an expression. The expression form was John McCarthy’s contribution:

x := if y > 0 then y else -y

This was new in 1960. Fortran’s arithmetic IF was a control-flow statement, not an expression that returned a value. The if-expression is now standard in every language with any functional sensibility: Scheme, ML, Haskell, Rust, Kotlin, Scala, modern Python (via the a if c else b form).

The For Loop

ALGOL 60’s for loop is almost bizarrely flexible:

for i := 1 step 2 until 100 do S;          comment 1, 3, 5, ..., 99;
for i := 1, 3, 5, 7, 11 do S;               comment explicit list;
for i := 1 step 2 until 10, 20 step 5 until 50, 100 do S;  comment mixed;
for i := 1 step 1 while x < 100 do S;      comment while condition;

The for-list can mix steps, explicit values, while clauses. This was too clever and led to genuine difficulties compiling it correctly. Pascal and its descendants simplified to a cleaner form. But it was a design choice: the committee believed iteration should be expressible as a declarative specification of the sequence of values the index took.

The Dangling Else

ALGOL 60 introduced the notorious “dangling else” ambiguity:

if A then if B then S1 else S2

Does S2 belong to the inner if or the outer? The Report’s convention was that else binds to the nearest unmatched if. This is a context-free grammar ambiguity that every subsequent block-structured language has had to handle somehow — usually by the same convention, sometimes by forcing braces. Python eliminates it by using indentation. Go eliminates it by requiring braces always.

Hoare’s Quicksort — Algorithm 64

The most cited algorithm in computing was first published in ALGOL 60. Here is Hoare’s Algorithm 64: Quicksort from Communications of the ACM, July 1961, with the Algorithm 65: Find (the partition procedure) as its helper:

procedure quicksort (A, M, N);
  value M, N;
  array A; integer M, N;
  comment Quicksort. Sorts A[M:N] in place;
begin
  integer I, J;
  if M < N then
  begin
    partition (A, M, N, I, J);
    quicksort (A, M, J);
    quicksort (A, I, N)
  end
end quicksort;

procedure partition (A, M, N, I, J);
  value M, N;
  array A; integer M, N, I, J;
begin
  real F;
  I := M;
  J := N;
  F := A[(M+N) div 2];
up: if A[I] < F then
    begin I := I + 1; goto up end;
down: if A[J] > F then
    begin J := J - 1; goto down end;
  if I <= J then
  begin
    real t;
    t := A[I]; A[I] := A[J]; A[J] := t;
    I := I + 1; J := J - 1
  end;
  if I <= J then goto up
end partition;

Note the use of goto with labels (up:, down:) — this was before Dijkstra’s 1968 letter made gotos unfashionable. Note the comment syntax. Note div for integer division. Note the block-structured begin/end with local variable t. This is ALGOL 60 exactly as Naur specified it.

Every Quicksort implementation in every language since is a translation of this code. Hoare won the Turing Award in 1980, partly on the strength of this algorithm.

Common Lisp of Europe

ALGOL 60 was to Europe what Lisp was to MIT: the language of the theoretical computing community. Dijkstra, Wirth, Hoare, Dahl, Nygaard, Bauer, Samelson, Randell, van Wijngaarden — nearly every European computer science pioneer of the 1960s and 1970s learned their craft on ALGOL 60. The language shaped how they thought about programming, and when they built their own languages — Pascal, Simula, Concurrent Pascal, Modula, Oberon — they carried ALGOL’s sensibilities with them.

Meanwhile in America, FORTRAN and COBOL dominated commercial work, and Lisp owned the research AI world. Europe’s programming culture was shaped by a language that America barely noticed. The gap mattered: European universities taught structured programming, formal specification, compiler theory, and correctness proofs, all rooted in ALGOL 60. American universities taught Fortran numerical methods and, later, C systems programming.

Some of this cultural divide is still visible today in the difference between the Pascal/Modula/Oberon teaching tradition (European) and the C/Unix professional tradition (American). The two converged in Java, C#, Rust, and Go — all of which are, in different proportions, ALGOL-family languages that inherited both traditions.
