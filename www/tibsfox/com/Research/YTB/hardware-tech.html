1. VLIW: The Impossible Computer

Very Long Instruction Word (VLIW) architecture represents one of the most ambitious and instructive experiments in computer architecture history. The idea is seductively elegant: instead of having expensive hardware dynamically discover instruction-level parallelism at runtime, encode the parallelism explicitly at compile time. Bundle multiple independent operations into a single very long instruction word, and let the hardware execute them all simultaneously with minimal control logic. The compiler does the hard work; the processor stays simple, fast, and power-efficient.

Origins and the Yale Experiments

The VLIW concept emerged from Josh Fisher's research at Yale University in the early 1980s. Fisher's insight was that a sufficiently intelligent compiler could analyze program dependencies, schedule operations across functional units, and pack independent instructions into wide instruction words—eliminating the need for the complex scoreboarding and reservation station hardware that dynamic scheduling required. Fisher went on to co-found Multiflow Computer in 1984, producing the TRACE series of VLIW minicomputers. Multiflow's machines had instruction words up to 1,024 bits wide, encoding up to 28 operations per cycle. The company produced working systems and real customers, but the minicomputer market was collapsing under the weight of increasingly powerful microprocessors. Multiflow folded in 1990.

Cydrome, another VLIW startup founded by Bob Rau, pursued a similar approach for numerical computation. Both companies demonstrated that VLIW could deliver impressive performance for structured, loop-heavy code—but neither could overcome the fundamental market reality that general-purpose computing demanded something more adaptable.

EPIC and the Itanium Gamble

The largest VLIW-derived bet in computing history began in 1994, when Intel and Hewlett-Packard partnered to develop IA-64, a new instruction set architecture based on a refined version of VLIW concepts that Intel branded Explicitly Parallel Instruction Computing (EPIC). The architecture encoded parallelism information directly in 128-bit instruction bundles, each containing three operations plus a template field indicating which operations could execute simultaneously. The processor had 128 general-purpose registers, 128 floating-point registers, 64 predicate registers for speculative execution, and a register stack engine for procedure calls.

The first Itanium processor (codenamed Merced) shipped in 2001—years behind schedule. It was immediately apparent that the architecture had fundamental problems:

  • Compiler complexity was underestimated by an order of magnitude. Donald Knuth observed that "the wished-for compilers were basically impossible to write." The compiler had to predict cache behavior, branch outcomes, and memory latencies at compile time—information that varies dramatically based on runtime data patterns.
  • Code size exploded. VLIW bundles contain explicit NOP operations when the compiler cannot find enough parallelism to fill all slots. Real-world programs, with their irregular control flow and pointer-heavy data structures, left many slots empty.
  • Branch prediction at compile time proved intractable. Branches are fundamentally unpredictable in general-purpose code. Hardware branch predictors in out-of-order processors could adapt to runtime patterns; static prediction could not.
  • x86 emulation was catastrophically slow. Running legacy 32-bit x86 code on Itanium required software emulation that performed at a fraction of native x86 speed, destroying the migration path.
  • AMD64 flanked the entire strategy. While Intel was betting on a clean architectural break, AMD simply extended x86 to 64 bits. AMD64 provided the 64-bit address space that customers needed while maintaining full backward compatibility. Intel was forced to adopt AMD's extensions as "Intel 64."
"The Itanium approach asked the compiler to do something it fundamentally could not: predict the future behavior of programs whose execution depends on data that does not exist until runtime." — Analysis based on Clemson University EPIC historical review

Intel shipped Itanium processors through 2017 (Itanium 9700 series, codenamed Kittson), but the architecture never achieved significant market share. Total Itanium revenue over its lifetime was a fraction of what Intel and HP invested in the program, making it one of the most expensive architectural failures in semiconductor history.

Where VLIW Succeeded

While VLIW failed catastrophically for general-purpose computing, it found extraordinary success in domains where its strengths—simple hardware, deterministic execution, power efficiency—aligned with workload characteristics:

The Texas Instruments C6000 family, launched in 1997, remains one of VLIW's greatest commercial successes. Its eight functional units—two multipliers (M1, M2), two ALUs per side (L1, L2, S1, S2), and two load/store units (D1, D2)—can execute eight 32-bit instructions per cycle from a single 256-bit fetch packet. For the structured, loop-heavy, highly parallel code that DSP workloads produce, the compiler can reliably fill these slots.

The Qualcomm Hexagon is arguably the most successful VLIW processor ever deployed by volume. Found in every Snapdragon SoC—and therefore in hundreds of millions of smartphones—Hexagon handles always-on sensor processing, audio wake-word detection, camera image processing, and neural network inference. Its VLIW design achieves significant power savings by avoiding the complex out-of-order execution hardware that general-purpose cores require. By the 2020s, Hexagon evolved from pure DSP into a hybrid DSP/NPU, adding tensor acceleration while retaining its VLIW core.

The Lesson: Hardware Adapts, Compilers Predict

The fundamental lesson of VLIW is about the boundary between compile-time and runtime knowledge. Out-of-order execution, based on Tomasulo's algorithm (IBM System/360 Model 91, 1966), won because it dissolves dependencies dynamically using register renaming, reservation stations, and a reorder buffer. Modern superscalar processors like AMD Zen 5 and Apple M4 execute six or more instructions per cycle using hardware that adapts to whatever the program actually does at runtime—not what a compiler predicted it might do. For structured, predictable workloads (DSP, media, sensors), VLIW remains ideal. For the irregular, branch-heavy, pointer-chasing code of general-purpose computing, dynamic scheduling won decisively.

2. Mexico City: The Sinking Lands

Mexico City is experiencing one of the most dramatic geological crises of any major city on Earth. Built on the drained bed of a pre-Columbian lake system, the city of 21 million people is sinking at rates that vary from negligible (where it rests on volcanic bedrock) to catastrophic—up to 50 centimeters per year in the worst-affected zones. Since the beginning of the twentieth century, some areas have subsided by more than 12 meters—the height of a four-story building. The process is irreversible, accelerating, and threatens the habitability of entire districts.

The Geology: A City Built on a Lake

The Valley of Mexico was once a closed hydrological basin containing a system of interconnected lakes, the largest of which was Lake Texcoco. The Aztec capital of Tenochtitlan, founded around 1325, was built on an island in Lake Texcoco and connected to the mainland by causeways. The Aztecs engineered an elaborate system of dikes, aqueducts, and chinampas (floating gardens) that worked with the lake rather than against it.

After the Spanish conquest of 1521, Hernan Cortes ordered the reconstruction of the city atop Tenochtitlan's ruins. The Spanish approach was fundamentally different from the Aztec one: rather than managing water, they sought to eliminate it. Over the following centuries, the lake system was progressively drained through a series of canals and tunnels. By the early twentieth century, Lake Texcoco was essentially gone.

What remained beneath the city was a 100-meter-thick layer of highly compressible, salty, clay-rich lacustrine sediment—the accumulated deposits of thousands of years of lake-bottom sedimentation. When saturated with water, these clays maintain their volume. When the water is extracted, the very fine mineral grains repack themselves more tightly in a process called consolidation. The clay compresses irreversibly, and the surface above it drops.

The Aquifer Crisis

Mexico City's aquifer system provides approximately 70 percent of the water the city consumes. The remainder comes from the Cutzamala system, which transfers water from basins in neighboring Michoacan and the State of Mexico—pumping it over mountains at enormous energy cost. The Cutzamala reservoirs have been operating at approximately 30 percent capacity in recent years.

Extraction rates far exceed natural recharge. Experts disagree on how long the aquifer can sustain current pumping rates, with estimates ranging from 5 to 20 years before critical depletion. The cruel paradox is inescapable: the more water the city pumps, the faster it sinks, the more its water infrastructure breaks, the more water it loses to leaks (estimated at 40 percent of supply), and the more it must pump to compensate. Climate change amplifies the crisis through the urban heat island effect—higher temperatures drive higher water demand, and altered precipitation patterns reduce recharge.

Infrastructure Under Siege

Differential subsidence—where adjacent areas sink at different rates—is more destructive than uniform sinking. When one side of a building, pipeline, or rail line sinks faster than the other, the structure experiences shearing forces it was never designed to withstand.

  • The Metro system: Research published in 2024 found that nearly half of the Mexico City Metro's elevated segments are experiencing differential subsidence at rates that would require servicing well before the system's designed 50-year lifespan. Rail alignments shift, stations tilt, and structural joints are stressed beyond design tolerances.
  • Water infrastructure: Broken water mains are both a consequence of subsidence (pipes crack as the ground shifts) and a contributor (leaked water erodes the clay matrix, accelerating local compaction). The city loses an estimated 1,000 liters per second to infrastructure leaks.
  • Historic buildings: The Palacio de Bellas Artes has sunk several meters since its construction. The Metropolitan Cathedral, built on the ruins of the Aztec Templo Mayor, has required ongoing stabilization work to prevent structural failure.

Engineering Responses

The most ambitious engineering response is the Tunnel Emisor Oriente (TEO), at 62.5 kilometers the eighth-longest tunnel in the world and the largest wastewater tunnel ever constructed. Built between 2008 and 2019, it runs at depths of up to 150 meters, bypassing the subsiding surface entirely. The tunnel carries wastewater from Mexico City to the Atotonilco treatment plant in Hidalgo state, serving a drainage basin of 20 million people. It replaced the aging Gran Canal del Desague, which had been designed with a gravity-flow gradient that subsidence had reversed—sewage was flowing backwards toward the city center.

Other responses include deep wells regulation (limiting new extraction permits), rainwater harvesting programs, wastewater reuse, and the controversial Lago Texcoco Ecological Park project, which aims to restore a portion of the former lakebed as wetland to improve aquifer recharge.

3. From Fiber to AI: The Laser Giant's Rebirth

The companies that survived the 2000-2003 telecommunications crash are now at the center of the most explosive infrastructure buildout since the internet itself. The very technologies that nearly bankrupted them—optical transceivers, laser diodes, wavelength-division multiplexing—have become the critical bottleneck in AI datacenter construction. It is a remarkable second act, and it is reshaping the photonics industry.

The Telecom Bust and Its Survivors

The late 1990s telecom bubble produced a massive overbuilding of fiber optic infrastructure. Network operators deployed vastly more fiber capacity than the market demanded, driven by projections of exponential internet growth that were directionally correct but temporally premature. When the bubble collapsed in 2000-2001, spending on optical transport equipment cratered. Companies like JDS Uniphase, Nortel, Corning, and dozens of component makers saw revenues fall by 80-90 percent. JDS Uniphase wrote off $50 billion in goodwill from its bubble-era acquisitions—the largest corporate write-down in history at the time.

The survivors consolidated ruthlessly. II-VI Incorporated, founded in 1971 in Saxonburg, Pennsylvania, as a specialty materials company (named for the II-VI semiconductor compound families), began acquiring distressed optical companies: Laser Power Optics (2000), Marlow Industries (2004), HIGHYAG (2007), Photop Technologies (2010), and Finisar (2019, for $3.2 billion). In 2022, II-VI completed its transformative $6.56 billion acquisition of Coherent, Inc.—a laser pioneer founded in Palo Alto in 1966—and adopted the Coherent Corp. name for the combined entity.

Lumentum Holdings, spun off from JDS Uniphase (itself a survivor of the bubble) in 2015, focused on vertical-cavity surface-emitting lasers (VCSELs) and coherent optics. Ciena, Infinera, and Broadcom's optical division each carved out niches in the post-bust landscape.

The AI Bandwidth Explosion

The resurgence of these companies is driven by a simple physical fact: AI training clusters require staggering amounts of interconnect bandwidth, and electrical signaling cannot deliver it at the required distances. A single NVIDIA H100 GPU has 900 GB/s of NVLink bandwidth. A cluster of 32,768 such GPUs—common in frontier model training—requires an aggregate fabric bandwidth measured in petabits per second. The data must flow between GPU racks separated by tens to hundreds of meters, and at these distances, only optical interconnects can deliver the required bandwidth at acceptable power consumption and signal integrity.

Coherent Corp. demonstrated its 800G-DR4 transceiver in OSFP form factor (four CWDM wavelengths at 200G PAM4 each) and its 1.6T-DR8 module (eight optical lanes at 200 Gbps each, using an NVIDIA DSP). The company's DFB-MZ technology—a distributed feedback laser monolithically integrated with an InP Mach-Zehnder modulator—enables 1.6T transceivers with up to 10 km reach. 800GbE optics shipments grew over 60 percent in 2025, and the datacom optical component market exceeded $16 billion in revenue.

Silicon Photonics and Co-Packaged Optics

The next frontier is co-packaged optics (CPO)—integrating optical transceivers directly into the switch or GPU package rather than using separate pluggable modules. NVIDIA's Quantum-X photonic switch delivers 115.2 Tbps total throughput using CPO modules, claiming 3.5x lower power consumption compared to pluggable transceivers. The elimination of the electrical path between the ASIC and the transceiver reduces both latency and power consumption.

Silicon photonics—fabricating optical components on standard silicon wafers using CMOS-compatible processes—is the key enabling technology. The market for silicon photonics is projected to grow from $95 million in 2023 to over $863 million in 2029. Intel, Broadcom, Marvell, Ranovus, and Ayar Labs are all pursuing silicon photonics approaches, while Coherent and Lumentum bring decades of III-V semiconductor (InP, GaAs) laser expertise that silicon cannot yet replicate for the light source itself.

"All AI datacenter interconnects will be optical within five years. The only question is whether the optics are pluggable or co-packaged." — Semi Engineering, 2025 datacenter analysis

The irony is complete: the "dark fiber" that represented billions in wasted telecom investment is now being lit up for AI traffic. The laser companies that nearly died building telecom infrastructure are now the bottleneck suppliers for the AI revolution. Their expertise in indium phosphide lasers, high-speed modulators, and coherent detection—developed over decades for long-haul telecom—is exactly what AI datacenters need at unprecedented scale.

4. Singapore's Sound Card Hero

The story of Creative Technology and the Sound Blaster is the story of how a small Singaporean company created an entire category of consumer electronics, dominated it for a decade, watched it dissolve into commodity integrated circuits, and has been searching for its next act ever since. It is also the story of Sim Wong Hoo, one of Southeast Asia's most consequential technology entrepreneurs.

Origins: A Computer Repair Shop in Chinatown

Creative Technology was founded in 1981 by Sim Wong Hoo and his Ngee Ann Polytechnic classmate Ng Kai Wa. The company started as a computer repair shop in Pearl's Centre, Singapore's Chinatown district. Sim, born in 1955, had an engineering background and an intuition for what personal computers lacked: multimedia capability. The company's first product was a memory expansion board for the Apple II. By the mid-1980s, Sim recognized that the IBM PC's audio capabilities—limited to a single-voice piezoelectric speaker capable of beeps and square waves—represented an enormous gap.

The Sound Card Wars

In 1987, Creative released the Creative Music System (C/MS), a 12-voice sound card using Philips SAA1099 chips. When C/MS struggled against AdLib's already-established card (based on the Yamaha YM3812 OPL2 FM synthesis chip), Sim traveled to Silicon Valley and negotiated a deal with RadioShack's Tandy division to market it as the "Game Blaster." The Game Blaster sold modestly, but Sim learned two critical lessons: AdLib compatibility was essential, and digital audio (sampled sound, not just FM synthesis) was the future.

The original Sound Blaster, introduced at COMDEX in November 1989, was a masterpiece of market positioning. It included:

  • A Yamaha YM3812 (OPL2) chip for full AdLib compatibility
  • The original C/MS chips for backward compatibility with Game Blaster titles
  • A digital-to-analog converter for 8-bit sampled audio playback and recording
  • A MIDI interface
  • A joystick/game port

This bundling strategy was decisive. For the same price as an AdLib card (or less), customers got AdLib compatibility plus digital audio plus a game port that saved an expansion slot. When Microsoft announced the Multimedia PC (MPC) specification in 1990, the Sound Blaster was the closest match among shipping products. Game developers began targeting Sound Blaster as the standard, and the flywheel effect took hold: more games supported Sound Blaster, so more users bought Sound Blaster, so more games supported Sound Blaster.

Creative's revenue exploded from $5.4 million in 1989 to $658 million in 1994. The company listed on NASDAQ in 1992—the first Singaporean company to do so—reaching a market capitalization of $1.6 billion by 1994.

Competitors and Their Fates

AdLib: The original PC sound card standard-bearer, AdLib could not match Creative's bundling strategy and lacked digital audio capability. The company went bankrupt in 1992.

Gravis UltraSound (GUS): Technically superior with its wavetable synthesis using real sampled instruments stored in on-board RAM, the GUS arrived after Sound Blaster compatibility had become the market requirement. Its Sound Blaster emulation was imperfect, and the "huge problem that GUS had was lack of content." A cult favorite among the demoscene and early Linux audio communities, it never achieved mainstream adoption.

Turtle Beach: Positioned as the premium option with high-quality DACs and the eMu synthesizer chip, Turtle Beach's MultiSound card targeted professional and enthusiast markets. It survived by staying in the niche that Creative did not prioritize.

Aureal: Perhaps Sound Blaster's most dangerous challenger. Aureal's Vortex2 (AU8830) chipset offered superior 3D positional audio through A3D technology, which used head-related transfer functions to create genuinely immersive spatial sound. Creative and Aureal fought a brutal legal war. In December 1999, Aureal won a favorable court ruling but went bankrupt from legal costs and investor withdrawal. Creative acquired Aureal's assets through bankruptcy court in September 2000 for $32 million—gaining the A3D technology and eliminating its most technically capable competitor.

By January 1994, Computer Gaming World declared: "The de facto standard in the gaming world is Sound Blaster compatibility."

The Integration Extinction Event

The Sound Blaster's dominance contained the seeds of its irrelevance. As the functionality became standardized, motherboard chipset manufacturers incorporated audio capability directly:

  • AC'97 (Audio Codec '97): Developed by Intel in 1997, AC'97 defined a standard audio codec interface that motherboard manufacturers could implement cheaply using dedicated codec chips. Quality was mediocre but sufficient for most users.
  • Intel High Definition Audio (HDA): Released in 2004 as AC'97's successor, Intel HDA supported higher sample rates, more channels, and better signal-to-noise ratios. It became universal on motherboards within two years.

When every motherboard shipped with "good enough" integrated audio, the mass market for dedicated sound cards evaporated. Jon Peddie Research reported a 12.3 percent decline in the sound card market between 2015 and 2020. Creative Technology, which had made Sim Wong Hoo Singapore's youngest billionaire at age 45 in 2000, saw its market cap decline by over 90 percent from its peak.

Pivot Attempts and Legacy

Creative attempted multiple pivots. The ZEN line of portable media players competed directly with the iPod—and in a remarkable legal outcome, Creative filed suit against Apple in 2006 for violating its "Zen patent" on hierarchical music navigation interfaces. Apple settled by paying Creative $100 million to license the patent. The Super X-Fi technology, introduced in the late 2010s, uses head-related transfer function mapping to create holographic audio in headphones—a computational audio approach that echoes the spatial audio ambitions of the Aureal technology Creative acquired two decades earlier.

Sim Wong Hoo passed away on January 4, 2023, at age 67. His nephew, Sim Li Ern, was appointed CEO in January 2026. Creative Technology continues to operate, with a market capitalization of approximately $36 million Singapore dollars—a fraction of its former glory, but still producing Sound Blaster products and pushing audio innovation from its Singaporean base in a country that now accounts for one in ten semiconductors produced worldwide.

5. MCP Dev Summit NYC Day 1: The Protocol Matures

The inaugural MCP Dev Summit North America, held April 2-3, 2026, at the New York Marriott Marquis, marked the moment the Model Context Protocol transitioned from experimental specification to production infrastructure standard. Hosted by the Agentic AI Foundation (AAIF) under the Linux Foundation, the event drew representatives from 170+ member organizations, featured 17 keynotes and over 95 sessions, and was sponsored by AWS, Docker, Anthropic, Google Cloud, IBM, Red Hat, Microsoft, OpenAI, and Datadog.

The Agentic AI Foundation

The AAIF, which governs MCP alongside the goose project and AGENTS.md specification, achieved a remarkable milestone: within three months of its January 2026 launch, it surpassed the Cloud Native Computing Foundation in membership size. Mazin Gilbert, formerly of AT&T Labs and Google, was appointed as inaugural Executive Director. The Foundation's velocity reflects the industry's recognition that agent interoperability requires neutral governance—no single vendor can own the protocol that connects all AI agents to all tools.

Day 1: Protocol Foundations and Authentication

Day 1 focused on the protocol's technical foundations and its most critical unsolved problem: authentication. Six dedicated sessions addressed MCP authentication, reflecting both the urgency of the problem and the ecosystem's convergence on a solution.

OAuth 2.1 Consensus: Aaron Parecki, the OAuth 2.1 specification author (from Okta), delivered what multiple attendees described as the summit's most clarifying talk: "Don't overthink auth in MCP." The message was that MCP should use existing OAuth 2.1 standards rather than inventing a new authentication scheme. The Python SDK v1.27.0, released the day of the summit, shipped with RFC 8707 OAuth resource validation for clients.

Cross-App Access (XAA): A significant new pattern emerged: Cross-App Access, which allows agents to carry user identity across services without requiring separate login flows for each tool. Okta's reference implementation at xaa.dev demonstrated the pattern, with early adopters including Automation Anywhere, Boomi, Box, and Glean. XAA addresses a fundamental UX problem: if an agent needs to call 15 different tools, each with its own OAuth flow, the user experience becomes untenable.

Conformance Testing: Paul Carleton from Anthropic presented "One Spec, Ten SDKs, Zero Excuses: Conformance Testing MCP," introducing a testing framework to ensure that the growing number of SDK implementations (Python, TypeScript, Rust, Go, Java, C#, and others) all implement the protocol identically. This is a maturity signal—the ecosystem has grown beyond the point where Anthropic can review every implementation manually.

Scope Discipline

Perhaps the most strategically significant announcement was what MCP would not do. David Soria Parra from Anthropic stated explicitly: "MCP should not attempt to own those problems," referring to observability, identity management, and governance. MCP handles "connecting AI applications to data sources"—period. Observability belongs to OpenTelemetry. Identity belongs to OAuth/OIDC. Governance belongs to policy engines. This scope discipline is a lesson learned from protocols that became bloated by trying to solve adjacent problems.

Production Deployments

Keynote presentations from Uber, Nordstrom, Bloomberg, Duolingo, and PwC demonstrated that MCP has moved beyond experimentation into production infrastructure. These organizations shared operational patterns including orchestration frameworks, agent gateways, and skill management architectures. The protocol registered 97 million monthly downloads across its SDK ecosystem, confirming substantial adoption velocity.

6. MCP Dev Summit NYC Day 2: Ecosystem and Enterprise

Day 2 shifted focus from protocol foundations to the ecosystem growing around MCP: SDK development, enterprise deployment patterns, and the emerging commercial infrastructure layer.

SDK V2 Roadmap

Max Isbey from Anthropic presented "Path to V2 for MCP SDKs," revealing that V2 development is active on the main branch of the MCP SDK repositories, with v1.x maintenance continuing on a separate branch. The TypeScript SDK was further along, having shipped v2.0.0-alpha.1 and alpha.2 with Standard Schema support (enabling Zod v4, Valibot, and ArkType for schema definition), a refactored TaskManager, and a new Fastify framework package. The Python V2 had no firm release timeline, with the earliest community expectation being Q3 2026.

Key V2 features include:

  • StreamableHTTP idle timeout: Preventing resource exhaustion from abandoned connections
  • TasksCallCapability: Backported from V2 to v1.27.0, enabling long-running task management
  • Standard Schema support: Decoupling schema validation from any single library

Cross-Platform Convergence

A telling signal of MCP's emerging dominance: OpenAI's Agents SDK v0.13.0, released contemporaneously, shipped list_resources(), read_resource(), and list_resource_templates() methods that match Anthropic's Python SDK implementation. When your primary competitor implements your protocol's interface in their SDK, the standard has won. Nick Cooper from OpenAI presented an Agents SDK keynote, reinforcing the convergence message.

The Server Quality Problem

Multiple sessions addressed what speakers called the "server quality problem"—vendors wrapping existing REST APIs in MCP without redesigning for agent cognition. A raw API wrapper that exposes 200 endpoints as 200 MCP tools does not help an agent reason about which tool to use. Quality MCP servers should expose semantically meaningful actions with rich descriptions, grouped logically, with appropriate granularity. The conformance testing framework from Day 1 was positioned as part of the solution, but cultural change in how developers think about agent-tool interfaces was identified as equally important.

2026 Roadmap

The official 2026 MCP roadmap targets three priorities:

  1. Authentication hardening: Moving OAuth 2.1 from recommendation to requirement, with XAA as the standard pattern for multi-service agent identity
  2. OpenTelemetry integration: Standard observability for MCP connections, enabling tracing of agent-tool interactions through existing monitoring infrastructure
  3. Horizontal HTTP transport scaling: Addressing the scale limitations of the current transport layer for enterprise deployments handling thousands of concurrent agent connections

Companion Protocols

Google introduced Universal Commerce Protocol (UCP) v1.0, adding an economic transaction layer for agents—enabling agents to negotiate prices, execute purchases, and manage billing. The A2A (Agent-to-Agent) protocol reached v1.0, formalizing agent-to-agent coordination as complementary to MCP's agent-to-resource focus. Together, MCP + A2A + UCP form an emerging trinity of agent infrastructure protocols: MCP for tools, A2A for agent coordination, UCP for commerce.

Unresolved Issues

Server discovery remains ad hoc. The IETF agents.txt draft, which proposed a .well-known/agents.txt file for capability declaration (analogous to robots.txt), was expiring on April 10, 2026 without a successor filing. Without standardized discovery, agents cannot autonomously find the MCP servers they need—a human must configure each connection. This is the most significant gap between the current state and the vision of fully autonomous agents.

7. Centralized MCP Security: OAuth, Governance, and Control Planes

As MCP moves from developer experimentation to enterprise production, the security architecture around it must mature from optional to mandatory. The MCP specification itself is deliberately neutral on security—it defines the protocol, not the policy. This design choice, while enabling rapid adoption, creates what security practitioners call the "governance gap": a well-defined protocol surrounded by an undefined security perimeter.

The Authentication Layer: OAuth 2.1 for MCP

MCP's authorization model follows OAuth 2.1 conventions. A properly secured MCP deployment requires:

  • Per-user OAuth flows: Each user authenticates through the organization's identity provider (Okta, Entra ID, Auth0). Agents act on behalf of authenticated users, never with shared service accounts.
  • Short-lived JWTs: Access tokens with limited lifetimes (minutes to hours), issued by the OAuth authorization server, carried in MCP request headers.
  • Mutual TLS (mTLS): For server-to-server MCP connections, certificate-based mutual authentication prevents unauthorized servers from joining the mesh.
  • Scope-based access control: OAuth scopes map to MCP tool capabilities. An agent with scope database:read can call query tools but not mutation tools.

The critical gap in current deployments is inconsistency: some MCP servers implement full OAuth correctly, others use shared bearer tokens or API keys across entire organizations, and some have no authentication at all. The MCP specification does not mandate any particular approach—governance resides in the implementation, not the protocol.

The Control Plane Architecture

The answer that scales for enterprise MCP is a control plane—a centralized gateway layer between MCP clients (agents) and MCP servers (tools). The gateway handles identity, access control, token management, filtering, and audit logging once, for every tool connection. This architectural pattern has been proven by API gateways (Kong, Apigee) and service meshes (Istio, Linkerd); MCP gateways apply the same pattern to agent-tool communication.

A reference architecture from InfoQ describes a three-component system:

  1. MCP Gateway Service: A TypeScript (or Go/Rust) service handling JSON-RPC message routing, schema validation (using Zod or equivalent), and policy enforcement. The gateway does not execute infrastructure changes directly—it validates, authorizes, and queues.
  2. OPA Policy Engine: Open Policy Agent evaluates fine-grained authorization policies written in Rego. Policies encode role-based access control, plan integrity validation (SHA-256 hash matching against registered artifacts), safety constraints (blocking destructive operations), and temporal windows (deployments only during approved change windows).
  3. Ephemeral Runners: Isolated execution environments (Kubernetes namespaces with forced teardown) that execute authorized tool calls. Each runner has a unique namespace, mandatory cleanup, and no persistent state—ensuring that failed executions never leave orphaned resources.

Policy as Code with OPA

OPA integration transforms MCP security from static configuration to dynamic, context-aware policy evaluation. A typical policy file (agent_authz.rego) enforces four categories of rules:

OPA's decision logs include correlation IDs that trace from the user's original question through the agent, the MCP tool call, the OPA decision, and the final command execution. This audit trail is essential for compliance in regulated industries.

Operational SLOs for MCP Security

The InfoQ reference architecture defines operational targets: policy decision latency under 100ms, runner start latency under 2 seconds (dev) or 5 seconds (staging), sandbox teardown under 30 seconds, audit log availability within 5 minutes, and a target denial rate of 2 percent or less (higher denial rates suggest misconfigured policies or overly restrictive rules).

"The safety of AI-driven automation improves less through smarter models and more through explicit, enforceable boundaries." — InfoQ, "Building a Least-Privilege AI Agent Gateway"

The principle of least privilege, applied to AI agents, means that no single component—agent, gateway, runner, or policy engine—has enough authority on its own to cause damage. The intent-to-execution separation is key: agents describe what they want to do, while runners control how it happens.

8. Orchestrating Intelligence: MCP with NimbleBrain Studio

As the number of MCP servers in enterprise environments grows from a handful to hundreds, the need for an orchestration layer between LLMs and tools becomes acute. NimbleBrain Studio represents one approach to this problem: a managed platform that provides visual workflow design, multi-MCP-server coordination, and enterprise governance for agent operations.

Architecture: Management UI + Kubernetes Runtime

NimbleBrain's architecture separates concerns into two layers: Studio (the management interface for designing, deploying, and monitoring MCP infrastructure) and Runtime (the Kubernetes-native execution environment where MCP servers and workflows actually run). The platform can be deployed as managed SaaS or as a bring-your-own-cloud (BYOC) installation within the customer's VPC.

The core abstraction model consists of three primitives:

  • Agents: AI assistants that can be chatted with, scheduled, or triggered by events. Each agent orchestrates one or more playbooks and has access to a configured set of MCP servers.
  • Playbooks: Reusable automation sequences composed of steps that invoke MCP tools. Playbooks can be called from agents or run independently, providing composability.
  • MCP Servers: The tool providers, deployed and managed through Studio with centralized configuration, scaling, and monitoring.

Enterprise Features

NimbleBrain addresses the operational requirements that distinguish enterprise MCP from developer experimentation:

  • Observability: Built-in tracing, metrics, and logs monitor cost, latency, and reliability across all MCP connections. OpenTelemetry integration provides compatibility with existing monitoring stacks (Datadog, Grafana, New Relic).
  • Security: Workspace isolation, role-based access control (RBAC), SSO/SAML integration, audit logs, and KMS-backed secrets management. Each workspace can have its own set of MCP servers and access policies.
  • Deployment tiers: Starter, Scale, and Enterprise tiers accommodate different organizational needs, from individual developers to multi-team deployments with thousands of agent invocations per day.

9. Building Stateful Transactional MCP Servers with Concierge AI

The MCP specification defines a fundamentally stateless protocol: each tool call is independent, with no built-in mechanism for session persistence, transaction management, or multi-step workflow orchestration. For simple tool calls (search a database, read a file), statelessness is sufficient. For complex operations (database migrations, infrastructure provisioning, multi-step financial transactions), it is not. Concierge AI, an open-source framework (Apache 2.0) from Bangalore-based developer Arnav, addresses this gap through opinionated, stateful MCP workflow patterns.

The Problem with Flat Tool Lists

Standard MCP presents all available tools to the LLM simultaneously. At enterprise scale, this creates compounding problems:

  • Token explosion: 100 tools with descriptions consume approximately 16 percent of a 200K context window before any actual work begins.
  • Non-determinism: The LLM selects from all available tools based on description matching, with no enforcement of operational ordering. An agent could attempt a database migration before creating a backup.
  • Semantic dilution: Irrelevant tools in the context reduce the model's ability to focus on the tools that matter for the current step.

Stage-Based Workflows

Concierge replaces flat tool lists with "staged workflows"—a directed graph of stages, each exposing only the tools relevant to that phase of the operation. The LLM navigates through stages, unlocking new tools as it progresses:

@stage(name="preparation", transitions=["backup"])
@stage(name="backup", transitions=["migration", "preparation"])
@stage(name="migration", transitions=["verification", "rollback"])
@stage(name="verification", transitions=["complete"])
@stage(name="rollback", transitions=["preparation"])

This enforces operational ordering: you cannot reach the migration stage without passing through preparation and backup. The server rejects invalid transitions, preventing the LLM from making non-deterministic choices that violate operational safety. Token usage drops by 80-90 percent because only the current stage's tools (typically 2-5) are exposed instead of the entire catalog.

Transaction Management and Rollback

Concierge implements atomic operations through transaction decorators. All state modifications within a transaction are snapshotted. If the workflow terminates abnormally—whether from an LLM error, a network failure, or a tool exception—changes automatically revert to the pre-transaction snapshot:

@server.tool()
@transaction()
async def apply_migration(self, migration_file: str) -> Dict:
    server.global_state["migration_applied"] = True
    return {"status": "success"}

@server.tool()
async def rollback_migration(self) -> Dict:
    server.rollback_transaction()  # Reverts to snapshot
    return {"status": "rolled_back"}

The framework distinguishes between transient errors (network timeouts, rate limits) and permanent errors (schema violations, authorization failures). Transient errors trigger retry loops within the current stage with exponential backoff. Permanent errors trigger forced transitions to recovery stages, ensuring the workflow degrades gracefully rather than failing silently.

Two-Tier State Model

Concierge uses a two-tier state model:

  • Global state: Persisted across all stages. Stores credentials, workflow metadata, and context that every stage needs. Accessible via self.global_state.
  • Stage-local state: Scoped to the current stage only. Contains temporary calculations, retry counters, and intermediate results. Automatically cleaned up on stage transitions.

For production persistence, Concierge uses PostgreSQL:

CREATE TABLE mcp_sessions (
    session_id UUID PRIMARY KEY,
    current_stage VARCHAR(255),
    global_state JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP DEFAULT NOW() + INTERVAL '24 hours'
);

CREATE TABLE transaction_snapshots (
    id SERIAL PRIMARY KEY,
    session_id UUID,
    snapshot_data JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

Production Deployment Patterns

Production deployments require session affinity (sticky sessions) because Concierge maintains in-memory state alongside PostgreSQL persistence. Load balancers must route requests with the same session ID to the same pod. The framework supports multiple transports (stdio, StreamableHTTP, SSE), enabling deployment on serverless platforms, containers, or bare metal.

The built-in evaluation framework validates workflow behavior through simulated conversations, measuring task completion rate, token efficiency, safety score (adherence to enforced transitions), latency per stage, and determinism variance across repeated runs.

Real-World Use Cases

  • Database migrations: Enforce preparation, backup, migration, verification sequence with transaction-based rollback on failure.
  • Infrastructure provisioning: Prevent compute resource launch before network creation; ensure security hardening before workflow completion.
  • Secret rotation: Enable dual authentication (old and new credentials both valid), validate new credentials, then retire old credentials sequentially—with automatic rollback if validation fails.

Concierge's design philosophy—3 to 7 stages with 2 to 5 tools each—reflects a practical understanding of LLM cognitive limits. A model presented with 50 tools in a flat list will make poor selections. A model presented with 3 tools in a well-described stage, with clear transitions to the next stage, produces deterministic, auditable behavior. The framework represents evolution from "flat tool lists to production-grade agent workflows," and its open-source Apache 2.0 license enables adoption without vendor lock-in.
