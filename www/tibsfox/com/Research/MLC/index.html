Machine Language & Machine Code

Machine code is the actual binary patterns a CPU decodes and executes. Not text, not mnemonics — raw bits. Every “instruction” is a specific bit pattern that the decoder recognizes. Assembly language is a human-readable representation of machine code; machine code is what the CPU actually runs.

Why machine code? The ASM project covered assembly language — the human-readable layer. This project goes one level deeper: into the actual binary patterns the CPU decodes. Understanding machine code means understanding instruction encoding (how bits map to operations), binary formats (ELF, PE, Mach-O), the fetch-decode-execute cycle, microcode, and the security implications of raw code execution. This is the layer where compilers produce output, JIT compilers emit bytes, exploits craft payloads, and the CPU’s decoder turns bits into electrical signals.

The Escalation Ladder — Machine Code’s Place

One step below assembly, one step above the transistors:

  • High-level languages (Python, Java, Go, Rust, C++, C) — compiled or JIT-compiled
  • Assembly language — human-readable mnemonics (ASM project)
  • Machine code — you are here. Raw binary opcodes. 48 89 E5 = mov rbp, rsp.
  • Microcode (CISC) — micro-operations inside the CPU that implement machine instructions
  • Logic gates — AND, OR, NOT, XOR implemented in transistors
  • Transistors — the physics layer

A Concrete Example

The x86-64 instruction mov rbp, rsp is encoded as the bytes 48 89 E5:

  • 48 = REX prefix with W bit set (64-bit operand)
  • 89 = opcode for mov r/m64, r64
  • E5 = ModR/M byte: mod=11 (register), reg=100 (rsp), r/m=101 (rbp)

The CPU’s decoder reads these 3 bytes, determines the operation (move rsp into rbp), and sends the appropriate control signals to the register file. This entire process takes one clock cycle in the decode stage of the pipeline.
