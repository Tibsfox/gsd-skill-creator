Instruction Encoding

How Bits Become Instructions — x86-64 Variable-Length, ARM64 Fixed, RISC-V Typed Formats
Overview Encoding Execution Formats Microcode Security Rosetta Learn History
An instruction is a bit pattern the CPU decoder recognizes. The encoding defines how those bits are divided into fields: opcode (what to do), operands (what to do it to), and modifiers (how to do it).

x86-64 Encoding — The Most Complex

An x86-64 instruction can be 1 to 15 bytes, composed of up to 6 optional fields:

  1. Legacy prefixes (0–4 bytes) — LOCK F0, REP F3, operand-size 66, address-size 67, segment overrides
  2. REX prefix (0–1 byte) — 40–4F: W (64-bit), R/X/B (register extension bits)
  3. Opcode (1–3 bytes) — primary map, or 0F escape, or 0F38/0F3A three-byte
  4. ModR/M (0–1 byte) — mod (2 bits: addressing mode), reg (3 bits: register or opcode ext), r/m (3 bits: register or memory)
  5. SIB (0–1 byte) — scale (2 bits: 1/2/4/8), index (3 bits), base (3 bits). Used for complex addressing.
  6. Displacement + Immediate (0–8 bytes each) — memory offset and constant value

ARM64 Encoding — Clean and Fixed

Every ARM64 instruction is exactly 32 bits (4 bytes). The top bits (28–25) identify the instruction class. This makes decoding trivial compared to x86.

RISC-V Encoding — Modular and Regular

Six instruction formats, all 32 bits. The opcode is always bits 6–0:

FormatFieldsUse
R-typefunct7 | rs2 | rs1 | funct3 | rd | opcodeRegister-register: add x1, x2, x3
I-typeimm[11:0] | rs1 | funct3 | rd | opcodeImmediate: addi x1, x2, 10
S-typeimm[11:5] | rs2 | rs1 | funct3 | imm[4:0] | opcodeStore: sw x1, 0(x2)
B-typeimm bits split | rs2 | rs1 | funct3 | imm bits | opcodeBranch: beq x1, x2, offset
U-typeimm[31:12] | rd | opcodeUpper immediate: lui x1, 0x12345
J-typeimm bits shuffled | rd | opcodeJump: jal x1, offset

Variable-Length vs Fixed-Length

x86: variable-length (1–15 bytes). Pro: denser code, backward compatibility. Con: complex decoder, ambiguous instruction boundaries, harder to pipeline.

ARM64/RISC-V: fixed-length (32 bits). Pro: trivial decode, easy pipelining, predictable fetch. Con: lower code density (though RISC-V C extension adds 16-bit compressed instructions).

Modern x86 CPUs spend enormous silicon area on the decoder. ARM and RISC-V CPUs spend almost none.
