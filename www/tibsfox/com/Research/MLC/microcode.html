Microcode & Micro-Operations

The Layer Below Machine Code in CISC CPUs — uops, Microcode ROM, Firmware Updates
Overview Encoding Execution Formats Microcode Security Rosetta Learn History
In CISC processors like x86, complex machine instructions are decoded into simpler micro-operations (uops) that the execution engine actually runs. Microcode is the firmware that controls this translation. It is the layer between machine code and the hardware.

What Microcode Is

A simple x86 instruction like add eax, ebx maps to a single uop. A complex instruction like rep movsb (string copy) or cpuid maps to many uops, sequenced by a microcode ROM inside the CPU. The microcode ROM is a small read-only memory containing micro-programs for complex instructions.

The uop Cache (DSB)

Modern Intel CPUs have a Decoded Stream Buffer (DSB) — a cache of already-decoded uops. On a DSB hit, the CPU skips the expensive decode stage entirely. This is why hot loops run faster after the first iteration: the decoded uops are cached.

Microcode Updates

Intel and AMD can update microcode via firmware patches loaded at boot time (by the BIOS/UEFI or the OS). This is how Spectre/Meltdown mitigations were deployed: microcode patches changed the behavior of speculative execution without replacing the CPU. Linux loads microcode updates via /lib/firmware/intel-ucode/ or amd-ucode/.

RISC vs CISC at the Microcode Level

ArchitectureMicrocode?Why
x86/x86-64Yes (extensive)Complex variable-length instructions need a micro-program to sequence
ARM64Minimal or noneFixed-width RISC instructions map directly to hardware operations
RISC-VNoneSimple instructions by design; hardware implements them directly

The irony: modern x86 CPUs are externally CISC but internally RISC. The decoder translates complex x86 instructions into simple uops that execute on a RISC-like out-of-order core. The microcode is what makes this translation possible for the really complex instructions.
