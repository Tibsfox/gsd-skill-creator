ELF, PE, Mach-O

Executable Binary Formats — Linking, Relocations, Dynamic Loading, Debug Info
Overview Encoding Execution Formats Microcode Security Rosetta Learn History
Machine code doesn’t float free — it lives inside binary file formats. ELF on Linux, PE on Windows, Mach-O on macOS. The format tells the OS how to load the code into memory, where to find the entry point, and how to resolve external symbols.

ELF Structure

ELF header → program headers (segments for loading) → section headers (sections for linking). Key sections:

  • .text — executable machine code
  • .data — initialized global/static data
  • .bss — uninitialized data (zero-filled at load time, no bytes in the file)
  • .rodata — read-only constants (string literals, const arrays)
  • .symtab / .strtab — symbol table and string table
  • .plt / .got — Procedure Linkage Table / Global Offset Table (dynamic linking)
  • .rel / .rela — relocation entries
  • .debug_* — DWARF debug information
# Inspect an ELF binary
readelf -h myapp              # ELF header
readelf -l myapp              # Program headers (segments)
readelf -S myapp              # Section headers
readelf -s myapp              # Symbol table
objdump -d myapp              # Disassemble .text section
objdump -d -M intel myapp     # Intel syntax

Linking

Static linking: all code in one binary. Larger, self-contained. Dynamic linking: shared libraries (.so/.dll/.dylib) loaded at runtime. Smaller binaries, shared memory, but dependency management.

PLT/GOT (Lazy Binding)

The first call to a shared library function goes through the PLT, which calls ld.so to resolve the symbol. ld.so patches the GOT entry so subsequent calls go directly to the function. ASLR randomizes base addresses.

Loading an Executable (Linux)

  1. execve() → kernel reads ELF header
  2. Map .text, .data, .bss segments into memory
  3. Set up stack: argc, argv, envp, auxiliary vector
  4. If dynamically linked: map ld.so, transfer control
  5. ld.so loads shared libraries, resolves relocations
  6. Jump to _start → __libc_start_main → main()

Debug Information (DWARF)

DWARF maps machine code addresses back to source code: line numbers, variable locations, type info. GDB uses DWARF to show source while stepping through machine code. PDB is Windows’ equivalent.
