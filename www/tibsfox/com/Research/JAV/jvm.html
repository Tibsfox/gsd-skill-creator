The JVM & Concurrency

HotSpot, Garbage Collection, JIT, the Java Memory Model, and Project Loom’s Virtual Threads

Java the language has always been conservative. Java the runtime has always been ambitious. The gap between those two halves is where most of the interesting engineering lives.

What the JVM Actually Is

The JVM is an abstract computing machine whose instruction set is not the native instruction set of any real CPU. It is defined by The Java Virtual Machine Specification, maintained separately from the Java Language Specification. Both are published by Oracle and can evolve independently.

The JVM specification describes:

  • A binary file format (the .class file) starting with the magic number CAFEBABE
  • An instruction set of roughly 200 opcodes for a stack machine
  • A set of runtime data areas: heap, method area, stacks, PC register, native stacks
  • A verification algorithm that proves a .class file is structurally safe before any code runs
  • Rules for class loading, linking, initialization, and thread semantics

It explicitly does not describe a garbage collector, a JIT compiler, thread scheduling behavior, or any mapping to real hardware. Those are implementation concerns — which is why OpenJDK HotSpot, Eclipse OpenJ9, GraalVM, and Azul Zing can all legitimately call themselves JVMs while making wildly different engineering choices under the hood.

Class Loading

Classes are loaded on first use, not all at once. The delegation model: children ask parents first.

  1. Bootstrap class loader (native code) — loads java.base and core JDK classes
  2. Platform class loader (was “Extension” before Java 9) — loads JDK extensions
  3. System / Application class loader — loads classes from the classpath / module path
  4. Custom class loaders for plugin systems, web containers, OSGi, etc.

Every .class file passes through the bytecode verifier before any code in it runs. The verifier rejects malformed bytecode, type violations, and stack overflow / underflow. This verification is what makes Java “safe” to run untrusted code (historically, applets).

JVM Memory Layout

Heap

Where all Java objects live. Divided into generations: Young (Eden + two survivor spaces) and Old. Controlled by -Xms (initial) and -Xmx (max).

Metaspace

Class metadata: method bytecode, constant pool, class structures. Replaced the fixed-size PermGen in Java 8. Grows as classes load.

Thread Stacks

One per platform thread. Local variables, operand stacks, frame pointers. Fixed size per thread (-Xss).

Direct Memory

Outside the heap. Allocated via ByteBuffer.allocateDirect() for NIO and off-heap caching. Not managed by the GC.

Garbage Collection

HotSpot offers multiple garbage collectors, each tuned for different trade-offs between throughput, pause time, and memory overhead.

The generational hypothesis drives GC design: most objects die young. Young-gen collections are fast and frequent; old-gen collections are slower but rare. A minor GC collects the young generation; a major (or full) GC collects the old generation too. Modern collectors like G1 and ZGC blur this distinction.

JIT Compilation

HotSpot uses tiered compilation. Code starts interpreted, then compiles at progressively higher optimization levels as it runs more often.

  1. Interpreter (tier 0) — directly executes bytecode. Slow but starts immediately.
  2. C1 (client compiler, tiers 1–3) — fast compilation, simple optimizations. Good for startup.
  3. C2 (server compiler, tier 4) — slow compilation, aggressive optimization. The “hot path” compiler.

C2 applies: inlining, escape analysis (scalar replacement, stack allocation), loop unrolling, lock coarsening and elision, dead code elimination, branch prediction hints, and speculative optimizations with guarded deoptimization. When speculations fail, HotSpot deoptimizes: falls back to the interpreter and re-compiles.

GraalVM replaces C2 with a new JIT compiler written in Java. It produces comparable or better code in many cases and, via Native Image, can compile a Java program to a native binary with near-instant startup — at the cost of a closed-world assumption (no arbitrary reflection at runtime).

Bytecode

Java bytecode is a stack-based instruction set with roughly 200 opcodes. Use javap -c to disassemble a class:

public int add(int a, int b) { return a + b; }

// Disassembled:
public int add(int, int);
  Code:
     0: iload_1      // push a
     1: iload_2      // push b
     2: iadd         // pop two, push sum
     3: ireturn      // return top of stack

Key opcode categories:

  • Load / store — iload, istore, aload, astore
  • Arithmetic — iadd, imul, dadd, …
  • Control flow — ifeq, goto, tableswitch
  • Invocation — invokevirtual, invokestatic, invokespecial, invokeinterface, invokedynamic
  • Field access — getfield, putfield, getstatic, putstatic
  • Object — new, newarray, checkcast, instanceof

invokedynamic (Java 7, JSR 292) was added specifically to make dynamic languages fast on the JVM. It’s now also used by lambdas, string concatenation (Java 9+), and pattern matching — each lambda gets its own lazy-bound call site that the JIT can specialize.

The Java Memory Model (JMM)

JSR 133, shipped in Java 5, defined what concurrent access actually means in Java. It established:

  • volatile — guarantees visibility and ordering across threads (acquire/release semantics)
  • synchronized — exclusive access plus memory barriers on entry and exit
  • Happens-before relationship — the foundation for reasoning about concurrent code
  • Final field semantics — immutability guarantees after construction, even without synchronization
  • Safe publication — how to make a newly-constructed object visible to other threads

The JMM is what made Java 5 the first version where lock-free algorithms could be written portably. Before JSR 133, volatile was under-specified, and double-checked locking was broken on most JVMs.

Virtual Threads — Project Loom (Java 21)

Ron Pressler led Project Loom with a clear goal: make the simple “thread per request” programming model viable at cloud scale. Platform threads are expensive (each has a ~1 MB stack); virtual threads are cheap (~1 KB).

Virtual threads are scheduled by the JVM onto a small pool of carrier threads (platform threads backed by the OS). When a virtual thread blocks on I/O or a synchronizer, it unmounts from its carrier, freeing it for another virtual thread. When the block resolves, the virtual thread is scheduled back onto any carrier. Under the hood, the JVM uses continuations: a mechanism for suspending and resuming a thread’s call stack.

// Classic platform thread
Thread t = new Thread(() -> doWork());
t.start();
t.join();

// Virtual thread (Java 21)
Thread vt = Thread.ofVirtual().start(() -> doWork());
vt.join();

// A million virtual threads, no problem
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i ->
        executor.submit(() -> handleRequest(i))
    );
}

Pinning (and its resolution)

In Java 21, a virtual thread could be pinned to its carrier inside synchronized blocks — it could not unmount. The JEP 491 work (targeting Java 24) removes this restriction, so synchronized no longer pins.

Structured concurrency (incubator)

// StructuredTaskScope — JEP 428 incubator
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User>     user    = scope.fork(() -> fetchUser(id));

    scope.join().throwIfFailed();

}

Project Panama — Foreign Function & Memory API (Java 22)

Panama replaces JNI with a safer, faster native interop API. No more hand-written C stubs.

  • MemorySegment — safe access to native memory
  • MemoryLayout — describes the shape of native data
  • Linker and SymbolLookup — bind to native functions by name
  • jextract — generates bindings from C headers

Performance — Warmup, Profiling, Startup

“Java is slow” was true in 1996 and has been wrong for a long time. Modern HotSpot is routinely within 10–20% of hand-tuned C++ on long-running workloads, and sometimes faster after JIT specialization. The real performance issues are:

  • Startup time — JVM warmup can take seconds for complex apps. Addressed by CDS (Class Data Sharing), AppCDS, GraalVM Native Image, CRaC (checkpoint/restore), and Project Leyden (static/ahead-of-time images).
  • Memory overhead — JVMs are hungry compared to native binaries. ZGC and Shenandoah trade throughput for low pause time but still need generous heaps.
  • GC pauses — historically the #1 tuning problem. Modern G1 and ZGC keep pauses under 10 ms at hundreds of gigabytes.
