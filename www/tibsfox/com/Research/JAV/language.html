Design Philosophy

When Gosling and McGilton published The Java Language Environment in May 1996, they framed Java with a now-famous list of adjectives. That white paper reads like a point-by-point reaction to the pain of writing large C++ systems in the late 1980s and early 1990s.

Simple meant “fewer lurking surprises,” not “fewer features.” Familiar meant C-like syntax: curly braces, semicolons, if/while/for, the same operator precedence table. A C programmer should be able to read a Java source file on day one and guess correctly what most of it does. Object-oriented meant classes, single inheritance, interfaces, and dynamic dispatch as the default — not the bolt-on classes of early C++.

The Primitive / Reference Divide

Java’s type system has two disjoint worlds: primitives and reference types. Primitives sit directly on the stack or inside objects. Reference types are pointers (internally) to heap-allocated objects.

PrimitiveSizeRange
byte8 bits−128 to 127
short16 bits−32,768 to 32,767
int32 bits±2.1 × 109
long64 bits±9.2 × 1018
float32 bitsIEEE 754 single
double64 bitsIEEE 754 double
char16 bitsUTF-16 code unit, 0 to 65,535
boolean1 bittrue or false

Every primitive has a guaranteed size and range across all JVMs. This was a sharp break from C, where int might be 16 or 32 bits depending on the platform. Each primitive has a corresponding wrapper class (Integer, Long, Double, …). Since Java 5, the compiler inserts boxing and unboxing automatically.

The billion-dollar mistake. Tony Hoare called null references his “billion-dollar mistake” — the unchecked null reference, which he invented for ALGOL W in 1965. Java inherited it whole. Any reference variable can be null, and dereferencing null throws NullPointerException. Modern Java has tools (Optional, nullability annotations, pattern matching, Project Valhalla’s eventual value classes) that mitigate the problem without removing null.

var — Local Variable Type Inference (Java 10)

JEP 286 added var for local variable declarations:

var list = new ArrayList<String>();   // inferred as ArrayList<String>
var length = list.size();              // inferred as int

var is not a dynamic type. The compiler infers the exact static type from the initializer and bakes it into the bytecode. var is not allowed for fields, method parameters, or return types — Brian Goetz and the JEP explicitly scoped it to locals to keep API signatures explicit.

Classes and Objects

package com.example.geometry;

public final class Point {
    private final double x;
    private final double y;

    public Point(double x, double y) {
        this.x = x;
        this.y = y;
    }

    public double x() { return x; }
    public double y() { return y; }

    public double distanceTo(Point other) {
        double dx = x - other.x;
        double dy = y - other.y;
        return Math.sqrt(dx * dx + dy * dy);
    }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point p)) return false;
        return Double.compare(x, p.x) == 0
            && Double.compare(y, p.y) == 0;
    }

    @Override public int hashCode() { return Objects.hash(x, y); }
    @Override public String toString() { return "Point[x=" + x + ", y=" + y + "]"; }
}

This example covers: package declaration, public final class, private final fields, constructor, accessor methods, equals/hashCode/toString contract, and pattern matching instanceof (Java 16+).

Generics (Java 5, JSR 14)

Generics were added in Java 5 to catch collection type errors at compile time. The design was led by Gilad Bracha and Martin Odersky (who would go on to create Scala). The key compromise: type erasure. Generic type information exists only at compile time. At runtime, List<String> and List<Integer> are both just List. This preserved binary compatibility with pre-generics code but added the surprises that Java programmers learn to work around.

public static <T extends Comparable<T>> T max(List<T> list) {
    T result = list.get(0);
    for (T item : list) {
        if (item.compareTo(result) > 0) result = item;
    }
    return result;
}

// Wildcards and the PECS rule: Producer Extends, Consumer Super
public static void copy(List<? extends Number> src, List<? super Number> dst) {
    for (Number n : src) dst.add(n);
}

Java 7 added the diamond operator <> for inferring generic type arguments: Map<String, List<Integer>> map = new HashMap<>();.

Collections Framework (Java 1.2)

The Collections Framework was added to Java 1.2 (“Java 2,” December 1998) and designed primarily by Joshua Bloch. It replaced the ad-hoc Vector and Hashtable of Java 1.0/1.1 with a systematic hierarchy:

InterfaceImplementations
ListArrayList, LinkedList, Vector (legacy), Stack (legacy)
SetHashSet, TreeSet, LinkedHashSet
Queue / DequeArrayDeque, PriorityQueue, LinkedList
MapHashMap, TreeMap, LinkedHashMap, Hashtable (legacy)

The Iterator and Iterable interfaces enable the enhanced for loop (Java 5):

for (String name : names) {
    System.out.println(name);
}

Exceptions

Java famously distinguishes checked from unchecked exceptions:

  • Checked exceptions (subclasses of Exception but not RuntimeException) must be declared in the method’s throws clause or caught.
  • Unchecked exceptions (RuntimeException and subclasses) propagate freely without declaration.
  • Errors (Error and subclasses) represent serious problems like OutOfMemoryError, conventionally not caught.

The checked exceptions experiment is widely considered Java’s most controversial design choice — the only mainstream language to enforce them. Modern wisdom (Bloch, Goetz) is to use unchecked exceptions for most cases.

// Try-with-resources (Java 7, JEP 238)
try (BufferedReader br = Files.newBufferedReader(path)) {
    String line;
    while ((line = br.readLine()) != null) {
        process(line);
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

// Multi-catch (Java 7)
try {
    // ...
} catch (IOException | SQLException e) {
    log.error("data access failed", e);
}

Lambdas and Functional Interfaces (Java 8, JSR 335)

Java 8 (March 2014) was the single biggest language release since Java 5. Lambdas had been in design for eight years. A lambda is an anonymous function that can be passed around as a value:

// Lambda expression
Comparator<String> byLength = (a, b) -> a.length() - b.length();

// Method reference
Comparator<String> byLength2 = Comparator.comparingInt(String::length);

// Used with collections
List<String> names = new ArrayList<>(List.of("Charlie", "Al", "Bob"));
names.sort(Comparator.comparing(String::length));

// Functional interfaces from java.util.function
Function<String, Integer> parseInt = Integer::parseInt;
Predicate<String> nonEmpty = s -> !s.isEmpty();
Consumer<String> printer = System.out::println;
Supplier<List<String>> factory = ArrayList::new;

A functional interface is any interface with exactly one abstract method (a SAM: Single Abstract Method). The @FunctionalInterface annotation asserts the intent.

Streams (Java 8)

Streams decouple iteration from transformation. A pipeline describes what to compute, not how to loop:

import java.util.stream.*;

// Count the total length of strings starting with "J"
int total = names.stream()
    .filter(s -> s.startsWith("J"))
    .mapToInt(String::length)
    .sum();

// Group people by department
Map<String, List<Person>> byDept = people.stream()
    .collect(Collectors.groupingBy(Person::department));

// Parallel computation (uses ForkJoinPool.commonPool)
double average = measurements.parallelStream()
    .mapToDouble(Measurement::value)
    .average()
    .orElse(0.0);

Stream operations are lazy (intermediate ops don’t run until a terminal op is invoked) and single-use (you can’t iterate a stream twice).

Optional (Java 8)

Optional<T> is a container that may or may not hold a value. Brian Goetz’s guidance is strict: use it for return types of methods where the absence of a result is meaningful, not for fields or parameters.

public Optional<User> findByEmail(String email) {
    return users.stream()
        .filter(u -> u.email().equals(email))
        .findFirst();
}

// Chain it
String displayName = findByEmail("foxy@example.com")
    .map(User::displayName)
    .orElse("Unknown user");

Records (Java 14 preview, Java 16 standard, JEP 395)

Records are a compact syntax for immutable data carriers. One line replaces constructor, accessors, equals, hashCode, and toString:

public record Point(double x, double y) {
    // Compact constructor for validation
    public Point {
        if (Double.isNaN(x) || Double.isNaN(y))
            throw new IllegalArgumentException("NaN");
    }
}

var p = new Point(3.0, 4.0);
System.out.println(p.x());     // accessor
System.out.println(p);         // toString: "Point[x=3.0, y=4.0]"

Records are implicitly final, their fields are final, and they automatically get pattern matching support (Java 21+).

Sealed Classes (Java 17, JEP 409)

Sealed classes let you declare a closed hierarchy. Only the named types may extend the sealed parent, enabling exhaustive pattern matching:

public sealed interface Shape
    permits Circle, Square, Triangle {}

public record Circle(double r) implements Shape {}
public record Square(double side) implements Shape {}
public record Triangle(double base, double height) implements Shape {}

// Exhaustive switch — no default needed
double area(Shape s) {
    return switch (s) {
        case Circle c -> Math.PI * c.r() * c.r();
        case Square sq -> sq.side() * sq.side();
        case Triangle t -> 0.5 * t.base() * t.height();
    };
}

Pattern Matching (Java 16–21)

Pattern matching has rolled out gradually across several Java releases:

  • Java 14 — switch expressions with arrow syntax (preview in 12, standard in 14)
  • Java 16 — pattern instanceof: if (obj instanceof String s) { ... }
  • Java 21 — pattern matching for switch with type patterns, guarded patterns, and record patterns
// Java 21 record deconstruction + guarded patterns
String describe(Shape s) {
    return switch (s) {
        case Circle(double r) when r > 100 -> "huge circle";
        case Circle(double r) -> "circle of radius " + r;
        case Square(double side) -> "square of side " + side;
        case Triangle(double b, double h) -> "triangle " + b + "x" + h;
    };
}

Text Blocks (Java 15, JEP 378)

String json = """
    {
        "name": "Foxy",
        "role": "sysadmin",
        "languages": ["bash", "perl", "c", "cpp", "java"]
    }
    """;

Text blocks eliminate the escaping dance for multi-line string literals. Leading whitespace is normalized by the compiler based on the position of the closing delimiter.

Enums (Java 5)

public enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    VENUS  (4.869e+24, 6.0518e6),
    EARTH  (5.976e+24, 6.37814e6);

    private final double mass;
    private final double radius;

    Planet(double mass, double radius) {
        this.mass = mass;
        this.radius = radius;
    }

    public double surfaceGravity() {
        return 6.67300E-11 * mass / (radius * radius);
    }
}

Java enums are full classes: they can have fields, methods, and even override methods per-constant. This is a big upgrade over the C/C++ enum integer idiom.

Annotations

Annotations are metadata that can be attached to declarations (classes, methods, fields, parameters). Common built-ins:

  • @Override — assert this method overrides a superclass method
  • @Deprecated — mark as deprecated
  • @SuppressWarnings("unchecked") — quiet specific compiler warnings
  • @FunctionalInterface — assert SAM interface
  • @SafeVarargs — assert no heap pollution with varargs + generics

Frameworks like Spring, JPA, and JUnit use annotations extensively (@Autowired, @Entity, @Test) to replace XML config.

Modules (Java 9, JEP 261)

Project Jigsaw delivered a module system: a module-info.java declares what a module exports, what it requires, and what services it provides. The JDK itself is modular (java.base, java.sql, etc.). Modules were controversial — many argued they weren’t worth the cost — but they enabled jlink to produce small runtime images and clarified the JDK’s internal boundaries.
