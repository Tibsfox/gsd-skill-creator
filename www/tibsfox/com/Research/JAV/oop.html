Object-Oriented Java

Classes, Interfaces, Inheritance, Polymorphism, Records, Sealed Hierarchies — The Four Pillars, Java-Style
Overview Language OOP JVM Frameworks Ecosystem Rosetta Learn History
Java is the language that made object-oriented programming mandatory. For 25 years, every piece of Java code had to live inside a class. Java 21 finally relaxed that with unnamed classes — but the OO model remains the defining abstraction.

The Four Pillars, Java-Style

Encapsulation

private fields + public getters/setters — the JavaBeans convention. Access modifiers enforced at the JVM level, not just compile time. public, protected, package-private (default), private.

Inheritance

Single inheritance only via extends. java.lang.Object is the ancestor of every class. Multiple interface inheritance via implements. Sealed classes (Java 17) for closed hierarchies.

Polymorphism

Every non-final method is virtual by default — the opposite of C++. Dynamic dispatch via the JVM’s invokevirtual/invokeinterface. Pattern matching for switch (Java 21) adds structural polymorphism.

Abstraction

Abstract classes + interfaces + generics. Interfaces gained default methods in Java 8 to enable library evolution. Records (Java 16) for data-oriented abstractions.

Class Basics

A class declaration can include fields, methods, constructors, nested classes, and static and instance initializer blocks. The compiler produces one .class file per class.

package com.example.banking;

public class Account {
    private final String owner;
    private double balance;

    public Account(String owner, double initial) {
        this.owner = owner;
        this.balance = initial;
    }

    public String owner() { return owner; }
    public double balance() { return balance; }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("must be positive");
        balance += amount;
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalStateException("insufficient funds");
        balance -= amount;
    }
}

The Object Class

java.lang.Object is the root of every Java class hierarchy. Every class you write extends Object (directly or transitively). Object contributes these methods that every class gets for free (and usually needs to override):

MethodPurpose
equals(Object)Logical equality. Must be reflexive, symmetric, transitive, consistent, and x.equals(null) must be false.
hashCode()Consistent with equals: equal objects must have equal hash codes.
toString()Human-readable representation. The debugger, logger, and String concatenation all use it.
getClass()Runtime Class<?> token, used by reflection.
clone()Shallow copy; part of the flawed Cloneable design.
wait(), notify(), notifyAll()Legacy monitor-based concurrency primitives.
finalize() — deprecatedCalled by the GC before reclaiming the object. Replaced by Cleaner.

The equals/hashCode contract is the single most-violated Java rule. Joshua Bloch devotes two full chapters of Effective Java to getting it right. Records (Java 16) auto-generate both correctly; use them whenever you can.

Inheritance

Java supports only single inheritance of implementation. A class extends at most one superclass:

public abstract class Animal {
    protected final String name;
    public Animal(String name) { this.name = name; }
    public abstract String speak();
    public String name() { return name; }
}

public class Dog extends Animal {
    private final String breed;
    public Dog(String name, String breed) {
        super(name);                 // call superclass constructor
        this.breed = breed;
    }
    @Override
    public String speak() { return name + ": woof!"; }
}

Rules:

  • Subclass constructors must call a superclass constructor explicitly via super(…), or the compiler inserts an implicit super() call to the no-arg constructor.
  • Fields are initialized in this order: static fields and static blocks first (per class, in declaration order); then instance fields and instance blocks during construction.
  • final classes cannot be subclassed. final methods cannot be overridden.
  • @Override annotation is optional but strongly recommended — it catches typos that would otherwise silently create a new method instead of an override.

Interfaces

Interfaces describe behavior without implementation. A class implements as many interfaces as it wants. Before Java 8, interfaces could only declare method signatures. Java 8 added default methods:

public interface Greeter {
    String name();

    // Default method — Java 8
    default String greet() {
        return "Hello, " + name() + "!";
    }

    // Static method — Java 8
    static Greeter anonymous() {
        return () -> "stranger";
    }
}

public class Person implements Greeter {
    private final String name;
    public Person(String name) { this.name = name; }
    public String name() { return name; }
}

Why default methods were added: to evolve interfaces in the standard library (especially Collection, which gained stream(), forEach(), etc.) without breaking every existing implementation.

Functional interfaces (Java 8)

An interface with exactly one abstract method (a SAM, Single Abstract Method) is a functional interface. Lambda expressions and method references can be used wherever a functional interface is expected:

@FunctionalInterface
public interface Transformer<T, R> {
    R apply(T input);
}

Transformer<String, Integer> length = s -> s.length();
Transformer<String, Integer> length2 = String::length;

Modern guidance: Prefer interfaces for most abstractions (single inheritance of implementation is a sharp constraint). Use abstract classes only when you genuinely need shared state or shared protected helpers.

The Slicing Problem — Java Doesn’t Have It

Unlike C++, Java references are always pointer-like. A variable of type Shape does not hold a Shape value — it holds a reference to some object, which might be a Circle, a Square, or any subtype. There is no slicing; polymorphism just works:

Shape s = new Circle(2.0);    // s holds a reference; the object is still a Circle
System.out.println(s.area()); // dispatches to Circle.area()

Records (Java 16, JEP 395)

Records are data-oriented classes with auto-generated boilerplate. One line gives you a constructor, accessors, equals, hashCode, and toString:

public record Point(double x, double y) {
    // Compact constructor for validation / normalization
    public Point {
        if (Double.isNaN(x) || Double.isNaN(y))
            throw new IllegalArgumentException("NaN");
    }

    // Additional methods allowed
    public double distanceToOrigin() {
        return Math.sqrt(x * x + y * y);
    }
}

var p = new Point(3, 4);
System.out.println(p);                 // Point[x=3.0, y=4.0]
System.out.println(p.distanceToOrigin()); // 5.0

Records are implicitly final, their fields are final, and they cannot extend another class (but can implement interfaces). Use cases: DTOs, value objects, tuples, event records, immutable state.

Sealed Classes (Java 17, JEP 409)

A sealed type declares the exact list of permitted subtypes. This enables exhaustive pattern matching:

public sealed interface Shape
    permits Circle, Square, Triangle {}

public record Circle(double r) implements Shape {}
public record Square(double side) implements Shape {}
public record Triangle(double base, double height) implements Shape {}

// Exhaustive — compiler rejects adding a Shape without updating this
public static double area(Shape s) {
    return switch (s) {
        case Circle c -> Math.PI * c.r() * c.r();
        case Square sq -> sq.side() * sq.side();
        case Triangle t -> 0.5 * t.base() * t.height();
    };
}

Subtypes of a sealed class must be declared final, sealed, or non-sealed. non-sealed re-opens the hierarchy for arbitrary extension (rare, for framework extension points).

Pattern Matching for switch (Java 21)

Type patterns, record deconstruction, and guarded patterns combine to replace visitor pattern boilerplate with a clean switch:

String describe(Shape s) {
    return switch (s) {
        case Circle(double r) when r > 100 -> "huge circle";
        case Circle(double r) -> "circle of radius " + r;
        case Square(double side) -> "square of side " + side;
        case Triangle(double b, double h) -> "triangle " + b + "×" + h;
    };
}

Exceptions — The Checked Experiment

Java is the only mainstream language to enforce checked exceptions: exceptions that must be declared or caught, not merely thrown. Three categories:

  • Error — unchecked, VM-level problems (OutOfMemoryError, StackOverflowError). Don’t catch.
  • RuntimeException — unchecked, programming errors (NullPointerException, IllegalArgumentException, IndexOutOfBoundsException).
  • Other Exceptions — checked, recoverable conditions (IOException, SQLException).

The checked exception experiment is widely regarded as Java’s most controversial design choice. Modern wisdom (Bloch, Goetz, and the broader Java community): use unchecked exceptions for most cases. Kotlin, Scala, and C# all chose not to follow.
