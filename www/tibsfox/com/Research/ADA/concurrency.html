Concurrency & Real-Time

Tasks, rendezvous, protected objects, Ravenscar, Real-Time Annex — the flagship feature that makes airliners fly
Overview Language Concurrency History Implementations Safety-Critical SPARK Rosetta Learn
Ada is the only mainstream language whose first standard, in 1983, had concurrency built into the grammar. Not a library. Not a system-call wrapper. A first-class citizen, with the keyword task sitting next to procedure, function, and package. This single design decision — driven by the 1978 Steelman requirements — is why Airbus flight control software is written in Ada. It is why the Paris Metro Line 14 runs driverless trains. It is why ESA space probes fly on bare metal. Forty-three years later, Ada tasking remains the benchmark against which other concurrent languages are measured when the requirement is this had better not kill anybody.

Part I — Why Concurrency in the Language

The problem in 1978

In 1978 the U.S. Department of Defense was maintaining roughly 450 distinct programming languages across its embedded systems portfolio. The B-1 bomber used JOVIAL J73. The Navy used CMS-2. The Space Shuttle used HAL/S. Air traffic control used JOVIAL J3. Each had some notion of “do things in parallel” — because real-time embedded systems inherently have to — but each did it differently. HAL/S had explicit SCHEDULE statements. PL/I had TASK and EVENT. JOVIAL had nothing at all — you wrote your own scheduler. None were portable. None were provable.

The Steelman requirements (1978) Section 9 laid out what a unified replacement language would have to provide:

  • 9A. The language shall provide for parallel processing of logically independent activities.
  • 9B. The language shall permit tasks to communicate with one another at synchronization points.
  • 9C. Tasks shall not rely on details of the implementation for correctness.
  • 9D. The language shall permit access to shared variables to be synchronized.
  • 9E. Mutual exclusion shall be expressible.
  • 9F. Scheduling shall be definable.

Notice what Steelman does not say. It does not say “provide a library.” It says the language shall provide. That phrasing was deliberate: when you compile a program that uses concurrency, the compiler itself has to understand what you are doing — not just pass opaque pointers to a runtime library and hope for the best.

Ichbiah’s design (1979–83)

Jean Ichbiah’s Green team drew directly on three sources: C. A. R. Hoare’s CSP (August 1978), Per Brinch Hansen’s monitors, and the real-world embedded systems experience of the team members. Ichbiah chose CSP as the primary model. Ada 83’s tasking was built around the rendezvous. Monitors would come later, in Ada 95, once it became clear that pure rendezvous was too expensive for passive data.

The key insight — the thing that made the design radical for 1983 — was that a task is an object with a type, declared like any other object. You don’t call a library function to spawn a thread. You declare a task:

task My_Task;
task body My_Task is
begin
   Put_Line ("Hello from a task");
end My_Task;

That task starts running the moment the enclosing scope’s begin is reached. It stops when its body finishes. No pthread_create, no new Thread(), no go func() { }(). Just a declaration. The declaration is the spawn.

This is structured concurrency, roughly 35 years before Martin Sústrik coined the term in 2016.

Four properties Ada tasking guarantees

  1. Lexical scoping of tasks. A task declared inside a block cannot outlive that block. The enclosing scope waits for all child tasks before unwinding. You cannot accidentally leak a task.
  2. Compile-time synchronization checking. Entry signatures are type-checked. Protected barriers are compile-time expressions. A pthread mutex acquired in one function and released in another cannot be checked; an Ada protected body’s end is part of the syntax.
  3. Portable semantics. The Ada Reference Manual specifies task semantics precisely. An Ada program works identically on GNAT/Linux, GNAT/VxWorks, GNAT/LynxOS, and GNAT/ORK+ (bare-metal ARM for ESA missions), because the language defines what tasks do, not the OS.
  4. Integration with the type system. Tasks have discriminants. They live in arrays, records, and generics. A thread pool is Workers : array (1 .. N) of Worker_Task; — a one-liner. A thread pool in C is 200 lines of library code.

The cost

Nothing is free. Ada tasking imposes three costs: runtime weight (GNARL is larger than a bare C runtime — hence Ravenscar), language complexity (the ARM 2012 is 873 pages; tasking accounts for ~150), and compilation complexity (compilers must implement tasking correctly; historically, some commercial Ada compilers had tasking bugs). The Ada position is that these costs are worth paying when the alternative is shipping bugs in safety-critical software.

Part II — Tasks as Active Objects

Task declaration and task body

with Ada.Text_IO; use Ada.Text_IO;

procedure Hello_Task is
   task Greeter;

   task body Greeter is
   begin
      for I in 1 .. 5 loop
         Put_Line ("Greeter iteration" & I'Image);
         delay 0.5;
      end loop;
   end Greeter;

begin
   for I in 1 .. 5 loop
      Put_Line ("Main iteration" & I'Image);
      delay 0.3;
   end loop;
   -- Main reaches end here, but the procedure does not return
   -- until Greeter finishes. This is automatic.
end Hello_Task;

The procedure Hello_Task does not return until both the main body and the Greeter task have finished. If Greeter hasn’t finished by the time the main body reaches end, the runtime blocks there waiting. This is the structured concurrency guarantee: a task cannot outlive its enclosing scope.

Task types and discriminants

task type Worker (ID : Positive);

task body Worker is
begin
   Put_Line ("Worker" & ID'Image & " starting");
   delay Duration (ID) * 0.5;
   Put_Line ("Worker" & ID'Image & " done");
end Worker;

W1 : Worker (ID => 1);
W2 : Worker (ID => 2);
W3 : Worker (ID => 3);

A task type is a type. Instances are declared like any other object. Discriminants are the initialization parameters — read-only inside the body, set at creation time. Ada had discriminated task types in 1995 — other languages reached this level of integration much later, if at all.

Tasks as type-system citizens

task type Background_Worker (ID : Positive);
task body Background_Worker is ...;

type Device is record
   Name   : String (1 .. 10);
   Worker : Background_Worker (ID => 42);  -- Each Device has its OWN task
end record;

D1 : Device := (Name => "Sensor_001", Worker => (ID => 1));
D2 : Device := (Name => "Sensor_002", Worker => (ID => 2));

Here is a thing no other mainstream language can do. When you create a Device, you automatically get a Background_Worker running inside it. When the Device is destroyed, its worker is waited for automatically. This “active object” pattern falls out naturally from Ada’s tasking, but is extraordinarily awkward in Java or C++.

Part III — The Rendezvous (Ada 83’s Original)

The rendezvous is modeled on Hoare’s CSP. Task A says “I want to talk to task B,” task B says “I’m ready to talk to whoever calls,” and the two meet at a designated point, exchange information, and proceed on their own ways. In syntax:

  • Task B declares an entry — a named synchronization point with parameters.
  • Task A calls the entry: B.Entry_Name (arguments);
  • Task B executes accept Entry_Name (parameters) do ... end;

Both must reach their respective statements before either proceeds. The first to arrive blocks and waits. When both arrive, the body of the accept runs — and while it runs, A is still blocked. Parameters flow in and out during the handshake.

task Squarer is
   entry Square (N : Integer; Result : out Integer);
end Squarer;

task body Squarer is
begin
   loop
      accept Square (N : Integer; Result : out Integer) do
         Result := N * N;
      end Square;
      exit;
   end loop;
end Squarer;

-- Caller:
R : Integer;
Squarer.Square (12, R);
-- R is now 144

The producer-consumer pattern

task Buffer is
   entry Put (Item : Integer);
   entry Get (Item : out Integer);
end Buffer;

task body Buffer is
   Stored : Integer;
begin
   loop
      accept Put (Item : Integer) do
         Stored := Item;
      end Put;
      accept Get (Item : out Integer) do
         Item := Stored;
      end Get;
   end loop;
end Buffer;

This enforces strict alternation: producer puts, consumer gets, producer puts, consumer gets. The producer cannot get ahead. To allow buffering, you need to choose which entry to accept based on state — which is exactly what selective accept provides.

Part IV — Selective Accept

Ada’s selective accept is written with select. It presents alternatives, and the task waits until one becomes ready. Each can have a guard and there can be delay alternatives, an else clause for polling, and a terminate alternative for graceful shutdown.

Bounded buffer with guards

task Buffer is
   entry Put (Item : Integer);
   entry Get (Item : out Integer);
end Buffer;

task body Buffer is
   Max  : constant := 4;
   Data : array (0 .. Max - 1) of Integer;
   Head : Natural := 0;
   Tail : Natural := 0;
   Cnt  : Natural := 0;
begin
   loop
      select
         when Cnt < Max =>
            accept Put (Item : Integer) do
               Data (Tail) := Item;
            end Put;
            Tail := (Tail + 1) mod Max;
            Cnt  := Cnt + 1;
      or
         when Cnt > 0 =>
            accept Get (Item : out Integer) do
               Item := Data (Head);
            end Get;
            Head := (Head + 1) mod Max;
            Cnt  := Cnt - 1;
      or
         terminate;
      end select;
   end loop;
end Buffer;

Subtle optimization: updating Head, Tail, Cnt happens outside the accept body. Anything inside the accept blocks the caller; anything outside blocks only this task. Minimize work inside the accept.

The watchdog pattern

task body Watchdog is
begin
   loop
      select
         accept Ping;
         -- Got a ping, reset the watchdog.
      or
         delay 10.0;
         Put_Line ("WATCHDOG: 10 seconds elapsed, triggering reset");
      end select;
   end loop;
end Watchdog;

The terminate alternative

terminate is Ada’s graceful-shutdown mechanism. It means: if no one is going to call me anymore, I’m willing to stop. The runtime detects that all tasks are waiting for callers and that the master has finished its own work, concludes that the system has no useful work left, and terminates all tasks with open terminate alternatives simultaneously. No explicit shutdown messages. No leaked tasks. Just a rule that always does the right thing.

Part V — Protected Objects (Ada 95)

The rendezvous is elegant but expensive — hundreds of microseconds on 1980s hardware. By the late 1980s, Ada programs were hitting two walls: cost (full context switch per rendezvous) and awkwardness for passive data (you had to model every shared counter or queue as an active server task). Ada 95 fixed both with protected objects, modeled on Brinch Hansen’s and Hoare’s monitors but adapted to Ada’s type-based model.

Three kinds of operation

KindAccessSemantics
functionRead-onlyMultiple concurrent readers; no writes allowed; compiler-checked
procedureRead-writeExclusive; no other operation runs concurrently
entryRead-write + barrierQueued until the when barrier evaluates true

Readers-writers semantics is built into the language. Barriers handle condition synchronization. No explicit condition variables, no “forgetting to signal.”

The bounded buffer as a protected object

protected type Bounded_Buffer is
   entry Put (Item : Integer);
   entry Get (Item : out Integer);
private
   Max  : constant := 10;
   Data : array (0 .. Max - 1) of Integer;
   Head : Natural := 0;
   Tail : Natural := 0;
   Cnt  : Natural := 0;
end Bounded_Buffer;

protected body Bounded_Buffer is

   entry Put (Item : Integer) when Cnt < Max is
   begin
      Data (Tail) := Item;
      Tail := (Tail + 1) mod Max;
      Cnt  := Cnt + 1;
   end Put;

   entry Get (Item : out Integer) when Cnt > 0 is
   begin
      Item := Data (Head);
      Head := (Head + 1) mod Max;
      Cnt  := Cnt - 1;
   end Get;

end Bounded_Buffer;

Compare this to the POSIX threads equivalent:

pthread_mutex_lock(&buffer_mutex);
while (count == MAX) {
    pthread_cond_wait(&not_full, &buffer_mutex);
}
// ... put item ...
pthread_cond_signal(&not_empty);
pthread_mutex_unlock(&buffer_mutex);

If you forget the while loop (spurious wakeups), your program is broken. If you signal the wrong condition variable, your program is broken. If you release the lock in the wrong order, your program is broken. The Ada version is a when barrier; the runtime handles all the queueing and re-evaluation. There is no “forgetting to signal” because there are no signals.

Barrier re-evaluation rules

Every time a protected procedure or entry body completes, the runtime re-evaluates the barriers of all entries that have queued callers. This happens before the lock is released, as part of the same atomic operation. If multiple barriers become true, the runtime picks one queued caller (FIFO within an entry, implementation-defined between entries). Barriers must be “reasonably pure” — no side effects, no calls to other protected operations.

Counting semaphore

protected type Counting_Semaphore (Initial : Natural) is
   entry P;
   procedure V;
private
   Count : Natural := Initial;
end Counting_Semaphore;

protected body Counting_Semaphore is
   entry P when Count > 0 is
   begin
      Count := Count - 1;
   end P;

   procedure V is
   begin
      Count := Count + 1;
   end V;
end Counting_Semaphore;

The discriminant Initial sets the count at construction. P is an entry with a barrier; V is a procedure. When V increments Count, the runtime re-evaluates P’s barrier and releases one queued caller. No condition variables, no signaling discipline.

Performance

On modern GNAT runtimes, a protected procedure call with no contention takes roughly the same time as a futex-based mutex lock/unlock pair in C — low tens of nanoseconds on x86-64. A protected entry call with no queueing is slightly more expensive because of barrier evaluation, but still sub-microsecond. A full rendezvous costs a few microseconds because it involves task scheduling.

Rule of thumb: protected objects for passive data sharing, rendezvous for active coordination between peer tasks.

Part VI — Timing and Ada.Real_Time

Relative vs absolute delay

Ada has two forms of delay that look similar but have very different semantics.

Wrong — relative delay accumulates drift:

loop
   Do_Work;
   delay 0.1;  -- 100 ms from NOW
end loop;

Each iteration takes Do_Work’s time plus 100 ms plus scheduling jitter. If Do_Work takes 20 ms and the scheduler adds 5 ms of jitter, the task actually runs every 125 ms.

Right — absolute delay stays locked to the period:

declare
   use Ada.Real_Time;
   Period : constant Time_Span := Milliseconds (100);
   Next   : Time := Clock + Period;
begin
   loop
      Do_Work;
      delay until Next;
      Next := Next + Period;
   end loop;
end;

Each iteration sleeps until exactly the next period boundary. If Do_Work takes longer than expected, the next iteration happens sooner, compensating. The task stays locked to a precise 100 ms period. This is how you write a periodic real-time task.

Ada.Real_Time

Ada.Real_Time provides a monotonic clock and time arithmetic specifically designed for real-time programming:

type Time is private;
type Time_Span is private;

function Clock return Time;  -- monotonic

function Nanoseconds  (NS : Integer) return Time_Span;
function Microseconds (US : Integer) return Time_Span;
function Milliseconds (MS : Integer) return Time_Span;
function Seconds      (S  : Integer) return Time_Span;

Ada.Real_Time.Time is distinct from Ada.Calendar.Time. Calendar time is wall-clock and can jump when the system clock is adjusted. Real-time is monotonic — never goes backwards, never jumps forward, suitable for delay until in hard real-time contexts. On GNAT/Linux it is backed by clock_gettime(CLOCK_MONOTONIC, ...); on bare metal it is backed by a hardware timer with sub-microsecond resolution.

Part VII — Asynchronous Transfer of Control (ATC)

Ada 95 introduced one of the more controversial features in the language: asynchronous transfer of control. It lets one task interrupt another task’s current activity in response to an event. The syntax is a special form of select:

select
   Trigger_Statement;
then abort
   Abortable_Statements;
end select;

The Abortable_Statements begin executing. Simultaneously, the runtime watches the Trigger_Statement. If the trigger fires before the abortable statements finish, they are aborted and control transfers to after end select. If the abortable statements finish first, the trigger is abandoned.

Timeout on a long computation

select
   delay 2.0;
   Put_Line ("Timeout! Computation aborted.");
then abort
   Long_Computation;
end select;

If Long_Computation finishes within 2 seconds, the timeout is abandoned. Otherwise, the runtime aborts it and prints the timeout message. In C/pthreads, equivalent functionality requires pthread_cancel, cancellation points, RAII cleanup, and hope that library functions are cancellation-safe. In Ada, the runtime handles it.

Why ATC is controversial

ATC is powerful but dangerous. The abortable statements can be interrupted at essentially any point. The runtime provides abort-deferred regions (inside protected operations, inside accept bodies, inside Finalize), where the abort is delayed until the region exits. But anywhere else, your code must be prepared to be cut off in the middle of anything.

The Ravenscar profile forbids ATC. If you are writing safety-critical software, you do not use select ... then abort because you cannot easily prove correctness across the abort boundary. For non-safety-critical code where ATC is a clean match (GUI cancellation, timeouts on bounded operations), it is a valuable tool.

Part VIII — The Real-Time Annex (Annex D)

Ada 95 introduced annexes — optional language additions. Annex D (Real-Time Systems) is the most important for avionics, aerospace, and embedded systems. It provides the machinery for precise timing, priority-based scheduling, and the determinism that hard real-time needs.

Task priorities

task High_Priority_Task with Priority => 10;
task Low_Priority_Task  with Priority => 3;

Higher numeric values mean higher priority (opposite of POSIX nice). The range is implementation-defined but must include System.Priority (typically 0..30), with System.Interrupt_Priority extending higher for interrupt handlers.

Dispatching policies

PolicyBehavior
FIFO_Within_PrioritiesDefault. Highest priority runs; same-priority is FIFO; no same-priority preemption
Non_Preemptive_FIFO_Within_PrioritiesOnce a task runs, it runs until it blocks (no preemption)
Round_Robin_Within_PrioritiesRound-robin at each priority level (Ada 2005)
EDF_Across_PrioritiesEarliest Deadline First via Ada.Dispatching.EDF (Ada 2005)

Ceiling locking — the priority inversion fix

Priority inversion: a high-priority task waits for a resource held by a low-priority task, which is preempted by a medium-priority task, so the high-priority task ends up waiting for the medium-priority task to finish. The most famous instance was the 1997 Mars Pathfinder repeated-reset bug, caused by priority inversion in VxWorks C code. The fix was priority inheritance on the affected mutex.

Ada’s answer is the ceiling priority protocol, encoded in Annex D:

pragma Locking_Policy (Ceiling_Locking);

protected type Shared_Data
  with Priority => 15
is
   procedure Update (V : Integer);
   function  Read return Integer;
private
   Value : Integer := 0;
end Shared_Data;

The ceiling (15) is the maximum priority of any task that will ever call this protected object. When a task calls a protected operation, its priority is temporarily raised to the ceiling. This gives you:

  1. No preemption of protected operations by higher-priority callers. Any task that could call is now blocked by the priority raise.
  2. No deadlock on a single processor as long as protected objects are entered in ceiling order.
  3. Bounded priority inversion. Worst-case blocking time equals the longest protected operation, not an unbounded chain.

This makes response-time analysis tractable. Ravenscar mandates ceiling locking.

Interrupt handlers

protected Interrupt_Handler is
   procedure Handle;
   pragma Attach_Handler (Handle, SIGINT);
end Interrupt_Handler;

protected body Interrupt_Handler is
   procedure Handle is
   begin
      Put_Line ("Interrupt received");
   end Handle;
end Interrupt_Handler;

The pragma Attach_Handler wires a protected procedure to an interrupt. When the interrupt fires, the runtime calls Handle with all the protected-object machinery engaged — mutual exclusion, ceiling locking, etc. Interrupt handlers are ordinary Ada code with the safety of protected operations.

Part IX — The Ravenscar Profile

Annex D gives you enough rope to hang yourself. For highest-assurance systems (DO-178C DAL A avionics, IEC 61508 SIL 4 industrial, ECSS space software), you want a restricted subset that is analyzable and verifiable. That subset is the Ravenscar profile, defined at the 8th International Real-Time Ada Workshop held in Ravenscar, Yorkshire, in 1997, and standardized as ISO/IEC TR 15942:2000.

Applying the profile

pragma Profile (Ravenscar);

This one-line pragma is shorthand for a long list of restrictions and policies: FIFO-within-priorities dispatching, ceiling locking, blocking detection, no abort statements, no dynamic priorities, no implicit heap allocations, no local protected objects, no local timing events, no relative delay, no requeue, no select statements (with one exception), and several more.

What Ravenscar forbids and why

  • No task allocators — all tasks declared statically at library level. Gives a bounded, known-at-compile-time number of tasks. Essential for schedulability analysis.
  • No task hierarchy — only library-level tasks. Combined with no allocators, the entire task set is a flat collection known at compile time.
  • No task termination — Ravenscar tasks are expected to run forever. Periodic loops or server loops, no “scripts.”
  • No ATC, no abort — hard to analyze.
  • No multi-accept select — tasks communicate only through protected objects.
  • No relative delay — only delay until. Prevents accumulating drift.
  • Max entry queue length = 1 — the hallmark restriction. Each protected entry can have at most one queued task. If N tasks need to wait on an event, each gets its own entry. This keeps scheduling analysis tractable.
  • Simple barriers — barrier expressions are a single Boolean variable or a negation. No compound expressions, no function calls. Trivially fast and statically analyzable.
  • No requeue — adds analysis complexity.
  • No dynamic priorities — priorities are set at compile time and never change.
  • Detect_Blocking — the runtime must raise Program_Error if a protected operation attempts to block.

A complete Ravenscar program

pragma Profile (Ravenscar);

with Ada.Real_Time; use Ada.Real_Time;

package Shared is
   protected type Counter_T
     with Priority => 10
   is
      procedure Increment;
      function  Value return Integer;
   private
      Count : Integer := 0;
   end Counter_T;

   Counter : Counter_T;
end Shared;

package Workers is
   task Periodic_1 with Priority => 5;  -- 100 ms period
   task Periodic_2 with Priority => 6;  -- 50 ms period
   task Monitor    with Priority => 3;  -- 1 s period
end Workers;

package body Workers is

   task body Periodic_1 is
      Period : constant Time_Span := Milliseconds (100);
      Next   : Time := Clock + Period;
   begin
      loop
         Shared.Counter.Increment;
         delay until Next;
         Next := Next + Period;
      end loop;
   end Periodic_1;

   -- Periodic_2 and Monitor similar

end Workers;

Three tasks at library level. Ceiling is at least as high as the highest caller (Periodic_2 at 6), so bounded priority inversion is guaranteed. This program can be fed to a response-time analysis tool (MAST, Cheddar, SymTA/S) and produce precise worst-case response times, provably meeting all deadlines.

Ravenscar in real systems

Ravenscar runs production systems at an impressive list of places:

  • AdaCore GNAT Pro Safety-Critical — a Ravenscar-compliant runtime.
  • ORK+ — Open Ravenscar Real-Time Kernel, Universidad Politécnica de Madrid, used by ESA.
  • ESA space missions — BepiColombo (2018), Solar Orbiter (2020), JUICE (2023), Gaia, Sentinel, Herschel, Planck. All run Ada on bare-metal LEON processors.
  • Airbus A350, A400M, parts of A380 — flight control software.
  • Eurofighter Typhoon, Rafale — avionics.
  • Boeing 777 (largely) and 787 (partially).
  • F-22 Raptor, F-35 (partially).
  • Ariane 6 launch vehicle.
  • Paris Metro Ligne 14, London DLR.
  • Eurocontrol iFACTS, CAATS.

For all of these, Ravenscar is not optional — it is the difference between a certifiable system and an uncertifiable one. Jorvik (Ada 2012, named after the Viking name for York) is a slightly more permissive variant: multiple tasks can queue on an entry, barriers can be slightly more complex. Used where Ravenscar is too strict but full Annex D is too much.

SPARK + Ravenscar

The gold standard for verified concurrent Ada is SPARK 2014 with the Ravenscar profile. SPARK is the Ada subset that can be formally verified by the AdaCore gnatprove toolset. With SPARK + Ravenscar you can prove:

  • No task races on shared data (SPARK flow analysis)
  • Protected objects are used correctly
  • Schedulability (external tools with SPARK assertions)
  • Absence of runtime errors
  • Functional correctness (if you write contracts)

This is what AdaCore sells to customers building systems where a software bug means deaths. Airbus, Boeing, Lockheed Martin, and several national space agencies are among the customers. See the SPARK page for detail.

Part X — Parallel Constructs (Ada 2022)

Ada didn’t get parallel for loops until Ada 2022. Why so late? The answer is that Ada’s tasking model already handled most of what you’d want parallel-for for — you could write the worker pool pattern by hand. The committee waited to see what worked in OpenMP, TBB, Rayon, and Rust, then designed a parallel-for construct that fit cleanly into Ada’s existing semantics.

Parallel for loops

parallel for I in 1 .. N loop
   A (I) := Compute (I);
end loop;

Functionally equivalent to a sequential loop, except that iterations may run in parallel. The compiler and runtime decide how many threads to use, how to partition the iteration space, and how to schedule the work.

Parallel blocks

parallel
   Compute_X;
and
   Compute_Y;
and
   Compute_Z;
end parallel;

Runs three statements in parallel and waits for all to finish. The structured-concurrency version of Go’s go func(); go func(); wg.Wait().

Tasks vs parallel constructs

  • Tasks for long-running, coordinated activities — servers, periodic controllers, event handlers.
  • Parallel constructs for data parallelism over large iteration spaces — numerical computation, batch processing, bulk transformations.

Under the hood, the Ada 2022 runtime typically uses a work-stealing thread pool similar to Intel TBB or OpenMP. The threads in the pool are Ada tasks, so the whole thing is internally consistent.

Part XI — Canonical Patterns

Dining philosophers — no deadlock

type Fork_Index is mod 5;

protected Table is
   entry Pick_Up (Phil : Fork_Index);
   procedure Put_Down (Phil : Fork_Index);
private
   Fork_Free : array (Fork_Index) of Boolean := (others => True);
end Table;

protected body Table is

   entry Pick_Up (Phil : Fork_Index)
     when Fork_Free (Phil) and Fork_Free (Phil + 1) is
   begin
      Fork_Free (Phil)     := False;
      Fork_Free (Phil + 1) := False;
   end Pick_Up;

   procedure Put_Down (Phil : Fork_Index) is
   begin
      Fork_Free (Phil)     := True;
      Fork_Free (Phil + 1) := True;
   end Put_Down;

end Table;

This solution is correct because the Pick_Up barrier atomically checks both forks. A philosopher never picks up one fork and then waits for the other. The traditional deadlock (everyone picks up left, everyone waits for right) is impossible because the barrier atomically verifies both forks are free before the transfer. This is a nice illustration of what protected-object barriers give you for free: compound conditions checked atomically.

Generic bounded buffer

generic
   type Item is private;
   Capacity : Positive := 16;
package Bounded_Buffers is

   protected type Buffer is
      entry Put (X : Item);
      entry Get (X : out Item);
      function Current_Count return Natural;
   private
      Data  : array (0 .. Capacity - 1) of Item;
      Head  : Natural := 0;
      Tail  : Natural := 0;
      Count : Natural := 0;
   end Buffer;

end Bounded_Buffers;

Parameterized over the element type and capacity. Instantiate for any type; get a fully synchronized bounded buffer with no additional code. This is the kind of abstraction Ada’s generic system enables.

External abort via signal

protected Signal is
   entry Wait;
   procedure Fire;
private
   Fired : Boolean := False;
end Signal;

protected body Signal is
   entry Wait when Fired is
   begin
      null;
   end Wait;

   procedure Fire is
   begin
      Fired := True;
   end Fire;
end Signal;

-- In a worker task:
select
   Signal.Wait;
   Put_Line ("Worker: received abort signal");
then abort
   loop
      Do_Work;
      delay 0.5;
   end loop;
end select;

Part XII — Where Ada Concurrency Actually Runs

Every production system where you find Ada concurrency has the same fingerprints:

  1. Real-time deadlines that must be met.
  2. Safety-critical or mission-critical correctness requirements.
  3. Long-term maintainability (the software lives for decades).
  4. Certification against some standard (DO-178C, IEC 61508, EN 50128, IEC 62304, ECSS-Q-ST-80, IEC 60880).
  5. Budget and schedule pressure that requires finding bugs at compile time.

See the Safety-Critical page for the full roster. Here are the concurrency-relevant highlights:

  • Airbus A350, A400M, A380 — flight control loops as periodic Ada tasks at 50/100/200 Hz. Ceiling-locked protected objects for shared state. Rate-monotonic schedulability analysis.
  • Boeing 777 — primary flight computer in Ada 83/95 with tasks and protected objects.
  • Eurofighter Typhoon, Rafale — mission computer with dozens of concurrent tasks (sensor fusion, radar tracking, weapons management) communicating through protected objects.
  • ESA missions — BepiColombo, JUICE, Solar Orbiter, Gaia on bare-metal LEON with ORK+ Ravenscar runtime.
  • Paris Metro Ligne 14 — first driverless subway (1998), still running 28 years later on Ada tasks under a Ravenscar-like profile.
  • Eurocontrol iFACTS — UK upper-airspace ATC with extensive use of tasks for concurrent flight tracks.

The pattern: periodic tasks for control loops, sporadic tasks for events, protected objects for shared state, interrupt handlers as protected procedures, ceiling locking for bounded priority inversion, and rate-monotonic analysis proving every deadline is met. Forty-three years of Ada tasking means forty-three years of nobody died because the concurrency library had a bug we didn’t catch.

Part XIII — The Philosophical Position

Ada’s claim is simple: concurrency is too important to be left to libraries. A library can only check what its types allow, and most languages’ type systems were not designed for concurrency. Ada’s was. The results:

  1. Compile-time checking of synchronization correctness.
  2. Portable semantics that work identically across every target.
  3. Integration with the type system — tasks and protected objects are types.
  4. Analyzability by static analysis tools, schedulability tools, and SPARK formal verification.
  5. Restriction profiles (Ravenscar, Jorvik) for certification.

The trade-off: runtime weight, language complexity, compilation complexity. For a web server, pthreads are fine. For a flight control computer, the compile-time checking and portable semantics are the difference between certification and catastrophe.

Ada is a language from 1983 that solved concurrency problems we are still arguing about in 2026. It is the quietest important language in computing, and its concurrency model is why.
