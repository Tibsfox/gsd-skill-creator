Concurrency is the one domain where language-level choices matter more than anywhere else. A mutex library can only be as good as its host language’s ability to reason about ownership. A thread abstraction is only as safe as its host language’s type system. Ada committed to concurrency as a language feature in 1983, and that single decision has shaped what every high-integrity system has been able to accomplish for forty years.

One Problem, Seven Languages — Bounded Buffer

The canonical concurrency exercise: a fixed-size buffer shared between producer and consumer threads. Producer blocks when full, consumer blocks when empty. Watch how each language shapes the problem.

Ada (the reference — protected object)

protected type Buffer (Capacity : Positive) is
   entry Put (Item : Integer);
   entry Get (Item : out Integer);
private
   Data  : array (1 .. Capacity) of Integer;
   Head  : Positive := 1;
   Tail  : Positive := 1;
   Count : Natural  := 0;
end Buffer;

protected body Buffer is
   entry Put (Item : Integer) when Count < Capacity is
   begin
      Data (Tail) := Item;
      Tail        := (Tail mod Capacity) + 1;
      Count       := Count + 1;
   end Put;

   entry Get (Item : out Integer) when Count > 0 is
   begin
      Item  := Data (Head);
      Head  := (Head mod Capacity) + 1;
      Count := Count - 1;
   end Get;
end Buffer;

The when barriers on the entries are the entire synchronization logic. Callers queue until their barrier becomes true. The language guarantees mutual exclusion, FIFO queueing by default, and with the Real-Time Annex, ceiling-locked priority inheritance to prevent priority inversion. This is the cleanest bounded-buffer in any mainstream language, and it has been part of Ada since 1995.

Go (goroutines + channels)

func bufferedChannel(capacity int) (chan<- int, <-chan int) {
    ch := make(chan int, capacity)
    return ch, ch
}

// Producer
go func() {
    for i := 0; i < 100; i++ {
        ch <- i  // blocks when buffer is full
    }
    close(ch)
}()

// Consumer
for value := range ch {
    process(value)
}

Go’s buffered channel is the bounded buffer. The language built the abstraction in directly — no protected object, no monitor, no explicit mutex. This is the most concise expression in mainstream programming. The trade-off: Go’s goroutines are managed by a runtime with garbage collection, which makes Go unsuitable for hard real-time. Ada’s tasks are compiled to OS threads with predictable scheduling.

Rust (channels + std::sync::mpsc, or crossbeam)

use std::sync::mpsc;
use std::thread;

let (tx, rx) = mpsc::sync_channel(capacity);

thread::spawn(move || {
    for i in 0..100 {
        tx.send(i).unwrap();  // blocks when buffer is full
    }
});

for value in rx {
    process(value);
}

Rust’s sync_channel is in the standard library. Move semantics enforce that only one side owns the producer end and one side owns the consumer end — the compiler catches the common bug of sending a value while still holding a reference to it. Rust’s Send and Sync marker traits are the static typing framework. No protected entries, no barriers — just channels and ownership.

Erlang (mailboxes, actor-based)

-module(buffer).
-export([start/1, put/2, get/1]).

start(Capacity) ->
    spawn(fun() -> loop([], Capacity) end).

loop(Queue, Capacity) when length(Queue) < Capacity ->
    receive
        {put, From, Item} ->
            From ! ok,
            loop(Queue ++ [Item], Capacity);
        {get, From} when length(Queue) > 0 ->
            [H | T] = Queue,
            From ! {ok, H},
            loop(T, Capacity)
    end;
loop(Queue, Capacity) ->  % Full
    receive
        {get, From} ->
            [H | T] = Queue,
            From ! {ok, H},
            loop(T, Capacity)
    end.

put(Pid, Item) ->
    Pid ! {put, self(), Item},
    receive ok -> ok end.

Erlang uses the actor model: each process has a mailbox, communication is entirely via message passing, and there is no shared state. The buffer is implemented as a process that maintains its own state and selectively receives messages. Erlang’s approach is beautiful for distribution (processes can live on different machines) but not a fit for hard real-time — the runtime does soft scheduling with preemption.

Java (BlockingQueue)

import java.util.concurrent.ArrayBlockingQueue;

BlockingQueue<Integer> buffer = new ArrayBlockingQueue<>(capacity);

// Producer
new Thread(() -> {
    for (int i = 0; i < 100; i++) {
        try {
            buffer.put(i);  // blocks when full
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return;
        }
    }
}).start();

// Consumer
while (true) {
    try {
        Integer value = buffer.take();
        process(value);
    } catch (InterruptedException e) {
        return;
    }
}

Java’s java.util.concurrent library (Doug Lea, 2004) gave the JVM production-quality concurrency abstractions. BlockingQueue is the bounded buffer. The ceremony around InterruptedException is the JVM’s legacy of trying to retrofit interrupt handling onto threads that were not designed for it.

C + pthreads (the reference old-school)

#include <pthread.h>

typedef struct {
    int data[CAPACITY];
    int head, tail, count;
    pthread_mutex_t mutex;
    pthread_cond_t  not_full, not_empty;
} buffer_t;

void put(buffer_t *b, int item) {
    pthread_mutex_lock(&b->mutex);
    while (b->count == CAPACITY)
        pthread_cond_wait(&b->not_full, &b->mutex);
    b->data[b->tail] = item;
    b->tail = (b->tail + 1) % CAPACITY;
    b->count++;
    pthread_cond_signal(&b->not_empty);
    pthread_mutex_unlock(&b->mutex);
}

int get(buffer_t *b) {
    pthread_mutex_lock(&b->mutex);
    while (b->count == 0)
        pthread_cond_wait(&b->not_empty, &b->mutex);
    int item = b->data[b->head];
    b->head = (b->head + 1) % CAPACITY;
    b->count--;
    pthread_cond_signal(&b->not_full);
    pthread_mutex_unlock(&b->mutex);
    return item;
}

Manual mutex, manual condition variables, manual locking order. Every while loop around a pthread_cond_wait is there because spurious wakeups exist and pthreads cannot prevent them. Forget a pthread_mutex_unlock and you deadlock. Get the lock acquisition order wrong across two buffers and you deadlock. Every safety-critical system that still uses C in 2026 uses something like this code, and a small fraction of those systems have bugs that took decades to find.

C++ (std::condition_variable, since C++11)

#include <queue>
#include <mutex>
#include <condition_variable>

template <typename T>
class Buffer {
    std::queue<T> q;
    std::mutex mtx;
    std::condition_variable not_full, not_empty;
    size_t capacity;
public:
    Buffer(size_t c) : capacity(c) {}

    void put(T item) {
        std::unique_lock<std::mutex> lock(mtx);
        not_full.wait(lock, [&]{ return q.size() < capacity; });
        q.push(std::move(item));
        not_empty.notify_one();
    }

    T get() {
        std::unique_lock<std::mutex> lock(mtx);
        not_empty.wait(lock, [&]{ return !q.empty(); });
        T item = std::move(q.front());
        q.pop();
        not_full.notify_one();
        return item;
    }
};

C++11 added RAII locks and lambda-based wait predicates, which are genuinely nicer than raw pthreads. Still more verbose than Ada’s protected object and without Ada’s priority ceiling guarantees.

When Each Wins

Ada wins when you need

  • Hard real-time guarantees. Response-time analysis, rate-monotonic scheduling, priority ceiling protocol — all built in.
  • Safety certification. DO-178C DAL A, EN 50128 SIL 4, IEC 61508 SIL 3, ISO 26262 ASIL D — Ada is the language regulators expect to see.
  • Formal proof of tasking correctness. SPARK + Ravenscar is the gold standard. No other mainstream language has comparable tooling.
  • Long-lived code bases. Ada programs written in 1985 still compile and run on modern GNAT. This matters for avionics, where a plane flies for 40 years.
  • Bit-level hardware control with the safety of a high-level language. Representation clauses let you lay out registers precisely.

Go wins when you need

  • Cloud-scale concurrency for network servers, microservices, CLI tools.
  • Fast compilation, fast iteration, low-ceremony deployment.
  • Millions of lightweight goroutines rather than a few real-time-scheduled tasks.
  • To hire programmers in a hurry — Go’s learning curve is orders of magnitude shallower.

Rust wins when you need

  • Memory safety with zero-cost abstractions.
  • Systems programming without garbage collection.
  • A modern type system with traits, lifetimes, and async/await.
  • A cultural fit with open-source systems programming and modern tooling (Cargo, crates.io).

Erlang wins when you need

  • Massive distribution across many machines with built-in supervision trees.
  • Soft real-time telecom-style workloads.
  • Hot code reloading without stopping the system.
  • The actor model as a first-class programming paradigm.

Ada vs Rust — The Modern Conversation

In the 2020s, the most common question at Ada conferences is: “Why would I use Ada instead of Rust?” The honest answer is that Rust solves most of the memory-safety problems Ada solved first, while remaining fresher, friendlier, and better-tooled for modern open-source development. Rust pulled attention toward “safe systems programming” in a way that benefited Ada by validating the premise.

But Rust has no built-in tasking primitive comparable to Ada’s tasks and protected objects. Rust’s async/await is a library-level abstraction over executors (tokio, async-std, smol) with no real-time scheduling guarantees. Rust’s thread API is thin, and its channel primitives are in the standard library rather than the language. For web servers and CLI tools this is fine. For avionics it is not.

The pragmatic 2020s answer: Rust for the bulk of your codebase, Ada/SPARK for the safety-critical kernel. The Ferrocene project (Ferrous Systems) is bringing Rust into DO-178C territory by applying the same qualification discipline that has existed for Ada compilers for 40 years. Whether Rust can match Ada’s tasking story for certified real-time in the next decade is an open question. For now, airliners still fly on Ada.

Why Concurrency in the Language Matters

The short argument: a library can only check what its types allow, and most languages’ type systems were not designed for concurrency. Ada’s was. The results are visible at every level of abstraction:

  1. Compile-time checking. The compiler knows what a task is, what a protected object is, and what ceiling locking means. It can warn about unreachable accept statements, deadlock risks in nested protected calls, and priority inversions.
  2. Portable semantics. Ada tasking means the same thing on Linux, VxWorks, INTEGRITY, RTEMS, and bare metal. No “works on my OS.” The language defines the contract; the runtime implements it.
  3. Integration with the type system. Protected objects are types. Tasks have discriminants. You can pass a task reference through generics. The concurrency primitives live in the same universe as the rest of your code.
  4. Analyzability. Static analyzers (AdaCore CodePeer, SPARK GNATprove, external tools like Cheddar for response-time analysis) can reason about tasking because the language is explicit. Library-based concurrency defeats analysis.
  5. Restriction profiles. Ravenscar and Jorvik let you turn off the parts of tasking that are hard to certify, leaving a deterministic subset that regulators accept. You cannot do this with a library.

The trade-off: Ada’s runtime is heavier than a no-concurrency language’s, and adding concurrency primitives to the language itself costs complexity every implementor has to pay. Forty years in, that trade has paid off for the industries that use Ada. For cloud servers it has not.

When Ada Loses

  • Web servers and cloud workloads. Go or Rust is the answer. Ada’s concurrency is overkill; its ecosystem is thin for HTTP, JSON, databases, and deployment.
  • Scripting and quick prototypes. Python. Ada is too verbose.
  • Mobile and desktop applications. Swift, Kotlin, Rust, C# are more idiomatic fits.
  • Machine learning. Python + CUDA. Ada has no ML ecosystem.
  • Hiring. There are few Ada programmers compared to C or Go. Training new ones takes months, not weeks. This is often the decisive factor.
