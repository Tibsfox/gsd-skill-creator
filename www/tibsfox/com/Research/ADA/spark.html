SPARK & Formal Verification

Bernard Carré’s Ada subset, GNATprove + SMT, Stone→Bronze→Silver→Gold→Platinum, iFACTS, Tokeneer, IRONSIDES
Overview Language Concurrency History Implementations Safety-Critical SPARK Rosetta Learn
SPARK is the answer to a question most mainstream languages do not even ask: can we prove this program correct? For most software, the answer is “we test it, and hope.” For the software flying the airliner you’re sitting in, the software routing the train you’re riding, the software running the air traffic controller’s radar screen — the answer had better be yes, we can prove it.

1. Origin and Evolution

SPARK was created by Bernard Carré at the University of Southampton in the 1980s. The name is informal — there is no official expansion, though “SPADE Ada Kernel” and “Spec and Programming Ada Kernel” circulate as retroactive backronyms. Carré’s team founded Program Validation Limited (PVL), later Praxis Critical Systems, then Altran Praxis, and the SPARK technology eventually came to AdaCore through the 2015 acquisition of Altran’s Praxis unit.

The 2014 revision was the major leap. Previous versions used comment annotations starting with --#, which was workable but inelegant. SPARK 2014 moved contracts into first-class Ada 2012 aspects (with Pre => ..., with Post => ...), making SPARK a subset of Ada 2012 rather than an extension. The same source file can be compiled by a regular Ada compiler and proved by GNATprove.

2. What SPARK 2014 Restricts

SPARK excludes Ada features that make static reasoning hard. Historically the most significant exclusions:

  • Access types (pointers) — removed in early SPARK, partially re-added with ownership rules in SPARK 2020+. The ownership model is conceptually similar to Rust’s borrow checker.
  • Exceptions — added gradually. Classical SPARK forbade handling exceptions; modern SPARK allows them under restrictions.
  • Side effects in functions — functions must be pure. This is a bigger restriction than it sounds and forces much of the discipline.
  • Undeclared globals — every global a subprogram reads or writes must be declared in its with Global => ... aspect.
  • Aliasing — two out or in out parameters cannot refer to the same object.
  • Concurrency — restricted to a Ravenscar-compatible tasking subset. No ATC, no dynamic tasking, no unrestricted rendezvous.
  • Recursion — allowed but must be shown to terminate via Subprogram_Variant.

A file can mix SPARK and Ada: with SPARK_Mode => On marks a unit as SPARK, with SPARK_Mode => Off excludes it. This lets you prove the kernel and leave the user-interface glue in ordinary Ada.

3. What SPARK 2014 Proves

The assurance ladder

  1. Stone — syntactic SPARK subset. No analysis yet. You’ve proven nothing except that the code fits in the subset.
  2. Bronze — flow analysis. Catches uninitialized reads, dead code, unused variables, information leaks. 100% automatic, fast, finds real bugs.
  3. Silver — absence of runtime errors (AoRTE). No overflow, no bounds violations, no division by zero, no null dereferences. Entire classes of failure provably disappear.
  4. Gold — contract satisfaction. Preconditions, postconditions, and loop invariants all hold. The program behaves the way the contracts describe.
  5. Platinum — full functional correctness. The contracts describe the entire intended behavior, and the implementation is proved to match.

What SPARK does not prove

  • Termination by default — opt-in via Subprogram_Variant and Loop_Variant.
  • Real-time properties — WCET (worst-case execution time) analysis is done separately, typically with AbsInt aiT or Rapita Systems RVS.
  • Platform compliance — if the compiler or runtime is buggy, SPARK cannot catch it. For that you qualify the toolchain (DO-178C tool qualification).
  • Properties outside the contract — the proof is only as strong as what you wrote in the Post.

4. How SPARK Proofs Work Under the Hood

Ada/SPARK source
       |
       v
  GNAT front end (parsing, name resolution, semantic check)
       |
       v
  Frontend in SPARK mode (SPARK-legality checks, flow analysis)
       |
       v
  Why3 intermediate representation (WhyML)
       |
       v
  Verification condition generator
       |
       v
  SMT solvers: Alt-Ergo, CVC4/CVC5, Z3
       |
       v
  Discharge report (proved / unproved VCs)

GNATprove reuses GNAT’s parser and semantic analysis, translates SPARK code to Why3 (a deductive verification platform from INRIA Saclay, maintained by Jean-Christophe Filliâtre, Claude Marché, and colleagues), generates verification conditions from the contracts and code, and dispatches them to SMT solvers.

The three SMT solvers

SolverOriginStrengths
Alt-ErgoINRIA / OCamlProLinear arithmetic, program verification
CVC4 / CVC5Stanford / NYU / IowaArrays, strings, bitvectors
Z3Microsoft ResearchVery fast, bitvectors, nonlinear arithmetic

GNATprove runs multiple solvers in parallel and accepts the first proof. Different solvers succeed on different VCs, which is why the portfolio approach works.

Counterexamples and ghost code

When a VC fails, SPARK returns the failing conditions and (if the solver supports it) a counterexample — concrete values of the inputs that violate the contract. Instead of “property not proved,” you get “the property fails when X = 2_147_483_647 and Y = 1, causing X + Y to overflow.”

When automatic proof cannot close a VC, the developer adds ghost code: variables, subprograms, and annotations that exist only for verification. Ghost entities are compiled out of the release build, so they cost nothing at runtime, but they give the prover the hint it needs.

5. Worked SPARK Examples

A saturation add, proved free of overflow

package Saturation
  with SPARK_Mode => On
is
   type Int32 is range -2**31 .. 2**31 - 1;

   function Sat_Add (X, Y : Int32) return Int32
     with Post =>
       Sat_Add'Result =
         (if    Integer (X) + Integer (Y) > Integer (Int32'Last)  then Int32'Last
          elsif Integer (X) + Integer (Y) < Integer (Int32'First) then Int32'First
          else  Int32 (Integer (X) + Integer (Y)));
end Saturation;

package body Saturation
  with SPARK_Mode => On
is
   function Sat_Add (X, Y : Int32) return Int32 is
   begin
      if Y > 0 and then X > Int32'Last - Y then
         return Int32'Last;
      elsif Y < 0 and then X < Int32'First - Y then
         return Int32'First;
      else
         return X + Y;
      end if;
   end Sat_Add;
end Saturation;

Running gnatprove -P sat.gpr --level=2 proves:

  • No overflow in Int32'Last - Y (the precondition Y > 0 guarantees it’s in range)
  • No overflow in X + Y in the final branch (the preceding guards ensure the sum is representable)
  • The postcondition holds under all inputs

Binary search with a proved loop invariant

function Find (A : Arr; X : Integer) return Natural is
   Lo, Hi, Mid : Index;
begin
   if A'Length = 0 then
      return 0;
   end if;
   Lo := A'First;
   Hi := A'Last;
   while Lo <= Hi loop
      pragma Loop_Invariant
        (Lo in A'Range and then Hi in A'Range
         and then (for all I in A'First .. Index'Pred (Lo) => A (I) /= X)
         and then (for all I in Index'Succ (Hi) .. A'Last  => A (I) /= X));
      pragma Loop_Variant (Decreases => Hi - Lo);
      Mid := Lo + (Hi - Lo) / 2;
      if A (Mid) = X then
         return Natural (Mid);
      elsif A (Mid) < X then
         exit when Mid = Index'Last;
         Lo := Mid + 1;
      else
         exit when Mid = Index'First;
         Hi := Mid - 1;
      end if;
   end loop;
   return 0;
end Find;

The loop invariant says: “Lo and Hi are in range, and the target X is not in any slot outside the current search window.” Combined with the Sorted precondition, this is strong enough for the prover to show that on exit from the loop, the postcondition holds. The loop variant Hi - Lo proves termination.

6. SPARK in Industry

iFACTS — the flagship reference

iFACTS (Interim Future Area Control Tools Support) is the medium-term conflict detection and trajectory-prediction system that sits under the controllers’ radar screens at Swanwick en-route control in Hampshire, covering the upper airspace over England and Wales. Developed by Altran Praxis (née Praxis High Integrity Systems) using SPARK for the safety-critical core.

The claim, documented in several Altran Praxis papers from 2005–2010, was a defect rate of roughly 0.04 defects per KLOC post-delivery, compared with industry norms of 1–10 defects per KLOC for non-SPARK safety-critical software — roughly 30–250× reduction. Productivity was similarly favorable. iFACTS has been operational since 2011.

Tokeneer — the NSA-commissioned demonstration

Tokeneer is the NSA’s 2003 commission to Altran Praxis to build a complete, realistic, security-critical system using SPARK to show that high-assurance software could be delivered at reasonable cost. The result was the Tokeneer ID Station — a biometric access control system for a simulated secure facility, delivered to Common Criteria EAL 5 using SPARK throughout.

The full deliverable — requirements, specifications, SPARK source, proofs, test reports, all artifacts — was released publicly and is available from AdaCore. Metrics from Tokeneer:

  • ~10,000 lines of SPARK code
  • Zero post-delivery defects in initial rigorous test
  • Two defects later discovered, both in code that was not in the SPARK subset (Ada-level glue code to the biometric hardware)
  • Productivity on SPARK portions: ~38 lines per day per engineer, comparable to non-formal projects

Tokeneer remains the most accessible full-scale SPARK case study anyone can download and read.

Other industrial SPARK projects

  • Lockheed Martin C130J Hercules — mission computer in SPARK by Aerosystems International (AeI) in the 1990s. One of the earliest DO-178B Level A uses of SPARK at scale. In service since 1999.
  • EADS / Airbus Defence and Space — various avionics sub-systems, guided weapons, space platforms. Public details are limited for obvious reasons.
  • Rolls-Royce jet engine control — FADEC (Full Authority Digital Engine Control) development for civil engine families, supported by AdaCore’s qualification kits for DO-178C.
  • MULTOS smart card OS — multi-application smart card operating system whose security kernel was formally specified and implemented with SPARK. Carries EMV payment applications, e-passports, and national ID applications.
  • muen separation kernel — x86-64 separation kernel from HSR Rapperswil and secunet, written entirely in SPARK. Hardware-assisted virtualization (Intel VT-x, VT-d) with a very small trusted computing base. First release 2013.
  • IRONSIDES DNS server — Martin Carlisle and Barry Fagin at the US Air Force Academy, ~2012. Design goal: “zero buffer overflows and zero remote code execution vulnerabilities.” Competitive with BIND on query throughput while provably free of memory-safety vulnerabilities.
  • SPARKNaCl — SPARK port of TweetNaCl, compact crypto library, proved free of runtime errors.
  • FreeRTOS-in-SPARK — research project porting FreeRTOS to SPARK.
  • Hi-Lite / Project P — EU-funded research for high-integrity embedded software.

Why the SPARK reference list is short but deep

Formal methods projects are expensive to start but cheap to maintain. A SPARK project tends to be one of a small number of deeply-verified deliverables rather than a mass of ordinary ones. The ratio: Ada code is measured in tens of millions of lines across the industry; SPARK code is measured in millions. Small relative to C or Java — but every line of SPARK is carrying evidence of a theorem, and the evidence is durable.

7. SPARK vs Rust from a Safety Perspective

A useful way to say this: Rust prevents a specific set of bugs statically; SPARK proves that the code meets its specification. These are different levels of assurance.

  • Rust’s guarantee: no data races, no use-after-free, no buffer overflow (in safe code). If your Rust program compiles, it won’t have those bugs.
  • SPARK Silver: no runtime errors. If your SPARK program proves, it can’t raise any runtime exception.
  • SPARK Gold/Platinum: the code satisfies the contracts. If the contracts capture the spec, the code meets the spec.

Rust’s guarantee is “memory and threading primitives won’t betray you.” SPARK’s guarantee is “the program does what you said it does.” Those are complementary, not competing.

The “SPARK for the kernel, Rust for the bulk” pattern

Several recent projects combine the two: the majority of the system is in Rust (Cargo, the Rust ecosystem, Rust libraries), and a small, high-assurance safety kernel is in SPARK. The SPARK kernel is the “monitor” that enforces safety invariants on the Rust bulk. This is the same pattern that was classic for Ada + C (the Ada kernel monitoring the C bulk), updated to the Rust era. It may become the dominant architecture for new high-integrity systems in the 2030s: Rust wherever the ergonomics and ecosystem matter, SPARK for the kernel that the regulator signs off on.

Ferrocene — Rust in certification, inspired by Ada

Ferrocene (Ferrous Systems, Berlin) is bringing Rust into the safety-certification space Ada has occupied for decades. A qualified Rust toolchain for ISO 26262, IEC 61508, and eventually DO-178C. First qualified release shipped 2023. Ferrocene was explicitly modeled on the Ada-with-qualified-GNAT model; the founders have spoken publicly at Embedded World and FOSDEM about studying AdaCore’s business model, qualification process, customer relationships, and tooling philosophy.

In a real sense, Ferrocene is Ada’s gift to Rust.

8. Getting Started with SPARK

  1. Install Alire and GNAT. alr init --bin spark_demo, then alr toolchain --install gnat_native.
  2. Enable SPARK mode in your source file: with SPARK_Mode => On; at the top of the package.
  3. Add contracts. Start with Pre and Post on a simple procedure.
  4. Run GNATprove. gnatprove -P spark_demo.gpr --mode=all --level=2. Read the output.
  5. Escalate to loop invariants. When you have a loop, add pragma Loop_Invariant. This is the point where SPARK stops feeling like Ada-with-assertions and starts feeling like formal verification.
  6. Add ghost code when the prover can’t close a gap. Ghost variables, lemma procedures, explicit invariants.
  7. Work through the AdaCore SPARK tutorial on learn.adacore.com. It runs SPARK in your browser. Best free SPARK resource.
  8. Read Tokeneer. The full deliverable is downloadable from AdaCore. Reading someone else’s verified code is how you learn what good SPARK looks like.

9. Why SPARK Matters

Ten years ago, running GNATprove on non-trivial code was a research project. Today, with SPARK 2014+, modern SMT solvers (CVC5, Z3 v4.x), proof-by-default patterns, and ghost-code idioms that engineers can learn, the Bronze and Silver levels of the SPARK ladder are achievable in weeks of part-time effort for a small module and in months for a substantial codebase. The productivity numbers from iFACTS and Tokeneer were once curiosities; now they are reproducible.

The EU Cyber Resilience Act (2024), US secure-by-design mandates, and growing regulatory pressure on software liability all make languages with stronger static guarantees — Ada, SPARK, Rust — significantly more attractive for long-lifetime embedded devices. For the first time since the 1990s, the US federal government is actively recommending Ada again.

SPARK is not fashionable. It is, however, the kind of technology that keeps getting asked for when the stakes go up. If you are building software that cannot fail, you should know SPARK exists, and you should know what it can prove. Forty years of deployed SPARK code have been quietly accumulating evidence that formal verification is practical. The rest of the industry is finally catching up to the conversation.
