The Control Flow Graph and Basic Blocks

The control flow graph (CFG) is the foundational data structure upon which all global scheduling techniques are built. It is a directed graph G = (V, E, entry, exit) where V is a set of basic blocks, E is a set of directed edges representing transfers of control, and entry/exit are the unique entry and exit nodes.

Identifying Basic Blocks

A basic block is a maximal sequence of consecutive instructions such that control flow enters only at the first instruction (the leader), leaves only at the last, and no internal instruction is a branch or branch target.

ALGORITHM: IdentifyBasicBlocks(instructions[1..n])
Input:  A sequence of instructions instructions[1..n]
Output: A set of basic blocks B

1.  leaders ← {1}                     // First instruction is always a leader
2.  FOR i ← 1 TO n DO
3.      IF instructions[i] is a branch THEN
4.          leaders ← leaders ∪ {target(instructions[i])}
5.          IF i + 1 ≤ n THEN
6.              leaders ← leaders ∪ {i + 1}
7.          END IF
8.      END IF
9.  END FOR
10. Sort leaders: l₁ < l₂ < ... < lₖ
11. FOR j ← 1 TO k DO
12.     block_j ← instructions[lⱼ .. lⱼ₊₁ - 1]
13.     B ← B ∪ {block_j}
14. END FOR
15. RETURN B

Dominators and Post-Dominators

A node d dominates node n if every path from the entry to n passes through d. The dominator tree is rooted at the entry node; each node’s parent is its immediate dominator. The classic algorithm is due to Lengauer and Tarjan (1979), running in nearly O(n·α(n)) time.

Post-dominators are the reverse: node p post-dominates node n if every path from n to exit passes through p. Post-dominators are computed by running the dominator algorithm on the reverse CFG. Together, dominators and post-dominators yield control dependences: node B is control-dependent on node A if A is a decision point and B executes (or not) depending on the direction taken at A.

Back Edges and Natural Loops

An edge (N → H) is a back edge if H dominates N. The node H is the loop header. The natural loop is {H} ∪ {m : m can reach N without going through H}. Two natural loops are either disjoint, one nested within the other, or they share the same header (merged). This nesting induces the loop nest tree.

ALGORITHM: FindNaturalLoop(H, N)
Input:  Header node H, tail node N (where N → H is a back edge)
Output: Set of nodes in the natural loop

1.  loop ← {H}
2.  stack ← empty
3.  IF N ∉ loop THEN loop ← loop ∪ {N}; PUSH N
4.  WHILE stack is not empty DO
5.      m ← POP(stack)
6.      FOR each predecessor p of m DO
7.          IF p ∉ loop THEN loop ← loop ∪ {p}; PUSH p
8.      END FOR
9.  END WHILE
10. RETURN loop

Loop depth determines execution frequency estimates. A block at depth d in a loop nest with trip counts T₁, T₂, ..., T₅ has estimated execution frequency T₁ × T₂ × ... × T₅ relative to the function entry. For trace selection, blocks with higher loop depth are weighted more heavily.

Fisher’s Greedy Trace Selection

Core idea. Start at the most-frequently-executed block and greedily extend the trace in both directions by always following the most-probable successor (forward) or predecessor (backward). Extension stops when a block is already part of another trace, a loop back-edge is encountered, a maximum trace length is reached, or profitability drops below threshold.

ALGORITHM: GreedyTraceSelection(CFG, weights)
Input:  Control flow graph with edge weights
Output: A trace T = (B₁, B₂, ..., Bₖ)

1.  B_seed ← basic block with highest execution count
2.  T ← (B_seed)

    // Extend forward
3.  B_cur ← B_seed
4.  WHILE B_cur has successors:
5.      B_next ← successor with highest edge weight
6.      IF B_next in a trace OR unprofitable: BREAK
7.      Append B_next to T
8.      B_cur ← B_next

    // Extend backward
9.  B_cur ← B_seed
10. WHILE B_cur has predecessors:
11.     B_prev ← predecessor with highest edge weight
12.     IF B_prev in a trace OR unprofitable: BREAK
13.     Prepend B_prev to T
14.     B_cur ← B_prev

15. RETURN T

This algorithm runs in O(n) time where n is the number of blocks. It is not optimal — it can make locally greedy choices that produce suboptimal global coverage — but in practice it works well enough that it remained the standard approach for decades.

The Outer Loop

After the first trace is selected and scheduled, the algorithm marks those blocks as scheduled and repeats on the remaining blocks (which now include any compensation blocks from bookkeeping). The hottest trace is scheduled first because it accounts for the most dynamic instructions. Later traces are more constrained by compensation code from earlier traces.

Superblock Formation

The superblock (Hwu, Mahlke, Chen, 1992–93) is a trace with a critical restriction: single entry, multiple exits. No execution can join the superblock from the side. This eliminates above-the-join compensation code entirely.

To form a superblock from a trace that has side entries, the compiler performs tail duplication: it copies the portion of the trace below each side entry, creating a separate code path for the side-entering execution.

ALGORITHM: FormSuperblock(trace T)
Input:  A trace T with possible side entries
Output: A superblock S (single entry, multiple exits)

1.  S ← T
2.  FOR each side entry E in T (except the top):
3.      Let T_tail = portion of T from E to the end
4.      Create duplicate T_tail' for the side-entering path
5.      Redirect the side-entry edge to T_tail'
6.      Remove the side entry from S
7.  END FOR
8.  RETURN S

Tail duplication increases code size but in a controlled, predictable way — proportional to the number of side entries, not exponential. The scheduling freedom gained typically outweighs the code size cost.

Hyperblock Formation

The hyperblock (Mahlke, 1992) extends the superblock by incorporating predicated execution. Instead of executing separate paths and inserting compensation code, the hardware conditionally executes instructions based on predicate registers.

If-conversion transforms control flow into data flow:

BEFORE (branching):             AFTER (predicated):

    if (cond) {                     p1 ← (cond)
        x = a + b;                  (p1)  x = a + b
    } else {                        (!p1) x = c - d
        x = c - d;
    }

Both paths execute; the predicate register selects which results are committed. This eliminates branches entirely within the hyperblock, removing a major source of scheduling constraints. Hyperblocks are especially effective on architectures with hardware predication support (Itanium, ARM in conditional-execution mode).

Trace Compaction — List Scheduling

The core scheduling engine. Once a trace (or superblock/hyperblock) is selected, the compiler builds a data-dependency DAG and schedules it via list scheduling — the standard heuristic for packing operations into VLIW instruction words.

Building the Dependency DAG

The DAG has nodes representing individual operations and directed edges representing dependencies. Three types of dependencies are tracked:

TypeAbbreviationDescriptionCan be renamed?
True dependenceRAW (Read After Write)Operation B reads a value that A writes. Genuine data flow.No
Anti-dependenceWAR (Write After Read)Operation B writes a register that A reads. Name dependency from register reuse.Yes
Output dependenceWAW (Write After Write)Both A and B write to the same register. Name dependency.Yes

The DAG spans the entire trace, crossing basic block boundaries. An operation in block B₃ that reads a register written by an operation in block B₁ has a true dependency edge, even though they are separated by two branch instructions.

The List Scheduling Algorithm

ALGORITHM: ListSchedule(DAG, machine)
Input:  Dependency DAG, machine resource model
Output: Schedule mapping operations to cycles and functional units

Phase 1: Compute priorities (reverse topological order)
1.  FOR each node n in reverse topological order DO
2.      priority[n] ← latency(n) + max(priority[s] + edge_lat(n,s))
3.                       for all successors s of n
4.  END FOR

Phase 2: Schedule cycle by cycle
5.  cycle ← 0
6.  ready ← {nodes with no predecessors}
7.  WHILE unscheduled nodes remain DO
8.      FOR each n in ready, sorted by priority descending DO
9.          IF resource_available(n, cycle, machine) THEN
10.             schedule[n] ← cycle
11.             reserve_resources(n, cycle, machine)
12.         END IF
13.     END FOR
14.     cycle ← cycle + 1
15.     Update ready list: add nodes whose predecessors are
        all scheduled and whose latencies have elapsed
16. END WHILE
17. RETURN schedule

Priority Functions

Priority FunctionDescriptionWhen to use
Critical Path Length (CP)Longest path from node to any DAG exit. Ensures critical-path operations are scheduled first.Default, most widely used
Resource PressurePrefer operations using the scarcest functional unit. Prevents resource starvation.Tiebreaker when CP ties
Mobility / Slacklatest_start(n) − earliest_start(n). Zero-slack operations are on the critical path.Identifying schedule-critical operations

Worked Example

Consider a basic block on a 2-issue machine with unit A (ALU, 1-cycle latency) and unit M (multiply/load, 3-cycle latency):

Instructions:
  I1:  r1 ← load [r10]        // M unit, latency 3
  I2:  r2 ← load [r11]        // M unit, latency 3
  I3:  r3 ← r1 + r2           // A unit, latency 1 (depends I1, I2)
  I4:  r4 ← r1 * r2           // M unit, latency 3 (depends I1, I2)
  I5:  r5 ← r3 + r4           // A unit, latency 1 (depends I3, I4)
  I6:  store r5 → [r12]       // M unit, latency 1 (depends I5)

Critical path: I1 → I4 → I5 → I6 = 3 + 3 + 1 + 1 = 8 cycles

Schedule:
  Cycle | Unit A      | Unit M
  ------+-------------+-------------
    0   |             | I1: load
    1   |             | I2: load
    2   |             |
    3   | I3: r1+r2   | I4: r1*r2     ← I1,I2 results ready
    4   |             |
    5   |             |
    6   | I5: r3+r4   |               ← I4 result ready
    7   |             | I6: store
  ------+-------------+-------------
  Total: 8 cycles (matches critical path bound)

Bookkeeping — Compensation Code

After trace compaction, some operations will have been moved from their original basic blocks to different positions. If an operation crossed a control-flow boundary, the program semantics may be violated on the off-trace paths. Bookkeeping restores correctness.

Above-the-Join Compensation

When an operation originally in block C is moved upward past a join point into block B (on-trace), it will no longer execute when control arrives at C via the off-trace predecessor A. Fix: insert a copy of the operation on the edge from A to C.

Below-the-Branch Compensation

When an operation originally in block B is moved downward past a branch into block C (on-trace), it will no longer execute when control goes from B to the off-trace successor D. Fix: insert a copy of the operation on the edge from B to D.

Code-size impact. Freudenberger and Ruttenberg (ACM TOPLAS, 1994) studied “Avoidance and Suppression of Compensation Code in a Trace Scheduling Compiler” and showed two techniques could reduce compensation code by 30–50%:

  • Avoidance — restrict code motion to cases where no compensation is needed (e.g., moves within the same basic block).
  • Suppression — use global dataflow analysis to identify when a compensation copy is redundant (the value is dead on the off-trace path).

Speculative Code Motion

Moving instructions above branches means potentially executing them in a context where they would not have been executed in the original program. This is speculative execution at the compiler level.

Safe vs. Unsafe Speculation

CategoryDescriptionExample
Safe speculationThe speculated operation has no visible side effects if the branch goes the other way. Its result is simply unused.Integer ALU operations that write to a dead register on the off-trace path
Unsafe speculationThe speculated operation could fault or produce visible side effects. Requires hardware support (e.g., Itanium’s ld.s speculative load with deferred exception handling).Load from an address that might be invalid on the off-trace path

Modern VLIW architectures provide speculative load instructions that suppress exceptions until the result is actually used, making memory speculation safe. Without such hardware support, the compiler can only speculate on operations that are provably exception-free.

Software Pipelining — Modulo Scheduling

The loop-domain complement to trace scheduling. While trace scheduling targets irregular, acyclic code (traces through the CFG), software pipelining targets regular loop bodies — overlapping successive iterations of a loop to fill the VLIW instruction word.

Software pipelining was formalized by B. Ramakrishna Rau (Rau, 1994) as modulo scheduling. The key concepts:

ConceptDefinition
Initiation Interval (II)The number of cycles between the start of successive loop iterations. Determines throughput: one iteration starts every II cycles.
Minimum IImax(resource-bound II, recurrence-bound II). Resource-bound = ceil(total operations / available units). Recurrence-bound = longest loop-carried dependency chain.
Modulo resource tableA table of II rows representing the repeating schedule pattern. Each row tracks which functional units are occupied.
KernelThe steady-state portion of the pipelined loop where all stages overlap.
Prologue / EpilogueStartup/shutdown code to fill/drain the pipeline.

The Scheduling-Allocation Phase-Ordering Problem

Schedule first or allocate registers first? This is one of the most persistent design challenges in compiler engineering. Instruction scheduling wants to spread operations apart in time to expose ILP, which requires more registers to hold intermediate values. Register allocation wants to minimize live values at any point, which constrains how far apart operations can be scheduled. The two phases have conflicting objectives.

Goodman and Hsu (1988) demonstrated that an integrated approach — combining scheduling and allocation into a single pass that tracks register pressure while scheduling — can produce better results than either phase ordering. Their method keeps track of available registers and chooses the appropriate scheduling technique (latency-reducing or register-minimizing) based on current pressure.

In practice, most production compilers schedule in two passes:

  1. Pre-register-allocation scheduling — aggressive ILP extraction, may increase register pressure
  2. Register allocation — graph-coloring or linear-scan, may introduce spill code
  3. Post-register-allocation scheduling — clean up spill code scheduling, less aggressive

NP-Hardness of Optimal Scheduling

Optimal VLIW instruction scheduling is NP-hard. Even for a single basic block on a machine with two or more functional units and non-unit latencies, finding the minimum-length schedule is NP-complete (reduction from multiprocessor scheduling). For traces spanning multiple blocks with compensation code, the problem is strictly harder.

This is why list scheduling is a heuristic, not an algorithm that finds the optimal schedule. In practice, list scheduling with a good priority function (critical-path length) produces schedules within 5–10% of optimal for most code. The gap between the heuristic schedule and the optimal schedule is called the scheduling gap.

Theoretical bounds on the schedule length:

  • Critical path bound: No schedule can be shorter than the longest path through the dependency DAG.
  • Resource bound: No schedule can be shorter than ceil(total_operations / issue_width).
  • Lower bound: max(critical_path_bound, resource_bound).
  • List scheduling guarantee: Within 2× optimal for unit-latency operations on identical machines (Graham’s bound).
