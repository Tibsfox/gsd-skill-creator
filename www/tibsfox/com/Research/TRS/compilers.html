GCC — The GNU Compiler Collection

GCC implements instruction scheduling in two passes, both controlled by the -fschedule-insns and -fschedule-insns2 flags. The first pass runs before register allocation; the second runs after. The scheduling engine is a list-scheduling framework that uses machine-specific descriptions to model functional units, latencies, and pipeline hazards.

Key Scheduling Infrastructure

ComponentDescription
-fschedule-insnsPre-register-allocation scheduling. Aggressive ILP extraction; may increase register pressure.
-fschedule-insns2Post-register-allocation scheduling. Cleans up spill code, schedules around register constraints. This is the pass most users encounter.
Superblock formationGCC forms superblocks for scheduling regions that span multiple basic blocks. Tail duplication eliminates side entries.
If-conversionGCC performs if-conversion to transform short conditional branches into predicated or conditional-move sequences, enabling hyperblock-like scheduling on targets that support conditional execution (ARM, Itanium).
Machine descriptions (.md files)Each target architecture defines its functional units, instruction latencies, pipeline stages, and resource constraints in a .md (machine description) file. The scheduler uses these to model resource availability.

Profile-Guided Optimization in GCC

GCC supports PGO via a two-phase process:

  1. -fprofile-generate — Instruments the program to record branch frequencies, function call counts, and value profiles during execution.
  2. -fprofile-use — Recompiles using the collected profile data to guide:
    • Block reordering (hot blocks first for better I-cache behavior)
    • Function inlining decisions (inline hot callees)
    • Superblock formation and trace selection (schedule the hot path)
    • Loop unrolling decisions (unroll hot loops more aggressively)
    • Branch hint generation (for targets that support them)

This two-phase process descends directly from Fisher’s profile-driven compilation idea (1981). PGO typically improves GCC-compiled code by 10–20% on benchmarks with representative training inputs.

LLVM — The LLVM Compiler Infrastructure

LLVM’s scheduling infrastructure is more modular than GCC’s, with multiple scheduling passes and a pluggable framework for machine-specific scheduling strategies.

Key Scheduling Components

ComponentDescription
SelectionDAGISelInstruction selection via DAG pattern matching. Converts LLVM IR to target-specific machine instructions organized as a DAG. Initial scheduling happens here.
MachineSchedulerThe primary post-instruction-selection scheduler. Uses a bidirectional list-scheduling approach (can schedule top-down, bottom-up, or both). Tracks register pressure during scheduling to avoid excessive spilling.
ScheduleDAG frameworkThe generic DAG-based scheduling framework. Builds the dependency DAG (SUnit nodes, SDep edges with latency information), computes critical paths, and drives the scheduling heuristic.
PostRASchedulerPost-register-allocation scheduling pass. Handles scheduling of spill/reload code and final instruction ordering.
TableGen (.td files)LLVM uses TableGen to define machine descriptions declaratively. Each target describes its instruction set, functional units (via SchedModel), latencies, and pipeline resources. The scheduler reads these descriptions at compile time.

LLVM PGO

LLVM supports PGO through instrumentation-based profiling (-fprofile-instr-generate / -fprofile-instr-use) and sampling-based profiling (via llvm-profdata and AutoFDO). PGO instrumentation attaches precise branch-weight and indirect-call target metadata directly to IR instructions, driving block scheduling, hot/cold splitting, branch hint reordering, and inlining decisions.

The IMPACT Compiler (UIUC)

The research compiler that defined modern ILP techniques. Built by Wen-mei Hwu’s IMPACT group at the University of Illinois at Urbana-Champaign, IMPACT was the vehicle for the superblock and hyperblock research that refined Fisher’s trace scheduling into practical compiler technology.

Key Contributions from IMPACT

  • Superblock formation — trace selection + tail duplication to create single-entry, multiple-exit scheduling regions (Hwu, Mahlke, Chen, 1993)
  • Hyperblock formation — if-conversion + predication to merge multiple paths into a single predicated stream (Mahlke, 1992)
  • Modulo scheduling for superblocks — extending software pipelining beyond simple loops to complex loop bodies with internal control flow
  • Speculative execution support — compiler techniques for safe and unsafe speculation, deferred exception handling
  • Profile-guided compilation — extensive use of profiling data to guide all optimization decisions

IMPACT was not a production compiler — it targeted research architectures (HPL-PD, the Itanium-like experimental machine description). But its algorithms were adopted by production compilers: GCC’s superblock formation, LLVM’s if-conversion, and the Intel compiler’s Itanium scheduling all trace their lineage to IMPACT research.

Trimaran (1999)

Trimaran was an open-source compiler infrastructure for ILP research, developed as a successor to the IMPACT compiler by a consortium including UIUC, NYU, and HP Labs. It provided a complete compilation and simulation environment for EPIC/VLIW research.

Architecture

ComponentRole
ElcorThe back-end compiler. Performs instruction scheduling (list scheduling, superblock scheduling, modulo scheduling), register allocation, and code emission for the HPL-PD machine description.
HPL-PDThe machine description language developed at HP Labs for parameterized VLIW/EPIC machine models. Specifies functional units, latencies, register files, predication support, and speculation support.
SimuA cycle-accurate simulator for HPL-PD machines. Allows evaluating the performance of compiled code without building hardware.
OpenIMPACTThe front-end and middle-end (derived from IMPACT). Handles parsing, optimization, superblock/hyperblock formation, and profile-guided decisions.

Trimaran was widely used in academic research throughout the 2000s. Many Ph.D. dissertations on VLIW scheduling, predication, and compiler optimization used Trimaran as their experimental platform. The project is no longer actively maintained but remains a valuable reference for ILP research.

Open64 / Pro64

Open64 (originally Pro64) was SGI’s and Intel’s open-source compiler for IA-64 (Itanium), released under the GPL. It was one of the most sophisticated production compilers ever built for a VLIW-like target.

Key Features for VLIW/EPIC

  • Global instruction scheduling with superblock and hyperblock support
  • Software pipelining (modulo scheduling) with rotating register file support for Itanium
  • If-conversion using Itanium’s 64 predicate registers
  • Speculative code motion using Itanium’s ld.s (speculative load) and ld.a/chk.a (advanced load/check) instructions
  • Inter-procedural analysis and optimization (IPA) for whole-program scheduling
  • Aggressive PGO with feedback-directed optimization at every stage

Open64 demonstrated what was possible with a fully optimizing VLIW/EPIC compiler, but also exposed the limits: even with aggressive optimization, Itanium code was often full of NOPs on irregular workloads. The compiler could not overcome the fundamental information deficit — runtime behavior that was unknowable at compile time.

Profile-Guided Optimization (PGO / FDO)

The chicken-and-egg problem. Trace scheduling needs profile data to select the right traces. Profile data requires running the program. But you need to compile the program before you can run it. The solution: compile twice.

The PGO Pipeline

  1. Instrumented build. Compile with profiling instrumentation. The resulting binary records branch frequencies, function call counts, and value profiles during execution.
  2. Training runs. Execute the instrumented binary on representative inputs. The profile is only as good as the inputs — if training inputs are unrepresentative, the selected traces will be suboptimal.
  3. Profile analysis. Process the raw profiling data into a profile database (edge weights, block frequencies, call frequencies).
  4. Optimized build. Recompile using the profile data. The compiler uses profile-weighted trace selection, profile-guided inlining, profile-guided block placement, and profile-guided loop optimization.

The Representative Inputs Problem

PGO’s effectiveness depends entirely on the quality of the training inputs. If the training workload differs from the production workload, the compiler optimizes for the wrong traces. This is the compiler analog of the AI orchestration challenge of optimizing for the wrong task distribution.

Solutions include:

  • Multiple training runs with diverse inputs, merging profiles
  • AutoFDO (Automatic Feedback-Directed Optimization) — uses hardware performance counters and sampling to collect profiles from production binaries without instrumentation overhead
  • BOLT (Binary Optimization and Layout Tool, developed at Facebook/Meta) — post-link-time optimization using production profiles to reorder code, improving I-cache behavior

Machine Descriptions — How Compilers Model VLIW Machines

The compiler’s scheduling algorithm needs a formal model of the target machine. This machine description specifies:

ElementDescriptionExample (TI C6x)
Functional unitsTypes and counts of execution units2× .L (logic), 2× .S (shift), 2× .M (multiply), 2× .D (data/load-store)
Instruction latenciesCycles from issue to result availabilityInteger add: 1, multiply: 2, load: 5
Reservation tablesWhich functional unit(s) an instruction occupies and for how many cyclesMultiply occupies .M for 1 cycle, result available after 2
Issue widthMaximum operations per instruction word8 operations per 256-bit fetch packet
Register fileNumber and types of registers32 general-purpose registers per side (A and B banks)
Pipeline constraintsCross-path restrictions, bank conflicts, branch delay slots1 cross-path read per cycle per side

In GCC, machine descriptions are encoded in .md files using a Lisp-like syntax. In LLVM, they are encoded in .td (TableGen) files with a declarative syntax. Both systems generate scheduling tables at compiler build time that the runtime scheduler consults.

Trace Scheduling Ideas That Survived into Modern Compilers

The lasting legacy. Even though pure VLIW is niche, trace scheduling concepts are in every major compiler. The ideas Fisher pioneered became the foundation of modern compiler scheduling, regardless of target architecture.

ConceptOriginModern Compiler Usage
Profile-guided optimizationFisher, 1981GCC (-fprofile-use), LLVM (PGO, AutoFDO), ICC, MSVC. Standard practice for performance-critical software.
Superblock formationHwu/Mahlke, 1992GCC superblock pass, LLVM block placement. Used for both scheduling and code layout optimization.
If-conversionMahlke hyperblock, 1992GCC if-conversion pass, LLVM SimplifyCFG + select lowering. Transforms branches to conditional moves/selects on all targets.
Software pipeliningRau, 1994GCC modulo scheduling (for IA-64, PowerPC, SPARC), LLVM MachinePipeliner. Essential for VLIW DSP targets.
Global code motionFisher trace scheduling, 1981GCC and LLVM move instructions across blocks in their scheduling passes. Superblock scheduling is a restricted form of trace scheduling.
Speculative code motionIMPACT/Itanium researchGCC and LLVM support speculative scheduling on targets with hardware support (Itanium ld.s, ARM conditional execution).
Machine description frameworkFisher/ELI-512 machine modelGCC .md files, LLVM TableGen .td files. Every target architecture has a formal machine description.
List scheduling with critical-path priorityBulldog compiler (Ellis, 1985)The default scheduling algorithm in both GCC and LLVM. The fundamental workhorse of instruction scheduling.

The irony of trace scheduling’s legacy is that its most enduring contributions are not in VLIW compilation per se, but in the general-purpose compiler infrastructure that it inspired. PGO, superblock formation, if-conversion, and list scheduling are used every day by compilers targeting x86, ARM, RISC-V, and other superscalar architectures — architectures that use hardware, not software, for dynamic scheduling. Fisher’s ideas live on in every compiler, even on machines he did not design them for.
