Rosetta — Scheduling Paradigms

Superscalar vs VLIW vs EPIC vs GPU — Who Finds Parallelism, and Where
Overview History Algorithms VLIW AI Crossover Compilers Rosetta Learn
Every computer executes instructions. The question is: who decides which instructions run in parallel? Hardware (superscalar), software (VLIW), both (EPIC), or the data itself (dataflow)?

The Scheduling Paradigm Space

  1. Sequential — One instruction at a time, in order. The original von Neumann model. No parallelism. Every instruction completes before the next begins. Simple, correct, slow.
  2. Pipelined — Overlap stages of different instructions. While one instruction executes, the next is decoded, and the one after that is fetched. Throughput increases without changing the ISA. Hazards (data, control, structural) require stalls or forwarding.
  3. Superscalar In-Order — Multiple instructions issued per cycle, in program order. Hardware detects independent instructions and dispatches them to parallel functional units. Limited by in-order constraint — a stall blocks everything behind it.
  4. Superscalar Out-of-Order — Reorder buffer, register renaming, dynamic scheduling. Hardware looks ahead in the instruction stream, finds independent operations, executes them out of order, and retires them in order. The dominant paradigm for general-purpose CPUs since the Pentium Pro (1995). Powerful but power-hungry — the scheduling hardware itself consumes significant die area and energy.
  5. VLIW (Very Long Instruction Word) — The compiler finds ILP at compile time and packs multiple operations into a single wide instruction word. Hardware is simple — no reorder buffer, no register renaming, no dependency checking. The compiler does all the work. Binary compatibility is sacrificed: a VLIW binary is tied to the exact number and type of functional units.
  6. EPIC (Explicitly Parallel Instruction Computing) — VLIW extended with predication (conditional execution without branches), speculative loads (loads that may fault without trapping), and register rotation (for software pipelining). IA-64/Itanium is the canonical example. More flexible than pure VLIW, but the compiler burden is enormous.
  7. Dataflow — Execution driven by data availability, not program counter. An instruction fires when all its operands are ready. No sequential control flow. Explored in the 1970s–80s (MIT Tagged-Token, Manchester Dataflow Machine). Never reached commercial success as a general-purpose model, but the ideas survive in out-of-order execution (which is dataflow inside a sequential wrapper) and in modern accelerators.
  8. Spatial (CGRA, Systolic Arrays) — Operations mapped to physical locations on a chip. Data flows through a grid of processing elements. Google’s TPU uses systolic arrays for matrix multiplication. CGRAs (Coarse-Grained Reconfigurable Arrays) map computation graphs to hardware. The scheduling problem becomes a placement-and-routing problem.

The Compiler Scheduling Spectrum

From no scheduling to whole-program scheduling, each level trades compile time and code size for more ILP:

  1. No scheduling — instructions emitted in source order. The baseline.
  2. Basic block scheduling — reorder instructions within a basic block (straight-line code between branches). Safe, simple, limited — basic blocks average 4–7 instructions.
  3. Superblock scheduling — extended basic block with a single entry but multiple exits (side exits). No join points. Easier bookkeeping than full trace scheduling. Hwu/Mahlke/Chen, 1992.
  4. Hyperblock scheduling — superblock + if-conversion (predication). Branches become predicated instructions, eliminating control flow within the block. Requires hardware predication support.
  5. Trace scheduling — select the most-likely path through the CFG (a trace), schedule it as a single unit, then insert compensation code for all off-trace paths. Fisher, 1981. The breakthrough.
  6. Region-based scheduling — partition the CFG into larger regions (e.g., loops, function subgraphs) and schedule within each region. Trimaran, IMPACT.
  7. Whole-program scheduling — interprocedural scheduling. Inline functions, then schedule across the combined CFG. Maximum ILP, maximum compile time, maximum code bloat.

The Same Loop: Three Paradigms

Consider a simple loop: for (i = 0; i < N; i++) A[i] = B[i] + C[i];

Paradigm How it schedules
Superscalar (OOO) Hardware sees loads of B[i] and C[i] are independent, issues them in parallel. The add waits for both loads. Store of A[i] waits for the add. Next iteration’s loads can overlap with this iteration’s add/store via the reorder buffer. The hardware discovers this dynamically, every cycle.
VLIW + Trace Scheduling Compiler unrolls the loop (say 4x), then packs 4 loads of B, 4 loads of C, 4 adds, and 4 stores into wide instruction words, interleaving them to hide load latency. Each instruction word might contain: load B[i], load C[i+1], add for i-1, store for i-2 — all in one cycle. The schedule is fixed at compile time.
GPU (SIMT/Warp) Launch N threads. Each thread computes one element: load B[i], load C[i], add, store A[i]. All threads execute the same instructions. The hardware manages thousands of threads, hiding memory latency by switching warps. Parallelism is in the data, not the instructions.
