Closures Over State

;; Common Lisp
(defun make-counter ()
  (let ((count 0))
    (lambda () (incf count))))

;; Scheme
(define (make-counter)
  (let ((count 0))
    (lambda () (set! count (+ count 1)) count)))

;; Clojure
(defn make-counter []
  (let [count (atom 0)]
    #(swap! count inc)))

;; Python
def make_counter():
    count = 0
    def inner():
        nonlocal count
        count += 1
        return count
    return inner

;; JavaScript
const makeCounter = () => {
  let count = 0;
  return () => ++count;
};

;; Rust
fn make_counter() -> impl FnMut() -> i32 {
  let mut count = 0;
  move || { count += 1; count }
}

The Influence Map

If you removed Lisp from history, you would lose:

  • Every garbage-collected runtime — JVM, .NET, V8, CPython, Go runtime
  • Every REPL culture — Smalltalk, Python, Ruby, Julia, JavaScript debuggers
  • Every modern functional language — Haskell, OCaml, F#, Elm (via ML via the Lisp tradition of typed lambda calculus)
  • Every AI/ML computational graph framework — TensorFlow, PyTorch, JAX, TVM
  • Every serious macro system — Rust procedural macros, Julia macros, Elixir macros, Nim macros
  • Emacs, org-mode, Magit — and everything built on top of them
  • The entire Clojure ecosystem — Nubank, Metabase, Circle CI, Roam, Logseq
  • ITA Software / Google Flights
  • GNU Emacs and, by extension, the GNU Project, the GPL, Linux, GCC, Git, Bash, Firefox, Android (yes, all of that traces back to Stallman’s experience with the Symbolics split — see Lisp Machines)

The Hacker Culture Thread — Archie, Eggdrop, and the Joy of Building

The Lisp tradition never lived in isolation. It was one thread in a larger hacker culture that ran from the MIT AI Lab through McGill, Berkeley, Carnegie Mellon, and out onto the early Internet. A culture where people built things because building things was interesting, and then gave them away because giving them away was the whole point.

The following are not strictly Lisp projects. But they share Lisp’s intellectual DNA — the belief that you should be able to write your own tools, that the tools should be inspectable and modifiable, and that the cleverest solution is one you can explain to a friend over pizza at 3 AM.

Archie — The First Internet Search Engine (1990)

In 1989, Alan Emtage, a graduate student at McGill University in Montreal, got tired of manually keeping track of which FTP servers held which files. The early Internet had already sprouted thousands of anonymous FTP sites scattered across universities and research labs, each with its own idiosyncratic directory structure and its own collection of software, papers, source code, and data. If you knew which server held the file you wanted, you could download it. If you didn’t know, you were stuck reading mailing list posts and hoping someone else had already found it.

Emtage, Bill Heelan, and Peter J. Deutsch wrote a script that periodically logged into known FTP sites, walked their directory listings, and built a searchable database of every file on every known FTP server. They called it Archie — a truncation of “archive” that happened to not be the name of a comic book character. (Veronica, Jughead, and Archie’s other namesakes would follow.)

Archie was the first Internet search engine. It predated the World Wide Web by several years. It predated Gopher. It predated WAIS. It predated Yahoo by five years and Google by eight. When Emtage wrote Archie, the Internet had no idea it needed a search engine, and then Archie appeared and everyone realized they had needed one all along.

Archie ran at archie.mcgill.ca and was immediately flooded with queries from around the world. Alan Emtage and Peter Deutsch founded Bunyip Information Systems to commercialize it. Mirrors spread to every continent. By 1993, Archie was handling millions of queries a month. By 1995 the web had mostly replaced it, and Archie faded out of daily use, but for four or five years it was the entire searchable layer of the Internet.

It was also completely free, completely open, completely collaborative, and entirely built by graduate students at a university. This is the culture. You see a problem, you write code, you give it away, the world gets better. There was no business plan. There was no monetization strategy. There was a graduate student who wanted to find files.

Archie was written in C with shell scripts, not Lisp. The connection to this project is cultural, not technical. Archie is what happens when you let hackers on the Internet solve their own problems without asking permission. It is the same culture that produced the MIT AI Lab, the Lisp Machine, GNU Emacs, and eventually the whole free software movement. McCarthy, Stallman, Emtage, Hickey — different generations, different languages, same impulse: build the tool you need, share it, move on to the next thing.

The original comp.archives announcement post sits in the Google Groups archives as a preserved artifact of the moment before the web existed, when the Internet was small enough that every new tool was noticed by everyone at once. Reading those old threads is a visit to a world we no longer inhabit.

Eggdrop and the IRC Bot Culture (1993)

Internet Relay Chat, invented by Jarkko Oikarinen in 1988 at the University of Oulu in Finland, became the default real-time chat system of the 1990s Internet. IRC was a simple protocol — so simple that you could implement a client in an afternoon, and so simple that you could implement a bot in a weekend.

Eggdrop, written by Robey Pointer in 1993, was the first widely-used IRC bot designed for channel management and operator assistance. It was written in C with a Tcl scripting layer, which let users load custom behaviors without recompiling. Eggdrop could:

  • Hold operator status in a channel when no humans were around
  • Kick or ban users who violated channel rules
  • Greet new users, offer help, answer FAQs
  • Log conversations for later review
  • Share user lists across a botnet of cooperating bots
  • Run user-written Tcl scripts that could do literally anything

The Tcl scripting layer is the important part for our purposes. Eggdrop wasn’t homoiconic (Tcl is closer than most languages but not a Lisp) but it established the pattern: a small, fast core written in C, with a dynamic extension language layered on top where all the interesting work actually happens. This is the same pattern as GNU Emacs (C core + elisp), Vim (C core + VimL), AutoCAD (C++ core + AutoLISP), Blender (C core + Python), and every other extensible application of the 90s and 2000s.

Eggdrop begat a culture. Channel ops on 1990s IRC were often 14-year-olds with too much time on their hands and a passion for writing Tcl scripts to automate their channels. They formed botnets. They shared trivia databases. They wrote plugins that played games, tracked sports scores, fetched weather, monitored RSS feeds (before RSS existed, as custom web scrapers), and held running jokes across multiple channels. Every IRC channel worth visiting had at least one bot, often named something like TriviaBot or ChanServ or WikiBot or BeerBot.

The Lisp connection is direct in some cases: many IRC bots were written in Common Lisp (cl-irc is still maintained), in Scheme, or later in Clojure. The Emacs client ERC is an IRC client written entirely in Emacs Lisp, and running ERC inside Emacs gives you a full Lisp environment sitting next to your chat window — the ultimate bot-writer’s workbench. Many Lisp hackers ran ERC, wrote bots in elisp, and maintained a constant Lisp REPL–IRC–bot pipeline that let them build new features live during channel conversations.

ERC — The Emacs IRC Client

ERC (Emacs IRC Client, later “Emacs Regex Client” retcon) was started in 1997 by Alexander L. Belikoff and is now maintained as part of GNU Emacs core. ERC is written entirely in Emacs Lisp. Because it runs inside Emacs, every ERC buffer is a regular Emacs text buffer, which means you can search it with C-s, kill-ring copy from it, run M-x commands on it, and — crucially — extend it with your own elisp code.

An ERC user who wants to add auto-responses, log filtering, custom notifications, URL previews, nickname highlighting, or integration with org-mode can do so with a handful of lines of elisp and M-x eval-buffer. No compile step. No plugin manifest. No package installation. Just write the function, evaluate it, and it is part of your IRC client now.

This is the Lisp Machine workflow surviving into the IRC era. The bot and the client are the same thing, because the client is a Lisp environment that can load code at runtime, and a bot is just code that runs in response to incoming messages. The distinction between “user” and “bot” is a function of which side is typing. In a true Emacs+ERC setup, it is often unclear which side is human and which is the script, and the human and the script are in fact the same loop.

The Botnet Sub-Culture

The early 2000s saw the rise of botnets that were actually a form of social organization: groups of friends who ran bots on the same channels, shared databases between them, and built collaborative tools that spanned multiple IRC networks. These were not the criminal botnets of malware fame. They were trivia rings, weather-bot coops, sports-score aggregators, and shared programming assistants. The infrastructure was free, the servers were donated, the code was open, and the labor was volunteer.

This is where the hacker ethic met the social Internet. People built things together, not because there was money in it, but because it was fun and because the result was useful to their community. The bots that survived were the ones whose maintainers cared about the channels they ran in. The bots that died were the ones whose maintainers moved on.

The whole thing was held together by the same kind of software craftsmanship you see in the Lisp tradition: small tools, composable, inspectable, extendable, fun to work on. Nobody ever shipped a bot with a version number in the Gartner magic quadrant. Nobody ran a Jira sprint on an IRC bot. The bot worked because the maintainer noticed it broke and fixed it, or because a friend on the channel submitted a patch, or because someone forked it and wrote a better version. This is the culture of building because it is fun.

Make Computer Science Fun Again

“Love my work, hate my job.”

— Every hacker who has ever held down a day job

// The Philosophical Aside //

There is a distinction that every working programmer eventually learns to make. There is the work — the craft, the ideas, the problems you can’t stop thinking about at 2 AM, the satisfaction of a data structure that finally clicks, the joy of writing a macro that makes a hard thing easy. And then there is the job — the Jira tickets, the standups, the quarterly OKRs, the pull request templates, the code review comments that are really status games, the commute (if you still have one), the sprint retrospective that devolves into recrimination, the manager who does not understand what you do but has opinions about how fast you should do it.

Love my work, hate my job. You can love writing Lisp macros and hate writing TPS reports. You can love inventing data structures and hate explaining them to a project manager who wants to know the ETA. You can love the field of computer science and recoil from the industry it has become.

What We Have Lost

Computer science used to be fun. Not because the work was easy — it was never easy — but because the work was, overwhelmingly, a thing people chose to do because they loved it. The MIT AI Lab ran on passion. The Berkeley hackers who wrote BSD wrote it because they wanted to. Alan Emtage wrote Archie because he was tired of finding files by hand. Robey Pointer wrote Eggdrop because his IRC channel needed a bot. Richard Stallman wrote Emacs and fought the Symbolics war and founded GNU because the hacker ethic mattered to him more than a paycheck. McCarthy built Lisp because he wanted to do AI research and there was no language for it. Hickey built Clojure because he wanted a better Lisp for the JVM.

None of these people had a quarterly OKR. None of them had a pull request template. None of them were bottlenecked on a code review from a senior engineer who was themselves bottlenecked on a design review from a principal architect who was themselves bottlenecked on a planning meeting with a product manager. They had an idea. They had a computer. They had time. They built the thing. They gave it away. They moved to the next thing.

Somewhere between 1995 and now, computer science stopped being a craft that hackers did for fun and became an industry that employed them for money. Both things are still true — you can still be a hacker who writes Lisp macros for the joy of it — but the industry has gotten so large and so process-heavy that most working programmers spend most of their working hours not building things they love, but attending meetings, filing tickets, answering Slack, and navigating org charts. The fun is still there, if you know where to find it, but you increasingly have to find it outside the job.

Why Lisp Is Part of the Answer

The Lisp tradition is one of the surviving branches where fun is still the primary currency. The Common Lisp community is small, weird, and uncommercial, and that is a feature. There is no Common Lisp careerism, because there is no Common Lisp career. People write Common Lisp because they love it, or because their specific niche (ITA, Grammarly’s early stack, MAXIMA, Opusmodus) is the one place they can get paid to love it. The Scheme community is even smaller. The Racket community is a research group and a small ecosystem of weird languages. The Clojure community is larger and has real jobs attached to it (Nubank alone employs over a thousand Clojure developers) but the culture is still, audibly, one where people love what they do.

Read any Rich Hickey talk transcript. Listen to any SICP lecture. Flip through On Lisp or PAIP or SICP or The Little Schemer. The through-line in all of these texts is that the writers love what they are doing. They are not producing content under deadline. They are trying to explain a thing they find beautiful. That shows up in the writing, and it shows up in the code, and it shows up in the ecosystem. You can feel the difference between a Lisp codebase written by someone who loves Lisp and a Java codebase written by someone who is clocking in.

The Bot Path, The Emacs Path, The Racket Path

The Lisp tradition offers several escape hatches from the job-that-is-not-the-work:

  • The bot path. Run ERC inside Emacs. Write a bot in elisp. Watch it respond to channel events in real time. Tweak it. Share it with a friend. Nobody is tracking your velocity on this.
  • The Emacs path. Emacs is a Lisp Machine in a terminal window. Every piece of it can be rewritten. Every workflow can be automated. Every frustration can be smoothed over with fifteen lines of elisp. Three months in, you have a text editor that fits your brain better than anything else you have ever used. That is the job.
  • The Racket path. Racket’s #lang system lets you build a new language in a weekend. Matthew Butterick built Pollen for book publishing. Shriram Krishnamurthi built teaching languages. You can build a DSL for your own problem, learn more about programming languages in a week than a semester-long compilers course would teach, and have something useful to show for it at the end.
  • The Clojure path. If you need a day job, Clojure is the Lisp with jobs attached. Nubank, Metabase, Circle CI, Juxt, and the consulting shops hire Clojure developers. The work is often more craftwork-flavored than the same role in Java or Python, because Clojure attracts people who care.
  • The Scheme path. Read SICP. Write a meta-circular evaluator. Write another one with lazy evaluation. Write a third with a type system. Write a fourth that targets the GPU. None of these will make you money, but all of them will teach you more about what programming is than any bootcamp.
  • The research path. The academic computer science world is not paradise either — it has its own politics and its own misery — but the research programming language community is still unusually open to weird ideas and unusually tolerant of beautiful work that does not pay off for years. PLT, the Racket team, the ML community, the type theory folks, the PL semantics crowd — these are places where fun is still the primary currency.

The Small Things

The fun is in the small things. The joy of M-x toggle-debug-on-error popping up a stack trace and letting you poke around. The joy of writing a macro that makes ten lines into one. The joy of a transducer composition that does exactly what you meant. The joy of call/cc doing something you didn’t know was possible. The joy of a DSL that reads like the problem domain. The joy of your IRC bot saying something funny because of a pattern-match you wrote at 1 AM.

These joys are real. They are not nostalgia. They are available right now, today, in 2026, in exactly the same form they were available in 1985 — except you have a laptop that is six orders of magnitude faster than a Symbolics 3600 and the entire Internet’s worth of learning material at your fingertips.

The only thing you need to make computer science fun again is permission to follow what interests you, time to do it, and a willingness to build things that do not appear on any roadmap. The Lisp tradition is a permission slip. Pick it up and use it. Write your own editor. Write your own language. Write your own bot. Give it away. Move on to the next thing. That is what the people who invented this whole field were doing, and it is still what the best work in this field comes from, and nobody is going to give you permission except yourself.

Love the work. The job will sort itself out.
