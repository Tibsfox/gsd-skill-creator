Atoms and Cons Cells

Lisp has two building blocks. That is not an exaggeration.

  1. Atoms — symbols, numbers, strings, characters, the empty list nil, and a few other self-contained values.
  2. Cons cells — pairs of two things. Written (a . b) where a is the car and b is the cdr.

A list is a chain of cons cells where each cdr points to the next cell and the final cdr is nil.

(cons 1 (cons 2 (cons 3 nil)))   ; => (1 2 3)
(1 . (2 . (3 . nil)))            ; same thing, dotted notation
(1 2 3)                          ; same thing, sugar

The list (1 2 3) is not a primitive. It is three cons cells chained together, terminated by nil. You can tell when somebody really understands Lisp: they stop seeing lists and start seeing cons cells.

The primitive functions are cons (build a pair), car (first), cdr (rest, pronounced “cudder,” from “contents of the decrement register” — a 1959 IBM 704 detail that stuck), null?, eq?, and atom?. Given those six primitives plus lambda, quote, and cond, you have a complete programming language. McCarthy proved it in 1960.

S-Expressions and Homoiconicity

An S-expression (for “symbolic expression”) is recursively defined:

sexp ::= atom | (sexp . sexp)

An S-expression is either an atom or a cons cell of two S-expressions. That’s it. Every Lisp program, every Lisp data structure, every Lisp configuration file, every Lisp AST node is an S-expression. There is no distinction between “source code format” and “data format” and “serialization format.” They are all the same thing.

Compare to the modern world. We have source code in Python or JavaScript. ASTs in a completely different format (trees of classes with visitor patterns). Configuration in YAML, JSON, TOML, INI. IPC in Protobuf, MessagePack, Thrift, Cap’n Proto. Query languages in SQL, GraphQL, LINQ. Build systems in Makefile, Bazel, CMake. Templates in Jinja, ERB, Handlebars.

Every one of these is an attempt to reinvent S-expressions badly. Every parser built for each of these is a reinvention of what read does for free in Lisp.

A language is homoiconic if its source code representation is a primary data structure of the language itself. Lisp is the original homoiconic language. Program text = list data = AST. Every time you see a modern framework reach for string templating, code generation scripts, macro preprocessors, plugin systems, annotation processors, reflection, or dynamic bytecode emission, you are watching a non-homoiconic language strain to do what Lisp does as a side effect of its data model.

Quote: The Dark Room of Lisp

The single most important primitive in Lisp is quote. Understanding quote is the difference between programming in Lisp and programming in a language that happens to use parentheses.

When you type (+ 1 2) in a Lisp REPL, the reader parses that text into a data structure: a list of three elements, the symbol +, the number 1, the number 2. The evaluator then looks at that list, sees that + is a function, evaluates the arguments, and calls the function. The result is 3.

But what if you want the list itself, not the result of evaluating it? You quote it:

(quote (+ 1 2))    ; => (+ 1 2) — a three-element list
'(+ 1 2)           ; same, with reader sugar

Quote means “don’t evaluate this — give me the data structure the reader built.” Everything to the right of the quote is frozen as data.

Quote is the wormhole between the two worlds Lisp lives in: code (what the evaluator sees) and data (what cons cells actually contain). Quote takes you from code to data. eval takes you back.

(eval '(+ 1 2))    ; => 3

(let ((op '+)
      (args '(1 2)))
  (eval (cons op args)))   ; => 3

You can build a program as data and then run it. This is the ground floor of metaprogramming.

The Reader-Eval-Print Loop

The canonical Lisp pipeline has three stages:

text  →  [READER]  →  data  →  [EVALUATOR]  →  value  →  [PRINTER]  →  text

The reader is a parser that consumes characters and produces cons cells, symbols, and numbers. Exposed as read. The evaluator takes data from the reader and reduces it to a value. Exposed as eval. The printer takes a value and produces text. Exposed as print.

A REPL is literally:

(loop (print (eval (read))))

Read, Eval, Print, Loop. The acronym is the implementation. Every piece is an ordinary function you can call, override, replace, or compose.

Lambda: The Universal Function Constructor

Alonzo Church invented the lambda calculus in the 1930s as a formal system for computation. Lisp embeds the lambda calculus almost directly.

(lambda (x) (* x x))     ; a function that squares its argument
((lambda (x) (* x x)) 5) ; => 25

(define square (lambda (x) (* x x)))
(square 5)               ; => 25

In fact, (define (square x) (* x x)) is just sugar for the lambda form. The lambda is the primitive; named functions are lambdas with a label.

Closures

A closure is a function packaged with the environment in which it was defined. When a lambda references a free variable, the closure remembers where that variable lives and what it was bound to.

(define (make-adder n)
  (lambda (x) (+ x n)))

(define add5 (make-adder 5))
(define add10 (make-adder 10))

(add5 3)     ; => 8
(add10 3)    ; => 13

Each call to make-adder creates a new environment with its own n. The returned lambda closes over that environment. Even after make-adder returns, the environment persists as long as something references the closure.

Closures are the reason Lisp has objects for free. An object is just a closure over some state with a dispatcher. count here is entirely private. Only the closure has access. This is encapsulation without a class system:

(define (make-counter)
  (let ((count 0))
    (lambda (msg)
      (case msg
        ((inc) (set! count (+ count 1)))
        ((dec) (set! count (- count 1)))
        ((get) count)))))

(define c (make-counter))
(c 'inc) (c 'inc) (c 'inc) (c 'dec)
(c 'get)    ; => 2

Church Encoding: Everything from Lambdas

Church showed you don’t need booleans, numbers, pairs, or lists as primitives. You can encode them all with lambdas.

(define church-true  (lambda (a b) a))
(define church-false (lambda (a b) b))
(define church-if    (lambda (c t f) (c t f)))

(define church-zero  (lambda (f x) x))
(define church-one   (lambda (f x) (f x)))
(define church-two   (lambda (f x) (f (f x))))

(define church-succ
  (lambda (n) (lambda (f x) (f (n f x)))))

(define church-pair
  (lambda (a b) (lambda (selector) (selector a b))))
(define church-first  (lambda (p) (p (lambda (a b) a))))
(define church-second (lambda (p) (p (lambda (a b) b))))

Given only lambda, you can build every data structure and every control flow construct. Lisp’s primitives (cons, if, numbers) are conveniences, not necessities.

The Y Combinator: Anonymous Recursion

How do you write a recursive function without giving it a name? The function needs to reference itself, but if it’s anonymous, there’s nothing to reference. The solution is the fixed-point combinator:

(define Y
  (lambda (f)
    ((lambda (x) (f (lambda (v) ((x x) v))))
     (lambda (x) (f (lambda (v) ((x x) v)))))))

(define factorial
  (Y (lambda (fact)
       (lambda (n)
         (if (= n 0) 1 (* n (fact (- n 1))))))))

(factorial 5)    ; => 120

There is no define inside the lambda. The function has no name. Y manufactures self-reference out of pure lambda applications. The derivation of Y is a rite of passage for every functional programmer.

The Meta-Circular Evaluator

// Maxwell’s Equations of Software //

In April 1960, John McCarthy published a paper that introduced S-expressions, conditional expressions, recursive function definitions, and something more audacious: eval — a function that takes a Lisp expression (as data) and returns its value. Defined in Lisp itself. An interpreter for the language, written in the language. Twenty-some lines of code. A perfectly recursive definition.

Steve Russell read the paper, realized it could be implemented, and wrote the first actual Lisp interpreter. This is the meta-circular evaluator: a Lisp interpreter written in Lisp. The recursion mirrors itself. It’s self-describing.

The architecture is two mutually recursive functions. eval takes an expression and an environment and returns a value. apply takes a procedure and a list of argument values and returns a value. eval dispatches on the type of expression; apply dispatches on the type of procedure.

A minimal meta-circular evaluator in under 50 lines

(define (meval exp env)
  (cond
    ((number? exp) exp)
    ((string? exp) exp)
    ((symbol? exp) (lookup exp env))
    ((eq? (car exp) 'quote) (cadr exp))
    ((eq? (car exp) 'if)
     (if (meval (cadr exp) env)
         (meval (caddr exp) env)
         (meval (cadddr exp) env)))
    ((eq? (car exp) 'lambda)
     (list 'closure (cadr exp) (cddr exp) env))
    ((eq? (car exp) 'define)
     (extend-env! (cadr exp) (meval (caddr exp) env) env))
    (else
     (mapply (meval (car exp) env)
             (map (lambda (e) (meval e env)) (cdr exp))))))

(define (mapply proc args)
  (cond
    ((procedure? proc) (apply proc args))
    ((eq? (car proc) 'closure)
     (let ((params (cadr proc))
           (body   (caddr proc))
           (env    (cadddr proc)))
       (meval-seq body (extend-env params args env))))
    (else (error "unknown procedure" proc))))

You can now evaluate Lisp programs inside Lisp:

(meval '((lambda (x) (* x x)) 5) (initial-env))   ; => 25
(meval '(if (= 1 2) 'a 'b) (initial-env))         ; => b

You have just written a Lisp interpreter in Lisp. In 40 lines. The moment you internalize this is the moment everything changes.

Futamura projections: compilers from interpreters

Yoshihiko Futamura observed in 1971 that if you have an interpreter for language L written in language H, and you have a partial evaluator (a program that specializes another program with respect to some of its inputs), then:

  • First Futamura projection: Partially evaluate the interpreter with respect to a specific source program. The result is a compiled version of that program in H.
  • Second Futamura projection: Partially evaluate the partial evaluator with respect to the interpreter. The result is a compiler from L to H.
  • Third Futamura projection: Partially evaluate the partial evaluator with respect to itself. The result is a compiler generator — a program that takes an interpreter and produces a compiler.

This works. PyPy’s tracing JIT effectively implements the first and second projections for Python. The punchline: a compiler is an interpreter that has been specialized away. Every language with a meta-circular evaluator is implicitly a compiler too.

CLOS: The Common Lisp Object System

Most OOP languages have single dispatch: a method call obj.method(args) looks up the method on obj’s class. Dispatch is based on one object — the receiver.

CLOS has multiple dispatch: a generic function call (method obj1 obj2 obj3) looks at the types of all the arguments and picks the most specific matching method. Dispatch is based on all the arguments.

(defgeneric collide (a b))

(defmethod collide ((a asteroid) (b asteroid))
  (print "Two asteroids bump"))

(defmethod collide ((a ship) (b asteroid))
  (print "Ship takes damage"))

(defmethod collide ((a asteroid) (b ship))
  (print "Asteroid takes damage"))

(defmethod collide ((a ship) (b ship))
  (print "Ships exchange fire"))

In a single-dispatch language, you’d have to put the logic on Ship or Asteroid, which means one class has to know about the other. In CLOS, the generic function owns the dispatch, and any combination of classes can have a method. New classes can add methods for existing generic functions without modifying any existing class.

Method combinations

CLOS generic functions support method combinations — multiple methods that all run as part of a single generic function call:

(defmethod draw :before ((s shape)) (log-drawing-start s))
(defmethod draw ((s circle)) (draw-circle s))
(defmethod draw :after ((s shape)) (log-drawing-end s))
(defmethod draw :around ((s shape))
  (with-lock *draw-lock* (call-next-method)))

One call to (draw c) on a circle runs: :around, then :before, then primary, then :after. This is aspect-oriented programming without the plugin or the annotation processor. It’s just CLOS.

The metaobject protocol (MOP)

CLOS classes are themselves objects. An instance of point is an object. The point class is also an object — specifically, an instance of standard-class, which is itself an instance of standard-class. It turtles all the way down, bottoming out at a cyclic base case.

The metaobject protocol (MOP) is the published API that lets you inspect and modify the behavior of the CLOS implementation. You can subclass standard-class to make a new class type, intercept slot reads and writes, add persistence transparently, add type-checking transparently, implement prototype-based objects on top of CLOS classes.

The Art of the Metaobject Protocol (1991) by Kiczales, des Rivières, and Bobrow is the book that introduced the MOP concept. Kiczales later led the AspectJ project, which brought CLOS’s before/after/around ideas to Java. Every serious language designer has read AMOP.

Influence

Python borrowed single-dispatch methods and the descriptor protocol from CLOS. Ruby borrowed open classes and method-missing. Julia is essentially CLOS without inheritance-based encapsulation — it has multiple dispatch as its primary abstraction mechanism. Dylan was an attempt to put CLOS into a non-parenthetical syntax; it had a beautiful design and almost no users.

The Numeric Tower

Common Lisp has a rich hierarchy of numeric types, often called the numeric tower:

number
├── real
│   ├── rational
│   │   ├── integer
│   │   │   ├── fixnum (machine word)
│   │   │   └── bignum (arbitrary precision)
│   │   └── ratio (exact fractions, e.g., 1/3)
│   └── float (short, single, double, long)
└── complex (real + imaginary)

The operations preserve exactness when possible and promote automatically when needed:

(+ 1/2 1/3)      ; => 5/6            ratio + ratio = ratio
(* 1/3 3)        ; => 1              exact!
(sqrt -1)        ; => #C(0 1)        returns complex
(expt 2 1000)    ; => (~300 digit bignum)
(reduce #'+ (loop repeat 10 collect 1/10))   ; => 1    (exact!)
(reduce #'+ (loop repeat 10 collect 0.1))    ; => 1.0000001  (float error)

Floating point is fast and convenient but lies about numbers. With the numeric tower, you compute exactly. Only when you explicitly ask for a float do you get inexact arithmetic. This is why Common Lisp is still the go-to language for symbolic math and experimental number theory.

Conditions and Restarts

This is the best-kept secret of Common Lisp: its error handling is radically more powerful than what any mainstream language offers. It is not “exceptions, but different.” It is a completely different model.

In most languages, when an error occurs, the stack unwinds until something catches the exception. The caller decides how to recover. The code that detected the error is gone by the time recovery happens — it’s just a corpse on the stack.

In Common Lisp, when a condition is signaled, the stack does not unwind immediately. The condition handler runs in the context of the signaling code and can decide what to do. The handler can invoke a restart — a named recovery strategy that the signaling code published. The signaling code continues running with that recovery applied.

(define-condition invalid-input (error)
  ((value :initarg :value :accessor invalid-input-value)))

(defun process (x)
  (restart-case
      (if (numberp x)
          (* x 2)
          (error 'invalid-input :value x))
    (use-zero () 0)
    (use-value (v) v)
    (retry-with (new) (process new))))

(handler-bind
    ((invalid-input
      (lambda (c)
        (invoke-restart 'retry-with 42))))
  (process "not a number"))
;; => 84

The handler is completely decoupled from the restart list. The function offers recovery strategies; the caller picks one based on policy. This is error handling as a negotiation between caller and callee, not as a unilateral stack unwind.

No other mainstream language has this. Not Python. Not Ruby. Not Java. Not Rust. They all unwind on error. Common Lisp unwinds only when it chooses to, and restarts let the code publish recovery options in advance. When a long-running computation fails deep in its call tree, the interactive debugger pops up showing you the error and the list of restarts. You can fix the problem, pick a restart to continue, and the computation resumes from the point of failure. You don’t lose hours of computation because of a typo.

Tail Call Optimization

A tail call is a function call in tail position — the last thing a function does before returning. In a language with tail-call optimization (TCO), the compiler reuses the current stack frame for the tail call instead of pushing a new one. Scheme mandates TCO in the language spec. Common Lisp doesn’t, but all serious implementations provide it.

;; NOT tail-recursive — the `*` happens after the recursive call
(define (fact n)
  (if (= n 0) 1 (* n (fact (- n 1)))))

;; Tail-recursive — recursive call is the last thing
(define (fact n)
  (define (loop i acc)
    (if (= i 0) acc (loop (- i 1) (* i acc))))
  (loop n 1))

TCO is not an optimization in the trivial sense. It is the thing that makes functional iteration possible. Without it, you are forced back to imperative loops. With it, you can write any iteration as a recursive function and rely on the compiler.

Continuations and call/cc

Scheme’s call-with-current-continuation (abbreviated call/cc) captures “the rest of the computation” as a first-class function. This is the most powerful control operator any language has exposed. From call/cc you can implement exceptions, generators, coroutines, green threads, backtracking search, and effect systems — all as library code, not language extensions.

;; Exception-like escape
(call/cc
 (lambda (escape)
   (for-each
    (lambda (x)
      (if (negative? x)
          (escape 'found-negative)
          (process x)))
    some-list)))

Continuations are a direct extension of the lambda calculus. In continuation-passing style (CPS), every function takes an extra argument — the “what to do next” — and instead of returning, calls that continuation with its result. Every Lisp (and every language with a compiler) can be rewritten in CPS. The lambda calculus is that flexible.
