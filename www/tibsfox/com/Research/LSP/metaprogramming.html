Metaprogramming & Macros

Code that writes code — the superpower that makes Lisp Lisp, and why every other language spends decades reinventing pieces of it

“Lisp is a programmable programming language.”

— John Foderaro, CACM 34(9), 1991

// The Thesis //

A Lisp macro is a function from code to code that runs at compile time. Everything else on this page is commentary.

When you call a function, the arguments are evaluated first, then passed to the function. When you call a macro, the arguments are not evaluated — they are passed as unevaluated code. The macro returns new code. That new code is then compiled and run in place of the original macro call.

A function sees values. A macro sees code. A function returns a value at runtime. A macro returns code at compile time. This is the single most important distinction in Lisp programming, and it is the reason Lisp has been able to absorb every new programming paradigm for 68 years without a single change to the core language.

Defmacro

In Common Lisp, you define a macro with defmacro:

(defmacro when (condition &body body)
  `(if ,condition (progn ,@body)))

This defines when — a conditional that runs a body if the condition is true, with no else clause. The backtick, comma, and comma-at are quasiquote syntax.

(when (> x 0)
  (print "positive")
  (print x))

;; expands at compile time to:
(if (> x 0)
    (progn (print "positive") (print x)))

That expansion happens at compile time, once, when the compiler processes the source file. At runtime there is no when — only the if it expanded to. There is zero overhead.

Quasiquote, Unquote, Splicing

Writing macros that build code by hand gets tedious. Quasiquote (backtick) is a smarter quote. It acts like quote by default, but inside a quasiquoted form you can unquote (with comma) to insert a computed value, and unquote-splice (with comma-at) to insert a list as multiple elements.

`(a b c)                      ; => (a b c)         same as '(a b c)
`(a ,(+ 1 2) c)               ; => (a 3 c)         comma evaluates
`(a ,@(list 1 2 3) c)         ; => (a 1 2 3 c)    comma-at splices

Read the when macro slowly. We are constructing a piece of code. The template is (if ,condition (progn ,@body)). Backtick quotes the whole thing as a list. Comma says “evaluate condition here and insert the result.” Comma-at says “evaluate body (which is a list) and splice its elements in here.”

Quasiquote is the macro writer’s best friend. Without it, macros would be an unreadable soup of cons, list, and append. With it, macros look almost exactly like the code they produce.

The Canonical Examples

unless — opposite of when

(defmacro unless (condition &body body)
  `(if (not ,condition) (progn ,@body)))

swap — cannot be a function

(defmacro swap (a b)
  `(let ((tmp ,a))
     (setf ,a ,b)
     (setf ,b tmp)))

(let ((x 1) (y 2))
  (swap x y)
  (list x y))    ; => (2 1)

Swap has to assign to the names, not the values, so it can’t be a function. The function would only see the values.

with-open-file — resource management

(defmacro with-open-file ((var path) &body body)
  `(let ((,var (open ,path)))
     (unwind-protect
          (progn ,@body)
       (close ,var))))

(with-open-file (f "data.txt")
  (read-line f))

The pattern “acquire, use, release” cannot be expressed as a plain function in most languages. Python needs with. C++ needs RAII. Java needs try-with-resources. Rust needs Drop. Common Lisp needs... a three-line macro.

cond — a recursive macro over if

(defmacro cond (&rest clauses)
  (if (null clauses)
      nil
      (let ((first-clause (car clauses))
            (rest-clauses (cdr clauses)))
        `(if ,(car first-clause)
             (progn ,@(cdr first-clause))
             (cond ,@rest-clauses)))))

A recursive macro. It expands into nested ifs. Each clause becomes one level of nesting.

Notice what is happening. Almost everything you think of as “built-in” in other languages is a macro in Lisp. when, unless, cond, case, dotimes, dolist, while, do, let, let*, letrec, loop, with-open-file, with-slots, with-lock, handler-case, restart-case — every one of these is a macro. In most languages, adding a new control structure means a committee decision, a parser change, a compiler feature, a breaking change to the language standard. In Lisp it is a ten-line function that runs at compile time.

The Variable Capture Problem

Consider this macro:

(defmacro my-swap (a b)
  `(let ((tmp ,a))
     (setf ,a ,b)
     (setf ,b tmp)))

It works fine for normal use. But what happens here?

(let ((tmp 100) (x 1))
  (my-swap tmp x))

;; expands to:
(let ((tmp 100) (x 1))
  (let ((tmp tmp))       ; ← captures the outer tmp!
    (setf tmp x)
    (setf x tmp)))

The tmp introduced by the macro collides with the tmp in the user’s code. The macro has captured a variable from the surrounding scope. The user wanted to swap tmp and x; the macro silently produced broken code.

This is the variable capture problem. It is the fundamental hazard of Lisp macros, and different Lisps have solved it differently.

Three Approaches to Hygiene

Common Lisp: gensym (manual hygiene)

Common Lisp’s answer is gensym — a function that creates a brand new symbol guaranteed not to collide:

(defmacro my-swap (a b)
  (let ((tmp (gensym)))
    `(let ((,tmp ,a))
       (setf ,a ,b)
       (setf ,b ,tmp))))

Each macro expansion introduces a fresh symbol (#:G12345 or similar) that cannot collide with user code. The hygiene problem is solved manually, at the cost of remembering to gensym every temporary.

Scheme: syntax-rules (automatic hygiene, pattern-based)

The R4RS standard introduced syntax-rules, a pattern-based macro system that is automatically hygienic:

(define-syntax swap!
  (syntax-rules ()
    ((swap! a b)
     (let ((tmp a))
       (set! a b)
       (set! b tmp)))))

The pattern (swap! a b) binds a and b to whatever was in the call. Identifiers introduced by the template (like tmp) are automatically renamed to fresh symbols when there’s a risk of capture. No gensym needed.

syntax-rules supports ellipsis patterns for variadic macros:

(define-syntax when
  (syntax-rules ()
    ((when cond body ...)
     (if cond (begin body ...)))))

The body ... pattern matches zero or more expressions and expands them splicingly. Pattern-based code generation at its cleanest.

Clojure: syntax-quote (namespace resolution + auto-gensym)

Clojure (2007) took a middle path. Clojure’s backtick automatically resolves symbols to their fully qualified namespace at macro expansion time, and any symbol ending in # inside a syntax-quote is automatically gensymed:

(defmacro my-when [cond & body]
  `(if ~cond (do ~@body)))

;; Expansion:
;; (if (user/pos? x) (clojure.core/do (clojure.core/println x)))

(defmacro swap! [a b]
  `(let [tmp# ~a]
     (reset! ~a @~b)
     (reset! ~b tmp#)))

Clojure’s system is a pragmatic middle ground — most of the hygiene of syntax-case with the syntactic comfort of defmacro.

Anaphoric Macros: Deliberate Capture

An anaphoric macro deliberately captures a variable so the user can refer to it. The name comes from linguistics — an “anaphor” is a pronoun that refers back to an antecedent.

(defmacro aif (test then &optional else)
  `(let ((it ,test))
     (if it ,then ,else)))

(aif (find-user "foxy")
     (print (user-name it))     ; `it` refers to the result of find-user
     (print "not found"))

aif introduces it on purpose. Hygiene would prevent this — the whole point is that it is capturable. awhen, acond, alet are the rest of the anaphoric family. Paul Graham uses them throughout On Lisp.

Anaphoric macros are divisive, but they illustrate that hygiene is a choice, not a law.

When to Use Macros, and When Not To

Heuristic: If your “macro” calls all its arguments exactly once in order, it should be a function. Macros exist to not evaluate arguments, to evaluate them zero or multiple times, to evaluate them in a non-standard order, or to introduce new bindings.

Bad macro — just use +:

(defmacro add (a b) `(+ ,a ,b))

Good macro — and short-circuits (doesn’t evaluate later args if earlier is false):

(defmacro and (&rest args)
  (cond ((null args) t)
        ((null (cdr args)) (car args))
        (t `(if ,(car args) (and ,@(cdr args)) nil))))

The mature take: functions first, macros when you need new evaluation rules. If your proposed macro does something a function could do, use a function. If it needs to not evaluate an argument (like if), or introduce a binding (like let), or wrap the body in something (like with-open-file), then use a macro. The macro is a tool for shaping code; the function is a tool for shaping values.

Reader Macros: Extending the Parser Itself

Reader macros run during the read phase, before the evaluator or the normal macro expander has seen anything. A reader macro lets you add entirely new surface syntax:

(set-dispatch-macro-character #\# #\v
  (lambda (stream char arg)
    (declare (ignore char arg))
    (let ((lst (read-delimited-list #\] stream t)))
      (coerce lst 'vector))))

;; Now you can write:
#v[1 2 3 4]
;; which reads as #(1 2 3 4)

Reader macros are powerful enough to embed JSON, YAML, or entire other languages directly in a Lisp source file. Racket takes this to the extreme: a file starting with #lang datalog is read by the Datalog reader, not the Racket reader.

Macro-Writing Macros: There Is No Bottom

Macros can write macros. The idiom defmacro/g! (from Doug Hoyte’s Let Over Lambda) automatically gensyms any symbol in a macro body that starts with g!:

(defmacro defmacro/g! (name args &body body)
  (let* ((syms (remove-duplicates
                (remove-if-not
                 (lambda (s) (and (symbolp s)
                                  (> (length (symbol-name s)) 2)
                                  (string= "G!" (subseq (symbol-name s) 0 2))))
                 (flatten body)))))
    `(defmacro ,name ,args
       (let ,(mapcar (lambda (s) `(,s (gensym ,(symbol-name s)))) syms)
         ,@body))))

(defmacro/g! my-swap (a b)
  `(let ((,g!tmp ,a))
     (setf ,a ,b)
     (setf ,b ,g!tmp)))

Macros writing macros. Tools building tools. This is why Lisp is the meta language — the same mechanism that lets you extend the language also lets you extend the extension mechanism itself. There is no bottom.

A DSL in 20 Lines — HTML Generation

(defmacro html (&body body)
  `(with-output-to-string (*standard-output*)
     ,@(mapcar #'html-form body)))

(defun html-form (form)
  (cond ((stringp form) `(princ ,form))
        ((atom form) `(princ ,form))
        (t (let ((tag (car form))
                 (contents (cdr form)))
             `(progn
                (format t "<~a>" ',tag)
                ,@(mapcar #'html-form contents)
                (format t "</~a>" ',tag))))))

(html
  (html (head (title "Hi"))
        (body (h1 "Welcome")
              (p "Hello, world."))))

Twenty lines. No template language. No preprocessor. No string concatenation gymnastics. Just S-expressions and a code walker that transforms them at compile time.

Racket: Languages as Libraries

Racket (formerly PLT Scheme) has pushed the macro idea further than anyone. In Racket:

  • A language is a module that exports #%module-begin, #%app, #%datum, and #%top — the forms the reader uses when starting a module, applying a function, encountering a literal, and looking up an identifier.
  • By redefining these forms, you can redefine every fundamental operation of the language.
  • You can ship a new language as a regular library. Users include #lang my-language at the top of their file, and the file is read and compiled using your language’s rules.

Racket ships dozens of #langs:

LanguagePurpose
#lang racketThe default (modern Scheme descendant)
#lang typed/racketSound gradual typing
#lang lazyHaskell-style lazy evaluation
#lang datalogDatalog logic programming
#lang scribbleDocumentation as a language
#lang slideshowPresentations
#lang pollenMatthew Butterick’s book publishing language
#lang r5rsClassic Scheme compatibility

Each is a language-as-a-library. They all interoperate because they all compile to Racket’s core. The Racket team has published papers on “Languages as Libraries” and on how to build DSLs with proper binding, scoping, and type checking. When somebody says “in Lisp you can build your own language,” Racket is the existence proof.

What Macros Actually Enable

The macros in the Lisp tradition have grown entire families of powerful abstractions:

  • Pattern matching — match macros that destructure values against patterns. Standard in Haskell, retrofitted to Scala and Rust, built in a weekend to Common Lisp and Racket.
  • ORM/DSL syntax — Hibernate-like object-relational mappers in twenty lines. Datomic’s query language in Clojure is just a DSL on top of Clojure data.
  • Parser combinators — build a parser by combining small parsers with macros.
  • Web routing — (route "/users/:id" get (show-user id)). No annotations, no reflection, just a macro.
  • Build systems — Racket’s make ships as a module you include in a regular Racket file.
  • Language implementation — Racket literally ships languages as libraries. All are full-blown languages with their own readers, macros, and compilers, living side by side in the same system.
  • Automatic differentiation — walks an expression tree and produces a derivative expression. This is how every modern ML framework works, and Lisp did it first.
  • Compilers — including whole-program optimizing compilers targeting other languages.

If you’ve ever thought “I wish my language supported X,” in Lisp you just... write X. The language is infinitely extensible because the compiler is a library you can call.
