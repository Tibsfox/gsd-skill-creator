Upstream Movement

gsd-skill-creator exists at the intersection of two rapidly evolving upstreams: GSD (get-shit-done) and Claude Code. Both shipped significant features in March–April 2026 that directly affect the skill-creator’s architecture. This module tracks those changes and maps them to implementation opportunities across the other seven GSI modules.

1. GSD Upstream v1.30.0 SDK Additions

Release: gsd-pi v2.59.0 (March 27, 2026)
Key addition: SDK for programmatic GSD workflow access. 8 runtime support. Standalone CLI.

The GSD SDK enables programmatic access to workflow primitives that were previously only available through the CLI. For gsd-skill-creator, this means skills and agents can drive GSD workflows without shell-escaping through Bash tool calls.

2. Claude Code v2.1.88 New Hooks

Three new hooks expand the deterministic control points available to gsd-skill-creator:

HookFires WhenGSI Application
PostCompact After context compaction occurs Context Optimization: Re-inject critical state (STATE.md, current PLAN.md, safety constraints) into the compressed context. This is the single most impactful hook for combating context rot.
FileChanged When a watched file is modified externally Learning System: Monitor skill files for external modifications. If a skill changes outside the agent’s control, trigger integrity verification before the next activation.
PermissionDenied When a tool call is blocked by permissions Agent Identity: Log denied tool calls to the immutable audit trail. Track patterns of denied requests as a signal for misconfigured permissions or adversarial probing.

Hook + if-conditions

The new if-conditions feature allows hooks to fire conditionally based on context state. This enables hooks that are aware of the current GSD phase, the active skill profile, or the agent’s trust tier.

// Conditional PostCompact hook
{
  "hook": "PostCompact",
  "if": "phase.type === 'execute' && phase.risk > 'medium'",
  "action": "inject-safety-context",
  "files": [".planning/STATE.md", ".planning/SAFETY-CONSTRAINTS.md"]
}

4. Skill Description 250-char Cap

Claude Code now enforces a 250-character limit on skill descriptions. All 6 gsd-skill-creator skills that exceeded this limit were trimmed in the March ecosystem alignment update. This constraint has implications for skill security:

  • Reduced attack surface: Shorter descriptions mean fewer tokens for adversarial DDIPE payloads to hide in
  • Activation precision: Tighter descriptions reduce false-positive activations, combating ghost token accumulation
  • Token savings: 34 skills × ~50 char reduction = ~1,700 characters (~425 tokens) saved at session start

5. S-PATH RAG: Graph-Aware Retrieval

Source: arXiv:2603.23512 — S-PATH proposes graph-aware retrieval that traverses relationship edges between documents, using a verifier pattern and path budget to control exploration depth.

For gsd-skill-creator, S-PATH maps to the cross-reference traversal problem in the PNW Research Series. The corpus of 242 projects has rich inter-project connections (the Complex Plane analysis shows these as magnitude and phase values). S-PATH’s path budget concept aligns with the token budget tracking already implemented in the token-budget skill.

Integration Points

  • Skill discovery: When a task requires capabilities not in the loaded skill set, traverse the skill graph to find related skills (S-PATH’s graph layer above vector embeddings)
  • Research navigation: The research co-processor can use S-PATH to traverse research project connections when building synthesis documents
  • Context building: Instead of loading all potentially relevant context, use S-PATH’s verifier to evaluate each traversal step and stop when the path budget is exhausted

6. Context-Packet DAG Resolver

The gsd-build project introduced context-packet: a DAG context resolver with token budget awareness. It models context dependencies as a directed acyclic graph and resolves them within a token budget, dropping lowest-priority nodes when the budget is exceeded.

DAG for Skill Loading

// Context-packet DAG for a typical GSD execute-phase session
root
  |-- system-prompt (required, ~2K tokens)
  |-- safety-constraints (required, ~500 tokens)
  |-- project-state (required, ~1K tokens)
  |     |-- STATE.md
  |     |-- current PLAN.md
  |
  |-- active-skills (on-demand, budget-aware)
  |     |-- gsd-workflow (always)
  |     |-- security-hygiene (always)
  |     |-- [task-specific skills] (triggered)
  |
  |-- tool-definitions (deferred, loaded via ToolSearch)
  |     |-- core-tools: Read, Edit, Write, Bash, Grep, Glob
  |     |-- [MCP tools loaded on first use]
  |
  |-- conversation-history (compactable, lowest priority)
        |-- recent turns (high priority)
        |-- older turns (low priority, compaction candidates)

The DAG resolver ensures that safety-critical context is never dropped, even under severe budget pressure. This directly supports the context rot prevention strategy from module 2.
