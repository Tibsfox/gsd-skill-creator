From Flat to Hierarchical

OrgAgent’s 3-layer hierarchy achieved 102.73% improvement over flat multi-agent systems on SQuAD 2.0 while reducing token consumption by 74.52%. The Data Processing Inequality proves mathematically that single-agent systems preserve more information for multi-hop reasoning tasks. The question is not “single or multi” but “when each, and how to switch.”

gsd-skill-creator already uses multiple orchestration patterns: the mayor-coordinator for convoys, fleet-mission for parallel dispatch, and single-agent execution for focused tasks. The research from arXiv:2604.01020 and 2604.02460 provides the theoretical framework to make pattern selection systematic rather than ad hoc.

1. OrgAgent 3-Layer Mapping to GSD

OrgAgent organizes multi-agent systems into three layers modeled on corporate hierarchy: governance, execution, and compliance. This maps directly to the existing GSD architecture.

OrgAgent LayerFunctionGSD Equivalent
GovernanceTask decomposition, resource allocation, strategic decisionsmayor-coordinator — creates convoys, dispatches work via sling, monitors progress
ExecutionActual task work, tool invocation, artifact productionpolecat-worker — ephemeral autonomous execution, GUPP autonomous mode
ComplianceOutput verification, quality gates, constraint enforcementgsd-verify-work + adversarial-pr-review — UAT validation and spec-compliance checking

The key insight from OrgAgent: the governance layer should never execute tasks directly. Its sole job is decomposition and assignment. When the mayor-coordinator attempts direct execution (a pattern seen in early GSD versions), it conflates governance and execution, losing the hierarchical token savings.

2. Single-Agent for Deep Reasoning (Data Processing Inequality)

The DPI proof: For any chain of information processing X → Y → Z, mutual information can only decrease: I(X;Z) ≤ I(X;Y). Every inter-agent handoff is an information-lossy channel. For multi-hop reasoning tasks, a single agent preserves more information than a multi-agent pipeline.

arXiv:2604.02460 demonstrates that single-agent LLMs outperform multi-agent systems on multi-hop reasoning benchmarks. The information-theoretic explanation is rigorous: each agent boundary is a lossy channel. The more hops, the more information loss.

When Single-Agent Wins

  • Deep reasoning chains: Mathematical proofs, complex debugging, architectural analysis
  • Context-dependent decisions: Tasks where every prior step informs the next
  • Small search space: Tasks with well-defined scope that fit in one context window

When Multi-Agent Wins

  • Embarrassingly parallel tasks: Independent file processing, batch operations
  • Large search space: Tasks requiring exploration of many alternatives
  • Diverse tool requirements: Tasks needing different tool sets that would bloat a single context

3. Multi-Agent Fleet Pattern

The fleet-mission skill already implements parallel agent dispatch. Refine it based on the OrgAgent findings to enforce the governance/execution separation:

// Fleet dispatch with OrgAgent hierarchy
governance: mayor-coordinator
  |-- decomposes task into N independent work items
  |-- assigns each to a polecat-worker via sling-dispatch
  |
execution: polecat-worker[1..N]
  |-- each receives: task brief, scoped tools, token budget
  |-- each operates in isolated context (worktree isolation)
  |-- returns: structured result artifact
  |
compliance: witness-observer
  |-- monitors all workers for stalls, errors, drift
  |-- aggregates results into unified output
  |-- triggers adversarial review on suspicious results

4. Hierarchical Token Reduction (74.52%)

The 74.52% token reduction from OrgAgent comes from three mechanisms that apply directly to GSD:

MechanismToken SavingsGSD Implementation
Task decomposition~30%Break large tasks into focused sub-tasks with minimal context each
Scoped tool loading~25%Each worker loads only the tools needed for its specific sub-task
Result summarization~20%Workers return structured results, not full conversation transcripts

5. Execution Mode Selection: DIRECT / LIGHT / FULL

Instead of always launching the full GSD pipeline (discuss → plan → execute → verify), select execution mode based on task complexity:

ModeWhenPipelineToken Cost
DIRECTTrivial tasks, single-file changesExecute inline, no sub-agentsMinimal
LIGHTModerate tasks, 2–5 file changesPlan → execute (skip discuss, minimal verify)~40% of FULL
FULLComplex tasks, architectural changesDiscuss → plan → execute → verify with sub-agentsFull budget

The existing gsd-fast and gsd-quick commands already implement DIRECT and LIGHT modes respectively. The improvement is making mode selection automatic based on task analysis rather than requiring the user to choose.

Mode Selection Heuristics

  • File count: 1 file = DIRECT, 2–5 = LIGHT, 6+ = FULL
  • Reasoning depth: Single-step = DIRECT, multi-hop = FULL
  • Risk level: Read-only = DIRECT, writes = LIGHT, config changes = FULL
  • User override: Always allow explicit mode selection via command flags

6. Dynamic Skill Profile Assignment

When dispatching workers from the pool, assign each a dynamic skill profile based on its task rather than loading the full skill set. The skills field in the new subagent configuration enables this:

// Dynamic skill profile for a code review worker
{
  effort: 0.8,
  maxTurns: 15,
  isolation: "worktree",
  skills: ["code-review", "security-hygiene", "typescript-patterns"],
  mcpServers: []  // no external tools needed
}

// Dynamic skill profile for a research worker
{
  effort: 1.0,
  maxTurns: 30,
  isolation: "worktree",
  skills: ["research-engine", "data-fidelity"],
  mcpServers: ["gsd-math-coprocessor"]
}
