Safety Is Not Reliability

A system can be reliable (works correctly most of the time) while being fundamentally unsafe (catastrophic when it fails). CMU’s ML in Production course makes this the central distinction: reliability is about uptime; safety is about what happens during downtime. For gsd-skill-creator, a skill that works 99% of the time but exfiltrates data on the 1% failure is reliable but not safe.

The CMU framework (AGT cluster 3) provides three hazard analysis methodologies — FMEA, FTA, and STPA — that originated in aerospace and automotive safety engineering. Applying them to skill execution transforms the verification pipeline from a pass/fail gate into a systematic safety analysis framework.

2. FTA: Fault Tree Analysis

FTA works top-down from an undesired event to its root causes. For gsd-skill-creator, the top-level undesired event is “skill-creator produces harmful output.”

HARMFUL OUTPUT (top event)
  |
  +-- OR --+
  |        |
  |   ADVERSARIAL SKILL LOADED
  |     |
  |     +-- AND --+
  |     |         |
  |     |    Skill passes static analysis
  |     |    Skill passes single-model check
  |     |    Skill contains delayed-activation payload
  |     |
  |   CONTEXT CORRUPTION
  |     |
  |     +-- OR --+
  |     |        |
  |     |    Ghost tokens override safety instructions
  |     |    Compaction loses critical constraints
  |     |    Sub-agent inherits polluted context
  |     |
  |   VERIFICATION BYPASS
  |     |
  |     +-- AND --+
  |               |
  |          Agent self-grades (no external verifier)
  |          Output contains subtle errors
  |          User trusts output without review

The FTA reveals that verification bypass requires the conjunction of self-grading AND subtle errors AND user trust. Breaking any one of these three links prevents the fault.

3. STPA: Systems-Theoretic Process Analysis

STPA models safety as a control problem. Unsafe states arise from inadequate control actions, not just component failures. For the skill-creator, the key control loops are:

  • Skill loading control: The security-hygiene skill controls what enters the skill registry. Unsafe if: loading proceeds without safety scan, or scan results are ignored.
  • Execution control: The mayor-coordinator controls worker behavior. Unsafe if: workers exceed their scoped permissions, or governance layer fails to monitor.
  • Output control: The verifier agent controls what gets committed. Unsafe if: agent self-verifies, or verification criteria are too permissive.

4. Guardrails as System Primitives

The CMU framework insists guardrails must be system primitives, not afterthoughts bolted on. In gsd-skill-creator, this means guardrails are hooks that fire deterministically, not skills that activate probabilistically.

Guardrail TypeImplementationFires When
Input guardPreToolUse hookBefore any tool invocation — validates parameters, checks permissions
Output guardPostToolUse hookAfter tool execution — validates results, checks for data leakage
Commit guardPre-commit hookBefore git commit — validates conventional commit format, no .planning files
Session guardPostCompact hookAfter compaction — re-injects critical safety context

5. The SQL Injection Analogy

Unsanitized LLM output injected into downstream systems is the SQL injection of the AI era. Just as web applications learned to never trust user input, agentic systems must never trust agent output without sanitization.

Attack Surfaces in gsd-skill-creator

  • Skill-generated code: Skills that produce code (test-generator, code-review) could embed malicious patterns
  • Commit messages: Generated commit messages could contain escape sequences or injection payloads for downstream CI systems
  • File path handling: Generated file paths could use path traversal (../../../etc/passwd)
  • MCP tool arguments: Constructed tool arguments could contain injection payloads for MCP servers

The defense is the same as for SQL injection: parameterized interfaces, input validation, output encoding, and principle of least privilege.

6. Externalized Correctness

The agent does not grade its own homework. This principle from the p3nGu1nZz meta-agentic bootstrapping work (AGT cluster 1) is the single most important verification improvement.

Current state: gsd-verify-work uses the same agent that executed the work to verify it. This is self-grading.
Target state: Spawn a separate verifier agent with a clean context, the verification criteria, and the execution artifacts. The verifier has no access to the reasoning that produced the artifacts — only the artifacts themselves and the criteria they must meet.

7. Abstention Rate Improvement (39.78%)

OrgAgent’s hierarchical structure improved abstention rates by 39.78% compared to flat MAS. Abstention — the ability to say “I don’t know” or “I cannot safely do this” — is a safety-critical capability. Flat systems tend to always attempt an answer, even when they should abstain.

In gsd-skill-creator, improved abstention means:

  • Skills that recognize when they are outside their competence and defer rather than guess
  • Workers that escalate to the governance layer when they encounter unexpected situations
  • The system as a whole preferring “no action” over “wrong action” for safety-critical operations
