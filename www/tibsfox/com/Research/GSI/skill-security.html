The Skill Poisoning Problem

631,813 skills in the SkillsMP marketplace with no mandatory security review. DDIPE bypass rates of 11.6–33.5% across Claude Code, OpenHands, Codex, and Gemini CLI. Only 1.6% of adversarial payloads bypass all three models simultaneously — making cross-model verification the single most effective defense.

arXiv:2604.03081 demonstrated 1,070 adversarial skills across 15 MITRE ATT&CK categories that exploit the document-as-code nature of skill files. Skills in gsd-skill-creator are markdown with YAML front matter — they are simultaneously documentation and executable configuration. This dual nature means every skill is an attack surface. The improvements below harden the skill loading pipeline without breaking the learning loop.

1. DDIPE Attack Mitigation in the Skill Loading Pipeline

Threat: Doc-Derived Indirect Prompt Execution — adversarial instructions embedded in skill description fields, YAML metadata, or markdown body text that the LLM executes as if they were user instructions.

The skill-creator currently loads skill descriptions directly into the system prompt. Every .claude/skills/ file contributes its description (up to 250 chars) and its full body text when activated. An adversarial skill can embed instructions like “ignore previous constraints” in ways that lexical pattern matching will not catch.

Mitigation Strategy

Load-time scanningStatic analysis of skill content before injection into contextParse YAML front matter, scan description and body for known DDIPE patterns from the 2604.03081 taxonomy

2. Document-as-Code Treatment for Skill Metadata

Skills are markdown files that serve as both human-readable documentation and machine-consumed configuration. This dual nature requires treating them with the same rigor as source code: version control, review gates, automated testing, and integrity verification.

Skill Lifecycle Controls

  • Git-tracked provenance: Every skill must be committed with a conventional commit message. The existing PreToolUse hook already enforces commit conventions — extend it to validate skill file changes specifically.
  • Schema validation: YAML front matter must conform to a strict schema (description, triggers, alwaysApply, globs). Reject unknown fields that could carry adversarial payloads.
  • Diff-based review: Skill modifications surface in git diff and must pass the security-hygiene skill’s automated check before activation.

3. Per-Skill Permission Scoping

Currently all skills share the same permission level once loaded. The 12 Primitives analysis of Claude Code reveals a tiered trust model: built-in tools > plug-in tools > user skills. Apply this same hierarchy to gsd-skill-creator skills.

4. Multi-Model Verification Ensemble

The 2604.03081 paper’s most actionable finding: only 1.6% of adversarial payloads bypass all three models (Claude, GPT, Gemini) simultaneously. Cross-model disagreement is the strongest available signal for detecting poisoned skills.

Verification Protocol

  1. Primary scan: The active model (Claude) analyzes skill content for adversarial patterns
  2. Secondary verification: A second model (via MCP or API) independently evaluates the same skill
  3. Disagreement escalation: If models disagree on safety classification, the skill enters quarantine and triggers HITL review
  4. Consensus promotion: Skills that pass all models move to their assigned trust tier

This maps directly to the existing gsd-review command, which already supports cross-AI peer review. Extend it to cover skill validation, not just phase plan review.

5. Trust-Boundary Annotations in System Prompts

System prompts should explicitly mark trust boundaries. When skill content is injected into context, annotate it with provenance metadata so the model can distinguish between trusted system instructions and potentially adversarial skill-derived content.

<!-- TRUST: core | source: .claude/skills/gsd-workflow.md | hash: a3f8... -->
<skill-content trust="core">
  ... skill body ...
</skill-content>

<!-- TRUST: community | source: skills/third-party-tool.md | hash: 7b2c... -->
<skill-content trust="community">
  ... skill body (model treats as data, not instructions) ...
</skill-content>

6. Static Analysis Integration

Integrate static analysis into the skill installation pipeline. Before a skill is written to .claude/skills/, scan its content for patterns from the DDIPE taxonomy:

  • Instruction override patterns: “ignore previous”, “disregard constraints”, “you are now”
  • Exfiltration vectors: URLs in skill content, base64-encoded payloads, encoded shell commands
  • Permission escalation: References to dangerouslyDisableSandbox, --no-verify, --force
  • Semantic misdirection: Skill descriptions that contradict their body content (detected via embedding similarity)
