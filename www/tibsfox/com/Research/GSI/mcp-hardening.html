110 Million Downloads, Zero Mandatory Audits

MCP SDK downloads reached 110 million by April 2026 — a 3,600x increase since November 2024. The protocol is now critical infrastructure for agentic AI systems. Yet the MCP Dev Summit revealed fundamental gaps: no mandatory tool auditing, no built-in cost controls, and no supply-chain integrity for the tool registry. gsd-skill-creator uses two MCP servers (gsd-math-coprocessor, unison) and will add more as the research co-processor matures.

1. Overthinking Loop Detection (658x Cost Inflation)

The problem: arXiv:2602.14798 documented MCP overthinking loops where agents enter recursive tool-calling cycles that inflate costs by up to 658x. The agent calls a tool, receives a result it finds insufficient, calls the tool again with slightly different parameters, and repeats hundreds of times without making progress.

In gsd-skill-creator, overthinking manifests as:

  • Read loops: Reading the same file repeatedly because compaction lost the memory of having read it
  • Search spirals: Running Grep with increasingly broad patterns, filling context with irrelevant results
  • Tool retry cascades: Retrying failed Bash commands with minimal variation instead of diagnosing the root cause

2. Binary MCP for Edge Performance

Standard MCP uses JSON-RPC over stdio or HTTP, which introduces serialization overhead for large payloads. The MCP Dev Summit discussed binary MCP as a protocol extension for performance-critical use cases.

Applicability to gsd-skill-creator

  • Math co-processor: The gsd-math-coprocessor transfers matrices, vectors, and computation results. Binary encoding (MessagePack or Protocol Buffers) would reduce serialization overhead for large linear algebra operations.
  • Embedding pipeline: When the research co-processor moves embeddings (768-dimensional float vectors), binary transfer is approximately 4x more efficient than JSON encoding.
  • Audio/image artifacts: The Artemis II mission produces audio segments and weather map images. Binary MCP would enable efficient transfer without base64 encoding overhead (33% size inflation).

3. OAuth Governance Control Planes

MCP servers that access external resources need OAuth-based access control. The Dev Summit proposed OAuth control planes where the human user authorizes MCP server scopes, and the agent inherits only the scopes it has been granted.

Current state in gsd-skill-creator: MCP servers are configured in .claude/settings.json with full access. The math co-processor and Unison servers have no per-session scope restrictions.
Target state: Each MCP server connection includes an OAuth scope that limits what the agent can do. The math co-processor might be scoped to “read-only computation” for research tasks and “read-write computation” only for verified math phases.

4. Stateful Transactional MCP Server Patterns

MCP servers are currently stateless — each tool call is independent. For complex operations that span multiple tool calls, this forces the client to manage all intermediate state in its context window. Stateful transactional patterns push state management to the server side.

Transaction Pattern

// Transactional MCP sequence
BEGIN_TRANSACTION(session_id: "phase-72-math")
  |
  CALL: algebrus_gemm(A, B)          // server holds result
  CALL: statos_regression(result)     // server chains results
  CALL: vectora_gradient(regression)  // server chains again
  |
COMMIT_TRANSACTION(session_id)        // server returns final result
  |
// Only the final result enters the agent's context
// Intermediate results stay server-side, saving context tokens

This pattern directly addresses context optimization (module 2) by keeping intermediate computation results server-side rather than injecting them into the agent’s context window.

5. Tool Enumeration Security Audit

An agent can enumerate all available tools from all connected MCP servers. This enumeration reveals the full capability surface, which an adversarial skill could exploit to discover tools the agent would not normally use.

Audit Checklist

  • Tool inventory: List all tools exposed by all MCP servers. Currently: gsd-math-coprocessor (18 tools) + unison (30+ tools)
  • Sensitivity classification: Which tools can modify state? Which can access external resources? Which are read-only?
  • Scope alignment: Are all exposed tools necessary for the current task? Could any be hidden until explicitly needed?
  • Description audit: Do tool descriptions contain information that could be exploited by adversarial prompts?

6. Semantic Intent Analysis

Lexical pattern matching catches known attack patterns but misses novel attacks that use synonyms, circumlocutions, or multilingual evasion. Semantic intent analysis uses embedding similarity to detect adversarial tool usage even when the literal text looks benign.

Analysis LevelWhat It CatchesCost
LexicalKnown patterns: “ignore previous”, base64, shell metacharactersLow (regex)
StructuralUnusual parameter combinations, unexpected tool sequencesMedium (rule engine)
SemanticIntent mismatch: tool call description vs actual parametersHigh (embedding comparison)

The pgvector database (maple@tibsfox, schema artemis) already supports embedding operations. Semantic intent analysis could use it to compare tool call intent against established usage patterns from the 190+ completed missions.

7. Supply-Chain Integrity for MCP Tool Registry

MCP tools are software dependencies. They deserve the same supply-chain integrity controls as npm packages or Python wheels: version pinning, hash verification, provenance attestation, and vulnerability scanning.
