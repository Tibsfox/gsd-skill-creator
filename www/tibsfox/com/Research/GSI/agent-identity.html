Who Is This Agent?

The MCP Dev Summit (7h14m, April 2026) introduced the Agent Identity Protocol (AIP) with IETF-125 coordination. The core question: when an MCP server receives a tool call, how does it know which agent is calling, what permissions that agent has, and whether to trust its claimed identity? For gsd-skill-creator, where sub-agents spawn sub-agents that invoke MCP tools, the identity chain must be cryptographically verifiable at every hop.

Agent identity is the foundation that every other security improvement depends on. Without knowing who is making a request, permission scoping, trust boundaries, and audit logs are all meaningless. The improvements below establish a trust chain from the root user through every agent and sub-agent in the system.

1. AIP / Agent Attestation Token Integration

Agent Identity Protocol (AIP): A proposed standard for agent-to-agent and agent-to-server identity verification. Each agent carries an Agent Attestation Token (AAT) — a signed JWT containing the agent’s identity, capabilities, and delegation chain.

In gsd-skill-creator, the trust chain starts with the human user (Foxy) and flows through the agent hierarchy:

Human User (root trust anchor)
  |-- signs initial session token
  |
Claude Code (primary agent)
  |-- AAT: { agent: "claude-code", delegator: "human", scope: "full" }
  |
mayor-coordinator (governance sub-agent)
  |-- AAT: { agent: "mayor", delegator: "claude-code", scope: "orchestrate" }
  |
polecat-worker (execution sub-agent)
  |-- AAT: { agent: "worker-1", delegator: "mayor", scope: "execute-phase-72" }
  |
MCP Server (tool provider)
  |-- validates AAT chain: worker-1 ← mayor ← claude-code ← human
  |-- checks scope: is "execute-phase-72" authorized for this tool?

Every MCP tool call carries the full delegation chain. The MCP server can verify the chain cryptographically and reject calls where the scope does not match the requested tool.

3. Implicit Deny / Least Privilege for Tool Access

The default posture for tool access should be deny, not allow. Each agent starts with zero tool access and must be explicitly granted the tools it needs for its specific task.

Permission Grant Flow

  1. Governance layer (mayor) decomposes task into sub-tasks
  2. For each sub-task, governance determines the minimum tool set required
  3. Worker receives a scoped permission grant: { tools: ["Read", "Edit", "Bash"], paths: ["src/**"], duration: "task" }
  4. Any tool call outside the grant is rejected at the harness level, not the model level
  5. Permission grant expires when the task completes

This inverts the current model where sub-agents inherit their parent’s full permission set. The new Claude Code isolation field supports this scoping, but it must be used deliberately by the governance layer.

5. DLP Scanning for Response Content

Data Loss Prevention scanning applies to agent outputs, not just inputs. Before any agent response is committed to disk, transmitted via MCP, or presented to the user, scan it for sensitive content.

Scan Categories

  • Secrets: API keys, tokens, passwords, private keys (regex patterns + entropy analysis)
  • Personal data: Email addresses, phone numbers, IP addresses from .planning/ files
  • IP markers: Fox Companies proprietary content that must stay in .planning/ (enforced by the existing gitignore rule)
  • Path leakage: Absolute local file paths that reveal system configuration

The existing security-hygiene skill already handles some of this. The improvement is making it a deterministic hook (PostToolUse) rather than a probabilistic skill activation.

6. Immutable Audit Log

Every agent action must be logged in an append-only audit trail that cannot be modified by the agent itself. The log records: who (agent identity), what (tool call + parameters), when (timestamp), why (task context), and outcome (success/failure/escalation).

Implementation: The beads-state skill already provides crash-recoverable state persistence. Extend it to maintain an append-only audit log per session. Each entry is content-addressed (hash-linked to the previous entry), creating a tamper-evident chain. The agent can append to the log but never modify or delete entries.

7. Cross-Model Disagreement as Security Signal

When multiple models evaluate the same skill or output and disagree, the disagreement itself is the most valuable security signal available. From arXiv:2604.03081: only 1.6% of adversarial payloads bypass all three models simultaneously.

Disagreement Response Protocol

  • Agreement (all models pass): Proceed with normal execution
  • Minor disagreement (one model flags): Log the flag, proceed with enhanced monitoring
  • Major disagreement (two+ models flag): Quarantine the artifact, trigger HITL escalation
  • Universal rejection (all models flag): Block immediately, log for post-mortem analysis

This creates a probabilistic defense where the attack surface shrinks exponentially with each additional model in the verification ensemble.
