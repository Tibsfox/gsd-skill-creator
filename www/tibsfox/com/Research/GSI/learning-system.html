The Skill-Creator’s Core Loop

gsd-skill-creator is a self-modifying system. It observes developer behavior, suggests new skills, validates those skills against safety criteria, and installs them into the active skill set. The learning loop — observe, suggest, validate, install, monitor — is the system’s primary value proposition and its greatest vulnerability. Every improvement in this module must strengthen the loop without breaking it.

Six research findings from the Artemis II program map directly to improvements in the adaptive learning system. Each addresses a specific gap identified through 190+ completed missions and 34 active skills.

1. Meta-Agentic Bootstrapping for Skill Generation

Source: p3nGu1nZz (Dr. Dimitrios Geromichalos) — embed an already-capable agent inside a rule-based world that governs success. The scaffold defines what a valid skill looks like; the agent generates candidates within those constraints.

The current skill generation process is ad hoc: the sc:observe skill watches behavior, sc:suggest proposes new skills, and the human reviews. Meta-agentic bootstrapping formalizes this into a scaffold-agent-validator pattern:

ComponentRoleGSD Implementation
ScaffoldDefines the artifact grammar for valid skillsSkill schema: YAML front matter (description ≤250 chars, triggers, globs, alwaysApply) + markdown body
AgentGenerates skill candidates from observed patternssc:suggest enhanced with structured output conforming to the scaffold schema
ValidatorExternal quality gate that the agent cannot bypasssecurity-hygiene skill + multi-model verification ensemble (from module 1)

The Self-Correcting Engine Applied

  1. Agent generates a skill candidate from observed patterns
  2. Validator scans against scaffold rules: schema conformance, DDIPE patterns, permission scope
  3. Failures produce targeted error signals: “description exceeds 250 chars”, “trigger pattern too broad”, “references unauthorized tools”
  4. Agent repairs and regenerates until all quality gates pass
  5. Validated skill enters quarantine tier (module 1) for human review before promotion

2. Recursive Composition: Scaffold-Agent-Validation Nesting

The scaffold + agent + validation pattern is itself composable. A skill that generates other skills is a meta-skill. A team of meta-skills governed by a meta-scaffold is a skill-generation pipeline.

Level 0: Individual skill execution
  scaffold: skill schema
  agent: polecat-worker
  validator: security-hygiene

Level 1: Skill generation
  scaffold: meta-skill grammar (defines what a valid skill-generator looks like)
  agent: sc:suggest (generates skill candidates)
  validator: multi-model ensemble + human review

Level 2: Skill-generation pipeline
  scaffold: pipeline grammar (defines valid pipeline configurations)
  agent: mayor-coordinator (orchestrates multiple skill generators)
  validator: ecosystem-alignment skill (checks upstream compatibility)

Each level can be composed into a single component at the next level. The recursion has a natural depth limit: Level 0 is execution, Level 1 is generation, Level 2 is orchestration. Going deeper adds complexity without proportional value.

3. Skills-in-the-Wild Degradation Calibration

Source: arXiv:2604.04323 — Agentic Skills in the Wild documents how skills degrade when deployed in real-world conditions versus controlled benchmarks. Performance drops of 15–40% are typical, caused by distribution shift between training and deployment environments.

In gsd-skill-creator, skills degrade because:

  • Codebase drift: The codebase evolves but the skill’s assumptions remain static. A skill written for v1.45 may not work correctly at v1.50.
  • Context pollution: As more skills are added, interaction effects between skills create unexpected behaviors not present when each was tested individually.
  • Model updates: Upstream model changes alter how the model interprets skill instructions, causing subtle behavioral shifts.
  • Usage pattern shift: The developer’s workflow evolves, making some skills increasingly irrelevant while creating gaps that new skills should fill.

Calibration Protocol

Run periodic calibration sweeps that test each skill against current conditions. The sc:digest skill already generates performance data. Extend it with a degradation score that triggers re-evaluation when a skill’s effectiveness drops below threshold.

4. Agent Psychometrics: Task-Level Performance Prediction

Source: arXiv:2604.00594 — Agent Psychometrics applies psychometric testing methodology (IRT, CTT) to predict agent performance on specific task types before execution.

Instead of treating all agents as equally capable on all tasks, psychometric profiling creates a capability map per agent configuration (model + skill set + MCP servers). This enables better task routing in the orchestration layer.

Psychometric DimensionWhat It MeasuresGSD Application
Task difficultyHow hard is this task for agents in general?Mode selection: easy tasks get DIRECT, hard tasks get FULL
Agent abilityHow capable is this specific agent configuration?Worker assignment: match worker skill profiles to task requirements
DiscriminationHow well does this task distinguish between capable and incapable agents?Verification: high-discrimination tasks need externalized verification
Guessing parameterProbability of correct output by chanceConfidence calibration: high-guessing tasks need stronger verification

The 190+ completed missions provide the training data for psychometric calibration. Each mission’s execution logs, verification results, and retrospective notes contribute to the capability map.

5. Error-as-Feedback Loop for Skill Improvement

Errors are not failures — they are targeted learning signals. When a skill fails or produces suboptimal output, the error signal should feed back into the skill improvement cycle rather than being discarded.

Feedback Loop Architecture

SKILL EXECUTION
  |
  +-- success --> increment success counter, update psychometric profile
  |
  +-- failure --> CLASSIFY ERROR
                    |
                    +-- recoverable: retry with modified parameters
                    |     (max 3 retries, then escalate)
                    |
                    +-- systematic: log pattern for sc:digest analysis
                    |     (recurring errors trigger skill re-evaluation)
                    |
                    +-- novel: flag for human review
                          (new error types may indicate codebase drift)

The sc:observe skill captures error events. The sc:digest skill aggregates them into patterns. The sc:suggest skill uses those patterns to propose skill modifications. This closes the feedback loop from execution back to generation.

6. Pluripotent Differentiation Model for Muse Agents

Source: Amazon + UC Berkeley STEM Agent paper (March 2026) — undifferentiated agents that self-specialize based on environment and task. Borrowed from developmental biology: stem cells differentiate into specialized cell types based on environmental signals.

The Muse Team (13+1 agents, two arcs) already implements a form of differentiation: each muse has a specific personality, domain focus, and grove assignment. The pluripotent model makes this differentiation lifecycle explicit:

StageBiological AnalogMuse Agent Behavior
PluripotentStem cellNew muse with no specialization — pure reasoning capability
CrystallizedCell fate commitmentPattern detected from 3+ successful activations in a domain
CommittedProgenitor cell10+ successes — gets priority routing for matching tasks
MatureFully differentiatedEstablished domain expert (Cedar=filter, Hemlock=CAS, Foxy=ECO, etc.)
ApoptosisProgrammed cell deathSuccess rate drops below threshold — muse reverts to pluripotent state for re-differentiation

The key insight: apoptosis is not failure — it is the system recognizing that the environment has changed and the current specialization is no longer optimal. The muse reverts to a pluripotent state and can re-differentiate based on the new environment. This prevents the skill ecosystem from accumulating “zombie skills” that were once useful but are now dead weight consuming context tokens.

Differentiation Signals

  • Task success rate: Above threshold = promote to next stage; below = trigger apoptosis
  • Context cost: Skills that consume disproportionate context for their contribution are candidates for pruning
  • Activation frequency: Skills that activate fewer than once per 10 sessions may be ghost tokens
  • Cross-skill interaction: Skills that conflict with other skills more often than they complement are differentiation failures
