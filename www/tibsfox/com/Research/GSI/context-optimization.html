The Context Budget Problem

gsd-skill-creator loads 34 skills, 57 commands, and extensive system prompts into every session. Tool definitions alone consume approximately 100K tokens at baseline. With a 200K context window, half the budget is spent before the user types their first message. Every improvement here directly increases the usable context available for actual work.

Context is the scarcest resource in agentic systems. The OOPS ecosystem alignment analysis measured the token costs of the current skill-creator configuration and identified seven concrete optimization opportunities, each with measured or estimated savings.

1. Tool Definition Token Reduction

Current state: ~100K tokens consumed by tool definitions, skill descriptions, and system prompt content at session start
Target: Reduce to ~30K through progressive loading and definition compression

Tool definitions are the largest single consumer of context budget. Each tool’s JSON schema includes parameter descriptions, type annotations, and usage examples. Many tools share common parameter patterns (file paths, glob patterns, regex) that could use shared references instead of repeated inline definitions.

Compression Strategies

2. Progressive 3-Tier Skill Loading (94% Savings)

Not all 34 skills need to be loaded at session start. A 3-tier loading strategy activates skills based on actual need rather than speculative preloading.

At session start, only 3 skills consume context instead of 34. The alwaysApply flag in YAML front matter already distinguishes always-on from triggered skills. The 94% savings estimate comes from loading ~2K instead of ~35K in skill descriptions alone.

3. Ghost Token Detection and Elimination

Ghost tokens are context entries that consume budget but provide no value to the current task. They accumulate through several mechanisms:

  • Stale system reminders: MCP server instructions loaded for servers not in active use
  • Duplicate skill listings: Some skills appear in both their native and fully-qualified forms (e.g., gsd-workflow and gsd:workflow)
  • Expired session state: Context from previous tasks that survived compaction
  • Redundant tool schemas: Deferred tools listed by name that will never be invoked in this session

A ghost token audit of the current system-reminder block would likely recover 5–15% of context budget. The sc:status skill already tracks per-skill token consumption — extend it to flag ghost entries.

4. Read Deduplication (8–30% Savings)

In long sessions, the same files are often read multiple times. Each read injects the full file content into context even if it has not changed since the last read. A read cache that tracks file content hashes can suppress duplicate reads.

Implementation

// Read dedup pseudocode
const readCache = new Map<string, { hash: string, turnId: number }>();

function shouldRead(path: string): boolean {
  const cached = readCache.get(path);
  if (!cached) return true;
  const currentHash = hashFile(path);
  return currentHash !== cached.hash;
}

The 8–30% savings range depends on session type: code review sessions (high file re-reading) see the upper bound; greenfield development sessions see the lower bound.

5. Context Rot Prevention at 25% Fill

Context rot is the degradation of model output quality as the context window fills. Research from the AGT digest and OOPS alignment both point to 25% fill as the threshold where quality begins to degrade measurably. The model starts losing track of earlier instructions, forgets constraints, and produces outputs that conflict with established context.

Prevention Strategies

  • Proactive compaction: Trigger compaction at 25% fill rather than waiting for the window to approach capacity
  • Instruction anchoring: Pin critical instructions (safety constraints, project rules) at both the beginning and end of the context window
  • Staleness scoring: Track when each context entry was last referenced; deprioritize entries with high staleness for compaction

6. Compaction Strategy Improvement (60–70% Loss Reduction)

The current PostCompact hook fires after compaction occurs, but the compaction itself is a black box that loses 60–70% of context detail. The new Claude Code v2.1.88 PostCompact hook allows injecting recovery context after compaction.

Recovery injection: After compaction fires, read .planning/STATE.md and the current phase’s PLAN.md back into context. These files contain the most critical state that compaction tends to lose. This is a form of external memory that supplements the compressed conversation history.

7. Sub-Agent Context Isolation (90.2% Improvement)

Sub-agents spawned by the skill-creator inherit the parent’s full context, even when their task requires only a fraction of it. The new Claude Code subagent fields (isolation: worktree, memory, skills, mcpServers) enable precise context scoping per sub-agent.
