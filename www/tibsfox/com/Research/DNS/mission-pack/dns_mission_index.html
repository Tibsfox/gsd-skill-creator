How to Use with GSD

  1. Open the PDF and review Stage 1 Vision — confirm scope and module boundaries match your intent.
  2. Hand the PDF to your GSD orchestrator: gsd run --mission dns_mission.pdf --profile fleet
  3. Orchestrator begins at Wave 0 (Foundation) — RFC index, RR schema, glossary stubs.
  4. Wave 1 runs three parallel tracks simultaneously (Wire+RR, Resolution+Zone, DNSSEC+Extensions).
  5. Approve the first CAPCOM gate after Wave 1 — review module drafts and scope before synthesis.
  6. Wave 2 synthesizes the attack taxonomy, interaction diagrams, and assembles the full document.
  7. Approve the second CAPCOM gate — security warden (SECURE) must sign off before publication.
  8. Wave 3 compiles the final PDF and delivers the complete reference document.

Through-Line

DNS is not a phone book. It is a proof system, disguised as a lookup table. The NS record pointing away from itself, the glue record placed in a parent it doesn't own, the DS record anchoring a child's trust in its parent's zone — these are the connective tissue of the internet. Every domain name resolved is a traversal of those connections. Every DNSSEC validation is a cryptographic proof that the connections are trustworthy. The Amiga Principle, running at planetary scale since 1987.
