Two Traditions

The phrase “object-oriented programming” was coined by Alan Kay in the late 1960s, but the technique of bundling data with the procedures that operate on it predates the term. Two languages, born within a few years of each other on opposite sides of the world, defined the two great families of OOP that every later language would inherit from.

Simula 67 (Norway, 1967) — Kristen Nygaard and Ole-Johan Dahl at the Norwegian Computing Center. Built for simulation. First language to have class, object, inheritance, and virtual procedures. Nygaard and Dahl received the 2001 ACM Turing Award for this work. Stroustrup encountered Simula at Cambridge in the mid-1970s — the experience permanently shaped his thinking. Simula was slow; the dream of “Simula speed for systems programming” became C++.

Smalltalk (Xerox PARC, 1972–1980) — Alan Kay, Dan Ingalls, Adele Goldberg. Radically different: everything is an object, dispatch by message name at runtime, no static types, live and reflective environment. Smalltalk gave us the MVC pattern, refactoring tools, and the integrated graphical IDE. Where Simula was structural, Smalltalk was behavioral.

Classes and Objects

A class in C++ is a user-defined type that bundles data members (fields) and member functions (methods) into a single named entity, with access control to enforce encapsulation. In C, a struct is a layout specifier. In C++, struct and class are the same thing, with one difference: in struct, members default to public; in class, they default to private. Convention uses struct for aggregates and class for everything else, but the compiler does not care.

class Account {
    double balance_;
public:
    explicit Account(double initial) : balance_(initial) {}
    void deposit(double amount) { balance_ += amount; }
    void withdraw(double amount) {
        if (amount > balance_) throw std::runtime_error("insufficient funds");
        balance_ -= amount;
    }
    double balance() const { return balance_; }
};

The Four Pillars

Encapsulation

Hide the inside. A class exposes a contract; the fields are implementation detail. private, protected, public. These are compile-time access controls — the bytes are still there in memory, the compiler just refuses to name them from the wrong context.

Inheritance

Reuse a base. class Dog : public Animal gives Dog everything Animal has. Single, multiple, and virtual inheritance. Public vs private inheritance models “is-a” vs “implemented-in-terms-of.”

Polymorphism

One interface, many forms. A virtual function, called through a base pointer, dispatches to the most derived override. Vtables, dynamic dispatch, dynamic_cast. Templates give you compile-time polymorphism at zero cost.

Abstraction

Model the essential. A pure-virtual class defines “anything that can do X.” C++20 concepts give you the same idea at compile time, for templates. An interface is a contract a reader can understand without reading the implementation.

The Big Five (or Six)

Every C++ class has six special member functions the compiler can generate for you:

Special MemberSignatureWhen invoked
Default constructorFoo()Foo f; and array creation
Destructor~Foo()End of lifetime, in reverse construction order
Copy constructorFoo(const Foo&)Copying from an lvalue
Copy assignmentFoo& operator=(const Foo&)a = b; where b is an lvalue
Move constructorFoo(Foo&&) noexceptConstructing from an rvalue (C++11)
Move assignmentFoo& operator=(Foo&&) noexcepta = std::move(b); (C++11)

The Rule of Zero: If your class uses only RAII-managed members (vectors, unique_ptrs, strings, your own RAII types), declare none of these. The compiler will generate them correctly.

The Rule of Five: If you must write any of them (usually because you own a resource through a raw handle), write or = default all five.

RAII — The Soul of C++

Resource Acquisition Is Initialization. Stroustrup’s own coinage, and arguably C++’s most important contribution to programming language design. The idea: tie resource lifetime to object lifetime. Acquire in the constructor, release in the destructor, and let the compiler guarantee cleanup at scope exit — even when exceptions are thrown.

class FileHandle {
    std::FILE* fp_;
public:
    explicit FileHandle(const char* path, const char* mode)
        : fp_(std::fopen(path, mode)) {
        if (!fp_) throw std::runtime_error("fopen failed");
    }
    ~FileHandle() { if (fp_) std::fclose(fp_); }

    // Rule of Five: disable copy, allow move
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;
    FileHandle(FileHandle&& other) noexcept : fp_(other.fp_) { other.fp_ = nullptr; }
    FileHandle& operator=(FileHandle&& other) noexcept {
        if (this != &other) {
            if (fp_) std::fclose(fp_);
            fp_ = other.fp_;
            other.fp_ = nullptr;
        }
        return *this;
    }

    std::FILE* get() const { return fp_; }
};

void process() {
    FileHandle f("data.txt", "r");
    // ... use f ...
    // destructor runs automatically, even if an exception is thrown
}

RAII replaces every language feature other systems have for handling cleanup: Java’s try-finally, C#’s using, Python’s with, Go’s defer. In C++, you write a type and cleanup becomes automatic everywhere that type is used. The entire standard library is built on RAII: std::vector owns its memory, std::lock_guard owns a mutex lock, std::unique_ptr owns a heap allocation, std::fstream owns a file handle.

Inheritance

A derived class inherits members from its base. The access specifier on the inheritance list controls how users of the derived class see inherited members:

  • class Derived : public Base — public members of Base remain public. Models “is-a.”
  • class Derived : protected Base — public and protected members become protected. Rare.
  • class Derived : private Base — everything from Base becomes private. Models “implemented-in-terms-of.” Often replaced by composition.
class Animal {
    std::string name_;
public:
    explicit Animal(std::string name) : name_(std::move(name)) {}
    virtual ~Animal() = default;
    virtual std::string speak() const = 0;
    const std::string& name() const { return name_; }
};

class Dog : public Animal {
    std::string breed_;
public:
    Dog(std::string name, std::string breed)
        : Animal(std::move(name)), breed_(std::move(breed)) {}
    std::string speak() const override { return name() + ": woof!"; }
};

Bases are constructed before members, in order of declaration. Members are constructed in declaration order, not the order they appear in the initializer list — a common bug source that compilers will warn about.

The Slicing Problem

Polymorphism in C++ requires references or pointers. If you assign a derived object to a value of base type, only the base subobject is copied. The derived parts are sliced off:

Circle c(2.0);
Shape s = c;        // slicing — only Shape subobject is copied
Shape& sr = c;      // OK: reference, no slicing
Shape* sp = &c;     // OK: pointer, no slicing

This is a constant source of confusion for programmers coming from Java or C#, where all objects are reference types by default.

Virtual Destructors

Rule: If you ever delete a derived object through a base pointer, the base’s destructor must be virtual. Otherwise, only the base destructor runs, derived members leak, and behavior is undefined.

class Base { public: virtual ~Base() = default; };  // CORRECT
class Bad  { public: ~Bad()  = default; };           // BAD if used polymorphically

Any class intended as a polymorphic base needs a virtual destructor. If the class is not meant to be a base, mark it final to enable devirtualization optimizations.

Multiple Inheritance and the Diamond

C++ allows a class to inherit from multiple bases. This introduces the diamond problem: if D inherits from B1 and B2, both of which inherit from A, then D has two A subobjects unless you say otherwise. virtual inheritance collapses them into one shared base:

struct B1 : virtual A {};
struct B2 : virtual A {};
struct D : B1, B2 {};  // D has ONE A subobject

Virtual inheritance has a runtime cost (an extra pointer per object) and complicates construction — D is now responsible for constructing the virtual A. It is rarely used outside of carefully designed library hierarchies. The iostream library is the famous example: iostream inherits virtually from ios_base via both istream and ostream.

Virtual Functions and vtables

A virtual member function can be overridden by derived classes. When called through a base pointer or reference, the most derived override runs. This is the classic polymorphism of OOP textbooks.

struct Shape {
    virtual ~Shape() = default;
    virtual double area() const = 0;      // pure virtual
    virtual void draw() const = 0;
};

struct Circle : Shape {
    double r;
    explicit Circle(double r_) : r(r_) {}
    double area() const override { return 3.14159 * r * r; }
    void draw() const override { /* ... */ }
};

void render(const Shape& s) {
    s.draw();             // dispatches to the actual derived class
    auto a = s.area();
}

The C++ standard does not specify how virtual dispatch is implemented, but every major compiler uses a vtable (virtual function table). Each polymorphic class has a static array of function pointers — one slot per virtual function. Each instance carries a hidden pointer (the vptr) to its class’s vtable. A virtual call compiles to: dereference vptr, index by slot, indirect call. Two memory loads and an indirect branch. The cost is fixed and well understood, but it inhibits inlining.

override and final (C++11)

  • override asserts “this overrides a virtual base function.” If the signatures don’t match, compile error. Without it, a typo silently creates a new virtual function.
  • final on a virtual function says “no further overriding allowed.” On a class, “no further inheriting allowed.”

Static Polymorphism — CRTP

A class can inherit from a template instantiated with itself. This sounds nonsensical but is a powerful form of static polymorphism known as the Curiously Recurring Template Pattern:

template <typename Derived>
struct Shape {
    double scaled_area(double k) const {
        return k * k * static_cast<const Derived*>(this)->area();
    }
};

struct Square : Shape<Square> {
    double side;
    double area() const { return side * side; }
};

Square s{3.0};
auto a = s.scaled_area(2.0);  // no virtual call; area() is inlined

No vtable, no indirection, full inlining. CRTP is used heavily in Eigen, the standard library’s iostreams, and anywhere the performance of virtual dispatch matters.

Design Patterns in C++

The Gang of Four book — Design Patterns: Elements of Reusable Object-Oriented Software (Gamma, Helm, Johnson, Vlissides, 1994) — was written with Smalltalk and C++ as its working examples. Many of its 23 patterns are idiomatic in C++:

Encapsulation via PIMPL

The Pointer to Implementation idiom hides all implementation details behind an opaque pointer, making the header immune to changes in the implementation and removing heavy includes from the public interface:

// widget.h
class Widget {
public:
    Widget();
    ~Widget();
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;
    void draw();
private:
    class Impl;                          // forward declaration only
    std::unique_ptr<Impl> pimpl_;
};

// widget.cpp
class Widget::Impl {
    // any members, any heavy includes
    SomeHeavyThing heavy_;
};

Widget::Widget() : pimpl_(std::make_unique<Impl>()) {}
Widget::~Widget() = default;             // must be defined here, not in header
void Widget::draw() { /* uses pimpl_ */ }

Exceptions and Exception Safety

C++ exceptions integrate with RAII: when an exception propagates, destructors run for every automatic object in the unwinding scope. This gives you automatic cleanup without try-finally. The language provides:

  • throw an expression of any type; conventional to throw types derived from std::exception.
  • try { } catch (...) { } to catch. catch (const std::exception& e) is the idiomatic form.
  • noexcept (C++11) — function specification that no exceptions escape. Critical for move operations in containers.
  • std::expected<T, E> (C++23) — return-value alternative for expected failures.

The exception safety guarantees (due to Dave Abrahams) classify operations by what they promise on failure:

GuaranteePromise
No-throwWill never throw. Destructors, swap, move (when noexcept).
StrongOn failure, the program state is unchanged (commit-or-rollback).
BasicOn failure, invariants are preserved but state may have changed.
NoneOn failure, anything is possible. Avoid.
