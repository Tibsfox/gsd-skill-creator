The Founding Decisions

Every feature of C++ since 1979 is a refinement of a small set of founding choices Stroustrup made at Bell Labs. Understand them and you understand why the language looks the way it does:

  • Static type system — Types are checked at compile time. The compiler knows everything it can know before runtime.
  • Class-based, not prototype-based — A class is a compile-time entity; an object is a runtime instance laid out in a known way.
  • Direct memory layout — A class is fundamentally a struct. Fields are laid out in declaration order; sizeof is meaningful; you can reinterpret memory if you accept the consequences.
  • Optional virtual dispatch — Virtual functions cost a vtable lookup; non-virtual functions cost a direct call. You pay only when you ask for polymorphism.
  • Value semantics by default — Objects live on the stack or inside other objects. Heap allocation is opt-in. Copies make new objects, not aliases.
  • Deterministic destruction — When an object goes out of scope, its destructor runs now, not whenever a garbage collector feels like it. This single decision is the seed of RAII, exception safety, and modern C++.

Types and the Type System

C++ is statically typed: every expression has a type known at compile time. It is strongly typed: implicit conversions exist but are limited, and the compiler will reject nonsensical operations. It is nominally typed: two types with the same layout are still different types unless one is an alias of the other.

Compound types

  • Arrays — int a[10];. Fixed size, decay to pointers in most contexts, no bounds checking.
  • Pointers — T*. A memory address of a T. Can be null, dangling, or valid.
  • References — T& (lvalue) and T&& (rvalue, C++11). Aliases that must be bound at declaration and cannot be rebound.
  • Functions — R(Args...). Function pointers are R(*)(Args...).
  • Classes — class and struct. User-defined types with fields and methods.
  • Unions — Shared storage across members. Unsafe by default; std::variant (C++17) is the type-safe alternative.
  • Enumerations — enum (C) and enum class (C++11, strongly-typed, scoped).

Value Categories

C++11 introduced a formal taxonomy of expression value categories. Every expression in C++ is exactly one of these:

lvalue

Has identity (you can take its address), cannot be moved from. Named variables. The expression x where x is a variable. The result of *p.

xvalue

“eXpiring value.” Has identity, can be moved from. Result of std::move(x). Result of a function returning T&&.

prvalue

“Pure rvalue.” No identity, can be moved from. Literals (42), temporaries. The result of a + b where a, b are ints.

glvalue / rvalue

Composite categories. A glvalue is any lvalue or xvalue. An rvalue is any xvalue or prvalue. The two axes are “has identity” and “can be moved from.”

These categories matter because overload resolution uses them: a function can overload on T& (takes lvalue), const T& (takes anything), and T&& (takes rvalue), and the compiler picks based on the expression’s value category. This is what makes move semantics work.

References and Pointers

Stroustrup added references to C++ in 1983 as a specific solution to a specific problem: operator overloading needed a way to pass objects by reference without the syntactic noise of pointers. They grew into the idiomatic way to pass objects in and out of functions.

FormSyntaxUse for
Raw pointerT*Non-owning observation; nullable; C-interop.
Lvalue referenceT&In/out parameters; aliases; operator overloads.
const lvalue referenceconst T&Read-only pass of large objects; accepts both lvalues and rvalues.
Rvalue referenceT&&Move-from sources; overload resolution for temporaries.
Forwarding referenceT&& in template deductionPerfect forwarding; preserves value category of the argument.

The rule for passing parameters in modern C++ is simple:

  • Small, trivially copyable types: pass by value.
  • Objects you only read: pass by const T&.
  • Objects you modify: pass by T& or T* (the latter signals “may be null”).
  • Objects you want to take ownership of: pass by value (so the caller can either copy or move in).

const and const-correctness

const is C++’s contract-in-the-type-system. A const variable cannot be modified after initialization. A const member function promises not to modify the object. A const T& parameter promises the function will not change what it’s handed. Const-correctness is the discipline of applying const everywhere it could possibly apply:

class Matrix {
    std::vector<double> data_;
    std::size_t rows_, cols_;
public:
    // const member function: does not modify *this
    double at(std::size_t r, std::size_t c) const {
        return data_[r * cols_ + c];
    }

    // non-const overload: returns a mutable reference
    double& at(std::size_t r, std::size_t c) {
        return data_[r * cols_ + c];
    }

    // const parameter: caller guaranteed we won't touch it
    void copy_from(const Matrix& other);
};

Const-correctness propagates: if copy_from takes const Matrix&, it can only call const member functions on that object. This lets the compiler catch a large class of errors at build time and gives readers a reliable guide to what each function can and cannot do.

constexpr — Compile-Time Computation

C++11 introduced constexpr: the declaration that a function or variable can be evaluated at compile time. C++14, C++17, C++20, and C++23 have steadily loosened the restrictions until today almost anything can be constexpr, including containers and algorithms.

KeywordStandardMeaning
constC++98Value cannot change after initialization; may or may not be a compile-time constant.
constexprC++11Can be evaluated at compile time if all inputs are; required when used as a compile-time constant.
constevalC++20Immediate function — must be evaluated at compile time. Error if called with runtime arguments.
constinitC++20Forces static initialization; guarantees no dynamic init order fiasco.
 
constexpr int factorial(int n) {
    return n <= 1 ? 1 : n * factorial(n - 1);
}

constexpr int fact5 = factorial(5);  // computed at compile time, becomes literal 120
static_assert(fact5 == 120);          // checked at compile time

Scope, Linkage, and Namespaces

A name in C++ lives somewhere: block scope, function scope, class scope, namespace scope, or the special case of global scope. It has a linkage: internal (private to the translation unit), external (visible across translation units), or none. And it lives inside a namespace, an addressable region of the name space.

namespace physics {
    namespace detail {
        constexpr double speed_of_light_m_per_s = 299'792'458.0;
    }
    double relativistic_energy(double rest_mass);
}

using namespace physics;           // don't do this in headers
using physics::relativistic_energy; // better
namespace ph = physics;            // namespace alias

The std namespace contains the entire standard library. using namespace std; is convenient in small programs and considered poor style in large ones — it pulls everything into the current scope and creates ambiguity with user code.

Operator Overloading

C++ allows you to redefine the meaning of most operators when one or both operands are user-defined types. You cannot invent new operators, you cannot change precedence or arity, and you cannot overload a handful (::, ., .*, ?:, sizeof, typeid). Everything else is fair game.

struct Vec2 {
    double x, y;

    Vec2& operator+=(const Vec2& rhs) {
        x += rhs.x; y += rhs.y; return *this;
    }
    double operator[](std::size_t i) const { return i == 0 ? x : y; }
    bool operator==(const Vec2&) const = default;    // C++20: compiler-generated
    auto operator<=>(const Vec2&) const = default;    // C++20: spaceship, all 6 relations
};

// Non-member binary operator — avoids asymmetry
Vec2 operator+(Vec2 lhs, const Vec2& rhs) { return lhs += rhs; }

The spaceship operator <=> (C++20) is the most recent addition. A single = default declaration gives you all six comparison operators (==, !=, <, <=, >, >=) with the semantics you almost always want.

The Preprocessor

C++ inherited the C preprocessor, which runs before the compiler sees your code. It is a textual substitution engine that predates the language’s type system and knows nothing about scope or namespaces. Its sins are famous:

  • #define MAX 100 leaks across headers and ignores scope.
  • #define SQUARE(x) x*x breaks with SQUARE(1+2).
  • #include <windows.h> defines min and max as macros, breaking std::min.
  • Conditional compilation (#ifdef) lets you write code that different compilers see differently.

Modern C++ replaces most preprocessor uses with real language features: constexpr for constants, inline functions and templates for function-like macros, if constexpr for conditional code generation, and C++20 modules to replace #include entirely.

The Big Five (or Six)

Every class in C++ has six special member functions the compiler can generate: default constructor, destructor, copy constructor, copy assignment, move constructor, move assignment. The rules about when the compiler generates each, and how one user declaration affects the others, are intricate. The modern guidance is the Rule of Zero and the Rule of Five:

Rule of Zero: If your class uses only RAII-managed members (vectors, unique_ptrs, strings, your own RAII types), you need to declare none of the special members. The compiler will do the right thing.

Rule of Five: If you must write any of the destructor, copy constructor, copy assignment, move constructor, or move assignment (usually because you own a resource through a raw handle), you should write or explicitly default all five.

Lambda Expressions

C++11 introduced lambdas: anonymous function objects with capture. They are the most immediately visible marker of modern C++, and they enabled ergonomic use of the STL algorithms that C++98 buried under clunky functor boilerplate.

auto counter = 0;
std::vector<int> numbers{1, 2, 3, 4, 5};

// [&] captures everything by reference
std::for_each(numbers.begin(), numbers.end(), [&](int n) {
    if (n % 2 == 0) ++counter;
});

// [=] captures by value
auto make_adder = [=](int base) {
    return [base](int x) { return x + base; };
};

auto add_10 = make_adder(10);
auto result = add_10(5);  // 15

// Generic lambda (C++14)
auto print = [](const auto& x) { std::cout << x << '\n'; };

// Template lambda (C++20)
auto size_of = []<typename T>(const T&) { return sizeof(T); };

Lambdas compile to anonymous function objects. A lambda with no captures decays to a plain function pointer and is as efficient as the equivalent free function. Captures become data members. This is the zero-overhead principle at work: the abstraction is free.

Range-for, Structured Bindings, if constexpr

Three C++11/17 features that eliminate huge amounts of boilerplate:

// Range-for (C++11) — works with any type that has begin()/end()
for (const auto& x : container) { /* ... */ }

// Structured bindings (C++17) — destructure tuples, pairs, arrays, structs
for (const auto& [key, value] : map) { /* ... */ }
auto [x, y, z] = some_point;

// if constexpr (C++17) — compile-time branch; only the taken branch is compiled
template <typename T>
std::string stringify(const T& value) {
    if constexpr (std::is_integral_v<T>) {
        return std::to_string(value);
    } else if constexpr (std::is_convertible_v<T, std::string>) {
        return std::string(value);
    } else {
        return "";
    }
}
