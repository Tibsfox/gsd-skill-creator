C++ sits at the pivot of the escalation ladder — high enough to model the problem domain in classes, low enough to count cache lines. Every other language in this table is defined, in part, by its answer to some question C++ already has an answer for.

The Escalation Ladder Revisited

From the PRL research project’s opening quote, the sysadmin’s escalation ladder:

  1. Bash — pipes, loops, one-liners. Glue at the shell level.
  2. Perl / Python — dynamic languages. Real data structures, proper regex, libraries. See PRL, SNK.
  3. C — systems programming. No runtime, close to the metal. See CEE.
  4. C++ — where objects meet the metal. Full OOP, generics, templates, zero-overhead abstractions, and everything C gives you.
  5. Assembly — when C++ is too abstract. Boot loaders, interrupt handlers, SIMD optimization.
  6. Machine Code / Microcode — patching binaries, firmware.
  7. Mathematics — define the problem formally.
  8. Philosophy — what is the problem?
  9. Ask Your Sysadmin — “Hey Foxy? Can you fix this? ^.^”

C++ is the rung where two questions meet: “can I model this problem as objects?” and “can I afford any runtime overhead?” If both answers matter, C++ is often the only option that doesn’t force you to compromise one for the other.

Same Program, Different Languages — A Class with Polymorphism

C++

#include <memory>
#include <vector>
#include <iostream>

class Shape {
public:
    virtual ~Shape() = default;
    virtual double area() const = 0;
};

class Circle : public Shape {
    double r_;
public:
    explicit Circle(double r) : r_(r) {}
    double area() const override { return 3.14159 * r_ * r_; }
};

int main() {
    std::vector<std::unique_ptr<Shape>> shapes;
    shapes.push_back(std::make_unique<Circle>(2.0));
    for (const auto& s : shapes)
        std::cout << s->area() << '\n';
}

Rust (trait-based, no inheritance)

trait Shape {
    fn area(&self) -> f64;
}

struct Circle { r: f64 }
impl Shape for Circle {
    fn area(&self) -> f64 { 3.14159 * self.r * self.r }
}

fn main() {
    let shapes: Vec<Box<dyn Shape>> = vec![Box::new(Circle { r: 2.0 })];
    for s in &shapes { println!("{}", s.area()); }
}

Java

abstract class Shape { abstract double area(); }
class Circle extends Shape {
    private final double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return 3.14159 * r * r; }
}

public class Main {
    public static void main(String[] args) {
        List<Shape> shapes = List.of(new Circle(2.0));
        for (Shape s : shapes) System.out.println(s.area());
    }
}

Python

class Shape:
    def area(self): raise NotImplementedError

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return 3.14159 * self.r * self.r

shapes = [Circle(2.0)]
for s in shapes:
    print(s.area())

C vs C++ — The Substrate and Its Refinement

C++ is famously described as “a better C.” The truth is more interesting: the two languages share a substrate (syntax, preprocessor, type system foundations, ABI) but diverge wildly in their idioms:

ConceptC idiomC++ idiom
Memory managementmalloc / freeRAII, smart pointers, containers
EncapsulationOpaque structs in header, static functions in .cprivate class members
PolymorphismFunction pointer tables in structsVirtual functions + vtables (or templates)
GenericsMacros, void*, _Generic (C11)Templates with type safety
Error handlingReturn codes, errnoExceptions or std::expected
String handlingchar*, strcpy, manual lengthstd::string, std::string_view
Dynamic arraysPointer + size + capacity, manual reallocstd::vector
Hash mapsNone in the standard library; roll your own or use a librarystd::unordered_map

C++ vs Rust — The Modern Challenger

Rust (Mozilla, 2010) is the language C++ is most actively measuring itself against in the 2020s. The key Rust innovation is the borrow checker — a compile-time alias-and-lifetime analysis that prevents data races and use-after-free at the type level, without a runtime. C++ has no equivalent; its answer is discipline plus RAII plus sanitizers plus Safety Profiles.

Where Rust wins: the borrow checker rules out entire categories of bugs; Cargo makes dependency management painless; error messages are better; the culture is safety-first by default.

Where C++ still wins: 40+ years of libraries, ABI stability, template metaprogramming expressiveness, multi-paradigm flexibility (you can write C-style code when you want to), presence in every major industry and codebase, and the ability to optimize at the level of individual cache lines.

Many projects are starting as interop: new Rust code at the edges of existing C++ codebases (Firefox, Chromium, Linux kernel since 6.1, parts of Windows kernel).

C++ vs Python — The Scientific Stack Mirror

Python dominates data science, machine learning, and scientific computing — but the heavy lifting is C++. NumPy’s ndarray is C. SciPy wraps Fortran and C. PyTorch and TensorFlow are C++ libraries with Python bindings. This two-language pattern is so pervasive it has a name: “the two-language problem.” Python gives you interactivity, exploration, and glue; C++ gives you the kernels that actually run the work.

See SNK — The Snake (Python) for the full story on how Python built on top of C and C++.

“How Did They Do It in C++?” — The Cookbook Method

From the PRL research project: CPAN serves as a cookbook for Perl. You find a similar problem, read the source, adapt it. C++ has the same practice through:

  • Boost (boost.org) — decades of idiomatic C++, the library incubator for the standard
  • cppreference.com — searchable canonical reference with worked examples
  • GitHub — the world’s largest open-source C++ corpus
  • Compiler Explorer — paste code, see assembly output, verify the optimizer is doing what you expect
  • C++ Core Guidelines — Stroustrup and Sutter’s living canon of “is this idiom recommended?”

When to Reach for C++

Yes, use C++

Performance-critical work where runtime overhead matters. Game engines. Browsers. Databases. HFT. Scientific kernels. Embedded with enough ROM. When you want OOP and you want to count cache lines.

Maybe use C++

Desktop applications. System services. CLI tools that must be fast. Numerical libraries with Python bindings. Anything where the deployment story tolerates a native binary.

Probably use something else

Web services (Go, Rust, Java). Scripts (Python, Perl). Prototypes (Python). New memory-safety-critical greenfield work (Rust). Anywhere the toolchain complexity isn’t worth the performance.

Do NOT use C++

One-off data processing (Python). Shell glue (bash). A team that won’t commit to modern idioms (the language punishes partial buy-in).
