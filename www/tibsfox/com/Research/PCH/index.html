PUNCH CARDS

A Stack of Cards, A Budget, A Loop — The Amiga Principle, 60 Years Earlier
Overview Hardware Workflow The Loop Philosophy Modern AI MVP Rosetta Learn
Early general-purpose computers could only be operated by professionals, and the cost was extreme.
Data and instructions were input by punch cards. No input could be added while the program was running. The languages developed in this era are designed for minimal interaction. The workflow was: design carefully, punch cards, submit deck, wait, read printout, improve. The loop took hours. Every iteration cost money. Every mistake was paid for twice — once in the wasted run, once in the wall-clock time lost waiting to try again.

A Single Card

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
DO 10 I = 1, N 00000100

One card. One line of code. 80 columns. Columns 1-5 for a statement label. Column 6 for continuation. Columns 7-72 for the statement. Columns 73-80 for the sequence number — so you could re-sort if you dropped the deck. The corner cut in the upper right oriented the card. The printed text along the top let a human read what was punched.

A single card weighed about 0.8 grams. A deck of 2000 — a substantial program — weighed four pounds. Your program was something you carried. It existed in the world. If you lost it, it was gone. If you dropped it, you spent an hour re-sorting. If a card jammed in the reader, you had to find a replacement and punch it again from a fresh blank.

—— BLOWING YOUR STACK ——
A literal event. The vacuum belt fails. The reader ejects 600 cards across the room.
You spend the next hour sorting them by the sequence numbers in columns 73-80.
Your rubber band snapped. Your deck is now furniture decoration.

“Blowing your stack” was not always figurative. Card readers used vacuum belts and pinch rollers to feed cards at up to 1000 per minute. When a belt failed or a card caught a rough edge, the output stacker could reject cards violently — cards flying across the room, hundreds of them, the deck you spent all morning preparing turned into scattered paper.

The sequence numbers in columns 73-80 existed precisely because this happened. If you dropped a deck, or if the reader blew it, you could run the cards through a sorter and get them back in order. This was not a backup system. This was an essential part of normal operations. Programmers learned to number their cards because they knew, with certainty, that cards would eventually scatter.

Why a research project on punch cards in 2026? Because the constraints of punch card programming are returning — not in hardware, but in intelligence. An AI agent has a finite context window. A token budget is real money. Every tool call has a cost. Every iteration through the design-code-build-run-examine-improve loop burns resources. The programmer of 2026 working with AI agents has more in common with the programmer of 1965 working with card decks than with the programmer of 2015 working with unlimited cloud compute.

This is not nostalgia. This is recognition. The patterns we’re rediscovering in AI agent work were developed under the exact same pressures sixty years ago. The hardware has changed beyond recognition. The economic forces shaping craft are identical. We are not inventing new patterns — we are rediscovering old ones, because the underlying constraints have returned.

The Loop — Design, Code, Build, Run, Examine, Improve

  1. Design — Before writing code, understand the problem. In 1965: flowchart, pseudocode, desk-check. Hours of thinking before touching a coding sheet. In 2026: plan mode, task breakdown, explicit assumptions before tool calls. The discipline is identical: the cost of running without thinking is too high.
  2. Code — Transcribe the design into the medium. In 1965: fill out a coding sheet with pencil, column by column, letter by letter. In 2026: write the code change as a minimum-viable diff. The act of writing is secondary to the thinking that preceded it.
  3. Build — Convert source into executable form. In 1965: walk to the keypunch room, punch 400 cards, rubber-band the deck, carry it to the submission window. In 2026: run the build, wait for tests to green, stage the commit. The physical cost has collapsed. The ritual persists.
  4. Run — Submit to the machine. In 1965: the operator queued your deck, scheduled it, ran it overnight. In 2026: the agent dispatches a tool call and waits for the response. The agent is a programmer, the tool is a machine, the user is the shift supervisor.
  5. Examine — Read the output carefully. In 1965: green-bar printout, line by line, looking for the error code on page 47. In 2026: the tool output text, line by line, looking for the assertion that failed. Reading is the primary skill. It always was.
  6. Improve — Make the minimum necessary change. In 1965: punch one new card, swap it into the deck, resubmit. In 2026: edit one function, rerun the failing test. Do not “refactor while you’re in there.” Do not “fix some other stuff you noticed.” Minimum viable change. Iterate.

This is the loop. It has not changed in 60 years. The turnaround time has collapsed from hours to seconds, but the rhythm — the six steps, the discipline of each step, the compounding of learning across iterations — is the fundamental engineering rhythm of doing good work under real constraints.

The Thesis

Programming under extreme constraints produces a different kind of thinking. When every run costs money and every cycle takes hours, you think before you act. When you can’t iterate quickly, you design carefully. When machine time is expensive and error is expensive, you develop habits that modern programmers — with their free iterations and fast feedback — have largely lost.

And now, in the era of AI agents, context windows, token budgets, and model orchestration, those constraints are returning. Not because the machine is slow, but because the intelligence is expensive. A context window is finite. A token budget is real money. An agent’s attention span is measured in tool calls.

We are re-learning what the punch card era knew.

The Amiga Principle

The Commodore Amiga 1000 (1985) had 256KB of RAM. In that space, it ran a preemptive multitasking operating system (Exec), drove a 32-color graphics chip (Denise), rendered four-channel stereo audio (Paula), handled floppy disks and networking, and provided a full GUI (Intuition). Modern software running on 64GB machines often does less.

The Amiga principle: constraints force careful design. Careful design produces extraordinary results. Modern AI agents need to rediscover this. A 200K context window is a feast compared to 256KB, but it’s a constraint nonetheless. The agent that treats it as infinite produces bloated, confused output. The agent that treats it as scarce produces focused, effective output.

This is not about nostalgia or artificial scarcity. It is about recognizing that real limits produce real craft. The Amiga did extraordinary things because its designers had no choice but to do more with less. The same discipline applied to AI agents — token budgets as real budgets, context as precious, every tool call deliberate — produces the same quality of outcome. Constraints are not obstacles. They are the forcing function that makes craft possible.
