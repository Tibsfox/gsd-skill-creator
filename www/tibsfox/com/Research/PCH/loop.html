The Six-Step Loop

C DESIGN → CODE → BUILD → RUN → EXAMINE → IMPROVE
Overview Hardware Workflow The Loop Philosophy Modern AI MVP Rosetta Learn
The rhythm is eternal. The hardware changes, the turnaround collapses from hours to seconds, the currency shifts from dollars to tokens — but the six steps are the same in 1965 as in 2026. This is the fundamental engineering rhythm of doing good work under real constraints.

The Loop in Detail

  1. DESIGN — Think before you act. Understand the problem. Plan the approach. Identify edge cases. Write down test inputs and expected outputs. Check assumptions. This phase produces no code and no output — and it is the most valuable phase of the loop.
  2. CODE — Transcribe the design. The act of writing code should be short relative to design. If coding is hard, you didn’t design enough. If coding reveals new questions, you did. Either way, the code is a transcription, not an exploration.
  3. BUILD — Turn source into something runnable. Compile, bundle, package. Fix syntax errors. This is a mechanical step, not a creative one. If build reveals fundamental problems, something went wrong in design or code.
  4. RUN — Submit to the machine. Execute. Wait for results. Do not try to predict the output — that was the design phase. Do not try to intervene — the machine is running now.
  5. EXAMINE — Read the output carefully. This is the second-most-valuable phase. Most people do it badly. They skim for the answer. They miss warnings. They ignore timing data. The careful programmer reads every line, notes every anomaly, and catalogs everything they didn’t expect.
  6. IMPROVE — Make the minimum necessary change. One bug per iteration. Do not “clean up while you’re in there.” Do not “fix some other stuff you noticed.” Minimum viable change. Return to step 1 if the change reveals new design questions.

Compound Interest of Deliberate Iteration

If each iteration teaches you 10% more about the problem, after 10 iterations you understand 2.6x more. If each iteration teaches you 30% more, after 10 iterations you understand 14x more. The quality of iteration matters more than the quantity.

The punch card programmer iterated slowly but each iteration was high-quality: designed carefully, examined thoroughly, improved minimally. The modern REPL programmer iterates quickly but often at low quality: change something, run it, see what happens, repeat. Ten high-quality iterations outperform one hundred low-quality ones.

The lesson: iterate deliberately, not reactively. Make each cycle count. Extract maximum learning from each printout, each test run, each tool call.

The Loop in AI Agent Work

Modern AI agents running on token budgets rediscover the loop. An agent that iterates carelessly burns through its context budget quickly. An agent that designs first, verifies assumptions, and makes deliberate changes gets more done per token. Watching a well-designed AI agent work is like watching a 1965 programmer: you can see the discipline of thinking before acting, because the cost of not thinking is real.

The gsd-skill-creator project is built around this realization. Every phase has explicit design (discuss), plan (plan), execute, verify, and improve steps. The rhythm is enforced at the framework level. Skill activations, token budgets, checkpoint commits, and structured verification are all ways of forcing the loop to run deliberately.

Why the Rhythm Survives

The six-step loop has survived 70 years of hardware change because it matches the physical reality of learning under constraints. You cannot learn without observing (examine). You cannot act usefully without planning (design). You cannot improve without changing (improve). You cannot change safely without verifying (run). The steps are not arbitrary — they are the minimum set of operations needed to compound learning across iterations.

Hardware changes. The loop stays. Whether you’re carrying a 400-card deck to a submission window in 1965, or dispatching a coding agent with a 200K token budget in 2026, the rhythm is the same. Design. Code. Build. Run. Examine. Improve.
