Minimum Viable Program

C ERIC RIES DID NOT INVENT MVP. THE IBM 704 DID.
Overview Hardware Workflow The Loop Philosophy Modern AI MVP Rosetta Learn
Eric Ries wrote The Lean Startup in 2011 and popularized Minimum Viable Product. But the concept was forced on punch card programmers by economics fifty years earlier. Incremental development wasn’t a methodology. It was survival.

The MVP Before It Had a Name

You did not write a 5000-line program all at once in 1965. You wrote a 50-line skeleton, submitted it, saw it compile, added a subroutine, submitted it, saw it run, added real data, submitted it. Each submission verified one assumption. Each successful submission became the foundation for the next.

The economic logic was inescapable: a 5000-line program submitted all at once had 5000 potential failure points. If any one of them failed — a syntax error in card 847, a runtime error in a subroutine, a JCL error in the job header — you got a failed run and lost a full turnaround cycle. A 50-line skeleton had 50 potential failure points. Submit the skeleton first. Fix the skeleton. Then add the next 50 lines on top of a verified-good foundation. Each step was small enough to debug in a single turnaround.

The Hello World Discipline

Before writing your real program, you wrote Hello World. Then Hello World With Input. Then Hello World With A Loop. Each was a complete program you could submit. Each verified one assumption. The culture of “verify the environment before writing business logic” came directly from the punch card era.

C     FIRST SUBMISSION: VERIFY THE COMPILER AND JCL
      PROGRAM HELLO
      WRITE (6,10)
   10 FORMAT (1H ,'HELLO, WORLD')
      STOP
      END

If this ran and produced “HELLO, WORLD” on the printout, you knew: the compiler was working, the JCL was correct, the operator had mounted the right tape, the output device was connected, the card reader had processed your deck successfully. None of these were things you could take for granted. Each was a potential failure point you had just eliminated.

C     SECOND SUBMISSION: VERIFY ARITHMETIC AND VARIABLES
      PROGRAM ADD
      INTEGER I, J, K
      I = 5
      J = 3
      K = I + J
      WRITE (6,10) I, J, K
   10 FORMAT (1H ,I2,' + ',I2,' = ',I3)
      STOP
      END

If this ran and produced 5 + 3 = 8, you knew the arithmetic was working and variables were being stored correctly. Only then did you write the real program on top of a verified foundation.

Load-and-Go Compilers

Some compilers were designed specifically for the student workflow where each run was painful:

CompilerOriginInnovation
WATFORUniversity of Waterloo, 1965Waterloo FORTRAN: load-and-go compilation. Fast compile, aggressive error recovery, keep running even after errors.
WATFIVWaterloo, 1968Successor to WATFOR. The “V” was a joke (V = 5 in Roman numerals, and this was FORTRAN V).
PL/CCornell, 1970Student PL/I compiler. Aggressive error repair — would guess at the intended fix and keep compiling.
SPITBOL1971Fast SNOBOL4 compiler for interactive use.

These were designed for the student workflow where each run was painful. The design goal was cheap feedback on small programs, so students could iterate more often within a class period. The lesson: make feedback cheap where you can, so constraint can focus where it must.

The Rules of Incremental Submission

  1. Small decks. A 100-card program is manageable. A 1000-card program is risky. A 10,000-card program is a project. Keep the deck you’re actively modifying small. Split larger programs into subroutines that can be developed and tested separately.
  2. One bug per run. You cannot debug ten things at once. You fix one bug, resubmit, verify the fix, then fix the next one. Fixing two bugs at once means if the run fails, you don’t know which fix caused the failure.
  3. Skeleton first. Your first submission is the smallest thing that can possibly run. Test the skeleton. Then add one feature. Then another. Every addition rests on a verified foundation.
  4. Chesterton’s Fence. Never remove code you don’t understand. The cost of accidentally breaking something that was working is too high. If you don’t know why a piece of code is there, leave it until you do.
  5. Backup decks. Keep the last working version. If a change breaks things, you can fall back. The reproducer (IBM 519) existed partly for this reason: it could duplicate a deck so you had a spare.
  6. Plan the next submission before you punch. Know what you’ll change before you walk to the keypunch. Don’t arrive at the keypunch and then decide. Decide at your desk, walk to the keypunch with a specific list of card replacements.

MVP in Modern AI Agent Work

The same pattern applies, unchanged, to AI agent orchestration. An agent asked to “implement feature X, write tests, update documentation, and open a PR” in one prompt will waste enormous budget re-deriving the environment from scratch. The same task framed as three submissions — “confirm the test suite runs on this branch,” then “report on the existing validation patterns,” then “implement the new feature using those patterns” — uses similar total tokens but distributes them across verified-good baselines, each of which constrains the agent to work within confirmed facts rather than assumptions.

The 2026 Equivalent of Hello World

# First verification: is the environment right?
"Run `pwd && git status && npm test --silent 2>&1 | tail -5` and report."

# Second verification: can I read the relevant files?
"Read src/validation/skill-validation.ts (all of it) and
 summarize its structure in under 200 words."

# Third verification: do I understand the test patterns?
"Find the three most similar existing validation rules to the one
 we want to add. For each, report the rule name, file location,
 and test location."

# Only now: the real work.
"Implement the new maxLineCount rule following the pattern of
 the three examples you identified. Add a test for it in the
 same style. Run the test suite to confirm nothing regressed."

Four submissions where one would have been attempted. Similar total token cost, vastly better success rate, and the agent’s final action rests on three verified foundations: the environment is good, the file is understood, the patterns are known. This is the punch card era rediscovered, unchanged, applied to a 2026 tool.

Agile Before Agile

Kent Beck formalized Extreme Programming in the late 1990s. The Agile Manifesto was signed in 2001. Eric Ries’s The Lean Startup came in 2011. But the tight loop of “write the smallest thing, submit, observe, improve” had been forced on punch card programmers by economics 40 years earlier. The names changed. The practice did not.

What Agile did was make explicit, and give vocabulary to, disciplines that the punch card era had enforced through economics. When the economics went away in the REPL era, those disciplines atrophied, and Agile was the response: an attempt to re-impose the discipline by cultural agreement when no economic force was making it mandatory.

The AI agent era is now re-imposing the discipline through economics, exactly as the punch card era did. This time the currency is tokens instead of dollars, and the turnaround is seconds instead of hours, but the shape of the forcing function is identical. Careless iteration is expensive. Careful iteration is cheap. The market, once again, is teaching the lesson.

The lesson that compounds across all three eras: when you cannot afford to be wrong, you become careful. When you cannot afford to be careful, you become a craftsman. The craft is what survives the era. The economics that forced the craft are just the scaffolding.
