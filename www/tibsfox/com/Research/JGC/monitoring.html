You cannot tune what you do not measure. Every successful GC tuning story begins with a GC log, a Flight Recording, or a heap dump — never with a Stack Overflow flag.

The Observability Stack

A production JVM should have at least three observation layers running continuously, and several more available on demand:

Always-on

Unified GC logs, JFR continuous recording (default template), Micrometer metrics exposed to Prometheus, JMX connection for on-call debugging.

On demand

JFR profile recording, async-profiler, heap dumps, thread dumps, NMT baseline + diff. Triggered by incident response.

Post-mortem

Eclipse MAT for heap dumps, JDK Mission Control for JFR files, FastThread for thread dumps, GCEasy / GCViewer for GC logs.

Business view

Grafana dashboards on the Prometheus metrics, alerts on p99 pause time, heap usage trend, allocation rate spikes, class loader growth.

GC Logs — The Single Most Valuable Data Source

GC logs are free, low-overhead, and contain everything a tuner needs. Every production JVM should write them.

Java 9+ Unified Logging

-Xlog:gc*=info:file=gc.log:time,uptime,tags:filecount=10,filesize=10m

Breakdown:

  • gc*=info — all GC tags at info level
  • file=gc.log — output file (use a persistent path in production)
  • time,uptime,tags — prefix decorators
  • filecount=10,filesize=10m — rotate 10 files, 10 MB each

Useful tags to enable separately:

TagWhat it shows
gcPause summaries, cause, duration
gc+heapHeap sizes before/after, generation sizes
gc+ageTenuring distribution
safepointSafepoint timings — catches TTSP issues
gc+ergoErgonomics decisions (why G1 chose what it did)
gc+phasesSub-phase timings inside a pause
gc+refReference processing time

Pre-Java 9 (legacy, still in use)

-Xloggc:gc.log -XX:+PrintGCDetails -XX:+PrintGCDateStamps
-XX:+PrintGCApplicationStoppedTime
-XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=10 -XX:GCLogFileSize=10M

Analysis tools

  • GCViewer — open-source graphical analyzer (Java Swing)
  • GCEasy.io — free web-based analyzer, upload a log file
  • GCPlot — SaaS analyzer with dashboards
  • Universal GC Log Analyzer (IBM, for OpenJ9)
  • gc-log-to-csv.py — roll-your-own with Python + pandas + matplotlib

Java Flight Recorder (JFR)

Originally a BEA/Oracle JRockit commercial feature, JFR was open-sourced in Java 11 (JEP 328) and is now the lowest-overhead production profiler on any managed runtime. Roughly 1% CPU overhead for the default template — you can leave it running in production continuously.

Starting a recording

# Start a 60-second profile recording
jcmd <pid> JFR.start name=foo settings=profile duration=60s filename=recording.jfr

# Stop manually
jcmd <pid> JFR.stop name=foo

# Dump a currently-running recording
jcmd <pid> JFR.dump name=foo filename=snapshot.jfr

# List active recordings
jcmd <pid> JFR.check

Always-on continuous recording

-XX:StartFlightRecording=disk=true,dumponexit=true,filename=recording.jfr,\
    maxsize=1G,maxage=12h,settings=default

JFR templates

  • default — low overhead, always-on-safe
  • profile — more events, more detail, short-term use
  • Continuous, Profiling — named templates in $JAVA_HOME/lib/jfr
  • Custom .jfc XML files for specific event sets

Viewing JFR files

  • JDK Mission Control (JMC) — the canonical GUI viewer. Now a standalone download from Adoptium or Oracle.
  • IntelliJ IDEA Ultimate profiler — reads JFR files directly
  • jfr command-line tool — ships with the JDK; jfr print, jfr summary, jfr assemble

Custom JFR events

import jdk.jfr.*;

@Name("com.example.OrderPlaced")
@Label("Order Placed")
@Category("Business")
public class OrderPlacedEvent extends Event {
    @Label("Order ID") String orderId;
    @Label("Amount")   double amount;
}

var event = new OrderPlacedEvent();
event.orderId = "ABC123";
event.amount = 99.99;
event.commit();

JMX (Java Management Extensions)

JMX has been in the JVM since Java 5 and exposes MBeans for reading and controlling the running VM. Built-in MXBeans under java.lang:

MXBeanPurpose
java.lang:type=MemoryHeap and non-heap usage
java.lang:type=GarbageCollector,name=*Per-collector count and time
java.lang:type=MemoryPool,name=*Individual pools (Eden, Survivor, Old, Metaspace, Code Cache)
java.lang:type=ThreadingThread counts, CPU time, deadlocks
java.lang:type=ClassLoadingClasses loaded, unloaded, total
java.lang:type=RuntimeUptime, VM name, version, JVM flags
java.lang:type=OperatingSystemHost-level stats (load, RAM, CPU)

Exposing JMX

# Local / development only — no auth, no SSL
-Dcom.sun.management.jmxremote.port=9010
-Dcom.sun.management.jmxremote.authenticate=false
-Dcom.sun.management.jmxremote.ssl=false

# Production — SSL + password file + access file
-Dcom.sun.management.jmxremote.port=9010
-Dcom.sun.management.jmxremote.ssl=true
-Dcom.sun.management.jmxremote.password.file=/etc/jmx/jmxremote.password
-Dcom.sun.management.jmxremote.access.file=/etc/jmx/jmxremote.access

Prometheus + Grafana

The free open-source observability stack of choice for JVMs:

Useful PromQL queries

# Heap usage fraction
sum(jvm_memory_used_bytes{area="heap"}) / sum(jvm_memory_max_bytes{area="heap"})

# GC pause p99 over 5 minutes
histogram_quantile(0.99, rate(jvm_gc_pause_seconds_bucket[5m]))

# Allocation rate (MB/sec)
rate(jvm_gc_memory_allocated_bytes_total[1m]) / 1e6

# GC overhead (fraction of time spent in GC)
rate(jvm_gc_pause_seconds_sum[5m])

# Thread count trend
jvm_threads_live_threads

# Old gen fill rate
rate(jvm_gc_live_data_size_bytes[10m])

Alert rules

groups:
- name: jvm
  rules:
  - alert: JvmHeapHigh
    expr: sum(jvm_memory_used_bytes{area="heap"})
        / sum(jvm_memory_max_bytes{area="heap"}) > 0.9
    for: 10m
    annotations:
      summary: "JVM heap > 90% for 10 minutes"

  - alert: JvmGcPauseP99High
    expr: histogram_quantile(0.99,
            rate(jvm_gc_pause_seconds_bucket[5m])) > 0.2
    for: 5m
    annotations:
      summary: "GC pause p99 > 200 ms"

  - alert: JvmAllocationRateHigh
    expr: rate(jvm_gc_memory_allocated_bytes_total[1m]) > 1e9
    for: 5m
    annotations:
      summary: "Allocation rate > 1 GB/sec"

JMX Exporter

For apps that don’t have Micrometer integration, the jmx_exporter Java agent converts JMX MBeans to Prometheus metrics:

java -javaagent:./jmx_prometheus_javaagent.jar=9404:config.yaml \
     -jar app.jar

async-profiler

Andrei Pangin’s sampling profiler. Low overhead, wall-clock profiling, flame graph output, no JVM agent bugs. Uses Linux perf events under the hood.

# Profile for 30 seconds, output flame graph
./profiler.sh -d 30 -f flame.html <pid>

# Allocation profiling
./profiler.sh -e alloc -d 30 -f alloc.html <pid>

# Lock contention profiling
./profiler.sh -e lock -d 30 -f locks.html <pid>

# Attach at startup
java -agentpath:./libasyncProfiler.so=start,event=cpu,file=profile.jfr -jar app.jar

OpenTelemetry Java Agent

Auto-instrumentation for 100+ libraries — HTTP clients/servers, JDBC, messaging, database drivers. Exports OTLP traces, metrics, and (experimentally) logs.

export JAVA_TOOL_OPTIONS="-javaagent:./opentelemetry-javaagent.jar"
export OTEL_EXPORTER_OTLP_ENDPOINT="http://otel-collector:4317"
export OTEL_SERVICE_NAME="my-service"

java -jar app.jar

Commercial APM

  • Datadog APM — auto-instrumentation, distributed traces, JVM dashboards, profiler
  • New Relic APM — similar
  • Dynatrace OneAgent — auto-discovery, AI anomaly detection
  • AppDynamics — Cisco’s APM

Pros: turnkey. Cons: cost, lock-in, agent overhead. Use when the open-source stack is more maintenance burden than value.
