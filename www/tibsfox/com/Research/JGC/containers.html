Running the JVM in a container is not the same as running it on bare metal. The rules about memory, CPU, and lifecycle are different. Ignoring that difference is the single most common source of “our Java pods keep getting OOMKilled” in 2026.

The Container-Aware JVM

Before Java 8u131, the JVM ignored cgroups entirely. It read the host machine’s total RAM and CPU count and sized itself accordingly. A Java container on a 64-core host saw 64 cores and allocated GC threads for all of them — even when the cgroup limited it to 1 core. Heap auto-sizing pulled the host RAM, not the cgroup limit. The result was predictable: containers ran out of memory constantly.

-XX:+UseContainerSupport (added in 8u191, default in Java 10+) tells the JVM to read cgroup limits for RAM and CPU. In Java 15+, cgroup v2 support is also solid. Every production JVM in a container should have this default-on.

The New Sizing Flags

Modern container-aware sizing uses percentages of the cgroup limit, not absolute sizes:

FlagDefaultPurpose
-XX:InitialRAMPercentage=N1.5625%Initial heap as fraction of cgroup memory limit
-XX:MaxRAMPercentage=N25%Max heap as fraction of cgroup memory limit
-XX:MinRAMPercentage=N50%Min heap on small containers
-XX:ActiveProcessorCount=NautoOverride CPU count (for edge cases)

The defaults are conservative. 25% of the container limit for the heap leaves three-quarters for Metaspace, code cache, thread stacks, direct memory, JIT compilation, and native libraries. For most Java workloads this is too conservative. Typical production starting points:

  • Small container (256 MB–1 GB): 50–60%
  • Medium container (1–4 GB): 70–75%
  • Large container (4–32 GB): 75–80%
  • Very large (> 32 GB): 80–85%

Measure. Never assume.

The Gap Between Heap and RSS

In a container, setting -Xmx2g in a 2 GB pod is a guaranteed OOMKilled. The heap is only part of the JVM’s memory footprint.

Container memory limit:   2048 MB
  - Java heap (-Xmx):     1024 MB  (50%)
  - Metaspace:             256 MB
  - Code cache:            128 MB
  - Thread stacks (50 × 1 MB):  50 MB
  - Direct memory:         256 MB  (Netty, Kafka, Lucene)
  - JIT working memory:     64 MB
  - JVM internals:         128 MB
  - Native libraries:       96 MB
  = Total JVM:            2002 MB
  + Safety margin:          46 MB
  = Container limit:      2048 MB

Every number above is measurable via NMT + monitoring. Never guess.

Kubernetes Resource Requests and Limits

For JVM pods, set requests == limits. This places the pod in the Guaranteed QoS class and eliminates memory throttling / eviction surprises. Different from stateless Go or Rust services where bursty memory is fine.

apiVersion: v1
kind: Pod
metadata:
  name: my-java-service
spec:
  containers:
  - name: app
    image: my-java-app:1.0
    resources:
      requests:
        memory: "2Gi"
        cpu: "1000m"
      limits:
        memory: "2Gi"
        cpu: "2000m"
    env:
    - name: JAVA_OPTS
      value: >-
        -XX:MaxRAMPercentage=75.0
        -XX:+UseG1GC
        -XX:+AlwaysPreTouch
        -XX:+HeapDumpOnOutOfMemoryError
        -XX:HeapDumpPath=/var/dumps/
        -Xlog:gc*=info:file=/var/log/gc.log
    volumeMounts:
    - name: dumps
      mountPath: /var/dumps
  volumes:
  - name: dumps
    persistentVolumeClaim:
      claimName: heap-dumps

Diagnosing OOMKilled

OOMKilled is NOT the same as OutOfMemoryError.

  • OutOfMemoryError — the JVM heap (or Metaspace or direct memory) exceeded its Java-managed limit. A Java exception is thrown; the app can log it; a heap dump can be written.
  • OOMKilled — the Linux kernel (via cgroup memory.oom_control) killed the JVM process because the whole container exceeded the cgroup limit. SIGKILL. No exception. No heap dump unless it was already being written.

Investigation checklist

  1. kubectl describe pod <pod> — confirms OOMKilled reason
  2. kubectl logs <pod> --previous — look at the dying process’s last output
  3. Prometheus: container_memory_working_set_bytes over the hour before the kill — was it trending up steadily, or did it spike?
  4. NMT baseline from a healthy pod — compare to the last-known state of the dying pod
  5. Check direct memory allocations: Netty, Kafka, Lucene, Chronicle, Aeron
  6. Check Metaspace growth (classloader leaks)
  7. Check thread count growth (stack memory)
  8. Check JIT code cache size

Liveness and Readiness Probes

Liveness probes should not trigger a GC. A common mistake: a liveness probe hits an endpoint that allocates a bunch of objects, which degrades the health check itself. Under load, the probe times out, Kubernetes kills the pod, and a cascade begins.

Good practice:

  • Liveness: trivial HTTP 200 endpoint that does nothing but prove the process is alive
  • Readiness: actually checks dependencies, DB connections, caches
  • Startup probe: gives the JVM time to warm up (JIT + CDS + first requests)

Swap — Turn It Off

Kubernetes strongly recommends (and historically required) swap disabled on all nodes. This is correct for Java workloads: a JVM with swap will eventually page heap memory out, and the next GC will page it back in one slow disk I/O at a time. A 200 ms GC target becomes a 60-second stall.

Modern Kubernetes (1.28+) has experimental swap support, but for JVM pods the answer remains: no swap. Size the container correctly instead.

Sidecars and Startup Time

The JVM’s traditional weak point is cold startup. A Spring Boot app can take 5–20 seconds to reach a steady state, during which it’s consuming CPU for JIT compilation while also trying to serve traffic.

Startup time optimizations

  • CDS (Class Data Sharing) — share the JDK class metadata across JVMs
  • AppCDS — share your app’s classes too; generate with -XX:ArchiveClassesAtExit
  • AOT compilation — compile hot methods ahead of time
  • GraalVM Native Image — compile the whole app to a native binary; tens of milliseconds to start
  • CRaC (Coordinated Restore at Checkpoint) — checkpoint a warm JVM, restore later with JIT state intact
  • Project Leyden — the ongoing OpenJDK work on static images and ahead-of-time compilation

Dockerfile Best Practices for JVM

FROM eclipse-temurin:21-jre-alpine AS runtime

WORKDIR /app
COPY target/app.jar .

# Non-root user
RUN addgroup -S javauser && adduser -S javauser -G javauser
USER javauser

ENV JAVA_OPTS="\
  -XX:MaxRAMPercentage=75.0 \
  -XX:InitialRAMPercentage=75.0 \
  -XX:+UseG1GC \
  -XX:+AlwaysPreTouch \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/dumps/ \
  -Xlog:gc*=info:file=/var/log/gc.log:time,uptime:filecount=5,filesize=10m \
  -XX:NativeMemoryTracking=summary"

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]

Observability in the Cluster

  • cAdvisor — built into kubelet, exposes container_memory_* metrics
  • node_exporter — node-level memory, swap, page cache
  • kube-state-metrics — pod lifecycle, OOMKilled events, restart counts
  • Prometheus + Grafana — the standard stack
  • Micrometer in the app — JVM-level detail above cAdvisor
