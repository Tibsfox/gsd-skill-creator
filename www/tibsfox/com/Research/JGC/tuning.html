“Don’t tune until you have a problem, and even then, measure first.” — the consensus advice from the HotSpot team. GC tuning is the art of trading one thing you don’t care about for another thing you do. Most developers never need to tune GC. A small minority spend their careers doing it. This document is for the second group, and for the first group when they suddenly find themselves in the second group at 3am.

The Tuning Triangle

Throughput

Fraction of CPU time spent running application code vs GC. A 98% throughput JVM spends 2% of its cycles collecting garbage. For batch workloads running for hours, 1% of throughput translates directly to money.

Latency

The longest stop-the-world pause caused by GC. For an interactive service, 500 ms is a customer-visible hiccup. For HFT, 1 ms is a career-ending event.

Footprint

Memory the JVM uses to do its job. Bookkeeping structures (remembered sets, mark bitmaps, forwarding tables) can add 10–20% overhead. Some collectors need 3× the live set.

Pick two.

The iron law. Any GC tuning decision is implicitly a vote for which corner you’ll sacrifice.

Workload Taxonomies

Batch jobs — “pauses don’t matter”

Spark jobs, ETL pipelines, report generators, nightly aggregations. Care about total elapsed time, not tail latency. Tolerate multi-second pauses for 5% overall speedup. Use Parallel GC. G1 underperforms by 5–15% on throughput-bound batch; ZGC by 10–20%.

java -XX:+UseParallelGC \
     -Xms16g -Xmx16g \
     -XX:+AlwaysPreTouch \
     -jar etl-job.jar

Latency-sensitive services

HTTP APIs, gRPC services, GraphQL servers, Spring Boot microservices. Need predictable p99 and p99.9 latencies. Use G1 or ZGC. G1 is the default; ZGC when p99 pause budget is under 100 ms.

java -XX:+UseG1GC \
     -Xms4g -Xmx4g \
     -XX:MaxGCPauseMillis=100 \
     -Xlog:gc*=info:file=gc.log:time,uptime:filecount=10,filesize=10m \
     -jar api-service.jar

Large-heap analytics

Trino, Druid, Cassandra, Elasticsearch, Solr. Multi-GB to multi-TB heaps, want throughput and predictability. Use G1 or ZGC. For heaps > 32 GB, strongly prefer ZGC.

Short-lived CLI / scripts

Command-line tools, scripts, test harnesses. Startup time dominates; GC is almost irrelevant. Use Serial or Epsilon or GraalVM Native Image.

HFT / ultra-low-latency

Order matching, risk engines, market data. Sub-millisecond requirements. Use Azul C4 (commercial) or very carefully tuned ZGC. Often combined with object pools (Disruptor, Chronicle) to minimize allocation.

Embedded / small heaps

IoT devices, sidecars, small containers. Minimize footprint. Use Serial or tuned G1.

Sizing the Heap

Heap sizing is the single most important decision. Too small: constant GC. Too big: long pauses, wasted memory.

  1. Start with live set × 2 to × 3 for young-gen-heavy workloads
  2. More for pause-sensitive G1 (needs headroom for the pause budget)
  3. Less if memory-constrained (cost or container limit)
  4. In production: -Xms == -Xmx to avoid heap resizing pauses
  5. In production: -XX:+AlwaysPreTouch to force page allocation at startup (reveals OOM early, reduces first-request latency)

Counterintuitively: oversizing the heap can make GC pauses worse. More memory to scan, longer marks, larger remembered sets.

Flags Every Tuner Should Know

Essential flags (every production JVM)

# Heap size
-Xms4g -Xmx4g

# Pre-touch pages at startup
-XX:+AlwaysPreTouch

# Thread stack size
-Xss1m

# Metaspace cap
-XX:MaxMetaspaceSize=256m

# Code cache
-XX:ReservedCodeCacheSize=256m

# Direct memory cap
-XX:MaxDirectMemorySize=1g

# Crash on OOM (fail fast)
-XX:+ExitOnOutOfMemoryError

# Heap dump on OOM (for diagnosis)
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heap-%p.hprof

# Enable NMT for later diagnosis
-XX:NativeMemoryTracking=summary

# Unified GC logging (Java 9+)
-Xlog:gc*=info:file=gc.log:time,uptime,tags:filecount=10,filesize=10m

Container-aware sizing

# Old (don't do this in containers)
-Xmx2g

# New (container-aware)
-XX:MaxRAMPercentage=75.0
-XX:InitialRAMPercentage=75.0
-XX:MinRAMPercentage=75.0

75% is a common starting point, leaving 25% of the container limit for Metaspace, code cache, thread stacks, direct memory, and native libraries.

Allocation Rate — The Hidden Metric

Heap size is only half the story. The other half is allocation rate — how fast new objects are born.

  • Young GC frequency = allocation rate ÷ young gen size
  • Promotion rate = (live young after GC) ÷ GC cycle
  • Old gen fills at promotion rate; concurrent marking triggers at IHOP
  • If allocation rate is high, young gen needs to be larger to reduce GC frequency

Measure it in JFR or from GC logs. 500 MB/sec is typical for a busy microservice. 5 GB/sec needs serious attention.

Worked Examples

Spring Boot REST service, 10K rps, 2 GB heap, p99 pause < 50 ms

java -XX:+UseG1GC \
     -Xms2g -Xmx2g \
     -XX:+AlwaysPreTouch \
     -XX:MaxGCPauseMillis=40 \
     -XX:InitiatingHeapOccupancyPercent=35 \
     -XX:G1ReservePercent=15 \
     -Xss512k \
     -XX:MaxMetaspaceSize=256m \
     -XX:+HeapDumpOnOutOfMemoryError \
     -Xlog:gc*=info:file=gc.log:time,uptime:filecount=10,filesize=10m \
     -jar api.jar

Spark executor, 16 GB heap, throughput priority

java -XX:+UseParallelGC \
     -Xms16g -Xmx16g \
     -XX:+AlwaysPreTouch \
     -XX:-UseAdaptiveSizePolicy \
     -XX:ParallelGCThreads=8 \
     -jar spark-executor.jar

Kafka broker, 6 GB heap, latency priority

java -XX:+UseG1GC \
     -Xms6g -Xmx6g \
     -XX:+AlwaysPreTouch \
     -XX:MaxGCPauseMillis=20 \
     -XX:InitiatingHeapOccupancyPercent=35 \
     -XX:G1HeapRegionSize=16m \
     -XX:MaxDirectMemorySize=4g \
     -jar kafka.jar

Container at 512 MB

java -XX:MaxRAMPercentage=75.0 \
     -XX:+UseSerialGC \
     -XX:+AlwaysPreTouch \
     -Xss256k \
     -XX:MaxMetaspaceSize=96m \
     -XX:ReservedCodeCacheSize=96m \
     -jar tiny-app.jar

Anti-Patterns

  • Copying flags from Stack Overflow without understanding what they do
  • Setting -Xmx to the container limit — leaves nothing for off-heap, Metaspace, thread stacks
  • Ignoring direct memory — Netty and Kafka can allocate more than the heap without any heap pressure
  • Not monitoring allocation rate — tuning without knowing this is guessing
  • Tuning before measuring — always wrong
  • Tuning for average instead of worst-case — the p99 is what matters
  • Running CMS in 2026 — removed in Java 14
  • Using -XX:NewRatio aggressively on G1 — it’s mostly ignored; G1 uses G1NewSizePercent / G1MaxNewSizePercent
  • Setting -XX:+AggressiveOpts — removed; used to enable speculative tuning

When Not to Use the JVM Collector

Some workloads are better served by zero-GC architectures:

  • LMAX Disruptor — pre-allocated ring buffer with object pools
  • Chronicle Queue / Chronicle Map — off-heap persistent data structures
  • Aeron — off-heap messaging
  • Custom allocators via Unsafe or Panama FFM
  • Netty pooled byte buffers — reference counted, reused
