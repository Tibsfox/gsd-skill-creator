Every JVM GC is an implementation of a handful of classical ideas, composed differently to hit different points on the three-way trade-off: pause time, throughput, and footprint.

GC Theory Primer

The Weak Generational Hypothesis

The single most influential empirical observation in GC design: most objects die young. Decades of production heap telemetry confirm that the lifetime distribution of allocated objects is strongly bimodal — the vast majority become unreachable within milliseconds of allocation; a small minority survive for the entire process lifetime. This is the justification for generational garbage collection: if you can cheaply identify the newly-allocated region and collect only that, you reclaim most of the garbage for a fraction of the cost of scanning the entire heap.

STW, Concurrent, Parallel

These three terms are orthogonal but often conflated:

  • Stop-the-world (STW). All mutator threads are halted at a safepoint. Simple to reason about; hostile to latency.
  • Parallel. The GC uses multiple GC threads. Still STW, but faster because work is divided across CPUs. “Parallel” does not imply concurrent with the application.
  • Concurrent. The GC runs at the same time as application threads. Requires barriers to keep the collector and mutator coherent.

Modern low-pause collectors (G1, ZGC, Shenandoah) use a combination: short parallel STW phases (initial mark, final mark) bracketing long concurrent phases (concurrent mark, concurrent relocate).

Barriers, card tables, remembered sets

Barriers are code emitted by the JIT around heap accesses to maintain GC invariants concurrently with the mutator. Two families:

  • Write barriers — run on pointer stores. Used by generational collectors to record old-to-young pointers; used by concurrent markers (SATB) to remember overwritten references.
  • Read barriers (load barriers) — run on pointer loads. Used by concurrent relocating collectors (ZGC, Shenandoah) to intercept loads of potentially-relocated references and forward them.

A card table is a coarse bitmap that divides the heap into equal cards (typically 512 bytes in HotSpot). A write barrier sets the corresponding card bit. At collection time, the GC scans only dirty cards.

A remembered set is a finer structure that explicitly lists, for each region, the set of other regions that contain pointers into it. G1 uses per-region remembered sets.

Safepoints and TTSP

A safepoint is a point in compiled code where the JVM knows the thread’s register state well enough to walk its stack precisely. When the GC needs to stop the world, it sets a flag; each mutator thread polls that flag at safepoint locations (method entry, loop back-edges, call returns) and suspends itself. The time between setting the flag and the last mutator reaching its next safepoint is called time-to-safepoint (TTSP) — a hidden latency source. JDK 10+ uses thread-local handshakes (JEP 312) to stop individual threads without a global safepoint.

Precise GC

The HotSpot JVM is precise. Every compiled method carries an OopMap (ordinary-object-pointer map) for each safepoint, listing exactly which stack slots and registers hold references. This is what makes compacting and evacuating collectors safe — every pointer can be updated to the new location.

Serial GC

The simplest collector in HotSpot. Single-threaded, fully stop-the-world.

java -XX:+UseSerialGC -jar app.jar
  • Young generation: copying collector with Eden + two survivor spaces
  • Old generation: mark-sweep-compact
  • Throughput: surprisingly competitive for small heaps (no concurrency overhead)
  • Pause time: proportional to heap size; unusable for large heaps
  • Footprint: minimal — no remembered sets, no concurrent marking structures
  • Use when: small heaps (< 100 MB), embedded, single-core, short-lived CLI processes

Parallel GC — The Throughput Collector

java -XX:+UseParallelGC -jar app.jar

Multi-threaded, still fully stop-the-world. Design goal: spend as little total CPU on GC as possible, regardless of individual pause length.

  • Young generation: parallel copying collector. N GC threads divide Eden into strips and evacuate in parallel.
  • Old generation: parallel mark-sweep-compact. In modern JDKs (8+), parallel old is the default when UseParallelGC is enabled.
  • Throughput: highest of any HotSpot collector for CPU-bound batch work.
  • Pause time: longer than concurrent collectors; old-gen collections can pause for seconds on multi-GB heaps.
  • Default in Java 8 for server-class machines; still best for batch jobs where latency doesn’t matter.

CMS (Concurrent Mark Sweep) — Removed

CMS is deprecated and removed. The Concurrent Mark Sweep collector (-XX:+UseConcMarkSweepGC) was the JVM’s first serious attempt at low-pause old-generation collection. It did concurrent marking and sweeping, with STW pauses only for initial mark and remark. But it never compacted, which meant fragmentation would eventually trigger a long full GC.

Deprecated in Java 9 (JEP 291), removed in Java 14 (JEP 363). G1 replaced it for the same niche.

G1 — Garbage First, The Modern Default

java -XX:+UseG1GC -jar app.jar

The default collector since Java 9. Region-based heap with typical regions of 1–32 MB. Young and old regions are mixed across the heap, not in separate contiguous areas.

Key architectural ideas:

  • Region-based heap — divided into 2048 regions by default, sized as a power of 2 between 1 MB and 32 MB.
  • Per-region remembered sets — so any region can be collected independently.
  • Incremental evacuation — G1 collects a subset of regions per pause, not the whole heap.
  • Predictable pause target via -XX:MaxGCPauseMillis. G1 schedules which regions to collect to meet the target.
  • SATB (Snapshot-At-The-Beginning) marking — concurrent marker uses a write barrier to remember overwritten references.
  • Mixed collections — between full marking cycles, G1 collects young + some old regions.
  • Humongous objects (> 50% of a region) live in dedicated contiguous humongous regions.

Sweet spot: heaps from ~4 GB to ~32 GB; usable at hundreds of GB with tuning. Signs you need to tune G1: to-space exhaustion, long mixed GC pauses, humongous object allocation.

ZGC — Sub-10ms Pauses at TB Scale

java -XX:+UseZGC -XX:+ZGenerational -jar app.jar

Introduced as experimental in Java 11, production-ready in Java 15, generational ZGC landed in Java 21 (JEP 439). The ultra-low-pause collector.

The ZGC innovations:

  • Colored pointers — historically 42-bit addresses with metadata in high bits (mark0, mark1, remapped, finalizable). Java 18+ uses multi-mapped memory for some platforms.
  • Load barriers on every reference load — the read barrier intercepts loads of relocated references and forwards them to the new location.
  • Concurrent everything — marking, relocation, reference processing all run concurrently with the mutator.
  • Page-based — ZGC has its own page structure separate from OS pages; three page sizes (small, medium, large).
  • Target: < 10 ms max pause regardless of heap size; scales to multi-TB heaps.
  • Memory overhead: historically ~3× the live set; generational ZGC improves this significantly.

Generational ZGC (Java 21, JEP 439): Young and Old generations, but still entirely concurrent. Most workloads with generational patterns can now use ZGC without the old throughput penalty.

Shenandoah — Red Hat’s Low-Pause Alternative

java -XX:+UseShenandoahGC -jar app.jar

Red Hat’s OpenJDK collector, built by Christine Flood and Roman Kennke. Similar pause goals to ZGC but a different approach.

  • Brooks pointers historically (now Load Reference Barriers) — a forwarding pointer in every object’s header that directs reads and writes to the current location.
  • Concurrent compaction — live objects are relocated while the mutator runs.
  • Heuristics: adaptive (default), static, compact, aggressive, passive.
  • Target: < 10 ms max pause regardless of heap size.
  • Availability: OpenJDK builds from Red Hat, Adoptium, Azul, BellSoft; not in Oracle JDK historically.

Epsilon — The No-Op Collector

java -XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC -jar app.jar

Epsilon allocates but never collects. It exists for very specific use cases:

  • Short-lived processes that die before exhausting the heap (CLI tools, benchmarks)
  • Benchmarking the allocator itself
  • Testing VM performance without GC interference
  • Measuring the “GC tax” of a real workload by comparison

The JVM dies with OutOfMemoryError when the heap exhausts. That’s the whole design.

Azul C4 — The Commercial Pauseless Collector

Part of Azul Zing / Prime. C4 (Continuously Concurrent Compacting Collector) was the first truly pauseless concurrent compacting GC, predating ZGC by years.

  • Load Value Barrier (LVB) — concurrent read barrier
  • Guaranteed bounded pause time at TB heaps
  • Used heavily in high-frequency trading and financial services
  • Commercial; requires Azul’s JVM
  • Many of the techniques eventually influenced ZGC and Shenandoah
