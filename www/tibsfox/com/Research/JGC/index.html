Why a dedicated project on JVM GC? The JAV research project covered Java the language and its ecosystem. This project goes one level deeper: into the machinery that actually runs Java code. Garbage collection is the single most-discussed, most-misunderstood, and most-misconfigured part of the JVM. The 2026 production Java stack — Spring Boot microservices, Kafka brokers, Elasticsearch clusters, HFT engines — lives or dies by how well its operators understand the collector, the memory spaces, the tuning flags, and the diagnostic tools.

The Tuning Triangle

Every GC trade-off reduces to three goals. You can optimize two at the cost of the third:

Throughput

The fraction of CPU time spent running application code instead of the GC. High throughput = batch jobs, data pipelines, ETL. Parallel GC wins here.

Latency (Pause Time)

The longest single stop-the-world pause. Low latency = API servers, trading systems, user-facing services. G1, ZGC, Shenandoah target this.

Footprint

How much memory the JVM actually uses. Small footprint = containers, edge, embedded, dense packing. Serial GC has the smallest overhead; ZGC has the largest.

…pick two.

The iron law of garbage collection. No collector optimizes all three simultaneously. Every tuning decision is a vote for which corner you’ll compromise.

Anatomy of a Running JVM in Memory

When you type java MyApp, the JVM process allocates a mosaic of memory regions. The heap is only one of them — and surprisingly often, not the biggest one.

Eden — Young GenerationNew object allocation; collected by minor GC
Survivor 0 / Survivor 1Objects that survived one or more minor GCs
Old Generation (Tenured)Long-lived objects promoted from young; collected by major/mixed GC
MetaspaceClass metadata, methods, constant pools. Replaced PermGen in Java 8.
Code CacheJIT-compiled native code. Segmented in Java 9+.
Thread StacksPer platform thread, default ~1 MB. Virtual threads use ~1 KB on heap.
Direct Memory / Mapped BuffersOff-heap buffers (Netty, Lucene, Kafka, Aeron). -XX:MaxDirectMemorySize
Native MemoryJNI, libc, zlib, OpenSSL, JVM internals. Not managed by the GC.

The difference between “Java uses 8 GB” and “top shows 12 GB RSS” is everything below the heap line. Native Memory Tracking (jcmd <pid> VM.native_memory summary) is how you find it.

The Five Questions Every Tuner Asks

  1. What is your current collector? Check with java -XX:+PrintFlagsFinal -version | grep UseG1GC or read the GC log.
  2. What is your heap size? Min and max. Are they equal? (They should be.)
  3. What are your goals? Throughput? Max pause time? Memory footprint? You can’t have all three.
  4. What is your measurement infrastructure? GC logs are free. JFR is free. Prometheus + Grafana is free. If you have no data, you’re guessing.
  5. Are you in a container? What are the cgroup limits? Are you OOMKilled or ApplicationOOM? They are different problems.
