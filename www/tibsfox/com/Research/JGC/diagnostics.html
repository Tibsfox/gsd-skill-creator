When everything is on fire, diagnostics is the difference between “restart the JVM and hope” and “find the leaking classloader, fix the code, and never see it again.”

Heap Dumps

A heap dump is a snapshot of every object on the Java heap at a point in time, with all their fields and references. The canonical format is HPROF. File sizes match the used heap — a 4 GB heap produces a ~4 GB dump file.

Triggering heap dumps

# On OutOfMemoryError (set this in production)
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/dumps/heap-%p.hprof

# Manual dump via jcmd
jcmd <pid> GC.heap_dump /var/dumps/manual.hprof

# Manual dump via jmap (legacy but still works)
jmap -dump:format=b,file=heap.hprof <pid>

# Dump only live objects (forces a full GC first)
jcmd <pid> GC.heap_dump -all=false /var/dumps/live.hprof

MAT key concepts

  • Shallow size — the bytes directly occupied by an object (header + fields, not referenced objects)
  • Retained size — the bytes that would be freed if this object and only this object became unreachable (follows the dominator tree)
  • Dominator tree — the tree where A dominates B if every path from any GC root to B passes through A. This is the structural view that makes leak hunting possible.
  • GC roots — stack slots, static fields, JNI handles, monitors. The entry points the collector traces from.
  • Leak Suspects Report — MAT’s automatic analysis that flags suspicious large retained-size roots

Common leak patterns

  • ThreadLocal leaks — ThreadLocal values kept alive by long-lived pool threads
  • Classloader leaks — webapp reload without cleaning references; static fields in the leaked classloader hold the whole world
  • Listener / observer leaks — subscribers never unsubscribed, growing without bound
  • Unbounded caches — HashMap used as a cache with no eviction policy
  • Duplicate strings — millions of identical String instances (use String.intern or G1 string deduplication)
  • Huge collections — one ArrayList with millions of entries that was supposed to be paginated
  • Event loops retaining stale state — executor tasks that never complete

Thread Dumps

# jstack (works on living processes)
jstack <pid> > threads.txt

# jcmd (modern equivalent)
jcmd <pid> Thread.print > threads.txt

# Also via SIGQUIT — writes to the process’s stdout
kill -3 <pid>

# With synchronized locks and ownable synchronizers
jstack -l <pid>

Thread states

StateMeaning
NEWCreated but not yet started
RUNNABLEExecuting (or ready to execute)
BLOCKEDWaiting to acquire a monitor lock
WAITINGwait(), join(), or park() without timeout
TIMED_WAITINGSame but with a timeout (sleep, wait(timeout))
TERMINATEDFinished

Deadlock detection: Modern JVMs automatically detect and report monitor-lock deadlocks in thread dumps. Look for the Found N Java-level deadlock(s) section at the bottom of a jstack -l output.

Analysis tools

  • FastThread.io — free web-based thread dump analyzer with clustering and leak detection
  • Spotify thread dump analyzer — open source CLI
  • Thread Dump Analyzer (TDA) — standalone GUI

JMH — The Microbenchmark Harness

Aleksey Shipilev’s benchmarking framework. The only credible way to benchmark JVM code. Naive microbenchmarks lie because of JIT optimization, dead code elimination, on-stack replacement, warmup effects, and GC interference.

@BenchmarkMode(Mode.AverageTime)
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@State(Scope.Benchmark)
@Warmup(iterations = 5, time = 1)
@Measurement(iterations = 10, time = 1)
@Fork(3)
public class MyBenchmark {

    @Param({"100", "1000", "10000"})
    int size;

    int[] data;

    @Setup
    public void setup() {
        data = ThreadLocalRandom.current().ints(size).toArray();
    }

    @Benchmark
    public int sumLoop() {
        int sum = 0;
        for (int x : data) sum += x;
        return sum;
    }

    @Benchmark
    public int sumStream() {
        return Arrays.stream(data).sum();
    }
}

Run with -prof gc to see GC overhead during benchmarks.

Incident Response Playbook

“App is slow”

  1. Read GC logs — long pauses? Frequent full GCs? Allocation spikes?
  2. Thread dump — lock contention? Thread state distribution?
  3. async-profiler flame graph — where is CPU actually going?
  4. Check JFR — is there a hot compilation loop or method?

“App is OOM” (OutOfMemoryError thrown by Java)

  1. Check -XX:+HeapDumpOnOutOfMemoryError was set (it should have been)
  2. Load the heap dump into Eclipse MAT, run the Leak Suspects Report
  3. Check Metaspace separately — the OOM might be “Metaspace”
  4. Check direct memory — the OOM might be “Direct buffer memory”
  5. Check the “requested array size exceeds VM limit” case

“OOMKilled” (killed by Linux kernel or cgroup)

Different problem. The JVM process exceeded its cgroup memory limit and Linux killed it. The heap might have been fine.

  1. Check RSS history before the kill (Prometheus/cAdvisor if you have it)
  2. Compare RSS to the container limit
  3. Check NMT summary from before: heap + metaspace + code cache + threads + direct + native
  4. Check for direct memory leaks (Netty, Kafka clients) — invisible to heap dumps
  5. Check for classloader leaks growing Metaspace
  6. Check thread count explosion growing stack memory

“Long pauses”

  1. Read GC logs — which phase is slow?
  2. Check safepoint logs — is TTSP the problem?
  3. Check promotion rate — is old gen filling too fast?
  4. Check humongous object allocation in G1 logs
  5. Consider switching to ZGC or Shenandoah

“High CPU”

  1. async-profiler flame graph — first and best tool
  2. Check GC threads — is the collector the one burning CPU?
  3. Check JIT compiler threads (tier 4 C2 compilation loops)
  4. Check for busy-wait patterns in thread dumps
