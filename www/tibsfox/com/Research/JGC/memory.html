Memory Spaces & Layout

Virtual vs Physical, Heap, Metaspace, Native — Every Byte a Running JVM Touches
Overview Collectors Memory Tuning Monitoring Diagnostics Containers Rosetta Learn
Ask ten Java engineers “how much memory does your service use?” and you will get ten different answers — all of them correct, all of them measuring different things. The JVM is one of the most memory-complicated processes you can run on a Linux box.

RSS vs VSZ vs Committed vs Reserved

Before we talk about the JVM at all, we need to understand how Linux accounts for process memory. The JVM sits inside this model; it does not escape it.

MetricMeaningWhen it matters
VSZ / VIRTVirtual address space reserved — sum of every VMA the process has mappedUsually much larger than physical RAM; often meaningless on its own
RSS / RESResident set size — physical pages actually mapped for the processThe number you care about for OOM killer / cgroup limits
PSSProportional Set Size — USS plus fair share of shared pagesBest single “how much is this process costing” metric; from smem
USSUnique Set Size — pages belonging only to this processWhat would be freed if the process died
CommittedPages the kernel has promised to the processReported by NMT; not every committed page is resident
ReservedAddress space the JVM has requested but not yet committedCan be large even for tiny real usage

Canonical example:

$ ps -o pid,vsz,rss,cmd -p $(pgrep -f MyApp)
  PID      VSZ     RSS CMD
12345 14876520 3421084 java -Xmx4g -jar myapp.jar

That process has reserved ~14.2 GB of virtual address space but is only using ~3.3 GB of physical RAM. This is completely normal for a JVM and not a bug.

Anatomy of a JVM Process

When you launch java MyApp, the process allocates a mosaic of memory regions. The heap is only one of them — and surprisingly often, not the biggest one.

Eden — Young GenerationNew object allocation. Minor GC.
Survivor 0 / Survivor 1Objects that survived one or more minor GCs.
Old Generation (Tenured)Long-lived objects. Major / mixed GC.
MetaspaceClass metadata, methods, constant pools. Replaced PermGen in Java 8.
Code CacheJIT-compiled native code. Segmented in Java 9+.
Thread StacksPer platform thread, default ~1 MB. Virtual threads use ~1 KB on heap.
Direct Memory / Mapped BuffersOff-heap (Netty, Lucene, Kafka, Aeron).
Native MemoryJNI, libc, OpenSSL, zlib, JVM internals.

The Java Heap

The heap is where every Java object lives (except, in limited cases, those that the JIT can prove do not escape the method scope).

RegionPurposeKey flags
EdenNew objects are born here-XX:NewRatio, -XX:NewSize
Survivor 0 / 1Two semi-spaces for objects that survived minor GCs-XX:SurvivorRatio=8 (default)
Old / TenuredObjects promoted after surviving enough minor GCs-XX:MaxTenuringThreshold=15

Sizing: -Xms (initial heap) and -Xmx (max heap). In production, you almost always want -Xms == -Xmx to avoid heap resizing pauses.

Metaspace (Java 8+)

JEP 122 replaced PermGen with Metaspace in Java 8. Stores class metadata: klass structures, method bytecodes, method info, constant pools, annotation info.

Key facts:

  • Allocated from native memory, not the Java heap
  • Default: unlimited — grows until native memory is exhausted
  • -XX:MaxMetaspaceSize=256m to cap it
  • -XX:CompressedClassSpaceSize for the Compressed Class Space sub-region
  • Class loaders not being GC’d leads to Metaspace leaks
  • Classic failure mode: Tomcat webapp hot reload with leaked classloaders

Code Cache

JIT-compiled native code lives in the code cache. In Java 9+, it’s segmented (JEP 197) into three regions:

  • Non-method — internal JVM code (interpreter, compiler, stubs)
  • Profiled — JIT output from the C1 tier, with profiling counters
  • Non-profiled — fully optimized C2 output, no profiling

-XX:ReservedCodeCacheSize=256m controls the max. The warning nobody wants: “CodeCache is full. Compiler has been disabled.” At that point the JVM keeps running but any method that needed recompilation is stuck interpreted.

Thread Stacks

Each platform thread has its own stack. -Xss controls the size; default is roughly 1 MB on 64-bit Linux. This is virtual memory — the physical pages are only committed as the stack grows.

1,000 threads = 1 GB of stack address space. 10,000 threads = 10 GB. This is why “thread-per-request” at scale became impractical on the JVM before Project Loom. Virtual threads (Java 21) have tiny stacks (~1 KB initially, heap-allocated) — a million virtual threads uses less memory than a thousand platform threads.

Direct Memory (Off-Heap)

Allocated via ByteBuffer.allocateDirect(), sun.misc.Unsafe, jdk.internal.misc.Unsafe, or the new Panama FFM API. Lives outside the Java heap and is not tracked by the GC’s normal mechanisms.

  • -XX:MaxDirectMemorySize=1g to cap it (default: roughly -Xmx)
  • Netty, Lucene, Kafka, Aeron, Chronicle, Apache Arrow all use direct memory heavily
  • Reclaimed via Cleaner (formerly sun.misc.Cleaner) — non-deterministic
  • Netty adds explicit reference counting on top because the GC is too slow to clean direct buffers

Memory-Mapped Files

FileChannel.map() produces a MappedByteBuffer. The file appears in the process address space via mmap(); the OS page cache brings pages in on demand.

Kafka, Lucene, and LMDB rely on mmap heavily — it’s the fastest way to give the OS the freedom to cache file data in available RAM without the JVM having to manage it. RSS can include mmap’d pages that the OS has actually brought in, making RSS unpredictable for these workloads.

Native Memory Tracking (NMT)

The one tool that accounts for every byte the JVM tracks. Enable with -XX:NativeMemoryTracking=summary (or detail for more). Then:

jcmd <pid> VM.native_memory summary

Categories reported:

CategoryWhat’s there
Java HeapThe managed Java heap
ClassMetaspace + Compressed Class Space
ThreadThread stacks (reserved + committed)
CodeCode cache
GCCard tables, remembered sets, collector internal data
CompilerJIT compiler working memory
InternalJVM internal structures
SymbolString interning and symbol tables
Native Memory TrackingNMT’s own overhead
Arena ChunkArena-allocated memory (mostly GC and compiler)
UnknownEverything NMT can’t classify (often direct memory, JNI, native libraries)

NMT is the answer to “why is my JVM using more RSS than -Xmx?” It shows you the gap: heap + metaspace + code cache + thread stacks + GC overhead + JIT + direct memory + native library allocations. Every production JVM should enable it.

Compressed Oops and the 32 GB Cliff

-XX:+UseCompressedOops (default when heap < 32 GB) uses 32-bit object pointers on a 64-bit JVM, saving memory on every object header and every reference field. The savings are significant — typically 10–15% less memory for the same workload.

The 32 GB cliff: above that heap size, compressed oops turn off, pointers become 64-bit, and per-object memory usage jumps. Many workloads see worse performance with a 33 GB heap than a 31 GB heap — the pointer overhead outweighs the extra heap space. -XX:ObjectAlignmentInBytes=16 can extend the cliff to 64 GB at the cost of some internal padding.

Container-Aware Sizing

-XX:+UseContainerSupport (default in Java 10+) tells the JVM to read cgroup limits for RAM and CPU instead of the host’s values. The modern way to size the heap in a container:

# Old style (often wrong):
java -Xmx2g -jar app.jar

# Container-aware style:
java -XX:MaxRAMPercentage=75.0 -jar app.jar

75% is a common starting point for the heap, leaving 25% of the container limit for Metaspace, code cache, thread stacks, direct memory, and native libraries. Too high, and you get OOMKilled. Too low, and you waste memory.

Hugepages

  • -XX:+UseLargePages — requests explicit hugepages (requires OS configuration: vm.nr_hugepages)
  • -XX:+UseTransparentHugePages — opts into THP in “madvise” mode
  • General Linux advice: set THP to madvise, not always; the JVM knows when it benefits
  • Impact: fewer TLB misses on large heaps; can reduce GC pause time measurably

Swap and Java

Java and swap are a bad combination. If the JVM’s heap pages get swapped out, the next GC cycle must page them back in one at a time, and the pause time becomes unbounded. A 200 ms G1 target becomes a 60-second stall.

Standard advice for production JVMs: vm.swappiness=0 or very low. Disable swap inside Kubernetes. Size the container so you never need it.
