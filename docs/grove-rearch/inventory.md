## 1. Purpose

This document classifies every TypeScript source file in `src/memory/` and `src/mesh/` against the eight M-modules of the Living Sensoria milestone (M1 Semantic Memory Graph, M2 Hierarchical Hybrid Memory, M3 Decision-Trace Ledger, M4 Branch-Context Experimentation, M5 Agentic Orchestration, M6 Sensoria, M7 Umwelt, M8 Symbiosis). Each file receives one of four Grove re-architecture decisions (EXTEND / REWRITE / NEW-LAYER / UNTOUCHED) and a one-line rationale. This inventory is the cacheable synthesis prefix for Wave 2 integration prompts; it gates Track D by confirming that the module-to-file mapping is unambiguous before any implementation begins. Files in `src/memory/__tests__/` are classified separately where they test multiple sources or carry unique fixture scope; pure per-source companion test files are noted inline with their source row.

---

## 2. Decision Taxonomy

**EXTEND** — the existing file's public API and storage contracts are preserved in full; M-module obligations are met by adding types, methods, or configuration to the existing module without breaking callers. Any module choosing EXTEND commits to zero regression on the current test suite for that file.

**REWRITE** — the existing file's internal implementation is replaced while keeping the public shape byte-compatible where possible; used only when the existing code is structurally inadequate for module obligations (wrong schema, wrong write semantics, wrong storage model) and extending would produce an unmaintainable hybrid. Rewrites require an ARCH approval stub; this document flags them but does not commit the rewrite.

**NEW-LAYER** — the module adds net-new files that sit on top of existing Grove code without modifying any existing file; the existing file is a substrate read-only. Used when the module is strictly additive and has no need to alter existing public contracts.

**UNTOUCHED** — the file is not consumed by any M1–M8 module during this milestone; it continues to compile, pass its tests, and export its API unchanged. No Living Sensoria work touches it.

## 8. Open Questions for ARCH

**OQ-1: `skill-creator-mcp.ts` CLI registration point.** The M6 (`skill-creator sensoria <skill>`) and M8 (`skill-creator teach|co-evolution|quintessence`) CLI tools are assigned as NEW-LAYER files in `src/sensoria/cli.ts` and `src/symbiosis/cli.ts`. If these commands must be registered through `skill-creator-mcp.ts` (which centralizes tool registration), then `skill-creator-mcp.ts` must be EXTENDed by both M6 and M8 tracks. Track A and C executors should confirm the CLI registration mechanism before starting W1.A/W1.C, and notify ARCH if EXTEND of `skill-creator-mcp.ts` is required.

**OQ-2: `ram-cache.ts` eviction hook for M2 reflection trigger.** The M2 reflection pass can be triggered by either a count threshold or an eviction event from ram-cache. If the reflection trigger is count/time based only (checked on every write to short-term), `ram-cache.ts` stays UNTOUCHED. If eviction-triggered reflection is required for correctness (to avoid losing entries without reflecting), `ram-cache.ts` requires an EXTEND to add an optional `onEvict` callback. Track D executor should decide this design point at W1.D.2 start and update ARCH if EXTEND is confirmed.

**OQ-3: `session-activation-view.ts` ownership.** This file sits in `src/mesh/` but carries semantic concerns (session/activation audit trail) that are also central to M1, M3, and M6. Three modules touch it via EXTEND (M1 reads edges from it, M3 needs `traceId` correlation, M6 writes activations). ARCH should confirm that `src/mesh/` remains the correct home or propose relocation to `src/memory/` or a new `src/audit/` directory. Relocation is out of scope for Living Sensoria unless ARCH authorizes it; the inventory assumes current location for all decisions.

**OQ-4: `context-types.ts` vs. M3 shared types.** `context-types.ts` defines `DecisionRecord` (mesh execution decision) and M3's shared types define `DecisionTrace` (AMTP-compatible skill activation trace). These are structurally similar but serve different consumers. ARCH should confirm that these remain parallel types (current assumption) rather than consolidating them, to avoid coupling the mesh execution layer to the M3 ledger schema.
