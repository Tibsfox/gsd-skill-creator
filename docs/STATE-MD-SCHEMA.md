# STATE.md Schema Reference

**Version:** 1.0
**Location:** `docs/STATE-MD-SCHEMA.md` (tracked in git)
**Status:** Normative — enforced by `tools/state-md-normalizer.mjs`
**Introduced:** v1.49.650 C6 (closes v1.49.634 HANDOFF open-item §1)

---

## Overview

`.planning/STATE.md` is the live project-state document for gsd-skill-creator.
It is intentionally gitignored (`.gitignore` line 8) so it survives across
branches without being committed. The auto-update tool
(`tools/update-state-md-on-ship.mjs`) advances the `milestone` + `status`
fields on ship. The normalizer (`tools/state-md-normalizer.mjs`) enforces the
full schema shape described here.

---

## Target Schema

```yaml
---
gsd_state_version: 1.0
milestone: v1.49.XXX
milestone_name: <description>
status: <opened | planning | shipped>
counter_cadence: <bool>
no_engine_state_advance: <bool>
nasa_degree: <int>
predecessor:
  milestone: v1.49.YYY
  shipped_at_tag: vN.NN.YYY
  shipped_at_sha: <40-char-sha>
  post_ship_sync_sha: <40-char-sha or null>
  counter_cadence: <bool>
  degree_advancing_milestone: <vN.NN.YYY or null>
opened_on: <YYYY-MM-DD>
shipped_at: <ISO-8601 or null>
last_updated: <ISO-8601>
mission_package_pattern: <string>
progress:
  total_phases: <int>
  completed_phases: <int>
  total_plans: <int>
  completed_plans: <int>
  percent: <float>
---

## Current Position

<auto-generated from frontmatter: milestone, milestone_name, status, opened_on>

## Engine state baseline at <milestone> open

<auto-generated from predecessor.* fields + nasa_degree>

## Notes

<free-text section preserved verbatim across normalization runs>
```

## Field Definitions

### Top-level fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `gsd_state_version` | string | YES | Schema version; currently `"1.0"` |
| `milestone` | string | YES | Current milestone identifier, e.g. `v1.49.650` |
| `milestone_name` | string | YES | Human-readable description of the milestone |
| `status` | string | YES | One of: `opened`, `planning`, `shipped` |
| `counter_cadence` | bool | YES | `true` if this milestone is a counter-cadence (non-engine-advancing) milestone |
| `no_engine_state_advance` | bool | YES | `true` if engine state (NASA/MUS/ELC/SPS/TRS) must NOT advance this milestone |
| `nasa_degree` | int | YES | Current NASA degree number at milestone open |
| `predecessor` | object | YES | Nested block — see below. Replaces all flat `predecessor_*` keys |
| `opened_on` | string | YES | ISO-8601 date (`YYYY-MM-DD`) of when this milestone was opened |
| `shipped_at` | string or null | YES | ISO-8601 datetime when shipped, or `null` if still open |
| `last_updated` | string | YES | ISO-8601 datetime of last write to this file |
| `mission_package_pattern` | string | NO | Describes the mission package shape (e.g. `engine-cadence-degree-advancing`) |
| `progress` | object | NO | Phase/plan tracking. Sub-fields: `total_phases`, `completed_phases`, `total_plans`, `completed_plans`, `percent` |

### `predecessor` block

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `milestone` | string | YES | Predecessor milestone identifier |
| `shipped_at_tag` | string | YES | Git tag at which predecessor shipped |
| `shipped_at_sha` | string | YES | Git SHA at which predecessor shipped (short OK) |
| `post_ship_sync_sha` | string or null | NO | SHA after post-ship sync to main (if any) |
| `counter_cadence` | bool | YES | Whether the predecessor was a counter-cadence milestone |
| `degree_advancing_milestone` | string or null | NO | The most recent degree-advancing milestone at predecessor open |

## Migration Notes

### Flat-key collapse

Historical STATE.md files accumulated flat `predecessor_*` keys such as:

```yaml
predecessor_milestone: "v1.49.630"
predecessor_shipped_at_tag: "v1.49.630"
predecessor_shipped_at_sha: "412074630"
predecessor_post_ship_sync_sha: "c3d53f766"
predecessor_counter_cadence: false
predecessor_degree_advancing_milestone: "v1.49.630"
```

These are collapsed into the nested `predecessor:` block by the normalizer.
Flat keys are REMOVED from the top level during normalization (they become
redundant once nested). If `--prune-stale` is not passed, only the
`predecessor_*` flat keys are collapsed; other unknown keys are preserved.

### Duplicate key resolution

YAML parsers silently last-write-wins on duplicate keys. The normalizer
reads the raw frontmatter string and uses the **last occurrence** of each key
(consistent with standard YAML parser behavior). Duplicate blocks are merged:
the last `predecessor_*` group wins for each sub-field.

### "Current Position" section regeneration

The normalizer auto-generates the `## Current Position` section body from
frontmatter. The old free-text narrative (which could drift to describe a
prior milestone) is replaced with a structured template.

### "Notes" section preservation

Anything following `## Notes` is preserved verbatim. All other body sections
(`## Current Position`, `## Engine state baseline at ... open`) are
regenerated. Unrecognized body sections between the frontmatter and the first
known section header are preserved verbatim.

## Normalizer Invariants

1. **Idempotency:** running the normalizer twice on the same file produces
   byte-identical output on the second run.
2. **Backup-before-write:** every `--write` run produces a timestamped backup
   at `.planning/STATE.md.backup-before-normalize-<ISO-8601>`. The `--check`
   flag is read-only (no backup, no write).
3. **Conservative by default:** unknown top-level keys (other than the
   `predecessor_*` flat keys being collapsed) are preserved unless
   `--prune-stale` is passed.
4. **Free-text preservation:** `## Notes` section content is preserved
   verbatim.
