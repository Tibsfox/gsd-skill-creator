# v1.49 — Spatial Substrate License Audit

**Date:** 2026-05-10
**Mission:** gis-spatial-substrate (Part B build)
**Scope:** Component 07 verification gate — every absorbed dependency must be permissively licensed (BSD / MIT / PostgreSQL / Apache 2.0). No GPL kernels reach the atlas binary distribution.

---

## Audit policy

The Part A research mission identified a license-clean composition at the atlas substrate layer:

- BSD/MIT/PostgreSQL/Apache 2.0 → **PASS** (permissive; statically linkable)
- LGPL → **PASS only at the wire-protocol boundary** (PostGIS, GEOS — accessed via PostgreSQL wire protocol, not statically linked)
- GPL on the binary distribution → **BLOCK**

### Component 01 — PostGIS migration

| Item | License | Verdict |
|---|---|---|
| PostgreSQL 18.3 (existing) | PostgreSQL License (BSD-style) | PASS |
| PostGIS 3.6.3 extension | GPL-2.0 | PASS at wire boundary — atlas connects via libpq protocol, does not statically link |
| GEOS (called by PostGIS) | LGPL-2.1 | PASS — invoked through PostGIS, never linked into atlas |
| PROJ (called by PostGIS) | MIT/X11 | PASS |

The PostGIS GPL applies to the extension binary itself, not to the data stored in it nor to client applications connecting via the wire protocol. This is the same discipline every PostgreSQL+PostGIS commercial deployment uses. Citation: PostGIS FAQ at postgis.net.

## CI gate to install (post-deps)

Once the upstream PMTiles / FlatGeobuf / Turf packages are added to `package.json`, the integration gate should add:

```bash
# Verify no copyleft kernels slipped in transitively.
npx license-checker --onlyAllow 'BSD-2-Clause;BSD-3-Clause;MIT;Apache-2.0;PostgreSQL;ISC' \
  --excludePackages '...' --production
```

A failing run blocks ship. Place the check in `tools/audit-spatial-licenses.sh` and call it from `tools/pre-tag-gate.sh` step 9.
