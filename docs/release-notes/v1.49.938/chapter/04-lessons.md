## Carried-forward observation candidate

- **A staged-lane flip-readiness gate's counting model must match the lane's FAILURE MODE — do not copy a sibling gate's predicate.** Two staged CI lanes (`macos`, `cargo`) both flip via the #10463 gate, but their readiness checkers count differently because they de-risk different things: the macOS leg's risk is cross-platform behavioral divergence on new code (only test-surface churn is fresh evidence; docs/release commits are inert), while the cargo lane's risk is environment/lane reliability exercised by a full recompile on every push (so docs-only greens DO count; only lane-definition-modifying commits are excluded). Copying the organic-churn predicate onto cargo would have made it un-promotable in a TS-heavy repo. First instance of the *divergence* here; a third staged lane that again needs a failure-mode-specific model would promote it.

- **Pin the membership of "advance" and "break" sets in a defer-biased gate.** A readiness gate whose contract is "misclassification defers the flip, never advances it" has its highest-risk blind spot exactly at the set boundaries that decide advance-vs-transparent and break-vs-transparent. A suite that only exercises one green value (`success`), one break value (`failure`), and one transparent value (`skipped`) leaves the other set members unpinned — and a shrink/expand of either set then silently violates the defer-bias. Add an explicit test per boundary (a non-`failure` breaking conclusion still breaks; a non-`success` conclusion stays transparent), and mutation-prove it. Surfaced here by the adversarial-verify; the same gap may exist in the macOS sibling's suite.
