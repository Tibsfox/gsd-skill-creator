## Summary

**CLI tool becomes native desktop app.** The six prior releases in the v1.15 → v1.20 arc progressively built dashboard surfaces — embedded terminal, milestone console, staging queue, information design, budget gauge, assembly wiring — all rendered as HTML in a browser. v1.21 takes that cohered web surface and wraps it in a Tauri v2 shell with a Rust backend, a WebGL2 post-processing pipeline, a native pseudo-terminal, and an Amiga Workbench-inspired window manager. Eleven phases (158-168), thirty-four plans, one hundred and six commits, and six hundred and thirty-six tests later, `npm run dev` against the `desktop/` workspace launches a frameless window that boots through an Amiga chipset animation, runs a first-boot calibration wizard on cold start, presents a pixel-art desktop with draggable windows, and embeds a working xterm.js terminal bound to a Rust-backed PTY with watermark flow control. The web dashboard continues to work; the desktop app is additive, not a replacement.

**Tauri v2 + capability ACL is the security backbone.** Phase 158 scaffolds the Rust backend (`014a3a2e2`), wires the Vite frontend build pipeline (`02cbbfe6f`), implements bidirectional IPC across commands, events, and channels (`bc2e9b128`), and closes with an IPC benchmark runner plus the initial capability ACL (`d0e4b5981`). The ACL model is explicit: `src-tauri/capabilities/default.json` and `restricted.json` list every Rust command the webview is allowed to invoke by name, and the autogenerated `src-tauri/permissions/autogenerated/*.toml` files enumerate per-command permissions (pty_open, pty_close, start_watcher, tmux_ensure_session, generate_dashboard, ipc_benchmark, and so on). A webview that tries to call an unlisted command is rejected at the Tauri layer, not at the command body. This is the right security model for a desktop app that manages execution pipelines — the alternative (open IPC with in-command auth checks) routinely produces privilege-escalation bugs that the ACL model prevents by construction. Every later phase that adds a Rust command (file watcher in 159, PTY in 162, tmux in 163, Claude session manager in 163, dashboard bridge in 166) adds its permission TOML in the same motion, so the capability surface stays visible and reviewable.

**WebGL2 CRT engine carries the retro-computing tribute.** Phases 160 and 161 ship the engine in three plans each. Phase 160-01 defines CRT config types plus a performance measurer (`02c8cc0ef`). Phase 160-02 builds the WebGL2 infrastructure: context creation, shader programs, fullscreen quad, render targets, and the four shader files (`quad.vert`, `crt-distort.frag`, `crt-post.frag`, `passthrough.frag`, plus `palette-bg.frag` from phase 161-02, all in `desktop/src/engine/shaders/`). Phase 160-03 assembles the full pipeline with CSS fallback (`3f3157f12`) — the CRT effects compose as multi-pass post-processing (scanlines, barrel distortion simulating screen curvature, phosphor glow via bloom, chromatic aberration as per-channel RGB offset, vignette at screen edges), and when WebGL2 is unavailable a CSS equivalent takes over with `box-shadow` and `filter` primitives. Phase 161 layers on the indexed palette system: a 32-color palette with five retro presets (Amiga Kickstart 1.3, Workbench 2.0, Workbench 3.1, Commodore 64, custom), OKLCH-based palette generation for perceptually uniform interpolation, a palette texture sampled by the background shader, and copper list raster effects for per-scanline color manipulation. The copper list is technically unnecessary — the app would function without per-scanline color — but the Amiga Copper is the chip that made the Workbench feel alive in 1985, and replicating it in a fragment shader is the kind of load-bearing detail that tells the user what project they're in.

**Native PTY terminal earns the name "desktop."** Phase 162-01 implements the Rust PTY backend using the `portable-pty` crate with session management, flow control, and six IPC commands (`ff128821a`: `pty_open`, `pty_close`, `pty_pause`, `pty_resize`, `pty_resume`, `pty_write`). Phase 162-02 implements flow control with watermark-based backpressure (`fa1373901`) plus a PTY bridge with `Uint8Array` normalization on the TypeScript side (`415cedfe4`). Phase 162-03 wires xterm.js into the webview with a terminal emulator lifecycle (`826373b38`). Phase 162-04 registers the six PTY commands in the Tauri AppManifest and capability ACL (`699e673fa`) and verifies the full stack end-to-end. The watermark pattern is TCP's backpressure mechanism applied to terminal I/O: a high-water mark pauses the PTY when the webview's receive buffer fills, a low-water mark resumes it when the buffer drains. Without this, a fast-producing process (say, `find / -type f`) would overwhelm the xterm.js renderer and the UI would lock up for tens of seconds. With it, the terminal stays responsive at any PTY throughput because the producer is literally told to stop. Phase 163 adds tmux session binding on top: tmux detection, session lifecycle, a tmux-aware terminal factory, and a Claude session manager that uses tmux to keep long-running Claude Code sessions attachable across window reopens.

**Amiga Workbench shell makes the aesthetic commitment explicit.** Phase 164 builds the window manager: window state CRUD with z-order depth cycling (`af21d5949`), Amiga-style window chrome with pointer-event drag/resize (`5060933d6`), and a `WindowManager` orchestrator plus frameless Tauri configuration (`35efd9d72`). Depth cycling is the specific Amiga 1.x behavior where clicking a window's depth gadget sends it to the back and pulling the front window forward brings it to the top — this is not the modern z-index stacking that every other OS uses, and implementing it required an explicit z-order stack in `desktop/src/wm/z-order.ts` rather than a numeric index. Phase 165 builds the shell around the window manager: pixel-art icon registry with desktop icon component (`812860568`), taskbar with process monitor (`eb54dd725`), system menu popup with keyboard shortcut manager (`c7664e885`), and a `DesktopShell` orchestrator that wires the pieces together (`8ceb4e972`). The keyboard shortcut manager binds the core Amiga + modern shortcuts — Alt+Tab for window cycling, F10 for the system menu, Ctrl+Q for quit — using a single `KeyboardNav` module that every shell component subscribes to rather than each component attaching its own listeners.

**Phase 166 bridges the v1.20 dashboard into the native shell.** Phase 166-01 implements the dashboard IPC bridge with a Rust node-shell generator (`63fe29270`) — the Rust side runs the existing `src/dashboard/generator.ts` through a sandboxed Node subprocess and returns the generated HTML. Phase 166-02 implements the palette CSS bridge for `user-style.yaml` theming (`0db0d63ce`) — user palette choices from the calibration wizard map to CSS custom properties that the dashboard inherits. Phase 166-03 implements the `DashboardHost` with caching, navigation, and HTML sanitization (`3b2acc0c9`), so the desktop presents the same six dashboard pages that v1.20 generated for the web, but as a native surface with cache-aware navigation. Phase 166-04 adds a watcher-driven refresh with debounced invalidation (`cf5226ead`) — file changes in `.planning/` or `.claude/` invalidate the relevant cache keys and regenerate the affected pages without the user lifting a finger. Phase 166-05 ties it all together with barrel exports, an integration test, and host CSS (`3db18e540`). What this phase buys is the property that the desktop app is not a separate codebase; it is the same dashboard, rendered the same way, with the same data sources, presented in a native window. No dashboard feature diverges, no renderer is duplicated. The web dashboard stays live and identical.

**Phase 167 is the first-boot calibration wizard.** Phase 167-01 implements user-style types, a Zod schema, YAML serialization, and a CSS bridge from `user-style.yaml` to DOM custom properties (`3b4797274`, `8b85b8976`). Phase 167-02 implements the wizard state machine with navigation and skip logic (`13ca851f8`). Phase 167-03 implements the first two screens: color picker with swatch grid and palette preview (`c4d2994e0`), and CRT settings with sliders and engine preview (`7622c2c7d`). Phase 167-04 implements the theme mode screen with light/dark toggle and preset cards (`436c8a410`) plus the wizard orchestrator with barrel exports and `main.ts` first-boot gate (`87dfb1406`). The wizard is three screens, skippable, and idempotent — running it a second time produces the same result from the same inputs. The `user-style.yaml` output is human-readable and version-controllable, so a user who has tuned their wizard output on one machine can copy the file to another machine and skip the wizard entirely. The scope decision not to add "advanced" sub-screens is deliberate: the wizard is for first-boot orientation, not for every preference the app will eventually support. Later preference work lands in a dedicated settings surface.

**Phase 168 ties delight to decency.** Phase 168-01 implements boot types, chipset definitions, skip detection, and the chipset initialization state machine (`8004d26ab`, `8d53f442b`) — four Amiga chipsets (Agnus, Denise, Paula, Gary) initialize in sequence with timing-driven transitions. Phase 168-02 implements the desktop background with copper list mode switching (`e6cd1b239`), so the background can render either as a static palette or with copper list effects depending on user preference. Phase 168-03 implements the boot renderer with chipset animation and skip handling (`c672cfa0c`) — click or keypress skips the boot sequence, and the skip state persists across sessions for that option. Phase 168-04 implements the accessibility detector with auto-activation on OS preferences (`099fb8a98`) and integrates the boot sequence, desktop background, and accessibility detector into `main.ts` (`dc3f8535e`). The accessibility rule is non-negotiable: `prefers-reduced-motion` disables CRT effects automatically at boot, and `prefers-contrast` applies a high-contrast palette automatically at boot — both without user intervention. The retro aesthetic never comes at the cost of usability, because the OS-level signals take precedence over the app's defaults. This is the right default behavior for a desktop app that wants to be delightful, not aggressive.

**Quiet foundation — the Amiga tribute is load-bearing.** Every later GSD-OS feature — the Minecraft knowledge world in v1.22, the mission infrastructure in v1.23, the conformance audit in v1.24, the ecosystem integration in v1.25, the Aminet archive extension pack in v1.26, the knowledge packs in v1.27, the Den Operations in v1.28 — either runs inside this desktop shell or is tested against it. ## Retrospective

### What Worked

- **Tauri v2 with a capability-scoped ACL as the security backbone.** Restricting webview access to specifically-named Rust commands — enforced at the Tauri layer, not in command bodies — is the right security model for a desktop app that manages execution pipelines. Every later Rust command adds its permission TOML in the same motion, so the capability surface stays visible and reviewable during PR review. The 22 autogenerated TOMLs at tag v1.21 are the audit trail.
- **WebGL2 CRT engine with a CSS fallback path.** The multi-pass post-processing pipeline (scanlines, barrel distortion, phosphor glow, chromatic aberration, vignette) is the personality of the app, but the CSS fallback means non-WebGL browsers still work. Progressive enhancement done right — the delight is opt-out-by-OS-capability rather than all-or-nothing. The performance measurer in phase 160-01 means the engine can self-tune the effect intensity under frame-time pressure.
- **Accessibility auto-detection built into the boot sequence, not into a settings panel.** `prefers-reduced-motion` disables CRT effects at boot; `prefers-contrast` applies a high-contrast palette at boot. Both happen without user intervention. A settings toggle for accessibility is a settings toggle that most users never find; an OS-level auto-detection is a default that works for everyone who set their OS preferences correctly.
- **Watermark-based PTY flow control as the backpressure primitive.** Pause at high-water, resume at low-water. This is the same backpressure mechanism TCP uses, and it is the difference between a terminal that stays responsive at `find / -type f` throughput and one that locks up for tens of seconds. The pattern is proven at scale; applying it at the PTY boundary is cheap and correct.
- **Red-green TDD rhythm preserved across all 11 phases and 34 plans.** Every implementation commit is preceded by a test commit. Future readers can `git show <test-sha>` to read the requirement as code before the implementation exists — a built-in requirement-to-test trace that costs nothing extra at authoring time.
- **Dashboard reuse via IPC bridge rather than reimplementation.** Phase 166 runs the existing `src/dashboard/generator.ts` through a Rust-launched Node subprocess and renders the output in a sandboxed surface. The web dashboard stays live and identical. No renderer is duplicated. The desktop gains all six pages plus watcher-driven refresh for free.
- **The calibration wizard as three screens plus skip.** Color picker, CRT settings, theme mode. Three screens. Skippable. Idempotent. The `user-style.yaml` output is human-readable and version-controllable. Scope discipline — resisting the temptation to add "advanced" sub-screens — is what keeps the wizard under a minute on first boot.

### What Could Be Better

- **11 phases is the largest single release so far in the v1.x line.** The scope spans Tauri shell, WebGL engine, palette system, PTY terminal, tmux bridge, window manager, shell, dashboard bridge, calibration wizard, and boot sequence. Any one of these could have been its own release. The coupling risk is that a bug in the PTY layer affects the boot-sequence integration path through `main.ts`, and the 228-file diff makes bisect rounds more expensive than they should be. Future foundation releases should consider splitting at plausible seams (engine, terminal, shell, boot) if the scope nears this size again.
- **OKLCH palette generation is sophisticated but hard to validate by unit test alone.** Perceptually uniform color interpolation is the right choice theoretically, but verifying that the 5 presets (Amiga Kickstart 1.3, Workbench 2.0, Workbench 3.1, C64, custom) actually look correct requires visual inspection — and visual inspection is not part of the 636-test count. A screenshot-diff harness against reference palette outputs would close the validation gap; v1.21 does not ship that harness.
- **Copper list raster effects are a personality win but a maintenance debt.** The per-scanline color manipulation shader in phase 161-03 is the kind of detail that defines the project's aesthetic, but the shader has no cross-platform visual equivalent — if the WebGL2 path fails on a given GPU driver and the CSS fallback activates, the copper list effects silently become flat color. This is the correct graceful-degradation behavior, but it means the app looks different on different machines in a way that is not always obvious to the user.
- **Frameless window chrome is platform-sensitive.** Phase 164-02's window chrome + drag/resize uses pointer events to implement the drag semantics that frameless Tauri windows lose when you remove native chrome. The implementation works on Linux, macOS, and Windows at tag v1.21, but every desktop platform has subtly different window-manager conventions (snap-to-edge on Windows, Rectangle-style tiling on macOS, tiling WMs on Linux), and the v1.21 implementation makes one set of choices rather than adapting to each platform's conventions. A per-platform chrome adapter would close the gap.
- **The calibration wizard ships three screens; five would be defensible.** The scope decision to omit "advanced" sub-screens (font size, icon scale, animation speed, sound, keyboard repeat) is correct for a first-boot orientation flow but leaves those preferences without a home until a dedicated settings surface lands. Users who want to tune beyond the wizard's three screens must edit `user-style.yaml` directly — doable, documented, but not first-class.

## Lessons Learned

1. **Foundation releases should make their load-bearing decisions at v1.0-of-the-foundation, not later.** v1.21 made the security model (capability ACL), the webview boundary (named commands only), the accessibility defaults (OS-preference-driven), and the two-root layout (`desktop/` + `src-tauri/`) all at the same release. None of these were retrofitted. The difference between a foundation that ages well and one that rots is whether these decisions ship at foundation time or get bolted on after the surface has grown.
2. **A capability ACL is better than in-command auth checks.** The Tauri v2 ACL rejects unlisted commands at the framework layer, before the command body executes. In-command auth checks routinely produce privilege-escalation bugs where one code path forgot the check. The ACL model prevents the bug class by construction. Every later Rust command inherits the pattern because adding a command without its TOML fails the build.
3. **Bidirectional IPC needs three patterns, not one.** Commands (webview→Rust), events (Rust→webview), and channels (streaming) each serve a distinct communication need. Trying to unify them into a single pattern would add complexity, not reduce it. The phase 158-03 contract shipped all three as first-class types, so later phases never have to reach for the wrong primitive.
4. **Watermark flow control at the PTY boundary is essential, not optional.** Without watermark-based backpressure, a fast-producing process overwhelms the xterm.js renderer and the UI locks up. With it, the terminal stays responsive at any PTY throughput because the producer is literally told to stop. The same pattern works at any producer/consumer boundary where buffer pressure can exceed render rate.
5. **Boot sequences should be skippable, and the skip should persist.** The Amiga chipset animation is delightful on first boot and annoying on the tenth. Click-or-keypress skip respects the user's time without removing the experience entirely, and persisting the skip-preference across sessions means users who want to skip don't re-opt-out every cold start.
6. **Accessibility belongs in the boot path, not in the settings panel.** `prefers-reduced-motion` and `prefers-contrast` are OS-level signals the app can read before it renders its first frame. Using those signals to auto-configure the app produces the right default for every user who has set their OS preferences correctly, without requiring them to find the app's accessibility toggle. A settings toggle for accessibility is a toggle most users never find.
7. **Copper list raster effects — the technically-unnecessary detail is the personality.** Per-scanline color manipulation in a fragment shader is not what makes the app functional. It is what makes the app feel like GSD-OS and not like a generic Tauri wrapper. Every project needs one or two technically-unnecessary details that encode the project's taste. They do not need many. They do need at least one.
8. **Reuse the web dashboard via IPC bridge, not reimplementation.** Phase 166 runs the existing `src/dashboard/generator.ts` in a Rust-launched Node subprocess and renders the output in the desktop window. The desktop inherits all six pages plus watcher-driven refresh. No renderer is duplicated. The web version stays live. This is the right pattern for any desktop app that wraps an existing web surface — bridge, do not fork.
9. **Red-green TDD with separate `test(N-M)` and `feat(N-M)` commits makes archaeology free.** Every plan in v1.21 has a failing-test commit followed by a passing-implementation commit. Future readers can `git show <test-sha>` to read the requirement before the implementation exists. This is a built-in requirement-to-test trace that costs nothing at authoring time and pays off every time someone asks "why does this do this?"
10. **Foundation releases trade volume for durability.** v1.21 is 106 commits, 228 files, +28,477 lines — large by any measure. It is large because it ships the foundation. The next five v1.x releases do not have to ship another Tauri shell, another WebGL engine, another PTY, another window manager. They build on v1.21. A smaller foundation release would have shipped a smaller surface and forced the next five releases to ship the remainder piecemeal. Sized correctly, foundation releases amortize across the next arc.
11. **The wizard is three screens because three is enough.** Color picker, CRT settings, theme mode. Scope discipline — resisting the temptation to add font size, icon scale, animation speed, sound, and keyboard repeat as sub-screens — keeps the wizard under a minute on first boot. Advanced preferences belong in a dedicated settings surface that lands when the app actually needs it, not in a first-boot flow that users want to get through.
12. **Per-platform chrome is the right abstraction but was not the right scope.** Phase 164-02's frameless chrome works on Linux, macOS, and Windows at tag v1.21, but does not adapt to each platform's window-manager conventions (Windows snap-to-edge, macOS Rectangle tiling, Linux tiling WMs). A per-platform chrome adapter is the right future shape; v1.21 ships the cross-platform baseline and leaves the platform-specific polish for a later release. Recognizing the distinction between baseline and polish is the discipline that keeps foundation releases from bloating into platform-adapter work.

## Engine Position

v1.21 is the opening brace of the GSD-OS desktop arc and the first foundation release of the native-app track. The v1.x line up to v1.20 operated entirely as a CLI tool with a web dashboard generated to `dist/` for browser consumption. v1.21 introduces `desktop/` (TypeScript webview) and `src-tauri/` (Rust backend) as two new long-lived roots in the tree, establishes Tauri v2 as the desktop framework, WebGL2 as the visual substrate, `portable-pty` as the terminal primitive, and the Amiga Workbench as the visual vocabulary. v1.20 made the web dashboard cohere; v1.21 takes that cohered surface and presents it as a native app. Every GSD-OS feature after v1.21 — v1.22's Minecraft knowledge world, v1.23's mission infrastructure, v1.24's conformance audit, v1.25's ecosystem integration, v1.26's Aminet archive extension, v1.27's knowledge packs, v1.28's Den Operations, v1.31's MCP host, v1.39's Bootstrap & READY Prompt — either runs inside this desktop shell or is tested against it. The capability ACL introduced in phase 158-04 becomes the security model every new Rust command inherits. The WebGL CRT engine introduced in phase 160 becomes the visual vocabulary every GSD-OS surface inherits. The calibration wizard introduced in phase 167 becomes the first-boot template every future app in the project reuses. The accessibility auto-detection introduced in phase 168-04 becomes the non-negotiable accessibility default. The `desktop/` + `src-tauri/` two-root layout becomes the frontend/backend convention. None of these patterns were retrofitted; they all shipped at v1.21 because the scope was sized to make them shippable at foundation time. Every later GSD-OS release builds on this foundation additively.
