# v1.49.780 — Retrospective

## decisions

- **Infrastructure-first migration pattern.** First commit added `src/cli/dispatch.ts` with an empty `REGISTRY` and wired it into `cli.ts` main() with a fall-through-to-legacy-switch path. Every subsequent migration was a self-contained, bisectable commit that moved one case (or one batched group) from the switch into the registry. The alternative (extract all cases first, then build dispatch) would have left the codebase in a broken-mid-refactor state across multiple commits.
- **Registry contract uses `CliContext`; existing command modules unchanged.** When the user picked the strict-CliContext signature for the dispatch contract, I noticed that 102 existing command modules already used `(args: string[]) => Promise<number>` as their signature. Forcing all of them to take `CliContext` would have expanded the refactor by 10×. Resolution: registry handlers are `(ctx: CliContext) => Promise<number | void>`, but each handler is a thin closure that adapts to whatever signature the underlying command function uses. New commands can opt into CliContext directly.
- **Batched migrations for structurally-uniform cases; atomic for structurally-different ones.** The 12 large inline cases each got their own commit (suggest, feedback, refine, delete, invoke, agents, team, dacp, rollback, skill, plus printHelp). The 45 simple-dispatch cases (`(args) => commandFn(args.slice(1))` shape) went into a single batched commit because their structure is identical and bisect granularity stays useful per-category. Similarly the 14 inline-arg-parsing cases batched as inline closures in dispatch.ts.
- **Inline closures over per-module extraction for the final 14.** When deciding whether to extract the 14 remaining inline-arg-parsing cases into their own command modules, I picked inline closures in dispatch.ts instead. Reasoning: the arg-parsing IS the routing concern; extracting 14 thin modules whose only job is "parse flags then call commandFn" would have created module sprawl. dispatch.ts ended up at 543 lines as a result — a tradeoff worth accepting for the cleaner end-state of cli.ts at 120 lines.

## lessons_learned

- **Three-step verify pattern after risky Edits.** When I tried to extract `showHelp` via an in-place Edit that only matched the first chunk of the 328-line template literal, the file got corrupted (dangling free text outside any function body, ~260 lines of broken structure). Falling back to `sed -i 'Nd'` line-range deletion was the right move — surgical Edits on multi-hundred-line spans are fragile when the template literal contains no unique boundary markers. Discipline: for any function body >100 lines, plan to use Write (full file replacement) or sed (line-range deletion) rather than Edit (find-and-replace).
- **Pre-tag-gate CI status is checked against `origin/dev` SHA, not the local working tip.** When I ran pre-tag-gate before pushing, it grepped the GH API for the CI run on the v779 SHA — which had been failing for an unrelated reason (the `sc:start missing` cluster of test fixtures that aren't present in CI). That blocked the gate even though my local changes were clean. Resolution: push to origin/dev first to trigger a fresh CI run on the right SHA, then re-evaluate. The enumerated `SC_SKIP_CI_GATE_TESTS=<csv>` override (Lesson #10185 / v1.49.636 C6) handles the case where the failures are pre-existing and known-acceptable.
- **`createScopedStoreAndIndex` was duplicated.** After the migration, both cli.ts and dispatch.ts had their own copy of `createScopedStoreAndIndex(scope)`. Tolerable trade-off — extracting into a shared helper would mean a new file for a 5-line function — but if a third caller emerges, it's worth promoting to `src/storage/scoped-store-helper.ts` or similar.

## surprises

- **cli.ts had GROWN from 1864 → 2132 between when the v777 REVIEW ledger was authored and v780.** The original sweep at 2026-05-26 noted 1864 lines; my baseline at 2132 was 268 lines bigger. Three counter-cadence ships (v777-779) added or modified ~268 lines of dispatch surface during that window. The ledger's target of `<300` was based on the smaller starting point, so the actual reduction needed was larger.
- **Two source-grep tests in `src/cli/commands/{eval,critique}.test.ts` validated wiring by greping `src/cli.ts` for case strings.** This is a sensible pattern — verifies the command is actually registered in the CLI — but it became brittle when the registration moved to `dispatch.ts`. Three tests failed and needed updating. The fix was trivial (change the file path in the test), but the broader lesson is: source-grep wiring tests should grep the actual registration site, not assume it's in cli.ts forever. Future tests of this shape should be written against `dispatch.REGISTRY` directly (introspect the array) rather than the textual source.
- **The agents command's lazy import of `AgentSuggestionManager` survived the migration unchanged.** I assumed I'd need to convert the dynamic `await import('./agents/index.js')` to a static import in the extracted module. But the extracted `src/cli/commands/agents.ts` preserves the dynamic import — keeps the original startup-cost behavior, no surprises. Lesson: don't change import strategy during a dispatcher refactor; preserve the original lazy-vs-eager pattern.

## what_could_be_better

- **`dispatch.ts` is now 543 lines with 73 entries.** Some entries are 1-line inline closures (`(ctx) => commandFn(ctx.args.slice(1))`); others are 30-line inline-arg-parsing blocks (sensoria, export). The file scrolls. Possible future refactor: split the inline-arg-parsing blocks into per-command modules so dispatch.ts becomes a pure import-and-register file. But that's a follow-on, not a regression.
- **No automated test verifies that every command in `printHelp()`'s text actually exists in `REGISTRY`.** A test like "every command listed under `Commands:` in printHelp is also a registry alias" would catch help-text drift. Could be added in a future cleanup ship.
- **The `_unusedShowHelpMarker` stub I accidentally left during the printHelp extraction was caught only by reading the file afterward.** A linter rule against orphan function stubs would have caught it instantly. Not worth introducing for this single occurrence, but worth noting.

## what_worked

- **The fall-through pattern.** Adding dispatch skeleton + wiring it as try-first-then-legacy-switch in commit #1 meant every subsequent commit was independently shippable. At any point in the 13-commit sequence, the CLI was fully functional — half the commands resolved via dispatch, half via the legacy switch.
- **Running tsc + the cli.test.ts subset after every commit** kept me from accumulating drift. Each commit was verifiable in ~5 seconds before proceeding.
- **Reading the entire remaining switch surface BEFORE Batch A.** When I shifted from per-case extraction to batched migration, I read all ~777 remaining lines of main() in two Read calls. That gave me the full picture (which cases were simple-dispatch vs inline-arg-parsing) and let me write the dispatch.ts entries + the slim cli.ts in two focused commits instead of churning case-by-case.
- **Using `sed -i 'Nd'` to recover from the corrupted-Edit state for printHelp.** When in-place Edit broke the file, falling back to sed for line-range deletion + a fresh tsc check was 30 seconds. Knowing both tools' strengths matters.
