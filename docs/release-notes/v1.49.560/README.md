## Summary

**The release-uplift mission took every shipped release from its original quality score to A-grade documentation.** The mission ran across three sessions between 2026-04-17 and 2026-04-18. Phase 01 uplifted seven patches. Phase 02 uplifted 54 milestones. Phase 03 uplifted 129 feature releases. Phase 04 uplifted 57 NASA degrees. Phases 05 and 06 uplifted approximately 355 Seattle360 degrees across the full v1.49.135–v1.49.500 range. The final 44 degrees (v1.49.456–v1.49.499) shipped in this session through nine waves of 4–5 parallel subagents per wave, recovering through one rate-limit wall at Wave 51 by re-dispatching two degrees after the reset. Every release that ended the mission was at or above A(90) on the scorer, with most at A(98). The baseline now stands at 602 A · 0 B · 0 C · 0 D · 0 F.

**The scorer is calibrated against `tools/release-history/score-completeness.mjs` with ten dimensions summing to 100.** Each README is graded on: 10+ field header block (10 pts), 2,000-word Summary with 40+ proper nouns and 5+ cross-sync paragraphs (15 pts), Key Features table ≥5 rows (10 pts), Part A depth with 8+ bold lead-in bullets (10 pts), Part B depth with 8+ bold lead-in bullets (10 pts), Retrospective structure with `### What Worked` + `### What Could Be Better` (15 pts), 8+ numbered Lessons Learned (10 pts), 15+ Cross-References (10 pts), running ledgers (5 pts), infrastructure block with Dedication + Engine Position (5 pts). Non-degree releases (patches, features, milestones) skip the Part A / Part B dimensions and rescale to 100 — the scorer normalizes type-appropriate grading rather than penalizing structural differences.

**The mission produced an observable quality lift: avg 97.4 across 602 releases, recent-20 avg 98.8.** Baseline was captured by `tools/release-history/quality-drift-check.mjs --update-baseline` at the mission-complete moment. The recent-20 average is 1.4 points higher than the overall average because the last 20 releases were written under the most calibrated prompt template, while the earliest Phase 01 patches (v1.8.1, v1.49.1, v1.49.61, v1.49.75) absorbed the pipeline's shakedown costs and scored A(91) instead of A(98) on some rubric dimensions. No releases fell below A(90) at mission close. The scorer output is the authoritative record: `docs/RELEASE-HISTORY.md` carries a per-release grade column that reflects the live scorer state, refreshed by `tools/release-history/refresh.mjs`.

**RELEASE-HISTORY.md also got three cleanup passes after the uplift closed.** First, 258 verbose Degree H1 titles in range v1.49.230–v1.49.499 were shortened: the scorer pulls the release name from the H1 header, and many Seattle360 degrees had 300–500 character H1s stuffed with parenthetical marketing blurbs ("THE OLYMPIA SEQUENCE CONTINUES, THE NO-BASS TWO-GUITAR ARCHITECTURE AT FULL REALIZATION, ..."). The new `tools/release-history/shorten-degree-h1.mjs` walks `docs/release-notes/<v>/README.md`, detects verbose H1s using a balanced-parens parser plus em-dash tagline trimmer, and emits clean short names like "Degree 358: Sleater-Kinney Dig Me Out + All-Species Dawn Chorus" without touching any other part of the README. The tool is idempotent and accepts `--from`/`--to` version ranges for re-runs.

**Second, 1,310 chapter files were published into `docs/release-notes/<v>/chapter/` via `tools/release-history/publish.mjs --execute`.** The chapter files (00-summary.md, 01-features.md, 03-retrospective.md, 04-lessons.md, 99-context.md) are authored in `.planning/roadmap/<v>/` by the chapter generator and were already present on disk, but they had never been copied into the tracked release-notes tree. The publisher copies only allowlisted basenames, runs a leak-scan gate to block any forbidden pattern, and writes idempotently via source-checksum tracking in `release_history.publish_target`. After publish, the Retro and Lessons columns in `RELEASE-HISTORY.md` link to real files instead of showing the `(no file)` placeholder — every release in the table now has a clickable retrospective chapter.

**Third, 346 commit counts were corrected via a degree-aware git backfill.** The original extractor in `tools/release-history/ingest.mjs` matched `\*\*Commits?:\*?\s*(\d+)` greedily against README content, which silently captured SHA prefix digits from lines like `**Commits:** 4525387a4 (1)` — the 4525387 is the start of a SHA, not a count. The backfill from git tags also hit a snag: many Phase 06 tags point at later degree ship commits due to a bulk re-tag that happened during the mission execution window, so `git rev-list --count vPREV..vCURR` returned 0 for 219 releases. The new `tools/release-history/backfill-degree-commits.mjs` resolves both: it queries releases whose name matches `Degree N:`, finds the actual content commit with `git log --grep "feat(www): degree N"`, and counts commits between the degree's content commit and its predecessor's. For the two degrees that shipped as part of a wave commit ("degrees 106-108 — Knife Knights..."), the grep rightfully misses and they're patched to 1 commit explicitly. The extractor itself was hardened with three-pattern fallback: prefer `SHA (N)` form, then standalone `(N)`, then bare digits that are neither followed by hex chars nor over 100,000 in magnitude.

**The cumulative effect on `docs/RELEASE-HISTORY.md` is visible in every column.** Grade column: all 602 rows now show A. Retro column: all 601 rows that have a retrospective chapter now link to it (one outlier, v1.49.551, correctly shows `✓ (no file)` because its retrospective lives only in the README). Lessons column: same pattern — 601 rows link to their `04-lessons.md`. Commits column: zero bogus numbers, zero rows with 0 where a commit actually exists, and the two legitimate "0 commits because this release was a pure doc tag" rows are gone. Name column: every Degree release has a scannable short name under 120 characters. The table is now readable end-to-end for the first time since the bulk re-tag.

**The mission carried three tool additions into the `tools/release-history/` directory for future re-runs.** `shorten-degree-h1.mjs` is a pure front-matter editor that will stay relevant as long as new degree releases are authored with the engine's standard H1 pattern. `backfill-degree-commits.mjs` is specifically for the Phase 06 bulk-re-tag artifact and may never need to run again, but the degree-grep lookup pattern generalizes to any future situation where tags diverge from content commits. The patch to `extractCommits` in `ingest.mjs` is permanent calibration — any future README that writes `**Commits:** SHA (N)` will now parse correctly. All three tools ship with `--dry-run` defaults (or `--execute` for writes) and console.error progress reporting, consistent with the `tools/release-history/` convention established by `scan.mjs`, `backfill-git-stats.mjs`, and `publish.mjs`.

**This release does not touch source code, runtime behavior, or test surface.** Zero changes to `src/`, `src-tauri/`, `desktop/`. The test suite continues to pass at 23,808 tests (the count inherited from v1.49.550's final state). No new runtime dependencies. No migrations. The package.json / tauri.conf.json version fields bump from 1.49.557 to 1.49.560 to match the tag state — three version points catch up, reflecting v1.49.558 (Apollo 1), v1.49.559 (Lunar Orbiter 3), and this release. The next release in the engine (v1.49.561) would be Degree 56 (Surveyor 3, per the NASA chronological reorder session).

## Part A: Release-Uplift Mission Execution

- **278 per-release uplift commits across three sessions.** Phase 01 seven patches on 2026-04-17 morning. Phases 02–05 on 2026-04-17 afternoon and evening. Phase 06 through the pause, reset, and final 44 degrees across the 2026-04-18 window. Every commit is independently bisectable with a commit message of the form `docs(release-notes): uplift vX.Y.Z Degree N to A-grade`.
- **5-wide parallel subagent dispatch proved the reliable cadence.** Each wave dispatched one agent per degree, with all five agents running concurrently and the orchestrator collecting JSON results. Six or more agents per wave hit the per-message token limit; four or fewer underutilized the parallelism window. Wave timing averaged ~10 minutes wall clock per degree, with some outliers at 18+ minutes when the agent had to resolve cross-branch fact conflicts.
- **Fact reconciliation was the highest-leverage prompt instruction.** The line "Prefer retrospective.md + knowledge-nodes.json consensus over research.md and any prior README content" caught at least eight factual errors in prior READMEs across the final 44 degrees: wrong labels (Sub Pop vs Amphetamine Reptile for Gas Huffer, C/Z vs Combat for Skin Yard, Artist Direct vs V2 for Blood Brothers), wrong band lineups (Alcohol Funnycar members, Botch's Knudson/Latona reversal, Swallow's vocalist, Green River's drummer), wrong geographic origins (Oak Harbor vs Seattle for The Accused, U-District vs Aberdeen for Skin Yard), wrong species (grizzly bear vs black bear at degree 325), and wrong acronym expansions (TAAS = These Arms Are Snakes, not "The Art Anew Sound").
- **Tag-offset handling was built into every prompt.** Many Phase 06 tags point at the next degree's ship commit because of a bulk re-tag that happened during the execution window. The instruction "If the tag points at a LATER degree's ship commit, infer actual content commit from chapter files + git log 'degree N'" produced 100% content-commit resolution across the final 44 degrees. Agents learned to find the right commit via chapter file presence plus git log grep, independent of the tag position.
- **The scorer regex for Part A / Part B bold lead-ins was documented up front.** "Do NOT put italic asterisks inside bold lead-ins — breaks scorer regex" prevented the v1.49.457 Part-A-scoring-7/10 issue from recurring. The regex is `/^[-*]?\s*\*\*[^*\n]{5,200}\*\*/` and it intentionally rejects bold spans that contain asterisks, because those are ambiguous with italic inside bold.
- **Rate-limit salvage protocol recovered 488 and 490 cleanly.** Wave 51 dispatched five agents. Three completed (486, 487, 489). Two (488, 490) hit the weekly cap mid-run with no disk writes. After the 6pm PT reset, both were re-dispatched from the same template and completed on first try. No content was lost because the salvage protocol is: if the agent didn't write to disk, re-dispatch from scratch; if it wrote a partial file that scores A, commit manually.
- **DB refresh between waves kept scorer state current.** Every 2–3 waves, `tools/release-history/refresh.mjs --fast` ran in the background to update the Postgres mirror from the committed READMEs, so the next dispatch's "before" score came from live data rather than stale baseline. The refresh is idempotent and takes ~90 seconds on this repo.
- **Running ledgers 3/5 is the stable ceiling for degree uplifts.** The scorer expects two distinct ledger markers (one for artist / S36 side, one for species / SPS side). Most uplifts produced a single merged ledger and scored 3/5 instead of 5/5. The 2-point ceiling absorbs into the final average — effectively every other dimension scored perfect, with the mission baseline at 97.4 = 100 − 2.6 attributable mostly to running-ledgers partial credit. A future prompt template could add a "two distinct ledger markers required" line to push this to 5/5, but the return is marginal for the prompt-engineering cost.

## Part B: RELEASE-HISTORY.md Cleanup

- **Short-title shortener uses a balanced-parens parser, not a regex.** The first shortener draft tried `/\s*\(([^()]*)\)/g` to strip parentheticals, but nested parens (Mountain Quail's `(*Oreortyx pictus*, ... (2013), ...)`) defeated it. The second draft uses a stateful scanner that tracks paren depth character-by-character, so `(...(2013)...)` correctly gets dropped if the outer content has commas or exceeds 20 chars, and short isolated parens like `(Annual)` or `(Indie)` survive.
- **Em-dash tagline trimmer keeps the first short tag and drops the rest.** Many H1s carried multi-tagline chains like "Band + Species — TAG A — TAG B — TAG C". The trimmer keeps "Band + Species — TAG A" if TAG A is ≤60 chars and comma-free, otherwise drops all taglines. This preserves meaningful short tags like "THE CIRCLE CLOSES" and "THE CENTENNIAL" while stripping the "THE HYPNOTIC ENGINE AND THE STUTTERING DRUM — FIRST KRAUTROCK — THE E=6 THREE-PEAT ENDS — CAPITOL HILL BEGINS — PICIDAE FOUR-STREAK" chains.
- **Publisher moved 1,310 chapter files from .planning to docs in a single run.** `publish.mjs` was built with a per-file leak-scan gate, source-checksum idempotency, and an allowlist of basenames (00-summary.md, 01-features.md, 03-retrospective.md, 04-lessons.md, 99-context.md). Before this release, the publisher had never been executed at scale — the dry-run had been the default. The `--execute --since v1.49.229` flag drove the bulk publication in one pass.
- **Commit-count backfill required degree-grep, not tag-diff.** The standard `backfill-git-stats.mjs` counts commits between tags, but the Phase 06 bulk re-tag left many tags pointing at later commits. The degree-specific backfill finds the actual content commit (`feat(www): degree N`) and compares to the previous degree's content commit. For the Seattle360 + NASA overlap range, this produces correct counts like "1 commit between degree 357 and 358 on the www side," which is the truth of what shipped.
- **The extractCommits regex now has three-pattern fallback.** Pattern A matches `**Commits:** SHA (N)` and pulls N. Pattern B matches `**Commits:** (N)` for cases where no SHA is present. Pattern C matches bare digits only if they're neither followed by hex characters nor over 100,000 in magnitude — a belt-and-suspenders sanity check against SHA prefixes that happen to be all-digit.
- **Manual patching for wave-commit degrees.** Two degrees (107 and 108) shipped as part of a wave commit with message "feat(www): degrees 106-108 — Knife Knights to E=6 double maximum," so the degree-specific grep for "degree 107" or "degree 108" returns nothing. A one-line SQL patch set both to commits=1, reflecting the wave commit's reality: one commit shipped three degrees.
- **The final state is every row in docs/RELEASE-HISTORY.md is meaningful.** 602 releases listed. 602 A grades. 601 Retro links. 601 Lessons links. Zero bogus commit counts. Zero zero-commit rows. Short names across the entire v1.49.229–v1.49.499 range. The table works as both a human-readable release log and as the authoritative data source for `docs/RELEASE-HISTORY.md` downstream consumers.
- **The cleanup commit was 1,570 files / 35,335 insertions / 4,755 deletions.** The size reflects the reach: 258 README H1 edits, 1,008 chapter file moves, 562 new chapter files, plus the three tool additions. Every change is deterministic and re-runnable via the committed tools, so the next bulk re-tag or H1 drift can be repaired in under five minutes.

## Retrospective

### What Worked

- **The 5-parallel-subagent cadence held at production scale.** Over the full mission, approximately 50 waves dispatched. Zero runaway agents. Zero content conflicts between parallel commits. The per-release commit pattern means each agent's output is atomic and independently verifiable — the orchestrator never had to merge agent results, only collect and commit.
- **Fact reconciliation caught more bugs than the rubric did.** The scorer was already calibrated from earlier phases. What the final 44 degrees surfaced was content quality issues in the source material: prior READMEs with wrong band lineups, wrong labels, wrong dates. The fact-reconciliation prompt instruction produced a cleaner factual baseline than the original content.
- **Rate-limit salvage with re-dispatch worked exactly as documented.** The handoff noted "when a subagent hits the wall mid-commit, the README is usually already written." This held in most prior cases. For Wave 51, it didn't — and the backup protocol (re-dispatch after reset) handled it cleanly with no lost work.
- **The `tools/release-history/` conventions paid dividends.** The new tools (shorten-degree-h1.mjs, backfill-degree-commits.mjs) slot in with existing tool conventions: `--dry-run` default, `--execute` to write, console.error for progress, console.log JSON for results. Building new tools on these patterns took minutes, not hours.
- **Postgres as the source of truth isolated quality changes from README drift.** When the scorer flagged a release as failing, fixing the README was the remedy. When `extractCommits` was wrong, fixing the extractor and re-ingesting cleaned up the data. The separation between "source of truth on disk" (README) and "queryable derived state" (Postgres) made the pipeline compositional.
- **Short H1 enforcement caught a second-order content quality issue.** Beyond the RELEASE-HISTORY.md readability improvement, the short H1s are now the correct format for programmatic consumers that expect "Degree N: Artist + Species" patterns. Any future tool that parses release names (dashboard generators, index builders, search tools) gets a clean input.

### What Could Be Better

- **Rate limit dispatching near a soft-limit window is avoidable with better scheduling.** Wave 51 hit the 5-hour limit mid-run. The earlier handoff's lesson was "don't dispatch right before a weekly rollover"; the generalization is "check remaining quota before dispatching a new wave and pause if we're less than one-wave-worth of tokens away from the limit." Future missions should instrument this check into the dispatch loop.
- **The DB refresh step should run in-line after every 5 commits, not out-of-band.** Refresh-between-waves works but couples the orchestrator's rhythm to the refresh's 90-second window. If the refresh were triggered automatically by the commit hook (or by a debounced post-commit script), the scorer state would always be live and the orchestrator wouldn't need to manage refresh cadence.
- **Two-pattern-fallback regex extractors are the wrong architecture for complex input.** The extractCommits three-pattern fallback works, but the underlying issue is that README content is free-form and the extractor is a pattern match. A small AST-based parser for release headers would be more robust and more maintainable than `extractCommits`, `extractShipped`, `extractName`, `extractBranchTag`, each with its own regex.
- **Running-ledgers scoring is too strict for the typical prompt template.** Most releases produce one running ledger (combined S36 + SPS), and the scorer wants two (one per side). Either the rubric should accept the combined form as full credit, or the prompt template should enforce the two-ledger form. The current state (every release caps at 3/5 on this dimension) is a calibration issue.
- **Manual SQL patches for wave-commit degrees feel brittle.** The 106-108 wave commit case worked because I noticed it during verification. A more general-purpose fix would be to have `backfill-degree-commits.mjs` fall back to searching for "degrees X-Y" patterns, or to inspect the commit message for degree ranges. Two degrees isn't enough to justify the engineering cost, but if this pattern recurs, the fallback is worth building.
- **The mission retrospective lives in .planning/ and .planning/ is gitignored.** The mission's decision record, session notes, and final retrospective are all in `.planning/missions/release-uplift/` — intentionally local per the standing rule. But that means the learning lives only on my disk and doesn't travel with the repo. Future missions should consider whether the retrospective itself (or a summary) should ship as part of the release notes, so the history is shared.

## Lessons Learned

1. **Parallel subagents are cheaper than context.** Dispatching 5 agents in parallel per wave lets each one do deep factual reconciliation independently without bloating the orchestrator's context. The alternative (collapsing a wave into one big prompt) was tried early in the mission and consistently produced weaker output because no single pass has the depth to handle 5 degrees' worth of fact-checking.
2. **Fact-reconciliation beats prompt-rubric for quality.** The rubric gets the structure right. The "prefer retrospective.md + knowledge-nodes.json consensus" instruction gets the facts right. Both are needed, but the fact instruction is the one that caught 8+ content-quality issues that the rubric would never have detected.
3. **Tag position is not content position for bulk-re-tagged repos.** Any tool that relies on `git log vPREV..vCURR` needs to handle the bulk-re-tag case explicitly. Degree-grep backfill is one way. Storing content-commit SHAs in a release_history.release column would be another, and might be worth building for v1.50.
4. **Idempotency in cleanup tools means you can run them again.** `shorten-degree-h1.mjs`, `publish.mjs`, and `backfill-degree-commits.mjs` all check current state before writing. Running any of them on an already-clean repo is a no-op. This lets me re-run them confidently after subsequent drift without fear of double-applying changes.
5. **Magnitude sanity checks catch SHA-prefix regex bugs.** Commit counts between tags max out at a few thousand in any realistic repo. Numbers over 100,000 are almost certainly SHA prefixes being parsed as counts. The magnitude check in `extractCommits` is a one-line safety net that prevents the bogus-number class of bug across all future ingest passes.
6. **.planning/ gitignore is correct and the retrospective should still be surfaced.** The mission retrospective has lessons that future missions need. The correct resolution isn't to commit .planning/, it's to surface the retrospective summary in the release notes (as this README does, above) so the learning travels with the repo.
7. **Chapter publishing should have been a commit-time hook, not an end-of-mission batch.** 1,310 files moved in one commit is a reviewable blast radius, but it's bigger than ideal. If the publisher ran on every release-notes commit, the chapter files would stay in sync continuously and there'd never be a "publish everything" moment. Future release-notes tooling should consider hook-driven publishing.
8. **Short titles are a structural property of release naming, not a documentation style choice.** The H1 is consumed by the scorer, by RELEASE-HISTORY.md, by any external dashboard or search tool. When authors write 500-char H1s, they're implicitly shifting cost to every downstream consumer. The shortener is a repair tool; the long-term fix is a pre-commit linter that rejects H1s over ~150 chars.
9. **Re-dispatching from scratch is usually safer than salvaging partial work.** When Wave 51 had two agents hit the limit with no disk writes, I re-dispatched both with the same template. This produced clean results. The alternative (trying to extract partial work from the failed agent's transcript and finishing it in a new agent) would have been more token-efficient but carries higher fact-drift risk.
10. **The scorer is authoritative; the README is source-of-truth; Postgres is the mirror.** When the three disagree, the remedy depends on which is wrong. Scorer disagrees with README: fix the README or fix the scorer. README disagrees with Postgres: refresh. Postgres disagrees with git (commit counts): backfill. Having three layers with clear remediation paths made the cleanup pass tractable.
