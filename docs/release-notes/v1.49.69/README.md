## Summary

**K8S is the largest single-project entry in the PNW Research Series by line count, and the size is earned, not inflated.** The 69th Research project ships six modules totaling 3,003 research lines — Distributions & Architecture (392), Networking & Traffic Management (361), Storage & Persistence (421), GPU Scheduling & AI/ML (506), Security & Zero Trust (668), and Operations & Observability (655) — plus a 975-line LaTeX mission-pack source, a 174,258-byte pre-rendered PDF, and a four-file site shell (`index.html` 106 lines, `page.html` 184 lines, `mission.html` 89 lines, `style.css` 40 lines). Cumulatively the release lands +4,585 lines across 15 files in a single content commit (`147dca299`), a single docs commit (`e3156aa10`), and a merge commit (`4df8810c0`). The project crosses the Research-series threshold where a single project becomes a reference work rather than a study: other projects in the series cite K8S as the canonical Kubernetes-ecosystem map, and K8S in turn cites PSG (Pacific Spine Ground Truth), SAN (SANS Institute), and a forthcoming FoxCompute project as the infrastructure neighbors it composes with. The size is load-bearing because Kubernetes is a six-layer stack and cutting any layer would leave the map incomplete; the Research series does not do partial maps of load-bearing infrastructure.

**The six-module structure maps the actual Kubernetes stack, not a pedagogical one.** Module 01 (Distributions & Architecture) establishes the control-plane and node-plane components — kube-apiserver, etcd, kube-scheduler, kube-controller-manager, kubelet, kube-proxy, container runtime — and names the distribution landscape (K8s v1.35 upstream, K3s for edge, MicroK8s for development, KubeEdge for IoT, OpenShift and Rancher for managed enterprise). Module 02 (Networking & Traffic Management) traces the rewrite currently underway from iptables-based kube-proxy to eBPF-based Cilium, from Ingress to Gateway API v1.0 (graduated 2023, now the standard), from ClusterIP/NodePort/LoadBalancer service types to service mesh (Istio, Linkerd) and bare-metal load balancers (MetalLB). Module 03 (Storage & Persistence) walks the CSI (Container Storage Interface) that standardized volume plugins in v1.13, the spectrum from local-path (Longhorn) to distributed (Rook/Ceph, OpenEBS Mayastor) to cloud-native, and the storage decision framework for matching workload to backend. Module 04 (GPU Scheduling & AI/ML) is the most FoxCompute-relevant module. Module 05 (Security & Zero Trust) is the longest. Module 06 (Operations & Observability) is the second-longest. The ordering — infrastructure first, networking, then state, then the workloads that motivated Kubernetes in the first place, then the security posture that keeps it honest, then the operational loop that keeps it running — is the actual mental model practitioners hold, not the one docs teach.

**The GPU-scheduling module is the release's load-bearing FoxCompute bridge.** Module 04 (506 lines) maps NVIDIA's container platform end-to-end: the NVIDIA GPU Operator installation path, Dynamic Resource Allocation (DRA) which became GA in Kubernetes v1.34, Multi-Instance GPU (MIG) partitioning for A100/H100/B200 cards, Multi-Process Service (MPS) for co-scheduling, time-slicing for smaller GPUs, Kueue and Volcano for batch-queue scheduling, topology-aware scheduling that knows which GPUs share a PCIe switch or an NVLink fabric, and the RTX 5090 specifications relevant to a single-node or small-cluster ML workstation. For FoxCompute — the spine of Fox Companies' compute strategy — this module is directly actionable. The RTX 4060 Ti baseline that the gsd-skill-creator project currently runs on and the future multi-GPU fleet implied by the Erdős research program, the Artemis II mission, and the Memory Arena VRAM tier all require the scheduling primitives this module documents. Module 04 is the module the rest of the project could lose and still be useful, but the module that makes the rest of the project most valuable to the builder.

**The security module earned its 668 lines by walking five defense layers explicitly rather than abstractly.** Module 05 is the longest module in K8S and longer than most entire Research projects. The length is structural. Zero-trust Kubernetes is a five-layer discipline — (1) cluster hardening (CIS benchmarks, kube-bench, minimal base images, distroless), (2) network policies (Calico / Cilium NetworkPolicy, default-deny, namespace isolation), (3) RBAC (role binding, service accounts, least privilege), (4) admission control (OPA Gatekeeper, Kyverno policies, PodSecurity admission), and (5) runtime detection (Falco, Tetragon, eBPF-based anomaly detection). Each layer requires specific configuration examples, not abstractions, or readers cannot implement them. The module covers KEP-4317 pod certificates (which replace the long-standing service-account-token pattern for zero-trust pod identity), Kyverno policy examples for enforcing image signatures and resource limits, Falco versus Tetragon tradeoffs for runtime security (userland versus kernel-assisted), and a phased implementation roadmap that does not assume a greenfield cluster. Security modules that cover fewer than five layers produce false confidence; cutting the fifth layer (runtime detection) is the most common mistake in production Kubernetes and the module is explicit that the cut is not safe.

**The operations module closes the loop with GitOps, observability, and FinOps, and the combination is the actual job.** Module 06 (655 lines, second-longest) walks ArgoCD end-to-end — repository structure, App-of-Apps pattern, sync waves, progressive delivery with Argo Rollouts, drift detection, and automatic reconciliation — then pivots to three-pillar observability (Prometheus for metrics, Loki for logs, OpenTelemetry for traces, Grafana as the common pane), self-healing through liveness/readiness/startup probes and PodDisruptionBudgets, VPA (Vertical Pod Autoscaler) for right-sizing, HPA (Horizontal Pod Autoscaler) and KEDA for demand-based scaling, FinOps cost attribution with Kubecost, and Velero for backup and disaster recovery. The module argues that Kubernetes is not a container platform, it is an operations platform: the containers are substrate, and the job is the closed loop of observe → decide → act that Kubernetes runs continuously. This is the meta-level pattern the release's title ("The Orchestrator") names. Operations teams that deploy Kubernetes without the full closed loop end up with an expensive container runtime that they still have to operate by hand; the module is a checklist for not ending up there.

**K8S connects directly to FoxCompute and the CERN-scale infrastructure vision PSG documents.** The PSG (Pacific Spine Ground Truth) Research project maps CERN's OpenStack/Ceph deployment at petabyte scale as the ground truth for what planet-scale compute actually requires. K8S is the workload-orchestration layer that sits on top of that infrastructure. The GPU-scheduling module is the bridge that connects FoxCompute's AI/ML compute requirements to the Kubernetes primitives that schedule them. The security module is the bridge to SAN (SANS Institute) defensive posture. The GitOps section connects to GSD2's own orchestration patterns — GSD is the system that orchestrates development work, Kubernetes is the system that orchestrates container work, and the meta-pattern (a closed-loop orchestrator that runs other work) is the same in both cases. The cross-references are not ornamental; they are the scaffolding that lets the Research series accumulate into a coherent architecture study rather than a collection of isolated papers.

**The CNCF project landscape is the modern Linux-distribution problem, restated.** The Cloud Native Computing Foundation now hosts roughly 160 projects across graduated (Kubernetes, Prometheus, Envoy, containerd, Helm, Cilium, etcd, Harbor, Istio, and others), incubating, and sandbox tiers. Picking a Kubernetes distribution is really picking a CNCF stack: upstream Kubernetes with Cilium networking, Longhorn storage, Prometheus observability, ArgoCD GitOps, and Kyverno policy is a completely different operational profile than upstream Kubernetes with Calico networking, Rook/Ceph storage, OpenTelemetry observability, Flux GitOps, and OPA Gatekeeper policy. Both are valid. Neither is interchangeable after the fact. Module 01 makes the decision framework explicit: start from the workload, work backward through the stack choices, and the CNCF project list stops being a menu and becomes a set of coupled decisions. This is the pattern Linux distributions taught the industry thirty years ago; Kubernetes is learning it now.

**The mission-pack triad (LaTeX + PDF + HTML) continues the editorial discipline established by JNS.** The mission pack at `www/tibsfox/com/Research/K8S/mission-pack/` contains `kubernetes-mission.tex` (975 lines LaTeX source), `kubernetes-mission.pdf` (174,258 bytes pre-rendered), and `index.html` (103 lines landing page). Readers who want to tinker with the source take the LaTeX. Readers who arrive via the site take the HTML landing page and navigate from there. The triad is redundant by design — three copies of the same content at three different fidelities — because the Research series optimizes for reader autonomy over storage efficiency. The mission pack is the long-form companion to the six research modules; the six modules are the navigable Markdown version, the mission pack is the single-PDF version a reader can archive, print, or cite as a single artifact.

**The color palette is infrastructure-coded rather than decorative.** The K8S project uses a three-color palette — steel `#263238`, electric blue `#1565C0`, graphite `#37474F` — that reads as industrial infrastructure rather than as document branding. Steel is the dominant color, matching the grey-blue of datacenter racks and the cool-toned palette of every Kubernetes console screenshot in existence. Infrastructure has a color vocabulary (steel, blue, graphite) whether or not the documentation chooses to use it; choosing it deliberately is cheaper than fighting it.

## Retrospective

### What Worked

- **The six-module structure maps the actual Kubernetes stack cleanly.** Distributions → Networking → Storage → GPU → Security → Operations follows the mental model practitioners actually carry — infrastructure, then data flow, then state, then the workloads that motivated Kubernetes, then the security posture, then the operational loop. Each module is self-contained but connects to adjacent layers. Modules are ordered as a reader would discover them in a real deployment, not as a pedagogy would teach them. Future infrastructure Research projects should adopt the practitioner-ordering pattern rather than the pedagogy-ordering pattern.
- **The GPU-scheduling documentation fills a critical Research-series gap.** Module 04 covers DRA (Dynamic Resource Allocation, GA in v1.34), MIG (Multi-Instance GPU), MPS (Multi-Process Service), time-slicing, topology-aware scheduling across PCIe and NVLink fabrics, and batch-queue systems (Kueue, Volcano) — territory that most Kubernetes guides skip or cover superficially. For FoxCompute with its current RTX 4060 Ti and the future GPU fleet implied by the Memory Arena VRAM tier, the Erdős research program's arbitrary-precision compute requirements, and the Artemis II mission's embedding workloads, this is directly actionable infrastructure intelligence. The module exists because the gap in the broader documentation landscape is real.
- **Security and operations modules earned their length.** At 668 and 655 lines respectively, Modules 05 and 06 are the longest in the series. The length is structural rather than verbose. Security requires covering five defense layers with specific configurations (cluster hardening, network policies, RBAC, admission control, runtime detection); cutting any layer produces false confidence. Operations requires the full GitOps + observability + self-healing + FinOps stack; cutting any component leaves the closed loop incomplete. Catalog-style modules may exceed the nominal 300-line target when the subject has an irreducible minimum content surface, and this release documents that the minimum for Kubernetes security and operations is in the 650-700 line range.
- **The three-commit atomic-content discipline continues to pay off.** Content commit `147dca299` + docs commit `e3156aa10` + merge commit `4df8810c0` means the v1.49.68..v1.49.69 window contains exactly one meaningful state transition. Bisect finds it instantly. Forensic recovery from the repo alone is trivial. The discipline was formalized around v1.49.43 and has held across every Research project since.
- **The infrastructure color palette matches the subject's native vocabulary.** Steel `#263238`, electric blue `#1565C0`, graphite `#37474F` reads as datacenter infrastructure rather than as document branding. The three-color discipline continues from COI (v1.49.58, ritual subject) and ATC (v1.49.64, celestial subject) and extends naturally to cloud-native subject. Subjects with intrinsic color vocabularies get three-color palettes; subjects without stay two-color.

### What Could Be Better

- **The multi-cluster and federation story is thin.** K8S focuses on single-cluster architecture. Multi-cluster federation (Liqo, Admiralty, Clusternet, Karmada) and the emerging multi-cluster service-mesh patterns would strengthen the FoxCompute distributed-deployment model, which will eventually span multiple physical sites and cloud regions. A follow-on Research project (candidate name: KFD for Kubernetes Federation, or MCD for Multi-Cluster Deployments) should pick up the missing territory and compose directly with K8S's single-cluster baseline.
- **The edge-computing story is gestured at but not mapped.** K3s is named in Module 01 as the edge distribution of choice, KubeEdge is named as the IoT option, but the architectural differences between edge-cloud, edge-as-extension, and edge-native deployment models are not walked in depth. Edge Kubernetes is becoming its own discipline and would benefit from a dedicated module or a companion Research project.
- **Service-mesh coverage in Module 02 is proportionally light given the topic's depth.** Istio, Linkerd, and the emerging Ambient Mesh (sidecar-less) pattern each deserve more than the summary treatment they receive. A service-mesh-focused Research project would extend Module 02's foundation without bloating K8S itself further.

### What Needs Improvement

- **The upgrade-path and EOL version coverage is thin.** K8s versions go EOL on a 14-month cadence; clusters running pre-v1.30 versions are common in practice and need upgrade-path guidance, not just "upgrade to the latest." A revision pass adding an explicit version-matrix appendix (what's supported, what's EOL, what's recommended, what the skip-level upgrade path looks like) would materially strengthen the project for operators inheriting older clusters.
- **Cost modeling stops at Kubecost attribution.** Module 06 covers FinOps cost attribution but does not walk the actual cost-reduction playbook — right-sizing through VPA recommendations, workload packing, spot/preemptible capacity, reserved-capacity commitments, and multi-region cost arbitrage. The attribution is a prerequisite for the playbook; the playbook is the job. A follow-on module or companion project should close the gap.
- **The Windows-container and mixed-OS story is absent.** Windows containers on Kubernetes (Windows Server 2022/2025 nodes, hybrid Linux+Windows clusters) are a real production pattern for teams migrating legacy workloads; K8S does not cover them. The omission is acknowledged rather than denied, and a future project should close it if the series intends K8S to serve as a complete reference.
- **The release-notes chapter auto-generation policy should be formalized.** K8S, like every Research project since the parser landed, ships with parser-generated chapters at confidence 0.95. This README uplift rewrites the README to A-grade depth while leaving the chapters as DB-navigation substrate. The policy works, but it is not written down in a way that a future reader can discover without reverse-engineering the pipeline. An explicit policy document in `.planning/missions/release-uplift/` is overdue.

## Lessons Learned

- **Kubernetes is infrastructure for infrastructure, and the meta-pattern generalizes.** K8S doesn't run applications — it runs the system that runs applications. The same meta-pattern applies to GSD (the system that orchestrates development work), to orchestration frameworks generally, and to any closed-loop system where the orchestrator's job is to schedule other work. Module 06's operations section makes the pattern explicit: observe → decide → act, continuously. Every future Research project that engages an orchestration subject should name the meta-pattern rather than treat the orchestrator as a thing-in-itself.
- **The CNCF ecosystem is the modern Linux-distribution problem restated.** Just as Linux distributions package the kernel with different userland choices (glibc vs musl, systemd vs runit, apt vs pacman), CNCF projects package Kubernetes with different networking (Cilium/Calico), storage (Longhorn/Rook), observability (Prometheus/OpenTelemetry), GitOps (ArgoCD/Flux), and policy (Kyverno/OPA Gatekeeper) choices. The distribution decision is really a CNCF stack decision, and it is a coupled decision — components chosen together work together, components chosen independently often do not.
- **Security is five layers or it is zero layers.** The Module 05 discipline — cluster hardening, network policies, RBAC, admission control, runtime detection — is not a menu. Each layer assumes the preceding layers are present; cutting any layer produces false confidence that is worse than no claim at all. The most common production Kubernetes security failure mode is running the first four layers and declaring zero-trust accomplished. Runtime detection (Falco, Tetragon) is the layer most often cut and the layer whose absence is most destructive when a compromise happens.
- **GPU scheduling is a separate discipline from CPU scheduling and requires its own primitives.** Module 04 is long because GPU scheduling is genuinely a separate discipline. CPU scheduling assumes fungible cores; GPU scheduling handles heterogeneous devices (A100 vs H100 vs B200 vs consumer cards like RTX 5090), memory-per-device constraints, topology (PCIe groups, NVLink domains), and batch-queue semantics that CPU workloads rarely need. DRA (Dynamic Resource Allocation) is the Kubernetes feature that finally makes GPU scheduling first-class; before DRA, GPU scheduling was a pile of device-plugin workarounds. The module's length is structural, not verbose.
- **Catalog modules may exceed the 300-line target when the subject has an irreducible minimum surface.** K8S Modules 05 (668 lines) and 06 (655 lines) both exceed the nominal 300-line module target substantially. The overshoots are earned — Kubernetes security and operations each have irreducible minimum content surfaces that nothing short of 650-700 lines can cover without creating gaps. Future Research projects should document irreducible-minimum findings as precedents rather than treating module-length overshoots as failures. Analysis modules should hold the 300-line target; catalog modules may not be able to.
- **Mission-pack triads (LaTeX + PDF + HTML) continue to justify their redundancy.** The triad shape JNS (v1.49.53) established — one LaTeX source, one pre-rendered PDF, one HTML landing page — adds storage cost but buys reader autonomy. Readers who want the authoritative reference take the PDF. Readers who want to tinker take the LaTeX. Readers who arrive via the site take the HTML. The Research series optimizes for reader autonomy over storage efficiency and has been doing so since v1.49.53; the policy continues to hold.
- **Infrastructure subjects have intrinsic color vocabularies and projects should use them.** The steel + electric blue + graphite palette reads as datacenter infrastructure because those are the colors datacenter infrastructure actually has. Fighting a subject's intrinsic color vocabulary is more expensive than using it. COI used the ritual-sound palette (deep violet, alchemical gold, ritual purple); ATC used the celestial-terrestrial palette (Mars crimson, celestial blue, deep red); K8S uses the infrastructure palette. Three-color discipline with load-bearing semantics is the Research-series default for vocabulary-rich subjects and remains the cheapest defensible palette choice.
- **Cross-references are scaffolding, not ornament.** K8S cites PSG (infrastructure layer), SAN (defensive posture), and forthcoming FoxCompute work (AI/ML layer) as the neighbors it composes with. Those cross-references are the scaffolding that lets the Research series accumulate into a coherent architecture study rather than a collection of isolated papers. Future Research projects should name their compositional neighbors explicitly rather than leaving readers to discover the connections.
- **Version-specific claims age fast and need explicit dated-as-of footnotes.** K8S captures the Kubernetes ecosystem as of 2026 — K8s v1.35, DRA GA in v1.34, Gateway API v1.0, KEP-4317 pod certificates in beta, and so on. Every version-specific claim carries a footnote naming the date of the claim, because the Kubernetes release cadence guarantees the specifics will drift within two cycles. Future Research projects on fast-moving subjects should default to dated footnotes rather than retroactively adding them when drift becomes a reader problem.
- **The three-commit atomic-content discipline is invisible until forensics are needed.** Content commit `147dca299` + docs commit `e3156aa10` + merge commit `4df8810c0` is the same shape every Research project has shipped under since v1.49.43. The discipline produces no visible benefit until a repo-forensics pass becomes necessary — a corrupted working tree, a lost feature branch, a bisect across a many-release window. At that point, the discipline is the only artifact that matters. Every future Research project ships under the same three-commit shape, and the rationale is continuity of forensic recoverability.
