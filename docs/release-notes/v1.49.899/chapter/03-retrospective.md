# Retrospective — v1.49.899

## What Worked

**Three promotions converge at the same ship — pattern-cluster co-promotion.** v890 calibration-adjustment-store wired the first class-stored hoist-at-top instance AND established the first read-side-only-at-write-bearing instance — both 1-instance candidates from the same ship. v896 + v897 each strengthened both candidates simultaneously. By v897 close, three patterns were at 3-instance (class-stored hoist-at-top / audit-record-count test / read-side-only at write-bearing) — promotion-ready as a cluster. The codify ship absorbs all three in one pass rather than three separate ships. The cluster structure reflects the underlying truth: these three patterns are all properties of the same shape (class-based file with single fs-op method, derived methods, read+write separation). Codifying them together preserves the relationship.

**v898 surfaced two sub-disciplines without filing 1-instance candidates first.** v898's initial test failure (index-by-index sequence assertion broken by sync-substrate fire-and-forget non-determinism) generated two refinements of #10453: sync-substrate ordering subtlety + outcome-driven kind subtlety. Both became part of the #10453 ESTABLISHED promotion at v899 rather than carry-forward 1-instance candidates. The mechanism: when the first sub-discipline emerges during a 3-instance-promoting ship, fold it into the same promotion if it's a natural refinement (sub-axis of variation, not a separate pattern). Otherwise carry it forward.

**The variation-axis table is the right form for #10453.** Each of the 3 substrate→calibration end-to-end test instances varies along multiple axes (sync/async, polarity invariance, kind selection, boundary case, override mechanism). Trying to write the canonical test as a single skeleton + "...and also" prose would have buried the variation. The table format makes the axes explicit — the next person writing a 4th instance picks the closest matching row and adapts the rest.

**Codify-ship duration tracked the composition baseline.** v895 codify (3 promotions) was 60-90 min wall clock. v899 codify (3 NEW + 1 EXTENDED) ran ~90 min — same band. The duration scales with composition (number of separate-doc promotions), not individual lesson complexity. This data point strengthens the 1-instance candidate "codify-ship duration scales with composition" (5 data points + this one = 6).

## What Could Be Better

**The #10448 catalog table is getting wide.** Adding class-stored hoist-at-top as the 6th row pushes the table near terminal-width readability limits. Future sub-variant additions (e.g., promoting `module-singleton` if a 2nd instance arrives) will need either column abbreviation or a switch to a different representation (sub-section per variant with prose). The discipline-doc-as-living-catalog pattern composes well up to ~5-6 entries; beyond that, structure refactor is overdue.

**No new lessons emitted FROM this ship.** Pure codification — no new 1-instance candidates were generated by the act of codifying. The codify-axis cadence assumes promotion ships add zero new debt; this ship validates that assumption. But if a future codify ship DOES generate new candidates (e.g., "documenting variation axes is its own discipline" emerging from this ship's variation-axis-table work), the next cycle would need to absorb them.

**Carry-forward count after v899 still ~13.** v897 close had 16 carry-forward candidates; v899 promoted 4 (3 NEW + 1 EXTENDED). Net change: -4 + 1 from v898's sync-substrate non-determinism (folded into #10453 rather than separate) = ~13 remaining. The codify backlog is healthy but not shrinking dramatically. Next cycle target: 5-6 more promotions if v900-v910 generates new 2-instance triggers, or absorb the existing 13 carry-forward as a larger codify ship at v910.

**STATE.md normalizer might trip on em dash in milestone_name.** v899 has 4 em dashes in its milestone_name field. The state-md-normalizer has historically been sensitive to special characters per a memory note. Mitigation: this ship's STATE.md write was via `tools/state-md-set-shipped.mjs` (atomic-writer pattern from #10431), which uses js-yaml for safe scalar quoting. The atomic-writer SHOULD handle em dashes correctly. Will verify in the next session start.
