## What went well

1. **Mission-package-as-MILESTONE-CONTEXT worked smoothly.** A 22-file package staged in `.planning/staging/inbox/intelligence-dashboard/` carried PROJECT.md update + REQUIREMENTS.md derivation + ROADMAP.md prepend + STATE.md reset + 6 phase definitions through a single `/gsd-new-milestone` invocation without a separate `/gsd-discuss-milestone` cycle. Saves ~20 min of operator dialogue per milestone when the mission is fully shaped upstream.
2. **Wave-parallel execution honored the mission design.** Phase 822/823/824 dispatched as three concurrent gsd-executor subagents from a hot 5-min cache window; total Wave 1 wall-clock was ~3h vs the ~9h sequential floor. Phase 827's C01 + C02 also ran parallel cleanly (non-overlapping file trees: `src/intelligence/kb/store.ts` + `src/intelligence/events/`).
3. **The G-gate hierarchy held.** G0 (post-W0 type compile + SQLite migration) PASSED on first attempt; G1 (CAPCOM, end of Wave 2) PASSED with the mission-emitter format compliance against the existing vision-to-mission skill's input contract; G2 (Safety Warden BLOCK throughout Wave 3) caught the perf-test classification bug + the nav-shim ephemeral-edit bug before they could land permanently. The `dist/dashboard/index.html` "1-line edit" rule was demonstrably broken (file is gitignored build output) and the G2 review caught it at commit-message review.
4. **Three-transport-against-one-KB came together cleanly.** The Tauri shell + browser-tab + vitest paths each have their own dispatch surface, but they all converge on the same KBStore + IntelligenceEventBus singleton. C03's `CreateIntelligenceBridgeOptions` for testability was a small unplanned addition that made C04's integration tests dramatically simpler.

## What surprised

1. **Phase 826.5 emerged as a real component during operator testing of the Intelligence tab.** The mission spec promised browser-tab parity, but Phase 825's planned WS-bridge work silently dropped from the plan; the static-mode banner masked the gap. Phase 826.5 was added intra-826 to close 15 of 18 commands; the remaining 3 throw-stubs + missing server-side SSE broadcast became Phase 827. v1.49.598 staging (originally) captured the gap; folded back into v1.49.597 as Phase 827 once the operator chose "stay on dev, no push to main, continue building".
2. **Two carryover gaps from 826.5 only surfaced during operator manual smoke at ship time.** (a) `intelligence:build` script copied the bundles but not their dependency chunks under `desktop/dist/assets/`; bundle imports 404'd; static-mode banner showed even though the bridge was reachable. (b) The bundle's auto-mount only looked for `#planning-root` (Tauri standalone shell ID) but the dashboard tab uses `#intelligence-planning` (Phase 826 / C14 migration ID); bundle ran cleanly but found no mount root; CSS `:empty::after` placeholder stuck visible. Both bugs would have been caught by a single Playwright smoke test that opens `http://localhost:3030/intelligence.html` and asserts the planning UI renders. Forward lesson #10222.
3. **Pre-existing dashboard parser drift across MILESTONES.md / ROADMAP.md / REQUIREMENTS.md.** The dashboard pages had been silently rendering empty ("No milestones shipped yet" / "No phases defined yet" / "No requirements defined yet") for at least a milestone or two — the parser had been left behind as `.planning/` doc formats evolved. Caught by the ship-pipeline smoke. Phase 828 included a third commit fixing the three parsers to support both legacy fixtures (which the existing parser tests still use) and the current `.planning/` format. 53/53 parser tests still pass; live parse now returns 28 milestones + 106 phases + 177 requirements / 31 groups. Forward lesson #10223 — periodic doc-format-vs-parser-format audit.
4. **Test count grew faster than expected.** Cumulative Phase 821-827 + the C04 integration suite added ~+770 tests over the v1.49.596 baseline. The +50 vitest delta target was crushed ~15× over. Some of this is intelligence-dashboard heavy because the surface is large (3 IPC paths × 18 commands × 5 event types × 14 components); some of this is the safety-critical-test culture from earlier milestones bleeding through.
5. **C00 made the new IntelligenceKB methods optional, not required.** The spec called for required signatures; making them required would have broken ~10 implementing classes / test stubs (KBStore + IntelligenceKBStub + 5 mock factories + 2 internal cast sites). C00 used the lower-blast-radius optional `?:` pattern; C01 implemented them as concrete methods which satisfies both contracts. A future cleanup phase can promote to required after all implementers have real impls. This is consistent with the "lowest-blast-radius incremental rollout" pattern in TypeScript ecosystems generally.

## What we'd do differently

1. **Add a real-browser smoke test to the ship pipeline.** Phase 826.5 + 827's integration tests verified the bridge works and events flow at the contract level, but didn't load the actual bundle in a browser. A Playwright/headless-chromium smoke that opens `intelligence.html`, waits for the planning UI to mount, and asserts at least one project row renders would have caught BOTH carryover bugs in 5 seconds. Add to the v1.49.597 manifest of "post-ship follow-on tasks".
2. **Add a `dashboard generator output sanity` test.** The dashboard pages had been silently broken for a while because the empty-state fallback hid the parser drift. A test that runs the full generator against a small fixture `.planning/` directory and asserts the output HTML contains at least one `timeline-item` / `phase-card` / `REQ-` row would fail loudly when doc formats evolve past parser expectations.
3. **Don't sequence W0 in a 5-min cache window unless caching is actually load-bearing.** Phase 821's "must complete in <5 min for the cache" framing didn't matter in practice — the parallel subagents have their own context, not a shared cache with the parent. The framing led to 821 being slightly under-spec'd (it didn't include schema validation against existing `.planning/console/` examples; that became a follow-on assertion in C13). Cache-window sequencing should only be used when the agents share context, not when each has its own.
