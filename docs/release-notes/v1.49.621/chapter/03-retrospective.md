10. **Request-ID nanoid collision under birthday-paradox in CI parallel-load.** Discovered v1.49.621 expansion CI run (run 25606304935 on commit ea86ffb5b): `src/intelligence/__tests__/edge/emitter-edge-cases.test.ts > E13: request-id uniqueness > generateRequestId produces unique IDs on rapid successive calls` failed with 1 collision in 100 iterations. **Root cause:** request-ID format used a 4-char nanoid suffix (TS: 36^4 = 1,679,616 space → p(collision) ≈ 0.3% at 100 calls; Rust: UUID v4 first 4 hex chars = 16^4 = 65,536 space → p(collision) ≈ 7.4% at 100 calls — even more flaky). Same class of CI-contention flake as Lesson 8 but the fix shape differs: this is birthday-paradox math, not hookTimeout. **Resolution (closed in this milestone via operator decision (c)):** bumped suffix length to 8 chars on both sides (TS `nanoid8` 36^8 ≈ 2.8 trillion; Rust UUID `[..8]` 16^8 ≈ 4.3 billion). Collision probability at 100 calls now ~1.8e-9 — effectively never. Format change `XXXX → XXXXXXXX` is backward-compat (the consumer regex `/req_..._\d{4}_[a-z0-9]+/i` at `src/intelligence/emitter/compose.ts:569` matches both lengths; existing IDs in databases continue to be valid).
