## Summary

**v1.7 is the spine, not another limb.** Every release from v1.0 through v1.6 built a component — the adaptive loop, validation, measurement, documentation, teams, pattern discovery, examples. v1.7 is the release that connects them. The load-bearing deliverable is a two-layer GSD Master Orchestrator that routes free-text user utterances to the right GSD command, the right skill, the right agent, and the right next step in the lifecycle — without a hardcoded command list and without forcing the user to remember which subcommand does what. Above that sits a Gas Town-inspired platform stack: persistent work state, session continuity through snapshots and handoffs, ephemeral observation tiering, skill Workflows with crash recovery, Roles with constraint injection, Bundles with progress tracking, and an Events emit/listen system that lets skills activate each other causally. Sixteen phases shipped in TDD style, with a failing test commit before every implementation commit. The commit range `160437bef..9e8255e6b` contains 132 commits across 478 files and 108,155 insertions; the tag message headlines "16 phases, 38 plans, 124 commits, 27,617 lines added" and every numeric claim in this README can be checked against `git log`, `git show`, and `git diff --shortstat` for the v1.6..v1.7 range.

**Dynamic filesystem discovery replaced the hardcoded command list.** Phase 36 shipped the discovery service: `src/orchestrator/discovery/` scans `.claude/commands/gsd/`, `.claude/agents/`, and the team manifests, parses the YAML frontmatter of each command/agent/team file, and caches the results keyed by mtime so re-scans are free when nothing changed. Commit `c2b4fc149` landed the command parser and scanner (36-01), `499b6a3d4` added agent parser + team parser + discovery service with mtime caching (36-02), and `8f81300cb` added auto-detection, error tolerance, and dependency injection for the discovery service (36-03). The design decision here is that the orchestrator does not know what commands exist until it asks the filesystem; a new `.claude/commands/gsd/foo.md` file becomes callable the next time an utterance is classified. Error tolerance means a malformed command file does not break the scanner — it is logged and skipped. This is the right design for a system still growing rapidly, and it is the pattern that every later release (v1.8, v1.9, v1.25) inherits for discovering its own extensions.

**Intent classification runs a 5-stage cascade with a confidence gate at the end.** Phase 38 shipped `src/orchestrator/intent/`: intent types + exact-match pass-through at `6ff682c71` (38-01), utterance augmentation + a Bayes classifier wrapper at `566444dc9`, lifecycle-stage derivation + command filtering at `248de160c` (38-02), regex-based argument extraction at `06e9c83dd`, and the full classifier pipeline with confidence resolution at `e5a26ede7` (38-03). The stages, in order: (1) exact match on command name or alias — O(1) and always wins; (2) lifecycle filter — narrow candidates to commands valid in the current project phase derived from `.planning/` artifacts; (3) Bayesian classifier trained on utterance/command pairs; (4) embedding-based semantic matcher added in Phase 42 at `88c61467f` for fuzzy natural-language utterances; (5) confidence threshold — if the top candidate's confidence falls below the threshold, the orchestrator stops and asks the user instead of guessing. The cascade is not over-engineered: each stage does one thing, fails fast to the next stage when it cannot answer, and the confidence gate replaces guessing with asking.

**Two-layer orchestrator agent with extension awareness.** Phase 41 shipped the Layer 1 `.claude/agents/gsd-orchestrator.md` agent file (`dc927cdaa`) that runs standalone with the GSD command set, plus the Layer 2 skill-creator CLI integration (`f159d86f2`) that enhances the agent when skill-creator is present. Phase 44 added the extension detector (`0238c0f72`) with capability-based feature flags — the orchestrator detects which GSD extensions (v1.15 fixture snapshot, skill-creator, event system) are installed and only advertises features it can actually run. Integration tests with graceful degradation validation shipped at `1b536aa96`; the extension-awareness section was added to the orchestrator agent at `ec1944e04`. The design point is that the orchestrator does not assume it has every extension — it probes for them and degrades cleanly when they are absent, which is the only way a single agent definition works across user environments with different GSD installations.

**Verbosity controller and human-in-the-loop gate framework.** Phase 43 shipped `src/orchestrator/verbosity/` and `src/orchestrator/gates/` as two complementary UX primitives. The verbosity controller (`0e99eb48d`) filters orchestrator output across five levels — silent, minimal, normal, verbose, debug — and is exposed via the CLI `--verbosity` flag at `0d2910498`. The HITL gate evaluator (`49338d1db`) implements three gate types: confirmation (`yes/no` before a destructive op), selection (pick one of N options), and threshold (continue when a numeric threshold is met). Gates and verbosity were wired through the orchestrator index at `bd68f84fc` (refactor 43-02) and `2da98e795` (chore 43-01). The agent definition grew a verbosity-and-HITL section at `f39d3c13d`. These are not afterthoughts — a system that acts automatically without asking is terrifying and one that asks about everything is annoying, so the gates are the dial between those extremes, deliberately shipped as first-class CLI features rather than ad-hoc prompts.

**Persistent work state, session continuity, and ephemeral observation tiering.** Phase 45 shipped the work-state stack at `src/orchestrator/work-state/`: Zod schemas at `88aa95fd5` (45-01), writer at `e6409ab46`, reader with safe YAML parsing at `7ac35b502`, queue manager at `97862287f` (45-02), CLI subcommands at `7cccb2b95`, and SessionEnd/SessionStart hook scripts at `e977b885d` (45-03). Phase 46 built session continuity on top: SessionSnapshot schema at `744800109` (46-01), SnapshotManager with `generate` / `store` / `getLatest` at `ee5042411`, SkillPreloadSuggester and CLI snapshot subcommands at `ba7649c44`, plus SessionStart/SessionEnd snapshot hooks at `58b5d7450` (46-02). Phase 47 added the ephemeral observation tier at `src/orchestrator/session-continuity/` and beyond: EphemeralStore at `c3bdc2f28` (47-01), PromotionEvaluator with multi-factor scoring at `9099e35f9`, ObservationSquasher with additive merging at `da7d19c68`, and tiered routing/promotion/squash wired into SessionObserver at `f81805ea0` (47-02). The result: observations live in a cheap in-memory tier first, get scored for promotion to durable storage, and get squashed to prevent redundant records from accumulating.

**Skill Workflows, Roles, and Bundles as three coordination abstractions.** Phase 48 shipped Workflows — multi-step skill chains with dependency tracking and crash recovery — at `src/workflows/`: Zod schemas at `c0a478f2b` (48-01), parser + WorkflowDAG at `9d58c7aa2`, validator with DAG and skill checks at `371298f84`, WorkflowRunStore with JSONL persistence at `09b1892b0` (48-02), the `workflow extends` compositor and runner with crash recovery at `9b94b1afb`, CLI commands and barrel exports at `6e8de213b` (48-03), and integration tests for the workflow lifecycle at `088f02f10`. Phase 49 shipped Roles — behavioral constraints and tool scoping for agent personas — at `src/roles/`: schema + parser at `7e1a21328` (49-01), constraint injector at `bf960c068`, `role extends` with additive constraint merging at `3694a0759` (49-02), and CLI `role create`/`list` at `0efab0df4`. Phase 50 shipped Bundles — project-phase skill sets with progress tracking and auto-suggestion — at `src/bundles/`: schema + YAML parser at `1773f9c7f` (50-01), bundle activator with WorkState integration at `201c56612`, CLI bundle commands at `7140dce7f`, bundle progress tracker with JSONL store at `0e7b3d89d` (50-02), and the bundle auto-suggester with clique detection at `356bf3820`. Each extension adds one dimension: Workflows chain, Roles constrain, Bundles group by phase.

**Events emit/listen system enabling causal skill activation chains.** Phase 51 shipped `src/events/` — the final coordination layer that connects skill activations causally. Event schemas with extension field and skill-index integration at `4c14b4ca3` (51-01), the event store and lifecycle with JSONL persistence at `01735664b`, the event-boost activation adjuster and the co-activation event suggester at `1aace6f39` (51-02), and CLI `event list`/`emit`/`consume`/`suggest`/`expire` subcommands at `a4788f151`. Events are the missing primitive: before Phase 51, a skill that produced output could not notify another skill that the output was ready, so skill activations were either all manual or all bundled into workflows. With events, a skill emits a typed event when it finishes, other skills listen for it, and the activation chain forms organically. This is where the system starts behaving adaptively rather than procedurally; it is also the feature that makes the v1.8 capability-aware planner and v1.25 ecosystem DAG tractable, because the DAG has a runtime signal to fire on instead of a static dependency graph.

**TDD across every phase was the scaffolding discipline.** Every feature commit in the v1.7 range has a failing-test commit immediately before it in git log. `test(36-01): add failing tests for command parser and scanner` (`c9358a0a0`) precedes `feat(36-01): implement command parser and scanner for GSD discovery` (`c2b4fc149`). TDD for sixteen phases in one release is the discipline that kept the orchestrator integration from collapsing under its own surface area; it is also the scaffolding that lets a post-v1.7 contributor safely refactor any piece of the orchestrator without guessing what the contract was.

## Retrospective

### What Worked

- **Dynamic GSD command discovery from the filesystem eliminates hardcoded command lists.** When new commands land in `.claude/commands/gsd/`, the orchestrator finds them automatically via mtime-cached filesystem scans. Error tolerance means a malformed command file is logged and skipped rather than breaking the scanner, so the discovery service survives the kind of drift that accumulates across sixteen phases.
- **5-stage intent classification pipeline with confidence fallback is robust.** Exact match handles the obvious cases at O(1); lifecycle filter narrows to commands valid in the current phase; Bayes and embedding matchers catch fuzzy utterances; the confidence threshold gate replaces guessing with asking. The cascade avoids over-engineering any single classification method — each stage does one thing and fails fast to the next.
- **Lifecycle-aware routing narrows candidates based on the current project phase.** A command that is irrelevant during planning does not clutter the candidate list during execution. This is context-sensitive command resolution, not a global keyword match, and the reduction in candidates is what makes the Bayes classifier accurate at small scale.
- **Workflows, Roles, Bundles, and Events are the right four abstractions.** Workflows chain skills with dependency tracking. Roles constrain them with tool scoping. Bundles group them by project phase. Events connect them causally. Each extension adds exactly one dimension of coordination, and the four together cover the space without overlap.
- **TDD across every phase kept the surface area tractable.** Every `feat(...)` commit has a `test(...)` commit immediately before it in the v1.6..v1.7 range. Sixteen phases worth of orchestrator integration without TDD would have collapsed into rework; with TDD, a post-v1.7 contributor can bisect any subsystem and watch the tests go red-then-green across a pair of adjacent commits.
- **Two-layer agent with extension detection degrades cleanly.** Layer 1 runs standalone against the GSD command set; Layer 2 activates when skill-creator is present and the v1.15 fixture is installed. Extension-gated activation and graceful degradation (`ffa05a0ad`, `1b536aa96`) mean the same agent definition works in minimal, mid, and fully-loaded installations without per-install branching.
- **Verbosity and HITL gates shipped as first-class CLI primitives, not prompts.** The `--verbosity` flag and the three gate types (confirmation / selection / threshold) are part of the orchestrator's published contract, not ad-hoc runtime prompts. That means a scripted environment can set verbosity once and honor the gates programmatically; an interactive environment gets the dial between "silent auto-pilot" and "ask about everything."
- **JSONL persistence everywhere with consistent schemas.** Work state, workflow runs, bundle progress, events, and ephemeral observations all use append-only JSONL with Zod-validated schemas. This is the v1.0 pattern-store primitive generalized: immutable append, crash recovery, grep-friendly, no migration path needed.

### What Could Be Better

- **16 phases, 38 plans, 42 requirements, and 108,155 insertions make this the largest release so far.** The orchestrator is the integration point for everything built in v1.0–v1.6, so the scope is justified, but the surface area is significant and post-v1.7 documentation work has to catch up (which is exactly what v1.3's precedent said would happen).
- **Crash recovery for workflows is documented and tested at the unit level, but end-to-end crash scenarios are hard to simulate.** The WorkflowRunStore JSONL persistence at `09b1892b0` and the crash-recovery runner at `9b94b1afb` are unit-tested, but multi-step skill chains that actually die mid-execution in a user environment will surface edge cases the unit tests cannot reach. v1.7 ships the machinery; the live-fire verification is deferred to operational experience.
- **The confidence-threshold value in the intent classifier is not calibrated against real utterance data.** The classifier ships with a default threshold, but the right value depends on the user's phrasing habits, installed extensions, and project type. The v1.2 calibration pattern (F1 optimization via `sc calibrate`) should be extended to the intent classifier in a follow-up release.
- **The Layer 1 orchestrator agent hit the character budget ceiling.** Commit `1d5b06622` (41-02) was explicitly about "optimize character budget with disambiguation and error handling" — the agent prompt bumped up against Claude's per-agent size limit and had to be compressed. That pressure will return every time a new extension adds another capability the agent needs to know about; the long-term fix is to split the agent into smaller specialized sub-agents rather than keep compressing one.
- **132 commits across 16 phases is more than the CHANGELOG backfill pattern from v1.3 can comfortably absorb.** The commit log is the source of truth for this release, and the CHANGELOG entry for v1.7 will necessarily summarize rather than enumerate. Future milestones at this scope should write the CHANGELOG entry as phases complete rather than at the tag boundary.

## Lessons Learned

1. **The orchestrator is the system's spine.** Everything before v1.7 built components; v1.7 connects them. Without an orchestrator, skills, agents, and teams are isolated capabilities with no coordination layer. Get the spine right and every later release adds muscles; get it wrong and every later release fights the spine.
2. **Verbosity levels and confirmation gates are UX decisions, not afterthoughts.** A system that does things automatically without asking is terrifying; one that asks about everything is annoying. The five-level verbosity dial and the three-gate HITL framework are the control surfaces that let the same orchestrator serve a scripted batch workflow and an interactive REPL without two separate code paths.
3. **Event-based skill activation (emit/listen) enables emergent behavior.** Skills that react to other skills' typed output events create chains that weren't explicitly designed — this is where the system starts behaving adaptively rather than procedurally. The Phase 51 event primitive is the feature that makes every later composition release (v1.8 capability-aware planning, v1.25 ecosystem DAG) tractable.
4. **Dynamic discovery beats hardcoded lists when the system is still growing.** A hardcoded command list is faster to write on day one and wrong on day thirty. The mtime-cached filesystem scan costs a few milliseconds per cold scan and zero per warm scan, and every command added after v1.7 is reachable without a code change. The only systems that should hardcode the command list are the ones that have stopped growing.
5. **Lifecycle-aware routing is context-sensitive command resolution, not global keyword match.** Narrowing the candidate set to commands valid in the current project phase is what makes a small-vocabulary Bayes classifier accurate. A global classifier over 60+ commands would need ten times the training data to match the lifecycle-filtered classifier's precision.
6. **Ship Workflows, Roles, Bundles, and Events together or you ship half a coordination layer.** Each extension adds one dimension — chain, constrain, group, connect. Shipping three of the four leaves a coordination problem with no primitive to solve it. Phase 48–51 were scheduled as a single front because the four abstractions are partial coverage of the same problem space, and partial coverage is the worst kind of half-built system.
7. **TDD is the only way to ship sixteen phases in one release without rework.** Every feat commit has a failing-test commit immediately before it. The pattern is mechanical, the discipline is cheap per phase, and the payoff at release scope is that the bisect surface for any subsystem is exactly one commit pair. Without TDD, the integration debt from an orchestrator that touches every previous component would have eaten the release.
8. **Two-layer agents with capability detection are how you ship one definition across multiple installations.** Layer 1 runs standalone; Layer 2 enhances when extensions are present. The extension detector at `0238c0f72` and the graceful degradation tests at `1b536aa96` mean the orchestrator does not branch on installation state — it probes, advertises what it can do, and keeps quiet about what it cannot. This pattern is load-bearing at v1.8 and later when the capability set continues to grow.
9. **JSONL with Zod schemas is the durable-store primitive for every new subsystem.** Work state, workflow runs, bundle progress, events, observations — all use the same append-only JSONL pattern with a Zod schema at the type boundary. Choosing the same primitive for every subsystem means crash recovery, grep-ability, and audit trail are free; choosing a different store per subsystem would have doubled the surface area without buying anything.
10. **Consolidate agents when they converge on the same job.** Commit `9e8255e6b` merged four specialized debug agents (alpha/beta/gamma/lead) into a single `gsd-debugger` and retired `gsd-health-checker`, `gsd-milestone-advisor`, `gsd-plan-optimizer`, and the individual researcher agents. Specialization is cheap to create and expensive to maintain; consolidation happens when the specialized agents are doing the same work with different names, and v1.7 was the natural boundary to do the consolidation because the orchestrator now routes to agents dynamically.
11. **The release where the system becomes the spine is the release that sets the ceiling for every later release.** v1.7's coordination primitives — discovery, classification, lifecycle, workflows, roles, bundles, events — are the substrate v1.8, v1.9, v1.25, and v1.49 extend. A ceiling that is too low here means every later release builds around the ceiling; v1.7's ceiling was set deliberately high, and the payoff is that v1.8 capability-aware planning slotted in without changing the orchestrator contract.

## Engine Position

Every v1.x release from v1.8 onward extends v1.7's coordination layer rather than replacing it: v1.8 capability-aware planning slots into the intent classifier; v1.9 advanced orchestration layers on top of the Layer 2 agent; v1.25 ecosystem integration builds its dependency DAG as a runtime graph fired by v1.7's event primitive; v1.49's mega-release consolidations still rely on the same discovery service, the same work-state JSONL, and the same event bus.
