## Summary

**v1.44 is the release that proved `sc:learn` can ingest a third-party codebase it did not write.** v1.40 closed the format-breadth gap by dogfooding the pipeline against the 923-page PDF of The Space Between, a manuscript the project authored. That run verified PDF extraction, checkpoint ingestion, dual-track learning, three-track verification, refinement-as-proposal, and safety validation against content the system already knew. It did not answer the harder question: can the pipeline ingest a codebase written by other people, in another language, for another domain, and produce a usable SKILL.md plus working Python scripts at the far end? v1.44 answers that question by running PyDMD ‚Äî a mature open-source Python library implementing sixteen variants of Dynamic Mode Decomposition ‚Äî through the whole chain, from Python detection through venv provisioning, tutorial parsing, concept extraction, decision-tree synthesis, reference generation, script emission, accuracy checking, and observation-bridge publication. Five phases, eleven plans, twenty-two commits, two hundred eighty-four tests, twelve thousand nine hundred and thirty-two insertions. The pipeline now knows how to ingest somebody else's library.

**The install pipeline treats Python as a runtime dependency, not an assumption.** Phase 404 shipped two install-stage modules ‚Äî `python-detector.ts` and `venv-manager.ts` ‚Äî that probe the host for a usable Python interpreter before any ingestion begins. The detector walks a priority list of candidate interpreters (`python3.11`, `python3.10`, `python3`, `python`), checks each against a minimum-version floor, captures version string and executable path, and reports a structured `PythonDetection` result rather than a bare path. The venv manager creates an isolated virtual environment under the dogfood workspace, installs PyDMD plus its transitive dependencies into the venv, and emits a `VenvManifest` recording interpreter path, venv path, installed package list, and install timestamp. Phase 404-02 added the health-check runner and an install-pipeline index: the runner imports PyDMD inside the venv, instantiates a minimal DMD object, fits it against a synthetic dataset, and returns `ok | degraded | failed` with a named reason. If health fails, the pipeline halts at the install stage and the downstream phases never run. 408 health-check tests plus 408 python-detector tests plus 281 venv-manager tests land in this phase alone ‚Äî install hardening is the first third of the 284-test total.

**Structure analysis and API surface mapping read the codebase like a library, not a blob.** Phase 405-01 shipped `structure-analyzer.ts` and `api-surface-mapper.ts`, the first pair of learn-stage modules. The structure analyzer walks the PyDMD source tree, classifies files by role (core-implementation, variant, utility, example, test), extracts the module hierarchy, and emits a `StructureMap` capturing the library's topology. The API mapper inventories public classes and their methods, distinguishes base-class methods from variant-specific methods, captures parameter signatures and return types where docstrings disclose them, and catalogs the sixteen PyDMD variants (DMD, OptDMD, HankelDMD, MrDMD, SpDMD, CDMD, FbDMD, HAVOK, PiDMD, BOPDMD, EDMD, HOSVD-DMD, ParametricDMD, LANDO, KernelDMD, PreProcessingDMD) with their canonical class names and short purpose descriptions. The 464-line analyzer plus 455-line concept-extractor plus 172-line complex-plane-mapper together form the knowledge-intake layer for this dogfood run; the test suite for the learn stage alone (structure-analyzer, api-mapper, concept-extractor, complex-plane-mapper, tutorial-parser, pattern-synthesizer) accounts for 2,776 test lines across six files.

**Concept extraction wires PyDMD's mathematical primitives into the skill-creator knowledge graph.** Phase 405-02 shipped `concept-extractor.ts` (455 lines) and `complex-plane-mapper.ts` (172 lines). The concept extractor identifies the mathematical primitives that underpin every DMD variant ‚Äî Singular Value Decomposition, eigendecomposition, Koopman operator theory, rank truncation, time-delay embedding ‚Äî and attaches each concept to its algorithmic variants with a `purpose`, `mathBasis`, `distinguishingFeatures`, and `tutorialReferences` field. The complex-plane mapper produces a `ComplexPlaneMap` that ties DMD's eigenvalue interpretation (unit-circle residence indicates neutral stability; inside the unit disk indicates decay; outside indicates growth) to skill-creator's pre-existing complex-plane learning framework (v1.37). The mapper emits a pedagogical narrative connecting DMD-specific spectral analysis to the angular-position / radial-magnitude substrate the Complex Plane Learning Framework already tracks ‚Äî which means a user reading the generated skill sees DMD eigenvalues rendered in the same coordinate system as every other mathematical concept the system has ingested. This cross-reference is the payoff for specifying the plane substrate in v1.37 and demonstrating it on a manuscript in v1.40: in v1.44 it does real work against an external corpus.

**Tutorial parsing and pattern synthesis produce a decision tree, not a list.** Phase 405-03 shipped `tutorial-parser.ts` (315 lines) and `pattern-synthesizer.ts` (410 lines). The tutorial parser reads PyDMD's bundled tutorial Python files via pattern matching on the canonical import / instantiation / fit / plot sequence, captures variant identity, data-type characteristics (noise level, timescale count, nonlinearity, control inputs, size, sparsity, time-delay structure), and extracts key insights that the tutorial author highlights. The pattern synthesizer cross-references the structure map, concept catalog, and tutorial extractions to build a unified `KnowledgeGraph` with an `apiSurface` summary, representative usage patterns, five domain pitfalls (noise sensitivity, rank-truncation traps, time-delay embedding misuse, multi-scale miscalibration, control-input confusion), and a `DecisionNode[]` decision tree that routes all sixteen variants via data-characteristic questions. The tree's first split asks about noise sensitivity; subsequent splits cover timescale separation, nonlinearity, control, size, sparsity, and time-delay structure. Variants that do not fit the canonical tree (because the seven features under-discriminate them) land in an automatically generated overflow subtree rather than vanishing from the output. The decision tree is the form that an end-user consults when choosing between DMD variants for a new dataset, and it is emitted as a structured JSON object that a downstream composer can render as prose.

**Skill composition produced a SKILL.md under the five-thousand-word progressive-disclosure ceiling.** Phase 406-01 shipped `skill-composer.ts` (497 lines), `decision-tree-formatter.ts` (80 lines), and `word-counter.ts` (53 lines). The skill composer transforms a `KnowledgeGraph` into a ten-section SKILL.md with six-plus Python code examples ‚Äî front-matter, overview, core concepts, variant catalog, decision tree, API reference, usage examples, pitfalls, cross-references, further reading ‚Äî and enforces a hard five-thousand-word ceiling by word-counting the rendered markdown (stripping formatting, excluding code blocks) and automatically truncating the pitfalls section if the budget is exceeded. The decision-tree formatter converts the synthesizer's `DecisionNode[]` into structured prose with consistent indentation and explicit "if your data has X, use Y" routing. The word counter handles the under-counted cases that naive word counters miss (hyphenated compounds, LaTeX inline math, inline-code spans). The 350-line test suite for the skill composer exercises budget enforcement, code-block exclusion, section completeness, and truncation behavior. This is the first release where an `sc:learn` run produces a publishable skill artifact from an external codebase; v1.40 produced knowledge entries but no end-user skill file.

**Reference generation produces four markdown documents plus three runnable Python scripts.** Phase 406-02 shipped `reference-builder.ts` (281 lines), `script-generator.ts` (425 lines), and `cross-reference-checker.ts` (138 lines). `buildReferences()` emits four markdown references ‚Äî `algorithm-variants.md` (class name, purpose, parameters, usage example, math basis per variant), `math-foundations.md` (SVD, eigendecomposition, Koopman theory, stability interpretation), `api-reference.md` (the public surface catalog), and `complex-plane.md` (the pedagogical narrative from the plane mapper). `generateScripts()` emits three self-contained Python scripts (`demo-dmd.py`, `demo-variants.py`, `demo-complex-plane.py`) with argparse parameter handling, NumPy-compatible docstrings, a `__main__` guard, matplotlib plotting, and imports structured to run inside the venv created in phase 404. `checkCrossReferences()` walks every internal markdown link in the emitted reference docs, validates that each link resolves to a known output file, and emits a diagnostic list of broken links; no link-in-generated-output failures land in the v1.44 artifact because the checker catches them before publication. All reference docs begin with a back-link to the top-level SKILL.md so a reader never loses the entry point.

**Accuracy checking ran tutorial replay against ground truth and scored extraction fidelity.** Phase 406-03 shipped `accuracy-checker.ts` (362 lines), `tutorial-replay.ts` (136 lines), and `scoring.ts` (64 lines). The tutorial-replay engine executes each PyDMD tutorial inside the venv, captures the variant invoked, the parameters used, and the intermediate outputs, and compares that trace against the extracted `KnowledgeGraph` entry for the same variant. The accuracy checker computes per-concept recall (did extraction name every concept the tutorial demonstrates?), per-variant precision (did the catalog attribute each variant's parameters correctly?), and cross-reference consistency (does every decision-tree leaf actually map to a variant the extraction catalog includes?). The scoring module reduces these three signals to a composite `AccuracyReport` with a named severity for each drop below threshold. 240 tutorial-replay tests plus 377 accuracy-checker tests cover the replay loop, scoring logic, and report generation. The safety posture inherits from v1.40: every extraction mismatch is emitted as an improvement ticket, never applied silently, and the `requiresReview: true` flag rides through the ticket chain.

**The observation bridge makes `sc:learn` runs visible to the pattern-detection system that monitors the rest of skill-creator.** Phase 407-01 shipped `learn-to-observe.ts` (155 lines) and `dogfood-report.md` (84 lines). The bridge translates each pipeline-stage event ‚Äî python-detected, venv-provisioned, health-checked, structure-analyzed, concepts-extracted, tree-synthesized, skill-composed, references-built, scripts-emitted, accuracy-checked ‚Äî into an `Observation` record matching the schema that skill-creator's broader observation pipeline consumes. Before v1.44, `sc:learn` was a standalone ingestion tool whose output vanished at the generator boundary. After v1.44, every run emits observations that the detect-suggest-apply loop (v1.0) can see and act on. The dogfood-report template lands alongside the bridge as the canonical artifact shape for any future corpus run. Phase 407-02 wired 999 lines of end-to-end integration tests covering the full 13-step pipeline (detect ‚Üí venv ‚Üí health ‚Üí learn ‚Üí generate ‚Üí validate ‚Üí bridge) plus a health-gate test (pipeline halts when health check fails) plus a partial-knowledge-graph degradation test (pipeline handles empty tutorials and cross-references gracefully) plus five interface contract tests for the main data shapes (RepoManifest, KnowledgeGraph, GeneratedSkill, Observation, AccuracyReport). The integration tests call real modules with only I/O boundaries mocked; no handoff is tested in isolation.

**The release honestly flags what PyDMD did not close.** The 284-test total is lower per-LOC than v1.40 (362 tests for ~7,200 LOC vs. 284 tests for ~12,932 insertions). The shortfall is concentrated in the learn stage ‚Äî structure-analyzer and concept-extractor do heavy work against source code and dockstrings, and a handful of edge cases (private-by-convention classes starting with underscore, variants inheriting from a mixin rather than a direct base, docstrings in ReStructuredText vs. Google style) are exercised only in the tutorial corpus rather than in dedicated unit tests. Concept-mapping dependency edges were validated against PyDMD's runtime imports but not against the deeper mathematical prerequisites (SVD-before-DMD, eigendecomposition-before-mode-analysis); a downstream user studying the decision tree may see the algorithmic lineage correctly while missing the math-prerequisite lineage. And one dogfood run against one library (internal PyDMD tutorials plus the public API) does not exercise the extreme tails ‚Äî libraries with no tutorials at all, libraries with only C-extension interfaces, libraries whose Python surface is auto-generated from a schema. Format breadth on the install ‚Üí learn ‚Üí generate ‚Üí validate axis is proven on PyDMD; format depth against adversarial external corpora remains future work. The follow-on release (v1.45) applied these lessons by closing the HITL batch-ingestion gap that a single-corpus dogfood could not surface.

## Retrospective

### What Worked

- **Running `sc:learn` against a genuinely external codebase ‚Äî PyDMD ‚Äî closed the gap v1.40's internal-only dogfood left open.** PyDMD is a mature open-source library written by other authors in a different domain with different conventions. The pipeline handling it end-to-end is a stronger statement than handling content the project wrote itself. v1.44 graduates `sc:learn` from "works on our own stuff" to "works on someone else's stuff."
- **Health-gating the install before any ingestion runs prevented every downstream phase from operating on a broken environment.** The Phase 404-02 health check fits a minimal DMD instance against synthetic data inside the venv and returns a named failure reason if anything is wrong. The integration test explicitly exercises the halt path, so the gate is not a slogan ‚Äî it is a tested behavior.
- **Decision-tree output routes all 16 DMD variants through 7 data-characteristic questions rather than listing them as an undifferentiated catalog.** A user coming to DMD for the first time does not need a list of sixteen variants; they need to know which one to use. The synthesizer produces a tree that answers that question, with an overflow subtree for variants that under-discriminate on the canonical seven features.
- **Generated artifacts (SKILL.md + 4 reference docs + 3 Python scripts) are usable, not just extracted.** The skill composer enforces a 5,000-word ceiling with automatic pitfall-section truncation. The script generator emits runnable code with argparse, matplotlib, and proper `__main__` guards. The cross-reference checker validates internal links before publication. The output is a working artifact, not a markdown dump.
- **Tutorial replay against ground truth scores extraction fidelity rather than guessing at it.** The replay engine runs each PyDMD tutorial inside the venv, captures what happened, and compares against what the `KnowledgeGraph` said should happen. The accuracy checker emits per-concept recall, per-variant precision, and cross-reference consistency, so the quality signal is a number, not a vibe.
- **The observation bridge closes the loop with skill-creator's broader pattern-detection system.** Before v1.44, `sc:learn` runs were isolated ‚Äî output vanished at the generator boundary. After v1.44, each pipeline stage emits `Observation` records in the same schema that the detect-suggest-apply loop consumes. Learning becomes visible to the rest of the system.
- **The 999-line integration-pipeline test covered all 13 steps with real module calls and mocked I/O only.** Unit tests catch individual-module regressions; this integration test catches handoff bugs between modules. The five interface contract tests (`RepoManifest`, `KnowledgeGraph`, `GeneratedSkill`, `Observation`, `AccuracyReport`) pin the types that cross module boundaries so future refactors cannot drift the pipeline-wide contracts silently.

### What Could Be Better

- **284 tests for 54 files and 12,932 insertions is a lower test-to-LOC ratio than v1.40 delivered at comparable scope.** v1.40 shipped 362 tests against ~7,200 LOC; v1.44 shipped 284 tests against ~12,932 LOC. The install stage is well-tested (1,053 lines across three test files), the learn stage is moderately tested (2,776 lines across six files), but edge cases in the generate and validate stages lean on the tutorial corpus rather than dedicated unit tests. Private-by-convention classes (leading underscore), mixin-derived variants, and non-Google-style docstrings are exercised indirectly at best.
- **Concept-mapping dependency edges validate runtime imports but not mathematical prerequisites.** DMD depends on SVD; mode analysis depends on eigendecomposition; time-delay embedding depends on sequence-reshaping. The extracted dependency graph reflects PyDMD's import graph but does not encode that an operator-theoretic Koopman variant (EDMD) needs its kernel basis chosen before meaningful mode comparison is possible. A user studying the decision tree sees algorithmic lineage correctly but misses the math-prerequisite lineage.
- **Only one external library has been dogfooded.** PyDMD has tutorials, public docs, a consistent class hierarchy, and a Python-native interface. A library with C-extension only, no tutorials, RST docstrings, or auto-generated Python wrappers would surface different failure modes. Format breadth on the install ‚Üí learn ‚Üí generate ‚Üí validate axis is proven on PyDMD; format depth against adversarial external corpora is still open.
- **The decision-tree overflow subtree is a fallback, not a solved case.** Variants that under-discriminate on the seven canonical features (noise, timescale, nonlinearity, control, size, sparsity, time-delay) land in an overflow branch. This is the right engineering choice ‚Äî better than dropping variants ‚Äî but a user reading the tree sees a clean routing through the canonical seven and a "see also" list for the remainder. The overflow branch is where future feature-extraction work can concentrate.
- **The health check fits a single synthetic dataset; it does not probe the variant space.** If PyDMD loads, instantiates, and fits on a trivial dataset, the gate passes. Variants with heavier dependencies (LANDO's kernel choices, BOPDMD's optimization parameters, HAVOK's delay-coordinate tuning) are not exercised at install time, so a variant-specific install regression would escape the health gate and surface only during tutorial replay.

## Lessons Learned

- **External dogfooding is the harder test and the more useful test.** Ingesting content the project authored tells the team the pipeline doesn't crash on its own material. Ingesting content the project did not author tells the team the pipeline can learn something genuinely new. v1.40 was a necessary prerequisite; v1.44 is the release that actually measures the pipeline's external-content capability.
- **Health-gate the install stage before any ingestion runs.** The Phase 404-02 health-check runner prevents downstream phases from operating on a broken environment. A pipeline that runs learn / generate / validate against a half-installed venv produces garbage output ‚Äî and garbage output that looks correct is worse than a hard failure that surfaces the environment bug immediately.
- **Decision trees beat catalogs when the reader has to choose.** PyDMD has sixteen variants; an undifferentiated catalog asks the reader to read every entry before deciding. The pattern-synthesizer's seven-question tree answers the reader's actual question ‚Äî which variant should I use for this dataset? ‚Äî and the overflow subtree prevents the tree from silently dropping cases that under-discriminate on the canonical questions.
- **Enforce a word-count ceiling on generated SKILL.md at composition time, not after.** The five-thousand-word budget is enforced by the composer counting words and truncating the pitfalls section automatically when the budget is exceeded. Enforcing after emission means either shipping oversize skills or re-composing on overrun; enforcing inline means the output is always within budget by construction.
- **Generate runnable scripts, not just prose.** The script generator emits argparse-driven Python with proper `__main__` guards and matplotlib plotting that actually runs inside the install-stage venv. A reader can execute the output immediately, which is a stronger learning signal than a wall of prose example-code snippets that may or may not match the installed library version.
- **Validate internal links before publication.** The cross-reference checker walks every generated markdown link and emits diagnostics for broken resolutions. A skill with broken internal links erodes user trust faster than a skill with one missing section; catching it at generation time is cheap, catching it after publication is expensive.
- **Score extraction fidelity with ground-truth replay, not with extraction counts.** Counting extracted concepts tells the team something was extracted; it does not tell them whether the extraction was correct. The tutorial-replay engine runs each tutorial inside the venv and compares the observed trace against the extracted `KnowledgeGraph`, producing per-concept recall, per-variant precision, and cross-reference consistency ‚Äî a three-dimensional accuracy signal, not a one-dimensional count.
- **Route pipeline-stage events into the same observation pipeline that monitors the rest of the system.** Before v1.44, `sc:learn` was a black box whose output disappeared at the generator boundary. The Phase 407-01 observation bridge translates stage events into `Observation` records consumable by the detect-suggest-apply loop ‚Äî which means the rest of skill-creator can now see learning happen, and patterns that emerge from dogfood runs can feed back into skill suggestion.
- **Pin cross-module types with contract tests so refactors cannot drift the pipeline-wide API.** The five contract tests for `RepoManifest`, `KnowledgeGraph`, `GeneratedSkill`, `Observation`, and `AccuracyReport` guarantee that any future change to the shapes that cross module boundaries fails loudly at test time. Integration tests catch handoff bugs; contract tests prevent them.
- **A 13-step pipeline without an end-to-end integration test is a promise, not a proof.** The 999-line integration test wires the full chain (detect ‚Üí venv ‚Üí health ‚Üí learn ‚Üí generate ‚Üí validate ‚Üí bridge) with real modules and only I/O mocked. Unit tests cover each step in isolation; the integration test proves the steps compose into a working pipeline. Without it, every individual step could pass while the whole never runs.
