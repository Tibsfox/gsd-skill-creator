# v1.49.926 — Retrospective

## What went well

- **The four-axis survey triangulated the gap from three independent angles.** A 5-agent parallel survey (verify / calibrate / consume / chokepoint / forward-backlog) was launched to pick the next forward milestone after 14 counter-cadence/CI ships. Three agents independently converged on `token_budget.warn_at_percent` as the most-overdue gap — the verify agent found the missing e2e, the calibrate agent found the missing substrate, the forward-backlog agent found it via the consume-axis ledger. Independent convergence on the same target was the strongest signal that this was the right ship, not a contrived one.
- **The masked-debt catch (#10427 in action).** The `verify-overdue-scan.mjs` scanner reported all 7 thresholds `COVERED`, 0 overdue — a naive read says "no verify ship available." The real finding was one layer down: warn's `COVERED` verdict rested on a ground-truth entry (`tested @ v799`) that was a misattribution. Only checking the scanner's claim against `git log` (v799 = `bounded-learning audit log`, not an e2e) exposed it. The scanner's own data masked the debt the scanner exists to surface — the same "verify against ground truth, not against the artifact's self-report" discipline that bit the v925 checker.
- **The mirror was maximal, so design risk was minimal.** The substrate is a near-exact mirror of v893 `ceiling-substrate.ts`; the e2e is a 1:1 mirror of the v898 max e2e (8 tests, identical titles). The polarity mapping already existed in `token-budget-events.ts::eventKindToValue` and was not re-derived. This is the 4th instance of an ESTABLISHED pattern, not new design — adversarial review returned 4×OK / 0 blockers.

## What was tricky

- **The two-reading semantics are a genuine footgun.** Unlike `max_percent`'s single inequality, warn's outcome depends on the *follow-up* reading after a warn fired. A substrate that took a plain `usagePercent` and was called at warn-fire time would silently classify the fire as `ignored` — an inverted signal. Mitigated by naming the parameter `followUpUsagePercent` and documenting the precondition loudly in both the module and function docstrings. The review confirmed this is documentation-only (no runtime guard), acceptable for an observability-only substrate and matching the v893 sibling, but a real soft edge to close if a production caller is ever wired.
- **The event field's documented meaning drifted under a second writer.** `TokenBudgetEvent.usagePercent` was documented as "the reading that triggered the original warn" (the CLI flow). The substrate records the *follow-up* reading there. Resolved with a one-line, path-agnostic docstring clarification — verified against both writers (the CLI help text vs the substrate) so the field's two-writer semantics are honest and non-contradictory.
- **Streak-aware ship sequencing.** To earn macOS organic green #2 (`#10463`), the feature commit had to be pushed to `dev` *alone* so `ci.yml` ran the macOS leg on the organic commit — not bundled with the inert `chore(release)` tip (the v925 ship bundled them and never advanced the streak). Followed the v923 pattern: feat commit CI-green before the chore(release).
