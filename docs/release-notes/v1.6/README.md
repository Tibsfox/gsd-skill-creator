## Summary

**v1.6 turned the examples directory from a four-skill starter pack into a 31-artifact reference library.** v1.3 shipped four example skills — `git-commit`, `code-review`, `test-generator`, `typescript-patterns` — as the first reference implementations anyone could copy into their own project. v1.6 extended that directory into a multi-domain library: 20 skill examples, 8 agent examples, and 3 team examples, covering git workflows, code quality, TypeScript, decision-making, REST API design, Docker, environment configuration, session continuity, SQL, CI/CD, Claude Code hooks, web accessibility, supply chain security, and the GSD workflow surface. The authoring work landed in four thematic commits — `7ef90154f` (three GSD-complementary examples), `239f23f0e` (five more GSD-complementary examples with decision archaeology and plan-optimization agents), `c25c81357` (the standalone `beautiful-commits` skill), and `6048a3ecb` (the 18-example cross-domain batch: 10 general skills, 5 general agents, 3 teams). A separate commit at `400218454` installed the examples into the project's own `.claude/` tree so the project dogfoods its own starter pack. The tag message headlines "Cross-Domain Examples" and the evidence on disk matches: `examples/` grew from four directories to thirty-one.

**Examples are the skill-format's real specification.** v1.0 defined the YAML frontmatter + Markdown body schema. v1.3 documented it in `docs/architecture/storage.md` and shipped four reference skills. What v1.6 demonstrated is that the schema survives scaling across domains: a 385-line `sql-patterns` skill, a 688-line `gsd-trace` skill, a 790-line `gsd-migrate` skill, and a 106-line `git-commit` skill all parse cleanly against the same frontmatter validator. The spec does not need changes to accommodate the range from short to long, from procedural to reflective, from general-purpose to GSD-specific. Shipping 31 examples that all satisfy the v1.0 schema is empirical evidence that the schema is load-bearing, not aspirational. This is the kind of evidence that is hard to manufacture after the fact — it is generated by the act of authoring cross-domain content and discovering that the format does not push back.

**Three-topology validation for the v1.4 team design.** v1.4 shipped the team-composition surface with three declared topologies: leader-worker, pipeline, and swarm. v1.6 shipped one team example for each topology — `migration-team` (leader-worker with 5 mixed-access members), `doc-generation-team` (pipeline with 4 write-scoped members), and `code-review-team` (swarm with 5 read-only members for parallel multi-perspective review). If any of the three topologies could not support a real cross-domain use case, v1.6 would have surfaced it. None of them did. The `config.json` + `README.md` pair for each team, committed at `6048a3ecb`, compiles against the v1.4 team-schema validator and exercises the topology's coordination primitives as documented. That is the specific sense in which v1.6 is a validation of v1.4: the topologies were designed in the abstract at v1.4 and grounded in the concrete at v1.6.

**Dogfooding by install.** The commit at `400218454` runs `install.cjs` against the repo's own `.claude/` tree so that `examples/` and `.claude/` hold the same 31 artifacts — 20 skills under `.claude/skills/`, 8 agents under `.claude/agents/`, and 3 teams under `.claude/teams/`. The project now activates its own starter pack on every session: if a skill like `beautiful-commits` has a broken description, `beautiful-commits` fails to activate in a live session against the authoring team, and the defect surfaces immediately. If `gsd-trace` has a bad frontmatter field, it fails validation against the v1.0 index builder and `.skill-index.json` surfaces the error. This is the closed-loop-testing property that `install.cjs` unlocks: the examples are not just authored — they are used, so authoring mistakes become runtime failures for the authors themselves.

**GSD-complementary examples answer "how does this project use itself?"** Five of the examples — `gsd-explain`, `gsd-preflight`, `gsd-onboard`, `gsd-migrate`, `gsd-trace` (skills) and `gsd-health-checker`, `gsd-plan-optimizer`, `gsd-milestone-advisor` (agents) — are targeted at the GSD workflow surface this project ships inside `.claude/commands/gsd/`. They exist as examples rather than as built-in GSD features because they are recommended patterns external users can copy, not guarantees the project makes. `gsd-explain` previews what a GSD command will do before execution; `gsd-preflight` validates `.planning/` artifacts for STATE/ROADMAP drift; `gsd-onboard` is a 640-line four-stage interactive tutorial; `gsd-migrate` is a 740-line brownfield-project importer; `gsd-trace` is a 918-line decision-archaeology skill that traces decisions from vision through requirements through implementation. Each one is a reference implementation with a specific trigger pattern and a documented trade-off against the baseline GSD command surface. A user working on a GSD-managed project can copy any of them into their own `.claude/` tree and have the behavior immediately; the project shows what is possible without dictating what is required.

**Count discrepancy — 31 actual versus "34" in the original release notes.** The original v1.6 release notes claim "34 cross-domain examples" and `examples/README.md` summary table lists "Skills 23, Agents 8, Teams 3, Total 34." The actual file inventory at the v1.6 tag is 20 `SKILL.md` files, 8 `AGENT.md` files, and 3 team directories — 31 total artifacts. The "23 skills" figure in the examples README appears to double-count GSD-focused skills that also appear in the general skills table. This uplifted release note records the ground-truth count of 31 rather than repeating the original miscount. The content is the same; the headline number is corrected. Tabular inventory sources: `git ls-tree -r v1.6 -- examples/` yields 20 SKILL.md, 8 AGENT.md, 3 config.json (one per team).

**The 12-commit range understates the authoring investment.** The `v1.5..v1.6` range is 12 commits and 30,068 insertions. Four of those commits (`7ef90154f`, `239f23f0e`, `c25c81357`, `6048a3ecb`) contain the example authoring — roughly 15,000 insertions across ~60 Markdown files. One commit (`400218454`) is the `install.cjs` dogfooding wire-up. The remaining commits include the v1.5 documentation update (`13373117d`), two merge commits for the GSD-research-team audit (`b331b6ab0` at the start, `400218454`'s dependency), and the final README + CHANGELOG + version bump (`3780b5077`). The insertion count is dominated by prose — 30,068 lines of content, not code. A reader can open any of the 31 artifacts under `examples/` and get a complete, copy-pasteable, self-contained reference implementation with YAML frontmatter, a description that reliably activates the skill, a detailed body with worked examples, and for agents a tool-restriction list that encodes the agent's safety envelope. This is what 30,068 insertions bought: a starter pack that is genuinely usable rather than decorative.

## Retrospective

### What Worked

- **Cross-domain authoring exercises the schema more thoroughly than a formal test suite.** Writing 20 skills across 15+ domains surfaced exactly which schema fields pushed back and which did not. For the v1.0 frontmatter spec the answer was: none pushed back. A 106-line skill and a 918-line skill validate identically. A skill with one activation trigger and a skill with six activation triggers index identically. The schema scales on content density, not on field count.
- **Three team examples — one per v1.4 topology — completely validate the topology surface.** `migration-team` exercises leader-worker. `doc-generation-team` exercises pipeline. `code-review-team` exercises swarm. There are no v1.4-era topologies that went unused, and no topologies whose behavior required extension to support a real example. The v1.4 design landed correctly.
- **GSD-adjacent examples document patterns the project does not own.** `gsd-onboard`, `gsd-migrate`, `gsd-trace`, `gsd-plan-optimizer`, `gsd-milestone-advisor` extend the GSD surface without modifying `.claude/commands/gsd/`. Shipping them as examples rather than as built-ins means a user can opt in selectively — a brownfield migration team might want `gsd-migrate` and nothing else. Examples preserve the "reference implementation" contract: they show what is possible, they do not enforce what is required.
- **`install.cjs` closed the dogfooding loop.** Running the same install script the project documents for users against the project's own `.claude/` tree means every authoring mistake surfaces as a runtime failure for the authoring team. This is the same principle as v1.3's CLI-reference verification: the documentation and the code stay aligned because the people who write the documentation are the first people who have to use it.
- **Research-team sourcing kept the example library outward-facing.** The `6048a3ecb` commit message explicitly credits the `taches-cc-resources` research team for the source material for `decision-framework`, `context-handoff`, and `hook-recipes`. The `7ef90154f` commit credits the `gsd-research-team` parallel research with 4 specialist researchers for the GSD-adjacent examples. Using prior-art research for source material, then adapting it into skills rather than copying it verbatim, kept the examples from drifting into navel-gazing.
- **The `beautiful-commits` skill is a standalone deliverable that other projects can drop in tomorrow.** `git-commit` (v1.3) is a minimal Conventional Commits template. `beautiful-commits` (v1.6) is the full commit-crafting discipline — 11 commit types, imperative-mood test, WHY/WHAT body guidance, atomic-commit principles, worked examples, anti-patterns. The two skills co-exist because they target different user needs: `git-commit` for automation, `beautiful-commits` for reflection. Shipping both captures a v1.0–v1.6 design principle: skills are not required to be unique per domain, they are required to be useful per trigger pattern.

### What Could Be Better

- **The "34 examples" headline was wrong and propagated into `examples/README.md`.** The ground-truth count is 31 (20 skills + 8 agents + 3 teams). The "23 skills" figure in the examples-README summary table double-counts GSD-focused skills that also appear in the general-skills table. This uplift corrects the count in this README; a follow-up should reconcile `examples/README.md` against the on-disk file inventory so the two never disagree again. The lesson is general: headline counts should be generated by a script from the file tree, not typed by hand.
- **`install.cjs` is a manual step for external users.** Internal dogfooding is automatic because the authoring team runs `node install.cjs` as part of repo setup. External users who clone the repo have to know to run it — or have to know the examples live in `examples/` and copy them by hand. A future release should teach the CLI to discover examples directly (`skill-creator examples list`, `skill-creator examples install`) so the examples are first-class citizens rather than files that live in a directory a user might find.
- **No automated cross-schema validator shipped with v1.6.** Each example validates against the v1.0 frontmatter validator (for skills), the v1.4 agent schema (for agents), and the v1.4 team-config schema (for teams). But no single CI step checks that all three validators pass against the full `examples/` tree. A future release should add `skill-creator validate examples/` as a one-liner that exercises every schema against every artifact; the current workflow is manual validation during authoring, which does not catch post-authoring drift.
- **GSD-focused examples bleed into GSD's own surface area.** `gsd-health-checker`, `gsd-plan-optimizer`, `gsd-milestone-advisor`, `gsd-migrate`, `gsd-onboard`, `gsd-trace`, `gsd-explain`, `gsd-preflight` — eight of the 31 examples target the GSD workflow surface. That is 26% of the example library. A reader looking for general-purpose examples has to filter GSD examples out. The examples-README partially addresses this with a "GSD-Focused Examples" section, but the organization is by topic rather than by audience. A later release should split GSD-focused examples into a separate directory (e.g., `examples/gsd/`) so the audience separation is structural rather than sectional.

## Lessons Learned

1. **Examples are the best documentation of a schema.** 31 working examples across 15+ domains teach more about the v1.0 skill format than the v1.3 specification does. A reader can open any example directory and see the full schema exercised against a real problem; a reader of the specification has to reconstruct the schema from abstract field definitions. The implication for future releases: every schema change should ship with an updated example that demonstrates the change, and every release that does not change the schema should consider extending the example library to cover an uncovered domain.
2. **Three team examples completely validate three team topologies.** The v1.4 topology set — leader-worker, pipeline, swarm — was designed in advance of having any real team use cases. v1.6 grounded each topology in one team example. If a fourth topology had been needed, it would have surfaced during example authoring. None did. The lesson: a topology surface should be validated by one concrete example per topology before being declared stable; failing to produce an example for a declared topology is a design smell.
3. **Dogfooding via `install.cjs` turns examples into tests.** The moment the project's own `.claude/` tree holds the 31 examples, any broken frontmatter, any mis-typed activation trigger, any schema drift surfaces as a failed session against the authoring team's daily work. This is the cheapest test harness that could possibly work: the test is "does Claude Code load the skill?" and the test-runner is the session the authors are already running. Future releases that add example artifacts should add an `install.cjs` step in the same commit so dogfooding is not deferred.
4. **Ship headline counts from the file tree, not from memory.** The v1.6 original release notes and `examples/README.md` both claim "34" examples. The actual file tree contains 31. The four lost in translation do not exist anywhere on disk. A hand-typed count will diverge from the ground truth the moment an example is added, renamed, or removed. Counts that appear in documentation should be generated by a script that reads `git ls-tree` or `ls` output; the original miscount is a direct cost of hand-typing.
5. **Skills are not required to be unique per domain; they are required to be useful per trigger pattern.** `git-commit` (v1.3) and `beautiful-commits` (v1.6) both target git-commit authoring. They co-exist because their trigger patterns differ — `git-commit` activates on automation-style requests ("write a commit for X"); `beautiful-commits` activates on reflection-style requests ("how should I structure this commit message?"). A v1.0-era design concern — "won't two skills for the same domain collide?" — dissolves in practice when the trigger patterns are distinct. The v1.1 semantic-conflict-detection layer catches the case where two triggers actually do overlap; the v1.6 authoring experience confirms that the same topic supporting multiple skills is normal, not pathological.
6. **Tool-restriction lists are a safety surface, not documentation.** Every agent example in v1.6 has an explicit tool-restriction list in its frontmatter. `codebase-navigator` has Read, Glob, Grep (read-only). `security-reviewer` has the same (read-only by design for audit agents). `doc-linter` has the same. `gsd-health-checker` has Read + Bash (needs Bash to run validation scripts) but no Write. The restriction list encodes the safety envelope of the agent in a form the harness enforces at runtime, which means a misconfigured agent cannot accidentally escape its role. Writing tool-restriction lists for 8 agents across v1.6 made the design feature concrete: each restriction list is an engineering assertion about what the agent is allowed to do, not a courtesy comment for the reader.
7. **Research-team sourcing avoids navel-gazing.** Three of v1.6's general skills (`decision-framework`, `context-handoff`, `hook-recipes`) are adaptations of source material from `taches-cc-resources` — an external research corpus. Eight of v1.6's examples (the GSD-complementary set) came from a parallel-research exercise with four specialist researchers (`gsd-research-team`). Sourcing from external research kept the skills outward-facing; they are not skills about the project, they are skills about problems people actually have. The authoring rule that falls out: before writing an example, find real prior art to adapt — authoring from scratch without a source is a signal the skill may be idiosyncratic to the authors.
8. **Count and reconcile before shipping.** The "34 examples" headline survived from `examples/README.md` into the original `docs/release-notes/v1.6/README.md` and into downstream changelogs because no one ran a file-tree count at release time. A pre-commit or CI step that cross-checks every numeric claim in release notes against the repo state would have caught this at zero cost. The general principle: numeric claims in long-lived documentation are claims about repo state, and claims about repo state should be verified against repo state mechanically. Writing the verification once pays back across every release.
9. **GSD-adjacent features belong in `examples/`, not in `.claude/commands/gsd/`.** The eight GSD-focused examples are capabilities the project demonstrates but does not guarantee. Shipping them as examples keeps the GSD command surface lean (only the decisions the project commits to supporting get built-in treatment) while still making the extended pattern library available to users who want it. The lesson is structural: the line between "feature" and "reference pattern" is whether the project owns it, and the directory layout should reflect that line. `examples/` is where reference patterns live; `.claude/commands/gsd/` is where the project's commitments live.
10. **`install.cjs` is the simplest possible package manager.** It is `cp -r examples/*/SKILL.md .claude/skills/<name>/SKILL.md` plus a bit of logic. It does not resolve versions. It does not manage conflicts. It does not track installs. And at v1.6 that is exactly what was needed: the simplest possible script that turns `examples/` into `.claude/`. Future releases may need a real package manager (especially once `skill-creator` publishes examples via npm), but shipping the simplest script that worked meant v1.6 did not block on package-manager design. This is the same principle as v1.0's JSONL pattern store: solve the problem with the simplest tool that compiles in one's head, and escalate later only if the simpler tool fails.

## Engine Position

v1.6 sits six releases past the v1.0 zero-point, one release past v1.5 Pattern Discovery, and one release before v1.7 GSD Master Orchestration Agent. In the architecture the release extends v1.3's documentation-layer reference-implementation philosophy ("examples as real skills, not placeholders") and v1.4's team-topology surface ("three topologies that each support a real use case"), without modifying the v1.0 loop, the v1.1 validation layer, the v1.2 measurement layer, or the v1.5 discovery pipeline. Every subsequent release inherits v1.6's example library as the canonical starter pack; v1.7's intent-classification agent uses the v1.6 GSD-adjacent skill set as one of its candidate routing targets; v1.8's capability-aware planner uses the v1.6 agent tool-restriction lists as canonical safety envelopes; v1.25's 20-node dependency DAG includes the example-library as a node; v1.34's documentation ecosystem extends v1.6's "reference implementation" philosophy into a full narrative structure; v1.49's Grove-format import turns the example library into a content-addressed record set where the v1.6 31-artifact set is the seed. In the v1.x line, v1.6 is the release that demonstrated the skill format scales to a library, the team topologies ground in real use cases, and the examples directory is a first-class reference surface rather than marketing.
