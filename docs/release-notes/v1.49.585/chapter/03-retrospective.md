## New lessons emitted (#10168–#10174)

- **Lesson #10168 — Counter-cadence cleanup-mission pattern is operationally productive every ~30 forward milestones.** v1.49.585 demonstrates that periodically pausing forward-cadence ship velocity to convert accumulated social-rule debt into deterministic gates pays back at every subsequent forward ship. Cost: ~5-7 hours wall-clock for a 5-category cleanup sprint. Payback: lower friction on every subsequent forward ship + a meta-tested confidence boost (the gates are exercised against their own milestone).
- **Lesson #10169 — Gate-not-vigilance discipline.** When a social rule has been violated even once historically (v1.49.577-580 missing release-notes; pre-v1.49.585 accidental .planning commits; pre-v1.49.585 agent-source drift), the cost of converting it to a deterministic gate is paid back at the first re-violation. v1.49.585 proved it during its OWN execution: C01 self-mod-guard fired ≥7× during W1-W3, surfacing real false-positives that prose-only guidance would never have caught.
- **Lesson #10170 — Meta-test strategy at ship time.** Run the newly-installed gates against the milestone's own release-notes / push / chapters during the ship pipeline. The system gating itself is the strongest signal that the gate is real; if the meta-test passes, the gate is integrated. v1.49.585 W4 Phase 3 runs `check-completeness.mjs --current --strict` against v1.49.585's own release-notes, runs `chapter.mjs` against v1.49.585's own chapters (must produce zero diff), fires pre-push against v1.49.585's own push (must BLOCK on simulated-missing files; ALLOW on clean state).
- **Lesson #10171 — Architectural correction mid-mission: codebase-mapper outputs are starting points, not source-of-truth.** The CONCERNS audit's framing of `/media/foxy/ai/GSD/dev-tools/artemis-ii/` as a "sibling DEV-TOOLS PROJECT" was wrong; verification mid-session via `git worktree list` showed it's a git worktree of gsd-skill-creator's `artemis-ii` branch. The user pointed out the actual canonical .env lives at the project root. Future audits that produce filesystem-shape claims should be verified against `git worktree list` / `git remote -v` / `git config --get remote.origin.url` before treating them as ground truth.
- **Lesson #10172 — Scope-expansion mid-mission produces better outcome than scope-as-specified.** C08 was originally specced as "env-var override + fail-fast" (option B); user pivoted to "full deprecation" (option C) when reality contradicted the assumption that artemis-ii was a sibling-project. The result is a wrapper 30 lines shorter and structurally simpler — anonymous-password-list parsing and psql probe loop were both legacy complexity. Lesson: when reality contradicts a spec, ask the user for direction rather than mechanically execute the original.
- **Lesson #10173 — Hook self-tests must use `env -i` for full sterility.** v1.49.585 hooks installed live in the harness during their own milestone execution; the C01 self-test then inappropriately inherited the parent shell's `SC_SELF_MOD=1` env when set. Future hook self-tests should always run under `env -i ...` for fully-sterile invocation, not just `env -u VAR`. Cosmetic for v1.49.585 (the live BLOCK events ≥7× prove the hook is functional at runtime); v1.49.586+ tweak.
- **Lesson #10174 — `.claude/settings.json` is correctly gsd-config-guard-protected; new hook registrations route through `project-claude/`.** Edits to `settings.json` from Claude tool calls are BLOCKed by an existing pre-v1.49.585 hook. The right path for new hook registrations is via `project-claude/settings-hooks.json` + `node project-claude/install.cjs` (which spawns outside the hook layer). v1.49.585 added two hooks (C01 + C02) via this route cleanly.
