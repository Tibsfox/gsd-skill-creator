# v1.49.807 — Retrospective

**Wall-clock:** ~40-50 min from session-start to tag-push. Single-ship; first ship of a planned 4-ship chain (S5 → S2 → KNOWN_UNWIRED chip → T1.3 recon) per the v806 handoff selection.

## What worked

**The recon-first discipline (#10412) caught a framing error in the handoff.** The v806 handoff §175 framed S5 as "add normalizer to pre-tag-gate," but the recon (~5 min of reading `pre-tag-gate.sh` step 0.5 + step 17) revealed that both normalizers were already in pre-tag-gate. The actual gaps were narrower: (a) step 0.5 ran `--write` but never verified post-write idempotency; (b) step 17 surfaced drift as WARN-only with no patch-drift ceiling. The narrower framing produced a smaller diff (2 step modifications + 1 memory-note refresh + 1 PROJECT.md hand-edit) instead of the larger "add normalizer to pre-tag-gate" scope the handoff implied.

**The memory-note staleness check paid off twice.** The first payoff: the existing memory note `feedback_state-md-normalizer-non-idempotent.md` claimed the normalizer required two `--write` invocations to converge, but actual reproduction (4 synthetic drift cases — flat predecessor_*, empty body, mixed Z-containing prose, missing sections) showed the normalizer converged in a single `--write`. Git-log on `tools/state-md-normalizer.mjs` showed commit `e5d0cbc69` (2026-05-26, shipped in v1.49.783) had explicitly closed both the missing-`Engine state baseline` bug AND the `/\Z/` literal-Z strip-regex bug. The "run --write twice" advice was obsolete; updating the note prevents future ships from inheriting the stale habit. The second payoff: the v806 handoff explicitly cited this memory note ("Required 2× `--write` invocations (per memory note)"), which suggests the v806 operator ran `--write` twice on reflex rather than necessity — confirming the staleness was actively misleading recent ships.

**The "is this within tolerance" policy stayed in bash, not in the normalizer.** Initial sketch added a `--max-patch-drift=N` flag to `project-md-normalizer.mjs`. Reconsidered: the normalizer is a structural validator with a small, well-defined responsibility; the threshold is a pre-tag-gate-specific policy (no other consumer of the normalizer would want it). Putting the regex parse and arithmetic in the bash gate keeps the normalizer pure and makes the policy obvious to anyone reading `pre-tag-gate.sh`. Lightest-wire choice per #10416.

**No new test harnesses added.** The two gates are detector-shaped (loud-fail on regression). Adding vitest coverage for bash gate behavior would have meant: build a fixture STATE.md with synthetic drift, invoke pre-tag-gate.sh in a subprocess, assert exit code + stderr pattern. That's an entirely separate ship's worth of infrastructure for a gate whose correctness is obvious by inspection (5 lines for step 0.5; ~25 lines for step 17). Deferred to "extend `tools/pre-tag-gate.test.sh` when next touched" — a #10416 lightest-wire judgment, not laziness.

## What surprised

**The handoff's S5 framing was based on incomplete information.** The handoff §175 stated "the STATE.md drift caught this ship would have been caught at T14 if the normalizer were in pre-tag-gate." But step 0.5 IS in pre-tag-gate, and was during v805's T14. The actual cause of the v805→v806 drift was: STATE.md gets hand-rewritten as part of the post-ship "next-milestone reset" step in T14 (AFTER pre-tag-gate), and the post-reset STATE.md is never re-normalized. The drift surfaces on the NEXT milestone's pre-tag-gate step 0.5 (which silently `--write`s it back to canonical form) but is invisible to the operator until vitest's `--check` test catches it. The v807 fix doesn't close this exact recurrence — it makes any future regression to the non-idempotency loud — but a complete closure would require running the normalizer at the END of T14's STATE.md reset step, not just at the START of the next pre-tag-gate. Captured as a forward observation for the next counter-cadence.

**The PROJECT.md normalizer's `--check` exits 0 even with WARNs.** The normalizer's main() at lines 269-291 exits 0 unless there are BLOCKs (or `--strict` is set). So pre-tag-gate step 17 had to explicitly check the output text for "drift detected" rather than rely on exit code. This is a deliberate design choice (per the normalizer's docstring: "WARN-only by default — PROJECT.md is gitignored hand-authored prose; auto-fix is out of scope"), but the consequence is that pre-tag-gate's escalation logic has to live in bash rather than relying on `--strict`. Acceptable but worth noting.

**A scope-doubling moment got correctly resisted.** Mid-implementation, I considered adding a `--write` mode to `project-md-normalizer.mjs` that would auto-bump the version number in the "Latest shipped release" line. Recognized that this would create a worse state: the line carries a description and date that would go stale alongside, producing a "lying" line that technically passes the version check but reads incorrectly. The conservative-by-design discipline (no auto-rewrite of hand-authored prose) is load-bearing here, not vestigial. Bounded drift cap is the right shape.

## What to watch

- **STATE.md post-T14-reset drift recurrence.** v807 closes the regression detector for normalizer non-idempotency, but the v805→v806 drift class (post-reset hand-edit) is a different recurrence. Next observation: does v807→v808 produce another drift instance at the start of v808's pre-tag-gate step 0.5? If yes, a follow-on ship should add a normalizer pass to the post-ship STATE.md reset step itself.
- **PROJECT.md patch-drift ceiling tuning.** Default is 3 — derived from the principle that PROJECT.md should be re-touched every ~3 ships at minimum. If operators consistently hit the cap, raise to 5; if drift stays well below 3, keep tight. First data point will be v808.
- **The "tentative observations" count holds at 8 across this ship.** v806 carried 3 NEW from v805's 5. v807 adds none — purely a gate-tightening with no new observation classes. Lesson backlog stays at 0 candidates.
- **Verdict on the chain shape.** v807 is the first ship of a planned 4-ship chain. v804+v805 chained-session ran ~70-75 min for two ships (~37 min per ship); v806 ran ~75-90 min as a single substrate-introducing ship. v807 ran ~40-50 min as a single enforcement-tightening ship. The single-ship pattern matches the ~30-50 min "1 component / 30-50 min" target from Lesson #10193 (sub-agent dispatch sizing) — even in main-context. If the remaining 3 ships in the chain hit similar wall-clocks, the chain compresses to ~2-3h total — comparable to the v804+v805 compression but without the chained-session orchestration overhead.

## Verdict on scope

S5 closure landed in the smallest viable shape: 2 bash gates + 1 PROJECT.md hand-edit + 1 memory-note refresh + 5 release-notes files. Resisted: `--write` mode for project-md-normalizer; new vitest test harness for bash gate behavior; unification of the two normalizers; a complete post-T14-reset drift closure (deferred to forward observation). All resistances paid off — the diff stays surgical, the policy stays in one file, and the regression detector is loud enough to catch any future drift class that surfaces in the same shape.
