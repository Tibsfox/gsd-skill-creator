## What went unusually well

- **The "flake" investigation paid for itself twice.** The reported symptom was a non-bug (mis-attributed expected stderr), but chasing it surfaced a real latent defect (a stale policy allowlist producing a false-positive violation) AND the structural gap behind it (the audit was never gate-wired since v607). A 1-line console mis-read led to a real correctness fix + a new enforcement. Reactive flake-triage is worth doing properly even when the flake turns out to be benign.
- **Adversarial verification before touching a policy tool.** Rather than trust the scout's "allowlist `loader-context.js`" recommendation, the ADR-0003 text was read directly and confirmed the import falls under category (b) ("an existing repo-root primitive in `src/*` outside atlas"). The `execSync` stderr behavior was settled by an empirical probe (default echoes; explicit `stdio:['ignore','pipe','pipe']` suppresses while keeping `err.stderr`), not by memory. Editing a clean-room-policy enforcer on a guess would have been the wrong move.
- **The #10448 audit terminated cleanly at a verified no-op.** The operator asked "are any #10448 carry-forward candidates ready to promote?" and the answer — adversarially checked against disciplines.json, architecture-retrofit-patterns.md, and the v910–v914 lessons — is a confident "no, and here is exactly how close each of the five is." A clean negative is a real deliverable.

## What went less well

- **The v914 handoff propagated a mis-attribution.** v914's own retrospective already flagged the "atlas-deps-audit flake" as `atlas-index-cli.test.mjs` failing — and that attribution was wrong. It was recorded in good faith during a tool-channel-degraded session, but it cost this ship an investigation to disprove. Lesson re-affirmed: when a `FAIL`/error string shows in test console output, grep the suite for the literal string to find which test *emits* it before naming a culprit.
- **The enforcement gap was two years of latent debt in miniature.** `atlas-deps-audit` shipped with a documented acceptance test (ADR-0003 §Verification) that nothing ran for ~300 ships. The allowlist drifted the moment the v905 chokepoint wire landed and stayed silently wrong until a stray console line happened to surface it. A policy tool with an unwired acceptance test is a slow-motion false-positive generator.
- **origin/dev's CI was already red — twice — and the v914 ship missed it.** The v915 pre-tag-gate's CI-on-dev step failed: the tools-suite CI job that v914 added was red on `17f6db91d`. Fixing the first red uncovered a second: (1) `mus-smoke/build-template-instruction.test.mjs` reads a gitignored `.planning/` template (ENOENT in CI); (2) `release-history/chapter.mjs` ran `main()` on import (no entrypoint guard), so a test importing it triggered a PG query that, with no PG in CI, threw `undefined.version` → async `process.exit(2)`. v914 wired the tools suite into CI but never verified the *CI* run went green afterward (the v914 session's tool-channel degradation), so two real reds shipped silently. The fixes (a #10182 skip-guard + an entrypoint guard) were both cheap; the lesson is that wiring a suite into CI is only half the work — the other half is confirming the suite is CI-*safe* (no gitignored-file, no PG, no local-only or import-side-effect dependencies) and *watching the first CI run go green*. This is the deferred-maintenance discipline's "an open red on dev is the alarm" — and a reminder that `gh run watch --exit-status` can disagree with the authoritative `gh run view` conclusion, so the run's conclusion must be re-checked, not trusted from the watch's exit code alone.
