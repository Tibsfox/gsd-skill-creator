## Summary

**The PNW Research Series landed as the project's first regional-biodiversity content surface, wired to the math co-processor chipset in the same release window.** Before v1.49.22 the project had a mission-pipeline stack (vision-to-mission from v1.49.17 forward, image-to-mission from v1.49.21) and a pack-shape content pattern (heritage-skills-pack from v1.49.12), but no first-class regional research surface tied to the Pacific Northwest base dataset that anchors GSD-OS as an autobiographical framework. This release ships four complete static research browsers — Columbia Valley Rainforest (`www/PNW/COL/`), Cascade Range (`www/PNW/CAS/`), PNW Living Systems (`www/PNW/ECO/`), and PNW Gardening (`www/PNW/GDN/`) — each with client-side-rendered markdown research documents, mission pack PDFs, and verification matrices, unified under a master index at `www/PNW/index.html` that cross-links seven thematic threads (salmon, mycorrhizal web, vertical gradient, old-growth, climate futures, native plants, heritage connection) across all four browsers. The release name is descriptive rather than aspirational: the series exists on disk after v1.49.22, is navigable as a whole through a shared `series.js` navigation injection, and is paired with the math co-processor chipset (`math-coprocessor/` + `data/chipset/math-coprocessor.yaml`) that brings GPU-accelerated linear algebra, FFT, statistics, symbolic math, and vector calculus to the dashboard with a NumPy CPU fallback for every operation.

**Static HTML with client-side markdown rendering proved itself as the right shape for research browsers.** The COL, CAS, ECO, and GDN browsers all share the same architecture: an `index.html` landing page, a `page.html` document viewer that fetches and renders markdown on the client, a `mission.html` that frames the browser's purpose and scope, a shared `style.css`, and a `research/` directory of markdown documents. There is no build step, no dependency graph, no server requirement — the browsers work offline, load instantly, and open in any modern browser with zero configuration. Adding a new research topic to an existing browser is a matter of dropping a new markdown file into `research/` and linking it from `page.html`; adding a new browser to the series is a matter of copying an existing browser's shell and swapping content. The pattern scales across topics without engineering overhead and composes cleanly with the master index because each browser exposes the same surface area through the shared `series.js` navigation bar injection.

**The math co-processor foundation ships with the five-chip separation that maps to distinct computational domains.** `math-coprocessor/chips/algebrus.py` covers linear algebra (matrix ops, decompositions, solves — cuBLAS-backed), `chips/fourier.py` covers spectral analysis (FFT/IFFT — cuFFT-backed), `chips/statos.py` covers statistics (moments, distributions, sampling — cuRAND-backed), `chips/symbex.py` covers symbolic mathematics, and `chips/vectora.py` covers vector calculus. Each chip is a thin orchestration layer over a distinct GPU library backend with a NumPy CPU fallback defined in `fallback/cpu.py` (296 lines). The separation is load-bearing: finer granularity (one-chip-per-operation) would fragment the surface; coarser granularity (one-chip-for-all-math) would couple unrelated GPU library dependencies together. The 5-chip shape matches the way mathematical work actually clusters in the downstream consumers that will use this co-processor through the MCP protocol — linear algebra calls come in batches, spectral analysis calls come in batches, and statistical sampling calls come in batches, and each batch prefers to stay resident on the same GPU context.

**GPU acceleration rides on top of a CPU oracle, not instead of one.** Every operation in the math co-processor has two implementations: a GPU path that dispatches to the appropriate CUDA library (cuBLAS, cuFFT, cuSOLVER, cuRAND) and a NumPy CPU fallback that produces the same result shape. The CPU fallback is not a legacy or compatibility layer — it is the correctness oracle against which GPU results are verified, and it is the default code path when the GPU context is unavailable (no CUDA, no compatible device, contention with another tenant). The 125 tests in `math-coprocessor/tests/` cover correctness (`test_correctness.py`, 289 lines), edge cases (`test_edge_cases.py`, 147 lines), integration (`test_integration.py`, 230 lines), performance (`test_performance.py`, 123 lines), and safety (`test_safety.py`, 266 lines) — and they run in both paths. The implication for downstream consumers is that the co-processor is a GPU-accelerated accessory, not a GPU-required dependency: a dashboard panel, a mission pack, or a research notebook that calls into the co-processor keeps working on a machine that has no GPU, just slower. That property was the non-negotiable design constraint, and it carried through every chip.

**The master series index is the discoverability substrate for the four browsers.** `www/PNW/index.html` (300 lines) introduces each project card, links the research documents, surfaces aggregate statistics (total research modules, total words, total species/communities documented), spells out a recommended reading order, pins a geographic coverage map across the four browsers, and publishes a cross-reference matrix of the seven thematic threads (salmon ecology, mycorrhizal networks, vertical elevation gradient, old-growth forest structure, climate-change futures, native-plant communities, heritage connections to tribal knowledge). Each thread deep-links to specific research documents across whichever browsers touch that theme — the salmon thread threads through COL/aquatic, CAS/aquatic, ECO/fauna-marine-aquatic, and GDN/05-pest-disease-adaptation; the mycorrhizal thread threads through COL/fungi, CAS/fungi, and ECO/fungi-microbiome-survey. The index is what turns four separate research silos into a navigable series, and the shared `series.js` (38 lines) auto-injects a navigation bar into the 12 sub-project HTML files (index, page, mission for each of the four browsers) so a reader can jump back to the master index or laterally to another browser from any leaf page with one click.

**Research verification is an explicit artifact, not an afterthought.** Each browser ships a verification document: COL has `research/verification.md`, CAS has `research/verification.md` (218 lines), ECO has `research/verification-matrix.md` (121 lines), and GDN has `research/07-verification.md` (696 lines). These documents enumerate the research claims that matter, cite the primary sources, and note where the claim is strong (primary source corroborated by peer-reviewed literature), moderate (single peer-reviewed source), or provisional (popular literature or tribal oral record only). The PNW Gardening browser passes 24/24 tests (7 safety-critical, 12 core, 5 integration), and the PNW Living Systems browser passes 29/32 tests with 3 PARTIAL status and 6/6 safety-critical PASS. The verification artifact is the honesty layer: it tells a downstream reader which claims to trust without further lookup and which claims to verify independently before acting. Explicit verification is a better shape than silent confidence — a browser that ships a verification document is one whose claims have been cross-referenced, and a browser that ships no verification document has opted out of that discipline.

**Mission packs accompany each browser, carrying the research into printable form.** COL, CAS, ECO, and GDN each ship a mission pack under `mission-pack/`: PNW Living Systems has two (an ecosystem mission PDF at 27 pages and a fungi taxonomy mission PDF at 18 pages), PNW Gardening has one (a 26-page gardening mission PDF), and the other browsers carry analogous artifacts. The mission pack is the LaTeX-compiled PDF form of the browser's research, suitable for printing, offline reading, or hand-off to a downstream agent through the mission-package pattern established by v1.49.12's heritage-skills-pack. Each pack is accompanied by its LaTeX source (`.tex`) so the compilation is reproducible and the research is version-controlled in editable form. The pattern mirrors the v1.49.21 transmission-package discipline — a research artifact should be portable across context boundaries without secret dependencies, and a compiled PDF paired with its source is the most portable shape a research document can take.

**The Cascade Range browser stretched the content surface to 10 elevation zones.** `www/PNW/CAS/research/zones.md` (583 lines) documents the Cascade elevation gradient from sea-level riparian corridors through Douglas-fir forests, western hemlock, Pacific silver fir, mountain hemlock, subalpine parkland, alpine tundra, and summit fellfields, with each zone's flora, fauna, fungi, and aquatic communities described separately. The flora document (`flora.md`, 1,221 lines) is the largest single content artifact in the release, covering species composition across all 10 zones; the fungi document (`fungi.md`, 614 lines) and aquatic document (`aquatic.md`, 802 lines) follow the same cross-zone pattern. This vertical gradient is one of the seven thematic threads in the master index because it is a characteristic feature of Pacific Northwest ecology — the compressed climatic zonation that makes the Cascade Range a natural laboratory for climate-change study. Two of the Cascade research agents (cas-611 and cas-613-fauna) failed during the CAS run, and the recovery was to write the documents directly rather than re-dispatch the agents; that fallback is documented as a lesson because it generalizes — agent orchestration is an optimization, not a requirement, and a content pipeline should always have a direct-write path available.

**PNW Living Systems brought ecosystem taxonomy to life as the dual-mission-pack browser.** `www/PNW/ECO/` is the most content-dense browser in the series: 26 files, 1.4 MB on disk, research spanning flora (41 species), fauna-terrestrial (86 species), fauna-marine (39 species), fungi (23+ communities), ecological networks (5 pathways), heritage bridge (17+ tribal nations), cross-module merge (189 unified taxonomic records), a chipset definition (6 chips/5 buses, `pnw-ecosystem.chipset.yaml`), a Minecraft world spec, engineering optimization notes, and a college-biology seed document. The foundation schema includes `silicon.yaml` (5 GPU engines), `coordinate-projection.md` (40.05 ft/block projection rule), and `shared-attributes.md` (8 elevation bands, 17 habitats, 13 ecological roles). The two mission packs — ecosystem (27 pages) and fungi taxonomy (18 pages) — were co-authored alongside the browser so the research flows into printable packs without an export pass. The ECO browser demonstrates the upper end of the static-browser pattern: a single-browser content density large enough to support two mission packs, multiple cross-module merges, and an embedded chipset definition.

**The Minecraft SocketEdit skill is the first external consumer of the v1.49.21 transmission-package shape.** `3b2b27006 feat(skill): add minecraft-socketedit skill with complete pipeline docs` shipped the skill as a supporting artifact, with pipeline documentation that follows the four-wave perceive→synthesize→translate→transmit shape from the image-to-mission pipeline. The skill documents the socket-protocol path between Claude Code and a Minecraft world-editing server, including structured input types, translation to block-placement commands, and transmission-package self-containment checks. Its landing in the same release window as the PNW Research Series is deliberate: the PNW Living Systems browser ships a Minecraft world spec (`research/minecraft-world-spec.md`, 450 lines) that the SocketEdit skill consumes, so the skill and the research browser arrive as a matched pair. The pattern — a content browser that produces a world spec, a skill that consumes the spec and builds the world — is the first demonstration of the pack-to-skill-to-world pipeline that later releases expand.

**Dashboard integration brought the math co-processor into the Tauri desktop surface.** `desktop/src/dashboard/math-panel.ts` (317 lines) and `math-panel.test.ts` (333 lines) add a Math Panel to the GSD-OS dashboard with 16 UI tests covering panel rendering, chip status, VRAM monitoring, benchmark invocation, and result display. `math-coprocessor/streams.py` (134 lines) adds CUDA stream management so multiple chips can run concurrently on the same GPU context without serializing through a single default stream. `math-coprocessor/benchmarks.py` (302 lines) ships a benchmark suite that produces a performance baseline for each chip, both GPU and CPU paths, so later releases can track performance drift. `math-coprocessor/vram.py` (121 lines) exposes VRAM utilization to the dashboard so a user can see how much GPU memory each chip is holding. The integration is deliberately modular: the co-processor is not embedded in the dashboard, it is connected through the MCP protocol, so the same co-processor serves the desktop dashboard, the CLI, and any future downstream consumer without code duplication.

**Config-driven chip activation keeps the co-processor footprint small when chips aren't needed.** `math-coprocessor/config.py` (148 lines) reads `mcp-config.json` and loads only the chips that are enabled for the current session. A workflow that only needs linear algebra loads `algebrus` and leaves `fourier`, `statos`, `symbex`, and `vectora` dormant. A workflow that only needs statistics loads `statos`. The GPU context for a dormant chip is never allocated, so the VRAM footprint tracks actual usage rather than maximum-possible usage. The pattern is the chipset-version of tree-shaking: the static MCP configuration tells the runtime which chips to instantiate, and the runtime instantiates only those chips. Config-driven activation also means a downstream consumer that wants to extend the co-processor with a new chip can do so by writing a new `chips/<name>.py`, adding an entry to `mcp-config.json`, and having the runtime pick it up on the next session — no runtime code changes required.

## Retrospective

### What Worked

- **Research browser architecture is proven across four independent landings.** The COL, CAS, ECO, and GDN browsers were built independently and converged on the same shell (`index.html` + `page.html` + `mission.html` + `style.css` + `research/` markdown directory). Adding a new research browser after v1.49.22 is a matter of content authorship, not engineering. The pattern composed cleanly into a series through the shared `series.js` navigation injection without requiring any browser to know about any other browser.
- **Master index creates discoverability that silos cannot.** Before the master index, the four browsers were four separate URL trees with no awareness of each other. After the master index, they compose into a navigable series with seven thematic threads that deep-link across browsers. A reader arriving at COL/fungi through a salmon search does not just get COL/fungi — the index lets them follow the mycorrhizal thread to CAS/fungi and ECO/fungi-microbiome-survey without re-searching. Discoverability is a cross-cutting concern, and the index is the right place to carry it.
- **GPU math with a CPU oracle made correctness testing unconditional.** Every math-co-processor operation has a NumPy CPU fallback that serves as the correctness oracle for the GPU path. The 125 tests in `math-coprocessor/tests/` ran on a CI machine with no GPU because the CPU path is the default, and they ran on a GPU machine with full GPU coverage because the GPU path is tested against the CPU oracle. A single test suite exercised both paths with identical assertions.
- **Config-driven chip activation kept the co-processor footprint proportional to usage.** `mcp-config.json` declares which chips are enabled, and the runtime only allocates GPU contexts for enabled chips. A workflow that only uses linear algebra never pays the VRAM cost of loading the FFT, statistics, symbolic, and vector-calculus chips.
- **Mission packs co-authored with browsers avoided an export pass.** The LaTeX mission packs for ECO and GDN were authored alongside the browser content, not exported from browser content after the fact. The two artifacts stay in sync because they are written in the same window, not because a tool is trying to keep them aligned.
- **Verification documents made the trust layer explicit.** Each browser ships a verification artifact that enumerates claims, cites sources, and rates confidence (strong / moderate / provisional). A downstream reader knows which claims to trust without further lookup and which to verify before acting, rather than being forced to treat the whole browser as uniformly authoritative or uniformly suspect.
- **Direct-write fallback absorbed agent failures without blocking the release.** When the Cascade Range research agents `cas-611` and `cas-613-fauna` failed during the CAS run, the recovery was to write the affected documents directly. The release did not stall on retry loops or agent-orchestration debugging — the content shipped on schedule because the direct-write path was always available.

### What Could Be Better

- **Research content is large and lacks in-browser search.** Individual research files (`flora.md` at 1,221 lines, `aquatic.md` at 802 lines, `fauna-terrestrial.md` at 2,446 lines) are substantial. A reader looking for "Douglas squirrel" has to use the browser's Ctrl-F across whichever document they happen to be on; a search/filter mechanism within each browser — or better, across the whole series — would make the content more navigable. This is a v1.49.23+ improvement opportunity.
- **No automated verification checking across browsers.** Verification documents are manually authored. A claim in `COL/research/aquatic.md` about coho salmon lifecycle timing might contradict a claim in `CAS/research/aquatic.md`, and nothing in the release pipeline catches that. Automated cross-reference checking — at least for numeric claims (species counts, elevation ranges, spawn dates) — would turn silent drift into explicit diffs.
- **Math co-processor tests pass on CPU-only CI but lack cross-path verification.** The test suite runs both paths separately; it does not (yet) run the same input through both paths in the same test and assert they produce equal results. That would turn the CPU fallback from a documented oracle into a continuously-exercised oracle, catching GPU regressions against the CPU ground truth every commit.
- **Three PARTIAL statuses in the ECO verification matrix.** The PNW Living Systems browser passes 29/32 tests with 3 PARTIAL. A PARTIAL status means the verification found a claim that was not fully corroborated, not that the claim is wrong — but a follow-up research pass to either fully corroborate or explicitly mark as provisional would raise the trust floor of the browser.
- **Minecraft SocketEdit skill ships documentation-only.** The skill in `3b2b27006` is a pipeline-documentation artifact, not an operational skill binding. A downstream consumer wanting to actually drive a Minecraft server needs to implement the socket protocol themselves; a skill shell that wraps the protocol and exposes it as a typed operation would move the skill from "specification" to "tool."
- **Series-level search is missing.** Per-browser in-document search is one gap; series-level search across all four browsers is another. The master index carries thematic threads but does not carry a searchable index. An inverted-index artifact generated from all research documents in the series would enable "find every mention of Douglas-fir across COL, CAS, ECO, GDN" without a manual sweep.

## Lessons Learned

- **Static HTML with client-side markdown is the right architecture for research browsers at this scale.** No build step, no dependency graph, no server requirement. The pattern scales across topics without engineering overhead — four browsers converged independently on the same shell, and the master index composed them into a series without any browser having to know about any other browser. A future browser can be authored by copying the existing shell, swapping content, and linking into the master index. The architecture absorbed four independent content efforts without a single engineering coupling between them.
- **GPU compute needs CPU parity as an oracle, not just as a fallback.** Building the CPU fallback first, then adding GPU acceleration on top, ensured correctness testing was never blocked by hardware availability. The fallback is the ground truth against which GPU results are verified — a pattern that generalizes: any accelerator path should have a known-good slower path that doubles as the correctness oracle, because that is the only way to catch regressions in the accelerator without re-deriving correctness from scratch every test.
- **Five-chip separation is the right granularity for a math chipset.** Each chip maps to a distinct mathematical domain with a distinct GPU library backend (cuBLAS, cuFFT, cuSOLVER, cuRAND, NVRTC). Finer separation would fragment the surface (every operation becomes its own chip, management overhead dominates); coarser separation would couple unrelated library dependencies (one chip failure cascades across unrelated domains). The 5-chip shape matches how mathematical work actually clusters in consumers.
- **Three research agents plus a direct-write fallback is a reliable content pipeline.** When individual research agents fail (as `cas-611` and `cas-613-fauna` did), writing the document directly is always available as a fallback. Agent failures should never block content delivery. The implication is broader: any orchestration layer should have a known-good manual path, because orchestration is an optimization and a release schedule does not pause for optimization debugging.
- **Config-driven activation scales chipsets without scaling VRAM cost.** `mcp-config.json` declaring which chips are enabled — and the runtime only allocating GPU contexts for enabled chips — means a chipset with 5 chips does not pay the cost of 5 GPU contexts unless all 5 are used. A future chipset with 20 chips still pays only the cost of the chips the current workflow uses. The pattern avoids the trap where adding capability costs memory even when the capability is dormant.
- **Shared navigation injection beats copy-paste navigation across sub-projects.** `www/PNW/series.js` (38 lines) injects a navigation bar into every sub-project HTML file. Without the shared script, each of the 12 sub-project HTML files would carry a hardcoded navigation bar, and updating the series-wide navigation would require editing 12 files. With the shared script, the navigation is a single source-of-truth artifact that the sub-projects opt into by including one `<script>` tag. Cross-cutting UI concerns should live in a cross-cutting artifact, not in every leaf file.
- **Verification documents are the honesty layer for research content.** Every research browser should ship a verification artifact that enumerates claims, cites sources, and rates confidence. Without a verification artifact, a downstream reader is forced to treat the whole browser as uniformly authoritative or uniformly suspect. With a verification artifact, a reader knows which specific claims to trust and which specific claims to verify further — and the research author is forced to be explicit about provenance rather than hiding behind the overall framing.
- **Mission packs co-authored with browsers stay aligned without tooling.** The LaTeX mission packs for ECO and GDN were written in the same window as the browser content, not exported from browser content after the fact. The two artifacts stay synchronized because the author is looking at both forms simultaneously during authorship, not because a tool is reconciling them after divergence. Co-authorship is cheaper than synchronization tooling when the authorship window is a single release.
- **Master index deep-linking by theme outperforms flat listings.** `www/PNW/index.html` could have shipped as a flat four-item list of browsers. Instead it ships with seven thematic threads (salmon, mycorrhizal web, vertical gradient, old-growth, climate futures, native plants, heritage connection) that deep-link to specific research documents across multiple browsers. A reader arriving through a thematic thread lands on the exact document they need, not on a browser landing page they have to re-navigate. Thematic linking is what turns four browsers into a series.
- **Minecraft world spec as a research artifact is the hand-off to a skill consumer.** `www/PNW/ECO/research/minecraft-world-spec.md` (450 lines) is simultaneously a research document and an input to the Minecraft SocketEdit skill. A research browser that produces specs consumable by downstream tools is more valuable than a research browser that produces only prose. The pattern — research document as machine-readable spec — is the generalization: content artifacts should carry their downstream consumability as a first-class concern.
- **Pack-shape content composes with skill-shape consumers through specs.** The v1.49.12 heritage-skills-pack pattern, the v1.49.21 transmission-package pattern, and the v1.49.22 mission-pack-plus-world-spec pattern are all versions of the same idea: content becomes portable when it ships in a well-typed shape, and skills become useful when they consume well-typed content. The pairing of the ECO Minecraft world spec with the Minecraft SocketEdit skill is the first demonstration of that pipeline end-to-end.
- **Release-ledger updates in the same window as the content keep the documentation honest.** CHANGELOG, FEATURES, and RELEASE-HISTORY entries for v1.49.22 were written in the same commit window as the content itself. Deferring them to a later cleanup commit would have created a window where the published ledger disagreed with the shipped content. A documentation layer that lags the code by even one commit is a documentation layer that can drift; a documentation layer written in lockstep stays honest by construction.

The release's 9 commits, 107 files, and approximately 41,300 new LOC make it one of the largest v1.49.x feature releases by content volume — the ECO browser alone adds 14,764 insertions, and the GDN browser adds 8,403 — while staying structurally contained because every file is a member of one of two cohesive modules (`www/PNW/` for content, `math-coprocessor/` for compute). The architectural footprint is disproportionately large because v1.49.22 is where regional biodiversity research entered the mission pipeline as a first-class content surface and where the first accelerator-backed chipset landed on the dashboard.
