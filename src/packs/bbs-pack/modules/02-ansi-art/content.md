# Module 2: ANSI Art

> **Tier**: 1 | **Safety Mode**: Annotate

## Overview

ANSI art is the visual language of the BBS era. While Module 1 established the serial connection that carries data between computers, this module covers what that data looks like when it reaches the screen. ANSI art uses escape sequences defined by ANSI X3.64 (ECMA-48) to control text color, cursor position, and screen clearing, combined with the CP437 character set's box-drawing and block elements to create detailed visual compositions within the 80-column, 25-row terminal grid. Every BBS login screen, file menu, message board header, and door game interface was built with these tools. -- ANSI X3.64 / ECMA-48; CP437 / Unicode Mapping

The ANSI art scene emerged as a distinct digital art movement in the early 1990s, with organized groups like ACiD Productions and iCE competing to produce the most visually striking BBS artwork. Artists used specialized editors like TheDraw and ACiDDraw to place characters and set colors, working within the constraints of 16 foreground colors, 8 background colors, and 256 fixed-width characters. The SAUCE metadata standard allowed artists to embed their name, group affiliation, and the artwork's dimensions directly in the file. Understanding ANSI art requires understanding three layers: the escape sequence language that controls rendering, the character set that provides the visual vocabulary, and the metadata format that identifies the work. -- SAUCE spec; ANSI X3.64 / ECMA-48

## Topics

### Topic 1: ANSI Escape Sequences

ANSI escape sequences are in-band control codes embedded in the text stream that instruct the terminal to perform actions beyond simply displaying characters. Every sequence begins with the Control Sequence Introducer (CSI), which is ESC followed by "[" (bytes 0x1B 0x5B). The CSI is followed by zero or more numeric parameters separated by semicolons, and terminated by a single letter that identifies the command. For example, ESC[1;31m sets the text attribute to bold red. The terminal processes these sequences as they arrive in the serial stream, maintaining an internal state machine that tracks the current text attributes, cursor position, and screen contents. ANSI escape sequences are part of the ECMA-48 standard (also known as ANSI X3.64), which defines the complete set of control functions for coded character sets used by video terminals. -- ANSI X3.64 / ECMA-48

### Topic 2: Select Graphic Rendition (SGR)

The SGR command (ESC[<params>m) sets text display attributes. Multiple parameters can be combined in a single sequence separated by semicolons. The key attribute codes are: 0 (reset all attributes to default), 1 (bold/bright), 4 (underline), 5 (blink), 7 (reverse video -- swap foreground and background), and 8 (concealed/hidden). The bold attribute (1) serves double duty in the CGA color model: it does not just make text thicker, it selects the bright variant of the current foreground color. ESC[0m resets all attributes, which is why ANSI art files typically begin and end with a reset sequence. BBS software relied heavily on SGR for colorizing menus, status bars, and message headers, while ANSI artists used precise SGR sequences to paint with the full 16-color foreground palette. -- ANSI X3.64 / ECMA-48

### Topic 3: CGA 16-Color Palette

The Color Graphics Adapter (CGA) palette used by ANSI terminals consists of 8 base colors and 8 bright variants, for 16 foreground colors total. The base foreground colors are encoded as SGR parameters 30 through 37: 30 (black), 31 (red), 32 (green), 33 (yellow/brown), 34 (blue), 35 (magenta), 36 (cyan), 37 (white/light gray). Background colors use parameters 40 through 47 in the same order but are limited to the 8 base colors -- there are no bright backgrounds in standard ANSI. To select a bright foreground color, combine bold (1) with the base color: ESC[1;31m produces bright red, while ESC[0;31m produces dark red. The yellow/brown distinction is notable: SGR 33 without bold renders as brown on CGA hardware, but ESC[1;33m renders as bright yellow. This 16-foreground, 8-background palette defined the complete color space of ANSI art for over a decade. -- ANSI X3.64 / ECMA-48; CP437 / Unicode Mapping

### Topic 4: Cursor Positioning

Cursor positioning sequences allow ANSI art to place characters anywhere on the 80x25 terminal grid, not just sequentially left-to-right, top-to-bottom. The absolute positioning command ESC[<row>;<col>H moves the cursor to a specific row and column (1-indexed, with row 1, column 1 being the top-left corner). Relative movement commands move the cursor from its current position: ESC[<n>A moves up n rows, ESC[<n>B moves down, ESC[<n>C moves forward (right), and ESC[<n>D moves backward (left). ESC[s saves the current cursor position and ESC[u restores it, enabling the terminal to jump to a different location, draw something, and return. ANSI artists used cursor positioning for non-linear drawing and animation effects, while BBS software used it to update specific screen regions (like a status bar) without redrawing the entire screen. -- ANSI X3.64 / ECMA-48

### Topic 5: Erase Commands

Erase commands clear portions of the screen. ESC[2J clears the entire screen and moves the cursor to the home position (row 1, column 1). ESC[J (without the 2) clears from the cursor to the end of the screen. ESC[K clears from the cursor to the end of the current line, which is useful for overwriting a previous line of text without leaving trailing characters. ESC[1K clears from the beginning of the line to the cursor, and ESC[2K clears the entire current line. BBS software sent ESC[2J at the start of each new screen to clear the previous content, and door games used line-level erase commands to update individual display elements like score counters and status messages. -- ANSI X3.64 / ECMA-48

### Topic 6: CP437 Character Set

Code Page 437 is the 256-entry character encoding used by the IBM PC's BIOS and the visual vocabulary of ANSI art. Unlike ASCII, which defines only 95 printable characters (0x20-0x7E), CP437 assigns visible glyphs to all 256 byte values. Bytes 0x00 through 0x1F, which are ASCII control characters, instead display graphical characters: smiley faces (0x01, 0x02), hearts (0x03), diamonds (0x04), clubs (0x05), spades (0x06), and other symbols. Bytes 0x80 through 0x9F contain accented Latin letters and currency symbols. The most important range for ANSI art is 0xB0 through 0xDF: the box-drawing characters, block elements, and line-drawing characters that form the structural vocabulary of terminal-based visual design. CP437 is a single-byte encoding where every byte maps to exactly one glyph, making it fast to render but incompatible with Unicode without an explicit mapping table (see `shared/cp437.ts` for the conversion implementation). -- CP437 / Unicode Mapping

### Topic 7: Box Drawing and Block Elements

Characters in the CP437 range 0xB0 through 0xDF are the building blocks of ANSI art's visual structures. The block elements -- light shade (0xB0), medium shade (0xB1), dark shade (0xB2), and full block (0xDB) -- create gradient effects and solid fills. The half-block characters -- upper half block (0xDF) and lower half block (0xDC) -- effectively double the vertical resolution by allowing two colors in a single character cell (foreground for the block half, background for the empty half). Single-line box-drawing characters (0xB3, 0xC0-0xC4, 0xD9, 0xDA, 0xBF) and double-line variants (0xBA, 0xBB-0xBC, 0xC8-0xCC, 0xCD, 0xCE) create window borders, table frames, and decorative panels. ANSI artists combined these structural characters with color sequences to build interfaces that looked remarkably sophisticated for a text-mode display. The constraint of 256 fixed characters forced creative solutions -- artists learned which character combinations, when colored carefully, could simulate curves, shadows, and three-dimensional depth. -- CP437 / Unicode Mapping

### Topic 8: SAUCE Metadata

The Standard Architecture for Universal Comment Extensions (SAUCE) is a 128-byte metadata record appended to the end of an ANSI art file. Placing metadata at the end rather than the beginning means the file can be displayed by any terminal that simply reads bytes sequentially -- the SAUCE record is reached only after the art itself. The record begins with the 5-byte identifier "SAUCE" (bytes 0x53 0x41 0x55 0x43 0x45), followed by a 2-byte version field ("00"), then fixed-width fields: Title (35 bytes), Author (20 bytes), Group (20 bytes), Date (8 bytes, YYYYMMDD format), FileSize (4 bytes, little-endian), DataType (1 byte), FileType (1 byte), TInfo1-TInfo4 (2 bytes each), Comments (1 byte, count of comment lines), TFlags (1 byte), and TInfoS (22 bytes). For ANSI art (DataType 1, FileType 1), TInfo1 contains the width in columns and TInfo2 contains the number of lines. The SAUCE specification was created by ACiD Productions in 1994 and remains the standard metadata format for ANSI art files (see `shared/sauce.ts` for the parser implementation). -- SAUCE spec, field definitions

### Topic 9: The ANSI Art Scene

The ANSI art scene was a competitive digital art community that flourished from roughly 1990 to 1998. Art groups -- ACiD Productions (ANSI Creators in Demand), iCE (Insane Creators Enterprise), Dark, and Fire among the most prominent -- released monthly "art packs" containing their members' work. These packs were distributed through BBS file sections and FidoNet-connected distribution networks. Artists created their work using specialized editors: TheDraw (by Ian Davis, 1986) was the original ANSI editor, followed by ACiDDraw and PabloDraw. The community had its own infrastructure, including artpack review boards, talent recruitment, and inter-group rivalries. The art itself ranged from BBS login screens and file area headers (functional art) to standalone artistic compositions (creative art). The scene's influence persists in the demoscene and in modern terminal art created with tools like moebius and the Sixteen Colors archive. -- SAUCE spec

### Topic 10: The Rendering Pipeline

Rendering an ANSI art file requires a multi-stage pipeline. The raw file is a sequence of bytes that must be processed in order. First, each byte is decoded from CP437 to its Unicode equivalent using the 256-entry mapping table. Simultaneously, the byte stream is scanned for ESC (0x1B) characters that begin escape sequences. When an ESC is found, the parser enters a state machine: it reads the "[" CSI introducer, accumulates numeric parameters and semicolons, and dispatches on the terminal letter (m for SGR, H for cursor position, J for erase, etc.). The SGR handler maintains a current attribute state: foreground color, background color, bold, blink, underline, and reverse. Each non-escape character is placed at the current cursor position with the current attributes, then the cursor advances right (wrapping to the next line at column 80). The result is a two-dimensional grid of (character, foreground, background) tuples that can be rendered to screen. The existing shared parsers (`shared/ansi.ts`, `shared/cp437.ts`) implement this pipeline as reusable TypeScript functions. -- ANSI X3.64 / ECMA-48; CP437 / Unicode Mapping

### Topic 11: ANSI Animation

ANSI animation exploits cursor positioning to create the illusion of movement within the terminal. By repositioning the cursor to a previously drawn area and overwriting it with new characters and colors, an ANSI file can produce animated title screens, scrolling banners, and visual transitions. At BBS modem speeds, the serial transmission delay itself created an animation effect: characters appeared progressively as they arrived, and skilled artists designed their work to "draw" in an aesthetically pleasing order. This is called "draw speed" or "sauce speed" in scene terminology. Some BBS software implemented configurable display speed to simulate the modem-speed reveal even on local connections. Animation was limited by the lack of a frame buffer or double-buffering -- every cursor movement and character write was visible immediately, which could cause flicker during complex redraws. -- ANSI X3.64 / ECMA-48

### Topic 12: ANSI.SYS versus Modern Terminals

The original DOS ANSI driver (ANSI.SYS, loaded via CONFIG.SYS) interpreted ANSI escape sequences for the IBM PC's text-mode display. ANSI.SYS supported a subset of ECMA-48: SGR for color and attributes, cursor positioning, and screen clearing, but not scrolling regions or some advanced cursor operations. It also added the non-standard ESC[<n>p sequence for key remapping, which was a security risk (a downloaded ANSI file could remap keyboard keys). Modern terminal emulators (xterm, VT100-compatible terminals) support a much wider set of ECMA-48 sequences including 256-color and true-color extensions, scrolling regions, alternate screen buffers, and mouse tracking. However, they do not natively support CP437 -- they use Unicode. Rendering legacy ANSI art in a modern terminal requires explicit CP437-to-Unicode character mapping and may not display identically due to font differences (CP437's box-drawing characters assume a specific character cell aspect ratio that modern monospace fonts may not match). -- ANSI X3.64 / ECMA-48; CP437 / Unicode Mapping

## Learn Mode Depth Markers

### Level 1: Practical

> ANSI art uses escape sequences to set text color and position the cursor on screen. The CSI (ESC [) introduces every command: ESC[1;31m sets bright red text, ESC[2J clears the screen, ESC[10;20H moves the cursor to row 10, column 20. CP437's box-drawing characters create borders and visual structure. -- ANSI X3.64 / ECMA-48

### Level 2: Reference

> See ECMA-48 for the full catalog of control functions including SGR parameter codes, cursor movement sequences, and erase operations. The SAUCE specification defines the 128-byte metadata record (title, author, group, dimensions) appended to ANSI art files. The CP437 / Unicode Mapping provides the authoritative 256-entry byte-to-codepoint table. -- ANSI X3.64 / ECMA-48; SAUCE spec; CP437 / Unicode Mapping

### Level 3: Technical

> CSI is ESC [ (0x1B 0x5B). SGR foreground: 30-37 (base), with bold(1) for bright variants. Background: 40-47 (base only). SAUCE record: 128 bytes at EOF, starts with "SAUCE" (5 bytes), version "00" (2 bytes), Title (35), Author (20), Group (20), Date (8), FileSize (4), DataType (1), FileType (1), TInfo1-4 (2 each), Comments (1), TFlags (1), TInfoS (22). CP437 box-drawing range: 0xB0-0xDF. Half-blocks (0xDC lower, 0xDF upper) double effective vertical resolution. Rendering pipeline: raw bytes -> CP437 decode -> ESC sequence state machine -> (char, fg, bg) grid. -- ANSI X3.64 / ECMA-48; SAUCE spec, field definitions; CP437 / Unicode Mapping
