# Module 5: Door Games

> **Tier**: 3 | **Safety Mode**: Annotate

## Overview

Door games are the application layer of the BBS ecosystem. A "door" is an external program launched by the BBS software that inherits the caller's serial connection (Module 1) and uses ANSI art (Module 2) for its user interface. The BBS software writes a drop file containing the caller's information -- name, time remaining, terminal capabilities -- then spawns the door program, which reads the drop file, communicates with the caller through the serial port (or through a FOSSIL driver abstraction), and returns control to the BBS when the user exits. This mechanism turned BBS systems into gaming platforms: hundreds of door games were written for DOS, ranging from simple text adventures to complex multiplayer strategy games with inter-BBS leagues connected via FidoNet (Module 3). -- DOOR.SYS / CHAIN.TXT; FOSSIL Driver Specification

The door game era (roughly 1987-1998) produced distinctive design patterns driven by the constraints of single-line BBS access: per-day-turn limits prevented any-hours players from dominating, asynchronous multiplayer via shared state files enabled competition without simultaneous connections, and ANSI art UI design created visually rich game interfaces within the 80x25 terminal grid. Games like Legend of the Red Dragon (LORD), TradeWars 2002, and Barren Realms Elite became cultural touchstones of the BBS community, and their design patterns anticipated modern web game mechanics like daily energy systems and asynchronous PvP. -- DOOR.SYS / CHAIN.TXT; Jason Scott BBS Documentary

## Topics

### Topic 1: The Door Concept

A door is an external program that runs outside the BBS software but communicates with the remote caller through the BBS's serial connection. When a caller selects a door from the BBS menu, the BBS software performs a handoff: it writes session information to a drop file, suspends its own terminal handling, and executes the door program. The door program reads the drop file to learn about the caller (name, time limit, terminal type), takes over the serial port, runs its logic, and exits. The BBS software then resumes, detecting the door's exit and returning the caller to the menu. This architecture was remarkably similar to CGI (Common Gateway Interface) in early web servers -- both involve a host program passing session state to an external process through files and environment variables. -- DOOR.SYS / CHAIN.TXT

### Topic 2: DOOR.SYS Drop File Format

DOOR.SYS is the most widely supported drop file format, consisting of 52 lines of plain text (CRLF-terminated). Originally defined as 31 lines by Kenny Gardner for GAP BBS Software, it was extended to 52 lines to accommodate additional session data. Key fields include: Line 1 (COM port, e.g., "COM1:" or "COM0:" for local mode), Line 2 (baud rate, e.g., "38400"), Line 7 (caller's full name), Line 8 (caller's location/city), Line 12 (security level as integer), Line 18 (time remaining in minutes), Line 20 (screen dimensions), and Line 23 (ANSI mode: "GR" for graphics-capable, "NG" for no graphics). The format is positional -- each field is on a specific line number, with no field names or delimiters. A door program reads lines sequentially and assigns them to variables by position. The simplicity of this format (plain text, one field per line) made it easy to implement in any language, contributing to its near-universal adoption. -- DOOR.SYS / CHAIN.TXT

### Topic 3: CHAIN.TXT Drop File Format

CHAIN.TXT is the drop file format used by WWIV BBS software. It consists of 32 lines of plain text with fields including: user number, user alias, real name, callsign (ham radio), age, sex, gold (in-game currency), last date on, screen width, screen length, security level, number of uploads/downloads, ANSI support (1 or 0), time remaining in seconds, and the full paths to the user record file and system directories. CHAIN.TXT differs from DOOR.SYS in several ways: it includes WWIV-specific fields (user number, callsign), expresses time remaining in seconds rather than minutes, and includes system path information that allows the door program to access the BBS's data files directly. Some door games supported both formats, detecting which was present at startup. -- DOOR.SYS / CHAIN.TXT

### Topic 4: Comparing Drop File Formats

The BBS era produced over a dozen drop file formats because there was no standards body to enforce a single specification. Beyond DOOR.SYS and CHAIN.TXT, notable formats include: CALLINFO.BBS (Wildcat! BBS), DORINFOx.DEF (RBBS-PC, where x is the node number), SFDOORS.DAT (Spitfire BBS), and PCBOARD.SYS (PCBoard). Each BBS software package defined its own format, and door authors had to support multiple formats to maximize compatibility. This led to door "wrapper" libraries (like OpenDoors, DoorFrame, and DDPlus) that abstracted the differences: the library detected which drop file was present, parsed it, and provided a uniform API to the door program. The proliferation of formats is a case study in the costs of absent standardization -- every door author duplicated the same parsing logic for each format they supported. -- DOOR.SYS / CHAIN.TXT

### Topic 5: FOSSIL Driver Interface

The FOSSIL (Fido-Opus-Seadog Standard Interface Layer) driver, Version 5 (February 1988), provides a standardized API for serial port access. On IBM PC compatibles, the BIOS INT 14h interface for serial communication was minimal and lacked buffering, making it inadequate for BBS and door software. FOSSIL replaces INT 14h with an extended API that provides: initialization and deinitialization (functions 04h/05h), character send and receive with buffering (functions 01h/02h), status checking (function 03h), baud rate setting (function 00h), flow control management, and timer services. Door programs that used FOSSIL instead of direct hardware access were portable across different PC serial hardware (8250, 16550, multi-port boards). The FOSSIL driver loaded as a TSR (Terminate and Stay Resident) program, typically from AUTOEXEC.BAT, and intercepted INT 14h calls. BNU, X00, and ADF were popular FOSSIL implementations. -- FOSSIL Driver Specification

### Topic 6: The Door Game Loop

A door game follows a characteristic execution pattern that reflects the constraints of BBS operation. The loop proceeds as: (1) read the drop file (DOOR.SYS/CHAIN.TXT) to get the caller's identity, time remaining, and terminal capabilities; (2) initialize the serial port (directly or via FOSSIL) at the baud rate specified in the drop file; (3) display the main menu using ANSI escape sequences for color and positioning; (4) enter the input loop -- read keystrokes from the serial port, process game commands, update display; (5) periodically check the carrier detect signal (DCD) to handle disconnections and the time remaining counter to enforce the BBS's time limit; (6) on exit, write updated player state to the game's data files and optionally append to a score file; (7) terminate, returning control to the BBS software. The DCD check is critical: if the caller hangs up mid-game, the door must detect the lost carrier and exit cleanly rather than waiting indefinitely for input. -- DOOR.SYS / CHAIN.TXT; FOSSIL Driver Specification

### Topic 7: Per-Day-Turn Game Design

The most successful BBS door games used a per-day-turn system: each player receives a fixed number of actions (turns, fights, moves) per real-world day. This design pattern emerged from the fundamental constraint that BBS systems typically had 1-4 phone lines, meaning players had limited time per session. Per-day turns solved several problems simultaneously: they prevented dedicated players from monopolizing the phone line, they ensured fair competition regardless of how many hours per day a player could call, and they created a natural daily engagement cycle (call the BBS, take your turns, check on your progress, log off). Legend of the Red Dragon (LORD, by Seth Robinson) gave players a set number of forest encounters per day. TradeWars 2002 (by Gary Martin) allocated movement turns per day. Barren Realms Elite distributed resource production and attack turns daily. This pattern directly anticipated modern mobile game "energy" and "stamina" systems. -- Jason Scott BBS Documentary

### Topic 8: Multiplayer via Shared State

BBS door games achieved multiplayer interaction without real-time networking by using shared state files on the BBS's hard disk. When a player took actions that affected other players (attacking their territory, trading with them, leaving messages), the door program wrote updates to data files that all players shared. The next time an affected player logged in, their game session would read the updated state and reflect the changes. This asynchronous model meant players never needed to be online simultaneously -- they interacted through the shared game world, much like leaving chess moves on a bulletin board. Some games (like LORD) even included an "inn" where players could leave messages for each other, creating a social space within the game. Inter-BBS leagues extended this further: FidoNet echomail (Module 3) carried game data packets between BBS systems, allowing players on different BBS nodes in different cities to compete in the same game universe. -- DOOR.SYS / CHAIN.TXT; FTS-0001

### Topic 9: ANSI Game UI Design

Door games used the ANSI escape sequences and CP437 character set from Module 2 to create their user interfaces. The 80x25 terminal grid was the game's display canvas, and games used several UI conventions: a top-line status bar showing player stats (health, gold, turns remaining), a bottom-line input prompt, and a central area for game content rendered with box-drawing characters and color. Maps in games like TradeWars 2002 used colored ASCII characters to represent sectors, ports, and other players. LORD used full-screen ANSI art for location screens (the inn, the forest, the arena). Menu systems used numbered or lettered choices highlighted with color. Animation was limited to cursor repositioning (ESC[row;colH) for updating specific screen elements without redrawing the entire screen. The constraint of fixed-width CP437 characters forced creative solutions: careful color choices could suggest depth and atmosphere within the severe limitation of 16 foreground colors and 256 character shapes. -- ANSI X3.64 / ECMA-48; CP437 / Unicode Mapping

### Topic 10: Notable Door Games

The BBS door game ecosystem produced hundreds of titles, with a handful achieving widespread popularity across thousands of BBS systems. **Legend of the Red Dragon (LORD)**, by Seth Robinson (1989), was a fantasy RPG where players explored a forest, fought monsters, and battled each other, with a charming sense of humor and deep social features including the inn messaging system. **TradeWars 2002**, by Gary Martin (1986, later Chris Sherrick), was a space trading and combat game with a complex economy, sector-based map, and team play. **Barren Realms Elite (BRE)**, by Mehul Patel, was a kingdom-building strategy game that pioneered inter-BBS league play over FidoNet. **Usurper**, by Jakob Dangaard Olesen, was a fantasy combat RPG with team-based dungeon exploration. **Solar Realms Elite (SRE)** was a space empire strategy game similar to BRE. These games shared common design principles: per-day turns, shared-state multiplayer, ANSI art interfaces, and game data stored in flat files on the BBS's local disk. -- Jason Scott BBS Documentary

The FOSSIL Driver Specification Version 5 documents the INT 14h replacement API for portable serial I/O. The Jason Scott BBS Documentary (2005) provides cultural context for the door game community. -- DOOR.SYS / CHAIN.TXT; FOSSIL Driver Specification; Jason Scott BBS Documentary

### Level 3: Technical

> DOOR.SYS: 52 lines, CRLF-terminated, positional fields. Line 1: COM port ("COM1:"), Line 2: baud ("38400"), Line 7: user name, Line 12: security level, Line 18: time remaining (minutes), Line 23: ANSI ("GR"/"NG"). Originally 31 lines (GAP BBS, Kenny Gardner), extended to 52. FOSSIL INT 14h functions: 00h (set baud), 01h (send char), 02h (receive char), 03h (status), 04h (init), 05h (deinit). Door game loop: read drop file -> init serial/FOSSIL -> ANSI display loop -> check DCD + time -> write state -> exit. Per-day-turn games: LORD (Seth Robinson), TradeWars 2002 (Gary Martin), BRE (Mehul Patel). -- DOOR.SYS / CHAIN.TXT; FOSSIL Driver Specification
