# CODE-101 Assessment Framework

## Rubric Levels

### Beginning

**Algorithm Understanding:** Student follows given algorithms and understands basic sequences. Can execute a simple set of steps but struggles to generalize or apply to new situations.

**Code Implementation:** Student writes simple programs but struggles with syntax. Code often has errors. Needs explicit instruction to move forward.

**Problem-Solving:** Student solves well-defined problems with step-by-step guidance. Has difficulty breaking down larger problems independently.

**Debugging:** Student recognizes when code doesn't work but often tries random fixes. Limited systematic approach to finding errors.

### Developing

**Algorithm Understanding:** Student designs simple algorithms and understands why certain structures (loops, conditionals) are chosen. Can apply familiar patterns to similar problems.

**Code Implementation:** Student writes working programs with mostly correct syntax. Asks good questions when stuck. Understands most language constructs but needs reference materials.

**Problem-Solving:** Student breaks problems into parts and makes reasonable design choices. Plans basic projects but struggles with integration. Iterates but not always systematically.

**Debugging:** Student uses print statements and traces through code logically. Can identify many errors themselves. Still relies on trial-and-error for complex issues.

### Proficient

**Algorithm Understanding:** Student designs efficient algorithms and considers trade-offs between approaches. Can explain WHY an algorithm works, not just THAT it works. Applies patterns creatively.

**Code Implementation:** Student writes clean, readable code. Chooses appropriate data structures. Code is well-organized with meaningful variable names. Follows style conventions.

**Problem-Solving:** Student plans projects effectively, tests assumptions, and iterates based on evidence. Handles multi-step problems. Refactors code for clarity. Completes projects largely independently.

**Debugging:** Student debugs systematically using appropriate tools (print statements, debuggers, logging). Can find their own errors quickly. Understands root causes, not just symptoms.

### Advanced

**Algorithm Understanding:** Student understands complexity theory and can optimize algorithms. Proves correctness. Recognizes when different algorithms apply to different problems. Invents novel solutions.

**Code Implementation:** Student writes modular, maintainable code. Follows design patterns. Designs for extensibility. Code is clear enough that others can easily understand and modify it.

**Problem-Solving:** Student solves novel problems and teaches others computational thinking. Builds complex projects with multiple components. Considers performance, user experience, and edge cases.

**Debugging:** Student debugs subtle bugs involving system interactions, performance bottlenecks, and race conditions. Uses profilers and advanced debugging techniques. Can mentor others in debugging.

## Formative Assessment (During Learning)

Formative assessment guides instruction. Look for these signals:

### 1. Code Understanding & Explanation
- Can the student explain what code does, not just "it works"?
- Do they understand the difference between what code says and what it does?
- Can they trace through logic step-by-step?

**Look for:** Student explaining code in their own words; asking clarifying questions; noticing that "no error" doesn't mean "correct"

### 2. Testing & Verification
- Do they test their code, or assume it works?
- Do they think of edge cases?
- Do they verify output matches expectations?

**Look for:** Student running code with different inputs; trying to break their own code; checking outputs carefully

### 3. Response to Errors
- How do they respond when code fails?
- Do they panic, give up, or approach systematically?
- Do they use error messages to guide fixes?

**Look for:** Student reading error messages; trying one thing at a time; learning from mistakes

### 4. Pattern Recognition
- Do they notice when similar algorithms apply to different problems?
- Can they generalize from one example to many?
- Do they see connections across modules?

**Look for:** Student saying "I've done something like this before"; recognizing patterns; applying old knowledge to new problems

### 5. Curiosity & Persistence
- Are they asking "Why?" and "What if?" rather than just following instructions?
- Do they persist through difficulty without giving up?
- Do they explore beyond requirements?

**Look for:** Student asking questions; experimenting; extending projects; reading documentation

## Summative Assessment (Evidence of Mastery)

Summative assessment documents what the student knows and can do. Use these categories:

### Portfolio Evidence

**What to include:**

1. **Working Programs** — 2-3 programs that solve real problems (not toy exercises)
2. **Well-Documented Code** — Cleaned-up versions with comments explaining logic
3. **Project Write-ups** — Documenting:
   - What problem it solves
   - Why they chose their approach
   - What they learned
   - What they'd do differently next time
4. **Testing Evidence** — Test cases, debugging notes, how they verified correctness
5. **Refactoring Examples** — Before/after showing code improvements
6. **Reflection** — What went wrong? How did they fix it? What was hardest?

### Project Examples Demonstrating Mastery

Pick one of these completion types:

**Option A: Complete Game**
- A playable game with win conditions, scoring, or levels
- Responds to user input correctly
- Code is organized into functions or classes
- Has been tested and debugged

**Option B: Useful Tool**
- A program that automates something useful (data processor, file organizer, calculator)
- Handles multiple use cases
- Has clear input/output
- Could be used by non-programmers (with instructions)

**Option C: Data Analysis Project**
- Collects or loads data
- Analyzes it with code
- Produces visualizations or report
- Draws conclusions from the data

**Option D: Algorithmic Solution**
- Solves a challenging problem (puzzle, challenge problem, real-world issue)
- Efficient solution chosen (not brute force)
- Tested on multiple cases
- Code is clear enough others can understand approach

**Option E: Teaching Others**
- Taught a younger student or friend a coding concept
- Created materials to help them learn (tutorial, video, guide)
- Student can demonstrate what they taught

### Assessment by Module

#### Computational Thinking (M1)
- Can break a complex task into step-by-step algorithm
- Recognizes patterns and generalizes them
- Understands why one algorithm might be better than another
- Can design algorithm before coding

**Evidence:** Flowchart, pseudocode, explanation of algorithmic choices

#### Programming Fundamentals (M2)
- Writes syntactically correct code
- Uses variables, loops, conditionals, and functions appropriately
- Code runs without errors
- Can transition from visual to text-based programming

**Evidence:** Working code samples in multiple environments; clean syntax; good naming

#### Building Projects (M3)
- Plans projects with goals and checkpoints
- Iterates: build, test, improve
- Debugs systematically
- Code is organized and maintainable

**Evidence:** Complete working project; project plan; testing evidence; refactored code

#### Algorithms & Efficiency (M4)
- Understands complexity (Big-O notation)
- Analyzes algorithm efficiency
- Chooses efficient solutions
- Optimizes when appropriate

**Evidence:** Complexity analysis; efficiency comparison; optimized code; explanations of why

#### Computing & Society (M5)
- Thinks critically about technology's impact
- Recognizes ethical issues in code
- Practices digital citizenship responsibly
- Understands open source and collaboration

**Evidence:** Reflection essays; debate participation; discussion contributions; ethical analysis

## Formative Assessment Activities

### Code Reading Circles (Ongoing)
Read and discuss published code, focusing on clarity, efficiency, design choices.

**Frequency:** Weekly or bi-weekly
**Duration:** 20-30 minutes
**Assessment Focus:** Understanding, pattern recognition, analysis

### Peer Code Review (2-3x per term)
Student reviews peer's code, gives feedback, receives feedback.

**Frequency:** After major projects
**Duration:** 30 minutes per review
**Assessment Focus:** Communication, understanding, suggestion quality

### Algorithm Explanation Interview (2-3x per term)
One-on-one conversation where student explains their algorithm or code.

**Frequency:** After each major project
**Duration:** 10-15 minutes
**Assessment Focus:** Understanding, reasoning, debugging approach

### Debugging Challenges (Monthly)
Given code with bugs, student identifies and fixes errors.

**Frequency:** Monthly or bi-monthly
**Duration:** 20-30 minutes
**Assessment Focus:** Debugging strategies, error recognition, systematic thinking

---

## Summative Assessment Activities

### Portfolio Review (End of Unit/Year)
Review student's portfolio of code, projects, reflections, improvements.

**Frequency:** End of each unit or year
**Duration:** 30-60 minutes
**Assessment Focus:** Growth, mastery across modules, evidence of learning

### Capstone Project (End of Year)
Student designs and completes a significant project demonstrating mastery.

**Frequency:** End of course
**Duration:** 1-2 weeks
**Assessment Focus:** All competencies: planning, coding, testing, reflection

### Concept Test (Optional, after M4)
Written or practical assessment of algorithm understanding and efficiency analysis.

**Frequency:** After completing Algorithms module
**Duration:** 45-60 minutes
**Assessment Focus:** Conceptual understanding, complexity analysis

## Grading Rubric (If Needed)

If assigning grades, use this holistic rubric:

| Grade | Evidence |
|-------|----------|
| **A (Excellent)** | Advanced level in algorithm understanding, code quality, and problem-solving. Projects are complete, tested, and well-documented. Can explain and justify choices. |
| **B (Good)** | Proficient level. Code works, projects are complete. Explanations are clear. Minor improvements in efficiency or testing. |
| **C (Satisfactory)** | Developing level. Code mostly works. Projects are attempted. Can explain basic concepts. Needs guidance for complex problems. |
| **D (Below Standard)** | Beginning level. Code has significant errors. Struggles with explanations. Needs explicit guidance for most tasks. |
| **F (Not Passing)** | Little evidence of understanding. Code doesn't work. Major gaps in knowledge. |

## Tips for Assessment

### Emphasize Process Over Product
- Value the thinking more than the final code
- Celebrate good questions and productive debugging
- Recognize learning from mistakes

### Use Multiple Assessment Types
- Code samples show capability
- Explanations show understanding
- Projects show integration
- Reflections show growth

### Make Rubrics Clear
- Share expectations upfront
- Use examples of work at each level
- Let students self-assess first

### Provide Feedback Quickly
- Debugging feedback most useful within 24 hours
- Celebrate specific improvements
- Guide next steps clearly

### Recognize Different Strengths
- Some students excel at algorithm design but struggle with syntax
- Some love debugging, others prefer clean code first time
- Some think visually, others textually
- Assess across these dimensions

## Self-Assessment for Students

Ask students to reflect on their own learning:

1. **What concept took the longest to understand?** Why?
2. **When did debugging feel natural?** When was it frustrating?
3. **What project are you most proud of?** What made it work?
4. **What would you like to understand better?**
5. **What coding skill do you feel most confident in now?**
6. **How has learning to code changed how you think about problems?**

Student reflection is valuable assessment data and builds metacognitive awareness.
